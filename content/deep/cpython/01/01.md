# 1.CPython 是什么

CPython 是 Python 编程语言的主要实现。这是大多数人从 python.org、包管理器、Linux 发行版、Homebrew、pyenv、Docker 映像或大多数云运行时安装 Python 时使用的实现。

CPython 有两个角色。

首先，它实现了Python语言。它定义了Python源代码如何解析、编译、执行以及连接到标准库。

其次，它充当 Python 生态系统的实际参考点。其他实现可能遵循语言规范，但 CPython 是大多数包、工具、调试器、分析器和 C 扩展测试的实现。

CPython 源代码树包含解释器、编译器、对象系统、运行时、C API、标准库、测试套件、文档和构建机制。官方开发人员指南将 CPython 描述为不断变化的代码库，并在源代码树中维护内部文档。 （[Python 开发人员指南][1]）

## 1.1 Python 语言与 CPython 实现

Python 是一种语言。

CPython 是该语言的一种实现。

这种区别很重要，因为 Python 语言描述了可见的行为：语法、名称、表达式、语句、对象、模块、异常、迭代、异步行为、导入等等。

CPython 描述了一种使该行为在真实机器上运行的具体方法。

像这样的 Python 程序：```python
def add(a, b):
    return a + b

print(add(2, 3))
```是语言级别的Python。语言告诉我们`def`创建一个函数对象，`return`返回一个值，`+`调用加法语义，并且`print`编写文本表示。

CPython 通过较低级别的机制实现这些操作：```text
source text
    ↓
tokens
    ↓
parser
    ↓
abstract syntax tree
    ↓
symbol table
    ↓
code object
    ↓
bytecode
    ↓
evaluation loop
    ↓
object operations
    ↓
C functions and machine code
```程序员通常只能看到该管道的顶部。 CPython 开发人员在整个管道中工作。

## 1.2 CPython 既是编译器又是解释器

CPython 通常被称为解释器。这是正确的，但不完整。

CPython首先将Python源代码编译为字节码。然后它解释该字节码。

编译阶段不会产生通常的提前编译器意义上的本机机器代码。它产生`code`包含字节码指令、常量、名称、变量布局信息、异常表、行号信息和运行时所需的元数据的对象。

像这样的文件：```python
x = 10
y = 20
print(x + y)
```被转换成字节码，大致形状如下：```text
load constant 10
store name x
load constant 20
store name y
load name print
load name x
load name y
binary add
call
```实际的字节码在不同的 Python 版本中发生了变化，但模型仍然存在：CPython 将源代码降低到紧凑的指令流中，并在虚拟机中执行该流。

您可以使用以下命令检查字节码`dis`:

```python
import dis

def add(a, b):
    return a + b

dis.dis(add)
```这提供了 CPython 执行模型的 Python 级别视图。

## 1.3 CPython 虚拟机

CPython虚拟机是一个基于堆栈的字节码解释器。

大多数字节码指令在评估堆栈上运行。指令压入值、弹出值、调用函数、创建容器、分支、引发异常和返回结果。

例如，评估：```python
a + b
```需要 CPython 来加载`a`， 加载`b`，并应用二进制加法。从概念上讲：```text
LOAD_FAST a      push local variable a
LOAD_FAST b      push local variable b
BINARY_OP +      pop two values, add them, push result
```该堆栈机存在于当前正在执行的帧内。帧存储函数调用的执行状态：局部变量、值堆栈、指令指针、代码对象、全局变量、内置函数和异常状态。

在运行时，CPython 重复执行此模式：```text
fetch next bytecode instruction
decode instruction and operand
dispatch to implementation
operate on Python objects
update frame state
continue or return
```核心解释器循环是 CPython 最重要的部分之一。它将字节码与对象行为连接起来。

## 1.4 CPython 主要用 C 和 Python 编写

CPython主要用C实现，标准库中有大量Python代码。

C代码实现核心运行时：```text
object representation
reference counting
garbage collector
interpreter loop
compiler
parser support
built-in types
C API
threading primitives
import machinery support
memory allocators
```Python 代码实现了大部分标准库：```text
collections
pathlib
asyncio
typing
unittest
importlib portions
dataclasses
logging
email
venv
many utility modules
```有些模块同时具有 Python 和 C 实现。 C 版本通常用于提高性能，而 Python 版本可能是为了清晰、可移植性、测试或后备行为而存在。

这种混合设计很重要。 CPython 不仅仅是运行 Python 文件的 C 程序。它是C级和Python级组件协作的运行时。

## 1.5 CPython 定义内存中的对象模型

Python 中的每个值都是一个对象。

在 CPython 中，每个对象都以 C 级对象头开始。该标头至少存储引用计数和指向对象类型的指针。

从概念上讲：```c
typedef struct {
    Py_ssize_t ob_refcnt;
    PyTypeObject *ob_type;
} PyObject;
```可变大小的对象使用更大的标头：```c
typedef struct {
    PyObject ob_base;
    Py_ssize_t ob_size;
} PyVarObject;
```这意味着 Python 整数、字符串、列表、元组、函数、类、模块、异常、框架和生成器都有一个类型指针。类型对象描述了该对象支持哪些操作。

例如：```python
x = 42
```创建或引用 Python 整数对象。 CPython不存储`x`作为普通 Python 代码中的原始 CPU 整数。它存储对一个的引用`PyLongObject`。

这种区别解释了许多 Python 行为：```python
x = 42
y = x
```这里，`x`和`y`引用同一个对象。赋值将名称绑定到对象。默认情况下它不复制对象。

## 1.6 类型也是对象

在 CPython 中，类型也是对象。

表达式：```python
type(42)
```返回代表的对象`int`。

表达式：```python
type(int)
```回报`type`。

这不仅仅是语言技巧。 CPython 表示类型使用`PyTypeObject`，一个大型 C 结构，用于存储类型元数据和称为槽的函数指针。

类型对象回答以下问题：```text
How large is each instance?
How should instances be deallocated?
How is attribute lookup performed?
How is the object called?
How does addition work?
How does indexing work?
How does iteration work?
How is the object represented as text?
```对于列表，类型对象指向特定于列表的实现。

对于字典，它指向特定于字典的实现。

对于用户定义的类，CPython 动态构建类型对象。

这就是为什么 Python 的数据模型清晰地映射到 CPython 内部。方法如`__len__`, `__getitem__`, `__call__`, `__iter__`， 和`__add__`对应内部操作槽位。

## 1.7 名称是引用

Python 变量是绑定到对象的名称。

CPython 通过引用来实现这一点。

在Python中：```python
a = [1, 2, 3]
b = a
b.append(4)
print(a)
```输出是：```text
[1, 2, 3, 4]
```两个都`a`和`b`引用同一个列表对象。

在 CPython 级别，赋值`b = a`增加列表对象的引用计数。它不会重复该列表。

该模型影响性能和正确性。将列表传递给函数会传递对该对象的引用：```python
def push(xs):
    xs.append(4)

items = [1, 2, 3]
push(items)
```该函数接收对同一列表的另一个引用。

## 1.8 引用计数是主要的内存管理机制

CPython主要使用引用计数来回收对象。

每个对象都会记录有多少个强引用指向它。当计数达到零时，CPython 立即释放该对象。 Python文档将垃圾收集器描述为引用计数的补充，主要用于检测不可达的引用循环。 （[Python 文档][2]）

从概念上讲：```text
create object         refcount = 1
bind another name     refcount += 1
delete a name         refcount -= 1
refcount reaches 0    destroy object
```例子：```python
x = []
y = x
del x
del y
```该列表在之后仍然存在`del x`因为`y`仍然引用它。后`del y`，引用计数可以达到零，因此CPython可以释放列表。

在许多情况下，引用计数为 CPython 提供了可预测的对象终结。例如，当文件对象的最后一个引用消失时，它可能会立即关闭。但仅引用计数无法处理循环：```python
a = []
b = []
a.append(b)
b.append(a)
```这里，`a`和`b`互相参考。即使没有外部名称指向它们，它们的引用计数也可能保持非零。 CPython 的循环垃圾收集器就是为了查找并清理此类循环而存在的。可以参与循环的容器类型需要显式的 GC 支持。 （[Python 文档][3]）

## 1.9 CPython 有一个循环垃圾收集器

循环垃圾收集器处理相互引用但无法从实时程序根访问的对象组。

它主要跟踪容器对象：列表、字典、集合、包含引用的元组、函数、类、框架和其他可以指向其他对象的对象。

许多整数和字符串等原子对象通常不需要循环 GC 跟踪，因为它们不包含对其他 Python 对象的引用。

收集器与引用计数一起工作：```text
reference counting handles most objects immediately
cyclic GC handles unreachable reference cycles
```这种混合设计塑造了 CPython 的行为。内存往往释放得很快，但一些循环结构会稍后被回收。

这`gc`模块公开了该机器的一部分：```python
import gc

gc.collect()
print(gc.get_count())
print(gc.isenabled())
```该模块在研究对象生命周期、泄漏、终结器和循环行为时非常有用。

## 1.10 CPython 使用全局解释器锁

传统的 CPython 使用全局解释器锁，通常称为 GIL。

GIL 一次只允许一个线程在一个解释器中执行 Python 字节码。这简化了内存管理并保护内部解释器状态，特别是引用计数更新。

这并不意味着 CPython 不能使用线程。这意味着受 CPU 限制的 Python 字节码通常不会在同一解释器中跨多个操作系统线程并行运行。

线程仍然可以用于：```text
I/O-bound programs
network clients
file operations
concurrent waiting
C extensions that release the GIL
```对于 CPU 限制的并行性，Python 程序经常使用：```text
multiprocessing
native extensions
vectorized libraries
subinterpreters in newer designs
free-threaded builds in modern CPython work
```GIL 是具有语言级结果的实现细节。在 Python 级别出现并发的代码可能会在字节码执行期间序列化。

## 1.11 CPython 有公共 C API

CPython 公开了用于扩展模块和嵌入的 C API。

这是 CPython 最重要的生态系统功能之一。 NumPy、pandas、lxml、cryptography、Pillow 等库和许多数据库驱动程序都依赖于本机扩展机制。

最小的扩展功能可能会收到`PyObject *`值，解析参数，调用 C 代码，并返回一个新的`PyObject *`。

在此边界上，引用所有权至关重要。 CPython 区分借用引用、新引用和窃取引用。不正确的所有权处理可能会导致内存泄漏、释放后使用错误、崩溃或解释器状态损坏。

概念示例：```text
Py_INCREF     take another strong reference
Py_DECREF     release a strong reference
Py_XDECREF    release if pointer may be NULL
PyObject      base object pointer
PyTypeObject  type object structure
PyModuleDef   extension module definition
```C API 使 CPython 具有异常可扩展性，但它也将生态系统的大部分内容与 CPython 的对象表示和运行时行为联系在一起。

## 1.12 CPython 与 Python 语义不同

一些 CPython 行为是实现细节。

例如，通过引用计数立即销毁对象是 CPython 行为。 Python 代码通常应避免依赖于确切的销毁时间，除非该行为由语言或上下文管理器明确保证。

这是便携式的：```python
with open("data.txt") as f:
    data = f.read()
```这是 CPython 特定的计时：```python
f = open("data.txt")
data = f.read()
f = None
```在 CPython 中，赋值`None`如果删除最后一个引用，可能会立即关闭文件。在另一个 Python 实现中，清理可能会稍后发生。

一个好的 CPython 内部结构读者必须不断地分离三层：

|层 |示例|稳定性|
| ----------------------- | ---------------------------------------------------- | ---------------------------------- |
|语言语义|`for`循环调用迭代协议|非常稳定|
| CPython 行为 |对象使用引用计数 |稳定但特定于实现 |
| CPython 内部布局 |精确的结构字段和字节码操作码 |版本之间可以更改 |

## 1.13 CPython 和其他 Python 实现

CPython 并不是唯一的 Python 实现。

常见的替代方案包括：

|实施 |运行时基础 |主要区别|
| -------------- | ------------ | ---------------------------------------------------------------- |
| CPython | C |参考实现和最强的 C 扩展兼容性 |
| PyPy | RPython |面向JIT的实施|
| Jython | JVM | Java 平台上的 Python |
|铁蟒 | .NET | .NET 运行时上的 Python |
|微Python | C |用于微控制器和受限设备的 Python 子集 |

大多数 Python 用户运行 CPython。大多数 Python 包首先在 CPython 上进行测试。大多数本机扩展库都以 CPython 的 C API 为目标。

即使在学习 Python 作为一种语言时，这也使得 CPython 成为中心。

## 1.14 CPython 执行什么

CPython 不直接执行源文本。

它执行代码对象。

代码对象包含已编译的字节码和元数据：```text
bytecode instruction stream
constants
names
local variable names
free variables
cell variables
argument counts
stack size
filename
function name
line number table
exception table
flags
```您可以检查函数的代码对象：```python
def f(x):
    y = x + 1
    return y

code = f.__code__

print(code.co_name)
print(code.co_varnames)
print(code.co_consts)
print(code.co_names)
```函数对象将代码对象与运行时上下文包装在一起：```text
code object
globals dictionary
defaults
keyword defaults
closure cells
annotations
qualname
module
```这种区别很重要。相同的代码对象可以与不同的函数对象、全局变量、默认值或闭包重用。

## 1.15 CPython的主要执行单元

正在运行的 CPython 进程包含几个重要的执行单元。

|单位|意义|
| ------------ | -------------------------------------------------------------------------------- |
|流程|运行CPython的操作系统进程|
|运行时 |全局 CPython 运行时状态 |
|口译 |具有模块、内置函数、导入状态和执行上下文的解释器状态 |
|线程状态 |每线程执行状态 |
|框架|一个活动函数/模块/类执行 |
|代码对象|编译的字节码和元数据|
|对象|由字节码操作的运行时值 |

一个简单的函数调用创建或重用一个框架。该框架执行一个代码对象。该帧属于线程状态。线程状态属于解释器。解释器属于运行时。

从概念上讲：```text
CPython process
    runtime
        interpreter
            thread state
                frame
                    code object
                    value stack
                    locals
```这种层次结构出现在整个 CPython C 代码中。

## 1.16 CPython的标准库是实现的一部分

实际上，标准库并没有与 CPython 分开。

模块如`sys`, `types`, `gc`, `inspect`, `dis`, `importlib`， 和`ctypes`公开运行时内部结构或与运行时内部结构交互。

示例：```python
import sys
print(sys.getrefcount(object()))

import dis
dis.dis(lambda x: x + 1)

import gc
print(gc.get_objects())

import inspect
print(inspect.currentframe())
```有些模块是纯Python的。有些是用 C 实现的。有些同时使用两者。

解释器和标准库之间的界限是有意划分的。 CPython 在必要时将低级操作放在 C 中，并在可能的情况下将高级编排放在 Python 中。

## 1.17 CPython 有一个庞大的测试套件

CPython 包含一个大型回归测试套件。

该测试套件有几个目的：```text
protect language behavior
protect standard library behavior
catch platform-specific bugs
test C API behavior
test compiler and parser behavior
test memory management behavior
test edge cases from historical bugs
```对于内部工作，测试套件不是可选的。 CPython 更改可能会默默地影响解析、字节码、对象生存期、导入、异常、线程、异步行为或扩展兼容性。

正常的开发循环如下所示：```text
edit source
build CPython
run targeted tests
run broader tests
check reference leaks if relevant
inspect performance if relevant
```## 1.18 为什么 CPython 内部很重要

CPython 内部结构对于多种工作都很重要。

对于 Python 开发人员来说，内部结构解释了性能行为：```text
why attribute access has cost
why function calls matter
why small object allocation is fast
why dicts are central
why some loops are slower than C loops
why local variables are faster than globals
```对于扩展作者来说，内部知识是必需的：```text
reference ownership
type definitions
buffer protocol
error handling
thread state
GIL rules
module initialization
ABI compatibility
```对于工具作者，内部解释了如何构建调试器、分析器、跟踪器、linter、覆盖工具、字节码工具和导入挂钩。

对于 CPython 贡献者来说，内部结构是工作材料：对象布局、评估循环、编译器通道、内存分配器、平台抽象、测试和文档。

## 1.19 本书其余部分的心智模型

在阅读 CPython 源代码时使用此模型：```text
Python code is compiled into code objects.
Code objects contain bytecode.
Frames execute code objects.
The evaluation loop executes bytecode instructions.
Bytecode instructions manipulate PyObject pointers.
Objects point to type objects.
Type objects define behavior through slots.
Memory is managed mainly through reference counts.
The cyclic GC cleans up unreachable cycles.
The C API exposes much of this machinery to extensions.
```该模型足以适应几乎每个 CPython 子系统。

## 1.20 章节总结

CPython 是Python 的主要实现。它将 Python 源代码编译为字节码，在基于堆栈的虚拟机中执行该字节码，将所有运行时值表示为对象，主要通过引用计数管理内存，通过循环垃圾收集对其进行补充，并公开强大的 C API。

本书的其余部分将 CPython 视为一个具体的系统：编译器、虚拟机、对象运行时、内存管理器、扩展平台和标准库实现。
