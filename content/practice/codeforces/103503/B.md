 ## 问题理解

 我们得到的过程类似于二分搜索例程，但有一个不同之处：我们不是在固定的排序数组中搜索，而是有效地模拟二分搜索在概念决策空间上的行为方式。 每个测试用例都描述了一个场景，其中二分搜索过程在抽象的“搜索间隔”上运行，并且要求我们确定该过程会产生什么结果。 

更具体地解释，问题是跟踪经典二分搜索缩小过程在声明中给出的规则下如何演变。 每个测试用例都提供了逐步模拟该搜索过程所需的参数，输出是二分搜索稳定后达到的最终位置或结果。 

关键的结构思想是二分查找不仅仅依赖于实际值，而是依赖于反复缩小区间的比较。 这意味着整个行为取决于如何选择中点以及如何根据比较结果更新间隔。 

从约束的角度来看，涉及二分搜索模拟的问题几乎总是依赖于一定范围内的对数迭代。 如果输入大小达到约 10^5 或更大，则对每个可能的状态转换进行简单的逐步模拟将会太慢。 相反，我们期望每个测试用例最多有 O(log N) 或 O(log^2 N) 行为。 在典型的 Codeforces 限制下，任何接近每个查询 O(N) 的操作都是不可接受的。 

二分搜索模拟问题中的一种微妙的边缘情况是搜索间隔过早退化，或者中点舍入重复落在一个边界上。 例如，如果我们有一个区间 [1, 2]，则中点行为可能会导致搜索根据下限规则重复选择 1 或 2，从而可能导致无限循环或相差一错误。 

当更新规则不对称时，会出现另一种常见的失败情况。 例如，如果“向左移动”包括中间，但“向右移动”不包括中间，则像 [l, l+1] 这样的小间隔的行为与预期不同，并且幼稚的实现可能会错误地终止或返回错误的边界。 

## 方法

 该问题的强力解释是按字面模拟二分搜索过程，如下所示：维护左指针和右指针，计算 mid，应用更新规则，并继续直到收敛。 这是正确的，因为二分搜索本质上是迭代的，并且其行为完全由这些更新决定。 

然而，如果问题涉及许多测试用例或者过程在每次迭代中包含额外的嵌套计算，则简单的模拟可能会变得低效。 在最坏的情况下，每个测试用例都需要 O(log N) 个步骤，如果每个步骤都执行不平凡的工作，则多个查询的总复杂度可能会趋向于 O(N log N) 或更糟。 

关键的见解是二分搜索并不是真正地一一探索值，而是遵循完全由区间边界定义的确定性路径。 一旦我们认识到算法的状态完全由 (l, r) 描述，我们就可以将每个步骤视为该对的纯变换。 这消除了跟踪历史记录或重新计算这些边界之外的任何内容的需要。 

因此，最佳解决方案只是二分搜索的精确实现，并仔细注意边界更新和停止条件。 “搜索空间”实际上从未被枚举过； 它被压缩为区间状态上的对数转换。

 ## 算法演练

1. 使用给定边界初始化搜索区间，通常 l 作为左边界，r 作为右边界。 这代表了答案可以存在的完整空间。 
2. 虽然间隔没有减少到单个有效位置，但计算中点为 mid = (l + r) // 2. 该中点表示二分搜索过程中的下一个候选决策点。 
3. 应用问题中描述的比较规则来确定正确的区域是位于中点的左侧还是右侧。 这一步是模拟的核心，因为它编码了二分搜索所依赖的单调结构。 
4. 如果条件表明答案位于左侧，则更新 r = mid。 这会将中间保留在搜索空间中，因为它可能仍然是边界解。 
5. 否则，更新 l = mid + 1。这会丢弃 mid，因为我们确定答案必须严格位于其右侧。 
6. 重复直到 l 等于 r。 到那时，区间已经崩溃，剩下的位置就是最终的答案。 

该过程的正确性取决于真实答案始终包含在当前区间 [l, r] 内的不变量。 每个更新步骤都会保留这个不变量，因为决策规则是单调的：一旦排除一侧，该侧的所有值对于所有未来的迭代都保持无效。

 ## 边缘情况

 对于最小间隔，例如 n = 1，算法立即终止，因为 l 和 r 开始相等。 由于唯一的候选者是两个端点，因此不变量保持不变，因此不执行任何更新。 

对于大小为 2 的间隔，例如 n = 2，中点始终解析为 1，并且根据谓词，算法要么保持 1，要么转向 2。这里重要的行为是更新规则永远不会陷入振荡，因为每次迭代都会严格缩小间隔。 

对于较大的范围，关键的正确性属性是 mid 永远不会由于包容性边界处理而导致真实答案的丢失。 之间的选择`r = mid`和`l = mid + 1`即使 mid 等于边界之一，也能保证单调约简并防止无限循环。
