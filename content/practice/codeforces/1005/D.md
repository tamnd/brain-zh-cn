 ## 解决方案
 ## 问题理解

 我们得到一个非常长的十进制字符串，并且允许我们在相邻数字之间插入剪切，将其分成连续的块。 每个块都被解释为一个数字，但有一个限制：不允许使用带有前导零的块，除非该块恰好是一位数字`"0"`。 

分割后，我们计算这些块中有多少代表可以被 3 整除的数字，并且我们希望最大化这个计数。 

因此，任务不是计算数字本身的值，而是选择其数字序列的一个分区。 如果每个棋子的数字总和能被 3 整除，则贡献 1 分，否则贡献 0 分。 目标是最大化“好”细分的数量。 

输入长度最多可达 200,000 位。 这立即排除了尝试所有分区的任何策略。 分区是由 199,999 个二元切割决定决定的，因此在最坏的情况下暴力破解是指数级的。 

如果重复地重新评估子串和，则对所有分段进行简单的动态规划也是不可行的，因为存在 O(n^2) 子串。 

一个关键的微妙之处是仅在单字符段中允许使用前导零。 这很重要，因为它阻止我们自由地将零分组为更长的块来操纵总和。 

一些边缘情况需要牢记在心。 

例如，如果字符串全为零`"0000"`，最佳答案是 4，因为每个数字必须隔离为`"0"`，并且每个都可以被 3 整除。 

如果字符串包含类似数字`"111111"`，最好的策略是仅在有助于平衡 mod 3 和的情况下尽可能多地拆分，但任意拆分并不总是最佳的，因为分组会影响可分结构。 

如果字符串是`"303"`，取整个字符串的总和为 6，所以答案为 1，但分成`"3|0|3"`给出 3 个有效段。 这表明合并下的可分性不是单调的。 

## 方法

 蛮力视图是考虑切割字符串的每种可能的方式，然后评估每个结果分区。 对于长度为 n 的字符串，有 2^(n−1) 种放置切割的方法。 即使评估一个分区需要 O(n)，总数也会变为 O(n·2^n)，这对于 n 达到 200,000 时是不可能的。 

一种更结构化的强力方法是动态编程。 设 dp[i] 为前缀 i 中有效段的最大数量。 从 i 开始，我们尝试所有先前的剪切位置 j < i 并检查子字符串 s[j..i] 是否能被 3 整除。检查整除性可以通过前缀和来完成，因此每个转换是 O(1)，但 DP 是 O(n^2)，仍然太大。 

关键的观察结果是，能否被 3 整除仅取决于数字和模 3。如果一个段的数字和与 0 mod 3 同余，则它贡献 +1。 

现在我们用前缀和模 3 来重写问题。令 pref[i] 为直到 i 模 3 的数字之和。如果 pref[i] == pref[j]，则段 (j+1..i) 是好的。 

因此，我们希望将数组划分为尽可能多的段，以便每个选定的段对应于两个相等的前缀 mod 状态。 

这将问题转化为选择等模位置序列，但具有分段不相交且连续的约束。 

贪婪的见解是，每当我们看到位置 i 时，我们应该尝试在最早的先前位置 j 处关闭具有相同前缀 mod 状态且仍然有效的段。 这相当于以最大化对数的方式对相等前缀状态的出现进行配对，同时尊重顺序。 

最佳结构变成贪婪扫描，其中三个计数器跟踪每个前缀模类“可”作为段的起点的次数。 

事实证明，前导零约束不会影响最佳计数，因为个位数零始终是有效段，并且可以被视为值为 0 的正常数字。 

我们贪婪地维护当前打开的前缀状态数量，每当我们再次看到匹配的状态时，我们就会立即关闭一个段，以最大限度地提高未来的灵活性。

 ## 算法演练

 我们将数字字符串转换为前缀和模 3 的序列。 

1. 从左到右扫描字符串时计算前缀模值。 在位置 i，我们知道当前的前缀余数 r。 
2. 维护一个计数器 cnt[r]，表示该余数作为未闭合段的“起始边界”出现了多少次。 最初 cnt[0] = 1，因为字符串之前的空前缀充当起点。 
3. 当我们位于位置 i 时，我们检查当前余数 r。 如果 cnt[r] 为正，我们立即形成一个以 i 结尾的段。 我们增加答案并减少 cnt[r]，因为我们将此位置与前一个位置配对。 
4. 如果 cnt[r] 为零，我们通过递增 cnt[r] 将此余数存储为未来潜在的起点。 
5. 继续直到字符串末尾。 

关键思想是，每次关闭一个段时，我们都要确保其数字和可被 3 整除，因为两个端点共享相同的前缀和模 3。 

### 为什么它有效

 该算法本质上是在从左到右的扫描中配对相等的前缀残基，同时始终关闭最早可能的有效片段。 任何最佳解决方案都会导致具有相等前缀模值的位置之间的配对。 如果我们在可能的情况下延迟关闭一个段，我们只会降低未来的灵活性，因为该打开可能会在以后用于形成不同的有效段。 因此，立即贪婪关闭永远不会减少可能配对的数量，并且段的总数等于此类配对的数量。

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    s = input().strip()
    
    cnt = [0, 0, 0]
    cnt[0] = 1
    
    pref = 0
    ans = 0
    
    for ch in s:
        pref = (pref + (ord(ch) - 48)) % 3
        
        if cnt[pref] > 0:
            ans += 1
            cnt = [0, 0, 0]
            cnt[0] = 1
            pref = 0
        else:
            cnt[pref] += 1
    
    print(ans)

if __name__ == "__main__":
    solve()
```该实现仅保留当前前缀余数和一个小的频率数组。 重置步骤对应于切割字符串：一旦我们形成有效的段，我们就从该边界重新开始计数。 这一点至关重要，因为段是不相交的，因此前缀逻辑必须在每次切割后重新启动。 

微妙的一点是在成功切割后重置计数器状态和前缀余数。 如果不重置，我们将错误地允许重叠段。

 ## 边缘情况

 对于`"0000"`，该算法从 cnt[0] = 1 开始。在第一个数字处，前缀为 0，因此段立即关闭。 该状态重置并重复，产生四个段。 这符合每个零必须隔离的约束。 

为了`"111"`，前缀mod值为1,2,0。只有最后位置闭合一个段，所以只形成一个段。 该算法不会强制进行不必要的分割，因为 cnt[r] 仅在存在匹配状态时使用。 

为了`"303"`，重复的前缀零允许在每一步立即关闭，从而产生最大的分段。 重置确保每次切割都是独立的，并防止前缀状态的重叠重用。
