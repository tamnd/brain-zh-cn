 ## 解决方案
 ## 问题理解

 给定一个正整数的多重集，我们可以选择这些数字的任何子集。 对于选定的子集，我们计算所有选定元素的最大公约数。 目标不是最大化该 gcd，而是在选择尽可能少的元素的同时实现 gcd 等于 1。 

因此，任务是可除性结构的覆盖问题：我们想要一个最小子集，其公约数完全崩溃为一。 

输入规模很大，最多有30万个数字，每个值也最多有30万个。 这立即排除了任何检查所有子集的方法，因为子集的数量是指数级的。 在这种规模下，即使是成对或三重枚举也是不可能的。 任何解决方案都必须利用算术结构，尤其是除数和 gcd 属性，并依赖于值范围上的预计算而不是子集上的组合搜索。 

一个关键的微妙之处在于，答案不一定与值 1 的频率相关。 即使没有元素等于 1，我们仍然可以使用质因数互补的多个合数得到 gcd 1。 

一个常见的失败案例来自于贪婪的直觉。 例如，给定 6、10、15 这样的数字，人们可能会首先尝试选择具有较小 gcd 的对。 但贪婪选择下的 gcd 行为并不单调； 组合局部好的对并不能保证全局最优。 

另一种边缘情况是根本没有子集具有 gcd。 当所有数字共享一个共同的素数时，就会发生这种情况。 例如，输入`6 10 14`每个子集的 gcd 至少为 2，因此正确答案为 -1。 

## 方法

 强力方法会尝试所有子集并计算每个子集的 gcd，跟踪产生 gcd 1 的最小子集大小。 这是正确的，但不可行。 当 n 高达 300,000 时，即使检查大小为 2 的子集也已经意味着大约 4.5e10 对。 

我们需要不同的观点。 我们不是选择子集并计算 gcd，而是反转视角：固定一个可能的 gcd 值 d，并询问使它们的 gcd 恰好为 d 所需的最小元素数是多少。 如果我们将所有数字除以 d，问题就变成在变换后的数组中找到 gcd 为 1 的最小子集。 这将问题简化为原则上仅考虑 d = 1，但它建议对 gcd 状态进行动态规划。 

关键的见解是，对于每个可能的 gcd 值 g，跟踪获得 gcd 恰好为 g 的子集所需的最小元素数。 这是值空间而不是子集空间上的经典 gcd DP。 我们一一处理数字，使用 gcd(g, a[i]) 形式的转换更新状态。 

然而，除非经过压缩，否则所有子集上的直接 DP 仍然太大。 相反，我们维护一个字典或数组，其中 dp[g] 存储实现 gcd g 的最小子集大小。 每个新数字要么开始一个新的子集，要么与现有的 gcd 状态合并。 

由于 gcd 值受最大元素 (300,000) 限制，并且每个数字仅通过 gcd 状态的除数贡献转换，因此在仔细实施时，转换数量是可以管理的，因为 gcd 值形成递减链。 

最后，如果存在的话，答案就是 dp[1]。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力子集 | O(2^n·n) | O(2^n·n) | O(n) | 太慢了 |
 | GCD DP 超过各州 | O(n log A log A) | O(n log A log A) | O(A) | 已接受 |

 ## 算法演练

 我们维护从 gcd 值到从迄今为止看到的某个子集中获得该 gcd 所需的最小元素数的映射。 

1. 初始化一个空的DP结构。 这将存储对（gcd_value→最小子集大小）。 我们还跟踪使用单个元素启动新子集的最佳方式。 
2. 迭代数组中的每个数字 x。 
3. 对于当前数字 x，我们考虑两种可能性：开始一个仅包含 x 的新子集，或者通过将其与 x 组合来扩展每个先前已知的 gcd 状态 g 以形成 gcd(g, x)。 

这反映了这样一个事实：任何子集要么包含 x 作为新的开始，要么将 x 附加到现有子集。 
4. 我们构建一个临时字典 new_dp，首先在其中插入大小为 1 的状态 gcd(x) = x。 
5. 对于每个先前的状态（g，成本），我们计算 g2 = gcd（g，x）。 新的子集大小变为 cost + 1。我们用最小值更新 new_dp[g2]。 

这样做的原因是每个以 x 结尾的子集都必须来自先前的子集，并且 gcd 组合是关联且与顺序无关的。 
6. 处理完 x 后，我们将 new_dp 合并到 dp 中，仅保留每个 gcd 值的最小大小。 
7. 处理完所有元素后，我们检查 dp[1]。 如果存在，那就是达到 gcd 1 的最小子集大小。 否则输出-1。 

### 为什么它有效

 每个子集对应于其元素的排序，并且当按该顺序处理时，DP 准确地构造其 gcd 演化。 由于 gcd 是关联的，因此最终值仅取决于多重集，而不取决于顺序。 DP 递增地枚举形成 gcd 状态的所有可能方法，并且对于每个 gcd 值，它存储可以产生它的最小子集大小。 因此，如果 gcd 1 完全可以实现，则 DP 将捕获至少一个有效构造，并且最小存储大小是最佳的，因为所有子集构造都被考虑在内，并且每次合并时大小都被最小化。 

## Python 解决方案```python
import sys
input = sys.stdin.readline
from math import gcd

INF = 10**18

def solve():
    n = int(input())
    a = list(map(int, input().split()))
    
    dp = {}  # gcd -> minimum size
    
    for x in a:
        new_dp = {x: 1}
        
        for g, cnt in dp.items():
            ng = gcd(g, x)
            if ng in new_dp:
                if cnt + 1 < new_dp[ng]:
                    new_dp[ng] = cnt + 1
            else:
                new_dp[ng] = cnt + 1
        
        for g, v in new_dp.items():
            if g in dp:
                if v < dp[g]:
                    dp[g] = v
            else:
                dp[g] = v
    
    print(dp.get(1, -1))

if __name__ == "__main__":
    solve()
```该实现维护了一个可达 gcd 状态的字典。 每个新元素通过折叠 gcd 转换来更新所有现有状态。 初始化`new_dp = {x: 1}`正确地建模仅使用当前元素启动子集。 合并步骤确保我们永远不会失去达到相同 gcd 的更好（更小尺寸）的方法。 

一个微妙的点是我们永远不需要存储完整的子集，只需要存储它们的大小和生成的 gcd。 这种压缩使解决方案变得可行。

 ### 示例 1

 输入：```

```我们一步步追踪dp。 

| 步骤| x| x 之后的 new_dp | 合并后的 dp |
 | --- | --- | --- | --- |
 | 1 | 10 | 10 {10:1} | {10:1} |
 | 2 | 6 | {6:1, 2:2} | {10:1, 6:1, 2:2} |
 | 3 | 15 | 15 {15:1, 5:2, 1:3} | {10:1, 6:1, 2:2, 15:1, 5:2, 1:3} |

 关键转变在 15 处，与之前的 gcd 2 结合产生 1。达到 gcd 1 的最小大小是 3，这意味着所有元素都是必需的。 

这证实了中间 gcd 状态可以以非平凡的方式组合，并且最终答案取决于链接多个元素。

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 | O(n·k log A) | O(n·k log A) | 每个元素都会更新所有活动的 gcd 状态，每个转换都会计算 gcd |
 | 空间| O(k) | k 是任意时间存储的不同 gcd 状态的数量 |

 实际上，不同 gcd 状态的数量仍然很少，因为 gcd 值迅速崩溃并且不会形成大型独立集。 当 A 高达 300,000 时，这在此类 CF 问题的典型约束下保持在限制范围内。

 ## 边缘情况

 一个关键的边缘情况是所有数字共享一个共同的素因数。 用于输入`6 10 14`，无论子集选择如何，每个 gcd 计算都至少保持为 2。 DP 从状态 {6, 10, 14} 开始，然后合并但永远不会产生 gcd 1，因此 dp 永远不会包含密钥 1 并且输出正确为 -1。 

另一种情况是单个元素等于 1 时。对于输入`1 100 1000`，DP 在第一步立即插入大小为 1 的状态 1，并且后续操作无法对此进行改进，因此答案为 1。 

更微妙的情况是 gcd 1 仅在组合三个或更多数字后才出现，例如`10 6 15`。 DP 正确地将达到 1 延迟到最后一步，但仍然跟踪最小子集大小，确保不会过早剪枝删除最优构造。
