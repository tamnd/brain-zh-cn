 ## 解决方案
 ## 问题理解

 我们有一排奶牛，每头奶牛都有一个重量。 “raid”由两个整数定义：起始位置和步长。 从位置a开始，我们反复向前跳跃b个位置，收集我们落在的所有奶牛，一旦越过队伍的末端就停下来。 每个查询都会询问该特定袭击模式收集的所有奶牛的总重量。 

换句话说，每个查询都定义了数组内索引的算术级数，并且我们必须对这些索引处的值求和。 

约束很大：最多 3×10^5 头牛和 3×10^5 查询。 这立即排除了通过逐步推进进程来重新计算每个查询的总和的任何方法。 在最坏的情况下，b = 1 的单个查询会触及 O(n) 个元素，这将导致 O(n·q)，远远超出可接受的限制。 

当 b 较大或较小时，会出现微妙的边缘情况。 当 b 很大时，每个查询涉及的元素很少，而且成本很低。 当 b 很小时，许多索引在查询之间重叠，并且朴素迭代会多次重复相同的工作。 另一个问题是当 a = 1 且 b = 1 时，会退化为对整个数组求和，这种情况在暴力破解下会重复重新计算。 

一个幼稚的改进可能会尝试缓存每个 (a, b) 的结果，但由于最多可以有 3×10^5 个不同的对，这在最坏的情况下没有帮助。 

## 方法

 暴力破解的想法很简单：对于每个查询，从索引 a 开始，重复添加 w[a]、w[a + b]、w[a + 2b] 等，直到超过 n。 这是正确的，因为它完全遵循袭击的定义。 问题是成本。 在 b = 1 的最坏情况下，每个查询处理 n 个元素，当 n 和 q 均为 3×10^5 时，总共会产生大约 9×10^10 次操作。 

关键的观察结果是跳跃大小 b 将行为划分为两个根本不同的状态。 大 b 意味着每个查询访问的位置很少。 小 b 意味着许多查询共享以 b 为模的相似残差类，因此我们可以重用预先计算的信息。 我们利用的结构是固定步骤 b 访问的索引形成独立的算术序列，并且当 b 很小时，总体上只有很少的这样的序列。 

这导致了混合解决方案。 我们选择 √n 附近的阈值 B。 对于大于 B 的 b，我们直接计算每个查询，因为它最多只访问 n/B 个元素。 对于小于或等于 B 的 b，我们使用动态规划从右到左预先计算所有可能的（b，余数）对的答案，因此每个查询变为 O(1)。 

关键的转变是重写递推式：对于固定的 b，定义 dp[i] = w[i] + dp[i + b]。 这允许一次性计算给定 b 的所有答案。

 | --- | --- | --- | --- |
 | 蛮力 | O(nq) | O(1) | O(1) | 太慢了 |
 | 按步长进行 Sqrt 分解 | O(n√n + q√n) | O(n√n + q√n) | O(n√n) | O(n√n) | 已接受 |

 ## 算法演练

 我们根据步长 b 使用阈值 B ≈ √n 来分离查询。

1. 我们初始化一个结构来存储小步长的答案。 对于从 1 到 B 的每个 b，我们将计算一个 dp 数组，其中 dp[i] 表示从 i 开始并按 b 跳直到离开数组的总和。 
2. 对于从 1 到 B 的每个 b，我们按从 n 到 1 的倒序计算 dp。在位置 i，如果 i + b ≤ n，我们设置 dp[i] = w[i] + dp[i + b]，否则 dp[i] = w[i]。 这是可行的，因为同一算术级数中的下一个访问位置恰好是 i + b。 
3. 计算出固定 b 的 dp 后，我们可以通过返回 dp[a] 在 O(1) 时间内回答该 b 的任何查询。 
4. 对于 b > B 的查询，我们通过迭代 i = a、a + b、a + 2b 并求和权重直接回答它们，直到超过 n。 由于b很大，所以访问的元素数量很少。 
5. 我们处理所有查询，对于小步骤使用预先计算的 dp，对于大步骤使用直接模拟。 

它的工作原理与固定数组上的算术级数结构有关。 对于小步骤，数组会在重叠链中重复重用，因此预计算会在所有查询中分摊。 对于大步长，链足够稀疏，直接遍历的成本很低。 每个查询都由这两种机制之一处理，并且都涵盖所有可能的（a，b）情况，没有重叠或遗漏。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

n = int(input())
w = list(map(int, input().split()))
q = int(input())

B = int(n ** 0.5) + 1

# store answers for small b
dp = [[0] * n for _ in range(B + 1)]

# precompute for all b up to B
for b in range(1, B + 1):
    for i in range(n - 1, -1, -1):
        if i + b < n:
            dp[b][i] = w[i] + dp[b][i + b]
        else:
            dp[b][i] = w[i]

for _ in range(q):
    a, b = map(int, input().split())
    a -= 1

    if b <= B:
        print(dp[b][a])
    else:
        s = 0
        i = a
        while i < n:
            s += w[i]
            i += b
        print(s)
```dp表是核心优化。 每行对应一个固定的步长 b，每个条目将整个算术级数折叠为单个预先计算的值。 仅当级数足够短以至于预处理不值得时才使用直接循环。 

唯一微妙的实现细节是索引。 dp 数组基于基于 0 的索引构建，因此每个查询在查找之前都会将 a 转换为 a - 1。 另一个细节是边界检查 i + b < n，这确保我们只引用有效的未来索引。

 ## 边缘情况

 关键的边缘情况是 b = 1 时。在这种情况下，每个查询都变成 a 的完整后缀和，并且简单的解决方案会重复重新计算长段。 dp 递归干净地处理了这个问题，因为 dp[1][i] 自然地成为后缀和链。 

另一种边缘情况是当 a 接近末尾且 b 很大时。 例如，n = 6，a = 5，b = 10。循环应该只包含w[5]，并立即停止。 直接模拟可以正确处理这个问题，因为在每次加法之前都会检查条件 i < n。 

最后一个微妙的情况是多个查询共享相同的 (a, b)。 该解决方案不依赖于每个查询的缓存； 相反，它通过 dp 查找或确定性遍历独立地确保每个查询的正确性。
