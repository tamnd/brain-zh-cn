 ## 解决方案
 ## 问题理解

 列表中的每个输入字符串都可以被视为一个非常短的字符序列，最多长度为 6。 对于每个查询字符串，我们都会被问到是否可以通过从这些存储的字符串中的至少一个中删除一些字符而不改变剩余字符的顺序来形成它。 换句话说，要使查询有效，列表中必须存在包含该查询作为子序列的源字符串。 

主要挑战来自规模而不是个别检查。 最多可以有五万个源字符串和五万个查询。 尽管检查一个字符串是否是另一个字符串的子序列很快，因为两者都很小，但对每一对进行检查仍然会导致数十亿次比较。 因此，即使使用非常优化的内部循环，直接嵌套解决方案也会超出时间限制。 

一个微妙的点是源字符串和查询都非常短。 这强烈表明应该利用每个字符串的结构，而不是它们的数量。 任何将字符串视为任意长度对象的解决方案都忽略了关键的简化：每个字符串只有 2^6 个可能的子序列，这是一个恒定的上限。 

当尝试仅预处理前缀或后缀时，会出现一个幼稚的错误。 例如，如果源字符串是“abdc”并且查询是“abc”，则前缀匹配或子字符串匹配将错误地拒绝它，即使删除“d”使其有效也是如此。 另一个常见的错误是假设查询必须连续出现，但这不是必需的。 

## 方法

 一个简单的想法是独立处理每个查询并针对每个源字符串进行测试。 由于两个字符串都很短，因此我们可以在最多六个长度的线性时间内检查子序列状态。 这种方法是正确的，因为它直接实现了定义。 然而，对于最多 5×10^4 次查询和 5×10^4 源字符串，我们执行大约 2.5×10^9 次比较，每次比较最多需要六次操作。 这远远超出了可行的限度。 

关键的观察是我们重复解决同一类型的问题：“某个源字符串是否包含此查询作为子序列？” 我们可以预先计算任何源字符串中存在的所有子序列，而不是在线回答查询。 由于每个字符串的长度最多为 6，因此子序列的数量最多为 2^6 = 64。这将预处理变成有界扩展步骤。 

我们生成每个源字符串的每个子序列并将其存储在哈希集中。 一旦构建了该集合，每个查询就会减少为单个成员资格检查。 生成的字符串总数最多为5×10^4×64，即约320万条，易于管理。

 ## 算法演练

 我们将问题转换为所有子序列的预处理和查找结构。 

1. 读取所有源字符串并初始化一个空哈希集。 

该集合将表示可以由至少一个源字符串形成的每个可能的查询。 
2. 对于每个源字符串，使用其位置的位掩码枚举生成所有子序列。 

由于最大长度为 6，因此每个字符串最多贡献 64 个子序列。 

对于每个位掩码，我们通过选择设置位的字符来构造相应的子序列。 
3. 将每个生成的子序列插入到全局集中。 

这确保了如果任何字符串可以通过删除生成给定模式，则该模式可用于查询。 
4. 读取每个查询字符串并检查它是否存在于集合中。 

如果是，则输出“YES”，否则输出“NO”。 
5. 重复此操作，直至处理完所有查询。

### 为什么它有效

 源字符串的每个子序列都是显式生成和存储的。 任何有效答案必须对应于至少一个源字符串的子序列，因此它必须出现在构造的集合中。 相反，集合中的任何内容都保证来自真实的源字符串，因此它的构造是有效的。 这在集合中的成员资格和查询的有效性之间创建了完美的等价。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    s_list = [input().strip() for _ in range(n)]
    
    subseqs = set()

    for s in s_list:
        L = len(s)
        # generate all subsequences
        for mask in range(1 << L):
            t = []
            for i in range(L):
                if mask & (1 << i):
                    t.append(s[i])
            subseqs.add("".join(t))
    
    m = int(input())
    out = []
    for _ in range(m):
        t = input().strip()
        out.append("YES" if t in subseqs else "NO")
    
    sys.stdout.write("\n".join(out))

if __name__ == "__main__":
    solve()
```该解决方案是围绕预处理集构建的`subseqs`。 位掩码循环枚举每个源字符串的每个可能的子序列，包括空字符串，它是有效的，但与长度大于零的查询无关。 每个生成的字符串都会插入到 Python 集中，从而在查询期间实现平均 O(1) 查找。 

这里重要的一个常见实现细节是字符串构造。 由于字符串很短，因此会重复`"".join`运营成本仍然低廉。 总体复杂性主要由位掩码枚举决定，而不是字符串处理开销。

 ## 边缘情况

 一个微妙的情况是查询为空。 每个字符串都包含空子序列，因此正确的输出始终为 YES。 该算法自然地处理这个问题，因为在预处理期间总是插入空掩码。 

另一种情况涉及重复的字符，例如“aaaaaa”。 每个子序列都会折叠成“”、“a”、“aa”等字符串，但重复并不重要，因为该集合仅存储唯一值。 像“aaa”这样的查询可以被正确识别。 

当字符存在于多个源字符串中但在单个字符串中从未以正确的顺序存在时，会出现更具结构性的边缘情况。 例如，“a”可能出现在一个字符串中，“b”出现在另一个字符串中，但“ab”无效，除非两者按顺序出现在单个源中。 该算法避免了这个陷阱，因为它从不混合不同字符串中的字符； 每个子序列都是从单个原始字符串生成的，准确地保留了排序约束。
