 ## 解决方案
 ## 问题理解

 我们有一个系统$n$灯和$n$按钮索引从 1 到$n$。 所有灯开始关闭。 按下按钮$i$翻转索引可整除的每个灯的状态$i$，因此它影响灯上的常规算术结构而不是局部段。 

我们被要求选择一组非空的按钮来按下，每个按钮最多使用一次。 此外，还有定向依赖规则：如果我们按下按钮$u$，然后按钮$v$也必须按下去。 这些规则的行为就像暗示一样，因此选择一个按钮可以强制执行一系列其他按钮。 

选择组并应用所有开关后，我们看看最终有多少盏灯亮着。 要求是这个数量不超过$\lfloor \sqrt{n} \rfloor$。 如果不存在有效选择，我们必须报告不可能性。 

约束允许最多$2 \cdot 10^5$灯跨所有测试用例，因此任何解决方案都必须接近每个测试用例的线性或线性数学。 任何涉及枚举按钮子集的操作都是立即不可行的，因为$2^n$增长得太快了。 即使模拟没有结构的任意子集的效果也会太慢，因为每个按钮最多可以影响$O(n)$灯。 

一个微妙的问题来自于依赖规则。 一种幼稚的方法可能会尝试仅根据其切换效果来选择“好”按钮，但忽略强制包含。 这可能会破坏正确性。 

例如，假设一个小实例，按下按钮 5 会强制按下按钮 1，而按钮 1 会影响所有灯。 尽管单独 5 可能看起来不错，因为它影响很少的灯，但强制包含 1 会突然翻转所有灯并破坏约束。 这表明可行性共同取决于图结构和所选指标的算术效果。 

当人们认为选择多个影响较小的按钮总是更好时，就会出现另一种失败情况。 依赖关系可能会强制包含许多附加按钮，从而使最终效果比预期大得多。 

## 方法

 暴力策略将尝试所有按钮子集，传播每个子集的强制依赖关系，计算按下按钮的结果集，并模拟所有灯的切换。 对于每个子集，切换成本$O(n \log n)$或者$O(n)$如果使用可分性标记仔细完成。 由于有$2^n$子集，即使对于非常小的子集，这种方法也是不可行的$n$。 

关键的结构观察是依赖关系在按钮上形成有向图，并且任何有效的选择都必须在传出边缘下关闭：如果我们选择一个节点，我们还必须选择从它可到达的所有内容。 这意味着每个有效的解决方案都是通过选择在可达性下稳定的集合来有效确定的。 

我们不再考虑任意子集，而是将焦点转移到不强制任何额外内容的节点上。 用图术语来说，这些是没有传出边的节点，因为选择这样的节点不会强制在其自身之外进行额外的选择。 这样的节点的行为就像“安全终端选择”。 

一旦我们减少到单按钮选择，切换结构就会变得简单。 仅选择按钮$i$恰好打开的倍数$i$，因此点亮的灯的数量变为$\left\lfloor \frac{n}{i} \right\rfloor$。 为了满足约束，我们需要这个数量最多为$\lfloor \sqrt{n} \rfloor$，相当于选择一个索引$i \ge \sqrt{n}$。 

因此，问题简化为寻找一个没有传出依赖关系且索引足够大的节点。 如果存在这样的节点，我们将其输出。 否则，每个安全节点都太小，或者每个节点都强制进行额外的选择，从而无法将点亮的灯数量控制在所需的范围内。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 对子集的暴力破解 |$O(2^n \cdot n)$|$O(n)$| 太慢了 |
 | 图过滤+单选|$O(n + m)$|$O(n + m)$| 已接受 |

 ## 算法演练

 1. 构建一个有向图，其中每个约束$u \rightarrow v$意味着按下$u$压力$v$。 同时计算每个节点的出度。 
2. 识别所有出度为零的节点。 这些节点不会强制在其自身之外添加任何其他按钮，因此选择其中一个可以避免所选集的意外扩展。 
3. 计算阈值$B = \lfloor \sqrt{n} \rfloor$。 任何单个选择的按钮$i$准确地产生$\lfloor n/i \rfloor$点燃的灯。 
4. 从出度为零的节点中，选择任意节点$i$这样$i \ge B$。 这确保了灯数量限制得到满足。 
5. 如果存在这样的节点，则输出$k = 1$和选择的索引$i$。 如果不存在这样的节点，则输出$-1$。 

### 为什么它有效

 依赖图确保任何选定的节点都会拖入可从其到达的所有节点，因此必须在可达性下关闭有效的解决方案。 出度为零的节点不能强制任何其他节点，因此选择它不会引入额外的约束。 因此，选择单个这样的节点始终是一致的。 

单个按钮的灯效$i$完全由整除性决定：只有整除性的倍数$i$已切换。 此类灯的数量正好是$\lfloor n/i \rfloor$，当$i$很小。 通过限制为$i \ge \sqrt{n}$，我们保证最多$\sqrt{n}$灯具受到影响。 由于我们只使用一个按钮，因此切换之间没有交互，并且最终状态与该计数完全匹配。 

如果不存在合适的节点，则每个节点要么强制进行额外的选择，要么太小而无法将切换的灯的数量保持在所需的范围内，因此在约束下无法形成有效的构造。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    
    outdeg = [0] * (n + 1)
    
    for _ in range(m):
        u, v = map(int, input().split())
        outdeg[u] += 1

    limit = int(n ** 0.5)

    for i in range(1, n + 1):
        if outdeg[i] == 0 and i >= limit:
            print(1)
            print(i)
            return

    print(-1)

if __name__ == "__main__":
    t = int(input())
    for _ in range(t):
        solve()
```该实现仅跟踪传出依赖计数，因为只有传出边缘的存在对于单节点构造下的可行性至关重要。 我们永远不需要存储邻接表，这使得内存使用量保持在最低限度。 

每个测试用例计算一次平方根阈值，并且扫描是线性的$n$。 这确保了解决方案在总体约束下保持高效。 

一个常见的实施陷阱是忘记只有传出边缘才重要。 传入边缘不会限制必须选择的内容，因为它们不会触发强制添加。 

## 工作示例

 考虑一个小案例，其中$n = 4$和依赖项是$4 \rightarrow 3$和$1 \rightarrow 2$。 

我们计算出度并识别出度为零的节点。 

| 节点| 出度 | 候选人？ | 行动|
 | ---| ---| ---| ---|
 | 1 | 1 | 没有 | 部队2 |
 | 2 | 0 | 是的 | 太小了|
 | 3 | 0 | 是的 | 检查阈值 |
 | 4 | 1 | 没有 | 力量3 |

 这里$\lfloor \sqrt{4} \rfloor = 2$，因此节点 3 有效。 选择 3 仅切换灯 3，满足限制。 

此跟踪显示了依赖性过滤如何将问题减少为仅检查小型候选集而不是探索组合。 

现在考虑$n = 10$完全没有边缘。 

| 节点| 出度 | 候选人？ | 行动|
 | ---| ---| ---| ---|
 | 1-10 | 1-10 0 | 是的 | 选择任意 ≥ 3 |

 自从$\lfloor \sqrt{10} \rfloor = 3$，任何节点3或以上都是有效的。 例如，选择 5 会切换灯 5 和 10，从而恰好给出两个点亮的灯。 

这表明，在没有约束的情况下，解决方案完全取决于整除性的算术结构。

 ## 边缘情况

 一种关键的边缘情况是每个节点至少有一个传出依赖项。 在这种情况下，没有办法选择单个稳定节点。 例如，在输入中$1 \rightarrow 2, 2 \rightarrow 1$，两个节点互相强制，因此选择其中一个会立即触发另一个，这意味着不存在单例解决方案。 算法正确输出$-1$因为不存在零出度节点。 

另一种情况是零出度节点存在但都太小。 例如，如果$n = 16$唯一的终端节点是$i = 3$， 然后$\lfloor 16/3 \rfloor = 5$，这超过了$\lfloor \sqrt{16} \rfloor = 4$。 该算法拒绝该节点并输出$-1$，即使满足图约束，也能正确处理算术约束。
