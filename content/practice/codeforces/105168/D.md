 ## 解决方案
 ## 问题理解

 我们得到了几个独立的测试用例。 在每个测试用例中，都有一个整数数组和一个目标值$k$。 任务是统计有多少个无序索引对$(i, j)$和$i < j$具有两个所选数组元素的按位异或完全等于的属性$k$。 

输入是纯数字的：每个测试用例描述一组值，问题是有多少对值在 XOR 下交互以产生固定结果。 每个测试用例的输出是表示该计数的单个整数。 

这些约束清楚地表明，任何直接检查所有对的解决方案都无法扩展。 所有测试用例的元素总数可以达到$2 \cdot 10^5$，所以二次扫描意味着$4 \cdot 10^{10}$最坏情况下的异或运算，远远超出了 2 秒允许的范围。 这立即排除了朴素的对枚举。 

有两种微妙的边缘情况经常会导致错误的方法。 

首先，当$k = 0$，条件变为$a_i = a_j$。 不分离这种情况的简单实现可能仍然有效，但是一些优化假设$k > 0$或尝试“避免自逆计数”可能会失败。 例如，在数组中$[1, 1, 1]$，正确答案是 3 对，而不是 0。 

其次，重复的价值观非常重要。 如果一个元素出现多次，它就会以组合方式对答案做出贡献。 任何将数组视为集合而不是多重集的方法都会低估。 

## 方法

 蛮力的想法很简单。 对于每对索引$(i, j)$, 计算$a_i \oplus a_j$并检查它是否等于$k$。 这是正确的，因为它直接评估问题中陈述的条件。 然而，它对每对执行一次恒定时间异或，导致$\frac{n(n-1)}{2}$每个测试用例的操作。 和$n = 2 \cdot 10^5$，这是不可行的。 

关键的观察结果是 XOR 方程可以确定性地重新排列。 如果$a_i \oplus a_j = k$，然后固定$a_i$意味着一个独特的所需值$a_j$: 即$a_j = a_i \oplus k$。 这将问题变成了频率匹配任务，而不是配对枚举任务。 

我们不是搜索对，而是维护迄今为止看到的值的频率图。 当我们从左到右扫描数组时，对于每个元素$x$，我们计算值$x \oplus k$并检查它已经出现了多少次。 每次出现这样的情况都会与当前元素形成有效的对。 然后我们记录$x$进入频率图。 

这是可行的，因为 XOR 是可逆的：应用 XOR 与$k$将目标转变为其独特的合作伙伴。 此映射中不存在冲突或歧义，因此增量计数补码可准确产生有效对的数量。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 |$O(n^2)$|$O(1)$| 太慢了 |
 | 频率哈希图|$O(n)$平均 |$O(n)$| 已接受 |

 ## 算法演练

 我们独立处理每个测试用例。 

1.初始化一个空的哈希图`freq`存储我们已经处理过的值的计数。 
2. 初始化一个პასუხ变量`ans = 0`。 
3. 从左到右迭代数组，取出每个值`x`。 
4. 计算所需的合作伙伴价值`y = x XOR k`。 这是唯一可以配对的值`x`产生异或等于`k`。 
5. 添加`freq[y]`到`ans`。 这会计算所有先前出现的有效合作伙伴。 
6. 增量`freq[x]`加1来记录我们现在看到的`x`。 

顺序很重要：我们只计算第二个索引大于第一个索引的对，这是通过仅与先前看到的元素匹配来确保的。 

### 为什么它有效

 在每一个位置上，`freq`存储严格小于当前索引的索引中的精确多重值集。 对于每个当前元素`x`, 每个较早的元素`y`这样$y \oplus x = k$对应于有效对$i < j$。 因为 XOR 定义了双射$y = x \oplus k$，当处理正确的端点时，每个有效对都只计算一次。 没有一对被遗漏，也没有一对被重复计算。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    out = []
    
    for _ in range(t):
        n, k = map(int, input().split())
        a = list(map(int, input().split()))
        
        freq = {}
        ans = 0
        
        for x in a:
            y = x ^ k
            ans += freq.get(y, 0)
            freq[x] = freq.get(x, 0) + 1
        
        out.append(str(ans))
    
    print("\n".join(out))

if __name__ == "__main__":
    solve()
```该解决方案依赖于每个测试用例一次通过。 字典`freq`跟踪以前见过的数字的计数。 对于每个元素，我们在更新映射之前立即查询其异或补集，保留严格的索引排序要求。 

一个常见的实施错误是更新`freq[x]`查询之前`freq[y]`，这会错误地计算对，其中$i = j$。 另一个微妙的问题是忘记重置测试用例之间的频率图，这会混合独立数组之间的计数。

 ## 边缘情况

 当所有元素都相同并且$k = 0$，每对都是有效的。 频率法自然地累积计数：第一个元素贡献 0，第二个元素贡献 1，依此类推，得出$n(n-1)/2$。 

什么时候$k \neq 0$但数组包含许多重复值，只有特定的补码才重要。 该算法不假设唯一性，因此每次重复都会正确地将贡献乘以`freq[y]`。 

什么时候$k = 0$，每个元素的异或补码就是它本身。 因为我们在递增之前查询频率图，所以我们只计算之前出现的次数，确保每对$(i, j)$具有相同值的情况仅计算一次$j$已处理。
