 ## 解决方案
 ## 问题理解

 我们得到一系列查询，其中每个查询告诉我们“今天”是一周中的哪一天，编码为从 1 到 7 的整数。映射是固定的：1 对应星期一，2 对应星期二，依此类推，直到 7 对应星期日。 对于每个查询，我们必须输出每周循环中紧邻该查询之前的星期几。 

该结构是圆形的，这意味着周日之后又是周一。 因此，从任何一天向后移动一步，要么将数字减少 1，要么从 1 回到 7。 

输入大小最多可达 200,000 个测试用例，这会立即排除每个查询执行非常量工作的任何内容。 每个测试用例需要 O(1) 计算，否则解决方案就有超时的风险。 由于每个查询都是独立的，因此除了理解映射之外不需要共享状态或预处理。 

唯一微妙的边缘情况是周一的总结。 如果今天是 1（星期一），那么昨天就不是 0，而是 7（星期日）。 不处理此边界的简单减法方法将默默地产生无效输出。 

例如，如果输入为 1，粗心的实现可能会计算出 1 - 1 = 0，这超出了有效范围。 正确答案应该是7。 

## 方法

 强力解释会将每个查询视为遍历周列表，直到找到前一个元素。 我们可以显式存储序列`[1, 2, 3, 4, 5, 6, 7]`对于每个查询，从给定位置向后扫描，直到找到前一个。 这是可行的，因为结构很小且固定，所以正确性微不足道。 

然而，即使周大小是恒定的，每个查询使用循环的强力实现也是不必要的开销。 更重要的是，它不能很好地推广到预期的模式：一周是一个模块化周期。 关键的观察结果是，该问题只是模 7 周期上的算术运算，其中我们减一并绕回。 

这将每个查询减少为恒定时间转换：计算`(w - 2) % 7 + 1`。 此公式将 1-7 索引转换为 0-6 系统，应用安全减量，然后将其转换回来。 或者，我们可以明确地处理边界：如果`w == 1`， 输出`7`，否则输出`w - 1`。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(t) 但有不必要的每个查询扫描 | O(1) | O(1) | 已接受但矫枉过正|
 | 最佳| O(t) | O(1) | O(1) | 已接受 |

 ## 算法演练

 1.读取测试用例数量`t`。 每个测试用例都是独立的，因此我们按顺序处理它们。 
2. 对于每个输入值`w`，检查它是否等于 1。这是唯一的特殊情况，因为一周从周一向后循环到周日。 
3.如果`w == 1`，输出 7。这直接对日历周期的环绕行为进行编码。 
4.否则，输出`w - 1`。 这可以在恒定时间内处理所有其他转换。 

### 为什么它有效

 这些天形成一个大小为 7 的闭合循环，其中每一天都有一个前一天。 减去 1 沿着该循环正确地向后移动，但在边界处除外，在边界处表示会破坏线性。 显式更正为`w = 1`恢复循环结构。 由于每个有效输入恰好映射到一个前驱，并且没有两个输入共享冲突的输出，因此转换既完整又一致。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    out = []
    for _ in range(t):
        w = int(input())
        if w == 1:
            out.append("7")
        else:
            out.append(str(w - 1))
    sys.stdout.write("\n".join(out))

if __name__ == "__main__":
    solve()
```该实现读取所有查询并将输出存储在列表中以避免重复的 I/O 开销。 每个查询都通过单个条件检查进行处理，确保恒定的处理时间。 

唯一重要的实现细节是避免在循环内直接打印，因为 Python I/O 可能成为 200,000 次操作的瓶颈。 积累结果并编写一次可确保可预测的性能。

 ## 边缘情况

 唯一有意义的边缘情况是周一的循环边界。 

输入：```
1
1
```这里`w = 1`，所以算法采用特殊分支并输出`7`。 支票`w == 1`立即触发，因此不会尝试算术减法。 

对于第二次边界检查：

 输入：```
1
2
```自从`w != 1`，算法输出`2 - 1 = 1`。 这证实了非边界值的行为是一致的，并且不需要超出递减规则的特殊处理。 

这两种情况共同验证了转换函数正确地模拟了 7 天系统的完整循环前驱关系。
