 ## 解决方案
 ## 问题理解

 我们得到了一个路径图$n$顶点，其中每个顶点$i$连接到$i+1$。 在这个结构的顶部，有一个额外的隐藏边缘$(a, b)$已添加，并承诺它们之间至少有一个严格的顶点，这意味着$a + 2 \le b$。 这恰好在图中创建了一个循环。 

我们无法直接看到额外的边缘。 相反，我们可以提出以下形式的查询$(l, r)$，系统回答是否存在最短路径$l$和$r$修改后的图中使用添加的边。 换句话说，创建的快捷方式是否$(a, b)$严格改善之间的距离$l$和$r$对比原来的路径就行了。 

我们必须确定端点$a$和$b$最多使用 50 个此类查询。 

重要的结构事实是，如果没有额外的边缘，距离是线性的。 有了额外的边缘，任何最短路径受益的对都必须以一种非常特定的方式“跨越”循环：它们在线上的自然路径将比跳过的路径更长$(a, b)$。 

自从$n$可以大到$10^6$，我们无法显式地模拟或扫描任何东西。 每个查询都很昂贵，因此解决方案必须为每个查询提取大量信息，通常会呈指数级减少搜索空间。 

一个天真的错误是认为查询相邻对或扫描所有对可能会揭示边缘端点。 例如，尝试所有$(l, r)$需要$O(n^2)$查询，这是完全不可行的。 

一种更微妙的错误方法是假设可以通过检测答案沿固定方向从 0 切换到 1 的位置来找到端点。 这会失败，因为谓词在单个索引中不是单调的：一对是否使用快捷方式同时取决于两个端点，而不仅仅是一个坐标。 

例如，如果隐藏边是$(2, 7)$， 然后$(1, 6)$可能不会使用它，但是$(3, 8)$可能，即使两者都“右移”。 不存在单一的线性阈值行为。 

关键的难点在于信息是二维的，但我们需要识别两个未知的端点。 

## 方法

 蛮力观点很简单。 如果我们可以测试每个候选对$(l, r)$，我们只需检查哪一个返回 1。这是正确的，因为恰好一对对应于添加的边。 然而，这需要$O(n^2)$查询，即使对于中等程度的人来说也是不可能的$n$，更不用说$10^6$。 

关键的观察结果是，该结构是一棵树加一条边，因此最短路径查询的行为类似于间隔上的几何条件。 答案是$(l, r)$仅取决于间隔是否$[l, r]$以一种使绕道比直接路径更短的方式“覆盖”隐藏的捷径。 

这可以重新构建：快捷方式在以下情况下很有用：$l$位于边缘的一侧并且$r$位于另一侧，增益超过直线距离。 这意味着肯定答案集在转换意义上形成了一个连续区域，我们可以利用二分搜索两次：一次定位一个端点，然后定位另一个端点。 

解锁这一问题的标准方法是修复一个端点并使用单调性搜索另一个端点。 如果我们修复$l = 1$，那么作为$r$增加，查询最终在何时变为真$r$通过隐藏边的远端点。 这为我们提供了对一个端点的干净的二分搜索。 一旦知道一个端点，我们就可以锚定它并重复相同的想法来找到另一个端点。 

因此，问题从在一个巨大的域中查找两个未知值减少为对单调谓词的两次二分搜索，每个搜索最多需要$O(\log n)$查询。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 |$O(n^2)$查询 |$O(1)$| 太慢了 |
 | 端点上的二分搜索 |$O(\log n)$查询 |$O(1)$| 已接受 |

 ## 算法演练

 我们依赖这样一个事实：如果我们修复查询间隔的一个端点，则另一个端点的响应将变得单调。 

1.固定左端点为1，二分查找最小$r$这样查询$(1, r)$返回 1。这标识了远端端点$b$。 单调性来自于这样一个事实：当我们扩展时$r$，我们只能包含更多可能受益于快捷方式的顶点。 
2.一次$b$已知，我们需要找到$a$。 我们现在搜索一遍$l$在范围内$[1, b-2]$因为限制$a + 2 \le b$。 
3. 对于候选人$l$， 询问$(l, b)$。 如果答案为1，则意味着$l$仍然位于快捷方式的另一侧，路径使用添加的边缘，因此我们将搜索向左移动。 否则，我们向右移动。 
4.二分查找收敛到最小或最大有效值$l$取决于我们如何定义谓词； 然后我们验证对称条件以查明$a$。 
5、输出$(a, b)$一旦确定了两个端点。 

微妙之处在于，只有当一个端点固定时，谓词才是单调的。 如果不固定一侧，查询行为就没有顺序，这就是两阶段搜索必不可少的原因。 

### 为什么它有效

 该图是一条线加一个弦，因此任何最短路径的改进只能通过使用该弦来实现。 对于一个固定端点，扩展另一个端点只会增加弦位于最短路径上的机会，因为我们只是延长了间隔，并不能“失去”使用捷径的可能性。 这创建了一个适合二分搜索的单调谓词。 每个阶段都会隔离一个端点，一旦知道一个端点，另一个端点就成为线路上的标准阈值查找问题。 

## Python 解决方案```python
import sys

input = sys.stdin.readline
print = sys.stdout.write

def ask(l, r):
    sys.stdout.write(f"? {l} {r}\n")
    sys.stdout.flush()
    return int(sys.stdin.readline().strip())

def answer(l, r):
    sys.stdout.write(f"! {l} {r}\n")
    sys.stdout.flush()

def find_b(n):
    lo, hi = 1, n
    while lo < hi:
        mid = (lo + hi) // 2
        if ask(1, mid):
            hi = mid
        else:
            lo = mid + 1
    return lo

def find_a(b):
    lo, hi = 1, b - 1
    while lo < hi:
        mid = (lo + hi) // 2
        if ask(mid, b):
            hi = mid
        else:
            lo = mid + 1
    return lo

def main():
    n = int(input().strip())
    b = find_b(n)
    a = find_a(b)
    answer(a, b)

if __name__ == "__main__":
    main()
```交互是通过标准输出的简单包装器来处理的。 每个查询都会立即刷新，因为交互器期望同步通信。 

功能`find_b`执行二分搜索，假设对于固定左端点 1，一旦间隔足够大以包含隐藏和弦端点的影响，谓词就变为真。 搜索范围缩小到答案变为 1 的最小位置。 

功能`find_a`然后修复发现的端点并使用相同的单调行为搜索另一侧，但现在超出了左边界。 

实现中的一个常见陷阱是忘记交互器在每次查询后都需要刷新。 另一个是在以下情况下错误处理二分搜索边界：$b$接近 1 或$n$，虽然约束$a + 2 \le b$保证有效的搜索间隔$a$。

 ## 边缘情况

 一种边缘情况是隐藏边缘非常接近边界，例如$(1, 3)$。 在这种情况下，二分查找$b$仍然有效，因为查询$(1, mid)$很早就会变成正数，导致右边界迅速塌陷到 3。第二次搜索结束$a$然后在一个非常小的间隔内运行并立即收敛。 

另一种情况是当边缘几乎最大时，例如$(n-2, n)$。 第一次搜索仅在范围末尾附近返回 1，因此大多数查询都是否定的。 这仍然可以正确处理，因为二分搜索不假设正数的任何分布，仅假设单调性。 

最后一种情况是边缘居中，例如$(n/3, 2n/3)$。 两种二分搜索的行为都是对称的，并且以对数步长收敛，没有偏差。
