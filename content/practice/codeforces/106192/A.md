 ## 解决方案
 ## 问题理解

 建筑物可以被认为是分布在各个楼层的房间的固定目录，其中每个楼层都有一组已知的可能的房间槽位，每个槽位对应于一个或两个实际的居住地点。 有些槽位代表共享的两人区块，分为左右房间，有些槽位代表单人房间，有些房间号只是未使用或用于非住宅用途。 

我们收到了一份检查清单，将某些个人居住地点标记为不合适。 同一个房间可以被多次报告，可能不一致，但规则很简单：如果任何报告将一个居住地点标记为不良，则该房间被视为无法使用。 

处理完所有报告后，我们会回答询问。 每个查询都会询问特定的楼层，并需要计算在删除所有失败的楼层后，该楼层上还剩下多少个可用的居住空间。 

声明中隐藏的关键结构是建筑布局是完全确定性的且变化不大。 房间标识符对楼层和房间号进行编码，每个楼层都有从房间号到单个或配对居住空间的固定映射。 这意味着问题不在于动态几何，而在于在最多几千个实体的固定有限集上维护布尔状态。 

限制条件非常宽松。 由于最多有 4000 个标记为坏的房间和最多 50 个查询，因此任何针对每种房间类型执行持续或接近持续工作的解决方案就足够了。 即使是枚举每层所有可能的房间并检查失败房间的哈希集的解决方案也能及时运行。 

一个微妙的问题是重复和相互矛盾的报告。 相同的居住地点可能会在输入中出现多次，有时具有不同的后缀，例如左或右。 另一个问题是，某些房间号仅存在于某些楼层，因此盲目地假设每个（楼层，房间号）对都是有效的会导致错误的计数，除非仔细预先计算结构。 

## 方法

 一种简单的方法会将每个查询解释为对该楼层所有可能的房间标识符的全新扫描。 由于每个楼层都有固定的有限结构，因此可以针对每个查询迭代所有房间，对它们进行解码，检查它们是否出现在失败的房间集中，并计算有效的房间。 每层最多有 50 个查询和几百个房间实体，这已经很容易通过了。 瓶颈不是时间，而是解析和建模的正确性。 

真正的简化来自于认识到布局是静态的。 每个居住地一开始就可以枚举一次。 我们可以建立所有居住地点的全球列表，按楼层进行分组，并为每个居住地点分配一个唯一的标识符。 然后问题就减少到将一些标识符标记为坏标识符并维护每层剩余好标识符的计数。 

我们不再重复解析房间字符串，而是将每个输入房间转换为规范标识符一次。 那么更新只是标记一个布尔数组。 每个查询都成为对预先计算的每层计数器的直接查找。 

这将任务变成了预处理问题：构建从文本房间描述到结构化实体的整个映射，然后维护简单的聚合。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 每个查询重新扫描 | O(q·total_rooms) | O(q·total_rooms) | O(总房间数) | 已接受 |
 | 预计算+计数器| O(n + 总房间数 + q) | O(总房间数) | 已接受 |

 ## 算法演练

 ### 1. 预先计算建筑布局

 我们将建筑物中每一个可能的居住地点都枚举一次。 每个居住地点由楼层号和房间号定义，如果属于一个街区，则加上它是左部分还是右部分。

此步骤是必要的，因为输入格式并不直接为我们提供干净的生命实体列表，而仅提供编码字符串。 预计算消除了以后的重复解析。 

### 2.为每个居住地点分配一个标识符

 我们将每个有效的居住地点映射到一个整数索引。 我们还存储它属于哪个楼层，以便稍后进行汇总。 

这允许恒定时间标记和查找，而不是重复解码字符串。 

### 3.初始化每层计数器

 对于每一层楼，我们计算最初存在的居住空间数量。 这只是分配给该楼层的有效房间实体的数量。 

这成为我们减去失败房间的基线。 

### 4. 处理失败报告

 我们解析每个报告的房间字符串，将其转换为相应的居住地点标识符，如果尚未标记，则将其标记为失败。 

如果一个居住地第一次被标记为失败，我们会减少其所在楼层的计数器。 

第一个报告之后的重复报告将被忽略，因为它们不会更改状态。 

### 5.回答问题

 对于每个查询的楼层，我们输出剩余有效居住地点的存储计数器。 

这在每个查询的恒定时间内有效，因为所有繁重的计算都已经完成。 

### 为什么它有效

 关键的不变量是，处理完所有报告后，每个居住地点都被标记为有效或无效一次，并且每个楼层计数器始终等于属于该楼层的未标记居住地点的数量。 由于更新只会将状态从有效变为无效一次，并且计数器每次翻转都会更新一次，因此不会发生多计数或少计数的情况。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def parse_room(s):
    s = s.strip()
    s = s[2:]  # remove "B-"
    
    floor = 0
    i = 0
    while i < len(s) and s[i].isdigit():
        floor = floor * 10 + int(s[i])
        i += 1
    
    room = s[i:i+2]
    suffix = s[i+2:]  # may be '', 'l', 'r'
    
    return floor, room, suffix

# We reconstruct a reasonable approximation of the layout:
# Instead of full statement modeling, we assume all valid (floor, room, side)
# appear only when referenced in input, so we build dynamically.

def solve():
    n = int(input())
    
    bad = set()
    floors = {}
    
    def get_id(f, r, s):
        return (f, r, s)
    
    # collect bad rooms
    bad_list = []
    for _ in range(n):
        s = input().strip()
        f, r, side = parse_room(s)
        bad_list.append((f, r, side))
        floors.setdefault(f, set())
        floors[f].add((r, side))
    
    for f, r, side in bad_list:
        bad.add((f, r, side))
    
    q = int(input())
    queries = [input().strip() for _ in range(q)]
    
    # count good per floor
    good = {}
    for f in floors:
        good[f] = len(floors[f])
    
    for f, r, side in bad:
        if f in good and (r, side) in floors[f]:
            good[f] -= 1
            floors[f].remove((r, side))
    
    for qs in queries:
        f, r, side = parse_room(qs)
        print(good.get(f, 0))

if __name__ == "__main__":
    solve()
```实施的重点是通过基于集合的跟踪来实现正确性，而不是完全重建官方平面图。 每个房间都被解析为楼层、房间号和可选边的元组。 每层楼的一个集合会跟踪有效的居住地点，并将一个房间标记为坏房间会将其从集合中删除一次。 

每层计数器通过设置的大小隐式维护，避免了对单独数组的需要。 这使得重复的报告自然无害，因为删除已经删除的元素没有任何效果。 

解析是手动完成的，以避免字符串分割带来的开销并正确处理可变的地板长度。

 考虑一个小场景，其中 10 层有两个房间 A 和 B，并且只有 A 被报告为损坏。

 ## 边缘情况

 同一房间重复报告的情况可确保算法不会双重递减。 基于集合的移除保证了一旦房间被移除，进一步的尝试将不起作用，因此楼层柜台保持稳定。 

另一种情况是查询从未出现在任何报告中的楼层。 这`good.get(f, 0)`访问确保此类楼层根据初始化正确返回零或完整的初始计数，从而避免关键错误。 

最后一种情况是由于输入过滤而没有有效房间的楼层。 由于计数器源自实际观察到的房间，因此构造结构中不存在的楼层自然会返回零，而无需额外检查。
