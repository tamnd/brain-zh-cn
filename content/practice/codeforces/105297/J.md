 ## 解决方案
 ## 问题理解

 我们得到了一份潜在客户名单，其中每个客户都有他们愿意为产品支付的最高价格。 如果我们设定一个价格$P$，那么正是那些客户$p_i \ge P$会购买该产品，并且每个人都做出贡献$P$到收入。 因此总收入为$P \times \#\{i : p_i \ge P\}$。 

任务是选择一个单一的价格$P$来自最大化该收入的真实可能值集，并输出所选价格和由此产生的最大收入。 如果多个价格达到相同的最大收入，则可以退回其中任何一个。 

关键的限制是$N$可以大到$10^6$，以及每个$p_i$可以达到$10^9$。 这立即排除了尝试数字范围内所有可能价格的任何方法。 即使不排序地迭代所有候选价格也会太慢，因为对每个价格进行简单的检查会导致二次行为。 

当所有值都相等时，会出现微妙的边缘情况。 例如，如果所有客户都有$p_i = 10$，那么任意价格$P \le 10$产生全部收入$10 \cdot N$。 仅测试输入价格而不考虑最优价格可能小于所有元素的粗心方法在这里仍然有效，但前提是它正确评估重复的候选者。 

当最优价格不一定是最高值之一时，就会出现另一个问题。 例如，如果价格是$[5, 5, 5, 100]$， 环境$P = 5$产生收入$5 \cdot 4 = 20$， 尽管$P = 100$产量$100 \cdot 1 = 100$。 假设较低的价格总是会带来更高的收入的解决方案在这里会失败。 

该结构表明，收入函数仅在数组中存在的值处发生变化，因为在两个连续的不同值之间，买家数量保持不变，而价格上涨，这只能线性提高收入，直到出现断点。 

## 方法

 一个直接的暴力想法是尝试所有整数中的每一个可能的价格$1$到$\max(p_i)$。 对于每个候选价格$P$，我们数一下有多少个$p_i \ge P$，然后计算收入。 这是正确的，因为它评估了问题的确切定义。 

然而，这种方法是不可行的。 价格范围可以达到$10^9$，对于每个价格，我们可以扫描所有$N$顾客。 这导致$O(N \cdot \max p)$，这是一个天文数字。 

关键的观察结果是，仅当价格超过数组中的某个值时，买家数量才会发生变化。 在两个连续排序值之间，买家数量保持不变，因此收入函数是线性的$P$并在这些间隔的端点之一达到最大值。 这意味着我们只需要考虑等于某些价格的价格$p_i$。 

一旦我们对数组进行排序，我们就可以有效地计算有多少元素大于或等于每个不同的值。 如果我们按排序顺序处理值，我们可以跟踪在提高价格阈值时剩余的客户数量。 

通过扫描排序后的数组并将每个不同的值视为候选价格，我们可以在排序后计算线性时间内的收入。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 |$O(N \cdot \max p)$|$O(1)$| 太慢了|
 | 排序+扫一扫|$O(N \log N)$|$O(N)$| 已接受 |

 ## 算法演练

 我们首先对数组进行排序，以便我们可以推断出每个可能的价格阈值还剩下多少客户。 排序至关重要，因为它将问题转化为后缀计数问题。 

然后，我们迭代排序后的数组，但我们不是独立评估每个位置，而是将相等的值分组在一起。 对于每个不同的候选价格$P = p_i$，我们计算有多少个元素大于或等于$P$。 如果数组按非递减顺序排序，并且我们位于索引处$i$，那么购买者的数量就是$N - i$。 

我们将收入计算为$P \times (N - i)$，并跟踪所有此类候选者的最大值。 

该算法进行如下。 

## 算法演练

 1. 对数组进行排序$p$按非递减顺序。 这使得所有具有较高阈值的客户可以自然地分组在一起，因此后缀长度直接代表买家数量。 
2. 初始化一个变量来跟踪迄今为止找到的最佳收入以及相应的价格。 我们从零收入和任意价格开始。 
3. 从左到右遍历排序后的数组。 对于每个索引$i$， 对待$p[i]$作为候选售价。 
4. 计算购买者数量：$N - i$，因为所有元素都来自$i$到最后至少有$p[i]$。 这是有效的，因为排序保证了单调性。 
5. 计算收入$R = p[i] \times (N - i)$。 将其与迄今为止的最佳收入进行比较，如果更大则更新。 如果相等，我们可以保留其中一个值。 
6. 对所有指数继续此过程，确保重复值不会导致问题，因为它们产生相同的价格，但后缀计数可能不同； 两者都是安全的候选人。 

### 为什么它有效

 正确性来自于这样一个事实：对于任何固定价格$P$，购买的客户群正是那些拥有$p_i \ge P$。 在排序数组中的两个连续不同值之间，该集合不会改变。 增加$P$在这样的区间内只会线性减少收入，因此最优值不能严格位于两个相邻的不同值之间。 因此，仅测试等于数组元素的价格就足够了，并且在正确排序后计算后缀计数可以捕获所有此类情况。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = list(map(int, input().split()))
    a.sort()

    best_rev = 0
    best_p = a[0]

    i = 0
    while i < n:
        j = i
        while j < n and a[j] == a[i]:
            j += 1

        p = a[i]
        buyers = n - i
        rev = p * buyers

        if rev > best_rev:
            best_rev = rev
            best_p = p

        i = j

    print(best_p, best_rev)

if __name__ == "__main__":
    solve()
```排序步骤是解决方案的基础。 它确保当我们处于索引时$i$, 后缀$i \ldots n-1$完全对应所有能承受价格的客户$a[i]$。 

分组逻辑为$j$避免对重复值进行冗余重新计算。 虽然没有严格要求，但它使推理与只有不同的价格才重要的想法保持一致。 

收入计算在 Python 中隐式使用 64 位安全算术，但在具有固定整数类型的语言中，此步骤需要小心，因为$10^9 \cdot 10^6$适合 64 位有符号范围，但不适合 32 位。

 ## 边缘情况

 对于像这样的单元素输入$N=1$，该算法进行简单排序并考虑一个候选者$P = p_1$, 产生收入$p_1$。 后缀计数为1，因此结果是正确的。 

对于所有值都相同的情况，例如$[7,7,7,7,7]$，每个指数产生相同的收入$7 \cdot 5$。 分组步骤确保我们只评估一次，但即使不分组，每个重复项都会产生相同的结果，因此保留了正确性。 

对于一个大值与许多小值混合的情况，例如$[1,1,1,100]$，该算法评估两者$P=1$和$P=100$。 在$P=1$，收入为$4$，同时在$P=100$，收入为$100$。 基于后缀的计算正确地捕获了这种对比，因为排序将大值放在末尾，给它后缀大小 1，这直接产生了它的收入。
