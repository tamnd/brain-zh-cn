 ## 解决方案
 ## 问题理解

 我们得到了数轴上的一系列点，必须以固定的顺序访问这些点。 我们从位置 0 开始。在每一步中，我们都会移动到列表中的下一个所需位置，但我们可以在两种移动模式之间进行选择。 

正常的移动会将我们从当前位置移动到下一个点，并花费它们之间的绝对距离。 传送移动会忽略当前位置，而是使用存储的“传送锚点”位置：其成本是该锚点与下一个点之间的距离。 使用传送后，系统会更新，使锚点变为传送前我们所在的位置，并丢弃旧的锚点。 

所以这个过程是连续的：我们总是按顺序访问点，但每次转换都可以通过两种方式之一支付，要么从我们现在所在的位置开始，要么从最后一个传送锚点开始。 

目标是最小化到达终点的总成本。 

约束足够小，以至于$O(n^2)$或者$O(n^2 \log n)$动态规划解决方案是可以接受的，因为$n \le 1000$。 如果简单地实现的话，这会立即排除任何立方或更复杂的每个状态转换，但允许在索引上进行二维 DP。 

当所有点都靠近时，会出现微妙的边缘情况，但只有在某些“枢轴”点之后，传送才重复有益。 像“每当本地有帮助时就进行传送”这样的贪婪策略会失败，因为锚点的选择会影响所有未来的传送成本，而不仅仅是下一次转换。 另一个边缘情况是坐标振荡时，例如 0、100、1、99，其中最佳游戏取决于保持最佳历史锚点而不是最新位置。 

## 方法

 暴力方法会模拟每一个可能的决策序列。 在每一步中，我们都会决定是正常移动还是瞬移，如果我们瞬移，我们还会隐式决定哪个先前位置成为新的锚点。 这会导致指数分支，因为锚的演化取决于整个历史。 即使进行修剪，状态空间也会变得太大。 

关键的观察是，任何时刻唯一重要的信息是两个位置：当前位置和当前传送锚点。 每个决策都会更新这两个值之一，未来的成本仅取决于它们。 这将问题简化为对最后访问点和最后锚点的索引的动态编程状态。 

我们通过当前位置的索引和先前访问过的位置中的锚点的索引来定义 DP。 由于两个指数的范围均为$n$，我们得到$O(n^2)$州。 从每个状态，我们通过正常移动（保持锚点不变）或传送（将锚点更新到之前的当前位置）过渡到下一个点。 

这种结构消除了两个指数之外的历史依赖性，这正是问题易于处理的原因。

 | DP over (i, 锚点) | O(n^2) | O(n^2) | O(n^2) | O(n^2) | 已接受 |

 ## 算法演练

 我们按顺序处理这些点，构建一个 DP 表，其中每个状态代表已经访问过某个索引，并记住哪个较早的位置是当前的传送锚点。 

1. 在访问任何点之前，在起始位置初始化 DP，我们位于位置 0，锚点实际上也为 0。 
2. 对于第一点，计算从一开始就达到该点的成本。 这会初始化 DP 状态的第一层，因为每个后续状态都建立在有效的第一步之上。 
3. 对于每个下一个点索引 i，考虑由当前位置 j 和锚 k 定义的所有先前状态。 这些代表当传送锚点为 k 时到达点 j 的所有可能方式。 
4. 从状态 (j, k)，以两种方式计算移动到 i 的成本。 正常移动成本 |a[i] - a[j]| 并保持锚 k 不变。 这反映了在不改变传送配置的情况下简单地向前走。 
5. 瞬移移动需要花费|a[i] - a[k]|，使用后当前位置变为k，新锚点变为j。 这种交换反映了传送消耗先前锚点并将其替换为先前位置的规则。 
6. 相应地更新 DP 转换，在所有方法中采用最小成本来达到每个结果状态。 
7. 处理完所有点后，答案是当前位置是最后一个点的所有状态中的最小成本，无论锚点如何。 

为什么有效：每个状态都精确编码计算未来转换所需的信息，因为允许的成本仅取决于当前位置和锚点位置。 一旦这两个值被固定，早期的历史就不会影响未来的决策。 这使得 DP 最优子结构有效，并且每个有效的移动序列恰好对应于 DP 状态图中的一条路径。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = list(map(int, input().split()))

    # dp[j][k] = minimum cost where:
    # j = current position index
    # k = teleport anchor index (0..n-1, plus 0 as virtual start)
    INF = 10**30

    dp = [[INF] * (n + 1) for _ in range(n + 1)]

    start = n  # use index n as virtual node for position 0
    dp[start][start] = 0

    # helper to get coordinate
    def coord(i):
        return 0 if i == n else a[i]

    for i in range(n):
        ndp = [[INF] * (n + 1) for _ in range(n + 1)]

        ci = a[i]

        for j in range(n + 1):
            for k in range(n + 1):
                cur = dp[j][k]
                if cur >= INF:
                    continue

                cj = coord(j)
                ck = coord(k)

                # normal move: j -> i
                cost1 = cur + abs(ci - cj)
                if cost1 < ndp[i][k]:
                    ndp[i][k] = cost1

                # teleport move: k -> i, update anchor
                cost2 = cur + abs(ci - ck)
                if cost2 < ndp[j][i]:
                    ndp[j][i] = cost2

        dp = ndp

    ans = min(dp[n][k] for k in range(n + 1))
    print(ans)

if __name__ == "__main__":
    solve()
```该实现显式地将 DP 存储在索引对上，包括位置 0 的虚拟索引。辅助函数`coord`将该虚拟索引映射到坐标 0，因此距离计算保持统一。 

转换逻辑直接镜像两个允许的操作：在保留锚点的同时正常移动，或者传送并将锚点与当前位置交换。 第二个转换是唯一微妙的部分，因为它编码了传送消耗并替换其锚状态的规则。 

我们使用一个完整的$O(n^2)$每层 DP 表，这是必要的，因为两个维度都会随着转换而变化。

 ## 边缘情况

 一种边缘情况是当$n = 1$。 该算法减少到从 0 到$a_1$，并且由于没有先前的结构可供利用，因此任何传送都不会改善结果。 DP 正确初始化，因为第一次转换直接计算 |a[0] - 0|。 

另一个边缘情况是严格增加坐标，例如 1、2、3、4。这里传送没有任何帮助，因为任何锚点总是比未来点的当前位置更远。 DP 仍在探索传送状态，但它们从未占据主导地位，因此最终答案与简单的差异之和相匹配。 

更微妙的情况是交替使用大值和小值，例如 0、100、1、99，其中最佳锚点在极端值之间交替。 DP 正确地捕获了这一点，因为它评估所有锚点当前组合，确保即使在需要非贪婪决策时也不会错过最佳交换序列。
