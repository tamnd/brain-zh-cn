 ## 解决方案
 ## 问题理解

 任务是计算图表中目标节点的最早可能到达时间，其中每条边的行为就像交通控制的走廊。 您在固定的起始时间从节点 1 开始，每条边基本上不需要旅行时间，但只能在特定的周期性“打开窗口”期间遍历。 每个边缘在打开一段时间和关闭一段时间之间交替，并且该模式从时间零开始与开路阶段无限地重复。 

当你到达一个节点时，你可能必须等待多个传出边缘，直到它们各自的红绿灯允许通过。 目标是选择一条从节点 1 到节点 N 的路线，使到达节点 N 的最终到达时间最小化，同时考虑到这些循环约束所施加的所有等待时间。 

该图最多可以有 100,000 个节点和 200,000 个边，这会立即排除任何逐步模拟时间或重新计算每个可能路径的等待行为的解决方案。 最短路径框架是必要的，但与标准图不同，边成本取决于到达节点的时间，这会动态改变遍历成本。 

当贪婪的选择忽略等待周期时，就会出现微妙的失败情况。 例如，考虑两条路线：一条路线具有较少的边缘，但由于缺少打开的窗口而导致等待时间较长，而另一条路线具有更多的边缘，但与周期更好地对齐并导致更早到达。 忽略时序的简单 BFS 或 DFS 将产生不正确的结果，因为边缘成本不固定。 

另一个常见的陷阱是假设在节点等待一次就足够了。 由于每个边缘都有自己的周期，因此同一节点可能会导致非常不同的延迟，具体取决于选择的传出边缘以及到达那里的时间。 

## 方法

 蛮力的想法是在模拟时间的同时尝试探索从节点 1 到节点 N 的所有可能路径。 在每个节点，我们将尝试每个外出边缘并根据边缘的周期计算下一个可能的出发时间。 这本质上变成了对节点和时间的完整状态空间搜索，其中时间可以增长到 10^9。 即使我们离散化时间转换，状态数量也会爆炸，因为节点的每个到达时间都可能导致不同的未来行为，因此必须使用不同的时间戳多次重新访问同一节点。 在最坏的情况下，这就像路径的指数数量乘以循环对齐检查，这远远超出了任何可行的限制。 

关键的观察结果是，这仍然是一个最短路径问题，但具有依赖于时间的边缘松弛函数。 当到达某个节点时`t`，遍历一条边的成本不是恒定的，但可以在其周期上使用模算术在 O(1) 中计算。 如果我们将到每个节点的“距离”视为已知的最早到达时间，那么这种结构允许我们使用 Dijkstra 算法。 

转移规则变为：如果我们在时间`t`在一个节点并考虑具有开放时间的边`x`以及闭馆时间`y`，然后我们首先计算我们在长度循环中的位置`x + y`。 如果我们处于开放阶段，我们立即离开； 否则，我们会等到下一个开放阶段开始。 这给出了该边缘的确定性下一次到达时间。 

这将问题转换为具有非负边权重的标准单源最短路径问题，其中权重是动态计算的，但始终对 Dijkstra 的贪婪属性有效。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | 指数| 高| 太慢了|
 | 最优（Dijkstra）| O(M log N) | O(M log N) | O(N + M) | 已接受 |

 ## 算法演练

 1. 将每个交叉路口建模为图中的一个节点，将每条街道建模为带有参数的无向边`(x, y)`描述其开环和闭环。 这使我们能够将运动重新解释为具有时间相关成本的图遍历。 
2.初始化一个数组`dist[]`具有较大的值和集合`dist[1] = t`，因为我们当时从节点 1 开始`t`。 该数组跟踪每个节点的最早已知到达时间。 
3. 推`(t, 1)`进入最小堆优先级队列。 堆总是选择当前已知的最早到达的状态来进行下一步扩展，确保我们按时间递增的顺序处理节点。 
4. 反复提取节点`u`到达时间最短`cur`。 如果该值与之前相比已经过时`dist[u]`，跳过它。 这可以防止对次优路径进行冗余处理。 
5. 对于每个邻居`v`的`u`有循环`(x, y)`，计算下一次出发时间：

 首先计算`cycle = x + y`和`pos = cur % cycle`。 如果`pos < x`，边当前是开放的，我们可以立即遍历`cur`。 否则，我们必须等到下一个周期的开放阶段，这样出发就变成了`cur + (cycle - pos)`。 
6. 到达时间`v`等于计算的出发时间，因为行程时间可以忽略不计。 如果这个值提高`dist[v]`，更新并推送`(dist[v], v)`到堆中。 
7. 继续，直到处理完所有可达节点。 答案是`dist[N]`。 

正确性依赖于这样一个事实：一旦从优先级队列中弹出一个节点，我们就找到了该节点的最小可能到达时间。 任何较晚的到达都将需要严格更长的时间，并且由于边缘转换永远不会产生负时间减少，因此未来的松弛无法改善最终状态。 

## Python 解决方案```python
import sys
import heapq
input = sys.stdin.readline

def solve():
    n, m, t = map(int, input().split())
    g = [[] for _ in range(n + 1)]

    for _ in range(m):
        a, b, x, y = map(int, input().split())
        g[a].append((b, x, y))
        g[b].append((a, x, y))

    INF = 10**30
    dist = [INF] * (n + 1)
    dist[1] = t

    pq = [(t, 1)]

    while pq:
        cur, u = heapq.heappop(pq)
        if cur != dist[u]:
            continue

        for v, x, y in g[u]:
            cycle = x + y
            pos = cur % cycle

            if pos < x:
                nxt = cur
            else:
                nxt = cur + (cycle - pos)

            if nxt < dist[v]:
                dist[v] = nxt
                heapq.heappush(pq, (nxt, v))

    print(dist[n])

if __name__ == "__main__":
    solve()
```该图被存储为邻接列表，因为我们需要在 Dijkstra 松弛期间有效地遍历所有出边。 每次松弛使用单个模运算计算最早的可行交叉时间，从而保持转换 O(1)。 

优先级队列确保节点按照已知到达时间的升序进行处理。 陈旧状态检查`if cur != dist[u]`避免扩展过时的条目，这对于保持复杂性为对数而不是二次至关重要。 

一个常见的实施错误是忘记必须在到达时间而不是在出发计划时间评估循环条件。 另一个是错误地处理闭区间延迟，其中等待必须跳转到下一个开相边界，而不是仅仅添加`y`。 

## 工作示例

 ### 示例 1

 输入：```
2 1 3
1 2 4 4
```循环长度为 8。从时间 3 开始，循环中的位置为 3，位于开区间内`[0,4)`，所以遍历是立即的。 

| 步骤| 节点| 时间 | 行动|
 | ---| ---| ---| ---|
 | 1 | 1 | 3 | 开始|
 | 2 | 2 | 3 | 立即遍历边缘 |

 输出为3。 

这演示了不需要等待并且起始偏移已经位于开路相位内的情况。 

### 示例 2

 输入：```
2 1 3
1 2 3 4
```循环长度为 7。在时间 3，位置为 3，正好位于闭区间的开始处，因此我们必须等到时间 7。 

| 步骤| 节点| 时间 | 行动|
 | ---| ---| ---| ---|
 | 1 | 1 | 3 | 开始|
 | 2 | 1 | 7 | 等待下一个开放阶段 |
 | 3 | 2 | 7 | 横移边缘 |

 输出为7。 

这说明了为什么简单地添加固定权重会失败，因为等待取决于与周期的对齐。

 ## 边缘情况

 当开始时间恰好位于开区间和闭区间之间的过渡处时，就会出现临界边缘情况。 对于边缘`(x, y)`， 如果`cur % (x + y) == x`，边缘不能立即使用，即使它可能看起来仍处于边界状态。 该算法正确地强制完全等待下一个周期开始，因为`pos < x`是严格要求的。 

另一个微妙的情况是等待将时间推向多个周期。 例如，如果`cur`远远进入闭线段，则公式`cycle - pos`正确地直接跳转到下一个开放阶段，而不是逐周期迭代。 这可以防止等待的线性时间模拟。 

最后，当多条边以不同的时间通向同一节点时，优先级队列确保只有最早到达的节点能够生存。 任何较晚的到达都会通过过时检查被丢弃，这保证了没有次优路径在图中进一步传播。
