 ## 解决方案
 ## 问题理解

 我们有一个拥有固定数量树木槽的苗圃，以及一家出售多种树木的商店。 每种类型的相同种子的供应量有限，并且每种类型的种子都会产生具有固定美丽值的树。 

我们希望通过从这些类型中采摘种子来精确种植给定数量的树木，同时尊重可用性限制。 每颗选定的种子都会为整体贡献其美丽价值，目标是最大化所有种植树木的美丽总和。 

输入由少量树类型组成，每种树类型都通过可用种子的数量以及每个种子的价值来描述。 我们必须决定从每种类型中取出多少种子，以便所选种子的总数等于所需的花园大小，同时最大化整体美观。 

这些约束意味着贪婪或基于排序的方法就足够了。 类型的数量最多为 100，因此迭代它们的成本很低。 然而，要采摘的树总数可能高达 100000 棵，因此任何在没有结构的情况下模拟一次采摘一棵树的方法只有在总容量呈线性时才是安全的，但我们可以通过聚合选择做得更好。 

一个天真的想法是单独处理每个种子，将所有类型展开为一个最多包含 sum(t_i) 项的长列表，然后按美观度排序并选择最好的 m。 这是正确的，但如果在最坏的情况下种子总数达到 2e5 或更多，速度可能会太慢。 

当一种类型占主导地位或当 m 超过或接近可用种子总数时，边缘情况很重要。 

当我们忘记不能从类型中获取超过 t_i 的种子时，粗心实现的微妙失败案例就会出现。 例如，如果一种类型具有较高的美感，但只有3个种子，即使m很大，取10个也是无效的。 

另一个极端情况是 m 恰好等于总供应量。 任何算法都必须采取一切，而无需对部分选择进行不必要的逻辑假设。 

## 方法

 强力解释是创建所有种子的概念列表，其中每个种子出现的次数与其相关的美感一样多。 对该列表进行排序并直接取前 m 个值即可给出答案，因为每个种子都是独立的并且会相加。 正确性是立竿见影的，但是这个扩展列表的构造可能涉及多达 200000 个条目，并且对其进行排序的成本为 O(N log N)，这仍然是可以接受的但不是必需的。 

关键的观察是同一类型的所有种子的值都是相同的，因此我们永远不需要在类型中混合。 如果我们要使用一个类型，我们总是首先获取它最有价值的单位，但由于类型内所有单位都是相等的，所以我们只需获取尽可能多的单位，直到 t_i。 这将问题转化为从多重集中选择项目，其中每个组具有统一的值，因此按值对组进行排序就足够了。 

我们按美感降序对类型进行排序，然后贪婪地从最高美感类型中获取，直到填满 m 个槽。 每种类型要么贡献其所有可用种子，要么仅贡献剩余的所需数量。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 展开所有种子 + 排序 | O(S 日志 S) | O(S)| 已接受但没有必要 |
 | 排序类型+贪婪填充 | O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

1. 读取树的类型数量和所需数量。 每种类型都给出一对由可用计数和美观值组成的对。 
2. 按照美观度从高到低的顺序对类型进行排序，因为我们总是首先想要价值更高的树。 
3. 将答案初始化为零并跟踪我们还需要采摘多少棵树。 
4. 迭代排序后的类型。 对于每种类型，决定要采用多少棵树：要么该类型的所有可用种子，要么仅保留仍需要的种子，以较小者为准。 添加他们对整体美丽的贡献。 
5. 相应减少剩余配额。 
6. 一旦配额达到零，就尽早停止，因为不再需要更多的树木。 

关键思想是，在每一步中，我们都致力于最佳可用的边际选择，并且由于类型内的所有项目都是相同的，因此类型内不存在可以改善以后结果的隐藏排序。 

### 为什么它有效

 在这个过程中的任何一点，我们都认为所有剩余的未加工类型的美感都小于或等于当前正在考虑的类型。 由于同一类型中的所有物品都是相同的，因此将任何选定的物品与后续物品交换只会降低或保持整体美观。 这使得贪婪选择局部最优且全局一致，因为选择问题简化为没有内部结构的最高重量块的填充容量。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    types = []
    
    for _ in range(n):
        t, b = map(int, input().split())
        types.append((b, t))
    
    types.sort(reverse=True)  # sort by beauty descending
    
    ans = 0
    remaining = m
    
    for b, t in types:
        if remaining == 0:
            break
        take = min(remaining, t)
        ans += take * b
        remaining -= take
    
    print(ans)

if __name__ == "__main__":
    solve()
```代码直接实现了贪心策略。 按美观度降序排序，以便循环始终首先处理最有价值的树。 这`take = min(remaining, t)`线路是实施供应约束的关键保障。 

一个常见的实现错误是忘记限制每种类型所采用的树的数量，这会超出可用性的计算。 另一种是排序方向错误，这会颠倒贪婪逻辑并产生次优总和。

 ## 边缘情况

 一种极端情况是单一类型在数量和美观上都占主导地位。 例如：```
1 5
100 10
```该算法立即从唯一类型中取出 5 棵树并输出 50。循环执行一次，并且`take`正确地限制在 m，因此不会发生过度选择。 

另一种情况是多种类型具有相同的美观值：```
3 6
2 5
3 5
10 5
```排序后所有类型具有相同的优先级。 该算法只是从左到右累加，直到 m 被填满，并且它们之间的任何顺序都是有效的，因为所有贡献都是相同的。 

最后一种情况是 m 等于 0，如果在某些变体中解释不正确，就会发生这种情况。 由于以下原因，循环立即终止`remaining == 0`保护，确保类型不会意外添加。
