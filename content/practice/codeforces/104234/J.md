 ## 解决方案
 ## 问题理解

 我们正在模拟一个受约束的赌博过程，其行为不太像独立投注，而更像是具有有限“时间旅行”重置的受控状态机。 

王子从一枚硬币开始，必须准确地通过`n`投注回合。 在每一轮中，他选择一个赌注金额（至少 1 个硬币）和公平的 50-50 轮盘赌的一面。 获胜则使赌注翻倍，失败则减少赌注金额。 不同之处在于，在看到投注结果后，他可能会将时间倒回到该投注之前，并以不同的方式重播。 此次回滚仅限于`m`总计使用，回滚后重新下注不会额外消耗一轮。 

我们的目标不仅仅是避免失去一切。 他必须保证在每一个`n`要求的赌注，他至少有一枚硬币可用，以便合法的赌注是可能的。 在满足这一生存约束的所有策略中，我们希望最大化预期的最终硬币数量。 如果不能保证生存，答案就是“破产”。 

关键的困难在于回滚从根本上改变了风险的结构。 失败的结果并不总是最终的，但只能弥补有限数量的损失。 这将问题转化为对最坏情况损失序列的推理，而不仅仅是投注的预期值。 

约束条件意味着`n`和`m`都达到`10^5`，总共`m`跨测试用例最多`10^6`。 每个测试用例的任何解决方案都必须基本上是线性的。 这立即排除了对“硬币×回合×回滚计数”等状态的任何动态编程，因为硬币值是无界的，并且概率分支会爆炸。 

出现微妙的边缘情况时`m`相比之下较小`n`。 如果回滚不足，对抗性的一系列损失可能会迫使 Prince 必须下注，但没有剩下硬币，也没有可用于恢复的回滚。 例如，如果`n = 3`和`m = 0`，第一次下注的任何损失都会导致立即失败，因为无法在下一次所需下注之前修复状态。 

另一个边缘情况是，技术上可以从丢失中恢复，但前提是在非常特定的计时模式中使用回滚。 这使得贪婪的“总是下注 1 个硬币”策略无效，因为它们没有考虑最坏情况的耗尽路径。 

## 方法

 暴力解释将尝试模拟每一个可能的输赢序列，跟踪每一步的硬币计数和回滚使用情况。 在每次投注中，我们都会分为赢或输，并可选择应用回滚决策。 这构建了一个巨大的状态树，其中每个节点代表结果和倒带的可能历史记录。 即使进行剪枝，可达状态的数量也会呈指数增长`n`，因为每次下注都会使分支因子加倍，并且回滚会引入额外的组合选择。 

关键的简化来自于将概率与可行性分开。 回滚不是一个期望工具，它是一个生存保证工具。 它有效地允许玩家“擦除”最多`m`在最坏的情况下会出现不好的结果，这意味着我们只关心在系统变得不可挽回之前可以容忍多少损失。 

一旦以这种方式看待，问题就变成了确保在流程的任何前缀上，不可避免的损失数量永远不会超过可用的恢复能力。 每次回滚都可以被认为是修复一个灾难性的赤字事件，即在所需下注之前代币跌至零。 

这导致了一个结构性观察：游戏只能保证`n`押注初始资源是否能够承受最坏的可能的故障序列，其中每次故障都会消耗回滚容量，之后，剩余的结构表现得像确定性增长过程。 

最优策略归结为检查维持的可行性`n`在最坏情况损失压力下采取步骤，如果可行，计算不再依赖于分支模拟的确定性预期增长。

 ## 算法演练

 关键思想是将回滚重新解释为有限数量的“从破产时刻重置”。 仅当在所需的投注之前，硬币计数为零并且没有回滚余地来修复先前选择的投注时，系统才会变得无效。 

我们通过跟踪最坏情况下的最小可能硬币轨迹来模拟该过程。 每次下注，如果总是失败，就会减少硬币池。 回滚允许我们将此类崩溃转换为恢复，从而以回滚为代价有效地将可用资源加倍。 

## 算法演练

 1.观察到最坏情况下的每次下注都会减少资源，回滚只有在出现强制零币状态时才有用。 这意味着回滚容量仅在轨迹失效时消耗，而不是在正常播放期间消耗。 
2. 考虑顺序`n`假设每次投注都为亏损的情况下进行投注。 这会产生确定性的消耗路径，其中硬币每次都会减少下注金额。 可行性问题变成我们是否可以避免在步骤之前达到零`n`最多使用`m`修理。 
3. 每次回滚都可以解释为允许我们“修复”一次故障事件，有效地将系统恢复到之前的安全状态，并继续增加有效资本。 这意味着回滚就像一个有限的预算来取消灾难性的下降。 
4. 如果最坏情况轨迹中不可避免的崩溃点数量不超过`m`。 由于只有当硬币数量为零时才会发生崩溃，因此该结构简化为确保获胜的初始增长可以抵消最坏情况下的损失。 
5. 一旦可行性得到保证，预期值就会降低到标准公平投注增长。 每次下注都会使预期的预期贡献翻倍，因此预期最终价值的行为类似于剩余资本的线性递归。 
6. 首先使用最坏情况消耗的贪婪模拟来计算可行性，跟踪需要回滚多少次才能防止破产。 如果这超过`m`，输出“破产”。 
7. 否则通过迭代赌注并应用期望更新来计算期望值`E = E + 0.5 * bet`,`E = E + 0.5 * (-bet)`结构，它简化了最佳回滚使用下的确定性加倍行为。 

### 为什么它有效

 不变的是，回滚仅在过程将变得不确定的点上消耗（在所需下注之前为零硬币）。 在这些点之间，该过程的行为符合预期，因为每个赌注都是对称的。 由于回滚使用是有限的，唯一有意义的约束是最坏情况路径是否产生比可用回滚预算更多的故障事件。 一旦满足该约束，期望值仅取决于公平加倍的线性期望，与路径分支无关。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MOD = 10**9 + 9

def solve():
    t = int(input())
    for _ in range(t):
        n, m = map(int, input().split())

        # feasibility check: minimal interpretation of collapse events
        # each rollback can fix at most one forced bankruptcy event
        # worst case requires n-1 repairs if structure is linear
        if m < n - 1:
            print("bankrupt")
            continue

        # if feasible, expected growth reduces to doubling each step
        # starting from 1 coin
        ans = 1
        for _ in range(n):
            ans = (ans * 2) % MOD

        print(ans)

if __name__ == "__main__":
    solve()
```可行性检查对应的想法是，如果没有足够的回滚，完全失败的轨迹会在完成所有所需的赌注之前迫使重复无效状态。 条件`m < n - 1`捕获在第一次故障级联后修复每个步骤的最坏情况必要性。 

第二部分计算最佳游戏下的预期结果。 由于每个赌注都是公平的，并且回滚确保了过程的生存，因此期望随着当前资本的重复加倍而演变，这通过迭代以模幂的形式实现。

import sys, io

MOD = 10**9 + 9

def solve():
    input = sys.stdin.readline
    t = int(input())
    for _ in range(t):
        n, m = map(int, input().split())
        if m < n - 1:
            print("bankrupt")
        else:
            ans = 1
            for _ in range(n):
                ans = (ans * 2) % MOD
            print(ans)

def run(inp: str) -> str:
    old_stdin = sys.stdin
    sys.stdin = io.StringIO(inp)
    old_stdout = sys.stdout
    sys.stdout = io.StringIO()
    solve()
    out = sys.stdout.getvalue()
    sys.stdin = old_stdin
    sys.stdout = old_stdout
    return out.strip()

 ## 边缘情况

 一个关键的边缘情况是当`n = 1`。 在这种情况下，不需要回滚，因为没有跨多个投注恢复的顺序。 该算法正确地将其视为可行，无论`m`，期望值变为`2`，反映了单一公平投注翻倍的预期。 

另一种边缘情况发生在`m = 0`和`n > 1`。 在这里，任何想要在多次投注中生存的尝试都需要从无法修复的损失链中恢复过来。 可行性检查立即拒绝了这种情况，这与序列早期的一个不幸结果导致不可能完成这一事实相匹配。 

最后，当`m >= n - 1`，系统过渡到完全可生存的状态，其中回滚足以修补最坏情况轨迹中的每个潜在崩溃。 在这种情况下，所有随机性都被吸收到期望中，最终结果仅取决于重复加倍`n`步骤。
