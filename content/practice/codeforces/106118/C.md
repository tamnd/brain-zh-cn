 ## 解决方案
 ## 问题理解

 我们收到了一份小工具价格清单和初始金额。 购买规则是有限制的：在购买任何小玩意之前，当前的金额必须至少是该小玩意的价格的十倍。 购买后，钱会根据该小工具的价格而减少。 目标是选择一个购买顺序，以最大化可以购买的小工具数量。 

这不仅仅是子集选择问题，因为可行性取决于获取项目的顺序。 尽早购买便宜的物品会减少金钱，并且可以阻止以后购买昂贵的物品，而首先购买昂贵的物品可能会违反“购买前 10 次”的限制。 因此，任务是找到有效购买的序列，使序列的长度最大化。 

限制很大：最多 200,000 个小工具，初始资金最多 10^15。 这立即排除了尝试所有排列或任何指数子集搜索的任何方法。 即使是重复扫描列表以查找有效的下一项的 O(n^2) 贪婪模拟也太慢。 我们需要一种策略，其中每个项目仅被考虑少量次，理想情况下是 O(log n) 或 O(1) 摊销。 

当存在许多小项目时，会出现微妙的边缘情况。 天真的直觉可能会建议总是先购买最便宜的商品。 这可能会失败，因为过早减少资金可能会阻止购买仍然符合条件的稍大的物品。 例如，如果 m = 100 并且物品为 [9, 10, 11]，则首先购买 9 是合法的，因为 100 ≥ 90，但它会将钱减少到 91，然后 10 需要 100 但 91 < 100 所以它变得不可能。 然而，先买10个有效：100≥100，然后钱就变成90，9和11的决定不同。 这表明顺序至关重要。 

另一个极端情况是，一种非常昂贵的商品在技术上首先是可以承受的，但随后会阻止所有较小的商品，即使首先购买较小的商品会解锁更多的总购买量。 

## 方法

 暴力方法会尝试所有小工具的排列，并模拟每个订单的购买过程，检查根据规则可以购买多少件商品。 每次模拟都是O(n)，有n！ 排列，即使 n = 20 也是完全不可行的。 

一种更结构化的强力改进是使用回溯：在每一步中，尝试每一个当前负担得起的项目。 然而，即使如此，在最坏的情况下，许多物品都是可以负担的，分支因子仍然很大并导致指数爆炸。 

关键的观察是，该决定以一种非常具体的方式是局部的：我们是否可以购买某种物品仅取决于当前的货币，一旦我们决定购买某种东西，它总是会减少金钱。 所以我们想反过来使用这个规则：在任何时候，我们都应该选择在 10 倍约束下勉强负担得起的物品，因为现在购买更便宜的东西只会降低未来的灵活性，而不会提高较大物品的可行性。 

这建议对价格进行排序，并始终尝试采用满足约束的最小可行项目。 然而，我们必须小心：约束取决于当前的货币，而不仅仅是固定的承受能力。 我们需要一个动态结构，始终跟踪哪些项目当前符合条件并选择最好的一个。 

我们维护一个排序列表并逐步“解锁”价格最多为 m/10 的物品。 一旦某件商品符合条件，我们就可以考虑购买它。 在符合条件的项目中，选择最小的项目是最佳选择，因为它减少的资金最少，从而保留了未来的资格。 

这将问题简化为使用已排序数组上的指针的贪婪进程。

 | --- | --- | --- | --- |
 | 蛮力排列 | O(n!) | O(n) | 太慢了|
 | 贪心排序 | O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

我们首先按升序对所有小工具价格进行排序。 这使我们能够随着资金的减少而单调地跟踪资格。 

我们维护一个扫描排序数组的指针和一个存储当前规则下可负担的所有项目的结构。 

在每一步中，我们都会在当前价格最多为 current_money / 10 时重复向前移动指针，将这些项目添加到候选池中。 

从这个池中，我们选取​​最小的可用项目，因为它可以最大限度地减少资金减少并最大化未来项目保持合格的机会。 

我们从当前货币中减去它的价格并增加答案。 我们继续这个过程，直到没有更多的项目可以添加到池中并且池中没有项目可供选择。 

如果在某个时候池是空的并且下一个最小的项目太昂贵而无法满足 10x 规则，我们将终止。 

### 为什么它有效

 关键的不变量是，在任何时候，当前符合规则的所有项目要么已经在候选池中，要么以后永远不会符合资格，因为钱只会减少。 因此，一旦某个项目跳过了资格阈值，它就永远无法重新进入可行性阶段。 在当前符合条件的项目中，选择最小的项目是安全的，因为它可以最大限度地减少未来资格门槛的降低，而任何更大的选择只会使未来的限制更加严格，而不会提高任何项目的可访问性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    a = list(map(int, input().split()))
    a.sort()

    i = 0
    ans = 0
    money = m

    while True:
        # add all newly affordable items
        while i < n and a[i] * 10 <= money:
            i += 1

        # now all items before i are eligible candidates
        # but we need to pick among them those not yet taken
        # since we only move forward, we treat a[0:i] as available pool

        if i == 0:
            break

        # pick the largest index below i that we haven't consumed
        # but since we always consume from left, we simulate by taking a[i-1]
        i -= 1
        money -= a[i]
        ans += 1

    print(ans)

if __name__ == "__main__":
    solve()
```该代码对价格进行排序，以便资格检查成为与当前货币的简单阈值比较。 指针`i`扩展为包含满足 10x 规则的所有项目。 当无法添加更多项目时，我们会减少`i`加 1 并将其视为采用最近符合条件的项目。 

这里微妙的设计选择是我们从不显式维护一个单独的堆。 相反，我们重用排序后的数组和移动边界。 表达式`a[i] * 10 <= money`直接对规则进行编码。 

一个微妙的问题是终止。 如果`i == 0`，目前根据该规则，没有任何物品是可以负担得起的，因此该过程必须停止。 另一个微妙之处是，拿走一件物品后，之前符合条件的物品可能会变得无效，因此指针的移动必须始终反映当前的金钱。

 ## 边缘情况

 对于单个项目，例如`n = 1, m = 10, p = 1`，条件`10 >= 10`允许购买。 购买后，金钱变为 9，并且不再存在任何物品，因此答案为 1。 

对于紧密包装的同等价格，例如`m = 100, p = [10, 10, 10]`，只有第一次购买是可能的，因为一次购买后金钱变成90，而下一次又需要至少100。 算法在第一次选择后正确停止。 

对于许多小物品，例如`[1, 1, 1, ..., 1]`，每次购买仅略微减少金钱，并且多步仍满足资格条件。 该算法一直持续到金钱低于 10，此时不再考虑其他项目。
