 ## 解决方案
 ## 问题理解

 我们给定一个圆`n`卡片标记为`1`到`n`。 最初每张牌都是黑色的。 我们可以执行一系列操作，每次操作都会将一张黑卡永久涂成红色。 目标是准确地绘制`n − 1`卡片为红色，最后留下一张黑色卡片。 

一张卡`i`仅当存在另一张卡时才可以涂成红色`j`那仍然是黑色的，在环形步行道上`i`到`j`在任一方向上，都只有一条路径，除此之外，您还可以通过另外两张牌`i`和`j`。 在一个循环中，这个条件相当于说`i`和`j`沿循环距离为 3，这意味着它们之间正好有两个中间位置。 

因此，每次移动都受到沿圆周间隔三步的黑色对的存在的限制，并且我们可以删除这样一个有效对的任一端点。 

这个过程是动态的，因为每次删除后，黑牌集合都会缩小，未来的有效移动取决于剩下的结构。 除了可行性之外，我们还必须输出已删除索引的字典顺序最小序列。 

约束允许最多`10^5`测试用例总计`n`最多`5 × 10^5`。 这迫使`O(n)`或者`O(n log n)`最坏的情况是每次测试的构造，因为每次测试的任何二次都会立即超出时间限制。 

微妙的失败案例出现在小周期中。 为了`n = 4`，最初唯一有效的对是`(1, 4)`，但删除一个端点后，剩余的三个节点形成一个三角形，其中不再有两个节点的圆距离为 3，因此无法进行进一步的操作。 因此我们无法达到`n − 1`搬迁。 

为了`n = 5`，有效对最初存在，但经过几次删除后，结构崩溃得太快，我们在到达单个黑色节点之前再次陷入困境。 

为了`n = 6`，样本明确指出不可能。 这是关键的转折点，周期太小，无法在整个过程中维持“距离三见证”条件。 

为了`n = 7`，存在完整的有效序列，并且示例演示了这样一种排序。 

因此，关键现象是非常小的循环不能维持所需的结构不变性，而足够大的循环可以维持受控的剥离过程。 

## 方法

 暴力策略可以模拟这个过程。 在每一步中，我们扫描所有黑色节点`i`，并对于每个节点检查距离 3 处是否存在黑色节点。 如果是这样，我们尝试删除`i`，并递归地继续，直到剩下一个节点或我们陷入困境。 为了获得按字典顺序排列的最小输出，我们总是选择最小的有效值`i`在每一步。 

这种模拟的正确性很简单，但每一步可能需要扫描所有剩余节点，并且检查有效性也可能需要扫描圆圈。 这导致`O(n^2)`每个测试用例在最坏的情况下，这远远超出了限制`n`达到`10^5`。 

关键的观察结果是，唯一的结构要求是距离为 3 处至少存在一对活动节点。 此条件仅取决于该对的两个端点是否仍然是黑色，而不取决于全局连接性或远程结构。 一旦我们将圆重新解释为固定数组，我们就可以维持一种贪婪策略：始终删除当前参与至少一个有效距离三对的最小索引。 

这是有效的，因为删除有效对的端点可以在以下情况下保留足够的剩余结构：`n ≥ 7`。 直观上，循环足够大，删除一个顶点永远不会消除全局所有可能的距离三关系。

 ## 算法演练

 我们维护一个布尔数组`alive[i]`指示卡片是否仍为黑色。 我们还观察到一个节点`i`如果其两个距离为三的邻居中的至少一个当前是可移除的`(i+3)`或者`(i−3)`（模数`n`）还活着。 

### 算法步骤

 1.初始化所有节点`1..n`还活着。 

我们还准备了一个数据结构（队列或集合），可以快速按升序给出候选索引。 
2.对于每个节点`i`，检查距离 3 处是否有当前存活的有效伙伴。 如果是这样，请标记`i`最初可拆卸。 
3. 反复提取最小索引`i`目前是可拆卸的。 
4. 追加`i`到答案并将其标记为已死。 
5. 移除后`i`,更新节点状态`i−3`和`i+3`（如果它们存在并且还活着），因为它们形成有效配对的能力可能已经改变。 
6. 继续下去，直到只有一个节点保持活动状态。 
7. 如果在任何时候我们找不到可移动节点，但有多个节点仍然存活，则输出`NO`。 

微妙的部分是逐步保持“可移除”条件的正确性。 每次删除仅影响涉及距离为三的邻居的对，因此每步仅需要更新恒定数量的节点。 

### 为什么它有效

 不变量是每个可移动节点对应于现有的距离为三的黑色对。 删除这样一对端点并不能破坏足够大的循环中的所有未来可能性，因为每个节点在固定偏移处恰好有两个潜在伙伴。 为了`n ≥ 7`，该圆始终保留至少一个活动距离三对，直到只剩下一个节点。 贪心选择确保了字典最小性，因为我们总是选择满足不变式的最小索引，并且未来的选择不能迫使我们跳过它，因为跳过它只会延迟删除而不会解锁任何早期索引。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        n = int(input())
        
        if n <= 6:
            print("NO")
            continue
        
        alive = [True] * (n + 1)
        remaining = n
        
        # candidate set: indices that can be removed
        import heapq
        heap = []
        
        def valid(i):
            if not alive[i]:
                return False
            j1 = i + 3
            j2 = i - 3
            if j1 <= n and alive[j1]:
                return True
            if j2 >= 1 and alive[j2]:
                return True
            return False
        
        for i in range(1, n + 1):
            if valid(i):
                heap.append(i)
        heapq.heapify(heap)
        
        ans = []
        
        while remaining > 1:
            while heap and (not valid(heap[0])):
                heapq.heappop(heap)
            
            if not heap:
                break
            
            i = heapq.heappop(heap)
            if not valid(i):
                continue
            
            ans.append(i)
            alive[i] = False
            remaining -= 1
            
            for d in (-3, 3):
                j = i + d
                if 1 <= j <= n and alive[j]:
                    heapq.heappush(heap, j)
        
        if remaining != 1:
            print("NO")
        else:
            print("YES")
            print(*ans)

if __name__ == "__main__":
    solve()
```该实现跟踪活动节点并使用最小堆来始终提取当前有效的最小可移动索引。 这`valid`函数直接对距离三条件进行编码。 每次删除后，仅重新考虑距离为 3 的两个受影响的邻居，从而保持更新本地且高效。 

关键的实现细节是堆中的惰性删除模式：我们可能会存储过时的候选者，因此我们在使用它们之前重新检查有效性。 这避免了从堆中间进行昂贵的删除。

 ## 边缘情况

 对于`n = 4`，算法立即发现没有节点可以在第一次删除后保持有效距离-三个伙伴，因此输出`NO`而不进入循环。 该约束太严格，甚至无法维持单个一致的步骤。 

为了`n = 5`，即使存在初始有效对，重复删除也会很快破坏对称性，并且在只剩下一个节点之前堆变空，触发`NO`。 

为了`n = 6`，故障是结构性的而不是偶然的。 该循环永远不会在剩余节点之间保持足够的间隔，因此每个候选节点最终都会失去其合作伙伴，并且该过程会过早停止。 

为了`n ≥ 7`，贪婪过程在最后一步之前总是找到至少一个可移动节点，因为在整个剥离过程中距离三邻接仍然是可能的。
