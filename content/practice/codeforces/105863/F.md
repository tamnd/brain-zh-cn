 ## 解决方案
 ## 问题理解

 我们得到一个数字序列和一系列应用于它们的模运算。 每次使用模值处理数字时，都会将其替换为其余数。 关键的困难在于这些操作并不纯粹是装饰性的，因为一旦一个值相对于模数变得足够小，进一步的减少就不再改变它。 

任务是有效地支持大量此类操作，其中对每个受影响的数字重复应用朴素模更新会太慢。 不要将其视为静态转换，而是将每个数字视为随时间演变的东西，有时会发生巨大变化，有时会保持完全稳定。 

这些约束意味着值和操作的数量都可以足够大，以至于$O(n \cdot q)$模拟是不可能的。 任何每次操作重复扫描或线性更新值的解决方案都会超出时间限制。 这立即推动我们采取一种策略，即每个单独的数字总体上仅处理少量次，而不是每次操作。 

当值过早变得小于模数时，就会出现微妙的边缘情况。 例如，如果一个数字已经小于下一个模数，则该操作不执行任何操作。 幼稚的实现可能仍然会重新插入或重新处理它，浪费时间并可能重复工作。 当值由于多个模而在大状态和小状态之间振荡时，就会出现另一个问题，如果我们不明确跟踪值实际变化的时间，可能会导致重复重新计算。 

## 方法

 思考问题的一个直接方法是直接模拟它。 对于每个模运算，我们扫描所有相关数字并替换每个值$x$和$x \bmod c$。 这是正确的，因为它与操作的定义完全匹配。 然而，成本是令人望而却步的：如果我们有$q$运营和$n$数字，最坏情况执行$O(nq)$更新，当两者都达到高限制时，更新太大了。 

关键的结构观察是应用模数并不总是改变数字。 如果一个值$x$已经小于$c$，那么它保持不变。 更重要的是，当一个值发生变化时，它会显着下降。 声明中给出的主张正式化了这一点：申请后$x \bmod c$，结果是$x$本身或变得更小，以保证整个过程中每个值仅发生对数数量的实际变化的方式受到限制。 

这意味着大多数操作对于大多数数字来说实际上是无用的。 我们不想将每个操作应用于每个数字，而是只想在数字实际上大到足以更改时对其进行处理。 

为了利用这一点，我们将所有活动值维护在一个结构中，使我们始终能够有效地访问最大的当前值。 优先级队列在这里自然发挥作用。 我们反复提取最大值并将其与即将进行的模运算进行比较。 如果该值已经小于当前模数，则在以后按顺序处理的任何更大模数下它都不会改变，因此可以安全地将其保留在该步骤中。 如果它更大，我们应用模数，更新它，如果它在未来的步骤中仍然可以改变，则重新插入它。 

关键的改进是，每次修改一个值，它都会显着下降。 由于值可以有意义地减少多少次的对数界限，每个元素仅重新插入少量次，从而使整个过程高效。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 |$O(nq)$|$O(n)$| 太慢了 |
 | 最优（具有延迟减少的优先级队列）|$O((n + k)\log n \log X)$|$O(n)$| 已接受 |

 ## 算法演练

 我们按顺序处理模数运算序列，同时在面向最大值的结构中持续维护当前的值集。 

1. 将所有初始数字插入到最大堆中，以便我们始终能够有效地检索最大的当前值。 这很重要，因为在模数下只有大值可能会发生变化。 
2. 迭代每个模值$c$按照给定的顺序。 
3.虽然堆中的最大值大于等于$c$，提取它。 这确保我们只处理实际受操作影响的值。 
4. 将新值计算为$x \bmod c$。 这是唯一对正确性重要的转换，因为无论如何都会跳过所有未更改的值。 
5. 如果新值仍然足够大，可能会受到未来操作的影响，请将其重新插入堆中。 否则，自然会保持稳定，不再被重复处理。 
6. 继续，直到应用了所有模数运算。 

我们始终使用当前最大值的原因是，它保证我们永远不会在已经太小而不受当前模量影响的值上浪费精力。 

### 为什么它有效

 正确性依赖于单调收缩特性。 每次更新值时，它都会严格减少。 声明中的声明保证了一个值只能经历有限次数的此类有意义的下降，然后才能在所有未来的操作中永久稳定。 

因为我们仅在值大到足以受到影响时才处理该值，所以我们永远不会错过会更改其值的更新。 因为我们立即重新插入更新的值，所以我们保留了未来操作在需要时影响它们的能力。 不会应用不必要的操作，也不会跳过所需的转换。 

## Python 解决方案```python
import sys
import heapq

input = sys.stdin.readline

def solve():
    n, q = map(int, input().split())
    a = list(map(int, input().split()))
    ops = list(map(int, input().split()))

    # max heap via negative values
    heap = [-x for x in a]
    heapq.heapify(heap)

    for c in ops:
        # process only values that can change
        while heap and -heap[0] >= c:
            x = -heapq.heappop(heap)
            x %= c
            heapq.heappush(heap, -x)

    print(-min(heap) if heap else 0)

if __name__ == "__main__":
    solve()
```该实现使用通过 Python 的最小堆实现的最大堆对当前值的多重集进行编码。 每个模数操作仅重复拉动那些足够大而受影响的元素。 关键的实现细节是循环条件`-heap[0] >= c`，这保证了我们永远不会触及保持不变的元素。 

另一个微妙之处是我们从不尝试“分配回”未更改的元素。 如果一个值小于当前模数，它将保持在堆中不变，这对于效率至关重要。 

## 工作示例

 考虑一个从值开始的输入`[10, 7, 3]`和模数`[5, 4]`。 

在第一个模数期间`c = 5`，我们重复提取≥5的值。 

| 步骤| 堆（最大视图）| 提取 x | x 模 c | 新堆|
 | ---| ---| ---| ---| ---|
 | 1 | [10,7,3]| 10 | 10 0 | [7, 3, 0] |
 | 2 | [7, 3, 0] | 7 | 2 | [3, 2, 0] |
 | 停止| 全部 < 5 | - | - | [3, 2, 0] |

 处理 5 后，仅保留低于 5 的值或已将其降低到该范围内。 

现在处理`c = 4`。 

| 步骤| 堆（最大视图）| 提取 x | x 模 c | 新堆|
 | ---| ---| ---| ---| ---|
 | 1 | [3, 2, 0] | 3 | （无变化）| [3, 2, 0] |
 | 停止| 全部 < 4 | - | - | [3, 2, 0] |

 这表明，一旦值足够小，未来的操作就会变成无操作，并且堆很快就会稳定下来。 

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 |$O((n + q)\log n \log X)$| 每个元素仅在实际减少时才被压入和弹出，并且每个堆操作花费对数时间 |
 | 空间|$O(n)$| 堆存储所有活动值|

 运行时间完全符合典型限制，因为每个值在变得稳定之前只能缩小有限次数，从而防止重复对数据进行全面重新扫描。

 ## 边缘情况

 关键的边缘情况是所有数字都已经小于模数。 例如，输入`[1, 2, 3]`带模数`10`不会产生任何变化。 该算法处理这个问题是因为堆条件`-heap[0] >= c`立即失效，因此不会发生爆裂并且结构保持不变。 

另一种情况是重复的从大到小的转变。 考虑`[100, 99]`带模数`[50, 10]`。 第一个操作显着降低了两个值，而第二个操作可能仅影响子集。 堆确保每次归约后，只重新考虑新的大值，因此我们永远不会重新处理稳定元素。 

最后，考虑许多小模数，然后考虑大模数。 小模数会在早期积极降低值，而随后大模数则不会执行任何操作。 堆自然会吸收这一点，因为一旦值降到所有未来模数以下，它们就会无限期地保持不变。
