 ## 解决方案
 ## 问题理解

 我们有一排人，每个人都有一个绩效值。 每个人的奖金取决于他们的表现与左右两侧的直接邻居相比如何。 如果一个人在一侧表现优于邻居，他们会从该侧获得基本奖金贡献，等于固定值加上邻居的奖金。 如果他们的表现优于两个邻居，他们的最终奖金就是两个可能贡献中的最大值。 由于人们围坐成一圈，所以第一个人和最后一个人也是邻居。 

任务是计算每个人在这些依赖性规则下的奖金。 依赖性不是独立的：每个值都依赖于相邻的值，而相邻的值本身又依赖于其他值，形成一个循环上的约束系统。 

该结构意味着，如果不解决这些相互依赖性，就不可能进行直接评估。 每个位置都可能依赖于两个邻居，并且依赖的方向是通过性能值的比较动态确定的。 

如果人数很大，比如达到 200000 人，那么在最坏的情况下，任何重复重新计算值或天真地在数组中传播更新的解决方案都将变成二次方，因为每次更新都会触发循环的连锁反应。 这立即排除了没有记忆的迭代放松或简单递归。 

当性能值形成严格的增加或减少循环时，就会出现微妙的边缘情况。 在这种情况下，每个位置都依赖于一致的方向，并且幼稚的局部传播可能会错误地假设从左到右和从右到左贡献之间的独立性。 

例如，考虑三个有价值观的人围成一圈`[1, 2, 3]`。 值为 3 的人比两个邻居都好，因此他们的奖金取决于双方。 简单的从左到右传递可能会分配部分值，但无法正确传播从右到左的影响，从而产生不一致的结果。 

当所有值都相等时会出现另一种边缘情况，例如`[5, 5, 5, 5]`。 严格来说，没有人比任何邻居都好，因此所有奖金都应保持在基本水平。 任何假设每条边至少有一个严格不等方向的算法都会错误地传播更新。 

## 方法

 如果我们将每个定向比较解释为创建潜在的奖金贡献“流”，那么依赖性结构就会变得更加清晰。 对于任意相邻对，如果`p[i] > p[j]`， 然后`i`可以获得来自以下方面的贡献`j`。 这会在邻居之间创建有向边，但方向不是全局固定的，它取决于值的相对顺序。 

暴力方法会重复更新所有奖金，直到没有值发生变化。 在每次迭代中，每个位置都会检查其邻居，如果任一邻居提供了更好的贡献，则更新其值。 每次更新都取决于当前的邻居状态，因此值会围绕圆逐渐传播。 

这是有效的，因为系统是单调的：奖金只会增加。 然而，在最坏的情况下，每次更新可能只绕圆每次迭代传播一步。 和`n`元素，这可以采取`O(n)`迭代次数，以及每次迭代的成本`O(n)`, 生产`O(n^2)`复杂。 对于大量输入，这是不可行的。 

关键的观察结果是更新规则具有由值比较而不是索引引起的方向不对称。 一旦我们修复了由`p[i]`，我们只能处理沿着“下坡到上坡”方向的过渡。 一个人只能从性能严格较小的邻居那里获得贡献，而该贡献本身取决于更小的邻居。 

这将问题转化为计算图上最长的加权链，其中每个节点仅连接到具有较小值的邻居。 由于每个节点最多有两个邻居，因此在将边从较小值定向到较大值后，该结构就变成了有向无环图。 在此 DAG 上，我们可以使用记忆 DFS 或按降序迭代处理来计算值`p[i]`。 

通过降低性能对节点进行排序可确保当我们处理节点时，所有可能的贡献者（必须具有较小的值）都已被计算。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力（迭代松弛）| O(n²) | O(n) | 太慢了|
 | 最优（按值排序的 DP）| O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 我们根据相邻值的比较将循环依赖关系转换为有向关系，然后按性能降序计算 DP。 

1.构造一个按降序排序的索引数组`p[i]`。 这确保了在处理节点时，如果值较小的两个邻居都是有效的贡献者，则它们都已被处理。 排序是从依赖解析中消除循环的关键。 
2.初始化一个数组`f[i] = 0`对于所有职位。 这表示最初没有分配任何贡献。 
3. 按排序顺序迭代索引。 对于每个位置`i`，使用循环索引检查其左右邻居。 
4. 对于每个邻居`j`，检查是否`p[i] > p[j]`。 如果是这样，则该方向的候选人贡献为`B + f[j]`。 我们计算这个是因为`i`严格优于`j`， 所以`j`已经计算出的奖金可以向上扩展到`i`。 
5. 保持有效贡献中的最佳价值。 如果两个邻居都较小，则取两个候选贡献的最大值。 如果只有一项有效，则使用该一项。 如果没有一个有效，则该值保持为零。 
6. 分配`f[i]`到计算出的最佳值。 

我们以降序处理的原因是，每当`p[i] > p[j]`，值在`j`在我们处理之前必须已经完成`i`。 这可以防止重复更新并确保每个状态只计算一次。 

### 为什么它有效

 核心不变量是在处理节点时`i`, 每个邻居`j`这样`p[j] < p[i]`已经有了最终的价值`f[j]`。 由于贡献仅从严格较小的值流向严格较大的值，因此后续操作不会修改`f[j]`以某种方式影响`i`。 这使得依赖图在以下顺序下是非循环的`p`，并保证每个`f[i]`计算结果在分配时是最终的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, B = map(int, input().split())
    p = list(map(int, input().split()))
    
    order = sorted(range(n), key=lambda i: p[i], reverse=True)
    f = [0] * n
    
    for i in order:
        best = 0
        
        left = (i - 1) % n
        right = (i + 1) % n
        
        if p[i] > p[left]:
            best = max(best, B + f[left])
        if p[i] > p[right]:
            best = max(best, B + f[right])
        
        f[i] = best
    
    print(*f)

if __name__ == "__main__":
    solve()
```该代码首先读取人数和基本奖金值。 性能数组存储在`p`。 然后，我们通过降低性能对索引进行排序，以便在所有可能贡献较低性能的邻居之后计算出较高性能的位置。 

数组`f`存储计算的奖金。 对于按排序顺序的每个索引，我们使用模算术计算其两个循环邻居。 我们只考虑性能严格较小的邻居的贡献。 每个有效贡献都会增加`B`加上已经计算出的邻居值。 有效贡献的最大值成为该指数的最终奖金。 

一个微妙的点是使用模索引来强制循环邻接。 如果没有这个，第一个和最后一个元素的边界情况将需要特殊处理并且很容易出错。 

## 工作示例

 考虑`n = 3`,`B = 10`,`p = [1, 2, 3]`。 

我们按顺序处理`[2, 1, 0]`。 

| 我| p[i] | p[i] 左| 对| f[左] | f[右] | f[i] | f[i]
 | ---| ---| ---| ---| ---| ---| ---|
 | 2 | 3 | 1 | 0 | 0 | 0 | 0 |
 | 1 | 2 | 0 | 2 | 0 | 0 | 10 | 10
 | 0 | 1 | 2 | 1 | 0 | 10 | 10 20 | 20

 在索引 2 处，两个邻居都较小，但它们的`f`值对于传播尚无用处。 在索引 1 处，只有索引 0 较小，因此它增益`B + f[0] = 10`。 在索引 0 处，两个邻居在值排序上下文中都较大，但只有索引 1 较小，给出`10 + 10 = 20`。 

该跟踪显示了值如何随着性能顺序的增加而累积。 

现在考虑`n = 4`,`B = 5`,`p = [4, 1, 3, 2]`。 

| 我| p[i] | p[i] 左| 对| 有效贡献 | f[i] | f[i]
 | ---| ---| ---| ---| ---| ---|
 | 0 | 4 | 3 | 1 | 从 3,1 | 最大(5+f[3], 5+f[1]) |
 | 2 | 3 | 1 | 3 | 从 1,3 | 最大(5+f[1], 5+f[3]) |
 | 3 | 2 | 2 | 0 | 从 2 | 5+f[2] | 5+f[2]
 | 1 | 1 | 0 | 2 | 无 | 0 |

 这演示了不同节点如何根据本地比较而不是索引顺序来累积贡献。

 ## 边缘情况

 对于一个人的圈子来说，`n = 1`，由于循环索引，左右邻居都是其本身。 自从`p[i] > p[i]`为 false，不会触发任何贡献，结果是`0`。 该算法自然地处理这个问题，因为两个邻居检查都失败了。 

为了`p = [5, 5, 5, 5]`,每次比较`p[i] > p[j]`失败，所以每个节点分配`f[i] = 0`。 排序顺序无关紧要，因为没有激活任何转换，并且 DP 自始至终保持稳定在零。 

对于严格的单调循环，例如`[1, 2, 3, 4, 5]`，每个节点仅接收来自其较小邻居的贡献。 以降序处理可确保在评估节点时，所有较小的节点都已经有其最终贡献，因此链累积会干净地发生，而无需重新访问节点。
