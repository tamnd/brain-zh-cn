 ## 解决方案
 ## 问题理解

 该任务是关于规范 IPv6 地址表示形式。 输入是由冒号分隔的十六进制组组成的单个字符串。 在有效的扩展 IPv6 形式中，应该始终有八个组，每个组代表一个 16 位值，最多可写为四个十六进制数字。 然而，输入可以使用压缩形式，其中省略一个或多个连续组，由双冒号或由相邻分隔符引起的丢失段表示。 

目标是重建完整的规范形式。 这意味着我们必须解释分隔符，用零恢复丢失的组，并确保最终表示恰好有八个组件。 每个组件必须写为一个四字符的十六进制字符串，必要时用前导零填充。 

输入大小约束在语句中没有明确指出很大，但此类问题始终是为线性解析而设计的。 任何低效地重复拆分、插入或移动数组的解决方案仍然会通过，但不必要的二次行为是可以避免的。 

例如，当地址以压缩开始或结束时，会出现微妙的边缘情况`"::1"`或者`"2001::"`。 另一个极端情况是多个连续缺失部分，其中多个组隐含为零。 当压缩表示多个丢失片段时，假设每个空子字符串恰好对应于一个组的基于分割的天真方法会失败。 

例如，`"1::1"`两个之间应该变成七个零组，而不仅仅是一个零组。 

## 方法

 直接解释方法是用冒号字符分割字符串并将每个结果段视为一个组。 问题在于 IPv6 压缩打破了这一假设。 当两个冒号连续出现时，分割会产生一个空字符串，但该空字符串并不代表一个零组，它代表一整块缺失的组，其大小取决于地址中其他地方已经存在的段数。 

正确的推理是将地址视为具有固定的总共八个段。 分割后，我们计算有多少个非空组。 八与此计数之间的差是必须在空段出现的位置插入的零组的数量。 

一旦结构恢复，每个片段都会独立标准化。 每个组都必须从十六进制转换为整数，并用恰好四位数字打印，保留前导零。 

关键的观察结果是，在有效的 IPv6 地址中最多发生一次压缩，因此最多有一个区域需要扩展。 这保证了零块的单个插入点。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 直接分割，无需处理压缩| O(n) | O(n) | 错误 |
 | 零扩展拆分+重建 | O(n) | O(n) | 已接受 |

 ## 算法演练

 1. 按冒号字符将输入字符串拆分为原始段。 这会保留与压缩点相对应的空字符串。 分割数组的结构对丢失块发生的位置进行编码。 
2. 计算有多少个段是非空的。 这告诉我们地址中已经提供了多少个显式组。 
3. 通过从 8 中减去显式组的数量来确定缺少多少组。 该值是必须插入的零组的数量。 
4. 通过扫描分割段构建最终列表。 每当遇到空段时，它就代表压缩点。 将该位置替换为所需数量`"0000"`组。 
5. 对于每个非空段，通过用零向左填充来对其进行规范化，直到其长度变为四个字符。 这确保了每个 16 位块的统一表示。 
6. 重建后，确保结果恰好包含八组。 使用冒号将它们连接起来形成最终的 IPv6 地址。 

重要的结构思想是压缩形式仅删除零组的单个连续块，因此所有丢失的信息都是本地化的并且可以确定性地恢复。 

### 为什么它有效

 IPv6 压缩规则保证最多一个连续零组序列被双冒号替换。 这意味着分割表示恰好包含一个需要扩展的位置。 由于组的总数固定为八，因此缺失的长度是唯一确定的。 每个有效的重建都必须按顺序保留显式组，并且只有压缩区域是灵活的，因此用计算出的零数填充它会产生独特的规范形式。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def normalize(group: str) -> str:
    return group.zfill(4)

s = input().strip()
parts = s.split(':')

# count explicit groups
explicit = sum(1 for x in parts if x != '')

missing = 8 - explicit

res = []
i = 0
n = len(parts)

while i < n:
    if parts[i] == '':
        if missing > 0:
            res.extend(['0000'] * missing)
            missing = 0
        i += 1
    else:
        res.append(normalize(parts[i]))
        i += 1

print(':'.join(res))
```该解决方案的工作原理是首先解析分隔符的结构，然后在压缩点将缺失的零块恰好注入一次。 标准化步骤确保每组都是四个十六进制数字。 一个微妙的点是我们必须只消耗一次空段； 根据压缩的放置方式，可能会出现多个空条目，但只有其中之一代表扩展站点。 

## 工作示例

 考虑输入`"2001:db8::1"`。 

我们分裂成`["2001", "db8", "", "1"]`。 存在三个显式组，因此缺少五个组。 空条目会触发插入五个`"0000"`组。 最终序列变为：

 | 步骤| 行动| 结果 |
 | --- | --- | --- |
 | 1 | 阅读 2001 | [2001] |
 | 2 | 阅读 db8 | [2001, 0db8] |
 | 3 | 空块扩展| [2001, 0db8, 0000, 0000, 0000, 0000, 0000] |
 | 4 | 阅读 1 | [2001, 0db8, 0000, 0000, 0000, 0000, 0001] |

 最终输出是`"2001:0db8:0000:0000:0000:0000:0000:0001"`。 

现在考虑`"::1"`。 

分裂给出`["", "", "1"]`。 有一个明确的组，因此缺少七个。 空白区域扩展为七个零块，然后是最后的`"1"`已标准化。 

| 步骤| 行动| 结果 |
 | --- | --- | --- |
 | 1 | 空扩大| 七个 0000 块 |
 | 2 | 阅读 1 | 附加 0001 |

 结果变成`"0000:0000:0000:0000:0000:0000:0000:0001"`。 

这些痕迹确认空段始终充当整个压缩区域的占位符。

import sys, io

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    import sys as _sys
    input = _sys.stdin.readline

    s = input().strip()
    parts = s.split(':')

    explicit = sum(1 for x in parts if x != '')
    missing = 8 - explicit

    res = []
    i = 0
    while i < len(parts):
        if parts[i] == '':
            if missing > 0:
                res.extend(['0000'] * missing)
                missing = 0
            i += 1
        else:
            res.append(parts[i].zfill(4))
            i += 1

    return ':'.join(res)

# basic
assert run("1::1\n") == "0001:0000:0000:0000:0000:0000:0000:0001"
# already full
assert run("abcd:0:0:0:0:0:0:1\n") == "abcd:0000:0000:0000:0000:0000:0000:0001"
# leading compression
assert run("::\n") == "0000:0000:0000:0000:0000:0000:0000:0000"
# trailing compression
assert run("2001:db8::\n") == "2001:0db8:0000:0000:0000:0000:0000:0000"

 ## 边缘情况

 一种重要的边缘情况是当压缩出现在字符串的开头时，在分割后产生一个初始的空段。 在这种情况下，算法必须确保扩展发生在处理任何实际组之前。 构造逻辑会处理此问题，因为第一个空段会触发立即插入丢失的零块。 

另一种情况是输入已包含八个未进行任何压缩的显式组。 这里不存在空段，因此缺失计数变为零并且不会发生扩展。 该算法只是独立地对每个组进行标准化，无需修改即可保持正确性。
