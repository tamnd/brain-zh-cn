 ## 解决方案
 ## 问题理解

 给定一个由大写字母组成的字符串，我们希望形成尽可能多的由三个字母组成的不相交组。 每个有效组必须重新排列为单词“TAP”或单词“TUP”。 一旦某个字母在一组中使用，就不能在另一组中重复使用。 

更具体地重新定义任务，我们得到了多个字符集。 我们想要将其拆分为三元组，并且每个三元组必须恰好包含以下每种模式之一：一个 T、一个 P 和一个 A 或 U，具体取决于我们是形成“TAP”还是“TUP”。 因此，每个有效组恰好消耗一个 T 和一个 P，然后消耗一个 A 或一个 U。 

约束很小，字符串长度最多为 300。这立即告诉我们，即使是检查字母间分布的多种组合的解决方案也是可以接受的，但我们仍然应该以直接计数参数为目标，而不是搜索。 

一种简单的方法可能会尝试通过模拟所有可能的字母分组或排列来显式形成三元组。 然而，由于只有字母的存在计数很重要，任何基于排列的方法都会浪费精力探索等效状态。 问题的结构表明我们只关心每个相关字母有多少个：T、P、A 和 U。 

当一个字母丰富而另一个字母稀缺时，就会出现一种微妙的边缘情况。 例如，如果我们有很多 T 但很少有 P，则答案完全受 P 限制。另一种情况是 A 和 U 严重不平衡，因为它们竞争相同的 T-P 主干。 

一个小例子有助于阐明：

 如果字符串是`TAPU`，我们有 T=1，A=1，P=1，U=1。 我们可以制作一个花环。 

如果字符串是`TTTTPPPPAAAA`，我们有足够的T和P，但是A是限制因素，所以我们只形成TAP花环。 

如果字符串是`TPTPTPUU`，限制因素变为 T 或 P，并且 U 仅对 TUP 形成有贡献。 

关键的观察结果是，无论哪种类型，每个花环都会消耗一个 T 和一个 P。 唯一的决定是我们是否将 (T,P) 与 A 或 U 配对。 

## 方法

 暴力策略会尝试将每个字母分配为三元组，并测试每个三元组是否可以重新排列为“TAP”或“TUP”。 这很快就会变成组合包装问题。 即使使用回溯，我们也会探索最多 300 个字符的分区，并且分区数量的增长远远超出了可行的限制。 

蛮力的失败点在于它将字母视为可区分的对象，而实际上只计算物质。 一旦我们将状态压缩为频率，问题就变成了纯粹的算术问题。 

关键的见解是将问题分成两个独立的资源。 每个有效的花环需要一个 T 和一个 P，因此花环的数量不能超过 min(T, P)。 在确定我们可以使用多少个 (T,P) 对后，必须为每个这样的对分配一个 A 或 U。因此，我们有效地将 k 对分成两个桶，其中一个桶受 A 限制，另一个桶受 U 限制。 

如果我们决定形成 x 个 TAP 花环，我们需要 x 个 A，对于 TUP 花环我们需要 k - x 个 U。 因此 x 必须满足 x ≤ A 且 k - x ≤ U。这成为一个简单的可行性区间问题，我们在这些约束下最大化 k。 

我们不需要显式地尝试所有 k。 最大可能的 k 受 T 和 P 以及 A + U 组合的限制。 这是因为在固定 T 和 P 对后，每个花环恰好消耗 A ∪ U 中的一个。 

我们可以直接将答案计算为最大 k，这样就存在将 k 分为两部分的情况，即 A 和 U 容量。 这简化为检查 k ≤ min(T, P, A + U)，并确保 A 和 U 单独可以支持分割，只要两者相对于 k 没有过度约束，这总是可能的。 

这导致了一个直接的公式：我们取 k = min(T, P)。 然后我们检查使用 A 和 U 可以分配这 k 对中有多少，这仅受 A + U 的限制，但我们也必须确保不超过任何一边。 TUP 的最佳数量受 U 限制，因此最多有 U 个 TUP，同样最多有 A 个 TAP。 所以我们能做的最好的就是贪婪地分配：首先根据需要满足尽可能多的TUP或TAP； 由于两者都是对称的，因此总数为 k，但受到各个电源的限制。 

更清晰的解释是，我们总是取 k = min(T, P)，然后可用三元组的数量受到我们可以匹配到 k 个槽中的 A 和 U 数量的限制，这就是 min(k, A + U)，但由于 A + U 已经限制了可行性，所以 k 本身就是最终答案。 

因此，问题简化为计算字符并应用单个最小表达式。

 ## 算法演练

 1. 计算字符串中每个相关字母的出现次数，特别是 T、P、A 和 U。这是必要的，因为只有这些字母有助于有效的花环，所有其他字母都是不相关的噪音。 
2. 计算我们可以形成多少个(T,P)对，即min(T,P)。 每个花环必须恰好消耗一个，因此这是一个硬性上限。 
3. 计算使用 A 和 U 一起可以填充多少个“第三个槽”，即 A + U。这反映了每个花环必须恰好选择这些字母之一。 
4. 最终答案是可用的(T,P)对的数量和可用的A/U字母的数量之间的最小值。 这确保我们永远不会超出任何一个结构要求。 

### 为什么它有效

每个有效的花环必须恰好包含一个 T 和一个 P，因此任何解都不能超过 min(T, P)。 独立地，每个花环也必须恰好消耗 {A, U} 中的一个字母，因此我们总共不能超过 A + U。 由于这些约束是独立的，并且每个有效的花环仅使用每一侧的一个单位，因此可以通过将 T-P 槽与可用的 A 或 U 字母配对来实现这两个数量中最小值的任何分配。 除了这些之外，没有其他结构性限制。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

s = input().strip()

t = s.count('T')
p = s.count('P')
a = s.count('A')
u = s.count('U')

tp = min(t, p)
print(min(tp, a + u))
```该代码首先读取字符串并计算四个相关字符的出现次数。 变量`tp`代表我们可以构建的基本对的最大数量，每个基本对需要一个 T 和一个 P。最终答案受到 A 和 U 组合的总可用性的限制，因为每一对必须仅使用其中一个字母扩展为完整的三字母花环。 

一个常见的错误是试图分别决定形成多少个 TAP 和 TUP，而没有注意到只有 A 和 U 的总数才重要。 另一个微妙的问题是忘记未使用的 T 或 P 字母无法弥补缺失的 A/U，因为每个花环同时需要所有三个角色。

## 测试用例```python
import sys, io

def solve(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    s = input().strip()

    t = s.count('T')
    p = s.count('P')
    a = s.count('A')
    u = s.count('U')

    return str(min(min(t, p), a + u))

def run(inp: str) -> str:
    return solve(inp)

# provided samples (interpreted)

assert run("TULIPAN") == "1"
assert run("TAPTUPTAP") == "3"
assert run("TOP") == "0"

assert run("TPA") == "1", "exact single TAP"

 ## 边缘情况

 单字母或两个字母输入，例如`T`或者`TP`证明该算法正确返回零，因为 min(T, P) 为零，立即防止任何无效分组。 

例如，严重倾斜的分布，如许多 T 和 P，但没有 A 或 U`TTTTPPPP`，结果为 A + U = 0，即使存在 TP 对，也将答案强制为零。 计算自然地捕获了这一点，因为 min(tp, a + u) 变为零。 

A和U丰富但T或P之一缺失的情况，例如`AAAAUUUU`，也产生零，因为 min(T, P) 为零，表明同时需要两个结构组件。
