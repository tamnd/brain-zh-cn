 ## 解决方案
 ## 问题理解

 我们可以将这种情况重新解释如下。 

我们有一个起始整数强度`s`以及敌人名单。 每个敌人都由两个值描述：在战斗中生存所需的阈值力量和获胜后获得的奖励。 如果在任何时候我们选择的敌人的阈值严格大于或等于我们当前的实力，我们就会立即失败。 否则，我们会击败它并通过它的奖励来增强我们的力量。 

我们的目标不是模拟固定的序列，而是确定是否存在某种敌人的顺序，使我们能够击败所有敌人。 

输入大小很小：龙的数量最多约为 1000 条。这立即表明 O(n^2) 贪婪策略是可以接受的，而任何指数级的排列都是不可能的，因为 n! 即使 n 适中，订单也会爆炸。 

一个天真的错误是假设任何任意顺序都有效，或者仅按强度排序就足够了。 如果较弱的龙提供大量奖励并随后解锁更强的龙，则这种情况会失败。 

例如，考虑：

 输入：```
s = 2
(1, 99), (100, 0)
```如果我们先与强龙战斗，我们就会立即死亡。 如果我们先与弱龙战斗，我们就会变得强大到足以击败第二条龙。 因此顺序至关重要。 

另一个微妙的边缘情况是，所有龙最初都太强大，但其中一条几乎无法到达，并提供足够的奖励来解锁其余龙。 贪婪选择必须检测到这个增量解锁过程。 

## 方法

 暴力方法将尝试龙的所有排列并模拟战斗。 这是正确的但不可行：有 n！ 排列，甚至 n = 15 也已经变得太大了。 

关键的观察是，一旦我们强大到足以击败龙，如果它能帮助我们更快变得更强，就没有理由推迟与它的战斗。 拖延只会带来错失更早解锁更强大龙的机会的风险。 这导致了一种贪婪的策略：在每一步中，在我们当前可以击败的所有龙中选择一条。 

剩下的问题是首先选择哪条“可击败的”龙。 正确的选择是优先考虑力量要求较小的龙。 按所需强度对龙进行排序可确保每当我们向前扫描时，我们总是以结构化的方式遇到所有当前可到达的选项，并且我们以难度递增的顺序贪婪地消耗它们，同时不断扩大我们的范围。 

该算法变为：按阈值排序，然后重复获取所有可到达的龙，并以任何顺序消耗它们，通常使用线性扫描。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力（所有排列）| O(n!) | O(n) | 太慢了 |
 | 贪心+按强度排序 | O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 1. 将每条龙所需的力量和加值配对到一个元组列表中，并按所需的强度升序排列该列表。 排序确保我们总是先考虑较容易的龙，然后再考虑较难的龙，这与释放力量应该是渐进的理念是一致的。 
2. 维护一个变量`cur`初始化为起始强度。 这代表了流程中每个时刻的当前能力。 
3. 迭代排序列表，每当下一条龙所需的力量严格小于`cur`，将其标记为可失败。 这是有效的，因为排序保证了所有先前考虑​​的龙的要求也较弱或相同。 
4. 在所有当前可用的龙中，重复选择任何未访问过的龙，击败它，并将其加值添加到`cur`。 每次失败都可能解锁新的龙，因此可用性集必须动态更新。 
5. 继续此过程，直到当前关卡中不再有龙被击败为止。 然后继续在排序列表中向前扫描以查找新解锁的龙。 
6.处理完所有可能的扩展后，检查是否所有龙都被击败。 如果是，则输出“YES”，否则输出“NO”。 

### 为什么它有效

 正确性来自于单调可达性不变量：在任何时候，我们维护所需强度小于当前强度的龙的集合，并且我们确保如果存在任何这样的龙，我们最终会处理它。 由于击败龙只会增加力量，因此可达范围只会随着时间的推移而增长，而不会缩小。 这保证了如果存在完整排序，贪婪进程将发现有效序列，而不会跳过必要的中间解锁。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

n, s = map(int, input().split())
dragons = [tuple(map(int, input().split())) for _ in range(n)]

dragons.sort()  # sort by required strength

cur = s
i = 0
used = 0

visited = [False] * n

while True:
    progress = False

    while i < n and dragons[i][0] < cur:
        if not visited[i]:
            visited[i] = True
            cur += dragons[i][1]
            used += 1
            progress = True
        i += 1

    if not progress:
        break

print("YES" if used == n else "NO")
```代码首先根据龙的强度要求对龙进行排序，以便始终按升序检查可行性。 指针`i`确保我们只向前扫描一次，防止重复工作。 内循环收集所有当前可击败的龙并立即应用它们的奖励，这可能会解锁更强大的龙。 

一个微妙的细节是我们使用`< cur`而不是`<= cur`。 这符合实力相等不足以生存的法则。 

该算法依赖于每条龙最多被处理一次的事实，使得排序后整体复杂度呈线性。

 ## 边缘情况

 当多条龙具有相同的所需强度时，就会出现一种边缘情况。 该算法可以正确处理这个问题，因为一旦`cur`超过该阈值，所有这些都将同时可用，并且可以按任何顺序消耗，因为每个都独立地增加强度。 

另一种情况是，最佳策略需要延迟一条强大但可触及的龙，而选择一条奖金更高的较弱龙。 贪婪扫描自然会处理这个问题，因为所有当前可到达的龙都已处理，并且人为延迟其中任何一条龙都没有任何好处，因为奖金只会增加未来的可到达性。 

最后的边缘情况是初始强度已经超过所有阈值。 在这种情况下，算法只需在一次扫描中消耗所有内容，确认一旦所有内容均可访问，排序约束就不再相关。
