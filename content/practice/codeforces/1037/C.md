 ## 解决方案
 ## 问题理解

 我们有两个长度相等的二进制字符串。 将它们视为两排开关，每个位置要么打开，要么关闭。 目标是使用两种类型的操作将第一行转换为第二行：我们可以交换第一个字符串中的任意两个位置，支付的成本等于这些位置之间的距离，或者我们可以以单位成本翻转单个位。 

关键的困难在于交换不是免费的并且取决于距离，因此如果天真地完成，移动不匹配的位是昂贵的。 翻转很便宜，但它们会永久改变一点，因此决定通过移动或直接修正来修复不匹配是核心权衡。 

最多一百万个字符的限制排除了任何二次甚至二次的$O(n \log n)$具有重常数。 任何解决方案本质上都必须扫描字符串恒定次数并维护线性数据结构。 

一个天真的但不正确的想法是贪婪地从左到右修复不匹配，总是在本地翻转或交换。 例如，如果我们看到索引不匹配$i$，我们可能会立即翻转它。 这会失败，因为通过将一个位置的不匹配与稍后的不匹配配对并交换而不是支付两次翻转来解决一个位置的不匹配可能会更便宜。 

另一个微妙的失败来自于当存在两个不匹配的头寸时将掉期视为总是有利的。 如果我们在不考虑距离的情况下进行交换，我们可以假设存在完美的配对，但是将不匹配的位放在一起的成本可能会超过两次翻转，尤其是当不匹配的位置相距很远时。 

## 方法

 问题归结为仅理解不匹配的位置。 在任意索引处，如果$a[i] = b[i]$， 不要紧。 否则，我们会遇到一个不匹配，必须通过翻转或与另一个不匹配配对并交换来纠正。 

让我们将不匹配分为两种类型：$a[i] = 0, b[i] = 1$，以及位置$a[i] = 1, b[i] = 0$。 分别称它们为A型和B型。 一个 A 和一个 B 之间的交换可能会同时修复两个位置。 

蛮力的想法将尝试所有方法来配对不匹配并决定是交换还是翻转。 这很快就会变成组合，因为有可能$O(n)$不匹配，导致指数匹配可能性或至少在具有距离权重的线上出现最小成本匹配问题。 

关键的观察结果是该结构是一维的。 如果我们按顺序列出不匹配索引，则每当使用交换时，配对相反类型的相邻不匹配始终是最佳的。 任何非相邻配对都可以通过考虑交叉来改进或匹配：交换远处的不匹配会强制传递中间索引，与本地解决相比，这只会增加成本。 

这将问题简化为扫描字符串，维护最后一个未匹配的不匹配，并决定是否将其与当前的匹配或支付翻转。 

我们维护一个未解决的不匹配的堆栈式结构。 每当我们看到堆栈顶部的相反类型不匹配时，我们可以将它们与交换配对或将它们留作翻转。 掉期成本恰好是指数之间的距离，而两次翻转成本为 2。因此，对于每一对，我们采用这两个选项中的最小值。 

由于索引是按顺序处理的，因此可以立即知道距离。

 | --- | --- | --- | --- |
 | 暴力配对不匹配 | 指数| O(n) | 太慢了|
 | 相邻错配的贪婪配对 | O(n) | O(n) | 已接受 |

 ## 算法演练

 我们从左到右处理字符串并维护尚未解决的不匹配列表。 

1. 扫描索引$i$从 0 到$n-1$。 如果$a[i] = b[i]$，跳过它，因为它不需要任何操作。 
2. 如果没有未决的不匹配，则将当前不匹配索引及其类型推入结构中。 我们记住是 0 到 1 不匹配还是 1 到 0 不匹配，因为配对仅在相反类型之间才有意义。 
3. 如果存在相反类型的未决不匹配，我们考虑将其与当前索引进行匹配。 这将创建一个候选交换成本等于$i - j$， 在哪里$j$是先前的不匹配索引。 
4. 将交换选项与独立翻转两个头寸进行比较，其成本为 2。 将这两者中的最小值添加到答案中并清除未决的不匹配。 
5. 如果待处理的不匹配与当前不匹配的类型相同，我们无法将它们配对，因此我们推送当前的不匹配并让前一个不匹配等待。 
6. 处理完所有索引后，任何剩余的未匹配的不匹配必须通过翻转单独解决，每次翻转为答案贡献 1。 

### 为什么它有效

 该算法依赖于这样的事实：不匹配在一条线上形成两个有序序列。 任何最佳解决方案都可以进行转换，以便交换仅发生在索引顺序中连续的相反类型不匹配之间。 任何交叉配对都可以在不增加成本的情况下取消交叉，因为交换成本与距离成线性关系，并且取消交叉会减少总行驶距离。 一旦仅限于相邻配对，每个决策都变成本地决策：要么支付该对的交换成本，要么支付两次翻转，并且未来的决策都无法改善此选择。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = input().strip()
    b = input().strip()

    ans = 0

    prev_idx = -1
    prev_type = 0  # +1 for 0->1, -1 for 1->0

    for i in range(n):
        if a[i] == b[i]:
            continue

        cur_type = 1 if a[i] == '0' else -1

        if prev_idx == -1:
            prev_idx = i
            prev_type = cur_type
        else:
            if prev_type != cur_type:
                cost_swap = i - prev_idx
                ans += min(cost_swap, 2)
                prev_idx = -1
            else:
                ans += 1
                prev_idx = i
                prev_type = cur_type

    if prev_idx != -1:
        ans += 1

    print(ans)

if __name__ == "__main__":
    solve()
```该代码扫描一次字符串，一次最多只跟踪一个未解决的不匹配。 变量`prev_idx`存储最后一个不匹配的不匹配位置，同时`prev_type`存储其方向。 当出现兼容不匹配时，我们评估配对成本与翻转。 

一个微妙的点是，当两个不匹配属于同一类型时，我们无法形成交换，因此必须立即通过翻转来解决较早的不匹配。 这可以防止向前推进不兼容的状态并确保贪婪配对的正确性。 

最后，任何剩余的不匹配都将作为翻转支付。

 ## 边缘情况

 对于没有不匹配的字符串，该算法永远不会进入配对逻辑并立即返回零，因为从未创建挂起状态。 

对于单个不匹配，例如$a = 0, b = 1$，算法将其存储并到达末尾，然后应用单个翻转成本，匹配最佳行为。 

对于同一类型的多个连续不匹配，例如全部 0 到 1，该算法永远不会将它们配对并通过翻转来解决每个不匹配，这是最佳的，因为交换无法修复同向不匹配。
