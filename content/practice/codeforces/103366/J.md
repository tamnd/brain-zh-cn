 ## 解决方案
 ## 问题理解

 我们正在模拟缓存在最近最少使用策略下的行为，但我们不是直接模拟固定缓存大小，而是被问到一个相反的问题：在所有可能的缓存容量中，我们想要最小的容量，使得缓存在处理给定的请求序列时至少产生 K 次缓存命中。 

每个请求都需要一个块 ID。 如果该块已经在缓存中，则该请求被命中。 否则为未命中，该块将被加载到缓存中。 如果缓存已满，则当前存储的块中最近最少被请求的块将被逐出。 

困难在于，缓存行为取决于其容量，而容量变化会导致请求成为命中，因为命中会保留缓存中的项目并防止驱逐。 

约束为 N 至 100000，值至 10^9。 这立即排除了任何独立模拟每个候选容量的解决方案，因为即使是单个模拟也会花费 O(N)，并且尝试最多 N 的所有容量将导致 O(N^2)，这太慢了。 

命中和 LRU 结构之间的交互中出现了一个微妙的问题。 请求是否命中取决于该项目当时是否仍在缓存中，这取决于之前的整个历史记录。 一个天真的错误是假设频率本身很重要，但 LRU 是时间相关的，而不是频率相关的。 

另一种常见的失败情况是假设增加容量总是以简单的线性方式增加命中，而无需仔细验证用于搜索的谓词的单调性。 关键的单调属性不是“点击量增加得很好”，而是“如果一个容量有效，所有更大的容量也有效”。 

这是事实，因为较大的缓存在任何时候都不会比较小的缓存更早驱逐，因此它只能保留更多的项目。 

## 方法

 直接的暴力想法是固定容量 K，在整个序列上模拟 LRU 缓存，并计算命中次数。 我们将对从 1 到 N 的每个容量重复此操作，并选取至少产生所需命中次数的最小容量。 

可以使用哈希集和有序结构（通常是链表或有序字典）有效地模拟单个 LRU 缓存，为每个请求提供 O(1) 摊销更新。 这使得一次模拟的复杂度为 O(N)。 对 N 个容量重复此操作会产生 O(N^2)，在最坏的情况下约为 10^10 次操作，远远超出限制。 

关键的观察结果是，谓词“容量 C 产生至少 K 次命中”在 C 中是单调的。如果大小为 C 的缓存实现 K 次命中，那么容量为 C+1、C+2 等的任何缓存也将实现至少 K 次命中，因为添加容量只能延迟或防止驱逐。 

这种单调性允许对答案进行二分搜索。 缺失的部分是快速评估固定容量的有效方法。 我们使用支持检查成员资格、更新新近度、插入新元素和驱逐最近最少使用的元素的数据结构对每个候选者模拟一次 LRU。 

我们维护一个有序的结构，我们可以将访问的项目移动到最近的位置，并在需要时删除最近的项目。 使用哈希图加上双向链表或有序字典，在 O(1) 摊余时间内处理每个请求。 

这将完整的解决方案减少到 O(N log N)。

 ## 算法演练

 ## 算法演练

1. 我们首先定义一个函数，给定缓存容量C，模拟LRU过程并返回缓存命中数。 模拟使用的结构始终允许我们在恒定时间内识别最近使用最多和最近最少使用的项目。 这是至关重要的，因为驱逐决定取决于近期顺序，而不仅仅是存在。 
2. 对于每个容量，我们从左到右迭代请求序列。 对于每个请求的块，我们检查它当前是否在缓存中。 如果是，我们会计算一次点击并更新其新近度以将其标记为最近使用的。 如果不存在，我们将计算为未命中并将其插入缓存中。 
3. 每当我们插入一个新块并且缓存大小超过 C 时，我们就会删除最近最少使用的块。 此步骤强制执行容量约束并确保结构始终表示 LRU 规则下的有效缓存状态。 
4. 一旦我们可以评估单个容量，我们就在 C 上使用从 1 到 N 的二分搜索。对于每个中点，我们运行模拟并计算命中数。 
5. 如果模拟产生至少 K 个命中，我们尝试通过向左移动搜索范围来减少容量。 否则，我们会增加容量。 
6. 二分查找完成后，我们验证找到的容量是否确实产生至少 K 个命中。 如果没有，则没有容量可以满足要求，我们输出失败字符串。 

### 为什么它有效

 正确性依赖于缓存容量和可实现的命中数之间的单调关系。 增加容量永远不会以减少同一序列的总命中的方式将命中转变为未命中，因为保留在较小缓存中的任何项目也保证在相同的访问顺序下保留在较大缓存中，并且额外的空间只会防止驱逐。 这确保了谓词“容量 C 足够”是单调的，使得二分查找有效。 

模拟本身是正确的，因为它完全反映了 LRU 行为：成员资格检查确定命中或未命中，新近度更新保持顺序一致性，而逐出总是删除最近最少使用的元素，该元素由访问历史记录唯一确定。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

from collections import OrderedDict

def simulate(capacity, arr, K):
    if capacity == 0:
        return 0

    cache = OrderedDict()
    hits = 0

    for x in arr:
        if x in cache:
            hits += 1
            cache.move_to_end(x)
        else:
            cache[x] = None
            if len(cache) > capacity:
                cache.popitem(last=False)
        if hits >= K:
            # early exit possible
            pass

    return hits

def main():
    N, K = map(int, input().split())
    arr = list(map(int, input().split()))

    lo, hi = 1, N
    ans = -1

    while lo <= hi:
        mid = (lo + hi) // 2
        if simulate(mid, arr, K) >= K:
            ans = mid
            hi = mid - 1
        else:
            lo = mid + 1

    if ans == -1:
        print("cbddl")
    else:
        print(ans)

if __name__ == "__main__":
    main()
```模拟功能是使用Python的LRU直接实现`OrderedDict`，其中排序代表新近度。 每次访问都会将一个项目移动到末尾，使其成为最新的。 驱逐会删除第一个项目，这是最近的。 

二分查找部分完全依赖于可行性不随容量波动的事实。 一旦某个容量起作用，所有更大的容量也都起作用，所以我们只需要找到第一个成功的值。 

一个微妙的问题是容量 0 必须小心对待，尽管在这个问题中最小相关容量是 1。不过，防止退化情况可以保持实现的干净。

 ## 边缘情况

 当所有请求都不同时，就会出现一种边缘情况。 在这种情况下，没有缓存大小可以产生超出每个项目的第一次访问的命中，并且由于命中需要重复，所以答案一定是不可能的。 

另一种边缘情况是当 K 为 0 或非常小时。 由于立即重复，即使容量 1 也可能已经立即满足要求，因此二分搜索应该正确识别最小工作大小而不是超调。 

第三种情况是当重复访问非常接近时。 例如，在类似的序列中`1 2 1 2 1 2`，即使容量为 1 也可以产生一些命中，因为相同的元素在逐出模式强制退出之前返回。 该算法通过对新近度排序的完整模拟正确地捕获了这一点。
