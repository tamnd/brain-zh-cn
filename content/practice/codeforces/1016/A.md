 ## 解决方案
 ## 问题理解

 我们正在模拟如何通过无限笔记本进行书写，其中每个页面可以存储固定数量的名称。 在一系列的日子里，我们知道每天有多少个名字被写入，并且写入在几天内是连续的：如果一天结束时一页未满，第二天将继续填充它而不是开始新的一页。 

每当一页写满时，作者会立即翻到下一页并继续在那里书写。 我们需要计算的是，每天在处理当天的一批名称时，页面转换发生了多少次。 

关键的观察是，写作过程在所有日子里都是完全连续的。 我们有效地将一个单元流倒入大小为 m 的桶中，并且我们只关心在该流的每个段内从一个桶溢出到下一个桶的次数。 

约束表明 n 可以大到 200,000，每个 ai 可以达到 10^9。 在最坏的情况下，单独处理每个名称的简单模拟最多需要 10^14 次操作，这远远超出了 2 秒所允许的范围。 这迫使我们避免逐个元素模拟，而是使用分段算术。 

当一天恰好从页面边界开始时，就会出现微妙的边缘情况。 在这种情况下，当天的第一个名字会从新的一页开始，因此不应计算隐式结转。 另一种特殊情况是，当一天的贡献大到足以跨越多个整页时，需要在同一天内翻页，而不仅仅是在最后翻页。 

## 方法

 直接模拟将跟踪当前页面内的指针并一次减少一个名称的容量。 每次指针到达零时，我们都会增加页面计数器并将容量重置为 m。 虽然正确，但这种方法对每个写入的名称执行一个步骤，这意味着它可能会退化为处理数十亿次操作。 

问题的结构提出了不同的观点。 我们不是跟踪个人姓名，而是跟踪每天开始时当前页面的填充量。 设当前页剩余容量为rem。 当我们处理具有 ai 名称的一天时，仅当 ai 超过 rem 时才会发生第一页转换。 在使用 rem 名称之后，其他所有内容都被有效地拆分为大小为 m 的整页。 

这将问题转换为部分填充和完整整数除法的组合。 如果我们跨越当前页面边界，那么一天中的翻页数就会成为一个条件转换，加上之后消耗的整页数。 

这将每天的算术运算减少到 O(1)。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力模拟 | O(ai 总和) | O(1) | O(1) | 太慢了|
 | 算术跟踪 | O(n) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们维护一个变量 rem 表示当前页面中剩余多少个槽。

1. 初始化 rem = m 因为我们从一个可以容纳 m 个名称的新页面开始。 
2. 对于每一天i，读取ai 并设置turns = 0。 
3. 如果ai小于或等于rem，我们只需从rem中减去ai并相应地设置rem即可。 这一天没有翻页，因此翻页数仍为 0。发生这种情况是因为我们从未跨越页面边界。 
4. 如果 ai 大于 rem，我们首先消耗 rem 名称来完成当前页面。 我们将轮数增加 1，因为我们在这里恰好移动到新页面一次。 
5. 完成部分页面后，我们将 ai 减少 rem 并将 rem 重置为 0，因为我们现在在页面边界对齐。 
6.剩余的ai被分割成整页。 我们计算 Turns += ai // m，因为每个大小为 m 的完整块都会强制翻页。 
7. 如果 ai % m != 0，则更新 rem = m - (ai % m)，否则设置 rem = m。 这将恢复最后部分填充的页面上的剩余容量，或者如果我们恰好在边界处结束，则恢复新页面上的剩余容量。 
8. 当天的输出周转。 

每个步骤都反映了书写的物理过程：我们只有在实际跨越页面边界时才“接触”页面边界，并且所有内部消耗都被压缩为算术。 

### 为什么它有效

 不变的是 rem 始终正确地表示每天开始时当前页面上未使用的容量。 每次我们处理一天时，我们都会将其贡献分解为最多一个部分页面转换加上一些全页完成。 由于页边界仅在消耗 m 个单位时受到影响，因此将写入分组为大小为 m 的块可以保留精确的转换计数。 同一 ai 的两种不同分解不能产生不同数量的完整 m 大小的块，因此翻页次数是唯一确定的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    a = list(map(int, input().split()))

    rem = m
    res = []

    for x in a:
        turns = 0

        if x <= rem:
            rem -= x
        else:
            turns += 1
            x -= rem
            x //= 1  # no-op clarity: we are working with remaining full pages

            turns += x // m
            r = x % m

            if r == 0:
                rem = m
            else:
                rem = m - r

        res.append(str(turns))

    print(" ".join(res))

if __name__ == "__main__":
    solve()
```该解决方案保留了页面容量的运行剩余部分。 有条件的分割确保我们只在跨越边界时计算翻页次数。 整数除法批量处理所有整页转换，这消除了按名称模拟的需要。 

一个微妙的细节是在处理整页后正确更新 rem 。 如果最后一个段恰好在边界处结束，则 rem 必须重置为 m，否则它必须反映剩余的未使用空间。

 ## 边缘情况

 一个关键的边缘情况是一天恰好在页面边界处结束。 例如，如果 m = 4 并且一天写入 4 个名字，则页面干净地完成，第二天应该从新页面开始。 该算法通过在除法的余数为零时将 rem 重置为 m 来处理此问题，确保第二天不会错误地假定剩余容量。 

另一种情况是当ai远大于m时，例如ai = 10^9，m = 1。在这种情况下，每个名称都会导致翻页。 算术分解精确地产生了 ai 圈数，并且 rem 最终正确地保持为 1，从而保持了几天内的一致性。 

第三种微妙的情况是，一天正好填满剩余容量，然后在边界处结束。 在这种情况下，如果不显式地将 rem 重置为 m，第二天就会错误地认为页面仍然部分填充，从而导致丢失翻页。
