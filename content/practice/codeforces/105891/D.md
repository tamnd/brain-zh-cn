 ## 解决方案
 ## 问题理解

 我们从一个初始值开始$v$，并且我们得到$n$运营。 每个操作要么是某个实数的加法$x_i$，或乘以某个因子$x_i$。 我们可以任意重新排序这些操作。 选择顺序后，我们从以下位置开始依次应用它们$v$，产生一系列值$v_1, v_2, \dots, v_n$，其中每个$v_i$是应用后的值$i$-th 选择的操作。 

目标不仅仅是最大化最终值，而是最大化所有中间值的总和。 每次我们提前应用一个操作，它都会影响许多后来的前缀和，因为当前值被结转。 这使得排序变得至关重要：较早放置的操作会在最终目标中产生更多项。 

约束很小，有$n \le 30$，这立即表明任何指数或基于子集的推理都是可能可行的，而且也表明可能需要一个干净的结构性贪婪解决方案，而不是大量的 DP。 价值观$x_i$是浮点数，乘法因子总是略高于 1，而加法是正数，但可能很大。 弱乘数和潜在的大加法之间的这种不对称性是微妙之处的关键来源。 

一种天真的方法会尝试所有的操作排列，但是$30!$太大了。 如果没有结构，即使尝试模拟所有交错或子集也是不可行的。 

混合加法和乘法时会出现微妙的边缘情况。 例如，如果您有一个大的加法和一个小的乘法，则较早放置乘法可能会稍微增加所有后面的值，这可能比延迟一个大的加法更重要。 相反，即使乘数很小，早期的大量加法也可能占主导地位。 除非我们确定一致的全局排序原则，否则这种耦合会使幼稚的局部推理面临风险。 

## 方法

 蛮力的想法很简单：枚举每个排列$n$操作，模拟该过程，并计算前缀值的结果总和。 这是正确的，因为它直接遵循问题的定义。 然而，排列的数量随着$n!$，即使对于$n = 30$，使其完全无法使用。 

关键的观察是我们优化的不是最终值，而是前缀状态的总和。 每个操作不仅影响其应用的时刻，还影响所有未来的前缀。 这将问题转化为较早位置的权重严格更高的问题。 一旦我们认识到这种结构，我们就可以将操作分为两个行为类别：加法将值增加固定量，而乘法则缩放迄今为止累积的所有值。 

乘法出现较早时更有价值，因为它放大了所有未来的加法和初始值。 如果较早出现的加法可以进一步扩大其规模，则加法更有价值。 这创造了一个一致的方向：乘法应该用于放大尽可能多的“未来质量”，而加法应该放在从尽可能多的乘法器中受益的地方。 

由于乘数都大于1且比较接近1，因此它们的效果是单调且稳定的。 这导致了贪婪结构：首先优先考虑乘法，按递减因子排序，以便更强的乘法器放大更多后续结构，然后再进行加法，也按递减值排序，以便较大的加法从早期放大阶段受益更多。 

这将问题从全局调度优化转化为确定性排序规则。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O(n! \cdot n)$|$O(1)$| 太慢了 |
 | 最优贪心排序 |$O(n \log n)$|$O(1)$| 已接受 |

 ## 算法演练

 我们根据操作类型将操作分为两组。 乘法运算缩放当前值，而加法运算则移动它。 

然后，我们按照因子的降序对乘法运算进行排序。 这确保了最强的缩放效应尽早发生，从而最大化其对所有后续操作的影响。 

我们还按其值的降序对加法运算进行排序。 这确保了最大的加法收益更早出现，他们仍然能够从后来的乘法中受益。 

我们通过将所有乘法按排序顺序首先放置，然后按排序顺序放置所有加法来构造最终序列。 然后我们从初始值开始模拟这个过程$v$，累加所有中间值的总和。 

其原理在于影响力的传播方式。 乘法会影响其后的所有内容，缩放未来的加法和累积的基数。 因此，延迟乘法会严格减少它可以放大的结构量。 除了对当前价值的贡献之外，添加不会影响未来的运营，因此提前放置只会增加其对后续乘数的影响。 最终的排序将两种效果沿同一方向对齐，确保相邻元素的交换不会提高总体效果。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, v = input().split()
    n = int(n)
    v = float(v)

    adds = []
    mults = []

    for _ in range(n):
        op, x = input().split()
        x = float(x)
        if op == '+':
            adds.append(x)
        else:
            mults.append(x)

    mults.sort(reverse=True)
    adds.sort(reverse=True)

    ops = [('m', x) for x in mults] + [('a', x) for x in adds]

    cur = v
    ans = 0.0

    for t, x in ops:
        if t == 'm':
            cur *= x
        else:
            cur += x
        ans += cur

    print(f"{ans:.10f}")

if __name__ == "__main__":
    solve()
```该实现首先将输入操作划分为两个列表。 根据先前导出的贪心规则，在每种类型内独立完成排序。 模拟就很简单了：我们维护当前值并将每个前缀结果累加到最终答案中。 这种顺序保证了每次乘法都具有最大的杠杆作用，并且每次加法都可以从未来的扩展中获得尽可能多的好处。

 ## 边缘情况

 极端情况是仅进行添加的情况。 在这种情况下，按值排序仍然是正确的但无关紧要，因为所有排列都会产生相同的累积增长差异结构。 该算法在乘法器（不存在）之后仍然一致地处理它们，并且模拟正确地累积了前缀和。 

当只有乘法时，按递减因子排序是必要的。 例如，与$1.1$和$1.05$, 放置$1.05$首先减少更强乘数的放大，这会永久降低所有后面的前缀值。 排序顺序通过确保更早应用更强的乘数来避免这种损失。 

当将一个大的加法与许多小的乘法器混合时，该算法将所有乘法器放在第一位，这样即使是适度的加法也能从复合缩放中受益。 如果相反，加法将被放大不足，从而减少其对每个后续前缀和的贡献。
