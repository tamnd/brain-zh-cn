 ## 解决方案
 ## 问题理解

 我们得到一个固定的小写字符串，并且允许我们插入最多`k`其中任意位置的字符。 插入灵活：每次都可以自由选择位置和字符。 

执行这些插入后，我们查看结果字符串并计算特定模式出现的次数`"lose"`显示为连续的子字符串。 任务是选择插入以使该计数最大化。 

一个关键细节是我们计算的是子字符串，而不是子序列。 这意味着四个字符必须在最终字符串中彼此相邻并且拼写准确`"lose"`。 

约束大小，字符串长度可达`10^5`直至`10^5`插入，立即排除任何尝试显式模拟所有可能的插入位置或所有结果字符串的方法。 即使在每次插入后对所有窗口进行二次扫描也是不可能的。 解决方案对于输入的大小必须是线性的或接近线性的。 

一个微妙的点是插入可以改变邻接关系。 即使原始字符串没有类似的结构`"lose"`，插入可以完全创建它。 相反，贪婪的本地决策可能会意外地浪费插入，而这些插入可能会在以后形成更多的事件。 

一种重要的边缘情况是当字符串已经包含朝向模式的部分进展时。 

例如，考虑：

 输入：```
k = 1
s = "lxe"
```我们希望形成`"lose"`。 我们可以插入`'o'`和`'s'`和`'e'`围绕现有字符，但我们只有一个插入，因此我们甚至无法完成单个字符`"lose"`子串。 正确答案是`0`。 

这里的一个天真的错误是假设由于`"lose"`出现在某处，一次出现总是可以实现的。 这忽略了连续性要求和插入预算限制。 

另一个失败案例是过度计算重叠潜力：```
k = 0
s = "loselose"
```这已经包含 2 次出现，但是粗心的字符计数方法可能会通过重新组合字母来错误地估计更多字符，而如果没有插入，这是不可能的。 

## 方法

 蛮力的想法是将插入视为显式构建最终字符串的一部分。 我们会尝试所有方法来插入`k`字符，生成候选字符串，并统计出现的次数`"lose"`在每个。 即使我们只考虑在哪里插入字符，可能性的数量也会以组合方式增长，并且每个候选都需要线性扫描来计算子字符串。 即使对于非常小的输入，这也是完全不可行的。 

关键的观察结果是，插入不会创建复杂的结构，它们只会帮助我们“修复”或“完成”缺失的字符，以便构建固定模式的副本`"lose"`。 一旦我们确定了出现次数的目标，我们就可以考虑按顺序构建那么多副本。 

我们不是显式地构造最终字符串，而是模拟我们可以贪婪地“生成”该模式多少次`"lose"`从左到右扫描原始字符串。 每当当前位置缺少下一个所需字符时，我们都会使用一次插入来提供它。 一旦我们用完插入，我们就无法再完成进一步的复制。 

这将问题减少到重复匹配`"lose"`针对流，我们可以使用原始字符串中的字符，也可以花费一次插入来构造所需的字符。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力构造字符串 | k 的指数 | O(n + k) | 太慢了 |
 | 对模式的贪心模拟 | O(n + k) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们将这个过程视为重复构建模式`"lose"`从左到右使用原始字符串中的字符并在必要时插入。 

1. 初始化一个指针`i`在字符串的开头和一个计数器`k`对于剩余的插入。 我们还维护一个计数器，记录有多少个已满`"lose"`我们已经成功构建了模式。 
2. 尝试构造一个出现的`"lose"`按顺序处理其字符。 

对于中的每个字符`"lose"`，我们检查输入字符串中的当前字符是否匹配。 如果匹配，我们通过前进来消耗它`i`。 如果不匹配，我们会花费一次插入来创建所需的字符。 此插入不会提前`i`，因为它是人工放置的。 

这里的原因是我们将原始字符串对齐到支持该模式的序列中，并且插入充当填充符。 
3. 如果在任何时候我们需要插入但没有剩余插入，则构建过程会立即停止，因为无法完成更多完整模式。 
4.一次一次`"lose"`完成后，我们增加答案并立即开始使用相同的过程构建下一个答案。 
5. 我们继续，直到耗尽字符串并且无法继续或用完插入。 

### 为什么它有效

 该过程在源字符串和重复出现的字符串之间强制执行贪婪对齐`"lose"`。 原始字符串中的每个字符都按顺序使用，并且仅在无法匹配下一个所需字符时才进行插入。 这保证了不会在稍后的机会上浪费插入，而这些机会本来可以在不同的模式中较早地使用，因为考虑到子串形成的固定顺序结构，任何重新排序都不会提高未来匹配的可行性。 

每完成一个`"lose"`对应于将该模式以最小成本嵌入到不断演变的字符串中，因此在插入预算下最大化已完成嵌入的数量会产生子字符串出现的最佳数量。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    k = int(input().strip())
    s = input().strip()
    target = "lose"
    
    i = 0
    n = len(s)
    ans = 0
    
    while True:
        used = k
        j = i
        
        ok = True
        for c in target:
            if j < n and s[j] == c:
                j += 1
            else:
                if used == 0:
                    ok = False
                    break
                used -= 1
        
        if not ok:
            break
        
        k = used
        i = j
        ans += 1
    
    print(ans)

if __name__ == "__main__":
    solve()
```该实现维护一个指针`i`到原始字符串和剩余的插入预算`k`。 每次迭代都尝试构建一个`"lose"`堵塞。 在循环内结束`"lose"`，我们要么消耗一个匹配的字符`s`或者在匹配缺失时花费一次插入。 

一个微妙的点是插入不会移动指针`s`，因为插入的字符独立于原始字符串。 只有匹配才会使指针前进。 一旦构建了完整的模式，更新的指针和减少的预算都将被带入下一次迭代。 

一旦无法再完成完整模式，循环就会停止。

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 | O(n + ans) | O(n + ans) | 中的每个角色`s`最多消耗一次，每次模式尝试最多处理4步|
 | 空间| O(1) | O(1) | 仅使用了少量计数器和指针|

 约束允许最多`10^5`字符和插入，因此在时间限制内，以每步恒定工作量对字符串进行线性扫描很容易足够快。

 ## 边缘情况

 关键的边缘情况是字符串根本不包含有用的字符。 

输入：```
k = 4
s = "xxxx"
```该算法尝试构建`"lose"`仅使用插入。 每一步都会消耗一次插入，并且因为`k`等于模式长度，恰好形成一次出现。 指针在`s`从不前进，这是一致的，因为没有字符被重用。 

另一种情况是字符串已经部分匹配模式但未对齐：```
k = 0
s = "losxlose"
```前三个字符完美对齐，但第四个字符打破了这种模式。 由于不允许插入，因此构造在该点失败，并且仅计算有效的完整出现次数。 当在没有可用插入的情况下发生不匹配时，算法会准确停止，从而防止过度计数无效的部分段。
