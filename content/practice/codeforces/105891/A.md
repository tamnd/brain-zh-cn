 ## 解决方案
 ## 问题理解

 我们得到一个分为 n 个连续段的条带。 每个片段已经有一个初始颜色，但是除了作为结构的来源之外，起始配置对于最终目标并不重要。 我们可以执行的实际操作是选择一个连续的间隔并将其完全重新绘制为单一颜色 c。 这样做会花费该颜色的固定价格 wc ，再加上等于我们重新绘制的间隔长度的额外成本。 

任务是独立计算从 1 到 n 的每种颜色 c 重新绘制整个条带所需的最小总成本，以便每个片段最终都是颜色 c。 

约束 n 高达 2 × 10^5 立即排除任何显式尝试所有间隔的解决方案。 可能的重绘操作数量为 O(n^2)，任何隐式考虑所有段分区的方法都会失败。 我们需要一种结构来避免重复子结构的重新计算成本。 

一个微妙的点是，初始颜色的重要性仅在于它们如何允许合并相同目标颜色的片段，而无需支付额外的重绘成本。 当初始配置已经包含目标颜色的有用连续块时，忽略此结构并假设我们直接在一个操作中重新绘制所有内容的幼稚方法可能是错误的。 

一种典型的故障情况是，除了单个不匹配之外，条带已经几乎均匀。 例如，如果数组是`[1, 1, 2, 1, 1]`我们想要颜色`1`，一个简单的解决方案可能会重新绘制整个范围的成本`w1 + 5`，但最佳解决方案仅重新绘制中间块`[3,3]`和成本`w1 + 1`。 关键思想是，我们可以通过将已与目标颜色匹配的段视为有用操作之间的分隔符来选择性地避免重新绘制它们。 

## 方法

 强力策略是固定目标颜色 c 并尝试将数组划分为重绘操作的所有可能方法。 每个操作都会选择一个区间，并且由于区间可以以复杂的方式重叠，因此状态空间呈指数分布。 即使简化为“选择不相交间隔的子集”，也已经导致每个状态发生 O(n^2) 次转换，如果通过段上的动态编程实现，在最坏的情况下，总复杂度约为 O(n^3)。 

一旦我们从目标颜色的角度重新解释问题，结构就变得易于管理。 假设我们希望最终的数组完全是颜色 c。 初始数组中已经具有颜色 c 的任何段都不需要重新绘制，更重要的是，它充当自然分隔符：我们永远不需要在这些位置上重新绘制，因为这样做只会增加成本而没有好处。 

这将数组分割为最大的尚未颜色 c 的段块。 在每个这样的块内，我们必须执行重绘操作，将所有内容都转换为 c。 现在关键的观察是，在一个块内，成本仅取决于长度和操作数量，并且每个操作都有固定的开销 wc 加上长度的线性成本。 这种结构允许贪婪或区间 DP 简化，其中我们基本上独立地处理每个非 c 块并对最优成本求和。 

最佳解决方案减少为每种颜色扫描一次阵列，累积来自还不是该颜色的连续段的贡献。 每个这样的段根据其长度贡献一个成本，加上每个段块的一次 wc 激活。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 间隔暴力破解/DP | O(n^3) 或更糟 | O(n^2) | O(n^2) | 太慢了 |
 | 每色线性扫描 | 最坏情况总计 O(n^2)，通过重用进行优化 | O(1) 额外 | 已接受 |

 ## 算法演练

 我们固定目标颜色 c 并计算将整个数组转换为 c 的最小成本。 

1. 我们从左到右扫描数组并识别当前颜色不是 c 的最大连续段。 这些片段代表必须在某个时刻重新绘制的区域。 当在一次操作中重新绘制段已经是最佳选择时，我们永远不需要在段内进行拆分，因为拆分会引入额外的 wc 成本，而不会减少长度成本。 
2. 对于长度为 len 的每个这样的段，我们观察到在一次操作中重新绘制它的成本为 wc + len。 如果我们将其拆分为多个操作，每个操作都会增加另一个wc，因此拆分只能增加总成本。 这使得“每个段一次操作”成为最佳选择。 
3.我们将所有非c段的总长度累加为sumLen，并计算存在多少个这样的段作为blocksCnt。 
4. 颜色 c 的总成本变为 wc ×blocksCnt + sumLen。 
5. 我们对每种颜色 c 独立重复此过程并输出所有结果。 

这种方法有效的关键原因是，每当目标颜色已经存在时，初始数组就会将工作划分为独立的组件。 这些位置永远不值得重新绘制，因此它们充当固定边界。 在每个组件内部，线性成本占主导地位，合并操作总是有益的。 

### 为什么它有效

 该算法依赖于这样一个不变量：尚未等于 c 的每个最大连续位置间隔必须被至少一个重绘操作完全覆盖，并且用多个操作覆盖它会严格增加成本，因为每个操作都会添加额外的 wc 而不会减少总绘制长度。 因此，最佳解决方案总是为每个最大间隔选择一个操作，并且这些间隔的分解由初始配置固定，从而使解决方案具有确定性和最佳性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = list(map(int, input().split()))
    w = list(map(int, input().split()))

    # precompute positions of each color
    pos = [[] for _ in range(n + 1)]
    for i, c in enumerate(a):
        pos[c].append(i)

    res = [0] * (n + 1)

    for c in range(1, n + 1):
        if not pos[c]:
            # if color doesn't exist, we must repaint everything in one segment
            res[c] = w[c - 1] + n
            continue

        cost = 0
        prev = -2
        segments = 0

        for i in pos[c]:
            if i != prev + 1:
                segments += 1
            prev = i

        # sum lengths of non-c positions
        # total non-c length = n - len(pos[c])
        cost = w[c - 1] * segments + (n - len(pos[c]))
        res[c] = cost

    print(*res[1:])

if __name__ == "__main__":
    solve()
```代码首先对每种颜色的位置进行分组，以便我们可以快速检测原始数组中该颜色的连续运行。 对于每种颜色 c，我们迭代其出现次数并计算存在多少个连续块。 只要当前位置与前一个位置不相邻，就会开始新的块。 

必须重新绘制的总工作量就是不等于 c 的索引数。 计算公式为 n 减去 c 的频率。 每个连续的块只贡献一个 wc 成本，因此我们将 wc 乘以块的数量。 

这里的一个常见实施错误是混淆频率和块计数。 单独的频率是不够的，因为分散出现的颜色不会形成单个自由区域； 只有连续发生才重要。

 ### 示例 1

 输入：```
n = 5
a = [1, 3, 2, 3, 5]
w = [5, 5, 5, 5, 5]
```我们计算 c = 3。 

3的位置是`[1, 3]`（0-索引）。 这些不相邻，所以我们有 2 个街区。 非 3 位置总数为 3。 

| 步骤| c | 的位置 块数 | 非c长度| 成本|
 | ---| ---| ---| ---| ---|
 | 初始化| [1, 3] | 0 | 0 | 0 |
 | 扫描| [1, 3] | 2 | 3 | 0 + 2×5 + 3 |

 最终成本为13。 

这表明，即使目标颜色的稀疏出现也不会合并成本，除非它们是连续的。 

### 示例 2

 输入：```
n = 3
a = [1, 1, 1]
w = [1, 10, 2]
```对于 c = 1，位置为`[0, 1, 2]`，一个块，并且没有非 c 位置。 

| 步骤| 职位 | 积木| 非c | 成本|
 | ---| ---| ---| ---| ---|
 | 扫描| [0,1,2]| 1 | 0 | 1×1 |

 成本为1。 

这证实了如果数组的目标颜色已经是统一的，我们只需支付一次操作成本，并且不需要重绘长度成本。

 ## 边缘情况

 一种边缘情况是数组中根本没有出现某种颜色。 在这种情况下，没有空闲位置，因此整个数组是一个重绘块。 该算法正确地计算了 0 次出现，将块计数设置为 1，并对 wc + n 进行计费。 

另一种边缘情况是所有位置都已经是目标颜色。 这里，位置列表形成一个连续的块，并且非 c 长度为零，因此成本压缩为单个 wc。 该算法通过从不引入额外的块来避免过度计数。 

第三种情况是高度分散的发生模式，例如`[c, x, c, x, c, x]`。 邻接检查确保由间隙分隔的每个出现形成一个新块，这与每个间隙在任何最佳策略中强制执行新的重绘操作的​​事实相匹配。
