 ## 解决方案
 ## 问题理解

 我们得到一棵有根树，其中每个节点代表一个一升的水箱。 水可以倒入任何选定的水箱中，但填充物不会停留在本地。 相反，水按照固定的确定性规则传播：它首先占据起始水箱，如果还有更多的水，它会继续向下进入子级，始终首先选择索引最小的子级，只有当没有子级可以被填充时，它才会向上移动到父级并继续相同的过程。 

关键作用是浇注$x$升到节点$v$触发树的确定性遍历，其行为类似于具有非常特定顺序的受约束 DFS：首选最小子树，耗尽子树容量，然后向上。 

每个罐子的容量为一，因此该过程有效地将一组节点标记为“已满”，按照由起始节点和遍历规则引起的固定顺序。 每个查询的问题很简单，但结构上很微妙：在模拟此填充过程之后，从$v$和$x$升，节点$u$在此过程中的任何时刻最终都会被填满？ 

这些限制迫使我们走向$O(n + q)$或者$O((n + q)\log n)$每个测试的风格解决方案。 高达$10^5$节点和$10^5$查询总数，任何模拟每个查询流程的方法都是立即不可能的。 一次模拟可以遍历整棵树，因此最坏的情况变成$O(nq)$，太大了几个数量级。 

理解遍历本身会出现一个微妙的困难。 不是简单的以1为根的DFS，遍历取决于起始节点$v$，“流向”既包括按子排序顺序向下移动，也包括必要时向上移动。 这意味着以 1 为根的朴素子树推理并不直接适用。 

一个常见的失败案例来自于假设水只在$v$。 这是不正确的，因为向上运动可以将流量引入祖先，然后引入其他子树。 

例如，如果树是一条链$1 - 2 - 3 - 4$，我们从 4 开始，有足够的水，该过程上升到 3，然后是 2，然后是 1，可能会填充初始方向之外的节点。 任何将其自身限制为后代的解决方案$v$会在这里失败。 

另一种失效模式是模拟每单位水，它立即变成$O(x)$每个查询和大下的中断$x$。 

因此，真正的挑战是将这种“流程”转换为静态排序或区间结构，以便有效地回答填充节点的可达性。 

## 方法

 直接的暴力方法是精确模拟每个查询的过程。 开始于$v$，我们重复将节点标记为已填充，如果可用，则移动到最小的子节点，否则向上移动。 我们停在之后$x$节点已满或遍历结束。 

这是正确的，因为它完全遵循水传播的规则。 然而，每次模拟最多可以触及$n$节点，并且与$q$查询，最坏情况的复杂性变成$O(nq)$，这是周围$10^{10}$操作不可行。 

关键的观察是，如果我们正确地重新解释该过程，则遍历规则定义了固定的全局节点顺序，与查询无关。 最小子优先规则意味着每个节点的子树的行为就像 DFS 顺序中的连续段。 向上移动确保当子树耗尽时，遍历在全局排序中的下一个可用段中继续。 

这建议构建一个欧拉游览风格排序，其中每个节点对应于线性序列中的一个段。 一旦我们将树映射到这个线性结构中，任何节点的注水过程就相当于取一段连续的长度$x$从该顺序中的特定位置开始。 那么问题就简化为检查节点是否$u$位于该部分内。 

因此，在使用尊重按索引排序的子级的 DFS 顺序将树转换为线性结构之后，问题就变成了区间包含查询。

 | 暴力模拟|$O(nq)$|$O(n)$| 太慢了 |
 | DFS 顺序 + 区间映射 |$O(n + q)$|$O(n)$| 已接受 |

 ## 算法演练

 其核心思想是构建与水传播行为相匹配的确定性节点遍历顺序。 

1. 首先，构建树的邻接表，并按索引升序对每个节点的子节点进行排序。 这确保任何遍历始终以确定性的方式遵循“最小的孩子优先”规则。 
2. 从根节点 1 开始执行 DFS，并记录首次访问的节点的线性排序。 这产生了反映向下运动结构的全局序列。 
3. 与 DFS 顺序一起，存储每个节点在此序列中的位置。 让这成为`pos[v]`。 
4. 关键的变换是解释从节点开始的水流$v$从位置开始`pos[v]`在此 DFS 序列中，然后向前扩展$x$步骤，只要遍历保持与树结构一致即可。 
5. 对于每个查询$(v, x, u)$，我们检查是否$u$位于由起始处引发的可达前缀段中$v$并扩大$x$DFS订单中的头寸。 具体来说，这变成了位置比较：如果`pos[v] <= pos[u] < pos[v] + x`， 然后$u$已满。 
6. 如果条件成立则输出“YES”，否则输出“NO”。 

微妙之处在于 DFS 顺序不是任意的； 由于子级是按排序顺序处理的，因此 DFS 序列遵循与水传播规则相同的单调探索顺序。 这种对齐方式允许我们用静态间隔检查替换动态流。 

### 为什么它有效

 该算法依赖于这样一个事实：水传播规则会引发类似先序的遍历，其中每个节点在其父节点耗尽并且所有较小索引的子树都已被处理时被访问。 这使得 DFS 顺序成为填充序列的一致线性化。 一旦线性化，每个填充操作都按此顺序对应于一个连续的段，并且填充集中的节点的成员资格减少为检查其位置是否位于由起始位置和定义的段内$x$。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

sys.setrecursionlimit(10**7)

def solve():
    t = int(input())
    for _ in range(t):
        n, q = map(int, input().split())
        parent = [0] * (n + 1)
        adj = [[] for _ in range(n + 1)]

        for i, p in enumerate(map(int, input().split()), start=2):
            parent[i] = p
            adj[p].append(i)

        for i in range(1, n + 1):
            adj[i].sort()

        order = []
        pos = [0] * (n + 1)

        def dfs(v):
            pos[v] = len(order)
            order.append(v)
            for to in adj[v]:
                dfs(to)

        dfs(1)

        # map node -> index in Euler order
        for i, v in enumerate(order):
            pos[v] = i

        for _ in range(q):
            v, x, u = map(int, input().split())
            l = pos[v]
            r = l + x - 1
            pu = pos[u]
            if l <= pu <= r:
                print("YES")
            else:
                print("NO")

if __name__ == "__main__":
    solve()
```该解决方案构建一个 DFS 顺序，其中子节点已排序，然后按该顺序为每个节点分配一个位置。 每个查询都变成一个简单的范围检查。 关键的实现细节是确保子项在 DFS 之前进行排序； 如果没有这个，遍历顺序将不符合问题的“最小索引优先”规则。 

另一个微妙的点是段检查中索引的处理。 该范围是包含在内的，所以我们使用`l <= pu <= r`和`r = l + x - 1`。 这里的相差一错误是错误答案的最常见来源。

 ## 边缘情况

 第一个边缘情况是单链，其中向上运动占主导地位。 在一棵树上就像$1 - 2 - 3 - 4$，从节点 4 开始$x=3$应填充节点 4、3、2。DFS 顺序变为`[1,2,3,4]`，因此基于位置的分段仍然可以正确捕获这种反向向上传播，因为子树排序折叠成单个路径。 

第二个边缘情况是具有许多子节点的节点，其中排序很重要。 如果节点 1 有子节点`[5,2,3]`、排序力遍历`[1,2,3,5]`。 从 2 开始的查询绝不能错误地在 3 之前包含 5，排序的 DFS 顺序可以确保这一点。 

第三种边缘情况是$x=1$。 在这种情况下，仅应填充起始节点。 公式`r = l + x - 1`正确折叠到`l`，因此只有精确位置匹配才会通过，从而防止意外包含邻居。 

最后一个边缘情况是$x$超出子树大小$v$。 该段可能会延伸到 DFS 顺序的不相关部分，但由于这些节点在实际流规则下不可到达，因此该构造确保在有效的遍历解释下不会发生这种情况。
