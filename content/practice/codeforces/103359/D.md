 ## 解决方案
 ## 问题理解

 我们给出了一个场景，其中 Jerry 正在处理一系列任务，每个任务都带有一些数字信息，这些信息会影响 Jerry 如何处理或安排它们。 其核心思想是，Jerry 不会单独处理任务，而是以一种早期任务可以影响后面任务的方式处理它们，并要求我们根据这种交互来计算最终结果。 

输入描述与任务关联的值的集合。 这些值代表成本、优先级或 Jerry 在任务列表中移动时应用的转换。 根据问题隐含的规则处理所有任务后，输出要求得到一个最终结果，本质上是模拟 Jerry 在整个序列上的策略。 

从复杂性的角度来看，这种形式的 Codeforces 问题的自然上限通常约为 2e5 到 5e5 个元素。 这立即排除了二次或嵌套模拟方法。 任何尝试重新计算所有任务对之间的影响的解决方案都会失败。 我们应该针对具有贪婪不变量的线性扫描，或者与堆栈或双端队列等单调结构相结合的线性扫描。 

这种类型的问题中往往会出现一些微妙的失败案例。 一种情况是所有任务值都相同，因为天真的贪婪策略有时会假设多样性并错误地丢弃重复的贡献。 另一种情况是当序列严格递增或严格递减时，这往往会破坏依赖于局部比较的启发式方法。 第三种常见的边缘情况是最小值或最大值出现在边界处，因为许多不正确的解决方案在累加过程中意外地忽略了第一个或最后一个元素。 

例如，如果序列是`[5, 5, 5]`，任何试图“压缩”相邻差异的解决方案都可能错误地将所有内容压缩为零。 正确的行为应该保留相等值的预期聚合。 同样，对于`[1, 2, 3, 4]`，任何只对局部下降做出反应的贪婪算法将完全错过累积结构。 为了`[4, 3, 2, 1]`，假设结构增加的策略将在第一次转变时立即失败。 

关键要点是，问题不在于单个任务，而在于序列的结构如何编码必须增量跟踪的全局过程。 

## 方法

 暴力解释是逐步模拟 Jerry 的处理，在整个数组中重复应用规则，直到不再发生任何变化或直到所有交互都得到解决。 在最坏的情况下，每个操作可能需要扫描整个数组，并且可能有 O(n) 这样的操作。 这会导致 O(n²) 或更差的解决方案，具体取决于交互的实现方式。 对于 n 大约 2e5 来说，这是完全不可行的。 

关键的见解是，每个任务的效果并不以任意方式依赖于完整的历史记录，而是可以使用运行结构进行总结。 一旦我们认识到每一步只有“当前有效状态”很重要，我们就可以将历史压缩为单个累加器或类似堆栈的表示形式。 每个新任务要么扩展当前状态，要么取消部分状态，并且这种交互是本地的并且每个元素的时间恒定。 

这将问题从重复的全局重新计算减少到单遍，其中每个元素被处理一次并合并到演化状态中。

 | ---| ---| ---| ---|
 | 暴力模拟| O(n²) | O(n) | 太慢了|
 | 线性贪婪/堆栈压缩| O(n) | O(n) | 已接受 |

 ## 算法演练

 1. 我们初始化一个空结构，表示 Jerry 在已处理任务上的当前有效状态。 此结构仅保存在所有先前取消和合并之后仍然相关的信息。 
2. 我们从左到右迭代任务，一次处理一个值。 顺序很重要，因为每项任务都建立在先前任务的累积效果之上。 
3. 对于每个新任务值，我们将其与当前状态进行比较。 如果结构为空，我们只需插入值，因为没有任何东西可以与之交互。 
4. 如果结构体不为空，则判断新值是否与当前顶层状态合并或修改。 这取决于问题定义的交互规则，通常是两个相邻的贡献是否可以抵消或组合。 
5. 如果当前值抵消了最后存储的值，我们从结构中删除最后一个值。 这个模型消除了相反的影响。 
6. 如果没有取消，我们将使用新值追加或更新结构，保留累积的效果。 
7. 处理完所有任务后，剩余的结构对最终结果进行编码，我们将其转换为所需的输出形式。 

### 为什么它有效

 在每一步中，该结构都保持不变，即它代表了迄今为止看到的所有任务的完全简化形式。 当处理第二个元素时，任何可以在问题规则下交互的元素对都已立即得到解决。 由于交互是本地的，并且仅取决于最近未解决的状态，因此未来的元素无法追溯更改先前已解决的取消。 这保证了一旦删除或组合某个元素，就永远不需要重新考虑它，从而确保了单遍归约的正确性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = list(map(int, input().split()))

    st = []

    for x in a:
        if st and st[-1] == x:
            st.pop()
        else:
            st.append(x)

    print(len(st))

if __name__ == "__main__":
    solve()
```该解决方案是围绕存储序列的当前简化形式的堆栈构建的。 每个传入元素都会根据堆栈顶部进行检查。 如果它们匹配，我们将其视为取消事件并删除顶部元素。 否则，我们会将新元素作为演化状态的一部分推送。 该模式是针对相邻消除问题的标准线性缩减技术，并且它避免了重新扫描或重复模拟的任何需要。 

一个常见的实现陷阱是在访问最后一个元素之前忘记检查堆栈是否为空。 另一个微妙的问题是假设取消总是成对发生，无论顺序如何，而实际上该规则仅适用于当前缩减状态中的相邻相等元素，而不适用于原始数组。

 ## 边缘情况

 对于像这样的单元素输入`[7]`，堆栈只是存储元素并返回大小 1，确认正确处理最小输入而不会意外弹出。 

对于交替对，例如`[1, 1, 2, 2, 3, 3]`，每次插入都会立即取消前一个插入，并且堆栈始终保持为空。 这证实了重复的本地取消是增量处理的，无需重新处理。 

对于相同值的长时间运行，堆栈在入栈和出栈操作之间交替，但每个元素仍然只处理一次，从而确认即使在最坏情况的振荡输入中，算法也保持线性。
