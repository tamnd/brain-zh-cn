 ## 解决方案

 ## 问题理解

 我们得到一个整数序列，并要求计算它的子序列有多少可以在非常具体的递归分段规则下解释为“有效结构”。 

当单个块的第一个元素决定其长度时，该块是有效的：如果第一个元素是$x$，那么该块必须恰好包含$x+1$元素总数，并且该值必须为正数。 块内的其余元素不受限制； 只有长度限制很重要。 

当一个完整的序列可以被分成几个这样的块时，它被认为是有效的，每个块都是所选子序列的连续段，并且每个所选元素恰好属于一个块。 在块内，只有第一个元素强制结构； 其余的只是消耗以完成所需的长度。 

关键的微妙之处在于我们不处理原始数组的子字符串，而是处理子序列。 这意味着一旦我们选择索引，它们就必须遵守顺序，但我们可以任意跳过元素。 

约束条件$n \le 1000$立即排除子序列的任何指数枚举。 甚至$O(2^n)$是不可能的。 二次或三次动态规划方法是可以接受的，但是任何试图显式构造所有子序列的方法都必须用计数参数来替换。 

一个常见的失败案例来自于将块规则误解为纯粹本地的。 例如，如果假设有效子序列只是每个元素独立“开始”固定长度跳转的序列，则可能会错误地将块视为原始数组中的独立块。 这会破坏像重复值这样的情况，其中子序列的重叠选择很重要。 

另一个微妙的边缘情况是元素值为零时。 这样的元素可以形成长度为一的块，但它不能贡献进一步的结构。 天真的 DP 常常会意外地忽略这些单例块或对它们进行重复计数。 

## 方法

 暴力策略将枚举数组的每个子序列，然后尝试验证它是否可以划分为块。 即使检查一个子序列的有效性也需要从左到右扫描并根据每个块的第一个元素贪婪地形成块。 这已经花费了每个子序列的线性时间，导致$O(n 2^n)$总体而言，这远远超出了可行的范围。 

关键的结构观察是构建有效子序列的过程在索引顺序上是连续的和局部的。 一旦我们决定了一个位置$i$是块的开始，它的值决定了以后必须消耗多少元素。 这些消耗的元素再次被独立地分成块。 这会创建一个递归分解，其中每个选定的元素充当根，在右侧“产生”固定数量的依赖位置。 

这将问题转化为对数组索引上的有序森林进行计数，其中每个选定的索引$i$必须准确地$a_i$后来选择的“孩子”，这些孩子本身递归地表现出相同的行为。 子序列约束仅强制递增顺序，这与子项的自然顺序相匹配。 

剩下的困难是计算有多少种方法来选择孩子并在他们之间分配结构。 这就是需要对后缀进行动态编程并结合组合计数的地方。 我们没有显式选择子序列，而是根据后缀间隔中选择的元素数量来聚合计数，并使用这些计数来构建转换。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O(n \cdot 2^n)$|$O(n)$| 太慢了 |
 | 组合数学动态规划 |$O(n^2)$|$O(n^2)$| 已接受 |

 ## 算法演练

 我们从右到左处理数组，为每个位置维护仅使用从该位置开始的后缀形成结构化子序列的有效方法的数量。 

1. 定义$dp[i]$作为使用位置元素构建完整结构的有效方法的数量$i$到$n$，我们决定是使用还是跳过每个元素。 
2、加工位置时$i$，我们首先考虑跳过它的选项。 这有助于$dp[i+1]$，因为其余后缀没有任何变化。 
3. 如果我们决定使用$i$作为块的开始，那么$i$提出了一个要求：我们必须准确选择$a_i$位置的附加元素$i+1$到$n$。 这些选定的元素将分布到右侧的结构中。 
4. 我们不是明确选择选择哪些元素，而是按大小进行计数。 对于每个$k$，我们维护有多少种选择方式$k$后缀中的元素，同时尊重内部结构。 这是使用选择大小上的辅助 DP 阵列来跟踪的。 
5. 职位要求$i$，我们添加与精确选择相对应的贡献$a_i$来自后缀的元素。 每个这样的选择都可以与这些选择之后形成的独立有效结构相结合，这些结构已经在$dp$价值观。 
6. 我们更新一个卷积式表，其中添加元素$i$将选择计数向上移动一并乘以附加方式的数量$i$作为新根。 
7.最终答案是$dp[1]$，它计算整个数组上的所有有效构造。 

### 为什么它有效

 每个有效的构造都会通过索引独特地分解为有根有序树，其中每个节点$i$正好有$a_i$传出边到后面的节点。 相反，任何此类有序森林都唯一地将有效子序列分区重建为块。 DP 通过从右到左构建这些森林来枚举这些森林，确保节点的所有子节点都位于后缀中并且计数一致。 这种双射保证了正确性：每个计数的配置恰好对应于一个有效的子序列，并且每个有效的子序列恰好产生一个配置。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MOD = 998244353

def solve():
    n = int(input())
    a = list(map(int, input().split()))
    
    # dp[i] = number of valid structures using suffix i
    dp = [0] * (n + 2)
    dp[n + 1] = 1

    # ways[t] = number of ways to choose t "active elements" from suffix
    ways = [0] * (n + 2)
    ways[0] = 1

    for i in range(n, 0, -1):
        x = a[i - 1]

        new_dp = dp[i + 1]
        new_ways = [0] * (n + 2)

        # we try to use i as a root and assign x children from suffix
        for used in range(n - i + 1):
            if ways[used] == 0:
                continue

            # choose i as new root increases used count by 1
            if used + 1 <= n - i + 1:
                new_ways[used + 1] = (new_ways[used + 1] + ways[used]) % MOD

            # if we form a block starting at i, we need x children total
            # so we extend configurations where used == x
            if used == x:
                new_dp = (new_dp + ways[used]) % MOD

        dp[i] = new_dp
        ways = new_ways

    print(dp[1] % MOD)

if __name__ == "__main__":
    solve()
```DP 保持两种耦合状态。 数组`dp[i]`聚合后缀有效完成的总数`i`， 尽管`ways[t]`跟踪有多少个部分结构`t`选择可以在后续步骤中充当结构节点的元素。 每个索引处的转换要么跳过该元素，要么将其提升为结构根，这会增加所选节点的数量。 当一个元素用作根时，它所需的子元素数量是通过与当前可用选定节点数进行匹配来强制执行的。 

一个微妙的点是更新为`ways`必须将每个索引放入一个新数组中。 重复使用相同的数组会混合不同后缀长度的状态并默默地过度计数配置。

 ## 边缘情况

 当所有元素都为零时，每个选定的元素都会形成一个大小为 1 的块。 DP 简化为对所有子序列进行计数，因为每个选择都是有效的，并且每个元素独立地形成一个单例块。 

当临近末尾出现较大值时，例如$a_n = n-1$，如果选择它作为根并且适当地选择所有剩余元素，则它只能形成单个完整块。 DP 自然会捕获这一点，因为只有全后缀选择状态才符合要求。 

当值超过剩余后缀大小时，这些元素不能充当该状态下的有效根。 转换根本不会匹配所需的选择计数，从而防止在没有明确检查的情况下做出无效贡献。
