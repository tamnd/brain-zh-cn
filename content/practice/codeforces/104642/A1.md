 ## 解决方案
 ## 问题理解

 我们得到了一系列命令来控制一个简单的类似机器人的系统。 该程序是由两种字符组成的字符串。 一个角色增加系统的射击威力，另一个角色发射一发子弹，造成相当于当时当前威力的伤害。 

系统启动时功率等于1。当我们从左到右扫描程序时，每次遇到功率增加命令时，功率都会加倍。 每次遇到射击命令时，我们都会将当前功率添加到总伤害中。 

除了该计划外，我们还给出了最大允许损害。 任务是确定使程序产生不超过此限制的总损坏所需的相邻命令之间的相邻交换的最小数量。 如果即使重新排列命令后仍无法实现，我们必须报告这一事实。 

关键的困难在于，交换不仅会改变局部顺序，还会改变哪些射击发生在哪个功率级别。 由于威力取决于射击前已经出现的增加量，因此较早移动射击会减少其贡献，而稍后移动则会增加其贡献。 

约束通常允许程序长度达到 10^5 左右。 这立即排除了任何模拟所有可能的交换或尝试所有排列的方法。 在最坏的情况下，即使是重复局部改进的 O(n^2) 模拟也会太慢。 我们需要一种方法来直接解释交换如何影响总损害。 

当所有射击都以最小功率发生时，就会出现微妙的边缘情况。 在这种情况下，没有任何交换是有用的，我们必须直接将计算出的损害与限制进行比较。 当程序不包含射击命令时会出现另一种边缘情况，在这种情况下，无论排列如何，损坏始终为零。 

## 方法

 暴力破解的想法是将程序视为一个字符串，并重复尝试所有相邻的交换，每次交换后重新计算总损坏，并保留最佳结果。 这在原则上是正确的，因为它探索了交换可达到的排列的完整状态空间。 然而，每次重新计算损坏的成本为 O(n)，并且交换序列的数量呈指数增长。 即使将我们限制在贪婪的局部改进上，在最坏的情况下也会导致 O(n^3) 行为，这远远超出了大输入的可接受限制。 

关键的观察结果是，每个射击命令的贡献仅取决于其前面出现的威力增加命令的数量。 功率增加命令和射击命令之间的每次交换都会在位置之间移动一个单位的功率贡献，并且最有利的交换始终是那些将射击命令提前移动到最新可能的功率增加命令的交换。 这使我们能够始终定位出现问题的模式的最右侧。 

我们不是直接模拟交换，而是计算当前的损害。 如果已经在限制内，则不需要交换。 否则，我们会通过将射击命令之前出现的最右边的威力增加命令向右移动一步来反复减少伤害。 每一次这样的移动都会减少高能射击的贡献，并且我们会跟踪每次移动减少的伤害量，直到达到限制或用完有用的交换。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力模拟| O(2^n·n) | O(2^n·n) | O(n) | 太慢了 |
 | 贪心归约| O(n^2) | O(n^2) | O(n) | 太慢了 |
 | 通过位置跟踪优化贪婪 | O(n) 或 O(n log n) | O(n) | 已接受 |

 ## 算法演练

 我们将该程序视为一个序列，其中每个射击命令累积的伤害等于其之前出现的功率倍增命令的数量所确定的功率级别。

1. 通过从左到右扫描来计算初始总损坏，保持当前功率。 每次我们看到电源命令时，我们都会将电源加倍。 每次我们看到射击命令时，我们都会为总伤害增加力量。 这给出了我们需要降低的基线。 
2. 如果初始伤害已经小于或等于允许的限度，我们会立即停止，因为不需要交换。 任何重新排列只会冒增加某些镜头伤害的风险。 
3. 否则，我们会反复尝试减少损坏。 为了有效地做到这一点，我们找到了电源命令出现在射击命令之前的最右侧位置。 交换这两个角色将射击命令的贡献减少了一些较高功率上下文的一半，这是单次交换的最大可能减少。 
4. 我们执行该交换并相应地更新总伤害，而不是从头开始重新计算。 关键是只有受影响的射击指挥的贡献会以可预测的方式发生变化，而其他则保持不变。 
5. 我们继续此过程，直到损坏变得可以接受或不再存在有效的交换。 如果我们用完掉期机会而仍然超出限制，那么答案是不可能的。 

它之所以起作用是基于这样一个不变式：唯一可以减少伤害的交换是将射击命令向左移动到电源命令上。 任何此类交换都会将执行该射击的功率水平降低一倍。 由于更高功率的射击造成的伤害呈指数级增加，因此始终将交换应用到最新可能的功率射击边界可确保每次交换最大程度地减少伤害。 这种贪婪的选择保留了正确性，因为任何非本地交换都可以分解为一系列具有相同或更小的效果的相邻交换，并且延迟有益的交换永远不会有助于减少所需操作的数量。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def compute_damage(s):
    power = 1
    damage = 0
    for c in s:
        if c == 'C':
            power *= 2
        else:
            damage += power
    return damage

def solve():
    t = int(input())
    for _ in range(t):
        d, s = input().split()
        d = int(d)
        s = list(s.strip())

        if s.count('S') == 0:
            print(0)
            continue

        swaps = 0

        while True:
            damage = compute_damage(s)
            if damage <= d:
                print(swaps)
                break

            idx = -1
            for i in range(len(s) - 1):
                if s[i] == 'C' and s[i + 1] == 'S':
                    idx = i

            if idx == -1:
                print("IMPOSSIBLE")
                break

            s[idx], s[idx + 1] = s[idx + 1], s[idx]
            swaps += 1

if __name__ == "__main__":
    solve()
```该解决方案首先使用程序语义的直接模拟来计算损坏。 功能`compute_damage`反映了力量如何在琴弦上演变的确切定义。 

然后主循环反复检查当前配置是否有效。 如果没有，它会从最右端搜索有利的相邻对，在该对中功率增加后立即进行射击。 这种选择是有意为之的，因为相对于稍后的威力增加而言，较早地移动射击对于减少伤害具有最大的影响。 

交换操作是就地执行的，我们会计算已经应用了多少次此类操作。 当满足损坏条件或不存在改进交换时，算法终止。 

重要的微妙之处在于，每次交换后重新计算伤害在概念上很简单，但效率很低。 然而，它保持了实现的正确性，并且在该 A1 变体的典型小限制范围内。

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 | O(n·k) | O(n·k) | 每次交换可能需要完整扫描以重新计算损坏，最多重复 k 次 |
 | 空间| O(n) | 该字符串存储为可变列表 |

 考虑到 A1 版本的典型约束，其中 n 和总交换量仍然很小，这就足够了。 对于较大的约束，需要优化的贪心策略来减少重复的重新计算。

 ## 边缘情况

 一种重要的边缘情况是程序不包含拍摄命令。 在这种情况下，损害计算循环永远不会添加任何值，因此无论交换如何，结果始终为零。 该算法通过在进入交换循环之前检查是否存在射击命令来正确地短路这种情况。 

当所有拍摄命令已经处于尽可能低的功率水平时，会发生另一种边缘情况。 尽管从语法上来说交换仍然是可能的，但它们并不能进一步减少伤害。 该算法通过在每次交换后重新计算损坏来处理此问题，并在无法改善时终止。 

最后一种边缘情况是字符串仅包含射击命令。 在这里，没有交换可以改变结构，因此最右边对有益对的搜索立即失败，并且算法正确地输出不可能。
