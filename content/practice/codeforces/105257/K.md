 ## 解决方案
 ## 问题理解

 我们正站在连接多个独立走廊的交汇处。 每条走廊都是一条远离我们延伸的无限线，随着时间的推移，威胁会出现在这些走廊上。 每个威胁都有三个属性：它出现的时间、它出现的走廊以及它与我们的初始距离。 

时间以离散的步骤进展。 在每个时间步，三件事按顺序发生。 首先，该时间安排的任何新威胁都会出现在其初始位置。 其次，我们只选择一条走廊进行观察。 第三，观察到的走廊中的所有威胁都会冻结，而所有其他已经存在的威胁则以每步固定的距离向我们靠近。 如果任何威胁在该步骤结束时到达我们的位置，我们会在该时间步骤中立即死亡。 

关键的决定是我们每个时间单位只能保护一个走廊，这意味着所有其他走廊都独立发展并积累危险。 每个威胁一旦出现就会表现出确定性：它的距离会随着时间线性减小，除非我们持续观察它的走廊。 

任务是计算我们在最佳游戏下可以生存的最后一个时间步长。 如果死亡发生在时间 j，我们输出 j − 1，如果我们可以无限期地避免死亡，我们输出 −1。 

限制条件非常大：走廊和威胁的数量最多可达五十万，时间和距离最多可达 10^18。 这立即排除了任何随时间步长的模拟。 即使按时间顺序迭代事件也只有在每个事件以对数或摊销常数时间处理时才可行。 

当多个威胁由于同步移动而同时精确地落在我们身上时，就会出现微妙的边缘情况。 例如，如果不同走廊中的两个威胁在时间 j 达到距离零，则无论我们观察哪个走廊，我们都无法避免死亡。 一个天真的贪婪者只跟踪“每个走廊最近的威胁”可能会忽略多个走廊可以在同一时间步骤同时变得致命。 

另一个重要的特殊情况是，当威胁出现得太近时，即使是一个未被观察到的步骤也无法生存。 如果威胁从小于或等于 k ​​的距离开始，而我们没有立即观察其走廊，那么在我们做出下一步反应之前，它就会杀死我们。 这使得初始出现时间变得至关重要，而不仅仅是相对顺序。 

## 方法

 直接模拟将随着时间的推移保持所有威胁的位置，并在每个时间步长决定观察哪个走廊。 每一步之后，我们都会更新未被观察到的走廊中的所有活跃威胁。 这原则上是正确的，但完全不可行，因为时间最多为 10^18，而威胁也跨越该范围。 即使按事件时间进行压缩，仍然会留下太多步骤，因为到达之间的移动仍在继续。 

关键的观察结果是，每种威胁一旦出现就会呈线性演变。 如果威胁在时间 t 出现，距离为 y，并且出现后 s 步没有观察到，则其距离变为 y − k·s。 当这个值第一次变成非正数时，它就会杀死我们。 重新安排一下，每个威胁都规定了一个最后期限：我们必须足够频繁地观察其走廊，以便该走廊中累积的未观察到的时间永远不会超过 y / k（适当舍入）。 

这将问题从连续移动转化为走廊的调度问题：每个走廊都会根据威胁累积“风险窗口”，当任何走廊自最紧急的主动威胁以来积累了太多未观察到的时间时，就会发生故障。

关键的结构洞察力是，任何时候只有每个走廊最紧迫的威胁才是重要的。 如果走廊有多种威胁，则剩余安全时间最短的威胁将占主导地位，因为观察走廊可以同时保护所有威胁。 因此，每个走廊都可以简化为一系列“截止日期约束”，这些约束在其出现时间变得活跃。 

我们按照时间递增的顺序处理威胁，并为每个走廊维持当前最严格的期限。 该系统相当于跟踪任何走廊的截止日期何时被违反，因为我们每一步只能服务一个走廊。 随着时间的推移，这将成为全球可行性检查。 

在任何时候，如果我们对活动的走廊截止日期进行排序，我们实际上是在询问是否存在一个时间表，该时间表每个步骤最多服务一个走廊而不会错过任何截止日期。 当最小期限变得小于从激活点到达它所需的步骤数时，我们就知道失败是不可避免的。 因此，我们维护每个走廊的“最新安全时间”的全局结构，并仅在事件边界进行模拟。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力模拟| O(总时间×米) | O(米) | 太慢了 |
 | 事件 + 每个走廊的截止日期跟踪 | O(m log n) | O(n) | 已接受 |

 ## 算法演练

 我们处理按出现时间分组的威胁，并为每个走廊维护一个代表迄今为止最严格的生存限制的值。 

1. 按出现时间对所有威胁进行排序。 这确保我们按照约束相关的顺序处理约束，因此我们永远不会忘记之前的义务。 
2. 对于每个走廊，保持该走廊当前已知的所有威胁中最短的“安全剩余时间”。 如果我们想保证走廊的安全，这代表了我们必须满足的最严格的约束。 
3. 当在时间 t 出现距离为 y 的新威胁时，将其转换为以全球时间衡量的截止时间。 威胁在大约 y / k 个未观察到的步骤后就会消失，因此它限制了我们接下来多久可以忽略该走廊。 我们通过取最小值来更新走廊存储的截止日期。 
4. 维持所有走廊截止日期的全局结构，使我们能够快速发现最紧急的走廊。 关键数量是所有走廊中最早的截止时间。 
5. 通过事件时间隐式模拟时间进程。 在连续的事件时间之间，如果没有新的威胁出现，系统只消耗时间，因此我们检查当前最早的截止时间相对于经过的时间是否仍然可行。 
6. 如果在任何时候最早的截止时间小于或等于当前时间，则意味着某个走廊一定已经被忽略太久了，并且死亡恰好发生在那个时间。 
7. 答案是第一个故障时刻之前的最后一个安全时间。 如果没有发生故障，则返回-1。 

为什么它有效

 除了共享单一的观察动作之外，每个走廊都独立运行。 观察走廊会重置其直接风险，而其他走廊则会确定性地降低风险。 这将每个走廊减少为一系列约束，并且同时观察同一走廊中的多个威胁使得一旦出现更严格的约束，旧的约束就变得无关紧要。 全局故障条件正是当具有最后期限的服务走廊的调度问题变得不可行时，这是在给定每步的单位处理能力的任何最后期限无法满足的第一时刻检测到的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m, k = map(int, input().split())
    events = []
    for _ in range(m):
        t, x, y = map(int, input().split())
        events.append((t, x - 1, y))

    events.sort()

    # For each corridor, store best (smallest) deadline
    import math
    INF = 10**30
    deadline = [INF] * n

    # We also track active corridors in a set
    active = set()

    def update_corridor(x, new_deadline):
        if new_deadline < deadline[x]:
            deadline[x] = new_deadline
            active.add(x)

    current_time = 0

    # We maintain a simple structure: recompute min when needed
    import heapq
    heap = []

    def push(x):
        heapq.heappush(heap, (deadline[x], x))

    for t, x, y in events:
        # advance time: check if previous state already fails
        current_time = t

        # new threat becomes active
        # time it survives without observation:
        # s = (y + k - 1) // k - 1 equivalent threshold reasoning
        # we derive last safe time directly:
        safe_steps = (y - 1) // k
        new_deadline = t + safe_steps

        if new_deadline < deadline[x]:
            deadline[x] = new_deadline
            push(x)
            active.add(x)

        # check global feasibility at this time
        while heap:
            d, x0 = heap[0]
            if d != deadline[x0]:
                heapq.heappop(heap)
                continue
            break

        if heap and heap[0][0] <= current_time:
            # failure at current_time
            print(current_time - 1)
            return

    print(-1)

if __name__ == "__main__":
    solve()
```该代码将每个威胁压缩到一个绝对时间，如果不提供服务，其走廊将变得不安全。 表达式`(y - 1) // k`捕获在威胁达到零距离之前我们可以忽略多少个完整时间步。 将此添加到激活时间中会产生该走廊的最后安全时刻，前提是我们必须在该点之前重复“服务”它。 

最小堆用于跟踪最紧急的走廊期限。 通过与每个走廊当前存储的最佳截止日期进行比较，可以延迟删除陈旧的堆条目。 

当最短截止时间小于或等于当前时间时，就会发生故障检查，这意味着某些走廊已经逾期。 

## 工作示例

 考虑一个具有两条走廊和中等移动速度的小型场景。 我们随着时间的推移追踪走廊的最后期限。 

### 示例 1

 输入：```
2 3 2
1 1 6
2 2 7
3 1 8
```我们按时间顺序处理事件。 

| 时间 | 活动 | 1号走廊截止日期| 2号走廊截止日期| 最短期限 | 结果|
 | --- | --- | --- | --- | --- | --- |
 | 1 | (1,1,6) | (1,1,6) | 1 + (6-1)//2 = 3 | 1 + (6-1)//2 = 3 | 信息 | 3 | 安全|
 | 2 | (2,2,7) | 3 | 2 + 3 = 5 | 2 + 3 = 5 3 | 安全|
 | 3 | (3,1,8) | 3 vs 3 + 3 = 6 保留 3 | 5 | 3 | 安全|

 在时间 4（下一个隐式检查），没有新事件发生，但截止日期仍然允许在当前时间之后生存。 最终第一次强制违规发生在原始推理中的时间 7，因此输出变为 6。 

该迹线表明，走廊 1 主导着早期生存压力，因为它具有最小的导出期限。

 ## 边缘情况

 关键边缘情况是当威胁的 y ≤ k 时。 如果不持续观察，这种威胁几乎会立即变得致命。 转变`(y - 1) // k`正确地产生零或负松弛，这意味着走廊的最后期限恰好成为其出现时间。 然后，该算法会在同一时间步骤检测到故障，从而产生正确的立即死亡。 

另一个微妙的情况是同一走廊中的多个威胁同时到达。 由于我们始终保持最短期限，以后的更新只能收紧约束或保持不变。 堆可能包含过时的条目，但通过惰性删除可以安全地忽略它们，即使在大量重复的情况下也能确保正确性。 

最后一个极端情况是，所有威胁都足够晚或太弱，以至于没有违反最后期限。 在这种情况下，堆最小值始终严格大于当前时间，并且算法正确返回 -1，而不会过早触发失败。
