 ## 解决方案
 ## 问题理解

 我们被要求建造一个$n \times m$网格仅填充三个符号：`1`,`+`， 和`*`。 网格不仅仅是一个静态对象，它定义了两个方向的表达式。 

从左到右读取的每一行都成为有效的算术表达式。 从上到下阅读的每一列也成为有效的算术表达式。 有效表达式的定义是递归的：一个或多个连续的单个块`1`s 是有效的，并且如果两个有效表达式与`+`或者`*`，结果也是有效的。 

因此从概念上讲，每个表达式都是由串联运行构建的二叉树表达式`1`， 在哪里`+`和`*`充当子表达式之间的运算符。 

整个网格的值是通过评估所有网格来定义的$n$行表达式和$m$列表达式并对它们的数值结果求和。 我们必须构建能够最小化这个总价值的网格。 

关键的困难在于行和列不是独立的。 每个单元格只参与一个行表达式和一个列表达式，因此放置一个符号会同时影响两个计算。 

限制条件$3 \le n, m \le 9$非常小。 这立即表明网格模式上的指数或组合构造是可以接受的，特别是如果我们可以在本地修剪或构造选择。 一种更接近于对模式进行强力推理或贪婪局部构造的解决方案是合理的。 

递归语法产生了一个微妙的边缘情况：表达式不使用标准运算符优先级规则进行计算，而是作为由语法引入的完全括号结构。 将其误解为标准算术会导致完全错误的值。 例如，`1+1*1`不一定以乘法优先级解释； 它是由构造规则定义的结构化表达式树。 

## 方法

 一个天真的想法是独立对待每个细胞并尝试所有$3^{nm}$网格，验证每一个。 对于每个网格，我们需要提取所有行和列字符串，将它们解析为表达式，并计算它们的值。 即使与$n,m \le 9$， 这是$3^{81}$，太大了，甚至无法从概念上进行枚举。 

即使我们限制自己对每行进行推理，我们仍然面临跨行和列的依赖。 核心问题是一个单元格同时影响两个表达式，因此局部决策在两个方向上都是耦合的。 

关键的观察是有效表达式的结构严重限制了其中`+`和`*`可以出现。 有效的表达式是由纯的串联构建的`1`块，这意味着运算符仅作为连续段之间的分隔符。 这意味着表达式的数值仅取决于如何将其划分为段，而不取决于超出段边界的运算符的任意放置。 

这将问题从任意表达式解析转移到控制每行和列中的段结构。 每行和每列都可以看作是一系列连续的组的序列`1`s 由运算符分隔，值由这些组的大小决定。 

由于网格很小，我们可以利用对称性：每个单元格的选择恰好影响一个行段结构和一个列段结构。 最佳解决方案最终倾向于最大限度地分裂成许多小的配置`1`块，因为较小的乘法组件会比加法合并增加整体价值更积极地减少整体价值。 

这导致了一种建设性的贪婪模式：我们希望避免长时间连续运行`1`在两个方向上，而是以结构化的棋盘式方式引入运算符。 可以通过平衡行方向和列方向的分段最小化来导出最佳结构，从而得出以下模式：`1`s 被放置在稀疏结构中，运算符填充其余部分以防止出现大型乘法链。 

自从$n,m \le 9$，我们可以修复一个规范模式：交替`1`和`*`在受控网格中，同时确保每一行和每一列仍然是有效的表达式。 在所有有效配置中，最小的配置是通过最大化碎片来实现的`1`段。

 ## 算法演练

 1. 首先将每一行和每一列解释为一个序列，其中仅连续运行`1`重要的是价值贡献。 这告诉我们，长链`1`快速增加价值。 
2. 决定避免长时间不间断的序列`1`同时在两个方向上。 如果我们放置太多`1`在一条直线上，由于线段尺寸较大，该行或列会产生较大的值。 
3. 构建一个网格，其中`1`放置在防止在行和列方向上同时形成大的连续块的位置。 最简单的稳定结构是交替模式，以便没有行或列包含长连续的`1`s。 
4. 填充剩余位置`*`作为分隔符，因为乘法比加法更积极地分割表达式组件，从而在结构化评估中增加它们。 
5. 确保每一行和每一列仍然通过构造形成有效的表达式：因为所有行和列都至少包含一个`1`，并且分隔符仅分割有效的子表达式，语法始终满足。 
6. 输出结果网格。 

### 为什么它有效

 关键的不变量是没有行或列形成一个长的连续段`1`s，这将主导表达式值。 由于递归乘法结构，表达式值随段长度超线性增长，因此最小化段大小主导了重新排列运算符的任何好处。 该结构确保每个最大运行`1`s 的大小有限，任何合并段的替代配置必然会增加至少一行或列的贡献，而不是减少其他行或列的贡献。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())

    # We construct a checkerboard-like pattern:
    # put '1' on cells where (i + j) % 2 == 0, otherwise '*'
    # This guarantees no long consecutive runs of '1' in any row or column.

    grid = []
    for i in range(n):
        row = []
        for j in range(m):
            if (i + j) % 2 == 0:
                row.append('1')
            else:
                row.append('*')
        grid.append(''.join(row))

    print("\n".join(grid))

if __name__ == "__main__":
    solve()
```该实现直接编码了防止连续的想法`1`段。 棋盘格条件确保水平和垂直方向上没有两个相邻单元格同时存在`1`，它将每个游程长度限制为 1。这是网格中最强的可能碎片，因此可以最大限度地减少递归乘法和加法结构的表达式爆炸。 

一个微妙的点是我们从不显式地构造或评估表达式。 我们完全依赖于符号放置的结构控制，这已经足够了，因为考虑到分段，表达式语法是确定性的`1`s。

 ## 边缘情况

 潜在的边缘情况是两个维度都是奇数时，例如$3 \times 3$或者$5 \times 5$。 在这些情况下，可能会怀疑朴素的交替模式会造成行和列贡献的不平衡。 然而，棋盘结构保持一致，因为决定有效性的是邻接约束，而不是全局奇偶性。 

例如，在$3 \times 3$：```
1*1
*1*
1*1
```每行都有隔离`1`s，并且每列也有隔离的`1`s。 尽管数量`1`行和列之间的 s 略有不同，没有游程超过长度 1，因此没有表达式因乘法结构而膨胀。 

另一种边缘情况是，如果在不考虑列的情况下尝试贪婪的行式构造，则会意外形成单个长行或列段。 例如：```
111
1*1
111
```这将创建具有大的连续段的行，从而大大增加值。 棋盘模式通过同时强制列约束来完全避免这种情况，从而防止这种隐藏的爆炸。
