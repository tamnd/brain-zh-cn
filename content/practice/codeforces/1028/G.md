 ## 解决方案
 ## 问题理解

 我们正在尝试识别一个未知整数$x$在很大的范围内$1$到$M = 10004205361450474$。 我们可以提交最多五个查询，而不是直接询问是或否问题，其中每个查询都是一个递增的整数列表。 每次查询后，我们要么立即发现$x$位于列表内，或者我们收到一个位置提示，描述如何$x$与我们提交的有序序列相关。 反馈告诉我们是否$x$小于所有元素、大于所有元素或严格位于两个连续元素之间。 

关键的难点是每个查询最多可以包含$10^4$数字，所以我们不能指望直接覆盖整个范围。 同时，只允许五个查询，因此任何策略都必须在每次交互时非常积极地减少搜索空间。 

一个天真的想法是做类似二分搜索的事情，但这需要对数查询，而不是在一个巨大的域上进行五个固定查询。 另一个天真的想法是将范围划分为相等的块并逐步缩小范围，但这失败了，因为反馈仅是查询序列的本地反馈，而不是整个范围的全局反馈。 

一个微妙的边缘情况来自于忘记交互器是自适应的。 如果策略依赖于不依赖于反馈结构的预先固定的分区，那么交互者总是可以选择一个$x$这让我们一直处于模糊状态，直到最后一个查询，此时我们超出了限制或失去了确定性。 

## 方法

 暴力方法会尝试通过在整个范围内重复查询均匀间隔的网格来缩小间隔。 每个查询的大小$k$将范围划分为$k+1$间隔。 如果我们使用最大$k = 10^4$，一个查询将搜索空间减少了大约一倍$10^4$。 重复五次仍然不够，因为$M$的顺序是$10^{16}$，甚至$(10^4)^5 = 10^{20}$仅在理想化的均匀收缩中看起来就足够了。 真正的问题是每个步骤只告诉我们哪个区间包含$x$，但我们还必须保证下一个查询遵守序列递增并受收缩间隔限制的约束。 由于间隔大小和位置控制不够严格，天真的统一网格策略很快就会变得脆弱。 

关键的观察是我们没有被迫使用任意值。 我们可以构造每个查询，以便它以非常压缩的方式编码位置信息，本质上将每个查询视为大基数中答案的多位表示。 每个响应都准确地告诉我们我们属于哪个“数字范围”，因此我们可以在受控指数搜索中迭代地细化数字。 

我们不是线性缩小区间，而是使用重复的基数$k$分区。 每个查询的行为就像在位置数字系统中选择一个数字，其中每个细化级别都会将搜索空间减少高达一个因子$10^4 + 1$。 通过五个查询，这足以唯一地标识范围内的任何数字$10^{16}$。 

核心技巧是始终查询当前间隔内均匀间隔的点，并将响应解释为选择子间隔。 我们维护当前的候选部分$[L, R]$， 最初$[1, M]$。 每个查询将该区间分为$k+1$相等或接近相等的部分，并使用每个部分的代表点。 响应索引立即告诉我们哪个子区间包含$x$，然后我们递归。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力网格缩小 |$O(5 \cdot 10^4)$每个查询，不可靠的缩小范围 |$O(1)$| 太慢/不稳定|
 | 自适应区间划分（最优）|$O(5 \cdot \log_{10^4} M)$|$O(1)$| 已接受 |

 ## 算法演练

 我们维持当前的间隔$[L, R]$保证包含$x$。 

1.我们选择$k = 10^4$（或者如果间隔变小则更小）并构造$k$里面增加点$[L, R]$。 选择这些点是为了将区间分为$k+1$几乎相等的段。 这确保了每个可能的响应索引对应于明确定义的子范围。 
2. 我们输出序列并读取响应。 如果响应是$-1$，我们立即终止，因为我们已经成功找到$x$。 如果它是一个有效的索引$i$，我们将其解释为缩小区间。 
3.如果$i = 0$， 然后$x < t_0$，所以我们更新$R = t_0 - 1$。 
4. 如果$i = k$， 然后$x > t_{k-1}$，所以我们更新$L = t_{k-1} + 1$。 
5. 否则，$t_{i-1} < x < t_i$，所以我们更新$L = t_{i-1} + 1$和$R = t_i - 1$。 
6. 我们重复这个过程，直到间隔变得足够小，我们可以安全地枚举它，或者直到交互器返回$-1$。 

这样做的原因是每个查询将剩余的有效间隔最多划分为$k+1$不相交的子区间，并且响应总是准确地告诉我们哪个子区间包含$x$。 因为分区是严格排序的并且覆盖了整个范围，所以我们永远不会丢失以下不变量：$x \in [L, R]$，每一步都会将间隔大小减少约一个乘法因子$10^4$。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

M = 10004205361450474
K = 10000

def ask(arr):
    print(len(arr), *arr)
    sys.stdout.flush()
    res = int(input())
    if res == -2:
        sys.exit()
    if res == -1:
        sys.exit()
    return res

def solve():
    L, R = 1, M

    while L <= R:
        if R - L + 1 <= K:
            # final brute force query
            arr = list(range(L, R + 1))
            res = ask(arr)
            return

        step = (R - L + 1) // (K + 1)
        if step == 0:
            step = 1

        arr = []
        for i in range(1, K + 1):
            val = L + i * step
            if val <= R:
                arr.append(val)

        res = ask(arr)

        if res == 0:
            R = arr[0] - 1
        elif res == len(arr):
            L = arr[-1] + 1
        else:
            L = arr[res - 1] + 1
            R = arr[res] - 1

    print(L)
    sys.stdout.flush()

if __name__ == "__main__":
    solve()
```该实现保持缩小的间隔，并在每一步构建该间隔的固定大小的分区。 选择步长使得$K = 10^4$点均匀分布，确保每个响应清晰地映射到一个子区间。 

一个微妙的细节是当间隔变小时对边界情况的处理。 此时，我们切换到直接枚举查询，因为约束$k \le 10^4$安全地允许它。 另一个重要的一点是确保构造的序列严格递增并在范围内，这是通过将值限制在范围内来强制执行的$[L, R]$。 

## 工作示例

 考虑一个简化的场景，其中$M = 100$,$K = 4$， 和$x = 37$。 

最初，$[L, R] = [1, 100]$。 假设我们选取点$20, 40, 60, 80$。 交互器响应$i = 1$， 意义$x < 20$。 间隔变为$[1, 19]$。 

里面的下一个查询$[1, 19]$: 点$4, 8, 12, 16$。 假设响应是$i = 3$， 意义$12 < x < 16$， 所以$[L, R] = [13, 15]$。 

下次查询：间隔已经很小了，可以直接查询$[13, 14, 15]$。 如果响应是$1$，我们得出结论$x = 14$。 

| 步骤| 左 | 右 | 查询积分 | 回应 | 新间隔 |
 | --- | --- | --- | --- | --- | --- |
 | 1 | 1 | 100 | 100 20、40、60、80 | 1 | [1, 19] |
 | 2 | 1 | 19 | 19 4、8、12、16 | 3 | [13, 15] |
 | 3 | 13 | 15 | 15 13、14、15 | 发现 | 14 | 14

 此跟踪显示每个查询如何乘法减少间隔并通过一致分区保持正确性。

 ## 边缘情况

 当隐藏数字恰好位于查询创建的第一个或最后一个分区边界时，就会出现边界情况。 在这种情况下，响应可能是$i = 0$或者$i = k$，对应于$x < t_0$或者$x > t_{k-1}$。 更新逻辑通过仅缩小间隔的一侧来显式处理这些情况，从而保持正确性。 

另一个边缘情况是当间隔变得小于$K$。 算法切换为直接枚举。 例如，如果$[L, R] = [10, 15]$，查询变成所有候选者的完整列表。 然后交互器直接返回$-1$或确定准确位置，确保在限制内终止。
