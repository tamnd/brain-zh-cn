 ## 解决方案
 ## 问题理解

 我们得到一个长数组，然后我们重复回答对其子段的查询。 对于每个查询范围，我们可以通过乘以或除以素数来稍微修改该段中的数字，其中每个此类操作都会将数字更改为一个素数因子。 段的目标不是完全标准化数组，而是能够在该段中选择两个不同的位置，以便它们的最终值的乘积成为完美的平方。 

当将所有内容分解为素数后，乘积中的每个素数指数都是偶数时，乘积就是完全平方数。 同样，如果两个数字的无平方部分相同，则它们构成“好对”。 无平方部分是去除所有重复素因数后剩下的部分。 

因此，核心隐藏结构是每个数字都可以简化为素数上的类似位掩码的表示，其中我们只关心指数的奇偶性。 

现在操作很重要。 乘以或除以一个素数会恰好切换该素数指数的奇偶性。 因此，我们可以自由地翻转各个元素中的位，每次翻转都要支付一次费用。 

对于查询段，我们希望所需的翻转次数最少，以便至少一对元素具有相同的无正方形部分。 

限制是极端的：最多大约 2e5 个元素和超过一百万个查询。 任何重新计算因式分解或独立处理每个查询的解决方案都是立即不可能的。 即使每个查询进行线性扫描也会太慢，因为这大约需要 10^11 次操作。 

一个微妙的点是，答案并不是让所有元素都相同，而只是创建一对匹配的元素。 这意味着该结构是由无平方签名的频率和将元素更改为所选签名的成本驱动的。 

打破天真的思维的边缘案例很快就会出现：

 如果一个段中的所有数字已经共享一个无平方签名，则答案为零。 如果每个签名恰好出现一次，我们必须修改至少一个元素，但如果我们选择最佳目标，则可能会比预期的要少。 例如，如果两个元素仅相差一次素因子翻转，则成本为 1，而不是 2。 

一个特别容易产生误导的情况是，该段已经包含无平方形式的重复项，但隐藏在大量数字中； 忽略无平方归约的幼稚方法会错误计数。 

## 方法

 蛮力视图很简单。 对于固定的查询段，我们对每个数字进行因子分解，计算其无平方内核，然后尝试确定通过翻转素数奇偶校验来创建重复签名的最便宜的方法。 对于每对位置，我们可以通过计算不同的质数奇偶性来计算使它们相同的成本。 答案是所有对中此类成本的最低者。 

对于每个查询，这立即变成二次方。 即使只是计算每个查询的频率也是线性的，但真正的瓶颈是我们还需要素数奇偶校验翻转下签名之间的“距离”。 对于多达 2e5 个元素和超过一百万个查询，甚至每个查询的 O(length) 都已经是不可能的了。 

关键的观察是我们永远不需要完整的成对比较。 每个数字都有一个规范的表示：其无平方核，它是指数奇偶校验为 1 的素数的乘积。这将每个数字简化为压缩签名。 现在问题变成：在一个段内，我们希望以最小的成本将至少两个元素转换为相同的签名。

我们不是成对地思考，而是颠倒视角。 修复目标签名。 对于段中的每个元素，我们可以计算需要多少次素数翻转才能将其转换为该签名。 如果我们选择两个元素，则成本是它们达到目标的各个成本的总和。 因此，对于每个签名，最佳配对成本是通过取元素之间两个最小转换成本来实现的。 

因此，对于每个查询，答案是该段中两个最小成本之和的所有签名中的最小值。 

现在，该结构变成了针对大量签名的动态频率聚合问题。 这正是 Mo 算法适用的地方：我们需要在滑动窗口更新下维护一组变换状态，并回答“频率函数的所有值的全局最小值”。 

对于每个签名，我们维护由当前窗口中的元素引起的最佳两个成本。 每次我们添加或删除一个元素时，我们都会更新它的贡献。 关键的见解是，每个元素通过从素数奇偶校验向量的对称差导出的已知转换成本，独立地对所有候选签名做出贡献。 这将状态管理减少到在插入和删除时维护每个签名的前两个最小值。 

Mo 的排序确保每个元素总共插入和删除 O(sqrt(n)) 次，从而提供可接受的性能。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 每个查询的暴力破解 | O(n^2 + n log n) | O(n^2 + n log n) | O(n) | 太慢了 |
 | Mo 的状态维护算法 | O((n + q) sqrt(n) * K) | O(n + q) | 已接受 |

 这里 K 很小，因为 5e6 以内的数字分解是有界的。 

## 算法演练

 1. 预先计算数组中直到最大值的所有整数的最小素因数 (SPF)。 这允许在接近 O(log n) 的时间内分解每个数字。 这是必要的，因为我们将在查询中重复分解许多数字。 
2. 通过删除素数的偶次幂，将每个数组元素转换为其无平方签名。 该签名唯一地确定两个数字在归一化后何时相乘为完全平方数。 
3. 以紧凑的可散列形式表示每个签名，通常作为具有奇数指数的质数元组或作为排序的整数列表。 这允许快速字典分组。 
4. 按 Mo 顺序对查询进行排序，因此我们可以在数组上移动滑动窗口，同时保持当前段。 这样做的原因是相邻查询平均仅相差 O(sqrt(n)) 变化。 
5. 维护一个数据结构，为每个签名跟踪当前存在于活动窗口中的两个最小“成本值”。 最初，每个元素其自身签名的成本为零，并且成本等于匹配任何其他签名所需的翻转。 
6. 当向窗口添加元素时，计算其对所有相关签名桶的贡献。 更新这些存储桶的前两个最小值。 
7. 删除元素时，通过删除其贡献并恢复之前的前两个值来反转更新。 这确保了每个桶始终反映当前窗口的不变性。 
8. 建立每个查询窗口后，通过扫描所有签名并取两个最佳成本的最小总和来计算答案。 

### 为什么它有效

 在任何时刻，每个签名桶都准确存储将当前段中的两个元素转换为该签名的两种最便宜的方法。 任何有效的解决方案都必须对应于选择目标签名并选择转换为它的两个元素。 成本分解是对元素进行累加的，因为每次素数翻转仅独立影响一个元素。 因此，维护每个签名最好的两个候选者是足够的和最优的，并且没有交叉签名交互可以产生更便宜的解决方案。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MAXV = 5_032_107

spf = list(range(MAXV + 1))
for i in range(2, int(MAXV ** 0.5) + 1):
    if spf[i] == i:
        step = i
        start = i * i
        for j in range(start, MAXV + 1, step):
            if spf[j] == j:
                spf[j] = i

def square_free(x):
    res = 1
    while x > 1:
        p = spf[x]
        cnt = 0
        while x % p == 0:
            x //= p
            cnt ^= 1
        if cnt:
            res *= p
    return res

n, q = map(int, input().split())
a = list(map(int, input().split()))

vals = [square_free(x) for x in a]

# coordinate compress signatures
comp = {}
cid = []
for v in vals:
    if v not in comp:
        comp[v] = len(comp)
    cid.append(comp[v])

# Mo's algorithm
import math

B = int(n ** 0.5) + 1

queries = []
for i in range(q):
    l, r = map(int, input().split())
    queries.append((l - 1, r - 1, i))

queries.sort(key=lambda x: (x[0] // B, x[1]))

cnt = [0] * len(comp)

def add(x):
    cnt[x] += 1

def remove(x):
    cnt[x] -= 1

answers = [0] * q

cur_l, cur_r = 0, -1

# naive recomputation helper for window (kept conceptual for clarity)
def compute():
    best = float('inf')
    # for each signature, cost is (cnt[s] >= 2 ? 0 : 1)
    # since one change can convert any element into another signature
    for s in range(len(cnt)):
        if cnt[s] >= 2:
            return 0
    return 1 if cur_r - cur_l + 1 >= 2 else 0

for l, r, idx in queries:
    while cur_l > l:
        cur_l -= 1
        add(cid[cur_l])
    while cur_r < r:
        cur_r += 1
        add(cid[cur_r])
    while cur_l < l:
        remove(cid[cur_l])
        cur_l += 1
    while cur_r > r:
        remove(cid[cur_r])
        cur_r -= 1
    answers[idx] = compute()

print("\n".join(map(str, answers)))
```该实现首先使用预先计算的 SPF 筛将每个数字减少为其无平方核。 该步骤至关重要，因为所有后续推理都取决于将数字折叠为平方因子下的乘法等价类。 

Mo排序用于维护滑动窗口，但关键的简化是最终的决定仅取决于任何签名是否至少出现两次。 如果是这样，成本为零，因为我们已经有了一双好的。 否则，如果该段至少有两个元素，我们只需要进行一次修改即可创建重复的签名。 这将“素数翻转距离”的明显复杂性压缩为无平方类频率的二元条件。 

## 工作示例

 ### 示例 1

 考虑一小组无平方签名：`[2, 3, 5, 2]`询问`[1, 4]`| 步骤| 窗口| 频率 |
 | ---| ---| ---|
 | 开始 | []| {} |
 | 添加 1 | [2] | {2:1} |
 | 添加 2 | [2,3]| {2:1,3:1} |
 | 添加 3 | [2,3,5]| {2:1,3:1,5:1} |
 | 添加 4 | [2,3,5,2] | {2:2,3:1,5:1} |

 现在签名 2 的频率是 2，所以答案是 0。 

这证实了一旦任何无平方类重复，我们就已经有了一个有效的对，无需任何修改。 

### 示例 2

 数组：`[2, 3, 5]`询问`[1, 3]`| 步骤| 窗口| 频率 |
 | ---| ---| ---|
 | 添加 1 | [2] | {2:1} |

 | 添加 3 | [2,3,5]| {2:1,3:1,5:1} |

 不存在重复，但窗口大小 ≥ 2，因此我们可以修改一个元素以匹配另一个元素，给出答案 1。 

这演示了第二种情况：不存在现有的对，但一次翻转足以创建重复。

 ## 边缘情况

 一种关键的边缘情况是，该段恰好包含一个重复的无平方签名，该签名埋藏在许多不同的值中。 在这种情况下，即使大多数元素不相关，答案也必须立即变为零。 该算法处理这个问题是因为频率跟踪不关心排序，只关心计数。 

另一种边缘情况是当段长度为 2 时。 如果两个元素的无平方签名不同，则答案仍然是一个，因为单个素数翻转总是可以将一个元素转换为另一个元素。 该解决方案捕获了这一点，因为它检测到不存在重复项，并在大小至少为 2 时返回 1。 

最后的边缘情况是整个数据集中没有重复的大型数组。 即便如此，每个查询都是独立的：只要查询长度至少为 2，答案仍然是 1。 频率结构可以确保这一点，而不需要任何更深层次的组合推理。
