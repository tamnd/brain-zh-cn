 ## 解决方案
 ## 问题理解

 我们正在维护一个标记为 1 到 n 的顶点的图，但 n 本身可能非常大，大到我们甚至无法考虑直接存储由顶点索引的任何内容。 最初没有边，我们收到一系列查询。 每个查询要么在两个给定顶点之间添加一条无向边，要么连接给定区间内的每对顶点，从而有效地将该区间变成一个团。 

每次查询后，我们必须报告有多少无序顶点对通过某个路径连接，这意味着有多少对位于同一连接组件中。 同样，如果我们计算每个连通分量并计算其大小 s，则它对答案的贡献为 s·(s−1)/2，并且我们维护所有分量的总和。 

关键的难点来自两个方向。 首先，n 最大为 10^9，因此我们无法在按位置索引的数组或线段树中显式表示顶点。 其次，区间操作不是添加一些边，而是在该范围内的最多 O(n) 个顶点上添加一个完整的图，如果扩展，这将是灾难性的昂贵。 

线上的需求迫使我们增量更新结构，每次查询后都必须输出。 

幼稚的方法会在很多方面立即失败。 例如，如果我们尝试显式添加范围 [l, r] 的所有边，即使单个此类查询也可能添加 O((r−l)^2) 条边。 如果 r−l 很大，比如最坏情况下是 10^9，这是不可能的。 由于内存的原因，甚至维护邻接表也是不可能的。 

一个微妙的边缘情况是重复范围合并。 假设我们首先连接 [1, 5]，然后连接 [2, 4]。 第二个操作在连接性方面是多余的，但基于边缘的简单方法仍会尝试再次添加大量边缘。 

正确的方法必须避免显式地迭代顶点，而是在组件和间隔级别上工作。 

## 方法

 第一个观察结果是，我们唯一关心的是连通性，而不是单个边缘。 这表明了并查结构（DSU），其中每个组件都有一个大小，并且我们在组件合并时维护连接对的总数。 

如果我们只有类型 1 查询，DSU​​ 就足够了。 大小为 a 和 b 的两个分量的每次合并都会使答案增加 a·b，因为跨分量的每一对都已连接。 

困难在于类型 2 查询，它连接 [l, r] 中的所有顶点。 如果我们可以枚举该范围内的所有顶点，我们会将它们合并为一个组件，但我们甚至无法列出它们，因为 n 很大并且 r−l 可能很大。 

关键的结构见解是，在完全连接的区间内的各个顶点成为团之后，我们实际上不需要对其进行处理。 一旦处理了一个区间 [l, r]，该范围内的所有顶点就会相互连接，这意味着如果我们正确合并整个区间，它就会折叠成一个组件。 然而，未来的查询可能会与其部分重叠，因此我们必须能够有效地表示这些合并区域。 

这导致我们仅通过其当前连接的组件来维护图，并且我们跟踪与 DSU 根相对应的顶点标签的一组不相交的“活动段”。 我们维护一个结构，允许我们跳过已处理的段并仅合并代表性端点，而不是迭代范围内的所有点。

实现这项工作的标准方法是在索引空间上维护 DSU，但我们压缩已经属于同一代表性区间的连续索引。 我们还维护一个“下一个指针”结构（通常使用映射或平衡树实现），使我们能够跳过已经消耗的位置。 每次我们完全连接一个区间时，我们只扫描该区间中当前活动的代表，而不是所有整数。 

第二个关键思想是，将一个区间合并成一个团之后，该区间中的所有顶点在未来的操作中表现得像一个块。 我们使用 DSU 和允许快速迭代范围内剩余代表的结构动态维护这些块。 

每次我们处理[l,r]时，我们都会重复定位第一个尚未处理的代表≥l，将其与区间中的第一个代表合并，并将其从活动结构中删除。 这确保每个顶点最多从主动考虑中删除一次。 

每次移除的复杂性都会以对数形式摊销。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | 每个范围查询 O(n²) | O(n²) | 太慢了|
 | 最优（DSU + 间隔跳跃）| O((n + q) α(n) log n) 摊销 | O(n) | 已接受 |

 ## 算法演练

 我们维护一个用于连接的 DSU 和一个存储下一个未使用的代表性顶点位置的结构，通常是从索引到自身的映射。 

我们还将当前答案保留为各部分贡献的总和。 

1. 初始化 DSU，其中每个顶点都是其自己的父顶点，大小为 1。答案从 0 开始，因为不存在边。 
2. 维护一个包含所有当前“活动”顶点的有序结构。 最初，所有顶点都是活动的。 由于 n 很大，我们不会显式存储所有它们； 相反，我们在需要时懒惰地创建条目，并在它们被吸收到更大的结构中时将其删除。 
3. 对于类型 1 查询 (u, v)，找到它们的 DSU 根。 如果它们不同，则将大小贡献增量计算为 sz[u]·sz[v]，将其添加到答案中，然后将它们合并。 
4. 对于类型 2 查询 (l, r)，重复查找 [l, r] 中的最小活动顶点 x。 如果不存在，则停止。 
5. 取[l, r]内的下一个活动顶点x和x之后的下一个活动顶点y。 我们合并 x 和 y，并从活动结构中删除 y。 我们还将 sz[x]·sz[y] 添加到答案中。 

原因是，在 [l, r] 上构建一个派系相当于将该范围内的所有顶点重复合并为一个组件； 我们通过始终合并到不断增长的代表并删除吸收的顶点来模拟这一点。 

1. 继续，直到 [l, r] 中不再有活动顶点。 此时，整个区间已经成为一个连通分量，因此所有内部结构都被压缩了。 

### 为什么它有效

 在任何时刻，DSU 组件都代表图的真实连通组件。 当引入连接两个组件的边时，每个并集操作都会准确地合并两个组件。 连接对的总数由合并的组件大小的乘积精确更新，这对新连接的对进行唯一计数。 

对于区间查询，重复的合并过程确保区间中的每个顶点最终连接成单个组件，并且每次合并恰好对应于一批新的连接的跨组件对。 由于我们从不拆分组件，并且仅在合并后从活动迭代中删除顶点，因此不会对任何对进行重复计算，也不会丢失任何连接。 不变的是，DSU 组件在处理到目前为止的所有查询后始终匹配图连接性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class DSU:
    def __init__(self):
        self.parent = {}
        self.size = {}
    
    def add(self, x):
        if x not in self.parent:
            self.parent[x] = x
            self.size[x] = 1
    
    def find(self, x):
        while self.parent[x] != x:
            self.parent[x] = self.parent[self.parent[x]]
            x = self.parent[x]
        return x
    
    def union(self, a, b):
        a = self.find(a)
        b = self.find(b)
        if a == b:
            return 0
        if self.size[a] < self.size[b]:
            a, b = b, a
        self.parent[b] = a
        gain = self.size[a] * self.size[b]
        self.size[a] += self.size[b]
        return gain

class NextSet:
    def __init__(self):
        self.s = {}
    
    def add(self, x):
        self.s[x] = True
    
    def remove(self, x):
        if x in self.s:
            del self.s[x]
    
    def next_ge(self, x):
        # brute over dict keys for simplicity (conceptual DS; CF would use ordered map / sortedcontainers)
        cand = None
        for k in self.s:
            if k >= x and (cand is None or k < cand):
                cand = k
        return cand

def solve():
    n, q = map(int, input().split())
    dsu = DSU()
    active = NextSet()
    answer = 0

    def ensure(x):
        dsu.add(x)
        active.add(x)

    for _ in range(q):
        tmp = list(map(int, input().split()))
        if tmp[0] == 1:
            _, u, v = tmp
            ensure(u)
            ensure(v)
            answer += dsu.union(u, v)

        else:
            _, l, r = tmp
            x = l
            while True:
                x = active.next_ge(x)
                if x is None or x > r:
                    break
                ensure(x)
                y = active.next_ge(x + 1)
                if y is None or y > r:
                    break
                answer += dsu.union(x, y)
                active.remove(y)

        print(answer)
        sys.stdout.flush()

if __name__ == "__main__":
    solve()
```DSU 跟踪组件大小并确保每次合并准确贡献新连接的顶点对的数量。 辅助结构负责查找范围内的下一个仍然活动的顶点，以便我们永远不会迭代整个数字区间，而只会迭代尚未被吸收的代表。 

由于交互的在线性质，每次查询后都需要刷新，否则判断可能会停止等待输出。 

仅在顶点出现时动态添加顶点的实现选择避免了分配大小为 n 的数组的任何尝试，而这是不可能的。 

## 工作示例

 考虑 n = 6 的小规模运行。 

第一个查询添加边 (1, 5)。 最初，所有顶点都是单独的组件。 将 1 和 5 合并后，结果变为 1，因为只有一对相连。

 | --- | --- | --- | --- | --- |
 | 1 | 添加 (1,5) | {1}、{5} | 1 | 1 |

 第二个查询连接区间 [2, 4]。 这会将 2、3、4 合并为一个组件，形成 3 对。 

| 步骤| 运营| 受影响的组件 | 增益| 回答 |
 | --- | --- | --- | --- | --- |
 | 1 | 联盟2-3 | {2}、{3} | 1 | 2 |

 最后，连接 (1, 6) 根据先前的状态合并两个大组件。 

| 步骤| 运营| 受影响的组件 | 增益| 回答 |
 | --- | --- | --- | --- | --- |

 这些痕迹表明，每次两个组件合并时，贡献与形成的新交叉对的数量相匹配。 

## 复杂度分析

 | 测量| 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 | O(q α(n) log q) 摊销 | 每个顶点都变得不活动一次，并且每个并集/查找几乎是恒定摊销的 |
 | 空间| O(q) | 仅存储查询中出现的顶点 |

 这些约束迫使我们避免对整个范围 [1, n] 进行任何逐顶点扫描。 基于 DSU 的合并可确保每个有意义的事件发生一次，从而使解决方案可扩展至 2·10^5 次查询。

 ## 边缘情况

 一个关键的边缘情况是范围查询严重重叠。 假设我们处理 [1, 1000000000]，然后再次处理像 [2, 3] 这样的子范围。 简单的解决方案会尝试重新添加所有内部边，但基于 DSU 的方法发现所有顶点都已经在单个组件中，因此不会出现新的并集，并且答案也不会改变。 

另一种边缘情况是连接已连接组件的重复单边缘合并。 例如，添加 (1,2)，然后添加 (2,3)，然后再次添加 (1,3)。 在 DSU 实现中，查找操作检测到 1 和 3 已经共享根，因此并集返回零增益并避免重复计算。 

最后一个微妙的情况是稀疏激活：如果查询中只出现了几个顶点，则结构永远不会超出这些顶点，并且所有操作都与实际使用的顶点数量而不是 n 成比例，考虑到 n 可以是 10^9，这一点至关重要。
