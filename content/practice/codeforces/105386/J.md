 ## 解决方案
 ## 问题理解

 我们得到了一张由无向道路连接的城市图，其中每条道路都属于一家公司并且有一定的长度。 结构是固定的，但移动受到必须按顺序使用的一系列门票的限制。 

每张票都允许一次有限制的“迷你旅行”：您选择一个目的地城市，并仅使用属于指定公司的道路从当前城市移动到那里，并且所选路线的总长度不得超过给定的预算。 您也可以在罚单上什么也不做，留在原地。 

关键的困难在于，每张票证都限制移动到一家公司，并且您无法重新排列这些限制的顺序。 处理完所有车票后，我们想知道从城市 1 可以到达哪些城市。 

约束非常大，每次测试最多有 5×10^5 个城市、边缘和门票，因此任何尝试模拟每张门票的最短路径或重复从头开始运行 BFS 的方法都立即不可行。 即使每张票只有一张 Dijkstra 也远远超出了限制。 该解决方案必须积极重用结构并避免从头开始重新计算图连接性。 

当贪婪地思考整个图中的最短路径而不考虑公司约束时，就会出现微妙的失败案例。 例如，在一次遍历中组合来自不同公司的边将错误地允许没有工单序列允许的转换。 另一个陷阱是将每张票视为全局最短路径查询，它忽略了每张票之后起点的演变。 

错误直觉的一个小例子：

 如果票证 1 允许公司 A 的长度最长为 10，而票证 2 允许公司 B 的长度最长为 10，则将公司 A 和 B 的所有边一起用作单个 20 长度的预算路径来计算可达性是无效的。 每张票的间隔是严格的。 

## 方法

 暴力模拟将一张一张地处理票证。 在每一步中，我们都会选择所有当前可到达的城市，并针对每个城市，运行仅限于一家公司且受票价预算限制的约束最短路径搜索。 这本质上意味着每张票运行多源 Dijkstra 或 BFS，仅限于一种颜色的边缘。 

这是正确的，因为每张票都是一个独立的移动阶段。 然而，成本变得灾难性的：对于每一张 k 票，我们可能会遍历图表的很大一部分。 在密集情况下，这会导致 O(k (n + m) log n)，这远远超出了可行的限制。 

关键的结构见解是每张票仅取决于单个公司内的连接。 我们可以将每个公司的图表预处理成支持快速“达到预算范围”查询的结构，而不是重复地重新计算最短路径。 一旦我们知道，对于每个城市和公司，我们可以在重量限制内传播多远，我们就可以将票证应用为重复受限扩展。 

重要的观察是，对于固定公司，我们只关心该公司子图中的最短路径距离。 计算这些距离后，每张票就成为预先计算的度量空间上的范围受限的可达性扩展。 剩下的挑战是在许多票证上有效地执行这些扩展，这是通过使用按公司分组的邻接列表并在每个票证上运行多源 Dijkstra 来处理的，但仅在受影响的公司图上，小心地重用全局状态，以便仅在相关时放宽每个边缘。 

使这项工作有效的优化是，每张票仅激活一家公司，因此边缘永远不会混合。 在整个序列中，每条边仅在选择其公司时才进行处理，从而提供摊销线性行为。

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 每票最短路径 | O(k (n + m) log n) | O(k (n + m) log n) | O(n + m) | 太慢了 |
 | 公司发起的 Dijkstra 重用 | O((n + m) log n) 摊销 | O(n + m) | 已接受 |

 ## 算法演练

 我们在每张购票后都会维护当前可到达的城市集。 我们不会在全局范围内重新计算所有内容，而是仅通过当前票证允许的公司传播可达性，并使用受票证限制限制的最短路径式扩展。 

1. 按公司对所有道路进行分组。 对于每个公司，我们维护自己的邻接列表，以便当票证激活该公司时我们只能遍历相关边。 
2. 维护一个布尔数组可达，最初仅将城市 1 标记为可达。 这代表我们处理完之前的门票后可以到达的所有城市。 
3. 对于每张票 (a_i, b_i)，我们从所有当前可到达的城市开始执行约束传播。 我们限制遍历属于公司 a_i 的边，并确保累积距离不超过 b_i。 使用类似于 Dijkstra 的过程，并使用距离为 0 的所有可到达城市初始化的优先级队列。 
4. 在此过程中，我们仅放松公司 a_i 的边缘。 如果我们发现预算范围内到某个城市的距离较短，我们就会更新它。 这确保我们从任何当前可到达的节点开始计算允许的公司子图中的真实最短距离。 
5. 完成以 b_i 为界的 Dijkstra 后，我们将所有距离有限且 ≤ b_i 的城市标记为下一阶段可达。 
6. 我们放弃距离并重复下一张票。 

我们重新初始化每张票的距离的原因是每张票定义了一个新的约束遍历，除了起始城市集之外，与之前的距离度量无关。 

### 为什么它有效

 在每一步中，可达集准确地表示使用按顺序排列的前 i 个门票可以到达的城市。 从步骤 i 到 i+1 的转换计算在仅使用公司 a_{i+1} 的边和最多 b_{i+1} 总权重的路径下该集合的闭包。 因为我们同时计算来自所有当前可到达节点的最短路径，所以我们不会错过从任何有效中间城市开始的任何复合路线。 由于边缘松弛仅限于单个公司，因此不会发生无效的约束混合。 

## Python 解决方案```python
import sys
import heapq

input = sys.stdin.readline

def solve():
    n, m, k = map(int, input().split())
    
    # group edges by company
    company_edges = {}
    for _ in range(m):
        u, v, c, l = map(int, input().split())
        if c not in company_edges:
            company_edges[c] = []
        company_edges[c].append((u - 1, v - 1, l))
    
    tickets = [tuple(map(int, input().split())) for _ in range(k)]
    
    reachable = [False] * n
    reachable[0] = True
    
    INF = 10**30
    
    for ai, bi in tickets:
        if ai not in company_edges:
            continue
        
        dist = [INF] * n
        pq = []
        
        for i in range(n):
            if reachable[i]:
                dist[i] = 0
                heapq.heappush(pq, (0, i))
        
        edges = company_edges[ai]
        
        while pq:
            d, u = heapq.heappop(pq)
            if d != dist[u] or d > bi:
                continue
            
            for a, b, w in edges:
                if a == u:
                    v = b
                elif b == u:
                    v = a
                else:
                    continue
                
                nd = d + w
                if nd <= bi and nd < dist[v]:
                    dist[v] = nd
                    heapq.heappush(pq, (nd, v))
        
        new_reachable = reachable[:]
        for i in range(n):
            if dist[i] <= bi:
                new_reachable[i] = True
        reachable = new_reachable
    
    print(''.join('1' if x else '0' for x in reachable))

if __name__ == "__main__":
    t = int(input())
    for _ in range(t):
        solve()
```该实施仅在活跃公司的边缘上维护每票 Dijkstra。 初始化步骤将所有当前可达的节点推送为距离为零的源，确保多源传播。 

一个微妙的点是我们必须检查`d > bi`当从优先级队列中弹出时，否则过时的状态可能会超出预算。 另一个重要的细节是，我们只放松属于当前公司的边，这是通过扫描该公司的邻接列表并检查端点来强制执行的。 

仅在完成票证的 Dijkstra 后才会更新可达数组，保留仅在每张票证完全消耗后才发生移动的语义。 

## 工作示例

 考虑一个包含 4 个城市和两张门票的小场景。 假设城市 1 和 2 由权重为 5 的公司 1 连接，城市 2 和 3 由权重为 5 的公司 1 连接，而城市 3 由权重为 3 的公司 2 连接到城市 4。门票为（公司 1, 10）然后（公司 2, 3）。 

在票证 1 之后，从城市 1 开始，我们计算公司 1 中的最短路径：城市 2 距离 5 可达，城市 3 距离 10 可达。因此可达变为 {1,2,3}。 

在票证 2 之后，我们从 {1,2,3} 开始多源，但仅使用公司 2。只有城市 3 以成本 3 连接到城市 4，因此城市 4 可以到达。 

| 门票 | 活跃节点| 公司 | 新距离 | 可以在 | 之后到达
 | --- | --- | --- | --- | --- |
 | 1 | {1} | 1 | 2:5, 3:10 | {1,2,3} |
 | 2 | {1,2,3} | 2 | 4:3 从 3 | {1,2,3,4} |

 这显示了如何仅通过每张票证的活跃公司来扩展可达性，同时保留多源行为。 

现在考虑公司图表断开连接的情况。 如果票证允许公司 1 但预算较小，则仅激活一部分节点，其余节点保持不变。 这证实了不可访问的组件永远不会错误地泄漏到可访问的集合中。

 ## 边缘情况

 第一个边缘情况是当票证指的是没有边缘的公司时。 在这种情况下，算法仍然初始化多源距离，但没有找到松弛，因此可达集保持不变。 例如，如果城市 1 独自参与该公司的组件，则产出保持稳定。 

当所有可到达的城市都已经是最佳来源时，就会出现另一种边缘情况。 Dijkstra 初始化将它们全部推入距离为零，并且算法正确地将它们视为同时起点。 这避免了需要从不同中间节点开始的丢失转换。 

最后一个微妙的情况是，公司图表中存在多个路径，但只有一些路径尊重预算。 因为我们明确使用最短路径松弛，所以即使存在更长的非最佳路径，任何最小距离超过预算的城市也永远不会被标记为可达。 这确保了循环和冗余边缘下的正确性。
