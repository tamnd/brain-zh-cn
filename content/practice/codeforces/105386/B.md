 ## 解决方案
 ## 问题理解

 有几场比赛同时进行。 每场比赛已经有一定数量的参赛团队，您可以根据需要在这些比赛中分配额外的团队。 

每场比赛都以非常简单的方式颁发金牌：每当该场比赛的队伍总数达到固定值的倍数时$k$，该比赛产生了一枚金牌。 同样地，如果一场比赛的结果是$t$团队，它贡献$\lfloor t / k \rfloor$奖牌。 

您从初始配置开始$a_i$每场比赛的队伍，你也有$m$可以在比赛中任意分配额外的球队。 每个团队都必须被分配到某个地方，每次分配都会使相应比赛的团队数量增加一个。 

任务是分发所有$m$以最大化所有比赛中奖牌总数的方式对各队进行分组。 

这些限制以非常直接的方式产生影响。 最多有100场比赛，但额外的队伍数量$m$和初始计数$a_i$两者都可以大到$10^9$。 这立即排除了任何逐个团队模拟流程的策略，因为即使是单个测试用例也可能需要数十亿次操作。 该解决方案必须通过团队集体推理而不是个人任务来发挥作用。 

当你假设每个额外的团队都独立贡献局部最优的东西时，就会出现天真的贪婪思维的微妙失败案例。 例如，添加一支球队可能不会立即增加任何比赛的奖牌数量，但精心选择的添加顺序可以“完成”门槛并解锁奖牌。 如果你只关注每个团队的直接收益，你就会忽略一个事实：接下来的多个进展都很重要。 

另一个陷阱是假设分配了一些团队后，可以统一对待剩余的团队。 如果您仍然有未与倍数对齐的比赛，这将变得不正确$k$，因为不同比赛中的部分进步与未来的奖牌相互作用。 

## 方法

 蛮力的想法很简单：在每一步，尝试将剩余的一支球队分配给任何比赛，重新计算奖牌总数，并继续递归或迭代，直到分配完所有球队。 这是正确的，因为它探索了所有可能的分布。 然而，每个$m$团队最多有$n$选择，因此搜索空间变得像$n^m$，即使对于很小的值也是完全不可行的$m$。 

关键的观察是，奖牌仅取决于每场比赛跨越多个分数的次数$k$。 这意味着每次比赛的状态完全由其余数模捕获$k$，每个有用的动作要么完成余数以达到下一个倍数，要么贡献完整的大小块$k$之后。 

这将问题从每个团队的决策转变为每个奖牌的增量。 每场比赛都可以被认为是提供获得+1奖牌的“机会”，每个奖牌的成本都以额外的团队来衡量。 然后，我们希望首先选择最便宜的机会，但我们也意识到，在每次比赛的第一次调整之后，所有剩余的收益都会表现一致。 

这种结构允许我们将解决方案分为两个阶段：首先我们优化所有“剩余修复”，然后我们批量处理剩余团队。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | 指数为$m$| O(n) 递归 | 太慢了 |
 | 最佳| O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 我们首先将问题转化为“我们已经可以获得多少奖牌”加上“我们可以使用额外的团队强制获得多少额外的奖牌”。 

每场比赛$i$最初贡献$\lfloor a_i / k \rfloor$立即获得奖牌，因为这些奖牌已经完成了多次。 

现在我们关注每场比赛中剩余的结构。 让$r_i = a_i \bmod k$。 如果我们增加一些球队，有效增加奖牌数的唯一方法是将比赛推至下一个倍数$k$。 这正好需要$k - r_i$团队如果$r_i \neq 0$， 和$k$团队如果$r_i = 0$，因为恰好处于倍数并不意味着我们“接近”下一个。 

因此，每场比赛都提供一个有意义的升级：支付费用$c_i$团队通过完成下一个多重边界来获得 +1 奖牌。 

只要我们有足够的团队，我们就会对这些成本进行排序并贪婪地接受它们。 

经过这些升级后，选择的每场比赛现在都与多个比赛完美匹配$k$。 剩下的比赛仍然有固定的结构，但是剩下的$m$团队现在只能通过完整大小的块进行交互$k$，并且将它们分布在多个竞赛中并不能优于对它们进行最佳分组。 剩下的球队最好的可能结果就是$\lfloor m_{\text{left}} / k \rfloor$额外的奖牌。 

这会导致一个干净的过程：

 1. 计算初始奖牌为$\sum \lfloor a_i / k \rfloor$。 
2. 对于每场比赛，计算达到下一个倍数的成本$c_i$。 
3. 整理成本，贪婪地花钱$m$在他们身上。 
4. 添加剩余贡献$\lfloor m / k \rfloor$。 

它起作用的原因来自于状态的结构压缩。 每场比赛都有一个非均匀过渡（以达到下一个倍数），之后所有未来的收益在大小块中表现相同$k$。 这确保了所有不对称性最多在之后耗尽$n$决定，其余的就变得统一了。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    T = int(input())
    for _ in range(T):
        n, k = map(int, input().split())
        a = list(map(int, input().split()))
        m = int(input())

        base = 0
        costs = []

        for x in a:
            base += x // k
            r = x % k
            if r == 0:
                costs.append(k)
            else:
                costs.append(k - r)

        costs.sort()

        extra = 0
        for c in costs:
            if m < c:
                break
            m -= c
            extra += 1

        extra += m // k
        print(base + extra)

if __name__ == "__main__":
    solve()
```该代码首先从初始配置中提取保证的奖牌贡献。 然后，它为每场比赛建立一个“完成成本”列表，这意味着需要多少支球队才能将该比赛推向下一个比赛的倍数。$k$。 对这些成本进行排序使我们能够始终首先将团队用于最有效的升级。 

在用尽可承受的升级后，剩余的团队将使用整数除以进行批量处理$k$，因为一旦所有比赛都一致，只有完整的区块才能贡献额外的奖牌。 

这里的一个常见错误是将剩余的零视为零成本，这将错误地允许免费奖牌。 这就是代码显式分配成本的原因$k$在那种情况下。 

## 工作示例

 ### 示例 1

 考虑$k = 5$, 与$a = [4, 6]$， 和$m = 6$。 

最初的奖牌是：

 - 4 贡献 0
 - 6 贡献 1

 所以基数 = 1

 现在费用：

 - 比赛 1：需要 1 队达到 5 → 成本 1
 - 比赛 2：需要 4 支队伍才能达到 10 → 成本 4

 我们对成本进行排序：[1, 4]

 | 步骤| 米 | 行动| 额外奖牌|
 | ---| ---| ---| ---|
 | 0 | 6 | 开始 | 0 |
 | 1 | 5 | 取成本 1 | 1 |
 | 2 | 1 | 花费4？ 没有| 1 |

 现在剩余 m = 5？ 实际上在第一步之后m=5，第二次尝试之后不能取4，所以m=5。 

剩余奖牌 = 5 // 5 = 1

 总计 = 基础 1 + 额外 2 = 3

 这显示了如何优先考虑尽早完成，而剩余的团队则形成统一的区块。 

### 示例 2

 让$k = 3$,$a = [3, 1, 2]$,$m = 4$。 

基础奖牌：

 - 3 → 1
 - 1 → 0
 - 2 → 0

 基数 = 1

 费用：

 - [3,2,1]

 排序：[1,2,3]

 | 步骤| 米 | 行动| 额外奖牌|
 | ---| ---| ---| ---|
 | 0 | 4 | 开始 | 0 |
 | 1 | 3 | 取 1 | 1 |
 | 2 | 1 | 拿2个？ 没有| 1 |

 剩余 m = 3 → 3 // 3 = 1

 合计 = 1 + 2 = 3

 这证实了留下的团队只在完整的区块中才重要的关键思想。

def run(inp: str) -> str:

    import sys

    T = int(input())
    out = []
    for _ in range(T):
        n, k = map(int, input().split())
        a = list(map(int, input().split()))

        costs = []

            base += x // k
            r = x % k

                break

            extra += 1
        extra += m // k
        out.append(str(base + extra))

    return "\n".join(out)

# all equal values

 ## 边缘情况

 一个关键的边缘情况是当$k = 1$。 在这种情况下，每个团队总是贡献一枚奖牌，因此整个优化分解为简单的加法。 该算法自然地处理了这个问题，因为每个成本都变为 1，并且每个额外的团队都直接通过最终划分做出贡献。 

另一个边缘情况是当比赛已经是$k$。 成本设置为$k$，不为零，防止错误的免费升级。 该算法正确地将此类竞赛视为在下一次增益之前需要完整的块。 

当所有剩余团队都不足以完成任何升级时，就会出现最后一个微妙的情况。 在这种情况下，排序成本循环会提前终止，并且只有完整的大小块$k$做出贡献，这正确地反映了部分进展没有价值。
