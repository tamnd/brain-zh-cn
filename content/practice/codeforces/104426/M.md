 ## 解决方案
 ## 问题理解

 我们拥有一组机器，每台机器都有固定的容量，以及一组应用程序，其中每个应用程序由几个必须放置在这些机器上的相同 Pod 组成。 每个 Pod 都属于一个应用程序，并且每个 Pod 必须分配给一个节点，同时尊重节点容量。 

放置所有 pod 后，每个应用程序都有一个质量指标：对于该应用程序，我们查看每个节点并计算该应用程序最终在其上运行的 pod 数量，然后取所有节点中的最小值。 该值捕获应用程序分布的均匀程度，因为它会惩罚接收很少或不接收该应用程序 pod 的节点。 

目标是将所有 Pod 分布在节点上，以便每个应用程序的最小值尽可能大。 换句话说，我们希望每个应用程序在每个节点上至少出现一定次数，并且我们希望同时在所有应用程序中最大化保证的最小频率。 

这些约束表明节点和应用程序的数量可以多达 100,000 个，并且 Pod 数量也可以很大。 这立即排除了任何尝试显式模拟放置或迭代每个应用程序节点的解决方案。 任何二次甚至接近二次的方法$N \cdot M$是不可能的。 

一个关键的结构性限制是 Pod 总数不超过总容量，因此可行性绝不是全球空间不足的问题。 困难纯粹在于节点之间的分配平衡。 

当一个应用程序的 Pod 很少时，就会出现一种微妙的边缘情况。 例如，如果一个应用程序只有一个 pod，则无论放置如何，其每节点的最小值始终为零，因为至少$N-1$节点将接收该应用程序的零个 Pod。 在至少一个应用程序足够小而无法有意义地传播的任何情况下，这都会迫使全局答案为零。 

另一种极端情况是产能极度倾斜。 如果一个节点几乎拥有所有容量，天真的直觉可能会建议将所有内容打包到那里，但这会破坏每个应用程序的最小值，因为其他节点没有获得大多数应用程序的 pod。 

## 方法

 一个蛮力的想法是尝试直接为每个应用程序跨节点构建 Pod 分布，并计算所得的最小值。 甚至限制我们自己检查固定目标值的可行性$x$，我们需要决定是否可以安排每个应用程序，以便每个节点至少接收$x$该应用程序的 Pod。 由于共享节点容量，这变成了应用程序之间存在依赖关系的组合分配问题。 

如果我们尝试在没有结构的情况下贪婪地模拟分配，我们很快就会遇到这样的事实：应用程序仅通过节点容量进行交互，而不是直接进行交互。 这表明我们不应该明确地构建分配，而应该进行聚合推理：需要多少总容量才能支持给定的目标最小值。 

修复候选值$x$。 对于每个应用程序$j$，如果我们希望其每个节点的最小值至少为$x$，那么每个节点必须至少接收$x$该应用程序的 Pod。 由于有$N$节点，这已经意味着应用程序$j$至少需要$x \cdot N$总共有 pod。 因此，可行性的必要条件是$a_j \ge x \cdot N$为所有人$j$。 如果任何应用程序未能达到此目的，则该目标就不可能实现。 

然而，这本身还不够，因为节点容量限制了可以放置的 Pod 总数。 每个节点必须至少托管$x$每个应用程序的 Pod，这意味着每个节点至少需要容纳$x \cdot M$跨应用程序的 Pod 总数。 所以每个节点$i$必须满足$c_i \ge x \cdot M$。 这就给出了第二个必要条件。 

这两个条件也被证明是足够的，因为如果每个应用程序都有足够的 pod 总数并且每个节点都有足够的容量，我们可以从概念上准确地分配$x$将每个应用程序的 pod 分配到每个节点，并任意分配剩余的 pod。 剩余的灵活性由全球产能状况保证。 

因此，可行性简化为检查两个简单的不等式以确定固定的$x$，我们可以二分查找最大有效值$x$。 

构造赋值的暴力思想被单调谓词的可行性检查所取代，从而实现了二分搜索。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力建造| 指数/阶乘 | O(NM) | 太慢了|
 | 具有可行性检查的二分搜索 | O((N+M) 对数答案) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们二分查找答案$x$，它代表我们想要强制执行的每个节点每个应用程序的最小保证。 

1. 设置搜索范围$x$介于 0 和最大可能值之间，该值由总容量除以应用程序数量乘以节点来确定。 

上限并不重要； 我们可以安全地使用一个大值，例如$\sum c_i$。 
2. 对于固定候选人$x$，检查每个应用是否满足$a_j \ge x \cdot N$。 

这确保了我们至少有足够的 Pod 来放置$x$在该应用程序的每个节点上。 
3.同时检查每个节点是否满足$c_i \ge x \cdot M$。 

这确保每个节点都可以支持所有应用程序所需的 Pod 数量。 
4. 如果两个条件都成立，则候选人$x$是可行的。 否则，就不是。 
5. 二分查找$x$，可行时向右移动，不可行时向左移动，找到最大有效值。 

关键的实现细节是两次检查都是线性扫描，因此每次可行性测试都是$O(N+M)$，二分查找将其乘以$\log \max a_j$。 

### 为什么它有效

 核心不变量是目标的可行性$x$仅取决于全球供应和每个节点的容量是否可以支持统一的基线分配$x$每个节点每个应用程序的 Pod。 一旦满足这些基线，所有剩余的 Pod 就可以任意分配，而不会降低任何应用程序的最小值以下$x$，因为额外的 Pod 只会增加节点数量，而不会减少最小值。 这使得可行性谓词单调$x$，保证二分查找的正确性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def can(x, n, m, c, a):
    need_per_app = n * x
    need_per_node = m * x

    for v in a:
        if v < need_per_app:
            return False

    for v in c:
        if v < need_per_node:
            return False

    return True

def solve():
    n, m = map(int, input().split())
    c = list(map(int, input().split()))
    a = list(map(int, input().split()))

    lo, hi = 0, 10**18
    ans = 0

    while lo <= hi:
        mid = (lo + hi) // 2
        if can(mid, n, m, c, a):
            ans = mid
            lo = mid + 1
        else:
            hi = mid - 1

    print(ans)

if __name__ == "__main__":
    solve()
```该实现将可行性分离为一个干净的函数，用于检查两个派生的约束。 二分搜索探索可能答案的空间。 

一个微妙的一点是，上限设置得很大，而不是仔细推导。 这是安全的，因为一旦可行性就会很快失败$x$由于乘法超过任何有意义的界限$N$和$M$。 另一个细节是所有算术都是用 Python 整数完成的，避免了溢出问题。

 ## 边缘情况

 当一个应用程序的 Pod 很少时，就会出现严重的边缘情况。 对于输入：```
2 3
10 10
1 10 10
```任何$x \ge 1$立即失败，因为$1 < 2 \cdot x$。 该算法在可行性检查时正确拒绝所有正值，因为$a_j < n \cdot x$触发提前退出。 

另一种情况是节点严重倾斜：```
3 2
100 1 1
50 50
```尽管总容量足够，但节点限制迫使$x=0$，因为每个节点都必须支持$2x$Pod 和小节点立即失败。 

这些案例证实，该解决方案正确地将应用程序供应和节点容量视为独立的瓶颈，而不是仅仅依赖于总和。
