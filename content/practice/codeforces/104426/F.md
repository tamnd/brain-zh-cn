 ## 解决方案
 ## 问题理解

 我们得到一个二进制数组和一个固定的窗口大小。 一步操作中，我们选择任意连续的片段$k$元素并翻转其中的每个值，将零变成一，一变成零。 我们最多可以申请$n$这样的移动，目标不是完全修复数组，而是减少零的数量，使其最多变为$\lfloor k/2 \rfloor$。 

关键的难点在于每一次操作都会产生巨大的、结构化的效果。 一个动作就能产生准确的效果$k$连续的位置和重叠的移动以一种不平凡的方式相互作用，因为翻转以模 2 累加。我们没有被要求最小化操作，只是在允许的范围内构造任何有效的序列。 

约束条件$n \le 10^6$排除任何翻转窗口或每次操作重复扫描的二次模拟。 甚至一个$O(nk)$立即接近是不可能的。 解决方案必须是线性或接近线性的，并且必须避免重复访问先前的位置。 

一种幼稚的策略会尝试贪婪地逐个修复零，每当遇到零时就翻转窗口。 这会失败，因为旨在修复稍后位置的翻转可能会撤消之前的工作。 例如，如果$k=3$我们在位置 1 翻转以固定索引 2，然后在位置 2 翻转可能会在固定索引 3 的同时再次恢复索引 2。相互作用不是局部稳定的。 

另一个微妙的失败案例来自边界依赖性。 如果贪心算法总是在第一个零处翻转，它可能会将问题推向有效长度较少的末尾 -$k$段存在，最终达到剩余零不能被任何有效操作起始位置覆盖的状态。 

问题的真正结构是翻转是固定长度间隔上的累积奇偶校验操作。 这表明将该过程视为从左到右构建受控奇偶校验模式。 

## 方法

 暴力方法会模拟数组并尝试最多的所有可能序列$n$操作，选择减少零数量的有效操作。 每个操作都涉及翻转$k$元素，所以即使是单个模拟$O(n)$。 高达$n$操作，这导致$O(n^2)$时间，这远远超出了可行的范围$10^6$。 

关键的观察是，我们实际上不需要在全球范围内决定未来的行为。 相反，我们可以强制执行从左到右的不变量：一旦我们完成处理位置$i$，其最终值将不再受到影响。 如果我们使用跟踪当前覆盖每个索引的活动翻转数量的差异数组来维护先前翻转的效果，这是可能的。 

我们将每个操作解释为切换一个范围，并且不是直接修改数组，而是跟踪影响每个位置的翻转奇偶校验。 当我们到达索引时$i$，我们知道当前有效值是0还是1。如果是0，我们还有空间放置长度-$k$操作开始于$i$，我们立即应用它，确保索引$i$最终结果变为1。 这种贪婪的选择是安全的，因为后面的操作不会影响索引$i$如果我们总是向前移动并且仅在当前索引处或之后开始操作，那么再次。 

该过程将问题简化为维护滑动奇偶校验窗口并贪婪地决定是否在每个位置翻转。

 ## 算法演练

 我们维护一个差异数组，让我们知道有多少活动翻转会影响每个位置。 我们还维护一个运行奇偶变量。 

1.初始化一个数组`diff`大小的$n+1$跟踪翻转效果何时开始和结束，以及一个变量`cur`存储当前翻转奇偶校验。 还维护一个操作列表。 
2. 从索引迭代$i = 0$到$n-1$。 在每个位置，首先更新`cur`通过申请`diff[i]`，因为这告诉我们先前的翻转间隔是在这里开始还是结束。 此步骤确保`cur`表示当前元素是否已翻转奇数次或偶数次。 
3. 计算有效值$a[i]$作为$a[i] \oplus cur$。 这是我们在之前所有操作之后实际看到的值。 
4. 如果我们太接近终点了，意味着$i > n-k$，我们无法开始新的长度-$k$翻动。 在这种情况下，我们只是继续，因为我们无法再改变这个立场。 
5. 如果有效值在$i$为0，我们被迫立即修复它。 我们在位置开始翻转$i$， 记录$i+1$作为一个操作，并更新差异数组：我们在$i$并减 1$i+k$。 我们还更新了`cur`立即，因为翻转立即开始影响当前段。 
6. 继续扫描直至结束。 

关键的设计选择总是在当前状态错误的最早可能位置采取行动。 这可以防止将修正推迟到不再可能进行操作的区域。 

### 为什么它有效

 不变的是，在处理索引时$i$，所有指数严格小于$i$已最终确定其有效状态，并且不会再受到影响。 这是有保证的，因为每个操作都只在位置开始$\ge i$，所以未来的翻转不能向后延伸。 

每当我们遇到有效形式的零时，在该确切位置应用翻转可确保当前索引变为 1，并且任何干扰都会被推到仍可以纠正的未来位置。 由于我们从不重新访问早期的指数，因此该算法构建了一致的修正序列，而不会发生冲突。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, k = map(int, input().split())
    a = list(map(int, input().split()))

    diff = [0] * (n + 1)
    cur = 0
    ops = []

    for i in range(n):
        cur ^= diff[i]

        if i > n - k:
            continue

        if (a[i] ^ cur) == 0:
            ops.append(i + 1)
            cur ^= 1
            diff[i] ^= 1
            diff[i + k] ^= 1

    print(len(ops))
    if ops:
        print(*ops)

if __name__ == "__main__":
    solve()
```该解决方案依赖于差异数组来模拟每次操作在恒定时间内的范围翻转。 变量`cur`跟踪当前索引是否受到奇数个活动翻转的影响。 当放置新操作时，我们立即切换`cur`因为效果从当前位置开始。 

条件`i > n - k`防止超出数组边界的无效操作。 该算法从不尝试修复此后缀中的位置，因为没有合法的移动可以从那里开始。

 ## 边缘情况

 一种边缘情况是数组已经接近允许的零阈值。 例如，如果$k=4$,$\lfloor k/2 \rfloor = 2$，并且数组正好有两个零，算法必须避免不必要的翻转，否则可能会产生额外的零。 由于贪婪算法仅在遇到零时才翻转，并且从不主动引入翻转，因此它在此边界场景中保留了最优性。 

另一种情况是当后缀中出现零时，无法启动任何操作。 例如，如果$n=5$,$k=3$，最后两个元素为零，这些不能直接固定。 该算法自然会忽略它们，并且正确性取决于早期决策必须确保全局约束这一事实。 从左到右的不变量确保了这些位置所需的任何修复已经嵌入到早期的翻转决策中。 

最后一个微妙的情况是，在长期的情况下，一次翻转可能会暂时产生零。 贪婪不会在那里触发翻转，因为它只对有效形式的零做出反应，这可以防止对已经正确的区域进行不必要的干扰。
