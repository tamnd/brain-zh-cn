 ## 问题理解

 我们得到了一系列独立的观察结果，每个观察结果代表一个湖泊和其中的鱼的数量。 对于每个湖泊，我们必须决定 Kaitokid 是否可以安全地访问它。 安全的定义非常严格：只有完全没有鱼的湖泊才是可接受的。 任何鱼的数量为正数都表明该湖不适合。 

输入由一个整数 T 组成，描述了报告的湖泊数量。 之后，我们读取 T 个整数，每个整数的范围在 0 到 100 之间（包括 0 和 100）。 对于每个值，我们输出一个决策字符串，指示该特定湖泊是否安全。 

约束足够小，计算总量可以忽略不计。 即使最大 T 为 10^4，我们也仅对每个湖执行恒定时间检查。 这意味着直接线性扫描就足够了，任何具有 O(T) 复杂度的解决方案都可以在限制内轻松运行。 

不存在隐藏的结构复杂性，例如排序依赖性或跨湖交互。 每个湖泊都可以独立处理，这是简化的关键。 

唯一重要的边缘情况是鱼的数量恰好为零。 粗心的实现可能会意外地将零视为假值或根据语言结构错误地跳过它，但从逻辑上讲，这是唯一应该产生积极响应的情况。 

## 方法

 暴力解释仍然会逐一查看每个湖泊，检查其鱼数是否为零，然后打印结果。 无法减少检查次数，因为每个湖泊都必须至少检查一次才能产生相应的产出。 即使我们尝试先存储所有值并稍后处理它们，总工作量仍与 T 成正比。 

这里唯一真正的“优化”是认识到不需要额外的数据结构或预处理。 决策规则是每个元素进行一次比较：如果 x 等于 0，则打印 YES，否则打印 NO。 问题的结构已经与最优评估模式相匹配。 

强力视图和最优视图是一致的，但重要的认识是聚合、排序或前缀计算不会改变任何东西。 计算本质上是 O(T)。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 直接按湖检查 | O(T)| O(1) | O(1) | 已接受 |
 | 任何替代转换 | O(T)| O(T)| 已接受但没有必要 |

 ## 算法演练

 我们按照给定的顺序独立处理每个湖泊。 

1. 读取整数 T，它告诉我们必须评估多少个湖泊。 这定义了我们将执行的迭代次数。 
2. 对于接下来的 T 个整数中的每一个，读取代表单个湖泊中鱼类数量的 x 值。 
3. 检查x 是否等于0。 这是安全的定义条件，因为只有空湖才是可接受的。 
4. 如果 x 等于 0，则立即输出 YES。 否则输出NO。 
5. 重复此过程，直到处理完所有 T 湖。 

每个步骤背后的原因是没有湖泊影响任何其他湖泊，因此立即评估既安全又最佳。 

### 为什么它有效

 正确性依赖于问题定义和条件 x == 0 之间的直接等价性。该算法对每个输入值精确地计算该谓词一次，并生成相应的标签。 由于每个决策都是独立的并且完全由单个整数确定，因此迭代之间不存在不一致或交互影响的可能性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input().strip())
    for _ in range(t):
        x_line = input().strip()
        if not x_line:
            x_line = input().strip()
        x = int(x_line)
        if x == 0:
            print("YES")
        else:
            print("NO")

if __name__ == "__main__":
    solve()
```该实现首先读取 T，然后循环处理每个湖。 每个输入行都被解析为一个整数并检查零。 条件是解决方案的核心，直接反映了有效湖的数学定义。 

唯一微妙的实现细节是确保在转换之前正确删除每个输入行，这可以避免在某些环境中出现尾随空格或空读取的问题。 立即打印输出，这避免了不必要的存储结果。

 ## 边缘情况

 一个关键的边缘情况是鱼的数量恰好为零。 该算法明确将此视为唯一的接受条件。 用于输入`0`，条件 x == 0 计算结果为 true，输出为 YES，确认正确处理可接受和不可接受湖泊之间的边界。 

另一种边缘情况是所有值都不为零时。 例如，输入`3 1 2 3`每个步骤都会产生“否”，因为没有一个值满足接受条件。 该算法不依赖于正负性，只依赖于精确相等，因此不存在歧义。 

最后，正确处理 T = 1 时的最小可能输入，因为循环只执行一次并生成单个决策，而不需要任何初始化或累加逻辑。
