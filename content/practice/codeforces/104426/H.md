 ## 解决方案
 ## 问题理解

 给定两个长度的序列$n$。 第一个序列代表 Naseem 实际购买的冰沙，第二个序列代表每个朋友最希望收到的东西。 哪个朋友收到哪种冰沙没有限制，因此我们可以自由地在朋友之间分配购买的冰沙。 我们的目标是最大限度地增加多少朋友最终收到完全符合他们喜好的冰沙类型。 

另一种看待这个问题的方法是我们有两个大小的多重集$n$：一种来自购买，一种来自偏好。 我们希望将第一个多重集中的元素与第二个多重集中的元素进行匹配，从而最大化相等对的数量。 

限制条件$n, m \le 10^5$意味着任何比线性或线性时间更差的解决方案都不起作用。 所有对之间的二次比较方法将涉及最多$10^{10}$最坏情况下的操作，远远超出了极限。 这促使我们进行基于频率的推理，而不是成对匹配。 

当一侧存在大量重复项而另一侧却没有时，就会出现一个微妙的陷阱。 例如，如果所有购买的冰沙都是类型 1，但朋友想要许多不同的类型，则尝试按顺序匹配而不按类型分组的天真贪婪分配很容易根据遍历顺序对匹配进行少计数或多计数。 另一个问题是假设索引对齐很重要，而实际上我们可以任意排列。 

## 方法

 暴力解释是尝试以各种可能的方式将冰沙分配给朋友，并计算我们可以获得多少匹配。 这对应于生成两个数组之间的所有赋值排列。 对于每个排列，我们计算有多少个位置匹配。 这是正确的，因为它探索了整个解决方案空间，但它大致呈阶乘增长$n!$，即使对于$n = 20$。 

一种更结构化的暴力改进是固定朋友的顺序，并为每个朋友尝试分配任何未使用的冰沙，递归地最大化匹配。 这仍然是指数级的，因为每个步骤分支最多$n$选择。 

关键的观察是元素的同一性仅通过频率起作用。 对于给定的冰沙类型$x$，假设出现$c_a[x]$购买清单中的次数以及$c_b[x]$时间在偏好中。 不管怎样排列，最多只能匹配$\min(c_a[x], c_b[x])$类型的朋友$x$，因为每场比赛都会消耗一份冰沙和该类型的一种偏好。 相反，这个界限可以通过简单地在每种类型内进行贪婪配对来实现。 

这将问题从全局分配问题减少为每个值的独立计数问题。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O(n!)$|$O(n)$| 太慢了|
 | 频率匹配 |$O(n)$|$O(m)$| 已接受 |

 ## 算法演练

 我们通过计算每种冰沙类型在两个数组中出现的次数并对最小重叠次数求和来解决该问题。 

1. 为购买的冰沙阵列构建频率图$a$。 这捕获了每种类型我们有多少可用项目。 
2. 为偏好数组构建频率图$b$。 这捕获了有多少朋友想要每种类型。 
3. 对于任一图中出现的每种类型，计算匹配数作为其两个频率中的最小值。 
4. 将这些最小值累加为最终答案并输出结果。 

每个步骤都是由匹配在不同类型之间独立的想法驱动的。 一旦分配了冰沙，就无法重复使用，因此按类型计数可以精确模拟资源消耗。 

### 为什么它有效

 对于每个值$x$，任何有效的分配最多只能配对一种类型的 Smoothie$x$和一位想要打字的朋友$x$。 因此，该类型的有效对数量的上限受供应商和供应商的限制。$c_a[x]$和需求$c_b[x]$。 取最小值同时满足两个约束。 对所有类型求和是有效的，因为不同类型永远不会竞争同一项目。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    a = list(map(int, input().split()))
    b = list(map(int, input().split()))

    freq_a = {}
    freq_b = {}

    for x in a:
        freq_a[x] = freq_a.get(x, 0) + 1

    for x in b:
        freq_b[x] = freq_b.get(x, 0) + 1

    ans = 0
    for x in freq_a:
        if x in freq_b:
            ans += min(freq_a[x], freq_b[x])

    print(ans)

if __name__ == "__main__":
    solve()
```该解决方案将计数分为两个字典。 一种微妙的实现选择是仅迭代`freq_a`并检查是否存在`freq_b`，这避免了迭代整个可能的冰沙类型范围，直到$m$，其中可能包括未使用的值。 

另一个重要的细节是使用`get`进行频率更新以避免关键错误并保持代码整洁和线性。

 ## 边缘情况

 关键的边缘情况是一种类型仅出现在一侧。 例如：

 输入：```
3 3
1 1 1
2 2 2
```算法构建`freq_a = {1: 3}`和`freq_b = {2: 3}`。 由于键不存在交集，因此循环没有任何贡献，答案为 0，这与现实相符，因为没有朋友的偏好能够得到满足。 

另一个边缘情况是严重不平衡：

 输入：```
5 5
1 1 1 1 1
1 1 2 2 2
```频率给出：

 类型 1 贡献 min(5, 2) = 2，类型 2 隐式贡献 min(0, 3) = 0，因此答案为 2。 该算法正确地避免了将额外的类型 1 平滑过度分配给类型 2 偏好，因为它从不跨键混合类型。 

这些案例证实，即使在极端倾斜的情况下，基于频率的分解也能保持正确性。
