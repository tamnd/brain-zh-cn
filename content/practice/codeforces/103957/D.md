 ## 问题理解

 我们从一张有价的钞票开始$A$，我们希望最终能够准确支付$B$， 在哪里$A > B$。 唯一可用的工具是自动售货机：我们花一些钱购买商品，机器会找零。 关键的困难在于，这种变化在结构上是不可预测的，只不过它总是由有效的中国货币面额组成，并且总和总是精确到正确的余数。 

我们被允许通过以受控方式多付钱来“制造”一套有用的硬币。 经过一系列这样的操作后，我们希望保证我们可以组装硬币，总和精确到$B$，无论机器如何在有效面额之间分配零钱。 

所有值都限制在中国货币面额的标准集合内，因此状态空间不是连续的。 问题本质上是在问：我们必须向系统注入的“废物”的最小量是多少，以便在最糟糕的变化分解中，我们仍然可以重建形成所需的每个单元$B$。 

就不同的面额而言，约束很小，而不是就测试用例而言，因此$O(1)$或者每个测试用例的有限计算量非常小。 任何依赖于探索组合或模拟多个步骤的交换的方法，如果以面额或中间状态扩展，都会太慢，但这里的固定货币系统强烈建议采用贪婪或数字结构。 

出现微妙的边缘情况时$A$只是略大于$B$。 例如，如果$A = 1$和$B = 0.99$，如果假设任意改变行为，则最优策略并不明显。 一个天真的想法可能会尝试模拟分裂硬币或假设确定性变化，但变化分布的对抗性迫使我们根据最坏情况的分解进行推理。 

## 方法

 暴力解释会将购买物品的每种可能方式视为生成一些多组硬币，然后检查该多组是否总是可以重新排列成$B$。 由于每次购买都可能产生许多可能的变化结果，因此这很快就会变成一个涵盖所有组合的分支过程$A - x$， 在哪里$x$是选择的购买价格。 即使对于单个步骤，有效的变化分解的数量也很大，因为任何有效的面值组合求和到余数都是可能的。 将其扩展到多次购买会使状态空间组合爆炸，因为我们需要跟踪所有可到达的多组硬币。 这显然是不可行的。 

关键的观察是，我们实际上并不关心变化的确切分布，只关心是否能够保证形成$B$在最坏的情况下。 这将问题转化为固定规范硬币系统的覆盖问题。 关键的结构是，货币面额是规范的，并且由 2 和 5 的因子嵌套。这意味着每个金额都可以标准化为贪婪表示，并且变化分布的不确定性可以减少到每个面额级别是否被充分“覆盖”。 

我们考虑的不是模拟变化，而是确保对于代表所需的每个面额级别$B$，无论零钱如何分配，我们都可以强制至少有一枚该级别的硬币可用。 这导致了由货币结构定义的混合基础系统中的逐位推理。 

最优策略简化为分析距离$A$来自$B$面额门槛，并确定保证跨越这些门槛所需的最低超额付款。

 ## 算法演练

 我们首先将所有值标准化为整数 fen 单位以避免浮点问题。 每个面额都是1分的整数倍，形成一个固定的集合。 

1. 转换$A$和$B$转换为整数 fen 值。 

这消除了精度问题并允许对硬币价值进行直接算术。 
2. 计算差距$D = A - B$。 

这代表了我们拥有的总闲置量，任何策略都必须在这个盈余范围内运作。 
3. 确定会导致代表性不稳定的最小面额阈值。 

我们从最大到最小扫描面额，并推断差距是否跨越了边界，在该边界上，零钱会不可预测地分裂成更小的硬币。 
4.确定最低超额支付额$x$这样付款后$x$，最坏情况的变化仍然允许至少重建$B$。 

直觉是，稍微多付钱会迫使收银员在找零时加入较高面额的硬币，因为否则较低面额的硬币无法在对抗性分割中正确求和。 
5.输出最小的这样的$x$。 

关键的结构步骤是认识到自动售货机的对抗性分裂只在面额边界上起作用。 一旦我们确保有足够的“空间”来迫使进位进入更高的面额，所有较小的面额都将变得可重构。 

### 为什么它有效

 中国的面额体系是等级制的，每一个金额在以分为单位时都可以以独特的方式贪婪地分解。 对手只能影响零钱的分配方式，而不能影响代表余数的每个面额规模的总量。 通过确保购买后的剩余价值不能纯粹以较低面额表示，而不强制使用至少一枚较高面额的硬币，我们保证生成的多重集始终跨越可以重新排列为的规范表示$B$。 该算法有效地确保每个必要的面额级别都强制存在，无论零钱如何分配。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

DENOMS = [10000, 5000, 2000, 1000, 500, 200, 100, 50, 20, 10, 5, 2, 1]

def to_fen(x):
    # input is decimal like 0.01, 1, 0.1 etc
    return int(round(float(x) * 100))

def solve_case(A, B):
    a = to_fen(A)
    b = to_fen(B)

    if a == b:
        return 0

    # we want minimal x such that after spending x,
    # remaining structure always allows forming b
    # key idea: check smallest "forcing" overpay
    need = a - b

    # try candidate answer from smallest denomination upward
    # (since answer must be a denomination)
    ans = None

    for d in DENOMS[::-1]:
        # try forcing at this denomination scale
        # minimal spend that forces rounding/carry behavior
        if d >= need:
            ans = d - (need % d if need % d != 0 else d)
            break

    return ans if ans is not None else need

def main():
    T = int(input())
    for tc in range(1, T + 1):
        A, B = input().split()
        res = solve_case(A, B)
        print(f"Case #{tc}: {res/100:.2f}")

if __name__ == "__main__":
    main()
```该解决方案将所有货币值转换为整数分单位，以便面额推理变得准确。 核心计算是由只有面额边界重要的观察驱动的，因此我们从大到小迭代可能的阈值。 

变量`need`代表了我们所拥有的与我们必须能够构建的之间的差异。 循环结束`DENOMS`试图找到可以发生强制携带的最小面额比例。 表达式`d - (need % d)`捕获了我们必须调整支出的数量，以便剩余部分与面额边界保持一致，这确保了最坏情况的变化仍然保留了可构造性$B$。 

输出被格式化回 CNY，具有两位小数精度，与问题陈述相匹配。

import sys, io

def run(inp: str) -> str:

 ## 边缘情况

 当之间的差异时，就会出现微妙的边缘情况$A$和$B$已经是一个面值了。 在这种情况下，算法应该正确返回零，因为没有对抗性分裂可以阻止形成$B$。 

例如，使用输入$A = 1$,$B = 0.5$，我们有$need = 0.5$。 由于0.5是有效面额，任何找零都必须包含可以直接使用的硬币，所以不需要额外的支出。 

另一个边缘情况是当$A$和$B$相差很小，如 1 分。 这里的算法确保我们始终考虑最小面额级别，保证即使是对抗性分裂也不会破坏重建所需金额的能力。
