 ## 解决方案
 ## 问题理解

 我们得到了一个用高度数组表示的一维景观。 每个索引是山脉中的一个位置，高度值是该位置的海拔高度。 当降雨时，水被困在较高的边界点之间，在较低地形的邻近区域形成“水池”。 

位置上方的空间单位`i`如果在其左侧某处存在更高或相等的高度并且在其右侧某处存在更高或相等的高度，则属于一个池。 直观地说，水池是水被困在两壁之间的区域。 水池的表面高度由其两个边界墙中较低的一侧决定，因为水不能上升到较短的一侧之上。 

每个水池都有一个体积，定义为该边界区域内地面上方截留的水总量。 在所有这些池中，我们被要求识别那些表面高度最大的池，然后输出它们的体积之和。 如果多个不相交的池达到相同的最大表面高度，则它们的体积会加在一起。 

主要困难在于池的定义取决于全局结构：仓位两侧是否有界，以及限制边界高度是多少。 这使得朴素的本地推理变得不可靠。 

约束条件`n ≤ 100000`排除任何检查每个子数组并重复重新计算边界的方法。 间隔内的二次或三次扫描已经太慢了，甚至`O(n log n)`除非精心设计，否则需要大量重新计算的分治法会带来 TLE 风险。 

一些边缘情况凸显了朴素区间枚举失败的原因。 

考虑一个严格递增的数组，例如`1 2 3 4`。 两边都没有位置限制，因此不存在池，答案为零。 任何假设局部最小值之间的每个山谷形成一个池的方法都会错误地产生非零输出。 

考虑一个平坦的高原，例如`5 5 5 5`。 每个点都有相同的邻居，因此天真的解释可能会将整个段视为高度为 5 的水池，但由于水无法积聚在地形本身之上，因此不存在陷阱。 

考虑交替的峰值，例如`5 1 5 1 5`。 存在多个小盆地，但它们的表面水平因局部边界最小值而异，因此错误地合并所有山谷会导致过度计算。 

关键的挑战是正确识别独立的有界区域并有效计算其截留的水和峰值表面高度。 

## 方法

 暴力方法会尝试每一个可能的部分`[l, r]`，判断是否形成有效流域，计算其有效水位为`min(max on left boundary, max on right boundary)`，然后计算内部截留体积。 对于每个部分，扫描其内部以计算水的贡献将花费线性时间。 这导致大约`O(n^3)`行为，因为有`O(n^2)`各个部分都需要`O(n)`加工。 即使优化为预先计算范围最大值，检查所有边界对仍然存在`O(n^2)`太慢了`n = 10^5`。 

问题的结构提出了一个不同的观点：我们不应该枚举所有部分，而应该确定实际形成有意义的盆地的边界。 每个水池基本上都是由一对边界“墙”决定的，其中水位由两个墙中较低的一个固定，并且中间的所有东西都填充到该水位。 

这正是单调堆栈所公开的结构。 当从左到右扫描时，我们可以保持高度递减的堆栈。 每当我们遇到高于顶部的高度时，我们就找到了盆地的正确边界。 弹出的元素成为盆地底部，当前高度与新的堆栈顶部一起形成两堵墙。 可以使用这些边界在本地计算捕获的水，并且每个位置最多处理一次。 

这减少了从推理任意间隔到精确处理每个谷被右边界闭合时的问题。

 | ---| ---| ---| ---|
 | 蛮力 | O(n3) | O(1) | O(1) | 太慢了 |
 | 单调堆栈 | O(n) | O(n) | 已接受 |

 ## 算法演练

 我们从左到右处理数组，同时维护高度严格按递减顺序排列的索引堆栈。 

1. 初始化一个空堆栈和一个变量来累积池的结果。 
2. 迭代每个索引`i`从左到右。 对于每个位置，我们尝试“关闭”任何以`i`。 当当前高度大于堆栈顶部的高度时，我们就找到了捕获区域的右边界。 
3. 当堆栈不为空且`h[i] > h[stack[-1]]`，弹出顶部元素。 这个弹出的索引代表了潜在池的底部。 如果出栈后栈变空，则没有左边界，所以我们停止处理这个盆地。 
4. 如果出栈后栈不为空，则新的栈顶成为左边界。 当前指数`i`是右边界。 水位由下式确定`min(h[stack[-1]], h[i])`。 滞留水的宽度从`stack[-1] + 1`到`i - 1`。 
5. 该盆地部分贡献的体积是通过将水填充至水位并减去地面高度来计算的。 这会累积到全球总数中。 
6.推送当前索引`i`入栈并继续。 

关键的微妙之处在于，每个弹出的元素对应于一个盆地，该盆地在右边界出现时就完全解析了。 这可以防止重复计算，因为一旦计算了盆地，就永远不会重新访问它。 

### 为什么它有效

 堆栈以高度递减的顺序维护索引，这保证了每次我们找到较高的条形时，我们都会关闭依赖于其间较短的条形的所有盆地。 每个盆地都由可以从两侧包围盆地的第一对边界来唯一标识。 由于边界仅在出现更高或相等的条时才有效，因此不会部分计算或复制任何盆地。 每个被困区域在遇到其正确边界时都会被确定一次。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    h = list(map(int, input().split()))
    
    stack = []
    water = 0
    
    for i in range(n):
        while stack and h[i] > h[stack[-1]]:
            mid = stack.pop()
            if not stack:
                break
            left = stack[-1]
            width = i - left - 1
            bounded_height = min(h[left], h[i]) - h[mid]
            if bounded_height > 0:
                water += bounded_height * width
        stack.append(i)
    
    print(water)

if __name__ == "__main__":
    solve()
```该解决方案使用单调递减的索引堆栈。 每当我们遇到打破单调性的高度时，我们都会弹出山谷并计算当前索引和新堆栈顶部之间的截留水。 

重要的实现细节是计算`bounded_height`。 它从限制边界高度中减去山谷高度，确保我们只计算实际水量，而不是负面贡献。 宽度对应于边界之间的水平跨度，不包括边界本身。 

条件`if not stack: break`是必不可少的，因为如果没有左边界，当前弹出的元素就不可能捕获。

 ## 边缘情况

 平坦的地形，例如`5 5 5 5`穿过堆栈而不会形成有效的左右边界对。 每个元素都被压入，但没有弹出操作产生有效的有界区域，因此累积的水量保持为零。 

严格递增的数组，例如`1 2 3 4`不断推高指数而不会触发盆地关闭。 由于堆栈永远不会形成先减后增的模式，因此不会检测到捕获区域，并且输出正确地为零。 

像之字形图案`5 1 5 1 5`产生多个小盆地。 当出现更高的边界时，每个谷值都会准确闭合，并且每个贡献在其闭合步骤中计算一次，从而防止重叠或重复计算。
