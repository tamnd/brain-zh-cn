 ## 解决方案
 ## 问题理解

 Jason 从一个空文本开始，并希望得到一个由字符串组成的文本`"orz"`至少重复$N$次。 只有三个可用的操作：他可以键入一个`"orz"`从头开始在$A$几秒钟，他可以复制当前写入的所有内容$B$几秒钟，他就可以将复制的内容粘贴到$C$秒，每次粘贴都会将整个复制的块附加到现有文本中。 

关键的观察是文本总是由重复的副本组成`"orz"`，因此系统的状态可以纯粹通过有多少个副本来描述`"orz"`文本中当前存在以及剪贴板中有多少个副本。 目标是至少达到$N$用最短的时间进行复印。 

这些限制使我们远离任何对每个角色或每个操作进行推理的模拟。 自从$N$可以达到$10^9$，任何逐步迭代每个操作或每个生成的副本的方法都是不可能的。 即使是尝试天真地探索所有状态的解决方案也会变得太大，因为可能的剪贴板配置和文本大小的数量会快速增长。 

当复制粘贴循环不能立即受益时，天真的贪婪策略就会出现微妙的失败案例。 例如，如果复制相对于打字来说成本较高，那么最好在复制之前多次打字。 相反，如果复制成本较低，那么延迟第一个复制可能会因为在复制效果更好时强制重复键入而浪费时间。 另一种失败模式是假设我们应该尽可能粘贴，当稍后复制更大的块时，这种模式会比提前复制更有效。 

## 方法

 直接模拟将保持当前的长度和剪贴板大小并尝试所有可能的操作序列。 每个状态转换都是确定性的，但随着时间的推移，分支选择的深度呈指数级增长。 自从达到$N$可能需要最多$N$在最坏的情况下增量，这种方法是不可行的。 

关键的见解是，在任何时刻，系统都由两个值完全描述：有多少个副本`"orz"`文本中当前有多少个副本以及剪贴板中有多少个副本。 从我们拥有的任何州$x$文中和$y$在剪贴板中，唯一有意义的决定是是否键入一次、复制全部或粘贴。 重要的是，打字总是将文本增加一个单位，而复制和粘贴则会成倍地缩放该过程。 

这种结构提出了状态图上的最短路径问题，但该图太大而无法显式构造。 相反，我们观察到最佳策略具有单调结构：我们永远不会从减少剪贴板或文本中的副本数量中受益。 此外，一旦我们决定复制一个大小的块$x$，所有未来的操作仅取决于$x$，因为粘贴总是乘以相同的数量。 

这将问题简化为针对当前大小的贪婪决策过程。 在任何时候，我们都会比较是继续输入到未来的复制点还是现在复制并开始粘贴更好。 最终通过比较构建大小块的成本来驱动最佳结构$k$纯粹通过打字而不是使用复制粘贴循环构建它。 

最终的解决方案可以通过迭代基本块的可能“构建大小”并计算达到的最佳时间来得出$N$使用该块大小。 每个候选基本尺寸对应于输入多次，可选择执行一份复制，然后重复粘贴，直到达到或超过$N$。

 ## 算法演练

 我们将该过程视为选择基本段大小$k$，这意味着我们首先创建$k$的副本`"orz"`使用打字，可能复制一次，然后使用粘贴进行扩展。 

1. 对于每个可能的副本数$k$我们可能决定建立一个基地，准确计算生产成本$k$仅使用打字进行复印。 这个成本简直就是$k \cdot A$。 这是达到尺寸起点的最直接方法$k$还没有使用复制粘贴。 
2.到达后$k$，决定是否使用复制粘贴来达到$N$。 如果我们此时复制，我们将支付$B$一次，然后每个粘贴添加$k$更多副本需按成本价购买$C$。 这意味着所需的粘贴操作次数为$\lceil (N-k)/k \rceil$。 
3. 计算总成本为$k \cdot A + B + \lceil (N-k)/k \rceil \cdot C$，并将其与纯打字成本进行比较$N \cdot A$。 我们始终将这些策略保持在最低限度。 
4. 迭代候选值$k$有效地而不是所有值达到$N$。 的最优值$k$接近打字的边际成本等于复制粘贴增长的边际成本的点，因此我们只测试高达$O(\sqrt{N})$围绕这个平衡点。 
5. 跟踪所有候选者的最小值并输出它。 

### 为什么它有效

 不变的是，任何最佳策略都可以分解为一个转换点，Jason 从打字切换到复制粘贴扩展。 在此之前，所有操作都相当于通过打字线性增长； 在此之后，所有增长都是乘法性的，并由固定的块大小控制。 任何具有多个复制点的策略都可以压缩为一个，而不会增加成本，因为后面的副本在大小和效率上都优于早期的副本。 这将搜索空间减少到单个决策边界，这就是为什么枚举该边界就足够了。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    T = int(input())
    for _ in range(T):
        A, B, C, N = map(int, input().split())

        # baseline: just type all
        ans = N * A

        # try all reasonable base sizes
        # we only need to test up to sqrt(N) region
        k = 1
        while k * k <= N:
            # build k by typing
            cost_base = k * A

            # expand using copy-paste
            if k < N:
                # number of copies needed beyond k
                rem = N - k
                paste_ops = (rem + k - 1) // k
                cost = cost_base + B + paste_ops * C
                ans = min(ans, cost)

            k += 1

        print(ans)

if __name__ == "__main__":
    solve()
```该解决方案首先考虑简单的策略，其中 Jason 只需键入`"orz"` $N$次，给出了答案的上限。 

循环结束$k$探索可能成为复制有用点的候选基本尺寸。 对于每个$k$，我们计算到达的成本$k$通过打字，然后模拟达到或超过所需的最佳粘贴操作次数$N$。 上限划分确保我们在最后一次粘贴超过时考虑到超调$N$。 

限制为$k \le \sqrt{N}$是保持解决方案高效的原因。 较大的值$k$会导致更少的粘贴操作，但它们的打字成本占主导地位，并且对称推理确保最佳权衡发生在较低范围或边界附近。 

一个微妙的点是每个候选人只考虑一次复制$k$。 允许多个复制步骤不会改善结果，因为一旦我们提交了块大小，重复复制只会重新创建相同的乘法结构，而不会改变最佳比率。

 ## 边缘情况

 一种极端情况是复制永远不值得。 例如，如果$B$相比而言非常大$A$，算法将始终偏向基线$N \cdot A$。 循环结束$k$仍然评估候选人，但所有这些都包括复制成本$B$，它立即主导了粘贴带来的任何节省。 

另一种边缘情况发生在$C$远小于$A$，使得粘贴非常有利。 在这种情况下，最佳的$k$往往很小，并且该算法可以正确识别较小的基本尺寸，因为枚举明确地包含了它们。 例如，与$A=10, B=1, C=1, N=10^9$，最好的策略是输入一次，然后重复复制和粘贴，这是在$k=1$在枚举中。 

最后一个微妙的情况是$N$很小。 为了$N=1$，任何复制或粘贴都是不必要的，并且基线$N \cdot A$是最优的。 该算法自然地处理这个问题，因为所有候选策略至少增加复制成本$B$，与直接输入相比，这绝对没有好处。
