 ## 解决方案
 ## 问题理解

 我们在酒店休息室里保持着一条充满活力的客人队列，但这条队列的结构很不寻常。 每个人都有一个唯一的标识符，我们必须支持修改或查询当前排序的三种操作。 

与首选熟人一起到来的客人并不只是简单地加入到最后。 如果他们的朋友当前在休息室，他们会立即插入该朋友之后。 否则，它们将附加到行尾。 客人也可以随时离开，无论他们在队列中，我们都必须将他们带走。 最后，我们必须重复报告当前排队的第一人。 

关键的难点在于该结构不是一个简单的队列。 插入取决于定位任意现有元素，而删除可以发生在任何地方。 对于多达 200,000 次操作，任何扫描列表以查找插入点或删除的解决方案都会太慢，因为最坏情况的行为将​​变成二次方。 

使用 Python 列表或双端队列进行的简单模拟会失败，因为定位朋友或删除中间元素需要线性时间。 在最坏的情况下，重复`in x y`操作，其中`y`接近结束时将强制重复进行全面扫描，并且`out x`需要再次扫描才能定位`x`。 

当队列为空或引用的朋友不存在时，会出现微妙的边缘情况。 例如：

 输入：```
check
```输出：```
-1
```任何解决方案都必须显式处理空结构查询。 

另一个边缘情况是同一个人重复到达。 该语句允许访客随着时间的推移多次出现，但每个标识符在任何时刻在当前队列中都是唯一的。 粗心的实现假设跨时间的唯一性而不跟踪存在可能会尝试重复插入或删除失败。 

## 方法

 直接模拟建议将队列存储在列表中，并且对于每个队列`in x y`,扫描查找`y`并在其后插入。 删除同样需要扫描才能找到`x`。 虽然概念上很简单，但在最坏的情况下每个操作的成本为 O(n)。 如果有多达 200,000 次操作，这会退化为 O(n²)，这太慢了。 

关键的观察是我们不需要存储实际的连续序列。 我们只需要维持前任和后继关系即可。 每个元素在双向链接结构中都有唯一的位置，我们需要通过标识符快速访问节点。 

这自然会导致链表表示与从值到节点的哈希映射相结合。 哈希映射允许任何人 O(1) 访问，而链表允许一旦我们有了节点引用就可以 O(1) 插入和删除。 

该结构成为具有显式头指针和尾指针的动态双向链表。 每个客人都是一个节点。 当在某人之后插入时，我们会在该人与其下一个邻居之间拼接一个新节点。 删除时，我们直接重新连接邻居。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力破解列表模拟| O(q²) | O(q) | 太慢了 |
 | 哈希图+双向链表 | O(q) | O(q) | 已接受 |

 ## 算法演练

 我们维护一个双向链表，其中每个节点存储一个访客 ID，以及指向上一个和下一个节点的指针。 我们还维护一个将 id 映射到节点的字典。 

1. 初始化空结构体，将头指针和尾指针设置为空，以及一个空字典。 
2. 对于一个`in x y`操作，检查字典中是否存在y。 
3. 如果 y 存在，则通过重新连接指针将 x 直接插入到节点 y 之后。 如果 y 是尾部，则更新尾部。 
4. 如果 y 不存在，则将 x 追加到列表末尾并相应更新 tail。 如果list为空，head和tail都变成x。 
5. 对于`out x`，从字典中检索 O(1) 中的节点 x，然后重新连接其上一个和下一个邻居，如果需要更新头或尾，并从字典中删除 x。 
6. 对于`check`，如果存在则输出head id，否则输出-1。 

每次插入或删除仅涉及固定数量的指针，因此每个操作的运行时间复杂度为 O(1)。 

### 为什么它有效

 不变的是，双向链表始终代表客人的当前确切顺序，并且字典始终将每个当前客人 ID 映射到其相应的节点。 每次更新仅通过重新连接本地链接来保留邻接关系，而不会在全局范围内重建结构。 由于每个操作仅修改或查询固定数量的节点，因此结构保持一致并有效支持所有所需的操作。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class Node:
    __slots__ = ("val", "prev", "next")
    def __init__(self, val):
        self.val = val
        self.prev = None
        self.next = None

def solve():
    q = int(input())
    pos = {}

    head = None
    tail = None

    out = []

    for _ in range(q):
        cmd = input().split()

        if cmd[0] == "in":
            x = int(cmd[1])
            y = int(cmd[2])

            node = Node(x)
            pos[x] = node

            if y in pos:
                ynode = pos[y]
                nxt = ynode.next

                node.prev = ynode
                node.next = nxt
                ynode.next = node

                if nxt:
                    nxt.prev = node
                else:
                    tail = node
            else:
                if tail is None:
                    head = tail = node
                else:
                    tail.next = node
                    node.prev = tail
                    tail = node

        elif cmd[0] == "out":
            x = int(cmd[1])
            node = pos.pop(x)

            pv = node.prev
            nx = node.next

            if pv:
                pv.next = nx
            else:
                head = nx

            if nx:
                nx.prev = pv
            else:
                tail = pv

        else:  # check
            if head:
                out.append(str(head.val))
            else:
                out.append("-1")

    sys.stdout.write("\n".join(out))

if __name__ == "__main__":
    solve()
```该解决方案的核心是显式的节点结构。 每个节点仅存储本地指针，所有全局结构都通过`head`,`tail`，以及哈希图。 

在插入过程中，我们要么在现有节点之后进行拼接，要么在尾部追加。 关键细节是正确处理以下情况`y`是当前尾部，因为在这种情况下，新节点将成为新尾部。 

在删除过程中，我们仔细更新邻居并调整`head`和`tail`删除边界节点时。 忘记更新这些指针之一是错误输出的常见原因。 

字典`pos`确保定位任何人的时间恒定，避免全面扫描。

 ## 边缘情况

 一个重要的案例是在一位失踪的朋友之后重复插入。 例如，如果我们插入`x`和`y`不存在，`x`必须走到最后。 The algorithm handles this by checking`y in pos`在任何指针逻辑之前，确保一致地使用追加行为。 

另一种情况是删除当前头。 删除头节点时，`prev`指针为空，所以我们必须显式移动`head`到`node.next`。 链接结构仍然有效，因为下一个节点的`prev`已正确设置或更新后变为空。 

最后，删除尾节点时，我们更新`tail`到`node.prev`。 如果不这样做，将来在末尾的插入将附加到过时的指针，从而破坏序列。
