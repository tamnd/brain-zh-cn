 ## 解决方案
 ## 问题理解

 有$n$厨房里有不同的香料，每种香料都有一个名称。 每天的菜肴都是精心挑选的$m$不同的香料，但我们不知道选择了哪些。 

我们得到了一个重要的观察：一名助理厨师对一组已知的过敏$k$香料，已经尝过这道菜，没有出现任何过敏反应。 这限制了未知的菜肴成分：没有选择的菜品$m$香料可能属于帮手的过敏组。 

实际上，这道菜是尺寸的某个未知子集$m$，但只能使用对助手安全的香料。 

现在我们得到了$p$客人。 每位客人都有自己的过敏名单。 对于每位客人，我们必须确定该菜肴是否可能引发过敏的结论。 由于实际的菜肴并不唯一，我们对所有有效的菜肴进行推理，与助手的观察一致。 

对于每位客人来说，可能会出现三种结果。 如果每道有效的菜肴都避免了所有过敏原，那么答案是“否”，这意味着该菜肴对他们来说是安全的。 如果每道有效的菜肴都必须含有至少一种过敏原，那么答案是“是”，这意味着该菜肴肯定会引发反应。 否则，两种结果都有可能，具体取决于未知数如何$m$-subset被选择，所以答案是“MAYBE”。 

限制条件$n \le 100$,$p \le 100$，并且小字符串集表明我们可以进行基于集合的推理，甚至可以对每个查询进行重新计算，而不必担心简单计数和散列之外的渐近复杂性。 

一个微妙的一点是，助手的测试完全排除了一些香料。 他们过敏名单上的任何香料都保证不会出现在菜肴中，因此仅与那些被移除的香料重叠的客人过敏变得无关紧要。 

另一个常见的陷阱是独立对待每位客人，而不以有效香料的数量减少为条件。 盘子不是大小的任何子集$m$来自所有人$n$香料，但仅限于那些不属于助手过敏范围内的香料。 

## 方法

 一种强力的解释是枚举所有可能的有效菜肴：首先过滤掉助手的过敏香料，然后生成所有组合$m$剩下的一组香料，并为每位客人检查这些组合中是否有任何组合与他们的过敏清单相交叉。 这立即变得不可行，因为即使$n = 100$, 组合数$\binom{100}{50}$规模如天文数字，我们最多会对 100 名客人进行重复检查。 

关键的观察是我们永远不需要明确地构建盘子。 所有有效的菜肴都是简单的$m$-固定大小的缩减宇宙的子集$n - k$。 对于特定的客人来说，只有两个结构属性很重要：有多少允许的香料不在他们的过敏集中，以及他们的任何过敏香料是否有资格出现在菜肴中。 

一旦我们将每位客人的过敏设置与安全宇宙相结合，一切都会简化为简单的计数。 我们只追踪该客人存在多少“危险但仍然可能”的香料。

 ## 算法演练

 让助手的过敏香料定义一组禁忌。 香料的有效宇宙是除了这些违禁物品之外的一切。 

对于每位客人，我们将其过敏列表中的香料分为两类：那些仍然存在于有效宇宙中的香料和那些已经被帮助者的约束排除在外的香料。 只有第一类会影响答案。 

1. 构建从香料名称到索引的映射，以便我们可以高效地使用集合而不是字符串。 这允许恒定时间的成员资格检查。 
2. 阅读帮手的过敏设定，并将所有这些香料标记为禁用。 有效的宇宙是所有不在这个集合中的香料，它的大小是$N' = n - k$。 
3. 对于每位客人，计算有效宇宙中仍然存在多少过敏香料。 调用这个值$x$。 这代表了菜肴中实际出现并仍然触发它们的香料的数量。 
4. 计算宇宙中对于这位客人还剩下多少安全香料，即$N' - x$。 这些香料可以在菜肴中使用而不会引起过敏反应。 
5.如果$x = 0$，那么客人的任何过敏原都不能出现在任何有效的菜肴中。 每道有效的菜肴都会避免它们，所以答案是“不”。 
6. 否则，如果$N' - x < m$，那么就无法选择$m$不含至少一种过敏原的香料。 每道有效的菜肴都必须含有对他们来说危险的东西，所以答案是“是”。 
7. 在所有剩余的情况下，完全安全的选择和危险的选择都存在，所以答案是“也许”。 

### 为什么它有效

 所有有效菜肴均统一$m$-固定大小宇宙的子集$N'$。 客人避免对特定菜肴产生过敏反应的唯一方法是选择的子集避免所有菜肴$x$他们仍然可能的过敏原。 这是否总是可能仅取决于是否至少有$m$提供非过敏原香料。 如果少于$m$，每个子集必须包含至少一种过敏原； 如果没有，每个子集都会避免它们； 否则两种结构都存在。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

n, m = map(int, input().split())
k = int(input())

all_spices = set()
bad = set()

for _ in range(k):
    bad.add(input().strip())

p = int(input())

for _ in range(p):
    ni = int(input())
    guest = set()
    for _ in range(ni):
        guest.add(input().strip())

    # compute intersection with helper-banned set
    # and compute effective dangerous spices
    x = 0
    for s in guest:
        if s not in bad:
            x += 1

    # actually x = |guest ∩ safe|, but we want safe allergens count
    # recompute properly:
    x = len([s for s in guest if s not in bad])

    safe_pool_size = n - k
    safe_non_guest = safe_pool_size - x

    if x == 0:
        print("NO")
    elif safe_non_guest < m:
        print("YES")
    else:
        print("MAYBE")
```实施过程首先读取助手的过敏列表并将其视为禁止的过滤器。 通过计算有多少过敏原在过滤器中幸存下来，对每位客人进行独立处理。 

变量$x$代表有多少客人的过敏原仍然有资格出现在菜肴中。 一旦知道了这一点，其余的推理就会分解为可用安全香料的数量和所需的菜肴大小之间的简单比较$m$。 

分支顺序很重要。 必须首先检查“否”的情况，因为无论组合如何，它都代表客人在结构上不可能受到影响。 “是”的情况源于对剩余安全池的争论。 其他一切都是混合情况。

 ## 边缘情况

 当客人将所有过敏原都包含在帮手的禁止组中时，他们的整个过敏清单就被有效地排除在考虑范围之外。 在这种情况下，每道有效的菜肴都会自动避开它们，因为菜肴宇宙从来不包含任何触发香料。 该算法通过以下方式捕获这一点：$x = 0$，立即产生“NO”，无需任何组合推理。 

当客人过敏组合之外的安全香料数量太少而无法填满菜肴时$m$，每个有效的选择必须至少包含一种过敏原。 计算$N' - x < m$直接识别这种强制包含场景，即使过敏原分布稀疏，也能确保“是”。 

当这两个条件都失败时，就有足够的灵活性来构建一个完全避开客人的有效菜肴和另一个包含至少一种过敏原的菜肴。 该算法将其归类为“MAYBE”，反映了受限宇宙中两种可行组合结构的共存。
