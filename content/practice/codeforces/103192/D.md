 ## 解决方案
 ## 问题理解

 我们得到了几个独立的测试用例。 在每个测试用例中，都有几个咒语。 每个法术消耗两种资源，红色资源和蓝色资源。 关键的观察结果是，玩家目前无法施展任何法术，这意味着无论他们拥有多少红色和蓝色法力，都严格不足以同时施展每个法术。 我们被要求确定玩家在保持这种情况下仍然可以拥有的最大法力总量，即红色加蓝色。 

一种更具结构性的思考方式是，每个法术都在可能的法力值平面上定义了一个禁区。 如果玩家的 R 为红色，B 为蓝色，则咒语 i 可用当且仅当 R ≥ r_i 且 B ≥ b_i 时。 问题中的条件表明没有可用的咒语，因此对于每个 i，我们必须有 R < r_i 或 B < b_i。 我们希望在这些约束下最大化 R + B。 

约束表明每个测试组的 n 最多为 5000，全局总和为 5000，每个值可以大到 10^6。 这强烈表明每个测试用例解决方案的任何 O(n^2) 或 O(n log n) 都是可以接受的，但任何立方或指数都是不可行的。 每个测试用例可能会进行线性或近线性扫描。 

当所有法术仅需要一种类型的法术力时，就会出现微妙的边缘情况。 例如，如果所有法术都是（1,0），那么拥有任何蓝色法术力都无济于事，但红色约束占主导地位。 另一个重要的情况是当存在一个要求很小的咒语时，例如(1,1)。 在这种情况下，即使是少量的两种资源也会受到限制，最佳解决方案必须仔细平衡红色和蓝色。 

最重要的隐藏困难是，该约束不是按法术累加的。 每个咒语都禁止在第一象限中出现一个矩形，我们必须同时选择一个位于所有矩形之外的点。 

## 方法

 直接的强力方法是尝试 R 和 B 的所有可能值，直到达到最大观察到的要求。 由于值达到 10^6，这立即变得不可行。 即使我们只离散化 r_i 和 b_i 中的候选值，每个轴仍然有多达 5000 个候选值，从而导致 2500 万个状态，并且检查每个状态的所有拼写会太慢。 

为了减少这种情况，我们重新解释这个条件。 对于固定的总法力 S = R + B，我们想知道是否存在分割 R、B 以致没有法术可用。 如果我们固定 R，则 B 被确定为 S − R，我们只需要检查对于所有 i 是否有 R < r_i 或 S − R < b_i。 

这成为对给定 S 的一维可行性检查。关键的见解是，对于每个咒语，都有一个使其可用的 R 值区间。 具体来说，当 R ≥ r_i 且 R ≤ S − b_i 时，拼写 i 可用。 该区间为 [r_i, S − b_i]。 因此，没有咒语可用的条件变为：不存在 R 使得 R 位于任何这些区间内。 

因此，对于固定的 S，我们检查区间的并集是否涵盖从 0 到 S 的所有可能的 R 值。如果是，则 S 是不可能的。 如果存在间隙，则S是可行的。 

现在我们将问题简化为：找到最大的 S，使得区间不完全覆盖 [0, S]。 我们可以通过构建区间并检查排序和合并后的覆盖范围来测试S的可行性。 由于我们只需要尝试候选 S 值，因此我们可以对 S 从 0 到 max(r_i + b_i) 进行二分搜索，这已经足够了，因为任何最佳分割都必须对应于由 r_i + b_i 引起的某个边界。 

降低复杂性的观察结果是，关键结构是一条线上的区间覆盖，而不是二维搜索。

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力破解 R、B | O(maxR * maxB * n) | O(1) | O(1) | 太慢了 |
 | 二分查找+区间覆盖| O(n log V) | O(n) | 已接受 |

 ## 算法演练

 1. 计算 S 的上限，作为所有咒语中 r_i + b_i 的最大值。 这已经足够了，因为任何咒语都定义了一个边界，超出该边界增加两种资源都无法创建新的约束，并且最佳解决方案出现在这些边界处。 
2. 在 0 到该上限的范围内二分查找答案 S。 我们正在寻找仍然可行的最大的S。 
3.对于固定候选S，为每个咒语构建区间。 对于咒语 i，以 R 表示的可用区域为 [r_i, S − b_i]，但前提是 r_i ≤ S − b_i。 否则该间隔无效并被忽略。 
4. 按起始点对所有有效区间进行排序。 这种排序使我们能够合并重叠并有效地检测连续覆盖。 
5. 扫描已排序的间隔，同时保持迄今为止覆盖的最远点。 最初，覆盖范围为 0。如果下一个间隔在当前覆盖范围边界之后开始，则存在间隙并且 S 立即可行。 
6. 如果间隔从 0 到 S 没有间隙地完全扩展覆盖范围，则 S 是不可行的，因为 [0, S] 中的每个 R 都允许至少一个咒语。 
7. 使用二分搜索内部的可行性检查来收敛到最大有效S。 
8. 返回 S，或者检测可行性保持无界的情况，如果结构意味着不存在有限最大约束，则输出 INF。 

### 为什么它有效

 对于固定的S，每个咒语都对应一个R值的禁止区间。 条件“法术可施放”正好变成“R 位于该区间内”。 因此，确保没有法术可施放相当于确保 R 避开所有间隔。 可行性检查简化为验证所有区间的并集是否未覆盖整个段 [0, S]。 二分查找是有效的，因为可行性在 S 中是单调的：增加 S 只能扩大区间，而不能缩小区间，因此如果某个 S 是不可能的，那么所有更大的 S 也是不可能的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def possible(n, spells, S):
    intervals = []
    for r, b in spells:
        l = r
        rr = S - b
        if l <= rr:
            intervals.append((l, rr))

    if not intervals:
        return True

    intervals.sort()
    covered = 0

    for l, r in intervals:
        if l > covered:
            return True
        if r > covered:
            covered = r
        if covered >= S:
            return False

    return True

def solve():
    T = int(input())
    for _ in range(T):
        n = int(input())
        spells = [tuple(map(int, input().split())) for _ in range(n)]

        mx = 0
        for r, b in spells:
            mx = max(mx, r + b)

        lo, hi = 0, mx
        ans = 0

        while lo <= hi:
            mid = (lo + hi) // 2
            if possible(n, spells, mid):
                ans = mid
                lo = mid + 1
            else:
                hi = mid - 1

        print(ans)

if __name__ == "__main__":
    solve()
```该代码首先使用所有咒语的最大总和要求来计算搜索空间。 可行性检查将每个咒语转换为可能的红色分配 R 上的一个区间。排序后，它执行贪婪扫描以检测是否存在无法施放咒语的间隙。 然后二分查找最大化总法力 S。 

一个常见的实现陷阱是忘记必须丢弃 r_i > S − b_i 的无效区间。 另一个是覆盖初始化处理不当； 从 0 开始至关重要，因为 R 可以为零。 贪婪合并逻辑直接对区间并集覆盖进行编码。 

## 工作示例

 考虑咒语 (3, 0)、(2, 2) 和 (0, 3) 的情况。 我们测试候选者 S = 4。 

| 法术| 区间 [r, S-b] | 有效 |
 | --- | --- | --- |
 | (3,0) | [3, 4] | 是的 |
 | (2,2) | [2, 2] | 是的 |
 | (0,3) | [0, 1] | 是的 |

 排序后：[0,1]、[2,2]、[3,4]

 我们从 0 开始覆盖。第一个间隔覆盖 [0,1]，因此覆盖范围变为 1。下一个间隔从 2 开始，它大于 1，因此存在间隙。 这意味着S=4是可行的。 轨迹显示，即使存在一些区间，它们也没有完全覆盖 [0,4]。 

现在考虑一个更严格的情况，咒语为 (1,1)、(2,0)、(0,2) 且 S = 3。 

| 法术| 间隔 |
 | --- | --- |
 | (1,1) | [1,2]|
 | (2,0) | [2,3]|
 | (0,2) | [0,1]|

 排序：[0,1]、[1,2]、[2,3]

 覆盖范围从 0 开始。第一个间隔延伸到 1，第二个延伸到 2，第三个延伸到 3。覆盖范围连续到 3，这意味着不存在间隙。 因此S=3是不可行的。 

这些示例表明可行性如何完全取决于派生区间的并集是否留下间隙。

 ## 边缘情况

 当所有咒语仅依赖于一种颜色（例如（r，0））时，就会出现关键边缘情况。 在这种情况下，导出的间隔总是从 r 开始并延伸到 S，可行性完全取决于最小 r 之前是否存在 R 间隙。 该算法会处理此问题，因为间隔要么完全覆盖，要么留下未覆盖的前缀，这是在扫描中检测到的。 

另一种情况是当 r_i + b_i 非常小时。 例如，S = 2 时 (1,1) 会生成区间 [1,1]，它是一个点。 扫描仍然正确地将其视为必须跨越的覆盖范围，并且间隙检测仍然有效。 

最后，当 S 相对于所有咒语来说非常大时，所有间隔都会变得无效，因为 r_i > S − b_i 失败。 在这种情况下，算法立即返回可行，这对应于极大的 S 意味着玩家无法以有意义的方式同时满足任何咒语的两个要求的事实。
