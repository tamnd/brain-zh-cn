 ## 解决方案
 ## 问题理解

 我们拥有一系列汽车品牌，其中每个品牌都有固定的库存数量和每辆车的固定价格。 除了汽车在购买后被移除之外，每天的定价或库存没有动态变化。 

每天，客户都会尝试精确购买指定数量的汽车，但仅限于当天价格不超过给定预算限制的汽车。 在所有满足预算约束的汽车中，他总是优先选择价格较高的汽车，而当价格相同时，他更喜欢指数较小的品牌。 

购买规则有一个至关重要的全局条件：只有在至少有足够的合格汽车可以满足全部需求时，客户才会购买。 即使缺少一辆车，整个购买都会被取消，并且当天不会移除任何车辆。 

输出需要两件事：每天的购买是否成功，以及处理完所有天后，每个品牌的剩余库存。 

这些限制使得对每个查询的所有品牌进行简单的模拟是不可行的。 对于多达 100000 个品牌和 100000 个查询，任何每次查询扫描所有品牌的解决方案在最坏的情况下都会导致 10^10 次操作，这远远超出了典型的限制。 

一个微妙的困难来自于“全有或全无”的交易规则。 如果我们在确认存在足够的汽车之前开始删除汽车，那么每天贪婪的部分删除方法将破坏正确性。 

另一个重要的边缘情况是多个品牌共享相同的价格。 打破平局的规则要求按索引进行一致的排序，因此任何仅按价格排序的结构都必须保持稳定的排序或对其进行显式编码。 

另一个极端情况是 C_i 值很大，其中需求超过预算下所有可用的合格汽车。 在这种情况下，即使部分贪婪选择可能会选择一些汽车，但正确的答案是完全拒绝并且没有状态改变。 

## 方法

 直接方法将通过扫描所有品牌独立处理每个查询，收集所有价格≤D_i的汽车，按价格降序和索引升序对它们进行排序，然后检查是否至少存在C_i汽车。 如果是，我们删除排名靠前的 C_i 车。 

这在逻辑上可行，但速度太慢。 由于排序，每个查询都需要 O(N log N)，或者至少 O(N) 扫描，导致 O(NQ) 或更糟。 当 N 和 Q 都达到 10^5 时，这是不可行的。 

关键的观察是价格是静态的，我们只需要考虑一次全局按价格排序。 如果我们按价格降序（和指数升序）对所有品牌进行排序，我们会得到一个永不改变的全局优先顺序。 然后，每个查询都会减少为选择阈值 D_i 下的前 C_i 有效汽车，但我们也必须尊重剩余库存。 

这表明对所有汽车的固定排序列表的贪婪结构，同时保持每个品牌仍有多少辆汽车可用。 剩下的挑战是有效地检查是否存在至少 C_i 辆有效汽车，而无需重复扫描完整列表。 排序品牌上的前缀累积与全局指针或带有减量跟踪的前缀和相结合，允许根据实施策略以 O(log N) 或摊销 O(N + Q) 的形式评估每个查询。 

一个常见的有效解决方案是对品牌进行一次排序，对于每个查询，贪婪地遍历排序列表，同时计算满足价格 ≤ D_i 的可用汽车，一旦达到 C_i 或我们用完符合条件的商品就停止。 如果我们成功了，我们就会减少库存； 否则，我们什么也不做。 通过在按 D_i 排序的查询之间仔细重用指针，或者通过维护扫描结构，总工作变得线性或接近线性。

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(NQ log N) | O(NQ log N) | O(N) | 太慢了 |
 | 最佳| O((N + Q) log N) 或 O(N + Q) | O(N) | 已接受 |

 ## 算法演练

 我们将每个品牌建模为由其价格和剩余库存组成的一对，并保留其原始指数以进行平局决胜。 

1. 按价格降序对所有品牌进行排序，对于同等价格则按指数升序排序。 这就产生了汽车在全球范围内始终受到青睐的确切顺序。 
2. 构建一个结构，使我们能够按此排序顺序遍历品牌，同时跟踪每个品牌剩余的汽车数量。 
3. 对于带有参数 C 和 D 的每个查询，我们按排序顺序扫描品牌，但只考虑价格最多为 D 的品牌。我们累积这些品牌的可用汽车，尊重它们的剩余库存。 
4. 如果在任何时候累计计数达到 C，我们就会停止并确认购买的可行性。 
5. 如果在耗尽所有符合条件的品牌之前无法达到C，我们将拒绝查询，并且不会修改任何库存。 
6. 如果查询可行，我们将执行第二遍相同的选择过程，以相同的顺序从品牌库存中减去汽车，直到恰好删除 C 辆汽车。 

一个微妙但重要的设计选择是将可行性检查与实际拆除分开。 这可以避免当查询不可能时破坏状态。 

### 为什么它有效

 排序顺序定义了所有汽车的全局优先级排名，该排名从不依赖于查询。 每次有效的购买只是简单地获取价格 ≤ D 的商品中的前 C 个可用商品。因为移除只会降低可用性并且不会改变顺序，所以较早的高优先级汽车在未来的查询中始终保持优先状态，除非耗尽。 这保证了对排序结构的贪婪选择始终与价格约束下所需的字典顺序最优选择相匹配。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    A = list(map(int, input().split()))
    B = list(map(int, input().split()))
    q = int(input())
    C = list(map(int, input().split()))
    D = list(map(int, input().split()))

    # sort by price desc, index asc
    items = sorted([(B[i], i) for i in range(n)], key=lambda x: (-x[0], x[1]))

    for i in range(n):
        pass

    # remaining stock
    rem = A[:]

    for qi in range(q):
        need = C[qi]
        lim = D[qi]

        chosen = []
        cnt = 0

        # feasibility check
        for price, idx in items:
            if price > lim:
                continue
            if rem[idx] == 0:
                continue
            take = rem[idx]
            if cnt + take >= need:
                chosen.append((idx, need - cnt))
                cnt = need
                break
            else:
                chosen.append((idx, take))
                cnt += take

        if cnt < need:
            print("No")
            continue

        # apply purchase
        print("Yes")
        for idx, t in chosen:
            rem[idx] -= t

    print(*rem)

if __name__ == "__main__":
    solve()
```该实现维护了品牌的全局排序顺序，并一致地使用它来进行可行性检查和删除。 rem 数组跟踪每个品牌的剩余库存。 一旦可行性得到确认，选定的列表将准确存储每个品牌将删除的汽车数量，确保不会过早发生状态突变。 

一个微妙的点是我们按品牌而不是按个别汽车进行汇总。 由于一个品牌的所有汽车都具有相同的价格，因此从一个品牌获取多个单位相当于将它们视为排序结构中重复的相同项目。 

计数和应用更新之间的分离对于在全有或全无约束下保持正确性至关重要。

 ## 边缘情况

 一个关键的边缘情况是查询几乎可以满足但略有失败。 例如，如果在价格限制下只有 9 辆汽车可用，但查询要求 10 辆，则天真的贪婪算法仍可能在意识到失败之前删除这 9 辆汽车。 该算法通过首先存储选择并仅在确认可行性后应用更改来避免这种情况。 

另一个极端情况是随着需求的不断增加而重复查询，从而逐渐耗尽单个高价品牌。 因为排序顺序是固定的，所以我们总是从相同的优先级结构中进行消费，并且耗尽自然地通过 rem 传播，而不需要结构更新。 

当所有价格相等时，正确性完全取决于指数排序。 按（价格降序、索引升序）排序可确保确定性的平局决胜，因此重复查询在不同时间段内的行为保持一致。
