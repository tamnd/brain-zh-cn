 ## 解决方案
 ## 问题理解

 我们有一群斯巴达人，每个人都有一种固定的力量，称为忠诚。 我们还获得了几个敌方小队，每个小队都有一个力量值。 

如果满足两个条件，则可以使用单个斯巴达小队击败一个敌方小队。 首先，该小队中的每个斯巴达人的忠诚度都必须严格高于敌人的实力。 其次，分配到该小队的斯巴达人数必须至少等于敌人的力量值。 一旦小队战斗，其中的所有斯巴达人都会被消耗掉，并且无法重新使用。 

任务是决定是否可以将部分或所有斯巴达人分成不相交的小队，以便每个敌方小队都被击败一次。 

重要的是，每个敌人都会同时施加两个约束：所需的最小小队规模和该小队每个成员的最低忠诚度阈值。 

每次测试的约束上限为 100000 名斯巴达人和 100000 名敌人，整个测试的总和为 100000。 这立即排除了任何尝试模拟所有可能的分组或重复搜索子集的解决方案。 任何与总输入大小成二次方的事情都不会及时完成。 解决方案本质上必须以每次操作的对数时间对有序结构和流程元素进行排序或维护。 

一个常见的陷阱是假设我们可以贪婪地将每个敌人与任何有效子集匹配，而不考虑全局资源消耗。 另一个微妙的问题是，更强大的斯巴达人并不总是最好尽早消费，因为他们可能需要首先满足更严格的忠诚度门槛。 

当高强度的斯巴达人被浪费在弱小的敌人身上时，幼稚贪婪匹配的具体失败案例就会出现，留下足够强大的斯巴达人太少以供以后更强的约束。 例如，如果我们首先将最强的斯巴达人分配给小敌人，那么即使存在有效的全局分配，我们随后也可能无法满足大阈值敌人的要求。 

## 方法

 暴力解释将尝试明确地构建小队。 对于每个敌人，我们都会扫描所有剩余的斯巴达人，过滤那些具有足够忠诚度的人，并挑选足够的人。 在最坏的情况下，这已经花费了 O(NM)，因为对于每个敌人，我们都可以扫描整个斯巴达列表。 N和M高达100000，这远远超出了可行的极限。 

关键的观察是，斯巴达人的个人身份并不重要，重要的是他们的忠诚价值观以及他们是否仍然未被使用。 每个敌人只关心有多少未使用的斯巴达人超过了其阈值。 这将问题简化为在阈值约束下重复检查和使用多重集中的元素。 

这自然表明对忠诚度值进行排序或频率结构。 我们可以让所有斯巴达人保持有序的结构，并按照实力递减的顺序处理敌人。 在首先处理强敌时，我们确保保留高忠诚度的斯巴达人，以满足严格的要求，以免他们被较弱的约束所消耗。 

一旦敌人按照实力降序排列，每一步都会问一个类似的问题：在所有当前未使用的斯巴达人中，有多少人的忠诚度大于当前敌人的实力，我们可以删除那么多人吗？ 

为了有效支持计数和删除，我们在压缩的忠诚度值上使用 Fenwick 树。 它允许我们查询有多少斯巴达人超过阈值，并在分配时删除特定的斯巴达人。 移除策略不需要是任意的，但是首先选择最小的有效忠诚度可以确保我们为后续阶段保留更强大的斯巴达人，尽管只要计数保持一致，正确性并不取决于这个细节。

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(NM) | O(N) | 太慢了 |
 | 芬威克树贪婪 | O((N + M) log N) | O((N + M) log N) | O(N) | 已接受 |

 ## 算法演练

 ### 1.将所有敌人按强度降序排列

 我们首先处理最强的约束，这样我们就不会意外消耗高阈值要求所需的斯巴达。 

### 2. 协调压缩斯巴达忠诚度

 由于忠诚度值最多可达 10^9，因此我们将它们压缩到排序索引空间中，以便可以使用 Fenwick 树。 

### 3. 在斯巴达计数上建立一棵芬威克树

 每个位置存储有多少未使用的斯巴达人具有该忠诚度值。 

### 4. 对于每个敌人，按降序排列

 我们计算目前有多少斯巴达人的忠诚度严格高于敌人的实力。 这是对 Fenwick 树的后缀和查询。 

如果这个数字小于敌人的兵力，我们立即得出结论，不可能进行分配。 

### 5. 如果存在足够的斯巴达人，则分配他们

 我们反复从有效后缀中删除斯巴达人，直到我们完全删除了其中的 B_i 为止。 

每次删除都是通过查找超过阈值的最小可用忠诚度指数，然后减少其在芬威克树中的计数来完成的。 

在每一步中选择最小的可用有效斯巴达可以避免浪费高忠诚度的斯巴达，但有效集中的任何一致的删除策略都可以保留正确性。 

### 6.如果所有敌人都处理成功

 如果没有敌人满足其要求，则存在有效的分区。 

### 为什么它有效

 按降序处理敌人可确保一旦我们通过阈值，未来的步骤将不需要比我们已经处理的更严格的忠诚度。 芬威克树保持了不变式，即它始终准确地代表未使用的斯巴达人的集合。 在每一步中，我们都会验证当前的要求是否可以仅使用当前可用的具有足够忠诚度的斯巴达人来满足。 由于每个 Spartan 最多被删除一次，并且每次删除都会在结构中进行计算，因此不会发生隐藏的重用或重复计算。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class Fenwick:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)

    def add(self, i, v):
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

    def range_sum(self, l, r):
        if r < l:
            return 0
        return self.sum(r) - self.sum(l - 1)

    def find_first(self, l):
        lo, hi = l, self.n
        ans = -1
        while lo <= hi:
            mid = (lo + hi) // 2
            if self.range_sum(l, mid) > 0:
                ans = mid
                hi = mid - 1
            else:
                lo = mid + 1
        return ans

def solve():
    T = int(input())
    for _ in range(T):
        N = int(input())
        A = list(map(int, input().split()))
        M = int(input())
        B = list(map(int, input().split()))

        A.sort()
        B.sort(reverse=True)

        # coordinate compression
        vals = sorted(set(A))
        idx = {v: i + 1 for i, v in enumerate(vals)}

        fw = Fenwick(len(vals))

        for a in A:
            fw.add(idx[a], 1)

        ok = True

        for b in B:
            # count available A > b
            import bisect
            pos = bisect.bisect_right(vals, b)
            total = fw.range_sum(pos + 1, len(vals))

            if total < b:
                ok = False
                break

            need = b
            while need > 0:
                p = fw.find_first(pos + 1)
                fw.add(p, -1)
                need -= 1

        print("Yes" if ok else "No")

if __name__ == "__main__":
    solve()
```该解决方案首先压缩忠诚度值，以便可以在 Fenwick 树中对其进行管理。 该树保存未使用的斯巴达人的计数。 对于每个敌人，我们找到忠诚度超过敌人实力的第一个索引，然后检查该后缀中是否有足够的斯巴达人。 否则，施工将立即失败。 

功能`find_first`是关键部分：它对 Fenwick 结构执行二分搜索，以在有效范围内找到下一个可用的 Spartan。 每次删除都会更新树，以便将来的查询看到更新后的状态。 

一个微妙的细节是，我们总是从当前的 Fenwick 状态重新计算可用性，而不是缓存它。 这是必要的，因为每次任务都会永久移除斯巴达并改变未来的可行性。

 ## 边缘情况

 一个关键的边缘情况是所有斯巴达人都具有完全相同的忠诚度。 如果任何敌人要求严格的不平等等于该值，则没有斯巴达人有资格。 该算法可以正确处理此问题，因为后缀查询针对该阈值返回零。 

另一种情况是，一个非常大的敌人在处理较小的敌人之前消耗了所有斯巴达人。 降序确保如果发生这种情况，它是不可避免的，因为任何有效的解决方案也需要那些斯巴达人来实现最大的约束。 

最后一种情况是，只有足够的斯巴达人，但分布在不同的忠诚度值上。 芬威克树确保我们始终只计算当前可用的有效候选者，因此部分分配不会错误地假设重用。
