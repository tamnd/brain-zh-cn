 ## 解决方案
 ## 问题理解

 我们获得固定数额的现金和一系列银行存款机会。 每个机会都需要花费特定数量的资金来参与，作为回报，它会产生固定数量的利息。 限制是每个存款选项最多只能选择一次，并且所选选项花费的总金额不能超过可用现金。 目标是选择这些存款选项的子集，以最大化获得的总利息。 

从结构上讲，这是一个背包问题，其中每个项目都有成本和价值，我们希望在容量限制下最大化价值。 “成本”是所需的存款金额，“价值”是赚取的利息。 

约束足够小，二次动态规划解决方案是可行的。 银行数量最多为 100 家，现金限额最多为 1000 家。简单的暴力子集枚举将考虑所有$2^N$子集，增长到大约$10^{30}$在最坏的情况下是完全不可行的。 这立即表明在预算维度上采用动态规划方法。 

当所有存款要求超过可用现金时，就会出现微妙的边缘情况。 例如，如果现金为 10 并且所有$w_i > 10$，则无法进行选择，并且答案必须为 0。错误地初始化 DP（例如，没有正确设置无法到达的状态）的幼稚实现可能会意外地传播无效值并产生非零答案。 另一种情况是多种组合产生相同的成本但不同的价值； 算法必须确保它始终保持最大值。 

## 方法

 蛮力的想法是尝试银行报价的每个子集。 对于每个子集，我们计算总成本和总利息，并保留最佳有效的一个。 这是有效的，因为它直接评估所有可能性，但一旦$N$由于呈指数增长，超过 25 左右。 

问题的结构揭示了经典的 0/1 背包模式。 每个银行报价都是一个项目，存款限额是它的权重，利息是它的价值。 容量就是可用现金$C$。 我们可以构建一个动态编程数组，而不是显式跟踪子集，其中$dp[c]$代表精确使用或最多使用可实现的最大利息$c$现金。 

关键的改进来自于认识到每个项目最多使用一次。 这使我们能够按降序迭代每个项目的容量，确保我们不会多次重复使用同一项目。 这将复杂性从指数降低到多项式。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O(2^N \cdot N)$|$O(N)$| 太慢了 |
 | 0/1 背包 DP |$O(NC)$|$O(C)$| 已接受 |

 ## 算法演练

 我们将每项银行报价视为具有成本和价值的项目，并且我们为每项可能的预算建立最佳可实现的利息$C$。 

1.初始化一个大小为DP的数组$C + 1$，其中每个条目都从 0 开始。这表示如果条目为零，则无论预算如何，都无法赚取利息。 
2. 逐一处理每家银行的报价。 对于银行$i$，设其成本为$w_i$其值为$v_i$。 
3. 对于这家银行，从$C$下降到$w_i$。 向后方向确保我们不会在本次迭代中多次重复使用同一个存储体。 
4. 对于每个预算$c$，考虑采用这家银行是否会改善结果：我们比较当前值$dp[c]$和$dp[c - w_i] + v_i$。 我们更新$dp[c]$如果后者更大。 
5.处理完所有银行后，答案是DP数组中的最大值，它代表在不超出预算的情况下最佳可实现的利息。 

### 为什么它有效

 在处理过程中的任何时刻，DP 数组都会仅使用迄今为止处理的银行来编码可实现的最佳利息。 向后迭代保证了当我们更新时$dp[c]$，值$dp[c - w_i]$still 指的是不包含当前bank 的状态，保留0/1 约束。 这个不变量确保银行的每个有效子集都被考虑一次，并且不会发生无效的重用。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    C = int(input().strip())
    v = list(map(int, input().split()))
    w = list(map(int, input().split()))
    n = len(v)

    dp = [0] * (C + 1)

    for i in range(n):
        cost = w[i]
        val = v[i]
        for c in range(C, cost - 1, -1):
            dp[c] = max(dp[c], dp[c - cost] + val)

    print(max(dp))

if __name__ == "__main__":
    solve()
```该方案使用一维DP数组来压缩内存。 每次迭代都会就地更新数组，但仅按容量降序排列，以便以前的状态对于转换仍然有效。 最终答案被视为所有容量的最大值，因为我们可以花费最多的任何金额$C$，不一定完全是$C$。

 ## 边缘情况

 当每个项目的成本都大于预算时，DP阵列应保持为零不变。 例如，与$C = 3$,$w = [5, 6]$，不可能有任何转换，因为内部循环永远不会执行。 最终正确答案变成0。 

当某个项目完全符合预算时，例如$C = 10$,$w = [10]$，DP更新集$dp[10] = v_1$， 自从$dp[0]$为 0 并且转换有效。 这证实了平等边界的处理是正确的。 

当多个项目的成本重叠时，例如$C = 10$,$w = [5, 5]$，向后迭代确保在处理第一个项目后，第二个项目仍然可以与第一个项目组合，而不会违反一次性约束，从而产生正确的组合值。
