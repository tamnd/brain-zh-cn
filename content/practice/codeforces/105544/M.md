 ## 解决方案
 ## 问题理解

 每个测试用例都描述了一个非常小的调度系统，该系统接收任务列表。 每个任务都有一个标识符和一个优先级值，系统必须决定任务的执行顺序。 

关键细节是任务已经按照插入系统的顺序给出。 第二行按插入顺序列出任务 ID，第三行提供每个任务相应的优先级。 调度始终首先选择优先级值最小的任务。 如果多个任务具有相同的优先级，系统会选择较早插入的任务来打破平局。 

因此，输出只是输入任务 ID 的排列，按照在此规则下执行的顺序排列。 

限制非常小。 每个测试用例最多有 100 个任务，并且最多有 10 个测试用例，即使是 O(n^2) 排序方法在限制范围内运行也是微不足道的。 这立即告诉我们，该解决方案不需要任何超出标准排序的高级数据结构或优化。 

当多个任务共享相同的优先级时，会出现微妙的边缘情况。 在这种情况下，正确性完全取决于保留原始插入顺序。 例如，如果输入是

 ID：7 3 9

 优先级：5 1 5

 那么任务 3 由于优先级 1 必须首先执行，而在 7 和 9 之间（优先级都是 5），7 必须在 9 之前执行，因为它出现得更早。 仅按优先级排序的简单方法会错误地交换 7 和 9。 

## 方法

 模拟调度程序最直接的方法是重复扫描列表并选择具有最小优先级的剩余任务。 每次我们选择一个任务时，我们都会将其标记为已删除并继续。 这是正确的，因为它完全遵循调度程序的定义。 

但是，每次选择都需要扫描所有剩余任务以找到最低优先级。 对于 n 个任务，这意味着 n 个选择，每个选择的成本为 O(n)，导致每个测试用例的时间为 O(n^2)。 尽管这里 n 只有 100，但这种方法是不必要的工作，并且当扩展到更大的约束时在概念上变得笨拙。 

关键的观察是调度程序相当于按组合键对任务进行排序。 主键是优先级，辅助键是插入顺序。 一旦我们将原始索引附加到每个任务，我们就可以使用这两个字段一次性对所有任务进行排序。 排序操作自然地解决了所有调度决策，因为两个任务之间的任何比较完全由这些规则决定。 

因此，问题简化为构建一个三元组列表并按（优先级，索引）执行稳定排序。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力选择| O(n^2) | O(n^2) | O(n) | 已接受 |
 | 排序依据（优先级、索引）| O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 1. 读取任务 ID 列表和优先级列表。 将每个任务与其插入索引配对，因为该索引代表打破平局的规则。 
2. 为每个任务构建一个结构，用于存储其优先级、索引和 ID。 该索引至关重要，因为它对原始到达顺序进行了编码。 
3. 使用首先查看优先级然后查看索引的比较对所有任务进行排序。 该索引确保在同等优先级中，较早的任务保持在较早的位置。 
4. 排序后，按顺序提取任务ID并输出。 

排序在这里起作用的原因是，一旦任务按照这两个标准进行全局排名，每个调度决策都是独立的。 优先级或任务集没有动态变化，因此全局排序完全决定执行顺序。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    T = int(input())
    out = []
    
    for _ in range(T):
        n = int(input())
        ids = list(map(int, input().split()))
        pr = list(map(int, input().split()))
        
        tasks = []
        for i in range(n):
            tasks.append((pr[i], i, ids[i]))
        
        tasks.sort()
        
        res = [str(task[2]) for task in tasks]
        out.append(" ".join(res))
    
    print("\n".join(out))

if __name__ == "__main__":
    solve()
```该实现首先将每个任务附加到其索引，以便将插入顺序保留为辅助排序键。 对元组进行排序`(priority, index, id)`直接编码两个调度规则。 排序后，仅提取 ID 进行输出。 

这里的一个常见错误是忘记排序键中的索引。 Python 的排序是稳定的，但如果结构被修改或使用不同的语言，则依赖稳定性而不显式包含索引是脆弱的。 显式排序依据`(priority, index)`保证正确性。 

## 工作示例

 考虑一个具有以下任务的测试用例：

 ID：4 1 3 2

 优先级：2 1 2 1

 我们将元组构建为`(priority, index, id)`:

 | 索引 | 编号 | 优先| 元组|
 | --- | --- | --- | --- |
 | 0 | 4 | 2 | (2, 0, 4) |
 | 1 | 1 | 1 | (1, 1, 1) | (1, 1, 1) |
 | 2 | 3 | 2 | (2, 2, 3) |
 | 3 | 2 | 1 | (1, 3, 2) |

 排序后：

 | 位置| 元组|
 | --- | --- |
 | 0 | (1, 1, 1) | (1, 1, 1) |
 | 1 | (1, 3, 2) |
 | 2 | (2, 0, 4) |
 | 3 | (2, 2, 3) |

 输出变为：`1 2 4 3`。 

此跟踪显示了如何通过索引严格解决相等优先级，从而确保每个优先级组内稳定的插入顺序。 

现在考虑一个更简单的情况：

 ID：10 20 30

 优先级：5 5 5

 所有任务共享相同的优先级，因此排序完全取决于插入索引。 排序的顺序仍然保留`10 20 30`，确认当优先级相等时算法退化为恒等顺序。

 ## 边缘情况

 当所有优先级都相同时，算法完全依赖于存储的索引。 每个元组变成`(p, i, id)`与平等`p`，因此排序减少为排序依据`i`。 输出与原始插入序列完全匹配，这是所需的行为。 

当优先级最低的任务出现在输入末尾时，排序仍将其放在第一位，因为优先级字段在比较中占主导地位。 该索引仅在优先级匹配时才重要，因此迟到的低优先级任务会正确地跳到较早的高优先级任务之前。
