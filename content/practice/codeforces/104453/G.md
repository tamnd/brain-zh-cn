 ## 解决方案
 ## 问题理解

 我们有几种作物。 对于每种作物类型，我们知道收获了多少公斤以及单个存储箱可以容纳该特定作物的容量。 关键的限制是盒子是特定于作物的，这意味着您不能在同一个盒子内混合不同的作物。 

对于每种作物，我们必须确定需要多少个相同的盒子来存储该作物的所有收获重量。 由于箱子是不可分割的单位，因此任何未完全装满箱子的剩余重量仍需要额外的箱子。 在计算每种作物的数量后，我们将所有作物的盒子数量相加。 

输入大小最多可达 100,000 个作物。 这立即排除了任何模拟逐项填充盒子或迭代超过公斤的方法。 任何解决方案都必须在恒定时间内处理每种作物，从而给出整体线性 O(n) 复杂度。 

当总收获量恰好能除以盒子容量时，就会出现微妙的边缘情况。 例如，如果 20 公斤存放在 4 公斤大小的盒子中，则答案恰好是 5 个盒子。 但如果 21 公斤存放在 4 公斤大小的盒子中，则答案是 6 个盒子。 在第二种情况下，没有上限处理的简单整数除法会错误地计算出 5。 

另一个极端情况是 n 等于 0 时。 在这种情况下，没有作物，因此不需要盒子，正确的输出是 0。 

## 方法

 思考问题最直接的方法是独立处理每种作物，并模拟将其收获物装入盒子中。 对于单一作物，我们可以反复从剩余重量中减去盒子的容量，并计算这样做的次数，直到什么都没有为止。 这正确地模拟了真实的过程，但在最坏的情况下速度太慢。 如果农作物有 100,000 公斤，箱子大小为 1 公斤，则此模拟仅对一种农作物执行 100,000 次操作，而对于 100,000 种农作物，这将变成 10^10 次操作，这是完全不可行的。 

关键的观察结果是每种作物都是独立的并且遵循简单的算术结构。 我们本质上是在计算需要多少个大小为 b_i 的组来覆盖 a_i 项。 这正是一个天花板划分问题。 我们可以直接使用整数运算来计算满框的数量，而不是模拟分组。 

对于每种作物，盒子的数量为：

 ceil(a_i / b_i)

 这可以有效地计算为：

 (a_i + b_i - 1) // b_i

 这种转换之所以有效，是因为添加 b_i - 1 可确保任何非零余数将除法推高一整格，而精确倍数保持不变。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 模拟| O(总 a_i / b_i) | O(1) | O(1) | 太慢了 |
 | 按作物天花板划分 | O(n) | O(1) | O(1) | 已接受 |

 ## 算法演练

 1. 读取作物类型的数量n。 如果 n 为零，则答案立即为零，因为没有任何内容可存储。 
2. 将累加器变量total_boxes初始化为零。 这将存储所有作物所需盒子的总和。 
3. 对于每种作物 i，读取 (a_i, b_i) 对，代表收获重量和箱子容量。 
4. 使用上限除法计算该作物所需的盒子数量：(a_i + b_i - 1) // b_i。 这确保了即使是部分填充的最终盒子也能被计数。 
5. 将此值添加到total_boxes。 每种作物的贡献都是独立的，因此在作物之间没有相互作用的情况下求和是有效的。 
6.处理完所有作物后，输出total_boxes。 

### 为什么它有效

每种作物都简化为一维包装问题，其中我们将固定数量 a_i 划分为大小为 b_i 的组。 任何有效的包装必须至少使用楼层（a_i / b_i）个满箱，如果存在剩余，则额外一个箱子是不可避免的，因为我们无法拆分箱子或混合作物。 公式 (a_i + b_i - 1) // b_i 准确地捕获了这个下界，并且没有解决方案可以使用更少的框，这使得它既必要又充分。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input().strip())
    total = 0

    for _ in range(n):
        a, b = map(int, input().split())
        total += (a + b - 1) // b

    print(total)

if __name__ == "__main__":
    solve()
```该解决方案读取每种作物一次，并立即将其转换为所需数量的盒子。 表达式 (a + b - 1) // b 是上限除法的标准整数技巧，完全避免了浮点运算，从而使计算既快速又安全。 

累加器总计保持为单个整数，并且由于每个操作都是 O(1)，因此整体复杂度保持线性。

 ## 边缘情况

 对于 n = 0，循环永远不会执行，并且总计保持为零，因此输出是正确的，无需特殊处理。 

对于 a_i 可以被 b_i 整除的作物（例如 20 和 4），公式给出 (20 + 3) // 4 = 23 // 4 = 5，与预期的精确包装相匹配。 没有添加额外的框，因为余数为零，并且 b_i - 1 的加法不会跨越 b_i 的倍数。 

对于像 7 和 10 这样的作物，收获量小于盒子容量，(7 + 9) // 10 = 16 // 10 = 1，正确反映了即使盒子部分装满也仍然需要。
