 ## 解决方案
 ## 问题理解

 我们得到一个整数数组和两个试图对其进行转换的独立进程。 每个进程都会收到自己的同一数组的副本。 一个进程可以对各个元素重复应用按位“或”运算，另一个进程可以对各个元素重复应用按位“与”运算。 在每次操作中，仅修改一个位置，但操作中使用的掩码可以是任何非负整数。 

两个进程的目标条件是相同的：它们希望最终数组在所有位置上具有相同的位模式，使得整个数组的按位与等于整个数组的按位或。 此条件强制最终数组的所有元素变得相等，因为元素之间任何不同的位都会破坏全局 AND 和全局 OR 之间的相等性。 

每个进程都希望通过尽可能少的操作来达到这种状态。 OR 过程只允许设置位，而不能清除它们。 AND 过程只允许清除位，而不能设置它们。 我们比较两者使用的操作数量并决定获胜者。 

约束很大：测试用例中的元素总数可以达到 200000。这立即排除了任何尝试模拟转换或探索多个候选目标数组的解决方案。 每个测试用例的线性扫描是可以接受的，但任何 n 的二次方都会失败。 

当所有元素已经相同时，就会出现微妙的边缘情况。 在这种情况下，两个进程都需要零操作。 当所有元素不同但不共享结构时，会发生另一个有趣的情况：天真的推理可能会建议每个元素进行多个操作，但由于掩码是任意的，因此一旦目标已知，每个元素就可以在单个操作中固定为有效目标。 

## 方法

 一个蛮力的想法是考虑数组的每个可能的最终值，并计算每只猴子需要多少次操作才能将所有元素转换为该值。 对于固定的目标值，每个元素可能需要多次位调整，我们将模拟 OR 或 AND 序列直到匹配。 这种方法很快就变得不可行，因为目标空间很大，可能有多达 2^30 个位模式，并且对于每个候选者，我们将扫描整个数组。 这会导致指数或至少 O(n * range) 解无法通过。 

关键的简化来自于理解在每种操作类型下甚至可以到达的最终数组。 OR 过程只能增加位，因此每个元素只能按位顺序向上移动。 这意味着最终值必须包含数组中任何位置出现的每一位，否则某些元素永远无法到达它。 这迫使目标成为整个数组的按位或。 

同样，AND 过程只能减少位。 至少一个元素中缺少的任何位都不能被引入，因此最终值必须包含在每个元素中。 这迫使目标成为整个数组的按位与。 

一旦确定了这些目标，问题就归结为计算有多少元素已经与目标匹配。 任何不同的元素都可以在一次操作中修复，因为掩码可以一步直接设置或清除所有必需的位。

 | ---| ---| ---| ---|
 | 对目标进行暴力破解 | O(n·2^30) | O(n·2^30) | O(1) | O(1) | 太慢了|
 | 最佳比特减少| O(n) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们独立处理每个测试用例。

1. 计算所有元素的按位或。 这代表 OR 过程可以达到的唯一可能的最终值，因为它不能引入数组中某处尚未存在的位。 
2. 计算所有元素的按位与。 这代表 AND 过程可以达到的唯一可能的最终值，因为它无法保留任何元素中丢失的位。 
3. 统计有多少个元素不等于OR 值。 每个这样的元素都需要 OR 过程的一次操作，因为具有合适掩码的单个 OR 可以直接将其转换为全局 OR。 
4. 计算有多少个元素不等于 AND 值。 每个这样的元素都需要一次 AND 处理操作，因为具有合适掩码的单个 AND 可以直接将其转换为全局 AND。 
5. 比较两个计数。 如果“或”过程使用较少的运算，它就会获胜。 如果 AND 过程使用较少的运算，它就会获胜。 否则，结果是平局。 

### 为什么它有效

 OR 过程只能在位空间中向上移动值，这意味着唯一全局一致的上限是数组的 OR。 对于包含不在目标中的位的元素，任何选择较小目标的尝试都会失败。 相同的推理对称地适用于 AND 过程和位的全局交集。 

一旦目标被固定，每个元素都是独立的，因为掩码操作允许一步中的任意位校正。 这消除了位置之间的任何耦合，将问题简化为简单的不匹配计数。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        n = int(input())
        a = list(map(int, input().split()))
        
        or_all = 0
        and_all = (1 << 30) - 1
        
        for x in a:
            or_all |= x
            and_all &= x
        
        cost_or = 0
        cost_and = 0
        
        for x in a:
            if x != or_all:
                cost_or += 1
            if x != and_all:
                cost_and += 1
        
        if cost_or < cost_and:
            print("or")
        elif cost_and < cost_or:
            print("and")
        else:
            print("sad")

if __name__ == "__main__":
    solve()
```该解决方案将全局位约束的预处理与计数操作分开。 初始循环计算两个进程的唯一可行目标。 第二遍对不匹配进行计数，这直接对应于所需的操作，因为可以使用适当的掩码在单个步骤中纠正每个元素。 

的选择`(1 << 30) - 1`因为初始 AND 值安全地覆盖了约束`a_i ≤ 10^9`，确保所有相关位均已初始设置。

 ## 边缘情况

 一种特殊情况是数组已经满足相等条件。 例如，输入`[8, 8, 8]`OR 等于 AND 等于 8。该算法将两个不匹配计数计算为零，因为每个元素都已与两个目标匹配，因此输出为平局。 

当一个进程似乎具有更“平衡”的不匹配分布但仍然由于目标限制而失败时，会发生另一种情况。 例如`[1, 2]`具有 OR 等于`3`且与等于`0`。 两个元素都与两个目标不同，因此两个进程都需要两次操作。 该算法正确输出平局。 

更微妙的情况是某些元素已经匹配 OR 目标但不匹配 AND 目标，反之亦然。 例如`[4, 5]`产生 OR = 5 和 AND = 4。一个元素与每个目标完全匹配，从而导致操作计数相等。 该算法自然会平衡这些情况，因为每种不匹配都会独立地贡献一个单位成本。
