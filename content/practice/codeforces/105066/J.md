 ## 解决方案
 ## 问题理解

 我们给出了许多独立的查询，每个查询定义了一个区间$[L, R]$价值高达一百万。 对于每个这样的区间，我们必须想象所有的子区间$[l, r]$完全位于其中，意味着$L \le l \le r \le R$。 

对于固定子区间$[l, r]$，我们查看所有数字$x$其中可被 3 整除。对于每个这样的数字，我们计算其十进制表示形式中出现了多少个数字“3”。 价值$g(l, r)$是所有有效数字中出现数字“3”的总数$x$在$[l, r]$。 

查询的最终答案不仅仅是这样一个$g(l, r)$，但是总和$g(l, r)$遍历所有子区间$[L, R]$。 由于子区间的数量是范围长度的二次方，因此直接枚举所有$(l, r)$对太大了。 

限制条件清楚地表明了这一点。 高达$10^5$测试用例和值高达$10^6$，任何即使每个查询都是线性的解决方案都已经很困难，并且任何二次的都是不可能的。 这个问题迫使我们将“所有子数组的总和”转换为单个位置贡献的计数问题。 

当局部思考时，会出现一个微妙的问题：像 33 这样的数字贡献了两位数的“3”，但它的贡献是根据包含它的子区间的数量来加权的，并且还遵循被 3 整除的约束。 另一个边缘情况是$x = 0$，它可以被 3 整除并且贡献零位数字“3”，因此它影响结构但不直接计数。 

## 方法

 直接方法将迭代所有$l$和$r$，并且对于每个子数组扫描所有$x$里面并测试能被 3 整除并计算数字“3”。 这是正确的，但非常昂贵。 一定长度范围内子数组的数量$n$是关于$n^2/2$，每个子数组最多可以扫描$n$元素，导致$O(n^3)$以最坏的解释进行每次测试。 即使优化为重复使用数字计数的前缀和，我们仍然需要评估每个子数组的整除性约束，这使得它不可行$n = 10^6$。 

关键的观察结果是该问题是线性的$x$其核心。 每个号码$x$独立贡献，但其贡献取决于子数组的数量$[l, r]$包括它。 固定位置$x$恰好出现在$(x-L+1)\cdot(R-x+1)$的子数组$[L, R]$。 然而，我们只想要那些内部条件为“我们正在对所有内容求和”的子数组$l, r$，并在其中我们总结所有$x \in [l,r]$可以被 3 整除。这可以交换：我们不是迭代子数组然后是数字，而是迭代数字并计算有多少个子数组包含它们。 

所以每个有效数$x$贡献：$$\text{digit\_3\_count}(x) \times \#\{(l,r): L \le l \le x \le r \le R\}$$等于：$$\text{digit\_3\_count}(x) \times (x-L+1)(R-x+1)$$但只有当$x \equiv 0 \pmod{3}$。 

这将整个问题最多转化为一个总和$10^6$每个查询范围的值。 我们仍然需要有效地计算数字计数，但由于界限很小，我们可以预先计算“3”数字的数量和整除性掩码一次。 

因此预处理数字最多可达$10^6$并且在每个查询中迭代 3 的倍数就足够了。 剩下的挑战是处理最多$10^5$高效查询，我们通过仅扫描每个范围内 3 的倍数来实现。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O(T \cdot N^3)$|$O(1)$| 太慢了 |
 | 最佳 |$O(T \cdot (R-L)/3)$|$O(N)$| 已接受 |

 ## 算法演练

 我们将问题重写为各个数字的总和，然后有效地累积它们的贡献。 

1. 预先计算一个数组`cnt3[x]`存储有多少个数字“3”出现在`x`。 

这对所有数字执行一次，直到$10^6$，因为数字结构独立于查询而重复。 
2. 对于每个测试用例，请阅读$L$和$R$，并初始化答案的累加器。 
3. 迭代所有数字$x$在$[L, R]$满足$x \bmod 3 = 0$。 

此限制直接匹配问题条件，因此我们忽略所有其他值。 
4. 对于每一个这样的$x$，将其贡献计算为：$$cnt3[x] \cdot (x-L+1) \cdot (R-x+1)$$因素$(x-L+1)(R-x+1)$计算有多少个子数组$[l, r]$包括$x$， 自从$l$范围可以从$L$到$x$， 和$r$范围可以从$x$到$R$。 
5. 将贡献值以模数添加到答案中$998244353$。 
6. 输出查询结果。 

这样做的关键原因是每一对$(l, r)$独立贡献职位$x$，并且每个位置对数字出现的影响独立。 通过交换总和，我们完全避免了枚举子数组，并将问题简化为计算覆盖率。 

### 为什么它有效

 正确性依赖于重复计算论证。 最初，计算是针对所有三元组的$(l, r, x)$这样$L \le l \le x \le r \le R$和$x \equiv 0 \pmod{3}$。 每个这样的三元组贡献了数字“3”的数量$x$。 

而不是迭代$(l, r)$首先，我们修复$x$并计算有多少有效$(l, r)$包括它。 每个有效对都只计算一次，因为每个子数组对每个包含的子数组都有独立的贡献$x$。 不同位置之间不存在交互作用，因此重新排列求和顺序可以保留总贡献。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MOD = 998244353
MAXN = 10**6 + 5

# precompute digit '3' counts
cnt3 = [0] * MAXN
for i in range(1, MAXN):
    cnt3[i] = cnt3[i // 10] + (1 if i % 10 == 3 else 0)

def solve():
    L = int(input())
    R = int(input())
    
    ans = 0
    
    # iterate only multiples of 3
    start = (L + 2) // 3 * 3
    for x in range(start, R + 1, 3):
        c = cnt3[x]
        if c == 0:
            continue
        left = x - L + 1
        right = R - x + 1
        ans = (ans + c * left * right) % MOD
    
    print(ans)

def main():
    t = int(input())
    for _ in range(t):
        solve()

if __name__ == "__main__":
    main()
```该实现首先使用整数除以 10 的简单递归来预先计算 100 万以内的所有数字的数字计数，这使得每个条目在其前一个条目已知后的时间恒定。 

对于每个查询，我们仅迭代 3 的倍数，因为所有其他数字都没有贡献。 算术术语`(x - L + 1) * (R - x + 1)`直接编码包含的子数组的数量`x`。 我们将其乘以预先计算的数字贡献，并以所需素数为模累加。 

主要的微妙之处是确保迭代从区间内第一个 3 的倍数开始。 这避免了检查每个数字，并使运行时间仅与相关候选者的数量成正比。

 ## 边缘情况

 一个重要的边缘情况是当$x$能被 3 整除，但不包含数字“3”。 例如$x = 6$。 算法仍然会访问它，但是`cnt3[x]`为零，所以它没有任何贡献。 这确保我们不需要预先过滤这些数字。 

另一个边缘情况是最小范围$L = R$。 如果单个值不能被 3 整除，则循环将在空范围内运行并立即返回零。 如果它是可整除的，则贡献变为$\text{cnt3}(L)$，因为左重数和右重数都等于 1。 

最后一个微妙的情况是当间隔在 3 的倍数边界附近开始或结束时。 的计算`start = (L + 2) // 3 * 3`确保我们永远不会错过第一个有效的候选人。 例如，如果$L = 4$，我们正确地从 6 开始，而不是 3，防止超出范围的包含。
