 ## 解决方案
 ## 问题理解

 两个人轮流吃无限序列的编号物品，从 1 开始，向上不间断。 关键规则是每个人吃的量不是固定的：相反，轮到他们吃他们当前“容量”允许的尽可能多的食物。 两者的容量均从 1 开始。 

当一个人完成自己的回合后，另一个人会观察自己吃了多少东西，并根据该数量增加自己的容量。 然后角色交替：月日先行，然后历，然后月日再次，依此类推。 

因此，吃过的物品的序列是将正整数划分为连续的块，其大小根据两人反馈循环随时间变化。 每个区块都完全分配给一个人。 

任务不是完全模拟 10^18 以内的整个范围的过程。 相反，我们被要求多个独立的查询：对于无限序列中的给定位置 x，确定该物品是否在 Tsukihi 回合或 Koyomi 回合期间被吃掉。 

这些限制使得直接模拟变得不可行。 序列以块的形式增长，块的大小可以快速增加，并且 x 可以大到 10^18，这立即排除了线性构造。 即使是构建块直到超过最大查询的贪婪模拟，在最坏的情况下也需要 10^9 或更多迭代，这远远超过 2 秒。 

一个微妙的边缘情况是由于块大小增长的速度以及它们的交替方式而产生的。 一种简单的方法可能会尝试预先计算块，直到达到最大的查询，但是当查询很大或间隔很宽时，这种方法就会中断。 另一种失败模式是独立模拟每个查询，因为每个模拟都会从头开始并重复重新计算相同的前缀。 

## 方法

 直接模拟很容易描述：保持序列中的当前位置，保持两个玩家的当前能力，并交替轮流。 每回合生成一个该大小的块，将其分配给当前玩家，更新其他玩家的容量，然后继续，直到我们通过最大的查询索引。 

这是正确的，但速度非常慢。 所需的区块数量随着容量的演变而增长，而容量本身也不断增长。 在最坏的情况下，我们可能会在到达 x 之前生成 sqrt(x) 或更多块的顺序，并且每个查询都需要重复此推理或扫描大的前缀间隔。 

关键的观察结果是，该过程是完全确定性的，并产生一系列不相交的连续片段。 每个段都有固定的所有者和已知的长度。 我们不考虑单个项目，而是将序列压缩为 [L, R] 形式的间隔，并用 Tsukihi 或 Koyomi 标记。 

一旦看到这一点，问题就变成了一个经典的前缀区间分解问题：我们生成段，直到达到所需的最大范围一次，存储它们，然后通过二分查找哪个段包含 x 来回答每个查询。 

剩下的唯一挑战是有效计算段长度。 递归很简单：当前玩家的区块大小就是他们当前的容量； 当他们吃完后，对手的容量会增加该区块的大小。 这会产生一对不断演化的值，并且每一步都是 O(1)。 我们只需要生成段，直到通过最大查询值。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 每个查询的暴力模拟 | O(n·k) | O(n·k) | O(1) | O(1) | 太慢了|
 | 预计算段+二分查找| O(n + s log s) | O(n + s log s) | O(s) | 已接受 |

 这里 s 是直到最大查询位置为止生成的段的数量。 

## 算法演练

 我们将整个序列构建为连续的标记片段。

1.确定最大查询值。 这是我们需要检查的最远位置。 我们仅构建序列，直到累积长度超过该值。 
2. 将 Tsukihi 和 Koyomi 的当前容量初始化为 1 和 1。将位置计数器初始化为 1。设置一个标志，指示轮到谁了，从 Tsukihi 开始。 
3、当前位置未超过最大查询时：

 将当前玩家的容量作为下一段的大小。 这定义了一个块 [pos, pos + size − 1]。 

这样做的原因是，每个回合都会消耗同样数量的连续物品，并且该过程是严格顺序的，没有重叠或跳过。 
4. 记录该段及其所有者、起始索引和结束索引。 然后将位置指针前进到下一个未使用的整数。 
5. 通过添加刚刚消耗的段的大小来更新对手的容量。 这捕获了反馈规则：看到 x 件食物被吃掉会增加 x 的内疚感。 
6. 切换到另一位玩家并重复。 
7. 构建所有段后，通过定位哪个段包含 x 来独立回答每个查询。 由于段是按位置排序的，因此我们可以在段开头进行二分搜索。 

### 为什么它有效

 该过程将正整数划分为连续的不相交区间。 每个间隔都由单轮唯一确定，整个演化仅取决于累积容量，并且确定性地更新。 一旦生成了达到最大所需范围的所有间隔，每个查询都会减少到识别哪个间隔包含给定索引。 后面的片段不会影响前面的位置，因此预处理是充分且准确的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    xs = list(map(int, input().split()))
    max_x = max(xs)

    seg_l = []
    seg_r = []
    seg_owner = []

    tk = 1  # Tsukihi capacity
    kk = 1  # Koyomi capacity

    pos = 1
    turn = 0  # 0 Tsukihi, 1 Koyomi

    while pos <= max_x:
        if turn == 0:
            sz = tk
        else:
            sz = kk

        l = pos
        r = pos + sz - 1

        seg_l.append(l)
        seg_r.append(r)
        seg_owner.append('T' if turn == 0 else 'K')

        if turn == 0:
            kk += sz
        else:
            tk += sz

        pos = r + 1
        turn ^= 1

    import bisect

    def get_owner(x):
        i = bisect.bisect_left(seg_r, x)
        return seg_owner[i]

    out = []
    for x in xs:
        out.append(get_owner(x))

    print(''.join(out))

if __name__ == "__main__":
    solve()
```该代码构建段边界，直到达到所需的最大索引。 每个段都存储其范围和所有者。 能力完全按照流程中的描述进行更新，每回合后只有非活跃玩家发生变化。 

对段端点的二分搜索可确保每个查询在对数时间内得到解决。 关键细节是我们按右端点搜索，因为段是连续且不重叠的。

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 | O(s + n log s) | O(s + n log s) | s 段生成一次，每个查询都使用二分查找 |
 | 空间| O(s + n) | 段存储加输出数组|

 分段 s 的数量很少，因为能力会累积增长，并且每回合都会增加对手的能力。 即使 x 高达 10^18，这也可以使总模拟保持在限制范围内。 

该解决方案很容易满足时间和内存限制，因为 s 仍然是可管理的，并且查询是在对数时间内处理的。

 ## 边缘情况

 关键的边缘情况是当查询恰好位于段边界时。 例如，如果某个段在位置 6 处结束，则查询 6 必须返回该段的所有者，而不是下一个段的所有者。 对右端点的二分搜索通过选择末尾至少为 x 的第一个段来确保正确处理相等性。 

另一种情况是非常大的查询，例如 10^18。 该算法从不尝试直接模拟该值； 一旦构造的段超过最大查询，它就会停止。 即使各个查询相距很远，这也保证了可行性。 

最后，两个容量均为 1 的初始步骤确保第一个段始终是 Tsukihi 拥有的单个元素。 任何在记录第一个段之前错误地增加容量的实现都会改变整个序列并从第一个查询中产生错误的答案。
