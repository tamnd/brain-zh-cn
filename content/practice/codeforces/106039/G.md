 ## 问题理解

 我们得到一个仅由两种类型的符号组成的序列，一个左括号`(`代表杨舞者和结束括号`)`代表殷舞者。 每个`(`必须与稍后匹配`)`形成有效的配对，并且每个舞者都恰好参加一对。 保证存在有效的完全匹配意味着序列的行为类似于正确匹配的括号配置。 

任务不是构建配对，而是计算有多少特定配对被禁止。 一对`(i, j)`和`i < j`， 在哪里`s[i] = '('`和`s[j] = ')'`，如果选择该对作为完整匹配的一部分使得无法完成剩余位置的有效匹配，则被认为是不兼容的。 

思考这个问题的一种有用方法是，我们在括号的完美匹配中选择一条边，并且我们想知道有多少这样的边，剩余结构不再是有效的括号匹配问题。 

约束条件`N ≤ 10^6`强制线性或近线性解。 所有可能的对的任何二次方都是立即不可能的，因为最多有`O(N^2)`最坏情况下的候选对。 即使在内存中存储所有对也是不可能的。 

高度嵌套或交替的结构会产生微妙的边缘情况。 在完全嵌套的字符串中，例如`((()))`，每个有效的配对都会使剩余的结构保持有效，因此答案为零。 在像这样的字符串中`()()`，选择最外面的一对可能会破坏剩余的结构，因为它可能会留下一个`)`之前`(`，使得完成不可能。 这表明不兼容不仅仅与嵌套有关，而是与诱导的剩余序列是否仍然形成有效的括号序列有关。 

另一种边缘情况是，删除所选对的端点时会隔离前缀顺序不再平衡的段。 例如，在`()()`, 配对`1`和`4`离开职位`2`和`3`以相反的顺序`( )`与`) (`取决于解释，打破有效性。 

## 方法

 蛮力方法很简单：尝试每个有效的对`(i, j)`这样`s[i] = '('`和`s[j] = ')'`，暂时将其删除，检查剩余序列是否仍能完美匹配。 可以使用堆栈模拟来检查有效性`O(N)`时间，所以这给出`O(N^2)`候选人时间`O(N)`验证，导致`O(N^3)`以最坏的解释，或充其量`O(N^2)`如果我们仔细地重用结构。 和`N`达到一百万，这是完全不可能的。 

关键的见解是，我们实际上并不是被问到任意匹配，而是被问到删除匹配对是否会破坏类似 Dyck 的序列的结构。 关键的观察是，不兼容性是由有多少不匹配的局部决定的`(`和`)`在匹配完成之前存在于后缀中。 如果我们使用堆栈解释匹配，则每个`(`被推动，并且每个`)`弹出一个。 一对`(i, j)`确切地说，当删除两个端点后，剩余序列中存在一个前缀，其中`)`超过数量`(`，使得完成不可能。 

这可以使用标准规范匹配来重新构建：当使用堆栈处理字符串时，每个`)`匹配最近未匹配的`(`。 每对对应一个堆栈弹出。 不兼容条件等同于该匹配边是否“安全”，因为删除它可以保留所有剩余后缀的堆栈可行性。 这简化为一个经典的结构：我们需要计算有多少匹配对不是隐式嵌套结构中的“必要桥梁”。 

最后的简化是，当一对是“非规范”配对时，它是不兼容的，因为它跨越了区间内前缀平衡为零的区域。 使用前缀平衡分解，我们可以表明不兼容的对对应于匹配的情况`)`关闭一个在其原始分解中不是最小的组件。 每个基元段恰好贡献一个安全匹配模式，并且该段内的所有其他配对以结构化可数方式不兼容。 这导致了对基元块的基于线性堆栈的计算。

 ## 算法演练

 我们处理字符串的同时维护一堆不匹配的左括号索引。 同时，我们使用前缀平衡来跟踪原始平衡段。 

1. 我们计算前缀余额，其中`+1`是`(`和`-1`是`)`。 这让我们可以确定完整的有效组件何时结束，即余额何时返回零。 每个这样的部分都是独立的。 
2. 我们使用堆栈将每个右括号与最近未匹配的左括号进行匹配。 这为我们提供了一个与序列的自然嵌套相对应的规范配对结构。 
3. 对于每对匹配的`(i, j)`，我们检查它是否完全位于原始段内或者是否跨越结构边界。 这很重要，因为只有尊重原始分解的匹配才能在删除后保持有效性。 
4. 我们计算违反此属性的对。 直观上，当匹配不是其局部结构的“最外层可能的闭包”时，删除它会破坏该段剩余后缀的平衡。 
5. 最终答案是在结构上不是其原始块的必要闭包的匹配对的数量。 

关键的不变性是，在处理过程中的任何一点，堆栈都准确地表示当前前缀的不匹配的开头，并且前缀平衡零位置将字符串划分为独立的 Dyck 组件。 在每个组件中，只有最外层的匹配闭包在删除后仍保持有效性； 所有其他的都会导致残差结构不平衡，从而使完成变得不可能。 这确保了对兼容和不兼容的对的分类是一致且详尽的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    s = input().strip()
    n = len(s)

    bal = 0
    stack = []
    comp_id = [-1] * n
    cid = 0

    # assign primitive components
    for i, ch in enumerate(s):
        if ch == '(':
            bal += 1
        else:
            bal -= 1

        comp_id[i] = cid

        if bal == 0:
            cid += 1

    stack = []
    pair = [None] * n

    for i, ch in enumerate(s):
        if ch == '(':
            stack.append(i)
        else:
            j = stack.pop()
            pair[i] = j
            pair[j] = i

    # mark component spans
    comp_end = {}
    bal = 0
    start = 0
    cid = 0

    for i, ch in enumerate(s):
        if ch == '(':
            bal += 1
        else:
            bal -= 1

        if bal == 0:
            comp_end[cid] = i
            cid += 1

    ans = 0

    for i, ch in enumerate(s):
        if ch == ')':
            j = pair[i]
            if comp_id[i] != comp_id[j]:
                ans += 1

    print(ans)

if __name__ == "__main__":
    solve()
```该实现首先计算前缀平衡段来为每个索引分配一个原始组件标识符。 这是必要的，因为不兼容性是在这些平衡块的级别而不是全局确定的。 

堆栈阶段构造括号的规范匹配。 每个右括号都与最近未匹配的左括号配对，这保证了 Dyck 结构假设的正确性。 

最后，我们通过检查其端点是否位于同一基元组件中来对每个匹配对进行分类。 如果不这样做，配对就会跨越结构边界，这意味着删除它会破坏至少一个组件的平衡结构，使其不兼容。 

一个常见的陷阱是忘记组件边界必须通过返回零的前缀平衡来定义，而不是通过简单的索引分段来定义。 另一个微妙的问题是配对必须是基于堆栈的； 任意配对策略不符合正确性所需的结构分解。

 ## 边缘情况

 在像这样的字符串中`()()`，该算法通过前缀平衡两次归零来分配两个基元分量。 每对在其自己的组件内进行匹配，因此不会触发跨边界检测。 堆栈对`(1,2)`和`(3,4)`两者都留在各自的细分市场内，产生零不兼容性。 

在完全嵌套的情况下，例如`((()))`，前缀余额直到最后才返回零，因此只有一个组件。 所有堆栈匹配都发生在该组件内部，并且每一对都被归类为兼容。 该算法正确地避免了将内部对标记为有问题，因为在任何匹配端点都没有跨越结构边界。
