 ## 解决方案
 ## 问题理解

 我们收到一篇由单词和一组“强制”单词组成的帖子，这些单词必须全部出现在最终的主题标签中。 主题标签是根据帖子中选定的单词构建的，每个单词将其完整长度贡献给总成本。 目标是从帖子中删除尽可能少的单词，以便剩余单词仍然包含每个必需的关键字至少一次，并且所有剩余单词的总长度不超过限制$d$。 

换句话说，我们从多组字长开始。 有些位置很特殊，因为它们对应于必须至少包含一次的必需单词。 我们可以删除任意单词，但是删除后我们仍然必须保留每个所需单词至少出现一次，并且保留单词的长度之和不得超过$d$。 我们希望最大限度地减少我们删除的单词数量，或者同等地减少我们保留的单词数量。 

这些约束清楚地表明，解决方案在字数上必须接近线性或近线性。 文本中最多有 30,000 个关键字和 45,000 个单词，因此任何独立尝试子集或重新计算每次删除可行性的方法都会太慢。 甚至一个$O(m^2)$对删除的模拟已经超出了几个数量级的时间限制。 

当仅强制性单词已经超出限制时，就会出现微妙的边缘情况$d$。 例如，如果所需的单词是“a”（长度100,000）和“b”（长度100,000），并且$d = 150,000$，那么即使只保留这两个，总和也是 200,000，所以不存在解。 另一种情况是所需单词在文本中多次出现。 我们必须至少保留一份副本，但保留更多副本总是有害的，因此任何最佳解决方案都只会保留每个所需单词的恰好一次出现。 

第二个重要的场景是删除不需要的单词总是有益的，但删除需要的单词是被禁止的。 这导致了关键结构：所需的单词形成固定的主干，其他所有内容都是可选的权重，我们可以删除它们以满足总和约束。 

## 方法

 蛮力的想法是尝试保留每个所需单词至少出现一次的所有单词子集，并检查它们的总长度。 这在概念上是正确的，但立即不可行，因为有$2^m$子集。 即使将注意力限制在非必需的单词上，仍然会留下指数组合。 

关键的观察是，约束纯粹是附加的：只有总长度很重要，可行性仅取决于我们是否可以通过删除不需要的单词同时保持所需单词完整来减少总和。 这将问题转化为贡献的贪婪选择问题。 

我们从唯一强制的结构开始：对于每个必需的单词，我们必须至少保留一次出现。 在所需单词的所有出现中，我们应该始终保留最短的单词，因为它可以最大限度地减少不可避免的基线成本。 任何更长的发生只会增加总数，而不会提供额外的好处。 

在每个必填单词固定一个代表性出现后，所有其他单词都成为可选项目。 该问题简化为选择要保留的可选单词的子集，以使总长度保持在$d$。 由于我们希望最大限度地减少删除，因此我们同样希望保留尽可能多的可选单词，但前提是总和约束允许。 

这成为一个经典的贪婪归约问题：我们从所有必需的选择单词加上所有可选单词开始，计算总和，然后重复删除最大的可选单词，直到总和合适。 首先删除较大的单词可以最大限度地减少满足预算约束所需的删除次数，因为每次删除都会给每步带来最大的减少量。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力子集 |$O(2^m)$|$O(m)$| 太慢了 |
 | 贪心选择和排序 |$O(m \log m)$|$O(m)$| 已接受 |

 ## 算法演练

 1. 将所有必需的单词读入哈希集，以便我们可以在恒定时间内测试每个单词的成员资格。 
2. 扫描文本并对所需单词的出现进行分组。 对于每个所需的单词，我们只需要保留其出现的最小长度。 这给出了固定的强制成本。 
3. 对于文本中的每个单词，将其分类为必填或可选。 可选词都是要删除的候选词。 
4. 将基本成本计算为所选所需发生次数的总和。 如果这已经超过$d$，立即输出-1，因为没有删除可以减少所需的贡献。 
5. 如果我们最初保留所有单词，计算总成本，即所有单词长度的总和。 
6. 我们需要减少的量是$total - d$。 为了以最少的删除来实现这一点，我们按长度降序对可选单词进行排序，并首先重复删除最大的单词，累积保存的长度，直到达到所需的减少量。 
7. 删除的单词数是我们在此过程中丢弃的可选单词的数量。 

关键的想法是，所需的单词定义了一个固定的下限，而其他所有内容的行为就像独立的可移动权重。 排序可确保每次删除都有助于最大程度地减少。 

### 为什么它有效

 不变的是，在每一步中，我们都会维护一组保留的单词，其中始终包含每个所需单词至少出现一次。 在所有可行的方法中准确去除$k$可选单词，选择$k$最大的单词产生尽可能小的剩余总和。 由于可行性仅取决于总和是否低于$d$，这种贪心排序保证了如果有任何一组$k$删除有效，贪心选择也有效。 这确保我们永远不会高估所需删除的数量。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    
    keywords = set()
    for _ in range(n):
        keywords.add(input().strip())
    
    words = input().split()
    d = int(input())
    
    # required occurrences: keep at least one per keyword
    from collections import defaultdict
    best_required = {}
    
    optional = []
    total_sum = 0
    
    for w in words:
        L = len(w)
        total_sum += L
        
        if w in keywords:
            if w not in best_required:
                best_required[w] = L
            else:
                best_required[w] = min(best_required[w], L)
        else:
            optional.append(L)
    
    required_sum = sum(best_required.values())
    
    if required_sum > d:
        print(-1)
        return
    
    # we initially assume we keep all words, but we can drop optional ones
    optional.sort(reverse=True)
    
    removed = 0
    saved = 0
    
    for L in optional:
        if total_sum - saved <= d:
            break
        saved += L
        removed += 1
    
    print(removed)

if __name__ == "__main__":
    solve()
```实现首先将关键字分成一组以进行快速查找。 在扫描帖子时，它会维护每个关键字的最小长度出现次数，因为任何较长的重复出现次数严格来说都比最短的重复出现次数更差。 

可选单词仅存储为长度，因为只有它们的贡献才重要。 在计算总和和所需基线后，我们检查可行性。 如果即使强制基线超过$d$，我们提前终止。 

贪婪删除循环处理按长度递减排序的可选单词。 每次迭代都会模拟删除当前最大的剩余单词，跟踪总长度减少了多少。 一旦剩余金额适合$d$，我们停下来。

 ## 边缘情况

 一个关键的边缘情况是当单独所需的单词超过限制时。 例如，如果所需的单词足够长，以至于它们的最小出现次数已经违反$d$，算法在计算基线总和后立即返回 -1。 这避免了不必要的排序或处理。 

另一种情况是重复不同长度的所需单词。 假设某个所需单词出现多次，长度分别为 10、5 和 7。算法仅保留 5 作为强制成本。 这确保我们不会高估不可避免的长度。 如果我们错误地保留了所有出现的情况，我们可能会错误地得出结论：即使存在有效的解决方案，也不存在解决方案。 

最后一种边缘情况是可选单词都比必需单词短。 即使这样，排序仍然可以正确工作，因为删除顺序仅取决于每次删除的最大减少量，而不取决于与所需单词的比较。
