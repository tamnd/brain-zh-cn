 ## 解决方案
 ## 问题理解

 我们得到一排房屋，索引从 1 到 n。 每个房主都有自己想要居住在道路的哪一侧的偏好：左侧或右侧。 此偏好被编码为二进制字符串，其中 0 表示该人希望位于道路的左侧，1 表示他们希望位于道路的右侧。 

我们可以在任意两个连续的房屋之间放置一条道路，甚至可以在第一个房屋之前或最后一个房屋之后放置一条道路。 如果我们将道路放置在位置 i 之后，则房屋 1 到 i 形成左侧，房屋 i+1 到 n 形成右侧。 

如果在道路的每一侧，该侧至少有一半的居民对其一侧的分配感到满意，则该安置被认为是可接受的。 满意度纯粹是通过将侧面与人的偏好相匹配来确定的。 

任务是找到一个有效的割点位置i，使得双方都满足“至少一半满足”的条件，并且在所有这些有效割点中，我们选择最接近村庄中间的一个。 中间自然是在n/2左右，所以我们希望有效的i最小化到n/2的距离，如果有平局，则首选较小的i。 

输入大小高达 300,000，这会立即排除任何尝试所有剪切并通过每次扫描段重新计算满意度的解决方案。 对每个位置进行直接检查将花费 O(n²)，在这种规模下这太慢了。 该解决方案必须将每个位置的分段评估减少到恒定时间，这建议使用前缀和。 

一个微妙的点在于“至少一半”的定义。 对于大小为 k 的段，“half”表示 ceil(k/2)。 这对于奇数长度很重要，简单地检查 k/2 作为整数除法是不正确的。 

当一侧为空时，就会出现边缘情况。 如果 i = 0，则左侧没有居民，同样，如果 i = n，则右侧为空。 这些情况仍然必须满足条件，并且空段有效，因为不存在矛盾：零个居民中有零个满足满足要求。 

一种简单的方法可能会重新计算每次切割的满意度计数。 例如，在每个 i 处，我们可以通过扫描来计算前缀中的 0 和后缀中的 1，但这会重复遍历数组并在约束下失败。 

## 方法

 蛮力的想法很简单。 对于每个可能的切割位置 i，我们将数组分成两部分，并计算左边有多少人喜欢 0，右边有多少人喜欢 1。然后我们验证每一侧是否至少有一半的居民满意。 这需要为每个 i 扫描左右段，每次扫描的成本为 O(n)，从而导致总工作量为 O(n²)。 当 n 达到 300,000 时，这相当于 10^10 次运算，这远远超出了任何时间限制。 

关键的观察是任何段所需的所有信息都可以增量地预先计算。 如果我们维护零和一的前缀计数，我们可以在恒定时间内回答有关前缀的任何查询。 后缀可以从前缀和总计数中得出。 

这将问题转化为检查每个 i 的简单条件。 唯一剩下的微妙之处是有效地表达“至少一半满足”约束。 对于长度为k的段，要求至少有一半满足意味着satisfed_count * 2 >= k + (k mod 2)，这避免了浮点推理并正确处理奇数长度。 

一旦每次切割都可以在 O(1) 内得到验证，我们只需扫描所有位置并根据到中心的距离选择最佳的有效位置。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(n²) | O(1) | O(1) | 太慢了 |
 | 前缀和 | O(n) | O(n) | 已接受 |

## 算法演练

 1. 构建零和一计数的前缀数组。 

这使我们能够计算出，对于任何 i，左段中有多少居民喜欢每一侧，而无需再次扫描该段。 
2. 计算整个数组中 1 的总数。 

这是通过从前缀总数中减去来快速得出后缀段中的计数所必需的。 
3. 迭代从 0 到 n 的所有可能的切割位置 i。 

每个 i 代表将道路放置在房屋 i 之后，其中 i = 0 表示所有内容都在右侧，i = n 表示所有内容都在左侧。 
4. 对于每个 i，计算左段大小和右段大小。 

左侧尺寸为 i，右侧尺寸为 n - i。 
5. 计算双方满意的次数。 

左满足计数是前缀 [1..i] 中零的数量。 

正确满足的计数是后缀 [i+1..n] 中的 1 的数量，计算公式为：total_ones 减去 prefix_ones[i]。 
6. 使用双方的“至少一半”条件检查可行性。 

我们使用不等式 2 *atid_count>=segment_size+(segment_size mod 2)。 
7. 在所有有效的 i 中，选择距离 n/2 最小的一个，以较小的 i 打破平局。 

### 为什么它有效

 正确性在于这样一个事实：双方的满意度仅取决于总数，而不是排列。 前缀和完全保留了这些聚合。 由于每次切割都会将数组分成两个独立的段，因此单独检查它们就足够了。 半满意条件相对于满足计数而言是单调的，因此一旦准确地知道计数，就可以毫无歧义地确定有效性。 最后的选择步骤是对有限的有效候选集进行确定性优化。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

n = int(input().strip())
s = input().strip()

# prefix counts
pref0 = [0] * (n + 1)
pref1 = [0] * (n + 1)

for i, ch in enumerate(s, 1):
    pref0[i] = pref0[i - 1] + (ch == '0')
    pref1[i] = pref1[i - 1] + (ch == '1')

total1 = pref1[n]

def ok(i):
    left_size = i
    right_size = n - i

    left0 = pref0[i]
    right1 = total1 - pref1[i]

    # at least half condition: 2*x >= k + k%2
    if left_size > 0:
        if 2 * left0 < left_size + (left_size & 1):
            return False
    if right_size > 0:
        if 2 * right1 < right_size + (right_size & 1):
            return False

    return True

best_i = 0
best_dist = abs(n / 2 - 0)

for i in range(n + 1):
    if ok(i):
        dist = abs(n / 2 - i)
        if dist < best_dist or (dist == best_dist and i < best_i):
            best_dist = dist
            best_i = i

print(best_i)
```前缀数组`pref0`和`pref1`存储累积计数，允许对任何前缀进行恒定时间查询。 功能`ok(i)`通过使用这些前缀计算左右满意度计数来评估剪切是否有效。 奇偶校验调整的不等式避免了浮点除法并正确处理奇数段大小。 

最后一个循环检查所有位置，并根据距中心的距离保留最好的有效位置。

 ## 边缘情况

 当所有居民都喜欢同一方时，每次削减都有效，因为每一方总是至少有一半满意。 在这种情况下，答案纯粹取决于与中心的距离。 该算法仍然可以正确计算有效切割，因为前缀计数在一侧产生精确的饱和度，在另一侧产生零，满足所有段的不等式。 

当字符串交替频繁时，例如`010101`，即使全局计数平衡，许多削减也会在本地无效。 基于前缀的检查独立地正确过滤每个位置，确保部分不平衡不会跨段泄漏。 

当i为0或i为n时，一侧为空。 在这些情况下，满足条件减少为仅检查非空一侧。 由于代码显式跳过空边检查，因此如果剩余边满足条件，这些切割将自动处理为有效。
