 ## 解决方案
 ## 问题理解

 该任务描述了一个沿着一维站点路线从位置 1 移动到位置 N 的流动站。在每对连续站点之间都有一段路段。 对于每个路段，流动站都有两种穿越它的方式。 

它可以行走，这需要一些时间并会增加一定量的疲劳。 或者，它可以乘坐公共汽车，这要花钱，需要不同的时间，而且重要的是在使用后立即将疲劳重置为零。 每次乘坐公交车都是按路段付费的，而且漫游车有固定的预算。 整个旅程还有全球时间限制。 

目标不仅仅是最大限度地减少时间或金钱。 相反，我们希望最大限度地减少流动站在旅途中经历的最严重疲劳值，同时仍然确保总旅行时间不超过限制，总巴士成本不超过预算。 

一个关键的解释是，疲劳只会在步行时累积，而每当乘坐公共汽车时就会完全重置。 因此，这条道路自然地被分成由公交车分隔的步行街区。 答案是任何此类块上的最大步行积累，我们希望选择在哪里重置，以便最大限度地减少该最大值，同时尊重时间和预算限制。 

约束表明 N 最多为 50，而时间可能很大且预算最多为 1000。这立即表明该结构足够小，适合对段和预算状态进行动态规划。 每个路段步行或公交车的天真指数选择给出了 2^(N−1) 种可能性，大约是 2^49，太大了。 即使我们在本地尝试贪婪策略，时间、成本和疲劳之间的耦合也会使本地决策变得不可靠。 

当步行总是比公共汽车更快、更便宜，但会产生疲劳时，就会出现微妙的边缘情况。 一个幼稚的策略可能会尝试完全避免乘坐公共汽车，以最大限度地减少成本和时间，但即使未使用预算，也可能会违反时间限制。 另一个失败案例是，早点乘坐公共汽车确实更好，因为它可以消除疲劳，即使在当地看起来很浪费。 例如，如果步行导致巨大的疲劳峰值，迫使后来的决策在时间限制内变得不可能，那么延迟重置就是错误的，即使一开始看起来更便宜。 

## 方法

 蛮力的观点是独立考虑每个路段并决定是步行还是乘坐公共汽车。 这是正确的，因为每条完整路径都被覆盖，但它在规模上立即失败。 当 N 达到 50 时，有 2^(N−1) 种可能性，对于每种可能性，我们需要计算总时间、成本和最大疲劳。 即使进行有效的评估，这也是一个天文数字。 

关键的观察结果是，疲劳的表现就像重置之间的分段最大值。 一旦乘坐公共汽车，疲劳就会重置，因此问题就变成了选择断点，我们将序列“切割”成步行块。 这种结构建议动态编程，其中状态跟踪我们已经进步了多远，我们已经使用了多少钱，我们已经使用了多少时间，以及自上次重置以来当前的疲劳程度。 

我们还需要最小化最大值，这通常是通过对答案进行二分搜索来处理的。 如果我们确定候选疲劳极限 F，我们可以询问是否可以在不让步行段超过 F 的情况下遍历该路径。这会将问题转化为可行性检查。 在此检查过程中，任何 X_i > F 的路段都无法步行，因此必须乘坐公共汽车，这会花费预算和时间，但会重置疲劳。 否则，步行是允许的，但会增加当前的疲劳程度，并且我们必须确保它永远不会超过F。

这将问题简化为检查是否存在满足时间和预算约束的有效选择序列。 因为 N 很小，C 最多为 1000，所以位置和预算上的 DP 就足够了，其中转换反映了疲劳约束下的步行或公共汽车决策。

 | ---| ---| ---| ---|
 | 蛮力战胜所有选择| O(2^N) | O(2^N) | O(N) | 太慢了 |
 | DP 与二分搜索疲劳 | O(N * C * log(最大 X)) | O(N * C) | 已接受 |

 ## 算法演练

 我们通过二分搜索最小可能的最大疲劳并检查每个候选者的可行性来解决问题。 

1. 固定一个候选值 F，代表重置之间任意点的最大允许疲劳值。 这将问题转化为是或否问题。 
2. 对于这个F，对每一段进行预处理，决定是否允许步行。 如果X_i大于F，那么在该路段上步行是不可能的，所以如果我们穿越它，我们就被迫乘坐公共汽车。 
3. 将 DP 状态 dp[i][c] 定义为花费恰好 c 个钱到达站点 i 所需的最短时间，同时遵守疲劳约束 F。 
4. 初始化 dp[1][0] = 0，因为我们从第一个站开始，没有成本和时间。 
5. 对于从 1 到 N−1 的每个段 i，我们考虑从站点 i 到 i+1 的转换。 对于每个可达的 dp[i][c]，我们尝试两次转换。 
6. 如果第 i 段允许步行（即 X_i ≤ F），我们用 dp[i][c] + A_i 更新 dp[i+1][c]，并隐式更新当前疲劳度。 由于疲劳受 F 限制，因此仅当该段上的累积疲劳不超过 F 时，我们才接受此转换。 
7. 如果我们乘坐公共汽车，我们用 dp[i][c] + B_i 更新 dp[i+1][c + Y_i]。 巴士消除疲劳，让下一段重新开始。 
8. 处理完所有段后，我们检查对于某些 c ≤ C 是否有任何 dp[N][c] ≤ T。如果存在这样的状态，则候选 F 是可行的。 
9. 对 F 从 0 到 max(X_i) 进行二分搜索以找到最小的可行值。 

正确性依赖于以下事实：疲劳在步行段内是单调的，并在公交车使用时重置，因此每个有效策略都准确对应于该 DP 状态空间中的路径。 

### 为什么它有效

 在 DP 中的任何一点，状态都完全捕获所有相关历史记录：站点索引、花费的资金和自上次重置以来的隐式疲劳通过确保我们永远不会超过 F 来控制。任何两个达到相同 (i, c) 的部分解决方案对于未来决策都是可互换的，因为未来的转换仅取决于当前位置和剩余约束。 这给出了最佳子结构，并且二分搜索确保我们收紧全局最大疲劳界限，直到找到最小的可行界限。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def can(F, N, C, T, A, X, B, Y):
    INF = 10**18
    dp = [[INF] * (C + 1) for _ in range(N + 1)]
    dp[1][0] = 0

    for i in range(1, N):
        for c in range(C + 1):
            if dp[i][c] == INF:
                continue

            # walk
            if X[i] <= F:
                if dp[i][c] + A[i] <= T:
                    dp[i + 1][c] = min(dp[i + 1][c], dp[i][c] + A[i])

            # bus
            if c + Y[i] <= C:
                if dp[i][c] + B[i] <= T:
                    dp[i + 1][c + Y[i]] = min(dp[i + 1][c + Y[i]], dp[i][c] + B[i])

    return min(dp[N]) <= T

def solve():
    N, C, T = map(int, input().split())
    A = [0] * N
    X = [0] * N
    B = [0] * N
    Y = [0] * N

    for i in range(1, N):
        a, x, b, y = map(int, input().split())
        A[i] = a
        X[i] = x
        B[i] = b
        Y[i] = y

    lo, hi = 0, max(X)
    ans = -1

    while lo <= hi:
        mid = (lo + hi) // 2
        if can(mid, N, C, T, A, X, B, Y):
            ans = mid
            hi = mid - 1
        else:
            lo = mid + 1

    print(ans)

if __name__ == "__main__":
    solve()
```该代码将可行性检查与优化分开。 这`can`函数强制执行固定的疲劳限制，并在站点和预算上运行 DP。 然后，二分搜索包装器找到最小的可行限制。 

一个微妙的实现细节是索引：段存储从 1 到 N−1，以便过渡在站 i 和 i+1 之间干净地对齐。 另一个重要的一点是，在 DP 转换中时间修剪是必要的； 如果没有它，已经超过 T 的状态将会不必要地污染后来的转换。 DP 使用较大的哨兵值，而不是积极修剪早期状态，这使正确性保持简单。

 ## 边缘情况

 一个重要的边缘情况是所有步行时间都超过总时间限制，即使对于单个路段也是如此。 在这种情况下，任何可行的解决方案都必须依赖于总线。 DP 正确地处理了这个问题，因为步行转换立即超出了时间限制，只留下总线转换处于活动状态。 

另一种情况是预算为零。 然后，所有要求 Y_i > 0 的转换都被禁止，因此 DP 在疲劳极限 F 下退化为纯粹的行走可行性。如果不存在这样的 F 来将时间保持在 T 以下，则二分搜索正确地返回 -1。 

最后一个边缘情况是，即使步行在当地是最佳选择，也需要提早乘坐公共汽车来消除疲劳。 DP 捕捉到了这一点，因为它始终独立地考虑两个转变，并且不假设疲劳累积的单调性。
