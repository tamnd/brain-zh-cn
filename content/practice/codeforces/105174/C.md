 ## 解决方案
 ## 问题理解

 魔杖正好有`n`宝石槽。 宝石分为三个独立类别。 

第一类增加魔法攻击，第二类增加法力，第三类增加攻击速度。 每颗宝石都会占据一定数量的槽位，并为某一属性贡献正值。 每个宝石最多可以使用一次。 

选择任何总占用槽位不超过的宝石集合后`n`，魔杖的最终威力是`magic_attack × mana × attack_speed`。 

任务是最大化该产品。 

最重要的观察是，除了通过共享槽位限制之外，不同类别的宝石永远不会交互。 在一个类别中，唯一重要的是使用给定数量的槽位可以获得多少总属性值。 

插槽数量最多为2000个，每个类别最多包含2000颗宝石。 复杂度较高的动态规划解决方案`O(total_gems × n)`很容易足够快，因为这大约是`3 × 2000 × 2000 = 1.2 × 10^7`状态转换。 任何涉及 gem 子集或直接搜索 gem 之间的所有分配的操作都是完全不可行的。 

一些边缘情况值得关注。 

如果一个属性从未得到改善，那么无论其他两个属性如何，最终产品都是零。 

例如，```
2
1 1 1
2 5
3 10
3 10
```没有法力或攻击速度宝石适合两个插槽，因此每个可行的解决方案都有产品`0`。 

另一个微妙的情况是，最佳解决方案可能会留下未使用的插槽。 

例如，```
5
1 1 1
2 100
2 100
2 100
```使用每个类别中的一颗宝石会消耗六个插槽，这是不可能的。 最好的解决方案仅使用两颗宝石，给出产品`0`，因为一个属性保持为零。 不要求准确填充每个槽位。 

第三个错误是假设每个类别应该获得大约三分之一的名额。 

例如，```
10
3 1 1
2 100
2 100
2 100
2 1
4 1
```魔法宝石比其他宝石要强得多。 最优分配是高度不平衡的，任何提前固定槽位分配的策略都可能错过最优。 

## 方法

 一个简单的暴力解决方案将枚举宝石的每个子集。 对于每个子集，计算占用的总槽位和三个属性之和，然后保留最大有效乘积。 

这显然是正确的，因为每一个可行的选择都会被检查。 不幸的是，宝石数量可能多达 6000 个，因此搜索空间包含`2^6000`子集，这是一个天文数字。 

关键的观察是这三个类别是独立的。 假设我们确切地知道`x`插槽用于购买魔法宝石。 在该类别中，唯一的问题是使用可获得的最大魔法攻击`x`插槽。 所选宝石的个人身份不再重要。 

这立即建议解决三个独立的 0-1 背包问题。 

对于每个类别，定义`best[i] = maximum attribute value obtainable using exactly i slots`。 

经过这三个背包计算后，每个可能的槽位分配`a + b + c ≤ n`可以通过评估来检查`best_magic[a] × best_mana[b] × best_speed[c]`。 

背包阶段将指数级的多个子集压缩为`n + 1`每个类别的状态，使最终的枚举更加实用。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(2^(m1+m2+m3)) | O(2^(m1+m2+m3)) | O(1) | O(1) | 太慢了 |
 | 最佳| O((m1+m2+m3)n+n3) | O(n) | 已接受 |

 ## 算法演练

 1. 为一种宝石类别创建一个背包例程。 让`dp[i]`表示精确使用可获得的最大属性值`i`插槽。 初始化`dp[0]=0`而其他所有状态都是不可能的。 
2. 每个宝石处理一次。 按降序遍历槽计数以强制执行 0-1 限制。 如果使用当前的 gem 可以提高更大槽数的值，请更新该状态。 
3. 独立运行这个背包来获取魔法、法力和攻速宝石。 三个结果数组包含每个确切插槽使用情况的最佳可实现值。 
4. 枚举每一个可能的对`(a,b)`分配给前两个类别的槽位计数。 
5. 第三类剩余名额最多为`n-a-b`。 枚举每一个可行的`c`从`0`到那个限度。 
6. 忽略三个背包阵列之一中不可能出现的任何状态。 
7. 计算三个属性值的乘积并保留最大值。 

### 为什么它有效

 背包计算只考虑一个类别中的每个宝石子集一次，因此对于每个插槽计数，它都会存储可实现的最大属性值。 由于类别从不共享宝石并且仅通过总插槽预算进行交互，因此每个可行的整体解决方案恰好对应于插槽使用量的三倍`(a,b,c)`。 在最终枚举期间，将检查该三元组，并且每个类别的最佳值已经已知。 使用相同数量的槽将任何类别替换为较弱的子集只会减少乘积，因此将这些槽数的三个独立的最佳选择组合起来始终是最佳的。 由于检查了每个可行的时隙分配，因此找到了全局最大值。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

NEG = -10 ** 18

def knapsack(m, n):
    dp = [NEG] * (n + 1)
    dp[0] = 0
    for _ in range(m):
        v, w = map(int, input().split())
        for s in range(n - v, -1, -1):
            if dp[s] != NEG:
                dp[s + v] = max(dp[s + v], dp[s] + w)
    return dp

def solve():
    n = int(input())
    m1, m2, m3 = map(int, input().split())

    atk = knapsack(m1, n)
    mana = knapsack(m2, n)
    speed = knapsack(m3, n)

    ans = 0

    for a in range(n + 1):
        if atk[a] == NEG:
            continue
        for b in range(n - a + 1):
            if mana[b] == NEG:
                continue
            limit = n - a - b
            for c in range(limit + 1):
                if speed[c] == NEG:
                    continue
                val = atk[a] * mana[b] * speed[c]
                if val > ans:
                    ans = val

    print(ans)

if __name__ == "__main__":
    solve()
```辅助函数执行标准的 0-1 背包。 按降序迭代槽数可保证每个宝石最多使用一次。 

不可能的状态被初始化为一个大的负数而不是零。 这种区别很重要，因为当没有选择宝石时，零是合法的属性值。 

计算完三个动态规划表后，程序枚举每个可行的槽分布。 内部循环永远不会超过剩余容量，因此每个测试的分配都会自动满足总插槽限制。 

Python 整数具有任意精度，因此不需要对乘积进行特殊处理。

 ## 边缘情况

 考虑某一类别无法做出贡献的情况。```
2
1 1 1
2 5
3 10
3 10
```魔法值和攻击速度的背包表仅包含零槽状态。 每个枚举分配都会产生至少一个等于 0 的属性，因此算法正确返回`0`。 

现在考虑未使用的插槽。```
5
1 1 1
2 100
2 100
2 100
```提高这三个属性的唯一方法需要六个插槽，这超出了容量。 该枚举包括最多使用五个插槽的每个分配，其中没有一个为所有三个属性提供正值。 答案是正确的`0`。 

最后，考虑一个不平衡的最优值。```
10
3 1 1
2 100
2 100
2 100
2 1
4 1
```魔法背包强烈倾向于在那里花费很多槽位，而其他类别只需要几个槽位。 由于检查了每个可行的槽分布，因此算法自然会发现这种倾斜的分配，而不是假设均匀分配。
