 ## 解决方案
 ## 问题理解

 我们得到一摞书，每本书都有一个独特的标签。 堆栈顺序很重要：第一个数组描述哪本书位于顶部，最后一个元素位于底部。 我们还收到了第二个序列，描述了 Vasya 尝试获取特定书籍的固定顺序。 

在每一步中，Vasya 都会在当前书库中查找所请求的书籍。 如果它仍然存在，他会将该书连同其上方的所有书一起移走，并且所有这些移走的书都被视为移至他的背包中。 如果所请求的书籍之前已被删除，则该步骤不会发生任何事情。 

任务是计算每个步骤在该操作期间删除了多少本书。 

限制最多为 200,000 本书。 这立即排除了任何针对每个查询重复扫描或天真的模拟堆栈操作的解决方案。 在当前堆栈中搜索每本请求的书籍并物理删除元素的简单模拟可能会退化为二次行为，这对于时间限制来说太慢了。 

当多个查询引用已删除的书籍时，会出现一个微妙的问题。 在这些情况下，答案为零，并且不维护正确的“已删除”结构的幼稚实现可能会错误地尝试处理陈旧位置或重复计算已删除的段。 

粗心模拟的一个具体失败案例是每次重建或缩小列表时。 例如，如果堆栈是`[1, 2, 3, 4]`和查询是`[4, 3, 2, 1]`，如果删除后未更新索引，重复扫描和切片可能会意外地错误地重新计算位置。 正确的输出是`4 0 0 0`，因为第一个操作会删除整个堆栈，并且所有后续请求都不执行任何操作。 

## 方法

 直接模拟将当前堆栈保留为列表。 对于每个查询，我们搜索所请求的书籍，找到其位置，并删除该位置之前的所有内容。 这是正确的，因为它与问题定义完全匹配。 然而，搜索每本书需要线性时间，删除前缀也需要线性时间。 在所有查询中，当每次仅删除一小部分时，在最坏的情况下，这会导致二次复杂性。 

关键的观察结果是，书堆只会从上向下收缩，并且每本书只能被移除一次。 我们可以预先计算每本书在初始堆栈中的位置，而不是重复搜索。 然后我们维护一个指针，指示我们已经处理了堆栈的深度。 当查询请求一本书时，只有当它的位置仍在当前指针之上时，我们才会将其删除。 每当我们删除新的最低到达位置时，我们都会使指针前进。 每本书都被有效地触摸一次。 

这将重复扫描转变为按照移除事件的递增顺序对堆叠位置进行单次扫描。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(n^2) | O(n^2) | O(n) | 太慢了 |
 | 最佳 | O(n) | O(n) | 已接受 |

 ## 算法演练

 1. 构建数组`pos`在哪里`pos[x]`给出书的索引`x`在初始堆栈中。 这会将每个查询转换为直接位置查找而不是搜索。 
2. 维护一个变量`ptr`代表堆栈中已被删除的最高索引。 最初，没有书籍被删除，所以`ptr = 0`。 
3. 对于每本请求的书籍`b_i`,检查其位置`p = pos[b_i]`。 
4.如果`p <= ptr`，这本书已经作为先前操作的一部分被删除，因此输出`0`。 
5. 如果`p > ptr`，那么这本书仍然存在。 该操作从索引中删除所有书籍`ptr + 1`下降到`p`。 删除的书籍数量为`p - ptr`。 
6. 更新`ptr`到`p`，因为以上所有内容（包括）`p`现在已从未来的考虑中删除。 
7. 对所有查询重复此操作。 

正确性来自于这样的事实`ptr`始终跟踪已消耗的堆栈的最深前缀。 每次删除都会单调扩展此前缀，并且不会重新访问已删除的段。 

## 为什么它有效

 在任何时刻，剩余的堆栈都是从以下位置开始的后缀：`ptr + 1`。 每个查询要么引用该后缀内部或外部的一本书。 如果它在外部，则它已包含在先前的删除中，因此答案必须为零。 如果它在内部，则删除操作将精确地获取该书之前剩余后缀的前缀，该书是未使用的书的连续段。 因为`ptr`只向前移动，每个索引最多处理一次，保证线性总工作。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = list(map(int, input().split()))
    b = list(map(int, input().split()))

    pos = [0] * (n + 1)
    for i, x in enumerate(a):
        pos[x] = i + 1

    ptr = 0
    res = []

    for x in b:
        p = pos[x]
        if p <= ptr:
            res.append("0")
        else:
            res.append(str(p - ptr))
            ptr = p

    print(" ".join(res))

if __name__ == "__main__":
    solve()
```这`pos`数组是使用从 1 开始的索引构建的，因此堆栈位置与删除长度自然对齐。 这避免了计算时的相差一调整`p - ptr`。 

这`ptr`变量是解的中心状态。 每次我们回答一个非零查询时，我们都会将其精确扩展到新删除的书的位置。 这保证了我们永远不会重复计算任何段。 

为了提高效率，输出被累积为字符串，因为在循环内重复打印会减慢执行速度。 

## 工作示例

 ### 示例 1

 输入堆栈是`[1, 2, 3]`，查询是`[2, 1, 3]`。 

| 步骤| 预订 | 位置 | 之前的 ptr | 行动| 已删除 |
 | ---| ---| ---| ---| ---| ---|
 | 1 | 2 | 2 | 0 | 删除前缀 2 | 2 |
 | 2 | 1 | 1 | 2 | 已经删除 | 0 |
 | 3 | 3 | 3 | 2 | 删除前缀 3 | 1 |

 输出是`2 0 1`。 

这表明一旦`ptr`越过了某个位置，该书就不能再触发任何操作。 

### 示例 2

 堆栈`[3, 1, 4, 2, 5]`, 查询`[4, 2, 5, 1, 3]`。 

| 步骤| 预订 | 位置 | 之前的 ptr | 行动| 已删除 |
 | ---| ---| ---| ---| ---| ---|
 | 1 | 4 | 3 | 0 | 删除至 3 | 3 |
 | 2 | 2 | 4 | 3 | 扩展到 4 | 1 |
 | 3 | 5 | 5 | 4 | 扩展到 5 | 1 |
 | 4 | 1 | 2 | 5 | 已经删除 | 0 |
 | 5 | 3 | 1 | 5 | 已经删除 | 0 |

 这显示了如何`ptr`单调增长并最终覆盖整个堆栈。

 ## 边缘情况

 关键的边缘情况是当查询引用上一步中已删除的书籍时。 在这种情况下，正确的行为是输出零并且不修改任何状态。 该算法通过检查存储的位置是否等于或低于来处理这个问题`ptr`。 

例如，使用堆栈`[1, 2, 3, 4]`和查询`[3, 4, 2, 1]`, 处理后`3`，指针变为`3`。 加工时`4`，它延伸到`4`。 那时，双方`2`和`1`低于或等于`ptr`，因此它们无需进一步更改即可正确生成零。
