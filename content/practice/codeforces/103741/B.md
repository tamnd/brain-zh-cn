 ## 解决方案
 ## 问题理解

 我们被分配了几个独立的工作，每个工作都代表准备一个竞赛问题。 每个问题都包含两个连续的阶段：首先必须创建它，然后才能对其进行检查或验证。 

总共有 n 个问题，有 m 个可用人员。 每个人每小时只能完成一个工作单元，该单元可以是创建单元，也可以是验证单元。 一个人不能将一个小时分配给不同的任务，每个单独的任务也不能在不同的人之间分配。 唯一的依赖关系存在于每个问题内部：只有在相应的创建完成后才能进行验证。 

目标是安排所有工作，使总完成时间（以小时为单位）尽可能短。 

约束允许 n 和 m 最多为 10^9 以及最多 100,000 个测试用例。 这立即排除了任何随时间推移的模拟或任何每小时的调度。 即使每个测试用例的复杂度为 O(n) 也是不可能的。 该解决方案必须将每个测试用例简化为常数时间算术。 

幼稚推理的一个微妙的失败案例来自于试图独立地处理创建和验证。 例如，如果n = 2并且m = 2，人们可能会认为这两个任务可以在一小时内独立完成，但验证取决于创建的完成，因此至少需要两轮。 

另一个常见的错误是假设答案只是 ceil(2n / m)。 虽然这个公式被证明是正确的，但如果我们忽略依赖结构，很容易错误地证明它的合理性； 正确的论点必须表明依赖性不会在总工作量之外引入任何额外的瓶颈。 

## 方法

 暴力方法会每小时模拟该过程。 在每一步中，我们都会将最多 m 个可用工作人员分配给尚未完成的创建任务或满足先决条件的验证任务。 我们将维护待处理创建、就绪验证的队列，并跟踪每个问题的完成状态。 

这在概念上是有效的，因为它准确地反映了真实的调度过程。 但是，每小时最多处理 m 个任务，总共最多可以处理 2n 个任务。 在最坏的情况下，模拟将需要 O(n) 个步骤，并且每个步骤可能涉及许多任务的簿记。 当n达到10^9时，这是完全不可行的。 

关键的观察结果是结构极其均匀。 每个问题恰好贡献两个单元任务，唯一的依赖关系是长度为 2 的简单链。 除了通过共享工作人员能力之外，不同问题之间不存在交互作用。 这将问题变成了纯粹的吞吐量问题。 

在所有任务中，正好有 2n 个单元操作。 由于 m 个工人每小时最多可以处理 m 个单位，因此总时间不能小于 ceil(2n / m)。 唯一剩下的问题是依赖性是否会迫使额外的空闲时间超出此限制。 

他们没有。 任何有效的计划都可以转换为这样的计划：只要有可能，工作人员总是很忙，并且验证任务仅在其相应的创建尚未执行时才会延迟。 由于创建和验证是对称的单元任务，并且每个问题仅阻止其自身的验证，因此我们始终可以安排工作，使系统表现得像一个连续的管道，没有超出总容量限制的全局停顿。 

因此，最佳答案正是在 m 台相同的机器上处理 2n 个单元任务所需的时间。

 ## 算法演练

 我们直接使用容量参数计算最短时间。

1. 首先计算总工作量。 每个问题贡献 1 个创建任务和 1 个验证任务，因此总共有 2n 个单元任务。 这代表了所需的绝对最小处理量，与订购无关。 
2. 计算每小时可以处理多少个任务。 由于有 m 个人，每个人每小时可以执行一个单位任务，因此系统每小时最多处理 m 个任务。 
3. 将 2n 除以 m 并向上舍入，将总工作量转换为时间。 如果根本没有依赖关系，这给出了所需的最少小时数。 
4. 返回该值作为答案，因为创建和验证之间的依赖关系不会在总工作量之外引入任何额外的全局约束。 

### 为什么它有效

 唯一的结构约束是每次验证都依赖于其相应的创建，但这不会造成超出总处理能力的瓶颈。 在任何时候，只要我们始终尽可能优先考虑可用验证，那么在完成之前，可用任务的数量永远不会少于工作人员的数量。 这确保了系统的行为就像一个由 2n 个单位作业组成的池，没有超出机器容量所强制的额外空闲时间。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    T = int(input())
    out = []
    for _ in range(T):
        n, m = map(int, input().split())
        total = 2 * n
        ans = (total + m - 1) // m
        out.append(str(ans))
    print("\n".join(out))

if __name__ == "__main__":
    solve()
```该解决方案本质上是对系统使用 m 个并行工作线程处理 2n 个单元操作的观察结果的直接转换。 唯一的微妙之处是上限除法，实现为 (2n + m - 1) // m，它避免了浮点运算并安全地处理大值。 

由于输入大小很大，因此每个测试用例的所有计算都严格保持 O(1)。 

## 工作示例

 考虑 n = 2 且 m = 2 的输入。总功为 4 个单位。 除以 2 个工人就是 2 小时。 在第一个小时，两个工人都执行创建任务，完成两个问题的第一阶段。 在第二个小时内，两名工作人员都执行验证任务，完成所有工作。 计算出的答案与时间表完全匹配。 

现在考虑 n = 3 和 m = 2。总工作量为 6 个单位，因此公式得出 3 小时。 

| 小时 | 工人 1 | 工人2 | 笔记|
 | ---| ---| ---| ---|
 | 1 | 制作 P1 | 制作 P2 | P3 仍在等待中 |
 | 2 | 制作 P3 | 验证 P1 | P1 可用 |
 | 3 | 验证 P2 | 验证 P3 | 全部完成 |

 此跟踪显示，即使验证依赖于先前的创建，管道在第一次转换后仍保持满状态。 该依赖性仅影响本地排序，而不影响总吞吐量。

 ## 边缘情况

 当 n = 0 时，根本没有任务。 该公式给出 ceil(0 / m) = 0，因此算法正确输出零时间。 没有隐藏的开销，因为即使存在工人，也不需要工作。 

当 m 相对于 n 非常大时，只有在容量允许的情况下，所有任务都可以在单批创建中完成，然后在同一小时分组内进行验证，但由于验证依赖于创建，因此至少需要一次排序转换。 该公式通过上限除法自动处理这个问题。 

当 m = 1 时，系统退化为 2n 个操作的单个序列。 每个问题都强制执行两个连续的步骤，并且时间表变得严格串行。 该算法正确输出 2n，与强制依赖链匹配。
