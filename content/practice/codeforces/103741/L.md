 ## 解决方案
 ## 问题理解

 我们给出了长度的排列`n`，我们从概念上一一处理它的前缀。 对于每个前缀`A[1..k]`，我们想象运行一个给定的排序过程，称为`SORT`，并且我们被要求记录特定变量出现了多少次`m`在那次运行期间被分配。 

关键是我们实际上并不需要模拟完整的排序过程。 被计数的操作与前缀的结构相关联，并且每个前缀都是独立的，因为我们可以计算其贡献而无需从头开始重新运行所有内容。 

所以输出是一个数组`s`， 在哪里`s[k]`是变量的次数`m`期间被分配`SORT(A[1..k])`。 

约束条件`n ≤ 10^6`立即排除任何根据前缀重新计算排序过程的解决方案。 甚至一个`O(n log n)`方法重复`n`时间会太慢。 这促使我们寻求一种解决方案，在扩展前缀时保持增量状态。 

一个天真的解释是独立模拟每个前缀的排序。 这将花费大约`O(n^2)`甚至在考虑分类的内部成本之前就已经开始工作了。 

一个更微妙的失败案例来自于假设“排序工作”在前缀之间呈线性行为而无需重新计算。 例如，如果我们从头开始重新计算每个前缀，即使很小的输入也会正确运行，但复杂性会爆炸：

 输入：```
5
4 2 5 3 1
```如果我们重新计算每个前缀的排序工作，我们就会重复处理重叠结构。 这种重复正是必须避免的。 

## 方法

 缺少的部分是了解分拣过程实际测量的是什么。 虽然这里没有显示伪代码，但事实上它被描述为“改编的冒泡排序”，并且我们正在计算对变量的赋值`m`强烈建议每个这样的分配对应于在处理数组时发现类似反转的事件。 

在标准冒泡排序中，每次通过交换相邻元素来解决反转时，我们都会有效地“触及”该无序一次。 如果我们解释`m`当跟踪此类更新的数量时，对前缀进行排序的分配总数恰好对应于该前缀内的反转数量。 

所以对于每个前缀`A[1..k]`，我们需要反转计数：

 对`(i, j)`这样`1 ≤ i < j ≤ k`和`A[i] > A[j]`。 

暴力解决方案将为每个前缀独立计算此值。 这很简单：对于每个`k`，扫描前缀内的所有对。 这是正确的，但总体而言是二次的。 

关键的观察是前缀是嵌套的。 当我们从前缀移动时`k-1`到`k`，我们只引入一个新元素`A[k]`。 所有新的反转涉及`A[k]`仅依赖于先前的元素。 这使我们能够使用频率结构增量地维护反转计数。 

我们可以使用芬威克树（二叉索引树）来维护有多少先前值小于或大于当前值。 每个新元素贡献的反转等于大于它的已见过元素的数量。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(n²) | O(1) | O(1) | 太慢了 |
 | Fenwick 树（前缀倒装）| O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 我们从左到右处理排列，同时维护一个数据结构，该结构存储每个值到目前为止出现的次数。 

1. 初始化一个超过取值范围的Fenwick Tree（二叉索引树）`1..n`，最初为空。 这种结构允许我们查询有多少个可见元素≤某个值并以对数时间更新频率。 
2. 维护运行变量`inv`存储当前前缀的反转计数。 最初是`0`。 
3. 对于每个位置`k`从`1`到`n`，我们取值`x = A[k]`。 
4. 插入前`x`，我们查询之前插入的元素有多少个严格大于`x`。 This is equal to`(k-1) - count_leq(x)`， 在哪里`count_leq(x)`Fenwick 前缀总和为`x`。 
5. 我们将此数字添加到`inv`。 这代表通过放置引入的所有新反转`x`在前缀的末尾。 
6.我们插入`x`通过增加其频率进入芬威克树`1`。 
7. 我们输出当前值`inv`作为`s[k]`。 

这样做的原因是前缀中的每个反转`k`要么已经存在于前缀中`k-1`或者涉及新添加的元素`A[k]`。 没有其他方法可以形成新的反演，因此更新步骤完全捕获了增量。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class Fenwick:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)

    def add(self, i, v):
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

n = int(input())
a = list(map(int, input().split()))

fw = Fenwick(n)
inv = 0
out = []

for i, x in enumerate(a, 1):
    leq = fw.sum(x)
    inv += (i - 1 - leq)
    fw.add(x, 1)
    out.append(str(inv))

print(" ".join(out))
```芬威克树是这里的核心结构。 这`sum(x)`调用计算之前有多少个元素 ≤`x`，因此从中减去它`(i-1)`给出的元素数量大于`x`，它直接对应于这一步引入的新反转。 

一个常见的实现陷阱是忘记 Fenwick 树中的索引是从 1 开始的，这就是直接使用这些值的原因，因为排列值已经位于`1..n`。

 ## 边缘情况

 对于严格递增的排列，例如`[1, 2, 3, ..., n]`，芬威克树总是报告大于当前值的零个元素，因此任何更新都不会增加反转计数。 该算法正确输出恒定的零序列，因为每个查询`fw.sum(x)`总是等于`i-1`。 

对于严格递减排列，例如`[n, n-1, ..., 1]`，每个新元素都小于所有先前的元素。 在步骤`k`，查询返回零`fw.sum(x)`，所以我们添加`(k-1)`每次。 这会产生正确的反转计数三角序列，并且芬威克树正确地累积所有贡献而不会丢失交叉前缀依赖性。
