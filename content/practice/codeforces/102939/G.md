 ## 解决方案
 ## 问题理解

 任务是引导机器人穿过代表滑雪场的矩形网格，从网格的左侧移动到右侧。 每个单元格要么被阻挡，要么是正常的雪，要么是斜坡。 机器人可以从第一列中任何可到达的雪单元开始，并可以在最后一列中任何可到达的雪单元上结束。 目标是尽量减少移动次数。 

移动主要限于每一步向右移动一列。 机器人可以从正常的单元格移动到下一列中的三个相邻单元格之一：向右直行、向右上方或向右下方，前提是这些单元格未被阻挡。 这使得网格在列索引方面实际上成为有向非循环结构，但行维度仍然会产生分支。 

坡道引入了第二种类型的过渡。 当机器人踏上坡道单元时，它不会简单地移动到相邻的单元。 相反，它会沿着固定的“下坡”方向发射并降落在第一个可用的透明单元上。 准确地解释该语句，这意味着斜坡会触发强制跳跃，跳过中间单元格，直到找到有效的着陆单元格，并且整个跳跃仅花费一次移动。 

重要的结果是每个单元格转换都有单位成本，但某些转换会跳过许多中间网格位置。 这立即暗示了有向图上的最短路径问题，最多可达$N \cdot M$节点，其中边缘要么是本地移动，要么是从斜坡预先计算的长跳跃。 

限制条件$N, M \le 1000$意味着最多一百万个节点。 任何尝试动态扩展路径而不进行仔细预处理的解决方案都会遇到困难，特别是如果每​​个步骤都简单地进行斜坡模拟。 需要线性或近线性图遍历，例如隐式图上的 BFS。 

一些边缘情况很容易处理不当。 

斜坡可能会跳过长长的阻塞单元链，因此每个查询天真的“一步一步移动直到碰到某个东西”如果重复的话会太慢。 另一个问题是，坡道与其着陆单元之间可能存在多个坡道或障碍物，并且只有第一个有效的着陆单元很重要。 最后，起始位置和结束位置不是固定点，而是第一列和最后一列中可能的单元格的集合，因此算法必须同时将它们全部视为源和汇。 

## 方法

 蛮力的想法是将每个单元视为一个状态，每当我们遇到斜坡时，通过沿斜坡方向向前扫描来模拟其效果，直到找到着陆单元。 每个 BFS 扩展都会沿着网格重复行走以确定匝道目的地。 在最坏的情况下，单个斜坡转换可以扫描$O(N)$或者$O(M)$细胞，这可能发生在每个边缘松弛上，导致总复杂度接近$O(N^2 M)$或者更糟。 对于一百万个细胞来说，这是不可行的。 

关键的观察结果是，斜坡过渡是确定性的，并且仅取决于网格几何形状，而不取决于到达它们所采取的路径。 这意味着每个斜坡单元都有一个可以预先计算一次的固定目的地。 经过此预处理后，整个网格变成一个标准的未加权图，其中每个节点最多有四个传出边：三个正常移动和一个斜坡跳跃。 

一旦图形明确，就可以使用多源 BFS 从第一列中的所有有效单元格开始找到最短路径。 BFS 自然地传播最小距离值，当我们第一次到达最后一列中的任何单元格时，就会给出最佳答案。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 每一步的简单模拟 |$O(N^2 M)$|$O(NM)$| 太慢了 |
 | 预计算斜坡 + BFS |$O(NM)$|$O(NM)$| 已接受 |

 ## 算法演练

 1. 识别第一列中的所有有效起始状态。 第 0 列中的每个非阻塞单元都被插入到距离为零的队列中。 这反映出滑雪者可以从左边缘的任何地方开始。 
2. 预先计算每个斜坡单元的效果。 对于每个包含坡道的单元，模拟其发射一次并存储着陆单元。 模拟遵循斜坡的固定方向，直到找到透明单元。 此步骤对每个斜坡单元执行一次，以便以后的遍历不会重复工作。 
3. 为每个单元构建隐式转换。 如果它们有效，则允许从正常单元格移动到下一列中的三个相邻单元格。 从斜坡单元，将单个有向边添加到其预先计算的着陆单元。 
4. 使用队列对该图运行 BFS。 每次转换的成本为 1，因此 BFS 保证我们第一次到达节点时会通过尽可能短的移动次数。 
5. 跟踪最后一列中所有单元格之间的最佳距离。 一旦 BFS 到达最右边的列单元格，就更新答案。 所有此类到达的最小值就是最终结果。 

### 为什么它有效

 状态空间在列维度上形成有向非循环结构，因此循环不能仅由水平运动产生。 坡道边缘也总是使滑雪者在有效进展方面向前推进，因为它们跳过中间地形而不是重新访问之前的列。 这确保了 BFS 按路径长度递增的顺序探索状态，而不会错过任何通过坡道的替代较短绕道。 因为每个有效的移动在预先计算的图中都被精确地表示一次，所以该图上的 BFS 相当于原始隐式移动系统上的最短路径搜索。 

## Python 解决方案```python
import sys
input = sys.stdin.readline
from collections import deque

def solve():
    n, m = map(int, input().split())
    g = [list(input().strip()) for _ in range(n)]

    # Precompute ramp destinations
    # direction: assumed (r-1, c+1), moving "up-right"
    dest = [[None] * m for _ in range(n)]

    def find_landing(r, c):
        rr, cc = r - 1, c + 1
        while 0 <= rr < n and 0 <= cc < m:
            if g[rr][cc] == '.':
                return (rr, cc)
            rr -= 1
            cc += 1
        return None

    for i in range(n):
        for j in range(m):
            if g[i][j] == '>':
                dest[i][j] = find_landing(i, j)

    INF = 10**18
    dist = [[INF] * m for _ in range(n)]
    q = deque()

    # multi-source start
    for i in range(n):
        if g[i][0] != '#':
            dist[i][0] = 0
            q.append((i, 0))

    while q:
        r, c = q.popleft()
        d = dist[r][c]

        if c == m - 1:
            continue

        # normal moves to next column
        for dr in (-1, 0, 1):
            nr, nc = r + dr, c + 1
            if 0 <= nr < n and 0 <= nc < m and g[nr][nc] != '#':
                if dist[nr][nc] > d + 1:
                    dist[nr][nc] = d + 1
                    q.append((nr, nc))

        # ramp transition
        if g[r][c] == '>' and dest[r][c] is not None:
            nr, nc = dest[r][c]
            if dist[nr][nc] > d + 1:
                dist[nr][nc] = d + 1
                q.append((nr, nc))

    ans = min(dist[i][m - 1] for i in range(n))
    print(ans)

if __name__ == "__main__":
    solve()
```一旦斜坡过渡标准化为显式边缘，BFS 就是标准的。 微妙的部分是确保斜坡目的地计算一次并重复使用，因为在遍历期间重新计算它们会导致性能下降。 

一个值得注意的细节是，斜坡转换仅在站在斜坡单元上时才会触发，而不是在通过相邻区域穿过它时触发。 另一个原因是所有有效的起始单元必须首先排队，否则 BFS 将错误地采用单个起始状态。

 ## 边缘情况

 关键的边缘情况是来自单元的唯一有用的转换是斜坡。 在这种情况下，未能预先计算着陆单元将迫使重复扫描并降低性能。 通过预处理，算法直接使用存储的目的地。 

另一种情况是当多个坡道沿对角线路径分布时。 一种简单的方法可能会重复遍历中间斜坡单元，但 BFS 将每个斜坡视为单个边缘，确保正确的最小计数。 

最后，当起始列仅包含除一条孤立路径之外的阻塞单元时，多源 BFS 正确地将探索限制在该单个可行的入口点，从而避免网格中其他地方不必要的状态扩展。
