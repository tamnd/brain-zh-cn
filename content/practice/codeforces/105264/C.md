 ## 解决方案
 ## 问题理解

 我们得到一个整数数组，并且允许我们执行有限数量的单位调整。 每个操作都会选择一个位置，并将该值精确地增加或减少 1。 最多使用后$k$此类操作的目标是使数组在值多样性方面尽可能“统一”，这意味着我们希望最大限度地减少数组中保留的不同整数的数量。 

关键的自由是我们不需要使所有元素相等。 我们只想减少不同值的数量，因此我们可以通过将数字组移向共同值来合并数字组，并支付等于绝对差值的成本。 

这些限制非常具有启发性。 所有测试用例中每个测试的数组大小总共最多为 300，因此对值进行二次或三次推理就可以了。 操作次数$k$可以大到$10^{12}$，这立即告诉我们，任何解决方案都必须根据总成本进行推理，而不是逐步模拟操作。 我们需要有效地计算最小调整成本，然后决定在预算下可以合并多少组。 

一个天真的错误是认为我们可以贪婪地将数字“推”向最常见的值。 但这会失败，因为最佳合并值不一定是现有值，并且分组决策会相互作用。 

例如，考虑一个像这样的数组$[1, 10, 20]$与一个小$k$。 贪婪策略可能会尝试将所有内容转换为 10，但最优策略可能会形成两个簇，例如$[1, 10]$和$[20]$取决于预算。 另一个微妙的情况是值相距较远但稀疏。 除非我们正确地考虑了成本，否则按排序顺序合并相邻对并不总是最佳的。 

## 方法

 暴力视图是选择我们想要保留哪些值作为最终的不同组，并将每个元素分配给这些组之一，并支付将其移动到所选代表值的成本。 对于固定的组选择，我们可以将最小成本计算为一条线上的聚类问题。 

这变得昂贵，因为有多种方法可以选择哪些组保持不同，并且对于每个组，我们需要计算移动成本。 即使与$n \le 300$，枚举所有分区是不可能的。 

关键的观察结果是，对数组进行排序后，最佳组对应于连续的段。 如果我们决定以$g$不同的值，我们有效地将排序数组划分为$g$段，并且每个段将合并为一个选定的值，以最小化成本，这是段的中值。 某个细分市场的成本就是与其中位数的绝对差值之和。 

这将问题转变为选择将排序数组划分为尽可能少的段，以使总成本在预算之内$k$。 自从$k$很大时，我们不是最大化成本使用，而是最小化段数量。 

我们使用前缀和预先计算每个段的成本，然后使用动态规划，其中$dp[g][i]$是首先分区的最小成本$i$元素进入$g$段。 答案是最小的$g$这样$dp[g][n] \le k$。

 | 蛮力 | 指数| 指数| 太慢了 |
 | DP 与分段成本 |$O(n^3)$每次测试 |$O(n^2)$| 已接受 |

 ## 算法演练

 我们首先对数组进行排序，使组成为一条线上的间隔，这使得成本结构凸且表现良好。 

## 算法演练

 1. 以非降序对数组进行排序，以便任何最佳分组都遵循顺序。 这是有效的，因为与在排序结构中重新分配相比，跨顺序混合元素总是会增加调整成本。 
2. 预先计算排序数组的前缀和。 这允许在预处理后在恒定时间内快速计算分段成本。 
3. 对于每个细分市场$[l, r]$，计算使所有元素等于单个值的最小成本。 最佳目标是中值元素，因此我们将成本分为左右部分，并使用前缀和进行计算。 此步骤构建成本表。 
4. 定义一个 DP 状态，我们计算分割第一个状态的最小成本$i$元素精确地$g$组。 通过尝试最后的剪切位置进行过渡$j$，将先前的最优解与分段成本相结合$[j+1, i]$。 
5. 迭代可能的组数，从 1 到$n$，并针对每个计算整个阵列的最佳可实现成本。 
6. 成本不超过的最小组数$k$就是答案。 

关键的推理步骤是，一旦排序，结构就会简化为划分一条线，并且段内的最佳合并总是会折叠到中位数。 

### 为什么它有效

 排序强制任何最佳分组都可以转换为连续的段，而不会增加成本，因为交叉分配会引入不必要的距离。 在每个段内，中位数最小化绝对偏差，这是$L_1$优化上线。 DP 探索所有可能的分段，保证如果配置$g$在预算范围内可能有不同的最终值，我们将予以考虑。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve_case(n, k, arr):
    arr.sort()

    prefix = [0] * (n + 1)
    for i in range(n):
        prefix[i + 1] = prefix[i] + arr[i]

    def cost(l, r):
        m = (l + r) // 2
        median = arr[m]

        left_sum = median * (m - l + 1) - (prefix[m + 1] - prefix[l])
        right_sum = (prefix[r + 1] - prefix[m + 1]) - median * (r - m)
        return left_sum + right_sum

    INF = 10**18

    dp = [[INF] * n for _ in range(n + 1)]
    for i in range(n):
        dp[1][i] = cost(0, i)

    for g in range(2, n + 1):
        for i in range(n):
            best = INF
            for j in range(i):
                best = min(best, dp[g - 1][j] + cost(j + 1, i))
            dp[g][i] = best

    for g in range(1, n + 1):
        if dp[g][n - 1] <= k:
            return g

    return n

def main():
    t = int(input())
    for _ in range(t):
        n, k = map(int, input().split())
        arr = list(map(int, input().split()))
        print(solve_case(n, k, arr))

if __name__ == "__main__":
    main()
```代码首先对数组进行排序，这对于将问题简化为区间划分至关重要。 前缀总和可实现恒定时间段总和查询，这是快速成本评估所需的。 

这`cost(l, r)`函数实现基于中值的绝对偏差公式。 中值索引处的分割确保了最小的移动成本，并且前缀和避免了重复重新计算段总计。 

DP 表按组数增量构建解决方案。 对于每个端点，我们尝试所有先前的分割点，这就是$O(n^3)$复杂性出现了。 自从$n \le 300$，这仍然是可行的。 

最后，我们扫描成本不超过的最小数量的组$k$，与问题目标直接匹配。

## 测试用例```python
import sys, io

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    import sys
    input = sys.stdin.readline

    def solve():
        t = int(input())
        out = []
        for _ in range(t):
            n, k = map(int, input().split())
            a = list(map(int, input().split()))
            a.sort()
            pref = [0]
            for x in a:
                pref.append(pref[-1] + x)

            def cost(l, r):
                m = (l + r) // 2
                med = a[m]
                left = med * (m - l + 1) - (pref[m+1] - pref[l])
                right = (pref[r+1] - pref[m+1]) - med * (r - m)
                return left + right

            INF = 10**18
            n = len(a)
            dp = [[INF]*n for _ in range(n+1)]
            for i in range(n):
                dp[1][i] = cost(0, i)

            for g in range(2, n+1):
                for i in range(n):
                    for j in range(i):
                        dp[g][i] = min(dp[g][i], dp[g-1][j] + cost(j+1, i))

            for g in range(1, n+1):
                if dp[g][n-1] <= k:
                    out.append(str(g))
                    break
        return "\n".join(out)

    return solve()

 ## 边缘情况

 一个关键的边缘情况是当$k = 0$。 在这种情况下，不允许进行任何调整，因此答案只是已经存在的不同值的数量。 DP 自然会处理这个问题，因为每个段成本必须为零，只有当段包含相同值时才会发生这种情况。 

另一个微妙的情况是当数组有很大的间隙时，例如$[1, 100, 101, 200]$。 天真的贪婪合并可能会尝试将所有内容拉向一个值，但最佳分区会分裂成紧密的集群，例如$[100,101]$和单例，DP 捕获它们，因为分段成本是按间隔精确计算的，而不是按全局吸引力计算的。 

最后一个边缘情况是$k$非常大，高达$10^{12}$。 这可能表明存在数字溢出问题，但所有计算仍保持在 64 位范围内，因为成本受以下限制$n \cdot 10^9$，它安全地适合标准整数算术。
