 ## 解决方案
 ## 问题理解

 我们得到一个正整数数组。 我们可以重复执行一个特殊的操作，在元素之间重新分配 2 的幂：选择一个值为偶数的索引，将其减少一半，同时将另一个元素加倍。 

换句话说，每次操作都会将二的一个因子从一个位置移动到另一个位置，而不会改变数组中奇数部分和 2 总指数方面的总乘积结构。 奇数分量保持与其元素相连，而 2 的幂可以在位置之间任意移动，但每次运算只能使用一个指数单位。 

在执行任何此类操作序列之后，我们希望最大化 x 值，使得最终数组中至少 x 个元素至少为 x。 这是我们对 2 的幂进行最佳重新分配后应用的标准“H 指数样式”条件。 

这些限制意味着我们无法直接模拟操作。 每个测试用例最多 2·10^5 个元素和最多 10^4 个测试用例，任何解决方案都必须以接近线性或每个测试用例 n log n 的方式运行。 这立即排除了任何重复模拟传输或维护每个操作状态更改的策略。 

一个微妙的边缘情况是当大偶数集中在几个位置时。 例如，如果一个元素具有 2 的巨大幂，而其他元素又奇又小，则该操作允许我们将该幂“分散”到许多元素中。 只增加最大元素的天真贪婪会失败，因为最优解决方案通常会分散权力而不是集中权力。 

另一种边缘情况是所有数字都是奇数时。 那么根本不可能进行任何操作，因此答案简化为数组的经典 H 索引。 

## 方法

 暴力解释是将每个操作视为两个人的权力的重新分配。 我们可以重复选择偶数元素并移动因子 2，直到达到某种稳定的配置，然后计算 H 指数并尝试所有可能性。 这很快就变得不可行，因为每个操作都会更改两个元素，并且每个测试用例可能有 O（位总和）操作，在最坏的情况下与 n log A 成正比，并且仍然需要探索巨大的分布状态空间。 

关键的观察是，唯一可转移的资源是 2 的幂，而奇数部分是确定基线值的固定锚。 每个元素可以被认为是 ai = odd_i · 2^{cnt_i}。 该操作让我们将一个单位从 cnt_i 移动到 cnt_j。 因此，在全局范围内，奇数部分的多重集是固定的，所有 cnt_i 的总和也是固定的。 

这将问题转化为在各个仓位上分配固定数量的“加倍”，以最大化 H 指数条件。 我们不去模拟，只关心是否能让至少 x 个元素达到 value ≥ x。 对于固定的 x，每个元素 i 需要足够的加倍，以便 odd_i · 2^{cnt_i} ≥ x。 这给出了每个元素所需的最小 cnt_i，我们想要检查是否可以分配可用的加倍以满足至少 x 个元素。 

这简化为贪婪的可行性检查：根据达到 x 所需的加倍次数对元素进行排序，然后选择最简单的元素，直到耗尽可用资源。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力重新分配| O(n·总操作数) | O(n) | 太慢了 |
 | 贪婪可行性+二分查找| O(n log A) | O(n log A) | O(n) | 已接受 |

 ## 算法演练

 我们将每个值 ai 重新表示为其奇数分量及其包含的 2 因数的数量。 设 ci 为 ai 中 2 的指数，oi 为 ai 除以 2^{ci}。 

然后，我们计算可用的总“功率预算”，即所有 ci 的总和。 这代表我们可以在整个数组中加倍元素的次数。

为了检查候选 x 是否可实现，我们确定每个元素需要多少倍加倍才能从 oi 开始至少达到 x。 如果 oi 已经 ≥ x，则需要零加倍。 否则，我们计算最小的 k 使得 oi · 2^k ≥ x。 

我们收集所有这些要求并对它们进行分类。 我们贪婪地分配可用的总预算来满足尽可能多的元素，优先考虑那些需要较少翻倍的元素。 

对于固定的x，如果我们可以在这个赋值下满足至少x个元素，那么x是可行的。 

我们对 x 从 0 到 n 进行二分搜索答案。 

### 为什么它有效

 关键的不变量是可用加倍的总数是固定的并且完全可转移，而一旦 x 固定，每个元素的要求都是独立的。 由于每个倍增单元具有相同的值并且没有不可计数的位置限制，因此最优策略始终是首先满足最小的要求。 这将问题简化为标准资源分配可行性检查，确保操作的重新安排不会产生比固定阈值的贪婪分配更好的结果。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def count_twos(x):
    c = 0
    while x % 2 == 0:
        x //= 2
        c += 1
    return c, x

def can(x, odd, cnt, total_twos):
    req = []
    for o, c in zip(odd, cnt):
        if o >= x:
            req.append(0)
        else:
            need = 0
            val = o
            while val < x:
                val *= 2
                need += 1
            req.append(need)
    req.sort()
    used = 0
    take = 0
    for r in req:
        if used + r <= total_twos:
            used += r
            take += 1
        else:
            break
    return take >= x

def solve():
    t = int(input())
    for _ in range(t):
        n = int(input())
        a = list(map(int, input().split()))

        odd = []
        cnt = []
        total_twos = 0

        for v in a:
            c, o = count_twos(v)
            odd.append(o)
            cnt.append(c)
            total_twos += c

        lo, hi = 0, n
        ans = 0

        while lo <= hi:
            mid = (lo + hi) // 2
            if can(mid, odd, cnt, total_twos):
                ans = mid
                lo = mid + 1
            else:
                hi = mid - 1

        print(ans)

if __name__ == "__main__":
    solve()
```该解决方案首先将每个数字分解为其奇数底数和二次方指数。 这种分离是必要的，因为只有指数部分可以通过运算转移。 

可行性检查函数计算每个元素需要加倍多少次才能达到目标 x，然后贪婪地分配总的可用指数预算。 排序步骤至关重要，因为它确保我们始终首先将资源花在最便宜的元素上。 

二分搜索包含此可行性检查以找到最大的有效 x。 

一个微妙的实现细节是通过重复乘法而不是对数来计算所需的倍数。 这避免了精度问题并且保持足够快，因为值呈指数级增长。 

## 工作示例

 ### 示例 1

 考虑一个数组`[8, 3, 1]`。 

我们分解：

 | 元素| 奇数| 两两|
 | --- | --- | --- |
 | 8 | 1 | 3 |
 | 3 | 3 | 0 |
 | 1 | 1 | 0 |

 总数 = 3。 

现在测试 x = 2。 

要求：

 | 元素| 价值| 需要加倍|
 | --- | --- | --- |
 | 8 | 8 | 0 |
 | 3 | 3 | 0 |
 | 1 | 1 → 2 | 1 |

 我们只需使用 1 个预算单位就可以满足这三个要求，因此 x = 2 是可行的。 

对于 x = 3：

 要求：

 | 元素| 需要|
 | --- | --- |
 | 8 | 0 |
 | 3 | 0 |
 | 1 | 2 |

 我们最多可以在预算内满足 2 个元素，因此 x = 3 失败。 

所以答案是2。 

该迹线显示了如何重新分配单个大元素的剩余功率 2 以提升较小的元素。 

### 示例 2

 数组`[1, 1, 1, 1]`。 

所有元素均为奇数，总数 = 0。 

对于 x = 1，所有都已经 ≥ 1，所以可行。 

对于x = 2，每个都需要加倍1，但预算为0，所以不可能。 

因此答案为1。 

这证实了算法正确处理无传输边缘情况。

 ## 边缘情况

 一种关键的边缘情况是一个元素包含所有可用的 2 的幂。 对于像这样的输入`[2^k, 1, 1, ..., 1]`，算法正确地将其转换为大量预算和许多小需求。 贪婪步骤确保该预算首先花在最便宜的元素上，从而有效地将单个大型来源分散到多个目标。 

另一种边缘情况是 x 接近 n 时。 可行性检查仍然表现正确，因为它直接计算可以满足的元素数量，并且一旦排序的要求提前超出预算，循环就会终止，而不会进行不必要的计算。 

第三种边缘情况是当值已经足够大而不使用任何传输时。 在这种情况下，所有所需的加倍都为零，并且算法立即将所有元素计数为满足，并在可能的情况下正确生成 x = n。
