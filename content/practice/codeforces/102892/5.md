 ## 解决方案
 ## 问题理解

 我们获得了一系列电视频道，每个频道播放一系列分钟。 对于每个频道，每一分钟要么“有趣”，要么“不有趣”。 如果某一分钟很有趣，则在该分钟观看该频道会给出与该频道相关的固定享受值； 否则它什么也没有给出。 

在任何一分钟，电视都可以同时录制多达一定数量的频道。 决策问题是：如果我们最多允许记录`c`每分钟频道数，我们在所有频道和所有分钟内可以积累的最大总享受是多少？ 我们不需要以超出每分钟限制的受限方式每分钟主动选择不同的频道，因此可以根据我们选择录制的频道来独立对待每一分钟。 

任务是找到最小的`c`这样最大可实现的总享受至少是`k`。 如果即使每分钟记录所有通道也达不到`k`，我们必须输出`-1`。 

输入大小足够小，以至于问题的维度（通道数和分钟数）最多为 1000。这立即表明，在优化的 C++ 中，任何多达大约几亿个基本运算的解决方案仍然可接受，但 Python 解决方案必须避免三次或大量的每个候选重新计算。 

关键的限制是`k`可以大到 10^18，这迫使解决方案仔细累积大值，并清楚地表明我们正在优化全局数量，而不是进行组合搜索。 

幼稚的解读可能会表明每分钟的选择会带来组合爆炸，但该结构消除了这种依赖性。 

当任何地方都不存在有趣的内容时，就会出现微妙的边缘情况。 如果每个通道都为零，则所有增益都为零，无论`c`，所以答案一定是`-1`。 第二个边缘情况是当`k`非常小，例如`k = 1`，即使只有一个最优选择就足够了。 第三种情况是一个频道每分钟占据主导地位，这意味着增加`c`Beyond 1 在第一次选择后不执行任何操作。 

## 方法

 蛮力方法确定了一个值`c`并模拟最佳可能的策略。 对于每一分钟，我们都会计算记录并累积增益的通道。 然而，关键的观察结果是，由于通道不会在几分钟内交互，并且每分钟独立贡献附加值，因此固定的最佳选择`c`总是要选择`c`当前贡献最大的频道。 这意味着我们每分钟都会根据贡献对频道进行排序并选择顶部`c`。 

为每个候选人独立执行此操作`c`从 1 到`n`导致复杂度大致为`O(n^2 * m log n)`，因为对于每个`c`每分钟我们都会对最高值进行排序或选择。 和`n, m ≤ 1000`，这在 Python 中已经太慢了。 

关键的见解是反转观点：我们不是询问“每分钟固定数量的通道，最好的分数是多少”，而是计算每分钟每个通道在所有分钟内总共贡献了多少。 也就是说，我们预先计算每个频道的总“利润”，因为每次我们在感兴趣的一分钟选择一个频道时，我们总是获得相同的特定于频道的值。 这将问题转化为静态选择问题：每个通道都有一个总的潜在增益，选择一个通道意味着我们可以收集它的所有贡献。 

这将整个问题简化为按总增益对通道进行排序并采用前缀。 答案变成最小的`c`这样顶部的总和`c`信道增益至少为`k`。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 |---|---|---|---|
 | 暴力破解每`c`每分钟重新计算| O(n² m log n) | O(n² m log n) | O(纳米) | 太慢了 |
 | 预先计算总计并排序通道 | O(nm + n log n) | O(nm + n log n) | O(n) | 已接受 |

 ## 算法演练

 1. 通过扫描其二进制时间表，计算每个通道在所有分钟内的总贡献。 每当某一分钟被标记为有趣时，请添加该频道的娱乐价值。 此步骤将时间结构压缩为每个通道的单个标量。 

2. 将这些总计存储在数组中。 现在，无论调度如何，每个条目都代表了我们可以从该通道中提取的最大可能贡献。 

3. 按降序对总计数组进行排序。 这确保了如果我们决定“激活”`c`渠道，我们总是首先选择最有价值的渠道。 

4. 在已排序的数组上构建前缀和。 如果允许我们每分钟记录多达那么多通道，每个前缀总和代表可实现的最佳分数。 

5. 找到最小的索引`c`使得前缀和至少为`k`。 那`c`就是答案。 

6. 如果所有通道的总和小于`k`， 输出`-1`。 

微妙之处在于，排序不是关于时间排序，而是关于最大化独立的附加贡献，因为一旦聚合，通道就不会跨分钟竞争。 

### 为什么它有效

 正确性依赖于一个简单的交换论证。 对于任何固定的`c`，最优策略总是优先选择总贡献较高的通道，因为用较高贡献的通道替换较低贡献的通道永远不会降低任何一分钟的增益，并且会严格增加总增益。 由于超出每分钟上限的分钟内通道之间没有耦合，因此这种贪婪的选择是全局最优的。 因此，对于任何情况，最优解都是`c`仅取决于通道总数的多重集，并选择顶部`c`总数就足够了。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

n, m, k = map(int, input().split())
a = list(map(int, input().split()))

totals = [0] * n

for i in range(n):
    s = input().strip()
    ai = a[i]
    total = 0
    # accumulate contribution over all interesting minutes
    for ch in s:
        if ch == '1':
            total += ai
    totals[i] = total

totals.sort(reverse=True)

prefix = 0
answer = -1

for i, val in enumerate(totals):
    prefix += val
    if prefix >= k:
        answer = i + 1
        break

print(answer)
```该解决方案首先将每个通道的时间表转换为单个累计值。 这可以避免以后模拟每分钟的决策。 排序步骤是唯一的全局耦合操作，它确保增加`c`始终向最佳方向扩展解决方案。 然后前缀扫描直接回答单调可行性条件。 

一个常见的实施错误是尝试重新计算每分钟的最佳选择`c`，这会重新引入不必要的第二维并导致超时。 另一个原因是忘记了通道贡献是独立的，这会引发涉及每分钟复杂的贪婪匹配的解决方案。

## 测试用例```python
import sys, io

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    import sys
    input = sys.stdin.readline

    n, m, k = map(int, input().split())
    a = list(map(int, input().split()))

    totals = [0] * n
    for i in range(n):
        s = input().strip()
        ai = a[i]
        for ch in s:
            if ch == '1':
                totals[i] += ai

    totals.sort(reverse=True)

    prefix = 0
    ans = -1
    for i, v in enumerate(totals):
        prefix += v
        if prefix >= k:
            ans = i + 1
            break

    return str(ans)

# edge: impossible
assert run("2 3 10\n5 4\n000\n000\n") == "-1"

# edge: single channel sufficient
assert run("1 3 5\n2\n111\n") == "1"

# edge: all channels needed
assert run("3 3 10\n1 1 1\n111\n111\n111\n") == "3"

# edge: mixed pattern
assert run("4 4 7\n1 2 3 4\n1010\n0101\n1111\n0000\n") == "2"
```| 测试输入| 预期产出 | 它验证了什么 |
 |---|---|---|
 | 全零| -1 | 不可能的情况|
 | 单通道 | 1 | 最小边界|
 | 完整要求 | 3 | 需要所有渠道 |
 | 混合 | 2 | 贪婪前缀正确性 |

 ## 边缘情况

 一个完全空的时间表，其中每个通道字符串仅包含零，为每个通道产生总的零贡献。 在这种情况下，前缀和永​​远不会增加，因此算法正确返回`-1`。 

单通道输入的处理很简单，因为前缀和循环在添加第一个元素后进行检查，因此如果该通道单独达到`k`，答案是`1`。 

当所有通道在所有分钟内都完全激活时，每个总计变为`value × m`，并且排序简化为按值排序。 该算法仍然表现正确，因为按比例缩放`m`保留排序。 

仅后期通道有用的情况并不重要，因为时间结构已经分解为总数； 该算法完全忽略时间分布，这是安全的，因为每分钟的独立性消除了排序效应。
