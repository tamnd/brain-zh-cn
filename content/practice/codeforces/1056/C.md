 ## 解决方案
 ## 问题理解

 两名玩家通过交替从公共池中选取英雄来组建两支同等规模的队伍$2n$候选人。 每个英雄都有固定的力量，一旦被占领，它就会从游戏中消失。 除此之外，一些英雄以特殊的配对形式出现：如果选择了这样一对英雄中的一个，则该对中的另一个端点将立即被对方玩家在下一步行动中强制选择。 这意味着选择一对中的一个成员不仅仅会删除单个项目，它还可以有效地从未来的决策中删除两个端点，同时还根据移动顺序将其中一个端点分配给每个玩家。 

你方的目标是最大化你最终拥有的英雄的力量总和，而对手则遵循相同的规则，并试图获得尽可能多的收益。 交互方面很重要：你必须一一输出你的动作，对对手的选择做出反应，并尊重强制的配对约束。 

这些限制足够严格，以至于任何涉及搜索所有移动序列或模拟博弈树的方法都是不可能的。 高达$2n \le 2000$，甚至一个$O(n^2)$模拟是边缘性的，但只有在每一步都极其简单的情况下才仍然可行。 然而，完整的极小极大策略是不必要的，因为强制对的结构将游戏简化为动态收缩集上的更简单的贪婪过程。 

当人们试图仅在本地进行推理而不考虑强制删除时，就会出现一种微妙的失败情况。 例如，如果一个低价值英雄与一个非常高价值的英雄配对，忽略配对的天真贪婪策略可能会过早选择低价值英雄，从而让对手稍后占据高价值终点。 

考虑这个配置：

 输入：```
n = 2
p = [10, 1, 9, 2]
pair: (1, 3)
```如果一个幼稚的策略在不考虑交互时间的情况下选择最好看的可用移动，它可能需要 10 个，而对手则需要 9 个，而后来的强制结构可能会导致次优分配。 正确的行为取决于了解获取一个端点会从系统中删除整个端点对。 

## 方法

 思考游戏的暴力方式是模拟所有可能的移动序列，跟踪轮到谁，并维持选择一对端点中的一个端点强制在对手的下一步移动中删除另一个端点的约束。 这很快就会导致一个分支过程：每一步都有最多$O(n)$可能的移动，深度是$2n$，产生指数数量的状态。 即使使用记忆，交互状态也取决于确切的剩余集合，这使得这个问题变得棘手。 

关键的观察结果是，强制配对规则不会引入选择复杂性，它只会改变删除时间。 一旦选择了英雄，其配对（如果存在）将在下一步立即解决，这意味着无论策略如何，两个端点都会在相邻回合中离开游戏。 从未来决策的角度来看，每个动作只是从池中删除最多两个顶点，但删除的顺序并不会创建超出谁获得哪个端点的长期战略结构。 

这将问题简化为动态集合上的经典贪婪过程：在每一回合，您选择一个当前可用的英雄，对手将通过选择一个被迫的伙伴或任意可用的英雄来做出响应。 由于双方总是对相同的剩余英雄池采取行动，唯一稳定的策略是在轮到你的时候总是选择价值最高的可用英雄。 任何偏差都只能用低价值的选秀权来取代高价值的选秀权，而不会创造补偿性的未来优势，因为移除并不能保留隐藏的未来收益。 

因此，游戏就陷入了按顺序重复提取最大剩余价值的过程。

 ## 算法演练

 我们维护一个结构，始终允许我们查询尚未被采用的最高价值英雄。 

1.读取所有英雄及其数值，并记录哪些英雄配对。 这只是为了确保我们尊重有效性，而不是为了决策。 
2. 将所有英雄插入按力量递减排序的结构中。 这定义了我们想要使用它们的顺序。 
3. 维护一个布尔数组，标记每个英雄是否已被占领。 
4. 在你的回合中，重复选择仍然可用的最高价值的英雄。 
5.输出选择后，等待对手响应，并将该英雄标记为已吃掉。 
6. 继续交替，直到所有英雄都被消耗掉，确保每次选择总是选择最好的剩余选项。 

唯一的微妙之处在于，“可用”必须反映直接选择和由于配对限制而强制移除。 然而，由于每个被删除的英雄都会立即被标记为不可用，因此我们绝不会在以后的选择中意外地重复使用它。 

### 为什么它有效

 每时每刻，剩下的英雄们都充分捕捉到了未来的所有可能性。 对约束的影响纯粹是局部的：选择一个英雄可能会删除它的伙伴，但这不会在剩余的图中保留或创建任何额外的结构。 因此，每一步唯一有意义的决定是立即采用哪个可用值。 任何跳过价值较高的可用英雄而选择价值较低的英雄的策略都会严格减少最终总和，而不会以补偿方式提高未来的可用性，因为可用性仅通过删​​除而变化，而不是通过重新排列或解锁。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

n, m = map(int, input().split())
p = list(map(int, input().split()))

pair = [-1] * (2 * n)
for _ in range(m):
    a, b = map(int, input().split())
    a -= 1
    b -= 1
    pair[a] = b
    pair[b] = a

t = int(input())

used = [False] * (2 * n)

order = sorted(range(2 * n), key=lambda i: p[i], reverse=True)

ptr = 0

def get_best():
    global ptr
    while ptr < 2 * n and used[order[ptr]]:
        ptr += 1
    return order[ptr] if ptr < 2 * n else -1

for turn in range(2 * n):
    if (turn % 2 == 0 and t == 1) or (turn % 2 == 1 and t == 2):
        x = get_best()
        used[x] = True
        print(x + 1, flush=True)

        if pair[x] != -1:
            used[pair[x]] = True
    else:
        x = int(input())
        if x == -1:
            exit(0)
        x -= 1
        used[x] = True

        if pair[x] != -1:
            used[pair[x]] = True
```该代码根据强度维护英雄的全局排序，并始终前进到下一个未使用的候选者的指针。 每次轮到我们时，我们都会选择仍然可用的最强英雄。 当英雄被任何一方带走时，我们也会立即将其配对伙伴标记为不可用，这反映了强制规则。 

基于指针的扫描避免了重复排序或扫描整个列表，从而保持选择效率。 

一个常见的错误是尝试主动推理首先选择一对端点中的哪个端点。 该实现完全避免了这种情况：全局排序已经保证我们总是尝试最强的剩余移动，这通过强制删除隐式地处理对交互。

 ## 边缘情况

 一个关键的边缘情况是当一个非常高价值的英雄与一个低价值的英雄配对时，并且对手先移动。 在这种情况下，对手可能会立即拿走高价值端点，迫使你稍后再接收低价值端点。 该算法自然地处理这个问题，因为两个端点在对手移动后立即被删除，并且贪婪指针从不尝试重用它们。 结果反映了不可避免的损失，而不是错过的优化机会。 

另一种情况是所有英雄都未配对。 该算法简化为降序值的简单交替选择，并且指针确保每一侧始终采用下一个最佳可用值而不会受到干扰。 

最后，当存在多对时，链式交互永远不会发生，因为每个英雄最多属于一对。 这保证了将配对伙伴标记为已使用永远不会触发递归依赖关系，从而使模拟在行为上保持严格的线性。
