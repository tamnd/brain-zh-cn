 ## 解决方案
 ## 问题理解

 该任务是模拟粘性液体如何在排列成一行的一系列容器中扩散。 每个容器都有固定的容量，当液体倒入一个容器时，它会充满到极限，多余的部分会立即流入下一个容器，继续向前，直到液体被完全吸收或溢出最后一个容器。 

系统支持两种操作。 一项操作将给定量的液体倒入特定容器中，让溢出通过更高索引的容器向前传播。 另一个操作询问在所有先前的倾倒和溢出效应解决之后当前在特定容器中存储了多少液体。 

关键的挑战是，在最坏的情况下，每次倾倒都会影响所有后续容器，并且操作次数高达 200,000 次。 逐步推动流通过数组的简单模拟将在每次查询中花费线性时间，这会导致在最坏情况下总复杂度呈二次方且无法通过。 

当容量较小并且重复倾倒使液体不断重新分布到右侧时，就会出现微妙的边缘情况。 例如，如果容量是`[5, 5, 5]`我们反复倒`5`就位`1`，每个操作可能会一直级联到最后一个容器，即使大多数中间容器已经满了。 即使饱和后没有任何变化，反复向前推进的幼稚实现仍然每次都会遍历所有索引。 

另一个失败案例来自部分填充。 如果容器已经满了，新液体应该完全跳过它。 如果在传播过程中没有正确实施容量限制，总是添加然后纠正溢出的幼稚实现可能会错误地重复计数或重复调整同一容器。 

## 方法

 暴力方法遵循字面描述。 每次倒入时，我们将液体添加到目标容器中，然后反复检查是否超出容量。 如果是这样，我们将其减少到容量并将多余的部分转移到下一个容器，继续直到没有溢出或我们通过了最后一个容器。 每个查询可能会遍历所有`n`容器。 

这是正确的，因为它直接模拟物理过程，但其最坏的情况是灾难性的。 一次大倾注可以传播到所有`n`容器，并且有`m`操作，因此总工作变为`O(nm)`，对于`n, m ≤ 2⋅10^5`。 

关键的观察是溢出的结构是单调的和有方向的。 一旦容器达到满容量，它的行为就像一堵坚固的墙：任何额外的水都会立即穿过它，而不会改变其状态。 这表明我们应该避免重复处理满容器，而是“跳过”它们。 

这可以通过为每个位置维护可能仍然接受水的下一个索引来实现。 从概念上讲，我们将连续的完整容器压缩为单个代表。 浇注时，我们总是移动到下一个未满的位置，当一个位置满时，我们将其向前合并，以便将来的操作完全跳过它。 这种行为自然是通过跟踪下一个可用位置的不相交集联合结构来维护的。 

每个容器一旦装满，就会从活动处理中有效地删除，并且每次删除最多发生一次，因此每次操作的摊余成本几乎保持不变。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(纳米) | O(n) | 太慢了|
 | DSU“下一个可用”跳过 | O((n + m) α(n)) | O((n + m) α(n)) | O(n) | 已接受 |

 ## 算法演练

 我们维护一个数组`cap[i]`用于剩余容量（或等效的当前填充），以及不相交集联合结构`parent[i]`总是指向以下位置或之后的下一个索引`i`那还没有满。 我们还定义了一个虚拟索引`n + 1`代表超出系统范围的溢出。 

1. 初始化`parent[i] = i`对于所有职位，以及`parent[n+1] = n+1`。 所有容器一开始都是空的。 
2. 定义函数`find(x)`返回最小索引 ≥`x`仍然处于活动状态（未完全饱和）。 这使用路径压缩，因此重复查询变得更快。 
3. 处理浇注位置`p`有金额`x`，首先找到第一个可用的容器`i = find(p)`。 
4. 虽然`i`在范围内并且仍然有液体`x > 0`，计算容器可以吸收多少。 如果`cap[i]`至少是`x`，我们简单地减少`cap[i]`经过`x`并完成。 
5.如果`cap[i] < x`，我们完全填满容器，设置`x -= cap[i]`，并标记`cap[i] = 0`。 由于它现在已满，我们通过设置将其从结构中合并出来`parent[i] = find(i + 1)`，在以后的操作中有效地跳过它。 
6. 填充后，通过设置移至下一个可用容器`i = find(i)`再次，因为路径压缩确保我们跳过完整的段。 
7.对于位置的查询操作`k`，简单输出当前存储的数量，即`original_capacity[k] - cap[k]`如果我们跟踪剩余空间，或者直接维护当前填充值。 

核心不变量是`find(i)`始终返回第一个索引 ≥`i`仍有剩余容量。 一旦容器已满，通过将其联合到其后继者，它将被永久排除在未来的考虑之外。 这保证了每个容器最多作为“满事件”处理一次，因此结构更新的总数在整个运行过程中是线性的。 由于溢出的每一步要么完全填满容器，要么立即结束，因此不会不必要地重新访问容器。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

sys.setrecursionlimit(10**7)

def solve():
    n = int(input())
    cap = [0] + list(map(int, input().split()))
    
    parent = list(range(n + 2))

    def find(x):
        while parent[x] != x:
            parent[x] = parent[parent[x]]
            x = parent[x]
        return x

    def union(x):
        parent[x] = find(x + 1)

    m = int(input())

    for _ in range(m):
        tmp = list(map(int, input().split()))
        if tmp[0] == 1:
            p, x = tmp[1], tmp[2]
            i = find(p)
            while i <= n and x > 0:
                if cap[i] >= x:
                    cap[i] -= x
                    x = 0
                else:
                    x -= cap[i]
                    cap[i] = 0
                    union(i)
                i = find(i)
        else:
            k = tmp[1]
            print(cap[k])

if __name__ == "__main__":
    solve()
```代码直接反映了算法。 这`find`函数实现路径压缩，快速定位下一个可用容器。 这`union`操作通过将饱和容器链接到其后继容器，将其从未来考虑中删除。 在倾倒过程中，我们反复跳转到下一个可用容器并部分填充它或完全饱和它，在这种情况下它会被合并出来。 

一个常见的实施陷阱是忘记重新运行`find(i)`工会行动后。 否则，指针可能会停留在完全饱和的索引上。 另一个微妙的问题是处理溢出终止`i`变成`n + 1`，必须立即停止处理。 

## 工作示例

 考虑容量`[5, 5]`带查询：在位置 1 处倒入 1 和 4，然后查询这两个位置。 

| 步骤| 我| 上限[1] | 上限[2] | x|

 第一次操作后，只有第一个容器发生变化。 查询位置1返回1，位置2返回5，确认没有发生溢出。 

现在考虑`[5, 10]`与触发溢出的倾倒。 

| 步骤| 我| 上限[1] | 上限[2] | x|
 | ---| ---| ---| ---| ---|
 | 开始| 1 | 5 | 10 | 10 12 | 12

 | 填充 2 | 2 | 3 | 0 | 0 |

 这显示了溢出如何传播以及第二个容器如何吸收剩余液体。 

跟踪表明，一旦容器变满，它就会在后续操作中被跳过，从而确认并查找跳过机制的正确性。

 ## 边缘情况

 一个关键的边缘情况是早期容器的重复完全饱和。 用于输入`n = 3`, 容量`[1, 1, 1]`，并反复倾倒`1`在位置`1`，第一个操作填充索引 1，第二个操作填充索引 2，第三个操作填充索引 3。之后，所有未来的浇注都应立即终止。 DSU结构确保每个索引被删除一次，因此后面的操作直接跳转到`n + 1`并停止而不扫描已处理的位置。 

另一种边缘情况正在涌入已经满的位置。 例如，如果容器 2 已满并且我们倒入位置 2，算法会立即使用以下命令重定向到下一个可用容器`find(2)`，避免任何不正确的双重处理。
