 ## 解决方案
 ## 问题理解

 我们获得了一个重复的工作流程，可以从头到尾改变每个编程任务。 每个任务按顺序经历四个阶段。 第一阶段是编写代码，第三阶段是测试结果后修复代码，都需要Pasha在场，各自花费时间A。第二阶段是自动化测试，第四阶段是代码审查，都需要时间B，不需要Pasha主动参与，但仍然需要时间完成。 

重要的限制是整个管道必须连续运行。 一旦任务进入流程，各个阶段之间就没有空闲时间，并且任务在单个流程中被一个接一个地处理。 我们需要确定完成 N 个任务的最短总时间。 

关键是每个任务都有一个固定的内部结构：两个长度为 A 的“活动”段分开，后面跟着两个长度为 B 的“非活动”段。但是，由于任务是在管道中处理的，只要尊重依赖性，不同任务的不同阶段可以在时间上重叠。 

输入约束使得无法模拟逐个任务的调度。 N 可以大到 10^9，因此任何线性或每任务模拟都会立即变得太慢。 解决方案必须将问题简化为从管道结构派生的封闭式表达式。 

当 B 明显大于 A 时，就会出现微妙的边缘情况，反之亦然。 例如，如果 A 比 B 小得多，则天真的解释可能会假设空闲等待占主导地位，但重叠的阶段可能会隐藏 B 的大部分。相反，当 A 占主导地位时，管道几乎按顺序运行。 

## 方法

 思考该问题的一种直接方法是模拟每个任务的四阶段管道。 每个任务进入第一阶段，然后进入第二到第四阶段，而后面的任务则在其后面排队。 这自然会导致调度模拟，我们会跟踪每个阶段何时空闲。 

这种模拟是正确的，因为它直接尊重所有约束，但它很快就变得不可行。 每个任务都贡献恒定的工作，因此总复杂度为 O(N)。 当 N 达到 10^9 时，这在时间限制内是不可能的。 

关键的观察结果是，这不是一般的调度问题，而是具有确定性阶段持续时间的刚性管道。 每个任务都贡献相同的工作模式，并且系统的行为就像传送带一样。 一旦有足够的任务正在运行，管道就会达到稳定状态，完成时间线性增长。 

我们不是跟踪各个阶段，而是关注将新任务“注入”到管道中需要多长时间以及完成的任务退出的速度。 第一个任务支付所有四个阶段的全部成本。 此后，就会出现重叠：当一项任务正在测试或审查时，另一项任务可能已经在编码或修复中。 有效吞吐量由主动阶段 (A) 和被动阶段 (B) 之间的瓶颈决定，因为它们形成交替依赖关系。 

核心简化在于，系统的行为就像一个线性链，其中每个任务在初始预热后有效地添加固定增量。 事实证明，该增量由 A 和 B 之间的最大重叠约束控制，这会在管道填充后产生恒定的每个任务贡献。

 | ---| ---| ---| ---|
 | 暴力模拟| O(N) | O(1) | O(1) | 太慢了|

 ## 算法演练

1. 观察每个任务都有固定的内部结构，由四个连续阶段组成，持续时间为 A、B、B、A。顺序不能更改，但只要满足依赖关系，不同任务的阶段可以重叠。 
2. 单独考虑第一个任务。 它必须按顺序完成所有四个阶段，因此总共需要 2A + 2B 时间。 这确定了初始“管道填充”成本。 
3. 第一个任务进入测试后，只要第一个任务释放编码阶段，第二个任务就可以开始编码。 这会在任务之间造成重叠，这意味着后面的任务不会独立产生全部 2A + 2B 成本。 
4. 跟踪阶段之间的依赖链。 第二个A阶段（修复）要等到测试完成后才能开始，审核取决于修复。 这意味着 A 阶段形成了关键的串行依赖关系，而 B 阶段可以在任务之间部分重叠。 
5. 有效的限制因素是新任务能够以多快的速度完全“推进”通过两个 A 阶段，同时尊重 B 阶段强制执行的间隔。 这简化为比较主动工作 (A) 还是被动工作 (B) 主导连续任务完成之间的间隔。 
6. 稳态行为在第一个任务之后为每个任务产生恒定的吞吐量，该吞吐量由 max(A, B) 确定。 每个额外的任务都会有效地将总时间增加 2 * max(A, B)，因为主动或被动阶段都会形成阻碍更紧密打包的瓶颈。 
7. 将预热成本与稳态增长相结合：总时间为初始 2A + 2B 加上 (N − 1) 乘以每个任务增量 2 * max(A, B)。 

### 为什么它有效

 该系统是一个具有交替依赖性的两阶段管道。 A 阶段形成了一系列强制性的依赖于用户的工作，而 B 阶段则充当可以重叠但仍强制间隔的延迟。 一旦管道已满，每个新任务都必须等待两个资源限制中较慢的一个来释放容量。 这会在完成时间中产生固定的循环，确保具有恒定斜率的线性增长。 由于所有任务都是相同的，因此没有重新排序或局部优化会改变该斜率，因此表达式是精确的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    N = int(input().strip())
    A = int(input().strip())
    B = int(input().strip())

    if N == 1:
        print(2 * A + 2 * B)
        return

    print(2 * A + 2 * B + (N - 1) * 2 * max(A, B))

if __name__ == "__main__":
    solve()
```该解决方案直接应用导出的封闭形式。 第一个任务贡献了所有四个阶段的全部流水线成本，即 2A + 2B。 每个附加任务都会贡献一个由 A 和 B 之间的瓶颈确定的恒定增量，由 max(A, B) 捕获，乘以 2，因为管道的两半都必须进展才能完全完成任务。 

特殊情况 N = 1 避免乘以 (N − 1)，否则即使只有一个任务，也会错误地假设存在重叠。 

## 工作示例

 ### 示例 1

 令 N = 2、A = 3、B = 5。 

我们使用以下公式计算：

 | 步骤| 价值|
 | ---| ---|
 | 第一个任务| 2A + 2B = 6 + 10 = 16 |
 | 每个任务的增量| 2 * 最大值(A, B) = 10 |
 | 总计 | 16 + 1 * 10 = 26 |

 这表明当B占主导地位时，管道间距由测试和审查阶段控制。 

### 示例 2

 令 N = 3、A = 7、B = 4。 

| 步骤| 价值|
 | ---| ---|
 | 第一个任务| 2A + 2B = 14 + 8 = 22 |
 | 每个任务的增量| 2 * 最大值(A, B) = 14 |
 | 总计 | 22 + 2 * 14 = 50 |

 这里 A 占主导地位，因此编码和修复阶段决定了吞吐量。 

这些痕迹证实，在第一个任务之后，每个附加任务都会贡献一个恒定量，该量仅由 A 和 B 中的较大者决定。

 ## 边缘情况

 当 N = 1 时，管道永远不会达到稳定状态。 该算法正确返回 2A + 2B，因为不可能存在重叠。 

对于 A = B，主动阶段和被动阶段都具有同等约束力。 该公式完全简化为 2A + 2A + (N − 1) * 2A = 2AN，这意味着管道的行为就像一个完全顺序的过程。 这证实了对称性下的一致性。 

对于极端不平衡，例如A≫B，增量变为2A，这意味着编码和修复主导吞吐量，而测试和审查完全隐藏。 计算的调度仍然正确，因为 B 阶段永远不会成为瓶颈。 

对于 B ≫ A，情况相反：测试和审查占主导地位，A 阶段完全隐藏在重叠中。 该公式仍然会产生正确的线性增长，因为它始终通过 max(A, B) 选择真正的瓶颈。
