 ## 解决方案
 ## 问题理解

 我们有一棵根植于 1 号大厅的加权树，杰拉德就是从那里开始的。 宝藏均匀随机地隐藏在其他大厅之一中。 杰拉德只有在第一次进入正确的大厅时才发现宝藏。 

彻底改变问题的限制是，每条走廊在穿越两次后就会塌陷。 由于该图是一棵树，这意味着杰拉德必须计划一个遍历，该遍历在某些走廊变得不可用之前仍然可以精确地到达每个节点一次。 实际上，每条边可以使用一次，也可以使用两次，作为进入子树并随后返回的一部分。 

对于每个节点`v ≠ 1`， 定义`t[v]`当杰拉德第一次进入时`v`。 目标是最小化所有这些首次到达时间的平均值。 

输入描述了一棵树，最多可达`10^5`顶点和加权边。 二次或三次算法是立即不可能的。 甚至`O(n log^2 n)`这里会产生不必要的开销。 由于结构是一棵树，我们应该期望一个线性或`O(n log n)`基于 DFS 和本地决策的解决方案。 

困难的部分是杰拉德没有选择到达已知目标的最短路径。 在不知道宝藏在哪里之前，他正在选择探索顺序。 不同的子树顺序会极大地改变预期的发现时间。 

一个常见的错误是认为局部先取最短边总是最优的。 考虑这棵树：```
1 --(1)-- 2
|
(100)
|
3 --(1)-- 4
```如果我们首先访问节点 2，则发现时间变为：

 | 节点| 首次访问时间 |
 | --- | --- |
 | 2 | 1 |
 | 3 | 101 | 101
 | 4 | 102 | 102

 平均值=`(1 + 101 + 102) / 3 = 68`。 

如果我们先转到节点 3：

 | 节点| 首次访问时间 |
 | --- | --- |
 | 3 | 100 | 100
 | 4 | 101 | 101
 | 2 | 203 | 203

 平均值=`134.67`。 

较小的边缘确实应该排在第一位，但真正的原因不是边缘长度本身。 它是对剩余未探索节点施加的未来延迟量。 

当两个子树的大小非常不同时，就会出现另一个容易出现的错误。 考虑：```
1 --(10)-- A
1 --(11)-- B
```假设子树`A`子树包含 100 个节点`B`仅包含 1 个节点。 仅基于边权重的贪心规则会选择`A`首先因为`10 < 11`，但这还不够推理。 正确的顺序取决于我们探索另一个子树时有多少节点被延迟。 

最后一个微妙之处是，树根本身从来就不是一个宝藏地点。 期望值除以`n - 1`， 不是`n`。 忘记这一点会稍微改变每个答案，并且经常会意外地通过小型手动测试。 

## 方法

 暴力方法将枚举所有有效的遍历顺序。 在每个节点，我们可以选择探索子节点的顺序，并且这些选择在树上递归地组合。 

对于有度数的节点`d`， 有`d!`可能的本地订单。 纵观整棵树，全局遍历的次数变得巨大。 即使对于具有许多分支点的中等树，这也变得完全不可行。 

尽管如此，暴力破解的想法还是揭示了一些有用的东西。 唯一有意义的决定是兄弟子树的相对顺序。 一旦我们进入子树，我们最终必须在永久失去对其边缘的访问之前完成对它的探索。 因此，问题归结为决定以什么顺序处理子子树。 

假设我们站在节点`u`并比较两个子子树`A`和`B`。 

让：

-`sz[A]`是子树内宝藏候选节点的数量`A`-`cost[A]`是完全探索子树时花费的总遍历时间`A`并返回到`u`如果我们处理`A`前`B`，然后里面的每个节点`B`被延迟`cost[A]`。 

同样，处理`B`首先延迟里面的每个节点`A`经过`cost[B]`。 

所以：

-`A`前`B`增加额外的延迟`cost[A] * sz[B]`-`B`前`A`增加额外的延迟`cost[B] * sz[A]`我们应该放置`A`前`B`确切的时间：$$cost[A] \cdot sz[B] < cost[B] \cdot sz[A]$$这是关键的见解。 这个问题变成了一个调度问题，与作业排序中的史密斯规则非常相似。 

现在我们只需要：

 1. 子树大小。 
2. 每个子树的总探索成本。 
3. 通过上面的比较，属于本地类型的儿童。 

由于在完整探索过程中每条边都会被遍历两次，如果边`(u,v)`有重量`w`， 然后：$$cost[v] = dp[v] + 2w$$在哪里`dp[v]`是子树内的总探索成本`v`。 

对孩子进行最佳排序后，我们模拟 DFS 顺序并累积首次访问时间。 

整个解决方案变成一对 DFS 遍历，并在每个节点进行排序。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | 指数| 指数| 太慢了 |
 | 最佳 | O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 1. 在节点处建立树的根`1`。 

这为每个节点提供了一个父节点和一组子节点，使子树推理成为可能。 
2. 运行 DFS 来计算子树大小。 

让`sz[u]`表示子树内候选宝藏节点的数量`u`， 包括`u`本身如果`u ≠ 1`。 
3. 在同一 DFS 期间，计算勘探成本。 

让`dp[u]`是完全探索子树所需的总时间`u`并返回到`u`。 

为了每个孩子`v`通过边权重连接`w`：$$dp[u] += dp[v] + 2w$$因为我们必须下去，然后沿着那条边缘返回。 
4. 对于每个节点，对其子节点进行排序。 

对于两个孩子`a`和`b`， 定义：$$A = dp[a] + 2w_a$$

$$B = dp[b] + 2w_b$$我们放置`a`前`b`如果：$$A \cdot sz[b] < B \cdot sz[a]$$这最大限度地减少了未探索节点上的总等待时间。 
5. 按照排序顺序运行第二个 DFS。 

保持当前经过的时间。 

第一次进入节点时`v`，将当前时间添加到全局答案中，因为如果宝藏在那里，这正是发现时间。 
6. 完全探索子子树后，将当前时间提前其总探索成本。 

这反映了返回当前节点并在其他地方继续探索。 
7. 将累计总数除以`n - 1`。 

宝藏均匀分布在所有非根节点中。 

### 为什么它有效

 每个子树的行为就像一个具有两个参数的作业：

 - 完全处理需要多长时间，
 - 在结束之前有多少宝藏地点仍被封锁。 

如果子树`A`在子树之前探索`B`，那么所有节点`B`等待整个探索时间`A`。 

成对交换论证证明了最优性。 假设我们只比较两个连续的子树。 订购`A`前`B`贡献：$$cost[A] \cdot sz[B]$$到节点的总等待时间`B`。 

逆转它们有助于：$$cost[B] \cdot sz[A]$$我们选择较小的那个。 由于比较是传递的，因此按此规则排序会给出全局最优顺序。 

第二个 DFS 精确地模拟了最佳遍历，因此每个节点的首次到达时间都被最小化。 

## Python 解决方案```python
import sys
from functools import cmp_to_key

input = sys.stdin.readline

sys.setrecursionlimit(1 << 25)

n = int(input())

g = [[] for _ in range(n + 1)]

for _ in range(n - 1):
    a, b, w = map(int, input().split())
    g[a].append((b, w))
    g[b].append((a, w))

sz = [0] * (n + 1)
dp = [0] * (n + 1)

children = [[] for _ in range(n + 1)]

def dfs1(u, p):
    sz[u] = 1 if u != 1 else 0

    for v, w in g[u]:
        if v == p:
            continue

        dfs1(v, u)

        sz[u] += sz[v]
        dp[u] += dp[v] + 2 * w

        children[u].append((v, w))

    def cmp(x, y):
        vx, wx = x
        vy, wy = y

        cx = dp[vx] + 2 * wx
        cy = dp[vy] + 2 * wy

        left = cx * sz[vy]
        right = cy * sz[vx]

        if left < right:
            return -1
        if left > right:
            return 1
        return 0

    children[u].sort(key=cmp_to_key(cmp))

dfs1(1, 0)

ans = 0

def dfs2(u, p, cur_time):
    global ans

    if u != 1:
        ans += cur_time

    now = cur_time

    for v, w in children[u]:
        dfs2(v, u, now + w)
        now += dp[v] + 2 * w

dfs2(1, 0, 0)

print(ans / (n - 1))
```第一个 DFS 计算两个结构量。`sz[u]`计算子树中存在多少个有效的宝藏位置`u`。 根源被排除在外，因为宝藏永远不会从那里开始。`dp[u]`测量完全探索子树所需的总遍历时间`u`并回到`u`。 每条边都会贡献两倍，因为探索需要进入和返回。 

排序步骤是解决方案的核心。 一个常见的实现错误是使用浮点除法，例如：```
cost[a] / sz[a]
```这可能会带来精度问题。 交叉乘法完全避免了这种情况。 

第二个DFS模拟实际的遍历顺序。 变量`now`代表我们回到节点的时间`u`并准备开始下一个子子树。 

还有一个微妙的点就是递归调用进入child`v`在某个时间`now + w`， 不是`now`。 边缘遍历本身在第一次到达之前会消耗时间`v`。 

所有算术都可以安全地适合 Python 整数，因为最大总遍历长度最多约为：$$2 \cdot (10^5 - 1) \cdot 1000$$无论如何，这完全在 64 位范围内。

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 | O(n log n) | O(n log n) | 每个节点的子节点都排序一次 |
 | 空间| O(n) | 邻接表、DFS数组、递归栈 |

 所有节点的子条目总数为`n - 1`。 排序在运行时占据主导地位。 在最坏的情况下，一个节点可能有度数`n - 1`，生产`O(n log n)`整体复杂性。 

这很容易符合以下限制`n ≤ 10^5`。

 ## 边缘情况

 考虑星形树：```
5
1 2 1
1 3 1
1 4 1
1 5 1
```每片叶子都需要在访问另一片叶子之前返回根。 到达第一个访问的叶子的时间`1`，第二个在`3`，第三个在`5`，第四个在`7`。 

该算法计算：

 | 叶| 首次抵达 |
 | --- | --- |
 | 第一| 1 |
 | 第二 | 3 |
 | 第三| 5 |
 | 第四 | 7 |

 平均的：$$(1 + 3 + 5 + 7) / 4 = 4$$这证实了第二个 DFS 正确地累积了回程成本。 

现在考虑子树大小比边权重更重要的情况：```
6
1 2 1
2 3 1
3 4 1
1 5 2
1 6 2
```子树根于`2`包含三个宝藏节点，每个节点的额外成本相对较少。 该算法在单节点离开之前对其进行优先级排序`5`和`6`。 

仅基于直接边长的幼稚策略可能会错误地以同样的方式对待所有三个孩子。 

比较器可以正确处理这个问题，因为它最大限度地减少了：$$cost \times delayed\_nodes$$而不仅仅是本地遍历时间。 

最后，考虑一条深链的长度`10^5`。 不增加递归深度的递归 DFS 会因堆栈溢出而崩溃。 该实现明确提高了递归限制以安全地处理最坏情况的树。
