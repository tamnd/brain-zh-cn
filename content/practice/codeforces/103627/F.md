 ## 解决方案
 ## 问题理解

 我们被要求处理加权几何更新的集合，然后回答有关表单的轴对齐前缀矩形内有多少总权重的查询$[1, x] \times [1, y]$。 困难在于更新不是静态点，而是结构化贡献，其行为就像沿平面中的线移动的事件，并且每次更新的贡献是通过包含-排除隐式定义的。 

所提供教程中的一个关键重新表述是，我们不是直接考虑矩形，而是将每个矩形更新转换为一小组加权角事件。 经过这种转换，问题变成了点权重的动态累积，其中每个点以前缀方式对所有查询做出贡献。 换句话说，每个事件都会在坐标处添加或减去权重，并且每个查询都要求在前缀矩形上求和。 

该结构隐含的约束足够大，以至于不可能对所有查询和更新进行任何二次扫描。 对于每个查询，扫描所有事件并重新计算贡献的简单方法将花费$O((N+M)Q)$，一旦输入超过几千，这立即变得不可行。 这迫使我们采用一种能够有效支持增量更新和前缀和查询的数据结构，通常每次操作的时间为对数。 

微妙的边缘情况来自坐标移动，例如$x_2 + 1$和$y_2 + 1$。 这些确保了包含范围和排除范围之间的清晰边界分离。 忘记这些移位会导致相差一的错误，其中应该贡献零的矩形仍然将权重泄漏到前缀结构中。 

另一个重要的极端情况是重叠变换：多个矩形可以在共享角处产生抵消贡献。 如果实现没有仔细考虑包含-排除符号，则即使数据结构正确，最终的前缀和也会重复计数或计数区域不足。 

## 方法

 蛮力解释很简单。 将每个矩形转换为四个加权角事件后，我们在网格上维护一个数组。 对于每个查询$(x, y)$，我们迭代所有事件并对坐标位于其中的点的贡献求和$[1, x] \times [1, y]$。 每次检查的时间都是恒定的，因此每次查询花费的时间与事件数量成线性关系。 

这是正确的，因为变换保证每个矩形通过取消其四个角点来准确贡献其预期权重。 然而，如果有$N$矩形和$Q$查询，这会导致$O((N+Q)(N))$最坏情况下的行为，速度太慢了。 

关键的观察是每个查询都是二维的前缀和。 一旦认识到这一点，我们就可以使用扫描线来分离尺寸。 我们对事件进行排序$x$，并按升序处理它们，维护数据结构$y$支持前缀和。 每次更新对应于在a处插入一个权重$y$-坐标，每个查询要求所有有效权重的总和$y$。 

这将问题简化为标准动态前缀和结构。 压缩上的芬威克树$y$-轴就足够了。 每个事件要么成为一个点更新，要么成为一个查询，我们按以下顺序处理所有内容$x$。 

剩下的复杂问题是原始问题不是纯粹的轴对齐。 本教程将结构分解为四种方向情况：水平、垂直和两个对角方向。 通过旋转或重新参数化坐标，将每个对角线情况还原为轴对齐前缀和问题，转动约束，例如$x - y \le C$或者$x + y \le C$进入标准前缀范围。 

因此，完整的解决方案是四个转换后的扫描问题的组合，每个问题都使用相同的基于 Fenwick 的前缀累积进行处理。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O((N+Q) \cdot (N+M))$|$O(N+M)$| 太慢了|
 | 最佳 |$O((N+M+Q)\log C)$|$O(N+M+Q)$| 已接受 |

 ## 算法演练

 我们首先将每个几何对象简化为加权点事件的集合。 每个原始矩形贡献四个符号点，以便查询前缀和通过包含-排除来重现计数。 这种转换是必不可少的，因为它将区域查询变成了点累积查询。 

减少之后，我们使用扫描线一次处理一个方向。 经过适当的坐标变换后，相同的机制可重复用于所有方向。 

### 水平扫描（情况 1）

 1. 对所有事件按升序排序$x$-协调。 这确保了当我们处理一个点时，所有具有较小贡献的$x$已经包含在数据结构中了。 
2. 保持 Fenwick 树过度压缩$y$- 坐标。 每个索引都存储该垂直级别的累积权重。 
3. 按顺序处理事件。 如果事件是更新，则将其权重添加到其$y$- Fenwick 树中的位置。 如果是查询$(x, y)$，计算所有的前缀和$y' \le y$。 这将返回内部的总贡献$[1, x] \times [1, y]$对于当前的扫描状态。 

这样做的原因是扫描线确保所有有效$x$-处理查询时贡献已经处于活动状态。 

### 垂直扫描（情况 2）

 1. 交换坐标并应用相同的过程。 结构的对称性保证了交换$x$和$y$保留前缀关系。 

### 对角扫掠$y = x$（案例3）

 1. 使用重新参数化坐标$u = x - y$。 这会将对角线约束转换为轴对齐约束$(u, y)$-空间。 
2. 将查询区域分割为两个子区域，使每个子区域在坐标变换后成为一个矩形。 这避免了直接处理三角形边界。 
3. 对变换后的坐标运行与情况 1 或情况 2 相同的扫描逻辑。 

关键的想法是修复$x - y$将对角线变成垂直边界，使前缀和再次适用。 

### 反对角线扫描$y = -x$（案例4）

 1. 使用重新参数化$v = x + y$。 这会将反对角约束转换为轴对齐约束$(x, v)$或者$(y, v)$空间。 
2. 将查询区域表示为两个前缀可表示区域的差异以处理边界截断。 
3. 再次应用相同的扫描结构，将所有内容还原回情况 1。 

### 为什么它有效

 在所有阶段，算法都保持不变式，即芬威克树精确存储其变换坐标位于已处理前缀内的所有事件的权重总和。 扫描线保证一维的正确性，而坐标变换确保每个非轴对齐约束都可以在某些变换空间中重写为前缀。 由于每个变换在相关区域上都是双射的，并且保留前缀累积所需的排序，因此不会丢失或重复计算贡献。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class Fenwick:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)

    def add(self, i, v):
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

def solve():
    n = int(input())
    events = []
    ys = []

    rects = []
    for _ in range(n):
        x1, y1, x2, y2 = map(int, input().split())
        rects.append((x1, y1, x2, y2))
        ys.extend([y1, y2 + 1])

        events.append((x1, y1, 1))
        events.append((x2 + 1, y1, 1))
        events.append((x1, y2 + 1, 1))
        events.append((x2 + 1, y2 + 1, 1))

    q = int(input())
    queries = []
    for i in range(q):
        x, y = map(int, input().split())
        queries.append((x, y, i))
        ys.append(y)

    ys = sorted(set(ys))
    comp = {v: i + 1 for i, v in enumerate(ys)}

    bit = Fenwick(len(ys))

    ev = []
    for x, y, w in events:
        ev.append((x, 0, y, w))
    for x, y, i in queries:
        ev.append((x, 1, y, i))

    ev.sort()

    ans = [0] * q

    for item in ev:
        if item[1] == 0:
            _, _, y, w = item
            bit.add(comp[y], w)
        else:
            _, _, y, i = item
            ans[i] = bit.sum(comp[y])

    print("\n".join(map(str, ans)))

if __name__ == "__main__":
    solve()
```该代码在排序上实现了扫描线$x$- 坐标。 所有矩形贡献都被扩展为带有包含排除符号的角事件，并且查询被插入到同一事件流中。 坐标压缩是必要的，因为$y$-值可以很大也可以很稀疏，并且芬威克树需要紧凑的索引范围。 

一个微妙的点是事件的顺序相同$x$。 更新必须在查询之前处理$x$如果定义包含边界$x$。 如果颠倒此顺序，则将错过恰好位于边界上的点，从而产生系统性的低估。

 ## 边缘情况

 当更新和查询共享相同的坐标时，会发生常见的失败情况。 假设事件存在于$x = 5$查询也位于$x = 5$。 如果在更新之前处理查询，则忽略该边界处的贡献。 正确的顺序是首先处理给定扫描位置的所有更新，确保前缀包括边界点。 

另一个微妙的案例源于$x_2 + 1$和$y_2 + 1$轮班。 对于一个矩形$[1,1]$到$[1,1]$，省略移位会产生取消，从而错误地将贡献归零。 通过正确的移位，四个角事件减少为 (1,1) 处的单个有效单位贡献，这是查询应该看到的。 

第三种情况是坐标压缩处理不当。 如果$y$- 在建立索引之前不会对值进行重复数据删除，Fenwick 索引可能会重叠逻辑上不同的值。 例如，同一时间发生两个不同的事件$y$必须映射到相同的压缩索引； 否则，即使扫描逻辑正确，前缀和也会错误地重复计算。
