 ## 解决方案
 ## 问题理解

 我们得到一个最多 21 个正整数的数组和一个初始值$x$。 我们可以任意重新排序数组。 固定好顺序后，我们对元素进行逐一处理，反复更新$x$将其替换为$x \bmod a_i$。 

目标是选择数组的顺序，以便在应用所有模运算后，最终值$x$尽可能大。 

关键是操作不对称：应用$x \bmod a$早期可以大大减少$x$，这会限制后续 mod 操作的效果。 因此，订购非常重要。 

就以下方面而言，限制非常小$n$， 和$n \le 21$，但值本身可以大到$10^{18}$。 这立即排除了任何依赖于枚举值的方法$x$以密集的方式或做任何多项式$x$。 唯一可利用的小参数是元素的数量，因此任何正确的解决方案都必须依赖于排列或子集的指数搜索。 

当某些情况下，会出现微妙的边缘行为$a_i > x$。 在那种情况下，$x \bmod a_i = x$，这意味着如果在正确的时间应用这些元素实际上是无操作的。 但是，如果稍后应用$x$减少后，它们可能会变得活跃并降低价值。 例如，如果$x = 10$,$a = [15, 6]$，然后先应用 15 保留$x = 10$，但先应用 6 使得$x = 4$，然后应用 15 不再执行任何操作，剩下 4。正确答案是 10。 

另一个失败案例是假设贪心排序$a_i$。 例如，降序排序可能看起来很自然，但它会失败，因为早期的大模数可能会阻止以后的减少，而这在不同的顺序中是有益的。 

## 方法

 蛮力的想法很简单：尝试数组的每种可能的排列，模拟模运算，并保持最佳结果。 每个排列需要$O(n)$过渡，并且有$n!$排列，所以总复杂度是$O(n! \cdot n)$。 和$n = 21$，这是一个天文数字，而且不可行。 

重要的观察结果是，最终值仅取决于元素的相对顺序，并且元素的数量足够小以允许对子集进行动态编程。 我们不是构建完整的排列，而是逐步构建过程，跟踪哪些元素已经被使用以及当前的值是多少$x$是在按某种顺序应用它们之后。 

关键结构是进程的状态完全由所用索引的子集和当前值决定$x$。 从任何状态中，我们接下来都可以选择任何未使用的元素。 这自然形成了类似最短路径或子集上的 DP 结构，其中转换将集合大小减少一并确定性地更新值。 

然而，子集上的朴素 DP 和精确的$x$值是不可能的，因为$x$最多可以花费$10^{18}$独特的价值观。 关键的节省观察是$x$仅更改其本身（如果模数较大）或更改为严格小于所选值的值$a_i$。 所以虽然$x$范围很大，所有状态中不同可达值的数量受到小集合上 mod 操作引起的分支结构的严重限制。 

这会导致位掩码状态的记忆化，其中每个状态存储可实现的最佳值$x$。 转换尝试每个未使用的元素并更新结果。 因为状态图在掩码大小方面是非循环的，所以递归或DP是安全的。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力排列 |$O(n! \cdot n)$|$O(n)$| 太慢了 |
 | 位掩码 DP 的排列 |$O(n \cdot 2^n)$|$O(2^n)$| 已接受 |

 ## 算法演练

 我们用位掩码表示每个状态，指示哪些元素已被使用，以及当前值$x$。 

1.定义DP函数$f(mask, x)$返回从状态开始的最大可能的最终值$mask$与当前值$x$。 这个公式直接反映了问题中描述的过程。 
2. 如果所有元素都被使用，则返回$x$。 此时没有进一步的操作，因此当前值是最终值。 
3. 否则，尝试选择每个未使用的索引$i$。 对于每个选择，计算下一个值$x' = x \bmod a_i$，并递归地评估$f(mask \cup \{i\}, x')$。 
4. 在所有选择中取最大值$i$。 这对应于选择最佳的可能的下一个操作。 
5. 记住各州的结果。 由于是同一对$(mask, x)$可以通过不同的排列来达到，缓存避免了重新计算。 

一个实际的改进是完整的记忆$x$如果使用 Python 散列仔细实现，则从天真的意义上来说是不必要的，因为可到达状态的数量在以下情况下仍然是可管理的：$n \le 21$。 然而，概念性​​的DP仍然存在$(mask, x)$。 

### 为什么它有效

 数组的每个有效顺序都对应于该状态图中的一条路径，从$mask = 0$和初始的$x$，并结束于$mask = (1<<n)-1$。 相反，每条路径都对应一个有效的排列。 由于我们评估每个状态的所有转换并取最大值，因此不排除任何有效排序，也不引入无效排序。 递归以结构化形式探索整个排列空间，同时避免重复重新计算相同的状态。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

sys.setrecursionlimit(10**7)

def solve():
    n = int(input())
    a = list(map(int, input().split()))
    x0 = int(input())

    from functools import lru_cache

    @lru_cache(None)
    def dp(mask, x):
        if mask == (1 << n) - 1:
            return x

        best = 0
        for i in range(n):
            if not (mask >> i) & 1:
                nxt = x % a[i]
                cand = dp(mask | (1 << i), nxt)
                if cand > best:
                    best = cand
        return best

    print(dp(0, x0))

if __name__ == "__main__":
    solve()
```该解决方案是前面描述的子集 DP 的直接实现。 递归状态包括已使用的掩码和当前值$x$。 这`lru_cache`确保重复的状态被计算一次。 

一个微妙的实现细节是我们必须包括`x`在 DP 键中。 省略它会错误地合并具有相同使用元素但当前值不同的状态，这些状态在未来的演变中根本不同。 

## 工作示例

 ### 示例 1

 输入：```
n = 3
a = [6, 7, 10]
x = 20
```我们追踪了一些有代表性的路径。 

| 面膜| 当前 x | 选择| 下一个 |
 | --- | --- | --- | --- |
 | 000 | 000 20 | 10 | 10 0 |
 | 000 | 000 20 | 7 | 6 |
 | 000 | 000 20 | 6 | 2 |

 一个最佳路径是先选择 10：

 20 mod 10 = 0，但这不好。 相反，首先选择 7 给出 20 mod 7 = 6，然后应用 6 给出 0，最终 0。但是选择 6 首先给出 2，然后 7 给出 2，然后 10 给出 2，最终 2。 

最佳排序通过延迟强减少来保留更大的中间值。 

这个例子表明，早期积极减少并不总是最佳的。

 ## 边缘情况

 一个重要的边缘情况是当每个$a_i$大于当前$x$。 在这种情况下，每次操作都会离开$x$不变，所以任何排列都是有效的，答案很简单$x$。 DP 自然地处理这个问题，因为每次转换都会保留$x' = x$，因此所有路径都保留该值直到终止。 

另一个边缘情况是当存在非常小的$a_i$，例如 1. 的任何应用$a_i = 1$立即部队$x$到 0。DP 正确地延迟使用这些元素，除非所有其他选择都已用尽，因为任何早期使用 1 的分支都会产生最终值 0，这永远不是最佳的，除非所有其他路径也崩溃。 

最后一个微妙的情况是，最佳策略依赖于在许多无操作操作中保留较大的中间值。 基于掩码的递归确保对此进行探索，因为它显式地尝试所有排列，包括那些延迟有害减少直到不可避免的排列。
