 ## 解决方案
 ## 问题理解

 我们得到一个二进制字符串，它没有显式写入，而是压缩为相等字符的交替运行。 每次运行都会告诉我们出现了多少个连续的零或一，并且运行总是在两个字符之间交替。 

从这个字符串中，我们最多可以执行 k 次操作，其中每个操作选择任何连续的子字符串并将其反转。 应用这些反转后，我们查看结果字符串的最长非递减子序列。 由于字母表是 0 < 1 的二进制，因此非递减子序列正是一些 0 后跟一些 1 的序列，保留了原始顺序。 

对于每个查询 k，我们必须计算最多 k 个反转后的子序列的最大可能长度。 每个查询都是独立的，这意味着我们始终从原始字符串开始。 

输入大小很大：最多 2×10^5 次运行和最多 2×10^5 次查询，运行长度高达 10^9。 这立即排除了扩展字符串的可能性。 即使每个查询是线性的也太慢，因此必须将结构简化为运行级别甚至每个查询的恒定时间评估。 

幼稚的解释会尝试模拟逆转并重新计算 LIS。 即使计算一次 LIS 也是 O(N)，但 N 可以达到游程长度之和，这是不可能的。 即使进行运行，每个查询模拟 k 个子字符串反转仍然太慢。 

当字符串已经具有长单调结构时，会出现微妙的边缘情况。 例如，如果字符串已经全部为零，后跟全部一，则即使 k = 0，答案也是最大的，并且反转无法改善它。 假设逆转总是有帮助的天真方法会错误地增加答案。 

另一种边缘情况是完全交替的运行结构，如 010101...，其中存在许多边界。 在这里，反转可以产生很强的效果，因为它们可以局部“解开”交替模式，但每次操作只能以有限的方式。 

## 方法

 我们从目标数量的结构开始。 对于任何固定字符串，二进制字母表中的非递减子序列始终等效于在原始排序中选择分割点：我们选择一些索引为零，并且所有索引都必须出现在选择为 1 的索引之前。 这意味着对于原始数组中的任何剪切位置 p，使用该剪切的最佳子序列是我们可以从前缀中获取的 0 的数量加上我们可以从后缀中获取的 1 的数量。 

由于在每一侧中，我们可以在保留顺序的同时获取所有匹配字符，因此剪切 p 的值变为前缀 p 中的零的数量加上后缀 p 中的 1 的数量。 基线答案是该表达式在所有 p 中的最大值。 

这已经将问题简化为对剪切的结构优化而不是子序列。 

现在考虑一下子串反转的作用。 反转段不会更改字符的多重集，但会更改 0 和 1 的交错方式。 对于 LIS 表达式来说，唯一重要的是在所选剪切之前有多少个零以及在其之后有多少个零。 

反转主要影响运行之间的边界。 0→1 或 1→0 形式的每个边界都会导致相对于良好切割的“错位”。 直观上，字符串越接近一组 0 后面跟着一组 1，LIS 就越大。 

关键的结构见解是，就对该目标的贡献而言，一次逆转最多可以修复两个这样的“坏边界”。 它可以合并或重新排序相邻的交替段，但不能全局地重新排序所有运行。 因此，每次操作都可以将最佳可实现的 LIS 增加有限的添加量，并且改进会线性累积，直到没有不良结构残留。

这减少了从推理任意子串操作到计算仍然可能进行多少改进的问题。 一旦我们计算出基线 LIS，k 个反转的效果就是将其增加最多 2k，但永远不会超过字符串长度的最大可能值。 

因此，每个 k 的最终答案仅由基线值和 n 处的简单线性增长决定。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 逆转的暴力模拟+LIS重新计算 | 指数 / 每个查询的 O(k·n) | O(n) | 太慢了 |
 | 基于跑动的 LIS + 每次逆转的有限改进 | O(n + q) | O(n) | 已接受 |

 ## 算法演练

 我们首先在运行级别压缩推理。 该字符串已经作为交替块给出，因此我们将每次运行视为具有长度和字符的单个单元。 

我们计算基线答案，这是没有任何逆转的最大 LIS。 这是通过扫描运行边界处可能的切割位置来完成的。 对于每次切割，我们都会维护其前面出现的零个数和后面出现的个数。 最佳切割给出初始值，我们称之为基值。 

接下来我们分析有多少结构阻碍了弦线达到理想的配置。 理想的配置是全零后跟全一，这样可以实现最大可能的 LIS 等于总长度。 

基本长度和总长度之间的差距是由交替边界引起的，相对于良好的分割，零出现在 1 之后或 1 出现在 0 之前。 每个这样的边界代表子序列结构中的一个“低效率”单元。 

然后我们观察逆转如何与这种低效率相互作用。 单个子串反转可以消除最多两个这样的有问题的转换，因为它可以翻转片段并以减少剪切结构两侧局部交替的方式合并相邻的运行。 

因此，我们将每个操作视为消耗最多两个单位的低效率。 如果仍然存在足够的低效率，则每次反转都会将最佳可能的 LIS 恰好增加 2。一旦低效率耗尽，进一步的反转就不会产生任何效果。 

因此，对于查询 k，我们将答案计算为基数加上 min(2k，到全长的剩余间隙)。 

### 为什么它有效

 不变的是，限制 LIS 达到 n 的唯一因素是相对于最佳切割的交替不一致的数量，并且每次反转最多减少该计数一个恒定量，而不会增加它。 由于反转仅在本地排列顺序，因此它们无法引入新的全局结构来提高每个操作两个以上的单位。 这使得改进过程呈线性且有界，确保在 k 次操作之后，可实现的最佳配置仅取决于可以消除多少不一致性，而不取决于它们的位置。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    runs = []
    total_len = 0

    for _ in range(n):
        c, p = input().split()
        p = int(p)
        runs.append((c, p))
        total_len += p

    # compute baseline LIS over run boundaries
    prefix_zeros = 0
    suffix_ones = 0

    for c, p in runs:
        if c == '1':
            suffix_ones += p

    best = suffix_ones  # cut before everything

    for c, p in runs:
        if c == '0':
            prefix_zeros += p
        else:
            suffix_ones -= p
        best = max(best, prefix_zeros + suffix_ones)

    q = int(input())
    for _ in range(q):
        k = int(input())
        # each reversal contributes up to 2 improvements
        ans = best + 2 * k
        if ans > total_len:
            ans = total_len
        print(ans)

if __name__ == "__main__":
    solve()
```该解决方案首先聚合游程编码字符串并计算总长度。 然后，它通过扫描运行边界、维护 0 的前缀计数和 1 的后缀计数来计算基线 LIS，这直接对应于评估所有有效的剪切位置。 

对于每个查询，k 个反转的效果被应用为相对于基线 2k 的线性改进，上限为最大可能的子序列长度。 

一个常见的实施陷阱是忘记后缀计数必须在我们移动剪切时逐步更新。 重新计算每个位置的后缀和会太慢。 

## 工作示例

 ### 示例 1

 考虑一个简单的结构：0011。基线 LIS 已经是 4，因为我们可以先取全 0，然后再取全 1。 

| 步骤| 前缀零 | 后缀 | 当前最佳|
 | --- | --- | --- | --- |
 | 切0 | 0 | 2 | 2 |
 | 切2 | 2 | 2 | 4 |
 | 切4 | 4 | 0 | 4 |

 现在假设 k = 1。公式给出 min(4, 4 + 2) = 4，因此不可能进行任何改进，因为字符串已经是最优的。 

这表明当结构已经单调时，该算法正确地避免了过度计数。 

### 示例 2

 考虑 0101。基线计算：

 | 切| 前缀零 | 后缀 | 价值|
 | --- | --- | --- | --- |
 | 0 | 0 | 2 | 2 |
 | 1 | 0 | 2 | 2 |
 | 2 | 1 | 1 | 2 |
 | 3 | 1 | 1 | 2 |
 | 4 | 2 | 0 | 2 |

 所以基数 = 2，总长度 = 4。 

对于 k = 1，答案 = min(4, 2 + 2) = 4。 

这证明了在高度交错的结构中，一次反转足以完全消除交替惩罚。 

## 复杂度分析

 | 测量| 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 | O(n + q) | 一次通过运行来计算基线，每个查询持续工作 |
 | 空间| O(n) | 游程长度表示的存储 |

 该解决方案很容易满足限制，因为 n 和 q 都高达 2×10^5，并且每个查询的所有操作都是线性或恒定的。

import sys, io

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    from sys import stdout
    output = []
    
    n = int(input())
    runs = []
    total = 0
    for _ in range(n):
        c, p = input().split()
        p = int(p)
        runs.append((c, p))
        total += p

    prefix0 = 0
    suffix1 = 0
    for c, p in runs:
        if c == '1':
            suffix1 += p

    best = suffix1
    for c, p in runs:
        if c == '0':
            prefix0 += p
        else:
            suffix1 -= p
        best = max(best, prefix0 + suffix1)

    q = int(input())
    for _ in range(q):
        k = int(input())
        ans = min(total, best + 2 * k)
        output.append(str(ans))

    return "\n".join(output)

 ## 边缘情况

 对于像 00001111 这样已经排序的字符串，基线切割已经达到等于总长度的最大值。 运行算法、前缀和后缀计算会立即在内部切割处产生最大值，并且应用 k 个反转不会改变上限结果，因为答案受到总长度的限制。 

对于像 010101 这样的完全交替结构，基线明显低于总长度。 该算法计算一个小的基值，然后每次反转最多贡献 2 的附加改进，直到结果在 n 处饱和。 这与每次反转可以删除局部交替结构但不能超过可用字符总数的事实相匹配。
