 ## 解决方案
 ## 问题理解

 我们得到一个简短的程序，由存储在类似数组的结构中的两种类型的指令组成。 执行时不会直接按顺序运行这个列表一次； 相反，它构建第二个结构，即队列，并动态地执行其中的指令。 

最初，所有原始指令都按顺序复制到队列中。 然后通过重复弹出该队列的前端指令来继续执行。 每条指令要么立即产生输出，要么将更多指令附加到同一队列的后面。 普通指令打印单个字符。 复制指令不会打印任何内容； 相反，它采用原始指令列表的第一部分并将这些指令的新副本附加到执行队列中。 

任务是在执行队列中的 k 条指令后确定结果输出字符串，如果队列较早耗尽，则确定更少的指令。 

关键的困难在于队列的大小可能呈指数级增长，因为复制指令可以重复地重新引入较早的指令。 因此，简单的模拟可能会远远超过 k 步。 

约束 n 和 k 都达到 100000。这立即排除了任何显式具体化队列或逐字复制指令块的模拟。 在最坏的情况下，即使每次复制操作的线性扩展也会变成二次方。 

当复制操作重复复制本身包含进一步复制操作的前缀时，会出现微妙的边缘情况。 例如，如果每条指令都是整个前缀的副本，则队列呈指数增长，而所需的输出仍然很小。 简单的队列模拟会尝试实现不可行数量的指令。 

另一种边缘情况是 k 小于 n 时。 在这种情况下，我们甚至永远不会完成初始前缀，因此任何假设对所有复制扩展进行完全预处理的优化仍然必须提前正确停止。 

## 方法

 直接模拟维护指令队列并逐步处理它。 每次弹出一条指令时，我们要么将一个字符附加到输出中，要么通过复制原始列表中的前 m 条指令来扩展队列。 这是正确的，因为它准确地反映了该过程。 

然而，故障模式是立即发生的。 每个复制操作最多可以附加 O(n) 条指令。 由于最多有 k 个操作，并且每个新附加的指令本身可能会触发更多副本，因此排队指令的总数可能会远远超出 k 中的任何多项式界限。 即使达到 k 步骤也可能需要处理指数级大的隐式结构。 

关键的观察是我们实际上从来不需要完整的扩展队列。 我们只需要知道前k条执行的指令。 每条指令要么是产生字符的叶子，要么是扩展为原始程序的前缀的内部节点。 这自然形成一个有根扩展树，其中每个节点对应于一个指令出现，并且 cp m 创建到前 m 个原始节点的边。 

我们通过跟踪每条原始指令在最多 k 个步骤的执行中被“访问”的次数来避免扩展。 我们不是显式扩展子级，而是向前传播计数：每次处理 cp m 指令时，它都会增加前 m 个原始指令的使用计数。 这将问题简化为计算前 k 个步骤中每条指令执行的次数，然后根据这些计数加权的回显指令生成输出。 

这将指数过程转换为固定数组上的线性传播。

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力队列模拟| O(指数) | O(n + k 扩展) | 太慢了 |
 | 前缀图上的计数传播 | O(n + k) | O(n) | 已接受 |

 ## 算法演练

 我们通过跟踪每条指令的执行次数来模拟执行，而无需显式扩展队列。 

1. 初始化一个大小为 n 的数组 cnt，对于所有 i，cnt[i] = 1。 这表示每个原始指令在初始队列副本中至少执行一次。 
2. 维持指令按原始索引顺序执行的指针顺序。 我们以递增的索引顺序处理指令，就好像它们按顺序出列一样。 
3. 保持变量processed = 0，表示到目前为止总共消耗了多少条指令执行。 
4. 对于从 1 到 n 的每条指令 i，当处理 < k 时，我们以聚合形式而不是单独模拟执行指令 i cnt[i] 次。 
5. 如果指令 i 是 echo c，我们将重复 cnt[i] 次的 c 添加到答案中，但最多只能达到已处理的剩余配额 k。 
6. 如果指令 i 是 cp m，则通过相应地递增 cnt[j]，将 cnt[i] 附加执行分配给从 1 到 m 的所有指令。 这表示每次执行 cp m 都会将前缀重新插入到执行流中。 
7. 一旦处理达到k，则停止。 

关键思想是，我们不是将 cp 操作扩展为显式队列元素，而是将每个指令执行视为可以重新分配的“流”单元。 

### 为什么它有效

 在任何时候，cnt[i] 表示在到达位置 k 之前指令 i 将出现在概念执行队列中的次数。 初始队列贡献每条指令的一次执行。 每条 cp m 指令执行一次后，都会将前缀 [1..m] 的一份额外副本附加到队列中，因此 cp m 的每次出现都会使将来对这些前缀指令的访问量乘以一个额外单位。 由于我们只关心前 k 次执行，因此聚合这些贡献可以保留精确的多重性，而无需构建扩展序列。 该过程相当于展开执行树但在深度 k 处截断它。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

n, k = map(int, input().split())
ops = []

for _ in range(n):
    parts = input().split()
    if parts[0] == "echo":
        ops.append(("echo", parts[1]))
    else:
        ops.append(("cp", int(parts[1])))

cnt = [0] * n
cnt[0] = 1

res = []
processed = 0

for i in range(n):
    if processed >= k:
        break

    if cnt[i] == 0:
        continue

    take = min(cnt[i], k - processed)

    if ops[i][0] == "echo":
        res.append(ops[i][1] * take)
        processed += take

    else:
        processed += take
        m = ops[i][1]
        if m > 0:
            add = cnt[i]
            for j in range(m):
                cnt[j] += add

print("".join(res))
```该实现保持了每条指令都带有一个重数cnt[i]的思想，它表示它对执行前缀的贡献次数。 我们仅消耗最多 k 个总执行，通过已处理进行跟踪。 Echo 指令直接影响输出字符串，乘以它们的执行计数。 复制指令将其执行权重传播到较早的指令。 

一个微妙的点是传播完全使用 cnt[i]，而不是截断的 take 值。 这是因为，即使我们只需要前 k 个步骤的部分执行，所有这些在概念上仍然会生成未来的排队指令，即使我们没有完全消耗它们的输出。 

停止条件确保我们永远不会产生超出所需的输出。

 | 测量 | 复杂性 | 说明|

 | 时间 | O(n + k) | 每条指令被处理一次，并且传播触及以 n | 为界的前缀范围。 
| 空间| O(n) | 我们存储指令列表和执行计数|

        else:
            processed += take

 ## 边缘情况

 一个关键的边缘情况是 cp 重复定位完整前缀。 例如，如果每条指令都是 cp i 或 echo Early，则执行计数会爆炸。 该算法处理这个问题是因为它永远不会实现扩展； 它仅在固定数组中累积计数。 

另一种边缘情况是 k 小于初始指令的数量。 使用processed ≥ k 循环提前停止，因此即使cnt 值仍然很大，后面的指令也不会被触及。 

最后一种边缘情况是 cp 指令出现在所有 echo 指令之后。 在这种情况下，传播会增加计数，但不会产生额外的输出，因为没有回声剩余来消耗它。 该算法仍然正确执行更新，但不会产生额外的字符，这与在 k 步内无法到达进一步的回显指令的事实相匹配。
