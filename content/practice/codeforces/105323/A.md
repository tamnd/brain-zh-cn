 ## 解决方案
 ## 问题理解

 我们得到一棵有根树，其中每个节点最多有两个子节点。 根是节点 1，每个节点都有一个关联的权重。 最初只有根是红色的，所有其他节点都是白色的。 

该过程以离散的轮次进行。 在每一轮中，在发生任何其他事情之前，您都可以从树中删除一条边。 删除之后，颜色会扩散：直接连接到已经是红色的节点的每个节点都会变成红色。 这意味着红色总是从节点 1 开始沿着剩余的边生长。 

重复该过程，直到没有新节点可以被染成红色。 最后，每个节点要么是红色的，要么永远保持白色。 得分定义为白色节点的总权重减去红色节点的总权重。 由于所有权重的总和是固定的，因此最大化分数相当于最小化变为红色的节点的总权重。 

玩家拥有的关键控制能力是能够在扩散发生之前每轮删除一条边。 每次删除都可能阻止整个子树从根部访问，但前提是在感染到达树的该部分之前足够早地删除它。 

约束 n ≤ 10000 表明我们大约需要 O(n log n) 或 O(n) 个解决方案。 任何天真地逐轮模拟过程的方法（可能需要 O(n^2)）都会太慢，因为每一轮都可能涉及许多边缘和传播步骤。 

当子树很早变得可达时，就会出现微妙的边缘情况。 如果我们未能在那一刻之前切断连接边，则整个子树将永久丢失。 例如，如果一个深层节点只是在较晚的时候被阻止，那么它可能之前就已经被感染了，即使最终执行了切割，也使得切割毫无用处。 

另一种边缘情况是存在多个高权重子树，但每轮只能切割一条边。 像总是切割最大直接子树这样的贪婪局部决策不一定是最优的，因为时序约束与子树大小一样重要。 

## 方法

 如果我们直接模拟这个过程，每一轮我们都会跟踪当前的红色边界，然后决定要切割哪条边。 之后，我们扩大红色区域。 在最坏的情况下，这可能需要 O(n) 轮，并且每轮可能需要扫描许多边缘来模拟扩散和评估切割。 这会导致 O(n^2) 行为，对于 n 高达 10000 来说太慢了。 

关键的观察结果是，如果没有删除任何边，则传播过程是确定性的：深度 d 处的每个节点在时间 d 处变为红色。 防止节点变红的唯一方法是在时间 d 之前切断连接其父节点的边。 这将问题转化为在时序约束下选择一组要切割的边缘。 

从父项 u 到子项 v 的每条边都可以被视为一项“工作”，其截止日期等于到达 v 的时间，即其深度。 切断该边可以防止以 v 为根的整个子树变红。 切割它的利润是该子树的总重量。 

我们现在有一个经典的调度问题：我们有单位时间的工作，每个工作都有截止日期和利润，并且我们希望最大化总利润。 我们每轮最多可以安排一次切割，所以这是一个有截止日期的单机调度问题。 

暴力破解之所以有效，是因为我们直接模拟切割和传播，但它失败了，因为它没有利用感染时间由深度决定的事实。 认识到该结构将问题简化为计算子树和并解决贪婪调度问题。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 模拟| O(n^2) | O(n^2) | O(n) | 太慢了|
 | 截止日期调度+贪心| O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

我们将问题转化为树边上的调度任务。 

1. 以节点 1 为树的根并计算每个节点的深度。 深度表示如果没有任何内容被切割，该节点变为红色的时间。 
2. 计算每个子树中的权重总和。 这很重要，因为如果我们切断从父节点到节点的边，我们可以防止整个子树变成红色，因此保存的值正是子树的总和。 
3. 对于除根之外的每个节点，定义一个与从其父节点到自身的边相对应的“作业”。 每个作业的利润等于其子树总和，截止日期等于其深度。 
4. 现在，我们需要选择这些作业的子集，以便每个时间单位选择的作业不超过一个，并且每个选定的作业都在其截止日期之前安排。 所选择的工作的总利润应该最大化。 
5. 按利润降序对所有作业进行排序。 迭代它们，并为每个作业尝试将其分配到严格早于或等于其截止日期的最新可用时间段。 如果有空位，我们会安排； 否则我们会丢弃它。 
6. 调度作业的利润总和是保持白色的节点的总权重。 最终答案是total_sum - 2 * red_sum，或者等效的total_sum - 2 * saving_white_subtrees，具体取决于解释，但直接通过red = total - saving进行计算是最简单的。 

它起作用的原因来自于一个结构不变量：每个子树要么完全被感染，要么完全保存，唯一的控制机制是通向它的边是否在其深度截止日期之前被切割。 由于切割全局限制为每轮一次，因此问题变成了独立作业的独立调度，并且子树之间除了时隙竞争之外不存在交互。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    w = [0] * (n + 1)

    vals = list(map(int, input().split()))
    for i in range(2, n + 1):
        w[i] = vals[i - 2]

    g = [[] for _ in range(n + 1)]
    for _ in range(n - 1):
        u, v = map(int, input().split())
        g[u].append(v)
        g[v].append(u)

    parent = [0] * (n + 1)
    depth = [0] * (n + 1)
    order = []

    stack = [1]
    parent[1] = -1
    depth[1] = 0

    while stack:
        u = stack.pop()
        order.append(u)
        for v in g[u]:
            if v == parent[u]:
                continue
            parent[v] = u
            depth[v] = depth[u] + 1
            stack.append(v)

    sub = [0] * (n + 1)
    for i in range(1, n + 1):
        sub[i] = w[i]

    for u in reversed(order):
        for v in g[u]:
            if parent[v] == u:
                sub[u] += sub[v]

    jobs = []
    for v in range(2, n + 1):
        jobs.append((sub[v], depth[v]))

    jobs.sort(reverse=True)

    maxd = n
    used = [False] * (maxd + 1)
    saved = 0

    for val, d in jobs:
        for t in range(min(d, n), 0, -1):
            if not used[t]:
                used[t] = True
                saved += val
                break

    total = sum(w)
    # nodes saved correspond to white remaining subtrees;
    # red = total - saved, score = white - red = total - 2*red = 2*saved - total
    print(2 * saved - total)

if __name__ == "__main__":
    solve()
```实现的第一部分构建有根树并使用显式堆栈遍历计算深度。 这避免了递归深度问题并确保线性时间遍历。 

子树计算以与遍历相反的顺序完成，将子树的贡献累积到父树中。 这为每个节点提供了它所控制的子树的准确权重。 

除根节点外的每个节点都成为调度作业。 截止日期是它的深度，利润是它的子树之和。 

贪婪调度循环将每个作业分配到截止日期之前的最新空闲时间段。 这一点至关重要，因为尽可能晚地安排工作可以为更紧迫的期限保留较早的时间。 

最后，分数转换使用保存的权重和最终目标之间的一致性：最大化白色减去红色减少到最大化保存的子树权重贡献。 

## 工作示例

 考虑一棵小树，其中节点 1 连接到 2 和 3，并且 2 和 3 都是叶子。 令权重为 w2 = 4，w3 = 5。 

我们计算子树和：sub(2)=4，sub(3)=5。 深度均为 1。作业为 (4,1) 和 (5,1)。 

| 步骤| 考虑的工作 | 预定时段 | 已保存 |
 | ---| ---| ---| ---|
 | 1 | (5,1) | t = 1 | 5 |
 | 2 | (4,1) | 没有可用 | 5 |

 在时间 1 只能进行一次切割，因此我们选择利润较高的子树。 

这表明竞争的相同截止日期子树仅强制选择最大的子树。 

现在考虑权重分别为 1、10、100、1000 的链 1-2-3-4。 

子树总和是累积的：sub(4)=1000，sub(3)=1010，sub(2)=1020。 深度分别为 3、2、1。 

| 步骤| 工作 | 截止日期| 决定|
 | ---| ---| ---| ---|
 | 1 | (1020,1)| 1 | 预定 |
 | 2 | (1010,2)| 2 | 预定 |
 | 3 | (1000,3)| 3 | 预定 |

 这里所有的工作都适合，因为最后期限沿着链条增加，从而确认当时间限制足够宽松时不会发生冲突。

 ## 边缘情况

 当多个大型子树共享相同深度时，会发生关键的边缘情况。 在这种情况下，只能保留其中一个，因为每个时间单位只能执行一次剪切。 贪婪利润排序确保选择最大的子树，因为所有子树都有相同的截止日期，并且只有利润区分可行性。 

另一种情况发生在树中的大子树位于树深处但被从未及时切割的浅祖先阻挡时。 在这种情况下，即使子树本身很有价值，但它的作业无法在截止日期之前调度，因此它被调度循环正确地丢弃，反映出它不可避免地会被感染。 

最后一种情况是一条长链，其中每个节点的深度期限不断增加。 在这里，每项工作都是可行的，并且算法会安排所有这些工作，这与削减可以随着时间的推移分散而不会发生冲突的事实相匹配。
