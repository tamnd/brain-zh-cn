 ## 解决方案
 ## 问题理解

 我们给出了三个整数集合。 您可以将它们视为必须随着时间的推移而匹配的三种类型的资源：一组代表在损坏中“生存”的单位，另外两组代表在生存时保护它们或提高其效率的设备。 

建模的过程可以解释为这些资源之间交互的持续积累。 第一组中的每个单位都与第二组中的一个物品配对，并且根据该单位存活的时间，它还会与第三组中的物品进行交互。 我们需要计算的最终数量是在所有相关活动结束之前这些交互累积的总有效贡献。 

关键的困难在于，结果在很大程度上取决于三组的配对方式。 分配不当可能会导致强大的资源浪费在较弱的生命周期上，从而减少总贡献。 

这些约束（对于这种类型的问题来说是典型的，在多个测试用例中数组尺寸很大）意味着任何尝试所有配对或显式模拟交互的解决方案都会太慢。 一种简单的方法会考虑排列或匹配策略，导致阶乘或至少二次爆炸，当每个测试可以包含大型数组时，这是不可行的。 

当其中一组明显大于另一组时，就会出现微妙的边缘情况。 例如，如果我们贪婪地按输入顺序配对而不排序，我们很容易得到次优匹配。 

考虑生存时间不匹配的情况：

 输入：```
a = [10, 1]
b = [1, 10]
```如果按顺序配对，较强的幸存者 (10) 会获得较弱的支持 (1)，较弱的幸存者会获得较强的支持。 这会导致潜在贡献的过早丧失。 正确的策略显然是大与大搭配。 

另一种边缘情况是一个列表为空或在修剪后实际上不相关。 幼稚的实现可能仍会尝试访问无效索引或假设大小相等。 

## 方法

 强力解决方案将尝试以所有可能的排列将第一组中的每个元素分配给第二组中的元素，并类似地决定第三组如何交互。 这将正确地对系统进行建模，但需要迭代所有匹配。 即使限制于配对，这也已经是大小的阶乘，因为每个分配选择都会影响未来的贡献。 对于 n 达到很大的值，这变得完全不可行。 

关键的观察结果是系统仅依赖于相对顺序，而不依赖于同一性。 更强的单位应该始终与更强的支持相匹配，因为任何不匹配都可以通过交换对来局部改善。 如果两对在强度上“交叉”，则严格交换它们可以提高或保留所有贡献。 这是一个经典的交换论证，直接导致排序成为最佳结构。 

一旦我们接受每个组应按降序排序，交互结构就变成线性的。 然后，我们通过按“损坏耗尽点”升序处理阈值来模拟贡献的演变。 每个事件对应于某些资源被完全消耗的时刻，并且在事件之间，贡献以恒定的速率增长。 

我们维护剩余的活跃单位数量，并在没有任何变化的时间间隔内累积贡献。 当达到阈值时，我们减少活动计数并继续。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(n!) 或更糟 | O(n) | 太慢了 |
 | 最优（排序+扫描模拟）| O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 1. 按降序对所有三个数组进行排序，以便首先考虑较强的元素。 这确保了更高的容量或耐用性始终分配给更高的冲击位置。 
2.排序后将第一组和第二组索引按索引配对。 每对代表一个具有生存值和盾牌能力的单位。 
3. 对于每个单位，如果同一索引处存在第三组元素，则创建一个附加事件，表示受单位生存限制限制的额外潜在贡献。 
4. 将所有这些贡献收集到事件列表中，其中每个事件都有一个阈值和关联的贡献值。 
5. 按阈值对这些事件进行排序，以便我们按照“耗尽时间”增加的顺序处理它们。 
6. 清查事件，同时保持已过去的总“时间”或累积损坏的数量。 对于每个事件，计算在达到阈值之前将累积多少额外贡献。 
7. 累积捐款，但设定上限，以免超出活动的可用捐款。 当超过每个事件阈值时减少活动计数。 
8. 最终累计答案为所有资源耗尽前收集到的贡献总额。 

### 为什么它有效

 正确性来自于排序的交换论证。 如果一个更强的生存单位与一个比另一个可用单位更弱的护盾配对，交换它们不会减少任何贡献，并且严格改善至少一个配对。 这保证了排序对齐最佳配对。 

然后扫描就会起作用，因为系统在离散的耗尽点之间线性演化。 阈值之间没有任何变化，因此累积是均匀的，并且可以分块计算，而不是逐步模拟。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m, k = map(int, input().split())
    a = list(map(int, input().split()))
    b = list(map(int, input().split()))
    c = list(map(int, input().split()))

    a.sort(reverse=True)
    b.sort(reverse=True)
    c.sort(reverse=True)

    b = b[:n]
    c = c[:n]

    events = []

    for i in range(n):
        events.append([a[i], b[i]])
        if i < len(c):
            events.append([min(a[i], c[i]), 0])

    events.sort()

    ans = 0
    dec = 0
    tim = 0
    num = len(events) - 1

    for i in range(len(events)):
        diff = events[i][0] - dec
        tim += num * diff
        ans += min(tim, events[i][1])
        num -= 1
        dec = events[i][0]

    print(ans)

if __name__ == "__main__":
    t = int(input())
    for _ in range(t):
        solve()
```该实现首先按降序对所有数组进行排序，强制执行贪婪配对原则。 第二个和第三个数组被修剪以匹配主要单元的数量，因为排序后只有最上面的匹配很重要。 

事件列表对所有交互进行编码。 每对贡献一个基本事件`a[i]`和`b[i]`，以及可选的上限事件`c[i]`。 按阈值对这些事件进行排序使我们能够按耗尽的递增顺序处理它们。 

扫描变量维护累积时间的运行概念（`tim`），剩余多少个活动段（`num`），以及我们在阈值空间（`dec`）。 每次迭代都会前进到下一个阈值，累积该时间间隔内的贡献，并消耗一个事件。 

一个微妙的细节是使用`min(a[i], c[i])`，这就保证了第三种资源的贡献不能超出相应单位的生存能力。 这可以防止在支持超出生存能力的情况下出现过度计数。

 ## 边缘情况

 关键边缘情况是一个数组仅包含非常大的值，而另一个数组包含非常小的值。 如果不进行排序，配对就会错误地将极端分配到错误的方向，从而导致总贡献崩溃。 排序后，大值正确对齐并且扫描行为单调。 

另一种边缘情况是第三个数组中的所有值都为零。 在这种情况下，只有碱基对贡献很重要。 该算法仍然会创建事件，但所有次要贡献都会消失，并且扫描干净地减少为主要对上的线性累积。 

最后的边缘情况是所有数组都具有相同的值。 在这种情况下，每个配对都是等价的，算法的排序不会改变结构，而是确保确定性的处理顺序。 扫描仍然正确处理统一阈值，确认对称下的稳定性。
