 ## 解决方案
 ## 问题理解

 我们得到一行$n$人们，每个人都持有一些相同的物品。 项目总数可除以$n$，所以有一个明确的目标值：每个人最终都必须精确地得到$s/n$项目。 

最初，分布是$a_1, a_2, \dots, a_n$。 然后一系列更新永久地修改此分布：每次更新都会将一定数量的项目从一个人移动到另一个人，因此每次更新后我们都会获得一个新数组。 每次更新后，我们必须回答同一个问题：在给定非常具体的移动规则的情况下，小组可以多快地重新分配物品，以便每个人都达到准确的平均值。 

密钥重新分配过程受到限制。 一分钟内，每个人最多可以同时向左邻居发送一件物品，向右邻居发送最多一件物品，只要他们有足够的物品。 这意味着流沿着相邻边缘并行发生，但每条边缘每分钟最多可以在每个方向携带一个项目，并且每个节点每分钟最多可以向每一侧发送一个单元。 

每次更新后的输出是在这些移动规则下达到完美平等所需的最小分钟数。 

限制条件很大：$n$和$q$所有测试的总和最多为$3 \cdot 10^5$。 这立即排除了使用重新分配过程的任何二次甚至线性扫描大量模拟从头开始重新计算每个查询的答案。 解决方案必须维护数组的紧凑摘要，并以每次操作的对数或恒定摊销时间更新它。 

最微妙的困难在于，答案不仅仅在于完全不平衡。 局部结构很重要：即使全局总和是正确的，限制因素是必须跨越边缘的不平衡程度，这取决于前缀行为。 

一个天真的错误是假设只有最大赤字或盈余才重要。 例如，考虑如下配置$[2, 0, 2]$有目标$2$。 总不平衡为零，但重新分配需要时间，因为两个单元都必须穿过中间节点。 如果我们错误地只关注全球盈余，我们就会预测零时间。 

另一种失败模式是忽略方向冲突。 在这样的情况下$[0, 3, 0]$有目标$1$，两个单位必须向左移动，一个向右移动，并且它们在中心相互作用，如果不仔细建模，这会增加超出简单距离估计的时间。 

真正的困难是捕获由同时通过边缘的流量引起的瓶颈。 

## 方法

 对重新分配过程的直接模拟将尝试对每分钟进行建模：每个节点向左和向右推送项目，重复更新数组。 每分钟的成本$O(n)$，最坏情况下的分钟数也是$O(n)$，导致$O(n^2)$每个查询。 高达$3 \cdot 10^5$操作，这是不可行的。 

即使我们尝试预先计算静态数组的答案，我们仍然面临动态问题，因为更新会任意更改值。 因此，我们需要一种方法来从结构摘要而不是完整的模拟中重新计算答案。 

关键的见解是停止考虑单个项目，而是跟踪累积的不平衡。 定义$b_i = a_i - \text{avg}$。 问题变成：每分钟使用双向单位容量边缘消除所有不平衡需要多长时间？ 

现在我们将其解释为路径图上的流问题。 每个前缀都有净过剩或净赤字。 前缀不平衡必须跨越边界传输$i$和$i+1$。 当所有所需的传输都得到最佳调度时，所需的时间取决于任何边缘的最大拥塞程度。 

对此进行建模的标准方法是定义前缀和：$$p_i = \sum_{j=1}^i b_j$$每个$p_i$代表有多少盈余必须通过之间的削减$i$和$i+1$。 重新分配过程每分钟在每个方向上最多可以移动一个单位穿过每个边缘，这意味着每个边缘每分钟在每个方向上的容量为 1。 这将问题转变为测量必须同时穿过每个边缘的流量。 

关键的观察是，最短时间等于经过优化调度后必须穿过它们的绝对净流的最大边缘，这可以使用前缀极值来表示。 事实上，答案可以简化为跟踪在与全局约束进行平衡时正或负前缀和的大小。 

这导致了动态结构：每次更新后，我们只更改数组中的两个位置，因此前缀和以结构化方式更改。 我们可以维护一个段树，其中存储每个段的最小和最大前缀和差，使我们能够有效地计算所需的最大拥塞。 

因此，每个查询变成点更新加上从线段信息导出的全局最大值的线段树重新计算。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力模拟 |$O(n \cdot q \cdot \text{time})$|$O(n)$| 太慢了 |
 | 前缀不平衡的线段树|$O(q \log n)$|$O(n)$| 已接受 |

 ## 算法演练

 我们通过隐式减去平均值将数组转换为不平衡形式。 由于总和保持不变且可除以$n$，我们可以处理偏差。 

我们维护这些偏差的前缀和数组。 目标是通过调整值在此结构中的位置来支持值发生变化的更新。 

### 步骤

 1. 计算初始值$b_i = a_i - \frac{s}{n}$。 

这将问题转化为平衡正流和负流而不是绝对计数。 它确保总和$b_i$为零。 
2. 构建前缀和$p_i$隐式地通过维护以下内容的线段树：

 任何段上的最小和最大前缀总和。 

这使我们能够了解不平衡如何在不同时间间隔内累积，而无需显式存储所有前缀。 
3. 对于每次更新$u \to v$有价值$x$，调整底层数组：

 减去$x$从$a_u$， 添加$x$到$a_v$。 

这仅影响两个位置$b$，所以我们在线段树中应用两个点更新。 
4. 每次更新后，重新计算全局度量：

 所需时间是任一方向上最差前缀偏差的最大值。 

这对应于必须通过某些边缘的最大拥塞。 
5. 为每个查询直接输出该值。 

### 为什么它有效

 重新分配过程相当于沿着路径图的边缘推动单位流量，其中每个边缘每分钟每个方向的容量为 1。 任何位置不平衡$i$必须跨越边缘运输才能达到全球平衡。 前缀和精确编码必须穿过每个边界的净流量。 

线段树在更新时维护所有可能的前缀极值。 由于每次更新仅更改两个点，因此所有受影响的前缀结构都会在本地捕获，但会通过树传播。 最大绝对前缀偏差对应于瓶颈边缘，它决定了路由所有流所需的最短时间。 

数组中没有其他结构可以独立于前缀不平衡而增加时间，因此该值完全表征了答案。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class SegTree:
    def __init__(self, arr):
        self.n = len(arr)
        self.mx = [0] * (4 * self.n)
        self.mn = [0] * (4 * self.n)
        self.arr = arr
        self.build(1, 0, self.n - 1)

    def build(self, v, l, r):
        if l == r:
            self.mx[v] = self.arr[l]
            self.mn[v] = self.arr[l]
            return
        m = (l + r) // 2
        self.build(v * 2, l, m)
        self.build(v * 2 + 1, m + 1, r)
        self.mx[v] = max(self.mx[v * 2], self.mx[v * 2 + 1])
        self.mn[v] = min(self.mn[v * 2], self.mn[v * 2 + 1])

    def update(self, v, l, r, i, val):
        if l == r:
            self.mx[v] += val
            self.mn[v] += val
            return
        m = (l + r) // 2
        if i <= m:
            self.update(v * 2, l, m, i, val)
        else:
            self.update(v * 2 + 1, m + 1, r, i, val)
        self.mx[v] = max(self.mx[v * 2], self.mx[v * 2 + 1])
        self.mn[v] = min(self.mn[v * 2], self.mn[v * 2 + 1])

    def answer(self):
        return max(abs(self.mx[1]), abs(self.mn[1]))

def solve():
    t = int(input())
    for _ in range(t):
        n, q = map(int, input().split())
        a = list(map(int, input().split()))
        s = sum(a)
        avg = s // n

        b = [x - avg for x in a]
        st = SegTree(b)

        for _ in range(q):
            u, v, x = map(int, input().split())
            u -= 1
            v -= 1

            st.update(1, 0, n - 1, u, -x)
            st.update(1, 0, n - 1, v, +x)

            print(st.answer())

if __name__ == "__main__":
    solve()
```该代码直接维护不平衡数组，并将每次传输作为一对点更新应用。 每次更新都会修改线段树，该线段树跟踪底层不平衡结构的全局最大值和最小值。 

每次查询后的答案是所有位置的最大绝对偏差。 这捕获了隐含流量系统中最严重的拥塞点，从而确定了所需的最小分钟数。 

## 工作示例

 考虑一个小例子，其中$n = 3$, 初始数组$[2, 2, 2]$，平均值为$2$, so initial imbalance is zero everywhere. 更新后，将一个单位从位置 1 移动到位置 3，我们得到了不平衡$[-1, 0, 1]$。 

| 步骤| Update | 数组| Imbalance | Segment Tree max | Segment Tree min | Answer |
 | ---| ---| ---| ---| ---| ---| ---|
 | 1 | 初始| [2,2,2] | [0,0,0] | 0 | 0 | 0 |
 | 2 | 1→3 | [1,2,3]| [-1,0,1] | 1 | -1 | 1 |

 这表明，即使总和是平衡的，流也必须穿过至少一个携带一个单位的边，需要一步。 

现在考虑一个稍微大一点的情况：$n = 4$， 最初的$[0,4,0,4]$， 平均的$2$。 

Initial imbalance is$[-2, 2, -2, 2]$。 

| 步骤| 更新 | 不平衡| 最大| 分钟 | 回答 |
 | ---| ---| ---| ---| ---| ---|
 | 1 | 初始| [-2,2,-2,2] | 2 | -2 | 2 |
 | 2 | 将 1 从 2 移动到 3 | [-2,1,-1,2] | 2 | -2 | 2 |

 答案保持稳定，因为瓶颈仍然由最大前缀偏差决定，而不是由局部平滑决定。 

这证实了更新仅影响局部不平衡，但全局瓶颈结构占主导地位。

 ## 边缘情况

 一个重要的情况是，当更新造成不平衡时，这种不平衡会在全球范围内取消，但集中在本地。 例如，在远程节点之间来回传输项目不会改变总和，但会增加峰值前缀偏差。 线段树正确地捕获了这一点，因为每次更新都会修改两个点并通过前缀结构传播更改。 

另一种情况是对同一索引进行重复更新。 由于每次更新都作为增量应用，因此该结构自然会累积更改。 前缀最大/最小值继续反映真实的拥塞情况，无需从头开始重新计算。 

最后，更新后所有值都相等的情况应始终返回零。 由于所有前缀和都归零，因此两个存储的极值都变为零，并且答案正确地是最小的。
