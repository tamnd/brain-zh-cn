 ## 解决方案
 ## 问题理解

 我们得到一个仅由值 1、2 和 3 组成的长序列。从这个序列中，我们希望提取尽可能多的不相交的索引三元组，其中每个三元组使用三个不同的位置并尊重索引的顺序。 

如果一个三元组的值按照递增的索引顺序形成模式 1、2、3，或者按照递增的索引顺序形成相反的模式 3、2、1，则该三元组被认为是有效的。 每个索引最多只能属于一个选定的三元组，因此一旦使用了一个位置，就不能在另一组中重复使用。 

任务不仅仅是计算存在多少个这样的三元组，而是显式构造不相交的有效三元组的最大集合。 

N 最大为 600000 的约束立即排除了任何尝试枚举或测试三元组组合的方法。 任何解决方案都必须是线性或近线性的，因为在最坏的情况下，二次行为意味着 10^11 次运算。 

当在本地进行贪婪匹配而不考虑全局平衡时，会出现微妙的失败模式。 例如，如果序列严重倾斜，如 1 1 1 2 2 2 3 3 3，则过于积极地将早期出现的 1 和 3 配对可能会阻止形成最佳的 1-2-3 链。 如果我们只尝试匹配正向模式而忽略反向 3-2-1 结构，则会出现另一个问题，当分布不对称时，这可能会产生次优结果。 

## 方法

 一种简单的方法是尝试通过检查 i、j、k 的每个组合（其中 i < j < k）来构建三元组，并验证这些值是否匹配 1-2-3 或 3-2-1，同时确保未使用索引。 在最坏的情况下，这需要迭代 O(N^3) 个三元组，这对于 N 高达 600000 来说是不可能的。 

即使将其简化为固定中间索引 j 并搜索匹配的 i 和 k 仍然会导致 O(N^2)。 核心问题是每个元素都可能参与许多候选三元组，并且成对检查兼容性仍然会导致组合爆炸。 

关键的观察是每个有效的三元组都有一个非常严格的结构。 每个三元组的值要么增加（1 然后 2 然后 3），要么减少（3 然后 2 然后 1）。 在这两种情况下，中间元素始终是 2。这将问题简化为将每个选定的 2 与左侧一个 1 和右侧一个 3 配对，或者对称地左侧一个 3 和右侧一个 1。 

这表明将 2 视为锚点。 然后，我们尝试将每个 2 与左侧池和右侧池相匹配。 目标变成最大化可以满足多少个锚点。 由于每个索引最多使用一次，因此我们需要一种贪婪策略来为未来的匹配保留灵活性。 

我们扫描数组并维护三个有序的索引列表：1、2 和 3 的位置。 然后，我们通过将 2 与最接近的可用兼容端点配对来构造三元组。 对于位置 j 处的 2，我们尝试形成 1-2-3 或 3-2-1 三元组。 贪婪地选择最近的可用合作伙伴可以确保我们不会浪费以后可能需要的极端头寸。 

强制这种贪婪的正确方法是从左到右处理 2，并维护两个指向 1-list 和 3-list 的指针。 每次我们尝试形成一个三元组时，我们都会选择仍然位于当前 2 的正确一侧且尚未使用的最早有效的左元素和最早的有效右元素。 

这将问题转化为指针前进的线性扫描，其中每个索引最多被消耗一次。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(N^3) | O(N^3) | O(1) | O(1) | 太慢了|
 | 最优贪心匹配| O(N) | O(N) | 已接受 |

 ## 算法演练

我们首先将索引分成三个递增的列表，每个值一个。 这使我们能够推断可用性，而无需重复扫描原始数组。 

然后，我们从左到右处理潜在中心，即值为 2 的索引。 对于每个这样的索引，我们尝试构建一个有效的三元组。 

对于位置 j 处的每个 2，我们尝试首先构建一个 1-2-3 三元组。 我们检查是否存在索引 i < j 的未使用 1 和索引 k > j 的未使用 3。 如果两者都存在，我们分配最小的未使用的有效 i 和最小的未使用的有效 k。 这最大限度地减少了左侧大索引和右侧小索引的消耗。 

如果无法形成 1-2-3 阵型，我们会尝试相反的 3-2-1 阵型。 我们检查左侧是否有 3，右侧是否有 1，再次选取最接近的候选值。 

如果这两种模式都不可能，我们就跳过这个 2，因为在当前可用性下它无法对任何有效的三元组做出贡献。 

每次我们形成一个三元组时，我们都会标记使用过的索引，这样它们就不能再次参与。 

### 为什么它有效

 该算法保持了以下属性：所有未使用的 1、2 和 3 按索引递增顺序保持可用，并且我们始终消耗与当前 2 最接近的有效候选者。这可以防止对未来 2 的不必要的阻塞，因为当存在更接近的端点时使用极端端点只会降低灵活性，而不会增加可能的三元组的数量。 

任何有效的解决方案都可以转换为使用贪婪最近匹配的解决方案，而不会减少三元组的数量，因为将较远的端点与较近的未使用的端点交换可以保留有效性，同时严格提高或维持两侧的剩余可用性。 这种交换论证确保贪婪选择不会降低最优性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = list(map(int, input().split()))
    
    pos1 = []
    pos2 = []
    pos3 = []
    
    for i, x in enumerate(a):
        if x == 1:
            pos1.append(i)
        elif x == 2:
            pos2.append(i)
        else:
            pos3.append(i)
    
    used1 = [False] * len(pos1)
    used3 = [False] * len(pos3)
    
    p1 = 0
    p3 = 0
    
    ans = []
    
    # process each 2 as center
    for j in pos2:
        while p1 < len(pos1) and pos1[p1] < j and used1[p1]:
            p1 += 1
        
        while p3 < len(pos3) and pos3[p3] < j and used3[p3]:
            p3 += 1
        
        # try 1-2-3
        i = p1
        while i < len(pos1) and (pos1[i] >= j or used1[i]):
            i += 1
        
        k = p3
        while k < len(pos3) and (pos3[k] <= j or used3[k]):
            k += 1
        
        if i < len(pos1) and k < len(pos3):
            ans.append((pos1[i], j, pos3[k]))
            used1[i] = True
            used3[k] = True
            continue
        
        # try 3-2-1
        i = 0
        while i < len(pos3) and (pos3[i] >= j or used3[i]):
            i += 1
        
        k = 0
        while k < len(pos1) and (pos1[k] <= j or used1[k]):
            k += 1
        
        if i < len(pos3) and k < len(pos1):
            ans.append((pos3[i], j, pos1[k]))
            used3[i] = True
            used1[k] = True
    
    print(len(ans))
    for i, j, k in ans:
        print(i, j, k)

if __name__ == "__main__":
    solve()
```该解决方案首先按值拆分索引，这可以避免重复扫描原始数组。 数组`pos1`,`pos2`， 和`pos3`存储排序位置，这对于确保指针移动保持正确性至关重要。 

这`used1`和`used3`数组跟踪特定的出现是否已分配给三元组。 这可以避免重复计算。 

对于每个 2，我们尝试找到兼容的端点。 指针维护确保我们不会重复扫描已使用的索引。 支票`pos1[i] < j`和`pos3[k] > j`强制执行排序约束。 

一个微妙的点是我们总是首先尝试正向模式 1-2-3。 这种选择在正确性方面是任意的，但可以稳定匹配行为并避免过早逆转。 

## 工作示例

 考虑输入：```
6
1 2 3 2 1 3
```我们有：

 位置 1 = [0, 4], 位置 2 = [1, 3], 位置 3 = [2, 5]

 我们首先在索引 1 处处理 2。 

| 步骤| 2 索引 | 选择 1 | 选择 3 | 行动|
 | --- | --- | --- | --- | --- |
 | 1 | 1 | 0 | 2 | 表格 1-2-3 |
 | 2 | 3 | 4 | 5 | 表格 1-2-3 |

 输出变为：

 (0,1,2) 和 (4,3,5)

 该跟踪显示了贪婪匹配如何保留早期和晚期结构。 

现在考虑：```
5
3 2 1 3 2
```位置1 = [2]，位置2 = [1,4]，位置3 = [0,3]

 我们在索引 1 处处理 2：

 | 步骤| 2 索引 | 选择 3 | 选择 1 | 行动|
 | --- | --- | --- | --- | --- |
 | 1 | 1 | 0 | 2 | 表格 3-2-1 |

 索引 4 处的下一个 2 无法形成任何三元组，因为所有兼容端点均已使用或未对齐。 

这表明早期消耗最近的有效端点不会阻碍可行性。 

## 复杂度分析

 | 测量| 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 | O(N) | 每个索引最多被扫描并标记一次，指针移动是单调的 |

 线性结构非常适合 N 高达 600000 的约束，因为每个操作都是恒定的摊销工作。

 ## 边缘情况

 一个关键的边缘情况是存在有效的三元组，但贪婪匹配必须避免过早耗尽一侧。 考虑：```
7
1 2 3 1 2 3 1
```形成三元组的方法有多种，但只有两个完整的三元组是可能的。 该算法按顺序处理 2 秒，并且始终采用最近的有效端点，因此索引 1 处的前 2 个端点与 (0,2) 配对。 索引 4 处的第二个 2 与 (3,5) 配对。 索引 6 处的剩余 1 保持未使用，正确反映了最佳打包。 

另一个边缘情况是只有反向三元组是可能的：```
4
3 2 1 2
```使用索引 (0,1,2) 只能形成一个三元组：(3,2,1)。 第二个 2 无法匹配，因为没有剩余的有效端点满足排序约束。 该算法正确地消耗唯一可行的结构并停止，而不会强制无效配对。
