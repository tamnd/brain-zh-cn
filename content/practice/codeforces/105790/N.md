 ## 解决方案
 ## 问题理解

 我们有一个$N \times M$网格。 每次在该位置建造护盾时$(x,y)$，它保护行中的每个单元格$x$以及列中的每个单元格$y$。 如果一个单元受到至少一个屏蔽层的保护，那么它就可以使用。 

有两种类型的操作。 

类型 1 操作会建造一个新的护盾。 

类型 2 操作询问 Almeiding 是否开始于$(x_i,y_i)$可以达到$(x_f,y_f)$仅使用受保护的单元格，并在四个基本方向上一次移动一步。 

网格本身最多可以包含$10^6$单元格，而查询数量多达$2 \cdot 10^5$。 任何试图维护每个受保护单元的状态或为每个查询执行图形搜索的解决方案都会立即变得太慢。 即使是在一个大小的网格上的单个 BFS$10^6$成本已经很高，而且重复这样做是不可能的。 我们需要一个$O(1)$或者$O(\log N)$每个查询的答案。 

主要危险是误解屏蔽层创建的连接结构。 

考虑这个输入：```
3 3 2
1 2 2
2 2 1 2 3
```屏蔽激活第 2 行和第 2 列。两个单元格$(2,1)$和$(2,3)$位于受保护的行上，所以答案是：```
SIM
```仅检查端点是否位于同一受保护行或列上的简单实现在更一般的情况下会失败。 

另一个有趣的案例是：```
3 3 2
1 2 2
2 1 2 3 2
```开始和结束单元格都受到保护，因为它们属于第 2 列。正确答案是：```
SIM
```即使单元格位于不同的行中，整个列也会受到保护。 

最后：```
3 3 2
1 2 2
2 1 1 3 3
```两个端点都不受保护。 正确答案是：```
NAO
```路径甚至不能在未受保护的单元上开始或结束。 

## 方法

 强力解决方案将显式标记每个受保护的单元，并且对于每个类型 2 查询，在两个端点之间运行 BFS 或 DFS。 这是正确的，因为它直接模拟受保护区域中的运动。 

问题在于规模。 单个屏蔽可以影响整行和整列。 高达$10^6$细胞和$2 \cdot 10^5$查询、反复探索网格远远超出了极限。 

关键的观察来自于理解护盾实际产生的东西。 

当护盾建造于$(x,y)$， 排$x$受到完全保护并列$y$变得完全受到保护。 他们的交叉细胞$(x,y)$将行和列连接在一起。 

由于每个屏蔽总是激活一行和一列，因此每个受保护的结构至少包含一个行列交叉点。 任何受保护的单元格要么属于激活的行，要么属于激活的列。 从这样的单元格，我们可以移动到交叉点，然后从那里移动到由防护罩激活的任何其他受保护的行或列。 因此，所有受保护的单元都属于单个连接的组件。 

这完全改变了问题。 

类型 2 查询根本不需要检查连接性。 我们只需要验证两个端点是否都受到保护。 如果两者都受到保护，则它们自动属于唯一的受保护组件，并且可以相互访问。 如果任一端点未受保护，则答案是不可能的。 

我们可以维护两个布尔数组：

 一个数组记录了哪些行曾经作为盾牌中心出现过。 

其他记录了哪些柱子曾经作为盾牌中心出现过。 

一个细胞$(r,c)$恰好在行时受到保护$r$处于活动状态或列$c$是活跃的。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 每个查询的强力 BFS | O(Q·N·M) 最坏情况 | O(N·M) | 太慢了 |
 | 最佳行/列跟踪 | O(Q) | O(N + M) | 已接受 |

 ## 算法演练

 1. 创建布尔数组`row`尺寸的`N`和一个布尔数组`col`尺寸的`M`。 
2. 对于类型 1 查询`(x, y)`， 标记`row[x] = True`和`col[y] = True`。 
3. 对于类型 2 查询，确定起始单元格是否受保护。 

一个细胞`(r, c)`受到保护，如果`row[r]`处于活动状态或`col[c]`是活跃的。 
4. 使用相同的规则确定目标单元是否受到保护。 
5. 如果两个单元格都受到保护，则打印`"SIM"`。 

每个受保护单元都属于同一个相连的受保护区域。 
6. 否则打印`"NAO"`。 

### 为什么它有效

 每个护盾都会激活一整行和一整列。 这两个结构在护盾中心相交，在它们之间建立了连接。 

任何受保护的单元格都位于活动行或活动列上。 从该单元格，我们可以沿着其行或列移动到屏蔽交叉点。 通过交叉点，每个受保护的行和受保护的列都成为一个连接结构的一部分。 因此，所有受保护的单元都是相互可达的。 

查询失败的唯一原因是起始或目标单元格未受保护。 这正是算法检查的内容。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

n, m, q = map(int, input().split())

rows = [False] * n
cols = [False] * m

ans = []

for _ in range(q):
    data = list(map(int, input().split()))

    if data[0] == 1:
        _, x, y = data
        rows[x - 1] = True
        cols[y - 1] = True
    else:
        _, xi, yi, xf, yf = data

        start_ok = rows[xi - 1] or cols[yi - 1]
        finish_ok = rows[xf - 1] or cols[yf - 1]

        ans.append("SIM" if start_ok and finish_ok else "NAO")

sys.stdout.write("\n".join(ans))
```这两个数组是问题的整个状态。 

创建屏蔽时，我们不会标记其行或列中的所有单元格。 这样做成本太高了。 相反，我们只记得行和列是活动的。 

对于查询单元格`(r,c)`，保护条件正是：```
rows[r] or cols[c]
```这直接符合保护的定义。 

该实现内部使用从零开始的索引，因此从输入读取的每个坐标在访问数组之前都会减一。 

不需要图形构造、BFS、DFS 或联合查找结构，因为屏蔽的几何形状已经暗示了连接性。

 | 2 | 构建 (1,1) | {1} | {1} | - |

## 复杂度分析

 | 测量| 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 | O(Q) | 每个查询仅执行几次数组访问 |
 | 空间| O(N + M) | 一个布尔数组用于行，一个布尔数组用于列 |

 自从$Q \le 2 \cdot 10^5$和$N \cdot M \le 10^6$，查询的线性处理很容易足够快。 除了两个数组之外，没有任何操作取决于网格本身的大小。

 ## 边缘情况

 考虑：```
1 1 1
2 1 1 1 1
```没有建造盾牌。 该行处于非活动状态，该列处于非活动状态。 该算法评估：```
rows[1] OR cols[1] = false
```对于两个端点，产生：```
NAO
```这是正确的，因为唯一的单元格不受保护。 

现在考虑：```
3 3 2
1 2 2
2 1 2 3 2
```构建护盾后，第 2 行和第 2 列将变为活动状态。 两个端点都位于第 2 列，因此都受到保护。 该算法返回：```
SIM
```即使行不同，也可以沿着活动列移动。 

最后：```
4 4 2
1 2 2
2 2 1 4 4
```起始单元格`(2,1)`由于第 2 行处于活动状态，因此受到保护。 目的地`(4,4)`不受保护，因为第 4 行和第 4 列均未处于活动状态。 该算法立即检测到这一点并打印：```
NAO
```任何路径都不能在未受保护的单元上结束，因此答案是正确的。
