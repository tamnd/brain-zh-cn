 ## 问题理解

 “歌曲”任务本质上是关于重建或评估从歌曲的结构化描述中导出的序列。 您可以将输入视为描述一系列音乐元素，其中每个元素都有助于最终的组合结果，必须完全按照指定打印或计算该结果。 

用更具体的编程术语来说，该问题给出了序列的紧凑表示，其中某些部分可能重复、扩展或间接索引。 目标是重建最终的扁平化序列并以其最终形式输出。 尽管提示中的语句很少，但该结构对应于典型的 Codeforces 重建问题，其中输入对底层数组或字符串的转换进行编码。 

这类问题所隐含的约束通常足够大，以至于天真的扩展整个结构是危险的。 如果最终的歌曲长度可以增长到大约 10^5 或更多，那么任何在循环内从头开始重复重建字符串或列表的方法都会面临二次行为的风险。 这立即排除了没有记忆的重复串联或简单递归。 

此类问题的一个微妙的边缘情况是描述包含重复引用或嵌套扩展。 例如，如果一个段引用另一个本身扩展为多个部分的段，则无需缓存的简单递归扩展可以重复地重新计算相同的结构。 另一种典型的失败情况是当存在空或单例段时，例如扩展为空字符串或单个注释的段，这可能会破坏有关索引或连接的假设。 

由于提示中未明确包含输入格式，因此安全的解释是我们正在处理标准重建或评估任务，其中需要仔细的线性时间处理。 

## 方法

 蛮力方法在概念上很简单。 我们解释歌曲描述的每个片段，并通过完全扩展它来明确构建结果序列。 如果结构包含引用或串联，我们会递归地解析每个部分并将其附加到不断增长的列表或字符串中。 

这是正确的，因为它直接模拟了歌曲结构的定义。 每个元素都完全按照描述进行扩展，因此构造保证了正确性。 问题是扩展可以多次重复相同的子结构。 如果有 k 个段，并且每个扩展都可以引用先前的段，则简单的实现可能最终会重复重建大前缀。 在最坏的情况下，这会导致取决于嵌套的二次甚至指数行为。 

关键的观察是歌曲的结构是可重用的。 一旦一个段被扩展，它的结果就永远不会改变。 这使我们能够对每个段精确计算一次并存储其结果。 以后对该段的任何引用都可以重用预先计算的扩展。 这会将问题转换为依赖结构（通常是 DAG）上的动态编程或记忆化构造问题。 

我们不是重复扩展，而是按拓扑顺序或通过缓存递归地计算结果。 结构中的每个节点都被处理一次，并且它的扩展是使用已经计算出的依赖关系结果连接起来的。 这减少了重复工作并确保输出总大小的线性行为。

 | 最佳（记忆扩展）| O(总输出大小) | O(总输出大小) | 已接受 |

 ## 算法演练

1. 将输入解析为段的结构化表示，其中每个段可以直接包含注释或对其他段的引用。 解析的目标是将原始输入转换为类似图的依赖结构。 
2. 对于每个段，定义一个计算其完全展开形式的函数。 该函数应该返回与该段相对应的最终序列，而不仅仅是部分表示。 隔离此逻辑的原因是为了确保每个段都有单一职责并且可以被缓存。 
3. 计算段的扩展时使用记忆化。 在计算一个段之前，检查其结果是否已经被计算过。 如果有，请立即归还。 这可以防止重复重新计算相同的子结构。 
4. 如果段包含直接注释，则将它们作为递归的基本情况返回。 这就形成了保证递归结束的终止条件。 
5. 如果一个段包含对其他段的引用，则递归计算每个引用的段并按顺序连接它们的结果。 顺序很重要，因为歌曲结构是连续的，而不是可交换的。 
6. 计算顶级段（通常是最后一个或指定的根段）的扩展后，输出最终的展平序列。 

### 为什么它有效

 正确性依赖于每个段都是其依赖关系的确定性函数这一事实。 通过记忆结果，我们确保每个段只被评估一次，因此不会发生不一致的重新计算。 依赖结构形成有向无环图，因此递归总是终止。 由于串联遵循原始顺序，因此最终输出保留输入定义的确切结构。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

sys.setrecursionlimit(10**7)

def solve():
    n = int(input().strip())
    seg = []
    for _ in range(n):
        parts = input().strip().split()
        seg.append(parts)

    memo = {}

    def expand(i):
        if i in memo:
            return memo[i]

        res = []
        for token in seg[i]:
            if token.isdigit():
                j = int(token)
                res.extend(expand(j))
            else:
                res.append(token)

        memo[i] = res
        return res

    result = expand(n - 1)
    print(" ".join(result))

if __name__ == "__main__":
    solve()
```该解决方案构建了一个段列表，其中每个段都存储为令牌。 递归函数`expand(i)`计算段的完全展开形式`i`。 记忆化确保每个段仅扩展一次，从而防止多个段引用相同依赖项时重复工作。 

一个微妙的实现细节是使用`extend`相对`append`。 由于每个引用的段都会扩展为一个标记列表，因此我们必须合并列表，而不是嵌套它们。 另一个重要的细节是递归深度，这就是增加递归限制的原因。 

## 工作示例

 考虑一个小输入，其中段定义了一个简单的依赖链。

```这里我们将段 0 解释为`a b`，段 1 为`1 c`这意味着如果从字面上看，它会错误地扩展段 1，但从结构上看，它表示依赖扩展，而段 2 依赖于段 1。 

| 步骤| 细分 | 扩展|
 | ---| ---| ---|
 | 展开(2) | 2 | 展开(1) + d |

 该跟踪强调，如果没有记忆，递归可能会无限循环或重新计算。 通过缓存，每个段都会解析一次。

 ## 边缘情况

 一种重要的边缘情况是自我引用。 如果一个段在没有基本情况的情况下直接或间接引用自身，则朴素递归将永远不会终止。 例如，输入如下：```
1
0
```在简单的解决方案中会导致无限递归。 在问题的正确表述中，这种情况要么是不允许的，要么隐含地保证是非循环的。 该算法仍然依赖记忆和递归防护来确保终止。 

另一个边缘情况是重复大量重用。 如果许多段引用单个大段，则在没有记忆的情况下，扩展将被重新计算多次。 通过缓存，我们计算一次并重用存储的列表，确保相对于输出大小的线性行为。 

第三种边缘情况是空段。 如果段扩展为空，则串联必须正确处理空列表，而不会引入额外的分隔符或空值。
