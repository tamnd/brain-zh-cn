 ## 解决方案
 ## 问题理解

 我们得到了一个代表直山沿线积雪高度的数组。 该数组是单调非递减的，这意味着当我们从索引 0 移动到索引 N−1 时，值永远不会减少。 每个查询都要求一个连续的段，其值的总和恰好为 K，并且在所有有效段中，我们必须输出最早开始的段，如果存在平局，则输出从该位置开始的最短的段。 

关键任务不仅仅是找到任何具有总和 K 的子数组，而是以确定性的方式解决歧义。 由于可能存在多个有效段，因此解决方案必须首先遵循左端点排序，然后遵循长度排序。 

约束很重要：N 最多可达 100,000，而 Q 最多为 100。每个数组值至少为 1，因此前缀和严格递增。 K 可以与 N² 一样大，这表明长段的总和是预期的，并且每个查询的 O(N) 或 O(N log N) 是可以接受的，但每个查询内的繁重工作必须仔细控制 O(NQ)。 

检查每个查询的所有子数组的简单方法会考虑 O(N²) 段，并且在最坏的情况下，整个 Q 会变成 10^2 操作，这显然是不可能的。 

一个更微妙的失败案例来自于忽略平局决胜局。 假设多个间隔之和为 K，并且解返回从右侧或任意顺序扫描中遇到的第一个间隔。 例如，如果数组是`[1, 2, 1, 2]`且 K = 3，有效区间包括`[0,1]`,`[1,2]`， 和`[2,3]`。 The correct answer must be`[0,1]`因为它具有最小的起始索引，即使简单的两指针扫描可能会根据移动方向首先发现稍后的匹配项。 

另一个极端情况是由于假设唯一性而产生的。 即使值是正的且单调的，前缀结构也不能保证每个 K 都有一个解。可以存在多个不相交的解，并且正确性取决于确定性选择规则。 

## 方法

 暴力解决方案将迭代每个查询的所有对 (a, b) 并计算段的总和。 即使前缀总和将每个范围总和减少到 O(1)，每个查询的成本仍然是 O(N²)。 当 Q 达到 100 时，这变得太慢了。 

我们通过利用两个结构特性来改进这一点。 首先，所有值都是正数，因此前缀和严格递增。 其次，该数组是单调非递减的，这进一步稳定了当我们移动端点时总和的演变方式。 

仅正性就足以启用固定 K 的两指针滑动窗口：我们可以维护一个窗口 [l, r]，其总和在扩展 r 时增加，在移动 l 时减少。 每个查询的时间复杂度为 O(N)。 然而，我们还需要字典顺序上最小的左端点，因此我们必须确保第一次为给定 K 找到有效窗口时，我们已经以一致的方式从左到右扫描了。 

单调结构保证了一旦我们向前移动左指针，我们就永远不需要重新访问同一总和目标的较早位置，因为向左扩展只会由于正性而进一步增加总和。 这确保了标准滑动窗口方法的正确性。 

因此，每个查询都是使用两个指针在线性时间内独立解决的，并且 Q 足够小，因此总体上是高效的。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(N²Q) | O(1) | O(1) | 太慢了 |
 | 每个查询的滑动窗口 | O(NQ)| O(1) | O(1) | 已接受 |

 ## 算法演练

 对于目标总和为 K 的每个查询，我们对数组运行两指针扫描。 

1. 初始化两个指针 l = 0 和 r = 0，当前和 s = 0。这表示一个空窗口，将从最左边的可能位置开始贪婪地扩展。 
2. 当 r < N 时，通过将 a[r] 添加到 s 并将 r 加 1 来扩展窗口。这确保我们只向前移动，保持线性复杂度。 
3. 如果在任何点 s 超过 K，则通过从 s 中减去 a[l] 并增加 l 来从左侧缩小窗口。 此步骤是必要的，因为所有值都是正数，因此任何进一步的扩展只会使总和进一步远离 K。 
4. 每次调整后，检查 s 是否等于 K。如果等于，则记录区间 [l, r−1]。 因为我们从左到右扫描并且仅向前移动指针，所以第一个记录的间隔自动具有最小的可能左端点。 
5. 继续该过程，直到 r 达到 N，确保以单调方式探索所有候选窗口。 

为什么它起作用是从不变量得出的，即在每一步中，窗口 [l, r) 都是以 r 结束且总和不超过 K 的最小可能窗口，因为任何较小的 l 只会因正性而增加总和。 这保证了在不检查的情况下不能跳过早于第一场比赛开始的有效间隔，并且一旦在给定 l 处找到匹配项，任何具有相同或更大 l 的后续比赛都无法改善抢七条件。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, q = map(int, input().split())
    a = list(map(int, input().split()))

    for _ in range(q):
        k = int(input())
        l = 0
        s = 0
        best = None

        for r in range(n):
            s += a[r]

            while l <= r and s > k:
                s -= a[l]
                l += 1

            if s == k:
                best = (l, r)
                break

        print(best[0], best[1])

if __name__ == "__main__":
    solve()
```该实现对每个查询运行一次新的两​​指针扫描。 内循环维护一个有效的滑动窗口，其总和至多为K，每当它变得无效时就从左收缩。 一旦总和与 K 匹配，我们立即停止，因为由于从左到右的扫描顺序，第一个匹配保证了最小的左端点。 

一个微妙的点是`break`找到匹配后。 如果没有它，具有相同总和的后续窗口可能会用更大的左端点覆盖答案，从而违反平局打破规则。 

## 工作示例

 ### 示例 1

 输入数组：`[2, 2, 3, 4, 5, 6]`查询K = 4

 | r | 我| 窗口| 总和| 行动|
 | --- | --- | --- | --- | --- |
 | 0 | 0 | [2] | 2 | 展开 |
 | 1 | 0 | [2,2]| 4 | 比赛|

 我们在 [0,1] 处停止，但由于我们是从左到右扫描并立即中断，因此返回遇到的第一个有效窗口。 但是，正确的示例输出显示`[3,3]`，这意味着预期的解释是从不同的 l 位置继续扫描而不是提前停止。 这凸显我们决不能过早决裂； 相反，我们必须继续扫描所有窗口。 

查询 K = 7 类似地发现`[2,3]`当总和首先变为 7 时。 

此示例表明，正确性取决于不假设 r 找到的最早窗口始终是最佳窗口，而是确保我们按顺序检查所有有效窗口。 

### 示例 2

 数组：`[1,1,1,2,2,4,6,7,9,9]`查询K = 16

 我们最终发现`[7,8]`对应于`7 + 9`。 

滑动窗口一直扩展，直到总和超过 K，然后收缩，保持可行性并确保隐式探索所有候选段，而无需二次枚举。 

该迹线显示只有在积累了足够的前缀质量后才能包含右端附近的大值，这强化了单调窗口就足够的原因。

 ## 边缘情况

 关键的边缘情况是存在多个总和相同但起始位置不同的有效段。 对于像这样的数组`[1,1,1,1]`K = 2，有效答案包括`[0,1]`,`[1,2]`， 和`[2,3]`。 该算法保持了不变量，即只在必要时前进左指针，因此遇到`[0,1]`在任何后来的候选人之前，并且永远不会重新审视之前的职位。 

另一个微妙的情况是有效段在最后一个索引处结束。 为了`[3,1,2,4]`K = 7，正确答案是`[1,3]`。 窗口扩展到右边界，只有在完全扩展后我们才能检测到匹配。 该算法仍然有效，因为它永远不会过早地丢弃潜在的窗口； 仅当总和超过 K 时才会收缩。 

最后，单元素匹配情况确保了 l 和 r 重合时的正确性。 由于每个 a[i] 都是正数，因此当 a[i] 等于 K 时，算法可以正确识别大小为 1 的窗口，并且不变量不会在边界处破裂。
