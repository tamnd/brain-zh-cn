 ## 解决方案
 ## 问题理解

 我们被赋予了一个$n \times n$网格开始为空，两个玩家交替将代币放入列中。 每次移动都会选择一列，令牌会落到该列中最低的可用单元格，就像四子棋中的重力一样。 

不同之处在于，我们没有被要求从头开始模拟游戏，而是在一系列已经玩过的动作之后重建状态，并确定某人是否已经形成了一条长度的获胜线$k$。 获胜线是任意直线段$k$相同颜色的标记在任一方向水平、垂直或对角对齐。 

输入给出网格大小$n$, 移动次数$m$，和目标长度$k$，后面是每个动作的列选择。 玩家 1 (Ayumi) 在奇数步中下注，玩家 2 (Bunji) 在偶数步中下注。 输出必须标识玩家完成的第一步移动索引$k$-line，或报告没有出现这样的行。 

约束条件允许$n$最多 300 个并且$m$最多$n^2$，因此每次移动后对棋盘进行全面扫描是有限的，但如果仔细完成，则是可行的。 通过完整网格扫描对每次移动进行简单的重新计算会导致粗略的结果$O(m \cdot n^2)$，在最坏的情况下速度太慢$n=300$,$m=90000$。 

一些边缘情况对于正确性很重要。 单个动作可能会同时创建多个获胜方向，并且任何方向达到长度的最早的动作$k$是答案，而不是最终状态。 此外，由于堆叠，胜利可能是垂直的，如果只检查水平线，这是最常见的错误。 

一个小的说明性案例：$n=3, k=3$, 移动$1,1,1$。 所有令牌都堆叠在第 1 列中，玩家 1 在第 3 步中垂直获胜。任何仅检查行的解决方案都会错误地输出“无获胜者”。 

另一个微妙的情况是从单个位置同时创建多条线，特别是在新单元格处相交的对角线。 我们必须确保从每个新放置的令牌中检查所有四个方向。 

## 方法

 蛮力方法模拟丢弃过程并保持完整的网格。 每次移动后，我们都会扫描整个棋盘并检查所有可能的起点$k$- 四个方向的长度线。 每次扫描都是$O(n^2)$，我们就这么做了$m$次，生产$O(mn^2)$运营。 和$n=300$，这可以达到大约$9 \cdot 10^4 \times 9 \cdot 10^4$，太大了。 

关键的观察结果是，只有最后放置的令牌才能创建新的获胜线。 在前面的步骤中已经检查了所有其他单元格，并且除了沿着最新的列位置之外，没有早期的配置更改。 这允许我们将检查限制为通过最后移动位置的线。 

对于每个放置的标记，我们只需要向四个方向扩展并计算连续的同色标记。 每个方向都是独立的，我们将单元格两侧的计数相加。 这减少了每次移动$O(n)$，因为每次定向扫描最多可以延伸$n$步骤。 

蛮力之所以有效，是因为它在每次移动后将网格视为静态，但会浪费时间重新检查未更改的区域。 优化的方法将问题减少到围绕新令牌的本地更新，将全局重新计算转变为增量验证。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 每次移动后进行强力全面扫描 |$O(mn^2)$|$O(n^2)$| 太慢了 |
 | 增量定向扩张|$O(mn)$|$O(n^2)$| 已接受 |

 ## 算法演练

 我们维护一个网格并一一模拟移动，将令牌放置在具有重力的列中。 

1. 初始化一个$n \times n$带有空单元格的网格，每列都有一个高度指针，用于跟踪从底部开始的下一个可用行。 这避免了每次放置令牌时向下扫描。 
2. 按顺序处理每个移动，通过读取所选列的当前高度来确定令牌落在的行。 根据移动奇偶性放置玩家 1 或玩家 2 的标记。 
3. 放置代币后，将此单元格视为唯一可以新创建获胜线的位置。 任何存在的获胜行都必须包含此单元格。 
4. 对于该单元格，沿四个方向扫描：水平、垂直、对角线 () 和反对角线 (/)。 对于每个方向，在正方向和负方向上向外扩展，同时单元格包含相同玩家的标记。 
5. 计算该行中连续标记的总数，包括当前单元格。 如果在任何时候计数达到或超过$k$，记录当前的走法索引和当前玩家为获胜者，然后停止处理进一步的走法。 
6. 如果没有任何一步棋产生获胜线，则输出没有获胜者。 

这样做的原因是，只有通过最近放置的令牌扩展现有的对齐段才能出现新的获胜配置。 在此移动之前，任何不涉及新令牌的行都已经存在，因此它会更早被检测到。 这个不变量确保仅检查最新的单元就足以保证正确性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def check(grid, n, r, c, player, k):
    dirs = [(0, 1), (1, 0), (1, 1), (1, -1)]
    
    for dr, dc in dirs:
        cnt = 1
        
        rr, cc = r + dr, c + dc
        while 0 <= rr < n and 0 <= cc < n and grid[rr][cc] == player:
            cnt += 1
            rr += dr
            cc += dc
        
        rr, cc = r - dr, c - dc
        while 0 <= rr < n and 0 <= cc < n and grid[rr][cc] == player:
            cnt += 1
            rr -= dr
            cc -= dc
        
        if cnt >= k:
            return True
    
    return False

def solve():
    n, m, k = map(int, input().split())
    cols = list(map(int, input().split()))
    
    grid = [[0] * n for _ in range(n)]
    height = [0] * n
    
    for i in range(m):
        c = cols[i] - 1
        r = height[c]
        height[c] += 1
        
        player = 1 if i % 2 == 0 else 2
        grid[r][c] = player
        
        if check(grid, n, r, c, player, k):
            if player == 1:
                print("Ayumi", i + 1)
            else:
                print("Bunji", i + 1)
            return
    
    print("No winner")

if __name__ == "__main__":
    solve()
```网格以行优先形式存储，其中第 0 行对应于板的底部，这简化了重力处理，因为每列都使用高度计数器向上填充。 这避免了重复搜索空槽。 

这`check`函数是核心正确性组件。 它只从新放置的单元格开始检查四个方向，并对称地向外扩展。 关键细节是分别检查向前和向后方向，确保完全计数穿过新标记的线。 

移动循环使用移动索引的奇偶性来交替玩家，这直接来自问题定义。

 ## 边缘情况

 一种边缘情况是恰好在网格边界处获胜。 例如，在一个$k=3$在顶行结束的行中，扩展仍然有效，因为检查仅在索引超出范围时停止，而不是在到达网格边缘时停止。 

另一种情况是在同一移动中形成的重叠线，例如与水平线交叉的对角线。 该算法可以正确处理此问题，因为它独立评估所有四个方向，并在第一次成功时立即返回，无论哪个方向触发了它。 

最后一种情况是，尽管棋盘上其他地方存在一条线，但最后一步不是获胜线的一部分。 在正确的玩法下不会发生这种情况，因为早期的动作已经触发了检测； 然而，该算法仍然表现安全，因为它从不假设最终状态的排他性，并且总是直接检查当前的移动。
