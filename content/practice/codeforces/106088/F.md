 ## 解决方案
 ## 问题理解

 我们得到了一系列级别，严格按照从左到右的顺序进行处理。 每个级别都有一定数量的物品，在尝试之前必须提供这些物品。 如果玩家拥有足够的物品，他们就可以进入关卡，但在通关后，他们会立即永久失去一件物品。 

输入由一个数组组成，其中每个位置描述启动该级别所需的最小项目数。 除此之外，我们还收到多个查询。 每个查询都描述一个已保存的游戏状态：玩家当前处于某个级别索引$l$，并且他们目前拥有$x$项目。 从这个状态开始，玩家再次一层一层地前进，总是检查自己是否能够满足当前级别的要求，然后在每次成功完成后失去一件物品。 任务是计算在进程停止之前可以完成多少个连续级别。 

这些约束将解决方案推向线性或近线性预处理，并具有非常快速的每个查询处理。 两个都$n$和$q$可以达到$5 \cdot 10^5$，因此任何逐步模拟每个查询的方法都会立即变得太慢。 甚至$O(nq)$或者$O(\log n)$模拟中的每一步都是不可接受的，除非进行大量摊销。 

资源单调减少产生了一个微妙的问题。 每升一级后，可用物品数量就会减少一个，而要求则任意变化。 这种组合阻止了简单的前缀可行性检查，因为约束在遍历过程中动态变化。 

当人们认为独立检查每个级别的可行性就足够时，就会出现天真的陷阱。 例如，假设某个关卡需要 10 个物品，而玩家最初有 10 个物品，但在通过较早的关卡后，物品数量比预期更快地低于要求。 可行性取决于之前通关的关卡数量，而不仅仅是起始状态。 

另一个不正确的直觉是尝试贪婪跳过：预先计算第一层，其中$a_i > x$。 这失败了，因为$x$在各个级别之间不是恒定的； 随着我们的进步，它会减少。 

## 方法

 蛮力的想法很简单。 对于每个查询，从级别开始$l$和$x$项目并逐步模拟。 处于水平$i$，检查当前项目是否至少$a_i$。 如果没有，就停下来。 否则减少项目并继续。 这正确地模拟了该过程，但在最坏的情况下，每个查询可能会遍历数组的几乎整个后缀，从而导致$O(nq)$的行为，远远超出了界限。 

关键的观察结果是，级别之间的唯一相互作用是资源的确定性线性减少。 如果我们在级别之前定义当前项目数$i$作为$x - (i - l)$，那么通过级别的条件$i$变成：$$x - (i - l) \ge a_i$$重新排列给出：$$x \ge a_i + (i - l)$$这将每个级别转换为固定阈值$x$，但移动了其相对于查询开始的位置。 

我们可以将其重写为：$$x + l \ge a_i + i$$这是关键的简化：通过级别的条件仅取决于值$a_i + i$，并且查询贡献了一个恒定的转变$x + l$。 现在问题变成：从位置开始$l$，在保持不变的情况下我们可以向右走多远$$a_i + i \le x + l$$现在一切都变得静止了。 每个位置都有一个预先计算的值$b_i = a_i + i$。 对于查询，我们从索引开始$l$并需要后缀的最长前缀，其中所有$b_i \le x + l$。 这是一个经典的范围最大查询问题：我们想知道是否最大$b$段上的值在阈值内。 使用线段树或稀疏表，我们可以重复二分搜索最远可达位置。 

我们对端点进行二分搜索$r$对于每个查询。 对于候选人$r$，我们检查是否$\max(b_l, \dots, b_r) \le x + l$。 如果属实，我们可以进一步扩展； 否则我们就会退缩。 这产生$O(\log n)$每个查询$O(n)$预处理。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O(nq)$|$O(1)$| 太慢了|
 | 最优（RMQ + 二分查找）|$O((n + q)\log n)$|$O(n \log n)$| 已接受 |

 ## 算法演练

 1.将数组转化为静态阈值数组$b_i = a_i + i$。 这在单个值中捕获了基于位置的物品损失的要求和影响。 
2. 构建范围最大查询结构$b$。 这使我们能够通过检查其最大值是否超过限制来快速测试整个段是否可行。 
3. 对于每个查询$(l, x)$，计算有效阈值$T = x + l$。 这是最大允许值$b_i$对于我们尝试的任何级别。 
4. 对最远可达索引执行二分查找$r \in [l, n]$。 Each check verifies whether$\max(b_l, \dots, b_r) \le T$。 如果为 true，则扩大搜索范围； 否则减少它。 
5. 转换最终端点$r$通过返回得到答案$r - l + 1$。 

二分搜索之所以有效，是因为可行性是单调的$r$。 如果一个段$[l, r]$有效，任何较小的段也有效，因为删除约束不会引入违规。 

### 为什么它有效

 转变$b_i = a_i + i$将资源的线性消耗吸收到固定的每级阈值中。 原始过程依赖于递减的状态变量，但在重新排列后，每个级别独立地施加相对于查询的静态约束。 段的可行性等同于该段中的所有约束均由单个常量边界单独满足$T$。 由于段有效性仅取决于段内的最大值，因此范围最大查询正确地表征了可行性，并且二分搜索利用了扩展的单调性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class SegTree:
    def __init__(self, arr):
        n = len(arr)
        self.n = 1
        while self.n < n:
            self.n *= 2
        self.seg = [-10**18] * (2 * self.n)
        for i in range(n):
            self.seg[self.n + i] = arr[i]
        for i in range(self.n - 1, 0, -1):
            self.seg[i] = max(self.seg[2 * i], self.seg[2 * i + 1])

    def query(self, l, r):
        l += self.n
        r += self.n
        res = -10**18
        while l <= r:
            if l % 2 == 1:
                res = max(res, self.seg[l])
                l += 1
            if r % 2 == 0:
                res = max(res, self.seg[r])
                r -= 1
            l //= 2
            r //= 2
        return res

n = int(input())
a = list(map(int, input().split()))

b = [a[i] + i + 1 for i in range(n)]

st = SegTree(b)

q = int(input())
out = []

for _ in range(q):
    l, x = map(int, input().split())
    l -= 1
    T = x + (l + 1)

    lo, hi = l, n - 1
    ans = l - 1

    while lo <= hi:
        mid = (lo + hi) // 2
        if st.query(l, mid) <= T:
            ans = mid
            lo = mid + 1
        else:
            hi = mid - 1

    out.append(str(ans - l + 1))

print("\n".join(out))
```线段树存储变换后的值$b_i$，在任何时间间隔内实现快速最大查询。 每个查询将其起始状态转换为单个阈值$T$，然后使用二分查找尽可能扩展可达前缀。 

一个常见的微妙之处是索引的一致性。 从概念上讲，该转换使用基于 1 的索引，因此$b_i = a_i + i$必须与阈值中使用的相同索引匹配$x + l$。 基于 0 的索引和基于 1 的索引之间的任何不匹配都会通过改变可行性边界来破坏正确性。

 ## 边缘情况

 临界边缘情况是当段中的第一级已经违反阈值时。 例如：```
a = [100, 1, 1], l = 1, x = 0
```这里$b_1$已经远远高于$T$，所以答案为零。 二分查找正确地识别出即使是最小的段也会失败，因为范围最大值立即超过阈值。 

当所有级别都可行时，会出现另一种边缘情况。 在这种情况下，二分搜索扩展到$n$没有提前终止，并且线段树总是返回低于或等于的值$T$，确保正确报告完整遍历。 

最后一个微妙的情况是索引一致性。 自从$b_i$在其定义中使用基于 1 的索引，混合基于 0 的查询阈值可以将可行性精确地转变一级，从而产生仅出现在边界查询上的差一错误，其中$x$只够再增加一个级别。
