 ## 解决方案
 ## 问题理解

 我们得到了一组歌曲，其中每首歌曲都有两个属性：以字节为单位的存储成本（从其标题的长度得出）和奖励值。 Amber 拥有固定容量（以字节为单位）的数字黑胶唱片。 她可以多次将任何歌曲放入唱片中，每个副本都会以字节为单位贡献其成本，并将其价值再次添加到总奖励中。 

任务是选择要存储每首歌曲的多少个副本，以使总使用字节数不超过容量，同时最大化总累积值。 

这是一个经典的无界选择问题，但“权重”并不是直接给出的。 相反，每个项目的权重隐式地是其字符串的长度，并且该长度可以很大。 歌曲中所有字符串长度的总和是有界的，因此计算这些权重总体上在线性时间内是可行的。 

关键的困难在于，对高达 2·10^5 的容量进行朴素的动态规划是边界性的，但仍然可行，而真正的约束是认识到该问题干净地简化为容量相对较小的无界背包。 

这些约束意味着任何比 O(nm) 更差的解决方案都会失败。 由于 n 和 m 都可达 2·10^5，因此二次方法是不可能的。 即使在最坏的情况下 O(nm) 也太大了，但这里每个项目都被处理一次，因此只有仔细实现 O(nm) 背包才可以接受，并且我们必须确保不会出现隐藏的额外因素。 

当歌曲的长度非常小时（例如 1 个字节）时，就会出现微妙的边缘情况。 在这种情况下，最佳解决方案可能涉及最多 m 次。 仅基于值与长度比的贪婪方法并不安全，因为混合项目可能会主导局部最优选择。 

## 方法

 强力解释将尝试每首歌曲的所有可能计数，本质上是探索适合容量的所有重复组合。 如果一首歌曲最多可以重复 m 次，并且有 n 首歌曲，这会导致指数搜索空间，因为每个选择都会分支为许多可能的计数。 即使限制每首歌的重复次数，在最坏的解释中仍然会导致与 m^n 成比例的状态爆炸，这是完全不可行的。 

更结构化的暴力破解会将其视为标准的背包动态规划问题。 我们将 dp[x] 定义为使用恰好 x 个字节可实现的最大值。 对于每首歌曲，我们都会迭代所有容量并尝试多次添加。 然而，如果实施不当，这种幼稚的公式会变成每个项目的 O(nm)，因为每首歌曲都会在重复时重复更新嵌套循环中的所有状态。 

关键的观察是每首歌曲都是独立的并且可以无限次使用，因此我们正在处理一个容量为 m 的无界背包。 每个项目的成本是它的字符串长度，并且它的值是给定的。 这允许标准的 1D DP，我们可以从小到大迭代容量并使用每首歌曲更新一次。 这是有效的，因为一旦我们处理了一首歌曲，我们就可以在同一个 DP 通道中重复使用它以允许多个副本。 

这将问题分解为一个干净的优化结构，其中每个项目在线性时间内贡献所有容量的转换。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力枚举| 指数| O(1) | O(1) | 太慢了 |
 | 天真的重复背包每件物品| 每个项目 O(nm) | O(米) | 太慢了 |
 | 无界背包DP| O(纳米) | O(米) | 已接受 |

 ## 算法演练

 我们将每首歌曲视为一个项目，其权重等于其标题长度，价值等于其给定分数。

1. 计算每首歌曲标题的长度。 这定义了使用该歌曲一次的成本。 此步骤是必要的，因为问题将权重隐藏在字符串内，而不是明确给出它们。 
2. 创建一个大小为 m + 1 的 DP 数组 dp，用零初始化。 dp[x] 表示使用 x 字节或更少字节可实现的最佳值。 我们将其保持为一维，因为我们只需要先前和当前的状态。 
3. 对于每首歌曲，按递增顺序从其权重到 m 迭代容量。 在每个容量 c 处，尝试使用 dp[c - 权重] + 值来改进 dp[c]。 这种转变表示在已经以较小的容量达到有效状态之后再获取该歌曲的一份副本。 
4. 如果新值更大，则更新 dp[c]。 这确保我们始终为每种容量保留最知名的组合。 
5.处理完所有歌曲后，答案是0到m所有容量中的最大值。 这是必需的，因为我们不会被迫完全填满容量。 

我们以递增顺序迭代容量的原因至关重要。 它允许在一次迭代中多次重复使用同一首歌曲。 如果我们向后迭代，我们将限制每首歌曲最多使用一次，将问题变成 0/1 背包，这在这里是不正确的。 

### 为什么它有效

 DP 保持不变式，即在处理前 k 首歌曲后，dp[c] 包含在容量 c 内使用这 k 首歌曲的无限副本可实现的最大值。 每次更新都会通过当前歌曲的一份额外副本正确扩展有效配置。 因为转换仅取决于 dp[c - 权重]，所以所有先前计算的最优状态仍然有效，并且未来的状态不会使过去的最优子结构无效。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    
    dp = [0] * (m + 1)
    
    for _ in range(n):
        parts = input().rstrip().split()
        s = parts[0]
        v = int(parts[1])
        w = len(s)
        
        if w > m:
            continue
        
        for c in range(w, m + 1):
            val = dp[c - w] + v
            if val > dp[c]:
                dp[c] = val
    
    print(max(dp))

if __name__ == "__main__":
    solve()
```该解决方案使用单个 DP 阵列来存储最佳可实现值。 每首歌曲都只处理一次，并且对于每首歌曲，我们都会向前扫容量以允许重复使用。 提前跳过重量大于 m 的歌曲可以避免不必要的工作。 

一个常见的实施陷阱是在错误的方向上迭代能力。 如果我们从 m 向下迭代到 w，每首歌曲只会使用一次，这打破了无界要求。 另一个微妙之处是在最后取所有 dp 状态的最大值，因为允许使用部分容量。

 ## 边缘情况

 一个关键的边缘情况是当歌曲的长度为 1 时。在这种情况下，每个 DP 状态都可以不断改进，最终的答案变为该值的 m 倍。 前向迭代确保每个状态正确地累积重复使用，而不需要显式计数。 

另一个极端情况是所有歌曲都超出容量。 由于没有发生任何转换，DP 保持全零并且输出正确地为零。 

第三种情况是多首歌曲具有相同的长度但不同的值。 DP 自然更喜欢更高价值的转换，并且重复覆盖可确保只有最佳组合能够跨容量生存。
