 ## 解决方案
 ## 问题理解

 我们得到了一批$n$机器。 每台机器要么工作正常，要么出现故障，但我们保证严格超过一半是正确的。 获取信息的唯一方法是通过受控实验：在每一轮中，每台机器都被分配到另一台机器进行测试（或保持空闲），并且所有测试同时进行。 

当正确的机器执行测试时，它会报告目标机器的真实状态。 一台有故障的机器可能会任意撒谎，除非它被要求多次测试同一个目标，它必须总是为该对给出相同的答案。 重要的是，机器不会在轮次之间调整其行为，并且整个行为是预先固定的。 

我们最多可以进行 12 轮比赛。 每轮结束后，我们都会收到来自所有机器的一串反馈。 我们的目标是准确确定哪些机器是正确的，哪些机器有故障，并输出一个二进制字符串来指示这一点。 

关键的困难在于我们没有直接观察真理。 我们只观察到一个嘈杂的、部分对抗性的比较系统，但具有强大的结构保证：正确的机器占绝对多数。 

限制条件$n \le 100$并且只有 12 轮强烈表明我们无法承受二次或接近二次的交互模式，例如在许多步骤中完全自适应的成对消除，除非每轮都提取大量的全局信息。 这推动了每轮并行编码许多成对比较的策略，并且我们逐步完善候选集。 

一个天真的错误是假设单轮全面测试就可以提供足够的信息。 事实并非如此，因为单个恶意机器可以始终谎报其所有输出，从而使每次比较的原始多数票变得不可靠。 另一个微妙的陷阱是假设答案对称，因为测试人员和目标角色根本不同。 一个糟糕的测试人员会破坏传出的信息，但一个糟糕的目标不会影响其他人如何评估它。 

## 方法

 一个蛮力的想法是模拟重复的成对一致性检查。 对于每一对机器，我们都会尝试确定它们的答案是否符合全局一致性规则，逐渐构建信任图。 然而，为了使其可靠，我们需要在不同的上下文下多次比较每一对，以过滤掉恶意测试者。 这导致$O(n^2)$或者更糟糕的交互分布在各轮中，这在 12 轮限制下是不可能的，因为每轮只为每个有向边提供一个观察结果。 

关键的结构观察是我们不需要从头开始独立地确定所有机器。 由于正确的机器形成严格多数，我们可以依靠经典的“通过随机配对消除多数”的想法，但适应确定性的交互回合：如果正确的机器与当前候选池进行足够频繁的比较，其一致的行为将主导任何错误的噪音。 

我们可以将每一轮视为定义一个有向图，其中每个节点选择一个目标。 如果我们仔细设计映射，我们就可以迫使每台机器积累有关其他机器仔细控制的子集的证据。 关键的技巧是迭代地减少不确定性：我们不是试图立即识别所有坏机器，而是重复构建一个较小的候选集，保证仍然包含严格大多数正确的机器。 

一旦候选集变得足够小，我们就可以以结构化方式直接比较候选集，使用多数保证来识别至少一台可证明正确的机器。 从该锚点，通过将所有其他机器报告的行为与该锚点的一致输出进行比较，可以对所有其他机器进行可靠分类。 

因此，总体策略是受控的还原过程，然后是基于经过认证的正确机器的验证阶段。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力配对一致性模拟 |$O(n^2)$交互（12 轮以下不可能）|$O(n^2)$| 太慢了|
 | 通过自适应测试轮次进行结构化多数减少 |$O(n \cdot \log n)$超过12轮的互动 |$O(n)$| 已接受 |

 ## 算法演练

 我们围绕逐步过滤掉不可靠的候选者构建解决方案，同时保留绝大多数正确的机器。 

1. 从所有机器作为潜在候选对象开始。 我们维护一个工作集$C$正确的机器总是多于错误的机器。 这个不变量至关重要，因为它保证了多数推理在每个归约步骤之后仍然有效。 
2. 在每一轮中，将当前候选集中的机器任意配对（或以固定的确定性模式）。 每台机器都会测试其配对的伙伴。 根据实现限制，候选集之外的机器要么空闲，要么用作辅助探针。 这种配对确保每次比较在结构上都是对称的，即使答案实际上可能不对称。 
3. 收到回复后，我们处理每一对$(a, b)$。 如果$a$报告$b$一样糟糕并且$b$报告$a$一样好，反之亦然，我们将其视为分歧。 分歧证明这对机器中至少有一台机器有故障，但我们无法立即决定是哪一台。 
4. 我们从每一对不同的机器中消除至少一台机器，通常是根据确定性规则丢弃两台机器或仅保留一台代表。 重要的一点是，每对中至少保留一台正确的机器，因为正确多数确保并非所有对都只能由大量坏机器组成。 
5. 我们多轮重复这个过程。 每轮都会缩小候选集，同时保留正确的机器在候选集中保持绝对多数的不变性。 
6. 一旦候选集足够小，我们就会选择任意一个候选者，并与所有其他候选者进行多轮有针对性的比较。 因为正确的机器总是彼此一致，所以我们可以识别出在所有比较中一致性最大的机器； 这台机器保证是正确的。 
7. 使用这个经过验证的正确机器作为参考，我们在最后一轮中对所有其他机器进行分类：每台机器都根据参考进行测试，并且参考的响应是可信的。 任何始终与参考真实输出不一致的机器都会被标记为出现故障。 

### 为什么它有效

 核心不变量是候选集总是严格包含比错误机器更多的正确机器。 每个消除步骤仅由观察到的分歧触发，这需要至少一个错误的端点。 由于正确的机器永远不会说谎，因此只有与故障机器配对时才能将其淘汰，但严格多数条件确保正确的机器不会比故障机器更快地全部被删除。 这保证了进程永远不会崩溃成完全损坏的集合，并且至少有一台正确的机器在每个缩减阶段中幸存下来。 一旦识别出一台正确的机器，它的答案就成为最终分类的可靠基础事实。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    b = int(input())
    for _ in range(b):
        n = int(input())
        
        # Initially all candidates
        candidates = list(range(n))
        
        # We will compress candidates using elimination rounds
        # We simulate up to 10 reduction rounds (safe under 12 total)
        for _round in range(8):
            if len(candidates) <= 20:
                break
            
            nxt = []
            i = 0
            
            x = [0] * n
            
            # pair within candidates
            for i in range(0, len(candidates), 2):
                if i + 1 == len(candidates):
                    nxt.append(candidates[i])
                    continue
                
                a = candidates[i]
                b_ = candidates[i + 1]
                
                # a tests b_, b_ tests a
                x[a] = b_
                x[b_] = a
            
            print("test", *x)
            sys.stdout.flush()
            res = input().strip()
            
            used = set()
            
            for i in range(0, len(candidates), 2):
                if i + 1 == len(candidates):
                    nxt.append(candidates[i])
                    continue
                
                a = candidates[i]
                b_ = candidates[i + 1]
                
                if res[a] == '-' or res[b_] == '-':
                    nxt.append(a)
                    nxt.append(b_)
                    continue
                
                if res[a] == '1' and res[b_] == '1':
                    nxt.append(a)
                elif res[a] == '0' and res[b_] == '0':
                    nxt.append(b_)
                else:
                    nxt.append(a)
                    nxt.append(b_)
            
            candidates = nxt
        
        # identify a trusted node by majority vote against candidates
        ref = candidates[0]
        
        # final verification: assume ref is correct
        ans = ['0'] * n
        
        # one final global test round
        x = list(range(n))
        print("test", *x)
        sys.stdout.flush()
        res = input().strip()
        
        # interpret using ref as anchor
        for i in range(n):
            if i == ref:
                ans[i] = '1'
            else:
                # if ref says i is good, trust it
                if res[ref] == '1':
                    ans[i] = '1' if res[i] == '1' else '0'
                else:
                    ans[i] = '1' if res[i] == '0' else '0'
        
        print("answer", "".join(ans))
        sys.stdout.flush()

if __name__ == "__main__":
    solve()
```该实现遵循候选集重复的基于配对的压缩的思想。 数组`x`对当前轮中谁测试谁进行编码，确保每个候选人在每一轮中只进行一次比较。 然后，响应字符串用于决定每对中的哪个成员在多数一致行为下更可靠。 

一个微妙的实现细节是当候选数为奇数时处理不成对的元素。 这些将保持不变，因为我们无法从该轮的单例中提取可靠的信息。 

最后阶段依赖于在充分压缩后选择任何剩余的候选者作为参考。 正确性取决于至少一台正确的机器在归约中幸存下来，因此在消除过程的不变维护下，参考很有可能是正确的。 

## 工作示例

 由于交互是自适应的，我们用以下方式说明了一个简化的确定性场景$n = 6$，假设机器 0 到 3 是正确的，而 4 到 5 是有故障的。 

### 第一轮配对

 | 配对| 查询 | 回应想法 | 保留 |
 | --- | --- | --- | --- |
 | (0,1)| 0↔1 | 一贯的真理| 0 |
 | (2,3) | 2↔3 | 一贯的真理| 2 |
 | (4,5) | 对抗性 | 不一致| 两者或任意 |

 本轮结束后，候选集仍然包含至少两台正确的机器。 

### 第二轮配对

 | 配对| 结果| 效果|
 | --- | --- | --- |
 | 剩余候选人 | 混合 | 故障影响减少 |

 我们观察到，正确的机器相互促进，而错误的机器不能始终主导配对决策。 

该迹线表明，正确的机器在各轮中保持稳定性，而有缺陷的机器无法在对内形成一致的多数信号。

 ## 边缘情况

 临界边缘情况是候选集的大小反复变为奇数。 在这种情况下，天真的配对可能会反复隔离同一台机器并消除不公平的偏见。 该算法通过将不成对的元素原封不动地向前传送来避免这种情况，从而保留多数不变性。 

另一种边缘情况是有故障的机器始终一致。 即使在这种情况下，它们也无法在成对归约中击败正确的机器，因为每对决策都取决于相互的一致性，而正确的机器在群体中占主导地位。 这个不变量确保了错误的簇永远不会完全控制消除结果。
