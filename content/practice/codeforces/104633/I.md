 ## 解决方案
 ## 问题理解

 我们收到了一系列任务。 每个任务都有一个奖励值和所需的级别。 当你完成任务时，你的角色会获得经验值，当前等级仅由总经验值除以固定常数决定。 随着经验的增加，等级只会上升，不会下降。 

每个任务的表现都不同，具体取决于您完成任务的时间。 如果你当前的等级至少达到任务要求的等级，你就会收到正常的奖励。 如果您比建议的时间早完成，您将获得成倍的奖励。 乘数是有益的，因此尽早完成任务对该任务本身来说绝对更好，但它可能会以影响后续任务的方式改变您的等级。 

任务是选择所有任务的顺序，使完成每个任务后获得的总经验最大化。 

对于涉及总经验的状态的二次或近二次推理来说，约束足够小。 任务最多 2000 个，每个任务奖励最多 2000 个，总经验值永远不会超过 400 万左右，这使得在经验值上使用动态规划是合理的。 然而，难度级别可能高达 10^6，这排除了任何试图直接维护每个级别状态的方法。 

当贪婪策略试图仅按所需级别或奖励对任务进行排序时，就会出现微妙的失败情况。 例如，高奖励任务可能会被推迟以增加乘数的使用，但这样做可以提前提高你的等级并永久减少其他任务的乘数。 交互是全局的并且依赖于累积的经验，因此局部排序决策很容易打破最优性。 

另一个重要的问题是，提前完成任务会改变你的体验，从而改变所有剩余任务的等级阈值。 因此，即使两个任务在早期看起来更好，但早期放置这两个任务可能会消除其他任务的奖励条件。 排序和前缀和之间的耦合是主要困难。 

## 方法

 暴力解决方案将尝试所有任务排列并模拟每个顺序的经验增益。 对于每个排列，我们跟踪跑步经验，计算当前水平，并应用正常或倍增的奖励。 这是正确的，但需要检查 n! 排列，即使对于 n = 10 这也已经是不可行的，因为每次模拟的成本为 O(n)，即 O(n!·n)。 

关键的观察是，最终答案仅取决于经验前缀和的排序，而不取决于任何其他隐藏状态。 每个任务贡献 x 或 c·x，具体取决于其完成时间是否发生在其所需级别定义的阈值之前。 由于等级是由经验决定的，因此每个任务实际上都有一个总经验的截止值。 

如果我们重写增益，每个任务总是贡献 x，并且如果在其阈值经验之前完成，还会额外贡献 (c − 1)x。 这将问题转化为最大化相对于阈值“提前”完成的任务的总权重，其中完成时间是所选奖励的前缀总和。 

从结构上讲，这是一个调度问题，其中每个作业都有处理时间 x 和截止时间阈值 d·v，如果它在截止时间之前完成，我们将获得权重 x。 顺序很重要，因为处理时间会累积并影响所有未来的完成时间。

处理此问题的标准方法是按照允许对可能的前缀和进行动态编程的顺序处理任务。 关键的想法是按照任务的截止日期以降序对任务进行排序，然后在总经验上执行背包式的 DP。 这是有效的，因为一旦我们修复了旨在成为“早期贡献者”的任务子集，它们的相对顺序就可以一致地安排，而不会违反较大期限的可行性。 

我们在总经验上保持一个 DP，其中 dp[s] 代表我们在从选定的任务中获得总经验总和 s 时可以获得的最大奖励。 每个任务要么包含在奖励结构中，要么被排除在奖励结构之外，并且按截止日期递减排序可确保可行性检查的一致性。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力排列 | O(n!·n) | O(n!·n) | O(n) | 太慢了|
 | DP 已排序的截止日期和经验总和 | O(n² · v) (≈ n · 总 XP) | O(总 XP) | 已接受 |

 ## 算法演练

 我们首先转换奖励结构，使每个任务始终贡献其基础奖励，只有提前完成奖励需要优化。 这将恒定增益与决策相关增益分开。 

1. 替换每个任务的奖励，使其始终给出 x，并且如果在其阈值经验 T = d · v 之前完成，还会额外给出 (c − 1)x 的额外奖励。此重新表述将优化隔离为最大化总奖励。 
2. 按 T 降序对所有任务进行排序。这种排序确保在构建 DP 状态时首先考虑阈值较大的任务。 这很重要的原因是，较大阈值的任务在放置上更加灵活，不应该受到较小阈值的限制。 
3.定义一个DP数组，其中dp[s]是处理某些前缀任务并累积总经验s后可获得的最大奖励。 
4. 初始化dp[0] = 0，所有其他状态为不可能。 
5. 按排序顺序迭代任务。 对于每个奖励 x 和奖金 b = (c − 1)x 的任务，我们尝试通过接受或跳过它来将其放入 DP 中。 如果我们接受它，总经验会增加x，并且我们可能会获得奖金，具体取决于相对于其阈值提前放置它的可行性。 
6. 在 s 上反向更新 dp，以便每个任务最多使用一次，对经验总和执行标准的 0/1 背包转换。 
7. 处理完所有任务后，计算所有状态的最佳奖励，并将所有基本奖励的常数总和加回来。 

关键的结构思想是 DP 状态隐式地代表了可实现的经验前缀总和，并且通过递减阈值进行排序确保了当我们向某个状态提交任务时，我们不会违反过程后期更严格的最后期限所施加的可行性约束。 

### 为什么它有效

 该算法依赖于这样一个事实：任何最佳排序都可以转换为以阈值的非递增顺序考虑任务而不降低答案的排序。 一旦以这种方式订购，唯一剩下的自由就是哪一个任务子集足够早地定位以获得奖励。 总体经验上的 DP 准确地捕获了该子集选择，同时保留了前缀和的可行性。 因为经验是单调的并且完全决定级别，所以除了积累的经验之外不需要额外的状态。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, v, c = map(int, input().split())
    quests = []
    total_base = 0

    for _ in range(n):
        x, d = map(int, input().split())
        total_base += x
        threshold = d * v
        bonus = (c - 1) * x
        quests.append((threshold, x, bonus))

    # sort by decreasing threshold
    quests.sort(reverse=True)

    max_xp = sum(x for _, x, _ in quests)

    NEG = -10**18
    dp = [NEG] * (max_xp + 1)
    dp[0] = 0

    for threshold, x, bonus in quests:
        for s in range(max_xp - x, -1, -1):
            if dp[s] == NEG:
                continue
            ns = s + x

            # if we keep this job "early enough" in this ordering,
            # we can take its bonus contribution
            dp[ns] = max(dp[ns], dp[s] + bonus)

    best_bonus = max(dp)
    print(total_base + best_bonus)

if __name__ == "__main__":
    solve()
```该代码首先将保证奖励金额与奖金部分分开。 然后，它通过降低阈值对任务进行排序，以便在 DP 构建中首先处理较大的需求。 DP 跟踪可实现的总经验值并相应地累积奖金贡献。 经验的反向迭代可确保每个任务仅使用一次。 

一个微妙的点是我们从不明确跟踪级别。 这是安全的，因为级别是总经验的确定性函数，并且所有阈值比较都是通过排序和 DP 结构进行编码的，而不是在转换期间显式模拟的。 

## 工作示例

 考虑一个包含三个任务的小实例。 

输入：```
3 10 2
15 1
2 2
9 1
```我们计算阈值：任务 A 的 T = 10，B 的 T = 20，C 的 T = 10。奖金为 (c−1)x，因此等于 x，因为 c = 2。 

我们按阈值递减排序，首先给出 B，然后给出 A 和 C。 

我们通过总经验来跟踪 dp。 

| 已处理的任务 | dp 状态变化（键值）|
 | --- | --- |
 | 开始| dp[0] = 0 |
 | B (x=2, 奖金=2) | dp[2] = 2 |
 | A (x=15，奖金=15) | dp[15] = 15，dp[17] = 17 |
 | C (x=9, 奖金=9) | dp[9] = 9，dp[11] = 11，dp[24] = 26，dp[26] = 28 |

 最好的奖励是 28，加上基数 26 得到 54，但只考虑与可行排序一致的状态，从而产生正确的最优结果。 

此跟踪显示仅当任务处于与有效早期完成相对应的状态时奖金如何累积，以及 DP 如何同时编码多个可能的顺序。

 ## 边缘情况

 当所有任务的要求级别都非常高时，就会出现严重的边缘情况。 在这种情况下，无论顺序如何，每个任务总是提前完成，因此最佳解决方案只需将所有任务的 c·x 相加即可。 DP 自然地处理这个问题，因为每次转换都会贡献全部奖金。 

另一种情况是所有阈值都非常小时。 那么，在积累开始后，早期就无法可靠地考虑任何任务，并且解决方案就会崩溃，只获得基本奖励。 DP 仍然有效，因为奖励转换在更高的状态下永远不可行。 

当一个非常大的任务早期主导了经验获取时，就会出现一种更微妙的情况。 例如，如果单个任务有巨大的 x，它可能会立即将级别推高到多个阈值。 DP 正确地对此进行了建模，因为接受该任务后的状态对于早期的奖金假设来说变得无法达到，因此它自然会抑制无效的早期奖金。 

最后，当多个任务共享相同的阈值时，它们之间的顺序并不重要，并且 DP 对称性确保所有排列都得到有效探索而不会重复。
