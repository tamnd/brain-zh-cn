 ## 问题理解

 我们得到一个数字列表，每个数字代表学生愿意在购买中接受多少糖果。 商店按一定的顺序处理学生，但我们可以自由选择让他们满意的顺序。 一旦我们确定了已经“承诺”或出售的糖果总量，学生只有在他们的要求至少达到该数量时才能得到满足。 我们的任务是通过选择最佳的服务顺序来最大限度地满足多少学生的需求。 

关键的结构是每个学生都有一个阈值，服务一个学生会累积消耗一定的“能力”。 我们正在有效地尝试将尽可能多的约束纳入不断增加的累积总和中，其中早期的选择会影响后来的选择是否可行。 

如果学生数量很大，比如多达 200000 人，那么我们反复尝试不同子集或重新排序的 O(n^2) 模拟将会太慢。 对于大约 10^5 或 2×10^5 元素，我们期望 O(n log n) 或 O(n) 解决方案。 这立即表明排序或贪婪选择是唯一可行的方向。 

当试图按原始顺序或降序处理学生时，就会出现一种微妙的失败情况。 如果我们首先考虑大的需求，我们可能会过快地增加累积总和，这会阻止我们适应本来可以更早满足的较小的需求。 

例如，假设要求是`[5, 1, 2]`。 如果我们处理`5`首先，累积和立即变大，并且根据解释，我们可能会跳过`1`和`2`或错误评估可行性。 正确答案是服务`1`， 然后`2`， 然后`5`，这使得所有三个都可以在受控的累积过程中得到满足。 

核心困难是认识到处理顺序决定了累积约束增长的速度，而最佳策略是尽可能延迟增长的策略。 

## 方法

 蛮力的想法是尝试对学生的每一种可能的排序，并模拟在该排序下可以满足多少人。 对于每个排列，我们维护一个运行总数并计算满足的阈值数量。 这是有效的，因为它直接对过程进行建模，但它具有阶乘复杂性。 有n个学生，有n个！ 排列，甚至评估一个排列的成本为 O(n)，使得该方法在超过非常小的 n 时不可行。 

关键的观察是，最大化满意学生数量的唯一方法是在尽可能长时间内保持累积总和尽可能小。 每当我们提前挑选一个要求较高的学生时，我们都会冒着不必要地增加累积总数并阻止未来录取的风险。 这表明我们应该始终首先优先考虑较小的需求，因为它们对运行总数的限制最小。 

一旦我们按升序对需求进行排序，我们就可以从最小到最大贪婪地处理它们，并保持运行总和。 每当当前要求至少为当前总和时，我们就可以将该学生包括在内并相应地增加总和。 如果没有，我们会跳过它们，因为包含它们只会进一步增加总和，而无助于满足更多约束。 

这将问题从排列的组合搜索转换为排序后的单个线性扫描。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力（所有排列）| O(n!·n) | O(n!·n) | O(n) | 太慢了|
 | 排序+贪心扫描| O(n log n) | O(n log n) | O(1) 或 O(n) | 已接受 |

 ## 算法演练

 我们首先按非降序对所有学生要求进行排序，以便我们始终首先考虑最小的约束。 这确保了在给定当前累积总和的情况下，每一步都有最大的可行机会。 

我们初始化一个运行变量`cur = 0`，代表已提交的糖果总量，以及一个计数器`ans = 0`，它跟踪有多少学生感到满意。 

然后我们迭代排序后的列表。 对于每个值`x`，我们检查给定当前累计总和是否可以满足该学生的要求。 

如果`x >= cur`，我们包括这个学生，增量`ans`一位，并更新`cur += x`。 此更新之所以安全，是因为我们仅向要求足够大以适应当前状态的学生承诺。 

如果`x < cur`，我们跳过这个学生，因为在这个贪婪的框架中包括他们是没有意义的。 由于所有未来值都大于或等于（由于排序），因此跳过可确保我们不会在不可行的转换上浪费机会。 

最后我们输出`ans`。 

### 为什么它有效

 在任何时候，算法都保持不变式：`cur`是通过选择已处理元素的某些子集可实现的最小可能累积和。 通过按排序顺序处理，我们确保永远不会在较大的需求之后遇到较小的需求。 这种单调性保证了一旦一个值不能满足当前的总和，后面的结构将无法恢复它。 采用每个可行的最小元素的贪婪选择保留了未来选择的最大可扩展性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

n = int(input())
a = list(map(int, input().split()))

a.sort()

cur = 0
ans = 0

for x in a:
    if x >= cur:
        ans += 1
        cur += x

print(ans)
```该解决方案首先读取需求列表并对它进行排序，以便首先处理较弱的约束。 变量`cur`跟踪累积的承诺，以及`ans`计算有多少学生被成功服务。 每次迭代都会决定是否可以在不违反累积约束的情况下包含当前学生。 条件`x >= cur`是关键的可行性检查，确保我们仅在解决方案仍然有效时才对其进行扩展。 

一个常见的错误是扭转不平等或更新`cur`在检查可行性之前错误。 顺序很重要：我们必须先检查，然后更新。 另一个微妙的问题是忘记跳过学生不会重置或修改`cur`。

 ## 边缘情况

 一种重要的边缘情况是所有值都相同。 考虑输入`[3, 3, 3]`。 排序后保持不变。 该算法进行如下：`cur = 0`， 第一的`3 >= 0`所以接受它，`cur = 3`。 下一个`3 >= 3`所以接受它，`cur = 6`。 下一个`3 >= 6`是假的，所以我们跳过它。 输出是`2`。 这表明，一旦累积总和超过重复值，即使是统一的输入也可能会破坏完全包含。 

另一种边缘情况是一个非常小的值后面跟着一个大的值，例如`[1, 100, 100]`。 该算法需要`1`， 然后`100`，那么最后的`100`变得不可行，因为`cur`已经变得太大了。 该痕迹证实了早期的小夹杂物可以阻止后来相同的大夹杂物，这正是为什么需要排序和贪婪接受来控制生长的原因。
