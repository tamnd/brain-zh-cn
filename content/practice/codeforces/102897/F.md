 ## 解决方案
 ## 问题理解

 我们得到了一系列硬币类型。 每种类型都有固定面额和限量供应。 从这些硬币中，我们想知道使用任何硬币组合可以形成多少个不同的总和，但只考虑从 1 到 m 的总和。 

每个硬币类型的行为就像一个有界资源：对于第 i 个类型，我们最多可以使用它的值 ai 次。 所有类型的硬币的任何子集都会对总和做出贡献，我们感兴趣的是 1 到 m 范围内的总和至少可以实现一次。 

输出是区间 [1, m] 中有多少个整数可以表示为有效和的计数。 

这些约束极大地影响了解决方案。 硬币类型最多有 100 种，目标范围 m 最多为 100000。计数 bi 和值 ai 也可能很大，最多为 100000。枚举所有硬币组合的简单方法是不可能的，因为组合的数量随着 n 和 bi 呈指数增长。 

一个更微妙的问题是，将每个硬币独立地视为无界是不正确的。 例如，如果价值 5 的硬币只出现一次，我们就不能使用它两次，即使天真的无界背包转换允许这样做。 

打破粗心解决方案的一个简单边缘情况是所有硬币都相同但有限：

 输入：

 n = 1，m = 10

 a = [2]，b = [3]

 正确的输出是 3，因为我们可以组成 2、4、6。天真的无界背包会错误地包含 8 和 10。 

当值为零时会发生另一种失败情况。 价值为 0 且数量为正的硬币不会影响总和，但如果处理不当，可能会破坏幼稚的转换。 

## 方法

 暴力解释是考虑每种硬币类型并尝试从 0 到 bi 的每种可能的使用计数，递归地组合所有可能性。 这有效地探索了一个搜索树，其中每个级别对应于一种硬币类型，并且每个分支选择要获取多少该类型的硬币。 

对于每种硬币类型，分支因子为 bi + 1，因此状态总数是所有 i 的 (bi + 1) 的乘积。 即使对于像 10 这样的中等 bi，这也会变得天文数字。 每个状态都需要计算总和并标记可达值，这使得这种方法完全不可行。 

关键的观察结果是，这是一个经典的有界背包可达性问题。 我们不去枚举我们拿走了多少枚硬币，而只关心是否可以拿到一笔钱。 这使我们能够将每个有界项转换为可以有效处理的结构。 

直接0/1背包会把每个币分裂成bi个单独的物品，但bi最多可以达到100000，所以这种扩展也是不可能的。 关键的优化是计数的二进制分解。 任何整数 bi 都可以写成 2 的幂之和。 对于每种硬币类型，我们将其分成 O(log bi) 项，每个项代表一组总价值为 (2^k * ai) 的硬币。 这将问题转化为最多 100 * log(100000) 项（大约 1700 项）的 0/1 背包。 

在此转换之后，我们运行一个标准布尔背包，其中 dp[x] 指示总和 x 是否可达。 每个分组的项目使用反向转换应用一次。 

这将问题减少到可管理的大小并保持正确性，因为任何高达 bi 的使用计数都可以使用二进制分解唯一地表示。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力枚举| O(∏(bi+1)) | O(∏(bi+1)) | O(n) | 太慢了 |
 | 二进制分解背包| O(m log bi 和) | O(米) | 已接受 |

 ## 算法演练

1. 使用其数量 bi 的二进制分割将每种硬币类型转换为多种物品。 每个分割项的值等于 ai 乘以所选块大小。 这确保我们可以将任何使用计数表示为这些块的总和，而不会重叠。 
2. 初始化一个大小为 m + 1 的布尔数组 dp，其中 dp[0] 为 true，其他均为 false。 这代表目前可以达到的金额。 
3. 对于每个拆分项，从 m 向下迭代到项值。 对于每个位置 j，如果 dp[j - value] 为 true，则将 dp[j] 设置为 true。 这可以防止多次重复使用同一项目，从而保留每个分割块的 0/1 性质。 
4. 处理完所有项目后，计算从 1 到 m 有多少个索引 dp[i] 等于 true。 

正确性依赖于这样一个事实：二进制分解允许每个整数使用计数达到 bi 唯一地表示为所选块的总和。 每个块都是独立处理的，反向DP保证每个块最多被使用一次。 

始终保持的不变性是，在处理每个分割项后，dp 正确地表示仅使用处理后的块可实现的所有总和。 由于每个有效的有界选择恰好对应于二进制块的一个选择，并且每个块选择对应于有效的有界选择，因此DP恰好覆盖了可达和。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    a = list(map(int, input().split()))
    b = list(map(int, input().split()))

    items = []

    for i in range(n):
        val = a[i]
        cnt = b[i]
        k = 1
        while cnt > 0:
            take = min(k, cnt)
            items.append(val * take)
            cnt -= take
            k <<= 1

    dp = [False] * (m + 1)
    dp[0] = True

    for w in items:
        if w > m:
            continue
        for j in range(m, w - 1, -1):
            if dp[j - w]:
                dp[j] = True

    ans = sum(dp[1:])
    print(ans)

if __name__ == "__main__":
    solve()
```该代码首先对每种硬币类型执行二进制分割。 变量`items`存储分解的权重。 分裂循环的每次迭代都会消耗一部分 cnt（2 的幂），确保对数展开。 

DP 数组跟踪可达总和。 j 上的后向循环至关重要，因为前向迭代将允许同一项目重复使用多次，从而违反有界约束。 

dp[1:] 上的最终总和计算了直到 m 的所有可达到的正和。

 ## 边缘情况

 一种边缘情况是硬币价值为零。 对于输入：

 n = 2，m = 5

 a = [0, 1]

 b = [10, 1]

 零值硬币产生权重为 0 的分割项。在 DP 期间，这些项被忽略或不会引起任何变化，因为 dp[j - 0] 等于 dp[j]，这意味着它们永远不会引入新状态。 唯一有贡献的物品是价值 1 的硬币，它产生可达到的金额 {1}。 该算法正确地避免了可达计数的膨胀。 

另一个边缘情况是当所有硬币超过 m 时。 例如：

 n = 2，m = 5

 a = [10, 20]

 b = [5, 5]

 所有分割项大于 m 并被跳过。 dp 仍然是 {0}，所以答案是 0。这确认了过滤步骤`if w > m: continue`对于效率和正确性至关重要。 

第三种情况是仅存在一种硬币类型并且具有很大的多样性。 二进制分割确保即使 b 为 100000，我们仍然只处理大约 17 个项目。 然后，DP 的行为就像一个标准的有界背包，没有指数爆炸，从而保留了可行性。
