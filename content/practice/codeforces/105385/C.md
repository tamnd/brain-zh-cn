 ## 解决方案
 ## 问题理解

 我们得到了几个独立的测试用例。 每个测试用例都由数轴上的一组闭合线段组成，我们必须为每个线段分配 k 种颜色中的一种。 限制是，如果两条线段共享相同的颜色，则它们不得在线上的任何点相交。 换句话说，相同颜色的线段必须形成成对不相交间隔的集合。 

任务是计算存在多少种有效颜色，如果至少一个片段具有不同的指定颜色，则两种颜色被认为是不同的。 

重新解释条件的一个有用方法是将线段视为图中的顶点，其中边表示重叠。 任何有效的着色都会分配颜色，以便每个颜色类别在此区间图中都是一个独立的集合。 

约束很大：所有测试用例的段总数最多为 5 × 10^5，k 可以大到 10^9。 这立即排除了任何尝试枚举每个段的颜色或对重叠执行指数或二次检查的解决方案。 即使 O(n^2) 重叠检测也太慢，因此我们需要一种结构将问题简化为排序和线性或近线性扫描。 

当链中的段重叠时，就会出现一个天真的微妙陷阱。 例如，线段 A 与 B 重叠，B 与 C 重叠，但 A 不与 C 重叠。粗心的方法可能会独立处理每个重叠，并通过假设局部约束就足够来过多计算有效分配，而实际上颜色约束通过连接的重叠组件传播。 

另一种边缘情况是多个段共享相同的端点或嵌套。 例如，[1,10]、[2,3]、[4,5] 形成一个嵌套结构，其中对成对重叠的简单计数错误地表示了实际的禁止结构。 正确的解决方案取决于全局结构，而不仅仅是局部的成对冲突。 

## 方法

 如果我们尝试暴力破解，我们会为每个 n 段分配一个从 1 到 k 的颜色，总共给出 k^n 个分配。 对于每个作业，我们必须检查共享颜色的所有线段对并验证它们是否不重叠。 检查重叠需要在每次分配的最坏情况下进行 O(n^2) 次比较，因此总成本变为 O(k^n · n^2)，即使对于很小的 n，这也是完全不可行的。 

关键的观察结果是，约束的结构完全由线段沿线重叠的方式控制，并且在对端点进行排序后，这种结构变得简单。 如果我们按段的左端点对段进行排序，我们可以从左到右扫描，同时保持活动的重叠段。 在任何时刻，重叠的线段集都会在区间图中形成一个团，这意味着它们必须都具有不同的颜色。 

这将问题变成了动态约束：当一个段开始时，它必须选择与所有当前活动段不同的颜色。 当一个段结束时，它会释放一种颜色。 因此，每个分段的有效选择数量仅取决于当前有多少分段与其重叠。 

如果我们按照左端点递增的顺序处理片段并维护活动右端点的数据结构，则在每一步中禁止颜色的数量正是活动重叠片段的数量。 这会将计数转换为每个段的选择乘积，其中每一项是（k 减去当前活动计数）。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(k^n · n^2) | O(k^n · n^2) | O(n) | 太慢了|
 | 扫线计数| O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 我们按照左端点的递增顺序处理段，并维护哪些先前启动的段仍然处于活动状态。

1. 按左端点对所有线段进行排序。 这确保了我们总是按照数轴上开始的顺序看到线段。 
2. 维护一个跟踪活动段的结构，按其右端点排序。 这使我们能够有效地删除在当前片段开始之前已经结束的片段。 
3. 将答案变量初始化为 1。 
4. 按排序顺序扫描段。 在处理线段 i 之前，删除右端点严格小于左端点的所有活动线段，因为它们不再与未来的任何内容重叠。 
5. 令当前活动段的数量为d。 这些正是与当前片段重叠的片段，因为所有活动片段均较早开始且尚未结束。 
6. 当前段必须选择与所有 d 个活动段不同的颜色，因此它有 (k − d) 个有效选择。 将答案乘以 (k − d)，对 998244353 取模。 
7. 将当前段插入活动结构中。 

一个关键细节是，活动计数仅在跨越端点时发生变化，因此每个段根据其开始时的状态恰好贡献一个乘法因子。 

### 为什么它有效

 在处理线段时，每个活动线段都与它相交，因为所有活动线段都较早开始且尚未结束。 分配给当前段的任何颜色都必须与所有这些活动段不同，否则我们会在它们的重叠区域产生冲突。 

相反，任何已经结束的段都不能与当前段重叠，因此没有任何限制。 这意味着禁止颜色集完全由活动集大小决定，其他全局历史无关紧要。 有效选择的所有段的乘积精确地计算了所有有效分配，因为每个段选择都是独立于早期分配的条件的，并且扫描确保所有约束仅执行一次。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MOD = 998244353

def solve():
    t = int(input())
    for _ in range(t):
        n, k = map(int, input().split())
        segs = [tuple(map(int, input().split())) for _ in range(n)]
        
        # sort by left endpoint
        segs.sort()
        
        # active segments stored as (r)
        import heapq
        active = []
        
        ans = 1
        i = 0
        
        for l, r in segs:
            # remove ended segments
            while active and active[0] < l:
                heapq.heappop(active)
            
            d = len(active)
            ans = ans * (k - d) % MOD
            
            heapq.heappush(active, r)
        
        print(ans)

if __name__ == "__main__":
    solve()
```该解决方案对片段进行排序，以便我们按照出现的时间顺序处理它们。 堆存储当前活动段的右端点； 右端点小于当前左端点的任何线段都将被删除，因为它不再重叠。 

变量`d`表示有多少段与当前段重叠。 乘法步骤对与所有这些活动片段不同的颜色的选择进行编码。 模运算确保我们保持在限制范围内。 

一个微妙的点是我们从不显式存储颜色。 我们只计算每一步存在多少个选择，因为早期的分配完全确定了禁止集的大小。 

## 工作示例

 考虑一个具有重叠结构的简单情况：

 输入：

 n = 3，k = 3

 段：[1,4]、[2,5]、[6,7]

 排序顺序不变。 

| 细分 | 之前活跃 | d | 选择 (k - d) | 活动后 |
 | ---| ---| ---| ---| ---|
 | [1,4]| 0 | 0 | 3 | [1,4]|
 | [2,5]| 1 | 2 | 1 | [1,4],[2,5] |
 | [6,7]| 2 → 0 | 0 | 3 | [6,7]|

 答案变为 3 × 1 × 3 = 9。 

该跟踪显示了重叠如何仅在时间上局部增加约束，以及结束段如何完全消除限制。 

现在考虑嵌套段：

 输入：

 n = 3，k = 4

 段：[1,10]、[2,3]、[4,5]

 | 细分 | 之前活跃 | d | 选择| 活动后|
 | ---| ---| ---| ---| ---|
 | [1,10]| 0 | 0 | 4 | [1,10]|
 | [2,3]| 1 | 1 | 3 | [1,10],[2,3] |
 | [4,5]| 2 | 2 | 2 | [1,10],[4,5] |

 答案 = 4 × 3 × 2 = 24。 

这证实了嵌套段正确地积累了约束，因为大间隔与所有内部段重叠。 

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 | O(n log n) | O(n log n) | 排序占主导地位，每个段的堆操作为 O(log n) |
 | 空间| O(n) | 堆存储活动间隔 |

 约束最多允许 5 × 10^5 段，因此 O(n log n) 解决方案就足够了。 堆操作保持高效，因为每个段都只插入和删除一次。

 ## 边缘情况

 一种重要的边缘情况是所有段都不相交。 对于像 [1,2]、[3,4]、[5,6] 这样的输入，不会发生活动重叠，因此每个段应该始终有 k 个选择。 该算法在每一步都保持堆为空，每次都给出 d = 0，并正确计算 k^n。 

另一个边缘情况是完全嵌套。 对于段 [1,10]、[2,9]、[3,8]，每个新段都会增加活动集，直到较早的段在处理后面的段之前永不过期。 堆大小单调增长，并且算法正确地强制减少可用颜色。 

第三种边缘情况是紧密链接，其中重叠开始和结束交错，例如 [1,5]、[2,3]、[4,6]。 堆在处理 [4,6] 之前正确删除 [2,3]，确保只有当前活动的重叠影响计数，并且乘法因子反映真实的瞬时约束而不是历史约束。
