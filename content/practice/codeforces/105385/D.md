 ## 解决方案
 ## 问题理解

 我们有一个交易模拟，玩家可以反复将金钱转换成面粉，然后以更好的价格将面粉转换回金钱。 玩家一开始拥有一定数量的金币，并且时间有限。 每个动作都不是瞬时的，而是时间线性取决于该动作中处理的面粉袋数，以及额外的固定开销。 

有两个操作。 第一个操作让玩家从工厂购买 x 袋面粉。 每个这样的批次需要花费 p·x 金币并花费 a·x + b 秒。 第二个操作让玩家向小酒馆出售 x 袋面粉，赚取 q·x 黄金并花费 c·x + d 秒。 关键条件是 q 严格大于 p，因此如果买入然后卖出，每个袋子都会产生黄金利润。 

玩家可以按任意顺序重复这些操作，但每次操作都会消耗时间，总时间不能超过t。 目标是选择买入和卖出批次的顺序，以使最终黄金最大化。 

重要的结构是利润与 x 呈线性关系，但时间既有线性成本又有固定成本。 这意味着分组决策很重要：进行许多小额交易可能会由于重复的固定管理费用而浪费时间，而进行大额交易可以提高效率，但可能不适合时间或预算限制。 

这些约束推动每个测试解决方案的 O(T log some) 或 O(T)。 由于 T 可以高达 500，值高达 10^9，因此任何模拟每单位时间或每单位黄金的解决方案都是不可能的。 我们需要将操作视为聚合的“事务”并推理最佳批量大小。 

当贪婪策略总是选择每次操作的最大可能 x 时，就会出现微妙的失败情况。 例如，如果我们总是购买尽可能多的袋子，然后立即出售它们，则由于每批线性时间成本较大，我们可能会浪费时间。 当忽略固定开销 b 和 d 时，会出现另一种失败情况：如果 x 很小，则开销占主导地位，并且与较少的大周期相比，重复的小周期变得低效。 

## 方法

 暴力解决方案将模拟所有可能的操作序列，为每个买卖步骤尝试所有可能数量的袋子，并在所有有效的时间段上进行分支。 这将正确地探索所有策略，因为任何最佳策略都是一系列离散批量操作，但分支因子是巨大的。 即使将 x 限制为最多 m/p，操作数量在时间 t 上仍然是无界的，并且每一步都会以耦合的方式改变金钱和时间。 在最坏的情况下，时间允许大约 10^9 / 1 ≈ 10^9 单位规模的决策，使得任何逐步模拟都不可行。 

关键的观察结果是，该过程在结构上是单调的：一旦我们决定执行大小为 x 的买卖周期，最终效果就是状态的确定性转换。 每个完整周期都会将金币增加 (q − p)x 并消耗时间 (a + c)x + (b + d)。 以更细粒度部分交错周期不会带来中间好处，因为将一个周期分成多个较小的周期会严格增加总开销消耗，而不会增加利润。 

这将问题简化为选择我们可以执行多少个完整周期以及每个周期应该有多大。 由于每袋利润是固定的，唯一的权衡是根据线性时间成本摊销固定时间成本 b + d。 由此得出的结论是，最佳策略是在承受能力和时间限制的情况下，以尽可能大的 x 执行尽可能少的周期。 限制因素要么是初始资本 m，要么是时间预算 t。

然后，我们对周期大小 x 的可行性进行建模：我们必须使 p · x ≤ 当前黄金，并且每个周期的总时间必须适合剩余时间。 一旦 x 确定，我们就会重复应用这个循环，直到时间或金钱耗尽。 最佳 x 是通过平衡承受能力和时间效率来确定的，这减少了检查从约束导出的候选边界。

 ## 算法演练

 我们将流程压缩为重复的相同交易周期，并推断每个周期应该有多大以及可以执行多少次。 

1. 将完整周期解释为购买 x 单位，然后立即出售。 这通过增加利润 (q − p)x 和消耗时间 (a + c)x + (b + d) 来转换状态。 这是唯一有意义的原子操作，因为在未完成的情况下分离买卖并不会在线性定价下带来任何好处。 
2. 确定当前时刻可以承受的最大x。 由于每个周期预先需要 p·x 黄金，x 受到当前黄金除以 p 的约束。 
3. 确定剩余时间内可以执行的最大x。 每个周期消耗 (a + c)x + (b + d)，因此 x 还受到考虑固定开销后剩余时间的限制。 
4. 选择满足两个约束的最大 x。 这是最优的，因为利润随 x 线性增加，而管理费用被摊销，因此只要可行，大批量严格控制小批量。 
5. 计算在黄金或时间变得不足之前可以重复多少次这样的循环。 在每个最大批量执行后相应地更新黄金和剩余时间。 
6. 重复，直到在约束范围内无法执行进一步的循环。 

它之所以有效，是因为任何有效的策略都可以分解为完整的买卖周期，并且在每个周期内，分成较小的批次只会增加总开销，而不会提高利润。 由于除固定成本外，利润和时间均以 x 线性缩放，因此在每一步最大化 x 可以保留开销的最佳摊销，同时不会降低超出约束的可行性。 这确保了循环的重新安排不会产生更高的最终黄金量。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    T = int(input())
    for _ in range(T):
        p, a, b = map(int, input().split())
        q, c, d = map(int, input().split())
        m, t = map(int, input().split())

        gold = m
        time = t

        while True:
            if gold < p:
                break

            # maximum x we can afford
            x_money = gold // p

            # we need (a+c)x + (b+d) <= time
            if a + c == 0:
                x_time = x_money
            else:
                if time <= b + d:
                    break
                x_time = (time - (b + d)) // (a + c)

            x = min(x_money, x_time)

            if x <= 0:
                break

            cost_time = (a + c) * x + (b + d)
            gold -= p * x
            gold += q * x
            time -= cost_time

        print(gold)

if __name__ == "__main__":
    solve()
```该代码维护当前的黄金和剩余时间，重复执行最大的可行完整周期。 关键的实施细节是正确地将黄金的承受能力与时间可行性分开，然后取其最小值来确定批量大小。 

一个微妙的点是处理线性时间系数 (a + c) 为零的情况，其中每个周期的时间消耗是恒定的。 在这种情况下，批量大小仅受资金限制。 另一个细节是确保当剩余时间小于固定开销b + d时，即使金币足够，也无法进行进一步的循环。

 ## 边缘情况

 一种边缘情况是时间恰好等于固定开销 b + d。 在这种情况下，即使 x = 1 也是不可能的，因为没有时间留给线性分量。 该算法在计算 x_time 之前检查 time <= b + d，确保不会执行无效除法。 

当 gold 恰好小于 p 时，会出现另一种边缘情况。 循环立即终止，因为不可能进行购买。 这可以防止错误地尝试将 x_money 计算为零并意外执行退化循环。 

最后的边缘情况是 a + c 等于 0。 该算法单独处理这个问题，因为在时间限制中除以零是无效的。 在这种情况下，周期仅受金钱限制，并且时间永远不会减少，因此算法执行单个最大批次并完成。
