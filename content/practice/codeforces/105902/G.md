 ## 解决方案
 ## 问题理解

 我们有一个两人玩的确定性游戏，玩的是一堆串。 一开始有`x`串和参数`k`。 玩家轮流轮流，OC 总是先行动。 在玩家的回合中，他们选择要移除的串串数量，范围从 1 到当前值`k`，只要剩下足够的串。 每次移动后的值`k`正好减少 1。一次`k`达到 0 时，就无法进一步移动，因此无论谁在该点移动，都会立即失败。 

获胜者完全取决于最佳玩法，这意味着双方玩家总是选择能够最大化获胜机会的行动。 

输入给出多个独立游戏。 每个游戏由初始堆大小定义`x`和初始移动限制`k`。 输出只是获胜玩家的身份。 

这些限制迫使我们走向`O(1)`或者`O(log x)`每个测试解决方案。 高达`10^5`测试用例和值高达`10^18`，任何对动作或状态的模拟都是不可能的。 即使模拟单个游戏也是不可行的，因为`k`本身可以大到`10^18`。 

出现微妙的边缘情况时`k`相对于`x`。 例如，如果`x = 4, k = 10`，OC可以立即拿走所有串，因为第一步允许最多移除10个串，游戏立即结束。 一种天真的方法，假设游戏总是精确地持续下去`k`动作将是不正确的。 

另一个极端情况是当`k = 1`。 第一个玩家可以恰好移除一根串，但之后立即`k`变为 0，意味着第二个玩家无论剩余的串如何，都没有合法的移动。 这将创造直接的第一玩家胜利，独立于`x`只要`x ≥ 1`。 

最后，当`x`非常大并且`k`很小，串根本不重要，因为游戏纯粹是由于堆耗尽之前移动限制的减少而结束的。 

## 方法

 暴力模拟将明确地玩这个游戏：在每一回合尝试所有可能的删除，从 1 到`k`，递归地评估结果，并减少`k`每一步。 这正确地模拟了最佳游戏，因为它探索了完整的博弈树。 

然而，分支因子可达`k`，深度也达到`k`， 自从`k`减少每一步，直到达到零。 甚至无视`x`，状态的数量在实践中就像阶乘式爆炸一样增长。 和`k`最多`10^18`，这是完全不可行的。 

关键的见解是堆大小`x`只有当它变得足够大而不再限制游戏时才重要。 之后，游戏完全由移动限制递减的顺序控制。 每一步都会减少`k`正好 1，所以游戏有一个确定的最大回合数：最多在`k`除非堆提前用完，否则会移动。 

因此，真正的问题是：实际总共会发生多少次移动？ 每一步都会减少`k`减 1，所以之后`k`移动后数值变为0，下一个玩家无法移动。 因此，如果堆永远不会用完，游戏总是精确地持续下去`k`移动，并根据奇偶性确定获胜者：OC 获胜，如果`k`是奇数，KP 获胜，如果`k`是均匀的。 

唯一的复杂性是堆是否早于结束`k`移动。 但OC每回合总能移除至少1根串，所以之后`t`至少移动`t`烤串已被移除。 因此，如果`x < k`，游戏恰好在`x`移动，因为之后`x`移动堆是空的，下一个玩家不能移动。 如果`x ≥ k`，游戏恰好在`k`移动由于`k`达到零。 

所以移动次数就是`min(x, k)`，获胜者取决于该数字是奇数还是偶数。

 | 最优奇偶校验减少 | O(1) | O(1) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们将游戏简化为计算实际会发生多少步，然后确定谁走了最后一步。 

1. 计算`m = min(x, k)`。 这表示游戏无法继续之前实际可能发生的最大移动次数。 
2. 根据平价决定胜负`m`。 OC 率先行动，因此 OC 获胜，如果`m`是奇数，否则 KP 获胜。 

第一步背后的原因是，每次移动都保证至少一根串被移除并减少`k`恰好为 1，因此两种资源都不允许游戏超过`min(x, k)`移动。 

第二步来自交替转弯。 如果移动总数为奇数，则第一个玩家 OC 执行最后一个有效移动，而让第二个玩家不进行任何移动。 如果是偶数，KP执行最后一步，OC接下来卡住。 

### 为什么它有效

 在游戏的每个阶段，两个独立的“预算”都在缩减：串烧和移动津贴。 每次移动，串至少减少 1 个，并且每次移动移动限制正好减少 1 个。 因为两者都是单调递减且无法绕过，所以游戏完全取决于哪个资源先耗尽。 一旦我们认识到每一步棋都可以只移除一根串，那么最坏情况的持续时间就被强制，并且任何策略选择都不能延长或缩短游戏的时间。`min(x, k)`。 减少之后，游戏就变成了简单的交替回合计数。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        x, k = map(int, input().split())
        m = x if x < k else k
        if m % 2 == 1:
            print("OC")
        else:
            print("KP")

if __name__ == "__main__":
    solve()
```该代码使用简单的最小运算直接计算有效移动的数量。 比较`x < k`避免函数调用的开销并确保每个测试用例的时间恒定。 然后奇偶校验确定获胜者，依靠 OC 始终首先启动。 

一个常见的实施错误是假设游戏总是精确地持续`k`移动。 当`x < k`，因为堆较早清空。 另一个错误是尝试用可变的移除量贪婪地模拟移动，这是不必要的，因为最佳游戏不会影响总移动数。 

## 工作示例

 ### 示例 1

 输入：`x = 4, k = 4`我们跟踪有效移动限制：

 | 移动| 剩余 x | 剩余 k | 笔记|
 | --- | --- | --- | --- |
 | 0 | 4 | 4 | 开始 |
 | 1 | 3 | 3 | OC 行动 |
 | 2 | 2 | 2 | KP 动作 |
 | 3 | 1 | 1 | OC 行动 |
 | 4 | 0 | 0 | KP 动作结束比赛 |

 这里`m = min(4, 4) = 4`。 游戏共 4 步，为偶数，因此 KP 获胜。 这证实了当两种资源一起耗尽时，仅通过奇偶校验来决定获胜者。 

### 示例 2

 输入：`x = 2, k = 1`| 移动| 剩余 x | 剩余 k | 笔记|
 | --- | --- | --- | --- |
 | 0 | 2 | 1 | 开始 |
 | 1 | 1 | 0 | OC取1，k变为0 |

 这里`m = min(2, 1) = 1`。 只能进行一次移动，因为在那之后`k = 0`。 OC 采取了这一行动并立即获胜，这证明了移动限制主导堆大小的情况。

 ## 边缘情况

 当`k = 1`，游戏立即崩溃。 OC移动一次，减少`k`为0，无论剩下多少串，KP都没有合法动作。 该算法计算`m = min(x, 1) = 1`，这很奇怪，所以 OC 正确获胜。 

什么时候`x < k`， 例如`x = 2, k = 10`，游戏因串耗尽而不是移动限制而结束。 OC移除1串，KP移除1串，2步后堆变空。 该算法使用`m = 2`，由于第二步棋是最终棋，因此 KP 获胜。 

什么时候`x = k`，两个约束完美匹配，并且游戏总是精确地持续`k`移动。 该解决方案干净地减少到奇偶校验`k`，匹配匝的交替结构。
