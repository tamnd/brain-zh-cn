 ## 解决方案
 ## 问题理解

 我们将活动安排在固定的周数内，每个团队每周只派出一名代表。 因此，每周由长度为 n 的二进制字符串表示，其中第 i 个字符表示团队 i 的成员 1 或成员 2 是否在场。 

对于任何两个不同的团队 i 和 j，以及对于成员 (i, a) 和 (j, b) 的任何选择，我们都会查看这两个特定人员同时在办公室的几周。 换句话说，我们过滤团队 i 选择 a 而团队 j 选择 b 的周，并获得一系列周索引。 这对货币对的“隔离”被定义为连续此类会议之间的最大差距，包括在 w 周的循环时间线上从上次会议到第一次会议的环绕差距。 如果他们在任何一周都没有见面，那么隔离就是无限的。 

目标是设计每周的选择，以便来自不同团队的每对有序个体至少会面一次，并且所有这些对的最大隔离度尽可能小。 

输入大小清楚地表明，我们处于必须显式构建时间表而不是搜索时间表的状态。 我们最多可以有 10^4 个团队，但最多 52 周，因此每个团队都由一个短的二进制字符串表示。 这强烈表明解决方案是构建具有强成对交互属性的结构化二进制字符串族。 

天真的解释会尝试模拟或优化所有 2^(nw) 计划，这是完全不可能的。 即使关注每个团队的字符串，任何对 n = 10^4 的成对约束检查也已经将我们推向构造而不是搜索。 

如果两个团队从未实现四种组合 (1,1)、(1,2)、(2,1)、(2,2) 中的一种，则会出现微妙的失败情况。 在这种情况下，隔离是无限的。 如果组合仅出现一次，则会发生另一种失败：那么该对和该组合的隔离完全由到环绕边界的距离确定，因此即使所有组合都存在，它也会变得很大。 

## 方法

 我们首先将问题转化为二进制字符串的条件。 

每个队伍 i 都是一个长度为 w 的字符串。 对于两支球队 i 和 j，考虑所有 t 周的有序对 (s_i[t], s_j[t])。 有四种可能的模式：11、12、21、22。为了避免无限孤立，每对团队必须至少实现所有四种模式一次。 

因此，我们被要求构造 n 个二进制字符串，使得对于任何两个不同的字符串 x 和 y，坐标配对涵盖所有四种组合。 

暴力方法会尝试逐个分配字符串并测试与所有先前字符串的兼容性。 每个测试都需要扫描 w 个位置并验证是否所有四对都出现，进行 O(n^2 w) 检查。 由于 n 高达 10^4，w 高达 52，这已经接近 5×10^9 运算，这太大了。 

结构洞察力来自于分离四种所需模式所隐含的条件。 

如果我们将 A_i 定义为团队 i 派遣成员 1 的周数，则：

 条件11要求A_i∩A_j非空。 

条件22要求补数也相交，这意味着两者都选择成员2的周必须存在，所以(U \ A_i) ∩ (U \ A_j)非空，相当于A_i ∪ A_j ≠ U。 

条件12表示A_i不包含在A_j中。 

条件21表示A_i不包含A_j。 

因此，我们需要 {1..w} 的一系列子集，它们同时相交、共相交和反链。

这是一种经典的情况，我们通过固定坐标来强制结构，然后在剩余坐标上使用大型组合族。 我们可以固定两个位置，以便所有集合都包含一个固定坐标并排除另一个固定坐标。 这立即保证了全局的交叉口和同交叉口条件。 剩下的要求只是避免子集关系，这可以通过在剩余的 w−2 坐标内选择反链来解决。 由于中间的二项式层对于 w 来说非常巨大，高达 52，因此我们可以轻松地在那里挑选 n 个不同的子集。 

这将问题从全局约束系统转变为简单的组合构造问题。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力一致性检查| O(n^2w) | O(n^2w) | O(nw)| 太慢了|
 | 结构化组合构造| O(nw)| O(nw)| 已接受 |

 ## 算法演练

 1. 修复两个特殊周，例如第 1 周和第 2 周。我们强制每个团队在这两周有一个固定模式：成员 1 在第 1 周，成员 2 在第 2 周。这将创建一个全局锚，自动保证所有对之间的交叉属性。 
2. 关注剩下的 w−2 周。 每个团队的行为定义了这些位置的子集，在其中选择成员 1。我们将仔细构建这些子集。 
3. 从集合 {3..w} 中选择 n 个大小恰好为 k = (w−2)//2 的不同子集。 确切的大小选择对于正确性来说并不重要，但它保证我们留在子集格的单个级别内，防止任何两个选定集合之间的子集关系。 
4. 将第 i 个选定的子集分配给团队 i。 对于每周 t ≥ 3，如果 t 在其子集中，则将团队 i 设置为 1，否则将其设置为 2。 
5. 输出构造好的w个二进制字符串。 

为什么它起作用来自于前两个固定坐标强制的结构。 由于每盘都包含第 1 周，因此任何两支球队总是以 (1,1) 布局至​​少相遇一次。 由于每组都不包括第 2 周，因此两个补充始终包含第 2 周，保证至少有一次 (2,2) 次会议。 其余两个交叉模式是自动确保的，因为不允许任何集合完全包含另一个集合； 通过留在单一大小的层内，子集关系是不可能的。 这迫使每对字符串在两个方向上都不同，从而确保所有混合模式在坐标上至少出现一次。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, w = map(int, input().split())

    if w == 1:
        print("infinity")
        return

    m = w - 2

    # We will generate n subsets of size m//2 from [0..m-1]
    k = m // 2

    from itertools import combinations

    elems = list(range(m))

    subs = []
    for comb in combinations(elems, k):
        subs.append(set(comb))
        if len(subs) == n:
            break

    # build output strings
    ans = []

    for i in range(n):
        s = ['2'] * w

        # fixed coordinates
        s[0] = '1'
        s[1] = '2'

        subset = subs[i]

        for j in subset:
            s[j + 2] = '1'

        ans.append("".join(s))

    print(w)
    for row in ans:
        print(row)

if __name__ == "__main__":
    solve()
```该实现首先读取 n 和 w。 它处理退化情况 w = 1，其中没有时间表可以满足多个不同会议的要求，立即返回无穷大。 

然后它在剩余的 w−2 位置上构建子集。 每个子集对应于团队选择成员 1 的位置。使用组合可确保所有子集都是不同的，并通过使所有子集保持相同的基数来自动避免子集包含冲突。 

前两个位置是固定的，以执行全球结构性保证。 字符串的其余部分根据所选子集填充。 

打印的数值是w，代表达到的最大隔离度，后面是时间表。 

这里的一个常见陷阱是忘记在将子集位置映射到完整的 w 长度调度时索引必须移动 2。 

## 工作示例

 ### 示例：n = 2，w = 6

 我们有 m = 4 和 k = 2。每个团队的前两周固定为 1 和 2。 

我们从 {0,1,2,3} 中选择大小为 2 的子集。 

| 团队| 子集| 第 1 周 | 第 2 周 | 第 3-6 周模式 |
 | ---| ---| ---| ---| ---|
 | 1 | {0,1} | 1 | 2 | 1 1 2 2 | 1 1 2 2
 | 2 | {0,2} | 1 | 2 | 1 2 1 2 | 1 2 1 2

 所以最终的字符串是：

 团队1：112122

 第二队：112212

 这可确保每对团队在 6 周内实现所有四次配对，并且不会丢失任何模式。 

### 示例：n = 2，w = 1

 仅存在一周。 没有一对不同的团队可以在一个时间步内实现所有四种组合。 任何尝试都无法立即产生所有必需的会议，因此正确的输出是无穷大。 

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 | O(nw)| 我们构造 n 个长度为 w 的字符串，并在线性时间内填充每个字符串 |
 | 空间| O(nw)| 存储所有日程表|

 这些约束允许输出中最多包含 5×10^5 个字符，这很容易管理。 该构造避免了任何成对模拟，这对于 n = 10^4 来说是不可行的。

 ## 边缘情况

 当w = 1时，无法实现每对个体多次会面，因此无法实现所需的交互多样性。 该算法在任何构造之前直接返回无穷大。 

当 w 很小时，例如 w = 2 或 3，子集构造会退化为非常少的可用模式。 该算法仍然有效，因为固定前两个坐标可以确保剩余结构不需要很大； 即使是一个小的反链也足以分配最多 n 个团队，只要 n 不超过可用的组合。 

当 n 接近最大值时，w−2 位置上的组合层仍然包含远远超过 10^4 的子集，因此我们永远不会耗尽可用的配置。
