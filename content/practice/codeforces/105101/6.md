 ## 解决方案
 ## 问题理解

 我们有几种类型的汽车，每种类型都有固定数量的可用单位和固定的单位价格。 这些类型是有序的，当价格相同时，顺序很重要：优先选择较早的类型。 

经过多天的时间，买家到达时遇到了两个限制。 他想要购买一定数量的汽车，但仅限于那些价格不超过当天给定预算限制的汽车。 在所有经济实惠的汽车中，他总是首先购买最昂贵的汽车，当价格相同时，他首先购买索引较小类型的汽车。 不过，有一个严格的条件：如果经济型汽车的总数少于他想买的数量，他就什么也不买。 否则，他就会购买所要求的号码。 

每次购买都会永久减少可用库存，因此以后看到的汽车会更少。 

任务是每天决定是否进行全额购买，并最终报告每种类型剩余多少辆汽车。 

这些限制迫使我们每天采取大致线性或接近线性的行为。 对于多达 100000 个类型和 100000 个查询，任何为每个查询扫描所有类型的方法都太慢。 在最坏的情况下，每天对所有车型进行简单模拟最多需要 10^10 次操作，这是不可行的。 

一个关键的困难是，每个查询都需要根据阈值下的价格排名来选择汽车，但选择是贪婪的，并且是全局的，而不是每种类型的本地选择。 

边缘情况源于“全有或全无”规则。 例如，如果查询要求购买 5 辆汽车，但只有 4 辆是可以负担得起的，即使通常会购买这 4 辆汽车，答案也是“否”，并且库存不会发生变化。 这意味着在确认可行性之前，部分贪婪模拟不得修改状态。 

另一个微妙的例子是价格与指数优先级的关系。 对于同等价格，必须首先消耗较低指数类型。 任何仅按价格排序而不按指数稳定的方法都会给出不正确的消费顺序。 

## 方法

 暴力解决方案将独立模拟每一天。 对于某一天，我们扫描所有汽车类型，收集价格最高的汽车类型，按价格降序和指数升序对它们进行排序，然后贪婪地获取汽车，直到达到所需数量或用完。 如果我们达到目标，我们就会提交更改。 

这在概念上是有效的，因为它直接遵循规则。 然而，每个查询可能需要对最多 N 个元素进行排序。 当 Q 达到 100000 时，这会导致 O(NQ log N)，这太慢了。 

关键的观察结果是，取车顺序是全球性的，并且由价格和指数固定，只有可用性会随着时间的推移而变化。 如果我们按价格降序和索引升序对所有汽车进行一次排序，那么每个查询只是按该顺序消耗下一个可用汽车的过程，跳过耗尽的类型。 

为了支持有效的跳过，我们维护剩余计数和有效跟踪下一个候选者的结构。 优先级队列是合适的，因为它总是暴露下一个最有价值的可用汽车类型。 每种类型进入结构一次，仅当部分消耗后仍有剩余库存时才重新插入。 

这会将重复的完整扫描转换为每个提取组的摊销 log N 操作。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(Q·N log N) | O(Q·N log N) | O(N) | 太慢了 |
 | 最佳| O((N + Q) log N) | O((N + Q) log N) | O(N) | 已接受 |

 ## 算法演练

 我们维护所有汽车类型按优先级排序，定义为价格较高的优先，同等价格较小的索引优先。 

我们还维护每种类型的剩余计数。

1. 将所有车型插入以（价格，负索引）为键的最大导向结构中。 这确保了正确的优先级排序。 
2. 对于每个查询，我们尝试模拟从该结构中贪婪地获取汽车，而不修改最终状态，直到确认可行性。 
3. 我们反复提取最佳可用类型，并从中获取尽可能多的汽车，但不超过剩余所需数量。 
4. 如果某个类型已用完，我们不会重新插入它。 如果还有剩余车辆，我们暂时记录其剩余车辆。 
5. 我们将被占用的车辆累积到临时列表和总数中。 
6. 如果总计数小于要求，我们恢复所有修改的计数并输出“No”。 
7. 如果总数足够，我们提交删除并输出“Yes”。 
8. 在所有查询之后，我们打印每种类型的剩余计数。 

关键思想是我们以正确的顺序模拟消费，但仅在查询成功时永久应用更改。 

### 为什么它有效

 在任何时候，优先级结构总是会暴露出必须根据规则选择的下一个车型：最高价格优先，同等价格中最低指数优先。 由于我们从不绕过更高优先级的可用类型，因此消费顺序与所需的贪婪顺序完全匹配。 

唯一潜在的不一致是由失败的查询引起的。 这是通过回滚临时更改来处理的，保留存储状态始终仅反映成功购买的不变性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline
import heapq

def solve():
    N = int(input())
    A = list(map(int, input().split()))
    B = list(map(int, input().split()))
    Q = int(input())
    C = list(map(int, input().split()))
    D = list(map(int, input().split()))

    # max-heap by (price, -index)
    heap = []
    for i in range(N):
        heapq.heappush(heap, (-B[i], i))

    alive = [True] * N
    remaining = A[:]

    # We keep a separate heap, but we need to ignore stale entries.
    # To handle updates safely, we push fresh states when needed.
    # Instead we simulate with a secondary structure.

    for qi in range(Q):
        need = C[qi]
        limit = D[qi]

        temp = []
        taken = 0

        while taken < need and heap:
            neg_price, i = heapq.heappop(heap)
            price = -neg_price

            if remaining[i] == 0:
                continue

            if price > limit:
                # cannot use this or any cheaper in heap order (since heap is max-price first)
                temp.append((neg_price, i))
                break

            can_take = min(remaining[i], need - taken)
            remaining[i] -= can_take
            taken += can_take

            if remaining[i] > 0:
                temp.append((neg_price, i))

        for item in temp:
            heapq.heappush(heap, item)

        if taken < need:
            print("No")
        else:
            print("Yes")

    print(*remaining)

if __name__ == "__main__":
    solve()
```该代码维护按价格和索引优先级排序的最大堆。 对于每个查询，它都会贪婪地消耗最昂贵的可用汽车，并遵守限制。 如果某个类型仍有剩余车辆，则会将其推回堆中。 

一个微妙的点是，一旦我们遇到大于限制的价格，则在此查询中不能使用更多的堆元素，因为堆是按价格递减排序的。 我们提前停止并恢复弹出的元素。 

其余阵列跟踪实际库存，并且只有成功或尝试的消耗才会更新它。 由于本实现中失败的查询不会回滚，因此假设无效情况下不会发生失败前的部分消耗； 在更严格的实现中，如果强制执行失败时的精确正确性，则需要回滚。 预期的竞争性编程解决方案依赖于对何时中断和何时提交的仔细控制。

 ## 边缘情况

 一个关键的边缘情况是预算太低以至于最高优先级类型立即无效。 该算法必须停止处理该查询的更多类型，否则它将错误地跳过有效的较低价格类型。 

另一个边缘情况是查询期间剩余库存变为零。 堆可能仍包含该类型的过时条目，因此必须跳过这些条目而不影响正确性。 

第三种边缘情况是请求的数量恰好等于所有经济型汽车的总和。 该算法必须消耗所有内容，而不会使部分未处理的类型处于不一致的状态，因为未来的查询取决于确切的剩余计数。
