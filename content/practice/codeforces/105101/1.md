 ## 解决方案
 ## 问题理解

 我们有一组战士，每个人都有一个我们称之为忠诚度的力量值，以及一组敌人群体，每个人都有所需的力量。 一组战士，称为小队，用于对抗一个敌人团体。 

如果同时满足两个条件，一个小队就会击败一个敌方团体。 每个被选入小队的战士的忠诚度必须严格大于敌人的实力，并且小队的规模必须至少等于敌人的实力。 如果一个小队战斗，其中的每个战士都会死亡，因此每个战士在所有小队中最多可以使用一次。 

任务是决定是否可以将部分或所有战士分成不相交的小队，以便为每个敌方团体分配一个满足这些约束的小队。 

关键的困难在于，每个敌人都需要最少的战士数量和每个选定战士的最小阈值，并且这两个约束相互作用，因为尽早选择更强的敌人会降低较弱敌人的灵活性。 

所有测试中的限制都达到十万名战士和敌人，因此任何试图明确建立小队或检查所有任务的解决方案都太慢了。 输入大小的任何二次方都会被立即排除。 即使是重复扫描每个敌人整个阵列的贪婪结构也会超出限制。 

一些失败场景出现在天真的贪婪方法中。 

一种常见的错误是以任意顺序处理敌人。 例如，如果先处理较强的敌人，他们可能会消耗太多高忠诚度的战士，而即使不同的顺序会成功，也无法满足较弱的敌人。 另一个错误是忽视了忠诚度上严格的不平等，错误地允许使用与敌人实力相等的战士。 

一个更微妙的问题是，在不考虑全局重用的情况下，针对每个敌人单独处理小队编队。 由于每个战士被消耗一次，局部可行性并不意味着全局可行性。 

## 方法

 蛮力策略会尝试将一组合格的战士一一分配给每个敌人。 对于每个敌人，我们都会扫描忠诚度大于敌人实力的所有未使用的战士，并检查是否剩余足够的战士。 这已经花费了每个敌人的线性时间，导致最坏的情况为 O(NM)，这对于 10^5 规模的输入是完全不可行的。 

关键的结构见解是只计算物质，而不计算身份。 对于具有 b 要求的敌人，唯一有用的信息是当前有多少未使用的战士的忠诚度严格大于 b。 如果该人数至少为 b，我们就可以组建一支规模为 b 的小队。 现在可以忽略任何额外的符合条件的战士。 

这将问题转化为管理多组战士的优势并反复检查可用性阈值。 对战士和敌人进行排序可以实现贪婪的策略。 我们从最难到最简单的顺序处理敌人，以便首先满足大的要求，并且我们维护可用战士的多重集（通过堆或带有指针的排序列表实现）。 

对于每个敌人，我们都会聚集所有足够强大的战士，并确保我们可以保留其中的 b 个。 如果不是，答案是不可能的。 

这将问题简化为排序和线性处理。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(NM) | O(N) | 太慢了 |
 | 最优贪心排序 | O((N + M) log N) | O((N + M) log N) | O(N) | 已接受 |

 ## 算法演练

我们首先按忠诚度升序对战士进行排序，并按所需强度按降序对敌人进行排序。 我们将维护一个数据结构，跟踪当前符合当前敌人资格的所有战士以及以后将符合资格的所有较弱战士。 

我们使用最大堆来存储对当前敌人来说足够强大的可用战士。 

1. 按升序对战士忠诚度进行排序，以便我们可以随着敌人需求的减少逐步添加符合条件的战士。 这允许增量激活，而不是从头开始重新计算资格。 
2. 按降序对敌人的需求进行排序，以便首先处理最严格的约束。 这可以防止弱小的敌人消耗更强的敌人所需的结构。 
3. 初始化一个指向战士数组的指针和一个空的最大堆。 
4. 按排序顺序迭代敌人。 对于每个要求为 b 的敌人，将所有忠诚度严格大于 b 的战士插入堆中。 我们在添加这些战士的同时推进指针。 
5.插入后，检查堆中是否至少包含b个战士。 如果没有，就不可能组成有效的小队，所以我们立即返回失败。 
6. 否则，从堆中弹出b个战士，代表为该敌人组建一个小队。 这些战士现已从系统中删除，无法重新使用。 
7. 如果所有敌人都处理成功，我们返回成功。 

### 为什么它有效

 在任何时刻，堆都准确地代表尚未使用的战士集合，其忠诚度足以应对当前或稍后处理的任何较弱的敌人。 以降序处理敌人可以确保一旦我们将战士交给强大的敌人，我们就永远不需要他们去对付较弱的敌人。 总是为当前敌人消耗任何可用的合格战士的贪婪选择是安全的，因为任何这样的战士也有资格为所有剩余的敌人服务，而让他们闲置无助于未来的可行性。 因此，可行性只取决于每一步是否有足够的补给，而不取决于具体选择哪些战士。 

## Python 解决方案```python
import sys
input = sys.stdin.readline
import heapq

def solve():
    T = int(input())
    for _ in range(T):
        N = int(input())
        A = list(map(int, input().split()))
        M = int(input())
        B = list(map(int, input().split()))

        A.sort()
        B.sort(reverse=True)

        heap = []
        i = N - 1

        ok = True

        for b in B:
            while i >= 0 and A[i] > b:
                heapq.heappush(heap, A[i])
                i -= 1

            if len(heap) < b:
                ok = False
                break

            for _ in range(b):
                heapq.heappop(heap)

        print("Yes" if ok else "No")

if __name__ == "__main__":
    solve()
```该解决方案对两个数组进行排序，然后使用指针逐步将符合条件的战士插入最大堆中。 处理每个敌人时，首先确保所有可能的具有足够忠诚度的战士都可用，然后检查是否有足够的剩余。 堆只存储当前可用的战士。 恰好弹出 b 个元素模拟分配一个小队。 

一个微妙的点是严格的不平等`A[i] > b`，这一点绝对不能放松。 使用`>=`会错误地允许边界战士并破坏有效性。 

另一个重要的实现细节是我们总是按降序处理敌人。 颠倒这个顺序会破坏正确性，因为它可能会因为较弱的约束而过早消耗高忠诚度的战士。

 ## 边缘情况

 一个关键的边缘情况是所有战士的水平仅略高于敌人的要求。 例如，如果所有 A 值都等于 b，则没有战士符合资格，因为条件很严格。 该算法可以正确处理这个问题，因为它只推送 A[i] > b 的战士。 

另一种情况是，一个非常大的敌人需要消耗几乎所有强大的战士。 贪婪排序确保首先处理该敌人，因此它不会被早期的弱分配所阻止。 

最后一个边缘情况是多个敌人具有相同的实力。 排序使它们保持相邻，并且每一步独立地消耗同一池中的 b 个战士。 由于消费是明确的，因此重用不会意外发生。
