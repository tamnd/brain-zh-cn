 ## 解决方案
 ## 问题理解

 我们经过多轮重复的决策过程。 每轮都有一组$N+1$球员：伊内斯和$N$其他的。 每轮都提出两种可能的黄金分配机制。 

在第一种机制中，一部分玩家选择“分享”。 如果$k$共有玩家选择此共享选项，则总池数为$X$，以及其中每一个$k$玩家收到$\lfloor X / k \rfloor$。 该值仅取决于有多少玩家加入共享组，而不取决于他们的身份。 

在第二种机制中，选择它的玩家会收到固定金额$Y$，独立于他人。 

对于每一轮，我们都会被告知另一轮是什么$N$玩家选择。 然后伊内斯决定自己的选择。 她的每轮决定纯粹是贪婪的：她将自己在选项 1 和选项 2 下的收益进行比较，假设其他人的选择是固定的，并选择更好的一个。 如果两种选择都能给她带来相同的回报，她必须选择分享。 

使用此策略模拟所有回合后，我们必须计算包括 Inés 在内的每个玩家累积的总金币。 

限制条件$N, M \le 100$明确表示即使是简单的$O(M \cdot N)$或者$O(M \cdot (N+1))$模拟就完全足够了。 除了直接的每轮评估之外，不需要进行优化。 

一个微妙的点是，伊内斯的决定会影响其他人在共享选项中的收益，因为它改变了$k$。 这意味着我们必须评估每一轮的两个假设世界：一个是 Inés 选择选项 1，另一个是她选择选项 2。 

可以打破朴素推理的边缘情况通常来自重新计算$k$错误或忘记楼层划分适用于共享组中的每个参与者。 

一个最小的说明性极端情况是$X$小于共享者的数量。 例如，如果$X = 3$4位玩家分享，然后每个人都会收到$\lfloor 3/4 \rfloor = 0$。 任何忘记地板或假设每个玩家至少有 1 个单位的实现都是错误的。 

另一个特殊情况是打破平局。 如果两个选项都给 Inés 带来相同的回报，她必须选择选项 1。这会影响未来的分配，必须严格执行。 

## 方法

 一种直接的方法是在概念上模拟每一轮两次。 对于给定的回合，我们尝试 Inés 选择选项 1，计算共有多少玩家加入共享，并确定每个玩家的奖励$\lfloor X/k \rfloor$。 然后我们计算 Inés 的假设增益。 我们对选项 2 做同样的事情，她的收益就是$Y$。 

每轮这种强力评估已经足够了，因为计算$k$只是计算有多少$N$固定玩家选择选项 1，根据场景，可能还包括 Inés 本人。 那是$O(N)$每轮，给予$O(MN)$总工作量。 

自从$N, M \le 100$，这最多是$10^4$操作，这是微不足道的。 

不存在隐藏的组合爆炸，因为伊内斯以外的玩家不会改变他们的决定，而且一旦伊内斯的选择确定，每一轮都是独立的。 唯一的依赖关系是在单轮内通过的值$k$。

 | 蛮力 |$O(MN)$|$O(N)$| 已接受 |

 在实践中，“最佳”解决方案只是仔细模拟。 

## 算法演练

 我们独立处理每一轮，维护所有玩家的累积分数。 

1. 对于每一轮，阅读$X$,$Y$，以及选择的数组$N$玩家。 我们立即计算这些玩家中有多少人选择选项 1，将此值称为$c$。 该计数决定共享下的所有结果。 
2. 考虑 Inés 选择选项 1。在这种情况下，共享者的数量变为$k = c + 1$。 每个分享者，包括 Inés 和所有选择 1 的玩家，都会收到$\lfloor X / k \rfloor$。 选择选项 2 的玩家在此分支中不会收到任何内容。 我们计算了 Inés 在这种情况下的假设增益。 
3. 考虑 Inés 选择选项 2。在这里，她收到$Y$。 其他玩家的选择不会受到她的影响，因此他们的分配不需要重新计算； 只有伊内斯的回报对决策很重要。 
4. 比较 Inés 在两种情况下的收益。 如果选项 1 给出的值更高，她就会选择它。 如果选项 2 给出的值更高，她就会选择它。 如果相等，则根据平局规则，她选择选项 1。 
5. 一旦她的决定确定，我们就会计算该轮的最终结果，并将贡献添加到每个玩家的总数中。 对于选项 1，我们再次使用适当的$k$并分发$\lfloor X/k \rfloor$致所有分享者。 对于选项 2，只有 Inés 获益$Y$。 

关键细节是，我们必须每轮每个场景仅重新计算一次共享结果，而不是跨轮增量计算。 

### 为什么它有效

 该算法是正确的，因为除了非 Inés 玩家的固定选择之外，每一轮都是独立的。 对于任何固定回合，伊内斯的决定仅取决于比较两个确定性值：她在共享和直接奖励下的收益。 共享收益完全由参与者的数量决定，这对于两种假设情况都是事先已知的。 由于她的决策不会影响任何未来回合的输入，因此局部最优的每轮决策会产生正确的全局结果。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    N, M = map(int, input().split())
    ans = [0] * (N + 1)

    for _ in range(M):
        X, Y = map(int, input().split())
        A = list(map(int, input().split()))

        cnt1 = sum(1 for a in A if a == 1)

        # Option 1: Inés shares
        k1 = cnt1 + 1
        gain1 = X // k1

        # Option 2: Inés takes Y
        gain2 = Y

        if gain1 > gain2:
            take_share = True
        elif gain2 > gain1:
            take_share = False
        else:
            take_share = True

        if take_share:
            k = cnt1 + 1
            val = X // k
            for i in range(N):
                if A[i] == 1:
                    ans[i] += val
            ans[N] += val
        else:
            ans[N] += Y

            for i in range(N):
                if A[i] == 1:
                    ans[i] += 0

    print(*ans)

if __name__ == "__main__":
    solve()
```该解决方案为每个玩家保留了一个运行总数。 对于每一轮，它首先计算有多少玩家选择选项 1，这是唯一需要的全局统计数据。 然后它评估 Inés 在两种选择下的假设收益。 通过在平等时优先共享来明确处理平局。 

当应用最终决定时，共享分支将相同的价值分配给所有参与的玩家，包括 Inés。 非共享分支仅更新 Inés。 

一个常见的实施错误是忘记了选项 1 场景中的共享者集合包括 Inés 本人，这使得分母从$c$到$c+1$。

```| 步骤| cnt1 | k1 | 伊内斯选项 1 | 伊内斯选项 2 | 决定| 更新 |

 这里共享给出 5，直接给出 6，所以 Inés 选择选项 2。只有她得到 6。

这表明，当伊内斯拒绝分享时，其他玩家的选择并不重要。

 ## 边缘情况

 一种重要的边缘情况是，由于团体规模过大，共享变得极其无利可图。 例如，如果$X=1$所有玩家都选择选项1，然后$k=N+1$每个参与者都收到 0。算法自然地处理这个问题，因为整数除法产生 0，并且所有更新保持一致。 

另一个边缘情况是只有 Inés 选择选项 1，而所有其他人都选择选项 2。那么$k=1$，所以她收到了完整的$X$。 该实现正确地将自己包含在计数中，避免除以零或不正确的分母。 

最后一个微妙的情况是选项之间的平等。 如果$X//(c+1) = Y$，算法仍然必须选择共享。 决策分支显式检查相等性并分配共享路径，确保下游总数符合问题要求。
