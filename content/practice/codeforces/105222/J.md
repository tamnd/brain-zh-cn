 ## 解决方案
 ## 问题理解

 我们被要求使用由两种符号构建的奇怪的混合数字系统来表示正整数。 

第一种是通常的从 0 到 9 的十进制数字。每个数字都有一个给定的成本，在表示中使用一次就会产生该金额的成本。 

第二种由七个罗马符号组成，I、V、X、L、C、D、M。其中每一个不再被视为基于字符串规则的数字系统，而是表现为数值固定的数字：I 贡献 1，V 贡献 5，X 贡献 10，L 贡献 50，C 贡献 100，D 贡献 500，M 贡献 1000。每个罗马符号也有其自己的每次使用成本。 

数字以 10 为基数的位置形式书写，但每个位置都允许使用十进制数字或这些罗马符号之一。 如果将值为 v 的符号放置在位置 i（从右侧从 0 开始计数），则它将为总值贡献 v · 10^i。 最终编写的表达式必须精确计算为给定的整数 n。 

任务是为每个位置选择一个符号，以便总和与 n 完全匹配，同时最小化总成本。 

约束 n ≤ 10^18 意味着最多 19 位小数。 测试用例的数量很大，最多可达 2000 个，因此任何解决方案都必须在每个测试用例的位数上接近线性，而不是在值范围上呈指数关系。 成本很大，但与复杂性无关，只是它们需要 64 位或 Python 整数。 

一个微妙的问题是存在大于 9 的符号，这打破了数字相互独立的通常假设。 如果我们将像 M = 1000 这样的符号放在低位，它可以创建进位到更高的位置。 这意味着我们不能独立地对待每个十进制数字，并且贪婪的每个数字都会失败。 

尝试每个位置的所有选择而不跟踪进位的幼稚方法将会失败。 例如，如果我们想要表示 19，并且我们将 X (10) 单独放置在单位位置，将 I (1) 单独放置在单位位置，则我们可能会超出或需要调整更高的数字。 通过进位实现仓位之间的交互是核心难点。 

当高价值符号用于低位置时，会出现另一种失败情况。 例如，将 M (1000) 放在表示 1000 的个位处表现正确，但将其放在像 1000 + 10 这样的混合配置中需要补偿进位到下一个数字，这会改变更高位置的结构。 

## 方法

 一个蛮力的想法是独立处理每个小数位并尝试所有 17 个可能的符号（10 个数字加 7 个罗马符号）。 对于每个位置，我们计算结果总和并检查它是否等于 n。 这会失败，因为符号可能会溢出到更高的位置，因此局部决策不是独立的。 即使我们扩展暴力来尝试将所有符号分配给每个位置，最坏情况下的搜索空间也是 17^19，这是完全不可行的。 

关键的观察是，虽然符号通过进位交互，但相邻位置之间的交互是严格局部的。 当我们将值 v 放在位置 i 时，它仅影响位置 i 并创建到 i+1 的进位。 这意味着我们可以在保持进位状态的同时处理从最低有效位到最高有效位的数字。 

这将问题转换为 n 位置上的数字动态规划过程。 在每个位置，我们知道 n 的目标数字和传入的进位。 我们选择 17 个符号之一，将其值加上进位相加，并确定结果数字和下一个进位。 我们希望在每个位置都精确匹配目标数字。

唯一的复杂之处是进位可能会变大，因为罗马符号在单个位置最多可以贡献 1000。 然而，由于只有 19 个位置，并且每一步除以 10，进位不能任意爆炸，并且仍以几千为界。 这使得（位置、进位）上的状态空间 DP 变得可行，并且转换很小。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力破解分配| O(17^19) | O(17^19) | O(19) | 太慢了|
 | 具有进位状态的数字 DP | O(位置 × 状态 × 17) | O（状态）| 已接受 |

 ## 算法演练

 我们独立处理每个测试用例。 

1. 从最低有效位到最高有效位提取 n 的十进制数字。 这定义了每个位置的目标数字，在考虑进位后我们必须完全匹配。 
2. 将动态编程状态定义为处理前 i 个位置的最小成本，同时将当前进位值 c 放入位置 i。 该进位表示先前位置的累积溢出。 
3. 用进位 0 和成本 0 初始化位置 0 处的 DP。这对应于从个位开始，没有传入溢出。 
4. 对于每个位置 i 和每个可达进位 c，尝试放置 17 个可用符号中的每一个。 令符号具有值 v 和成本 cost[v]。 该位置贡献的总价值变为 v + c。 
5. 计算结果数字 rem = (v + c) mod 10。这必须等于位置 i 处 n 的目标数字。 如果不匹配，则该符号放置对于该状态无效。 
6. 如果匹配，则计算下一个进位 next_c = (v + c) // 10. 使用进位 next_c 将 DP 转移到位置 i + 1，更新最小成本。 
7. 处理完所有仓位后，我们可能还有剩余进位。 该进位必须与 n 的更高隐式数字一致，这意味着这些数字在超出 n 的长度后为零。 因此，我们继续转换，直到进位在有效的终止位置变为零。 
8. 答案是消耗了n的所有数字并以进位0结束的所有DP状态中的最小成本。 

其工作原理与以 10 为底的加法结构有关。 每个符号放置仅影响其自己的数字，并以确定性方式向前传播溢出。 DP 状态准确地捕获了继续下去所需的信息，没有歧义：位置和进位完全决定了未来的可行性。 除了通过进位之外，早期的选择不会影响未来，因此状态压缩是无损的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

romans = [1, 5, 10, 50, 100, 500, 1000]

def solve_case(n, cost_d, cost_r):
    digits = list(map(int, str(n)))[::-1]
    L = len(digits)

    values = list(range(10)) + romans
    costs = cost_d + cost_r

    INF = 10**30
    dp = {0: 0}  # carry -> cost at position 0

    for i in range(L):
        ndp = {}
        target = digits[i]

        for carry, cur_cost in dp.items():
            for v, cst in zip(values, costs):
                s = v + carry
                if s % 10 != target:
                    continue
                nc = s // 10
                nc_cost = cur_cost + cst
                if nc not in ndp or nc_cost < ndp[nc]:
                    ndp[nc] = nc_cost

        dp = ndp
        if not dp:
            return -1

    # process remaining carry
    i = L
    while dp:
        if 0 in dp:
            return dp[0]

        ndp = {}
        target = 0

        for carry, cur_cost in dp.items():
            for v, cst in zip(values, costs):
                s = v + carry
                if s % 10 != target:
                    continue
                nc = s // 10
                nc_cost = cur_cost + cst
                if nc not in ndp or nc_cost < ndp[nc]:
                    ndp[nc] = nc_cost

        dp = ndp
        i += 1
        if i > L + 60:
            break

    return min(dp.values()) if dp else -1

def main():
    t = int(input())
    for _ in range(t):
        n = int(input().strip())
        cost_d = list(map(int, input().split()))
        cost_r = list(map(int, input().split()))
        print(solve_case(n, cost_d, cost_r))

if __name__ == "__main__":
    main()
```该实现维护一个由进位键控的字典，这避免了存储无法到达的状态。 每层对应于数字的一位小数加上额外的层以刷新剩余进位。 

转换步骤严格执行数字匹配，这确保了构造表示的正确性。 成本逐渐累积，并且仅保留每个进位状态的最小成本。 

一个实际的微妙之处是我们继续处理超过 n 的长度，直到进位消失。 这是必要的，因为在较低位置选择大符号可能会将值推入超出原始长度的更高虚数位。 

## 工作示例

 为简单起见，考虑一个小示例，其中 n = 102 并且所有成本均为 1。 数字是 [2, 0, 1]。 

在位置 0，目标为 2。我们可以将数字 2 与进位 0 一起放置，从而得到进位 0，或者使用罗马符号 I 加上进位调整（如果需要），但只有产生余数 2 的位置才能幸存。 

在位置 1，目标为 0。DP 继承可能的状态，并且仅保留那些在模 10 后产生零的先前选择的组合。 

| 职位| 携带 | 所选符号 | 价值总和 | 数字检查 | 执行 | 成本|
 | ---| ---| ---| ---| ---| ---| ---|
 | 0 | 0 | 2 | 2 | 比赛| 0 | 1 |
 | 1 | 0 | 0 | 0 | 比赛| 0 | 2 |
 | 2 | 0 | 1 | 1 | 比赛| 0 | 3 |

 这证实了标准数字表示形式是有效的并且可以作为基线。 

现在考虑使用罗马符号是有益的情况。 假设在某个位置我们用 X = 10 替换一个数字； 这会创建一个将值移至下一位的进位。 DP 通过进入更高的进位状态而不是强制进行本地校正来捕获这一点，这演示了如何有意使用而不是避免溢出。

 ## 边缘情况

 当罗马符号放置在低位并生成进位链时，就会出现临界边缘情况。 例如，对于 1000 这样的数字，将 M = 1000 放在个位中会导致将 100 进位到更高的位置。 DP 通过从位置 0 处的进位 0 移动到位置 1 处的进位 100 来处理此问题，并继续处理直到解析出更高的数字，从而确保与 n 的其余结构的一致性。 

另一种边缘情况是当多个表示产生相同的数字余数但不同的进位时。 例如，某个位置上的 10 可以来自 X (10) 或来自 0，并从较低位置进位 1。 DP 通过将进位视为状态的一部分来明确区分这些情况，确保两种可能性都得到独立评估，并且只有更便宜的才能生存。
