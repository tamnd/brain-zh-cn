 ## 解决方案
 ## 问题理解

 我们有五种烈酒，索引从 1 到 5，每种烈酒都有一定数量的副本。 每种精神类型`i`具有固定的内在价值`i`。 

称为“召唤”的单个操作选择可用烈酒的某个子集。 对于每一种选择的烈酒，我们独立决定它是否有贡献`1`或贡献其全部价值`i`。 所有选定的武魂贡献值之和不得超过6，如果正好是6，则操作会产生一个新的武魂，价值为6。每个原始武魂每次召唤最多可以使用一次，如果在召唤中使用，则会被消耗。 

任务是计算我们可以生产的 6 值烈酒的最大数量。 

输入规模很大，最多有 100000 个测试用例，每种类型最多可计数 10^9。 这立即排除了任何尝试模拟单个选择或枚举每个测试用例子集的方法。 任何有效的解决方案都必须在恒定时间内处理每个测试用例，依赖于所有可能的有效召唤的固定结构特征。 

当一个人试图“贪婪地打包高值”而不考虑将贡献降级为 1 的能力时，就会出现一种微妙的失败情况。例如，使用 5 不会自动帮助您达到 6，除非它正确配对。 

考虑输入：```
1 0 0 0 1
```一个天真的想法可能会尝试单独使用 5，但单个 5 无法达到总计 6，因为即使取其全部值也只能得到 5，并且没有其他项目可以弥补差距。 正确答案是 0，但粗心的贪婪方法常常会多算。 

另一个陷阱是假设每种精神总是贡献其全部价值。 之间切换的灵活性`1`和`i`这使得分组结构变得不平凡。 

## 方法

 强力解释会尝试枚举可用烈酒的所有子集，并且对于每个子集，尝试“1 或 i”选项的所有分配，检查某些分配的总和是否为 6。这原则上是正确的，但会立即爆炸：即使对于中等计数，子集的数量也是指数级的，并且每个子集的分配选择会添加另一个指数因子。 即使将我们限制在一个测试用例中，这也是不可行的。 

关键的观察是，召唤中唯一重要的是选择了多少个项目，以及通过使用完整值而不是 1 提取了多少“额外价值”。如果子集包含 k 个灵魂，将它们全部分配给 1 给出基线 k，并选择其中一些贡献其完整值会增加额外的值`(i - 1)`每个选定的精神。 因此，每个有效召唤的特征都是由一组小的物品组成，其基本总和加奖金恰好等于 6。 

因为目标总数太小，所以每次有效召唤最多只能包含6个精灵。 这使我们能够枚举有效组的所有结构模式，这将问题转化为从计数中贪婪地重复提取这些模式。 

中心思想是，越小的团体总是更有价值，因为他们每生产 6 个烈酒消耗的精神就越少。因此，我们优先考虑先用 2 个物品形成有效的召唤物，然后是 3 个物品，依此类推，总是在每个级别获取尽可能多的物品。

 ## 算法演练

 我们首先按大小对所有可能的有效传票进行分类。 

大小为 2 的召唤兽必须满足以下条件：基础贡献为 2 后，奖励总和为 4。这迫使配对组合必须恰好是值总和为 6 的组合：(1,5)、(2,4) 和 (3,3)，因为只有这些配对才能提供所需的奖励结构。 

尺寸为 3 的召唤必须满足基数 3 和奖励 3，这意味着这些值的总和必须为 6，给出组合 (1,1,4)、(1,2,3) 和 (2,2,2)。 

存在更大的群体，但在效率方面确实更差，因为他们每产生结果消耗更多的烈酒。 由于我们的目标是最大化召唤数量，因此我们总是优先选择较小的有效组。 

该算法进行如下：

 1. 尝试使用可用计数形成尽可能多的有效 2 号传票。 我们贪婪地匹配（1,5），然后（2,4），然后（3,3），从库存中减去用过的烈酒。 每一场成功的比赛都会产生一个结果并消耗恰好两个灵魂。 
2. 在用尽所有可能的对后，我们转向尺寸 3 的召唤。 我们首先贪婪地形成 (2,2,2)，因为它只使用一种类型，然后是 (1,2,3)，然后是 (1,1,4)，相应地消耗计数。 
3. 如果由于分布限制而遗漏了任何较小的结构，则仅在必要时才会考虑较大的模式，但它们是占主导地位的，并且不会增加最终计数超出早期步骤实现的范围。 

这种排序背后的原因是，每个有效召唤都对应于固定数量的消耗灵魂，并且最小化每次召唤的消耗可以最大化从可用池中提取的不相交有效结构的数量。 

### 为什么它有效

 每个有效召唤对应最多 6 个灵魂的多重集合，其调整后的总和等于 6。任何此类多重集合都由其大小和组成完整描述，并且所有组成都由上面列举的模式涵盖。 由于每次召唤都是独立的，并且消耗不相交的资源，因此最大化召唤数量相当于贪婪地提取尽可能多的小有效模式。 由于在不增加总资源消耗的情况下无法将更大的模式拆分为更多的召唤，因此通过增加大小的贪婪排序保留了最优性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve_case(a):
    c1, c2, c3, c4, c5 = a
    ans = 0

    # size-2 patterns: (1,5), (2,4), (3,3)
    x = min(c1, c5)
    ans += x
    c1 -= x
    c5 -= x

    x = min(c2, c4)
    ans += x
    c2 -= x
    c4 -= x

    x = c3 // 2
    ans += x
    c3 -= 2 * x

    # size-3 patterns: (2,2,2)
    x = c2 // 3
    ans += x
    c2 -= 3 * x

    # (1,2,3)
    x = min(c1, c2, c3)
    ans += x
    c1 -= x
    c2 -= x
    c3 -= x

    # (1,1,4)
    x = min(c4, c1 // 2)
    ans += x
    c4 -= x
    c1 -= 2 * x

    return ans

def main():
    t = int(input())
    out = []
    for _ in range(t):
        a = list(map(int, input().split()))
        out.append(str(solve_case(a)))
    print("\n".join(out))

if __name__ == "__main__":
    main()
```该实现直接对模式提取顺序进行编码。 首先处理配对编队，因为它们是创建召唤的最节省资源的方式。 之后，使用三元组来消耗剩余的结构。 每个操作都是对计数的直接最小或贪心减法，确保每个测试用例的处理时间恒定。 

一个常见的微妙之处是每个模式必须按固定的顺序应用。 对它们重新排序可能会破坏最优性，因为早期的选择会影响更多受限模式（如（1,2,3））的烈酒的可用性。

 ## 边缘情况

 当仅存在类型 5 和类型 1 时，算法首先尝试将它们配对。 每个 (1,5) 都会消耗两种资源，但不会创建无效的部分结构，因此剩余的 5 或 1 根本无法形成任何有效的配置并被正确忽略。 

当仅存在类型 3 时，算法会在尝试三元组之前优先考虑形成 (3,3) 对，以确保不会在不可能的部分配置中浪费单个 3。 剩下的单个3不能组成有效召唤，这符合每个有效组需要至少两个兼容元素的规则。 

当计数非常大时，例如每种类型 10^9，算法永远不会迭代每个项目。 所有运算均通过整数除法和最小值运算执行，因此缩放不会影响正确性或性能。
