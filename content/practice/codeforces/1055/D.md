 ## 解决方案
 ## 问题理解

 我们得到了几对字符串，其中每对描述了变量名称当前的外观以及在单个全局重构操作后它应该如何外观。 重构工具以非常具体的方式工作：我们选择两个字符串，称它们为$s$和$t$，然后对于每个变量名称，我们从左侧扫描。 如果$s$显示为子字符串，仅其第一次出现被替换为$t$。 如果它没有出现，则该字符串保持不变。 

任务是判断是否存在单对$(s, t)$这样，将此规则同时应用于所有初始名称就会准确地产生所有目标名称。 如果可能的话，我们必须输出一对有效的。 

关键的困难在于，相同的子字符串替换必须立即解释每个更改的字符串，同时保持未更改的字符串完全不变。 这创建了全局一致性约束：相同的模式必须负责所有转换。 

约束允许最多 3000 个长度为 3000 的字符串，因此总字符数可以达到大约$9 \times 10^6$。 这排除了任何尝试所有子字符串或独立模拟许多候选者的替换的方法。 二次或更差的每串方法会太慢。 

一个天真的错误是假设我们可以选择一个字符串与其目标不同的任何位置并构建$s$仅从该字符串本地获取。 这会失败，因为其他字符串可能需要不同的上下文。 

例如，假设一对是：```
abcde -> abXde
```另一个是：```
abfde -> abYde
```如果我们选择$s = c$从第一个开始，第二个就没有任何作用了。 如果我们选择$s = d$，它可能会意外出现在未更改的字符串中的非预期位置，从而导致错误的替换。 

当候选子字符串出现在“好”字符串的较早位置时，会发生另一个微妙的失败。 即使它在预期位置正确转换，该规则也仅替换第一次出现的情况，因此较早出现的情况会默默地破坏正确性。 

## 方法

 蛮力的想法是尝试所有可能的选择$s$作为任何初始字符串的任何子字符串，对于每个候选者模拟所有字符串的转换，并检查我们是否可以通过选择适当的匹配所有目标$t$。 这在概念上是正确的，因为任何有效的解决方案都必须使用至少一个输入字符串中存在的某个子字符串。 然而，有$O(nL^2)$子串，以及每次模拟的成本$O(nL)$，这会导致不可行的$O(n^2 L^3)$最坏的情况。 

关键的观察结果是，转换完全由任何字符串与其目标不同的第一个位置决定。 此位置强制子字符串必须在所有受影响的字符串中“开始”，因为任何较早的位置都不会影响已更改的字符串，或者会错误地影响未更改的字符串。 

从这个锚点位置，我们可以尝试构建最小的子串$s$在所有修改过的字符串中保持一致，并同时导出$t$来自目标字符串。 一旦我们确定了起始位置，子字符串就可以贪婪地扩展，同时保持所有更改对的一致性。 

构建候选者后$(s, t)$，最后一步是验证：确保每个更改的字符串在“首次出现替换”规则下产生其目标，并确保每个未更改的字符串不包含$s$任何地方。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 |$O(n^2 L^3)$|$O(L)$| 太慢了 |
 | 锚定子串构造|$O(nL)$|$O(nL)$| 已接受 |

 ## 算法演练

 我们专注于最优的建设策略。 

首先我们找到最早的位置$p$使得至少存在一个索引$i$和$w_i[p] \ne w'_i[p]$。 这个位置是任何转变必须生效的第一个位置。 任何有效的子字符串$s$必须包含这个位置，否则不会改变不同的字符。 

其次，我们收集所有索引$i$在哪里$w_i \ne w'_i$。 这些是重构操作必须更改的“活动”字符串。 

第三，我们尝试从位置开始构建候选子串$p$。 我们设定$s$最初到$w_i[p]$对于任何活动字符串$i$，因为所有活动字符串都必须同意该字符。 如果他们不同意，就没有解决方案。 

第四，我们尽可能将子串向右延伸。 在每个扩展步骤中，我们要求所有活动字符串仍然共享原始字符串中的相同字符$s$，并且在目标字符串中共享相同的字符$t$。 这确保了两者$s$和$t$在所有转换中保持一致。 

第五，一旦我们停止扩展，我们定义$s$从原始字符串和$t$在此间隔内的目标字符串。 

第六，我们验证候选人。 对于每个活动字符串，我们模拟替换第一次出现的效果$s$。 关键的检查是第一次出现必须恰好从位置开始$p$，否则将替换不同的事件并且结果会出现分歧。 

第七，对于每个不活动的字符串（其中$w_i = w'_i$），我们确保$s$没有出现在字符串中的任何位置。 否则，操作将错误地修改它。 

如果所有检查都通过，我们输出该对$(s, t)$。 否则，答案是不可能的。 

### 为什么它有效

 该结构迫使替换窗口与第一个全局失配位置对齐。 任何有效的解决方案都必须影响至少一个不同的字符，并且最早的此类位置固定左边界。 一旦这个边界被固定，所有改变的字符串之间的一致性就会强制子字符串的唯一扩展。 验证步骤保证不存在意外的早期事件，从而保留“仅首次出现”规则的正确性。 这可以防止未更改的字符串中的过度应用以及已更改的字符串中的未对齐替换。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    w = [input().strip() for _ in range(n)]
    t = [input().strip() for _ in range(n)]

    changed = []
    first_diff = None

    for i in range(n):
        if w[i] != t[i]:
            changed.append(i)
            if first_diff is None:
                for j in range(len(w[i])):
                    if w[i][j] != t[i][j]:
                        first_diff = j
                        break

    if not changed:
        print("YES")
        print("a")
        print("a")
        return

    p = first_diff

    # build s and t by extending to the right
    l = p
    r = p

    m = len(w[changed[0]])

    while True:
        if r + 1 >= m:
            break

        ok = True
        for i in changed:
            if w[i][r + 1] != w[changed[0]][r + 1]:
                ok = False
                break
            if t[i][r + 1] != t[changed[0]][r + 1]:
                ok = False
                break

        if not ok:
            break

        r += 1

    s = w[changed[0]][l:r+1]
    tt = t[changed[0]][l:r+1]

    def apply(s, tt, x):
        pos = x.find(s)
        if pos == -1:
            return x
        return x[:pos] + tt + x[pos+len(s):]

    # validate changed
    for i in changed:
        if apply(s, tt, w[i]) != t[i]:
            print("NO")
            return

    # validate unchanged
    for i in range(n):
        if i not in changed:
            if s in w[i]:
                print("NO")
                return

    print("YES")
    print(s)
    print(tt)

if __name__ == "__main__":
    solve()
```该实现首先识别所有必须更改的字符串，并找到其中最早的不匹配位置。 该位置锚定候选子串。 仅当所有更改的字符串都与原始字符和目标字符一致时，扩展循环才会增长子字符串，从而确保一致性。 

这`apply`函数通过使用精确地反映编辑器行为`find`，它返回从左侧开始的第一个匹配项。 这一点至关重要，因为即使一个子字符串出现多次，也只有最早出现的才会受到影响。 

最后，验证确保两个方向都是安全的：更改的字符串在转换后必须完全匹配，而未更改的字符串不得包含任何模式。

 ## 边缘情况

 一种棘手的情况是候选子字符串在未更改的字符串中出现较早。 该算法通过显式拒绝任何解决方案来处理此问题$s$出现在应保持不变的字符串中。 这可以防止由于“仅首次出现”规则而导致的意外修改。 

另一个微妙的情况是多个不同的字符串建议子字符串的不同扩展名。 该结构通过要求所有活动字符串在扩展期间逐个字符一致来强制一致性。 如果发生任何分歧，扩展会立即停止，确保我们不会过度拟合一根字符串而牺牲其他字符串。 

最后的边缘情况是当第一个不匹配位置导致子字符串不会唯一触发更改的字符串中的替换时。 验证步骤通过模拟实际的替换行为而不是依赖于结构假设来捕获这一点。
