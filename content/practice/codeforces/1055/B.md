 ## 解决方案
 ## 问题理解

 我们在一系列位置上维护一系列头发长度，并且随着某些位置的增长，系统会随着时间的推移而演变。 在任何时刻，我们都可以想象一个假设的理发操作：头发严格长于固定阈值的每个位置$l$必须精确地减少到$l$，但是理发工具有一个限制。 一个操作只能作用于一个连续的段，并且只有当该段中的每个位置当前超过$l$。 理发师的目标是最小化所需的此类段操作的数量，以使所有值最多降至$l$。 每个类型零的查询都要求当前状态下的最小操作数，而类型一的查询会永久增加一个位置。 

关键对象不是确切的值本身，而是当前索引超过的模式$l$。 任何指数等于或低于$l$与切割无关，因为它不需要接触并且也会破坏片段。 重要的是有多少连续的索引块满足$a_i > l$，因为每个这样的块都可以在一次操作中删除。 

这些限制推动了在线解决方案的复杂性接近线性。 高达$10^5$更新和查询，为每个类型零查询从头开始重新计算答案将需要每次扫描整个数组，从而导致$O(nm)$，这太慢了。 

更新只会增加价值这一事实引起了一个微妙的问题。 这种单调性至关重要。 已经在上面的位置$l$永远不会再改变其状态，并且位置低于或等于$l$可能会恰好翻转一次，从非活动状态变为活动状态，但永远不会返回。 

一个天真的错误是通过仅扫描更改的区域或通过尝试维护全局段的端点而不正确处理合并来重新计算段数。 例如，考虑如下配置$[1, 5, 1, 5]$和$l = 3$。 正确答案是两段。 如果我们错误地假设每个活动位置独立贡献一个操作，我们将输出四个，完全忽略邻接。 

另一种失败模式来自于忘记激活单个位置可以合并两个先前独立的段。 例如，如果我们在指数上有活跃头寸$1$和$3$，答案是两段。 如果索引$2$变为活动状态，正确答案变为 1，但是仅递增计数器的本地更新方法将错过此减少。 

## 方法

 蛮力策略很简单。 对于类型为零的每个查询，我们扫描整个数组并构建一个布尔数组，指示每个位置是否超过$l$。 然后，我们计算真值紧随假值出现的次数，如果存在第一个活动段，则加一。 这可以正确计算连续块的数量，但每个查询都会花费$O(n)$，使得总复杂度$O(nm)$，这对于最坏的情况来说太慢了。 

关键的观察结果是每个位置的状态都是二元的，并且具有一次性转换。 一旦某个值跨越$l$，它仍然在上面$l$永远。 这意味着我们不是在处理任意的动态间隔，而是在处理点的增量激活。 我们维护的结构是一条线上动态活动索引集中的连接分量的数量。 每次激活要么创建一个新组件，要么合并两个现有组件。 

这减少了在线图上点插入下维护连接组件数量的问题。 由于邻接是一维的，不相交的集合并集结构或简单的邻居检查就足够了。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 |$O(nm)$|$O(n)$| 太慢了 |
 | 增量组件跟踪 |$O(n + m)$|$O(n)$| 已接受 |

 ## 算法演练

 我们将每个索引解释为不活动（值$\le l$）或活动（值$> l$）。 我们维护存在多少个连续的活动块。 

1.初始化一个数组`active`在哪里`active[i] = 1`如果$a_i > l$，否则为 0。 

这代表了段的初始结构。 
2. 遍历数组一次并计算有多少个索引开始一个新的活动段。 

如果某个位置处于活动状态并且它是第一个索引或前一个索引处于非活动状态，则该位置将启动一个段。 

这给出了初始段数。 
3. 维护一个变量`segments`存储当前活动块的数量。 
4.对于每个增加位置的更新查询$p$，首先检查该位置是否已经激活。 如果是的话，结构上不会发生任何变化。 
5. 如果该位置第一次变为活动状态，请将其标记为活动状态并确定它如何影响连接。 

如果两个邻居都处于非活动状态，则该位置会创建一个新的隔离块并增加`segments`一个。 

如果恰好有一个邻居处于活动状态，则它会附加到该块并且不会更改计数。 

如果两个邻居都处于活动状态，则会将两个现有块合并为一个，从而减少`segments`一个。 
6. 对于每个类型为零的查询，输出当前值`segments`。 

关键的不变量是`segments`始终等于索引集中连接分量​​的数量$i$这样$a_i > l$。 每次更新都会保留此不变性，因为激活仅影响本地邻接。 除非激活边界点，否则不活动区域不会分裂或合并，并且通过检查两个邻居可以完全捕获该效果。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m, l = map(int, input().split())
    a = list(map(int, input().split()))

    active = [0] * n
    segments = 0

    for i in range(n):
        if a[i] > l:
            active[i] = 1
            if i == 0 or active[i - 1] == 0:
                segments += 1

    for _ in range(m):
        tmp = input().split()
        t = int(tmp[0])

        if t == 0:
            print(segments)
        else:
            p = int(tmp[1]) - 1
            d = int(tmp[2])

            was = active[p]
            a[p] += d

            if was == 0 and a[p] > l:
                active[p] = 1

                left = active[p - 1] if p > 0 else 0
                right = active[p + 1] if p < n - 1 else 0

                if left and right:
                    segments -= 1
                elif not left and not right:
                    segments += 1
                # else: merges with one side, no change

solve()
```该解决方案首先将初始数组转换为二进制激活数组。 然后，它利用新段恰好在某个位置处于活动状态而前一个位置不处于活动状态时开始这一事实，计算单次传递中活动段的数量。 

每次更新最多将一个位置从非活动修改为活动。 代码在更新段结构之前会仔细检查此转换。 邻居检查就足够了，因为只有局部邻接才能改变一维线中连通分量的数量。 

## 工作示例

 考虑示例输入：

 初始状态是$[1, 2, 3, 4]$和$l = 3$。 只有索引 4 处于活动状态，因此只有一个段。 

| 步骤| 数组 | 活跃模式| 细分 |
 | ---| ---| ---| ---|
 | 开始 | 1 2 3 4 | 1 2 3 4 0 0 0 1 | 0 0 0 1 1 |
 | + (2, +3) | 1 5 3 4 | 1 5 3 4 0 1 0 1 | 0 1 0 1 2 |
 | 查询 | - | - | 2 |
 | + (1, +3) | 4 5 3 4 | 4 5 3 4 1 1 0 1 | 1 1 0 1 2 |
 | 查询 | - | - | 2 |
 | + (3, +1) | 4 5 4 4 | 4 5 4 4 1 1 1 1 | 1 1 1 1 1 |
 | 查询 | - | - | 1 |

 第一次更新在索引 2 处创建一个新的隔离活动位置，产生两个段。 第二次更新将左侧合并到现有块中，但不更改段数。 最终的更新填补了所有活动区域之间的空白，将所有内容合并到一个连续的块中。 

该迹线表明，只有边界激活才重要； 已经活跃的区域内的内部增长不会影响结构。 

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 |$O(n + m)$| 每个索引最多激活一次，并且每个查询在常数时间内处理 |
 | 空间|$O(n)$| 我们存储每个位置的激活状态 |

 该算法完全符合限制，因为每个操作的时间都是恒定的，并且有意义的状态变化的总数受下式限制：$n$。
