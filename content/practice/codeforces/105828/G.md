 ## 解决方案
 ## 问题理解

 我们有固定的书籍顺序，每本书都有阅读时间和必须完成的最晚可接受的日期。 阅读顺序不灵活：书籍按照给定的顺序从头到尾阅读。 每本书必须在一天内完全完成，一天可能包含几本书。 

任务是将每本书分配到 1 到 m 之间的一天，尊重每本书 i 必须分配到不晚于 di 的某一天，同时保持顺序。 一旦分配了书籍，每天的工作量就等于分配给它的书籍 ti 的总和。 在所有有效的作业中，我们希望最小化每日最大工作量。 找到这样的最佳时间表后，我们必须输出尽可能小的最大工作量，以及每天分配给该天的最后一本书，如果当天是空的，则输出 0。 

对于书籍和天数，约束都达到 3·10^5，因此任何尝试所有分区或对天数和位置进行动态编程的解决方案都将失败。 解决方案需要大致线性或线性算数行为，通常对答案进行二分搜索和贪婪可行性检查。 

一个微妙的困难是截止日期与分区相互作用。 仅仅将序列分割成有界总和的块是不够的。 即使一个块符合当前的工作量限制，如果延迟它使一本书超过了截止日期，也可能是非法的。 

当一本书的 ti 大于测试的每日限制时，就会出现一种边缘情况。 例如，如果一本书需要时间 10，但我们测试限制为 7，则无论截止日期如何，都不可能进行作业。 另一个有问题的情况是当最后期限迫使提前分裂时。 如果书籍的 di 值较小，则仅考虑总和的天真贪婪可能会为时过晚分配一本书，即使后来的容量存在。 

## 方法

 一个蛮力的想法是将其视为 m 天的分区问题，并尝试所有方法将序列切成 m 段，然后检查每个段是否遵守截止日期并计算其最大总和。 这是正确的，因为每个有效的调度都对应于序列的一个分段。 然而，这种分割的数量随着 n 和 m 的组合而增长，甚至限制为 m 切割也会导致大约$\binom{n}{m}$的可能性，这远远超出了可行的计算。 

关键的结构观察是，我们不是每天选择任意分配，而是构建固定序列的单调分段。 如果我们固定一个候选最大每日工作量S，我们可以贪婪地检查是否可以按顺序处理书籍，形成最早可能的有效日块。 问题的单调性意味着如果 S 太小，则失败，如果 S 足够大，则成功，因此可行性是单调的，可以进行二分搜索。 

贪婪的可行性检查之所以有效，是因为当我们被迫按顺序分配书籍时，将书籍推迟一天只会使未来的约束变得更加困难。 所以每一天都应该在不违反容量S的情况下尽可能地填满，并且只有在必要时我们才开始新的一天。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力分割| 指数| O(1) | O(1) | 太慢了 |
 | 二分查找+贪心检查| O(n log A) | O(n log A) | O(n) | 已接受 |

 这里 A 是所有 ti 值的总和，它限制了答案。 

## 算法演练

 我们搜索最小可能值 S，以便我们可以将书籍分配到 m 个有序天，而不超过 S 每天的总时间，并且不违反截止日期。 

1.我们在最大单个ti和所有ti的总和之间二分搜索S。 下限是必要的，因为没有哪一天可以容纳比 S 更大的书，并且上限始终是可行的，通过将每次约束放松都放在一天中。 
2.对于固定的S，我们模拟从左到右分配书籍。 我们维护当天的索引和当天的当前累计时间。 
3. 当考虑书 i 时，如果 ti 超过 S，我们立即得出结论 S 无效。 这是因为没有有效的作业可以将这本书放在任何地方。 
4.如果将书i添加到当天不超过S并且不违反我们不超过di的约束，我们将其分配给当天。 
5. 如果添加书 i 会超过 S，我们必须关闭当天并移至第二天。 在此之前，我们确保尚未到达本书允许的最后日期； 否则，就没有有效的位置。 
6. 一天结束后，我们开始新的一天并将书籍分配在那里。 我们再次检查新的日指数是否超过 di，这将使 S 无效。 
7. 如果我们在 m 天内处理完所有书籍，则 S 是可行的。 
8. 在二分搜索找到最小可行 S 后，我们再次重新运行贪婪模拟，记录每天分配给它的最后一本书索引。 

关键的微妙之处在于，贪婪的人总是会在几天之内创造最早的终点。 这很重要，因为任何将书本推迟一天的替代作业只会降低未来的灵活性，而不会增加它。 

正确性取决于一个不变量：处理完前 i 本书后，贪心程序在遵守截止日期的同时使用 S 下所有有效作业中的最小可能天数。 如果存在任何分配，这种贪婪不会超过它，因为它只会在容量或可行性的迫使下打开新的一天。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def check(S, n, m, t, d):
    day = 1
    used = 1
    cur = 0

    for i in range(n):
        if t[i] > S:
            return False

        if cur + t[i] <= S:
            cur += t[i]
        else:
            day += 1
            if day > m:
                return False
            cur = t[i]

        if day > d[i]:
            return False

    return True

def build(S, n, m, t, d):
    day = 1
    cur = 0
    last = [0] * (m + 1)

    for i in range(n):
        if cur + t[i] <= S:
            cur += t[i]
        else:
            last[day] = i
            day += 1
            cur = t[i]

        last[day] = i

    return last

def solve():
    n, m = map(int, input().split())
    t = []
    d = []
    for _ in range(n):
        ti, di = map(int, input().split())
        t.append(ti)
        d.append(di)

    lo = max(t)
    hi = sum(t)

    while lo < hi:
        mid = (lo + hi) // 2
        if check(mid, n, m, t, d):
            hi = mid
        else:
            lo = mid + 1

    ans = lo
    last = build(ans, n, m, t, d)

    print(ans)
    for i in range(1, m + 1):
        print(last[i] if i <= m else 0)

if __name__ == "__main__":
    solve()
```该解决方案分为可行性检查和重建阶段。 检查器强制执行两个约束：每日容量和截止日期。 重建使用相同的贪婪打包逻辑，确保与所选最优 S 的一致性。 

一个微妙的实现细节是重建必须反映与检查相同的决策结构； 否则，即使值 S 是正确的，所得划分也可能不对应于有效的最优解。

 ## 边缘情况

 当一本书的 ti 大于任何可行的每日限制时，就会出现严重的边缘情况。 在这种情况下，每个低于 ti 的候选 S 都必须立即失败。 该算法在进行任何分配之前在可行性检查中捕获这一点，从而防止浪费模拟。 

当 di 非常小时，会出现另一种边缘情况，例如对于所有 i，di = i。 这迫使每本书的放置时间不晚于其位置，实际上要求几乎每天一本书的行为。 当容量不足时，贪婪自然会提前打开新的天，而可行性条件day > di可以防止无效分配的泄漏。 

第三种情况是m大于n时。 这里很多天都没有使用，重建必须正确输出尾随零。 由于贪婪仅在需要时提前天数，因此未使用的天数保持不变，并在输出数组中默认为零。
