 ## 解决方案
 ## 问题理解

 我们正在模拟书架上越来越多的书籍。 每本书都有一个唯一的标识符，我们只将书籍添加到当前排列的最左端或最右端。 随着时间的推移，这会产生所有插入书籍的固定线性顺序。 

除了插入之外，我们还收到询问有关书架上已有的特定书籍的查询。 对于这样的查询，我们必须计算需要从左端或右端删除多少次，才能使该目标书在其中一端暴露出来。 我们实际上并没有永久删除任何内容，这纯粹是关于当前排序的假设问题。 

从约束中观察到的关键是，有多达 200,000 个操作，因此我们无法使用数组并在两端重复插入来重建或模拟每个查询的完整列表，因为在最坏的情况下，在 Python 列表左侧插入或从中间删除等操作会降级为二次行为。 我们需要一种支持 O(1) 两端插入和 O(1) 位置查询的表示。 

一个微妙的点是结构不是任意的。 每本书都只插入一次，而且只在极端情况下插入。 这保证了最终的顺序完全由插入的顺序决定，并且每本书在该演变的顺序中都有一个固定的位置。 

值得强调的边缘案例：

 如果所有插入都位于同一侧，例如所有“L”，则该结构是一个简单的反向列表。 简单的模拟仍然有效，但关于“向左意味着指数增加”的错误假设可能会破坏逻辑。 

如果在插入第一本书后立即发生查询，则答案必须始终为零，因为该书已经是最左边和最右边的。 

如果一本书在多次交替插入后位于中间，我们必须确保以最终的固定顺序（而不是插入顺序）正确测量到两端的距离。 

## 方法

 暴力方法将维护一个代表架子的明确列表。 对于每次插入，我们将为“L”执行 list.insert(0, id) 或为“R”执行追加操作。 对于每个查询，我们将扫描列表以查找该书的索引，然后将其到两端的距离计算为 min(index, n - 1 - index)。 这是正确的，但太慢了。 

问题的出现是因为查找一本书的索引是 O(n)，而且我们可能会执行多达 200,000 次。 这导致 O(nq)，远远超出了限制。 

关键的结构见解是我们永远不需要显式存储完整列表。 我们只需要知道每本书在最终排序中的相对位置即可。 由于每次插入都处于极值，因此我们可以为每本书分配一个数字坐标，表示其在数轴上的位置。 每个新的左插入都会比之前的所有插入获得更小的坐标，而每个右插入都会获得更大的坐标。 这将架子变成一组保持顺序的整数。 

一旦每本书有了一个坐标，回答查询就简化为查找在该坐标排序中严格位于其左侧和右侧的书籍有多少本。 我们不是动态计数，而是使用类似前缀的计数器来跟踪每本书在每一侧之前插入了多少本书。 

一个更简单的等效视图是，我们使用类似双端队列的模型在最终序列中为每本书分配一个索引，但我们从不显式构建双端队列。 相反，我们模拟两个向外扩展的指针。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(n²) | O(n) | 太慢了|
 | 与哈希图的坐标分配 | O(n) | O(n) | 已接受 |

 ## 算法演练

我们维护两个增长指针，表示隐式数组中最左边和最右边占用的位置。 我们还维护一个字典，将每本书 id 映射到其最终位置。 

1. 初始化两个计数器，left = 0 和 right = 0。它们代表虚拟坐标系中货架的当前边界。 
2.维护一个字典pos，其中pos[id]存储每本书的指定坐标。 
3. 当处理“L id”查询时，我们将 left 减 1 并赋值 pos[id] = left。 这确保每个新的左插入都严格小于所有先前的位置。 
4. 当处理“R id”查询时，我们将 right 加 1 并分配 pos[id] = right。 这确保每个正确的插入都严格大于所有先前的位置。 
5. 当处理“?id”查询时，我们计算将这本书带到任一端需要多少步骤。 如果 pos[id] 为 k，则：

 它左边的书数是 k - left，右边的书数是 right - k。 答案是min(k - 左，右 - k)。 

原因是左和右跟踪极端坐标，因此该坐标系中的差异与最终货架顺序中的距离完全对应。 

### 为什么它有效

 不变的是，在每次操作之后，所有书籍都位于不同的整数坐标处，并且书架上书籍的顺序与递增的坐标值完全对应。 每个“L”插入都会创建一个新的最小坐标，每个“R”插入都会创建一个新的最大坐标。 因此，货架顺序始终与按坐标排序一致，并且通过坐标差正确测量到末端的距离。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    q = int(input())
    pos = {}
    left = 0
    right = 0

    first = True

    for _ in range(q):
        parts = input().split()
        if parts[0] == 'L':
            x = int(parts[1])
            if first:
                pos[x] = 0
                left = right = 0
                first = False
            else:
                left -= 1
                pos[x] = left

        elif parts[0] == 'R':
            x = int(parts[1])
            if first:
                pos[x] = 0
                left = right = 0
                first = False
            else:
                right += 1
                pos[x] = right

        else:
            x = int(parts[1])
            k = pos[x]
            print(min(k - left, right - k))

if __name__ == "__main__":
    solve()
```该解决方案避免维护任何显式列表。 相反，它为每本书分配一个坐标，就像我们扩展一条数轴一样。 第一本插入的书很特殊，因为两个边界都未定义，因此我们在该点将左侧和右侧初始化为零。 

每次插入仅更新一个端点。 查询完全依赖于存储坐标和当前边界之间的算术差异，因此它们的复杂度为 O(1)。 

一个微妙的细节是干净地处理第一次插入。 如果没有正确初始化两个边界，坐标系将会漂移或产生不正确的距离。

 ## 边缘情况

 一个边缘情况是单次插入后第一个操作已经是查询。 对于输入：```
L 10
? 10
```系统分配 pos[10] = 0 并且 left 和 right 都为 0。查询计算出 min(0,0) = 0，这正确地反映了书已经在两端。 

另一种情况是交替插入，两侧不对称扩展。 例如：```
L 1
R 2
L 3
R 4
? 2
```处理后，根据排序变化，最左边的坐标变为 3，最右边的坐标变为 4。 该查询仍然仅通过坐标间隙（而不是插入顺序）正确测量距离。 

最后一个微妙的情况是，许多左插入发生在任何右插入之前。 左边界移动得很远，但由于所有计算都是相对差异，因此不会发生溢出或排序不一致，并且距离保持正确。
