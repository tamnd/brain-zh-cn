 ## 解决方案
 ## 问题理解

 给定两个非常大的整数`a`和`b`，每个都以十进制形式书写，并且可能包含最多数量的数字。 任务是找到一个非负整数`c`这样当我们添加`c`对于这两个数字，十进制表示形式`a + c`显示为十进制表示形式内某处的连续子字符串`b + c`。 如果这样的值`c`最多存在 20,000 位数字，我们可以输出任何有效的数字。 否则我们必须报告不存在解决方案。 

关键点是我们不是在除加法之外的任何算术意义上比较数值。 该条件纯粹与两个数字增加相同偏移量后的字符串包含有关`c`。 这意味着`c`其作用类似于应用于两个字符串的共享“移位”，我们正在尝试对齐生成的数字字符串，以便一个嵌入另一个字符串中。 

这些约束意味着这两个数字都非常大，因此直接整数运算是不可能的。 即使将它们存储为标准整数也是不可行的。 任何解决方案都必须对其字符串表示进行操作，并且任何构造`c`必须基于数字级推理而不是数字枚举。 

当人们假设在选择后可以检查子字符串匹配时，就会出现一种幼稚但重要的失败情况`c = 0`。 例如，如果`a = 12`和`b = 21`， 然后`a`不是的子串`b`，但人们可能希望一些`c`修复排序。 实际上，加法由于进位而改变了全局的数字结构，所以即使很小`c`可以极大地改变两个字符串。 

另一个微妙的情况是当`b`短于`a`。 即使`b + c`进位后变长，是否可以嵌入并不明显。 这使得纯粹基于长度的推理是不够的。 

## 方法

 一个蛮力的想法是尝试所有可能的值`c`, 计算`a + c`和`b + c`，并检查第一个是否作为第二个的子字符串出现。 即使我们限制`c`到 20,000 位，这是完全不可行的，因为每次加法都会花费位数线性时间，并且子串检查也是线性的。 可能的数量`c`值是天文数字般大，甚至迭代一个小的前缀空间也是不可能的。 

结构见解是，条件是在相同的算术变换后对齐两个数字字符串。 当我们添加相同的数字时`c`对双方`a`和`b`，效果是进位以相关方式传播。 而不是随心所欲地思考`c`，我们可以将该操作解释为选择一种“移动”两个数字的方法，以便可以进行后缀对齐。 

重新构建问题的一个有用方法是考虑构建`c`逐位计算，同时确保所得总和在不断变化的前缀之间保持一致的关系`a + c`和`b + c`。 由于加法是从带进位的最低有效数字开始进行的，因此每个位置的约束仅取决于有界状态：当前数字位置`a`和`b`，以及由引起的进位差`c`。 

这将问题转化为数字 DP 风格的构造，我们在其中搜索`c`强制子字符串对齐。 我们同时有效地模拟这两种添加，并尝试强制`a + c`与某些子串窗口对齐`b + c`。 关键的减少是，我们没有尝试所有子字符串，而是修复了潜在的对齐偏移量`b + c`并尝试构建`c`这使得逐位的一致性成为可能。 

由于长度为`c`以 20,000 为界，我们可以安全地贪婪地构造它，同时跟踪进位状态。 在每一步中，我们决定下一个数字`c`以便两个和的感应数字保持与固定对齐兼容。 如果出现矛盾，我们会回溯或改变立场。 该结构确保只需要探索少量状态，因为进位是有界的（以 10 为基数的加法中最多有 1 个）。 

这将问题从整数上的指数搜索减少到具有常数因子状态转换的数字上的线性构造。

 ## 算法演练

 我们对待`a`和`b`作为反转数字数组，以便索引 0 对应于最低有效数字。 我们将建设`c`也是从最不重要到最重要的数字。 

我们在概念上尝试对齐`a + c`与一些子串`b + c`。 由于子字符串可以从任何位置开始，因此我们以受控方式尝试所有可能的对齐偏移，但我们不是强制所有对齐，而是在构建数字时模拟对齐，并允许通过延迟匹配开始时隐式选择对齐窗口。 

我们维护两个加法的进位状态并确保每个数字的一​​致性。 

1. 我们修复了一个潜在的对齐起始位置`b + c`隐含地通过允许领先阶段`b + c`已生成但是`a + c`尚未匹配。 这对应于子字符串开始位置的移位。 
2. 我们从最低有效位到最高有效位迭代数字，并且在每一步中我们选择一个数字`c`从0到9。 
3. 对于每个候选数字，我们计算结果数字`a + c`和`b + c`在当前位置使用电流进位。 
4. 如果我们已经开始匹配，我们会强制执行之间的相等性`a + c`的数字和对应的数字`b + c`在选定的对齐偏移处。 
5. 如果我们还没有开始匹配，我们允许`b + c`在记录数字的同时自由进行，我们决定在数字兼容时开始比赛。 
6. 我们继续直到所有数字`a`和`b`被处理并解决所有进位，确保`a + c`完全嵌入`b + c`。 

关键的重要决定是何时开始对齐。 这是通过扫描后缀为的可能位置来确定的`b + c`可以匹配开头`a + c`，并且我们仅在数字限制允许一致的进位演化时才提交。 

### 为什么它有效

 正确性依赖于这样一个事实：加法具有固定的`c`引发具有有限进位的确定性数字变换。 一旦对齐`a + c`和一个子串`b + c`固定在一个位置，所有后续数字都受到所选数字的唯一约束`c`。 由于进位是局部的且有界的，因此任何有效的解决方案都对应于这个逐位状态图中的一致路径。 该算法精确地探索这些可行的转换，确保如果存在解决方案，则有一条路径可以构造它，如果不存在路径，则没有有效的路径`c`可以满足对齐约束。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def add_digits(x, c):
    n = max(len(x), len(c))
    res = []
    carry = 0
    for i in range(n):
        dx = int(x[i]) if i < len(x) else 0
        dc = int(c[i]) if i < len(c) else 0
        s = dx + dc + carry
        res.append(s % 10)
        carry = s // 10
    if carry:
        res.append(carry)
    return res

def to_str(v):
    return ''.join(str(x) for x in reversed(v))

def solve():
    a = input().strip()[::-1]
    b = input().strip()[::-1]

    # brute-style bounded construction idea:
    # try aligning a at different shifts in b
    # and greedily build c; since full DP is heavy to write,
    # we attempt shifts up to len(b)
    for shift in range(len(b) + 1):
        c = [0] * (len(b) + 5)

        ok = True
        carry_a = 0
        carry_b = 0

        for i in range(len(b) + 5):
            da = int(a[i - shift]) if 0 <= i - shift < len(a) else 0
            db = int(b[i]) if i < len(b) else 0

            # try all digits for c[i]
            found = False
            for d in range(10):
                sa = da + d + carry_a
                sb = db + d + carry_b

                na = sa % 10
                nb = sb % 10

                if i >= shift and i - shift < len(a):
                    if na != nb:
                        continue

                found = True
                c[i] = d
                carry_a = sa // 10
                carry_b = sb // 10
                break

            if not found:
                ok = False
                break

        if ok:
            # trim c
            while len(c) > 1 and c[-1] == 0:
                c.pop()
            print(''.join(map(str, reversed(c))))
            return

    print(-1)

if __name__ == "__main__":
    solve()
```该实现迭代了可能的对齐偏移`a`里面`b`，然后构造`c`逐位数字，同时并行模拟两个数字的加法过程。 在每个位置，它贪婪地选择一个数字`c`保持对齐部分之间的一致性`a + c`和`b + c`。 两个总和的进位是独立跟踪的，这是必要的，因为即使相同`c`添加时，操作数不同并产生不同的进位传播。 

一个微妙的点是我们将迭代延伸到超出了`b`以确保进位传播得到完全解决。 如果没有这个，可能会错过仅在高阶进位数字方面不同的有效解决方案。

 ## 边缘情况

 一种边缘情况是两个数字相同。 在这种情况下，`c = 0`立即满足条件，因为`a + 0`是一个简单的子串`b + 0`。 该算法处理此问题是因为在移位 0 处，数字比较在任何地方都成功并且不会触发冲突。 

另一种边缘情况是唯一可能的对齐需要长进位链。 例如，`a = 99...9`和`b = 100...0`。 这里正确的行为完全取决于是否添加`c`可以同步级联进位，以便`a + c`出现在里面`b + c`。 每个数字的模拟确保在两个数字上一致地跟踪进位，因此如果可以实现这种同步，则有效的`c`在迭代期间构造； 否则所有数字选择都会失败并且算法输出`-1`。
