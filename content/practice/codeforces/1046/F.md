 ## 解决方案
 ## 问题理解

 我们有几个比特币钱包，每个钱包都持有一定数量的聪。 目标是将这些代币重新分配到可能更多的钱包中，这样钱包最终所持有的数量就不会超过固定限制$x$。 创建新钱包是免费的，但每次将代币从一个钱包转移到另一个钱包都需要支付固定费用$f$，从源钱包支付。 

单次转账不仅仅是记账操作，它实际上将一定数量的代币从一个钱包转移到新的或现有的钱包中，并且无论转移多少代币，每次此类操作都会产生相同的费用。 任务是最小化所需的总费用，以便在所有操作之后，每个钱包最多包含$x$聪。 

输入描述了钱包余额的初始多重集和两个参数：每个钱包允许的最大余额和每次传输成本。 输出是实现约束的最小总成本。 

关键的困难在于分裂不是一步一步强制进行的。 太大的钱包可以使用多个交易分成多个较小的钱包，每笔交易都会减少源余额，同时创建新的目标钱包。 

这些限制使得简单的模拟变得不可能。 拥有多达 200,000 个钱包和价值高达$10^9$，任何重复模拟转账或跟踪单个代币的方法都太慢了。 该解决方案必须将每个钱包独立地减少到确定的操作次数。 

一些边缘情况暴露了幼稚的推理失败。 

第一个失败案例是假设每个钱包都需要$\lceil a_i / x \rceil - 1$operations without considering leftover aggregation logic incorrectly. 例如，如果$a_i = x$，不需要任何运算，但是粗心的公式往往还是算一次分割。 

第二个失败案例是试图贪婪地跨不同来源“填充新钱包”，这导致钱包之间的不正确耦合。 由于每笔交易成本都是独立的，因此协调钱包之间的分割没有任何好处。 

## 方法

 暴力解释将逐一模拟钱包。 对于每个有余额的钱包$a_i > x$，我们反复创建一个新钱包并移动到$x$将硬币存入其中，直到源钱包变得有效。 每一次这样的举动都会花费$f$。 该过程持续进行，直到所有钱包都满足约束。 

这是正确的，但价格昂贵。 在最坏的情况下，单个钱包的大小$10^9$可能需要大约$10^9 / x$转移，以及跨$200{,}000$钱包这变得完全不可行。 

关键的观察是每个钱包都是独立的。 一个大小的钱包$a_i$contributes a fixed number of required “extra wallets” regardless of other wallets. 每个钱包最多可容纳$x$，所以需要存储的最小钱包数量$a_i$是：$$\left\lceil \frac{a_i}{x} \right\rceil$$如果钱包已经存在，它会贡献一个插槽。 每一个额外的槽位都对应于一次转账操作，因为每次转账都会创建一个新钱包。 

因此，钱包所需的操作次数$i$是：$$\left\lceil \frac{a_i}{x} \right\rceil - 1$$将所有钱包的总和得出转账总数，然后乘以$f$给出了答案。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力模拟|$O(\sum a_i / x)$|$O(N)$| 太慢了 |
 | 数学计数|$O(N)$|$O(1)$| 已接受 |

 ## 算法演练

 1. 对于每个钱包价值$a_i$，计算需要多少个全容量钱包。 这是使用整数算术完成的$(a_i + x - 1) // x$。 该表达式执行上限除法而不进行浮点运算。 
2. 从此计数中减一以确定必须创建多少个额外钱包。 减法反映了原来的钱包已经存在，不需要创建。 
3. 将此值添加到操作的运行总计中。 该金额中的每个单位恰好对应一次转账。 
4. 处理完所有钱包后，将操作总数乘以固定费用$f$以获得最终成本。 

步骤 2 背后的原因是，只有当钱包超出容量时才需要分割。 完全合适的钱包不需要任何操作，因为它已经满足约束条件。 

### 为什么它有效

 每个钱包都是独立的，因为转账永远不会合并不同来源之间的约束。 一个大小的钱包$a_i$必须划分为最多大小的组$x$，并且组的数量仅由容量决定。 由于每次转账都会创建一个新钱包，因此额外的组和操作之间存在一一对应的关系。 这使得总成本纯粹是跨钱包的累加。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = list(map(int, input().split()))
    x, f = map(int, input().split())

    ops = 0
    for v in a:
        ops += (v + x - 1) // x - 1

    print(ops * f)

if __name__ == "__main__":
    solve()
```代码直接实现了推导公式。 表达式`(v + x - 1) // x`computes the number of required wallets for a given balance. Subtracting one removes the original wallet from the count, leaving only the number of transfers needed.

 累加器`ops`跟踪传输总数。 乘以`f`将操作转换为聪成本。

 ## 边缘情况

 当钱包已经完全满足限制时，就会出现微妙的边缘情况。 用于输入`a = [6]`和`x = 6`，计算得出`(6 + 5) // 6 - 1 = 1 - 1 = 0`，因此不计算任何操作。 这避免了总是计算至少一组的常见错误。 

另一种边缘情况是当值略高于$x$。 为了`a = [7]`和`x = 6`，公式给出`(7 + 5) // 6 - 1 = 2 - 1 = 1`，匹配单个所需的拆分为 6 和 1。 

大输入不会改变行为，因为每个操作都是在每个钱包的恒定时间内计算的，从而防止任何模拟崩溃。
