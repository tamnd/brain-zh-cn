 ## 解决方案
 ## 问题理解

 我们被要求表达一个给定的整数`n`作为精确总和`k`数字，其中每个被加数必须是 2 的幂。 因此，每个选定的数字的形式为`1, 2, 4, 8, ...`，并且允许重复，因为我们可以多次使用相同的功率。 

输入描述了目标金额`n`和所需数量的被加数`k`。 输出要么是分解不可能的陈述，要么是具体的多重集`k`两个之和等于的幂`n`。 

问题的结构受到两个独立方向的约束。 价值`n`限制了我们可以分配的总质量，同时`k`迫使我们将质量分成固定数量的碎片。 这造成了“少数大国”和“许多小国”之间的紧张关系。 

边界足够严格，任何尝试显式枚举分区或构建组合的方法都会失败。 的价值`n`上升到`10^9`，所以即使用二进制表示也最多有 30 个有效位。 的价值`k`上升到`2 * 10^5`，因此输出本身可能很大，这意味着算法的最终答案大小必须接近线性。 

出现关键边缘情况时`k`大于`n`。 由于最小允许值为`1`，总和`k`正整数至少是`k`，所以任何情况下`k > n`立刻就不可能了。 

另一个微妙的情况出现时`n`二进制中的设置位很少。 例如，如果`n = 8`，自然表示就是`[8]`。 如果`k = 3`，我们必须分裂`8`化为二的三次方。 这可以通过分裂实现`8 -> 4 + 4`，然后进一步分割一个`4 -> 2 + 2`，取决于需要多少分割。 正确性取决于我们能否继续分解二的幂。 

## 方法

 一个直接但幼稚的方法是从`n`写成二进制形式的二的幂和。 每一位`1 << i`成为一种元素。 这使得被加数的最小数量等于设置位的数量`n`。 

如果这个数字已经大于`k`，我们立即失败，因为合并两个力量是不可能的。 2的幂不能与另一个2的幂相结合，除非它们相等，所以我们不能任意减少计数。 

如果初始项的数量最多为`k`，我们尝试增加项数，直到完全达到`k`。 在增加元素数量的同时保留总和的唯一操作是将 2 的幂分成相等的两半。 每次拆分都会使计数恰好增加 1。 

蛮力版本会重复选择任何大于的数字`1`并将其分割直至达到`k`条款。 在最坏的情况下，每次分裂都会将高功率降低到许多，从而导致大约`O(n)`如果将操作作为数组操作实现效率低下。 这对于`k`最多`2e5`。 

关键的观察是我们永远不需要考虑任意分割。 我们只需要一个贪心策略：总是将最大的可用幂分为二。 这保证了我们保持可行性，同时最大限度地提高未来拆分的灵活性。 

我们维护一个二的幂计数结构。 每次我们分开`2^i`，我们将其替换为两个`2^(i-1)`价值观。 我们继续直到元素数量达到`k`或者不可能进一步分裂。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力破解列表 | 最坏情况 | O(n) O(k) | 太慢了 |
 | 贪婪二幂分裂 | O(k log n) | O(k log n) | O(k) | 已接受 |

 ## 算法演练

 1. 转换`n`进入其二进制分解，存储 2 的每个幂的计数。 这给出了尽可能少的被加数。 此步骤是强制执行的，因为 2 的幂构成所有整数的不相交基。 
2. 计算初始被加数的个数。 如果这个计数大于`k`， 输出`NO`，因为我们无法将不同的 2 的幂合并为单个有效的 2 的幂。 
3. 将 2 的所有幂插入到最大结构中（概念上是按指数排列的优先级队列）。 我们总是希望分配最大的可用功率，因为​​它在未来的分配中产生最大的灵活性。 
4. 当被加数的个数小于`k`，提取最大功率`2^x`。 如果`x = 0`，意味着值为`1`，停止，因为它无法进一步拆分。 否则用两份副本替换`2^(x-1)`并将计数加一。 
5. 如果我们正好达到`k`元素，输出多重集。 如果我们在到达之前被卡住了`k`， 输出`NO`。 

### 为什么它有效

 该算法维护一个表示`n`作为二的幂的多重集，总和为`n`。 每个操作都会保留总和，因为替换`2^x`有两个`2^(x-1)`保持值不变。 每个分割元素的数量单调增加一个，因此我们精确地控制最终的大小。 分割最大幂的贪婪选择确保了当存在可行解时我们永远不会过早耗尽可分割质量。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

import heapq

def solve():
    n, k = map(int, input().split())

    # initial decomposition
    cnt = 0
    heap = []

    for i in range(31):
        if n & (1 << i):
            heap.append(-i)
            cnt += 1

    if cnt > k:
        print("NO")
        return

    heapq.heapify(heap)

    # we store exponents as negatives for max-heap behavior

    while cnt < k:
        if not heap:
            break

        x = -heapq.heappop(heap)

        if x == 0:
            break

        # split 2^x into two 2^(x-1)
        heapq.heappush(heap, -(x - 1))
        heapq.heappush(heap, -(x - 1))
        cnt += 1

    if cnt != k:
        print("NO")
        return

    print("YES")
    res = []
    while heap:
        x = -heapq.heappop(heap)
        res.append(1 << x)

    print(*res)
```解决方案从分解开始`n`化为二的幂，形成最小有效表示。 堆用于始终有效地提取最大指数。 这很重要，因为较大的权力是唯一可以重复分裂的权力，并且首先选择它们可以避免陷入许多困境`1`还太早了。 

每个拆分操作都会删除一个元素并添加两个较小的元素，从而使总计数恰好增加 1。 这使得跟踪进展变得容易`k`。 

终止条件要么恰好达到`k`元素或用尽所有可分割的权力。 

## 工作示例

 ### 示例 1

 输入：`n = 9, k = 4`初始分解为`9 = 8 + 1`，所以指数是`{3, 0}`。 

| 步骤| 堆（指数）| 计数 | 行动|
 | --- | --- | --- | --- |
 | 0 | [3, 0] | 2 | 开始 |
 | 1 | [2, 2, 0] | 3 | 分割 3 → 2 + 2 |
 | 2 | [2, 1, 1, 0] | 4 | 分割 2 → 1 + 1 |

 我们到达 4 个元素，因此输出有效：`8 → 4 4 → 2 2 4 → 1 1 2 4`。 

这表明，最大可用功率的重复分裂可以正确地将质量分配到所需数量的被加数中。 

### 示例 2

 输入：`n = 7, k = 5`初步分解：`7 = 4 + 2 + 1`。 

| 步骤| 堆| 计数 | 行动|
 | --- | --- | --- | --- |
 | 0 | [2, 1, 0] | 3 | 开始 |
 | 1 | [1, 1, 1, 0] | 4 | 分割 2 → 1 + 1 |
 | 2 | [1, 1, 1, 0] | 4 | 无法进一步分裂以获得进步|

 我们无法达到 5 个元素，因为所有剩余值都已`1`，因此不可能进一步分裂。 答案是`NO`。 

这表明在到达之前检测可分割权力耗尽的重要性`k`。 

## 复杂度分析

 | 测量| 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 | O(k log n) | O(k log n) | 每次分割都是一个堆操作，我们最多执行 k 次分割 |

 约束允许最多`2 * 10^5`输出元素，并且每个操作在此大小下都是对数，因此该解决方案可以轻松地满足限制。

 ## 边缘情况

 当`n = 1`和`k > 1`，堆仅包含一个指数`0`。 该算法立即停止，因为`0`无法拆分，因此它正确返回`NO`。 

什么时候`n`已经很大但是位稀疏，例如`n = 2^30`，算法从单个元素开始。 它反复分裂成`1`s 直到达到`k`或耗尽所有分流容量。 每个步骤都保持正确性，因为每次分割都会保持总和。 

什么时候`k`等于初始位数`n`，不需要分裂。 该算法直接输出二进制表示，与最优最小分解相匹配。
