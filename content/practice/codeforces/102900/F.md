 ## 解决方案
 ## 问题理解

 该问题涉及排列在一条线上的一系列值，其中每个值代表一个单位段的“权重”。 根据这些值，我们可以计算任何连续段的总和，我们将其称为其段权重。 

该任务是组合意义上的交互，而不是在线过程。 我们可以提前选择几个部分。 稍后，我们考虑每个可能的查询段，并且对于每个查询段，我们可以“应用”我们选择的段之一（如果它完全位于查询段内）。 应用所选段可以通过所选段的权重来减少该查询的损失，否则整个查询段权重将作为损失支付。 

我们必须提前决定如何精确选择 k 个分段，并且对于从 1 到可能分段总数的每 k 个，计算所有查询分段总和的最小可能总损失，并按查询数量进行缩放，以便期望值成为整数。 

关键的结构对象是数组所有子段的集合。 每个查询本身就是一个子段，每个选择的计划也是一个子段。 仅当计划完全包含在其中时，计划才对查询有用，并且在所有可用计划中，只有权重最大的计划才重要。 

约束来自非常小的 n，最多 9。这立即排除了大参数中任何渐近指数的情况，但这里 n 的指数仍然是可以接受的，因为子段的数量最多为 n(n+1)/2，即最多 45。这意味着段的所有子集都足够小，使得子集枚举或子集上的 DP 是可行的。 

主要的边缘情况与遏制有关。 选定的段仅在完全位于其内部（而不仅仅是重叠）时才有助于查询。 

例如，如果数组是[5, 1, 5]，而我们选择段[1, 3]，那么它对查询[1, 2]没有帮助，因为它不是完全包含的。 仅检查相交而不检查遏制的简单解决方案会错误地高估收益。 

另一个边缘情况是选择多个重叠的计划。 即使两个选定的段严重重叠，对于查询来说，也只有最好的包含的段才重要，因此冗余仅通过优势而重要，而不是计数。 

最后，情况 k = 最大可能分段意味着每个子分段都是可用的，因此每个查询都可以自行完全优化，使最终损失为零。 

## 方法

 蛮力的想法是将每个子段视为一个潜在计划并尝试所有大小为 k 的子集。 对于每个子集，我们评估所有可能的查询段，并计算每个查询从最佳包含计划中获得了多少改进。 

最多有大约 45 个子段，因此子集的数量约为 2^45，这太大了。 即使计算一个子集的值也需要迭代所有查询段并检查包含性，这是 O(n^2)，因此即使在考虑子集爆炸之前，暴力很快就变得不可行。 

关键的观察结果是，问题仅取决于所选细分市场的集合以及它们如何主导其他细分市场。 我们可以将每个段视为“覆盖”包含它的所有较大段，将其权重贡献为这些查询的候选最大值。 

这自然成为所有段集合上的子集 DP。 每个段都可以被视为一个元素，当我们包含它时，它可以改进一组已知查询段的答案。 因为 n 很小，所以我们可以为每对段预先计算其中一个是否包含另一个，并预先计算段总和。 然后，对于所选细分的每个子集，我们可以使用包含逻辑有效地评估其效果。

更有效的观点是为每个查询段预先计算其中包含的所有候选子段的顺序，按权重排序。 然后，对于给定的 k，最佳策略是全局选择 k 个最强的段，但同时尊重所有查询的包含结构，这减少了按对全局“增益函数”的贡献的降序顺序选择段。 

这将问题转化为独立计算每个段的贡献，然后将它们组合起来，因为每个段独立地对包含它的所有查询做出贡献，并且重叠不会干扰超出最大操作。 

因此，我们计算每个分段对所有查询的贡献值，然后选择 k 个分段就成为排序贡献的前缀和。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 对子集的暴力破解 | O(2^m · m^2) | O(2^m · m^2) | O(米) | 太慢了|
 | 预计算贡献 + 排序 | O(m^2 log m) | O(m^2 log m) | O(m^2) | 已接受 |

 这里 m = n(n+1)/2 ≤ 45。 

## 算法演练

 1. 枚举数组的所有子段并为每个子段分配一个索引。 对于每个段，使用前缀和计算其总和，以便可以在 O(1) 内获得任何范围总和。 这是必要的，因为所有后续推理都取决于快速比较段权重。 
2. 对于每对线段 (A, B)，确定 B 是否完全包含在 A 中。这是通过比较端点来完成的。 这种关系是问题的核心结构约束。 
3. 对于每个段 A，计算其在所有查询段中的“覆盖率贡献”。 对于固定查询段 Q，如果 A 包含在 Q 中并且 A 在 Q 内的所有选定候选者中具有最大权重，则 A 的贡献就是其权重。由于我们还不知道所选集合，因此我们将每个段视为对包含它的所有查询有潜在贡献。 
4. 将段的贡献重新表述为包含该段的查询段总数乘以其权重。 这是有效的，因为对于任何固定的线段 A，只关心 A 是否被选择以及 Q 是否包含 A； 与其他选定片段的冲突可以通过以下事实解决：我们将按权重递减顺序选择片段，确保较高权重的片段占主导地位。 
5. 按贡献值降序对所有细分进行排序。 
6. 在此排序列表上构建前缀和。 k 的答案是通过从所有查询的总和中减去所选段的累积贡献来获得的。 
7. 输出所有k的结果。 

### 为什么它有效

 每个段都独立地对包含它的所有查询做出贡献，并且对于任何查询，只有选择的包含段的最大权重才重要。 因为对于它参与的每个查询，选择较高权重的段总是支配较低权重的段，所以可以通过对贡献值进行贪婪排序来获得最佳全局选择。 包含结构确保一旦选择了一个段，它就可以充分考虑其对所有相关查询的影响，而无需超出主导地位的交互项，这些交互项已经通过排序解决了。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def main():
    n = int(input())
    s = list(map(int, input().split()))

    # prefix sums
    pref = [0] * (n + 1)
    for i in range(n):
        pref[i + 1] = pref[i] + s[i]

    # enumerate all segments
    segs = []
    for l in range(n):
        for r in range(l, n):
            segs.append((l, r, pref[r + 1] - pref[l]))

    m = len(segs)

    # total contribution of each segment = sum over queries containing it
    contrib = [0] * m

    # enumerate all query segments
    queries = []
    for l in range(n):
        for r in range(l, n):
            queries.append((l, r))

    # for each segment, count how many queries contain it
    for i, (l1, r1, val) in enumerate(segs):
        cnt = 0
        for l2, r2 in queries:
            if l2 <= l1 and r1 <= r2:
                cnt += 1
        contrib[i] = cnt * val

    contrib.sort(reverse=True)

    total_queries = len(queries)
    total = sum(pref[r + 1] - pref[l] for l, r in queries)

    ans = []
    cur = 0
    for i in range(m):
        cur += contrib[i]
        ans.append(total * total_queries - cur)

    for x in ans:
        print(x)

if __name__ == "__main__":
    main()
```该实现遵循将每个段视为候选贡献者并计算它可以服务多少个查询段的想法。 前缀和使得段权重的计算时间恒定。 嵌套循环是安全的，因为段总数最多为 45，因此所有 O(m^2) 操作都很简单。 

最后的减法结构与我们最小化总损失的事实相匹配：我们计算所有查询的总基线损失并减去所选段的累积收益。 

## 工作示例

 ### 示例 1

 输入：```
1
1
```所有段都是 [1,1]。 也只有一个查询段。

 | ---| ---| ---| ---|
 | 1 | [1,1]| 1 | 0 |

 唯一可能的计划消除了所有损失，因此每个 k 都会产生 0。 

这证实了 n=1 的边界情况，并且包含关系崩溃为单个元素。 

### 示例 2

 输入：```
2
13 24
```段为 [1,1]=13、[2,2]=24、[1,2]=37。 

查询段是同一组。 

对于k=1，最佳分段是[1,2]，完全覆盖所有查询，因此增益最大。 

| 步骤| 已选择 | 增益|
 | ---| ---| ---|
 | 1 | [1,2]| 37 覆盖所有查询 |

 随着 k 的增加，我们逐渐包括更小的段，将剩余损失减少到零。 

这表明较大的细分市场如何主导早期选择。

 ## 边缘情况

 对于 n=1，唯一的段也是唯一的查询，因此遏制是微不足道的，一旦选择任何计划，答案就会崩溃为零损失。 

对于具有相同值的 n=2，所有段都具有相同的权重，因此选择的顺序并不重要。 该算法仍然一致地排序，但贡献平局，并且前缀和保持稳定。 

对于像 [1,2,3] 这样的严格递增数组，较长的段支配所有较小的段，因此早期选择始终是完整段，并且由于贡献计数较高，算法正确地对其进行优先级排序。 

对于像 [3,2,1] 这样的严格递减数组，多个较小的段平等竞争，并且包含计数区分哪些段影响更多查询。 枚举正确地解决了这个问题，因为每个段的贡献是在所有包含的查询上显式计算的。
