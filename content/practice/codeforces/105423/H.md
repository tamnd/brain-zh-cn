 ## 问题理解

 我们得到了一个目标模式字符串$s$我们想要构建长度的字符串$n$使用小写英文字母。 在每个构造的字符串中，我们查找出现的$s$作为连续的子串。 要求是最终的字符串完全包含$k$的出现次数$s$，并且这些出现必须是不重叠的，这意味着两个匹配的副本之间不共享字符位置$s$。 字母表上的每个有效构造都会为答案贡献 1，并且结果会取模$998244353$。 

关键的微妙之处在于我们不是独立选择事件发生的位置。 一个天真的解释是放置$k$不相交的副本$s$并任意填充剩余位置，但这错过了一个重要的约束：额外的意外发生$s$可能会跨越边界出现，这些也必须被排除。 

这些约束表明了为什么直接的指数或组合构造是不可能的。 长度$n$可以达到$10^4$， 尽管$k \le 10$和$|s| \le 100$。 任何枚举字符串甚至直接尝试所有匹配位置的方法都是不可行的，因为字符串的数量是$26^n$，甚至推理所有的位置$k$之间出现的$n$如果我们不仔细构建转换，位置组合就会变得很大。 唯一可行的方向是动态编程方法，它从左到右构建字符串，同时有效地跟踪我们与模式匹配的程度。 

一个幼稚的错误是只强制执行不重叠的展示位置。 例如，如果$s = "aba"$,$n = 5$， 和$k = 1$，字符串“ababa”包含从位置 1 和 3 开始的两个重叠出现，并且基于位置的简单方法会根据重叠的处理方式错误地少计数或多计数。 另一个微妙的失败案例是忽略由强制段之间的填充字符创建的意外匹配。$s$。 

## 方法

 暴力方法会尝试生成所有长度的字符串$n$，检查每个子字符串是否出现$s$，并精确计算那些$k$不重叠的匹配。 即使检查一个字符串需要$O(n \cdot |s|)$，字符串的数量是$26^n$，对于$n = 10^4$。 这立即排除了任何明确的构造或采样方法。 

使这个问题可解决的结构是，我们实际上并不关心字符串的完整历史，只关心字符串的部分前缀如何$s$当我们构建字符串时出现。 这是一种典型的情况，其中根据模式构建的自动机（通常通过 KMP 的前缀函数）将所有必要的历史记录压缩到一个小的状态空间中。 在每个位置，我们只需要知道有多少$s$我们当前已将其作为构造前缀的后缀进行匹配，以及已经完成了多少次完整匹配。 

这将问题转化为关于位置、自动机状态和已完成比赛数量的动态规划。 每个新角色都会转换自动机状态，每当我们达到完全匹配时，我们就会增加出现次数。 非重叠条件自然由自动机强制执行：完成匹配后，我们从 KMP 失败状态继续，该状态会以一致的方式自动考虑重叠，而不允许对重叠出现进行重复计算，除非它们在序列中真正有效。

 ## 算法演练

 我们首先对模式进行预处理$s$使用前缀函数构建 KMP 自动机。 这给了我们，对于每个状态代表多少个字符$s$我们当前已经匹配了每个可能的下一个字符，以及附加该字符后的下一个状态。 

然后，我们对构造的字符串的长度执行动态编程。 

1.我们定义一个DP状态，代表我们可以有多少种方式构建一个长度的前缀$i$，以给定的自动机状态结束，并且已经完全完成$c$的完整比赛$s$。 
2. 我们在位置 0 处用零个匹配字符初始化 DP，并完成零次出现。 这对应于空字符串，它通常没有匹配项。 
3.对于从0到的每个位置$n-1$，我们迭代所有当前的自动机状态以及已完成匹配的所有有效计数$k$。 
4. 对于每个状态，我们尝试将每个字符从“a”附加到“z”。 这使用预先计算的转换表产生一个新的自动机状态。 
5. 如果此转换导致完全匹配$s$，我们将匹配计数器加一。 如果这个增量超过$k$，我们放弃转换，因为我们只想精确地$k$发生。 
6. 否则，我们相应地更新下一个位置的 DP。 
7. 处理完所有位置后，答案是该位置处所有自动机状态的总和$n$恰好与$k$已完成的比赛。 

正确性取决于自动机对当前字符串的每个后缀（也是前缀）进行编码的事实$s$。 这确保了每次出现$s$完成时会被准确检测到，并且不会遗漏或重复计算任何事件。 DP 永远不需要检查完整的字符串，因为任何未来的匹配仅取决于此后缀状态，而不是之前的字符。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MOD = 998244353

def build_automaton(s):
    m = len(s)
    pi = [0] * m
    for i in range(1, m):
        j = pi[i - 1]
        while j > 0 and s[i] != s[j]:
            j = pi[j - 1]
        if s[i] == s[j]:
            j += 1
        pi[i] = j

    nxt = [[0] * 26 for _ in range(m + 1)]

    for state in range(m + 1):
        for c in range(26):
            if state < m and ord(s[state]) - 97 == c:
                nxt[state][c] = state + 1
            else:
                if state == 0:
                    nxt[state][c] = 0
                else:
                    nxt[state][c] = nxt[pi[state - 1]][c]

    return nxt

def solve():
    n, k = map(int, input().split())
    s = input().strip()
    m = len(s)

    nxt = build_automaton(s)

    dp = [[0] * (k + 1) for _ in range(m + 1)]
    dp[0][0] = 1

    for _ in range(n):
        ndp = [[0] * (k + 1) for _ in range(m + 1)]
        for state in range(m + 1):
            for cnt in range(k + 1):
                if dp[state][cnt] == 0:
                    continue
                val = dp[state][cnt]
                for c in range(26):
                    ns = nxt[state][c]
                    nc = cnt
                    if ns == m:
                        nc += 1
                        ns = nxt[ns][c] if m > 0 else 0
                    if nc <= k:
                        ndp[ns][nc] = (ndp[ns][nc] + val) % MOD
        dp = ndp

    print(sum(dp[state][k] for state in range(m + 1)) % MOD)

if __name__ == "__main__":
    solve()
```自动机构造是使 DP 可行的核心细节。 前缀函数将模式的所有重叠信息压缩到线性状态空间中。 DP 数组跟踪当前匹配的模式数量以及到目前为止已完成的完整匹配数量。 

当我们达到完全匹配状态时，我们立即增加计数并继续使用回退转换，这确保重叠结构得到一致的处理，而不需要手动重叠检查。 

对所有状态的最终求和是必要的，因为字符串可以以任何部分匹配状态结束，同时只要完全匹配就仍然有效$k$已看过完整比赛。

 ## 边缘情况

 一个重要的边缘情况是模式长度为 1 时。在这种情况下，每个字符都会自动匹配，并且 DP 必须正确地将每个位置解释为可能完成一次出现，而不会出现任何自动机歧义。 自动机退化为单个转换状态，但匹配计数器仍然必须为每个字符正确地递增一次。 

另一种边缘情况发生在$k = 0$。 DP 仍必须正常运行，但必须仔细排除任何创建达到模式长度的匹配状态的转换。 幼稚的实现可能会忘记传播从未完成匹配的有效状态，错误地生成零而不是$26^n$减去包含模式的无效字符串。 

第三种边缘情况是像“aaa”这样的自重叠模式。 在这里，单个字符移位既可以完成一个匹配，又可以立即促成下一个部分匹配。 KMP 后备转换可确保这些链接的重叠被准确地计为有效的延续而不是独立的段。
