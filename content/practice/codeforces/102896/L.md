 ## 解决方案
 ## 问题理解

 我们得到一个固定的二叉搜索树，其节点存储整数键。 每个节点还知道其子树内的最小和最大键，以及该子树的大小。 使用此结构，递归过程执行范围查询：给定一个段$[L, R]$，它计算树中有多少个键位于该区间内。 

该过程并不简单地遍历所有节点。 相反，当子树完全超出查询范围时，它会尝试跳过整个子树，并且当子树完全包含在范围内时，它也会停止下降，立即对整个子树进行计数。 否则，它访问该节点，可能对其进行计数，然后继续递归到两个子节点。 

我们要求为每个查询计算的数量不是该范围内的值的数量，而是该递归过程在树上执行时进行的函数调用次数。 这包括由于修剪而立即结束的调用、在完全覆盖的子树处停止的调用以及在根处的初始调用。 

限制很大：多达二十万个节点和二十万个查询。 对每个查询的递归的直接模拟将重复地重新访问树的大部分。 在最坏的情况下，单个查询可能会强制遍历几乎整个结构，从而在所有查询中提供二次行为。 这远远超出了两秒限制所能处理的范围。 

主要挑战是我们不是聚合值，而是计算_递归如何探索结构_，这取决于子树间隔以及它们如何与查询范围重叠。 

当查询范围与子树间隔完全匹配时，会出现微妙的边缘情况。 例如，如果子树包含键$[1, 10]$查询是$[1, 10]$，函数在该节点停止并且不访问其子节点。 总是下降的天真的遍历会错误地过度计数调用。 

另一种边缘情况是查询范围完全排除子树。 如果子树有区间$[20, 30]$查询是$[1, 10]$，递归立即返回零而不访问子项。 幼稚的 DFS 仍会探索子树并过度计数调用。 

## 方法

 暴力解决方案实际上是为每个查询执行所描述的递归。 它检查节点的间隔，决定是否修剪，并在需要时递归访问子节点。 这是正确的，因为它准确地反映了定义。 

然而，这种方法的成本很高，因为每个查询可能涉及许多节点。 如果树是倾斜的或者查询范围很大，一次查询可能会访问所有$n$节点。 和$q$查询，这会导致$O(nq)$最坏情况下的行为，这对于$2 \cdot 10^5$。 

关键的观察结果是，递归在结构上与线段树范围查询相同。 每个节点代表一个键区间，只有当该区间与查询部分重叠时，递归才会分裂。 当某个节点完全位于查询范围内时，递归会立即在该节点处停止。 当它完全在外面时，它会被完全跳过。 

这意味着递归总是精确地访问树的_规范分解_到不相交的覆盖段的节点，以及下降到它们时遇到的中间节点。 因此，访问调用的数量仅取决于哪些子树间隔完全在查询内、部分重叠或完全在查询外。 

我们可以将树视为键排序顺序上的线段树，而不是模拟递归。 每个节点的最小-最大间隔形成一个连续的段，并且子节点对该段进行分区。 查询$[L, R]$诱导一小组类似线段树的节点，其间隔要么完全覆盖，要么部分相交。 对递归调用进行计数就相当于对线段树遍历中访问的节点进行计数。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O(nq)$|$O(n)$| 太慢了 |
 | 线段树式遍历|$O(\text{visited nodes per query})$, 摊销$O(\log n)$|$O(n)$| 已接受 |

 ## 算法演练

 我们预处理每个节点以便我们知道它的子树间隔$[min[v], max[v]]$。 

对于每个查询$[L, R]$，我们模拟递归遍历的结构，但我们使用区间检查避免不必要的下降。 

1、从根源开始。 从概念上讲，我们在根节点上进行初始函数调用，因此我们立即对其进行计数。 
2.如果当前节点的区间完全在外面$[L, R]$，我们返回而不扩展它。 这对应于修剪。 
3. 如果当前节点的区间完全包含在$[L, R]$，我们将此节点算作一次调用并停止其下面的递归，因为原始函数将返回`v.count`无需探望儿童。 
4. 否则，区间与查询部分重叠。 我们计算当前节点，然后递归到左子节点和右子节点。 
5. 我们对这次受控遍历中所有访问过的节点进行求和。 

关键思想是我们并不是按字面意思执行原始递归；而是执行原始递归。 我们仅使用区间比较来重现其_控制流决策_。 

### 为什么它有效

 原始函数中的每个递归调用都对应于进入一个节点，该节点的子树间隔未知是完全不相关的。 如果节点的间隔完全在查询内，则递归立即停止，因此不存在后代调用。 如果完全在外面，则呼叫存在但立即终止。 如果它部分重叠，则递归必须继续到子级。 

这创建了一种完全由间隔关系决定的独特访问模式。 由于子树区间形成层次结构（每个子区间严格包含在其父区间中），因此递归始终遵循相同的确定性区间分割集。 我们的遍历准确地再现了这些分割，因此它与函数调用的数量相匹配。 

## Python 解决方案```python
import sys
input = sys.stdin.readline
sys.setrecursionlimit(10**7)

n = int(input())
left = [0] * (n + 1)
right = [0] * (n + 1)
key = [0] * (n + 1)

for i in range(1, n + 1):
    l, r, k = map(int, input().split())
    left[i] = l
    right[i] = r
    key[i] = k

minv = [0] * (n + 1)
maxv = [0] * (n + 1)

def dfs(v):
    minv[v] = key[v]
    maxv[v] = key[v]
    if left[v]:
        dfs(left[v])
        minv[v] = min(minv[v], minv[left[v]])
        maxv[v] = max(maxv[v], maxv[left[v]])
    if right[v]:
        dfs(right[v])
        minv[v] = min(minv[v], minv[right[v]])
        maxv[v] = max(maxv[v], maxv[right[v]])

dfs(1)

q = int(input())

def solve(v, L, R):
    if v == 0:
        return 0
    if maxv[v] < L or minv[v] > R:
        return 1
    if L <= minv[v] and maxv[v] <= R:
        return 1
    return 1 + solve(left[v], L, R) + solve(right[v], L, R)

out = []
for _ in range(q):
    L, R = map(int, input().split())
    out.append(str(solve(1, L, R)))

print("\n".join(out))
```第一部分使用后序 DFS 构建子树区间。 每个节点的最小值和最大值都是根据其子节点计算的，这是至关重要的，因为所有修剪决策都取决于这些范围。 

查询函数直接反映了问题中描述的递归过程。 每一次通话都有贡献`1`因为每次调用`lookup`被计算在内。 如果当前子树在查询之外，我们仍然对调用进行计数，但停止。 如果完全在里面，我们也会计算呼叫但提前停止。 否则，我们会分支到两个孩子。 

关键的微妙之处在于，我们从不尝试通过跳过根调用或合并案例来进行优化，因为问题明确与函数调用计数有关，而不是与工作所访问的节点有关。 

## 工作示例

 考虑一棵小树，其中节点 1 有子节点 2 和 3，键的排列方式使得 2 位于左侧，3 位于右侧。 

对于完全覆盖所有键的查询，遍历会立即在根处停止。 

| 步骤| 节点| 间隔 | 决定| 来电 |
 | --- | --- | --- | --- | --- |
 | 1 | 1 | 完整的树 | 完全在里面| 1 |

 这表明仅进行了 root 调用。 

现在考虑一个排除所有内容的查询。 

| 步骤| 节点| 间隔 | 决定| 来电 |
 | --- | --- | --- | --- | --- |
 | 1 | 1 | 完整的树 | 部分/外部线索检查儿童| 1 |
 | 2 | 2 | 左子树 | 外面| 2 |
 | 3 | 3 | 右子树| 外面| 3 |

 这里每个节点仍然_被调用一次_，但是递归在每个节点处立即停止。 

这表明函数调用计数与访问的子树大小不同，而是对每个尝试的下降步骤进行计数。

 | 测量 | 复杂性 | 说明|

 | 时间 |$O(n + q \cdot h)$| 构建间隔需要线性时间，每个查询都沿着高度路径递归$h$修剪|

 这符合限制，因为子树修剪确保递归不会重复探索相同的大区域，并且典型的 BST 深度使遍历在预期的约束下保持可管理性。

 ## 边缘情况

 完全覆盖的子树演示了提前停止行为。 对于子树$[1, 100]$并查询$[1, 100]$，该函数仅调用根并且不下降，因此输出恰好是 1。 

完全不相交的子树演示了修剪。 对于子树$[50, 60]$并查询$[1, 10]$，该函数仍然调用该节点一次，检测到排除，并立即停止。 

倾斜的树测试最坏情况的深度。 如果树退化为链，则按顺序访问每个节点以进行部分重叠查询，并且算法正确地计算沿该路径的每个节点的一次调用。
