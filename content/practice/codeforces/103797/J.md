 ## 解决方案
 ## 问题理解

 我们根据问题给出了一个参赛者网格，其中每对$(c, p)$代表一名参赛者解决一个问题。 随着时间的推移，我们会收到按时间顺序排列的提交内容，每个提交内容都属于一个这样的单元并带有一个判决。 

关键规则是我们只关心在第一个接受的相同解决方案之前发生的错误提交$(c, p)$。 一旦单元接收到第一个 AC，该单元就会变得不活动，并且以后向其提交的内容必须被完全忽略。 

处理完所有提交后，我们必须回答许多疑问。 每个查询都会给出一个由一系列参赛者和一系列问题定义的矩形子网格。 对于该矩形，我们需要计算其中所有单元格中错误提交的总数（在 AC 之前发生的提交）。 

这些约束已经表明了解决方案的形状。 网格大小最多为$500 \times 500$，大约有 250,000 个细胞。 提交量和查询量均可达到$2 \times 10^5$。 这立即排除了任何扫描每个查询的提交或每个查询的每个单元重新计算的解决方案。 该结构强烈暗示我们应该将所有提交信息压缩到静态二维数组中，然后支持快速子矩阵求和查询。 

一个微妙的点是“仅在 AC 之前”条件。 一个幼稚的错误是计算全局所有非 AC 提交的内容，然后尝试减去 AC 之后的提交。 这会失败，因为 AC 之后的提交根本不能影响单元格，但它们仍然出现在输入流中，并且如果不仔细过滤每个单元格，可能会错误地增加计数。 

当单元从未接收到 AC 时，会出现另一种边缘情况。 在这种情况下，即使有很多 WA 或 TLE 条目，其提交的任何内容都不应被计算在内。 只有最终达到 AC 的细胞才会做出贡献。 

## 方法

 一种直接的方法是独立模拟每个查询。 对于给定的矩形，我们可以迭代所有提交并检查它们是否属于矩形内的单元格并且发生在该单元格的第一个 AC 之前。 这需要扫描最多$2 \times 10^5$每个查询的提交次数，给出最坏情况的复杂性$4 \times 10^{10}$的操作，远远超出了可行的限度。 

关键的观察结果是每个细胞的行为独立于其他细胞。 提交仅影响其自身$(c, p)$，一旦我们确定有多少有效的错误提交属于该单元格，它就不会再改变。 这意味着整个动态过程可以简化为计算每个网格单元的单个整数。 

一旦每个单元格都有固定值，问题就变成了经典的二维范围求和查询问题。 二维前缀和允许在恒定时间内回答每个查询。 

剩下的唯一任务是计算每个单元格在其第一个 AC 之前有多少提交是错误的。 我们可以按顺序处理提交日志，维护每个单元是否已经看到 AC，如果没有，则每当出现非 AC 判决时就增加其计数器。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 每个查询的暴力破解 |$O(S \cdot Q)$|$O(1)$| 太慢了 |
 | 每单元累加 + 2D 前缀和 |$O(S + NM + Q)$|$O(NM)$| 已接受 |

 ## 算法演练

 我们将网格视为一个表，其中每个条目都存储有效错误提交的计数器。 

1. 创建两个$N \times M$arrays：一个布尔数组来标记一个cell是否已经收到AC，一个整数数组来统计AC之前的错误提交。 这种分离是必要的，因为在 AC 之后我们必须冻结该单元的状态。 
2. 按时间顺序处理提交的内容。 对于每个提交$(c, p, v)$，首先检查该小区是否已经有AC。 如果是这样，请完全跳过提交，因为根据定义，AC 之后的任何内容都是不相关的。 
3. 如果结论是 AC 并且单元格尚未标记，则将其标记为已解决。 从这一刻起，以后的任何提交都不会影响该单元格。 
4. 如果判决不是 AC 并且该单元仍未解决，则增加其计数器。 这确保我们只计算第一次成功提交之前发生的错误。 
5. 处理完所有提交后，在计数器网格上构建 2D 前缀和。 每个前缀单元存储矩形中所有值的总和$(1,1)$到$(i,j)$。 
6. 对于每个查询矩形，在常数时间内使用前缀网格上的包含-排除计算总和。 

### 为什么它有效

 每个单元格都是独立的，因为提交仅修改自己的坐标。 当一个细胞接收到 AC 的那一刻，它的最终贡献就完全确定了，以后的事件都不能改变它。 因此，问题分解为计算每个单元格的固定值。 前缀总和转换保留子矩形上的精确总和，因此仅​​使用预先计算的信息来回答每个查询，而无需重新访问提交日志。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def main():
    N, M = map(int, input().split())
    S = int(input())

    solved = [[False] * (M + 1) for _ in range(N + 1)]
    bad = [[0] * (M + 1) for _ in range(N + 1)]

    for _ in range(S):
        c, p, v = input().split()
        c = int(c)
        p = int(p)

        if solved[c][p]:
            continue

        if v == "AC":
            solved[c][p] = True
        else:
            bad[c][p] += 1

    pref = [[0] * (M + 1) for _ in range(N + 1)]

    for i in range(1, N + 1):
        row_sum = 0
        for j in range(1, M + 1):
            row_sum += bad[i][j]
            pref[i][j] = pref[i - 1][j] + row_sum

    Q = int(input())
    out = []

    for _ in range(Q):
        c1, p1, c2, p2 = map(int, input().split())

        res = pref[c2][p2]
        if c1 > 1:
            res -= pref[c1 - 1][p2]
        if p1 > 1:
            res -= pref[c2][p1 - 1]
        if c1 > 1 and p1 > 1:
            res += pref[c1 - 1][p1 - 1]

        out.append(str(res))

    print("\n".join(out))

if __name__ == "__main__":
    main()
```核心实现维护每个单元的状态机。 这`solved`表确保一旦达到 AC，后面的提交将被忽略。 这`bad`表仅累积有效的 AC 前错误尝试。 

前缀和构造是逐行完成的，以避免嵌套重新计算。 查询应答使用子矩阵的标准包含-排除公式，完全依赖于预先计算的前缀表。 

## 工作示例

 ### 示例 1

 考虑一个有 2 个参赛者和 2 个问题的微型系统。 假设我们处理提交并最终在 AC 之前得到以下有效错误提交的每个单元格计数：

 | 细胞| 错误计数 |
 | ---| ---|
 | (1,1) | 0 |
 | (1,2) | 1 |
 | (2,1) | 3 |
 | (2,2) | 0 |

 现在我们构建前缀和：

 | 我\j | 1 | 2 |
 | ---| ---| ---|
 | 1 | 0 | 1 |
 | 2 | 3 | 4 |

 对于覆盖整个网格的查询，我们返回 4。这与 (2,1) 中仅发生 3 次错误提交和 (1,2) 中仅发生 1 次错误提交这一事实相符。 

### 示例 2

 如果一个cell有多次错误的提交但从未收到AC，例如(1,1)有10个WA但没有AC，它的贡献仍然是0。前缀网格完全忽略它。 

对包含 (1,1) 的任何区域的查询将不包括这 10 次尝试，从而确认只有具有最终 AC 的单元格才会做出贡献。 

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 |$O(S + NM + Q)$| 一次传递提交，一次传递前缀总和的网格，每个查询的恒定时间 |
 | 空间|$O(NM)$| 两个存储状态和前缀和的网格 |

 界限$N, M \le 500$使$NM$存储微不足道，同时$S, Q \le 2 \times 10^5$确保仅接受线性时间的提交和查询处理。 该解决方案在一定范围内非常适合。

 ## 边缘情况

 一个常见的陷阱是计算所有错误提交，无论 AC 时间如何。 例如：```
1 1
3
1 1 WA
1 1 AC
1 1 WA
1
1 1 1 1
```这里正确的答案是1，而不是2。第三次提交必须被忽略，因为AC已经发生了。 该算法通过将单元标记为已解决并跳过所有后续更新来处理此问题。 

另一个边缘情况是电池永远不会获得交流电：```
1 1
2
1 1 WA
1 1 TLE
1
1 1 1 1
```正确的输出是 0。`solved`flag 从未设置，但由于没有 AC，规则明确表示我们根本不应该计算这些提交。 该实现仅通过递增来强制执行此操作`bad`在 AC 之前，并且永远不会将其添加到最终总和中，除非发生 AC。
