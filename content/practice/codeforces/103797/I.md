 ## 解决方案
 ## 问题理解

 我们得到了一长串编码为字符串的日子。 每个字符代表科森扎当天是否有考试。 标记的一天`E`是考试日，同时`F`是自由活动的一天。 

科森扎遵循一条非常严格的规则：每次考试都需要整整一天的自由学习时间，而且他拒绝在考试当天学习。 这意味着每个`E`必须与独特的配对`F`在日程安排中，它发生在它之前，因为学习发生在考试之前，并且占用了一整天的空闲时间。 如果在任何时候考试到来时没有事先安排空闲时间作为学习时间，那么时间表就会失效，他就会“哭泣”。 

同时，科森扎只在真正自由的日子里玩CS:GO：它们不是考试日，也不用于学习。 最初，他已经连续 100 天这样，我们需要计算在安排完所有必需的学习课程后，他还可以累积多少个额外的自由游戏天数。 

关键的相互作用是一些`F`几天的时间都用来学习，剩下的时间`F`天成为可玩的连续扩展。 输出是安排后可玩的免费天数的总数，或者`"I cry"`如果不可能为每次考试分配一个学习日。 

字符串长度可达 10^6，因此任何二次或重复扫描方法立即不可能。 我们需要一个仅维护简单计数器的单线性通道解决方案。 

当考试在足够的空闲天数之前出现时，就会出现微妙的失败情况。 例如，`EFFFFF`立即失败，因为第一天是一场考试，之前没有空闲日可供学习，即使稍后出现空闲日。 

另一种失败情况是，全球范围内有足够的空闲天数，但在某些考试之前却不够。 例如，`FEFE`看起来全球平衡，但第二个`E`没有足够的早期未使用过的`F`天数取决于消费订单。 

## 方法

 强力策略会尝试通过迭代每次考试并向后搜索未使用的空闲日来指定为学习时间来显式模拟日程安排。 每个分配都可能需要扫描字符串的整个前缀。 在最坏的情况下，交替模式如`FFFFFF...EEE...`，每次检查可能会扫描近 O(n) 个位置，导致总复杂度为 O(n^2)，这对于 10^6 个字符来说太慢了。 

关键的观察是，我们永远不需要记住哪一天是用来学习的，只需要记住有多少空闲时间即可。 每个`F`增加可用学习时段池，并且每个`E`消耗一个。 如果在任何时候我们需要使用一个学习时段但不存在，则时间表将无效。 

一旦可行性得到保证，剩下的问题就纯粹是算术问题了：每一个没有用于学习的空闲日都将成为可玩的 CS:GO 日。 如果我们跟踪用于学习的空闲天数，答案是总空闲天数减去已用的学习天数。 

这将问题简化为具有两个计数器的单次传递。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力破解（每次考试回溯）| O(n^2) | O(n^2) | O(1) | O(1) | 太慢了 |
 | 贪心计数| O(n) | O(1) | O(1) | 已接受 |

 ## 算法演练

 ### 最优策略

 我们维护两个计数器：可用的空闲天数和已用的学习天数。 

1.初始化一个计数器`free = 0`，代表可用于学习的未使用的空闲天数，以及`used = 0`，代表到目前为止我们已经投入学习的空闲天数。 
2. 从左到右遍历字符串，按顺序处理每一天。 这是必要的，因为学习必须在考试之前进行，因此前缀的可用性很重要。 
3. 如果当前字符是`F`, 增量`free`因为这一天可能会用于以后的学习。 
4. 如果当前字符是`E`，我们需要分配一个学习日。 如果`free`大于零，我们递减`free`并增加`used`。 如果`free`为零，我们立即得出结论不可能满足这个考试要求，所以我们输出`"I cry"`。 
5. 成功处理整个字符串后，计算总空闲天数`F`在输入中。 最终的答案是`total_F - used`。 

### 为什么它有效

 核心不变量是在扫描中的任何点，`free`准确地表示之前看到的尚未分配为学习天数的空闲天数。 每场考试都消耗一个这样的单元，如果不存在，未来的空闲日也无法解决失败问题，因为学习必须在考试前进行。 因此，可行性完全取决于前缀有效性。 一旦可行性成立，剩余的免费天数就是固定的，并且与作业选择无关，因为每项考试恰好消耗一天的免费时间，并且所有作业的成本都是相等的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

s = input().strip()

free = 0
used = 0
total_f = 0

for ch in s:
    if ch == 'F':
        free += 1
        total_f += 1
    else:  # 'E'
        if free == 0:
            print("I cry")
            sys.exit(0)
        free -= 1
        used += 1

print(total_f - used)
```其实现直接反映了贪婪的解释。 这`free`计数器跟踪可用的学习能力，以及`used`跟踪有多少`F`日子都被考试消耗掉了。 我们还维护`total_f`因此我们可以在最后计算剩余的可玩天数，而无需重新扫描字符串。 

失败后提前退出是安全的，因为一旦无法满足给定前缀的考试要求，未来就没有了`F`可以追溯修复。

## 测试用例```python
import sys, io

def solve():
    s = sys.stdin.readline().strip()

    free = 0
    used = 0
    total_f = 0

    for ch in s:
        if ch == 'F':
            free += 1
            total_f += 1
        else:
            if free == 0:
                return "I cry"
            free -= 1
            used += 1

    return str(total_f - used)

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    return solve()

# provided samples
assert run("EFFFFF") == "I cry"
assert run("FFEE") == "0"

# custom cases
assert run("F") == "1"
assert run("E") == "I cry"
assert run("FEFE") == "1"
assert run("FFFFEEEE") == "0"
```| 测试输入| 预期产出 | 它验证了什么 |
 | ---| ---| ---|
 |`F`|`1`| 单日自由活动，无考试|
 |`E`|`I cry`| 不可能立即失败|
 |`FEFE`|`1`| 交错大小写，前缀依赖 |
 |`FFFFEEEE`|`0`| 完全消耗边缘情况|

 ## 边缘情况

 对于输入`EFFFFF`，算法在第一个字符处立即失败，因为`free = 0`加工时`E`。 模拟停止并输出`"I cry"`，正确捕获未来的空闲日不能用于满足过去的考试要求。 

对于像这样的情况`FFFFEEEE`，算法累积四个空闲天，然后消耗所有四个空闲天进行考试。 最终计数器达到`free = 0`,`used = 4`， 和`total_f = 4`，产生答案`0`，这正确地反映了没有剩余可玩的日子。 

为了`FEFE`，算法在获取和消耗之间交替。 每个`E`始终与最近未使用的匹配`F`，保持不变式`free`永远不会错误地代表未来的可用性。 最终的结果只取决于有多少`F`保持未消耗，而不是其位置，确认贪婪前缀匹配策略的正确性。
