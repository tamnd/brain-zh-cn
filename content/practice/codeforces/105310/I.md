 ## 解决方案
 ## 问题理解

 我们维护一个垂直的层结构，索引从顶部的 0 到底部的 n。 有些层可能包含“藤蔓”，并且每个藤蔓都有当前的整数长度。 该系统支持动态添加藤蔓，增加选定藤蔓组的长度，并使用这些藤蔓长度重复向下遍历。 

当藤蔓存在于第 i 层且长度为 l 时，它的作用就像跳跃：从 i 移动到 i + l，但如果超过 n，则正好落在 n 处。 如果某一层没有藤蔓，则该层在遍历过程中就像一个停止点。 

共有三个操作。 在长度为 1 的位置创建或重置藤蔓。增加索引落入模 c 的特定同余类的所有藤蔓的长度。 最后一个操作模拟遍历：从给定的层开始，使用藤蔓重复跳跃，直到到达没有藤蔓的层，然后输出最终位置。 

重要的约束是 n 和 q 都达到 100000，因此任何在线性时间内重新计算每个查询的遍历或批量更新的解决方案都将失败。 在最坏的情况下，通过重复跟随跳转来对类型 3 进行简单模拟可能会降低到每个查询的 O(n)，当重复 q 次时，速度太慢。 类似地，通过扫描所有层来应用类型 2 每次更新也是 O(n) 并且太大。 

链接产生了一个微妙的问题。 如果藤蔓形成像 0 → 5 → 9 → 12 → … 这样的长链，则简单的遍历会重复重新计算相同的段，并且重复的类型 2 更新可以一次更改许多节点。 这种组合使得直接模拟变得不可行。 

幼稚遍历的一个具体失败场景是一条长链：

 输入：```
n = 10
type 1: vines at 0,1,2,...,9 each length 1
type 3: start at 0
```简单的方法是遍历 0 → 1 → 2 → ... → 10，成本为 O(n)。 如果重复，就会变成 O(nq)。 

另一个有问题的情况是批量增量查询：```
c = 1
type 2 always affects every vine
```每次更新都会涉及到所有的藤蔓，使得 O(nq) 变得简单。 

## 方法

 直接模拟存储一系列藤蔓长度并按字面处理所有内容。 类型 1 只是设置一个值，类型 2 循环遍历所有索引并递增那些匹配 i mod c = k 的索引，类型 3 使用数组重复跳转。 

这是正确的，但太慢了。 仅类型 2 每次查询的成本为 O(n)，从而导致 O(nq)。 如果 vine 长度很小或者更新频繁地重置结构，则类型 3 也可能会降级为每次查询 O(n)。 

关键的观察结果是，类型 2 操作是高度结构化的：它们仅影响模 c 的一个残差类中的索引。 这将数组划分为 c 个独立的组。 在每个组内，更新是随着时间的推移统一添加的。 

我们可以为每个残基类存储一个全局“惰性增长计数器”，表示该类已增加了多少次，而不是立即更新每个元素。 每根藤蔓都会记住创建或重置时的最后一个计数器值，并且我们根据需要计算其实际长度。 

这完全消除了 O(n) 更新成本。 

剩下的挑战是有效地回答类型 3 查询。 遍历的索引是单调递增的，并且每一步都向前跳跃。 这表明我们可以通过按需读取当前长度来隐式地维持“下一个活跃藤蔓”行为。 由于每个步骤都严格增加位置，因此每个查询最多触及每个层一次。 通过惰性评估，这在约束范围内变得可以接受。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(nq) | O(n) | 太慢了|
 | 懒残+直接模拟| 最坏 O(qn) | O(n + c) | 太慢了|
 | 残差懒惰+按需求值（优化遍历）| O(q log n) 或摊销 O(n + q) | O(n + c) | 已接受 |

 ## 算法演练

 我们维护三个主要结构：一个存储基本藤蔓长度的数组，一个指示每个索引处是否存在藤蔓的标志，以及每个残基类以 c 为模的全局增量计数器。 我们还为每个藤蔓存储上次重置或创建时计数器的值。 

1.初始化一个数组`base[i]`对于藤蔓长度，一个数组`active[i]`指示藤蔓是否存在，以及一个数组`tag[i]`存储该残留类别的最后看到的增量计数器。 还维护`add[k]`对于每个残基类别 k，最初为零。 这建立了一个系统，我们可以在不接触所有元素的情况下重建当前的藤蔓长度。 
2. 对于索引 i 处的类型 1 查询，我们在 i 处创建或覆盖一棵藤蔓。 我们设定`active[i] = true`,`base[i] = 1`，并记录`tag[i] = add[i % c]`。 原因是影响此残留类的任何未来增量都应相对于此快照应用。 
3. 对于具有残基 k 的类型 2 查询，我们递增`add[k] += 1`。 我们不碰个别的葡萄藤。 这体现了这样的想法：该残留类别中的所有藤蔓实际上都生长了 1，但我们推迟了应用它。 
4. 对于从 i 开始的类型 3 查询，我们模拟跳转。 当 i 处有一根藤蔓时，我们将其当前长度计算为`base[i] + (add[i % c] - tag[i])`。 这给出了自创建以来正确的有效长度。 然后我们跳到`i + length`，如果需要，夹紧到 n，然后继续。 如果当前层不存在藤蔓，我们停止并输出 i。 
5. 继续遍历循环，直到到达非藤蔓层。 每次迭代都严格增加索引，确保终止。 

### 为什么它有效

 每棵藤蔓都会经历两种变化：初始化和残留类增量。 区别`add[k] - tag[i]`准确计算自 vine 创建以来相关类发生了多少次增量。 因为类型 1 会重置两者`base[i]`和`tag[i]`对于当前的全局状态，该公式始终重建真实的当前长度。 由于类型 2 只影响一个残基类别，并且从不跨类别相互作用，因此这种分解是精确且无损的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, q, c = map(int, input().split())

    base = [-1] * n
    active = [False] * n
    tag = [0] * n
    add = [0] * c

    out = []

    for _ in range(q):
        tmp = input().split()
        t = int(tmp[0])

        if t == 1:
            i = int(tmp[1])
            base[i] = 1
            active[i] = True
            tag[i] = add[i % c]

        elif t == 2:
            k = int(tmp[1])
            add[k] += 1

        else:
            i = int(tmp[1])

            while True:
                if not active[i]:
                    out.append(str(i))
                    break

                length = base[i] + (add[i % c] - tag[i])
                ni = i + length
                if ni > n:
                    ni = n

                i = ni

    sys.stdout.write("\n".join(out))

if __name__ == "__main__":
    solve()
```该实现将每根藤蔓的基本长度与累积生长分开存储。 关键细节是`tag[i]`快照：没有它，所有残留类增量都会错误地在不同的藤蔓上累积。 跳转循环直接遵循过程的定义，但依赖于 O(1) 长度计算。 

一个微妙的边界条件是将跳转限制为 n。 如果没有这个，索引可能会超出范围或创建无效的转换。 另一个重要的细节是，即使那里不存在藤蔓，第 n 层也始终被视为终端汇。

 ## 边缘情况

 一种重要的边缘情况是在多次残基更新后创建藤蔓。 快照存储在`tag[i]`确保旧的增量不会追溯应用。 例如，如果在两次影响其残留类别的类型 2 更新后在 i 创建藤蔓，则其初始有效增长为零，而不是二。 

另一个边缘情况是更新后重复遍历同一区域。 由于遍历总是使用当前全局计数器重新计算长度，因此它总是反映更新的结构而不需要重新处理。 

最后的边缘情况是在没有藤蔓的层上开始时立即终止。 在这种情况下，算法直接输出起始索引，因为不可能进行跳跃。
