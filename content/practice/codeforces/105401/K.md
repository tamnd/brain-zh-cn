 ## 解决方案
 ## 问题理解

 我们给出了从 1 到 N 的一系列位置，我们必须为每个位置分配一个 0 到 K 之间的值。在这个数组的顶部，有 M 个约束。 每个约束指定一个段 [l, r] 并要求该段内的值之和恰好为 K。 

任务不是计算此类数组的数量，而是构造任何有效的数组或报告不存在。 

这些约束意味着段之间的全局耦合。 每个位置同时对许多段总和做出贡献，因此我们不能单独对待段。 相反，一个索引上的每个选择都会传播到覆盖它的所有段。 

最重要的结构观察来自这样一个事实：K 非常小，最多为 20。这强烈表明每个位置的行为就像有限的资源，必须分布在重叠的区间上。 

输入规模很大，测试用例中的总 N 和 M 分别高达 4e5 和 2e5。 这排除了任何与段有关的二次方或幼稚的每段扫描。 每个测试用例的任何解决方案都必须大致线性或接近线性。 

天真的解释会尝试贪婪地赋值，然后重复检查所有段。 这在两个方面是失败的。 首先，重新计算段和使得速度太慢。 其次，不了解重叠的贪婪分配很容易破坏可行性。 

当段以嵌套方式重叠时，会出现微妙的失败情况。 例如，如果我们有 K = 1 的线段 [1,3] 和 [2,4]，那么在位置 2 处选择一个值来满足第一个线段可能会妨碍满足第二个线段。 局部贪婪决策并不安全，因为约束是非局部相互作用的。 

另一个有问题的情况是相互矛盾的报道。 如果一个段迫使一个区域积累过多的价值，但重叠的段以不同的方式限制同一区域，则天真的分配可能直到很晚或根本才发现不可能性。 

## 方法

 暴力方法会尝试为所有 N 个位置赋值并检查所有 M 个段。 每次检查都需要在一个范围内求和，在最坏的情况下给出 O(NM)。 即使使用前缀和来加速检查，我们仍然需要反复调整分配，而 (K+1)^N 的搜索空间使得这完全不可行。 

关键的观察结果是 K 非常小，并且每个段都强制执行精确的总和。 我们可以考虑跨区间分布 K 个单位，而不是考虑任意实值约束。 每个部分都需要固定的总质量，并且每个位置最多只能贡献 K 个总质量。 

这表明将每个位置视为承载一定量的“容量消耗”，必须满足通过它的所有段。 真正的突破是以结构化顺序处理约束并逐步执行它们，以便每个段在处理时都得到准确满足，而不会违反之前的约束。 

我们可以将问题解释为维护一个数组，其中每个段都施加流量要求。 如果我们以一致的顺序处理片段，并始终将所需的质量推到片段内最右边的可用位置，我们就可以避免未来的冲突。 这是基于区间处理的经典贪心构造。 

我们维护一个跟踪当前分配值的结构，并确保当我们处理一个段时，我们仍然可以在其中分配 K 个单元。 如果不是，则该实例是不可能的。 由于 K 很小，我们可以在保持可行性的同时显式分配单元。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O((K+1)^N·M) | O((K+1)^N·M) | O(N) | 太慢了|
 | 最佳 | O((N + M) log N) 或 O(N + M) | O(N) | 已接受 |

 ## 算法演练

关键思想是将每个部分视为必须放入当前可用容量的 K 个单位的需求。 

我们按照右端点的升序处理片段。 这种顺序确保当我们满足某个段时，任何未来的段要么进一步扩展，要么稍后开始，因此我们不会意外破坏已处理约束的可行性。 

1. 按右端点 r 对所有线段进行排序。 如果两个线段共享 r，则按左端点递增对它们进行排序。 
2. 维护一个长度为 N 的数组 a，初始化为 0。当满足段时，我们将增加条目。 
3. 对于按排序顺序的每个段 [l, r]，计算该段中已分配了多少 sum。 这可以使用 Fenwick 树或线段树来维护，以便我们可以有效地查询范围总和。 
4. 如果 [l, r] 中的当前总和超过 K，我们立即知道此配置无效，因为我们无法在不破坏先前段的情况下稍后减小值。 
5. 如果当前总和小于K，则需要在[l,r]内添加（K减去当前总和）个单位。 我们贪婪地将这些单元从位置 r 开始向左放置，总是填充到 K 的位置。 
6. 在添加单位时，我们始终遵守上限 a[i] ≤ K。如果在达到 K 个附加单位之前我们用完 [l, r] 内的空间，则该实例是不可能的。 
7. 处理完所有段后，输出构造好的数组。 

从右到左的贪婪放置之所以有效，是因为较早的段被锚定到较小的右端点。 从右侧填充可以避免消耗较早结束的段所需的潜在空间。 

### 为什么它有效

 在每一步中，我们都确保所有已处理的部分都完全满意。 我们从不减少值，因此之前满意的段仍然有效。 因为我们通过增加右端点进行处理，所以任何可能重叠的线段都会以受控的方式共享其右边界：一旦我们完成了一个线段，后面的操作就不会在其间隔之外的位置引入多余的质量，而不会在至少延伸到最右侧的线段中得到考虑。 

不变的是，在处理到给定右端点 R 的段之后，每个 r ≤ R 的段都完全满足，并且没有位置超过 K。贪婪填充确保段中的任何剩余缺陷始终放置在最新的可能位置，为未来的约束保留灵活性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class Fenwick:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)

    def add(self, i, v):
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

    def range_sum(self, l, r):
        return self.sum(r) - self.sum(l - 1)

def solve():
    n, m, k = map(int, input().split())
    segs = []
    for _ in range(m):
        l, r = map(int, input().split())
        segs.append((r, l))
    segs.sort()

    bit = Fenwick(n)
    a = [0] * (n + 1)

    for r, l in segs:
        cur = bit.range_sum(l, r)
        if cur > k:
            print(-1)
            return
        need = k - cur
        i = r
        while need > 0 and i >= l:
            if a[i] < k:
                add = min(k - a[i], need)
                a[i] += add
                bit.add(i, add)
                need -= add
            i -= 1
        if need > 0:
            print(-1)
            return

    print(*a[1:])

t = int(input())
for _ in range(t):
    solve()
```芬威克树维护前缀和，以便可以有效地查询每个段。 如果没有它，我们将重复扫描范围，这在上限时会变得太慢。 

贪心内循环是解决方案的建设性部分。 我们总是尝试将剩余的所需金额尽可能放在段内的右侧。 这避免了干扰较早结束的较早段，因为如果需要，这些较早段已经填充了左侧位置。 

数组`a`强制执行值不能超过 K 的每个位置约束。Fenwick 树镜像这些更新，以便将来的段查询反映所有分配。 

一个常见的微妙错误是填充段时从左到右迭代。 这破坏了正确性，因为它太早消耗了低索引，这可能是未来处理顺序中较早结束的段所需要的。 

## 工作示例

 考虑一个 N = 4、K = 2 和段 [1,2]、[2,4] 的小实例。 

我们按右端点排序：[1,2] 然后是 [2,4]。 

| 细分 | Range | Current sum | Need | 行动|
 | ---| ---| ---| ---| ---|
 | [1,2]| 1-2 | 0 | 2 | fill at 2 then 1 |
 | [2,4]| 2-4 | 0 | 2 | fill at 4 then 3 |

 在第一段之后，我们可能有 a = [1,1,0,0]。 第二次之后，我们得到 a = [1,1,1,1]。 

这表明较早的部分将质量集中到右侧，为后面的部分留下了灵活性。 

现在考虑一个不可能的情况：N = 3，K = 2，其中段为 [1,2]、[2,3]、[1,3]。 

处理 [1,2] 时，我们放置 2 个单位，可能先在位置 2，然后在位置 1，给出 a = [1,1,0]。 处理 [2,3] 时，我们放置了 2 个单元，但位置 2 已满，因此我们使用位置 3，但无法满足最终段 [1,3] 的第一个段约束。 最终，当[1,3]在不超出容量的情况下无法平衡时，我们发现不可能性。 

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 | O((N + M) log N) | 每个segment查询和更新都使用Fenwick运算，每个位置最多递增K次 |
 | 空间| O(N) | 数组和 Fenwick 树存储 |

 约束允许最多 4e5 个总 N 和 2e5 个总段，因此对数因子是安全的。 由于 K 很小，内部填充循环总体上保持有界。

 ## 边缘情况

 临界边缘情况是当一个段完全包含在一个已经被早期段饱和的区域内时。 假设段 [2,3] 在 [1,4] 已经消耗该区域中的所有可用容量之后到达。 当处理[2,3]时，算法将计算当前总和已经等于K或更大。 如果超过K，我们立即拒绝。 如果它等于 K，则不采取任何操作，从而保持正确性。 

另一个微妙的情况是长度为 1 的段。例如，[i,i] 强制 a[i] = K。如果另一个段也覆盖 i，则通过 Fenwick 和检查，任何不匹配都会立即可见。 这样就能及早发现矛盾。 

第三种情况涉及重叠细分市场的长链，其中每个新细分市场都会将需求稍微向右移动。 从右到左的填充确保较早的段永远不会缺乏所需的容量，因为它们始终比排序处理顺序中较早的索引具有优先级。
