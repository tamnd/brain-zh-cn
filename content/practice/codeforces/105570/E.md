 ## 解决方案
 ## 问题理解

 我们被给予$N$竞争对手。 每位参赛者$i$有一个未知的参数$h_i$，代表他们完成的速度。 较小的值$h_i$总是对应于较早的完成时间，因此最终排名恰好是所有的排序$h_i$按递增顺序。 如果我们对竞争对手进行排序$h_i$，我们得到一个排列$1..N$描述比赛结果。 

各参赛者并未完全透露$h_i$，只有一个区间$[L_i, R_i]$。 我们必须选择一个有效的整数$h_i$在那个区间内。 

此外，我们还获得有关最终排名的部分信息：一个数组$p$， 在哪里$p_i$是竞争对手的位置$i$按最终顺序，或$0$如果未知。 必须填补缺失的位置，以便最终$p$变为完全排列$1..N$，并且此排列必须与所选的排序相匹配$h_i$。 

所以任务是双重的。 我们必须为每个竞争对手分配一个值$h_i \in [L_i, R_i]$，并且我们必须为它们分配一个与递增一致的总顺序$h_i$。 同时，任何已经固定的排名位置$p$必须受到尊重。 

一个关键的观察结果是，一旦$h_i$值是固定的，排名完全由排序决定。 除了打破平局之外，最终的排列没有任何自由，并且平局可以被忽略，因为我们可以在保持在区间内的同时稍微扰乱值。 

约束条件很大，有$N$最多$2 \cdot 10^5$。 这排除了尝试所有排列或使用的任何解决方案$O(N^2)$可行性检查。 我们需要一些接近的东西$O(N \log N)$，通常涉及排序或优先级队列。 

当间隔紧密且重叠时，会出现微妙的边缘情况。 例如，如果一位参赛者必须排在前面，但其下限非常大$L_i$，我们可能被迫分配一个大的$h_i$，这可能会阻止以后的分配。 另一种失败情况发生在固定排名位置迫使排序无法在区间内实现时，即使每个区间单独可行。 

## 方法

 直接的尝试是尝试竞争对手的所有排列，并且对于每个排列模拟我们是否可以分配递增的$h_i$其区间内的值。 给定固定的顺序，可行性很容易：我们从左到右遍历并设置每个$h_i$尽可能小，同时尊重单调性及其间隔。 具体来说，如果先前的值为$t$，我们设置$h_i = \max(L_i, t+1)$，并检查是否超过$R_i$。 

这种暴力的想法是正确的，但完全不可行，因为有$N!$排列，甚至检查一个排列是$O(N)$，给出阶乘时间。 

关键的见解是我们实际上不需要尝试所有排列。 我们想要的排列正是最终的排序顺序$h_i$。 因此，我们不应该猜测排列然后验证它，而是应该构造一个保证允许有效赋值的排列。 

施工变成了一个贪婪的调度问题。 我们从左到右构建最终顺序。 在每个位置，我们选择一个未使用的竞争对手，其间隔允许将其放置在序列中的下一个，同时保持未来的选择尽可能灵活。 订单确定后，我们分配$h_i$贪婪地。 

困难在于如何选择下一个竞争对手。 正确的结构来自于将每个竞争对手视为具有发布时间的任务$L_i$和截止日期$R_i$，但附加的约束是分配的值必须严格沿顺序增加。 这将问题转化为选择一个避免过早强制大幅跳跃的顺序。 

出现了一个标准的贪婪策略：总是选择具有最小的可用候选者$R_i$。 直观上来说，如果我们推迟了紧迫期限的工作，我们就有可能让以后的工作变得不可能，而尽早选择宽松的时间间隔并不会限制未来的灵活性。 

我们还必须尊重部分固定的排列$p$。 这意味着某些位置是被迫的，所以在步骤$k$， 如果$p_k \neq 0$，我们必须放置该特定的竞争对手。 如果当时不可行，则施工立即失败。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力排列 |$O(N! \cdot N)$|$O(N)$| 太慢了 |
 | 贪心+优先级队列构建|$O(N \log N)$|$O(N)$| 已接受 |

 ## 算法演练

 我们构建最终的排列$p'$从左到右同时决定顺序。 

1. 收集所有尚未排名的参赛者。 我们还跟踪哪些竞争对手已经被给定的部分排列固定$p$。 我们将处理职位$1$通过$N$。 
2. 维持当前时间值$t$，代表最后分配的$h$构造顺序中的值。 最初$t = 0$。 
3. 维护一个包含所有未使用的竞争对手的数据结构，按递增顺序排序$R_i$。 最小堆是合适的。 
4. 各位置$k$， 如果$p_k$是固定的，我们尝试选择那个竞争对手。 否则，我们从堆中选择一个候选者。 
5. 考虑候选人时$i$，我们计算它将在这个位置收到的值：$h_i = \max(L_i, t+1)$。 如果这个值超过$R_i$，然后放置$i$这里使得施工变得不可能。 
6. 当我们挑选有效候选人时$i$，我们将其分配给位置$k$，从堆中删除它，并更新$t = h_i$。 
7. 如果在任何一步我们都找不到有效的候选人，那么答案是不可能的。 

原因最小$R_i$策略的工作原理是任何有效的解决方案最终都必须放置每个元素，并且元素具有较小的$R_i$移动空间较小。 如果我们推迟它们，我们就有超出其允许范围的风险，而提早放置它们不会降低其他人的可行性，超出必要的增加$t$。 

### 为什么它有效

 在任何步骤中，算法都认为所有先前选择的分配都是可行的并且严格递增。 在剩下的元素中，选择一个最小的$R_i$确保我们永远不会以一种会阻塞所有有效计划的方式将受约束较大的元素推迟到受约束较少的元素后面。 

如果存在解决方案，则总有一种方法可以对其进行转换，以便在每一步中选择的元素都是最小的元素$R_i$可以出现在某些有效完成中的所有元素之间。 这个交换论证保证贪婪选择永远不会删除所有解决方案（如果存在）。 

施工后$p'$, 分配$h_i$按这个顺序贪婪总是成功的，因为构造明确地确保了$h_i \le R_i$每一步都保持递增的序列可行性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline
import heapq

def solve():
    N = int(input())
    p = list(map(int, input().split()))
    L = list(map(int, input().split()))
    R = list(map(int, input().split()))

    # convert p into position constraints
    pos_of = [-1] * (N + 1)
    fixed_at = [-1] * (N + 1)

    for i in range(N):
        if p[i] != 0:
            pos_of[i + 1] = p[i]
            fixed_at[p[i]] = i + 1

    used = [False] * (N + 1)
    heap = []

    # push all items into heap by R
    for i in range(1, N + 1):
        heapq.heappush(heap, (R[i - 1], i))

    res = [0] * (N + 1)
    t = 0

    for k in range(1, N + 1):
        chosen = None

        if fixed_at[k] != -1:
            i = fixed_at[k]
            # extract i from heap lazily
            temp = []
            found = False

            while heap:
                r, x = heapq.heappop(heap)
                if x == i and not used[x]:
                    found = True
                    break
                if not used[x]:
                    temp.append((r, x))

            for item in temp:
                heapq.heappush(heap, item)

            if not found:
                print("No")
                return

            chosen = i
        else:
            temp = []
            found = False

            while heap:
                r, i = heapq.heappop(heap)
                if used[i]:
                    continue
                hi = max(L[i - 1], t + 1)
                if hi <= R[i - 1]:
                    chosen = i
                    found = True
                    break
                temp.append((r, i))

            for item in temp:
                heapq.heappush(heap, item)

            if not found:
                print("No")
                return

        used[chosen] = True
        hi = max(L[chosen - 1], t + 1)
        t = hi
        res[chosen] = k

    # build final arrays
    p_out = [0] * N
    h_out = [0] * N

    for i in range(1, N + 1):
        p_out[i - 1] = res[i]
        h_out[i - 1] = max(L[i - 1], 0)  # placeholder

    # reconstruct h in order of p_out
    order = sorted(range(1, N + 1), key=lambda x: res[x])
    t = 0
    for i in order:
        hi = max(L[i - 1], t + 1)
        if hi > R[i - 1]:
            print("No")
            return
        h_out[i - 1] = hi
        t = hi

    print("Yes")
    print(*p_out)
    print(*h_out)

if __name__ == "__main__":
    solve()
```第一阶段在尊重固定位置的同时构建排列。 堆强制执行“最小$R_i$第一”策略，而可行性检查使用$t$确保我们永远不会在区间之外分配值。 

第二阶段重新计算$h_i$使用最终订单。 这种分离很重要，因为构造阶段仅保证排序可行性，而第二遍保证实际的数字分配。 

一个常见的陷阱是尝试分配$h_i$在排列构造期间并同时强制执行排序约束。 将这些阶段分开可以避免意外的依赖循环。

 ## 边缘情况

 当固定位置迫使竞争对手拥有非常大的优势时，就会出现关键的边缘情况。$L_i$出现得太早。 在这种情况下，算法尝试将其分配到所需的位置，计算$h_i = \max(L_i, t+1)$，如果超过则立即拒绝$R_i$。 这捕获了不可能的配置，其中约束迫使值超出区间。 

当所有间隔严重重叠时，会发生另一种微妙的情况，但错误的贪婪选择会推迟紧密的间隔。 堆策略通过始终暴露最小的来防止这种情况发生$R_i$候选者优先，确保高度受限的元素不会意外延迟。 

最后一种情况是多个参赛者具有相同的间隔。 在这种情况下，它们之间的任何排序都是有效的，并且贪心算法仍然会选择一致的序列，因为可行性仅取决于维持递增$t$，而不是区分相等的约束。
