 ## 解决方案
 ## 问题理解

 我们正在构建一个由字符串“orz”的重复副本组成的文本。 目标是至少生产$N$在最短的时间内复制该字符串。 

在任何时刻，Jason 都拥有当前的文本量，并且他可以执行三个操作。 他可以输入一个“orz”，这会将计数加一并花费$A$秒。 他可以复制整个当前文本$B$秒，将其存储在剪贴板中。 他可以粘贴剪贴板$C$秒，每次粘贴都精确添加已复制的“orz”数量。 

该过程一开始是空的，所以开始的唯一方法就是打字。 

困难来自于复制和粘贴之间的交互。 复制是固定开销，而粘贴则缩放当前文本量。 一旦我们有$x$副本，复制操作后跟$k-1$糊剂总数可以从$x$到$k \cdot x$，这通常比再次手动输入所有内容要便宜得多。 因此，决定是何时停止打字并切换到指数增长。 

这些限制使得对所有操作序列进行强力探索变得不可能。 自从$N$可以达到$10^9$，任何模拟每个步骤或在所有状态上执行 DP 的解决方案$N$将会失败。 甚至$O(N)$或者$O(N \log N)$过渡太大。 

一个天真的尝试会模拟所有可能的打字、复制和粘贴操作序列，跟踪当前的大小和时间。 这会失败，因为可达状态的数量增长得非常快。 另一个天真的想法是 DP 超过当前“orz”的数量，但状态空间太大了。 

当过早复制不好时会出现微妙的边缘情况。 例如，如果$A$非常小并且$B$很大，打字可能完全占主导地位。 相反，如果$C$很小，早期的复制就变得极其强大。 正确的解决方案必须平衡这两种制度，而不是遵循固定的策略。 

## 方法

 暴力方法试图将过程建模为最短路径问题，其中每个状态是当前的“orz”副本数量，并且转换对应于打字、复制和粘贴。 这在原则上是正确的，因为每个操作都被明确地表示，但是状态空间增长到$N$，并且每个状态都有多个转换。 在最坏的情况下，这至少会变成$O(N)$州，这远远超出了限制。 

关键的观察是，一旦我们决定以某种大小复制$x$，之后我们能做的最好的事情就是重复粘贴，直到达到或超过$N$。 在完成复制粘贴突发之前没有理由再次交错输入，因为与从更大的基数粘贴相比，输入会将我们重置为更糟糕的状态。 

这将问题简化为单一的结构决策：选择时机$x$当我们停止打字并执行一次复制粘贴扩展以达到$N$。 在那之后，我们再也不会打字了。 

如果我们选择停止输入$x$，成本由三部分组成：打字成本$A \cdot x$, 一份成本$B$，以及足够的焊膏至少达到$N$。 每个粘贴添加$x$，所以粘贴的数量是$\lceil N/x \rceil - 1$，每项成本$C$。 

这将问题转化为最小化所有表达式中的单个表达式$x$，可以通过仅检查楼层划分断点周围的候选值来有效优化。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力状态搜索 |$O(N)$或者更糟|$O(N)$| 太慢了 |
 | 优化的单分支评估 |$O(\sqrt{N})$每次测试 |$O(1)$| 已接受 |

 ## 算法演练

 我们将问题重新定义为在切换到复制粘贴增长之前选择手动键入的次数。 

1. 考虑我们首先建立一些数字$x$纯粹通过打字来获得“orz”。 这正好花费$A \cdot x$。 在进入最终增长阶段之前混合复制操作没有任何好处，因为只有在我们拥有稳定的基础大小后，早期复制才有意义。 
2. 一旦我们修复$x$，我们执行一次复制操作。 这需要花费$B$秒并给我们一个剪贴板，其中包含$x$。 
3. 从那时起，每个粘贴都会添加$x$。 我们希望至少达到$N$，所以我们需要$k = \lceil N/x \rceil$总块大小$x$， 意义$k-1$粘贴。 这有助于$(k-1)\cdot C$。 
4. 固定成本的总成本$x$因此是$$A \cdot x + B + (\lceil N/x \rceil - 1)\cdot C.$$5. 我们评估所有相关的这个表达式$x$。 而不是检查所有值$N$，我们只需要其中的值$\lceil N/x \rceil$变化。 这些变化发生在除数附近$N$，所以检查$x$最多$\sqrt{N}$还检查$x = \lfloor N/i \rfloor$因为这些价值观涵盖了所有候选人。 

### 为什么它有效

 该函数在上限项中是分段常数，并且在每个区间内，其中$\lceil N/x \rceil$是固定的，成本线性增长$x$。 这意味着最小值不能位于大平坦区间的深处而不靠近其边界。 通过评估划分变化引起的边界点，每个潜在的最佳配置都包含在候选集中，确保达到真正的最小值。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve_case(A, B, C, N):
    ans = float('inf')

    # try all x up to sqrt(N)
    x = 1
    while x * x <= N:
        # case 1: x as typing count
        k = (N + x - 1) // x
        cost = A * x + B + (k - 1) * C
        ans = min(ans, cost)

        # also try paired value N // x
        y = N // x
        if y > 0:
            k = (N + y - 1) // y
            cost = A * y + B + (k - 1) * C
            ans = min(ans, cost)

        x += 1

    return ans

def main():
    T = int(input())
    out = []
    for _ in range(T):
        A, B, C, N = map(int, input().split())
        out.append(str(solve_case(A, B, C, N)))
    print("\n".join(out))

if __name__ == "__main__":
    main()
```该解决方案枚举了以下候选值$x$以围绕平方根分解的对称方式。 对于每位候选人$x$，它计算需要多少个完整的复制粘贴块才能达到$N$，然后评估总成本。 对配对值重复相同的操作$N/x$，它捕获了上限发生变化的除法断点的另一侧。 

关键的实现细节是仔细使用整数算术进行上限除法。 表达式$(N + x - 1) // x$确保正确舍入而没有浮点错误，这在以下情况下至关重要$N$很大。

 ## 边缘情况

 当$N = 1$，算法仍然考虑输入一次并进行复制粘贴循环，但公式自然更喜欢$x = 1$因为任何副本都会产生额外费用$B$。 计算表达式正确地减少为$A$。 

当复制成本极其昂贵时，每个候选人$x$产生大量固定开销$B$，并且最小值出现在最小的位置$x$，对应于纯打字。 枚举包括$x = 1$，所以直接处理这个case。 

当粘贴极其便宜时，最佳解决方案会转向小尺寸$x$值，因为放大几乎是免费的。 平方根枚举仍然明确地捕获这些小值，确保正确性而不需要更深的 DP 状态。 

什么时候$N$是完全平方或有大除数，配对评估为$N/x$确保边界转换在哪里$\lceil N/x \rceil$不会错过任何变化，从而防止候选人覆盖范围出现空白。
