 ## 解决方案
 ## 问题理解

 我们给定一个非常大的整数`a`和另一个整数`b`，都写为十进制字符串。 数量`a`正好多四位数字`b`。 我们可以删除正好四位数字`a`，而不重新排列剩余的数字，并将结果解释为一个新的整数（在构造过程中允许使用前导零，但在值比较中自然会被忽略）。 

任务是计算有多少种不同的方式选择要删除的四个位置`a`产生一个严格小于的结果数`b`。 

从输入限制中观察到的一个关键是数字可能非常大，最多可达 10,000 位。 这立即排除了任何到内置整数的转换或任何试图通过每个候选者的大量字符串操作来简单地枚举所有子序列的方法。 即使生成四个删除的所有组合也只是组合可行的，因为它大致是$\binom{n}{4}$，这对于$n \le 10000$是关于$10^{15}$，太大了。 

影响解决方案的真正约束是删除四位数字后，所得数字的长度与`b`。 这意味着比较是在相同长度的字符串上按字典顺序进行的，使用通常的整数比较规则，包括前导零的影响。 

边缘情况主要源于数字相等边界。 如果结果数字与以下共享一个长前缀`b`，那么一位数的差异决定了比较。 另一个微妙的问题是构造数字中的前导零：它们不会影响数值，但如果处理不仔细，它们确实会影响字典比较。 例如，`"0123"`在数值上等于`"123"`，但从字典顺序来看它更小。 由于比较是数字的，而不是基于字符串的，因此我们必须仔细规范比较或以避免歧义的方式进行推理。 

## 方法

 暴力方法会选择四个索引的每个子集来从中删除`a`，构建结果字符串，去掉前导零，然后与`b`。 每张支票费用$O(n)$，并且有$O(n^4)$子集，大致导致$10^{16}$最坏情况下的操作。 这远远超出了任何可行的限度。 

关键的见解是只进行了四次删除，这是一个常数。 这建议采用组合 DP 或数字 DP 风格的解决方案，其中我们从左到右处理字符串并跟踪我们使用了多少次删除，同时与`b`。 

在每个位置`a`，我们决定是否删除该数字或保留它。 如果我们保留它，它就会对构建的数字做出贡献； 如果我们删除它，我们会使用四个允许删除之一。 结果数字的长度必须等于`b`，所以确切地说$|a| - 4 = |b|$数字被保留。 

与的比较`b`引入了一个额外的维度：在构造保留序列时，我们必须跟踪我们是否已经严格小于`b`, 到目前为止等于或超过`b`。 然而，超出可以被修剪掉，因为一旦前缀大于`b`，它永远不会再次有效。 

这导致数字 DP 的状态基于位置`a`，使用了多少次删除，以及与`b`。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O(n^4 \cdot n)$|$O(n)$| 太慢了 |
 | 数字DP |$O(n \cdot 4 \cdot 2)$|$O(n \cdot 4 \cdot 2)$| 已接受 |

 ## 算法演练

 我们将构建视为精确选择`len(b)`数字来自`a`按顺序排列，相当于删除四位数字。 

我们在字符串上定义一个 DP`a`。 

1. 初始化一个DP表，其中`dp[i][k][t]`表示使用第一种方法的数量`i`的数字`a`，完全删除`k`数字，以及在哪里`t`表示构造的前缀是否仍然等于`b`（`t = 0`）或已经更小（`t = 1`）。 

我们只需要两个状态进行比较的原因是，一旦我们变小，我们就不再需要对它们施加约束`b`。 
2. 从位置 0 开始`k = 0`和`t = 0`。 尚未处理任何数字，因此构造的前缀基本上等于`b`。 
3. 各位置`i`在`a`，考虑两个动作：删除`a[i]`或保留`a[i]`。 我们跳过超过四次删除的转换。 

删除很简单：我们转到`dp[i+1][k+1][t]`因为删除并不影响构造数。 
4.如果我们保留`a[i]`，我们将其附加到构造的数字上。 然后我们确定新的比较状态`t'`。 如果我们已经在`t = 1`，我们留在那里。 如果我们在`t = 0`，我们将新添加的数字与中对应的数字进行比较`b`。 如果它更小，我们移动到`t = 1`。 如果相等，我们留在`t = 0`。 如果它更大，我们就放弃这个转变。 

此步骤强制我们只构建小于或等于的数字`b`按字典顺序排列，同时允许 DP 仅计算有效的结构。 
5. 处理完所有位置后，我们取所有状态的总和，其中恰好使用了四次删除，并且构造的数字的长度等于`b`。 其中，我们只计算最终比较相等或已经较小的状态； 因为完整的相等意味着等于`b`，仅当在某个时刻强制执行严格较少的条件时，它仍然有效。 因此我们只计算`t = 1`。 

### 为什么它有效

 DP 保持了每个状态代表一个唯一的子序列的不变量`a`具有固定数量的删除，并且`t`正确捕获当前前缀是否严格小于或仍然与相应的前缀相关`b`。 因为我们决不允许超出字典顺序的转换`b`，每个计数的终止状态对应一个有效的子序列。 相反，每个有效子序列都恰好对应于通过 DP 的一条路径，因为保留或删除的选择完全决定了子序列。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve(a, b):
    n = len(a)
    m = len(b)
    # dp[k][t]: k deletions used, t in {0 equal, 1 already less}
    dp = [[0, 0] for _ in range(5)]
    dp[0][0] = 1

    for i in range(n):
        ndp = [[0, 0] for _ in range(5)]
        for k in range(5):
            for t in range(2):
                cur = dp[k][t]
                if not cur:
                    continue

                # delete a[i]
                if k < 4:
                    ndp[k + 1][t] += cur

                # keep a[i]
                if k - (i - (k)) < m:  # conceptual guard, not strictly needed
                    if t == 1:
                        nt = 1
                        if k <= 4:
                            ndp[k][nt] += cur
                    else:
                        # determine position in constructed string
                        pos = i - k
                        if pos < m:
                            if a[i] < b[pos]:
                                ndp[k][1] += cur
                            elif a[i] == b[pos]:
                                ndp[k][0] += cur
                            else:
                                pass
        dp = ndp

    return dp[4][1]

def main():
    t = int(input())
    for _ in range(t):
        a, b = input().split()
        print(solve(a, b))

if __name__ == "__main__":
    main()
```DP 是围绕这样的想法构建的：删除改变了之间的对齐`a`以及结果字符串，这就是为什么位置`b`导出为`i - k`。 这避免了显式构造字符串。 

比较逻辑嵌入在转换中：一旦一个数字使构造的序列小于`b`，我们不再需要强制执行平等约束。 

关键的微妙之处在于确保我们永远不会进行超出长度的比较`b`，因为一旦进行了四次删除，构造的数字总是具有完全相同的长度。

 ## 边缘情况

 一种微妙的边缘情况是删除数字会在结果数字中产生前导零。 例如，删除数字`"10005"`可以产生`"0005"`，其数值等于`5`。 DP 可以正确处理这个问题，因为比较是逐位进行的`b`，当适当的前缀时，前导零自然会比较为较小的数字`b`均非零。 

另一种情况是当所有删除都聚集在前面时`a`。 对于像这样的输入`a = "1000001234"`和一个小`b`，早期的删除会改变对齐方式，使得后面的数字突然主导比较。 民主党通过以下方式解释了这一点：`pos = i - k`映射，确保构造的字符串保持正确对齐`b`不管删除是如何分布的。 

当构造的前缀匹配时会出现第三种情况`b`很长一段时间，直到接近尾声才出现分歧。 这正是`t`状态很重要：它可以防止过早计数并确保只有真正较小的前缀才被视为有效。
