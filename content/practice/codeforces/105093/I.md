 ## 解决方案
 ## 问题理解

 我们得到了一系列按固定顺序战斗的 Boss。 对于每个老板来说，有两种方法来应对战斗。 直接战斗会消耗boss的真实力量，而伏击则消耗较小的值，但它会带来全局副作用：每次伏击boss时，所有剩余的boss在真实形态和伏击形态下都会变得更强，特别是它们的力量会加倍。 

这会在决策之间产生耦合。 选择尽早伏击会增加以后一切的成本，而选择推迟伏击可以避免通货膨胀，但可能会迫使你为早期的老板支付更高的直接成本。 

任务是为每个老板选择直接战斗或伏击，以最小化所有双倍效果后的总累积成本。 

所有测试用例中的 boss 总数最多为 2 · 10^5 的约束意味着任何比每个测试用例的线性或线性数学更差的解决方案都将陷入困境。 针对位置和伏击次数的二次动态规划方法立刻就太慢了，因为它需要跟踪每个位置最多 n 个可能的“先前伏击次数”状态。 

一个微妙的困难来自于这样一个事实：每次伏击不仅会影响当前的老板，还会永久地改变所有未来决策的规模。 一个幼稚的错误是独立对待每个 boss 并选择 min(t_i, a_i)，这完全忽略了加倍传播。 另一种常见的错误尝试是根据 t_i 与 a_i 的局部比较来贪婪地做出决定，这也会失败，因为后来的 Boss 的成本取决于已经选择了多少伏击。 

## 方法

 蛮力的想法很简单：尝试伏击老板的每个子集，模拟过程，并计算最终的成本。 对固定子集的模拟需要线性时间，因为我们维持当前的倍增因子并相应地累积成本。 由于有 2^n 个子集，因此当 n 非常小（大约 20 左右）时，这很快就会变得不可行。 

困难在于伏击不是本地修改。 每次伏击都会将所有未来的贡献乘以 2，这意味着决策是乘法而不是加法相互作用。 关键的观察是，我们可以通过从右到左处理并跟踪后缀中已经选择了多少伏击来线性化这种交互。 这个后缀的数量完全决定了当前老板的比例因子，更重要的是，它还决定了未来的决策将受到怎样的影响。 

这导致了一种贪婪的解释：在每个位置，我们唯一需要携带的状态是我们已经将后缀加倍了多少次（相当于，选择了右侧的伏击次数）。 一旦知道了这一点，当前老板的成本贡献就完全确定为任一选择，而未来仅取决于该决定如何改变加倍计数。 

我们可以将其与朴素的动态规划公式 dp[i][k] 进行比较，其中 k 是后缀中的伏击数量。 这立即表明 O(n^2)。 关键的简化是，当我们从右向左移动时，k 会确定性地演化，因此我们永远不需要明确地对其进行分支。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(2^n·n) | O(2^n·n) | O(1) | O(1) | 太慢了 |
 | DP 超过位置和伏击数 | O(n^2) | O(n^2) | O(n^2) | O(n^2) | 太慢了 |
 | 最优后缀贪心模拟 | O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 我们从右到左处理 Boss，保持所有先前选择的伏击的效果，作为应用于剩余未处理前缀的乘数。

1. 初始化一个运行乘数 m = 1。这表示所有已处理的未来贡献由于早期的伏击而被缩放了多少次。 开始时，不存在伏击，因此乘数是中性的。 
2. 从最后一个 Boss 迭代到第一个 Boss。 在每个位置 i，两个选择都在当前乘数下进行评估，因为右侧后缀的所有成本都已经固定。 
3.如果我们直接与boss i战斗，我们支付t_i * m。 这不会改变 m，因为我们没有引入新的伏击。 
4. 如果我们伏击老板 i，我们支付 a_i * m。 此后，我们必须考虑问题的结构性影响：每个未来的老板（即所有早期的指数）的价值将再次翻倍。 即通过将 m 更新为 2m 来表示。 
5. 每一步的决策都是通过比较伏击与不伏击的边际效应，即它如何影响当前成本和应用于所有剩余决策的乘数来做出的。 关键是乘数已经编码了所有未来的缩放效应，因此局部决策是在正确的全局背景下做出的。 
6. 在我们继续进行时累计成本，始终应用当前的乘数。 

微妙的一点是，我们并没有将其视为独立的局部优化。 相反，乘数带有所有历史依赖性，确保每个决策都在正确的缩放环境中进行评估。 

### 为什么它有效

 不变量是，处理位置 i 后，乘数 m 恰好等于 2 乘以位置 i+1 到 n 中选择的伏击数量。 这保证了每个已处理的成本都已通过正确的缩放比例进行计算，并且每个未来的决策都将在正确的加倍次数下进行评估。 

由于每次伏击仅通过统一的乘法因子影响较早的位置，因此除了该指数计数之外没有隐藏状态。 任何最优策略都可以通过以相反的顺序做出决策来重构，同时保持不变性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    T = int(input())
    for _ in range(T):
        n = int(input())
        t = list(map(int, input().split()))
        a = list(map(int, input().split()))
        
        m = 1
        ans = 0
        
        for i in range(n - 1, -1, -1):
            # If we fight directly
            cost_head = t[i] * m
            
            # If we ambush
            cost_ambush = a[i] * m
            
            # Choose the cheaper option in current scaled state
            if cost_ambush < cost_head:
                ans += cost_ambush
                m *= 2
            else:
                ans += cost_head
        
        print(ans)

if __name__ == "__main__":
    solve()
```该实现直接反映了逆向处理的思想。 乘数 m 是我们维护的唯一状态，它捕获了所有先前伏击决策的累积效果。 每个步骤都会计算相同缩放下的两种可能成本，以确保比较的一致性。 

一个常见的实施陷阱是忘记乘数只有在选择伏击后才必须更新。 过早更新它会错误地增加当前老板的成本。 另一个微妙的问题是没有大整数的语言中的整数溢出，因为 m 可以呈指数增长； Python 自然地处理这个问题。

 ## 边缘情况

 最小的单老板案例很重要，因为当不存在未来时，乘数逻辑不能被错误地应用。 对于一个 boss，该算法可以正确地简化为选择 min(t_1, a_1)，因为 m 始终为 1 并且永远不会改变。 

当所有伏击值都极小时，算法会反复将乘数加倍，但仍继续选择伏击，因为每次局部比较在当前规模下仍然有利。 该不变量确保即使是大的指数增长也不会影响正确性，只会影响幅度。 

当伏击没有任何好处时，乘数在整个运行过程中保持固定为 1，并且解决方案崩溃为 t_i 的简单线性和，这表明该算法优雅地退化为普通情况，无需任何特殊处理。
