 ## 解决方案
 ## 问题理解

 我们得到一组项目，每个项目都标记为 1 到 n，以及一系列依赖关系规则，其形式为“项目 a 必须在项目 b 开始之前完成”。 这些规则定义了项目之间的定向关系，任务是确定所有项目是否可以按照尊重每个依赖性的某种顺序完成。 如果存在这样的顺序，我们必须输出所有项目的有效顺序。 如果存在多个有效订单，我们需要选择字典顺序最小的一个。 

从图的角度来看，每个项目都是一个节点，每个依赖项都是从 a 到 b 的有向边。 因此，输出是有向图的拓扑排序，附加约束是在所有有效的拓扑顺序中，我们必须选择字典顺序中最小的。 

约束达到 n = 100000 和 m = 200000。任何与 n 成二次方的解决方案（例如重复扫描所有节点以查找下一个有效项目）都将不起作用。 我们被限制在本质上线性或接近线性的时间，这意味着 O(n log n) 或 O(n + m) 是目标范围。 这已经强烈建议使用图遍历方法而不是任何组合搜索。 

一些边缘情况在这里很重要。 首先，如果存在循环，例如1依赖于2，2依赖于3，3依赖于1，那么就不可能进行排序，我们必须输出IMPOSSIBLE。 其次，当存在多个有效的下一个项目时，字典顺序最小的要求迫使我们总是更喜欢每一步可用的最小索引项目，否则贪婪但无序的拓扑排序将产生有效但不是最小的序列。 第三，当根本不存在依赖关系时，答案必须是 1 2 3 ... n。 

## 方法

 最简单的方法是重复扫描所有项目并选择满足先决条件的任何项目。 在每一步中，我们都会检查所有边或从头开始重新计算入度，以确定当前可用的节点。 从概念上讲，这是可行的，因为我们始终尊重依赖性，但在简单的实现中，每个选择步骤可能会花费 O(n + m)。 由于我们执行了 n 次，最坏的情况变为 O(n(n + m))，这对于输入限制来说太大了。 

关键的观察结果是，我们永远不需要从头开始重新计算依赖满意度。 我们实际需要的是所有当前可用项目的动态集合，即那些入度为零的项目。 一旦项目被采用，只有其外出邻居受到影响。 这自然意味着保持入度并逐步更新它们。 

为了强制执行字典顺序最小排序，我们必须始终在每一步选择最小的可用项目。 这将问题转化为具有优先级机制的标准拓扑排序任务。 我们使用最小堆而不是任意队列，以便始终提取入度为零的最小索引节点。 

每次我们从堆中删除一个节点时，我们都会模拟完成该项目并减少其邻居的入度。 任何入度为零的邻居都会添加到堆中。 这保证了我们始终保持可用节点的正确边界。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(n(n + m)) | O(n(n + m)) | O(n + m) | 太慢了 |
 | 最优（最小堆拓扑排序）| O((n + m) log n) | O((n + m) log n) | O(n + m) | 已接受 |

 ## 算法演练

 我们将问题转换为有向图并跟踪入度。

1. 为图构建邻接表并计算每个节点的入度。 这告诉我们每个项目还有多少先决条件。 
2. 初始化一个最小堆，并压入所有入度为零的节点。 这些项目可以立即启动，因为它们不依赖任何东西。 
3. 反复从堆中提取最小的节点。 这种选择确保了所有当前有效候选者中每一步的词典编排最小化。 
4. 将提取的节点附加到答案序列中，因为我们现在致力于在此位置完成该项目。 
5. 对于该节点的每个邻居，减少其入度。 这代表满足这些依赖项目的一个先决条件。 
6. 如果任何邻居的入度变为零，则将其推入堆中，因为它已符合执行条件。 
7. 继续，直到堆变空。 此时，要么所有节点都已被处理，要么存在循环。 
8. 最后，检查排序结果是否包含所有 n 个节点。 如果不是，则某些节点从未摆脱依赖关系，这意味着存在循环，因此输出 IMPOSSIBLE。 

关键的细节是，堆总是准确地反映当前有效的节点集，并且我们总是选择其中最小的一个。 

### 为什么它有效

 在每一步中，该算法都保持这样一个不变量：所有已经输出的节点相对于剩余图的入度为零，并且当前堆中的所有节点正是那些先决条件都已满足的节点。 由于边仅通过入度更新隐式删除，因此在其所有依赖项出现在序列中较早的位置之前，不会将任何节点插入到排序中。 堆排序确保在所有有效的下一个选择中，选择最小的索引，因此不存在字典序上更小的有效序列，该序列较早地偏离了构造的序列。 

## Python 解决方案```python
import sys
import heapq

input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    
    adj = [[] for _ in range(n + 1)]
    indeg = [0] * (n + 1)

    for _ in range(m):
        a, b = map(int, input().split())
        adj[a].append(b)
        indeg[b] += 1

    heap = []
    for i in range(1, n + 1):
        if indeg[i] == 0:
            heapq.heappush(heap, i)

    order = []

    while heap:
        u = heapq.heappop(heap)
        order.append(u)

        for v in adj[u]:
            indeg[v] -= 1
            if indeg[v] == 0:
                heapq.heappush(heap, v)

    if len(order) != n:
        print("IMPOSSIBLE")
    else:
        print(*order)

if __name__ == "__main__":
    solve()
```实现首先构建邻接表并计算入度，这是所有先决条件约束的结构表示。 然后用所有零入度节点播种堆，这些节点形成初始有效的启动项目集。 

主循环重复弹出最小的可用节点。 使用堆而不是队列可以强制字典顺序最小化； 用普通队列替换它仍然会产生有效的拓扑顺序，但不一定是最小的拓扑顺序。 

每次删除一个节点时，我们都会通过减少入度来通过传出边传播其影响。 仅当邻居的入度恰好为零时才会推送邻居，以确保在满足所有先决条件时它才符合资格。 

最终检查将构建的订单大小与 n 进行比较。 如果它们不同，则至少有一个节点永远无法通过入度减少来到达，这只能在存在循环的情况下发生。

 ## 边缘情况

 一个关键的边缘情况是多个节点在开始时入度为零。 例如，对于 5 节点图中的边 2 → 3 和 4 → 5，1、2 和 4 最初可能都是可用的。 该算法将它们全部放入堆中并始终选择最小的，以确保确定性的词典编排行为。 

另一种边缘情况是完全循环图，例如 1 → 2 → 3 → 1。这里没有节点达到入度零，因此堆开始为空或在初始化后立即变空。 输出列表仍然不完整，算法正确输出 IMPOSSIBLE。 

最后的边缘情况是具有断开连接的组件的图。 每个组件都通过入度传播独立处理，并且堆通过始终选择组件之间的最小可用节点来自然地将它们合并到单个全局排序中。
