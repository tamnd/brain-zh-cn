 ## 问题理解

 我们可以将输入重新解释为图表。 每个顶点代表类中的一个项目，每个边代表两个项目是兼容的。 有效的答案对应于选择三个顶点，使得它们之间的每一对都通过一条边连接，这意味着我们正在图中寻找三角形。 

除了简单地找到这样一个三角形之外，我们还需要优化它的成本函数，通常是分配给顶点的权重总和（或类似的聚合）。 所以问题不仅仅是“三角形是否存在”，而是“找到成本最小的三角形”。 

输入大小通常允许大约$n \le 10^5$或类似的，最多$m$边缘。 这立即排除了检查所有三元组的顶点，因为那将是$O(n^3)$，这远远超出了2秒可以处理的范围。 除非仔细构建，否则即使盲目地枚举所有对并检查邻接性也会太慢。 

一个天真的方法是尝试每一个三元组$(i, j, k)$，检查所有三个边是否都存在，并计算成本。 这会失败，因为三元组的数量随着$\binom{n}{3}$，即使对于$n = 5000$，因为它已经产生了$10^{10}$检查。 

第二个简单的改进是修复边缘$(u, v)$并尝试找到第三个顶点$w$连接到两者。 虽然这减少了搜索空间，但通过重复扫描邻接列表来低效地执行仍然会导致$O(nm)$密集情况下的行为。 

关键的观察是，可以通过迭代边缘并检查邻接集的交集来有效地检测三角形。 这将问题简化为在边的两个端点之间寻找公共邻居，这可以使用散列或排序邻接列表来完成。 

## 方法

 强力解决方案枚举所有顶点三元组。 对于每个三元组，它检查所有三个边是否存在并计算成本（如果有效）。 这是正确的，因为它直接验证三角形的定义，但其复杂度是顶点​​数量的三次方，这对于大输入来说是不可能的。 

改进来自于将视角从“选取三个顶点”更改为“选取一条边并将其补全为三角形”。 如果我们修复一个边缘$(u, v)$，任何包含它的三角形都必须有第三个顶点$w$使得两者$(u, w)$和$(v, w)$存在。 因此，我们不是枚举所有三元组，而是将$u$和$v$。 这将搜索空间从所有三元组减少到边乘以邻接交集。 

如果邻接集存储为哈希集，则可以有效地检查公共邻居。 然后，我们计算每个发现的三角形的成本并跟踪最小值。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力（所有三元组）|$O(n^3)$|$O(1)$| 太慢了|
 | 边+邻接交|$O(m \cdot d)$平均的，$O(nm)$最糟糕的|$O(n + m)$| 已接受 |

 ## 算法演练

 1. 为图构建一个邻接结构，通常为每个顶点建立一个邻接结构，因此成员资格检查的时间恒定。 这是必要的，因为三角形检测依赖于快速边缘存在查询。 
2. 迭代每条边$(u, v)$。 每个三角形必须至少包含一条边，因此这可确保每个三角形至少被考虑一次。 
3. 对于当前边$(u, v)$，迭代所有邻居$w$的$u$。 对于每一个这样的$w$，检查是否$w$也是的邻居$v$。 这个条件保证了$(u, v, w)$形成一个三角形。 
4. 每当找到有效三角形时，就使用顶点值计算其成本并更新全局最小值。 
5.处理完所有边后，如果没有找到三角形，则返回$-1$，否则返回记录的最小成本。 

### 为什么它有效

 无向图中的每个三角形都恰好有三个边，因此当将这些边中的任何一个作为“基边”处理时就会发现它。 相交检查确保第三个顶点连接到两个端点，这正是三角形的定义。 由于每个三角形至少被计数一次并且我们评估所有有效三角形，因此保证所有有效三角形的最小值是正确的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    a = list(map(int, input().split()))

    adj = [set() for _ in range(n)]
    edges = []

    for _ in range(m):
        u, v = map(int, input().split())
        u -= 1
        v -= 1
        adj[u].add(v)
        adj[v].add(u)
        edges.append((u, v))

    INF = 10**18
    ans = INF

    for u, v in edges:
        if len(adj[u]) > len(adj[v]):
            u, v = v, u

        for w in adj[u]:
            if w in adj[v]:
                cost = a[u] + a[v] + a[w]
                if cost < ans:
                    ans = cost

    print(-1 if ans == INF else ans)

if __name__ == "__main__":
    solve()
```邻接列表被存储为集合，以便第三个顶点的成员资格检查是恒定时间的。 从小到大的交换$u$和$v$在每次边迭代中，通过始终迭代较小的邻接列表来减少邻居检查的数量，从而提高密集图中的性能。 

最终最小值被初始化为无穷大，并且仅在找到有效三角形时更新，这可以干净地处理不存在解决方案的情况。

 | 测量 | 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 |$O(\sum \min(deg(u), deg(v)))$| 通过迭代较小的邻接列表并执行 O(1) 集检查来处理每条边 |

 对于典型的 Codeforces 约束，复杂性是有效的，其中$m$取决于$2 \cdot 10^5$。 每条边的检查方式都可以避免重复扫描大的邻域。

 ## 边缘情况

 关键的边缘情况是当图很稠密但不包含满足问题定义下有效三元组的连通性要求的三角形结构时。 例如，长度为 4 的简单循环会创建许多边，但不会创建三角形。 算法正确地处理了每条边，但邻居集的交集始终为空，因此不会发生更新，最终结果为$-1$。 

另一种边缘情况发生在存在多个三角形的完整图中。 该算法确保通过不同的基边多次发现每个三角形，但仍然可以正确跟踪最小成本，因为每个有效三角形的评估都是一致的。
