 ## 解决方案
 ## 问题理解

 该问题描述了一个以非常结构化的方式增长的利润过程。 第一天，哈什什么也没赚到，第二天他的每日收益就比前一天增加一个单位。 所以每日利润的顺序是累加的：第1天给0，第2天给1，第3天给3，第4天给6，依此类推。 

我们真正被问到的是：给定一个天数$d$，计算在这种逐日递增规则下截至当日累计的利润总额。 

从约束中观察到的关键是$d$可以大到$10^9$，同时有多达$10^4$测试用例。 任何模拟日复一日积累的解决方案都是不可能的，因为即使是单个测试用例也最多$d$需要迭代十亿步。 

这立即迫使我们采用封闭式公式或直接算术解释。 

微妙的边缘情况是最小的输入。 什么时候$d = 1$，利润必须为 0。当$d = 2$，利润为 1。如果错误地将第一项视为 1 而不是 0，从而将整个序列向上移动并从错误的基数开始生成三角形数字，则经常会发生天真的差一错误。 

另一个潜在的问题是每日增量和累计利润之间的混淆。 该语句定义了一个过程，而不是一个直接公式，因此仅实现部分逻辑（例如“前 d 个整数的总和”）而不调整索引将给出小值的错误结果。 

## 方法

 如果我们从字面上模拟该过程，我们会保持运行利润和每天增加 1 的每日增量。 在第 1 天，我们添加 0，在第 2 天，我们添加 1，在第 3 天，我们添加 2，依此类推。 后$d$天，我们会计算总和：$$0 + 1 + 2 + \dots + (d-1)$$这是一个标准的算术级数。 暴力法将从 1 循环到$d$并累计每次增量利润。 这在概念上是可行的，但它的运行时间是$O(d)$每个测试用例。 和$d$最多$10^9$，即使对于单个查询，这也变得完全不可行。 

序列的结构使问题变得简单。 我们不进行模拟，而是认识到总利润是第一个利润的总和$d-1$整数。 该总和有一个封闭形式：$$\frac{(d-1)d}{2}$$这消除了对迭代的所有依赖，并将每个查询减少为常数时间算术。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 |$O(d)$每次测试 |$O(1)$| 太慢了 |
 | 最佳|$O(1)$每次测试 |$O(1)$| 已接受 |

 ## 算法演练

 1.读取测试用例数量$t$。 每个测试用例都是独立的，需要计算给定日期的直接公式。 
2. 对于每个输入日$d$，将过程解释为从 0 开始的每日增量的累积和。这将问题转换为对序列求和$0, 1, 2, \dots, d-1$。 
3. 使用算术级数公式计算总和$\frac{(d-1)d}{2}$。 这一步完全避免了迭代，直接评估最终结果。 
4. 立即输出每个测试用例的计算值。 

### 为什么它有效

 该过程将每日增益定义为从 0 开始、每天增加 1 的严格递增序列。 这意味着当天$i$，添加的增量为$i-1$。 后的总利润$d$天数正是所有这些增量的总和。 由于这形成了线性算术级数，因此其总和由其第一项、最后一项和项数唯一确定。 不存在替代的累积顺序，因此封闭形式必须与任何有效的模拟相匹配。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        d = int(input())
        print((d - 1) * d // 2)

if __name__ == "__main__":
    solve()
```实现的核心是直接算术公式。 减 1 至关重要，因为第一天的利润为零，这意味着贡献序列结束于$d-1$， 不是$d$。 使用整数除法可确保正确性，而不会出现浮点精度问题。 

测试用例的循环非常简单，并且每个查询都在恒定时间内处理。 

## 工作示例

 ### 示例 1

 输入：```
d = 5
```我们计算每日增量：0、1、2、3、4。 

| 日 | 增量 | 运行总计 |
 | ---| ---| ---|
 | 1 | 0 | 0 |
 | 2 | 1 | 1 |
 | 3 | 2 | 3 |
 | 4 | 3 | 6 |
 | 5 | 4 | 10 | 10

 最终输出为10，匹配$\frac{4 \cdot 5}{2} = 10$。 

这证实了该公式无需模拟即可正确捕获累积增长。 

### 示例 2

 输入：```

```我们计算增量：0、1、2、3、4、5。 

| 日 | 增量 | 运行总计 |
 | ---| ---| ---|
 | 1 | 0 | 0 |
 | 2 | 1 | 1 |
 | 3 | 2 | 3 |
 | 4 | 3 | 6 |
 | 5 | 4 | 10 | 10
 | 6 | 5 | 15 | 15

 最终输出为15，匹配$\frac{5 \cdot 6}{2} = 15$。 

这证明了稍大案例的正确性，并清楚地显示了线性增长模式。

 ## 边缘情况

 对于$d = 1$，除了单个 0 贡献之外，增量序列为空。 公式给出$(1-1)\cdot1/2 = 0$，与预期结果完全匹配。 

为了$d = 2$，增量为 0 和 1，总计为 1。该公式得出$(2-1)\cdot2/2 = 1$，确认最小的非平凡情况的正确性。 

对于非常大的$d$， 例如$10^9$，由于任意精度整数，乘法在 Python 中是安全的。 在固定宽度语言中，这需要 64 位存储，但中间步骤不会超过$10^{18}$，因此溢出安全类型就足够了。
