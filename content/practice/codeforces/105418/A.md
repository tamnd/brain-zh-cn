 ## 解决方案
 ## 问题理解

 我们有一个由道路连接的城市的有向网络，其中每条道路都有一个非负成本，代表菲尼克斯沿着它行驶必须对抗多少怪物。 菲尼克斯从城市 1 出发，必须使用任意顺序的有向道路到达城市 n。 这是标准的最短路径设置，其中路径成本是边权重的总和。 

不同的是，凤凰有一个可以减少怪物数量的特殊操作。 在他的整个旅程中，他可以选择以下选项之一：要么选择一条道路并将其权重w替换为floor(w / 2)，要么沿着他选择的路径选择两条连续的道路并将其权重替换为floor(w / 2)。 应用一次此操作后，他仍然沿着正常路径行进并累积修改后的边权重。 

目标是在最优选择路径和使用单次减半能力后，最小化从城市 1 到城市 n 的总成本。 

这些约束立即表明对所有路径进行简单枚举是不可能的。 对于多达 10^5 个城市和 2×10^5 条道路，任何依赖于探索所有路径或尝试所有边缘子集的方法都会发生组合爆炸。 即使单个 Dijkstra 运行也是可行的，但复杂的是额外的“一次性修改”，它引入了对已修改的边数以及它们是否连续的状态依赖性。 

当能力的最佳使用不在原始图中的最短路径上时，就会出现微妙的边缘情况。 例如，在将两个大的相邻边减半后，稍长的路径可能会变得更便宜，因此没有状态跟踪的贪婪最短路径会失败。 

另一种故障模式是假设每个边缘的减半是独立的。 如果我们贪婪地将最大边缘减半，我们就会错过将两个中等边缘依次减半所产生的总减少量比将单个大边缘减半更大的情况。 

## 方法

 蛮力的想法是考虑从 1 到 n 的每条可能路径，并为每条路径模拟三种变体：无操作、将一条边减半或将两个连续边减半。 对于长度为 k 的固定路径，评估所有此类选择的复杂度为 O(k)。 但有向图中的路径数量可能是 n 的指数，因此这种方法立即变得不可行。 即使限制最短路径也无济于事，因为最佳路径结构取决于应用减半的位置。 

关键的观察结果是操作的结构引入了少量的、固定数量的附加状态。 当按顺序遍历边时，任何时候我们只关心我们是否还没有使用该能力，当前是否处于“双边减半”的中间，或者该能力是否已经使用并完成。 这将问题转化为每个节点少量状态的分层最短路径问题。 

我们在扩展图上有效地运行最短路径算法，其中每个节点都分为表示我们使用特殊操作的程度的状态。 从每个状态，转换模拟正常获取边缘，在当前边缘开始减半，或者在下一个边缘完成减半（如果适用）。 这是一种标准技术，其中涉及局部邻接的路径约束被编码到图状态中。 

结果是在多达 3n 个状态上进行类似于 Dijkstra 的过程，因为每个城市最多可以处于三种有意义的配置中。 这使得复杂性易于管理，同时保持精确的正确性。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 对所有路径进行暴力破解 | 指数| 指数| 太慢了 |

 ## 算法演练

我们用三个状态对每个城市进行建模：状态 0 表示尚未使用任何操作，状态 1 表示我们已将该操作用作“选择的第一条边”，并正在等待选择性地将其与下一条边配对，状态 2 表示该操作已完全消耗。 

然后我们在这些状态上运行最短路径算法。 

1. 初始化距离表 dist[node][state] 为无穷大，并设置 dist[1][0] = 0。这反映了我们从城市 1 开始，没有使用任何修改。 
2. 使用按当前距离排序的优先级队列。 每个条目代表（当前成本、节点、状态）。 我们总是首先扩展最小的已知成本，确保我们按照最终成本的升序处理状态。 
3. 从状态 (u, 0)，我们可以通过三种不同的概念方式遍历出边 (u → v, w)。 我们可以正常地采取它，并在 v 处保持状态 0，成本为 +w。 我们可以在这条边上开始减半操作，支付下限（w/2），并在 v 处移动到状态 1，这意味着我们现在正在向前进行部分应用的操作。 
4. 从状态 (u, 1) 开始，我们已经选择了双边操作的第一个边。 当我们遍历一条边 (u → v, w) 时，我们也通过对第二条边应用减半来完成操作。 该转换在 v 处转移到状态 2，并增加了成本下限 (w/2)。 这强制了连续性，因为状态 1 强制下一条边成为同一路径段的一部分。 
5. 从状态 (u, 0)，我们还允许通过采用成本 w 来简单地移动到 v 处的状态 2，而根本不使用该操作。 这代表我们在这条道路上从未使用过特殊能力的情况。 
6. 从状态 (u, 2) 开始，所有转换都是标准的最短路径转换：我们只需添加 w 并保持在状态 2，因为该操作已被完全使用。 
7. 继续Dijkstra，直到处理完所有可达状态。 答案是 dist[n][0]、dist[n][1]、dist[n][2] 中的最小值。 

关键的结构限制是状态 1 不能无限期地持续。 它强制精确地执行一个后续边来解析操作，确保我们只应用一次“两个连续边”修改。 

### 为什么它有效

 在有效解决方案中的任何一点，唯一相关的历史记录是我们是否使用过该操作，以及我们是否正在应用它的双边版本。 关于较早边缘的任何更长的记忆都是无关紧要的，因为该操作最多只与两个连续的边缘交互。 这使得过程在扩展的状态空间上成为马尔可夫式。 

Dijkstra 算法保证了正确性，因为所有转换都保留非负边缘成本，并且每条有效的修改路径恰好对应于该扩展图中的一条路径。 相反，扩展图中的每条路径都对应于原始问题中操作的有效使用，因此不会引入无效的配置。 

## Python 解决方案```python
import sys
import heapq

input = sys.stdin.readline
INF = 10**30

def solve():
    n, m = map(int, input().split())
    g = [[] for _ in range(n + 1)]
    for _ in range(m):
        u, v, w = map(int, input().split())
        g[u].append((v, w))

    dist = [[INF] * 3 for _ in range(n + 1)]
    dist[1][0] = 0
    pq = [(0, 1, 0)]

    while pq:
        d, u, st = heapq.heappop(pq)
        if d != dist[u][st]:
            continue

        if st == 0:
            for v, w in g[u]:
                nd = d + w
                if nd < dist[v][0]:
                    dist[v][0] = nd
                    heapq.heappush(pq, (nd, v, 0))

                nd2 = d + (w // 2)
                if nd2 < dist[v][1]:
                    dist[v][1] = nd2
                    heapq.heappush(pq, (nd2, v, 1))

                nd3 = d + w
                if nd3 < dist[v][2]:
                    dist[v][2] = nd3
                    heapq.heappush(pq, (nd3, v, 2))

        elif st == 1:
            for v, w in g[u]:
                nd = d + (w // 2)
                if nd < dist[v][2]:
                    dist[v][2] = nd
                    heapq.heappush(pq, (nd, v, 2))

        else:
            for v, w in g[u]:
                nd = d + w
                if nd < dist[v][2]:
                    dist[v][2] = nd
                    heapq.heappush(pq, (nd, v, 2))

    print(min(dist[n]))

if __name__ == "__main__":
    solve()
```该实现维护三个状态的完整距离表并运行标准 Dijkstra 循环。 关键细节是状态1只允许完成双边操作的转换，防止无效重用或非连续应用。 

一个微妙的实现问题是从状态 0 开始的转换的重复。我们明确允许三个选项：正常遍历、开始减半或过早地将操作标记为已使用。 最后一个转换是必要的，因为该操作可能永远不会被使用或稍后可能会被使用，并且状态 2 充当“完成”状态，其中不保留任何特殊处理。

### 示例 1 草图

 最优路径依次使用三个边。 该算法探索正常路径和变体，其中操作应用于路线中间的两个连续边缘。 状态转换迫使减半精确地应用于相邻边缘，并且 Dijkstra 确保选择更便宜的组合配置。 

跟踪证实该算法并未预先假设固定路径； 它在放松过程中动态评估操作的不同位置。

 ## 边缘情况

 单边图测试算法是否正确允许立即应用减半操作。 有输入`1 2 1`，从状态 0 的唯一转换包括以成本 0 移动到状态 1，并且算法正确地以节点 2 处的距离 0 结束。 

双边链，例如`1 → 2 → 3`有重量`100, 100`确保正确使用状态 1 转换。 该算法在第一个边缘之后进入状态 1，然后强制第二个边缘完成操作，产生成本`50 + 50 = 100`。 

竞争路径示例表明该算法不假设固定路线。 如果存在一条长但高度可简化的路径和一条短但昂贵的路径，则通过 Dijkstra 并行探索两条路径，并且状态表确保选择最佳组合，而不偏向原始最短路径。
