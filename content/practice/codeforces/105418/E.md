 ## 解决方案
 ## 问题理解

 我们得到一组硬币面额，它们遵循一种非常结构化的模式：每枚硬币都是一个完全由数字 1 组成的数字，并且长度以特殊方式增长。 因此我们会得到 11、111、1111、11111 等值，并无限期地继续下去。 每个测试用例给出一个目标量`n`，我们需要确定是否可以表示`n`作为这些硬币的总和，其中每种硬币类型都可以无限次使用。 

重新构建这个，我们正在检查是否`n`可以表示为 11、111、1111 等具有非负整数系数的数字的线性组合。 该结构提出了一个经典的无界硬币找零可行性问题，但关键的困难在于硬币的价值不是任意的，它们是高度相关的。 

约束条件`n ≤ 10^9`是关键的指导。 任何尝试对所有值进行动态规划的解决方案`n`时间和内存都太大了。 即使每个测试用例进行线性扫描也会太慢`t = 10^4`。 

尝试枚举这些硬币的所有组合的简单方法很快就会失败，因为组合方式的数量会增加。 即使将自己限制在贪婪的构造上也是不安全的，因为当货币系统不规范时，贪婪的选择可能会失败。 

一个微妙的边缘情况是，许多小值无法形成，即使它们看起来应该形成。 例如，不能组成12，因为最小的硬币是11，其次是111，与11少量组合时已经超出了。 另一个例子是 23，它可能会引发像 11 + 11 + 1 这样的分解，但 1 不能作为硬币使用，所以这样的推理就破裂了。 

## 方法

 一个蛮力的想法是将其视为标准的无界硬币找零可行性问题。 我们首先生成所有硬币价值高达`n`，这是可行的，因为序列增长很快。 然后我们运行经典的动态规划方法，其中`dp[x]`表示是否求和`x`可以形成。 对于每个硬币，我们将所有可到达的状态放宽到`n`。 

这种方法是正确的，因为它系统地探索了所有组合。 然而，它的运行时间与`O(n * number_of_coins)`，在最坏的情况下大约是`10^9 * 30`，远远超出了可行的限度。 即使将其减少为位集优化仍然很困难`10^4`测试用例。 

关键的观察结果是，所有硬币都仅由数字 1 组成，这赋予了它们强大的加法结构。 每枚硬币的长度`k`可以看作`111...1 = (10^k - 1) / 9`。 这种结构意味着较大的硬币几乎是具有移位位置值的较小硬币的线性组合。 这表明并非所有硬币都提供独立的约束。 

一个更有用的观点是仅使用最小的几个硬币来考虑模块化结构和可构建性。 一旦我们包括 11 和 111，就可以通过这两个数字移位的组合来表示更高的硬币，从而使更大的硬币对于给定约束的可行性检查变得多余。 

因此，问题简化为检查是否`n`可以仅使用两种硬币类型来表示：11 和 111。从这里开始，问题就变成了非负整数的简单线性丢番图可行性检查。 

我们尝试计算较大硬币 (111) 的所有可能数，并检查余数是否能被 11 整除。`n ≤ 10^9`，在最坏的情况下迭代次数最多约为 10^7，但我们可以通过将循环限制为进一步减少它`n / 111`。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力破解所有硬币| O(nk) | O(nk) | O(n) | 太慢了 |
 | 使用 11 和 111 减少搜索 | 每次测试 O(n/111) | O(1) | O(1) | 已接受 |

 ## 算法演练

 1. 预先计算或固定两个基本硬币值 11 和 111。不需要更大的硬币，因为它们可以被这两个硬币的组合替换，而不会失去达到约束的值的可达性。 
2. 对于每个测试用例，取值`n`并迭代我们可能使用多少 111 个硬币。 范围从 0 到`n // 111`。 每个选择都代表使用最大的有意义面额来确定总和的一部分。 
3. 对于固定数量的 111 个硬币，计算剩余价值`r = n - 111 * x`。 此步骤隔离是否可以仅使用 11 个硬币组成其余部分。 
4. 检查是否`r`能被 11 整除。如果是，那么我们可以用 11 个硬币精确地填充余数，这意味着该表示是有效的。 
5. 如果有任何选择`x`导致有效余数，我们立即得出结论`n`是可以实现的并且输出YES。 如果这样的分割不起作用，则输出 NO。 

### 为什么它有效

 该算法依赖于这样一个事实：任何有效的表示都可以重新排列，以便所有 111 个硬币首先分组，剩余的值必须完全位于 11 生成的半群中。由于 11 是最小的面额，任何不能被 11 整除的剩余部分都不能使用更大的硬币来修复，而不会将总数增加到超过`n`。 这使得超过 111 个硬币的搜索完整且充分。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        n = int(input())
        ok = False

        max_x = n // 111
        for x in range(max_x + 1):
            rem = n - 111 * x
            if rem % 11 == 0:
                ok = True
                break

        print("YES" if ok else "NO")

if __name__ == "__main__":
    solve()
```该解决方案循环遍历 111 个硬币的可能计数，并检查余数能否被 11 整除。提前退出可确保我们在找到有效分解后不会探索不必要的组合。 

一个微妙的实现细节是确保在限制循环时使用整数除法。 使用浮点或不正确的边界会错过有效案例或引入性能问题。 

## 工作示例

 考虑`n = 144`。 

| x (111 秒) | 111 * x | 余数 | 余数 % 11 | 决定|
 | ---| ---| ---| ---| ---|
 | 0 | 0 | 144 | 144 1 | 没有|
 | 1 | 111 | 111 33 | 33 0 | 是的 |

 这表明使用一枚 111 的硬币剩下 33，正好是三个 11 的硬币，所以答案是 YES。 

现在考虑`n = 69`。 

| x (111 秒) | 111 * x | 余数 | 余数 % 11 | 决定|
 | ---| ---| ---| ---| ---|
 | 0 | 0 | 69 | 69 3 | 没有|
 | 1 | 111 | 111 负| 无效| 停止|

 分解不起作用，所以答案是否定的。 

这些痕迹证实该算法正确地探索了较大面额的所有可行分配，并验证了剩余部分是否位于较小的硬币系统中。

import sys, io

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    import sys
    input = sys.stdin.readline

    t = int(input())
    out = []
    for _ in range(t):
        n = int(input())
        ok = False
        for x in range(n // 111 + 1):
            if (n - 111 * x) % 11 == 0:
                ok = True
                break
        out.append("YES" if ok else "NO")
    return "\n".join(out)

 ## 边缘情况

 对于`n = 11`，循环立即发现`x = 0`，余数是11，可以整除，所以答案是YES。 这直接检查基本面额是否有效。 

为了`n = 12`，唯一的可能性是`12`或者`12 - 111`这是无效的。 由于 12 不能被 11 整除，因此不存在分解，算法正确输出“否”。 

对于非常大的值，例如`n = 10^9`，循环仍然很快终止，因为迭代次数的上限为`n / 111`，一旦找到有效余数，我们就提前退出。
