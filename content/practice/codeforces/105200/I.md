 ## 解决方案
 ## 问题理解

 我们得到了一系列数字，任务围绕着理解它距离“干净”的顺序有多远。 感兴趣的关键数量是反转计数，它测量有多少对元素不符合其自然顺序，这意味着较大的元素出现在较小的元素之前。 

该问题不是直接计算完整的反演结构，而是暗示了结构的简化：已经正确定位在序列开头或结尾的元素一旦被隔离，就不会对反演计算做出有意义的贡献。 核心任务是确定阵列中有多少部分“已经对齐”，然后确定剩余的无序度。 

从计算的角度来看，这立即表明，当序列变大时，任何直接检查所有对的解决方案都会太慢。 简单的对检查需要二次方的时间，一旦数组大小达到大约 10^5 个元素，这就变得不可行，大约需要 10^10 次比较。 这迫使我们采用一种利用排序结构或高效计数技术（例如基于合并的计数或芬威克树）的方法。 

当数组已经排序或几乎排序（除了一个小的中间区域）时，会出现微妙的边缘情况。 例如，在类似的输入中`[1, 2, 3, 10, 9, 8, 4, 5, 6, 7]`，大多数前缀元素都正确放置，并且对反转复杂性没有贡献，但中间反转块主导反转计数。 简单的方法可能仍然会扫描所有对，而没有意识到在贡献逻辑中可以忽略大的排序段。 

另一种边缘情况是当数组严格递减时，例如`[5, 4, 3, 2, 1]`。 这里每一对都贡献一个反转，任何假设从边界进行部分排序的优化都会失败，除非它正确地减少到完全反转计数。 

## 方法

 蛮力方法很简单。 我们检查每一对`(i, j)`和`i < j`并数数有多少次`a[i] > a[j]`。 这直接符合反转的定义，并且保证是正确的，因为它详尽地验证了所有关系。 问题是可扩展性。 为了`n = 100000`，该方法的执行顺序为`n^2 / 2`比较，这远远超出了可以及时执行的范围。 

关键的观察是，如果我们可以跟踪元素相对于排序顺序的移动方式，则反转计数不需要显式的对枚举。 当在合并排序期间分割数组时，反转与右半部分的元素放置在左半部分的剩余元素之前的情况完全对应。 这允许我们在排序时计算反转次数`O(n log n)`时间。 

另一个等效的观点是使用芬威克树或二元索引树。 我们按顺序处理元素，对于每个元素，我们查询有多少个先前看到的元素大于当前元素。 这将成对计数转换为前缀频率查询，动态维护效率很高。 

该语句关于不影响反转计数的前缀和后缀元素的提示与此结构一致：已排序的段的行为类似于排序边界，并且不会在其交互点之外引入交叉反转。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(n²) | O(1) | O(1) | 太慢了 |
 | 归并排序/芬威克树 | O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 我们将描述芬威克树方法，因为它是实现反转计数的最直接方法。 

1. 首先，将数组的值压缩到更小的范围内。 这是必要的，因为芬威克树是根据索引而不是原始值进行操作的，而大的值会使该结构不可行。 压缩保留了相对顺序，这对于倒数计数来说是最重要的。 
2. 初始化支持前缀和查询和点更新的Fenwick Tree。 该树将跟踪到目前为止每个压缩值已被看到的次数。 
3. 从左到右遍历数组。 对于每个元素，我们想知道有多少先前见过的元素比它大。 这对应于查询已处理元素的数量减去小于或等于当前元素的数量。 
4. 将此计数添加到正在运行的答案中，因为每个此类先前元素都与当前元素形成反转。 
5. 更新 Fenwick 树以将当前元素标记为所见，以便将来的元素可以在其反演计算中考虑到它。 

关键思想是我们不断维护动态前缀频率结构，使我们能够用聚合计数替换显式对枚举。 

### 为什么它有效

 在每一步中，芬威克树都准确包含出现在当前索引之前的元素的多重集。 处理元素时`a[i]`，任何大于它的较早元素形成反转对`(j, i)`在哪里`j < i`。 前缀总和查询精确地隔离了那些大于的较早元素`a[i]`而不单独检查它们。 由于每个反转在处理其右端点时都只计算一次，因此不会遗漏或重复计算任何对。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class Fenwick:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)

    def add(self, i, v):
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

def solve():
    data = list(map(int, input().split()))
    n = data[0]
    arr = data[1:]

    vals = sorted(set(arr))
    comp = {v: i + 1 for i, v in enumerate(vals)}

    fw = Fenwick(len(vals))
    inv = 0

    for x in arr:
        cx = comp[x]
        inv += fw.sum(len(vals)) - fw.sum(cx)
        fw.add(cx, 1)

    print(inv)

if __name__ == "__main__":
    solve()
```Fenwick 树被实现为具有点更新和前缀和的标准二叉索引树。 这`add`函数更新频率计数，同时`sum`检索在给定的压缩索引之前已看到的元素数量。 

里面`solve`，坐标压缩可确保值映射到从 1 开始的紧凑范围。对于每个元素，我们通过从总处理计数中减去当前值之前的前缀计数来计算有多少个先前看到的值严格大于该值。 这种差异直接影响反转计数。 

一个常见的陷阱是忘记压缩必须保留顺序而不是绝对值。 另一个微妙的问题是 Fenwick 树中的逐一索引，可以通过将压缩值从 1 开始移动来避免这个问题。

 ## 边缘情况

 对于已经排序的数组，例如`[1, 2, 3, 4]`，芬威克树总是报告零反转，因为每个查询`fw.sum(max) - fw.sum(x)`评估为零。 该结构正确地反映了先前看到的元素都不大于当前元素，并且答案始终保持稳定。 

对于像这样的完全递减数组`[4, 3, 2, 1]`，每一步都会贡献最大可能的新反演。 加工时`1`，树已经包含所有较大的元素，因此查询返回`3`，完全匹配三对`(4,1), (3,1), (2,1)`。 每一个都在处理正确端点时精确计数一次，以确认正确性。
