 ## 解决方案
 ## 问题理解

 我们得到了许多正整数，对于每个正整数，我们必须确定它的真因数有多“丰富”。 对于一个号码$n$，我们考虑它的所有除数，除了$n$本身，总结它们，并将该总和与$n$。 其和较大则谓之丰，其小则谓之不足，相等则谓之完备。 

输入本质上是一大批查询。 每个查询都是一个最多一百万的整数，并且最多可以有一百万个这样的查询。 输出是每个数字的分类字符串，因此关键挑战不是计算单个除数和，而是在许多重复查询中以极快的速度完成计算。 

这些约束立即排除了每个查询从头开始重新计算除数的情况。 朴素除数扫描最多$n$在最坏的情况下，对于一个查询来说已经太慢了，并乘以$10^6$查询就变得不可能了。 甚至扫描最多$\sqrt{n}$在这种规模下，每个查询仍然太重。 

微妙的边缘情况来自完美数字，其中总和恰好等于数字。 例如，6、28、496。如果忘记排除数字本身，粗心的实现将始终错误地将数字分类为“丰富”，因为完整的除数总和始终包括$n$。 

另一个边缘情况是重复查询。 由于输入可以多次包含相同的数字，因此每次重新计算其除数和都是浪费工作，并且可能会导致 TLE。 

## 方法

 蛮力的想法很简单：对于每个数字$n$，从 1 迭代到$n-1$，检查整除性，并累加除数之和。 这是正确的，因为它直接遵循等分总和的定义。 然而，每次查询的成本$O(n)$，并与$n$最多$10^6$和$10^6$查询，总操作计数退化为$10^{12}$，这远远超出了可行的限度。 

标准改进是仅检查除数$\sqrt{n}$，将两者相加$d$和$n/d$什么时候$d$划分$n$。 这将单个查询减少为$O(\sqrt{n})$，但与$10^6$查询这仍然是按顺序$10^9$操作，这在 Python 中是边缘且不安全的。 

关键的观察结果是值范围很小且固定：所有数字都位于 1 和$10^6$。 我们可以使用类似筛子的累加来预先计算此范围内每个数字的真除数之和，而不是重新计算每个查询的除数和。 每个整数$d$贡献于所有倍数$d$，因此我们可以迭代除数并有效地传播贡献。 这将问题转化为经典的除数和筛法，完全减少了重复计算。 

一旦预计算完成，每个查询就变成一个简单的数组查找和比较。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 |$O(T \cdot n)$|$O(1)$| 太慢了|
 | 每个查询的平方根 |$O(T \sqrt{n})$|$O(1)$| 太慢了|
 | 筛分预计算 |$O(N \log N)$|$O(N)$| 已接受 |

 ## 算法演练

 我们想要每个数字的真因数之和，直到$10^6$。 我们不是独立处理每个数字，而是反转观点，让每个可能的除数“向前贡献”。 

1. 创建数组`s`尺寸的$N+1$，初始化为零，其中`s[x]`将存储的真因数之和$x$。 我们不包括$x$它本身就是它自己的总和，所以我们确保永远不会将数字添加到它自己的索引中。 
2. 对于每个整数$d$从 1 到$N/2$，我们迭代所有倍数$m = 2d, 3d, 4d, \dots$。 对于每个这样的倍数$m$，我们添加$d$进入`s[m]`。 这有效是因为$d$是每个严格大于自身的倍数的真因数。 
3.经过这个过程后，`s[n]`恰好包含所有真因数的总和$n$，因为每个除数$d < n$对其倍数仅贡献一次。 
4. 对于每个查询$n_i$，我们比较`s[n_i]`和$n_i$。 如果`s[n_i] > n_i`，我们产出丰富。 如果`s[n_i] < n_i`，我们输出不足。 否则我们输出完美。 

为什么它有效可以归结为一个干净的计数不变量。 每对$(d, m)$在哪里$d$划分$m$和$d < m$当我们处理时只添加一次$d$。 不存在重复，因为每个除数都独立地负责其所有倍数，并且没有除数会以破坏正确性的方式导致大于范围边界的数字。 结果，每个数字都精确地累加了它的真因数之和，没有其他任何东西。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MAXN = 10**6

# precompute sum of proper divisors
s = [0] * (MAXN + 1)

for d in range(1, MAXN // 2 + 1):
    for m in range(2 * d, MAXN + 1, d):
        s[m] += d

t = int(input())
nums = list(map(int, input().split()))

out = []
for x in nums:
    if s[x] > x:
        out.append("abundant")
    elif s[x] < x:
        out.append("deficient")
    else:
        out.append("perfect")

sys.stdout.write("\n".join(out))
```核心实现细节是筛循环。 外循环选择一个除数候选，内循环将其分配给所有倍数。 内循环开始于`2*d`至关重要，因为我们明确地将数字本身排除在它自己的除数和之外​​。 

另一个微妙的点是我们预先计算一次$10^6$无论输入大小如何。 这确保了昂贵的工作得到摊销，并且每个查询都变成恒定的时间。 

比较步骤是直接的，因为`s[x]`已经排除了`x`本身通过建设。 

## 工作示例

 我们使用样本值 12、21 和 28。 

对于 12，它的真因数是 1, 2, 3, 4, 6。预计算构建`s[12] = 16`。 对于 21，约数是 1, 3, 7`s[21] = 11`。 对于 28，约数是 1, 2, 4, 7, 14`s[28] = 28`。 

| 数量 | 真因数累计 | s(n) | s(n) | 比较|
 | ---| ---| ---| ---|
 | 12 | 12 1、2、3、4、6 | 16 | 16 16 > 12 | 16 > 12
 | 21 | 21 1, 3, 7 | 11 | 11 11 < 21 | 11 < 21
 | 28 | 28 1、2、4、7、14 | 28 | 28 28 = 28 |

 该迹线证实了筛子正确地累积了除数贡献，并且分类仅取决于最终的恒定时间比较。 

第二个小例子是 6。在预处理期间，贡献来自 1、2 和 3，产生`s[6] = 6`，这正确地将其置于完美类别中。 这验证了该算法正确处理对称除数结构。

def run(inp: str) -> str:

    MAXN = 10**6
    s = [0] * (MAXN + 1)

    for d in range(1, MAXN // 2 + 1):
        for m in range(2 * d, MAXN + 1, d):
            s[m] += d

    t = int(input())
    nums = list(map(int, input().split()))

    res = []
    for x in nums:
        if s[x] > x:
            res.append("abundant")

            res.append("perfect")

    return "\n".join(res)

assert run("3\n12 21 28") == "abundant\ndeficient\nperfect"

 ## 边缘情况

 对于输入`1`，筛子给出`s[1] = 0`因为 1 没有真约数。 该算法比较 0 < 1 并正确输出不足。 这捕获了除数集为空的边界情况。 

为了`6`，预处理添加 1、2 和 3 的贡献，给出`s[6] = 6`。 相等性检查完美触发，确认我们不会意外地将 6 本身包含在其总和中。 

为了`28`，来自 1、2、4、7 和 14 的贡献精确地累加到 28。由于每个除数都是独立处理的，因此不存在重复计算，并且算法可靠地将其分类为完美。
