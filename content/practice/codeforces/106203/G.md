 ## 解决方案
 ## 问题理解

 我们得到一个源字符串`s`和两个目标字符串`t1`和`t2`其总长度等于`|s|`。 任务是分区`s`通过在选定的位置将其切割成一系列连续的片段。 切割后，我们从左到右扫描这些碎片，并将每个碎片完全分配给两个累加器之一，`a`或者`b`。 碎片的顺序被保留，但每个碎片都是不可分割的，并且必须完全属于其中一个`a`或者`b`。 最后，我们想要`a`准确地成为`t1`和`b`准确地成为`t2`。 我们的目标是尽量减少我们的削减次数`s`。 

切割定义了碎片的边界。 和`k`削减，我们得到`k+1`件。 然后，每个部分选择是否有助于构建`t1`或者`t2`，保留每个目标字符串内的相对顺序。 

约束条件允许`|s|`最多`10^4`，所以任何解决方案`O(n^2)`已经到了边缘，任何立方体都是不可能的。 然而，真实的结构表明我们希望避免显式枚举分区，因为分段的数量呈指数增长。 

当角色重复时会出现一个微妙的问题。 如果两者中出现相同的字符`t1`和`t2`，单个字符的贪婪分配或朴素匹配可能会失败，因为决策取决于整个段的未来兼容性，而不是本地字符匹配。 

当最佳解决方案需要延迟分离时，就会出现第二种重要情况。 如果我们削减得太早，我们可能会强制分配无法一致分配的部分，即使稍微大一点的部分也可以。 

## 方法

 直接的做法就是从分割的角度来思考：尝试各种方法来分割`s`分成连续的块，并为每个分割尝试将块分配给`t1`或者`t2`同时维护秩序。 这相当于在其中选择剪切位置`n-1`间隙，然后检查块的双向交织是否可以重建两个目标字符串。 

即使检查固定分段也需要验证块序列是否可以贪婪地分配以匹配`t1`和`t2`，因此完整的暴力破解次数会呈指数级增长。 即使我们限制切割并尝试对分区进行动态编程，状态空间很快就会变得不可行，因为每次切割都会同时更改两个字符串的未来匹配约束。 

关键的观察结果是，问题不在于任意分区，而在于对可能进行确定性分配的最大延伸进行分组。 一旦我们确定了每个位置的分配`s`到任一`t1`或者`t2`，我们需要切割的唯一原因是当相邻位置`s`去不同的目标。 但是，我们不能自由地独立分配职位； 赋值必须遵守每个目标字符串中的字符顺序。 

这表明将问题重新定义为选择如何交错`t1`和`t2`形成`s`，同时最大限度地减少从字符中获取字符之间的切换次数`t1`和来自`t2`。 每个开关完全对应于原始字符串表示形式中的一个切口。 

因此，我们首先考虑的不是切割，而是构建`s`穿过两者`t1`和`t2`，在每一步选择下一个字符是否来自`t1`或者`t2`，只要它匹配下一个字符`s`。 该约束成为一个经典的两指针交错可行性问题，并增加了一个目标：最大限度地减少源之间的交替。 

这简化为状态上的最短路径式 DP`(i, j, last)`， 在哪里`i`是位置在`t1`,`j`在`t2`， 和`last`指示我们最后从中获取的字符串。 转换消耗下一个字符`s`如果它与所选源中的下一个字符匹配，则切换源会花费一次。

 ## 算法演练

 我们定义一个 DP，在其中模拟构建`s`从左到右，同时跟踪我们消耗了多少`t1`和`t2`。 关键是位置在`s`是隐式的，因为在步骤`k`我们必须匹配`s[k]`。 

1.我们创建一个DP表`dp[i][j][p]`， 在哪里`i`是我们使用了多少个字符`t1`,`j`从`t2`， 和`p`指示最后获取的字符是否来自`t1`或者`t2`。 这准确地编码了每个目标已消耗多少来形成前缀`s`。 
2. 我们从没有使用任何字符的开始状态初始化DP，并且我们对第一个字符尝试两种可能性`s`。 如果`s[0] == t1[0]`，我们可以从`t1`，类似地对于`t2`。 这很重要，因为第一段不会发生剪切。 
3. 对于每个状态，我们考虑通过采用下一个字符来扩展构造`s`从任一`t1`或者`t2`，前提是下一个字符匹配。 如果我们从与上一步相同的源继续，我们不会添加剪切； 否则我们将切割计数加一。 
4.我们迭代所有可能的`(i, j)`按总进度递增顺序排列的状态`i + j`，因为每次转换仅消耗一个字符`s`。 
5. 答案是所有状态中的最小值`i + j = |s|`和`(i, j)`比赛`(len(t1), len(t2))`，考虑最后两个可能的来源。 

关键的一点是，每次我们在从`t1`和`t2`，我们在原始字符串分段中有效地引入了剪切。 

### 为什么它有效

 DP 对每个有效的交织进行编码`t1`和`t2`形成`s`，并且每个这样的交错都唯一对应于一个分段`s`其中相同源选择的每个最大运行形成一个片段。 任何剪辑都会简化为源发生变化的过渡。 由于 DP 最大限度地减少了源之间的转换，因此它直接最大限度地减少了剪切次数。 该模型之外不能存在分段，因为任何有效的赋值都必须遵守每个目标字符串内的字符顺序，这正是由两指针消耗结构强制执行的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    s = input().strip()
    t1 = input().strip()
    t2 = input().strip()

    n = len(s)
    n1 = len(t1)
    n2 = len(t2)

    INF = 10**9

    # dp[i][j][p] = min cuts, p=0 from t1, p=1 from t2
    dp = [[[INF] * 2 for _ in range(n2 + 1)] for _ in range(n1 + 1)]

    # initialize
    if n1 > 0 and s[0] == t1[0]:
        dp[1][0][0] = 0
    if n2 > 0 and s[0] == t2[0]:
        dp[0][1][1] = 0

    for i in range(n1 + 1):
        for j in range(n2 + 1):
            for p in range(2):
                cur = dp[i][j][p]
                if cur == INF:
                    continue
                k = i + j
                if k == n:
                    continue
                if i < n1 and s[k] == t1[i]:
                    cost = cur + (0 if p == 0 else 1)
                    if cost < dp[i + 1][j][0]:
                        dp[i + 1][j][0] = cost
                if j < n2 and s[k] == t2[j]:
                    cost = cur + (0 if p == 1 else 1)
                    if cost < dp[i][j + 1][1]:
                        dp[i][j + 1][1] = cost

    ans = min(dp[n1][n2][0], dp[n1][n2][1])
    print(ans)

if __name__ == "__main__":
    solve()
```DP 通过每个目标字符串中已消耗的字符数进行索引。 隐含位置`s`总是`i + j`，保证所有三个字符串之间的同步。 状态转换要么从`t1`或者`t2`，并且切换源会增加成本，这恰好对应于引入新的切割边界。 

一个微妙的点是初始化：我们允许从`t1`或者`t2`不计算切割，因为切割是在块之间定义的，并且第一块总是空闲的。

 ## 边缘情况

 一个重要的边缘情况是当两者`t1`和`t2`从相同的角色开始。 在这种情况下，两个 DP 初始转换都是有效的，并且算法会无偏差地探索两种可能性。 例如，如果`s = "ab"`,`t1 = "a"`,`t2 = "b"`，两个起始选择都是有效的并产生单切解。 

当一个目标字符串完全交错在另一个目标字符串内部时，会出现另一种边缘情况`s`。 DP 自然地处理这个问题，因为切换成本仅在更改源时应用，而不是在消耗连续字符时应用。 

最后一种情况是存在多个有效对齐的重复字符。 DP 不依赖于字符匹配的唯一性，因此会探索所有有效路径，并通过跨状态松弛来保留最小切割配置。
