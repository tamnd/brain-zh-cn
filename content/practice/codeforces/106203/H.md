 ## 解决方案
 ## 问题理解

 我们得到一个正数数组。 我们必须选择三个不同的位置$i, j, k$并计算由这些值的比率形成的对称表达式。 具体来说，每个选定的三元组贡献一个值，该值取决于每个元素除以其他元素之一，并且我们希望所有三元组中的最小可能值。 

表达式的结构比索引本身更重要。 由于表达式在三个选定元素中完全对称，因此索引仅强制唯一性，而实际值仅取决于从数组中选择的三个数字。 

输入大小可达$2 \cdot 10^5$，这立即排除了三元组的任何三次或二次枚举。 一个天真的$O(n^3)$方法需要按顺序$10^{15}$行动，这远远超出了任何可行的时间限制。 甚至$O(n^2)$除非经过大量优化，否则将成为边界，但这里仍然没有必要，因为表达式的结构允许更小的候选集。 

一个微妙的问题是表达式涉及除法，因此浮点精度很重要。 所需精度为$10^{-9}$，只要我们避免不稳定的重排，就可以使用标准双精度轻松实现。 

朴素推理的一个关键失败案例是假设必须检查任意三元组。 

例如，如果数组是$[1, 1000, 1001]$，暴力可能会考虑多种组合，但最佳三元组必须来自结构更加结构化的子集。 正确答案是由一小组极值而不是任意中间值确定的。 

## 方法

 蛮力方法很简单：迭代所有三元组$(i, j, k)$，计算表达式并跟踪最小值。 这是有效的，因为它直接评估定义。 然而，三元组的数量是$\binom{n}{3}$，这大约变成$2 \cdot 10^{15}$什么时候$n = 2 \cdot 10^5$。 即使每次评估都是恒定的时间，这也是不可能的。 

关键的观察结果是，该表达式是高度结构化的：每个术语都会奖励所选值之间的不平衡，并且与极端候选值相比，中间值往往不是最佳的。 这是对称比率最小化问题中的典型模式，其中最优配置折叠为围绕排序顺序的一小部分候选。 

数组排序后，最佳三元组必须来自排序顺序接近的值。 直观上，在本地簇之外插入一个非常大或非常小的元素会不成比例地增加一项，从而使其无法达到最佳效果。 

这将问题简化为仅检查排序数组中的连续三元组，排序后这是线性的。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 |$O(n^3)$|$O(1)$| 太慢了 |
 | 排序三重扫描|$O(n \log n)$|$O(1)$| 已接受 |

 ## 算法演练

 1. 对数组进行非降序排序。 排序会组织候选者，使“局部三元组”变得有意义，因为大的跳跃只会恶化不对称的比率贡献。 
2. 用一个大值初始化一个答案变量。 这存储了迄今为止遇到的最佳表达值。 
3. 迭代每个连续的三元组$(a[i], a[i+1], a[i+2])$。 此限制起作用的原因是任何最佳配置都可以转换为这些本地窗口之一，而无需增加值。 
4. 对于每个三元组，直接使用浮点除法计算表达式的值。 
5. 如果当前三元组产生较小的值，则更新答案。 
6. 输出最终的最小值。 

### 为什么它有效

 排序后，考虑任何最佳三元组。 如果它包含按排序顺序不相邻的元素，则它们之间至少存在一个更接近三元组中位数的元素。 用中间元素替换相距较远的元素不能同时增加所有比率项，而且实际上会减少不平衡。 这迫使最佳配置压缩成大小为三的连续段。 因此，检查所有连续的三元组涵盖了可能是最佳的所有候选者。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = list(map(int, input().split()))
    a.sort()

    ans = float('inf')

    for i in range(n - 2):
        x, y, z = a[i], a[i + 1], a[i + 2]
        val = x / y + y / z + z / x
        if val < ans:
            ans = val

    print(ans)

if __name__ == "__main__":
    solve()
```该解决方案首先对数组进行排序，这对于将搜索空间限制为有意义的本地配置至关重要。 然后，循环评估每个大小为 3 的滑动窗口，直接计算表达式。 

计算使用浮点除法，除了 Python 默认的双精度之外，没有任何特殊的预防措施，这足以满足所需的要求$10^{-9}$宽容。 

主要的微妙之处在于仅考虑连续的三元组。 如果不排序，这个限制将无效。 正确性完全取决于首先对数组进行排序。 

## 工作示例

 ### 示例 1

 输入：```
3
6 2 4
```排序后的数组变为$[2, 4, 6]$。 

| 我| 三重| 价值|
 | ---| ---| ---|
 | 0 | (2,4,6) | 2/4 + 4/6 + 6/2 = 0.5 + 0.666... + 3 = 4.166... |

 答案是$4.1666666667$。 

该轨迹表明，只有一个可能的三元组，算法会直接对其进行评估。 它还强调了表达式对顺序很敏感。 

### 示例 2

 输入：```
5
1 10 2 9 3
```排序后的数组变为$[1, 2, 3, 9, 10]$。 

| 我| 三重| 价值|
 | ---| ---| ---|
 | 0 | (1,2,3) | (1,2,3) | 1/2 + 2/3 + 3/1 = 0.5 + 0.666... + 3 = 4.166... |

 | 2 | (3,9,10) | 3/9 + 9/10 + 10/3 ≈ 0.333... + 0.9 + 3.333... = 4.566... |

 最小值为$4.166...$。 

这演示了算法如何自然地过滤到最小的局部簇而不是大的分离值。

 ## 边缘情况

 一种边缘情况是所有数字都相等。 该算法计算单个三元组并返回 3，因为每个比率都变为 1。排序不会改变任何内容，并且每个窗口都会生成相同的值。 

另一种情况是当数字严格增加且差距较大时，例如$[1, 2, 1000, 100000]$。 该算法仍然只检查局部三元组。 对于第一个窗口$(1,2,1000)$，该值主要由$1000/1$，它很大，而其他窗口由于比率更大，情况甚至更糟。 即使全局直觉可能建议混合极端情况，该算法也能正确识别最佳局部簇。 

最后一个微妙的情况是，由于浮点接近性，最小三元组在视觉上并不明显。 因为所有计算都是以双精度完成的，并且所需的公差是$10^{-9}$，除了一致的评估顺序之外不需要特殊的精度处理。
