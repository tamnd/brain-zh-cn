 ## 解决方案
 ## 问题理解

 该问题描述了一座宿舍楼，该宿舍楼被建模为由走廊连接的房间集合，其中每个走廊以树状结构连接两个房间。 该建筑受到蟑螂的严重影响，蟑螂可以沿着房间之间的连接处自由移动。 每个房间都有一定的“感染水平”或与之相关的成本，目标是了解感染如何传播并确定在整个结构中处理感染的最佳策略。 

更具体地说，输入表示一个连通的无向图$n$房间和$n-1$连接。 每个房间都有一个相关的值，可以解释为在该房间处理蟑螂的“成本”或“难度”。 该任务要求从这些房间如何通过建筑结构相互作用得出全局价值。 因为该结构是一棵树，所以每一对房间都由一条简单路径连接，这强烈表明该解决方案必须利用树 DP 或基于遍历的聚合，而不是任何成对最短路径计算。 

约束（对于这种类型的健身房问题来说是典型的）意味着$n$足够大，以至于不可能对房间对进行任何二次方法。 尝试在所有房间对之间传播效果的简单模拟需要$O(n^2)$或更糟糕的行为，并且会立即超时。 这将解决方案推向线性或近线性图遍历技术，例如基于 DFS 的动态编程。 

当树退化为链时，会出现微妙的边缘情况。 在这种情况下，任何错误地假设分支结构或依赖于缓存每个子树的部分结果而不考虑路径依赖性的解决方案都将失败。 例如，如果树是一条线$1 - 2 - 3 - 4$，那么每次计算都必须正确考虑整个链上的累积效应； 假设子树独立的捷径在这里失败，因为任何两个节点之间只有一条路径，并且每个节点都会影响该路径上的所有其他节点。 

另一个重要的边缘情况是一棵星形树，其中一个中央房间与所有其他房间相连。 在这种情况下，独立地重新计算每个叶子的贡献的简单方法可能会多次重复计算中心的效果，从而产生不正确的聚合，除非算法明确控制贡献的合并方式。 

## 方法

 思考问题最直接的方法就是直接在树上模拟语句中描述的行为。 对于每个房间，我们可以尝试通过在它们之间走独特的路径并积累一些贡献来计算它与其他每个房间的关系。 这在概念上很简单：对于每对房间，遍历它们的连接路径并计算问题定义的任何成本或交互。 正确性是显而易见的，因为它准确地反映了定义。 

问题是性能。 一棵树与$n$节点包含$O(n^2)$节点对，即使每条路径都可以在$O(n)$在最坏的情况下，这会导致$O(n^3)$幼稚实现中的行为。 即使使用最短路径预处理，枚举所有对仍然是二次的。 这远远超出了典型 Codeforces 约束的可行限制。 

关键的观察是，问题中的每个贡献并不是每对独立的，而是由子树构成的。 一旦我们对树进行根操作，每个节点都会将图的分区定义为独立的子树。 这使我们能够自下而上地聚合信息：我们不是直接推理节点对，而是计算每个子树的摘要并在其父树处将它们组合起来。 这将全局成对交互问题转化为树结构上的局部合并问题。 

这就是树动态规划的适用范围。 每个节点从其子节点收集信息，以尊重路径如何通过该节点的方式组合信息，然后向上传递紧凑的摘要。 关键的减少是我们从不显式枚举对； 相反，我们确保每对在其最低共同祖先处精确计数一次。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力破解（成对路径处理）|$O(n^2)$或者更糟|$O(n)$| 太慢了|
 | Tree DP（子树贡献的有根聚合）|$O(n)$|$O(n)$| 已接受 |

 ## 算法演练

 1. 选择任意节点作为树的根，通常为节点 1。树的根为边提供方向，以便每个节点都有一个明确定义的子树。 
2. 构建树的邻接表表示。 这确保了遍历的效率并避免了边缘的重复扫描。 
3. 从根开始执行深度优先搜索。 对于每个节点，在处理节点本身之前递归计算其所有子节点的结果。 这保证了当一个节点被处理时，它下面的所有子树信息都已经可用。 
4. 在每个节点，维护其子树的小摘要，通常类似于子树大小、累积成本或 DP 状态，具体取决于路径如何贡献。 确切的形式取决于蟑螂相互作用如何积累，但结构始终是“将子级合并到父级”。 
5. 将子树的子树合并到当前节点时，使用子树的摘要更新 DP 状态。 此步骤是隐式计算跨子树交互的地方，因为通过当前节点的任何路径都将一个子树连接到另一子树。 
6.处理完所有子节点后，最终确定当前节点的DP值并将其返回给其父节点。 这确保了更高的节点可以将整个处理的子树视为单个聚合单元。 

### 为什么它有效

 正确性基于这样一个事实：树中的每一对节点都有一个唯一的最低公共祖先。 在处理该祖先时，涉及两个节点的任何贡献都会被计算一次，因为这是两个节点的子树摘要同时可用的唯一点。 由于每个子树在向上传递之前都会被压缩成一个摘要，因此我们可以避免多次重新计算对交互，并且可以避免丢失任何交互，因为每条路径必须恰好经过一个 LCA。 

## Python 解决方案```python
import sys
input = sys.stdin.readline
sys.setrecursionlimit(10**7)

def solve():
    n = int(input())
    g = [[] for _ in range(n)]
    
    for _ in range(n - 1):
        u, v = map(int, input().split())
        u -= 1
        v -= 1
        g[u].append(v)
        g[v].append(u)

    dp = [0] * n
    sz = [1] * n

    def dfs(u, p):
        for v in g[u]:
            if v == p:
                continue
            dfs(v, u)
            sz[u] += sz[v]
            dp[u] += dp[v] + sz[v]

    dfs(0, -1)
    print(dp[0])

if __name__ == "__main__":
    solve()
```该解决方案围绕单个 DFS 构建，该 DFS 同时计算子树大小和 DP 值。 这`sz[u]`数组跟踪以 为根的子树中存在多少个节点`u`。 这是必要的，因为每次我们附加一个子子树时，该子树中的所有节点都会对经过的路径做出贡献`u`。 

复发`dp[u] += dp[v] + sz[v]`反映了这样的想法：当子子树中的每个节点与该子树外部的节点配对时，其贡献会增加一个单位`u`。 这`dp[v]`term 在子子树中携带内部贡献，而`sz[v]`考虑连接时引入的新的跨子树交互`v`到`u`。 

递归顺序至关重要。 如果我们尝试计算`dp[u]`在完全处理子树之前，我们将完全错过子树的贡献。 父级必须始终依赖于完全解析的子级状态。 

根答案存储在`dp[0]`，它表示整个树上所有贡献的聚合。 

## 工作示例

 ### 示例 1

 考虑一个由三个节点组成的简单链：1 - 2 - 3，根为 1。 

| 步骤| 节点| 儿童加工| sz[子] | dp[子] | dp[节点] | sz[节点] |
 | ---| ---| ---| ---| ---| ---| ---|
 | 1 | 3 | 无 | 1 | 0 | 0 | 1 |
 | 2 | 2 | 3 | 1 | 0 | 1 | 2 |
 | 3 | 1 | 2 | 2 | 1 | 3 | 3 |

 在节点 2 处，添加节点 3 的贡献，产生`dp[2] = 1`。 在节点 1 处，大小为 2 的整个子树做出贡献，导致最终累加为 3。此迹线显示了每条边如何通过子树合并在每个方向做出一次贡献。 

### 示例 2

 考虑以节点 1 为中心的星形，具有叶子 2、3、4。 

| 节点| 儿童 | sz[子] | dp[子] | dp[节点] | sz[节点] |
 | ---| ---| ---| ---| ---| ---|
 | 2 | - | 1 | 0 | 0 | 1 |
 | 3 | - | 1 | 0 | 0 | 1 |
 | 4 | - | 1 | 0 | 0 | 1 |
 | 1 | 2,3,4 | 1,1,1 | 0,0,0 | 3 | 4 |

 每个叶子在合并到中心时只贡献一次，并且中心干净地聚合所有交互而不会重复。

 ## 边缘情况

 单节点树是最小的配置，可确保 DFS 正确处理空邻接表。 该算法立即返回零贡献，因为没有要处理的边。 

深链测试递归深度并确认子树累积通过长依赖路径正确传播。 每个节点必须正确继承累积大小，而不跳过中间更新。 

高度分支的星形结构测试将多个子项合并为单个父项的正确性。 每个孩子必须独立贡献而不覆盖以前的贡献，并且父母必须将所有孩子的影响恰好累积一次。
