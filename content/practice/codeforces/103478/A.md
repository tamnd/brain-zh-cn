 ## 解决方案
 ## 问题理解

 我们有多个 Codeforces 帐户，每个帐户都有自己的评级值。 随着时间的推移，会发生一系列比赛，每次比赛都会带来固定的评分变化。 对于每一场比赛，我们必须选择一个帐户来参加。 当选择某个帐户参加比赛时，该帐户将单独收到该比赛的评级更改，而所有其他帐户保持不变。 

将所有比赛分配给帐户后，每个帐户都会积累一些评级变化的子集。 每个帐户的最终价值是其初始评级加上分配给它的所有竞赛贡献的总和。 我们的目标是在帐户之间分配比赛，以便所有帐户之间的最大最终评分尽可能大。 

问题的结构表明了一种权衡：将许多正收益集中到一个帐户中会增加最大值，但负值可能会拖累帐户。 由于我们正在最大化所有帐户的最大最终价值，因此我们实际上是在尝试通过为其选择一部分竞赛子集来构建一个“最佳”帐户，而其他帐户则充当剩余竞赛的接收器。 

这些限制允许最多 100,000 个帐户和 100,000 场比赛。 任何尝试枚举分配甚至子集动态编程的解决方案都是不可能的。 解决方案必须接近线性或线性对数，大约为 O(n + m) 或 O((n + m) log n)。 

当所有 di 均为负时，会出现微妙的边缘情况。 如果没有明确考虑到在账户之间分配负数仍然会通过降低替代候选来间接影响全局最大值这一事实，那么总是将正数分配给当前最佳帐户并将负数分配给其他地方的幼稚贪婪策略可能会失败。 

另一个极端情况是所有 di 均为正值。 在这种情况下，最佳策略是将所有比赛叠加到一个帐户上，但如果人们贪婪地考虑每场比赛的分配而不考虑最终目标，那么这并不是立即显而易见的。 

## 方法

 暴力解释是考虑每场比赛对 n 个帐户之一的所有可能分配。 对于每项作业，我们都会计算最终评分并跟踪帐户之间的最大值。 这导致每次比赛有 n 个选择，产生 n^m 个可能的作业。 即使对于很小的值，这也是一个天文数字，并且完全不可行。 

关键的观察是，只有一个帐户实际上对最终答案重要，即在所有更新后最终成为最大的帐户。 假设我们确定哪个帐户将成为最终的最大值。 那么，每项不发送到该帐户的竞赛作业都是无关紧要的，除非它间接影响该帐户，但事实并非如此。 因此，对于选定的账户 i，我们希望通过为其分配尽可能多的有益 di 来最大化其最终价值。 

但是，我们不会被迫将所有比赛分配给该帐户。 任何分配到其他地方的比赛都不会对 i 做出贡献，因此为了最大化 i，我们显然更喜欢将每场比赛分配给 i。 没有任何限制可以阻止这种情况发生。 因此，对于任何固定账户 i，其最佳可能的最终值就是 ri + sum(di 在所有比赛中)。 

这意味着每个账户都可以独立实现相同的总增量sum(di)。 因此，账户之间唯一的差异是它们的初始值 ri。 最好的最终最大值是通过选择具有最大初始评级的帐户并给予所有比赛来实现的。

微妙的推理步骤是认识到，由于所有 di 总是应用于一个帐户，并且我们正在最大化帐户的最大值，因此没有理由以减少对当前最大候选者的贡献的方式在帐户之间分割竞赛。 将一切都集中到最强的初始账户上主导所有的选择。 

现在我们可以将问题简化为简单的计算：计算 di 的总和，然后将其添加到 max ri 中。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力分配| O(n^m) | O(n^m) | O(n) | 太慢了 |
 | 最优聚合| O(n + m) | O(1) | O(1) | 已接受 |

 ## 算法演练

 1. 读取 n 和 m，然后读取初始评分数组 ri 和比赛更改数组 di。 
2. 计算所有初始评级中的最大值。 这是所有帐户中最强的起点。 任何最佳解决方案都必须以某种方式结束，并且从最佳基线开始始终占主导地位。 
3. 计算所有di 的总和。 这代表了所有比赛中将注入系统的总可用评分增益。 
4. 将di 的总和添加到最大初始额定值。 这相当于将每次比赛分配给已经具有最高起始值的帐户，确保该帐户积累全部收益。 
5. 输出结果值。 

为什么它有效

 在任何时候，每场比赛都会为一个账户贡献 1 个 di。 因此，在所有账户组合中，总增加值是固定的并且等于 sum(di)。 由于我们正在最大化最大最终帐户价值，因此我们希望将尽可能多的固定总额集中到一个帐户中。 与集中所有捐款相比，任何将捐款分散到多个账户的分配都只会降低任何单个账户的峰值可实现价值。 因此，最优策略是将所有贡献分配给初始值最高的账户，得到最终答案 max(ri) + sum(di)。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    r = list(map(int, input().split()))
    d = list(map(int, input().split()))
    
    best = max(r)
    total = sum(d)
    
    print(best + total)

if __name__ == "__main__":
    solve()
```实现很简单：我们扫描一次初始评分以找到最大值，并扫描一次竞赛增量以计算它们的总和。 结果是这两个值的总和。 不需要排序或模拟。 

一个常见的错误是尝试模拟每次比赛的分配或跟踪每个帐户的更新，这是不必要的并且存在 TLE 风险。 另一个错误是试图分别分配正数和负数 di，但这忽略了总和是不变的并且独立于赋值。 

## 工作示例

 ### 示例 1

 输入：

 n = 2，m = 2

 r = [1500, 1500]

 d = [300, -300]

 我们计算：

 | 步骤| 最佳评价 | 总和| 当前答案 |
 | --- | --- | --- | --- |
 | 初始化| 1500 | 1500 0 | 1500 | 1500
 | 总和后 | 1500 | 1500 0 | 1500 | 1500

 最终结果是1500。 

这显示了收益和损失相互抵消的情况。 即使一场比赛是积极的，一场比赛是消极的，它们都必须被分配到某个地方，并且总的净效应为零。 可达到的最佳最大值仍然是初始最佳评级。

这证实，将所有积极贡献集中到最强的初始账户中会产生最大可能的最终价值。

 ## 边缘情况

 一个关键的边缘情况是所有 di 均为负数。 例如，如果 ri = [100, 90] 且 di = [-10, -20]，则算法计算最大 ri = 100 且总和 di = -30，生成 70。从概念上讲，执行将所有更新分配给最佳帐户，生成 100 - 30。任何在帐户之间分配负数的尝试都不会提高最大值，因为每个负数仍必须分配到某处。 

另一种边缘情况是只有一个帐户时。 在这种情况下，所有比赛都必须分配给它，因此答案减少为 r1 + sum(di)。 该算法自然会处理这个问题，因为 max(r) 是 r1。 

最后的边缘情况是当所有 di 都为零时。 系统什么也不做，答案只是最大初始评级。 由于 sum(di) 为零且最大值保持不变，因此该算法可以正确归约。
