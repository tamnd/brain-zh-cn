 ## 解决方案
 ## 问题理解

 我们给出的时间表是`n`天。 公司每天需要最多`k`CPU 核心，但不是购买固定套餐，而是可以从多个重叠的租赁报价中租用核心。 每个优惠仅在连续的一天间隔内有效，并且在每个有效日内最多提供`c_i`核心。 当天从该报价租用的每个核心的成本`p_i`。 

在任何一天，我们都会选择从每个有效报价中获取多少个核心。 如果所有活动报价的总可用容量至少为`k`，我们只取`k`核心，否则我们会利用一切可用的东西并接受需求未得到充分满足的事实。 目标是最小化全天的总成本。 

关键的困难在于提供时间重叠，但每天的决定都是本地的：每天我们都想首先选择最便宜的可用核心，最多`k`。 

这些限制迫使我们走向`O((n + m) log m)`或者`O(n log m)`解决方案。 和`n`高达一百万并且`m`多达二十万，任何每日排序或每日扫描所有活动报价都太慢。 甚至`O(nm)`是不可能的，因为它会达到 2e11 次操作。 

一个幼稚的错误是重新计算最好的`k`每天通过收集所有活跃报价并按价格对它们进行排序来获取核心。 那将是`O(n m log m)`在最坏的情况下。 

当昂贵的报价较早出现但较便宜的报价较晚出现时，就会出现另一种微妙的边缘情况。 如果我们不维护按价格排序的全局结构，我们可能会错误地提前购买昂贵的核心，而不是等待更便宜的核心可用。 

## 方法

 暴力方法独立对待每一天。 对于给定的一天，我们收集间隔包括该天的所有报价，按价格对它们进行排序，并贪婪地获取核心，直到我们达到`k`。 这是正确的，因为在一天之内，更便宜的核心应该始终是首选。 然而，每天重新计算活跃报价的成本很高。 在最坏的情况下，所有报价都跨越所有天，我们会重复排序`m`项目用于`n`天，给予`O(n m log m)`复杂性，远远超出了极限。 

关键的观察结果是，尽管可用性随着时间的推移而变化，但决策规则是静态的：始终首先采用最便宜的可用核心。 这意味着维护当前活跃报价的全局结构（按价格排序），并在时间间隔开始和结束时进行更新。 我们可以使用随时间变化的扫描线和支持插入、删除和提取最便宜容量的数据结构按顺序处理天。 

主要挑战是每个报价都有容量限制`c_i`每天，我们可能会部分使用它。 我们需要跟踪按价格排序的多集结构中的剩余容量。 每天，我们都会反复从最便宜的可用报价中消费，直到达到`k`或排气供应。 

我们可以使用平衡结构（例如与延迟删除或价格值上的线段树相结合的堆）对主动报价进行建模。 常见的 CF 解决方案通过处理事件并维护每个价格的多组容量来进行压缩，始终首先从最低价格桶中消耗。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(n·m log m) | O(n·m log m) | O(米) | 太慢了|
 | 最佳扫描 + 多重集 | O((n + m) log m) | O((n + m) log m) | O(米) | 已接受 |

 ## 算法演练

 我们处理一天中的时间`1`今天`n`同时保留按价格排序的所有活跃报价。 

1. 将每个报价转换为两个事件：当天`l_i`它变得活跃，并且在白天`r_i + 1`它变得不活跃。 这使我们能够增量更新活动容量，而不是每天从头开始重新计算。 这是必要的，因为每天重新检查所有报价会太慢。 
2. 维护一个以价格为关键的结构，该结构存储所有活动报价中该价格的可用总容量。 当报价生效时，我们添加`c_i`到它的价格范围； 当它到期时，我们减去它。 
3. 每天，我们都会反复从最便宜的价格桶中获取，直到我们收集到`k`核心或没有剩余容量。 我们总是首先用尽更便宜的价格，因为任何从更昂贵的桶中取出的单元，而存在更便宜的桶，都可以更换以降低成本。 
4.随着容量的消耗而积累成本。 当我们采取`x`核心价格`p`，我们添加`x * p`到答案。 
5. 处理完当前配置的所有消耗后移至第二天。 

### 为什么它有效

 在任何固定的一天，问题减少为选择最多`k`每个容量单位都有固定重量（价格）的物品，我们希望最大程度地降低总成本。 这相当于一个分数背包，其中所有物品都可以分为单位核心。 由于除了激活间隔之外，各天之间没有耦合，因此独立维护每天的正确性就足够了。 不变的是，在每天开始时，该结构恰好包含活跃报价的所有可用容量，并且通过增加价格贪婪地消费总是会产生当天的最佳选择。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, k, m = map(int, input().split())

    add = [[] for _ in range(n + 2)]
    rem = [[] for _ in range(n + 2)]

    for _ in range(m):
        l, r, c, p = map(int, input().split())
        add[l].append((p, c))
        rem[r + 1].append((p, c))

    import heapq

    active = {}
    heap = []

    def add_offer(p, c):
        active[p] = active.get(p, 0) + c
        heapq.heappush(heap, p)

    def remove_offer(p, c):
        active[p] -= c
        if active[p] == 0:
            del active[p]

    ans = 0

    for day in range(1, n + 1):
        for p, c in add[day]:
            add_offer(p, c)

        for p, c in rem[day]:
            remove_offer(p, c)

        need = k

        while need > 0 and heap:
            p = heap[0]
            if p not in active:
                heapq.heappop(heap)
                continue

            avail = active[p]
            take = min(need, avail)
            ans += take * p
            need -= take

            if take == avail:
                heapq.heappop(heap)
                del active[p]
            else:
                active[p] -= take

    print(ans)

if __name__ == "__main__":
    solve()
```该解决方案使用事件列表`add`和`rem`维护每天哪些优惠开始和结束。 堆存储候选价格。 字典`active`存储每个价格的总剩余容量。 

由于多个报价或容量发生变化，堆可能包含过时的条目； 因此，在使用顶部元素之前，我们验证它仍然存在于`active`。 如果没有，我们就懒惰地丢弃它。 

消费逻辑总是先从最便宜的价格中移除，并在需要时部分消费桶。 

一个微妙的一点是，我们从不跟踪单独的报价，只跟踪每个价格的聚合容量。 这是有效的，因为相同价格的所有单位都可以互换。

 ## 边缘情况

 一个重要的边缘情况是报价仅持续一天。 基于事件的更新可确保在同一迭代中正确添加和删除它。 例如：```
1 2 3 5
```在第 1 天激活并在第 3 天删除，因此它仅在第 1 天和第 2 天起作用。add/rem 列表保证了这一点，无需特殊的大小写。 

另一个边缘情况是相同价格的多个报价。 由于我们按价格聚合容量，因此我们避免了分裂逻辑。 堆可能包含重复的价格条目，但延迟删除可确保正确性。 

最后的边缘情况是总容量小于`k`。 当堆为空时循环停止，并且我们正确地仅累积可用核心而不强制完全分配。
