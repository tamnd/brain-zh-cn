 ## 解决方案
 ## 问题理解

 每天都会产生一定数量的垃圾单位，每个单位都必须在其出现的当天或第二天将其丢弃。 通过将设备装入袋子中进行处置，每个袋子可容纳固定容量$k$，并且在准备好当天立即清空袋子。 目标是安排最多一天的垃圾运送方式和分装袋的方式，从而最大程度地减少使用的袋子总数。 

关键的限制是垃圾的生命周期非常短：最多连续两天可用。 这会产生相邻日期之间的局部交互，而不是全局调度问题。 输出只是覆盖这些邻接约束下的所有单元所需的最小容量限制组数。 

输入大小达到$2 \cdot 10^5$，因此任何尝试模拟所有天数之间分配的二次策略都太慢。 唯一可行的方法是线性或近线性，因为大约$10^8$简单操作是 Python 类 C 效率中 3 秒的安全上限，而 Python 开销使我们更接近$10^7$有意义的操作。 

当需要推迟垃圾处理以避免浪费袋子容量时，就会出现微妙的边缘情况。 例如，如果一天的数量较少，而第二天的数量较多，那么天真地每天单独打包可能会浪费容量，而在允许的窗口内向前或向后移动单位可以减少行李数量。 

另一个极端情况发生在最后一天。 当天产生的垃圾$n$不能进一步推迟，因此任何隐含假设未来平衡能力的策略都必须谨慎处理边界。 

## 方法

 一个蛮力的想法是模拟所有可能的决策：每天，决定第二天保留多少单位以及立即处置多少单位，然后尝试将所有分配放入袋子中。 这将需要探索指数级的多种配置，因为每个单元可以分配给两天中的一天，然后每天分组到袋子中会引入另一个组合层。 即使我们限制自己对剩余单元进行日常动态规划，状态空间也会随着可能的进位值而增长$k$，进行过渡$O(nk)$，当$k$取决于$10^9$。 

结构的简化来自于我们从不需要显式模拟单个单元的观察。 重要的是一天中有多少单位$i$推迟到今天$i+1$，因为从当天开始推迟单位$i$必须与日一起处理$i+1$。 一旦我们确定了结转的单位数量，当天的剩余决策就变得确定了：我们只是贪婪地装满袋子以尽量减少数量。 

贪婪的核心洞察是，对于每一天，我们希望通过将前一天的剩余单元与当前单元配对来最大限度地重用容量。 我们不是跟踪单个任务，而是跟踪每天开始时仍有多少个单元仍在“待处理”，然后计算必须延迟到第二天的额外单元数，以最大限度地减少本地行李数量。 

这将问题简化为单个从左到右的传递，并使用表示未处理的遗留垃圾的简单状态变量。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | 指数 /$O(nk)$|$O(nk)$| 太慢了|
 | 最佳 |$O(n)$|$O(1)$| 已接受 |

 ## 算法演练

 我们按顺序处理日期，同时维护当天可包装的垃圾单元数量，包括新垃圾和前一天的剩余垃圾。 

1. 初始化变量`carry`为0，代表前一天推迟的垃圾，并初始化`ans`为 0 表示使用的袋子数量。 
2. 每天$i$， 添加$a_i$到`carry`，因为所有新生成的垃圾在今天或明天都可用。 
3. 计算今天我们可以使用所有可用垃圾形成多少个装满的袋子。 每个袋子可容纳$k$单位，所以我们可以完成的袋子数量`carry`是`carry // k`。 将此值添加到答案中并减少`carry`相应地使用模$k$。 
4. 剩余的`carry`形成完整袋子后代表今天无法装满整个袋子的剩余单元。 这些剩余物必须与第二天的垃圾合并，或者在第二天完成，因此它们保持原样。 
5、转移到第二天之前，确保剩余的不超过可以合理处理的范围。 由于每个单位最多只能向前移动一天，因此任何剩余的单位都自动有效作为结转。 
6. 加工日后$n$, 剩下的全部`carry`必须在最后一天处理。 添加`carry // k`再次回答。 

正确性取决于这样一个事实：某一天的任何部分行李都应使用可用单位尽快完成，因为延迟完成不会增加容量，但可能会失去配对机会。 

### 为什么它有效

 在任何一天的边界，所有剩余的单位只能与第二天进行交互，而不能再进行交互。 因此，本地的最佳策略是始终立即填充尽可能多的满袋，因为任何推迟形成满袋的分组都无法提高一天之后的未来配对潜力。 不变的是处理一天后$i$，所有剩余的单位正是那些必须重新考虑的单位$i+1$，并且所有已完成的包都针对该前缀进行了最佳固定。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, k = map(int, input().split())
    a = list(map(int, input().split()))

    carry = 0
    ans = 0

    for x in a:
        carry += x
        ans += carry // k
        carry %= k

    ans += carry // k
    print(ans)

if __name__ == "__main__":
    solve()
```该代码维护一个可用垃圾单元的运行池，并在每一步贪婪地提取完整的垃圾袋。 整数除法准确地捕获了当前可用性可以形成多少个完整的袋子，而模数仅保留必须与第二天交互的剩余单元。 最后的添加处理最后一天的清理工作。 

一个微妙的点是，我们从来没有明确决定有多少单位转移到第二天； 在形成完整的组之后，这被隐式编码在剩余部分中，这已经足够了，因为只有剩余部分可以从未来的组合中受益。

 ## 边缘情况

 单日少于$k$单位，例如输入`1 10 / 3`，算法集`carry = 3`，在迭代期间计算没有满袋，然后丢弃最终的进位，因为`3 // 10 = 0`，正确生成 0。 正确性来自于不可能推迟的事实，因此每个单位必须在整袋形成方面保持未使用状态。 

一天正好等于$k$， 例如`1 10 / 10`，该算法通过以下方式恰好形成一个包`carry // k = 1`并将进位减少到零。 这证实了在每个步骤中贪婪地提取满袋可以正确处理精确配合的场景，而不需要任何特殊的外壳。
