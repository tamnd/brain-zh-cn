 ## 解决方案
 ## 问题理解

 我们得到了一个简化的类似斯诺克的计分系统的目标状态。 任何时刻都有两名玩家，其中一名目前在牌桌上。 我们知道玩家 A 和玩家 B 的当前分数，我们知道桌子上还剩多少个球，我们知道轮到谁了。 

我们的任务不是计算分数，而是重建从帧开始到这个精确状态的任何可能的事件序列。 每个事件要么是成功投进一个已知分值从 1 到 7 的球，要么是失手，这会导致回合切换而不改变得分。 

因此，输出是一个描述有效的按时间顺序排列的操作历史记录的字符串。 我们可以输出任何有效的序列，只要它与最终得分、剩余球数以及轮到谁一致即可。 如果不存在这样的序列，我们输出 NA。 

就状态空间而言，约束很小。 分数和剩余球都最多为 200，测试用例的数量最多为 2000。这立即表明我们不希望模拟长分支历史或做任何指数级的事情。 相反，我们正在处理一个受限的可行性重建问题，其中的关键困难是分数增量和剩余球数之间的一致性。 

朴素推理中最常见的失败案例是将其视为一个简单的分区问题：将分数差异与一些底池值的多重集相匹配。 这忽略了两个关键限制。 首先，回合会在未命中时交替进行，因此顺序必须尊重当前玩家。 其次，剩余的球数量限制了可能发生的成功彩池数量，因为每个彩池都会减少可用的彩池。 

当分数非零但没有球剩余时，就会出现微妙的边缘情况。 在这种情况下，所有得分都必须更早发生，并且任何超出可用球的得分尝试都会立即无效。 另一种边缘情况是当两个分数都为零但转弯指示器与任何有效的起始移动顺序不一致时，特别是当需要强制失误但由于缺乏剩余球而无法放置时。 

## 方法

 暴力的观点是想象生成所有可能的长度序列，直到某个界限，模拟彩池和未命中，跟踪分数变化和剩余球，并检查我们是否达到目标状态。 这在概念上是正确的，因为每个有效的游戏历史都包含在搜索空间中。 

然而，即使有乐观的界限，分支因子也很大。 在每一步中，球员都可以错过七种球类型中的一种，或者将其中一种击入，每个状态至少有八次转换。 即使深度为 30 也已经产生了天文数字般的序列。 最终序列长度最多为 100 的约束也无助于暴力破解，因为它只限制答案，而不限制搜索空间。 

关键的结构观察是状态被极度压缩。 除了比分、剩余球和回合的一致性之外，我们不关心顺序。 这意味着我们可以通过仅关注发生了多少得分动作以及需要多少次失误来调整最终球员来对解决方案进行逆向工程。 

一旦我们忽略排序自由度，问题就简化为构造一组得分动作，其总和恰好等于分数差，同时考虑到每个得分动作消耗一个球，并且剩余球约束给出总得分动作的上限。 之后，未击中的球只是用来翻转回合以匹配最终球员的填充。 

本质上的简化在于得分事件和未命中事件的解耦：得分决定得分的可行性，未命中决定奇偶性和回合对齐。

 ## 算法演练

 我们将分数差表示为 Δ = a + b，但我们必须分别对待 A 和 B，因为我们只控制当前玩家的动作，而不是全局的。 相反，我们将游戏解释为一系列交替的回合，其中每次移动要么增加分数值，要么结束回合。 

### 1. 验证基本可行性边界

 我们首先确保所需的评分动作总数最多为 n + k，其中 k 隐式地受到游戏结构的限制。 由于每个成功的罐子都会消耗一个球，因此罐子的总数不能超过初始球的数量加上任何隐式补充，而这里不存在隐式补充。 因此，我们必须保证a+b≤n和规则所隐含的初始有效评分能力。 如果违反，我们会立即返回 NA。 

存在这一步的原因是光有分数是不够的，还需要有分数。 拥有足够的球来产生这个分数必须在身体上是可能的。 

### 2. 将分数分解为球值

 我们使用可用值 {1,2,3,4,5,6,7} 独立构造 a 和 b 的表示。 由于我们只需要任何解决方案，因此贪婪分解有效：重复减去不超过剩余分数的最大可能值。 

这是可行的，因为硬币系统是有限的且很小，并且我们不会针对超出可行性的最小长度进行优化。 

### 3. 确保球消耗量与剩余球台相符

 每个成功的底池消耗一个球。 因此得分动作的总数为S，并且在达到状态之前我们必须有S≤可用的初始球。 由于我们给定了 n 个剩余球，消耗球的总数是固定的，因此我们通过检查隐含的初始球数减去 S 是否与 n 匹配来确保一致性。 

如果发生不匹配，则不存在有效的历史记录。 

### 4.构造交替序列

 现在，我们将得分动作放入从初始玩家 p = 0 或 1 开始的时间轴中。我们模拟向前回合：

 每当当前玩家仍有剩余分数可供分配时，我们就会附加底池操作。 否则我们插入一个未执行动作“/”来切换回合。 

这构建了一个有效的序列，尊重回合交替并确保所有得分都正确归因。 

关键思想是，失误是免费的结构性工具：它们不会影响得分或球数，只会影响平局。 

### 5.修复最终玩家对齐

 构建序列后，我们计算重播后哪个玩家将出现在牌桌上。 如果它与 p 不匹配，我们会附加一个额外的未命中或通过在开头插入最小数量的未命中来进行调整。 由于仅错过切换回合，奇偶校验调整始终是可能的，除非受到不可能的评分分解的限制。 

### 为什么它有效

 该算法依赖于评分动作完全确定数字状态转换的不变量，而错过仅控制奇偶校验。 由于除了得分之外没有任何动作会影响得分，并且得分不依赖于归因之外的回合历史记录，因此我们可以任意重新排序未命中而不影响可行性。 因此，当且仅当满足数字约束时，将分数有效分解为允许的值，并结合足够的奇偶校验校正，才能产生有效的游戏历史记录。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

vals = [7, 6, 5, 4, 3, 2, 1]

def decompose(x):
    res = []
    for v in vals:
        while x >= v:
            res.append(v)
            x -= v
    if x != 0:
        return None
    return res

def solve_case(a, b, n, p):
    A = decompose(a)
    B = decompose(b)
    if A is None or B is None:
        return "NA"

    total_scores = len(A) + len(B)

    if total_scores > n + 21:
        return "NA"

    # build actions: we interleave greedily
    i = j = 0
    turn = 0
    res = []

    while i < len(A) or j < len(B):
        if turn == 0:
            if i < len(A):
                res.append(str(A[i]))
                i += 1
            elif j < len(B):
                res.append("/")
            else:
                break
        else:
            if j < len(B):
                res.append(str(B[j]))
                j += 1
            elif i < len(A):
                res.append("/")
            else:
                break
        if res[-1] == "/":
            turn ^= 1

    cur = 0
    for ch in res:
        if ch == "/":
            cur ^= 1

    if cur != p:
        res.append("/")

    if len(res) > 100:
        return "NA"

    return "".join(res)

def main():
    T = int(input())
    out = []
    for _ in range(T):
        a, b, n, p = map(int, input().split())
        out.append(solve_case(a, b, n, p))
    print("\n".join(out))

if __name__ == "__main__":
    main()
```分解步骤使用从 7 到 1 的固定评分值的贪心减法。这是安全的，因为我们不需要最小的射击次数，只要求可行性，任何剩余失败都会立即证明不可能。 

构建阶段独立处理 A 和 B，然后在尊重转弯变化的同时将它们交错。 每当当前玩家不能或不应该行动时，就会插入一次未命中，并且该未命中会翻转回合。 

最后，我们调整奇偶性，最多增加一次未命中。 这已经足够了，因为单个切换可以修复计算的起始玩家奇偶性与所需的起始玩家奇偶性之间的任何不匹配。

 ## 边缘情况

 一个关键的边缘情况是当两个分数都为零但所需的玩家是 B 时。算法会生成一个空序列，然后附加一个未命中来翻转回合。 这是有效的，因为不存在评分限制，因此奇偶校验是唯一的要求。 

当分数很小但在贪婪减法下无法完全分解时，会出现另一种边缘情况，例如 1 时仅错误地假设了更高的值。 在我们的系统中，这种情况不会发生，因为 1 始终可用，但任何删除 1 的变体都会立即破坏通过余数检查进行的可行性检测。 

最后的边缘情况是当总构造长度超过 100 时。在这种情况下，即使存在有效的分解，我们也必须拒绝，因为无法满足输出约束。 该算法显式检查这一点并返回 NA，确保符合问题的格式限制。
