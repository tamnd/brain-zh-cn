 ## 解决方案
 ## 问题理解

 我们得到了一系列物品。 每件物品都有重量和价值，但对于某些物品，这两个数字中恰好缺少一个，因此我们必须将其指定为不超过 10 亿的正整数。 

稍后将在完成的数据集上运行两个贪婪程序。 爱丽丝通过增加重量对物品进行排序，然后按该顺序取出物品，直到下一个物品超出背包容量，此时她完全停止。 Bob 按价值递减对项目进行排序，然后按该顺序进行扫描，每当项目适合剩余容量时就取出该项目。 

我们的任务是分配缺失的权重和值，以便两个过程最终选择完全相同的一组项目。 

关键的难点在于，分配同时影响排序顺序和背包可行性，因此我们在单一容量约束下同时控制排序和累积总和。 

约束足够小，以至于每个测试用例的二次甚至稍微超三次推理在所有情况下都是不可接受的。 总共最多有 3000 个项目，但真正的困难不是原始计算，而是在来自排序和容量的混合约束下构建一致的分配。 

如果试图独立对待爱丽丝和鲍勃，一种微妙的失败模式会立即出现。 Alice 的选择仅取决于权重排序和前缀和，而 Bob 的选择取决于值排序和增量可行性。 首先修复一个排序的天真尝试通常会使另一个贪婪过程发散，因为第二个过程对中间剩余容量敏感，而不仅仅是最终总和。 

## 方法

 蛮力视角将尝试分配所有缺失值，然后模拟这两种策略。 这显然是不可能的，因为每个未知数最多可以占用十亿个值，从而给出指数或无限的搜索空间。 

更结构化的观察来自于了解爱丽丝的程序真正产生的结果。 一旦权重固定，Alice 不会随意选择，她总是按权重顺序取一个前缀，并在第一个不适合的项目处停止。 这意味着爱丽丝选择的集合完全由该排序中的阈值决定。 

鲍勃的过程在选择顺序上更加灵活，但仍然是贪婪的，因为一旦项目按值排序，他就会尝试在可能的情况下打包它们。 

关键思想是，我们应该避免任何交互，即 Alice 获取的项目与 Bob 未获取的项目在任一顺序中交错。 如果我们可以强制两个过程就在两个排序标准下稳定的项目的共同“接受区域”达成一致，那么两个贪婪过程的行为相同。 

保证相同行为的最简单方法是使两种算法有效地选择所有项目，这将问题减少为使两个贪婪程序都接受每个项目，而不会因容量或排序冲突而拒绝。 

为此，我们分配所有缺失的权重和值，以便每个项目都合适，并且每个项目都以一致的顺序处理，不会发生拒绝的情况。 

我们确保权重总和至多为W，并且我们确保值排序不能阻止包含。 一旦所有项目都适合，Alice 就会获取所有内容，因为前缀步骤不会溢出，而 Bob 会获取所有内容，因为遇到的每个项目都适合。 

唯一的障碍是有些重量是固定的，无法减少。 然而，由于我们可以分配未知的权重，我们总是可以选择足够小的权重，以便只要固定权重不超过容量就满足总和约束。

这导致了直接的可行性检查：如果已经固定的权重之和大于 W，则任何分配都无法挽救 Alice 的过程，因为这些权重是不可变的。 否则，我们可以将每个未知重量分配为 1，并确保总重量保持在容量范围内。 值可以指定为任意正数，例如全部相等，这使得 Bob 的排序与平局决胜无关。 

这使得两个贪婪程序都接受每一项。

 ## 算法演练

 我们构建一个解决方案，其中 Alice 和 Bob 都选择所有项目。 

1. 计算所有已固定的给定权重（非零条目）的总和。 这些重量无法更改，因此它们形成了背包总使用量的硬性下限。 
2. 如果这个固定总和已经超过 W，则不可能进行分配，因为即使在添加任何未知权重之前，Alice 在尝试获取所有项目时也会超出容量。 在这种情况下，我们立即输出该任务是不可能的。 
3. 否则，为每个未知权重分配值 1。这使总权重尽可能小，同时满足权重必须为正整数的要求。 
4. 现在考虑 Alice 的过程。 由于所有权重都是正数，并且它们的总和不超过 W，因此权重排序顺序的每个前缀仍然可行，因此 Alice 永远不会遇到会溢出背包的物品。 因此，她选择了每一项。 
5. 将所有未知值也指定为 1。 这使得所有值都相同，因此 Bob 的排序变得稳定并且仅取决于原始索引。 
6. Bob 按某种顺序处理所有项目，但由于每个项目都适合剩余容量（因为总重量在 W 内），因此他永远不会因容量而跳过。 因此鲍勃也选择了每一项。 

### 为什么它有效

 该结构将两个贪婪过程简化为一个微不足道的机制，其中容量限制永远不会触发拒绝。 Alice 的停止条件永远不会激活，因为总重量受 W 限制。Bob 的跳跃条件永远不会激活，因为每个项目在考虑时都是单独适合的。 由于两个过程都包含所有项目，因此它们选择的集合是相同的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    T = int(input())
    for _ in range(T):
        n, W = map(int, input().split())
        w = list(map(int, input().split()))
        r = list(map(int, input().split()))
        
        fixed_sum = 0
        for x in w:
            if x != 0:
                fixed_sum += x
        
        if fixed_sum > W:
            print("No")
            continue
        
        print("Yes")
        
        # assign weights
        res_w = []
        for x in w:
            if x == 0:
                res_w.append(1)
            else:
                res_w.append(x)
        
        # assign values
        res_r = []
        for x in r:
            if x == 0:
                res_r.append(1)
            else:
                res_r.append(x)
        
        print(*res_w)
        print(*res_r)

if __name__ == "__main__":
    solve()
```实施直接跟随构造。 唯一重要的步骤是计算固定权重的总和并验证 W 的可行性。其他一切都是直接将 1 分配给缺失的条目。 

一个微妙的一点是我们永远不会修改给定的值。 这很重要，因为该问题保证了每件物品最多缺少重量或价值之一，因此覆盖时不会发生冲突。 

## 工作示例

 ### 示例 1

 考虑容量 W = 10 的三个项目。 

假设权重是`[3, 0, 2]`和价值观是`[0, 5, 0]`。 

固定权重总和为`3 + 2 = 5`，在能力范围内。 

我们将缺失的权重和值指定为 1。 

| 步骤| 重量| 价值观 | 固定金额 | 有效 |
 | ---| ---| ---| ---| ---|
 | 初始| 3, 0, 2 | 0, 5, 0 | 5 | 是的 |
 | 填写后| 3, 1, 2 | 1, 5, 1 | 1, 5, 1 | 5 | 是的 |

 Alice 按照重量递增的顺序处理物品，最多累计 6 个，≤ 10，所以她拿走了所有物品。 鲍勃按价值递减的顺序处理物品，并且每个物品都适合，因此他也拿走所有物品。 

这证实了相同的选择发生。 

### 示例 2

 令 W = 6，权重`[2, 0, 5]`, 值`[0, 3, 0]`。 

固定权重总和为`2 + 5 = 7`，已经超出了容量。 

| 步骤| 固定金额 | 决定|
 | ---| ---| ---|
 | 初始| 7 | 不可能|

 由于即使不可变的权重也超出了容量，Alice 在任何分配之前都将被迫失败，因此不存在解决方案。 

这表明了唯一的障碍：单独固定权重可能会违反可行性。

 ## 边缘情况

 一个关键的边缘情况是当固定重量已经超过容量时。 例如，如果某项的权重为 10^9 并且 W 较小，则任何分配都无法减少它。 该算法立即正确拒绝，而不尝试分配。 

另一种边缘情况是所有值均缺失但权重固定。 即使在这种情况下，价值分配也是无关紧要的，因为一旦满足 Alice 的可行性，选择就完全由权重决定。 

最后一个微妙的情况是存在多个零。 由于我们总是分配 1，因此我们确保总重量的膨胀最小，同时保持正确性，并且我们永远不会意外地超出所构造解决方案的容量。
