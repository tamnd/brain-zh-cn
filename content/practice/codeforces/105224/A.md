 ## 解决方案
 ## 问题理解

 我们得到一个从 1 到 n 的一维位置板。 每个位置 i 包含一个跳跃长度 t[i]。 如果一个球落在位置 i，它会重复执行确定性跳跃：从 i 移动到 i + t[i]，然后从新位置 j 移动到 j + t[j]，依此类推，直到位置超过 n。 此时球被认为已离开棋盘。 

有趣的是数组不是静态的。 我们必须支持两项操作。 一项操作会更改单个索引处的跳转长度。 另一个要求我们从给定的索引开始模拟整个跳跃过程，并报告两个值：球离开棋盘之前仍在棋盘内的最后一个位置，以及退出之前进行了多少次跳跃。 

约束 n 和 q 都高达 100000，并且每次跳跃可能会移动很远或保持很小，具体取决于数组。 如果结构是对抗性的，每个查询的直接模拟很容易退化为二次行为，因为单个查询可能遍历许多位置，并且可能有许多这样的查询。 

一个天真的期望是，在最坏的情况下，每个查询类型 1 可能需要 O(n)，这导致总工作量为 O(nq)。 对于 100000 次操作，这远远超出了可行的极限。 

一个微妙的边缘情况是跳跃始终停留在一个小区域内，例如对于所有 i，t[i] = 1。 从位置 1 开始，我们将按顺序访问每个位置，直到 n，每个查询生成 n 个步骤。 如果有很多查询，这会立即变得太慢。 另一个边缘情况是频繁的更新，它会改变单个值并完全改变路径，使得跨查询的记忆不可靠，除非仔细构建。 

## 方法

 蛮力方法很简单。 对于从 i 开始的查询，我们使用当前数组重复跳转，直到离开范围。 我们计算步数并记住最后一个有效位置。 每次更新仅更改一个值。 

这是正确的，因为它确实模拟了流程的定义。 问题是在最坏的情况下单个查询可以遍历 O(n) 个位置。 当 q 达到 100000 时，最坏情况总数变为 O(nq)，这太大了。 

关键的观察是，每个位置要么跳转到较远的位置，要么保留在重复小跳转行为可预测的区域。 我们希望避免单独模拟每个中间步骤。 在更新下加速这种“下一个指针跳跃”过程的标准方法是，对于每个位置，维护我们将到达的下一个位置，该位置保持在受控块结构内，并且还维护到达下一个位置需要采取多少步。 

一种众所周知的技术是索引的 sqrt 分解。 我们将数组分成大小约为 sqrt(n) 的块。 对于每个索引，我们预先计算退出当前块的位置以及退出当前块需要多少步。 处理查询时，我们重复地逐块跳转，而不是逐步跳转。 当更新发生时，我们仅重新计算受影响的块。 

在块内部，跳转是显式模拟的，但每个块遍历在查询期间都被压缩为单个跳转。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | 每次查询 O(n)，更新 O(1) | O(n) | 太慢了 |
 | 块分解| 每次查询 O(√n)，更新 O(√n) | O(n) | 已接受 |

 ## 算法演练

 我们将索引 1 到 n 分成大小为 B 的块，通常在 sqrt(n) 左右。 对于每个位置 i，我们维护两个值：nxt[i]，我们按照跳跃规则离开 i 块后的下一个位置，以及 cnt[i]，到达 nxt[i] 或退出块所需的跳跃次数。

1. 构建块并从右到左初始化nxt和cnt。 我们处理索引，以便在计算 i 时，我们已经知道 i + t[i] 的结果是否位于同一块内部或外部。 
2. 对于每个索引 i，计算 j = i + t[i]。 如果 j > n，则 nxt[i] 位于数组外部，并且 cnt[i] 为 1。这表示导致退出的单次跳转。 
3. 如果j与i在同一个块内，则nxt[i]和cnt[i]继承自j。 我们设置 nxt[i] = nxt[j] 和 cnt[i] = cnt[j] + 1。这将多个内部跳转压缩为一个摘要。 
4. 如果 j 位于不同的块中但仍在数组内，则设置 nxt[i] = j 且 cnt[i] = 1。这意味着下一个压缩跳立即离开该块。 
5. 对于从 i 开始的查询，我们重复应用这些块跳转。 我们累积总计数并跟踪最后一个有效位置。 每次从 i 跳转到 nxt[i] 时，最后一个位置是跳转前的 i，并将 i 更新为 nxt[i]。 
6. 当 i 超过 n 时，过程停止，我们输出最后一个有效位置和总步数。 
7. 对于位置 i 的更新，我们更改 t[i] 并重新计算 i 块内的所有 nxt 和 cnt 值，因为只有该块的内部结构可能已更改。 

其起作用的关键原因是，在一个块内，所有链在重新计算后都是局部一致的，并且跨块跳跃极大地减少了路径长度。 每个查询最多跨越 O(√n) 个块，因为每个块遍历一次会跳过许多位置。 

### 为什么它有效

 nxt 和 cnt 维护的结构充当由跳转规则引起的功能图的压缩表示。 在块内，每条路径最终都会退出块或数组，并且我们会预先计算退出发生的确切位置。 由于更新仅影响一个块，并且重新计算会在本地恢复正确性，因此每次跳转要么遵循有效的预计算链，要么转换到新的块边界。 这确保了压缩过程中的每个模拟步骤与原始跳跃序列精确对应，从而保留最终位置和步数。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

n, q = map(int, input().split())
t = list(map(int, input().split()))

B = int(n ** 0.5) + 1

nxt = [0] * n
cnt = [0] * n

def rebuild(block):
    l = block * B
    r = min(n, (block + 1) * B)
    for i in range(r - 1, l - 1, -1):
        j = i + t[i]
        if j >= n:
            nxt[i] = n
            cnt[i] = 1
        else:
            if j // B == i // B:
                nxt[i] = nxt[j]
                cnt[i] = cnt[j] + 1
            else:
                nxt[i] = j
                cnt[i] = 1

for b in range((n + B - 1) // B):
    rebuild(b)

for _ in range(q):
    tmp = input().split()
    if tmp[0] == '0':
        i = int(tmp[1]) - 1
        x = int(tmp[2])
        t[i] = x
        rebuild(i // B)
    else:
        i = int(tmp[1]) - 1
        pos = i
        steps = 0
        last = i

        while pos < n:
            last = pos
            steps += cnt[pos]
            pos = nxt[pos]

        print(last + 1, steps)
```实现的核心是`rebuild`函数，它重新计算一个块的压缩跳转信息。 它从右到左处理索引，以便每当我们使用`nxt[j]`和`cnt[j]`，它们已经有效。 

在查询期间，我们不是一次单步执行一个索引，而是使用重复跳转`nxt[pos]`，它会跳过块内的整个链。 最后一个有效位置始终是退出数组的最终跳转的开始。 

更新操作仅涉及一个块，因此重新计算保持在本地。 最微妙的细节是一致地处理索引：存储结构内部使用基于 0 的索引，而输出需要基于 1 的位置。

    t = list(map(int, input().split()))

                    cnt[i] = cnt[j] + 1
                else:
                    nxt[i] = j
                    cnt[i] = 1

    for b in range((n + B - 1) // B):
        rebuild(b)

    out = []
    for _ in range(q):
        tmp = sys.stdin.readline().split()

 ## 边缘情况

 一个关键的边缘情况是每次跳转都停留在单个块内，这通常会强制逐步遍历。 块分解可以处理这个问题，因为内部链是预先计算并折叠成 nxt 指针的。 即使我们直到最后一步才离开块，cnt 也会累积完整的内部路径长度。 

另一种边缘情况是单次更新更改块边界附近的值时。 如果不重新计算整个块，后续查询可能会遵循陈旧的 nxt 指针。 重建整个块可确保所有内部依赖关系再次保持一致。 

最后，当 t[i] 大到足以立即退出数组时，nxt[i] 设置为 n，cnt[i] 为 1。这可以防止在链压缩期间发生任何意外的超出范围的取消引用，并确保查询干净地终止，即使连续发生多个此类跳转也是如此。
