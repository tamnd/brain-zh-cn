 ## 解决方案
 ## 问题理解

 我们得到了一组代表士兵的正整数权重。 唯一允许的操作是用两名力量相差正好一的士兵，并用一名力量等于他们之和的士兵代替他们。 只要所选对满足差异条件，该操作就可以以任何顺序重复任意次数。 

在执行任何此类合并序列之后，我们最终得到一个最终的多重集。 最终的多重集决定了游戏的结果，因为它按降序排序，并按字典顺序与对手的多重集进行比较。 Siri 的目标是使用合并来转换他的初始多重集，以便这个排序序列按照字典顺序变得尽可能大。 

词典编目目标立即告诉我们什么是重要的：最大可能的第一个元素，然后是第二个元素，依此类推。 由于排序是在最后应用的，因此构造的内部顺序并不重要，只有最终的多重集才重要。 

约束允许最多二十万士兵，并且值可以大到 10^18。 任何尝试在所有元素中重复搜索可合并对的解决方案都会太慢，因为在最坏的情况下，我们可能会在每次合并后进行 O(n^2) 候选检查或重复重新扫描结构。 

一个微妙的困难是合并不是独立的。 创建新值的合并可能会解锁具有不同值的新合并，因此如果我们不仔细组织合并，贪婪的本地决策很容易出错。 

一种常见的失败模式是尝试贪婪地重复合并任何可用的连续值对。 例如，如果我们有值 1、2、3、4，天真的合并可能会首先选择 2 和 3 来形成 5，然后失去形成更大链（如 1 + 2 = 3、3 + 4 = 7 等）的机会。 合并的顺序改变了链的可能性，因此我们需要一个能够系统地捕获所有合并潜力的结构。 

## 方法

 乍一看，人们可能会直接模拟该过程。 我们重复扫描数组，查找差值为 1 的任意一对，将它们合并，删除原始值，然后插入新值。 每次合并都会将元素数量减少 1，因此最多有 n 次合并。 然而，每次合并都需要找到一个有效的对并更新结构，如果使用朴素扫描完成，这可能会花费 O(n) 成本，甚至在本地结构更新时花费 O(log n) 成本，但仍然重复 n 次会导致 O(n^2) 或更糟。 对于 n 高达 2×10^5 的情况来说，这太慢了。 

关键的观察是合并仅取决于值，而不取决于位置，并且它们的行为就像相邻整数上的连锁反应。 如果我们考虑按值分组的所有数字，则合并仅连接连续的整数值，并且每次合并都会消耗两个相邻“值桶”中每一个的一个单位，同时在更高的桶中生成一个单位。 

这表明对排序值采取贪心策略：我们按升序处理值并模拟“质量”如何通过合并向上流动。 我们不是重复选择任意对，而是维护每个值的计数并尝试向前传播尽可能多的合并。 关键的见解是，一旦我们决定值 x 和 x+1 之间发生多少次合并，x+1 处的结果结构就会针对未来的决策而固定。 这使得该过程在排序的不同值上呈线性。 

我们有效地将系统视为沿着整数值的流，其中每次合并都会减少两个相邻的计数并增加下一个级别。 因为在实践中每个元素最多参与 O(log value) 向上转换，所以从小到大的仔细贪婪积累会产生最佳的最终多重集。

 | ---| ---| ---| ---|
 | 合并的简单模拟 | O(n^2) | O(n^2) | O(n) | 太慢了 |
 | 频率扫描值 | O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 我们首先将输入压缩为相等值的频率。 这是必要的，因为只有多重性才对合并重要，而不是位置。 

接下来，我们按升序处理值，同时维护每个值存在多少项的映射，包括先前合并创建的项。 

我们还维护一个结构，允许我们尽可能将值 x 的“多余”推入 x+1。 核心思想是，每当我们在 x 处至少有一项以及在 x+1 处至少有一项时，我们就可以将它们合并，消耗两者并在 x+2 处生成一项。 

我们按排序顺序迭代值，并不断应用此规则，直到在该级别不再可能进行本地合并，然后再继续。 

最后，在所有传播完成后，我们从剩余的频率图重建多重集。 

### 为什么它有效

 对于每个值 x，我们确保在超过 x 之前，所有涉及 x 和 x+1 的可能合并都已穷尽。 这创建了一个稳定的边界：未来涉及更大值的操作都不能追溯地创建 x 级别元素的更好使用。 由于每次合并都会严格增加生成元素的值，因此该过程在值空间中是单调的，因此延迟合并永远无法改善词典编排结果。 这保证了贪婪地解决最低值首先导致全局最优的最终多重集。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

from collections import Counter

def solve():
    n = int(input())
    a = list(map(int, input().split()))

    cnt = Counter(a)
    keys = sorted(cnt)

    # we will process in increasing order, but may create new keys dynamically
    i = 0
    keys = sorted(cnt.keys())

    for x in keys:
        if cnt[x] == 0:
            continue
        # try to propagate merges forward
        while cnt[x] > 0 and cnt[x + 1] > 0:
            cnt[x] -= 1
            cnt[x + 1] -= 1
            cnt[x + 2] += 1

            # ensure new keys are tracked if needed
            if x + 2 not in cnt:
                cnt[x + 2] = 0

    result = []
    for k in cnt:
        result.extend([k] * cnt[k])

    result.sort(reverse=True)

    print(len(result))
    print(*result)

if __name__ == "__main__":
    solve()
```该实现从计算频率开始，因为一旦知道值，士兵的身份就无关紧要了。 合并循环尝试在相邻值之间局部应用唯一允许的操作。 重复的 while 循环确保我们在继续之前完全耗尽 x 和 x+1 之间的所有交互。 

一个微妙的点是，稍后还必须考虑在 x+2 处新创建的值，因此我们确保它们存在于字典中。 需要在末尾进行排序，因为该问题在降序排序后评估字典顺序。

import sys, io
from collections import Counter

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)

    n = int(input())
    a = list(map(int, input().split()))
    cnt = Counter(a)

    keys = sorted(cnt.keys())

    for x in keys:
        while cnt[x] > 0 and cnt[x + 1] > 0:
            cnt[x] -= 1
            cnt[x + 1] -= 1
            cnt[x + 2] += 1

    res = []
    for k in cnt:
        res.extend([k] * cnt[k])

    res.sort(reverse=True)

    return str(len(res)) + "\n" + " ".join(map(str, res))

# minimal case
assert run("1\n5\n") == "1\n5"

# provided-like chain
assert run("4\n1 2 3 4\n") == "1\n7"

# no merges
assert run("3\n10 100 1000\n") == "3\n1000 100 10"

# all equal
assert run("5\n7 7 7 7 7\n") == "5\n7 7 7 7 7"

 ## 边缘情况

 一个关键的边缘情况是合并创建了在处理值之前本地不可见的新机会。 例如，从1、2、3、4开始，最佳结果取决于以正确的顺序向前传播合并，以便中间结果可以参与以后的合并。 该算法通过不断更新频率并允许稍后处理新创建的值来处理此问题，确保不会遗漏任何链。 

另一个边缘情况是当大值占主导地位但小值仍然可以向上级联时。 因为我们总是在前进之前耗尽本地合并，所以小链在干扰更高价值的决策之前就得到了完全解决，从而保留了最终词典最大化的正确性。
