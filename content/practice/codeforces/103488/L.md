 ## 解决方案
 ## 问题理解

 我们得到了一个参考字符串`s`长度`n`，和一个上限`m`另一个字符串的长度`t`我们被允许建造。 任务是产生一个字符串`t`超过小写英文字母，使其长度最多为`m`，并且严格按字典顺序小于`s`。 在所有这些有效字符串中，我们想要字典顺序上最大的一个。 

关键的压力是我们同时在两个方向上进行优化。 我们想要`t`按字典顺序尽可能大，但仍然严格小于`s`。 同时，我们可以调整长度到`m`，并且用小字符增加长度有时可以在不违反约束的情况下增加词典价值。 

这里使用的字典顺序是标准的：两个字符串逐个字符进行比较，第一个不匹配的地方决定顺序。 如果一个字符串是另一个字符串的前缀，则较短的字符串较小。 

约束条件允许`n`和`m`最多 1e6。 任何尝试枚举候选字符串的解决方案都立即不可行，因为即使迭代所有字符串直至达到长度`m`将是指数级的。 即使每个候选构造的线性扫描也必须仔细限制，因此解决方案在最坏的情况下必须本质上是 O(n) 或 O(n log Alphabet)。 

一些边缘情况很重要。 

如果`s`开始于`'a'`重复多次，说`s = "aaaa"`，那么答案一定会变成这样`"aaa"`或者`"aa..."`根据`m`，因为任何以`'a'`存在等于或超过的风险，任何较大的字符都会立即违反约束。 

如果`s`有一个像这样的角色`'b'`或早更高，说`s = "caaaa"`，那么我们通常可以用较小的字符替换该位置，例如`'b'`或者`'a'`然后自由最大化后缀。 

如果没有正确处理严格减少的第一个点以及后缀上的后果，只在每个位置局部查看并尝试尽可能长时间地保持相等的天真贪婪可能会失败。 

## 方法

 一个蛮力的想法是构造所有长度达到`m`，检查它们按字典顺序是否小于`s`，并取其中的最大值。 甚至限制长度`m`， 这是$26^m$，这是不可能的。 

一个不太天真的想法是从左到右逐个字符地构建答案，始终尝试最大可能的字符并通过与`s`。 然而，如果我们模拟每个候选扩展的完整比较，我们仍然会得到一个额外的因子`O(n)`每次检查，都会导致二次行为。 

关键的观察是我们只需要考虑一个结构事件：我们构造的字符串变得严格小于的第一个位置`s`。 在那个位置之前，我们被迫匹配`s`如果我们想保持尽可能大的话。 在第一次不匹配时，我们选择一个严格小于的字符`s[i]`，在那之后，我们可以独立地最大化字符串的其余部分`s`。 

这将问题转化为扫描分裂点。 对于每个位置`i`，我们可以考虑在`i`，意味着我们匹配`s[0..i-1]`，然后选择最大的字符`< s[i]`在位置`i`，然后填充余数（直到长度`m`） 和`'z'`。 我们还考虑了匹配整个前缀的可能性`s`然后缩短字符串，因为正确的前缀按字典顺序也更小。 

最佳答案是所有此类选择中最好的。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(26^m) | O(26^m) | O(米) | 太慢了 |
 | 最佳 | O(26·n) | O(26·n) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们在处理字符串的同时跟踪最佳的可能结构。 

1. 考虑每个位置`i`从`0`到`n-1`作为潜在的第一个失配点。 在这个位置，我们要选择一个严格小于的字符`s[i]`。 最好的这样的角色是`s[i] - 1`，因为最大化字典顺序意味着选择仍然使字符串更小的最大有效字母。 
2. 对于每个位置`i`，形成一个候选字符串，其组成为`s[0:i] + (s[i]-1) + 'z' * (m - i - 1)`，只要`i < m`。 这确保了我们在强制严格的职位不平等之后保持尽可能大的规模`i`。 
3. 还要考虑我们在第一个过程中不引入任何不匹配的情况`n`人物。 在这种情况下，唯一的方法是严格小于`s`是一个正确的前缀`s`，所以我们考虑`s[0:k]`为所有人`k < n`和`k ≤ m`。 
4. 在所有有效候选者中，选择字典顺序最好的一个。 
5.返回最佳候选者。 

填充背后的原因`'z'`是一旦我们已经确保字符串在第一个不同位置按字典顺序更小，所有后面的位置就不再影响与`s`，所以我们贪婪地最大化后缀。 

### 为什么它有效

 任何有效的解决方案都必须有第一个索引，它不同于`s`，或者它必须是严格的前缀。 在第一个不同的索引处`i`，字符必须严格小于`s[i]`，任何比最大可能值更小的选择只会使结果变得更糟，而不会获得可行性。 在那之后，后缀不受以下方面的约束`s`，所以选择`'z'`无处不在最大化词典编纂价值。 由于我们枚举了所有可能的第一个不匹配位置加上前缀情况，因此我们涵盖了所有有效的结构。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    s = input().strip()

    best = ""

    # Case 1: prefix-only answers
    limit = min(n, m)
    for k in range(limit):
        cand = s[:k]
        if cand > best:
            best = cand

    # Case 2: introduce first mismatch at position i
    for i in range(min(n, m)):
        if s[i] == 'a':
            continue
        prefix = s[:i]
        c = chr(ord(s[i]) - 1)
        rem_len = m - i - 1
        if rem_len < 0:
            continue
        cand = prefix + c + ('z' * rem_len)
        if cand > best:
            best = cand

    print(best)

if __name__ == "__main__":
    solve()
```该代码首先尝试所有前缀选项，因为任何前缀`s`自动变小。 这涵盖了最佳答案避免修改的情况`s`而是缩短它。 

然后它尝试将每个位置作为严格减少的第一个点。 在位置`i`，我们选择最好的角色`s[i] - 1`如果有效，然后贪婪地扩展`'z'`。 边界检查`m - i - 1 < 0`确保我们不超过长度限制。 

一个微妙的点是我们不需要考虑位置上的多个较小的字符`i`，因为任何小于的字符`s[i]-1`产生严格更差的词典结果。

这表明确保不匹配保证的重要性`t < s`，而不仅仅是一味地最大化后缀。

 ## 边缘情况

 一种边缘情况是当`s`完全由`'a'`。 在这种情况下，没有位置允许减少不匹配结构中的字符。 唯一有效的答案是严格的前缀。 该算法自然地处理这个问题，因为不匹配循环会跳过所有位置，只保留前缀候选，所以最好的是`s[:m-1]`如果`m < n`。 

另一个边缘情况是当`m > n`。 在这里，仅前缀候选永远不会达到长度`n`，并且错配结构可以扩展到`n`安全。 由于后缀充满`'z'`，该算法仍然会产生最大允许长度的有效候选者。 

最后一个微妙的情况是最佳解决方案是非常短的前缀。 例如，如果`s = "baaa"`和`m = 4`, 前缀`"b"`是有效的，并且可能会超出一些不匹配结构，具体取决于词典比较规则。 前缀循环确保不会错过这些，因为它们被明确视为独立候选者。
