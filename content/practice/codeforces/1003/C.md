 ## 解决方案
 ## 问题理解

 我们得到了一系列每日温度，并要求评估长度至少为给定阈值的所有连续时间间隔。 对于每个这样的间隔，我们计算其平均温度，即其值的总和除以其长度。 任务是找到所有有效区间的最大可能平均值。 

因此，我们优化的对象不是固定的窗口大小，而是长度从最小值开始并可以扩展到整个数组的每个段。 这立即使问题比固定窗口最大平均值更加微妙，因为较长的段可以用较低的和密度换取较长的跨度。 

输入大小限制最多允许 5000 天。 对所有子数组进行简单的三次方法并重新计算总和已经是临界点了。 即使是重新计算总和的双循环也会太慢，除非使用前缀总和进行优化。 然而，即使使用前缀和，在 n = 5000 时枚举所有 O(n²) 段仍然可行，因为在 Python 中大约可以接受 2500 万次操作。 

隐藏的困难在于我们正在优化一个比率，而不是一个线性函数。 这意味着最佳段不一定是允许的最短或最长的段之一，并且段长度不存在单调结构。 

一些边缘案例暴露了不正确的贪婪想法。 

如果所有温度都相等，则每个段具有相同的平均值，因此任何有效段都是最佳的。 正确的解决方案不得依赖于选择特定的长度。 

如果数组严格递增，则最佳平均值可能来自短后缀段而不是长前缀，因为包含较早的低值会降低平均值。 

如果 k 等于 n，则答案就是整个数组的平均值。 任何在选择段长度方面具有灵活性的解决方案仍然必须干净地处理此边界。 

## 方法

 蛮力的想法很简单。 我们考虑每个起始索引，然后扩展结束索引，并且仅评估长度至少为 k 的段。 对于每个候选片段，我们计算其总和并除以其长度，跟踪最大值。 

通过前缀和，我们可以在 O(1) 内计算任何段和。 这将总复杂度降低到 O(n²)，因为大约有 n²/2 个段。 当 n = 5000 时，这大约是 1250 万次评估，每个评估都包含一些算术运算，这是可以接受的。 

关键的观察是，虽然我们无法避免检查许多段，但我们可以避免重复重新计算总和。 比率结构不允许更快的修剪或二分搜索技巧，因为目标在段长度或端点上不是单调的。 

因此，最佳解决方案本质上是通过前缀和使暴力破解思想变得高效。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(n3) | O(1) | O(1) | 太慢了 |
 | 所有段的前缀总和 | O(n²) | O(n) | 已接受 |

 ## 算法演练

 1. 构建一个前缀和数组，其中 prefix[i] 存储前 i 个元素的和。 这允许在恒定时间内计算任何段总和。 
2. 迭代从 1 到 n 的所有可能的左端点 l。 每个选择都固定候选片段的开始。 
3. 对于每个 l，迭代从 l + k - 1 到 n 的所有右端点 r。 这保证了每个段的长度至少为 k。 
4. 对于每对 (l, r)，使用 prefix[r] − prefix[l − 1] 计算段和。 
5. 将总和除以 (r − l + 1) 计算平均值。 
6. 跟踪所有有效段的最大平均值。 

我们从 l + k − 1 开始 r 的原因是不允许使用较短的段，并且只会引入无效的候选者，而不会为答案做出贡献。 

### 为什么它有效

每个有效段都由一对端点 (l, r) 唯一表示，其中 r − l + 1 ≥ k。 该算法仅枚举所有此类对一次。 由于平均值是针对每个段精确计算的，并且最大值是在完整的候选集上取的，因此结果必须与全局最优值相匹配。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, k = map(int, input().split())
    a = list(map(int, input().split()))

    pref = [0] * (n + 1)
    for i in range(n):
        pref[i + 1] = pref[i] + a[i]

    best = 0.0

    for l in range(n):
        for r in range(l + k - 1, n):
            total = pref[r + 1] - pref[l]
            length = r - l + 1
            avg = total / length
            if avg > best:
                best = avg

    print(best)

if __name__ == "__main__":
    solve()
```构造前缀和数组，以便每个范围和查询成为单个减法。 这很关键，因为没有它，内循环将变成 O(n3)。 然后，嵌套循环枚举所有有效段，同时直接遵守索引范围中的最小长度约束。 

浮点除法是安全的，因为所需的精度是 1e-6，而 Python 的双精度足以满足大约 5000 × 5000 的总和。

## 测试用例```python
import sys, io

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    import sys
    input = sys.stdin.readline

    n, k = map(int, input().split())
    a = list(map(int, input().split()))

    pref = [0] * (n + 1)
    for i in range(n):
        pref[i + 1] = pref[i] + a[i]

    best = 0.0
    for l in range(n):
        for r in range(l + k - 1, n):
            total = pref[r + 1] - pref[l]
            best = max(best, total / (r - l + 1))

    return str(best)

assert run("4 3\n3 4 1 2\n")[:5] == "2.666", "sample 1"
assert run("3 3\n1 2 3\n") == "2.0", "whole array only"

assert run("5 1\n5 5 5 5 5\n") == "5.0", "all equal"

assert run("5 2\n1 100 1 1 100\n")[:4] == "50.5", "multiple peaks"

assert run("1 1\n7\n") == "7.0", "single element"

assert run("6 3\n1 2 3 4 5 6\n")[:4] == "4.5", "increasing array"

 ## 边缘情况

 当 n 等于 k 时，循环结构仍然有效，因为对于每个 l 恰好有一个有效的 r。 在这种情况下，该算法有效地计算整个数组的平均值。 

对于像 [1, 2, 3, 4, 5] 这样的严格递增序列，长度至少为 k = 2 的最佳段不一定是完整数组。 枚举检查像 [4, 5] 这样的段，其平均值为 4.5，这击败了较长的段，如 [1, 2, 3, 4, 5]，其平均值为 3。这证实了将注意力限制在全范围段上是不正确的，并且详尽的枚举是必要的。
