 ## 解决方案
 ## 问题理解

 我们得到了一组硬币值，其中每个硬币都是 2 的幂。 每个查询都会询问我们是否可以使用这些硬币的子集形成目标总和，如果可以，则需要的最小硬币数量是多少。 每个币每次查询最多可以使用一次，不同查询之间互不影响。 

输入的结构非常重要：一切都不是任意值，而是 2 的幂。 这立即表明每个值的行为都类似于二进制数字贡献，而问题实际上在于我们是否有足够的2的每个幂的供应来组装目标的二进制表示。 

限制很严格：币数和查询数最多可达 200,000 个。 任何通过扫描所有硬币来处理每个查询的解决方案都太慢，因为在最坏的情况下这将导致大约 400 亿次操作。 即使对所有硬币类型的每次查询贪婪扫描只有在 2 的不同幂的数量很小的情况下才是可接受的，但在最坏的情况下，值的范围可以达到 2×10^9，因此最多大约 31 个不同的幂是相关的。 n 和不同指数的数量之间的差异是关键的结构简化。 

当将此问题视为一般的背包问题并试图贪婪地挑选不超过剩余金额的最大硬币时，就会出现天真的错误。 这种方法可能会失败，因为它忽略了全球范围内较小硬币的可用性，而这些硬币是弥补缺失的较高功率所必需的。 

例如，如果我们有硬币 [8, 2, 2, 2] 并且我们想要 10 个，那么贪婪的方法可能会需要 8 个，然后陷入错误的推理，如果它对可用性簿记管理不善，它就无法形成 2。 正确答案是 2 个硬币 (8 + 2)，问题不在于排序，而在于固定功率桶中的正确计数。 

另一个微妙的边缘情况是当目标需要跨越二进制级别时，例如 7 需要 4 + 2 + 1。如果任何级别缺乏供应，则更高级别无法补偿，因为硬币在超出其 2 的幂之外是不可分割的。 

## 方法

 暴力方法会尝试每个查询的所有硬币子集，检查总和并跟踪最小大小。 这立即是不可行的，因为它会在每个查询中探索 2^n 个子集。 即使限制求和仍然会导致指数行为。 

一个更结构化的粗暴想法是独立处理每个查询，并贪婪地尝试使用可用硬币来构建总和，始终采用不超过剩余目标的最大可能硬币。 这更接近正确，因为硬币值是 2 的幂，但如果我们直接消耗硬币而不预处理每个幂的计数，我们会重复扫描或递减数组，导致 O(nq) 或更糟。 

关键的见解是，二的幂的行为就像二进制系统一样。 每种硬币类型对应于一个比特位置，并且问题归结为具有多组比特容量。 对于每个查询，我们模拟目标的二进制分解，同时在需要时借用更高位。 

我们不再考虑单个硬币，而是汇总每种力量的数量。 然后，对于查询，我们尝试通过使用每个级别的可用硬币来满足 b 的二进制表示，贪婪地偏好较低级别，但在必要时从较高级别借用。 借贷始终是最佳的，因为一枚更高功率的硬币可以替换两枚次低功率的硬币。 

此结构将最多 31 位级别的每个查询减少到 O(log A) 处理。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(2^n·q) | O(2^n·q) | O(1) | O(1) | 太慢了 |
 | 每查询贪婪扫描 | O(nq) | O(n) | 太慢了 |
 | 最优位贪心计数 | O(n + q log A) | O(n + q log A) | O(log A) | 已接受 |

 ## 算法演练

1. 计算 2 的每一次方有多少枚硬币。 我们将其存储在一个数组中，其中索引 i 表示我们拥有多少个价值 2^i 的硬币。 这会将输入压缩到最多 31 个有用的存储桶中。 
2. 对于每个查询，我们都会使用这些计数的临时副本。 我们不能修改全局计数，因为查询是独立的。 
3. 我们从最低到最高迭代位位置。 在每一位 i 处，我们决定应该使用多少硬币来满足目标的第 i 位。 
4. 如果查询的第i位是1，我们需要产生一个单位2^i。 如果我们至少有一枚具有这种能力的硬币，我们就会使用它。 如果没有，我们向上搜索可以分割的最小的更高功率的硬币。 
5. 当在 j > i 级使用更高功率的硬币时，我们在概念上将其分解为 2^i 的 2^(j-i) 个单位。 我们重复分裂直到达到第 i 层，然后使用一个单元并将剩余部分传播回中间层。 这确保我们在各个级别上保持正确的可用性。 
6. 满足所需位后，我们通过存储的计数自然地将剩余容量结转到更高位。 
7. 如果在任何时候即使从所有更高级别借用后我们仍无法满足所需位，则答案为 -1。 
8. 结果是本次建造过程中使用的硬币总数。 

正确性依赖于以下不变量：在每一步中，可用硬币的多重集恰好代表将任何借来的更高功率硬币完全扩展至当前水平后的剩余资源。 由于分割是确定性的并保留总价值，因此一旦提交了更高的代币，没有其他安排可以减少使用的代币数量。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MAXB = 31

n, q = map(int, input().split())
cnt = [0] * MAXB

a = list(map(int, input().split()))
for x in a:
    b = x.bit_length() - 1
    cnt[b] += 1

for _ in range(q):
    b = int(input())
    cur = cnt[:]
    ans = 0

    possible = True

    for i in range(MAXB):
        if (b >> i) & 1:
            if cur[i] > 0:
                cur[i] -= 1
                ans += 1
            else:
                j = i + 1
                while j < MAXB and cur[j] == 0:
                    j += 1
                if j == MAXB:
                    possible = False
                    break
                cur[j] -= 1
                for k in range(j - 1, i, -1):
                    cur[k] += 1
                ans += 1
        cur[i + 1 if i + 1 < MAXB else i] += cur[i] // 2
        cur[i] %= 2

    print(ans if possible else -1)
```该解决方案首先将所有硬币压缩到按指数索引的频率桶中。 每个查询都在复制的状态上工作，以便转换保持独立。 主循环从低位到高位处理位，确保较低位先于较高位被解析，这一点至关重要，因为较高级别的分割会向下传播。 

关键的实现细节是传播步骤`cur[i+1] += cur[i] // 2`。 这模拟了向上携带未使用的较低硬币，保持与二进制结构的一致性。 如果没有这种标准化，剩余的小额硬币就会被错误计数或重复使用。 

借用逻辑显式搜索最近的更高可用硬币，将其拆分到所需的级别，并更新中间计数。 这保留了正确性，因为每次分割的价值都是可逆的，但会增加硬币数量，这正是我们正在测量的。

 ## 边缘情况

 一个关键的边缘情况是解决方案必须完全依赖于分裂单个高功率代币。 对于输入一个硬币 16 和查询 6，算法首先尝试满足位 1，找不到 2，然后借用 16，将其拆分为 8+8，继续拆分直到达到 2，最后使用三个 2。 不变的是，每次分割都保留总价值，同时增加硬币数量，并且由于不存在替代硬币，这是唯一可行的结构。 

另一个边缘情况是存在多个更高功率的硬币，但选择最接近的更高功率很重要。 例如，使用 8 而不是 16 可以减少不必要的分割深度。 贪婪向上搜索确保了最少的分解步骤，同时仍然产生最佳的硬币计数。
