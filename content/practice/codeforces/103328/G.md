 ## 解决方案
 ## 问题理解

 我们正在控制一个简单的工厂，它每天只能生产一个单位，每天我们都必须选择该单位是产品 A 还是产品 B。这个决定是全局性的，因为我们从第一天开始建立一个单一的时间线，并且每一天都对两个计数器之一做出贡献。 

沿着时间线，有 N 个检查事件。 每次检查都在特定的日子 ti 进行，并询问截至该天工厂是否已生产至少 xi 单位的指定产品类型 pi（A 或 B）。如果当时满足要求，我们将收到奖励 vi。 

任务是随着时间的推移选择生产计划，以最大化所有满意检查的总回报。 

重要的隐藏结构是每次检查仅独立于 A 和 B 的前缀计数，并且生产是每天的单一共享资源。 因此，分配给 A 的每一天都会减少 B 的可用时间，反之亦然，从而产生必须进行全局优化的权衡。 

约束条件很大，N 可达 100000，乘以 10^9。 这排除了任何尝试日复一日模拟的方法。 即使迭代所有天也是不可能的，因为时间线很稀疏并且天数实际上是无限的。 我们必须压缩事件点的时间，并根据分配而不是明确的时间表进行推理。 

当两个检查竞争相同的有限前缀资源时，就会出现天真的贪婪推理的微妙失败情况。 

例如，假设我们同时进行检查：

 在时间 5，A 需要 3 个单位来获得奖励 10，B 需要 3 个单位来获得奖励 10，但距离该时间只有 5 天。 同时满足两者是不可能的，因此除非我们了解全局权衡结构，否则通过更高的奖励或更早的约束进行本地选择可能会产生误导。 

核心难点在于，每次满足的检查都会消耗 A 端或 B 端的容量，并且容量以前缀约束的方式跨时间共享。 

## 方法

 思考该问题的一种强力方法是明确考虑生产计划。 对于每天直到最大时间，我们选择 A 或 B，然后模拟所有检查并对奖励求和。 这已经变得不可行，因为时间达到 10^9，即使我们将自己限制为仅事件日，在 k 个相关日内 A 和 B 分配的每种配置都会产生 2^k 种可能性，这种可能性会立即爆炸。 

一个稍微结构化一点的蛮力是考虑我们在每个相关时间生产了多少个 A 单位。 一旦我们解决了这个问题，B 就会自动确定。 对于时间 t 的每次检查，我们可以计算它是否满足。 这将问题简化为选择前缀分配序列，但决策空间仍然是指数的，因为不同时间的分配会相互作用。 

关键的观察结果是每次检查仅取决于时间 ti 之前生产的 A 和 B 单元的数量。 如果我们按时间对所有检查进行排序，我们可以按升序处理它们，并维护到目前为止每次分配的 A 和 B 的数量。 

在固定的时间边界上，我们只关心截至该时间存在多少个生产槽位，以及如何将它们划分为 A 和 B。这将问题转化为为每个时间前缀选择分配多少个 A 单元，同时确保在所有约束条件下的可行性。 

随着时间的推移，这将成为一种经典的资源分配，其中每个约束都对 A 或 B 前缀和施加下限。 我们可以将每个检查重写为时间 ti 上 A 单位数量的约束，因为 B 单位总数减去 A 单位。

因此，对于每次检查，我们都会获得 A 前缀在时间 ti 的可能值的约束区间。 每个约束都会贡献一个利润 vi，我们必须选择一个满足所有所选约束的可行分配。 

这自然可以通过按时间顺序处理约束并维护可行范围的结构来解决，使用带有前缀可行性检查的贪婪或扫描线。 最终的减少是，我们每次只需要跟踪在满足所有活动约束的情况下允许分配多少个 A 单元，并且我们贪婪地接受保持可行性的约束。

 ## 算法演练

 我们首先将问题转换为 A 的前缀计数的约束系统。 

1. 我们通过增加时间 ti 对所有检查事件进行排序。 这是必要的，因为时间 t 的可行性仅取决于早期的生产决策，并且按时间顺序处理可确保我们仅遵守仍可能受到影响的约束。 
2. 我们将时间 ti 的总产能定义为 ti 生产单位，因为每天生产一个单位。 如果我们在时间 ti 之前确定 A 的 x 单位，则 B 具有 ti − x 单位。 
3、每次检查都成为一个约束：

 如果 pi 是 A，那么我们需要 x ≥ xi。 

如果 pi 是 B，那么我们需要 ti − x ≥ xi，即 x ≤ ti − xi。 

因此，每次检查都会定义当时 A 单元数量的下限或上限。 
4. 我们以增加的时间处理约束并维护我们选择满足的所有约束的交集。 在任何一点，该交集都是一个区间 [L, R]，描述 A 截至当前时间的可行值。 
5. 每个约束都有一个值vi。 我们希望选择约束子集，使得结果间隔在每个前缀时间保持非空，从而最大化总价值。 这相当于在选择最大奖励子集的同时保持可行性。 
6.我们使用贪婪结构：对于每次约束，我们尝试将它们包括在内并更新L和R。如果区间变得无效（L>R），我们丢弃影响冲突的约束中最不有价值的约束。 这是使用按值键控的优先级队列来实现的。 
7.处理完所有约束后，保留的vi之和就是答案。 

优先级队列起作用的原因是，当出现不可行性时，至少必须移除一个约束。 删除最小的 vi 是最优的，因为约束仅在价值上相加，并且除了通过可行性之外不会相互作用。 

### 为什么它有效

 在任何时间前缀，所选约束集定义了 A 单元数量的凸可行区域。 添加新约束会缩小该区域或使其变空。 如果它变成空，则冲突是由不兼容的边界引起的，并且需要删除约束才能恢复可行性。 

由于所有约束都独立地影响目标，并且仅通过可行性相互作用，因此最优选择始终保持可行性，同时最大化保留权重。 贪婪地删除最小值约束可确保我们在每次冲突时损失尽可能少的奖励，并且未来的决策不依赖于删除哪个特定的低值约束超出其对区间边界的影响。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    events = []
    
    for _ in range(n):
        t, x, v, p = input().split()
        t = int(t)
        x = int(x)
        v = int(v)
        
        if p == 'A':
            # A constraint: A(t) >= x
            events.append((t, 1, x, v))  # type 1 = lower bound
        else:
            # B constraint: B(t) >= x => A(t) <= t - x
            events.append((t, 0, t - x, v))  # type 0 = upper bound
    
    events.sort()

    import heapq

    L = -10**18
    R = 10**18
    pq = []  # store (-value, bound_type, bound)

    total = 0

    for t, typ, bound, v in events:
        total += v
        heapq.heappush(pq, (v, typ, bound))

        if typ == 1:
            L = max(L, bound)
        else:
            R = min(R, bound)

        while L > R:
            # remove smallest value event
            v2, typ2, bound2 = heapq.heappop(pq)
            total -= v2

            # recompute bounds from scratch for safety
            L = -10**18
            R = 10**18
            tmp = []
            for vv, tt, bb in pq:
                if tt == 1:
                    L = max(L, bb)
                else:
                    R = min(R, bb)
                tmp.append((vv, tt, bb))
            pq = tmp
            heapq.heapify(pq)

    print(total)

if __name__ == "__main__":
    solve()
```该代码首先将每次检查转换为对时间 t 时 A 单元数量的约束。 A 类检查成为下限，而 B 类检查在与总时间补足后成为上限。 

然后，我们按时间对事件进行排序，以便按时间顺序应用约束。 堆存储当前接受其值的约束，因此我们可以在发生冲突时删除最不有利的值。 

变量 L 和 R 跟踪所有活动约束的交集。 当它们交叉时，可行性就被破坏了，因此我们删除约束，直到间隔再次有效。 重新计算步骤用于确保删除后的正确性，但以效率为代价。 

## 工作示例

 考虑一个小情况，其中约束出现的次数不断增加。 

输入：

 t=3 A 需要 2 给出 5

 t=3 B 需要 1 给出 4

 t=3 A 需要 3 给出 10

 我们按时间顺序处理。 在 t=3 时，我们转换：

 A ≥ 2 则 L = 2

 B ≥ 1 给出 A ≤ 2

 A ≥ 3 则 L = 3

 | 步骤| 左 | 右 | 主动约束| 行动|
 | --- | --- | --- | --- | --- |
 | 1 | 2 | 3 | 第一个约束 | 好的 |
 | 2 | 2 | 2 | + B 约束 | 好的 |
 | 3 | 3 | 2 | + A≥3 约束 | 冲突，删除最低值 |

 删除最小值约束可以解决冲突，留下最佳可行子集。 

这表明冲突的边界如何直接对应于 A 单元的不可行分配。

 ## 边缘情况

 关键的边缘情况是所有约束同时发生。 在这种情况下，整个问题就归结为选择相交的区间约束子集，而贪婪移除就变得至关重要。 

输入：

 t=5 A≥3 v=10

 t=5 B≥4 v=8

 t=5 A≥5 v=6

 处理产生L和R立即冲突，因为如果B≥4，A≥3和A≤1不能共存。 该算法删除最小值约束，直到恢复可行性，最终保持最佳一致子集。 

另一种边缘情况是当约束随着时间的推移在 A 和 B 之间交替时，会产生一个变窄的可行范围。 堆确保如果后期的高值约束与许多早期的低值约束发生冲突，则首先删除低值约束，从而保留最佳的高值结构，同时恢复可行性。
