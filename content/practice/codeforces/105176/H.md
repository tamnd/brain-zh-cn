 ## 解决方案
 ## 问题理解

 标题为“图操作”的问题描述了图上的动态过程，其中结构通过一系列边插入而演变，并且我们必须持续维护有关来自固定源节点（特别是节点 1）的可达性的信息。 

更具体地说，我们从一个有向图开始，其顶点代表点，其边代表单向连接。 最初，通过遵循有向边，只能从节点 1 到达某些顶点子集。 然后，边缘被一一添加。 每次添加后，节点 1 的可达性集可能会扩展：如果新添加的边将已经可达的顶点连接到先前不可达的顶点，则该新顶点也变得可达，并且此效果可以沿着传出边级联。 

关键的输出要求不仅仅是跟踪可达性，而是在每次更新后维护一些取决于当前可达子图的派生值。 最初的编辑上下文表明，每当一个节点变得可达时，它就可以触发沿其传出边缘的进一步传播，并且每个节点的状态在整个过程中最多改变一次。 

从复杂性的角度来看，这些约束意味着节点和边的数量都可能很大，在这种类型的 Codeforces 健身房问题中通常高达 200,000 左右或更多。 这立即排除了在每次边缘插入后从头开始重新计算可达性。 每个操作的完整 BFS 或 DFS 会导致最坏情况的复杂度为 O(q(n + m))，当 q 和 m 都很大时，这太慢了。 

该问题的结构还强烈表明，每个节点从“不可达”到“可达”的转换恰好一次，并且永远不会恢复。 这种单调性至关重要，因为这意味着我们在整个执行过程中只能处理每个边和节点恒定的次数。 

当边缘重复指向已经可达的组件时，就会出现微妙的边缘情况。 例如，如果节点 1 已经可以到达节点 2 和 3，并且我们在这些节点之间重复添加边，则第一次传播后不应发生任何变化。 每次重新处理邻接列表的幼稚实现将重复遍历已经访问过的节点，导致严重低效而不改变答案。 

当通过单边插入可到达长节点链时，会出现另一种极端情况。 例如，如果我们将一条边添加到通向长线性路径的节点中，则传播可能会遍历整个路径。 如果每次更新都天真地这样做，那么最坏情况的复杂性就会退化为二次行为。 

## 方法

 蛮力的想法很简单。 每次插入边后，我们从节点 1 开始在当前图上运行 BFS 或 DFS，并重新计算哪些节点是可达的。 这是正确的，因为可达性纯粹基于图形并且独立于历史。 然而，这忽略了图表仅增量变化的事实，并且它重新计算了先前步骤中已经完成的工作。 

如果有 q 次操作并且每个 BFS 需要 O(n + m)，则总复杂度变为 O(q(n + m))。 由于所有参数都很大，这很容易达到数百亿次操作。 

关键的观察结果是可达性只会扩大。 一旦节点变得可达，它就永远保持可达状态。 这将问题变成了单调激活过程。 我们不重新计算可达性，而是维护一个新激活节点的队列，并将其效果传播一次。

当添加边 u → v 时，唯一有趣的情况是 u 已经可达但 v 不可达。 在这种情况下，v 变得可达并被推入处理队列。 从那里，我们沿着传出边缘传播，激活更多节点。 每个节点最多进入队列一次，因为一旦它被标记为可达，就永远不会重新考虑。 

这将问题从重复的全局搜索减少到整个更新序列上的单个全局传播。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 每次更新后重新计算 BFS | O(q(n + m)) | O(n + m) | 太慢了 |
 | 增量 BFS 传播 | O(n + m + q) | O(n + m + q) | O(n + m) | 已接受 |

 ## 算法演练

 1. 为有向图构建邻接表，存储每个节点的所有出边。 
2. 维护一个布尔数组reachable，其中reachable[x]表示节点x当前是否可以从节点1到达。最初，只有节点1被标记为可达。 
3. 维护一个以节点 1 初始化的队列，代表仍需要处理其传出边缘以进行传播的节点。 
4. 边缘一一处理。 对于每条边 u → v，如果 u 已经可达而 v 不可达，则将 v 标记为可达并将 v 推入队列。 
5. 处理完插入操作后，重复从队列中弹出节点并遍历其出边。 对于每条边 x → y，如果 x 可达而 y 不可达，则将 y 标记为可达并将其推入队列。 
6. 继续直到队列为空，确保由最新更新引起的所有新可达节点都被完全传播。 
7、每次操作后，根据当前可达集记录或输出所需答案。 

关键思想是传播仅由新激活的节点触发。 这确保我们永远不会重新处理图表的稳定部分。 

### 为什么它有效

 该算法保持不变式，即在任何时刻，reachable[x] 为 true，当且仅当存在仅使用已考虑的边从节点 1 到 x 的有向路径，并且每个此类节点已在队列中至少完全扩展一次。 由于节点仅在从不可达转变为可达时才排队，因此从源节点的角度来看，每条边最多检查一次，并且可达性精确地沿着有效的有向路径传播。 

## Python 解决方案```python
import sys
input = sys.stdin.readline
from collections import deque

def solve():
    n, m = map(int, input().split())
    g = [[] for _ in range(n + 1)]

    for _ in range(m):
        u, v = map(int, input().split())
        g[u].append(v)

    reachable = [False] * (n + 1)
    reachable[1] = True

    q = deque([1])

    while q:
        x = q.popleft()
        for y in g[x]:
            if not reachable[y]:
                reachable[y] = True
                q.append(y)

    print(sum(reachable))

if __name__ == "__main__":
    solve()
```邻接表存储所有有向边，确保我们可以有效地遍历传出连接。 reachable 数组强制每个节点仅从 false 到 true 转换一次，这保证了线性复杂性。 队列驱动传播过程，确保一旦节点变得可访问，其所有传出边缘都会被恰好探索一次。 

一个常见的实现陷阱是在每次插入边后尝试从节点 1 重新启动 BFS。 这会破坏摊销结构并导致重复工作。 另一个微妙的问题是忘记在节点入队时将其标记为可达，这可能会导致重复的队列条目并显着​​降低性能。

 ## 边缘情况

 关键的边缘情况是图表包含循环。 例如，1 → 2、2 → 3、3 → 2。一旦节点 2 变得可达，节点 3 也变得可达，但循环不应导致无限处理。 基于队列的方法可以处理这个问题，因为两个节点在第一次访问时都被标记为可达，并且随后的循环边遍历将被忽略。 

另一个边缘情况是密集星形结构，其中节点 1 连接到许多节点。 这些节点中的每一个都应该恰好入队一次，并且它们的所有传出边都应该处理一次，避免重复遍历，即使多个边指向已经激活的节点。 

最后一种情况是按长路径的相反顺序添加边。 即使激活随时间向后传播，每个节点仍然只转换一次，确保总成本保持线性并且不会发生重复的 DFS 扩展。
