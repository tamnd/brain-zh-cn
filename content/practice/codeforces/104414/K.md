 ## 解决方案
 ## 问题理解

 我们有一条一维街道$n$位置，每个位置$i$有计划的最终高度$h_i$。 将此视为一个天际线项目，其中每个索引都是一个建筑工地，并且数字$h_i$是该地点需要多少“施工层”。 

我们被允许进行建筑作业。 每个操作选择一个段$[l, r]$并增加每个$h_i$在该部分中，完成进度为 1 个单位。 解释是，每个单位的增加代表一批工作，同时为该区间内的所有建筑物贡献一层高度。 

项目定义是动态的。 有两种类型的操作。 一项操作通过添加值来修改计划高度$k$到某个范围内的所有位置$[l, r]$，有效地增加了这些建筑物最终需要的建筑层数。 另一项行动提出了一个受限的规划问题：如果我们忽略外面的一切$[l, r]$通过将这些位置视为目标高度为零，完成所有建筑物所需的最少施工间隔操作次数是多少$[l, r]$？ 

关键输出是每个第二类查询的答案。 

限制条件$n, m \le 10^5$和高度可达$10^6$立即排除任何针对每个查询从头开始重新计算答案的解决方案。 对构建步骤的简单模拟或每个查询的重复完整扫描将无法生存。 我们需要更新和查询都是对数的或接近的。 

一个微妙的问题是，高度都会通过更新而增加，然后在子段上单独查询。 一个天真的错误是认为问题只是前缀和或范围加法查询。 第二个操作不是要求总和或最大值，而是要求实现高度轮廓所需的最小数量的区间“覆盖”操作。 

当更新严重重叠时，或者当查询隔离已间接增量多次的区域时，就会出现边缘情况。 例如，如果我们增加$[1, 5]$重复乘以1，然后查询一个子段，答案取决于高度的结构，而不仅仅是总和。 

一个揭示陷阱的小例子：

 输入：```
3 3
1 2 3
1 1 3 1
2 1 3
2 2 2
```更新后高度变为$[2,3,4]$。 查询$[2,2]$应该返回 1，而$[1,3]$是 4，因为我们需要 4 个全局单元操作。 天真的解释可能会尝试单独使用 sum 或 max，但这会失败，因为成本取决于相邻差异的结构。 

## 方法

 暴力方式是独立模拟每个查询的构造。 对于固定间隔$[l, r]$，我们想知道需要多少间隔增量才能使每个位置达到其目标高度。 每个操作都会均匀地增加一​​个连续的段，因此相当于通过跨段堆叠水平“层”来构建高度轮廓。 

如果我们从层的角度考虑，每个操作都会为连续的块贡献一个高度单位。 最小操作次数等于从左到右扫描数组时必须开始新层的总次数。 具体来说，这就是计算与之前的位置相比高度增加了多少倍。 

因此对于固定数组，答案为$[l, r]$是：$$h[l] + \sum_{i=l+1}^{r} \max(0, h[i] - h[i-1])$$这是一个已知的特征：每次增加都代表一个新的“堆栈开始”。 

暴力计算每个查询的总和$O(n)$，并且更新需要$O(n)$如果直接应用。 和$m = 10^5$，最坏情况的复杂度变为$10^{10}$，这太慢了。 

关键的见解是答案仅取决于相邻差异$d_i = \max(0, h_i - h_{i-1})$。 范围更新于$h$转化为这些差异的结构化更新。 每次更新仅以受控方式影响边界位置。 这将问题转化为通过范围加法和求和查询来维护差异的动态数组。 

我们维持：$$d_i = \max(0, h_i - h_{i-1})$$并注意查询的答案$[l, r]$变成：$$h[l] + \sum_{i=l+1}^{r} d_i$$所以我们需要：

 1. 范围附加$h$2. 维护$d$3.查询前缀调整后的总和$d$这是通过支持范围添加和点查询的线段树或芬威克树来处理的$h$，加上第二个结构用于维护$d$，仅在更改边界处更新。 

每个范围添加$[l, r, k]$仅更改两个比较：

 - 在$l$：$h_l$增加，影响$d_l$- 在$r+1$：$h_{r+1}$比较变化相对于$h_r$因此更新是$O(\log n)$, 还查询$O(\log n)$。

 ## 算法演练

 我们维护两个线段树（或 BIT 结构）。 1.存储数组$h$在范围添加和点查询下。 第二个存储派生结构$d_i = \max(0, h_i - h_{i-1})$通过点更新。 

### 步骤

 1. 构建支持范围添加和点查询的结构$h_i$。 

这使我们可以立即检索任何当前高度，这是更新后重新计算局部差异所必需的。 
2. 构建数组$d$最初：$d_i = \max(0, h_i - h_{i-1})$。 

我们还维护一个前缀和结构$d$。 

这是因为查询答案减少为前缀和$d$。 
3. 对于类型 1 更新$[l, r, k]$，将范围加法应用到$h$。 

的高度在$[l, r]$变化，但只有邻接边界才重要$d$。 
4. 重新计算受影响的差异：

 仅职位$l$和$r+1$可以改变比较结果。 

我们更新：$d_l = \max(0, h_l - h_{l-1})$

$d_{r+1} = \max(0, h_{r+1} - h_r)$如果$r+1 \le n$这是有效的，因为所有内部差异统一移动，保持相对顺序。 
5. 对于类型 2 查询$[l, r]$，计算：$h[l] + \sum_{i=l+1}^{r} d_i$。 

这是通过点查询获得的$h[l]$和前缀范围总和$d$。 

### 为什么它有效

 施工成本解释仅取决于从左向右移动时高度序列增加的位置。 每次增加都会在任何有效的施工计划中强制增加一个新层。 由于范围添加保留了除边界之外的相对差异，因此除了更新边缘之外，增加的结构是稳定的。 这将全局问题简化为跟踪相邻比较中的一小部分局部变化。 不变的是$d_i$始终等于当前相邻差值的正部分，因此前缀和$d$准确计算层开始的数量。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class BIT:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 2)

    def add(self, i, v):
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

class RangeAddPointQuery:
    def __init__(self, n):
        self.n = n
        self.bit = BIT(n)

    def range_add(self, l, r, v):
        self.bit.add(l, v)
        if r + 1 <= self.n:
            self.bit.add(r + 1, -v)

    def get(self, i):
        return self.bit.sum(i)

class Fenwick:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 2)

    def add(self, i, v):
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

n, m = map(int, input().split())
h0 = list(map(int, input().split()))

rapq = RangeAddPointQuery(n)
for i, v in enumerate(h0, 1):
    rapq.range_add(i, i, v)

d = Fenwick(n)

def get(i):
    if i == 1:
        return rapq.get(1)
    return max(0, rapq.get(i) - rapq.get(i - 1))

for i in range(1, n + 1):
    d.add(i, get(i))

def update(i):
    if 1 <= i <= n:
        if i == 1:
            new = rapq.get(1)
            prev = 0
        else:
            new = rapq.get(i)
            prev = rapq.get(i - 1)
        val = max(0, new - prev)
        cur = d.sum(i) - d.sum(i - 1)
        d.add(i, val - cur)

for _ in range(m):
    tmp = input().split()
    if tmp[0] == '1':
        _, l, r, k = map(int, tmp)
        rapq.range_add(l, r, k)
        update(l)
        update(r + 1)
    else:
        _, l, r = map(int, tmp)
        res = rapq.get(l)
        res += d.sum(r) - d.sum(l)
        print(res)
```该实现将高度维护与差异维护分开。 第一个结构支持快速范围增量和点查询，这避免了重建整个数组。 第二个结构维护驱动查询答案的压缩“增量”表示。 

更新函数被小心地限制为仅重新计算边界位置。 一个常见的错误是尝试更新中的所有索引$[l, r]$，这会太慢。 正确性依赖于内部差异值保持不变的事实，因为每个内部边缘的两个端点均等地移动。 

查询使用的点值位于$l$加上差异的前缀和$l+1$到$r$，匹配重构公式。

 ## 边缘情况

 一个微妙的边缘情况是更新触及数组的边界。 例如，如果我们应用$[1, n, k]$， 仅有的$d_1$有意义的改变，因为没有$d_{n+1}$，并且算法必须避免访问超出范围的索引。 该实现在更新时显式检查边界$r+1$，防止 Fenwick 树腐败。 

另一种情况是重复重叠更新。 由于高度结构是通过基于差异的 Fenwick 树来维护的，因此重叠的添加可以正确累积，无需重新计算。 每次更新都会线性地影响内部 BIT 状态，并且边界重新计算可确保派生差异的正确性。 

最后一个案例是单点查询$[l, l]$。 公式简化为返回$h[l]$，因为没有邻接贡献。 该实现自然地处理这个问题，因为前缀差和变为零。
