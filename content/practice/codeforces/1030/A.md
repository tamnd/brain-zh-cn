[CF 1030A - 寻找一个简单的问题](https://codeforces.com/problemset/problem/1030/A)

 **标签：** 实施
 **求解时间：** 3m 6s
 **已验证：** 是的

 ## 解决方案
 ## 问题理解

 我们有一小群人，每个人都对一个问题给出二元意见。 每个回答要么是 0，表示该人认为问题简单，要么是 1，表示该人认为问题很难。 协调员的规则很简单：只有当每个人都认为很容易时，问题才会保持原样。 如果连一个人都说很难，那么这个问题就被拒绝了。 

任务是确定答复中是否出现任何“强硬”意见。 如果列表中至少有一个值为 1，则答案将变为“HARD”。 否则，如果所有值均为 0，则答案为“EASY”。 

n 的限制最多为 100，这意味着任何线性扫描都非常快。 即使检查所有子集或重新计算多个通道在性能方面也是可以接受的，但没有必要。 问题的结构已经表明单遍就足够了。 

不存在涉及排序或算术的棘手的隐藏边缘情况。 唯一有意义的边缘情况是 n 等于 1 时，因为决策完全取决于单个值。 另一种情况是当所有值都为 0 时，应该正确返回“EASY”。 一个常见的错误是错误地假设 0 的存在意味着简单，而不验证 1 是否存在。 

第二个微妙的问题是输出格式。 接受任何大小写变体，但无论格式选择如何，逻辑条件仍然必须正确。 

## 方法

 暴力方法只需扫描列表并明确计算有多少人说“很难”。 如果计数大于零，我们打印“HARD”，否则打印“EASY”。 这在结构上已经是最优的，但也可以说是重复检查每个元素并累积一个结果。 

在最坏的情况下，我们检查所有 n 值，对每个元素执行恒定的工作。 由于 n 最多为 100，因此即使更复杂的方法也没关系，但关键的观察是我们不需要处理元素之间的关系，只需检测单个值的存在。 

我们的见解是，这是一个存在性检查问题，而不是一个聚合问题。 我们不需要 1 的总数，只需要是否至少存在一个。 这样我们一找到单个 1 就可以提前终止。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力（全部计数）| O(n) | O(1) | O(1) | 已接受 |
 | 最佳（提前停止检查）| O(n) | O(1) | O(1) | 已接受 |

 ## 算法演练

 ### 最佳方法

 1. 读取整数n，它告诉我们将处理多少个意见。 这定义了我们将执行的迭代次数。 
2. 读取代表意见的n 个整数的列表。 每个值不是 0 就是 1，因此我们只关心检测 1 是否存在。 
3.初始化一个flag变量，例如`has_hard = False`。 该变量跟踪到目前为止我们是否至少看到了一个强硬的意见。 
4. 迭代列表中的每个值。 对于每个值，检查它是否等于 1。 
5. 如果某个值等于 1，则设置`has_hard = True`。 此时，我们已经知道最终答案将是“困难”，因此我们可以选择提前停止。 
6. 处理完所有值后（或提前停止），检查标志。 如果`has_hard`为 True，输出“HARD”。 否则输出“EASY”。 

### 为什么它有效

 该决定仅取决于是否存在至少一个等于 1 的元素。该标志保持不变式，即在处理数组的任何前缀后，它正确反映在该前缀中是否看到 1。 由于最终的数组只是完整的前缀，因此该标志正确地表示输入中是否存在任何硬响应。 这保证了正确性，而不需要任何额外的结构或计算。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

n = int(input().strip())
arr = list(map(int, input().split()))

has_hard = False

for x in arr:
    if x == 1:
        has_hard = True
        break

if has_hard:
    print("HARD")
else:
    print("EASY")
```该解决方案以线性时间读取输入并将响应存储在列表中。 然后它扫描列表一次，当遇到 1 时立即停止。提前中断对于正确性来说不是必需的，但反映了问题的存在检查性质并避免了不必要的迭代。 

关键的实现细节是布尔标志。 它避免了计数，直接对我们关心的条件进行编码。 另一个微妙之处是正确处理输入分割，因为所有响应都在一行上给出。

 ## 边缘情况

 最重要的边缘情况是 n 等于 1 时。对于输入`1 0`，循环处理单个元素并且从不设置标志，因此输出正确为“EASY”。 用于输入`1 1`，标志立即设置并且输出变为“HARD”。 这证实了即使没有“批量”输入结构，算法也能正确运行。 

另一种边缘情况是所有值均为 0 时。例如，`4 0 0 0 0`在整个迭代过程中保持标志不变，最终决策正确输出“EASY”。 这表明缺席检测处理得当。 

最后，有多个1的情况，例如`5 0 1 0 1 0`，测试算法是否错误地依赖于计数。 由于标志设置在第一个 1 上并且从未重置，因此最终结果仍然是“HARD”，与出现多少个附加 1 无关。
