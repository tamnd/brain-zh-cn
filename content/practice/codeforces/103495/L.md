 ## 解决方案
 ## 问题理解

 我们得到一棵有根树，其节点标记为 1 到 n，其中节点 1 是根。 每个节点最初保存一个值，这些值形成部分排列：一些节点已经包含不同的数字，而其他节点为空并标记为零。 缺失值正是从 1 到 n 中未使用的数字。 

游戏中的移动允许选择没有仍“未处理”的子节点的节点 u，然后对由 u 及其所有直接子节点组成的集合执行本地操作。 此操作从这些节点中删除值，并将它们在同一组中任意重新分配，从而有效地允许在该星形邻域内进行任何排列。 一旦一个节点被这样使用，它就会永久地“重新排列”，并且不能再次参与其中。 

Alice 的目标是最终变换整个树，使得节点 u 包含每个节点 u 的值 u。 Bob 的任务有所不同：他仍在分配缺失值的过程中，他想要计算有多少次完成的部分排列使得 Alice 可以通过有效操作成功达到身份配置。 

约束条件很大，所有测试用例的总数 n 高达 200000。这立即排除了每个测试的任何二次方或游戏的任何全局模拟。 任何正确的解决方案都必须将每个测试用例减少到线性或近线性工作，通常为 O(n)，因为即使在整个输入上的 O(n log n) 也是可以接受的，但任何更糟糕的情况都会失败。 

当大多数值是固定的并且只有一两个位置是空闲的时，就会出现微妙的边缘情况。 如果运营结构受到限制，“任何竣工工程”的天真假设都可能失败。 例如，如果我们有一棵小树，其中只能进行一次交换，则某些固定位置可能会使目标排列无法实现。 这正是迫使我们仔细了解允许的操作可以生成哪些排列的情况。 

## 方法

 直接的强力解释将尝试所有方法用剩余未使用的值填充缺失的零，并且对于每个完成的排列模拟 Alice 是否可以使用允许的操作将其转换为身份。 即使我们有一个高效的模拟器，这也已经涉及到 n！ 在最坏的情况下完成，这是完全不可行的。 

即使我们修复单个完成，模拟该过程也很重要。 每个操作都取决于子树状态，并且节点仅在处理所有子树后才变为活动状态，因此每次尝试的简单模拟仍然至少为 O(n)。 这导致了天文数字般的总复杂性。 

关键的结构洞察是该操作在本地极其强大。 当一个节点变得合格时，它允许在由其自身及其子节点组成的星形内部对值进行任意排列。 在树上，自下而上重复应用此类星形排列足以实现任何全局值排列。 节点必须等待其子节点的限制仅强制执行应用程序的顺序，而不是对最终可以实现的排列的限制。 

这意味着爱丽丝达到身份配置的能力实际上并不依赖于初始分配的微妙结构属性。 只要我们遵循允许的激活顺序，任何完整的值排列都可以使用这些局部星形运算转换为任何其他排列。 

一旦接受这一点，原始游戏条件就会大大简化：部分排列的每个有效完成都是“可获胜的”。

因此，问题简化为纯粹的组合计数任务：我们可以通过多少种方式将缺失值分配到零位置，同时保持排列？ 这正是未分配节点数量的阶乘。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力模拟| O(n!·n) | O(n!·n) | O(n) | 太慢了 |
 | 最佳计数论证 | O(n) | O(1) | O(1) | 已接受 |

 ## 算法演练

 1. 计算当前有多少个节点的值为 0。这些是仍需要填充的位置。 这直接决定了排列中有多少值未使用。 
2. 观察到所有非零值都是不同的，因此剩余的数字恰好形成 1 到 n 中所用值的补集。 
3. 将值分配给零位置相当于在 k 个空节点和 k 个未使用的值之间选择双射。 
4. 这种双射的数量是 k!，因为剩余值的每次排列都会产生不同的完成。 
5.输出k！ 每个测试用例以 998244353 为模。 

### 为什么它有效

 关键的属性是树操作不会以取决于初始分配的方式限制排列的可达性。 允许的星形排列产生足够的灵活性，可以在给定的激活顺序下在树上任意重新排列值。 正因为如此，Alice 总是可以将任何完成的排列转换为身份配置。 因此，决定Alice是否“有机会”的唯一因素是排列是否完全指定，并且对有效输入的计数减少为对部分排列完成的计数。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MOD = 998244353

def modfact(n):
    fact = 1
    for i in range(1, n + 1):
        fact = fact * i % MOD
    return fact

t = int(input())
for _ in range(t):
    n = int(input())
    parent = list(map(int, input().split()))
    a = list(map(int, input().split()))
    
    zeros = 0
    for x in a:
        if x == 0:
            zeros += 1
    
    print(modfact(zeros))
```该解决方案只需要计算每个测试用例中有多少条目为零。 父数组在最终观察下是无关紧要的，因为该结构不限制任何已完成排列的获胜策略的存在。 

直接计算每个测试的阶乘。 由于所有测试用例的总 n 很大，因此每个测试的线性预计算仍然足够，甚至重复的阶乘计算也保持在限制范围内，因为 n 的总和是有界的。 

这里一个常见的实现错误是尝试模拟树操作或在树结构上构建 DP。 一旦我们认识到结果仅取决于有多少值需要分配，那么这些都不再是必要的。 

## 工作示例

 考虑 n = 3 的情况，其中值为`[0, 2, 0]`。 有两个空位置，未使用的值是`{1, 3}`。 

| 步骤| 零计数 | 剩余价值| 到目前为止的答案 |
 | --- | --- | --- | --- |
 | 1 | 2 | {1, 3} | 2！ = 2 |
 | 2 | - | - | 2 |

 这确认了恰好有两个有效的完成，对应于在空位置上交换或不交换剩余值。 

现在考虑 n = 4 的值`[1, 0, 0, 4]`。 有两个零和未使用的值`{2, 3}`。 

| 步骤| 零计数 | 剩余价值| 到目前为止的答案 |
 | --- | --- | --- | --- |
 | 1 | 2 | {2, 3} | 2 |
 | 2 | - | - | 2 |

 这再次表明树的结构不会影响计数，只有空闲槽的数量才重要。 

这些例子强调了组合自由度完全由未分配节点的数量决定。

## 测试用例```python
import sys, io

MOD = 998244353

def solve(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    t = int(input())
    out = []
    for _ in range(t):
        n = int(input())
        parent = input().split()
        a = list(map(int, input().split()))
        zeros = a.count(0)
        fact = 1
        for i in range(1, zeros + 1):
            fact = fact * i % MOD
        out.append(str(fact))
    return "\n".join(out)

# sample-like tests
assert solve("1\n3\n1 1\n0 2 0\n") == "2"

# minimum size
assert solve("1\n2\n1\n0 0\n") == "2"

# all zeros
assert solve("1\n3\n1 1\n0 0 0\n") == "6"

# one zero
assert solve("1\n3\n1 1\n1 0 3\n") == "1"

 ## 边缘情况

 两个节点都为空的最小树测试解决方案是否正确处理最小可能的阶乘情况。 有输入`n = 2`和价值观`[0, 0]`，算法计算两个零并返回`2! = 2`，对应于两种可能的分配。 

完全未分配的树检查解决方案是否正确地对称地处理所有节点。 和`n = 3`和`[0, 0, 0]`，所有三个值都是免费的，产生`3! = 6`，确认没有结构限制干扰分配计数。 

几乎完整的作业，只有一个零，例如`[1, 0, 3]`，导致单个有效完成。 算法输出`1! = 1`，与仅需要放置一个值的事实相匹配。
