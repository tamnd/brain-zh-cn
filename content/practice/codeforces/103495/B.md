 ## 解决方案
 ## 问题理解

 我们得到了一张由加权无向路径连接的房间地图。 两个冒名顶替者从固定的房间开始，只能沿着这些加权的秘密路径移动。 时间是连续的，沿着边缘移动的重量以秒为单位，因此冒名顶替者实际上是在标准的最短路径度量上行进。 

最多有八名船员。 每个船员都有一份预计出现的列表，这意味着在某些时间他们将出现在特定的房间中。 如果在某个确切的时间，冒名顶替者与船员出现在同一个房间，则该船员会立即被淘汰。 

冒名顶替者想要最小化所有船员被杀死的时间，或者确定不可能保证在允许的时间范围 tmax 内所有杀死。 

关键的困难在于杀戮并不是要到达一组静态节点。 它们是关于将多个代理之间的行程时间与预定事件同步，而只有两个移动者存在并且它们共享相同的图形约束。 

这些约束强烈地塑造了解决方案。 该图最多有 10^4 个节点和 2×10^4 条边，因此每个源的最短路径计算必须接近线性。 船员数量很少，最多 8 人，这立即暗示了位掩码状态空间。 事件数量最多为 10^5，因此任何针对每个事件的昂贵模拟都是不安全的。 

天真的解释可能会尝试模拟冒名顶替者的移动，同时跟踪与所有船员的所有可能的会面时间，但这种解释会爆炸，因为每个冒名顶替者的选择都会在图形路径和事件时间上分支。 

当多名船员需要协调访问时，就会出现更微妙的边缘情况。 例如，如果船员 A 出现在（房间 1，时间 5），船员 B 出现在（房间 2，时间 6），但到达这两个地方需要不同的冒名顶替者或仔细划分路线，则天真的贪婪分配可能会失败。 另一种失败模式是假设一名船员“到达后就被杀死”，而忽略了冒名顶替者必须恰好在活动时间到达。 

第三个陷阱是将其视为从每个冒名顶替者开始到每个事件的独立单源最短路径。 这忽略了属于不同船员的事件之间的相互作用，以及在共享时间线中涵盖所有事件的需要。 

## 方法

 蛮力观点试图将每个冒名顶替者视为一个连续的旅行者，可以选择路径并决定接下来参加哪个活动。 对于每个可能的事件分配给冒名顶替者以及每个冒充者的事件排序，我们将通过最短路径时序检查来计算可行性。 这在概念上是正确的，但立即组合起来：对于最多 10^5 个事件和最多 8 个船员，即使对每个船员进行分组事件也会导致其外观上的阶乘行为以及两个冒名顶替者之间的指数分裂。 

关键的观察是船员很少，所以我们不应该直接推理个别事件。 相反，我们将所有预测压缩为“如果冒名顶替者在时间 t 在 x 房间，何时可以杀死船员 p”的概念。 对于每个船员 p，我们只关心在给定时间冒名顶替者是否可以在所需的房间中。 这表明我们应该使用最短路径预先计算房间之间的最早到达时间。 

现在结构变得更加清晰：每个冒充者开始都定义了房间上的距离函数。 如果冒名顶替者能够在指定时间之前到达活动室，则击杀是可行的。 由于允许等待，因此只有不等式 dist(start, x) ≤ t 重要。 

因此，对于每个冒名顶替者的开始，我们计算图表上的最短路径。 然后对于每个事件（p，x，t），我们检查哪些冒名顶替者可以满足它。 这将每个船员变成一组带有时间戳的候选击杀机会，这些机会必须全部由冒名顶替者 1 或冒名顶替者 2 覆盖。

现在剩下的问题是分配：每个船员最多有许多事件，但由于 k ≤ 8，我们可以将每个船员压缩成一个位掩码，其中冒名顶替者负责其所有事件。 对于固定分配掩码，可行性降低为根据所选的冒名顶替者距离检查所有事件。 

最后，我们对答案时间 T 进行二分搜索。对于固定的 T，我们忽略 t > T 的事件，并检查是否存在将船员分配给冒名顶替者的情况，以便覆盖每个事件。 由于 k ≤ 8，我们可以强制所有 2^k 分配并使用预先计算的距离在 O(e) 中验证每个分配。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力事件调度| e 中的指数 | O(e) | 太慢了 |
 | 最短路径+位掩码分配+二分查找| O((n+m) log n + 2^k · e · log tmax) | O(n + e) | 已接受 |

 ## 算法演练

 我们首先使用 Dijkstra 算法计算距两个冒名顶替者起始房间的最短路径距离。 这给出了所有房间 x 的 dist1[x] 和 dist2[x]。 

然后，我们对所有预测事件进行预处理，并按机组成员对它们进行逻辑分组，但在测试可行性时，我们仍将对其进行全局评估。 

接下来我们二分搜索可以消除所有船员的最小时间 T。 下限为 0，上限为 tmax，因为在 tmax 之后，如果有人幸存，游戏就会失败。 

对于固定的候选时间 T，我们只过滤那些 t ≤ T 的事件，因为后面的事件对于在时间 T 之前实现消除并不重要。 

我们现在尝试将所有 2^k 名船员分配给冒名顶替者。 在一次分配中，每个船员 p 都被分配为冒名顶替者 0 或冒名顶替者 1。 

对于给定的作业，我们检查每个事件 (p, x, t)。 如果 p 被分配给 impostor 0，我们要求 dist1[x] ≤ t，否则我们要求 dist2[x] ≤ t。 如果任何事件失败，则分配无效。 

如果至少有一项分配有效，则时间 T 是可行的。 

我们输出最小的可行 T，如果 T ≤ tmax 不成立，则输出 -1。 

为什么这种方法有效与图距离创建的独立性有关。 一旦距离固定，每个事件就成为一个局部约束，将单个冒名顶替者的可达性与时间戳进行比较。 唯一的全局耦合是哪个冒名顶替者处理哪个船员，并且该耦合在 k ≤ 8 上由位掩码完全捕获。不需要事件排序，因为等待消除了排序约束，并且一旦分配固定，每个事件都是独立的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

import heapq

INF = 10**30

def dijkstra(n, adj, src):
    dist = [INF] * (n + 1)
    dist[src] = 0
    pq = [(0, src)]
    while pq:
        d, u = heapq.heappop(pq)
        if d != dist[u]:
            continue
        for v, w in adj[u]:
            nd = d + w
            if nd < dist[v]:
                dist[v] = nd
                heapq.heappush(pq, (nd, v))
    return dist

def solve():
    T = int(input())
    for _ in range(T):
        n, m, k = map(int, input().split())
        adj = [[] for _ in range(n + 1)]
        for _ in range(m):
            u, v, w = map(int, input().split())
            adj[u].append((v, w))
            adj[v].append((u, w))

        e, tmax = map(int, input().split())
        events = []
        for _ in range(e):
            p, x, t = map(int, input().split())
            events.append((p - 1, x, t))

        s1, s2 = map(int, input().split())

        dist1 = dijkstra(n, adj, s1)
        dist2 = dijkstra(n, adj, s2)

        lo, hi = 0, tmax
        ans = -1

        def feasible(Tlim):
            # try all assignments of k crewmates
            for mask in range(1 << k):
                ok = True
                for p, x, t in events:
                    if t > Tlim:
                        continue
                    if (mask >> p) & 1:
                        if dist2[x] > t:
                            ok = False
                            break
                    else:
                        if dist1[x] > t:
                            ok = False
                            break
                if ok:
                    return True
            return False

        if feasible(tmax):
            lo, hi = 0, tmax
            while lo < hi:
                mid = (lo + hi) // 2
                if feasible(mid):
                    hi = mid
                else:
                    lo = mid + 1
            ans = lo

        print(ans)

if __name__ == "__main__":
    solve()
```Dijkstra 在每次冒名顶替者启动时运行一次，捕获所有移动限制。 可行性函数是核心简化：它不是模拟运动，而是将问题简化为检查每个事件是否位于所选冒名顶替者当时的范围内。 

位掩码循环枚举了所有船员对冒名顶替者的分配。 这是有效的，因为 k ≤ 8，因此即使乘以最多 10^5 个事件，256 个配置也足够小。 

使用二分搜索是因为可行性在时间上是单调的：如果所有杀戮都可以在时间 T 之前完成，那么任何更大的时间 T' 也可以工作，因为事件只会变得更容易满足。 

## 工作示例

 考虑一个最小的场景，其中两个房间直接相连，一名船员具有两种外观。 每个房间都有一名冒名顶替者。 第一次出现较早出现在 A 房间，第二次出现较晚出现在 B 房间。该算法计算这两个距离，然后测试分配。 其中一项任务将船员分配给冒名顶替者 1，但由于距离原因，第二项任务失败。 如果冒充者 2 可以及时到达两个事件，则另一个分配会成功，这说明了为什么分配灵活性很重要。 

现在考虑这样一种情况：在时间戳之前，一名船员出现在距离两个冒名顶替者太远的房间里。 对于该事件，dist1[x] 和 dist2[x] 都超过 t，因此每个分配都会立即失败。 可行性检查对所有 T 都返回 false，最终答案变为 -1。 

这两条轨迹表明，正确性完全取决于将最短路径可达性与事件截止日期进行比较，而分配则处理唯一的组合自由度。 

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 | O((n + m) log n + 2^k · e · log tmax) | Dijkstra 两次加上对可行性检查的二分搜索，每次检查所有掩码的所有事件 |
 | 空间| O(n + e) | 图形存储、距离数组和事件列表 |

 约束的设计使 k 保持很小，使指数因子无害，而图的大小需要有效的最短路径计算。 该解决方案完全符合限制，因为 2^8 只有 256，并且 Dijkstra 在每个测试用例中占主导地位。

 ## 边缘情况

 一个关键的边缘情况是多个事件在同一房间但不同时间发生。 该算法自然地处理这个问题，因为每个事件都是根据相同的距离约束独立检查的，因此较早的事件不会错误地阻止后面的事件。 

另一种情况是当一个冒名顶替者被严格控制时，这意味着它的距离阵列对于所有房间来说都更差。 位掩码枚举仍然有效，因为它会简单地将所有船员分配给更好的冒名顶替者，而无需特殊的外壳。 

最后的边缘情况是所有事件都在 tmax 之后发生。 在这种情况下，最佳答案通常为 0，因为在故障截止时间之前没有任何约束处于活动状态，并且二分搜索在 T = 0 时立即崩溃为可行性。
