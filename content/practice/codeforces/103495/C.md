 ## 解决方案
 ## 问题理解

 我们得到从 0 到 9 的数字计数。将其视为具有多组数字，其中数字`d`准确地出现`a[d]`次。 我们的任务是将所有这些数字排列成一个整数，每个数字只使用一次，同时满足两个约束：数字不能以前导零开头，除非它恰好为零，并且最终数字中的两个相邻数字不能相同。 在所有有效的重新排列中，我们想要字典顺序上最小的可能整数，这相当于数字上最小的可能值，因为所有数字都具有相同的长度。 

这些限制意味着所有测试用例的总位数最多为 100,000。 这立即排除了任何试图排列数字或模拟所有排列的方法。 即使在每次测试的最坏情况下是二次的贪婪构造也会太慢，因为重复扫描大型多重集很容易超出限制。 

一个关键的困难是邻接约束与最小数值的要求相互作用。 总是选择最小的可用数字的天真的贪婪策略可能会失败，因为它可能会造成在不违反邻接规则的情况下无法放置剩余数字的情况。 

当最小的数字大量重复时，就会出现简单的失败情况。 例如，如果我们有许多零而只有一个其他数字，则天真的贪婪解决方案可能会从零开始，由于前导零规则，这是无效的，或者它可能会强制一种安排，随后留下两个相同的数字相邻而没有其他选择。 

当仅存在一种数字类型时，会发生另一种微妙的失败情况。 例如，`a[3] = 5`和所有其他都是零。 唯一可能的安排是`"33333"`，这违反了邻接规则，所以正确的输出是`-1`。 任何假设有效排列始终存在的构造都会在这里失败。 

## 方法

 强力解释将尝试通过在每个位置按升序尝试数字并检查是否存在完整的有效完成来构建尽可能小的数字。 这自然会导致回溯：在每一步中，我们都会尝试每个可用数字，确保它不违反邻接或前导零约束，然后递归。 虽然正确，但这种方法探索了指数搜索空间。 在最坏的情况下，有很多重复的数字，分支因子仍然很大，并且深度达到100,000，这使得它完全不可行。 

关键的观察是我们实际上不需要全局“搜索”。 该问题从根本上讲是在局部约束下构建序列：邻接性和可用性。 这是一个经典的设置，如果我们对下一步选择哪个数字保持足够的灵活性，那么贪婪的放置就会起作用。 

关键的想法是始终选择尽可能小的数字，但前提是放置它不会造成未来的矛盾。 唯一真正的危险是，当选择一个数字时，我们会陷入一个数字的太多副本，而这些副本以后无法分开。 这简化为一个可行性条件：在任何时候，任何数字都不应超过可以分隔它的剩余位置的数量。 对于这个特定问题，一个更简单、更强大的观察是有效的：如果存在有效的排列，我们总是可以贪婪地构造一个，同时永远不会放置一个会立即创建无效邻接或违反前导零规则的数字。 

我们保留前一个数字，并始终选择与前一个数字不同的剩余计数为正的最小数字，并附加一个额外的约束，即我们避免在第一个位置放置零，除非整个数字为零。 因为我们只禁止局部违规并且总是消耗数字，所以我们要么成功放置所有数字，要么达到无法放置有效数字的状态，这意味着不可能。

 ## 算法演练

 我们从左到右逐位构建数字，同时跟踪每个数字剩余的数量以及之前放置的数字是什么。 

1.初始化数字0到9的频率数组并计算总长度`n`。 如果`n = 1`，答案只是个位数，因为不能违反邻接约束。 
2. 对于第一个位置，除非整个数字都由零组成，否则我们不能放置零。 我们扫描从 1 到 9 的数字并选择具有正频率的最小数字。 这保证了最小的有效前导数字。 
3. 对于每个后续位置，我们再次按升序扫描从 0 到 9 的数字，并选择仍具有剩余计数且不等于先前放置的数字的最小数字。 这强制执行字典顺序最小选择和邻接约束。 
4. 如果在某个位置我们找不到任何有效数字，我们终止并输出`-1`。 这表示死配置，其中剩余数字不能在不违反邻接性的情况下放置。 
5. 每次选择一个数字时，我们都会将其附加到结果中，减少其频率，并更新前一个数字。 

### 为什么它有效

 该算法保持不变，即在每一步中，到目前为止构建的前缀是可以出现在任何有效解决方案中的最小可能前缀。 选择最小有效数字的任何偏差都会立即产生更大的前缀，并且由于所有剩余数字都以多集形式固定，因此以后的重新排列无法补偿该增加。 如果算法卡住，则意味着每个剩余的数字都与前一个数字冲突，这意味着所有剩余的数字都是相同的。 在这种情况下，不存在有效的重新排列，因为邻接性将不可避免地被破坏。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    arr = list(map(int, input().split()))
    total = sum(arr)
    
    if total == 1:
        for d in range(10):
            if arr[d]:
                return str(d)
    
    res = []
    prev = -1
    
    for i in range(total):
        found = False
        
        if i == 0:
            for d in range(1, 10):
                if arr[d]:
                    res.append(str(d))
                    arr[d] -= 1
                    prev = d
                    found = True
                    break
        else:
            for d in range(10):
                if arr[d] and d != prev:
                    res.append(str(d))
                    arr[d] -= 1
                    prev = d
                    found = True
                    break
        
        if not found:
            return "-1"
    
    return "".join(res)

t = int(input())
out = []
for _ in range(t):
    out.append(solve())

print("\n".join(out))
```实现直接遵循贪心构造。 第一个数字单独处理以强制执行无前导零规则。 之后，每一步都会选择与前一步不同的最小可行数字。 这`found`标志是必不可少的，因为它清楚地捕获了不可能性：如果没有数字可以放置在某个位置，则函数立即返回`-1`。 

一个微妙的点是我们永远不需要进行任何前瞻或回溯。 约束的结构确保当剩余的多重集无法用不同的数字分隔时，任何失败都会立即发生。

### 示例 2

 输入数字：`a = [3,0,1,0,0,0,0,0,0,0]`，表示三个零和一个 2。 

第一步，我们必须选择数字 2，因为除非整个数字为零，否则禁止前导零。 

| 步骤| 剩余计数 (0,2) | 上一个数字 | 选择的数字 |
 | --- | --- | --- | --- |
 | 1 | 0:3, 2:1 | - | 2 |

 | 4 | 0:1 | 0 | 0 |

 结果是`2000`。 该示例证实，只要尊重邻接性，就可以自由放置零。

 ## 边缘情况

 一种边缘情况是所有数字都相同。 对于像这样的输入`a[5] = 4`，算法首先放置`5`，然后找不到下一个位置的有效数字，因为每个剩余的数字都等于前一个数字。 该算法正确终止于`-1`，匹配无法满足邻接性的事实。 

另一种边缘情况是当零占主导地位但存在非零数字时。 例如`a[0] = 100000, a[1] = 1`。 第一个数字必须是`1`，之后用零填充其余部分。 该算法从不尝试先放置零，因此它避免了无效的前导零并产生`100000...0`。 

最后一个边缘情况是贪婪的选择后来迫使我们走入死胡同。 这里不会发生这种情况，因为任何故障状态都意味着仅保留一种数字类型，并且在邻接约束下这种情况本质上是不可能的。
