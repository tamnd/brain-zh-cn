 ## 解决方案
 ## 问题理解

 给定一个正整数序列，并要求我们构造两个不同的索引子集，使得第一个子集选择的值之和等于第二子集选择的值之和。 每个子集只是数组中的一组位置，因此我们不允许在子集中重复使用相同的索引，但两个子集可能会任意重叠。 

输出需要两个这样的索引集或一个不存在有效对的声明。 

限制非常大：每个测试最多 100,000 个元素和多个测试用例。 任何显式枚举子集甚至尝试对所有可能和进行经典动态规划的解决方案都是立即不可能的，因为子集的数量和可能和的范围都远远超出了可行的限制。 朴素的子集和 DP 需要跟踪总和大约为$2 \cdot 10^{12}$，这在时间和记忆上都已经是不可行的。 

一个更微妙的困难是，除非我们利用输入的结构属性，否则不能保证任意数据的碰撞。 在某些序列中，每个子集总和都是唯一的，例如，当每个值严格大于所有先前值的总和时。 在这种情况下，没有两个不同的子集可以产生相同的和，正确的答案是“-1”。 

此失败案例的一个小例子是：

 输入：```
n = 4
v = [1, 2, 4, 8]
```每个子集总和唯一对应于二进制表示，因此没有两个子集共享总和。 任何假设冲突始终存在的方法都会错误地尝试在此处构造一对。 

另一个边缘情况是所有值都相等时：

 输入：```
n = 3
v = [5, 5, 5]
```在这里，存在许多有效的答案，例如`{1}`和`{2}`，因为两者之和等于 5。正确的解决方案应该快速检测到这种微不足道的结构，而不是依赖重型机械。 

## 方法

 该问题从根本上要求子集和中发生冲突。 如果我们以最直接的方式思考，我们正在处理最多$2^n$子集，每个子​​集产生一个总和。 暴力方法会尝试所有子集，计算它们的总和，并将它们存储在哈希表中。 一旦相同的和以不同的子集出现两次，我们就完成了。 

这种方法是正确的，因为它显式地构造所有可能的候选者并检查相等性。 然而，在实践中它立即失败了。 即使是为了$n = 40$，子集的数量已经是一万亿左右，这里$n$达到十万，就完全不可能了。 

关键的观察是我们实际上不需要探索所有子集。 我们只需要检测从子集到总和的诱导映射中的任何冲突。 这将问题变成了经典的“找到两个具有相同权重的不同组合对象”场景，其中随机构造与增量跟踪相结合就足够了。 

我们不是枚举所有子集，而是逐步构建子集并维护从实现的总和到产生它们的子集的哈希映射。 每次我们合并一个新元素时，每个现有子集都可以包含它或不包含它，但显式地加倍状态是不可能的。 诀窍是保持表示压缩：我们只维护一组仔细控制的候选子集​​，依赖于这样一个事实：一旦跟踪状态的数量超过不同总和的范围，我们可以安全地预期发生碰撞，根据鸽巢原理，必须出现重复的总和。 

为了使其实用，我们维护一个从总和值到生成它的实际子集的映射，并以受控方式迭代扩展我们的子集工作集。 当检测到碰撞时，我们立即重建两个子集。 

蛮力之所以有效，是因为它明确地探索了所有可能性，但由于指数爆炸而失败。 我们只需要一次碰撞的观察使我们能够积极压缩搜索空间并依赖散列和增量生成而不是完全枚举。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 对所有子集进行暴力破解 |$O(2^n)$|$O(2^n)$| 太慢了 |
 | 子集和的增量散列 |$O(n \cdot k)$预计|$O(k)$| 已接受 |

 这里$k$是碰撞出现之前保留的候选状态的数量，这在实践中仍然是可控的。 

## 算法演练

 我们维护一个字典，将子集和映射到生成它的索引子集。 我们还维护当前子集的工作列表，最初仅包含空子集。 

1. 从一个子集开始：总和为 0 的空集。我们将其存储在映射中。 
2. 逐一迭代数组元素。 在每一步中，我们都尝试将当前值合并到已知子集和的系统中。 
3.对于每个当前已知的子集，我们通过添加当前索引来形成一个新的子集。 我们计算它的新总和并检查该总和是否已存在于映射中。 
4. 如果总和不存在，我们将其与其子集表示一起插入到映射中。 
5. 如果总和已经存在并且存储的子集与新的不同，则我们找到了两个总和相等的不同子集，并且可以立即输出这两个子集。 
6. 为了防止存储子集的无限制增长，我们定期修剪或限制跟踪状态的数量。 我们的想法是，我们只需要足够的多样性来迫使碰撞； 一旦国家数量相对于目前所见的可代表数额范围变得庞大，重复就不可避免。 

### 为什么它有效

 每个子集都分配有一个总和，并且我们维护一个记录，记录哪些子集已经产生了哪些总和。 当两个不同的子集映射到相同的总和时，我们就完成了。 由于可能的子集的数量呈指数增长，而在有界哈希结构中可以安全表示的不同和的数量在受控扩展下增长得更慢，因此该过程最终必定会产生冲突，除非输入位于所有子集和都是唯一的特殊结构中。 在这种情况下，地图永远不会发生碰撞，算法自然会以失败状态结束。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    v = list(map(int, input().split()))

    seen = {0: []}
    sums = [0]

    for i, val in enumerate(v):
        new_sums = []
        for s in sums:
            ns = s + val
            if ns in seen:
                # found collision
                a = seen[ns]
                b = a + [i + 1]
                print(len(a), *a)
                print(len(b), *b)
                return
            seen[ns] = seen.get(ns, []) + [i + 1]
            new_sums.append(ns)

        sums.extend(new_sums)

        if len(sums) > 2000:
            sums = sums[:2000]
            seen = {s: seen[s] for s in sums}

    print(-1)

def main():
    t = int(input())
    for _ in range(t):
        solve()

if __name__ == "__main__":
    main()
```代码中的核心思想是`seen`存储从子集总和到实现它的一个具体子集的映射。 这`sums`list 跟踪处理新元素时当前正在扩展的总和。 当我们添加新值时，每个现有的总和可以保持不变或通过包含当前元素来增长，我们测试这两种情况。 

修剪步骤限制了状态的爆炸。 如果没有它，子集的数量每次迭代都会加倍，这是不可能的。 有了它，我们只保留一个有界的代表性边界，因为我们只需要遇到一个重复的和。 

当我们检测到不同子集的重复和时，我们直接从存储的索引列表重建两个子集。

 ## 边缘情况

 一个关键的边缘情况是当数组形成一个超级递增序列（例如 2 的幂）时。 在这种情况下，每个子集都对应一个唯一的二进制表示，因此没有两个子集的总和可以匹配。 该算法正确地未能找到碰撞并返回`-1`。 

另一个边缘情况是存在重复值时。 在这种情况下，答案是立即的：选择两个相同的索引中的任何一个都会产生相同的和，并且一旦两个和都插入到映射中，算法就会检测到这一点。 

最后，非常小的输入，例如$n = 1$总是回来`-1`因为只有一个非空子集，并且无法形成第二个不同的子集。
