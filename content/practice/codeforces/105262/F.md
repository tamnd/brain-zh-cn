 ## 解决方案
 ## 问题理解

 我们有两个基本字符串，称它们为$F_1$和$F_2$，我们定义一个字符串序列，其中后面的每个字符串都是通过按顺序连接前两个字符串而形成的。 所以$F_3 = F_2 + F_1$,$F_4 = F_3 + F_2$， 等等。 这与斐波那契数的结构规则相同，只不过我们不是将值相加，而是连接字符串。 

对于每个测试用例，我们都会被问到一个查询：给定一个索引$x$和一个位置$k$，我们必须确定哪个字符出现在位置$k$（1-索引）在字符串内$F_x$。 困难在于$F_x$很快就会变得天文数字般大，因此明确构建它是不可能的。 

这些限制强化了这一点：最多可以有$10^4$测试用例，输入字符串长度之和仅为$10^5$，这意味着预处理总体上必须是线性的。 指数$x$上升到$1000$，因此我们可能需要通过串联的深度递归进行推理，但我们无法构建字符串，甚至无法在跟踪其长度之外显式存储它们。 位置$k$可以大到$10^{18}$，这清楚地表明只有基于长度的推理才是可行的。 

一种天真的方法会重复构造斐波那契字符串，直到达到$F_x$，但即使存储$F_{40}$在大多数情况下已经超出了内存限制，并且$F_{100}$是完全不可行的。 另一个天真的想法是递归，动态构建字符串，但这也会呈指数级增长。 

当两个初始字符串中的所有字符都相同时，会出现微妙的边缘情况。 在这种情况下，每个$F_x$是统一的，任何查询都会简化为返回单个字符。 仍然尝试递归的粗心实现可能会溢出或不必要地浪费时间。 

另一个边缘情况是当$k$正好等于之间边界的长度$F_{i-1}$和$F_{i-2}$分解过程中。 这里的差一错误很常见，因为串联将字符串分成两部分，而正确的索引决定了我们在递归中是向左还是向右移动。 

## 方法

 蛮力的想法很简单。 我们构建每个字符串$F_i$迭代地使用串联直到达到$F_x$，然后直接对其进行索引。 这是正确的，因为它完全遵循定义。 然而，每个串联的结果大小都是线性的，斐波那契增长使长度呈指数增长。 甚至达到$F_{40}$已经需要构造一个包含数十亿个字符的字符串，这在时间和内存上都是不可能的。 

关键的观察是我们实际上从来不需要完整的字符串。 我们只需要知道每个时间有多长$F_i$是以及串联如何分割位置。 自从$F_i = F_{i-1} + F_{i-2}$，我们可以使用相同的递归来预先计算长度，但将它们限制为一个大值（例如$10^{18}$）以避免溢出。 一旦长度已知，回答查询就变成了从$F_x$, 决定是否位置$k$位于左侧部分$F_{x-1}$或正确的部分$F_{x-2}$。 这将问题转化为对数下降$x$，类似于导航二叉分解树。

 ## 算法演练

 我们首先为所有斐波那契字符串构建一个长度数组，直到所需的最大索引。 

1. 初始化$len[1] = |F_1|$和$len[2] = |F_2|$。 这些可以从输入字符串直接得知。 
2. 对于每个$i \ge 3$, 计算$len[i] = len[i-1] + len[i-2]$，但如果总和超过$10^{18}$，将其夹至$10^{18}$。 这可以防止溢出，同时保持索引的正确性。 
3. 长度表构建完成后，处理查询$(x, k)$。 从索引开始$x$和位置$k$。 
4. 虽然$x > 2$， 比较$k$和$len[x-1]$。 如果$k \le len[x-1]$，移至$x-1$并保留$k$不变，因为位置位于左子串中。 否则，减去$len[x-1]$从$k$并移至$x-2$，因为我们现在位于正确的子串内。 
5. 什么时候$x$变为1或2，直接返回$k$第 - 个字符$F_1$或者$F_2$。 

每个决策步骤都通过遵循串联结构而不是扩展它来减小问题的大小。 

### 为什么它有效

 该结构保证了每个$F_x$完全分成两个连续的部分：前缀$F_{x-1}$后跟后缀$F_{x-2}$。 长度数组对该分区进行精确编码。 在每一步中，算法都会保留当前对的不变量$(x, k)$指的是与原作中相同的字符$F_x$，只是映射到一个更小的子问题。 由于每次转换都严格减少$x$，进程必须终止，并且最终的基本情况直接索引显式字符串。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MAX = 10**18

def solve():
    t = int(input())
    for _ in range(t):
        n, m = map(int, input().split())
        f1 = input().strip()
        f2 = input().strip()
        x, k = map(int, input().split())

        # build length table up to x
        mx = x
        ln = [0] * (mx + 1)
        ln[1] = len(f1)
        if mx >= 2:
            ln[2] = len(f2)

        for i in range(3, mx + 1):
            val = ln[i - 1] + ln[i - 2]
            if val > MAX:
                val = MAX
            ln[i] = val

        cur_x = x
        cur_k = k

        while cur_x > 2:
            if cur_k <= ln[cur_x - 1]:
                cur_x -= 1
            else:
                cur_k -= ln[cur_x - 1]
                cur_x -= 2

        if cur_x == 1:
            print(f1[cur_k - 1])
        else:
            print(f2[cur_k - 1])

if __name__ == "__main__":
    solve()
```该解决方案将两个问题分开：长度计算和导航。 长度数组是迭代构建的，因此每个值仅取决于前两个值，与斐波那契结构相匹配。 钳位防止溢出，同时保持比较对任何可达的有效$k$。 

下降循环是核心逻辑。 每次迭代都会决定查询位置是位于串联的第一部分还是第二部分。 当进入第二个分量时，减去$len[x-1]$是必不可少的，因为索引相对于新的子字符串发生变化。 

一个常见的实施错误是忘记了$F_x = F_{x-1} + F_{x-2}$（不颠倒）。 颠倒此顺序会导致始终错误的分支。 另一个微妙的问题是不限制长度，它可能会溢出其他语言中的 Python 列表并破坏比较。

 ## 边缘情况

 ### 统一字符串

 如果$F_1 = F_2 = "n"$，那么每一个$F_x$是同一字符的重复。 该算法仍然计算长度和下降，但任何基本情况都会返回相同的字符，无论路径如何。 例如，与$x = 1000$和$k = 10^{18}$，下降可能会访问很多州，但最终的结果总是`"n"`。 

### 以精确前缀长度进行边界分割

 考虑一个步骤，其中$k = len[x-1]$。 该条件必须将其视为属于左侧。 规则`if cur_k <= ln[cur_x - 1]`确保正确性。 如果写成严格的不等式，算法将错误地转移到正确的段，从而产生相差一的错误。 

### 极大的 k 值

 当$k$超过实际长度，夹紧可确保比较保持有效。 即使$len[x]$上限为$10^{18}$，逻辑仍然一致地路由查询，因为所有无法访问的后缀都被视为完全包含在正确的分支中。
