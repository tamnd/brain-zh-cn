 ## 解决方案
 ## 问题理解

 我们给定了多个字符串，对于每个字符串，我们必须计算有多少子序列形成了一个非常具体的结构。 

有效的子序列分两个阶段构建。 首先，它必须包含固定字符串`nunhehheh`作为顺序的子序列。 之后，它必须包含一个仅由字符组成的非空序列`a`。 这些`a`字符不需要是连续的，但它们必须按子序列顺序出现在固定模式完成之后。 

所以对于一个字符串`S`，我们正在有效地计算有多少种方法可以选择指数`i1 < i2 < ... < ik`这样得到的字符就形成了`nunhehheh`随后至少有一个`a`。 

关键的困难在于我们计算的是子序列，而不是子字符串，因此可以任意跳过字符，并且即使结果字符相同，不同的索引选择也会产生不同的子序列。 

所有字符串的总长度可以达到的约束`10^6`排除任何尝试显式枚举子序列的方法。 对子序列的暴力破解的长度将呈指数级增长，甚至对所有子集进行动态编程也是不可能的。 我们被迫对每个字符串进行线性或接近线性的扫描，并且常数因子非常小。 

一个微妙的边缘情况是当字符串包含模式时`nunhehheh`以重叠的方式多次。 形成模式的每个不同的指数选择都必须单独计算，并且每个选择都可以独立地扩展到不同的选择方式`a`之后的人物。 

另一个边缘情况是包含模式但不包含模式的字符串`a`在它之后。 例如，`nunhehhehbbb`具有零个有效子序列，即使前缀模式存在多次，因为不满足后缀要求。 

同样，像这样的字符串`aaaaaaaa`有许多子序列`a`，但除非完全`nunhehheh`子序列较早存在。 

## 方法

 暴力方法将尝试生成输入字符串的所有子序列并检查每个子序列是否与模式匹配`nunhehheh`随后至少有一个`a`。 这会立即失败，因为长度的字符串`n`有`2^n`子序列。 即使是为了`n = 40`，这变得不可行，而这里`n`可以是`10^5`。 

我们可以通过认识到不需要显式构造子序列来减少问题。 我们只需要在从左到右扫描字符串的同时，计算可以按顺序匹配模式的方式有多少种。 这是一个经典的子序列计数动态规划结构。 

我们首先跟踪有多少种方法可以匹配前缀`nunhehheh`。 一旦我们完成了该模式，我们就进入第二阶段，在这个阶段中，我们计算有多少种方法可以选择非空子序列`a`稍后出现的角色。 

关键的观察是模式部分和后缀部分可以分为两个 DP 阶段，但它们必须小心地相互作用。 每场已完成的比赛`nunhehheh`成为以后可以收集的“来源”`a`字符以多种方式。 每个`a`要么开始后缀，要么将现有后缀组合加倍。 

蛮力在概念上是有效的，因为它枚举了所有内容，但由于指数爆炸而失败。 DP 之所以有效，是因为对于每个字符，我们只维护固定模式和后缀状态的部分匹配的计数。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力破解子序列 | O(2^n) | O(2^n) | O(n) | 太慢了 |
 | DP 超过模式 + 后缀状态 | O(n) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们定义模式`P = "nunhehheh"`长度为 9。我们在扫描字符串时维护两种类型的状态。 

一种状态计算有多少子序列与前缀匹配`P`。 第二个状态计算已完成的比赛`P`已经开始收集至少一份`a`。 

我们还需要仔细处理转换，以便每个字符要么用于扩展子序列计数，要么被跳过。 

1.初始化一个DP数组`dp`大小为 10 其中`dp[j]`表示匹配第一种的方式数`j`模式的字符`P`。 放`dp[0] = 1`因为只有一种方法可以匹配空前缀。 
2.初始化另一个变量`dpA = 0`它表示已经选择了至少一个的已完成模式匹配的数量`a`。 
3. 从左到右扫描字符串。 对于每个角色`c`，首先以相反的顺序更新模式 DP，以便每个字符在每个子序列扩展中最多使用一次。 对于每个`j`从 8 降到 0，如果`c == P[j]`，然后添加`dp[j]`进入`dp[j+1]`。 

此步骤确保我们正确计算扩展模式的部分匹配的所有方法，而不会覆盖较短前缀所需的状态。 
4. 如果当前字符不是`a`，不影响`dpA`我们继续。 
5. 如果角色是`a`，我们处理后缀转换。 每个完成的模式匹配`dp[9]`可以开始一个新的后缀或扩展现有的后缀选择。 所以我们更新`dpA = 2 * dpA + dp[9]`。 

乘以 2 对应于选择或不选择此`a`对于每个已经激活的后缀子序列。 添加的`dp[9]`对应于从已完成的模式匹配开始新的后缀子序列。 
6.处理完所有字符后，答案为`dpA`。 

正确性依赖于这样一个事实：一旦我们进入后缀阶段，所有进一步的`a`字符是每个活动的已完成模式实例的独立二进制选择。 

### 为什么它有效

 前缀上的 DP 保持一个不变式`dp[j]`始终计算与第一个完全匹配的已处理前缀的子序列`j`的字符`P`。 反向更新顺序可确保每个角色正确贡献，而无需在同一步骤内重复使用。 

一旦子序列到达`dp[9]`，它就成为一个完整的模式实例。 从那一刻起，接下来的每一次`a`要么被忽略，要么包含在子序列中。 这为每个活动的已完成实例创建了加倍结构，而新的已完成实例为后缀 DP 提供了新的起点。 前缀计数和后缀累积的分离确保了不会重复计数，也不会漏掉组合。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MOD = 998244353

PAT = "nunhehheh"

t = int(input())
for _ in range(t):
    s = input().strip()

    dp = [0] * 10
    dp[0] = 1
    dpA = 0

    for ch in s:
        for j in range(8, -1, -1):
            if ch == PAT[j]:
                dp[j + 1] = (dp[j + 1] + dp[j]) % MOD

        if ch == 'a':
            dpA = (dpA * 2 + dp[9]) % MOD

    print(dpA % MOD)
```其实现直接反映了 DP 的表述。 模式状态的反向循环至关重要，因为向前更新将允许在同一转换步骤中多次使用单个字符。 

变量`dpA`仅在遇到时更新`a`，因为只有`a`有助于后缀构造。 期限`dp[9]`将新完成的模式匹配注入后缀状态。 

## 工作示例

 考虑字符串`nunhehhehaa`。 

我们仅跟踪相关状态。

 | 图案形成后| n-u-n-h-e-h-h-e-h | 1 | 0 |

 | + 第二个 | 一个 | 1 | 3 |

 第二个`a`将现有的后缀选择加倍（选择或不选择它），并且还允许与先前的模式完成进行新的组合。 

这显示了后缀累积一旦多倍后如何呈指数增长`a`字符出现。 

现在考虑`nunhehhehbbb`。 

| 步骤| 查尔 | dp[9] | dp[9] | dpA |

这证实了如果没有后缀条件，仅靠模式完成是不够的。

import sys, io

def solve(inp: str) -> str:
    sys.stdin = io.StringIO(inp)

    t = int(input())
    out = []

        out.append(str(dpA % MOD))

    return "\n".join(out)

 ## 边缘情况

 第一个边缘情况是当多个重叠出现时`nunhehheh`存在于同一个字符串中。 DP 自然会计算所有这些，因为每次出现都对应于一种不同的填充方式`dp[9]`，并且每个单独贡献后缀形成。 该算法处理这个问题是因为`dp[9]`是计数，而不是布尔状态。 

第二个边缘情况是没有的字符串`a`根本不。 即使图案已形成多次，`dpA`永远不会转变，所以答案仍然为零。 这正确地强制要求后缀必须非空。 

第三个边缘情况是长期运行`a`在任何模式完成之前。 这些没有任何作用，因为`dp[9]`仍然为零，因此后缀积累没有来源。 只有在至少一次完整模式匹配之后，后缀计数才会开始，这保证了交错的正确性。
