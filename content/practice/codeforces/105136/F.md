 ## 解决方案
 ## 问题理解

 我们得到了一系列水库。 每个水库都有固定的容量和当前储存的水量。 允许的操作非常具体：我们最多可以选择两个水库，即i和j，并将j中的所有水倒入i中。 如果 i 溢出，它最终会充满其容量，否则最终会变成合并的水。 

目标是确定在最多执行一次此类倾倒操作后，我们是否可以获得最终水量恰好为 x 的水库。 我们可以什么都不做（如果某个水库已经有 x 水），或者选择一个目标水库 i 和一个源水库 j 并将 j 倒入 i 中。 

输出是描述此类操作的一对 i j ，或者如果储存器 i 最初已包含 x 则为 i 0 ，或者如果不存在有效配置则为 -1 -1 。 

约束条件高达 n = 10^5，因此任何直接检查所有储层对的解决方案在最坏的情况下都需要多达 10^10 次操作，这远远超出了 1 秒的限制。 这立即排除了二次方法。 

一个微妙的点是，如果达到容量限制，从 j 倒入 i 不需要完全传输。 结果值为 min(vi, wi + wj)。 这会产生单调但非线性的饱和行为。 

重要的边缘情况：

 水库可能已经含有 x 水，我们必须立即输出 i 0。例如，如果对于某个 i，x = 3 且 wi = 3，则这已经是有效答案。 

即使水库总水量很大，组合也可能不起作用，因为容量限制可能无法准确达到 x。 例如，vi = [2, 2]、wi = [2, 2]、x = 3 就不可能。 

同样重要的是，i 和 j 在输入解释中可能相等，但实际上这不会带来任何新内容，因为倒入同一个容器中不会改变任何有意义的内容。 

## 方法

 蛮力方法会考虑每个可能的对 (i, j)。 对于每一对，模拟将 j 倒入 i 中并计算 min(vi, wi + wj)。 如果它等于 x，我们返回该对。 这需要评估 n^2 个转换，每个转换都需要恒定的时间，这会导致 O(n^2) 复杂度。 当n达到100000时，这是不可行的。 

关键的观察是，从 j 浇注后，水库 i 的最终值仅取决于 wi 和 wj，而不取决于任何其他结构。 我们只关心是否可以使用两种情况之一到达 x。 

第一种情况很简单：某些 wi 已经等于 x。 

第二种情况很重要：我们需要一对 (i, j) 使得 min(vi, wi + wj) = x。 这种情况可以分为两种互斥的情况。 

要么 wi + wj = x 且 wi + wj 不超过 vi，意味着不会发生饱和，要么 vi 足够小，结果会饱和到 vi 且 vi = x，但这已经包含在第一种情况中。 

因此，对于 j 贡献非零效应的有效对，我们需要 wi + wj = x 且 x ≤ vi。 这意味着 j 必须提供 wi 的补集。 

重写它给出了一个经典的补搜索：对于每个 i，我们需要找到一个 j 使得 wj = x - wi，同时还要确保 i 可以容纳至少 x 单位的水容量，vi ≥ x。 

这减少了存储每个水位wi 的位置和匹配补数的问题。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(n^2) | O(n^2) | O(1) | O(1) | 太慢了 |
 | 哈希图补搜索| O(n) | O(n) | 已接受 |

 ## 算法演练

1. 扫描所有水库并检查是否有 wi 等于 x。 如果是，则输出该索引为 0 并停止。 这处理不需要任何操作的情况。 
2.构建从水量wi到具有该值的索引列表的哈希图。 这允许快速补集查找。 
3. 将每个水库 i 作为潜在目标进行迭代。 
4. 如果 vi 小于 x，则完全跳过 i。 即使浇注后，由于容量限制也无法达到x。 
5. 计算 j 所需的贡献，即 need = x - wi。 
6. 如果need 为负，则跳过此i，因为它已经超过x 并且无法通过加法来校正。 
7. 检查map中是否存在有值需要的j。 
8. 如果存在这样的 j，则确保 j 与索引 i 不同，除非存在多个相同的条目，然后输出 i j。 
9. 如果扫描完所有 i 后没有对起作用，则输出 -1 -1。 

工作原理：该算法将所有有效配置划分为直接命中或精确补码总和。 哈希图保证在恒定的平均时间内找到每个可能的贡献库 j。 容量过滤器 vi ≥ x 确保我们只考虑可以合法保留最终值的目标。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, x = map(int, input().split())
    v = list(map(int, input().split()))
    w = list(map(int, input().split()))

    pos = {}
    for i, val in enumerate(w):
        if val == x:
            print(i + 1, 0)
            return
        if val not in pos:
            pos[val] = []
        pos[val].append(i)

    for i in range(n):
        if v[i] < x:
            continue
        need = x - w[i]
        if need < 0:
            continue
        if need not in pos:
            continue

        for j in pos[need]:
            if j != i:
                print(i + 1, j + 1)
                return
            if len(pos[need]) > 1:
                print(i + 1, j + 1)
                return

    print(-1, -1)

if __name__ == "__main__":
    solve()
```该解决方案首先处理立即成功的情况，其中某些水库已经包含恰好 x 水。 这是必需的，因为该操作是可选的，并且这是其唯一有效的输出格式。 

该字典按水位对索引进行分组，从而能够恒定时间查找补集。 i 上的循环强制我们只尝试可以根据容量合法容纳 x 的水库。 

内部选择小心地避免了使用相同索引的退化情况，除非存在重复项，因为从存储库倒入自身不会改变任何有意义的内容。

 ## 边缘情况

 一个重要的边缘情况是多个水库共享相同的水值作为补充。 例如，如果 x = 5 且 wi = 2，并且有两个 wj = 3 的水库，我们必须确保可以选择与 i 不同的有效 j 或重复使用另一个相同的索引。 该算法通过在哈希映射中存储完整列表来处理此问题，从而允许在必要时选择不同的索引。 

当 wi 已经不可能超过 x 时，会出现另一种边缘情况，因为 wi ≤ vi 且 wi ≤ vi 意味着 wi 不能超过容量，但 wi 仍然可以大于 x。 在这种情况下，需求变为负数，并且算法正确地跳过该水库。 

最后的边缘情况是当 i 和 j 重合时。 如果 wi = x - wi，则 j 可以等于 i。 仅当有另一个相同的水库可用或问题允许简单的自用时，这才有效。 该实现显式检查列表大小，以允许在需要时再次发生，从而防止无效的自传输。
