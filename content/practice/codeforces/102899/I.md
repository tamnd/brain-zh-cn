 ## 解决方案
 ## 问题理解

 我们得到了 n 天的股票价格序列。 在每一天 i，股票都有一个已知的价格 p[i]。 KK 最多可以执行一笔完整的交易：他可以在某一天买入一次，然后在严格的晚些时候卖出一次。 他一开始拥有无限的现金，因此目标纯粹是利润最大化，利润定义为卖出价减去买入价。 如果每笔可能的交易都会导致非正利润，则答案定义为零。 

该结构是一个单数组扫描问题，我们试图选择两个索引 i < j 来最大化 p[j] − p[i]。 

约束 n 高达 100000 立即排除所有对的任何 O(n²) 枚举。 在最坏的情况下，二次解需要大约 10^10 次比较，这远远超出了 Python 中 1 秒的限制。 这促使我们采用线性或近线性的方法，在一次或少量的过程中处理数组。 

当价格从未上涨时，就会出现一种微妙的边缘情况。 例如，如果序列是 5 4 3 2，则任何买卖对都会产生负结果，但由于 KK 并未被迫进行交易，因此正确答案为 0。盲目采用最佳差值的幼稚方法可能会错误地返回负数，而不是钳制为零。 

另一个边缘情况是通过在较晚发生的全局最低价处购买来实现最佳利润。 例如，在 10 1 7 3 8 6 中，最佳买入价为 1，卖出价为 8，但如果我们不仔细跟踪历史记录，最低价可能会出现在局部良好的卖出之后。 

## 方法

 蛮力的想法很简单：尝试每对可能的买卖日。 对于每个 i，我们考虑所有 j > i 并计算 p[j] − p[i]，跟踪最大值。 这是正确的，因为它耗尽了所有有效交易。 然而，对于n个起始点中的每一个，我们可能会扫描最多n个后续点，导致大约n(n−1)/2次操作，当n为100000时，这大约是5×10⁹。这无法在时间限制内运行。 

关键的观察是，当我们确定卖出日 j 时，最佳可能的买入日就是之前所有日子中价格最低的那一天。 我们不是重复地重新计算这个最小值，而是在从左到右扫描时维护它。 在每一步中，我们都知道迄今为止看到的最便宜的价格，并将当前价格作为潜在的卖出价格与最佳的买入价格进行比较。 

这将问题简化为一次传递，我们不断更新两条信息：最小前缀值和迄今为止获得的最佳利润。

 | --- | --- | --- | --- |
 | 蛮力 | O(n²) | O(1) | O(1) | 太慢了|
 | 最少一次通过跟踪 | O(n) | O(1) | O(1) | 已接受 |

 ## 算法演练

 1. 初始化变量`min_price`具有非常大的价值并且`best_profit`为 0。`min_price`代表截至当日的最低股价，这是迄今为止最好的买入点。 
2. 从左到右迭代价格数组。 
3. 对于每个价格 p[i]，计算如果我们在前一天最便宜的价格买入后在第 i 天卖出的利润，即 p[i] − min_price。 
4. 更新`best_profit`如果计算出的利润大于当前的最佳利润。 
5. 更新`min_price`如果 p[i] 小于当前最小值，因为未来销售应将其视为更好的购买选择。 
6.处理全天后，输出`best_profit`。 

关键点是循环内的顺序：我们必须使用之前的最小值计算利润，然后才能使用当前价格进行更新。 否则，我们会错误地允许在同一天进行买卖。 

### 为什么它有效

 在每个索引 i 处，`min_price`存储 p[0..i−1] 的最小值。 这保证了在评估第 i 天作为卖点时，我们严格考虑 i 之前的所有有效购买日。 任何最佳解决方案 (i, j) 都必须以 j 作为销售日，并且我们的扫描确保当我们到达 j 时，最佳可能的 i 已被捕获`min_price`。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    p = list(map(int, input().split()))
    
    min_price = p[0]
    best = 0
    
    for i in range(1, n):
        price = p[i]
        
        # sell today using best previous buy
        best = max(best, price - min_price)
        
        # update best buy candidate
        if price < min_price:
            min_price = price
    
    print(best)

if __name__ == "__main__":
    solve()
```该解决方案使用单次遍历阵列，保持运行最小值和最佳利润。 使用 p[0] 进行初始化是安全的，因为第一个可能的交易必须至少涉及较早的一天，并且我们只从第 2 天开始计算利润。 

一个常见的错误是更新`min_price`在计算当日利润之前。 这将允许使用同一天进行买卖，这违反了问题约束。

 ## 边缘情况

 对于像 5 4 3 2 这样的严格递减序列，算法设置`min_price`最初为 5。 随后的每一天都会产生负利润或零利润，并且`best`始终保持为 0。 运行的最小值会不断更新，但永远不会产生积极的差异，以匹配所需的输出。 

对于最小值出现较晚的序列（例如 8 7 6 1 10），算法正确地避免了过早固定较早的最小值。 当达到1时，`min_price`变为 1，最后第 10 天产生利润 9，该利润被正确捕获，因为更新`min_price`计算每天的利润后发生。
