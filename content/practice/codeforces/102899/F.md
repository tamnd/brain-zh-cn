 ## 解决方案
 ## 问题理解

 我们给出了一系列的日子，每天 kk 记录他做出了多少个错误答案（WA）。 这些值形成一个数组`a[1..n]`。 

从零分开始，分数一天天演变。 处理日时`i`，我们比较 WA 计数`a[i]`与之前的每一天`j < i`。 每个比较都独立贡献：如果`a[j] < a[i]`，分数加一，如果`a[j] > a[i]`，分数减少一，如果相等，则没有任何变化。 在应用与前几天的所有比较后，这个新分数将成为当天的当前分数`i`。 第一天没有之前的日子，因此总是以零分结束。 

我们需要输出两个值：所有天中任意时刻获得的最高分数，以及处理所有天后的最终分数。 

约束条件`n ≤ 100000`迫使我们避免任何明确比较每一天的事情。 一个简单的双循环将涉及大约 n 次平方的比较，这远远超出了可行的范围。 任何解决方案都必须将重复的“计算有多少个先前值小于或大于当前值”减少为每天可以有效回答的问题。 

当许多值相等时，会出现微妙的边缘情况。 由于相等的值没有任何贡献，因此正确的解决方案必须确保它们被排除在“较大”和“较小”计数之外，而不会意外地重复计算或丢失相等值组之间的转换。 

当序列严格递增或严格递减时，会出现另一个极端情况。 在这种情况下，随着时间的推移，在对对贡献的简单解释中，分数呈二次方增长，但实际的演化对于先前元素的贡献而言，每一步都是线性的，如果试图错误地模拟对更新，这种差异很容易被误解。 

## 方法

 蛮力解释很简单。 对于每一天`i`，我们扫描前几天`j < i`并明确比较`a[j]`和`a[i]`。 我们保持当前分数并跟踪最大值。 这是正确的，因为它直接遵循分数更新规则的定义。 然而，它的性能约为`1 + 2 + ... + (n-1)`比较，其顺序为`n^2 / 2`，并且对于`n = 100000`这是完全不可行的。 

关键的观察是当天的得分`i`仅取决于之前的值有多少较小和有多少较大。 如果我们定义：

 -`less(i)`= 数量`j < i`这样`a[j] < a[i]`-`greater(i)`= 数量`j < i`这样`a[j] > a[i]`那么转换很简单：`score[i] = score[i-1] + less(i) - greater(i)`因此，问题简化为维护相对于当前值的先前元素的计数。 我们需要一个数据结构，当我们从左到右处理数组时，它可以支持：

 1. 插入值`a[i]`2. 查询有多少个现有值严格小于或严格大于

 由于值高达`10^5`，我们可以将它们压缩成 Fenwick 树（或线段树）。 这使我们能够有效地维护频率和查询前缀和。 每一天都成为`O(log n)`操作代替`O(n)`扫描。 

最终的答案需要跟踪过程中的最大前缀分数，而不仅仅是最终值。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(n²) | O(1) | O(1) | 太慢了|
 | 芬威克树计数 | O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 我们从左到右处理数组，同时维持所有先前看到的值的频率结构。 

1. 初始化 Fenwick 树（或类似结构）并将所有计数设置为零。 同时初始化`score = 0`和`best = 0`。 这表示进行任何比较之前的第一天。 
2. 每天`i`从 1 到 n，我们对待`a[i]`作为我们插入历史记录的当前值。 
3. 插入前`a[i]`，我们查询有多少个先前的值严格小于。 这是一个前缀和查询，最多可达`a[i] - 1`。 这给出了`less(i)`。 
4. 我们还查询有多少个先前值严格大于。 这计算为`(i - 1) - count(≤ a[i])`。 前缀总和为`a[i]`给出小于或等于的值的数量，因此从之前的元素总数中减去可以隔离严格更大的元素。 这给出了`greater(i)`。 
5. 我们将分数更新为`score += less(i) - greater(i)`。 这直接应用了前一天如何对今天的变化做出贡献的定义。 
6.我们更新`best = max(best, score)`每天之后，因为最高分数可能出现在最后一天之前。 
7. 我们插入`a[i]`通过增加频率进入 Fenwick 树。 

经过一整天的处理，`best`是看到的最高分数，并且`score`是最终值。 

### 为什么它有效

 在任何时刻，Fenwick 树都准确地代表了前几天 WA 计数的多重集。 每一天对分数的贡献`i`仅取决于之间的比较`a[i]`和早期的元素。 该结构确保每个较早的元素在“较少”、“较大”或忽略类别中精确计数一次，并且这些类别是不相交的。 由于分数被定义为独立成对贡献的总和，因此维护汇总计数可以保留准确的值，而无需重新访问过去的元素。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class Fenwick:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)

    def add(self, i, v):
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

def solve():
    n = int(input())
    a = list(map(int, input().split()))

    maxv = max(a)
    fw = Fenwick(maxv)

    score = 0
    best = 0

    for i, x in enumerate(a):
        less = fw.sum(x - 1)
        total_prev = i
        leq = fw.sum(x)
        greater = total_prev - leq

        score += less - greater
        best = max(best, score)

        fw.add(x, 1)

    print(best, score)

if __name__ == "__main__":
    solve()
```Fenwick 树存储迄今为止看到的 WA 计数频率。 前缀总和为`x-1`直接给出严格较小的先前值的数量。 前缀总和为`x`给出小于或等于值的计数，因此从已处理元素的数量中减去会产生严格更大的值。 分数更新完全遵循配对贡献规则，并增量跟踪最佳值。 

一个微妙的细节是`total_prev`只是索引`i`，自处理日之前开始`i`正是有`i`之前的元素。 这可以避免额外的查询并保持更新干净。

 ## 边缘情况

 对于所有值都相同的序列，每次比较都会产生相等，因此两者都不是`less`也不`greater`改变分数。 Fenwick 树对于两个查询始终返回零，并且分数始终保持为零。 用于输入`5 2 2 2 2 2`，算法每天都会处理，但每个`less`和`greater`为零，因此最终分数和最高分数都为零。 

对于像这样的严格递增序列`1 2 3 4 5`，每个新元素都比所有以前的元素大，所以`less(i) = i-1`和`greater(i) = 0`。 分数增加了`0, 1, 2, 3, 4`，累积到 10，芬威克树正确地反映了每一步不断增长的前缀分布。
