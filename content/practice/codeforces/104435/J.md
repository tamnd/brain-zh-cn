 ## 解决方案
 ## 问题理解

 我们有一栋大楼，有四个房间，分别标记为 A、B、C 和 D，由三个走廊相连。 每个走廊都是两个房间之间的物理通道，每次有人通过走廊从一个房间移动到另一个房间时，都会记录一个日志条目，该日志条目由该人的 ID 和他们使用的走廊组成。 

所有员工都从 A 房间开始。对于每位员工，我们都会按时间顺序获得他们据报告使用过的走廊的顺序。 关键的困难在于系统没有明确记录他们在哪些房间之间移动，只记录了哪个走廊被触发。 

还有一个额外的复杂性：通风口将所有走廊相互连接。 这意味着一个人可能会在不经过有效的房间过渡的情况下从一个走廊移动到另一个走廊。 在这种情况下，他们的移动将不符合在建筑布局中物理上一致的行走。 任务是确定哪些员工只有在滥用通风口至少一次的情况下才能产生观察到的走廊使用顺序。 这些员工被标记为可疑。 

输出是员工的二进制字符串。 如果该员工的序列无法通过从 A 开始且不使用通风口的有效步行来解释，则字符为 1，否则为 0。 

就员工数量而言，约束很小，最多 1000 人，但日志条目的数量可以大到 100000 条。这立即表明我们必须以线性时间处理日志并维护每个员工的增量状态，而不是从头开始为每个查询重新模拟。 

为每个员工独立重建完整路径并尝试所有可能的房间分配的简单方法会太慢，并且可能会导致每个人的日志条目数量呈指数级增长。 

主要边缘情况来自具有单个日志条目的员工以及连续走廊过渡与底层图中的共享端点不对应的员工。 特别是，即使路径的其余部分是一致的，单个无效转换也足以引起怀疑。 

## 方法

 强力解释是独立对待每个员工，并尝试将一系列房间分配到他们的走廊序列。 对于每个日志条目，我们会尝试猜测该人从走廊的哪个端点进入以及从哪个端点退出，从而与之前的分配保持一致。 这成为每个走廊使用的两个可能方向的受限搜索问题。 在最坏的情况下，对于具有 k 个日志条目的单个员工，这种分支会导致 2^k 种可能性，这在 k 很大时是不可行的。 

关键的简化是我们实际上不需要重建完整路径。 我们只需要检测是否存在有效路径。 问题的结构简化为检查连续走廊用途之间的局部一致性。 

每条走廊正好连接两个房间。 如果一个人在不使用通风口的情况下从走廊 ci 移动到走廊 ci+1，那么他们在 ci 之后结束的房间必定是 ci+1 的端点之一。 这意味着 ci 和 ci+1 必须在建筑图中至少共享一个端点。 如果它们不共享端点，则无法仅使用有效的房间移动在它们之间进行转换，因此必须使用通风口。 

因此，对于每个员工，我们只需要跟踪他们之前的走廊并验证每对连续的走廊是否共享至少一个房间端点。 我们还需要确保从房间 A 可以到达第一个走廊，这意味着 A 必须是该走廊的一个端点。 如果情况并非如此，那么第一个动作就已经需要通风口了。 

这将问题简化为简单的每个员工线性扫描和恒定时间邻接检查。

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力路径重建| 每个员工 O(2^k) | O(k) | 太慢了 |
 | 邻接一致性检查| O(ℓ) | O(n + 3) | 已接受 |

 ## 算法演练

 我们通过为每个走廊存储它连接的两个房间来预处理建筑物。 这为我们提供了一种恒定时间的方法来检查两个走廊是否在共享一个房间的意义上物理上相邻。 

我们为每个员工维护他们的顺序是否仍然与有效步行一致。 我们还追踪他们最后使用的走廊。 

1. 初始化一个大小为 n 的可疑数组，并将所有值设置为 false。 这将标记被证明需要使用通风口的员工。 
2. 对于按时间顺序排列的每个日志条目，处理该对（员工、走廊）。 
3. 如果这是我们第一次见到该员工，我们会检查走廊是否以房间 A 作为其端点之一。 如果不是，那么从A开始，该员工不可能合法进入该走廊，因此我们将其标记为可疑。 
4. 如果这不是第一次发生，我们会将该员工当前使用的走廊与之前使用的走廊进行比较。 我们检查两个走廊是否共享至少一个端点房间。 如果他们不这样做，那么就没有员工可以在这两次走廊穿越之间合法过渡的房间，因此我们将他们标记为可疑。 
5. 将员工最后一次看到的走廊更新为当前走廊，无论他们是否已被标记为可疑，因为仍需要后续检查以确保正确性。 

关键思想是我们从不尝试明确地重建位置。 我们仅确保每个台阶在连续的走廊使用之间至少有一个有效的“集合点”房间。 

### 为什么它有效

 在两次记录的走廊使用之间的任何时刻，没有通风口的有效移动意味着该人位于作为先前走廊遍历的端点的某个房间中。 下一个走廊遍历必须从同一个房间开始，因此它也必须与该房间相关。 因此，连续的走廊必须至少共享一个端点。 如果不这样做，则该转换不存在一致的房间分配，因此必须至少发生一次通风转换。 相反，如果每个连续的对共享一个端点，并且第一个走廊与 A 相关，我们总是可以沿着序列贪婪地选择房间分配，确保存在一致的行走。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, l = map(int, input().split())

    # Each corridor connects exactly two rooms.
    # We assume this mapping is given implicitly by the problem statement.
    # For a CF-style solution, these would normally be read or predefined.
    # Here we represent them as adjacency lists.
    corridors = {
        0: (0, 1),  # A-B (example encoding A=0, B=1, C=2, D=3)
        1: (1, 2),  # B-C
        2: (2, 3),  # C-D
    }

    def shares(u, v):
        return (corridors[u][0] == corridors[v][0] or
                corridors[u][0] == corridors[v][1] or
                corridors[u][1] == corridors[v][0] or
                corridors[u][1] == corridors[v][1])

    last = [-1] * n
    bad = [False] * n

    start_room = 0  # room A

    for _ in range(l):
        e, c = map(int, input().split())

        if last[e] == -1:
            # first move must be reachable from A
            if corridors[c][0] != start_room and corridors[c][1] != start_room:
                bad[e] = True
        else:
            if not shares(last[e], c):
                bad[e] = True

        last[e] = c

    print("".join("1" if bad[i] else "0" for i in range(n)))

if __name__ == "__main__":
    solve()
```该解决方案仅维护每个员工最后使用的走廊，这已经足够了，因为一旦违反或本地保留了一致性，任何更长的历史记录都无关紧要。 

一个常见的错误是试图跟踪每个员工的整个房间状态。 这是不必要的，因为唯一重要的是是否存在至少一个与两个连续走廊用途一致的房间。 

另一个微妙之处是，第一个走廊必须对照房间 A 进行检查。如果没有这一点，第一个日志已经与起始位置相矛盾的员工将被错误地标记为有效。

 ## 边缘情况

 一种重要的边缘情况是员工只有一个日志条目。 在这种情况下，唯一的要求是走廊必须与 A 房间相连。如果不是，员工会立即产生怀疑，因为没有可能合法地开始他们的移动。 

当员工的所有日志在 A 中单独有效，但两个连续走廊之间的单一转换是不可能的时，就会出现另一种边缘情况。 例如，如果走廊 0 连接 A-B，走廊 2 连接 C-D，并且员工先记录 0，后记录 2，则没有可以连接这两个台阶的共享房间。 该算法在第二步中正确地将它们标记为可疑。 

最后一个边缘情况是重复使用同一走廊。 这始终有效，因为走廊始终与其自身共享两个端点，因此邻接检查自然会通过，并且不会引入错误怀疑。
