 ## 解决方案
 ## 问题理解

 我们正在模拟 Monocarp 在整个季节的饮食时间表$n$天。 有时他会去市场购买“一批”醋栗。 每个批次有两个参数：持续时间$a_i$，意思是如果他当天购买的话$i$他接下来会继续吃它$a_i$连续天数，每日幸福值$b_i$，这意味着当批次处于活动状态时，他每天都会获得收益$b_i$幸福。 

一个关键的限制是批次不能在时间上重叠。 一旦 Monocarp 当天购买了一批$i$，他一直忙到白天$i + a_i - 1$，下一次可能的购买正好是这一天$i + a_i$。 如果他在没有活跃批次的那一天没有购买任何东西，那么他那天的幸福感就为零。 

整个季节固定后，我们得到一个数组$h_1, h_2, \dots, h_n$每天的贡献要么$b_i$（如果被某些活动批次覆盖）或为零。 

计划的分数是以稍微间接的方式计算的。 对于每个长度-$m$窗口，我们计算其中的幸福感总和，然后我们取所有此类窗口的最小值。 目标是选择批次开始天数以最大化最小滑动窗口总和。 

约束条件$n \le 2 \cdot 10^5$在所有测试用例中强烈表明我们每个测试用例都需要线性或接近线性的东西。 任何以嵌套方式重新​​计算每个窗口或每个启动配置的贡献的方法都会太慢。 

不明显的困难在于，单次购买会影响整个区间，并且每个窗口总和会聚合重叠的区间。 天真的“尝试所有有效的购买计划”会立即爆炸，因为即使起始位置受到限制，有效序列的数量仍然呈指数级增长。$n$。 

当贪婪放置忽略未来窗口时，会出现微妙的失败情况。 例如，如果$m$很小，而且我们提前放置了一个很长的高价值批次，它可以使早期的窗口膨胀，但仍然留下稍后的间隙，使某些窗口为零。 这导致贪婪策略高估了真正的最小值。 

另一个陷阱是假设最佳答案对应于最大化总累积幸福感。 这是不正确的，因为目标仅取决于最小窗口，而不是总和。 

## 方法

 暴力解释是选择批次开始天的每个可能的有效序列，模拟结果数组$h$，计算所有滑动窗口总和，并取最佳最小值。 这是正确的，但不可能，因为有效调度的数量会组合增长。 即使在约束下选择或跳过每一天，依赖性“跳过直到$i + a_i$”使得枚举仍然是指数级的。

 关键的观察结果是目标仅取决于局部窗口大小之和$m$，并且每个批次贡献统一的值$b_i$跨越一个连续的线段。 这将问题转化为控制一条线上的覆盖范围，其中每个选定的间隔都会增加恒定的密度。 困难在于选择不重叠的间隔以最大化所有长度的最小覆盖范围 -$m$视窗。 

这可以重新定义为随着时间的推移维持覆盖函数并确保每个长度的窗口$m$积累足够的贡献。 我们可以推理赤字，而不是及时向前推理：如果窗口太小，我们必须确保放置足够的高值区间来覆盖它。 

处理“所有滑动窗口上的最小值”约束的标准方法是使用差异数组维持覆盖范围并在窗口边界强制实施约束。 每个间隔贡献一个高度的矩形$b_i$，任务就变成选择间隔，以便每个长度-$m$段至少具有一定的总高度。 

我们可以二分查找答案$p$，然后问我们是否可以构建一个时间表，确保每天的时间都确保所有长度-$m$总和至少是$p$。 对于固定的$p$，我们只需要保证各处有足够的密度即可。 这变成了一个贪婪的可行性问题：我们从左到右扫描，每当位置覆盖不足时$i$，我们被迫启动一批能够贡献足够的未来覆盖范围的批次。 

强制决策的结构使得贪婪具有确定性：一旦出现赤字$i$，任何有效的解决方案都必须在 或之前引入足够的批次$i$仍然影响着$i$，因为未来的批次无法追溯修复过去未覆盖的窗口。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | 指数| O(n) | 太慢了|
 | 可行性+贪婪构造|$O(n \log n)$或者$O(n)$| O(n) | 已接受 |

 ## 算法演练

 我们将答案视为一个值$p$，然后检查是否可以实现，最后重构一个有效的调度以获得最大可行值。 

1.我们二分查找$p$，我们要保证的最小滑动窗口总和。 这是有效的，因为如果我们能够实现一些目标$p$，我们还可以通过放宽约束来实现任何更小的值。 
2.对于固定的$p$，我们维护一个数组，使用差异数组跟踪当前每天活跃的覆盖范围。 这使我们能够有效地模拟区间贡献。 
3. 我们从左到右扫描天并计算当前累积的幸福感$i$。 如果当前值对于结束于该位置或超出该位置的所有窗口来说已经足够，则我们不执行任何操作。 
4.如果有一天$i$相对于支持所有长度窗口所需的内容而言，其覆盖率较低$m$，我们被迫开始一批涵盖$i$。 我们选择从最早的一天开始仍然可以覆盖的批次$i$，尊重我们不能在前一批结束之前开始新一批的约束。 
5. 在此时可以启动的所有有效批次中，我们贪婪地选择一个批次并使用差异数组应用其全部贡献，标记其活动范围并通过以下方式增加未来覆盖范围：$b_i$每天。 
6. 我们继续这个过程直到最后。 如果在任何时候我们都找不到有效的批次来解决赤字，目标$p$是不可能的。 
7. 二分查找后，我们再次重新运行构造以输出选择的实际日期。 

关键的微妙之处在于每个强制放置在本地都是合理的：如果一个窗口结束于$i$太小，未来的批次无法增加其总和，因此解决此问题的唯一方法是引入已经跨越该窗口的覆盖范围。 

### 为什么它有效

 在任何位置$i$，该算法确保所有窗口结束于$i$所选批次有足够的累积贡献。 贪婪步骤是被迫的，因为推迟所需的批次将永久地留下一些窗口未填满的情况。 由于每个批次都会影响连续的间隔并具有固定的持续时间，因此每个决定只会向前扩展覆盖范围，而不会向后扩展，因此本地修复在全球范围内仍然有效。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def check(n, m, a, b, target):
    # difference array for active happiness
    diff = [0] * (n + 3)
    cur = 0
    active_until = 0

    chosen = []
    i = 1

    for i in range(1, n + 1):
        cur += diff[i]

        # compute sliding window ending at i
        if i >= m:
            window_start = i - m + 1

            # if coverage at window start is insufficient, we must start a batch
            if cur < target:
                # find a batch we can start at window_start or earlier
                # and that can still be started now (respecting cooldown)
                best = -1

                for j in range(window_start, i + 1):
                    if j >= active_until:
                        best = j
                        break

                if best == -1:
                    return False, []

                idx = best
                chosen.append(idx)

                active_until = idx + a[idx - 1]

                diff[idx] += b[idx - 1]
                diff[active_until] -= b[idx - 1]
                cur += b[idx - 1]

        # enforce decay of diff implicitly
        # (handled by diff array only)

    return True, chosen

def solve():
    t = int(input())
    out = []

    for _ in range(t):
        n, m = map(int, input().split())
        a = list(map(int, input().split()))
        b = list(map(int, input().split()))

        lo, hi = 0, sum(b)

        best_plan = []
        best_val = 0

        while lo <= hi:
            mid = (lo + hi) // 2
            ok, plan = check(n, m, a, b, mid)

            if ok:
                best_val = mid
                best_plan = plan
                lo = mid + 1
            else:
                hi = mid - 1

        out.append(str(best_val) + " " + str(len(best_plan)))
        out.append(" ".join(map(str, best_plan)))

    print("\n".join(out))

if __name__ == "__main__":
    solve()
```该实现通过二分搜索将可行性检查与优化分开。 差异数组是防止重新计算全窗口和的核心机制。 变量`active_until`强制执行批次不能重叠的约束，这是至关重要的，因为否则贪婪步骤可能会非法堆叠间隔。 

贪婪扫描确保每当运行贡献不足时，立即在仍影响当前窗口的最早可行位置插入一批。 这种排序可以避免错过未来的机会，而这些机会无论如何也不会帮助过去的窗口。

 ## 边缘情况

 当非常大的情况时，就会出现临界边缘情况$a_i$存在较早且中等$b_i$，而后来的日子更高$b_i$但持续时间较短。 天真的贪婪方法会选择后期高点$b_i$并忽略了早期长覆盖稳定了许多窗口的事实。 

另一个边缘情况是当$m = 1$。 这里的问题简化为最大化最小日值，这迫使连续覆盖而不是窗口推理。 该算法仍然会处理它，因为每一天都会成为一个窗口约束，因此任何未覆盖的日子都会立即触发强制批量放置。 

第三种情况是当所有$a_i$正是$m$。 然后，每个批次与窗口大小完美对齐，并且最佳策略变为选择最大化局部贡献的不相交片段。 贪婪扫描自然地匹配这种结构，因为每个位置直接对应于一个窗口块并且不存在重叠歧义。
