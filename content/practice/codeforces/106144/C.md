 ## 解决方案
 ## 问题理解

 我们得到了一串括号。 两个玩家交替删除该字符串末尾的字符。 每次移动时，玩家都会选择当前字符串的最左边或最右边的字符并将其删除。 不同之处在于，Monocarp 在移动时还将删除的字符附加到第二个字符串，而 Polycarp 只是将其丢弃。 

第二个字符串仅由 Monocarp 构建，开头为空。 如果在任何时候这个字符串变成一个非空的常规括号序列，Monocarp 立即获胜。 如果在此之前从原始字符串中删除所有字符，则波利卡普获胜。 

关键的困难在于 Monocarp 并不试图完成所有括号的完整解析，他只需要收集的序列的一些前缀就已经形成了正确的括号序列，并且这可以在所有移动完成之前发生。 Polycarp 的角色纯粹是破坏性的，试图阻止 Monocarp 形成这样的前缀。 

约束的总大小很小，测试用例的字符串长度总和高达 6000。这立即排除了对所有游戏状态或所有可能的移动序列的任何指数模拟。 每个测试用例的三次或二次解是可以接受的，但任何涉及完整博弈树探索的东西都是不可接受的。 

一个微妙的点是，只要他收集的字符串的任何前缀是正确的括号序列，而不一定是他收集的整个字符串，Monocarp 就会获胜。 这使得尽早形成“平衡前缀”成为真正的目标。 

## 方法

 蛮力的想法会将游戏模拟为完整的极小极大过程。 每个状态由 s 的当前子串、当前串 t 以及轮到谁来确定。 从每个州开始，我们都会向左或向右分支。 Monocarp 另外更新 t，并且我们在每次移动后检查 t 的任何前缀是否形成规则的括号序列。 

这很快就会爆炸。 即使我们只考虑端点的选择，游戏路径的数量也会以 n 为指数，大约为 2^n，并且在每个节点检查括号有效性会使情况变得更糟。 最优游戏的结构隐藏在这棵树中，因此暴力破解是不可用的。 

关键的观察是波利卡普的着法根本不依赖于 t 的结构。 他删除角色只是为了影响 Monocarp 以后可能选择的内容。 Monocarp 唯一的希望是快速强制出现一种情况，无论 Polycarp 的删除如何，他都可以获得一个 t 的前缀，从而成为正确的括号序列。 

这将问题简化为初始字符串的结构属性：Monocarp 能否保证在 Polycarp 破坏所有机会之前以正确的顺序获得两个匹配的括号。 关键的简化在于，只有存在一对可以与两端隔离的“安全”括号才重要，并且游戏会在开始时检查字符串是否已经“过于受限”。 

最终已知的特征非常简单：只有当每个可能的第一步都迫使他收集的字符串以右括号开头时，Monocarp 才会失败，因为这样就没有前缀可以成为有效的 RBS。 否则，Monocarp 可以通过尽早选择开局来强制获胜配置。 

这导致对字符串结构的线性检查：是否存在可以以 Polycarp 无法立即阻止的方式确保左括号作为 t 的第一个字符的移动。

 ## 算法演练

游戏本质上是从 Monocarp 选择 t 的第一个角色开始的。 在这第一步之后，波利卡普做出了回应，但他无法追溯改变莫诺卡普已经采取的行动。 所以决定性的问题是 Monocarp 能否确保他的第一个选择的角色是开括号。 

如果 Monocarp 的第一步动作可以导致左括号，那么他已经有了一个有效的单字符前缀 t，因为稍后可以将其扩展为完整的 RBS。 从那时起，最佳发挥允许他继续延伸，直到出现完整的有效序列。 

如果 Monocarp 无论选择左还是右都被迫在第一步采取右括号，那么 t 以 ')' 开头。 以 ')' 开头的字符串永远不能有任何作为常规括号序列的非空前缀，因为有效的 RBS 必须在每个前缀边界处至少多一个开头而不是结尾。 

因此，我们只需要检查初始字符串的任一端点是否包含左括号。 如果至少一个末端是“(”，Monocarp 选择它并为 t 确保一个开始。如果两个末端都是“)”，那么任何第一步都会产生一个结束括号，并且 Monocarp 注定失败。 

整个游戏简化为这个端点检查，因为波利卡普只能移除内部结构，但无法阻止初始移动选择。 

### 为什么它有效

 不变的是，Monocarp 的获胜条件仅取决于他能否使 t 以作为括号序列不会立即无效的前缀开头。 有效的括号序列不能以“)”开头，因此 Monocarp 的成功相当于确保他的第一个收集到的字符是“(”。

 Polycarp 的后续动作无法改变这个初始事实，并且由于 Monocarp 只需要存在某个作为 RBS 的 t 前缀，因此第一个字符“(”保证了在最佳玩法下存在有效的完成策略。如果两端都是“)”，则每一个可能的第一步都会迫使失败，并且没有后续策略可以修复它。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    out = []
    for _ in range(t):
        n = int(input())
        s = input().strip()
        
        if s[0] == '(' or s[-1] == '(':
            out.append("Monocarp")
        else:
            out.append("Polycarp")
    
    print("\n".join(out))

if __name__ == "__main__":
    solve()
```该代码反映了将整个游戏减少为每个测试用例的恒定时间检查。 读取每个字符串，并且仅检查其端点。 字符串结构的其余部分是无关紧要的，因为 Polycarp 的删除永远不会影响 Monocarp 从两端选择第一个字符的能力。 

为了提高效率，该决策会附加到输出列表中，并在最后打印一次，从而避免重复的 I/O 开销。 

## 工作示例

 考虑输入`s = "(()())"`。 端点都是“(”，因此 Monocarp 可以立即选择一个左括号。

 | 转 | s（当前）| 单果树摘 | t | 结果检查|

 | 1 | (()()) | (()()) | '(' | ( | 存在有效前缀 |

 这表明，一旦 t 的第一个字符是“(”，Monocarp 就可以构建完整的有效序列。

 现在考虑`s = "))))"`。 

| 转 | s（当前）| 单果树摘 | t | 结果检查|

 这里每个可能的第一步都会产生 ')'，因此 Monocarp 永远无法形成有效的 t 前缀。 

这两个案例展示了核心二分法：端点处是否存在可访问的“(”决定了整个游戏。

 ## 边缘情况

 一种重要的边缘情况是唯一的开口支架位于中间。 例如，`s = "))()("`。 Monocarp 可以选择最右边的字符，即“(”，立即确保 t 的有效开始。该算法可以正确处理此问题，因为它只检查最后一个字符。

 另一种边缘情况是两端都是')'，但里面有很多'('，比如`s = ")()()()"`。 尽管该字符串富含左括号，Monocarp 也无法首先选择它们，因为 Polycarp 始终只允许端点删除。 该算法正确输出 Polycarp，因为两个端点都是 ')'，并且没有第一步可以避免以右括号开始。 

最后的边缘情况是最小长度场景，例如`s = "()()"`。 这里两端都是“(”和“)”，但由于至少一端是“(”，所以 Monocarp 立即获胜，这符合预期的逻辑。
