 ## 解决方案
 ## 问题理解

 我们得到一组隐藏的位置，标记为 1 到 N。其中一些位置是“损坏的”，其余的是“工作的”。 断仓数量并没有直接给出，但是我们可以交互式地查询系统。 

每个查询包括发送长度为 N 的二进制字符串。法官通过仅获取与工作位置相对应的位、保留它们的相对顺序并将它们连接成较短的字符串来做出响应。 换句话说，如果您从查询字符串中删除了所有损坏的位置，则会得到响应。 

任务是使用尽可能少的查询来准确识别哪些位置被破坏。 

N 可以很大（在该问题系列的完整版本中最多约为 10^5）的约束意味着任何尝试单独测试每个位置的方法都太慢。 仅当每个查询都非常便宜时，具有 N 个查询的线性扫描才是可接受的，但交互规则通常使这种操作变得不必要。 相反，响应的结构强烈建议并行编码所有索引的信息。 

当没有被破坏的仓位或者除了一个仓位以外的所有仓位都被破坏时，就会出现一种微妙的边缘情况。 在所有工作情况下，每个查询都会返回未更改的原始字符串，因此任何假定“缺少缩短”信号的逻辑都将失败。 在单一工作情况下，每个查询的每个响应只有一个字符，因此即使仅保留一列，重建也必须保持一致。 

## 方法

 一个蛮力的想法是通过在查询中翻转每个位置来独立测试每个位置，同时保持其他位置固定并观察输出是否以揭示其状态的方式发生变化。 然而，这很快就会陷入歧义，因为响应压缩了损坏的位置，这意味着我们失去了直接的位置对应关系。 尝试隔离每个索引将需要 O(N) 查询，并且每个查询仅提供部分信息，使得这种方法既缓慢又概念上不可靠。 

关键的见解是停止将查询视为“每个索引的测试”，而是将它们同时视为所有索引的并行位编码。 如果我们为每个索引 i 分配一个二进制表示，那么我们可以构造同时请求所有索引的一位位置的查询。 法官的响应保留了工作索引之间的顺序，因此每个响应列始终对应于所有查询中相同的幸存索引。 

通过发出大约 log N 个查询，每个查询编码索引的一位，我们可以重建每个工作位置的完整身份。 一旦工作集已知，断开的位置就只是补充。 

关键的结构属性是在所有响应中保留工作索引的顺序。 这种对齐使得可以将响应字符串中的每个位置视为属于不同查询中的相同原始索引。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 每个索引的暴力破解 | O(N^2) 交互 | O(N) | 太慢/不可靠 |
 | 按位编码重构 | O(N log N) 预处理/O(log N) 查询 | O(N) | 已接受 |

 ## 算法演练

 我们假设 N 个位置标记为从 1 到 N，并令 B = ⌈log2 N⌉。

1. 对于从 0 到 B−1 的每一位 k，构造一个查询字符串，其中位置 i 包含 i 的第 k 位。 这会在该位位置同时对每个索引进行编码。 
2. 发送此查询并接收响应字符串 r_k。 该字符串准确包含位位置 k 处的所有工作索引的位，在所有查询中以相同的相对顺序。 
3. 存储所有响应字符串r_k。 每个 r_k 的长度等于工作索引的数量。 
4. 对于响应字符串中的每个位置 j，通过从每个 r_k 中取出第 j 个字符来重建二进制数。 这个二进制数对应第j个工作位置的原始索引。 
5. 将所有重建索引标记为工作。 
6. 不在重构的工作集中的从1到N的任何索引都被破坏，因此将其输出。 

这样做的原因是每个响应 r_k 的第 j 个字符始终对应于相同的基础工作索引，因为判断器保留所有查询中幸存元素的相对顺序。 

### 为什么它有效

 每个查询将所有索引投影到一位维度上并过滤掉损坏的索引，而不对剩余元素重新排序。 这意味着每个查询中的第 i 个幸存元素对应于所有位查询中的相同原始索引。 因此，收集到的响应的每一列形成一个工作索引的一致二进制编码，并且索引之间不可能混合。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input().strip())
    
    # number of bits needed
    b = n.bit_length()
    
    res = []
    
    # build bitwise queries
    for k in range(b):
        query = []
        for i in range(1, n + 1):
            if (i >> k) & 1:
                query.append('1')
            else:
                query.append('0')
        print("".join(query))
        sys.stdout.flush()
        
        r = input().strip()
        res.append(r)
    
    w = len(res[0])
    working = set()
    
    for j in range(w):
        idx = 0
        for k in range(b):
            if res[k][j] == '1':
                idx |= (1 << k)
        working.add(idx)
    
    broken = []
    for i in range(1, n + 1):
        if i not in working:
            broken.append(i)
    
    print("!".join(map(str, broken)) if broken else "!")

if __name__ == "__main__":
    solve()
```实现的核心是位查询的构造。 每个查询都是确定性的，并且仅取决于位位置，因此我们永远不需要根据以前的响应进行调整。 

重建步骤依赖于将每个列索引 j 视为所有响应中的固定幸存者身份。 k 上的按位或重建了该幸存者的原始索引。 唯一微妙的一点是确保索引的一致性，这是通过问题的顺序保存来保证的。 

最后，我们计算补集。 这避免了有关损坏索引的任何歧义，并确保即使在极端情况下（例如没有损坏元素）的正确性。 

## 工作示例

 考虑一个小情况，其中 N = 5 且工作索引为 {1, 3, 5}。 那么每个查询的响应将始终包含 3 个字符。 

对于 k = 0（最低有效位），查询为 [1,0,1,0,1]，响应为 [1,1,1]。 

对于 k = 1，查询为 [0,1,1,0,0]，响应为 [0,1,0]。 

对于 k = 2，查询为 [0,0,0,1,1]，响应为 [0,0,1]。 

现在我们按列重建：

 | j | r0[j] | r0[j] | r1[j] | r1[j] | r2[j] | r2[j] | 重建索引|
 | --- | --- | --- | --- | --- |
 | 1 | 1 | 0 | 0 | 1 |
 | 2 | 1 | 1 | 0 | 3 |
 | 3 | 1 | 0 | 1 | 5 |

 这确认了每一列对应于一个工作索引。 

现在考虑 N = 4，只有索引 2 起作用。 

所有查询都返回一个单字符字符串，始终是索引 2 处的值。重建仅生成索引 2，所有其他查询都被正确分类为损坏。 

这些示例证实，即使工作集最小或最大，列一致性仍然保持。

 ## 边缘情况

 当所有位置都工作时，每个查询都会准确返回构造的字符串。 重建步骤生成所有索引 1 到 N，并且最终补集为空，因此输出正确报告没有损坏的位置。 

当只有一个位置工作时，每个查询都会返回一个字符。 重建循环处理单个列并根据其二进制表示正确重建该索引，这表明该算法不依赖于具有多个幸存者。 

当除一个位置以外的所有列都被破坏时，响应长度为 1，并且所有列都会折叠为单个一致索引。 按位重建仍然有效，因为每个位查询在所有响应中保持对齐。
