 ## 问题理解

 我们正在处理一个隐藏的长度为二进制的数组$N$。 每个位置对应一个正在工作或已损坏的设备，但我们不知道哪些是哪些。 获取信息的唯一方法是通过询问：我们提交另一个相同长度的二进制数组，作为回报，我们收到一个整数，它告诉我们有多少工作设备在我们放置 1 的位置与我们的查询匹配。 

换句话说，每个查询的行为就像对隐藏的“工作掩码”的选择性求和。 无论我们发送什么内容，损坏的设备都不会产生任何答案。 这使得系统的行为就像一个嘈杂的线性测量，其中只有坐标的子集实际上很重要。 

这个更简单版本中的任务是使用尽可能少的查询来定位至少一个损坏的设备索引。 

的限制$N$足够大以至于不可能单独扫描每个索引。 线性方法需要$O(N)$查询，这远远超出了交互式设置所允许的范围。 由于每个查询已经花费了一次交互，因此我们应该期望更接近对数行为，通常$O(\log N)$，当结构允许重复减半或分区时，这是自然目标。 

一个常见的陷阱是假设响应直接告诉我们有关各个指数的信息。 事实并非如此。 它仅聚合所选子集，因此任何解决方案都必须仔细设计子集，其响应可以隔离结构差异。 

当所有设备都工作时会出现边缘情况。 在这种情况下，不存在损坏的索引，任何盲目继续分区的策略最终都会尝试隔离不存在的目标。 例如，如果$N = 1$并且单个设备正在工作，除非明确处理这种情况，否则简单的二分搜索式方法可能仍会尝试“查找”损坏的索引并表现不正确。 

## 方法

 蛮力的想法很简单。 我们通过查询仅包含该索引的集合来逐一测试每个索引。 如果响应为1，则设备正在工作。 如果为0，则设备已损坏。 这是有效的，因为单例查询完全隔离了单个位置的贡献。 

然而，这种方法需要$N$最坏情况下的查询。 什么时候$N$如果很大，这立即变得不可行，因为每个查询都涉及与法官的交互，并且交互的总数受到严格限制。 

关键的观察结果是响应在子集上表现出线性。 如果我们查询一个集合$S$，答案等于内部工作设备的数量$S$。 如果所有设备都在$S$正在工作，答案等于$|S|$。 之间的任何差异$|S|$并且响应立即保证内部至少有一个损坏的设备$S$。 这将问题转化为在子集级别可检测到“不良”的空间中的搜索。 

此属性允许采用分而治之的策略。 我们没有单独检查元素，而是反复将搜索间隔分成两半，并确定哪一半包含至少一个损坏的设备。 每个查询都会将候选范围缩小两倍，从而导致查询数量呈对数。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O(N)$查询 |$O(1)$| 太慢了|
 | 最佳（通过查询进行二进制拆分）|$O(\log N)$查询 |$O(1)$| 已接受 |

 ## 算法演练

 我们将索引视为当前候选段，保证至少包含一个损坏的设备。 

1、从全系列入手$[1, N]$。 我们假设该范围内至少存在一个损坏的索引。 如果问题不允许，我们首先检查全范围； 如果一切正常，我们会立即终止。 
2. 将当前范围分成两半。 让左半边成为$[l, mid]$右半部分是$[mid+1, r]$。 
3. 通过发送一个二进制数组来查询左半部分，该数组的索引恰好为 1$[l, mid]$其他地方为 0。 
4. 将响应与左半部分的大小进行比较。 如果响应严格小于该大小，则至少有一个损坏的设备必须位于左半部分。 否则，左半部分仅包含工作设备，因此损坏的索引必须位于右半部分。 
5. 将搜索范围移至包含损坏设备的一半。 
6. 重复此操作，直到范围缩小为单个索引。 该指数肯定会被打破。 

关键思想是每个查询都充当一个证书：它告诉我们某个段是“纯粹工作”还是“被至少一个损坏的设备污染”。 

### 为什么它有效

 该算法保持一个不变量：当前搜索段始终包含至少一个损坏的索引。 每个步骤的正确性依赖于这样一个事实：当且仅当查询响应等于其长度时，段才没有损坏的设备。 因此，每当我们丢弃一半时，我们都可以保证它不会包含损坏的设备。 由于我们只丢弃被证明完全正常工作的段，因此剩余段始终保留至少一个损坏索引的存在，以确保进度，直到隔离单个索引。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def ask(l, r):
    # build query string with 1s on [l, r]
    n = N
    s = ['0'] * n
    for i in range(l, r + 1):
        s[i] = '1'
    print(''.join(s))
    sys.stdout.flush()
    return int(input().strip())

def solve():
    global N
    N = int(input().strip())

    l, r = 0, N - 1

    # assume at least one broken exists
    while l < r:
        mid = (l + r) // 2

        res = ask(l, mid)
        expected = mid - l + 1

        if res < expected:
            r = mid
        else:
            l = mid + 1

    print(l + 1)
    sys.stdout.flush()

if __name__ == "__main__":
    solve()
```核心结构是标准的二分搜索，但我们不是比较值，而是将子集总和与预期大小进行比较。 功能`ask(l, r)`为每个查询构造所需的二进制字符串，仅在所选间隔中放置 1。 比较`res < expected`是消除一半搜索空间所需的唯一决策点。 

一个微妙的实现细节是索引。 内部搜索是从 0 开始的，但输出是从 1 开始的，因此最终索引在打印之前会递增。 另一个重要的一点是每次输出后刷新，这在交互式问题中是强制性的，以确保法官收到查询。

 ## 边缘情况

 当唯一设备工作时，每个段都包含损坏索引的假设变得无效。 正确的实现必须依赖于保证至少存在一个损坏的设备，或者明确处理“无损坏”响应模式。 在二分搜索过程中，这表现为每个查询都准确返回段大小，导致算法始终向右移动。 如果没有停止条件，这将错误地返回最后一个索引，即使它没有被破坏。 

当损坏的设备位于索引 1 时，涉及最左侧段的每个查询都会立即检测到差异。 该算法将反复向左边界缩小范围，并且不变量确保搜索永远不会丢弃包含索引 1 的段。 

当损坏的设备位于索引处时$N$，所有左半查询都将符合预期，导致重复消除左段，直到只剩下最终索引。 这强调了“相等时向右移动”规则的正确性，这对于正确性至关重要。
