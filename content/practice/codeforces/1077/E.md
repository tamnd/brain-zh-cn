 ## 解决方案
 ## 问题理解

 我们得到了多组问题，其中每个问题都属于一个主题。 同一主题可以出现多次。 我们想从这个池子中组织几场比赛。 每次竞赛必须仅使用单个主题的问题，因此它是同质的。 不同的比赛必须使用不同的主题，因此每个主题最多只能为一场比赛做出贡献。 

这些比赛是按顺序排列的。 如果我们决定第一场比赛使用$x$出现问题，那么接下来必须使用$2x$，下一个$4x$等等，形成严格的双链。 我们可以自由选择创建多少比赛，也可以自由选择起始规模$x$。 目标不是最大化竞赛数量，而是最大化所有选定竞赛中使用的问题总数。 

关键结构是每次竞赛都会消耗固定的问题预算，并且预算呈指数增长。 由于主题不能重复，因此序列中的每个步骤都必须分配一个不同的主题，并且该主题必须有足够的可用问题来支持该阶段大小。 

约束条件$n \le 2 \cdot 10^5$意味着我们需要一个$O(n \log n)$或者$O(n)$方法。 任何尝试显式模拟所有可能的起始大小和主题分配的方法都会太慢，特别是因为每个主题可能出现多次，并且朴素匹配会重复扫描大型频率列表。 

当频率不均匀时，会出现微妙的故障情况。 例如，如果一个主题的计数很大，而许多其他主题的计数很小，那么简单地首先获取最大频率而不考虑加倍顺序的贪婪分配可能会在早期阶段浪费大量计数，或者即使存在更好的安排，也无法扩展链。 

另一个边缘情况是多个主题具有相同的频率。 一种幼稚的方法可能会互换对待它们，但分配顺序很重要，因为早期阶段消耗的问题比后期阶段少得多，而且排序错误可能会阻止可行的扩展。 

## 方法

 一个蛮力的想法是考虑每个可能的起始主题和每个可能的起始大小。 对于固定的起始尺寸$x$，我们尝试建立一个链：选择任何至少有$x$问题，然后是任何其他主题至少$2x$， 然后$4x$， 等等。 对于每个选择，我们计算所使用问题的总和并取最大值。 

这在原则上是正确的，因为它探索了所有有效的序列。 然而，对于每次尝试，我们都会在主题中反复搜索可行的下一个频率。 高达$2 \cdot 10^5$问题，不同频率的数量很大，并且可能的链的数量也很大。 即使我们压缩频率，检查所有链也会导致接近二次或更糟的结果。 

关键的观察结果是问题的结构仅取决于主题频率，而不取决于单个问题。 一旦我们计算出每个主题存在多少问题，我们就可以将每个主题视为一个“容量”值。 然后我们想将这些能力分配给几何级数的级别。 

我们不是选择开始并模拟前进，而是扭转观点：确定我们将参加多少场比赛，比如说$k$，并尝试查看主题到级别的最佳分配$1, 2, 4, \dots, 2^{k-1}$。 对于固定的$k$，为了最大限度地提高使用率，我们希望将最大的可用频率分配给最大的所需级别。 这建议对频率进行排序并从最大需求到最大容量进行贪婪匹配。 

剩下的问题是如何确定最佳的$k$。 由于每个增加的级别都会使需求增加一倍，因此所需的总量呈指数增长，因此$k$最多是周围$\log n$。 这使得尝试所有的方法成为可能$k$值并贪婪地评估。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力破解链 | 指数 /$O(n^2)$|$O(n)$| 太慢了 |
 | 尝试所有长度+贪心赋值|$O(n \log n)$|$O(n)$| 已接受 |

 ## 算法演练

 1. 统计每个主题的频率。 

每个主题都变成一个数字，代表我们可以从中分配多少个问题。 
2. 按频率降序排列。 

大主题更加灵活，因此我们希望它们尽可能满足更大的需求。 
3. 构建频率前缀和，以快速计算任意选定的热门主题子集有多少问题可用。 
4.尝试所有可能的比赛次数$k$。 

由于需求增长$1, 2, 4, \dots$，当所需的总数超过可用问题时，我们就会停止。 
5.对于固定的$k$，计算所需的尺寸：$1, 2, 4, \dots, 2^{k-1}$。 
6. 从最大需求到最小需求贪婪地检查可行性：首先将最大频率主题分配给最大需求。 

这是可行的，因为更大的需求更难满足，因此它们应该消耗最灵活的资源。 
7. 如果可行，将总贡献计算为所有分配的竞赛规模的总和并更新答案。 

### 为什么它有效

 核心不变量是，对于任何固定数量的竞赛，当较大频率可用时，最佳分配永远不会从向较小频率提供较大要求中受益。 任何将更大频率移动到更大要求的交换只能保留或提高可行性，因为约束是单调的。 这将问题简化为将排序的需求与排序的容量相匹配，从而消除了分配的组合爆炸。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = list(map(int, input().split()))
    
    freq = {}
    for x in a:
        freq[x] = freq.get(x, 0) + 1
    
    vals = sorted(freq.values(), reverse=True)
    
    best = 0
    
    # try number of contests k
    for k in range(1, len(vals) + 1):
        need = []
        x = 1
        total_need = 0
        
        for i in range(k):
            need.append(x)
            total_need += x
            x *= 2
        
        # prune if even total requirement exceeds sum of all problems
        if total_need > n:
            break
        
        # check feasibility: assign largest needs to largest frequencies
        ok = True
        for i in range(k):
            if vals[i] < need[k - 1 - i]:
                ok = False
                break
        
        if ok:
            best = max(best, total_need)
    
    print(best)

if __name__ == "__main__":
    solve()
```该代码首先将输入压缩为主题频率。 然后，它对这些频率进行排序，以便我们始终可以首先推理出最强的主题。 对于每个可能的比赛次数，它构造所需的几何序列并检查最大频率是否可以满足最大要求。 反向比较确保我们以最有利的方式匹配约束。 

一旦需求超出可用问题，使用总需求量的修剪步骤可以防止不必要的检查。 

## 工作示例

 ### 示例 1

 输入：```
18
2 1 2 10 2 10 10 2 2 1 10 10 10 10 1 1 10 10
```频率：

 | 主题 | 计数 |
 | ---| ---|
 | 10 | 10 8 |
 | 2 | 5 |
 | 1 | 3 |

 排序：`[8, 5, 3]`我们测试$k = 1, 2, 3$。 

为了$k = 3$，要求是`[1, 2, 4]`，总数为 7。 

分配检查：

 | 级别（最大的在前）| 要求 | 频率|
 | ---| ---| ---|
 | 4 | 4 | 8 |
 | 2 | 2 | 5 |
 | 1 | 1 | 3 |

 全部满意，所以总计 = 7。 

这表明所有三个主题都可以参与双链。 

### 示例 2

 输入：```
2
3 3
```频率：

 | 主题 | 计数 |
 | ---| ---|
 | 3 | 2 |

 仅存在一个主题，因此仅$k = 1$是可能的。 

要求`[1]`，总计 = 1。 

该算法正确地避免了尝试$k = 2$因为没有第二个主题。

 ## 边缘情况

 一种关键的边缘情况是当一个主题主导频率分布时。 例如：```
8
1 1 1 1 1 1 1 1
```这里只能形成一场竞赛，并且算法正确地输出 1，因为不存在第二个主题。 

另一种情况是许多小的平等主题：```
6
1 2 3 4 5 6
```每个主题的频率为 1。仅$k = 1$有效，因为$k = 2$已经需要第二个主题了。 该算法正确地防止了过度扩展，因为排序的频率未通过要求检查。 

最后一种情况是存在大主题但无法支持深链时。 该算法确保即使总和很大，指数要求也会强制提前停止，从而防止无效的长序列。
