 ## 解决方案
 ## 问题理解

 我们得到了一系列学生，每个学生都与一个信任值相关联，该信任值随着我们处理系统而变化。 在任何时刻，都有一些学生被认为具有积极的信任，我们对两件事感兴趣：当前有多少学生具有积极的信任，以及他们调整后的贡献之和。 

每个学生都有一个初始信任值，并且还有一个由之前更新得出的全局累积数量。 为了方便起见，我们定义一个运行值$d_i$，这仅取决于截至当前点的全局更新，而不取决于任何单个学生。 当学生具有积极的信任时，他们对答案的贡献是他们当前的信任值减去此全局偏移量$d_i$。 

关键结构是所有活跃的学生共享相同的减法项$d_i$，因此，当对所有活跃学生的贡献求和时，这个全局术语将其分解为活跃学生数量和$d_i$。 这意味着我们永远不需要明确跟踪每个学生的个人价值观变化，只需汇总有关谁仍然活跃以及他们的原始信任值总和的信息。 

输入可以解释为一系列更新，这些更新可以修改信任值或推进全局状态。 处理完所有操作后计算输出，报告具有正调整信任的学生数量以及相应的总和。 

在约束方面，预期的解决方案运行在$O(n \log n + q)$，这表明大约$2 \cdot 10^5$元素与更新相结合。 这立即排除了每次更新都会扫描所有学生的二次模拟。 任何重复重新计算每个操作的所有活动状态的方法都会大致达到$10^{10}$在最坏的情况下进行操作，这是不可行的。 

当人们试图在每次全局更新后急切地为每个学生重新计算调整后的信任值时，就会出现微妙的失败案例。 例如，如果所有学生一开始都具有较大的信任值，但全局偏移量逐渐增长，则幼稚的方法可能会反复减少所有值。 

一个小例子可以清楚地说明这一点。 假设我们有三个学生，信任值为 10、9、8，全局偏移量变为 7。正确的活动集仅取决于每个值是否超过 7，因此这三个值仍然为正。 然而，简单的重复减法实现可能会重复更新每个学生的每个操作并降低性能，即使逻辑条件仅取决于与共享值的比较。 

当学生被除名或变得不活跃时，就会出现另一个陷阱：未能在汇总中正确计算他们的贡献，导致重复计算或在最终结果中包含陈旧的值。 

## 方法

 蛮力的想法很简单。 我们直接模拟系统。 对于每次操作，我们都会更新所有受影响学生的信任值，重新计算哪些信任值保持正值，然后从头开始重新计算贡献的计数和总和。 这是有效的，因为它直接遵循过程的定义，但每次重新计算都会扫描所有学生。 和$n$学生和$q$操作，这导致$O(nq)$，在最坏的情况下是二次到三次尺度，具体取决于输入结构。 在$n = 2 \cdot 10^5$，即使每个操作进行一次完整扫描也已经太大了。 

关键的观察结果是，所有活跃的学生都受到全球抵消的相同影响$d_i$。 这意味着我们只需要维护两件事：活跃学生的原始信任值总和，以及其中有多少人是活跃的。 一旦我们知道了这些，总贡献就可以通过减去$d_i \times s$， 在哪里$s$是活跃学生的数量。 然后，问题简化为维护一组动态变化的值，其中一旦元素低于由下式定义的当前阈值，元素就会被删除$d_i$。 

这正是最小堆有效捕获的结构类型。 我们总是想知道当前活跃学生中的最小信任值。 如果最小的一个低于阈值$d_i$，它必须被删除，因为所有的贡献都是单调的，一旦最小的贡献不满足条件，它可能会使当前的活动集失效。 重复此删除过程可确保仅保留有效的学生，并且每个学生最多插入和删除一次，从而提供摊销的对数复杂性。 

另一种观点是按信任度对所有学生进行排序，并按阈值递增的顺序对其进行处理。 当阈值单调时，这种方法也有效，但堆方法更通用并且与动态更新保持一致。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 |$O(nq)$|$O(n)$| 太慢了 |
 | 最小堆维护 |$O(n \log n + q)$|$O(n)$| 已接受 |

 ## 算法演练

 我们根据候选学生的信任值、信任值的运行总和以及活跃学生的数量来维护候选学生的最小堆。 我们还跟踪全球偏移量$d_i$。 

1.初始化一个空的最小堆，一个变量`active_sum = 0`， 和`active_count = 0`。 这些代表当前的活动集。 
2. 通过将每个学生的信任值插入到堆中来处理每个学生，并增加两者`active_sum`和`active_count`因此。 这构建了初始候选池。 
3. 当我们处理全局更新时，保持当前的偏移量$d_i$。 该值表示学生保持有效所需的最低信任阈值。 
4. 每次更新后，重复检查堆中的最小信任值。 如果它严格小于或等于由$d_i$，将其从堆中移除并从中减去`active_sum`，同时递减`active_count`。 此步骤可确保活动集中没有无效学生。 
5. 继续删除，直到堆为空或最小元素满足保持活动状态的条件。 
6. 一旦达到稳定，使用以下公式计算答案`active_sum - active_count * d_i`总贡献，并报告`active_count`作为活跃学生的数量。 

移除循环是核心决策点。 仅当学生在当前阈值下被证明无效时，我们才会将其删除。 这样可以避免不必要的重新计算，同时保证正确性。 

### 为什么它有效

 该算法保持一致的不变量：当前堆内的所有元素都满足在当前全局偏移量下被视为活动的条件。 任何违反此条件的元素都必须位于最小堆的顶部，因为它是最小的剩余信任值，因此如果失败，则不存在更小的候选元素来保持正确性。 由于每个元素最多被删除一次，因此堆操作的总数与学生的数量成线性关系，并且每次操作花费的时间是对数的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline
import heapq

def solve():
    n, q = map(int, input().split())
    arr = list(map(int, input().split()))

    heap = []
    active_sum = 0
    active_count = 0

    for x in arr:
        heapq.heappush(heap, x)
        active_sum += x
        active_count += 1

    d = 0

    for _ in range(q):
        typ, val = map(int, input().split())

        if typ == 1:
            d += val
        else:
            heapq.heappush(heap, val)
            active_sum += val
            active_count += 1

        while heap and heap[0] <= d:
            x = heapq.heappop(heap)
            active_sum -= x
            active_count -= 1

        if active_count <= 0:
            print(0, 0)
        else:
            print(active_count, active_sum - active_count * d)

def main():
    solve()

if __name__ == "__main__":
    main()
```代码保留全局偏移量`d`这代表了应用于所有学生的累积转变。 每次插入都会更新堆和聚合和，以便我们稍后可以在恒定时间内重建移位的和。 堆确保我们始终首先删除最小的违规元素，这是必要的，因为只有最小元素才能确定是否需要删除。 

减法`active_sum - active_count * d`是重建的关键步骤。 我们没有调整每个存储的值，而是延迟应用偏移量并在查询时计算一次。 这可以避免重复更新并保持解决方案的高效。 

从堆中弹出的循环是安全的，因为每个元素都插入一次并删除一次，保证了摊销对数行为。 

## 工作示例

 考虑一个初始数组`[5, 3, 10]`更新增加了全局偏移量。 

开始时，堆是`[3, 5, 10]`，总和为 18，计数为 3，并且`d = 0`。 

| 步骤| 运营| 堆| d | 活动计数 | 活动总和 |
 | ---| ---| ---| ---| ---| ---|
 | 1 | 初始化| [3,5,10] | 0 | 3 | 18 | 18
 | 2 | +2 至 d | [3,5,10] | 2 | 3 | 18 | 18
 | 3 | 删除 <= d | [5,10]| 2 | 2 | 15 | 15

 去掉3后，剩下的元素都是有效的。 答案变成 2 并且$15 - 2 \cdot 2 = 11$。 

该跟踪显示仅检查最小的元素，并且一旦将其删除，则不需要进一步的级联删除。 

现在考虑插入的情况：从空开始，然后插入 4、1、6，并逐渐增加偏移量。 

| 步骤| 运营| 堆| d | 活动计数 | 活动总和 |
 | ---| ---| ---| ---| ---| ---|
 | 1 | +4 | [4] | 0 | 1 | 4 |
 | 2 | +1 | [1,4]| 0 | 2 | 5 |
 | 3 | +6 | [1,4,6]| 0 | 3 | 11 | 11
 | 4 | d += 3 | [1,4,6]| 3 | 3 | 11 | 11
 | 5 | 删除 <= d | [4,6]| 3 | 2 | 10 | 10

 这演示了插入和阈值增加如何干净地交互：仅在需要时才延迟删除无效元素。 

## 复杂度分析

 | 测量| 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 |$O(n \log n + q \log n)$| 每个元素在堆中插入一次并删除一次，每次操作花费对数时间 |
 | 空间|$O(n)$| 堆最多存储所有活动或挂起的元素 |

 复杂性完全符合最多可达的典型限制$2 \cdot 10^5$，因为即使在最坏情况的插入和删除序列中，堆操作仍然有效。 

## 测试用例```python
import sys, io

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    import heapq

    n, q = map(int, input().split())
    arr = list(map(int, input().split()))

    heap = []
    s = 0
    c = 0
    d = 0

    for x in arr:
        heapq.heappush(heap, x)
        s += x
        c += 1

    out = []
    for _ in range(q):
        t, v = map(int, input().split())
        if t == 1:
            d += v
        else:
            heapq.heappush(heap, v)
            s += v
            c += 1

        while heap and heap[0] <= d:
            x = heapq.heappop(heap)
            s -= x
            c -= 1

        if c <= 0:
            out.append("0 0")
        else:
            out.append(f"{c} {s - c*d}")

    return "\n".join(out)

# custom cases
assert run("3 3\n5 3 10\n1 2\n1 1\n1 5") is not None
assert run("1 2\n1\n1 1\n1 1") is not None
assert run("2 2\n1 100\n2 50\n1 60") is not None
assert run("5 1\n1 2 3 4 5\n1 10") is not None

 ## 边缘情况

 当全局偏移量变得足够大以至于所有学生都无效时，就会出现严重的边缘情况。 用于输入`[2, 3, 5]`增量很大，比如说$d = 10$，堆最终会弹出所有元素。 该算法正确地处理了这个问题，因为 while 循环清空了堆并且最终检查`active_count <= 0`触发`(0, 0)`输出。 

另一个微妙的情况是在偏移量已经增长后重复插入小值。 例如，如果$d = 100$我们插入值`[1, 2, 3]`，每个新插入都会在清理循环期间立即删除。 堆永远不会保留无效元素，因此状态保持一致，无需追溯修复。 

最后，当所有值都相等并且偏移量逐渐增加时，就会批量进行删除。 为了`[5, 5, 5, 5]`和重复增量$d$，每个增量可能会一次删除多个元素。 该不变量确保批量删除是安全的，因为排序保留在堆中并且所有元素都是相同的，因此不会出现隐藏的排序问题。
