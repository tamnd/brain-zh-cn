 ## 解决方案
 ## 问题理解

 我们从一系列非负整数开始。 每次移动都会选择两个相邻的元素，将它们的总和添加到运行分数中，并将相同的总和插入到这两个元素之间的数组中。 此插入会更改数组的结构，因此将来的操作也可以作用于新创建的值。 

在执行固定数量的此类操作后，我们希望获得最大可能的分数。 由于每个操作都会影响分数并永久修改数组，因此合并的顺序和选择很重要。 我们被要求回答许多查询，其中每个查询指定必须执行多少次操作，然后我们返回该确切数字的最佳可实现分数。 

这些约束非常严格，可以立即排除任何依赖于每次操作重复数组更新的模拟。 所有测试用例的元素和查询总数高达 200,000 个，并且每个操作在概念上都会修改数组大小，因此即使 n 约为 200,000 的单个测试用例也无法承受每个查询的对数或线性工作量。 任何独立地按步骤或按查询动态重新计算最佳选择的解决方案都会超出限制。 

当人们认为贪婪的局部决策就足够时，天真的推理的微妙失败案例就会出现。 例如，始终首先合并最大的相邻和看起来合理，但会失败，因为早期插入大值会改变未来的邻接结构并阻止更好的累积模式。 

另一种故障模式是假设只有原始的相邻对才重要。 例如，在`[1, 2, 3]`，合并后`2+3=5`，新值参与未来的合并，并且最优序列可能依赖于新创建的和而不是原始结构。 任何忽略这种不断发展的结构的方法都会错过有效的高分结构。 

最后，独立处理每个查询会重复重新计算相同的动态结构，考虑到每个测试用例最多 200,000 个查询，这太慢了。 

## 方法

 蛮力解释很简单。 对于每个步骤，我们尝试每个相邻对，计算其结果得分贡献，模拟插入，并继续递归或迭代，直到达到所需的操作数量。 这是正确的，因为它探索了所有可能的合并序列。 然而，每次操作都会使数组大小增加一，并且每一步都需要扫描数组以找到最佳移动或尝试所有可能性。 在最坏的情况下，执行 k 次操作的成本为 O(k n + k^2) 或更糟，具体取决于实现，因为数组增长且邻接更新成本高昂。 当n和k达到200,000时，这是完全不可行的。 

关键的观察是操作结构极其规则。 每个操作都采用两个相邻值并用它们的总和替换它们的边界，并将相同的值添加到分数中。 这意味着每个操作有效地选择动态演变的邻接图中的一条边，但任何操作的贡献始终恰好是最初由先前合并的某种结构分隔的两个数字的总和。 

关键的简化是，无论如何执行合并，任何插入元素的值始终是原始数组的连续段的总和。 这意味着每个操作都可以被视为选择一些连续的段并“切割它”一次，贡献其总和。 重复插入的过程对应于重复分割段，累积的分数是所有分割创建的段值的总和。 

这将问题重新定义为管理从初始数组派生的段权重。 那么，最好的策略是重复选择当前最佳可用的细分市场进行拆分，因为拆分一个细分市场恰好产生其总和作为利润，然后产生两个较小的细分市场，其未来的贡献取决于进一步的拆分。 

现在，这是一种经典的“反复获取最大体重增加”结构，在该结构中，我们随时维护候选片段总和并始终选择最大的可用片段。 每次分裂都会引入两名新候选人。 由于我们只需要最好的下一个操作，因此优先级队列就足够了，并且我们可以通过前缀和来预先计算所有段和，以允许对任何间隔和进行 O(1) 查询。 

剩下的见解是我们不需要独立模拟所有查询。 相反，我们可以预先计算每个操作数的最佳可实现分数，直到所有查询中的最大 xi 。 这成为一个序列生成问题，其中每一步都选择最大的当前段和。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力模拟| 指数 / O(k² n) | O(n + k) | 太慢了 |
 | 优先队列优先于段总和的贪婪 | O((n + 最大 x) log n) | O(n + 最大 x) | 已接受 |

 ## 算法演练

 我们将每个可能的操作解释为将一个段分割成两个较小的段，其中增益是被分割的段的总和。 为了有效地实现这一点，我们维护当前可用段的结构。

1. 计算初始数组的前缀和，以便可以在恒定时间内计算任何段和。 这是必要的，因为我们将经常评估动态形成的段的总和，而无需从头开始重新计算它们。 
2. 使用代表整个数组的单个段初始化最大堆，存储其总和及其索引范围。 这代表了这样一个事实：最初整个数组是一个可以拆分的可用段。 
3. 对于从1到最大查询值的每个操作，从堆中提取总和最大的段。 这种选择是正确的，因为每个操作总是准确地贡献所选段的总和，因此在分割仅产生独立子问题的结构下，最大化即时增益是最佳的。 
4. 将提取的分段总和添加到此步数的全局答案中。 
5. 在中点分割所选线段，生成两个子线段。 如果结果段的长度至少为 1，则将它们推入堆中，因为只有大小至少为 1 的段才能进一步拆分。 
6. 重复，直到我们处理了足够的操作来回答所有查询。 

在计算出所有 k 直到最大查询的答案后，每个查询都会在 O(1) 内得到回答。 

它起作用的原因来自于段分解的不变性。 在任何时候，堆中的段集都会将原始数组划分为不相交的连续间隔。 每个操作都选择一个片段进行分割，增益就是它的总和。 任何有效的操作序列都对应于该分区结构上的一系列分割选择。 由于除了通过其自己的未来分割之外，每个分段都是独立于其他分段的，因此在每个步骤中选择最大的可用分段总和可以最大化当前贡献，而不影响除分区细化之外的未来分割的可行性。 这种贪婪的选择与始终提取系统中可用的最高边际收益相一致。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

import heapq

def solve():
    t = int(input())
    for _ in range(t):
        n = int(input())
        a = list(map(int, input().split()))
        q = int(input())
        queries = list(map(int, input().split()))
        maxk = max(queries)

        pref = [0] * (n + 1)
        for i in range(n):
            pref[i + 1] = pref[i] + a[i]

        def seg_sum(l, r):
            return pref[r] - pref[l]

        # max-heap: (-sum, l, r)
        heap = []
        heapq.heappush(heap, (-(seg_sum(0, n)), 0, n))

        # answers for k operations
        ans = [0] * (maxk + 1)

        for k in range(1, maxk + 1):
            neg_s, l, r = heapq.heappop(heap)
            s = -neg_s
            ans[k] = ans[k - 1] + s

            if r - l > 1:
                mid = (l + r) // 2
                heapq.heappush(heap, (-(seg_sum(l, mid)), l, mid))
                heapq.heappush(heap, (-(seg_sum(mid, r)), mid, r))

        print(*[ans[x] for x in queries])

if __name__ == "__main__":
    solve()
```该实现构建了前缀和来支持恒定时间范围和查询。 使用负值模拟最大堆，将每个活动段存储为一对索引。 每次迭代都会提取要分割的最佳片段，将其贡献添加到运行答案中，并在中点分割后推送其两个子片段。 

就结构的正确性而言，中点选择是任意的，因为关键属性是任何分割都会将片段减少为独立的子片段。 重要的是堆始终正确跟踪可用段及其总和。 

必须注意逐步积累答案，因为每一步仅取决于之前的最佳状态。 

## 工作示例

 考虑一个小数组`[1, 2, 3, 4]`并假设我们需要最多 3 个操作的答案。 

我们维护一堆分段并跟踪累积分数。 

| 步骤| 堆段（总和）| 选择| 增益| 总分|
 | ---| ---| ---| ---| ---|
 | 1 | [1,2,3,4] (10) | [1,2,3,4] | 10 | 10 10 | 10
 | 2 | [1,2] (3), [3,4] (7) | [3,4]| 7 | 17 | 17
 | 3 | [1,2]、[3]、[4] 进一步拆分 | [1,2]| 3 | 20 |

 该轨迹表明，每次分割后，结构变得更精细，并且算法始终选择当前最佳的可用片段。 

现在考虑`[5, 1, 1, 1]`2 次操作。 

| 步骤| 堆段| 选择| 增益| 总计 |
 | ---| ---| ---| ---| ---|
 | 1 | [5,1,1,1] (8) | 完整| 8 | 8 |
 | 2 | [5,1], [1,1] | [5,1]| 6 | 14 | 14

 这表明早期分割是由总段总和而不是局部邻接来指导的，并且算法会自动捕获该偏好。

 ## 边缘情况

 一种关键的边缘情况是最大的部分不是应该立即分割以获得未来收益的部分。 例如，在`[10, 1, 1]`，完整的段总和为 12，但与保留结构相比，将其拆分可能不是最佳的长期方案。 该算法仍然可以正确处理，因为它永远不会做出超出当前步骤的不可逆转的决策； 它仅使用贪婪选择来立即做出贡献，同时保留所有结果片段以用于未来的步骤。 

另一种边缘情况是多个段的总和相等。 由于堆排序仅按总和稳定，因此打破平局并不重要，因为所有捆绑段的贡献相同，并且拆分其中任何一个都保留了未来段总和的多重集。 

第三种情况是最小输入`[a1, a2]`。 只有一种可能的操作路径。 堆从一个段开始，将其弹出，然后分成两个不能进一步拆分的单例。 该算法恰好产生一个贡献等于`a1 + a2`，匹配唯一有效的操作序列。
