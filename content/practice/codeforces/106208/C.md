 ## 解决方案
 ## 问题理解

 我们被赋予了无限的楼层。 每层楼都有准确的$n$房间，每个房间可容纳一个人。 将一个人放在地板上$x$成本$2x$预算单位，并且我们允许在同一楼层安置多人，只要不超过$n$那里的人。 

对于每个测试用例，我们都会指定每层的容量$n$和总预算$m$。 我们必须准确使用预算，不能少也不能多。 在将人员分配到楼层并准确支出的所有有效方法中$m$，我们希望最大化我们安置的人数。 如果不可能完全达到预算，我们会输出$-1$。 

关键结构是成本仅取决于楼层指数，而不取决于人员，并且每个楼层的行为就像相同“成本项目”的有限供应：楼层$x$提供高达$n$项目，每项成本$2x$，每贡献一个人。 

约束条件达到$10^{18}$，这会立即排除任何以简单方式迭代每层楼或每人的解决方案。 任何在任一方面呈线性的方法$n$,$m$，或者楼层数是不可能的。 解决方案必须将问题简化为每个测试用例的对数或平方根行为。 

一个微妙的问题是，我们不仅仅优化成本可行性，而是在精确总和约束下最大化人数。 这使得贪婪的“首先填充最便宜的”方法变得非常重要，因为过度调整或留下不可填充的剩余部分将使构造无效。 

打破天真的推理的边缘情况包括预算平价失败的情况。 因为每一项成本都是$2x$，总预算必须是偶数。 例如，如果$m = 5$，无论怎样都不存在赋值$n$，因为所有总和都是二的倍数。 

另一种故障情况出现在容量$n$非常小。 例如，如果$n = 1$和$m = 30$，我们不能在同一层楼堆放多个人，因此解决方案变成选择不同的楼层成本$2,4,6,\dots$，其行为类似于受约束的硬币系统，而不是简单的前缀填充。 

## 方法

 蛮力观点是对待每一层$x$因为有最多$n$相同的成本项目$2x$，并尝试选择每层放置多少人，使得总成本恰好为$m$同时最大化所选项目的总数。 这直接变成了一个有界背包问题，最多可达$10^{18}$可能的项目类型，因此任何显式动态编程或迭代都是没有希望的。 甚至迭代楼层直到成本超出的地方$m$放弃$10^{18}$最坏情况下的状态。 

关键的观察结果是，成本结构随着楼层数的增加而严格增加，并且所有项目都具有相同的价值（每个人对答案的贡献恰好为 1）。 这意味着只要有可能，我们总是更喜欢便宜的楼层，因为它们可以最有效地增加每单位成本的人数。 

然而，由于精确求和的要求，我们不能简单地贪婪地获取尽可能多的廉价物品而不检查剩余的可行性。 

我们通过将问题分为两层来解决这个问题。 首先，我们以半单位的方式工作，将所有成本除以二，因此楼层中的每个人$x$成本$x$，总预算变为$m' = m/2$。 如果$m$很奇怪，答案是不可能的。 

其次，我们按照楼层而不是个人来进行决策。 如果我们充分利用地板$x$，我们取$n$成本人员$x$每个贡献一个大小的块$n$。 这将问题转化为选择我们占用多少个完整楼层，以及可能的下一层的部分楼层。 

在我们负担得起的情况下，我们贪婪地租用从一楼开始的整层楼。 如果我们完全占领楼层$1$通过$k$，消耗的成本为$$n(1 + 2 + \dots + k) = n \cdot \frac{k(k+1)}{2}.$$我们选择最大的这样$k$不超过$m'$。 之后，我们将剩余预算分配到楼层$k+1$，其中每增加一个人的成本$k+1$。 

这将问题减少到每个测试用例的少量算术运算。

 ## 算法演练

 我们现在以直接构建最优分配的方式逐步描述构建过程。 

## 算法演练

 1. 通过设置将预算转换为半个单位$m' = m / 2$。 如果$m$为奇数，返回$-1$立即，因为每个有效成本都是偶数。 
2. 找到最大的完整楼层数$k$以便完全填满地板$1$通过$k$和$n$人均不超出预算。 完全填满这些楼层的成本是$n \cdot k(k+1)/2$。 我们增加$k$贪婪地，而这仍然在$m'$。 
3. 减去这些整层楼的成本$m'$。 剩余预算现在仅与下一层相关$k+1$。 
4.在地板上$k+1$, 每个人的花费$k+1$，我们最多可以放置$n$人们。 我们在不超出剩余预算的情况下尽可能多地拿走，所以我们添加$\min(n, \lfloor m' / (k+1) \rfloor)$人们。 
5. 如果经过这些步骤后剩余预算不完全为零，则构建无效并返回$-1$。 否则，总人数为所有全楼层贡献加上部分楼层贡献的总和。 

### 为什么它有效

 在每个阶段，我们都会优先考虑较便宜的楼层，然后再考虑昂贵的楼层，因为每个人对目标的贡献均等，同时在较低楼层消耗的预算严格较少。 任何将更少的人放在更便宜的楼层而使用更昂贵的楼层的偏差只能减少或维持总计数，而永远不会提高总计数。 

该结构确保所有楼层$1$通过$k$在考虑地板之前已经饱和$k+1$。 由于成本是线性的并且每层的容量是均匀的，因此任何最优解决方案都可以转化为这种贪婪形式，而不会改变可行性或总成本，同时不会减少人数。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    T = int(input())
    for _ in range(T):
        n, m = map(int, input().split())

        if m % 2 == 1:
            print(-1)
            continue

        m //= 2

        # find maximum k such that n * k(k+1)/2 <= m
        k = 0
        while True:
            cost = n * (k + 1) * (k + 2) // 2
            if cost <= m:
                k += 1
            else:
                break

        used = n * k * (k + 1) // 2
        rem = m - used

        ans = n * k
        ans += min(n, rem // (k + 1))

        # check exact spend
        if used + min(n, rem // (k + 1)) * (k + 1) != m:
            print(-1)
        else:
            print(ans)

if __name__ == "__main__":
    solve()
```解决方案首先要使成本标准化，以便地板上的每个人$x$成本$x$。 这避免了在整个推理过程中携带两倍的因子。 

循环为$k$增加满层楼的数量，同时总成本仍然可行。 每次迭代都意味着致力于让下一个最便宜的楼层充满$n$人们。 

修复完整楼层后，剩余的部分将在下一层处理，每个额外的人都有统一的成本，因此我们可以使用整数除法来计算最佳可能的填充。 

最终的可行性检查确保我们不依赖部分可分性，否则会留下无法使用当前底线成本表达的剩余预算。

 ## 边缘情况

 当$m$是奇数，算法立即拒绝这种情况，因为每个有效的构造仅使用偶数总成本。 这是在任何楼层推理开始之前处理的。 

什么时候$n = 1$，每个楼层最多可以贡献一个人，因此解决方案退化为在精确总和约束下选择不同的楼层索引。 贪婪的全层结构仍然有效，因为它反映了选择具有单位重数的连续整数。 

什么时候$n$非常大，解决方案实际上会崩溃为仅采用第 1 层，直到预算耗尽，因为标准化后第 1 层本身就可以吸收单位成本步骤中的所有预算，而更高的楼层永远不会有利于最大化计数。 

当预算略高于整层楼数的三角形数时，其余部分将踩在楼层上$k+1$变得至关重要，正确性依赖于使用整数除法来确保我们不会超出可用预算或底线容量。
