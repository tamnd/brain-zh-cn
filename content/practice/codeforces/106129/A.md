 ## 解决方案
 ## 问题理解

 我们正在模拟一个非常结构化的游戏，涉及两队玩家站在桌子的两侧。 左边有ℓ个玩家排列在一个队列中，右边有r个玩家排列在另一个队列中。 每时每刻，只有每个队列的前面的球员很重要：他们面对面，其中一个球员击球，然后该球员立即离开前面的位置，加入对面队列的后面。 这以确定性的方式无限期地持续下去。 

问题的关键不在于完整的配置，而在于每次命中时出现的配对顺序。 每次命中都会生成一对，由左队列的当前前端和右队列的当前前端组成。 在足够的命中之后，这些配对最终开始重复，因为系统的两个循环队列的可能配置数量是有限的。 

输入给出 ℓ 和 r，两个队列的初始大小。 任务是确定在配对序列开始重复之前有多少个不同有序的玩家出现在前面的位置。 

尽管该声明提到了非常大量的点击次数，高达 10^10，但该值并不是真正的瓶颈。 关键是系统早在规模变得重要之前就已经成为周期性的，所以我们只需要了解周期的结构。 

约束 ℓ, r ≤ 10⁹ 立即排除对单个命中的任何模拟。 甚至存储队列的状态也是不可能的，因为状态数量随着 ℓ 和 r 组合增长。 这迫使对该过程进行纯粹的算术或数论解释。 

可能会误导天真的模拟的一个微妙之处是假设该过程取决于玩家的身份而不仅仅是他们在循环轮换中的位置。 例如，当 ℓ = 2 且 r = 2 时，系统会快速循环，即使玩家不断在队列之间移动，相同的配对也会重新出现。 模拟方法需要检测两个序列的重复配置，这太大而无法直接管理。 

## 方法

 直接模拟将显式地维护两个队列并重复地将前面的玩家移动到相反的两端。 每个步骤都是 O(1)，但每次操作后配置都会发生变化，我们需要检测完整状态何时重复。 可能状态的数量是巨大的，因为每个队列可以是其元素的任意排列，并且跟踪需要对大小为 ℓ + r 的结构进行散列或完全比较。 即使忽略这一点，模拟最多 1010 个步骤也是不可能的。 

关键的观察是我们实际上并不需要完整的状态。 我们只关心每一步出现哪一对。 系统的演化是完全确定性和对称的：每一步都以可预测的模块化模式推进“前沿位置”。 每个玩家以固定的节奏有效地循环两侧，并且两个循环之间的相互作用完全由相对大小 ℓ 和 r 控制。 

这将问题简化为理解两个独立循环进程何时同步。 一个周期的长度为 ℓ，另一个周期的长度为 r。 每个步骤的配对仅取决于这些循环的对齐。 当两个循环同时返回到其原始对齐时，对的序列会精确重复，这发生在 lcm(ℓ, r) 步骤之后。 

由于每个步骤只产生一个配对，并且在整个循环完成之前不会重复任何配对，因此不同配对的数量正是该循环的长度。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 队列模拟| O(steps) 或更差 | O(ℓ + r) | 太慢了|
 | 使用 LCM 进行循环分析 | O(log min(ℓ, r)) | O(1) | O(1) | 已接受 |

## 算法演练

 ### 算法演练

 1. 读取值 ℓ 和 r，它们代表两个旋转队列的大小。 目标是确定两个周期的交互模式何时完全重复。 
2. 观察每个队列的行为就像一个循环结构：每个玩家在等于队列大小的固定旋转次数后返回到其原始相对位置。 这表明系统由长度为 ℓ 和 r 的两个独立循环驱动。 
3. 认识到每一步的配对仅取决于这两个循环的相对排列。 仅当两个循环同时返回到其起始对齐时，才会完全重复所有观察到的配对。 
4. 计算两个周期再次对齐的最小时间步长，这是 ℓ 和 r 的最小公倍数。 这是两个前沿和所有相对位置都与初始配置匹配的第一个时刻。 
5. 返回此值作为不同配对的数量，因为重复之前的每个步骤都会生成唯一的配对。 

### 为什么它有效

 该过程可以被视为两个同步的模块化计数器：一个以 ℓ 为模，另一个以 r 为模。 在步骤 t 中观察到的对完全由一对残基 (t mod ℓ, t mod r) 决定。 仅当两个残基同时匹配时，两个步骤才会产生相同的配对，当 t 是 ℓ 和 r 的倍数时，就会发生这种情况。 第一个这样的时间是 lcm(l, r)，直到该点，所有对都是不同的，因为两个残基不会发生较早的碰撞。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def lcm(a, b):
    import math
    return a // math.gcd(a, b) * b

def main():
    l, r = map(int, input().split())
    print(lcm(l, r))

if __name__ == "__main__":
    main()
```故意将实现最小化，因为整个问题简化为计算最大公约数，然后导出最小公倍数。 尽管 Python 可以安全地处理大整数，但乘法是在除以 gcd 后执行的，以避免具有固定整数边界的语言中的溢出问题。 

唯一的微妙之处是确保首先计算 gcd。 写作`a * b // gcd(a, b)`数学上是正确的，但是计算`a // gcd(a, b) * b`一般来说更安全，因为它使中间值更小。 

## 工作示例

 ### 示例 1：ℓ = 2，r = 3

 我们跟踪 t 的值以及两个周期的隐含对齐。 

| 步骤 t | t 模 2 | t 模 3 | 配对唯一性 |
 | --- | --- | --- | --- |
 | 0 | 0 | 0 | 新 |
 | 1 | 1 | 1 | 新 |
 | 2 | 0 | 2 | 新 |
 | 3 | 1 | 0 | 新 |
 | 4 | 0 | 1 | 新 |
 | 5 | 1 | 2 | 新 |
 | 6 | 0 | 0 | 重复初始 |

 完整状态的第一次重复发生在 t = 6，等于 lcm(2, 3)。 所有 6 个步骤都会产生不同的配对。 

这证实了配对序列完全由模块比对控制。 

### 示例 2：ℓ = 2，r = 2

 | 步骤 t | t 模 2 | t 模 2 | 配对唯一性 |
 | --- | --- | --- | --- |
 | 0 | 0 | 0 | 新 |
 | 1 | 1 | 1 | 新 |
 | 2 | 0 | 0 | 重复初始 |

 这里，循环在 2 个步骤后重复，匹配 lcm(2, 2) = 2。即使玩家在队列之间移动，配对结构也会在两个循环重新对齐后立即重复。 

这表明相同的队列大小会将系统折叠成非常短的周期性结构。

 | 测量 | 复杂性 | 说明|

 ## 边缘情况

 当一个队列的大小为 1 时，系统在另一侧减少为单个循环周期。 例如，ℓ = 1, r = 5 会产生每 5 个步骤重复一次的配对序列。 该算法正确返回 lcm(1, 5) = 5，因为 gcd(1, 5) = 1，因此不会发生归约。 

当 ℓ 和 r 相等时，每个对齐都会立即同步。 对于 ℓ = r = 2，两个循环同步移动，并且配对序列在恰好 2 步后重复。 由于 gcd(2, 2) = 2，计算 lcm(2, 2) 自然地处理了这个问题。 

当 ℓ 和 r 互质时，循环仅在 ℓ·r 步后重新排列。 例如，ℓ = 7 和 r = 3 产生 21 个不同配对的全长循环。 该算法捕获了这种最大行为，因为 gcd(7, 3) = 1，使乘积保持不变。
