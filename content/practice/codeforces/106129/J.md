 ## 解决方案
 ## 问题理解

 我们有一个系统，其中发送者接收二进制字符串，并且必须通过非常不可靠的通道传输它。 通道不保留传输数据包的边界。 相反，到达的是实际发送的循环移位。 换句话说，接收者以相同的顺序看到相同的字符，但字符串已在未知的位置旋转。 

编码步骤允许我们将原始二进制字符串转换为另一个相同长度的字符串，但这次是三个符号的字母表。 在解码步骤中，我们得到该编码字符串的任何循环旋转，并且我们必须准确地恢复原始二进制消息。 

关键的困难是我们失去了一致性。 我们不知道编码消息从哪里开始，因此任何解决方案都必须对循环移位具有鲁棒性。 这立即排除了任何依赖于字符串中绝对位置的编码，除非可以从字符串本身恢复这些位置。 

约束 n 高达 100000 意味着我们需要线性或近线性构造。 任何涉及显式检查所有旋转或尝试通过对所有移位进行强力模式匹配来对齐的操作都会太慢，因为与所有 n 次旋转进行比较将导致 O(n^2) 行为。 

当考虑如何“锚定”绳子时，会出现一种微妙的边缘情况。 如果我们尝试用特殊符号标记一个位置，然后相对于它进行重建，一个天真的想法就是假设该位置是固定的。 然而，该频道打破了这一假设。 标记仍然存在，但其索引被任意移动，因此它不能表示绝对原点，除非字符串的结构使该原点可以以旋转不变的方式恢复。 

另一个重要的边缘情况是 n = 1。在这种情况下，旋转是微不足道的，但编码仍然需要生成正确解码的有效三元字符串，因此任何依赖于拆分为多个部分的构造都必须显式处理此问题。 

## 方法

 一种直接的暴力想法是尝试接收到的字符串的所有可能的旋转，并尝试将每个旋转解释为候选编码，然后检查哪一个对应于二进制消息的有效解码。 这立即会遇到最坏情况的 O(n^2) 解决方案，因为每次旋转都需要 O(n) 工作来解释和验证，并且有 n 次旋转。 

关键的观察是我们实际上并不需要尝试所有的轮换。 相反，我们可以设计编码，使循环结构包含一个独特的、可识别的锚点，使我们能够恢复字符串的一致线性化，无论它如何旋转。 一旦我们能够恢复一致的线性顺序，问题就简化为二进制字符串和固定长度的三元字符串之间的确定性编码和解码。 

关键的想法是使用三进制字母表的额外容量。 长度为 n 的二进制字符串有 2^n 种可能的值，而长度为 n 的三元字符串有 3^n 种可能的值。 即使考虑了循环等价（n 次旋转），仍然有足够的空间将所有二进制字符串唯一地嵌入到三元字符串的旋转类中。 这使我们能够以结构化方式对二进制消息进行编码，并在标准化后恢复它​​。 

我们构造编码字符串，使其包含一个独特的结构标记，可以在旋转中幸存下来，并且可以在 O(n) 时间内找到。 该标记允许我们将接收到的字符串旋转回规范形式。 规范化之后，解码就变成了直接逆映射。

 | --- | --- | --- | --- |
 | 暴力旋转 | O(n^2) | O(n^2) | O(n) | 太慢了|

 ## 算法演练

我们构建一种编码，在循环字符串中创建单个明确的锚点，并使用剩余位置来存储二进制消​​息的无损表示。 

1. 将二进制字符串解释为 [0, 2^n) 范围内的整数值 x。 这提供了一种将消息推理为单个数字的紧凑方法。 
2. 使用除以 n 将 x 分成两部分。 定义 p = x mod n 和 r = x // n。 这种分解是可逆的，因为 x = r * n + p 并且每个有效对 (r, p) 对应于允许范围内的唯一 x。 
3. 构造一个长度为 n 的三进制字符串，其中恰好有一个位置为特殊符号“2”保留，所有其他位置都以二进制形式存储 r 的位。 ‘2’的位置是p，剩下的n-1个位置按顺序填充r的位。 
4. 发送该三进制字符串。 频道将任意旋转它。 
5. 解码时，扫描接收到的字符串并找到唯一出现的“2”。 这个位置成为锚点。 旋转字符串，使锚点位于索引 0 处，从而生成规范表示。 
6. 将剩余的 n − 1 个符号读取为二进制数 r，因为它们只包含“0”和“1”。 
7. 将原始整数重构为 x = r * n + p，其中 p 是规范表示中的锚点位置。 
8. 将 x 转换回 n 位二进制字符串并输出。 

重要的想法是，单个“2”充当同步点，在旋转下保持不变，并允许我们定义一致的解码原点。 

### 为什么它有效

 正确性依赖于这样一个事实：编码的字符串恰好包含一个不属于二进制有效负载的符号。 这保证了任何循环移位仍然恰好包含一个可识别的锚点。 一旦找到锚点，将其旋转到前面就会为每个旋转类产生独特的规范形式。 

其余位置形成 r 的固定长度二进制表示，一旦字符串对齐，它就不受旋转的影响。 由于分解 x = r * n + p 在允许的范围内是双射的，因此恢复 r 和 p 可以唯一确定原始消息。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def encode(n, s):
    x = int(s, 2)
    p = x % n
    r = x // n

    bits = list(bin(r)[2:])
    bits = ['0'] * (n - 1 - len(bits)) + bits

    res = ['0'] * n
    j = 0

    for i in range(n):
        if i == p:
            res[i] = '2'
        else:
            res[i] = bits[j]
            j += 1

    return ''.join(res)

def decode(n, s):
    s = list(s)
    p = s.index('2')

    # rotate so that '2' is at front
    s = s[p:] + s[:p]

    bits = []
    for i in range(1, n):
        bits.append(s[i])

    r = int(''.join(bits), 2)

    x = r * n + p
    return bin(x)[2:].zfill(n)

op = input().strip()
n = int(input())
s = input().strip()

if op == "Encode":
    print(encode(n, s))
else:
    print(decode(n, s))
```在编码中，我们明确地将消息分为商和余数形式。 余数决定了我们放置唯一标记的位置，商填充字符串的其余部分。 这确保了完全的单射性和稳定的解码锚。 

在解码过程中，唯一的结构工作是定位标记并旋转。 之后的一切都是简单的二进制重建。 

## 工作示例

 ### 示例 1

 假设 n = 5，二进制字符串为 s = 10110。 

我们计算 x = 22。然后 p = 22 mod 5 = 2 和 r = 22 // 5 = 4，即二进制 100。 

我们将“2”放在位置 2 并将 r 的位分配到其余位置：

 | 步骤| 数组状态 |
 | --- | --- |
 | 开始 | _ _ _ _ _ |
 | 将 2 置于 p=2 | _ _ 2 _ _ |
 | 填充 r 位 | 1 0 2 0 0 | 1 0 2 0 0

 现在假设频道将其旋转到类似 00210 的值。 

解码器找到“2”，将其旋转回到前面，读取r = 100，恢复x = 4 * 5 + 2 = 22，并重建10110。 

该迹线显示旋转如何仅影响对齐，而不影响可恢复性。 

### 示例 2

 令 n = 4 且 s = 0110。 

那么 x = 6，p = 6 mod 4 = 2，r = 6 // 4 = 1（二进制 01）。 

我们建造：

 | 步骤| 数组状态 |
 | --- | --- |
 | 开始 | _ _ _ _ |
 | 地点 2 | _ _ 2 _ |
 | 填充 r | 0 1 2 0 | 0 1 2 0

 旋转后，假设我们收到2010。解码器找到唯一的“2”，旋转，提取r = 01，并重建x = 1 * 4 + 2 = 6，返回0110。

 ## 边缘情况

 对于 n = 1，该构造会退化，因为没有剩余位置来存储 r 位。 在这种情况下，二进制字符串只有两个可能的值，因此编码器可以直接映射 0 →“0”和 1 →“1”，或者由于旋转不执行任何操作，因此可以简单地放置标记策略。 解码器仍然有效，因为单个字符始终是锚点。 

对于 r 恰好需要 n − 1 位的字符串，填充步骤可确保二进制表示适合而不会溢出。 除以 n 可以保证 r 对于这种固定宽度表示总是足够小，因此不会发生截断。 

如果标记位于位置 0，则旋转不执行任何操作，并且解码过程相同。 如果处于任何其他位置，旋转步骤会恢复一致的视图，因此不需要特殊的外壳。
