 ## 解决方案
 ## 问题理解

 我们得到一个字符串和一个附加到每个字符位置的整数权重。 我们可以删除字符，每次删除都会花费被删除位置的权重，因为它最初出现在字符串中。 删除后，我们按顺序查看剩余的字符，并且我们希望确保结果字符串不包含“hard”作为子序列。 

关键点是我们并不是试图禁止“hard”作为子字符串，而是作为子序列。 这使得结构更加宽松：我们只需要四个字符`h → a → r → d`按顺序出现在剩余字符串中的某个位置，不一定是连续的。 我们的目标是删除字符，以便不存在这样的有序四元组，同时最小化删除成本的总和。 

字符串长度最多为 100,000，每个字符都有关联的成本。 任何尝试显式枚举子集或位置的解决方案都将立即失败，因为$2^n$行为是不可能的。 即使对所有子串进行二次 DP 也太慢了。 该结构表明线性或$O(n \cdot k)$接近哪里$k$很小。 

一个微妙的问题来自于成本定义：删除字符的成本取决于其原始索引，而不是其先前删除后的位置。 这消除了删除之间的任何交互，因此我们可以独立地处理每个删除，这对于动态编程的正确性很重要。 

值得注意的一个边缘情况是当字符串已经不包含`h`,`a`,`r`， 或者`d`按可用顺序。 在这种情况下，答案为零，因为不需要删除。 另一种是当角色高度集中时，例如`"hhharddd"`，其中存在许多重叠的“hard”子序列，我们必须仔细选择要删除的字符，而不是贪婪地删除所有出现的一个字母。 

## 方法

 一个直接的想法是考虑所有删除字符的方法并检查“hard”是否仍然作为子序列。 这将涉及选择索引的子集、在线性时间内验证子序列条件以及对删除成本求和。 即使我们尝试修剪，在最坏的情况下子集的数量仍然呈指数增长，因此这种方法不可行。 

禁止模式的结构提出了不同的观点。 我们不是考虑要删除哪些字符，而是考虑如何在扫描字符串时跟踪子序列“hard”的形成。 在该过程中的任何时刻，有关前缀的唯一相关信息是我们已经将多少个“hard”字符匹配为子序列。 这自然会导致表示模式进展的动态编程状态。 

我们定义一个状态，表示处理字符串前缀后删除的最小成本，同时将某个“hard”前缀匹配为剩余内容中的子序列。 当我们处理一个新字符时，我们要么删除它（支付其成本并保持状态不变），要么保留它，在这种情况下，它可能会根据字符推进匹配的前缀。 

重要的结构见解是匹配子序列仅取决于相对顺序，因此转换是单调的：一旦我们匹配了`h`，我们永远不会“不匹配”它。 这使得 DP 小而稳定，只有四个有意义的状态。

 ## 算法演练

 我们将问题解释为维持我们在剩余字符串中形成子序列“hard”的距离，同时决定删除哪些字符。 

1.定义四种DP状态，分别对应有多少个“hard”字符已经被匹配：0表示没有匹配到，1表示已匹配`h`, 2 表示`ha`, 3 表示`har`，4表示禁止子序列已经形成。 最后一种状态是不可取的，在最佳解决方案中将被避免。 
2. 在处理状态 0 具有成本 0 且所有其他状态具有无限成本的字符串之前初始化 DP。 这反映出我们一开始就没有删除，也没有匹配的子序列。 
3. 从左到右处理字符。 在每个位置，我们决定是否删除或保留该角色。 如果我们删除它，我们会添加它的成本并保持所有 DP 状态不变。 
4. 如果我们保留角色，我们会根据当前状态和角色本身更新转换。 例如，从状态 0 开始，看到`h`允许转换到状态 1； 看到其他字母会使状态保持不变。 
5. 对于状态 1，如果我们看到`a`，我们可以前进到状态2； 否则我们停留在状态 1。同样的逻辑继续：状态 2 前进到 3`r`，并且状态 3 前进到 4`d`。 
6. 我们确保转换永远不会明确允许在最终答案中达到状态 4。 如果保留一个字符将完成“困难”，则该选项将被丢弃，因为它违反了最终字符串必须简单的要求。 
7. 处理完所有字符后，答案是状态 0 到 3 中的最小 DP 值，因为状态 4 表示无效配置。 

正确性取决于这样一个事实：在任何前缀上，决定未来可行性所需的唯一信息是到目前为止有多少“hard”已作为子序列进行匹配。 任何两个达到相同状态的历史对于未来的决策都是等价的，因此 DP 压缩是有效的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    s = input().strip()
    a = list(map(int, input().split()))

    INF = 10**18
    dp = [0, INF, INF, INF]  # states: '', 'h', 'ha', 'har'

    for i in range(n):
        ndp = [INF, INF, INF, INF]

        # delete current character
        for j in range(4):
            ndp[j] = min(ndp[j], dp[j] + a[i])

        c = s[i]

        for j in range(4):
            if dp[j] >= INF:
                continue

            if j == 0:
                if c == 'h':
                    ndp[1] = min(ndp[1], dp[0])
                else:
                    ndp[0] = min(ndp[0], dp[0])

            elif j == 1:
                if c == 'a':
                    ndp[2] = min(ndp[2], dp[1])
                elif c == 'h':
                    ndp[1] = min(ndp[1], dp[1])
                else:
                    ndp[1] = min(ndp[1], dp[1])

            elif j == 2:
                if c == 'r':
                    ndp[3] = min(ndp[3], dp[2])
                elif c == 'h':
                    ndp[2] = min(ndp[2], dp[2])
                else:
                    ndp[2] = min(ndp[2], dp[2])

            elif j == 3:
                if c == 'd':
                    continue
                else:
                    ndp[3] = min(ndp[3], dp[3])

        dp = ndp

    print(min(dp))

if __name__ == "__main__":
    solve()
```DP 数组跟踪我们构建禁止子序列的进度。 删除转换始终有效，并且只会累积成本。 keep 转换通过禁止角色从状态 3 进行最终转换来小心地防止完成“困难”`d`。 

一个常见的陷阱是成本索引处理不当：由于成本与原始头寸相关，因此我们总是使用`a[i]`直接进行，无需进行删除调整。 另一个微妙的问题是忘记多个 DP 状态可以共存并且必须在每一步独立更新，否则您将失去最佳组合。

 ## 边缘情况

 长度为 1 或 2 的最小字符串永远不会包含“hard”作为子序列，因此最佳成本始终为零，因为 DP 永远不会达到状态 3。如果贪婪删除策略试图“安全”而不是跟踪实际的子序列进度，则它会错误地删除不必要的字符。 

完全匹配的字符串，例如`"hard"`证明了阻止完成而不是在完成后做出反应的必要性。 DP 明确阻止转换到禁止的终端状态，确保算法永远不会接受无效的配置。
