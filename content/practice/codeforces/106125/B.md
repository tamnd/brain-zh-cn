 ## 解决方案
 ## 问题理解

 我们得到一个瓶子，里面最初装有两种液体：酒精和其他液体。 随着时间的推移，这两个部分都会蒸发，但每天以不同的恒定速率蒸发。 瓶子打开固定天数后，我们想要计算剩余液体中酒精的比例，以百分比表示。 

更具体地说，瓶子的状态从初始酒精体积开始`a`和初始非酒精体积`o`。 每天，酒精含量都会减少`∆a`而另一种液体则减少`∆o`。 后`d`天，剩余量是时间的线性函数，我们被要求计算剩余量中的酒精比例。 

输出是浮点百分比：$$\text{answer} = \frac{a - d \cdot \Delta a}{(a - d \cdot \Delta a) + (o - d \cdot \Delta o)} \cdot 100$$约束允许值高达$10^{12}$和天数$10^6$，所以中间值可以达到大约$10^{18}$。 这在 64 位整数范围内是安全的，但如果我们不注意操作顺序，则在浮点整数精度内就不安全。 

一个微妙的点是这个问题保证瓶子在之后不是空的`d`天。 这确保了分母始终严格为正，因此我们不需要处理被零除的情况。 

可能破坏简单解决方案的边缘情况通常来自精度和溢出：

 一种失败模式是计算 32 位整数的中间值。 例如，如果`a = 10^12`,`d = 10^6`,`∆a = 10^6`， 然后`d * ∆a = 10^12`，它已经溢出 32 位但适合 64 位。 任何过早截断的语言或方法都会悄然崩溃。 

另一种失败模式是过早转换为浮点。 如果我们计算`(a - d*∆a)`和`(o - d*∆o)`由于在减法之前浮点数，我们会失去整数精度，最终的比率可能会超出所需的范围$10^{-6}$宽容。 

最后一种微妙的情况是一种成分完全蒸发。 例如，酒精可能会达到零，而其他液体仍然存在。 该公式仍然有效，但逐步除法或提前标准化的简单代码可能会产生不正确的中间无穷大或 NaN。 

## 方法

 直接解释就是简单的模拟。 我们计算剩余的酒精和剩余的其他液体`d`天，然后计算它们的比率。 这是正确的，因为蒸发是线性的并且对于两种组分而言是独立的。 

暴力方法会减去`∆a`和`∆o`一天一次。 那需要`d`迭代，并且自从`d`可以达到$10^6$，这已经是临界点，但仍然是可能的。 然而，如果约束更大（就像类似问题中的典型情况），这将立即失败。 无论如何，问题的结构使得它没有必要，因为这两个过程都是算术级数。 

关键的观察是，之后`d`天，每个数量的计算时间为 O(1)：$$a' = a - d \cdot \Delta a,\quad o' = o - d \cdot \Delta o$$一旦我们直接计算这些最终值，答案只是一个单一的比率计算。 酒精和其他液体之间没有相互作用，因此没有动态过程，每一步都没有舍入，也不需要模拟。 

唯一真正需要注意的是数字安全：所有中间算术都应该以 64 位整数或 Python 整数完成，并且浮点转换应该仅在最终除法时发生。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力破解（日常模拟）| O(d) | O(1) | O(1) | 一般形式太慢|
 | 直接公式计算 | O(1) | O(1) | O(1) | O(1) | 已接受 |

 ## 算法演练

 1.读取天数`d`, 初始酒精`a`、及其他液体`o`。 它们定义了两个独立的线性衰减过程的初始状态。 
2. 读取蒸发率`∆a`和`∆o`。 这些代表每天不断减少，因此系统线性发展，组件之间没有相互作用。 
3. 计算之后的剩余酒精含量`d`天作为`a' = a - d * ∆a`。 这是有效的，因为每天都会精确删除`∆a`，所以之后`d`天我们一步减去总损失。 
4. 类似地计算剩余的其他液体`o' = o - d * ∆o`。 
5. 计算剩余液体总量为`t = a' + o'`。 这是对比率分母唯一重要的数量。 
6. 将酒精百分比计算为`100 * a' / t`。 这直接符合百分比构成的定义。 
7. 使用浮点格式以足够的精度打印结果，以保证误差在$10^{-6}$。 

### 为什么它有效

 该过程由两个独立的时间线性函数控制。 在每个时间步，系统状态完全由下式决定：`(a - t∆a, o - t∆o)`。 两个组件之间没有耦合，因此跳过中间步骤不会丢失任何信息。 最终比率仅取决于端点值，使得直接评估等同于完整模拟。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def main():
    d = int(input())
    a, o = map(int, input().split())
    da, do = map(int, input().split())

    a_rem = a - d * da
    o_rem = o - d * do

    total = a_rem + o_rem
    ans = 100.0 * a_rem / total

    print(ans)

if __name__ == "__main__":
    main()
```该解决方案依赖于直接计算最终剩余数量。 乘法`d * da`和`d * do`必须在减法之前完成，但在 Python 中是安全的，因为整数是无界的。 最后只执行一次除法，这样可以避免累积浮点错误。 

一个微妙的实现细节是在最后一步之前将分子和分母保持为整数。 这保留了最大精度。 到浮点数的转换被延迟到最终比率计算。

 ## 边缘情况

 一种重要的边缘情况是蒸发率为零。 对于像这样的输入`d = 100`,`da = 0`,`do = 0`，公式减少到原来的比例。 该算法自然地处理这个问题，因为`d * 0 = 0`，因此不会发生减法并且状态保持不变。 

另一种情况是一种成分完全蒸发。 认为`a = 10`,`da = 2`,`d = 5`。 然后`a' = 0`。 该算法仍然有效，因为分子变为零，而分母通过问题保证保持为正值。 

更微妙的情况是减法产生非常大的中间值，接近 64 位整数的限制。 既然两者`d`费率最高可达$10^{12}$，他们的产品可以达到$10^{18}$。 该算法通过依赖 Python 整数来避免溢出，但在 C++ 等语言中，这需要 128 位中间存储或在乘法之前仔细转换。
