 ## 解决方案
 ## 问题理解

 我们得到三个由字符组成的字符串`1`,`2`， 和`3`。 

隐藏的物体是游戏获胜者的真实序列。 如果玩家`1`赢得一场比赛，然后是玩家`2`和`3`写下`1`，但是玩家`1`忘记录制那场比赛。 同样的规则对称地适用于其他玩家。 

因此，每个输入字符串都是完整的获胜者序列，删除了一个特定字符：

 -`s1`是所有的获胜者序列`1`已删除。 
-`s2`是所有的获胜者序列`2`已删除。 
-`s3`是所有的获胜者序列`3`已删除。 

任务是重建原始的获胜者序列。 该声明保证答案存在并且是唯一的。 

字符串的长度可达`10^5`，因此隐藏序列本身也可以具有该比例的长度。 任何重复扫描字符串、执行昂贵的子字符串操作或大量回溯的算法都会太慢。 我们需要接近线性时间的东西。 

一个微妙的方面是，获胜者的每次出现恰好出现在三个字符串中的两个字符串中。 例如，如果下一个获胜者是`2`，那么`2`出现在`s1`和`s3`，但不在`s2`。 恢复序列意味着根据三个字符串当前未读的位置来确定下一个必须出现的字符。 

考虑一个小例子：```
s1 = 2
s2 = 1
s3 = 21
```答案是：```
21
```第一个获胜者是`2`。 那`2`出现在前面`s1`和`s3`。 从这两个字符串中消耗完之后，剩余的未读字符是`""`,`"1"`， 和`"1"`，所以下一个获胜者一定是`1`。 

一个常见的错误是尝试在任意字符串对之间贪婪地匹配字符而不考虑顺序。 例如：```
s1 = 23
s2 = 13
s3 = 12
```答案是：```
123
```一开始，所有三个字符串都以不同的字符开头。 只看频率没有任何线索。 关键的观察是，恰好有两个字符串必须就下一个获胜者达成一致。 

当同一个获胜者连续多次出现时，就会出现另一个简单的陷阱：```
s1 = 2222
s2 = ""
s3 = 2222
```这将对应于隐藏序列`2222`。 任何重建方法都必须连续多次正确地使用匹配的前缀，而不是尝试在不同的获胜者之间交替。 

## 方法

 暴力重建将尝试通过测试所有可能性来确定下一个获胜者。 

假设我们在三个字符串中都有当前位置。 我们可以尝试附加`1`,`2`， 或者`3`到答案并检查选择是否与字符串一致。 选择获胜者后，应包含该获胜者的两个字符串前进一位。 

这种方法是正确的，因为隐藏序列的每一步都对应于三个可能的获胜者之一。 不幸的是，递归地探索可能性会导致指数搜索树。 即使经过修剪，长度可达`10^5`使这样的策略变得不可能。 

关键的观察是，下一个获胜者实际上是被迫的。 

获取每个字符串的第一个未使用的字符。 如果下一个获胜者是`1`，然后是下一个未读字符`s2`和`s3`必须两者都是`1`。 同样，获胜者`2`接下来将出现在`s1`和`s3`，以及获胜者`3`接下来将出现在`s1`和`s2`。 

因为保证答案是唯一的，所以在每一步中，只有一个字符作为当前字符出现在两个字符串中。 这个角色一定是下一个获胜者。 输出后，我们将这两个字符串的指针前进并重复。 

另一种观察方式是从末端观察。 第一场比赛的获胜者恰好出现在两个列表中的第一位。 一旦我们确定了获胜者，我们就会将其从这两个列表中删除。 同样的论点也适用于其余的后缀。 这产生了一个简单的基于指针的线性算法。 官方的解决方案正是使用了这一观察结果。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力搜索 | 指数| 指数| 太慢了|
 | 三分重建| O(n) | 除了输出 | O(1) 已接受 |

 ## 算法演练

 1.读取三个字符串`s1`,`s2`， 和`s3`。 
2. 维护三个指标`i1`,`i2`， 和`i3`，最初为零。 每个索引都指向其字符串的第一个未读字符。 
3. 当至少一个字符串仍有未读字符时，查看指针未到达末尾的每个字符串的当前字符。 
4. 统计其中的哪个字符`1`,`2`， 和`3`恰好出现在两个字符串的前面。 
5. 该角色是隐藏序列中的下一个获胜者。 将其附加到答案中。 
6. 将当前字符等于所选获胜者的两个字符串的指针向前推进。 
7. 重复此操作，直到所有字符串中的所有字符都已被消耗。 

### 为什么它有效

 获胜者的每次出现都恰好出现在两个列表中，并且不存在于获胜者自己的列表中。 

假设下一个隐藏的获胜者是`x`。 那么属于其他玩家的两个列表中的下一个未读字符必须都是`x`，因为这些列表保留了获胜者的原始顺序。 第三个列表根本不包含这种情况。 

所以在任何时刻，下一个获胜者正是恰好出现在两个字符串的当前位置的字符。 使用这两个字符串中的该字符将从所有表示中删除下一个游戏的一次出现。 剩余的后缀对应于剩余的游戏，因此同样的论证适用于归纳。 该算法以正确的顺序重建每个获胜者，并且不能选择错误的字符。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    s = [input().strip() for _ in range(3)]

    p = [0, 0, 0]
    ans = []

    while True:
        cur = []

        for k in range(3):
            if p[k] < len(s[k]):
                cur.append((k, s[k][p[k]]))

        if not cur:
            break

        cnt = {'1': 0, '2': 0, '3': 0}

        for _, ch in cur:
            cnt[ch] += 1

        winner = None
        for ch in "123":
            if cnt[ch] == 2:
                winner = ch
                break

        ans.append(winner)

        for k, ch in cur:
            if ch == winner:
                p[k] += 1

    print("".join(ans))

solve()
```实现直接反映了证明。 

数组`p`存储每个字符串中当前未读的位置。 在每次迭代中，我们仅检查当前指向的字符，从不向后扫描或向前搜索。 

字典`cnt`记录当前有多少个字符串以每个可能的获胜者开头。 由于只存在三个符号，因此这项工作是恒定时间的。 

确定获胜者后，我们将当前字符等于该获胜者的指针精确推进。 这是关键的一步。 推进任何其他指针都会破坏列表保留的排序信息。 

每个指针最多向前移动其字符串的长度。 指针不会向后移动，因此总工作量是线性的。

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 | O(n) | 每个指针仅向前前进，并且每个字符被处理一次 |
 | 空间| O(n) | 答案字符串长度为n，辅助内存为O(1) |

 如果隐藏序列有长度`n`，那么三个输入字符串一起也只包含`2n`人物。 每个字符都会导致一个指针递增一次，因此该算法可以轻松处理大小的输入`10^5`。

 ## 边缘情况

 考虑：```
2
1
21
```最初，前线是`(2,1,2)`。 人物`2`出现两次，所以算法输出`2`并推进第一和第三指针。 其余战线变为`(end,1,1)`, 迫使获胜者`1`。 输出是`21`，这是正确的。 

现在考虑屡次获胜的人：```
22
1
221
```前面是`(2,1,2)`，所以获胜者`2`被选中。 推进两个指针后，锋线再次出现`(2,1,2)`。 该算法正确地选择了另一个`2`。 只有这样才可以`1`成为出现两次的角色。 结果是`221`。 

一个更有趣的案例是：```
23
13
12
```所有三个字符串都以不同的符号开头。 对整个字符串进行基于频率的重建将会失败。 仅查看当前未读位置即可得出计数`{1:2, 2:1, 3:0}`, 迫使获胜者`1`。 重复相同的逻辑重构`123`确切地。 

这些例子说明了为什么局部“恰好出现在两个当前位置”不变量足以唯一地恢复整个序列。
