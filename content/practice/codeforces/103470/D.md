 ## 解决方案
 ## 问题理解

 给定一个整数数组，然后我们重复应用一个非常不寻常的排序过程。 该过程不会像冒泡排序那样比较相邻元素，而是比较每对位置。 对于每个有序索引对，如果第一个索引处的元素小于第二个索引处的元素，则交换这两个值。 这是在所有位置上的双循环中完成的，因此每个元素都可以与其他元素多次交互。 

我们需要的关键数量不是最终排序的数组，而是该过程在数组的每个前缀上运行时执行了多少次交换。 对于从 1 到 n 的每个 k，我们获取前 k 个元素，运行这个完整的过程，并计算整个过程中执行的交换次数。 

输出是所有前缀的交换计数的序列。 

这些约束意味着每个测试用例 n 最多可达 10^5，测试中的总 n 最多可达 10^6。 排序过程的直接模拟将为每个前缀执行最多 n2 次比较，如果对每个前缀独立完成，则可能执行 n3 总工作。 即使单次运行也已经是 O(n²)，这在规模上太慢了。 该解决方案必须将每个测试用例的问题减少到更接近线性或近线性。 

当元素相等时，会出现微妙的边缘情况。 由于条件严格为 ai < aj，因此相等的值永远不会触发交换。 这很重要，因为重复项在此过程中表现为稳定的块。 例如，对于数组 [2, 2, 1]，前两个元素永远不会相互交换，但都与 1 交互。任何假设严格排序而不单独处理重复项的方法都会错误地计算交换贡献。 

另一个微妙的情况是增量前缀。 一个幼稚的实现可能会尝试独立地为每个前缀重新运行整个排序过程，但是中间工作不能以明显的方式重用，因为交换大量地对数组进行了重新排序。 关键的困难在于，处理前缀 k 后的状态不是前缀 k−1 的简单扩展，除非我们了解该过程在全局范围内实际执行的操作。 

## 方法

 暴力解释遵循字面算法。 对于每个前缀，我们模拟嵌套循环，重复扫描所有对 (i, j) 并在 ai < aj 时进行交换。 每次运行都需要 O(k²) 次比较，并且可能需要 O(k²) 次交换。 对所有前缀求和，在最坏的情况下变成 O(n3)，因为我们从头开始重新计算，并且每次运行本身都是二次的。 对于 n 达到 10^5 来说这是不可行的。 

关键的见解是停止将其视为动态交换过程，而是重新解释每个元素对交换计数的贡献。 

修复前缀并查看单个元素 x。 每次 x 参与交换时，它都必须与某个元素 y 配对，其中比较时 x < y。 该算法的结构确保最终较大的元素通过交换反复“拉”较小的元素向前。 每对元素都会贡献可预测的交换次数，仅取决于它们的相对顺序和交互频率。 

如果我们按照价值递增的顺序处理元素，我们可以计算出有多少先前看到的元素更大或更小，并逐渐累积贡献。 这将问题转化为维护值的频率结构，其中每个新元素根据有多少先前元素大于它来贡献交换。

这减少了维护频率数组上的前缀统计信息的任务，通常使用 Fenwick 树或线段树来处理。 每个新元素贡献的先前插入元素的数量严格大于它，并且还通过前缀聚合间接影响未来的计算。 每个前缀的总交换计数成为动态增长的多重集上类似反转贡献的累积。

 ## 算法演练

 我们将交换过程重新解释为计算每个元素与先前见过的元素参与“无序交互”的次数。 每个前缀都是增量处理的。 

1. 我们在值上维护一个 Fenwick 树（或类似的结构），跟踪每个值到目前为止在当前前缀中出现的次数。 该结构支持对数时间的前缀和。 
2. 我们从左到右迭代数组。 在位置 k 处，我们将 a[k] 插入结构中。 在插入之前，我们查询之前插入的元素有多少个严格大于a[k]。 该值被添加到前缀 k 的答案中。 
3. 我们维持捐款总额。 前缀 k 的答案是先前的答案加上 a[k] 的贡献。 
4. 为了计算“较大元素的数量”，我们使用total_so_far − prefix_sum(a[k])。 这是有效的，因为 prefix_sum 给出的元素计数 ≤ a[k]。 
5. 我们随时存储每个前缀结果，因为每个前缀都建立在前一个前缀的基础上。 

关键点是每次插入仅依赖于之前的元素，因此我们永远不需要重建较早的前缀。 

### 为什么它有效

 交换过程有效地迫使每对元素进行重复比较，直到它们的相对顺序稳定为止。 每当比较循环中较小的元素遇到较大的元素时，就会发生交换。 在整个执行过程中，每对都会贡献固定数量的交换，该数量仅由它们的顺序和在前缀中的存在决定。 

这使得总交换计数相当于对每个元素求和，即有多少个先前见过的元素更大。 这正是 Fenwick 树所维护的反转结构。 因为每个前缀只是前一个前缀的扩展，所以这些贡献会持续累积，而无需重新处理过去的交互。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class Fenwick:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)

    def add(self, i, v):
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

def solve():
    t = int(input())
    for _ in range(t):
        n = int(input())
        a = list(map(int, input().split()))

        fw = Fenwick(n)
        total = 0
        res = []

        for x in a:
            greater = total - fw.sum(x)
            total += 1
            fw.add(x, 1)
            res.append(str(greater))

        print(" ".join(res))

if __name__ == "__main__":
    solve()
```芬威克树存储迄今为止看到的值的频率。 每个步骤使用总计数减去前缀和来计算有多少先前元素超过当前值。 

一个常见的实现陷阱是忘记问题约束中的值是 1 索引的，这允许直接索引到 Fenwick 树中。 另一个微妙的问题是操作的顺序：查询必须在插入当前元素之前发生，否则元素将错误地计算自身。

 ## 边缘情况

 对于像[7]这样的单元素输入，芬威克树是空的并且不存在先前的元素，因此贡献为零并且在整个计算过程中保持为零。 

对于完全递减序列 [5, 4, 3, 2, 1]，每个元素贡献的数量恰好等于其之前看到的较大元素的数量。 第一个元素贡献 0，第二个元素贡献 1，第三个元素贡献 2，依此类推。 该算法会正确累积这些元素，因为每个查询都会捕获插入前大于当前元素的所有先前元素。 

对于具有重复值的数组，例如 [2, 2, 2]，每个前缀查询都返回零，因为条件严格为 ai < aj，因此相等的值永远不会产生交换。 Fenwick 结构正确区分了“严格大于”和“大于或等于”，确保重复项不会夸大计数。
