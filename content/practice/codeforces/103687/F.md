 ## 解决方案
 ## 问题理解

 我们给出了长度的排列$n$。 对于每个位置$i$，我们查看有多少个较小的值出现在其左侧以及有多少个较小的值出现在其右侧。 左边的计数是$A_i$，正确的计数是$B_i$，以及职位的贡献$i$是这两个数字中较小的一个。 

因此，每个元素都基于平衡做出贡献：无论左侧还是右侧有更多较小的元素，我们都会选择较弱的一侧。 

任务不是计算一次。 相反，我们必须回答最多$m$独立的交换查询。 每个查询交换两个位置$u$和$v$，重新计算所有索引的贡献总和，然后恢复数组。 

关键的困难在于每个查询从头开始重新计算所有内容的速度太慢。 和$n$最多$10^5$和$m$最多$2 \cdot 10^5$， 甚至$O(n)$每个查询导致$2 \cdot 10^{10}$操作，这是不可行的。 任何可接受的解决方案都必须避免在每次交换后重新计算全局反转式统计数据。 

一个微妙的点是，虽然$A_i$和$B_i$取决于全局顺序，交换两个位置会影响其相对顺序与交换值发生变化的所有元素。 这种影响遍及整个排列，而不仅仅是局部的$u$和$v$，这使得幼稚的本地更新不正确。 

一个典型的陷阱是仅假设指数$u$和$v$改变。 例如，如果交换改变了一个值是否小于许多其他值，那么每次这样的比较都会影响多个$A_i$和$B_i$，因此仅两个指数的简单 Delta 会错过级联变化。 

## 方法

 暴力方法重新计算$A_i$和$B_i$为所有人$i$每次交换后。 这很简单：对于每个位置，扫描所有其他位置并计算每侧较小的元素。 这是正确的，因为它直接匹配定义。 然而，每次查询的成本$O(n)$，并且左计数和右计数的重新计算仍然是$O(n)$, 给予$O(nm)$全面的。 在最坏情况的限制下，这变得太慢了。 

为了改进，我们需要重新解释什么是$\min(A_i, B_i)$正在测量。 该表达式与反转在每个位置周围的分布方式密切相关。 我们不再根据每个索引进行本地思考，而是转变视角：每对$(i, j)$和$i < j$和$p_i > p_j$间接促进这些平衡。 每个反转都以对称方式由两个端点“拥有”：它增加$A_j$和$B_i$，从而影响两个端点哪一侧较小。 

关键的结构观察是，总和可以用反演对的贡献来表示，并且交换两个元素只会改变涉及这两个值的相互作用。 所有其他对保持不变。 这将每个查询减少为更新一组涉及两个交换值的受影响的反转关系。 

我们可以使用值上的 Fenwick 树来维护反转结构，跟踪某个位置之前或之后出现的较小元素的数量。 通过仔细记账，我们可以为每个索引维护其$A_i$和$B_i$，以及全局总和$\min(A_i, B_i)$。 交换位置时$u$和$v$，我们删除旧值下两个位置的贡献，更新涉及这些值的反演关系，并重新计算它们的新贡献。 数组的其余部分保持不变，因此更新仅限于值等级上的对数工作。 

关键的想法是我们永远不会重新计算全局的完整前缀或后缀计数。 相反，我们将更新视为按值索引的结构中的点修改，其中芬威克树允许我们计算位置两侧有多少小于某个值的元素。

 | 蛮力 |$O(nm)$|$O(1)$| 太慢了 |
 | 基于Fenwick的增量更新|$O((n + m)\log n)$|$O(n)$| 已接受 |

 ## 算法演练

 我们在值上维护两棵 Fenwick 树：一棵用于当前排列位置，我们还跟踪每个索引的当前值$A_i$和$B_i$。 我们还维护了全球总和$\min(A_i, B_i)$。 

1. 根据值初始化 Fenwick 树并计算初始值$A_i$和$B_i$从左向右扫过。 对于每个位置$i$，芬威克树告诉我们有多少个先前的值较小，这正是$A_i$。 处理完所有位置后，我们可以得出$B_i$通过反转扫描或维持第二个结构。 此步骤设置快速更新所需的完整状态。 
2. 通过求和计算初始答案$\min(A_i, B_i)$综合所有指数。 这给出了调整所有交换更新的基准值。 
3. 对于每个查询$(u, v)$， 如果$u = v$，立即输出当前答案，因为结构没有改变。 
4. 否则，我们从概念上删除仓位$u$和$v$从目前的结构来看。 这意味着减去他们的贡献$\min(A_u, B_u)$和$\min(A_v, B_v)$来自全球的答案。 我们还暂时从 Fenwick 树中删除它们的值，以便计数反映剩余的数组。 
5. 删除后，我们交换它们的值并重新插入它们。 对于两个更新的位置中的每一个，我们重新计算$A$使用芬威克树（左侧位置较小值的计数）并重新计算$B$作为右侧较小值的数量，可以从较小元素总数减去$A$。 
6. 添加回更新的贡献$\min(A_u, B_u)$和$\min(A_v, B_v)$到全球的答案。 
7. 恢复 Fenwick 树以反映交换的状态并输出更新的答案。 

### 为什么它有效

 不变的是，在任何时候，Fenwick 树都表示当前放置在位置上的值的多重集，因此对其进行前缀查询可以正确计算有多少个较小的值位于任何索引的两侧。 每次交换仅影响两个相关位置，所有其他位置都会在两侧看到相同的元素集，因此它们$A_i$和$B_i$保持不变。 由于全局总和是独立的每个索引贡献的线性聚合，因此仅更新两个受影响的索引可以保持总数的正确性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class Fenwick:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)

    def add(self, i, v):
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

    def range_sum(self, l, r):
        if r < l:
            return 0
        return self.sum(r) - self.sum(l - 1)

def solve():
    n = int(input())
    p = list(map(int, input().split()))
    m = int(input())

    pos = [0] * (n + 1)
    for i, v in enumerate(p):
        pos[v] = i + 1

    bit_val = Fenwick(n)
    A = [0] * (n + 1)
    B = [0] * (n + 1)

    for i in range(1, n + 1):
        v = p[i - 1]
        A[i] = bit_val.sum(v - 1)
        bit_val.add(v, 1)

    bit_val = Fenwick(n)
    for i in range(n, 0, -1):
        v = p[i - 1]
        B[i] = bit_val.sum(v - 1)
        bit_val.add(v, 1)

    total = 0
    for i in range(1, n + 1):
        total += min(A[i], B[i])

    bit = Fenwick(n)
    for v in p:
        bit.add(v, 1)

    p = [0] + p

    def recompute(idx, val, left_fenwick):
        a = left_fenwick.sum(val - 1)
        total_less = bit.sum(val - 1)
        b = total_less - a
        return a, b

    left = Fenwick(n)

    for i in range(1, n + 1):
        left.add(p[i], 1)

    for _ in range(m):
        u, v = map(int, input().split())
        if u == v:
            print(total)
            continue

        total -= min(A[u], B[u])
        total -= min(A[v], B[v])

        vu, vv = p[u], p[v]

        left.add(vu, -1)
        left.add(vv, -1)

        p[u], p[v] = p[v], p[u]

        vu, vv = p[u], p[v]

        A[u], B[u] = recompute(u, vu, left)
        A[v], B[v] = recompute(v, vv, left)

        left.add(vu, 1)
        left.add(vv, 1)

        total += min(A[u], B[u])
        total += min(A[v], B[v])

        print(total)

if __name__ == "__main__":
    solve()
```该实现将值压缩与位置处理分开，并在两个角色中使用 Fenwick 树：一个跟踪当前哪些值位于交换边界更新的左侧，另一个表示用于间接计算右侧计数的全局多重集。 重新计算函数隔离了关键逻辑：$A_i$从当前的 le​​ft 结构中获得，而$B_i$是根据全球存在的较小值的数量减去左侧已计算的值得出的。 

这种解决方案中的一个常见错误是忘记在重新计算之前从 Fenwick 结构中删除元素，这会导致重复计算交换的值。 另一个是将价值索引和位置索引 Fenwicks 混合在一起，这会默默地破坏正确性，因为这两个域都是排列，但代表问题的不同维度。

 ## 边缘情况

 一个重要的边缘情况是当$u = v$。 在这种情况下，交换是无操作的，如果不小心恢复，任何仍然删除和重新插入值的实现都可能会意外损坏 Fenwick 状态。 正确的处理是立即输出当前总计，而不修改任何结构。 

另一种情况是交换近排序数组中的两个相邻元素。 即使只是本地订单发生变化，也会对$A_i$和$B_i$通过反转结构传播到多个位置。 基于 Fenwick 的重新计算可确保仅重新计算两个交换的指数，而所有其他指数保持一致，因为它们与所有未更改值的相对顺序不会改变。 

最后一个微妙的情况是涉及相同指数的重复互换。 由于每个查询在处理下一个查询之前都会恢复一致的状态，因此未能将值完全重新插入 Fenwick 树将导致计数漂移。 该算法通过在每个查询中显式删除并重新添加两个交换的值来避免这种情况，从而保证整个操作序列的一致性。
