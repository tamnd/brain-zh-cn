 ## 解决方案
 ## 问题理解

 我们正在尝试从起始位置移动一个点$S$到目标位置$T$在 2D 平面上。 运动是连续的且方向不受限制。 正常情况下，角色匀速行走$V_1$。 这意味着欧几里德距离的任何位移$d$需要时间$d / V_1$。 

还有特殊的“滑翔点”。 如果角色触及这些点之一，他们会立即获得暂时的提升：在接下来的 3 秒游戏时间内，他们的移动速度会变为$V_2$， 在哪里$V_2 > V_1$。 3秒后，他们恢复步行速度。 

接触滑行点所消耗的时间不会超出移动到那里所花费的时间。 关键的难点在于滑翔是有时间限制的，并且可以通过访问多个点来多次触发。 我们希望从以下地点出发的最短时间$S$到$T$。 

输入给出最多 1000 个滑翔点的坐标，然后是$S$和$T$，以及两个速度。 

输出是一个实数，精度高达$10^{-6}$，代表最快的旅行时间。 

约束条件$n \le 1000$表明二次关系$n$， 例如$O(n^2)$，是可以接受的。 然而，任何直接模拟连续运动或时间演化的尝试都是不可能的，因为状态包括位置和剩余升压时间，这是连续的。 

一些边缘情况很容易被错误处理：

 如果没有滑翔点且$S$距离$T$，答案一定是直线行走时间$dist(S,T)/V_1$。 任何假设必须存在至少一次滑翔的方法都会失败。 

如果滑翔点正好位于从$S$到$T$，最佳解决方案可能会忽略它或根据几何形状使用它。 像“总是经过滑翔点”这样天真的贪婪选择可能会更糟糕。 

如果滑翔点聚集，链接它们可能会重复刷新 3 秒的提升。 独立处理提升而不跟踪重叠的简单模拟可能会低估或高估总速度优势。 

## 方法

 蛮力观点首先将问题视为具有时间依赖性状态的连续最短路径问题。 从任何一点，您都可以直接移动到$T$，或者首先到达任意滑翔点，到达该点后，您将获得 3 秒的更快移动窗口。 这表明状态包括位置和剩余升压时间，该状态是连续的，因此直接的最短路径搜索是不可行的。 

如果我们尝试仅离散化位置，我们可以观察到任何有用的轨迹仅在滑行点或在$S$和$T$。 这将问题简化为一个图，其中节点就是这些点。 然而，边缘不仅仅是距离除以单个速度，因为速度取决于我们是否以主动提升的方式到达。 

关键的观察结果是，每当我们在两个固定点之间旅行时，沿该路段的最佳策略很简单：要么我们仅使用步行速度，要么我们以最佳方式消耗部分可用的助推力。 如果我们将状态建模为“当前处于有或没有主动提升的节点并且有一些剩余时间”，则该结构变得易于管理。 我们没有跟踪确切的剩余时间，而是利用只有充分利用升压才能实现最佳转换的目的。 

这导致扩展状态图上的最短路径：每个节点都有两种状态，“无活动滑行”和“滑行可用”。 从一个点移动到另一个点总是需要步行时间，但如果我们降落在滑翔点，我们会切换到持续 3 秒加速旅行的增强状态。 在增强状态下，我们可以在单位时间内移动更多的距离。 

我们可以预先计算相关点之间的所有成对距离（滑翔点加上$S$和$T$）。 然后，转换对应于从一个节点到另一个节点的行进，根据我们当前是否处于提升窗口内来更新我们是否可以从滑行时间中受益。 

这将问题简化为图上的最短路径$O(n)$节点和$O(n^2)$边，其中每条边都有一个权重，具体取决于我们是否处于增强模式。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力连续状态搜索 | 无限/棘手| 无限 | 不可能|
 | 具有扩展状态的图最短路径 |$O(n^2 \log n)$|$O(n^2)$| 已接受 |

 ## 算法演练

 我们对待每一个滑行点，加上$S$和$T$，作为完整图中的节点。 我们预先计算每对之间的欧几里德距离。 

然后，我们在扩展的状态空间上运行最短路径算法。 每个状态都由节点索引以及当前是否有可用的活动滑动窗口来定义。 由于滑行持续时间固定为 3 秒，因此我们只需要跟踪过渡时是否处于“滑行可用段”，而不需要在边缘松弛期间连续跟踪剩余时间。 

我们定义两个距离数组：一个用于到达没有主动滑行效果的节点，另一个用于假设我们处于滑行模式下到达该节点。 

我们初始化起始节点$S$没有先前的滑行优势，可以像 Dijkstra 一样放松。 

## 算法演练

 1. 构造一个由所有滑翔点加上的节点列表$S$和$T$。 这使我们能够将问题简化为离散位置之间的转换。 
2. 预先计算每对节点之间的欧几里德距离。 这是必要的，因为每个移动成本完全取决于几何距离。 
3. 定义一个状态为$(i, t)$， 在哪里$i$是一个节点索引并且$t \in \{0,1\}$表明我们当前是否受益于滑动窗口。 第二个部分确定从该状态开始的行驶是否使用速度$V_2$或者$V_1$。 
4.将除起始状态外的所有距离初始化为无穷大$(S, 0)$，设置为 0。 
5. 使用优先级队列重复提取已知时间最小的状态。 
6.来自一个州$(i, t)$，尝试转换到每个其他节点$j$。 计算距离$d = dist(i, j)$。 如果$t = 1$，有效速度为$V_2$旅行时间是$d / V_2$; 否则就是$d / V_1$。 
7. 如果节点$j$是一个滑翔点，那么到达后我们激活滑翔状态以进行未来的转换，将下一个状态设置为$(j, 1)$。 否则就变成$(j, 0)$。 
8. 如果找到更短的时间，则放宽距离，并将更新后的状态推入优先级队列。 
9. 答案是达到的最小值$T$在任一状态下。 

正确性取决于以下事实：每个最优路径都可以分解为离散事件点之间的直线段，唯一的状态影响是滑动窗口在遍历期间是否处于活动状态。 任何与按顺序访问节点的偏差都可以用直线段代替，而无需增加时间。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

import heapq
import math

def solve():
    n = int(input())
    pts = []
    
    for _ in range(n):
        x, y = map(int, input().split())
        pts.append((x, y))
    
    sx, sy, tx, ty = map(int, input().split())
    v1, v2 = map(int, input().split())
    
    S = (sx, sy)
    T = (tx, ty)
    
    nodes = pts + [S, T]
    s_idx = n
    t_idx = n + 1
    m = n + 2
    
    dist = [[0.0] * m for _ in range(m)]
    
    for i in range(m):
        x1, y1 = nodes[i]
        for j in range(m):
            x2, y2 = nodes[j]
            dx = x1 - x2
            dy = y1 - y2
            dist[i][j] = math.hypot(dx, dy)
    
    INF = 1e100
    dp = [[INF] * 2 for _ in range(m)]
    dp[s_idx][0] = 0.0
    
    pq = [(0.0, s_idx, 0)]
    
    while pq:
        cur, i, state = heapq.heappop(pq)
        if cur != dp[i][state]:
            continue
        
        for j in range(m):
            if j == i:
                continue
            
            d = dist[i][j]
            speed = v2 if state == 1 else v1
            t = cur + d / speed
            
            nxt_state = 1 if j < n else 0
            
            if t < dp[j][nxt_state]:
                dp[j][nxt_state] = t
                heapq.heappush(pq, (t, j, nxt_state))
    
    ans = min(dp[t_idx])
    print("{:.12f}".format(ans))

if __name__ == "__main__":
    solve()
```实施首先针对所有相关点（包括起点和目标）构建完整的图表。 距离矩阵使用欧几里德距离，因此以后的每个转换都变成恒定时间查找。 

DP 数组存储每个节点状态对的最佳已知时间。 状态标志表示到达节点是否授予新的滑行效果。 当我们移动到一个滑翔点（任何第一个$n$节点），我们将状态重置为活动； 否则它会变得不活动。 

优先级队列确保我们始终扩展当前最快的已知部分路径，匹配 Dijkstra 的非负边权重的正确性条件。 唯一微妙的细节是边缘权重取决于当前状态，而不是目标状态，这就是为什么在计算转换之前选择速度的原因。

 ## 边缘情况

 当没有滑翔点时，就会出现极端情况。 该算法仍然有效，因为 DP 图仅包含$S$和$T$，并且所有过渡都使用步行速度。 最短路径减少为单边$S \to T$，生产$dist(S,T)/V_1$。 

另一个微妙的情况是，滑翔点距离直接路径很远，但由于高度较高，仍然很有吸引力。$V_2$。 DP 正确评估了两种可能性：直接移动和通过滑行节点绕行，因为两者都是完整图中的显式边。 

当多个滑翔点靠近时，会出现更微妙的情况。 该算法自然地处理重复激活，因为每次访问滑行节点都会重置状态。 即使重新访问同一区域是有益的，最短路径松弛也确保只有改进的序列才能在优先级队列中生存。
