 ## 解决方案
 ## 问题理解

 我们正在模拟一系列线性阶段的进展，其中每个阶段都必须在前进之前被清除。 在任何阶段，一次尝试要么成功，让我们进入下一个阶段，要么失败，让我们保持在同一阶段，但会降低生命值。 

该角色的生命值有限，法力值较小。 仅当尝试失败时，健康状况才会下降。 如果生命值达到零，则视为失败，因此任何明智的策略都必须避免允许这种情况发生。 

一个阶段的每次尝试都会花费一个单位时间。 一次成功的尝试会将我们从阶段 i 移至 i + 1，而失败则使我们保持在 i 阶段，但生命值会减少 1。 除了尝试这个阶段之外，我们还可以花时间使用两种不同的治疗机制来恢复生命值：一种会消耗法力，另一种有特殊限制，只能在受到伤害后立即使用。 

有些阶段包含特殊的“图腾”，可以立即将法力补充到最大值，而无需花费时间。 这意味着法力并不是在全球范围内单调减少； 它可以根据阶段定期重置。 

我们的目标是尽量减少从满生命值和法力值开始达到第 n 阶段所需的预期总时间。 每个阶段的最佳策略并不固定，而是取决于当前的生命值、法力值以及是否提供特殊的伤害后治疗。 

这些约束在结构意义上是严格的，而不是在数字尺度上。 n 最多为 1000，生命值最多为 9，法力最多为 6，状态空间足够小，我们可以在所有配置上进行动态编程。 然而，阶段内概率转移和循环行为的存在意味着我们正在解决期望值优化问题，而不是简单的最短路径。 

当健康为一时，就会出现一种微妙的边缘情况。 失败将立即结束运行，因此任何 hp = 1 的状态都必须非常倾向于治疗或非常仔细地考虑尝试的风险。 另一个棘手的情况是当一个阶段有图腾时：法力可以有效地多次重置为满，因此假设单调法力消耗的策略可能是不正确的。 

## 方法

 蛮力方法将每个状态视为概率图中的一个节点，并尝试通过重复松弛转换直到收敛来计算期望值。 每个状态都由阶段、生命值、法力以及是否可以进行伤害后治疗来定义。 对于每个状态，我们可以在三个动作中进行选择，每个动作都会产生概率性或确定性的转变。 

这导致方程组具有多达大约 1000 × 9 × 7 × 2 状态，大约 126000 个状态。 跨所有状态的朴素迭代期望传播在大小上仍然是可以接受的，但如果我们尝试逐步模拟每个查询的转换或使用带有重新计算的递归，由于重叠状态和阶段内概率循环的重复重新计算，它很快就会变得不可行。 

关键的观察结果是，尽管存在随机转变，但系统在精心选择的方向上是非循环的。 阶段指数在成功时严格增加，健康在失败时严格降低。 法力会发生变化，但受到限制，并且伤害后的可用性也受到限制。 这意味着我们可以以相反的阶段顺序执行动态编程，并在一个阶段内以递增的顺序处理运行状况，以便所有“故障转换”已经指向已知的较小运行状况。 

挑战尝试的概率转变是期望出现的唯一地方，但它不会对同一状态产生循环依赖，因为失败总是会降低健康状况。 这消除了对迭代求解器的需要，并允许直接 DP 公式。

| 方法| 时间复杂度| 空间复杂度| 判决 |

 | 蛮力迭代松弛 | O(n·H·S·迭代) | O(n·H·S) | O(n·H·S) | 太慢/不稳定 |
 | 结构化 DP over (i, hp, mp, flag) | O(n·H·S·动作) | O(n·H·S) | O(n·H·S) | 已接受 |

 ## 算法演练

 我们定义一个 DP 状态，表示在给定配置下完成游戏的最短预期剩余时间。 该配置包括当前阶段 i、当前生命值、当前法力值以及指示特殊伤害后治疗当前是否可用的布尔标志。 

1. 我们以相反的顺序处理阶段，从 n 开始到 1。原因是成功的尝试总是向前移动到 i + 1，所以未来的值必须是已知的。 
2. 我们为阶段 n + 1 定义一个基本情况。一旦我们通过最后一个阶段，无论资源如何，预期剩余时间都为零，因为任务已完成。 
3. 在计算第 i 阶段的任何状态之前，我们应用该阶段的图腾效应（如果存在）。 这意味着每当我们进入第一阶段时，法力就会立即重置为满容量。 这种重置不是可选的，也不会花费时间，因此第 i 阶段的每个 DP 状态都假设 mp 已经符合此规则。 
4. 对于每个状态，我们评估三个可能的决定：尝试该阶段、专注于使用法力恢复生命值以及使用伤害后治疗（如果可用）。 
5. 在考虑尝试时，我们总是立即支付一个时间单位。 之后，我们以 pi 的概率进入阶段 i + 1，而不改变生命值或法力值。 以 1 − pi 的概率，我们停留在阶段 i，但失去一个生命值并为下一个决定解锁伤害后治疗。 预期值是通过结合这两个结果而形成的。 
6. 在考虑焦点时，我们检查法力是否可用。 如果是这样，我们支付 T1 时间并增加 1 点生命值，同时减少 1 点法力值。 这种转变是确定性的。 
7. 在考虑伤害后治疗时，我们只能在旗帜处于活动状态时使用它。 如果允许，我们支付 T2 时间，将生命值增加 1，并消耗该标志，这样它就无法重复使用，直到另一个损坏事件发生。 
8. 每个状态的DP值是所有有效动作中的最小值。 

### 为什么它有效

 每次转换要么会增加阶段索引，要么会降低运行状况，并且这两种操作都不会循环回到之前无法解析的状态。 唯一的随机元素，即成功概率，产生一个线性期望方程，其中递归项始终指的是已计算的严格较小的健康状态或较高阶段的状态。 这保证了每个 DP 状态都是明确定义的，并且可以按相反顺序精确计算，而无需迭代近似。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, H, S = map(int, input().split())
    p_raw = list(map(int, input().split()))
    p = [x / 100.0 for x in p_raw]

    tmp = list(map(int, input().split()))
    K = tmp[0]
    totem = set(tmp[1:]) if K > 0 else set()

    T1, T2 = map(int, input().split())

    NEG = 1e100

    # dp[i][hp][mp][flag], but we compress by rolling over i
    dp_next = [[[ [0.0]*2 for _ in range(S+1)] for __ in range(H+1)] for ___ in range(2)]
    dp_cur  = [[[ [0.0]*2 for _ in range(S+1)] for __ in range(H+1)] for ___ in range(2)]

    # stage n+1 is 0
    for hp in range(H+1):
        for mp in range(S+1):
            for f in range(2):
                dp_next[hp][mp][f] = 0.0

    for i in range(n, 0, -1):

        for hp in range(H+1):
            for mp in range(S+1):
                for f in range(2):
                    dp_cur[hp][mp][f] = NEG

        for hp in range(1, H+1):
            for mp in range(S+1):
                for f in range(2):

                    # totem refill
                    cur_mp = S if i in totem else mp

                    best = NEG

                    # action 1: challenge
                    if hp > 0:
                        succ = dp_next[hp][cur_mp][0]
                        fail = dp_cur[hp-1][cur_mp][1] if hp-1 > 0 else NEG
                        val = 1 + p[i-1] * succ + (1 - p[i-1]) * fail
                        best = min(best, val)

                    # action 2: focus
                    if mp > 0 and hp < H:
                        val = T1 + dp_cur[hp+1][mp-1][f]
                        best = min(best, val)

                    # action 3: hiveblood
                    if f == 1 and hp < H:
                        val = T2 + dp_cur[hp+1][mp][0]
                        best = min(best, val)

                    dp_cur[hp][mp][f] = best

        dp_next, dp_cur = dp_cur, dp_next

    ans = dp_next[H][S][0]
    print(f"{ans:.10f}")

if __name__ == "__main__":
    solve()
```该解决方案使用反向动态规划扫描各个阶段。 关键的实现细节是，在固定阶段内，我们使用相同的 dp 数组填充状态，因为故障转换只会导致严格的较低运行状况，而该运行状况已在同一迭代中计算出来。 

在评估转换之前，图腾处理在每个阶段状态本地应用，确保法力始终与阶段进入条件一致。 

概率转换直接通过线性期望处理，分为成功和失败分支。 故障分支在 hp − 1 处使用 dp_cur，由于迭代顺序超过运行状况，该分支已被填充。

 ## 边缘情况

 当运行状况为 1 时，会发生临界边缘情况。在这种情况下，故障将导致运行状况为零，这是无效的，因此 DP 必须有效地将该分支视为不可用。 该转换通过为无效的 hp − 1 状态分配无限成本来处理此问题，确保算法在死亡迫在眉睫时永远不会选择赌博，除非没有其他选择。 

另一个重要的情况是当一个阶段包含图腾时。 由于法力值在进入时被强制重置，任何 DP 状态都必须忽略传入的法力值并在本地重新计算。 这可以防止跨阶段转换时错误地重复使用耗尽的法力。 

蜂巢血限制也很微妙。 它的可用性取决于之前是否受到过损坏，因此不能将其视为全局资源。 该标志确保它仅在发生故障后立即可用，并且 DP 在使用后显式重置它，以匹配问题约束。
