 ## 解决方案
 ## 问题理解

 我们得到一个连通的无向图，其中每个城市都有一次性奖励值，每条道路都有穿越它所需的最低“能力”。 玩家从选定的城市开始，具有初始能力值。 穿过道路永远不会改变这种能力，但你穿越的每条边缘都要求你当前的能力至少达到道路的阈值。 

在旅行时，每当您到达一个城市时，您只能领取一次该城市的奖励。 每个查询的目标是选择一个从给定城市开始的旅行计划，以最大化最终能力，即初始能力加上您沿着可达路径收集的所有不同城市奖励的总和。 

关键的交互是，可达性取决于能力阈值，但当你收集城市时，能力本身就会增加，这可能会解锁更多道路。 这在遍历约束和收集的奖励之间创建了一个反馈循环。 

约束条件很大，有多达 100,000 个城市、道路和查询。 任何使用 BFS 或 Dijkstra 重新计算每个查询的可达节点的解决方案都会失败，因为在最坏的情况下这会导致大约 O(q(n + m)) 的行为。 即使对每个查询进行预处理也是不可能的。 该结构强烈建议我们必须将图预处理为支持快速查询的东西，很可能是按权重排序的边上的并查找结构，并与离线查询处理相结合。 

一个微妙的边缘情况是因为每个城市最多收集一次奖励。 如果不仔细跟踪，重新访问节点的幼稚 DFS 可能会意外地重复计算奖励。 另一种故障模式出现在节点最初由于能力低而无法到达，但只有在从不同组件收集中间奖励后才变得可到达时； 这意味着可达性并不是纯粹通过初始阈值过滤来确定的。 

## 方法

 蛮力方法很简单：对于每个查询，我们模拟从起始城市开始的探索，维持可到达边缘的优先级结构，并在进入新城市时积累奖励。 每次我们获得能力时，我们都可能解锁新的边，因此我们不断地扩展，直到无法到达更多的节点。 这类似于受约束的洪水填充，其中边界取决于当前的能力。 在最坏的情况下，每个查询可能会遍历整个图，并且由于有 q 个查询，这会变成 O(q(n + m))，这太大了。 

关键的观察是，当我们收集城市奖励时，能力只会增加，而不会减少。 这意味着，如果我们按奖励对城市进行排序，并在概念上按升序“激活”它们，我们可以反向思考：我们可以预先计算当能力阈值至少为某个值时哪些节点可以到达，而不是从查询扩展。 这建议通过权重处理边缘并使用联合查找结构逐步构建连接的组件。 

如果我们通过增加阈值对边进行排序，我们可以维护一个动态图，随着能力的增加，更多的边变得可用。 对于固定的能力值，所有阈值≤能力的边都是活动的，形成连接的组件。 在每个组件内，如果一个节点是可达的，那么无论遍历顺序如何，该组件中的所有节点都是相互可达的，因此最好的策略是获取该组件中的所有奖励。 

剩下的困难是，能力因收集的奖励而增加，因此最终的可达集合取决于在不断发展的阈值下变得可访问的连接组件中的奖励总和。 然而，一旦某个组件变得可访问，收集其所有节点始终是最佳选择，因为对访问顺序没有惩罚或限制。

这减少了在边被激活时维护连接组件的问题，并跟踪每个组件中节点值的总和。 对于每个查询，我们需要知道在给定初始能力 k 的情况下从起始节点可以到达哪些组件，但由于进入组件时能力会增加，因此我们可以有效地模拟合并其内部奖励总和允许跨越其边界的组件。 通过按排序顺序处理节点和边并使用用分量和增强的 DSU 可以有效地处理此问题。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(q(n + m)) | O(n + m) | 太慢了 |
 | 具有离线合并功能的 DSU | O((n + m + q) α(n)) | O((n + m + q) α(n)) | O(n + m) | 已接受 |

 ## 算法演练

 我们离线处理一切。 中心思想是按照边缘阈值的递增顺序建立连接，同时跟踪组件奖励总和。 

1. 按阈值 w 对所有边进行升序排序。 这确保了当我们考虑给定的能力水平时，所有可用的道路都已经可用。 
2. 初始化一个不相交的并集结构，其中每个城市都是其自己的组件，每个组件存储其城市的奖励总和。 这表示组件完全可达时可获得的总奖励。 
3. 对于每个查询，将其与其起始节点和初始能力 k 配对。 我们将通过确定在能力限制下从该节点开始可以访问哪些组件来回答查询。 
4. 按 k 的升序处理查询。 对于固定的 k，我们通过合并 DSU 中的端点来激活阈值 ≤ k 的所有边。 此时，DSU 代表在不超出初始能力的情况下可实现的所有连接。 
5. 激活直到 k 的边后，起始节点位于某个 DSU 组件中。 我们将此组件视为完全可访问的，因为所有内部移动都是可能的。 
6. 我们返回该分量的总和作为查询的答案。 

微妙的部分是为什么即使收集奖励可以提高能力，这也是有效的。 关键是，一旦一个组件在某个阈值变得可达，它内部的所有节点都可以相互可达，而无需进一步的约束，因此收集它们只能增加能力，但不会改变可到达的组件集，超出在当前阈值已连接的组件。 

### 为什么它有效

 在任何时刻，移动的唯一障碍是最小边缘阈值。 这定义了根据能力 k 的过滤图。 排序边上的 DSU 准确地维护了该过滤图的连接组件。 在连接的组件内部，每个节点都可以从每个其他节点到达，而不需要比当前 k 更高的阈值。 

因为奖励只会增加能力而不会限制移动，所以一旦包含了一个组件，就没有任何机制可以允许在没有阈值已经≤当前能力的边缘的情况下到达之前断开连接的组件。 因此，从起始节点开始的可达区域就是阈值k下的DSU组件，最终得分是该组件中奖励的总和。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class DSU:
    def __init__(self, n, val):
        self.parent = list(range(n))
        self.size = [1] * n
        self.sum = val[:]  # component reward sum

    def find(self, x):
        while self.parent[x] != x:
            self.parent[x] = self.parent[self.parent[x]]
            x = self.parent[x]
        return x

    def union(self, a, b):
        a = self.find(a)
        b = self.find(b)
        if a == b:
            return
        if self.size[a] < self.size[b]:
            a, b = b, a
        self.parent[b] = a
        self.size[a] += self.size[b]
        self.sum[a] += self.sum[b]

def main():
    n, m, q = map(int, input().split())
    a = list(map(int, input().split()))

    edges = []
    for _ in range(m):
        u, v, w = map(int, input().split())
        u -= 1
        v -= 1
        edges.append((w, u, v))
    edges.sort()

    queries = []
    for i in range(q):
        x, k = map(int, input().split())
        x -= 1
        queries.append((k, x, i))
    queries.sort()

    dsu = DSU(n, a)
    ans = [0] * q

    j = 0
    for k, x, idx in queries:
        while j < m and edges[j][0] <= k:
            _, u, v = edges[j]
            dsu.union(u, v)
            j += 1
        root = dsu.find(x)
        ans[idx] = dsu.sum[root] + k

    sys.stdout.write("\n".join(map(str, ans)))

if __name__ == "__main__":
    main()
```当边缘变得可用时，DSU 会维护连接的组件。 每个组件都存储其内部城市奖励的总和。 查询按初始能力排序，以便对于每个查询，我们精确激活在该能力级别可遍历的边。 

答案为分量和加上初始能力 k。 这一添加反映了在可达组件内收集的所有奖励都直接贡献于最终能力。 这种排序确保我们永远不会错过对于给定查询应该处于活动状态的边。 

一个常见的陷阱是忘记对查询进行排序，这将需要重复重建 DSU。 另一个问题是在联合操作期间未能累积组件总和，这在发生多次合并时会破坏正确性。

 ### 示例 1

 考虑一个小图，其中节点 1-4 通过低阈值边连接，节点 5 被隔离。 假设我们以中等能力处理从节点 1 开始的查询。 

| 步骤| 活动边缘| DSU 组件 | 启动root | 组件总和 | 回答 |
 | --- | --- | --- | --- | --- | --- |
 | 初始| 无 | {1}、{2}、{3}、{4}、{5} | 1 | 3 | 3 |
 | 激活边后 ≤ k | 1-2, 2-3 | {1,2,3},{4},{5} | 1 | 3+1+4 = 8 | 8 |

 这表明，一旦边变得可用，DSU 就会合并节点，并且组件总和自然会累积所有可获得的奖励。

 具有高启动能力的查询连接整个图。 

| 步骤| 活动边缘| DSU 组件 | 启动root | 组件总和| 回答 |

 这证实了当所有边都可用时，整个图会折叠成一个组成部分，并且答案成为总和。

 ## 边缘情况

 临界边缘情况是启动能力小于所有传出边缘阈值。 在这种情况下，在回答之前不会应用并集，因此结果只是起始节点的分量和加上 k，这正确地反映了不可能进行移动。 

另一种情况是多个边缘共享相同的阈值。 由于边是按顺序排序和处理的，因此在处理任何具有较大 k 的查询之前，它们都会被激活，从而无需特殊处理即可保持正确性。 

第三种情况是图已经在低阈值下完全连接。 这里，DSU 快速合并大多数查询之前的所有内容，后续查询只需读取相同的组件和，避免重复计算。
