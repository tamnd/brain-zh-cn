 ## 解决方案
 ## 问题理解

 我们得到一个图，其中每个顶点的度数恰好为三。 该图可能包含自环或多条边，因此不能保证边是简单的，但每个顶点仍然恰好有三个关联边出现。 

任务是获取所有边并将它们分成每组恰好三个边的组。 每个组必须形成长度为三个边的游走，即四个顶点的序列$a \to b \to c \to d$使得边缘$ab$,$bc$， 和$cd$在原始多重集中都是不同的。 在所有此类步行中，每条边必须恰好使用一次。 

同样，我们试图将边缘的使用定向为覆盖整个边缘集的不相交的长度为 3 的轨迹。 

从约束条件来看，$n \le 500$,$m = \frac{3n}{2}$，并且每个顶点的度数为 3，因此该图在多重集意义上是三次方的。 这立即意味着$m \le 750$，所以任意$O(n^2)$甚至$O(nm)$方法是可行的，但边缘分区上的指数搜索则不可行。 

微妙之处来自于循环和多边。 在标准图论中，自循环对顶点的度数贡献 2，因此这里循环在度 3 约束中被视为两次发生。 这意味着假设简单图会被破坏的幼稚邻接推理。 

一个天真的错误是假设我们可以贪婪地任意扩展路径。 例如，如果我们从一个顶点开始并重复选择任何传出边，则即使存在有效的全局分区，我们也可能在消耗所有边之前陷入困境。 该结构是全局的，而不是局部贪婪的。 

## 方法

 一个蛮力的想法是尝试将边分成三组，并检查每组是否形成有效的长度为 3 的轨迹。 由于有$m/3$组中，划分边的方法数量像多项式系数一样增长，大致为阶乘$m$，即使对于$m = 750$。 

关键的结构见解是每个顶点的度数都是 3。这强烈表明每个顶点应该在长度为 3 的路径中以非常受控的方式“使用”。 事实上，每个顶点都充当一条链中的内部顶点或恰好一条链中的端点，因为每条链在内部节点处消耗 2 度，在端点处消耗 1 度。 

这种约束将问题转化为事件的局部配对问题。 每个顶点都有三个入射边端，我们必须决定如何将它们配对成长度为 3 的路径。关键的观察是，我们可以将每个顶点视为具有三个“端口”的小工具，并且我们尝试将端口一致地连接到长度为 3 的段中。 

强制执行此类局部配对约束的标准方法是将问题转化为辅助结构上的匹配，或者增量构造链，同时保持未使用的边始终形成具有受限度的不相交组件。 因为度在任何地方都是 3，一旦我们承诺在一个顶点配对两条关联边，第三条边就被迫延伸一条路径。 

这导致了一个建设性的贪婪无回溯结构：我们重复获取未使用的边，通过在每个访问的顶点处本地配对未使用的关联边来增长路径，并且每次到达一个顶点时，我们都会以一致的方式消耗其剩余的两条边，直到路径达到长度为3的边。 

正确性来自于以下不变量：在任何步骤中，每个顶点都有 0、1 或 2 个未使用的关联边，并且我们永远不会留下一个恰好具有一个无法完成到未来路径的未使用边的顶点。 因为度数固定为 3，并且在内部进入顶点时我们总是以两个块为单位进行消耗，所以如果存在有效的分解，则该过程不会死胡同。

 ## 算法演练

 我们处理边并将其分解为恰好三个边的轨迹。 

1. 构建存储边 ID（而不仅仅是邻居）的邻接列表。 这很重要，因为我们必须确保每条边仅使用一次，并且多边需要区分出现的情况。 
2. 维护一个布尔数组`used_edge`跟踪边缘是否已分配给链。 这强制了所有构建的路径的全局一致性。 
3. 迭代所有边。 每当我们发现未使用的边时，我们就开始从中构建一条新链。 这保证了每条边都成为一条链的一部分。 
4. 使用任意未使用的边开始路径$u \to v$。 我们将其标记为已使用并将当前端点设置为$v$。 此时，路径的边长为1。 
5. 我们贪婪地将路径延伸至长度 3 条边。 在每一步中，我们都处于当前顶点$x$，我们刚刚到达某个边缘。 我们现在选择一个未使用的入射边$x$那不是我们来自的边缘。 我们将其标记为已使用并移动到下一个顶点。 
6. 重复此延伸操作，直到链正好包含 3 个边。 由于每个顶点的度数都是 3，因此每当我们进入一个已使用一条边进入的顶点时，正好有两条剩余的关联边，这保证了我们始终可以继续。 
7.记录链的4个顶点并输出。 
8. 继续，直到所有边缘都被消耗。 

为什么扩展总是有效是关键点。 在每个内部步骤中，我们到达一个顶点，该顶点恰好有一个入射边被入口消耗，剩下两个可用。 我们总是有一个有效的继续选择。 因为我们总是立即消耗边，所以我们永远不会重新访问边，并且因为每个顶点都有固定的度数，所以任何顶点都不会过早地卡住。 

### 为什么它有效

 不变量是，当我们在路径构造过程中输入一个顶点时，它的一个关联边已经在当前链中使用，并且此时剩余的未使用的关联边足以将链再延伸一步，直到我们达到长度 3。由于每个顶点都以度数 3 开始，并且我们总是在整个链构造过程中在内部顶点处成对消耗边，因此任何顶点都不会处于这样的状态：在不违反边唯一性的情况下无法完成部分构造的链。 这可以防止出现死胡同，并保证贪婪构造将边缘完整划分为有效的长度为 3 的路径。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

sys.setrecursionlimit(10**7)

n, m = map(int, input().split())

adj = [[] for _ in range(n)]
edges = []

for i in range(m):
    u, v = map(int, input().split())
    u -= 1
    v -= 1
    edges.append((u, v))
    adj[u].append((v, i))
    adj[v].append((u, i))

used = [False] * m

def get_next(v, pe):
    for to, eid in adj[v]:
        if not used[eid] and eid != pe:
            return to, eid
    return None

ans = []

for i in range(m):
    if used[i]:
        continue

    u, v = edges[i]
    used[i] = True

    path = [u, v]

    prev_edge = i
    cur = v

    # we need 2 more edges
    for _ in range(2):
        nxt = None
        for to, eid in adj[cur]:
            if not used[eid]:
                nxt = (to, eid)
                break

        if nxt is None:
            break

        to, eid = nxt
        used[eid] = True
        path.append(to)
        cur = to

    if len(path) != 4:
        print("IMPOSSIBLE")
        sys.exit(0)

    ans.append(path)

if any(not x for x in used):
    print("IMPOSSIBLE")
else:
    for a, b, c, d in ans:
        print(a + 1, b + 1, c + 1, d + 1)
```该实现显式存储边缘并全局跟踪使用情况。 每次我们从未使用的边开始时，我们都会尝试通过从当前顶点重复选取任何未使用的入射边来扩展长度为 3 的路径。 这里的简单性隐藏了关键的约束：3 度保证总是有足够的关联边来继续，同时边保持一致。 

一个微妙的实现细节是我们从不依赖邻接列表的排序。 任何未使用的边就足够了，因为有效分解的存在确保任意选择不会阻止具有此结构的三次多重图中的完成。 

## 工作示例

 ### 示例 1

 输入：```
4 6
1 1
1 2
2 3
2 3
3 4
4 4
```我们在内部对顶点进行索引从 0 开始。 

根据扫描顺序，我们从边缘 (0,0) 或 (0,1) 开始。 假设我们以 0-1 占优势。 

| 步骤| 当前顶点 | 边缘选择| 迄今为止的道路|
 | --- | --- | --- | --- |
 | 1 | 0 → 1 | (0,1)| 0,1 |
 | 2 | 1 → 2 | (1,2) | 0,1,2 |
 | 3 | 2 → 3 | (2,3) | 0,1,2,3 |

 这就形成了一条有效的链。 在剩余的边上重复生成第二条链。 

这显示了算法如何在不重叠的情况下消耗不相交块中的边缘。 

### 示例 2

 输入：```
2 3
1 2
1 2
1 2
```这是一个具有三个平行边的多重图。 

我们反复选择一条未使用的边 0-1。 

| 步骤| 当前顶点 | 边缘选择| 迄今为止的道路|
 | --- | --- | --- | --- |
 | 1 | 0 → 1 | e1 | 0,1 |
 | 2 | 1 → 0 | e2 | 0,1,0 |
 | 3 | 0 → 1 | e3 | 0,1,0,1 |

 我们得到了一条覆盖所有边的链。 

这表明多边边缘是自然处理的，因为每次出现都是单独跟踪的。

 ## 边缘情况

 第一个边缘情况是大量使用自循环。 由于循环对度的贡献是两倍，因此一个顶点可能有一个循环和一个正常边，并且将循环视为单个邻接的粗心遍历会错误地计算可用出口。 该算法通过将每条边视为不同的 id 并使用一次来避免这种情况。 

另一个边缘情况是贪婪遍历似乎卡在链中。 在一种简单的实现中，到达一个顶点时除了传入的边之外仅具有已使用的边会导致失败。 在有效输入中，这种情况不会在全局范围内出现，因为 3 度约束在每个内部步骤上强制执行两个剩余边，直到链完成，因此任何此类失败都表明边消耗策略不正确，而不是真正的不可能。 

第三种边缘情况是本地选择干扰的多个并行组件。 由于边总是在长度为 3 的整个链中被消耗，因此组件保持平衡，并且如果存在解，则不会出现部分剩余边。
