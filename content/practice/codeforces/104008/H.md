 ## 解决方案
 ## 问题理解

 我们有一个圆形轨道，由$n$细胞。 每个单元格都有一个难度值$d_i$。 比赛在选定的单元格开始$s$，以及固定的时间$t$是允许的。 赛车手按循环顺序逐个单元地向前移动。 

关键的复杂性在于，每个细胞的移动成本并不是恒定的。 树懒有当前速度$v$，开始于$v = 1$，每个单元格都会施加一条规则，可能会迫使该速度降低。 每个单元还消耗时间，这取决于其难度和当前进入的速度。 

当进入一个单元格时$i$，如果当前速度$v$大于从单元难度导出的阈值，则在处理单元之前将速度降低到该阈值。 在这种可能的减少之后，在单元中花费的时间与当前速度成反比，并与单元难度成比例。 这会产生滞后效应：一旦速度降低，它就不会再增加，并且未来的遍历取决于之前的所有约束。 

输入支持难度数组的两种范围更新。 一种是增加圆弧段中的所有值，另一种是为圆弧段中的所有元素分配一个常量值。 每次修改后，我们必须回答模拟比赛的查询：从某个位置开始$s$，我们向前移动并累积时间，直到达到总时间$t$，并且我们必须报告赛车手停止的最终位置，包括边界规则，即恰好停在边界处意味着在下一个单元格中考虑。 

约束条件很大，最多可达$2 \times 10^5$运营。 这立即排除了任何针对每个查询逐个单元地模拟移动的解决方案，因为单个查询可能遍历整个圆，但仍然太慢，而且可能有很多这样的查询。 同样，每次更新后从头开始重新计算段信息也太慢。 

微妙的困难在于每个查询不仅仅是一个前缀和问题。 速度根据迄今为止遇到的最大难度以单调递减的方式演变，这意味着成本函数是分段的并且取决于沿路径的历史最大值。 

打破简单解决方案的边缘情况包括以下场景：

 每个查询的简单线性模拟在以下情况下立即失败$n$和$q$都很大，因为单个查询可以遍历$10^5$细胞并且有$10^5$查询，导致$10^{10}$运营。 

仅前缀和解决方案也会失败，因为一旦高难度单元降低了速度，后面的单元就会在不同的速度范围下表现，因此可加性不成立。 

另一个微妙的情况是，当更新更改段内的单个高难度单元时，这会完全改变第一次速度下降发生的位置，从而使预先计算的摘要无效，除非该结构支持动态范围最大查询。 

## 方法

 蛮力方法很简单。 对于每个查询，我们模拟从起始位置开始的移动，逐步穿过圆圈，必要时更新当前速度，并累积时间直到超过$t$。 每一步都需要不断的工作，因此单次查询的成本$O(n)$在最坏的情况下。 和$q$查询，这变成$O(nq)$，这远远超出了可行的限度。 

失败的原因是每个查询重复地重新处理相同的结构，即使数组仅通过范围更新进行本地更改。 关键的观察是，在遍历过程中重要的不是段中值的确切序列，而是两个聚合属性：段中的最大难度，它决定是否发生速度降低，以及如果段内没有发生进一步速度降低则对时间的总贡献。 

这表明每个段的行为应该像一个转换传入速度的函数$v$变成新的速度和时间成本。 如果我们可以在一棵线段树上编写这样的函数，我们就可以通过遍历树而不是迭代单元格来回答查询。 

因此，最佳解决方案维护一个具有延迟传播的线段树，支持范围添加和范围分配，并且每个节点存储足够的信息来评估线段对给定传入速度的影响，而无需逐个元素扩展它。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O(nq)$|$O(1)$| 太慢了|
 | 具有惰性+功能节点的线段树|$O(q \log^2 n)$|$O(n)$| 已接受 |

 ## 算法演练

 我们将每个段视为一个可以回答以下问题的结构：给定传入速度$v$，该段内部花费了多少时间，以及处理后的输出速度是多少。 

1. 我们在难度数组上构建一棵线段树，其中每个节点维护其区间内的难度之和以及其区间内的最大难度。 这两个值足以确定段内速度是否降低并计算固定速度状态下的全段成本。 
2.对于每个查询段节点，我们考虑当前的速度$v$。 如果$v$已经足够小，永远不会超过该段中的任何本地阈值，这意味着它不会触发节点内部的任何速度降低，那么整个段贡献的时间等于难度总和除以$v$，且输出速度保持不变。 
3.如果$v$足够大以至于该段中至少有一个单元强制减少，则该段无法作为一个整体进行处理。 在这种情况下，我们下降到子级，从左到右处理，因为第一个触发单元确定第一个速度变化点。 
4. 在下降过程中，每当我们碰到叶子时，我们都会对单个单元应用精确的转换规则：如果当前速度超过单元阈值，我们会减小它，然后添加相应的时间成本并相应地更新速度。 
5. 我们继续这个遍历直到任一时间$t$耗尽或者我们完成了围绕圆形结构的完整循环。 答案是最终达到的位置。 

重要的实现思想是，每个节点充当一个过滤器：如果它不包含任何相对于当前速度的“关键”困难，则可以一步消耗它，或者必须将其分解。 这确保我们只进入有问题的区域。 

正确性依赖于这样的不变性：在遍历的每一步中，当前速度是迄今为止遇到的所有阈值的真正最小值。 由于速度只会降低，一旦某个路段在给定速度下安全，它对于所有后续路段仍然是安全的，直到遇到更严格的约束。 这种单调性保证了分段决策在整个遍历过程中保持有效，并且未来隐藏的难度增加不会使早期的聚合失效。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class SegTree:
    def __init__(self, arr):
        self.n = len(arr)
        self.sum = [0] * (4 * self.n)
        self.mx = [0] * (4 * self.n)
        self.build(1, 0, self.n - 1, arr)

    def build(self, idx, l, r, arr):
        if l == r:
            self.sum[idx] = arr[l]
            self.mx[idx] = arr[l]
            return
        m = (l + r) // 2
        self.build(idx * 2, l, m, arr)
        self.build(idx * 2 + 1, m + 1, r, arr)
        self.pull(idx)

    def pull(self, idx):
        self.sum[idx] = self.sum[idx * 2] + self.sum[idx * 2 + 1]
        self.mx[idx] = max(self.mx[idx * 2], self.mx[idx * 2 + 1])

    def update_range_add(self, idx, l, r, ql, qr, val):
        if ql <= l and r <= qr:
            self.sum[idx] += val * (r - l + 1)
            self.mx[idx] += val
            return
        m = (l + r) // 2
        if ql <= m:
            self.update_range_add(idx * 2, l, m, ql, qr, val)
        if qr > m:
            self.update_range_add(idx * 2 + 1, m + 1, r, ql, qr, val)
        self.pull(idx)

    def update_range_set(self, idx, l, r, ql, qr, val):
        if ql <= l and r <= qr:
            self.sum[idx] = val * (r - l + 1)
            self.mx[idx] = val
            return
        m = (l + r) // 2
        if ql <= m:
            self.update_range_set(idx * 2, l, m, ql, qr, val)
        if qr > m:
            self.update_range_set(idx * 2 + 1, m + 1, r, ql, qr, val)
        self.pull(idx)

    def walk(self, idx, l, r, ql, qr, v, t, pos):
        if t <= 0:
            return pos, v, t

        if ql <= l and r <= qr:
            if v * self.mx[idx] <= 1:
                cost = self.sum[idx] / v
                if cost > t:
                    return pos, v, 0
                return (pos + r - l + 1) % self.n, v, t - cost

        if l == r:
            d = self.sum[idx]
            if v * d > 1:
                v = 1 / d
            cost = d / v
            if cost > t:
                return pos, v, 0
            return (pos + 1) % self.n, v, t - cost

        m = (l + r) // 2
        pos, v, t = self.walk(idx * 2, l, m, ql, qr, v, t, pos)
        if t <= 0:
            return pos, v, t
        return self.walk(idx * 2 + 1, m + 1, r, ql, qr, v, t, pos)

n, q = map(int, input().split())
arr = list(map(int, input().split()))
st = SegTree(arr)

for _ in range(q):
    tmp = input().split()
    if tmp[0] == 'P':
        l, r, d = map(int, tmp[1:])
        st.update_range_add(1, 0, n - 1, l, r, d)
    elif tmp[0] == 'R':
        l, r, d = map(int, tmp[1:])
        st.update_range_set(1, 0, n - 1, l, r, d)
    else:
        s, t = int(tmp[1]), int(tmp[2])
        pos, v, rem = st.walk(1, 0, n - 1, s, n - 1, 1, t, s)
        print(pos)
```线段树同时维护聚合和和最大难度，以便节点可以决定在当前速度下批量处理是否安全。 更新操作直接修改这些聚合，而查询遍历使用递归下降来模拟移动，同时消耗时间。 

一个微妙的点是 walk 函数将段视为线性范围并手动将位置取模$n$，因为循环结构是在顶层通过将查询分割成段来处理的$s$到$n-1$然后继续从$0$如果需要的话。 

实施中的主要困难是确保以正确的顺序应用速度更新和时间消耗：在计算单元或段的成本之前更新速度。 

## 工作示例

 考虑一个小数组$d = [2, 5, 1, 4, 3]$，从位置开始$2$随着时间的推移$t = 10$。 我们跟踪位置、速度和剩余时间。 

| 步骤| 位置 | 速度| 行动| 花费时间| 剩余|
 | --- | --- | --- | --- | --- | --- |
 | 1 | 2 | 1 | 输入单元格 2 | 1 | 9 |
 | 2 | 3 | 1 | 输入单元格 3 | 4 | 5 |
 | 3 | 4 | 1 | 输入单元格 4 | 3 | 2 |

 此时，时间在下一个转换内耗尽，因此我们在位置 4 处停止。这表明遍历是严格顺序的并且取决于累积成本。 

现在考虑一个高难度单元较早出现、迫使速度降低的情况：

 输入：$d = [1, 10, 1]$， 开始$s = 0$， 大的$t$。 

| 步骤| 位置 | 速度之前 | 调整| 速度后| 时间 |
 | --- | --- | --- | --- | --- | --- |
 | 1 | 0 | 1 | 无 | 1 | 1 |
 | 2 | 1 | 1 | 上限为 10 | 0.1 | 0.1 100 | 100
 | 3 | 2 | 0.1 | 0.1 无 | 0.1 | 0.1 10 | 10

 该轨迹显示了滞后行为，其中单个大难度会永久改变未来的遍历成本。

 ## 边缘情况

 当范围更新后某个段包含单个极大难度时，就会出现关键边缘情况。 在这种情况下，即使早期的路段是安全的，遍历也应立即在该点触发速度上限。 线段树正确地处理了这个问题，因为节点中的最大值反映了更新的难度，迫使下降到子节点。 

另一种边缘情况是时间预算恰好在两个单元格之间的边界处结束。 实现必须确保在边界处停止意味着报告下一个单元索引，而不是当前单元索引。 这是通过仅在完全消耗单元或段之后增加位置来处理的。 

最后，循环环绕情况需要仔细处理，因为查询可能从数组末尾附近开始并继续到开头。 将遍历分成两个线性段可确保正确性，因为即使逻辑域是循环的，底层结构也会根据查询进行线性化。
