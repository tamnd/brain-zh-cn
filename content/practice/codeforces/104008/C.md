 ## 解决方案
 ## 问题理解

 我们从一个数组开始，并允许重复地精确地转换它$m$次。 每次转换都会用以下两种方式之一形成的新数组替换当前数组：要么复制数组并将其附加到自身，要么获取反向副本并将其放置在原始数组的前面。 

毕竟$m$运算，我们得到最终的数组$b$。 目标不是最大化元素的简单和，而是最大化加权和，其中每个位置贡献其前缀和。 具体来说，如果我们定义$S_i = \sum_{j=1}^{i} b_j$，那么分数就是$\sum_{i=1}^{|b|} S_i$, 计算模数$10^9+7$，但有一个重要的变化，即比较是在取模之前完成的：我们想要最大可能的整数值，然后才应用取模。 

限制条件$n, m \le 10^5$立即排除对阵列的任何模拟。 即使单个操作也会使大小加倍，因此数组长度变为$n \cdot 2^m$，显式构造是完全不可行的。 这迫使我们只能推理变换下元素的总贡献。 

一个天真的错误是假设我们可以贪婪地独立决定每个操作。 例如，在数组上$[1,2]$，两种操作都会产生不同的结构，并且重复选择而不跟踪前缀权重的全局影响会导致错误的答案。 另一个微妙的陷阱是误解目标：它不是元素的总和，而是所有前缀总和的总和，这会严重影响较早位置的权重。 

一个小的说明性边缘情况是$a = [1, 100]$,$m = 1$。 追加给出$[1,100,1,100]$，同时反转和添加前缀给出$[100,1,1,100]$。 即使元素多重集相同，前缀结构也会显着改变分数，因为较早出现的大值对许多前缀的贡献更大。 

## 方法

 暴力方法模拟了这个过程：从数组开始，应用所有$m$以所有可能的方式进行操作，生成所有结果数组，并计算分数。 每次操作都会使数组大小加倍，并且有$2^m$可能的操作顺序。 即使忽略内存，这也已经呈指数级爆炸。 另外，计算长度的分数$n2^m$数组本身在该大小上是线性的，使得总工作量变得天文数字般巨大。 

关键的见解是，两个允许的操作都保留值的多集并且仅更改排序，但分数取决于通过前缀累积的排序。 我们不跟踪整个数组，而是跟踪两个聚合统计信息：数组的总和以及前缀和的总和。 这些就足够了，因为这两种操作都可以表示为这两个值的转换。 

连接时$b + b$，后半部分的前缀和移位前半部分的总和。 当反转和添加前缀时，我们利用对称性：反转改变前缀贡献，但总和保持不变，并且前缀和结构可以通过已知的恒等式从原始数据导出。 这将每个操作减少为对一对数字的确定性更新，使我们能够迭代$m$线性时间内的时间。

 ## 算法演练

 我们维护两个值：元素之和$S$以及前缀和的总和$P$。 

1. 计算初始值$S$和$P$在线性时间内从原始数组中得到。 这在任何转换之前建立了基线贡献。 
2. 对于每一个$m$运营，决定转型如何影响$(S, P)$对于两种可能的操作。 我们观察到我们正在最大化，而不是选择，所以我们采取更好的结果状态。 
3. 对于复制操作$b \to b + b$，总和变为$2S$。 前缀和之和变为$2P + S \cdot n$，因为第二个副本的前缀移动了$S$对于它的每一个$n$元素。 
4. 对于反向前缀操作，我们通过前缀反转恒等式来分析效果：反转将前缀结构变成后缀结构。 总金额仍为$S$，而前缀和之和变换为$S \cdot (n+1) - P$。 这是因为前缀和加上后缀和的和具有仅取决于总和的封闭形式。 
5. 在每一步中，选择产量较大的操作$P$，因为最终目标正是$P$后$m$转变。 
6. 迭代这个过程$m$次更新$(S, P, n)$因此，注意到$n$两次操作都加倍。 
7.输出最终结果$P \bmod 10^9+7$。 

### 为什么它有效

 关键的不变量是，在任何操作序列之后，可以仅使用其长度、总和以及前缀和的总和来汇总整个数组以实现此目标。 两种操作都允许将这些数量映射到新值，而不需要了解这些聚合编码之外的内部排序。 由于每个可能的结果数组都可以通过重复应用这些确定性变换来表示，因此贪婪地选择在每个步骤中最大化结果前缀和值的操作可以保持最优性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MOD = 10**9 + 7

def prefix_sum(arr):
    s = 0
    p = 0
    for x in arr:
        s += x
        p += s
    return s, p

def transform_concat(s, p, n):
    ns = 2 * s
    np = 2 * p + s * n
    return ns, np, 2 * n

def transform_reverse_prefix(s, p, n):
    # reversed prefix identity
    ns = s
    np = s * (n + 1) - p
    return ns, np, n * 2

def solve():
    n, m = map(int, input().split())
    a = list(map(int, input().split()))
    
    s, p = prefix_sum(a)
    cur_n = n

    for _ in range(m):
        s1, p1, n1 = transform_concat(s, p, cur_n)
        s2, p2, n2 = transform_reverse_prefix(s, p, cur_n)

        if p1 >= p2:
            s, p, cur_n = s1, p1, n1
        else:
            s, p, cur_n = s2, p2, n2

        p %= MOD
        s %= MOD

    print(p % MOD)

if __name__ == "__main__":
    solve()
```该实现仅跟踪三个状态变量。 功能`prefix_sum`计算初始对$(S, P)$。 每个转换函数都会在恒定时间内更新这些值。 循环贪婪地选择产生较大前缀和贡献的操作。 

一个微妙的实现细节是，必须在取模之前进行比较，因为取模仅用于输出。 过早应用模数会破坏排序决策。 

## 工作示例

 ### 示例 1

 输入：```
2 1
1 2
```我们计算初始值。 

| 步骤| 数组| S | 普 |
 | --- | --- | --- | --- |
 | 初始化| [1,2]| 3 | 1 + 3 = 4 |

 现在评估这两个操作。 

重复给出：

 数组 [1,2,1,2]，S = 6，P = 4 * 2 + 3 * 2 = 14

 反向前缀给出：

 数组 [2,1,1,2]，S = 3，P = 3 * 3 - 4 = 5

 我们选择重复，所以答案是 14。 

这表明当初始排序已经有利时，重复如何更快地放大前缀增长。 

### 示例 2

 输入：```
3 2
1 3 2
```初始：

 S = 6，P = 1 + 4 + 6 = 11

 步骤一：

 对数：S=12，P=22 + 18 = 40

 修订版：S=6，P=6*4 - 11 = 13

 选择重复。 

步骤2：

 从 (12,40)：

 重复：S=24，P=80 + 144 = 224

 修订版：S=12，P=12*7 - 40 = 44

 选择重复。 

最终 P = 224。 

这表明，一旦重复占据主导地位，它就会继续占据主导地位，因为它使前缀增长速度成倍加快。

 ## 边缘情况

 对于$n=1$，在任一操作下数组都不会发生有意义的更改。 国家$(S,P)$通过重复以可预测的方式将两个值加倍而确定性地演变，并且反向前缀使结构实际上保持不变。 

对于像这样的数组$[x, x, x, \dots]$，两个操作都会产生相同的多集行为，但重复严格占主导地位，因为前缀积累受益于相同早期贡献的重复强化。
