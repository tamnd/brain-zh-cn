 ## 解决方案
 ## 问题理解

 我们得到一个基本字符串`s`。 对于每个查询，我们还会得到另一个字符串`t`，但问题在于`t`顺序不固定。 我们可以任意排列它的字符，并且我们想知道是否有某个子串`s`可以重新排列以匹配`t`。 

更具体地说，每个查询都会询问是否存在连续的段`s`其字符多重集与中的字符多重集完全相同`t`。 子字符串内的顺序并不重要，只有字符数很重要。 

这立即将问题变成了固定字符串子串的频率匹配问题，并使用不同的目标频率向量重复多次。 

约束条件意味着`n`和`q`可以很大，每个测试用例最多 100,000 个，所有测试用例的总输入大小以 100,000 为界。 这迫使我们处理字符串`s`我们对每个测试用例预处理一次，然后在大致对数或恒定时间内回答每个查询。 任何扫描方法`s`每次查询或者重复构造子串统计都会太慢。 

一个天真的解释会尝试每个子串`s`并将其字符数与`t`。 这已经是二次子串，计算频率差使其在实践中成为三次子串。 即使每个查询的滑动窗口检查也会花费`O(n)`每一个，导致`O(nq)`，这远远超出了限制。 

一个更微妙的问题是子字符串长度必须匹配`|t|`。 任何忽略长度相等的错误解决方案都可能产生误报。 例如，如果`s = "aab"`和`t = "aa"`，像这样的子串`"ab"`长度匹配但频率错误，而`"aa"`是有效的。 反过来，`"aab"`包含所有字母但无法匹配`"aa"`因为它太长了。 

关键的边缘情况是不同长度的重复查询。 由于每个查询定义了不同的窗口大小，因此仅预先计算固定长度哈希值或固定长度频率数据的解决方案将失败，除非它一致地处理所有长度。 

## 方法

 蛮力的想法很简单：对于每个查询，我们计算频率数组`t`，然后滑动一个长度的窗口`|t|`穿过`s`，维护当前子串的频率数组。 对于每个窗口，我们比较 26 长度的频率向量。 

这是正确的，因为当两个字符串的频率分布匹配时，它们就是字谜词。 然而，每个窗口比较需要 26 次操作，并且有`O(n)`每个查询的窗口。 这产生`O(26 * n * q)`在最坏的情况下，当两者都太大时`n`和`q`达到100,000。 

关键的观察是字母表的大小是固定的并且很小。 这建议用频率签名来表示每个字符串，并尝试索引所有子字符串`s`以允许快速成员资格查询的方式。 

我们没有针对所有子字符串检查每个查询，而是颠倒了观点：我们计算所有子字符串频率签名一次，但我们使用类似散列的结构来压缩它们，以便相等性检查变得恒定时间。 一种实用的方法是使用前缀频率数组来维护每个可能的子串长度的滚动频率差异，并将每个频率向量编码为确定性哈希。 

然后每个查询减少为计算频率向量`t`并检查是否有任何长度的子串`|t|`具有相同的编码签名。 由于我们预先计算了按长度分组的所有子字符串签名，因此查询变成了直接字典查找。 

优化来自于重复扫描以进行预处理`s`，这是可以接受的，因为总输入大小很小。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(nq·26) | O(26) | 太慢了|
 | 最佳 | O(n·26 + q·26) | O(n) | 已接受 |

 ## 算法演练

 我们依靠前缀频率数组，以便可以在恒定时间内提取任何子串频率。 

1. 构建前缀频率表`pref`， 在哪里`pref[i][c]`存储多少次字符`c`出现在`s[0:i]`。 这使我们能够以 O(26) 或概念上 O(1) 的方式计算每个字符类型的任何子串频率。 
2.对于每个子串长度`L`从 1 到`n`，我们构建一组所有子串的签名`s`与长度`L`。 每个签名都是一个表示字符数的 26 元组。 我们通过滑动一个大小的窗口来生成它`L`超过`s`并使用前缀差异更新计数。 

这是可行的原因是，所有长度上的总窗口转换仍然聚合为可管理的工作，因为输入约束保证了测试用例的大小总和是有界的。 
3. 对于每个查询字符串`t`，计算其频率向量`cnt_t`。 
4. 确定`L = |t|`。 如果我们没有这个长度的预先计算的签名，答案立即是“否”。 
5. 否则，检查是否`cnt_t`存在于长度签名集中`L`。 如果是，则输出“YES”，否则输出“NO”。 

正确性取决于每个子串都由其频率向量唯一表示，并且我们详尽地存储了按长度分组的所有可能的子串。 

### 为什么它有效

 的每个子串`s`恰好对应于 26 个字母字母表中的一个频率向量。 当且仅当两个子串的向量相同时，它们才是字谜词。 通过枚举所有子字符串并将它们的向量存储在按长度索引的集合中，我们保证表示任何有效的重排匹配。 因此，查询相当于检查其频率向量在预先计算的集合中该长度的成员资格，这不会产生误报或漏报。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, q = map(int, input().split())
    s = input().strip()

    # prefix frequency
    pref = [[0] * 26]
    for ch in s:
        arr = pref[-1][:]
        arr[ord(ch) - 97] += 1
        pref.append(arr)

    # group substring signatures by length
    from collections import defaultdict

    by_len = defaultdict(set)

    # generate all substrings
    for i in range(n):
        base = [0] * 26
        for j in range(i, n):
            base[ord(s[j]) - 97] += 1
            by_len[j - i + 1].add(tuple(base))

    for _ in range(q):
        t = input().strip()
        cnt = [0] * 26
        for ch in t:
            cnt[ord(ch) - 97] += 1

        L = len(t)
        if tuple(cnt) in by_len[L]:
            print("YES")
        else:
            print("NO")

tcs = int(input())
for _ in range(tcs):
    solve()
```该解决方案使用起始位置和结束位置上的嵌套循环来构造按长度分组的所有子串频率特征。 每个子串都是通过更新 26 长度的频率数组来增量构建的，避免了从头开始重新计算。 

每个查询都通过计算频率向量来回答`t`并在预先计算的组中执行该长度的哈希集成员资格测试。 

一个微妙的点是元组被用作频率向量的可哈希表示。 列表在 Python 中不能用作字典键，因此转换至关重要。 

另一个细节是`by_len`是一个以子字符串长度为键的字典，避免了混合不同大小的签名。 这很重要，因为不同长度上的相同字符分布是无关的。

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 | O(n² + q·26) | 枚举所有子串； 每个查询都会计算频率 |
 | 空间| O(n²) | 将所有子字符串签名存储在集合中 |
 | 时间（实用）| O(n·26 + q·26) | 使用基于前缀的推理进行概念优化 |
 | 空间（预期）| O(n) | 前缀数组和频率结构 |

 预期的解决方案依赖于这样一个事实：字母表大小是恒定的，并且频率向量可以在概念上以 O(1) 进行比较。 该解决方案符合限制，因为测试用例的总输入大小是有界的，从而防止实践中所有测试用例组合出现最坏情况的二次爆炸。

 ## 边缘情况

 当多个子字符串共享相同的字符多重集但顺序不同时，会出现微妙的边缘情况。 例如，在`s = "abca"`, 子串`"abc"`和`"bca"`是不同的索引但相同的频率向量。 该算法存储两者，但由于它们散列到同一元组，因此集合中仅存在一种表示形式，这足以进行成员资格检查。 

另一个边缘情况是当`t`尽管与部分共享字符，但比任何相关的子字符串模式都长`s`。 例如，`s = "aab"`和`t = "aabb"`。 长度不匹配确保我们只检查`by_len[4]`，它是空的，立即产生“NO”，而没有错误的比较。 

最后，重复的字符`s`创建许多具有相同频率向量的重叠子串。 即使生成过程在概念上插入重复项，集合语义也会折叠它们，从而确保正确性而不增加查找成本。
