 ## 解决方案
 ## 问题理解

 我们得到了几个独立的测试用例。 在每个测试用例中，我们都会看到一排参赛者，每个参赛者都拿着一个由整数表示的某种颜色的气球。 任务是找到这条线的一个连续段，该段仍然包含出现在整个数组中任何位置的每种不同颜色，并且在所有这些段中，我们想要尽可能短的一个。 

换句话说，我们首先隐式确定数组中所有唯一值的集合。 然后我们搜索其值集与该全局集完全匹配的最小子数组。 

关键约束是所有测试用例的元素总数可以达到 10^6。 这立即排除了任何显式检查所有子数组的解决方案。 在最坏的情况下，对所有子数组进行简单的 O(n^2) 枚举将需要大约 10^12 次操作，这远远超出了 2.5 秒内的可行限制。 即使是 O(n^2 log n) 的方法也是没有希望的。 

问题本质上是要求序列中覆盖所有不同元素的最小窗口。 困难不在于识别颜色集，而在于将该集压缩到最小的连续范围。 

一些边缘案例凸显了幼稚推理的陷阱。 如果所有元素都相同，例如 [5, 5, 5, 5]，则答案为 1，因为任何单个元素已经涵盖了所有不同的颜色。 忘记预先计算不同计数的幼稚方法可能仍然有效，但许多不正确的实现意外地从每个位置向外扫描并计数过多。 

另一个微妙的情况是当数组中间已经最小时，例如 [1, 2, 3, 2, 1]。 最佳段是整个数组，因为需要所有颜色，但在 [1, 2, 1, 3, 2] 中，即使存在重复，最小段也是整个数组。 这通常会破坏试图仅从一侧收缩的贪婪尝试。 

## 方法

 强力解决方案将考虑每个可能的子数组，计算其中的元素集，并将其与所有元素的全局集进行比较。 对于每个 O(n^2) 子数组，如果天真地完成，在最坏的情况下构造集合的成本为 O(n)，或者通过仔细维护至少摊销 O(1)，但总更新仍然为 O(n^2)。 这会导致 O(n^3) 或 O(n^2) 行为，具体取决于实现风格，这对于 n 高达 2 × 10^5 来说太慢。 

关键的观察是我们不需要任意子数组。 我们只关心包含所有不同值的窗口。 一旦我们知道全局存在多少个不同的值，问题就变成了经典的“包含所有必需元素的最小窗口”问题。 我们不是从头开始重新计算集合，而是维护一个带有频率计数的滑动窗口。 

我们扩展右指针以包含元素，直到窗口包含所有不同的颜色。 然后我们在保持覆盖范围的同时尽可能缩小左指针。 每次窗口有效时，我们都会更新答案。 每个指针最多移动 n 次，因此每个测试用例的总复杂度呈线性。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(n^3) 或 O(n^2 · 不同) | O(n) | 太慢了|
 | 最优滑动窗口| O(n) | O(n) | 已接受 |

 ## 算法演练

 我们使用数组上的两指针滑动窗口独立地解决每个测试用例。

1. 首先计算数组中不同颜色的总数。 这是通过将所有值插入到集合或频率图中来完成的。 该值表示我们的窗口必须包含的唯一元素的确切数量才有效。 
2. 初始化两个指针，左和右，都从位置 0 开始。我们还维护当前窗口内元素的频率图，以及跟踪窗口中当前满足多少种不同颜色的计数器。 
3. 逐步展开右侧指针。 每次我们包含一个新元素时，我们都会更新它的频率。 如果此元素第一次出现在窗口中，我们将增加覆盖的不同颜色的计数。 我们继续扩展，直到窗口包含所有需要的不同颜色。 
4. 一旦窗口变得有效，我们尝试将其从左侧缩小。 我们向前移动左指针，同时仍然保留所有不同的颜色。 每次删除一个元素时，我们都会更新其频率，如果频率降至零，我们就会减少覆盖的非重复计数。 
5. 在窗口仍然有效的每次成功收缩操作之后，我们用当前窗口长度更新答案。 
6. 继续此过程，直到右指针到达数组末尾。 

关键的想法是，每次修复右边界时，我们都会在不失去有效性的情况下将左边界推得尽可能远，确保每个最小有效窗口都被恰好考虑一次。 

### 为什么它有效

 在任何时刻，算法都会维护一个准确反映左右元素频率分布的窗口。 不变的是，只要窗口被标记为有效，它就包含完整数组的所有不同元素。 因为我们只会在保留有效性的情况下缩小，所以我们永远不会跳过候选最小窗口。 由于每次向左移动时，我们只丢弃不必要的重复元素，因此任何最佳解决方案都必须与某个右端点对齐，并且我们系统地枚举所有这些端点。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        n = int(input())
        a = list(map(int, input().split()))

        total_distinct = len(set(a))

        freq = {}
        have = 0
        left = 0
        ans = n

        for right in range(n):
            x = a[right]
            freq[x] = freq.get(x, 0) + 1
            if freq[x] == 1:
                have += 1

            while have == total_distinct:
                ans = min(ans, right - left + 1)
                y = a[left]
                freq[y] -= 1
                if freq[y] == 0:
                    have -= 1
                left += 1

        print(ans)

if __name__ == "__main__":
    solve()
```该解决方案首先使用一组计算全局存在多少种不同的颜色，该组定义了任何有效窗口的目标条件。 频率字典跟踪每种颜色在当前窗口内出现的次数，并且变量 has 跟踪当前至少表示一次的不同颜色。 

右侧指针扩展窗口，每个新元素要么引入新的独特颜色，要么增加现有频率。 一旦 has 等于total_distinct，窗口就有效，我们尝试从左侧收缩它，同时保持有效性。 答案在每个有效的收缩步骤中更新，确保我们捕获在每个右边界结束的最小窗口。 

一个常见的微妙错误是仅在完全扩展阶段之后才更新答案。 这会错过收缩期间创建的较短的有效窗口。 另一个问题是当频率达到零时未能递减 has，这错误地假设颜色仍然存在。

    import sys
    input = sys.stdin.readline

    t = int(input())
    out = []
    for _ in range(t):
        n = int(input())
        a = list(map(int, input().split()))

        total_distinct = len(set(a))
        freq = {}
        have = 0
        left = 0
        ans = n

        for right in range(n):
            x = a[right]
            freq[x] = freq.get(x, 0) + 1
            if freq[x] == 1:
                have += 1

            while have == total_distinct:
                ans = min(ans, right - left + 1)
                y = a[left]
                freq[y] -= 1
                if freq[y] == 0:
                    have -= 1
                left += 1

        out.append(str(ans))

    return "\n".join(out) + "\n"

# provided sample
assert run("1\n4\n2 2 1 3\n") == "3\n"

# all equal
assert run("1\n5\n7 7 7 7 7\n") == "1\n"

# already minimal spread
assert run("1\n3\n1 2 3\n") == "3\n"

# duplicates with shrink possibility
assert run("1\n5\n1 2 1 3 2\n") == "3\n"

assert run("1\n6\n1 1 2 2 3 3\n") == "3\n"

 ## 边缘情况

 一种边缘情况是所有元素都相同，例如 [4, 4, 4, 4]。 该算法计算出的total_distinct为1。第一次右移到索引0时，窗口已经有效。 收缩步长立即向左增加，直到向右传递，但最小记录窗口仍为 1。答案是正确的 1。 

另一种情况是最小有效窗口位于数组末尾，例如 [1, 2, 3, 3, 4]。 该算法仅在右到达最后一个元素时才有效。 此时，收缩仅删除冗余前缀，最终的最小段正确地变为 [1, 2, 3, 4]，产生长度 4。 

最后一个微妙的情况是重复项严重交错，例如 [1, 2, 1, 2, 3]。 该窗口仅在最后一个元素处变得有效，并且早期的部分窗口永远不会错误地触发有效性，因为它精确地跟踪了不同的存在，从而防止了错误的早期答案。
