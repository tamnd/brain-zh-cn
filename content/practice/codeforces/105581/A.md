 ## 解决方案
 ## 问题理解

 我们正在观察一个只能在离散时刻开始的过程。 Vitya 定期访问扫描仪，每次`t`从时间 0 开始的分钟，一天结束于 时间`T`。 每次访问都是开始扫描新文档的机会，但前提是扫描仪当时处于空闲状态。 一旦文档开始扫描，它就会占用机器一段时间`k`分钟不间断。 如果 Vitya 在扫描仪仍然忙碌时到达，他会立即离开并失去这个机会。 

任务是确定从时间 0 到（包括时间）的时间间隔内 Vitya 成功启动扫描的次数`T`。 

约束很小，所有参数最多为 100。这意味着即使对每次可能的访问进行直接模拟也是高效的。 访问次数最多为 100，因此任何与访问次数呈线性关系的解决方案都可以在限制范围内轻松运行。 

一天的边界出现了微妙的边缘情况。 如果维蒂亚准时到达`T`，他仍然可以尝试开始扫描。 如果扫描仪此时空闲，则会开始新的扫描，即使它可能在工作日结束后结束。 在最后到达有效的情况下，排除最终时间点的简单方法会低估。 

另一个极端情况发生时`k`大于`t`。 在这种情况下，Vitya 经常在繁忙时段到达并错过了很多机会。 例如，如果`t = 2`和`k = 10`，除非两次尝试之间有足够的空闲时间，否则只有第一次成功启动才重要。 

## 方法

 直接模拟是最自然的起点。 每次 Vitya 访问扫描仪时我们都会枚举：`0, t, 2t, ...`最多`T`。 每次访问时，我们都会检查扫描仪是否空闲。 如果是，我们开始新的扫描并将机器标记为忙碌，直到当前时间加上`k`。 否则我们什么也不做。 这是正确的，因为该过程完全由每个到达时间的本地决策定义。 

这种暴力方法最多执行`T / t + 1`检查，最坏情况下最多 101 次操作。 每次检查都是持续不断的工作，因此速度非常快。 

除了这个模拟之外，不需要任何有意义的渐近改进，但关键的观察是我们需要维持的唯一状态是扫描仪下次可用时。 这将问题简化为单个运行变量，而不是一分钟一分钟地跟踪整个时间线。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力模拟| O(T / t) | O(1) | O(1) | 已接受 |
 | 基于事件的贪婪模拟| O(T / t) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们将过程压缩为一系列到达事件，并维护扫描仪下次空闲的时间。 

1. 初始化变量`free_time = 0`，表示新扫描可以开始的最早时刻。 同时初始化`count = 0`存储成功启动的次数。 
2. 迭代所有到达时间`time = 0, t, 2t, ...`尽管`time ≤ T`。 每个值代表 Vitya 检查扫描仪的时刻。 
3. 对于每个`time`，将其与`free_time`。 如果`time ≥ free_time`，此时扫描仪处于空闲状态，因此可以开始新文档。 
4. 当扫描开始于`time`, 增量`count`并设置`free_time = time + k`，因为扫描仪将在下一个任务中保持忙碌状态`k`分钟。 
5. 如果`time < free_time`，Vitya 在正在进行的扫描期间到达，但不执行任何操作，保持状态不变。 

该过程一直持续到所有到达时刻为止（包括）`T`被处理。 

关键属性是`free_time`始终代表最近开始的扫描的确切结束时间。 每个决定仅取决于当前到达是否已到达或超过该边界。 早期历史不再重要`free_time`已更新。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

t, k, T = map(int, input().split())

free_time = 0
count = 0

time = 0
while time <= T:
    if time >= free_time:
        count += 1
        free_time = time + k
    time += t

print(count)
```该代码维护一个时间线指针`time`跳跃的步骤为`t`，这与 Vitya 的抵达时间表完全一致。 变量`free_time`在每次到达时对扫描仪是忙还是空闲进行编码。 

一个常见的错误是忘记扫描开始时间`T`即使在之后完成也有效`T`。 该条件仅检查开始时间，而不检查扫描是否在工作日内完成。 

另一个微妙的点是初始化。 环境`free_time = 0`确保在时间 0 的第一个到达总是开始扫描，与系统开始空闲的事实相匹配。

## 测试用例```python
import sys, io

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    import sys as _sys
    input = _sys.stdin.readline

    t, k, T = map(int, input().split())

    free_time = 0
    count = 0

    time = 0
    while time <= T:
        if time >= free_time:
            count += 1
            free_time = time + k
        time += t

    return str(count)

# provided samples
assert run("4 5 17\n") == "3"
assert run("5 5 20\n") == "5"

# minimum values
assert run("1 1 1\n") == "2"

# large k blocking most attempts
assert run("2 10 10\n") == "1"

# t equals T boundary case
assert run("7 3 7\n") == "2"

# no overlap case
assert run("3 1 9\n") == "4"
```| 测试输入| 预期产出 | 它验证了什么 |
 | --- | --- | --- |
 | 1 1 1 | 1 1 1 2 | 时间 0 和 T 处的边界包含 |
 | 2 10 10 | 2 10 10 1 | 长扫描会阻止未来的尝试
 | 7 3 7 | 7 3 7 2 | 恰好在当天结束时抵达计算 |

 ## 边缘情况

 当首次到达发生在时间 0 时，系统必须将扫描仪视为立即可用。 有输入`1 5 3`，到达的顺序是 0, 1, 2, 3。在时间 0 开始扫描并阻塞，直到时间 5。每个后来的到达都被忽略，因为每个都小于`free_time`，所以答案是 1。模拟自然地处理这个问题，因为`free_time`被初始化为0。 

当`k`小于`t`，每次到达都会开始扫描，因为扫描仪总是在下次访问之前完成。 为了`t = 4, k = 1, T = 12`，到达时间为 0、4、8、12，并且每次都满足`time ≥ free_time`，因此每一步都会增加计数。 国家演变为`free_time = 1, 5, 9, 13`。 

什么时候`k`大于`t`，重叠扫描占主导地位。 为了`t = 2, k = 6, T = 10`，到达次数为 0、2、4、6、8、10。只有 0、6 和 10 开始扫描，因为中间到达次数落在活动间隔内。 条件`time ≥ free_time`正确过滤这些情况，而不需要任何关于间隔的明确推理。
