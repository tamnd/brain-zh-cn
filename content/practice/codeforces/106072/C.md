 ## 解决方案
 ## 问题理解

 我们得到了一系列编程问题，每个问题都已经准确地标记了三个学生中哪一个可以解决它。 每个问题根据其在学生 1、2 和 3 中设置的可解决性分为七个类别之一。有些问题只能由一名学生解决，有些问题只能由两个特定学生解决，有些问题可以由所有三个学生解决。 

我们的任务不是决定一个问题是否可以解决，而是将每个问题分配给能够解决它的学生之一。 分配后，每个问题都被视为由一名学生解决，即我们分配给该学生的学生。 目标是分配作业，以便分配问题最少的学生尽可能多。 换句话说，我们希望在所有作业确定后，最大化三个学生的最小工作量。 

这些约束允许最多 100,000 个测试用例，每个类别的值最多为 7 × 10^8，因此任何解决方案都必须在每个测试用例的恒定时间内运行。 任何涉及问题模拟或作业搜索的事情都是立即不可行的，因为问题总数太大而无法明确列举。 

当尝试没有结构的贪婪分配时，会出现微妙的失败模式。 例如，如果我们总是把共同的问题交给当前最弱的学生，那么当我们可以更好地重新分配时，我们可能会陷入困境。 

考虑一个简单的情况，其中 F1 = F4 = F7 = 1，其他为零。 天真的贪婪方法可能会首先任意分配 F7，然后无法最佳地平衡以后的分配，即使所有学生的正确答案显然都是 1。 这表明共享类别的局部决策可能会破坏全局最优性。 

主要困难在于类别 F3、F5、F6 和 F7 在学生之间造成耦合。 这些共同的问题是灵活的资源，必须仔细分配。 

## 方法

 强力解释是单独处理每个问题，并尝试将每个问题的所有可能分配给有效的学生。 每个F7题有3个选择，F3、F5、F6各有2个选择，其他都有固定的作业。 这导致配置数量呈指数级增长，大约为 3^{F7} × 2^{F3+F5+F6}，即使对于小输入来说，这也是一个天文数字。 

从概念上讲，这是可行的，因为它探索了每个有效的分布，但由于组合爆炸，它立即失败。 问题的结构呈现出对称性：仅涉及每个学生的总数，而不涉及单个问题的身份。 

关键的观察结果是，我们只关心每个学生的最终计数，并且每个类别都以线性、结构化的方式做出贡献。 单学生类别 F1、F2、F4 为固定贡献。 混合类别可以重新分配，但只能在有限的自由度内进行。 

我们不再考虑作业，而是考虑每个类别提供的灵活性。 每个 F7 问题在所有三个学生之间贡献 1 个单位的灵活性，每个 F3 问题在学生 1 和 2 之间贡献灵活性，依此类推。 该问题变成了一个约束平衡问题：我们希望最大化 x 值，以便每个学生至少收到 x 份作业。 

我们可以将 x 作为候选答案并检查可行性。 对于固定的 x，我们尝试看看是否可以分配共享问题以确保每个学生至少达到 x。 这将问题简化为检查是否可以使用共享池来平衡剩余容量。

为了可行性，我们首先分配所有强制贡献。 然后我们计算每个学生的赤字。 然后以结构化方式贪婪地使用共享类别：我们首先使用两个学生共享池来弥补赤字，最后使用完全灵活的池 F7 来弥补剩余的不平衡。 这是可行的，因为 F7 是唯一可以任意定向的资源。 

我们可以通过推导封闭形式而不是二分搜索来进一步简化。 限制因素是在尊重总和限制的情况下，每个学生可以从共享类别中获得多少提升。

 ## 算法演练

 我们首先将这些类别重新解释为对三个总数的贡献。 学生 1 最初收到 F1 的所有问题以及我们决定从 F3、F5 和 F7 分配的问题。 对于其他人来说也是如此。 

1. 首先布置强制问题。 学生 1 获得 F1，学生 2 获得 F2，学生 3 获得 F4。 这部分无法更改，因此它构成了基线。 
2. 将 F3 视为只能在学生 1 和 2 之间分配的池。类似地，F5 在 1 和 3 之间分配，F6 在 2 和 3 之间分配。这些充当成对平衡工具。 
3. 关键的限制是每个池只能帮助两个特定的学生，因此不能完全解决全局失衡问题。 仅可使用 F7 来同时调整所有三个。 
4. 假设我们希望每个学生至少达到 x。 在使用所有单个学生的贡献后，我们计算每个学生相对于基线的缺失量。 
5. 然后，我们贪婪地分配成对池，以尽可能减少赤字。 例如，F3用于以最有效的方式同时减少学生1和学生2的赤字。 
6. 耗尽成对池后，必须使用 F7 弥补剩余的不足。 每个F7问题为任何学生贡献一个单位，因此可以自由地减少最大赤字差距。 
7. 答案是最大x，使得所需补偿总额不超过可用的灵活资源。 

### 为什么它有效

 该结构简化为仅具有一种全局灵活资源的流式平衡系统。 成对资源只能减少局部不平衡，无法在其端点之外创建全局重新分配。 因此，任何可行的解决方案完全取决于完全灵活的池F7在优化使用成对池后是否可以覆盖剩余的不平衡。 由于所有其他分配都是线性且独立的，因此可行性条件变得精确并且不需要对分配进行搜索。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    T = int(input())
    out = []
    
    for _ in range(T):
        S = int(input())
        f1, f2, f3, f4, f5, f6, f7 = map(int, input().split())
        
        # baseline contributions
        a1 = f1
        a2 = f2
        a3 = f4
        
        # pair pools
        p12 = f3
        p13 = f5
        p23 = f6
        
        # total sum constraint: a1+a2+a3 + p12+p13+p23+f7 = S
        
        # We binary search x
        lo, hi = 0, S
        
        def can(x):
            need1 = max(0, x - a1)
            need2 = max(0, x - a2)
            need3 = max(0, x - a3)
            
            # use pair pools greedily
            use12 = min(p12, need1 + need2)
            # split as much as possible
            take1 = min(need1, use12)
            need1 -= take1
            need2 -= (use12 - take1)
            
            use13 = min(p13, need1 + need3)
            take1 = min(need1, use13)
            need1 -= take1
            need3 -= (use13 - take1)
            
            use23 = min(p23, need2 + need3)
            take2 = min(need2, use23)
            need2 -= take2
            need3 -= (use23 - take2)
            
            total_need = need1 + need2 + need3
            return total_need <= f7
        
        ans = 0
        while lo <= hi:
            mid = (lo + hi) // 2
            if can(mid):
                ans = mid
                lo = mid + 1
            else:
                hi = mid - 1
        
        out.append(str(ans))
    
    print("\n".join(out))

if __name__ == "__main__":
    solve()
```该代码对候选最小值 x 实施可行性检查。 首先分离基线计数，反映强制类别。 功能`can(x)`计算每个学生需要多少额外作业。 

关键的实现细节是成对池的贪婪消耗。 每个池的应用方式首先满足其两个端点中尽可能多的未满足需求。 这种顺序很重要，因为每个池都是独立的，一旦使用，其容量就消失了。 

即使可行性检查在 x 中被视为单调，二分搜索也能确保正确性。 增加x只能增加赤字，所以一旦某个值变得不可行，所有更大的值也都不可行。

 ## 边缘情况

 一个关键的边缘情况是当所有问题都属于单个学生类别 F1、F2 和 F4 时。 在这种情况下，不存在共享灵活性，因此答案只是这三个计数中的最小值。 该算法处理此问题是因为所有成对池和 F7 池均为零，因此 x 的可行性仅取决于基线值。 

另一个边缘情况是所有问题都在 F7 中。 这里每个问题都是完全灵活的，因此最佳解决方案是 S // 3。该算法通过 F7 有效地对此进行建模，覆盖基线零初始化后的所有缺陷。 

第三种边缘情况是当成对池很大但 F7 为零时。 这暴露了成对平衡无法修复三向不平衡，并且只要一个学生在用尽成对转移后仍然不满意，可行性检查就会正确失败。
