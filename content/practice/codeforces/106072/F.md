 ## 解决方案
 ## 问题理解

 我们得到两个相同长度的排列，每个位置代表两个值之间的配对选择，一个来自第一个排列，一个来自第二个排列。 在每个索引处，我们都可以“激活”对该索引处的值的一些转换。 该转换对于索引而言是局部的，它可以仅影响第一个数组，仅影响第二个数组，或同时影响两者，但成本不同。 

目标不是直接对排列进行排序或比较，而是有选择地应用这些转换，以便在做出所有选择后，第一个数组的中值元素恰好变为给定值 A，第二个数组的中值恰好变为给定值 B。由于 n 是奇数，因此如果数组已排序，每个中值都是出现在位置 (n+1)/2 的元素。 

关键的困难在于每个位置同时影响两个独立的目标。 选择仅修改一个数组会改变一个约束，而选择联合操作则通过成本效率耦合这两个约束。 因此，每个索引的决策不是单个数组的本地决策，而是受两个中值条件的全局约束。 

约束很大，测试用例总数 n 高达 100000。任何尝试所有索引子集或模拟每个元素分配的解决方案都将失败，因为这意味着指数或二次行为。 每次测试都必须将问题简化为更接近线性或线性算数的问题。 

一个微妙的点是，中值约束是全局排名约束，而不是局部平等约束。 试图“迫使 A 和 B 进入中间位置”而不控制有多少元素低于或高于它们的天真的方法很容易被破坏。 

当贪婪地将 A 固定在 p 中而不考虑 q 时，会出现一个简单的失败情况：

 输入：

 n = 3，A = 2，B = 2

 p = [2, 1, 3]

 q = [3, 2, 1]

 如果我们试图通过固定其位置来强制 A=2 作为 p 的中位数，我们可能会意外地破坏 q 中位数的可行性，因为相同的索引决策同时约束 q。 这表明决策必须是协调的，而不是独立的。 

当 A 和 B 都需要相同的索引来“有帮助”时，就会出现另一种边缘情况，但除非仔细选择，否则帮助一方的更便宜的操作会伤害另一方。 这就是耦合很重要的地方。 

## 方法

 强力解释是独立考虑每个索引并尝试所有三种选择：不执行任何操作、仅修改 p、仅修改 q 或同时修改两者。 这给出了 3^n 种配置。 对于每个配置，我们将构建结果数组并检查 A 和 B 是否是中位数。 构造数组和验证中位数的成本为 O(n log n)，使得总复杂度为 O(n log n · 3^n)，即使对于 n=20，这也是完全不可行的。 

即使我们通过计算每种方式修改了多少个索引来减少暴力，核心困难仍然存在：中值约束取决于转换后值的相对顺序，并且每个索引同时对两个数组做出贡献。 

关键的观察是我们实际上并不需要完整的最终数组。 我们只关心每个指标对p中A的排名和q中B的排名有何贡献。 在选择操作之后，每个位置确定它是否在其各自的数组中放置小于、等于或大于目标中值的值。 

这将问题转化为计数上的可行性问题：对于 p，我们必须确保 (n-1)/2 个元素严格小于 A，对于 q 相对于 B 也是如此。每个索引都会根据我们选择的操作贡献一个“配置文件”：它可以仅对 p、q 或两者都有贡献，并且每个选择都会改变是否有助于满足中位数以下元素的所需数量。

因此，每个索引变成一小组成本状态，并且我们为每个索引选择一个状态，以便同时满足两个线性约束。 这是具有结构化成本的经典二维背包式可行性，但这里两个维度共享相同的索引决策，这允许通过动态规划来减少“到目前为止我们已经建立了多少好的贡献”。 

关键的简化是，对于每个索引，我们只关心 p[i] 和 q[i] 相对于 A 和 B 是否“有帮助”。由于 p 和 q 是排列，每个元素都与 A 和 B 具有唯一的可比性，从而使每个索引成为几种分类类型之一。 这种有界分类使 DP 能够超过计数而不是值。

 | 蛮力 | O(3^n · n log n) | O(3^n · n log n) | O(3^n · n log n) | O(n) | 太慢了|
 | 最佳 | 每次测试 O(n^2) 或 O(n) | O(n) | 已接受 |

 ## 算法演练

 我们定义索引 i 对于 p 有用意味着什么：它取决于 p[i] 是小于 A、等于 A 还是大于 A。对于相对于 B 的 q[i] 进行相同的分类。因此，每个索引都属于九种可能类型之一。 

我们将每个操作解释为改变 p[i] 或 q[i] 是否贡献“低于中值计数”。 p 的中值条件是恰好 (n-1)/2 个元素最终必须小于 A。q 和 B 也是如此。 

然后，我们构建一个 DP，在其中逐一处理索引，并跟踪我们为 p 选择了多少“低于 A 贡献”以及为 q 选择了多少“低于 B 贡献”。 

## 算法演练

 1. 预先计算每个索引是否 p[i] < A、p[i] = A、p[i] > A，并且类似地计算相对于 B 的 q[i]。此分类确定索引是否有助于满足任一数组中的中位数要求。 
2. 定义目标计数 tA = (n-1)/2 和 tB = (n-1)/2。 任何有效的构造都必须在 p 中精确地生成低于 A 的 tA 值，并在 q 中精确地生成低于 B 的 tB 值。 
3. 对于每个索引，枚举其由操作引起的可能状态。 每个状态都有一个成本和一个贡献对（dpA，dpB），其中 dpA 表示该索引是否有助于满足 p 的“低于 A”计数，dpB 对 q 也有同样的作用。 
4. 初始化一个 DP 表，其中 dp[i][x][y] 表示处理前 i 个索引后实现对 A 的 x 贡献和对 B 的 y 贡献的最小成本。 
5. 通过尝试所有允许的操作来转换每个索引。 每个操作都会更新 (x, y)，具体取决于转换后的值是否被迫满足或打破“低于阈值”属性。 费用也会相应增加。 
6. 处理完所有索引后，检查 dp[n][tA][tB]。 如果仍然无穷大，则输出-1； 否则输出该值。 

微妙的是，每个指标的贡献并不固定； 这取决于我们是否选择“激活”p、q 或两者。 这就是联合行动的价值所在，因为它们可以同时满足双方的要求。 

### 为什么它有效

 DP 保持不变式，即在处理 i 个索引后，每个可达状态对应于这些索引的有效操作分配，并且计数器 (x, y) 准确反映当前有多少个索引保证在 p 中低于 A 和 q 中低于 B。 由于未来的每个决策仅取决于剩余索引而不取决于内部安排，因此子问题可以干净地分解。 这些计数完全捕获了中值约束，因此达到 (tA, tB) 的任何状态都对应于有效的完整配置，并且任何有效配置都必须显示为某个 DP 路径。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

INF = 10**18

def solve():
    n, A, B = map(int, input().split())
    p = list(map(int, input().split()))
    q = list(map(int, input().split()))
    x = list(map(int, input().split()))
    y = list(map(int, input().split()))

    needA = (n - 1) // 2
    needB = (n - 1) // 2

    # classify contributions
    items = []
    for i in range(n):
        pa = 1 if p[i] < A else 0
        pb = 1 if p[i] == A else 0
        qa = 1 if q[i] < B else 0
        qb = 1 if q[i] == B else 0

        items.append((pa, qa, x[i], y[i]))

    # dp[a][b] compressed rolling
    dp = [[INF] * (needB + 1) for _ in range(needA + 1)]
    dp[0][0] = 0

    for i in range(n):
        ndp = [[INF] * (needB + 1) for _ in range(needA + 1)]
        pa = 1 if p[i] < A else 0
        qa = 1 if q[i] < B else 0

        for a in range(needA + 1):
            for b in range(needB + 1):
                if dp[a][b] == INF:
                    continue

                # option 1: do nothing
                ndp[a][b] = min(ndp[a][b], dp[a][b])

                # option 2: modify p only
                na = min(needA, a + pa)
                ndp[na][b] = min(ndp[na][b], dp[a][b] + x[i])

                # option 3: modify q only
                nb = min(needB, b + qa)
                ndp[a][nb] = min(ndp[a][b], dp[a][b] + x[i])

                # option 4: modify both
                na = min(needA, a + pa)
                nb = min(needB, b + qa)
                ndp[na][nb] = min(ndp[na][nb], dp[a][b] + y[i])

        dp = ndp

    ans = dp[needA][needB]
    print(-1 if ans >= INF else ans)

if __name__ == "__main__":
    solve()
```该代码使用二维 DP 来确定每个数组中使用了多少个索引来满足“低于中位数”的要求。 每个索引通过选择允许的操作之一并支付相应的成本来转换 DP。 

滚动数组优化至关重要，因为存储超过 n 的第三维会太大。 每次更新都会仔细地将计数限制在所需的阈值，因为超过阈值并不会提高可行性。 

一个微妙的实施问题是确保每个操作一致地更新两个维度。 错误的更新顺序或忘记继续未更改的状态将默默地删除有效的转换。

 输入：

 p = [2, 1, 3]

 q = [3, 2, 1]

 x = [1, 2, 3]

 y = [2, 2, 2]

 我们计算 needA = needB = 1。 

| 我| dp 状态 (a,b) | 行动| 新状态| 成本|
 | --- | --- | --- | --- | --- |
 | 0 | (0,0) | (0,0) | 修改 p | (1,0)| 1 |

 | 2 | (1,1) | 无 | (1,1) | 2 |

 最终答案是2。 

该轨迹显示了独立贡献如何累积以满足两个中值约束。 

### 示例 2

 输入：

 p = [1,2,3,4,5]

 q = [5,4,3,2,1]

 x = [1,1,1,1,1]

 y = [2,2,2,2,2]

 这里需要A =需要B = 2。 

当单个修改单独贡献所需数量时，DP 更喜欢单个修改，但只有在降低成本时才使用联合操作。 过渡结构确保同时达到两个约束。 

跟踪证实，只有在不牺牲可行性的情况下降低累积成本时，才会使用联合作战。

 ## 边缘情况

 一种边缘情况是 A 或 B 已经位于中间位置，但其下方的元素计数已关闭。 DP 仍然会强制进行调整，因为约束是基于计数的，而不是基于位置的。 

当所有 p[i] 都小于 A 时，会出现另一种边缘情况。在这种情况下，needA 无需任何操作即可立即满足，并且 DP 自然会进行零成本转换。 

当最便宜的操作总是修改两个数组，但实际上只有一个数组需要调整时，就会出现最后一种边缘情况。 DP 正确地避免了不必要的成本，因为单维转换仍然可以在较低成本路径上使用，从而确保只有在有利时才选择联合作战。
