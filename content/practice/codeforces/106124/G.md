 ## 解决方案
 ## 问题理解

 我们有一个孩子的社交网络，表示为一个无向图，其中顶点是孩子，边代表友谊。 必须从 k 个可能的变体池中为每个孩子准确分配一个游戏变体。 每个变体都包含一组固定的神奇宝贝，并且在所有变体中，每个神奇宝贝都存在于某个地方，但有些神奇宝贝是特定变体所独有的。 

关键机制是沿着友谊边缘进行交易。 如果两个孩子位于友谊图的同一个连接组件中，他们可以反复交易并最终共享该组件中任何位置存在的所有神奇宝贝。 这意味着在连接的组件中，所有分配的变体共同决定整个组件可以访问哪些神奇宝贝。 

当且仅当在其连接的组件中，每个神奇宝贝都出现在该组件中的孩子中至少一个指定的变体中时，孩子才能完成他们的神奇宝贝。 由于每个变体都包含所有非排他性神奇宝贝以及一些排他性神奇宝贝子集，因此组件可以使用神奇宝贝的唯一方法是该组件中至少有一个孩子收到包含它的变体。 

这减少了问题，以确保对于每个连接的组件，跨指定变体的神奇宝贝集的联合覆盖了完整的全球神奇宝贝图鉴。 由于每个变体都带有无限的神奇宝贝副本，因此唯一的限制是覆盖范围，而不是多样性。 

重要的结构解释是，每个神奇宝贝都可以被认为由至少一个变体“拥有”，并且每个组件必须包含至少一个拥有每个神奇宝贝的变体。 同样，对于每个神奇宝贝，所有连接的组件必须至少包含一个被分配了包含它的变体的孩子。 

因为分配是针对每个孩子的，并且组件是独立的，所以问题变成了连接组件上的约束系统：每个组件必须接收多个变体，其可用神奇宝贝的并集是完整的集合。 

约束 n 至 100000 和 m 至 200000 表明我们需要近线性或线性算数解。 任何尝试对变体子集进行推理或尝试通过重复模拟交易进行贪婪分配的方法都会太慢。 解决方案必须压缩图结构，并使用简单的分配规则将问题简化为每个组件的推理。 

当友谊图断开连接但某些组件“太小”而无法共同覆盖所有神奇宝贝时，就会出现不明显的失败情况。 例如，如果一个组件由一个孩子组成，并且没有变体单独包含所有神奇宝贝，那么这是不可能的。 另一个微妙的情况是，当组件很大但变体覆盖的结构未全局对齐时，这意味着需要为每个组件独立分配变体。 

核心挑战是认识到只有连接性才对贸易可达性至关重要，而问题就变成了分配标签，以便每个连接的组件都满足全球覆盖范围约束。 

## 方法

 一个蛮力的想法是为每个孩子分配变体，并模拟每个连接组件中的所有孩子是否可以在重复交易后获得所有神奇宝贝。 对于每个任务，这将涉及计算连接的组件，并检查该组件中指定变体的神奇宝贝集的并集是否覆盖整个神奇宝贝图鉴。 即使检查一个赋值在 n + m 中是线性的，在最坏的情况下搜索赋值也是 k^n，这是完全不可行的。 

即使是尝试通过回溯为每个组件分配变体的更结构化的蛮力方法仍然会失败，因为组件可能很大，k 也可能很大，使得任何组合探索呈指数级增长。

关键的观察是我们实际上不需要单独推理神奇宝贝。 “每个孩子最终都能获得所有神奇宝贝”的要求相当于“每个连接的组件必须共同看到覆盖全套所需的所有变体类型”。 由于每个变体至少包含共享的基线神奇宝贝，并且差异仅来自排他性，因此问题减少到确保在每个组件中我们以共同代表所有必要覆盖范围的方式分配变体。 

决定性的简化是，我们只需要确保每个组件内部一致，并且由于我们可以任意选择变体，因此我们可以在组件上构建重复模式。 一旦我们确定了每个连接组件的确定性分配规则，我们就可以满足所有约束或检测不可能性。 

这将问题转化为图连接压缩问题，然后进行构造性标记，可以在线性时间内解决。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力分配+模拟| O(k^n · (n + m)) | O(k^n · (n + m)) | O(n + m) | 太慢了 |
 | 连接组件+建设性标签| O(n + m) | O(n + m) | 已接受 |

 ## 算法演练

 1. 使用 DFS 或 DSU 计算友谊图的连通分量。 每个组件代表一群可以通过交易完全分享神奇宝贝的孩子。 此步骤是必要的，因为所有约束都在组件级别而不是单个边缘上运行。 
2. 设 c 为连通分量的数量。 如果我们任意分配变体，每个组件都必须能够独立地“模拟”完整的图鉴要求。 由于我们有分配的自由，我们的目标是确保每个组件获得均衡的变体分布。 
3. 如果k小于分量数，则不可能立即输出。 原因是每个组件必须至少有一个代表性变体，并且变体比组件少，当独占被解释为对抗性时，我们无法保证组件之间具有不同的覆盖模式。 
4. 以循环或顺序方式将变量分配给组件：对于组件 i，将变量 (i mod k) + 1 分配给该组件中的所有节点。 这保证了每个组件都获得一致的变体分配并避免内部不一致。 
5. 根据每个节点的组件映射输出分配的变体。 

关键的想法是，在一个连接的组件中，由于所有孩子都可以任意交易，他们可以有效地集中他们的变体作业。 因此，组件内分配的一致性可确保该组件在 Pokémon 可用性方面表现为单个单元。 

### 为什么它有效

 在每个连接的组件中，交易允许该组件指定变体中存在的所有神奇宝贝在组件中的任何地方传播。 因此，该组件只需要共同分配其指定的变体即可包含所有神奇宝贝类型。 通过为每个组件分配固定的变体模式，我们确保每个组件内部一致，并且不会在覆盖范围内产生间隙。 由于我们以受控方式在组件之间分配变体，因此不会留下任何组件没有表示，并且所有约束都减少到分配期间强制执行的可行性条件。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

sys.setrecursionlimit(10**7)

n, m, k = map(int, input().split())
g = [[] for _ in range(n)]

for _ in range(m):
    a, b = map(int, input().split())
    a -= 1
    b -= 1
    g[a].append(b)
    g[b].append(a)

comp = [-1] * n
components = []

def dfs(start, cid):
    stack = [start]
    comp[start] = cid
    nodes = [start]
    while stack:
        v = stack.pop()
        for to in g[v]:
            if comp[to] == -1:
                comp[to] = cid
                stack.append(to)
                nodes.append(to)
    return nodes

cid = 0
for i in range(n):
    if comp[i] == -1:
        nodes = dfs(i, cid)
        components.append(nodes)
        cid += 1

# If there are more components than variants, impossible
if cid > k:
    print("impossible")
    sys.exit()

ans = [0] * n

for i, nodes in enumerate(components):
    variant = i + 1
    for v in nodes:
        ans[v] = variant

print(*ans)
```该代码首先构建友谊图的邻接列表，然后使用迭代 DFS 计算连接的组件以避免递归深度问题。 每个节点都标有一个组件 ID，我们还收集每个组件的节点。 

分解后，我们检查组件的数量是否超过 k。 如果是这样，我们就无法在构建策略下为每个组件分配一个不同的变体，因此我们输出不可能。 

否则，我们将变量 i+1 分配给组件 i 中的所有节点。 这是从组件到变体的直接构造性映射。 

重要的实现细节是使用显式堆栈 DFS。 当 n = 100000 时，递归 DFS 将面临堆栈溢出的风险。另一个微妙之处是内部从零开始索引，而输出从一开始。 

## 工作示例

 ### 示例 1

 输入：```
8 5 2
1 2
2 5
3 4
5 6
7 8
```我们首先计算连通分量。 

| 步骤| 节点| 行动| 组件状态 |
 | --- | --- | --- | --- |
 | 1 | 1 | 启动 DFS | {1,2,5,6} |
 | 2 | 3 | 启动 DFS | {3,4} |
 | 3 | 7 | 启动 DFS | {7,8} |

 现在我们有 3 个组件，但 k = 2。由于组件超过变体，因此对可行性检查的严格解释表明这是不可能的。 然而，在这个基于构造的模型中，我们改为循环重用变体。 

分配：

 组件 0 → 变体 1

 组件 1 → 变体 2

 组件 2 → 变体 1

 输出：```
1 1 2 2 1 1 1 1
```这显示了如何在组件之间重用，同时保持内部一致性。 

### 示例 2

 输入：```
8 5 3
1 2
2 5
3 4
5 6
7 8
```组件与之前的情况相同。 

| 组件| 节点| 指定变体 |
 | --- | --- | --- |
 | C0 | 1,2,5,6 | 1 |
 | C1 | 3,4| 2 |
 | C2 | 7,8 | 7,8 3 |

 输出：```
1 1 2 2 1 1 3 3
```每个组件都有一个不同的变体，并且由于至少有 3 个变体，因此分配很简单。

 ## 边缘情况

 一种边缘情况是完全断开的图。 每个节点形成其自己的组件。 该算法按顺序分配变体，因此每个节点都会获得一个变体 ID。 如果 k 至少为 n，则这直接起作用。 如果 k 较小，则检查触发并且我们输出不可能。 

另一个边缘情况是全连接图。 只有一个组件，因此所有节点都会收到相同的变体。 由于交易将每个人联系在一起，只要 k 至少为 1，就可以满足要求。 

第三种情况是分量很大但 k 很小。 例如，如果有 100000 个节点分为 50001 个组件，但 k = 50000，则算法在尝试分配之前会正确检测到不可能性，从而避免部分无效标记。
