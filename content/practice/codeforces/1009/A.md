 ## 解决方案
 ## 问题理解

 商店里有一系列游戏，每种都有固定的价格，马克西姆严格地从左到右浏览它们。 与此同时，他携带的钱包就像一排钞票：只考虑前面的钞票，它要么被消耗掉，要么留在原地，具体取决于它是否足以支付当前游戏的费用。 

在每个游戏位置，马克西姆只进行一次尝试：他查看钱包中的第一张钞票。 如果钱包是空的，他根本无法与该游戏互动，只能继续前进。 如果第一张钞票的价值至少等于游戏价格，他就会购买该游戏，然后该钞票就会从钱包中消失。 否则，他无法购买并保留相同的钞票，然后继续进行下一场游戏。 关键的一点是，不成功的尝试不会使钱包指针前进，只有成功的购买才会前进。 

任务是按顺序处理完所有游戏后，确定有多少游戏被成功购买。 

限制很小，游戏数量和账单数量都高达 1000。这可以立即允许在二次最坏情况下进行直接模拟，而不会出现超时风险。 任何最多大约 10^6 次操作的解决方案都是非常安全的。 

主要的微妙之处在于正确地对购买失败时相同账单的持久性进行建模。 一个常见的错误是无论成功与否，总是将钱包指针向前推进，这完全改变了流程并导致少算购买。 

当游戏结束前钱包变空时，就会出现第二种边缘情况。 在这种情况下，必须跳过所有剩余的比赛，因为不可能进行进一步的比较。 例如，如果所有账单都提前消耗完毕，则答案只是迄今为止的购买数量，没有进一步的交互。 

## 方法

 思考这个过程的简单方法是逐字模拟所描述的每个步骤。 我们维护游戏索引和指向第一个未使用账单的指针。 对于每场比赛，我们都会检查当前的账单。 如果足够，我们就消耗它并将账单指针向前移动。 如果没有，我们保持钞票指针固定并进入下一个游戏。 

这个模拟已经足够高效了，因为每张账单只能消费一次，每个游戏也被处理一次。 这意味着最多 n + m 次指针移动会实际改变状态，最多 n 次不成功的检查不会改变钱包状态。 最坏的情况仍然与游戏数量呈线性关系。 

更概念化的观点是，该过程是一个两指针行走。 一根指针在游戏中严格向前移动。 仅当成功购买时，第二个指针才会在钱包中移动。 关键的见解是，没有任何事情需要回溯或重新审视早期的游戏或账单。 每个决定都是局部的且不可逆转。 

这消除了任何嵌套循环或搜索的需要。 模拟直接反映了流程的结构。

 ## 算法演练

 我们维护两个索引：一个用于游戏，一个用于钱包中的当前账单。 我们还设有一个成功购买的柜台。 

1. 初始化一个指针`j = 0`第一张账单和柜台`ans = 0`。 指针代表钱包中当前有效的账单。 
2. 迭代每个游戏`i`从左到右。 在每一步中，我们都会考虑是否可以尝试购买。 
3. 如果钱包是空的（`j == m`），我们完全停止检查钱包并继续跳过剩余的游戏。 这是有效的，因为一旦所有账单用完，就不可能再进行购买。 
4. 否则，比较当前账单金额`a[j]`以当前游戏成本`c[i]`。 
5. 如果`a[j] >= c[i]`，增加答案，因为游戏已购买，并通过设置将钱包指针向前移动`j += 1`。 这对账单的消费进行建模。 
6.如果`a[j] < c[i]`，对钱包指针不执行任何操作。 相同的账单仍可用于下一场比赛，因此我们只需转到下一场比赛索引即可。 
7. 继续，直到处理完所有游戏。 

关键的设计选择是钱包指针仅在成功时移动，这完全符合失败的尝试不会消耗账单的规则。 

### 为什么它有效

 在任何时刻，该算法都会保留以下事实：`a[j]`是第一个未消费的账单。 每个游戏要么消耗这笔账单，要么保持不变。 由于没有任何操作会跳过或修改较早的账单，因此指针`j`始终正确跟踪下一个可用资源。 每次成功购买都会严格减少钱包大小，并且每个游戏都会被评估一次，因此不会跳过或重复任何有效的购买机会。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    c = list(map(int, input().split()))
    a = list(map(int, input().split()))
    
    j = 0
    ans = 0
    
    for i in range(n):
        if j == m:
            break
        if a[j] >= c[i]:
            ans += 1
            j += 1
    
    print(ans)

if __name__ == "__main__":
    solve()
```该解决方案是围绕游戏的单次传递构建的。 指针`j`仅在消费账单时才会预支，与流程中描述的确切行为相匹配。 提前退出时`j == m`一旦钱包空了，就可以防止不必要的比较，尽管即使没有钱包，解决方案仍然是正确的。 

一个常见的实施错误是增加`j`即使购买失败。 这打破了不变量`a[j]`始终是第一个可用的账单，并导致人为地跳过可用的账单。

 ## 边缘情况

 当每场游戏的第一张钞票都太小时，就会出现一种边缘情况。 在这种情况下，钱包指针永远不会移动，答案仍然为零。 例如，`c = [5, 6, 7]`和`a = [1]`结果根本没有任何进展，因为单张账单从未被消耗过。 

另一个边缘情况是所有游戏与账单相比都非常便宜。 然后，每个游戏都会消耗一张账单，直到任一列表结束。 为了`c = [1, 1, 1]`和`a = [10, 10]`，每次游戏时指针都会前进，直到第二张钞票被消耗完为止，产生答案 2。
