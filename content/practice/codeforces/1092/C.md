 ## 解决方案
 ## 问题理解

 我们得到一个字符串集合，这些字符串全部来自一个长度为的隐藏字符串$n$。 从该隐藏字符串中，获取每个正确的前缀和每个正确的后缀，因此对于每个长度$1$到$n-1$正好有两个字符串：一个前缀和一个后缀，但它们混合在一起并打乱了顺序。 

任务不是唯一地重建字符串。 相反，我们只需要决定，对于每个提供的字符串，是否应该将其标记为前缀或后缀，其方式与至少一个有效的原始字符串一致。 可能存在许多有效的分配，并且其中任何一项都是可接受的。 

关键的结构性约束是$2n-2$字符串，每个长度正好有两个。 这立即表明相同长度的字符串必须紧密相关，因为每个长度恰好对应于一个前缀和一个后缀。 

约束条件$n \le 100$意味着输入大小足够小，我们可以承受$O(n^3)$推理或重复字符串比较而无需担心性能。 重要的是正确性和利用结构，而不是渐近优化压力。 

当两个长度相同的字符串相等或几乎相同时，歧义就会产生一个微妙的困难。 例如，如果多个前缀和后缀一致，天真的贪婪标记可能会选择错误的早期分配，该分配随后会变得不一致。 另一个有问题的情况是，当真实字符串高度重复时，例如“aaaaa”，其中每个前缀等于相同长度的每个后缀，使得早期决策难以区分。 

一个含糊不清的最小例子是$n=3$，输入为：```
a
a
aa
aa
```“第一个是前缀，第二个后缀”和相反的分配都是有效的，并且本地决策不提供任何指导。 这说明了为什么我们必须使用全长结构来锚定解决方案，而不是进行孤立的按长度选择。 

## 方法

 思考这个问题的一种强力方法是假设我们尝试重建原始字符串。 我们可以选择两个长度的字符串之一$n-1$一个作为候选前缀，一个作为候选后缀，以与重叠一致的方式将它们拼接在一起，并验证所有提供的字符串是否与结果字符串的前缀和后缀匹配。 这导致尝试$O(n^2)$候选对，对于每次重建我们都会验证$O(n^2)$子串，给出$O(n^4)$最坏情况的行为。 这是不必要的，因为大多数检查都会重复相同的前缀比较。 

关键的观察是我们不需要立即完全重建字符串。 两个字符串的长度$n-1$信息量非常大，因为它们几乎是除一个字符之外的整个字符串。 一是长度前缀$n-1$，另一个是长度后缀$n-1$。 如果我们假设一种排序，我们可以尝试构建完整的字符串，然后验证一致性。 

一旦形成候选完整字符串，每个其他字符串必须与其相应长度的前缀或后缀匹配。 这将问题简化为验证和分类步骤，而不是搜索所有可能性。 

使这项工作有效的结构是两个长度的候选字符串$n-1$如果它们来自有效的解决方案，则只有一个字符不同。 该单个不匹配确定第一个是前缀，第二个是后缀，反之亦然。 尝试这两种可能性最多给出两个候选者，并且只有一个是一致的。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力重建超越所有可能性|$O(n^4)$|$O(n)$| 太慢了 |
 | 两者都尝试一下$n-1$候选人并核实|$O(n^3)$|$O(n)$| 已接受 |

 ## 算法演练

 我们首先按长度对输入字符串进行分组，以便我们可以快速访问每个前缀长度的两个候选字符串。 

1. 将所有字符串按长度收集到桶中。 每个桶恰好包含两个字符串。 
2.关注桶的大小$n-1$。 设这两个字符串为$A$和$B$。 这些是前缀的唯一候选者$n-1$和后缀$n-1$。 
3. 对于每个订购$(A, B)$,构造一个候选完整字符串$S$通过采取$A$作为前缀并附加最后一个字符$B$。 这是有效的，因为长度的后缀$n-1$几乎完全与前缀重叠。 
4.一次$S$构造完毕后，我们尝试对每个输入字符串进行分类：

 对于每个字符串$t$长度$k$，检查是否$t == S[:k]$或者$t == S[n-k:]$。 我们相应地分配它。 
5. 如果在任何时候字符串无法匹配前缀或后缀形式，则丢弃该候选者$S$并尝试其他排序。 
6. 当找到有效的赋值时，输出一个长度的字符串$2n-2$将每个输入行标记为前缀或后缀。 

关键的设计选择是构建$S$直接而不是尝试增量分配。 这消除了传播歧义并将问题转化为确定性验证。 

### 为什么它有效

 任何有效的解决方案都必须与真实长度的前缀和后缀一致$n-1$。 这两个字符串仅在相对于完整字符串的最后一个或第一个字符方面有所不同。 因此，至少两个候选者的一种排序可以准确地重建原始字符串。 一旦完整的字符串被固定，每个较短的前缀和后缀都是唯一确定的，因此分类变得强制和一致。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    arr = [input().strip() for _ in range(2 * n - 2)]

    by_len = {}
    for i, s in enumerate(arr):
        by_len.setdefault(len(s), []).append((s, i))

    # candidates of length n-1
    cands = by_len[n - 1]

    def try_build(A, B):
        # build full string candidate
        # A is assumed prefix of length n-1
        S = A + B[-1]

        res = [''] * (2 * n - 2)

        for s, i in arr_with_idx:
            k = len(s)
            if S[:k] == s:
                res[i] = 'P'
            elif S[n - k:] == s:
                res[i] = 'S'
            else:
                return None
        return res

    arr_with_idx = list(enumerate(arr))

    A, iA = cands[0]
    B, iB = cands[1]

    # try A as prefix
    res = try_build(A, B)
    if res:
        print(''.join(res))
        return

    # try B as prefix
    res = try_build(B, A)
    if res:
        print(''.join(res))
        return

solve()
```该实现依赖于按长度对字符串进行分组，因此我们可以立即找到两个关键候选者。 功能`try_build`在一个假设下构造一个完整的字符串，并在一次传递中验证所有约束。 

一个微妙的点是，分类是通过直接字符串与前缀和后缀切片进行比较来完成的。 这避免了存储预先计算的哈希值或构建 trie 结构的任何需要，考虑到约束，这是不必要的。 

之间的顺序交换$A$和$B$是必不可少的，因为我们不知道哪一个对应于原始字符串的前缀侧。

 ## 边缘情况

 关键的边缘情况是给定长度的所有字符串都相同。 在这种情况下，没有本地决策区分前缀和后缀，只有全字符串重建才能解决歧义。 该算法可以正确处理这一问题，因为两种候选排序均已显式尝试，并且两者都会产生相同的中间匹配，从而确保返回至少一个一致的分​​配。
