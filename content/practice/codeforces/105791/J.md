 ## 解决方案
 ## 问题理解

 竞赛中的每份提交作品都属于一位参赛者，并且每位参赛者对其可能收到的判决都有个人限制。 对于每次提交，系统不会产生确定性结果。 相反，它会在提交提交的参赛者允许的判决中统一随机选择。 

任务不是模拟这种随机性，而是计算所有提交都被评判后每个判决出现的预期次数。 

重新构建流程的一个有用方法是根据提交进行思考。 每次提交都会为判决子集贡献概率质量。 如果参赛者有$t$允许的判决，那么每个判决都会收到概率$1/t$从该意见中，所有其他判决的贡献为零。 最终答案只是所有提交的这些贡献的总和。 

限制很重要，因为提交的数量可以达到$10^5$，而判决类型的数量非常少，最多 10 个。这立即表明任何解决方案都应该与提交数量呈线性关系，并与每个参赛者的判决数量成正比，而不是涉及大量成对计算或对提交进行状态跟踪的任何解决方案。 

一个幼稚但不正确的方向是模拟随机性或重复采样结果并平均结果。 这会引入方差并且不会精确收敛，即使重复多次也会太慢并且仍然只是近似值。 

一个更微妙的错误是尝试独立处理提交的内容而不由参赛者汇总。 如果我们使用昂贵的结构从头开始重新计算每个提交的允许集，我们仍然可能通过约束，但关键的见解是同一参赛者出现多次，因此预先计算他们的概率分布至关重要。 

概率模型本身不存在棘手的极端情况，但一种微妙的边缘情况是参赛者只有一个允许的判决。 在这种情况下，该参赛者的每次提交都会以概率 1 确定性地对该判决做出贡献。另一种情况是，任何参赛者都不允许作出判决，在这种情况下，其预期频率恰好为零。 

## 方法

 如果我们单独查看一项提交，计算就很简单。 假设参赛者$c$有一组允许的大小$t_c$。 然后对于每个允许的判决$v$，本次提交贡献$1/t_c$到预期计数$v$。 总结所有提交的内容给出了最终的期望。 

蛮力方法将独立处理每个提交，并对每个提交迭代所有提交$K$判决，检查该参赛者是否被允许。 这产生$O(NK)$时间，这已经是可以接受的$K \le 10$，但它不是最干净的结构。 更糟糕的暴力变体会尝试动态计算每次提交的概率，而不预先计算参赛者分布，从而导致与输入大小乘以判决集大小成比例的重复工作。 

关键的观察结果是，分配仅取决于参赛者，而不取决于提交的内容。 因此，我们可以为每个参赛者预先计算一个概率向量$K$判决。 一旦完成，每次提交就变成简单的查找和向量加法。 

这将问题转化为构建$M$小概率向量，然后求和$N$根据提交所有权的副本。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 每次提交重新计算 | O(NK) | O(K) | 已接受但多余 |
 | 每个参赛者的预计算 | O(MK + N) | O(MK) | 已接受 |

 ## 算法演练

 1.查看提交数量$N$, 参赛人数$M$，以及判决类型的数量$K$。 我们将维护一个大小的数组$K$累积预期计数。 
2. 对于每个参赛者，读取他们的允许判决列表并存储它。 我们还计算它的大小$t$。 这个大小直接决定了从该参赛者提交的内容中收到的每个判决的概率质量。 
3. 为每个参赛者建立一个概率向量。 对于每个允许的判决$v$，分配概率$1/t$。 该向量代表该参赛者的任何单个提交的预期贡献。 
4. 阅读提交的顺序。 每次提交仅告诉我们哪个参赛者产生了它，因此我们使用该索引来获取预先计算的概率向量。 
5. 对于每次提交，将参赛者的概率向量添加到全局答案数组中。 这相当于将独立试验的预期贡献相加。 
6. 以足够的精度输出最终的累加向量。 

这是有效的原因是期望是线性的。 每个提交都会独立地影响每个判决的预期计数，因此对每个提交的期望求和即可给出正确的全局期望，而无需对任何联合分布进行建模。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def main():
    N, M, K = map(int, input().split())
    
    allowed = [[] for _ in range(M)]
    prob = [[0.0] * K for _ in range(M)]
    
    for i in range(M):
        qt = int(input())
        arr = list(map(int, input().split()))
        allowed[i] = arr
        p = 1.0 / qt
        for v in arr:
            prob[i][v - 1] = p
    
    ans = [0.0] * K
    
    for _ in range(N):
        c = int(input()) - 1
        pc = prob[c]
        for i in range(K):
            ans[i] += pc[i]
    
    print(" ".join(f"{x:.6f}" for x in ans))

if __name__ == "__main__":
    main()
```核心结构将预处理与聚合分开。 预处理步骤为每个参赛者构建固定的概率向量，这避免了提交循环期间的重复划分和重复列表处理。 

一个微妙的实现细节是索引。 判决在输入中给出为基于 1 的索引，因此在存储到概率向量中之前必须将其转换为基于 0 的索引。 另一个细节是确保浮点累加以双精度完成，这是 Python 的 float 自然提供的。

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 | O(MK + N·K) | 构建概率向量需要MK，每次提交都会添加一个K长度的向量|
 | 空间| O(MK + K) | 每个参赛者分布的存储以及最终答案 |

 界限足够小，即使是最坏的情况$10^5 \times 10$积累在Python中是微不足道的。 内存也是安全的，因为$M \le N$和$K \le 10$，因此存储完整的概率表可以忽略不计。

 ## 边缘情况

 一种关键的边缘情况是参赛者只有一个允许的判决。 在这种情况下，概率向量包含一个 1 和其余的 0。 例如，如果参赛者 0 只允许判决 2，那么每次提交都为判决 2 贡献了 1。算法自然地处理这个问题，因为$1/t = 1$，因此不需要特殊的分支。 

另一个边缘情况是不同的参赛者拥有不相交的允许集。 该算法仍然可以正确运行，因为每个提交仅在其自己的本地分布内做出贡献。 参赛者之间不存在干扰，因为积累纯粹是累加的。 

最后一个边缘情况是判决从未出现在任何参赛者允许的集合中。 在这种情况下，它在所有概率向量中的列保持为零，因此它在聚合期间永远不会收到任何贡献。
