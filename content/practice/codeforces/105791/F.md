 ## 解决方案
 ## 问题理解

 我们有一群 X 马拉松运动员，他们需要交通到目的地。 有 N 个可用的乘车选项，每个乘车选项的行为就像最多只能使用一次的单一车辆。 每个游乐设施都有两个属性：容量 pi，这意味着它最多可以承载团体中的 pi 名乘客，以及成本 vi，这意味着如果我们选择该游乐设施，无论我们实际使用多少个座位，我们都必须支付 vi。 

目标是选择游乐设施的子集，使总运输能力至少为 X，同时最大限度地降低总成本。 我们不会以超出容量限制的方式将乘客单独分配到游乐设施，因为一旦选择了某个游乐设施，它就会贡献其全部容量来覆盖整个团体。 

约束 X、N ≤ 1000 和 pi ≤ 4 足够小，二次或低三次动态规划解决方案是可行的。 尝试所有游乐设施子集的解决方案在 N 中呈指数级增长，并且立即不可行，因为它需要评估 2^1000 种组合。 即使是试图贪婪地选择每单位容量最便宜的乘车服务的天真方法也可能会失败，因为成本效率并不能以保证最佳包装的方式进行累加。 

当稍微昂贵且容量更大的游乐设施能够避免多个较小的游乐设施时，就会出现贪婪推理的微妙失败案例。 例如，考虑 X = 6，游乐设施为 (4, 20)、(3, 10)、(3, 8)。 首先选择最低成本的贪心策略将采用 (3, 8) 和 (3, 10)，实现成本 18。但是选择 (4, 20) 加上其他任何内容是无效的，选择 (4, 20) 加上 (3, 8) 会得到容量 7，成本为 28，这更糟糕。 这说明基于成本或单位容量成本的本地决策表现不佳。 

另一个边缘情况是总可用容量不足。 对于 X = 6 且有游乐设施 (3, 10) 和 (2, 5) 的情况，总容量为 5，因此任何选择都无法达到目标，正确答案为 -1。 任何 DP 或贪婪方法都必须显式处理这种无法到达的情况。 

## 方法

 蛮力的想法是考虑游乐设施的每个子集并计算其总容量和总成本。 这是可行的，因为每个有效的解决方案都包含在搜索空间中，因此我们可以在达到至少 X 容量的解决方案中选择最小成本。 然而，这需要迭代 2^N 个子集，并且为每个子集求和，导致 O(N·2^N) 次操作，这远远超出了 N 为 1000 时的任何合理限制。 

关键的观察结果是，这是一个经典的背包结构，但有一个变化：我们并不是试图完全匹配重量，而是要达到至少 X 的容量。 每次乘坐都是一个可以乘坐一次的物品，贡献少量重量（最多 4 个）和成本。 这立即表明了可实现的容量的动态规划公式。 

我们定义一个状态，跟踪从 0 到 X 的所有 j 个乘客准确运送 j 个所需的最低成本。任何高于 X 的容量都可以安全地视为 X，因为超过要求不会进一步改善答案。 这将问题压缩到一个标准的 0/1 背包中，其中权重很小，目标尺寸以 1000 为界。 

我们迭代游乐设施并反向更新 DP 数组，因此每个游乐设施最多使用一次。 处理完所有游乐设施后，答案是所有 j ≥ X 的最小 dp[j]，实际上就是 dp[X]。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(N·2^N) | O(N·2^N) | O(N) | 太慢了 |
 | DP超容量| O(N·X) | O(X) | O(X) | 已接受 |

 ## 算法演练

 我们将其视为成本最小化背包，其中“重量”是载客量，“价值”是负成本最小化。

1. 我们创建一个 DP 数组 dp，其中 dp[j] 表示恰好运送 j 名乘客所需的最小成本，j 从 0 到 X。所有值都初始化为无穷大，除了 dp[0] 为零，因为运送零名乘客不需要任何成本。 这建立了所有转换都源自的基础状态。 
2. 对于每次骑行 (pi, vi)，我们尝试使用它来改进之前可到达的状态。 由于每个行程只能乘坐一次，因此我们将 j 从 X 迭代到 0。反向迭代可防止在单个转换步骤中多次重复使用同一行程。 
3. 对于每个 j，如果 dp[j] 已经可达，我们考虑转移到状态 min(j + pi, X)。 如果可以提高成本，我们用 dp[j] + vi 更新 dp[min(j + pi, X)]。 钳位在 X 处确保我们不会区分“恰好 X”和“超过 X”，因为两者都足够了。 
4. 处理完所有游乐设施后，我们检查 dp[X]。 如果它仍然是无穷大，则意味着没有任何游乐设施子集可以达到所需的容量，因此我们输出-1。 否则 dp[X] 是最小成本。 

### 为什么它有效

 DP 状态保持不变，即在处理前 k 个游乐设施后，dp[j] 是这 k 个游乐设施的所有子集中精确实现 j（上限为 X）容量的最小成本。 每个转换要么排除或包括当前骑行，并且反向迭代保证每个骑行对于每个子集最多贡献一次。 因为所有有效子集都可以通过此类决策的序列来表示，所以不会错过任何可行的解决方案，并且始终存储每个可达容量的最小成本可以得出最优性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    X, N = map(int, input().split())
    INF = 10**18
    
    dp = [INF] * (X + 1)
    dp[0] = 0
    
    for _ in range(N):
        p, v = map(int, input().split())
        
        for j in range(X, -1, -1):
            if dp[j] == INF:
                continue
            nj = j + p
            if nj > X:
                nj = X
            dp[nj] = min(dp[nj], dp[j] + v)
    
    print(-1 if dp[X] == INF else dp[X])

if __name__ == "__main__":
    solve()
```DP 数组被初始化为一个大的哨兵值来表示不可达状态。 j 上的反向循环是强制选择的 0/1 性质的关键实现细节。 将下一个状态限制为 X 可以避免浪费内存或跟踪剩余容量的时间。 

一个常见的错误是按递增顺序迭代 j，这将允许在单次迭代中多次重复使用同一个骑行，从而有效地将问题转化为无界背包并产生不正确的结果。 

## 工作示例

 ### 示例 1

 输入：```
6 3
3 10
4 20
3 8
```我们在每次骑行后跟踪 dp 状态，仅显示相关的压缩值。 

| 步骤| 骑行 | dp[0..6]（压缩为最著名的 X=6 状态）|
 | ---| ---| ---|
 | 初始化| - | dp[0]=0, 其他=∞ |
 | 1 | (3,10) | dp[3]=10 | dp[3]=10 |
 | 2 | (4,20) | dp[4]=20，dp[6]=30（通过3+3无效，所以只有4和3+4状态）|
 | 3 | (3,8) | dp[6]=18 | dp[6]=18 |

 最终答案是 18，通过选择容量为 3、3 和 4 的游乐设施来实现，但最佳情况下 (3,8) + (3,10) 比不同组合更糟糕； DP 确保在所有子集中找到最佳组合。 

此跟踪显示了如何重用中间部分容量以形成更大的容量，以及为什么仅存储每个容量的最佳成本就足够了。

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 | O(N·X) | 每次骑行最多更新 X 个 DP 状态一次 |
 | 空间| O(X) | O(X) | 仅存储超过容量的单个 DP 数组 |

 N·X 的乘积最多为 10^6，这在 Python 中在 1 秒的限制下非常快，特别是在简单的整数运算并且除了 DP 循环之外没有嵌套开销的情况下。

 ## 边缘情况

 一种重要的边缘情况是，所有游乐设施组合起来仍无法达到 X。对于输入 X = 10 和游乐设施 (3, 5)、(4, 6)、(2, 3)，DP 永远不会达到 dp[10]，最终状态仍为无穷大。 该算法正确输出 -1，因为没有任何转换序列产生有效的全容量状态。 

另一种边缘情况是单次骑行已经满足要求。 对于 X = 3 和骑行 (4, 100)，转换立即将结果钳位为 dp[3] = 100。 钳位步骤至关重要，因为如果没有它，DP 就会错误地将超出 X 的容量视为单独的状态，并且可能会错过最佳转换。 

最后的边缘情况涉及多种重叠组合，以不同的成本产生相同的容量。 DP 确保在每个容量级别仅存在最小成本，因此即使许多子集达到相同的 j，也只会保留最好的一个。
