 ## 解决方案
 ## 问题理解

 我们被要求计算给定区间内有多少个数字可以用一种非常具体的形式表示：三个连续整数的乘积。 每个这样的数字都来自选择一个整数$x$并形成$x \cdot (x+1) \cdot (x+2)$。 然后问题给出了许多查询，每个查询提供一个范围$[L, R]$，并且我们必须确定有多少这种形式的不同值落在该范围内。 

因此，有效值的范围完全由单个整数参数确定，而不是使用任意数字$x$。 每个$x$准确生成一个候选值，任务就变成计算这些生成值中有多少位于给定区间内。 

限制条件非常大：$L$和$R$可以达到$10^{16}$，并且最多可以有$10^6$查询。 这立即排除了间隔内的任何每次查询迭代，因为即使扫描范围或单独检查每个值也将远远超出可行的限制。 该解决方案必须依赖于有效候选者的预处理或快速数学定位。 

一个微妙的边缘考虑因素是有效值稀疏。 功能$f(x) = x(x+1)(x+2)$呈立方增长，因此值很快就会变得非常遥远。 这表明我们也许能够预先计算所有有效值，直至最大可能值$R$，并通过二分搜索回答查询。 

一个潜在的陷阱是假设多个不同的$x$值可能映射到同一产品。 对于正整数，这种情况不会发生，因为该函数严格递增$x \ge 1$。 另一个问题是生成期间的溢出，因为值会增长到$10^{16}$，所以我们必须小心地停止生成。 

## 方法

 一个简单的想法就是尝试一切可能的方法$x$并计算$x(x+1)(x+2)$，然后对于每个查询扫描所有值并计算有多少个落在其中$[L, R]$。 这是正确的，但完全不切实际。 即使我们限制$x$，最大$x$这样$x^3 \le 10^{16}$就在附近$10^5$。 这已经给出了关于$10^5$价值观。 高达$10^6$查询，每个查询的线性扫描导致$10^{11}$操作，规模太大了。 

关键的观察结果是有效数字的结构根本不依赖于查询。 所有有效值都可以独立生成一次，因为它们只依赖于$x$。 由于该函数是单调递增的，因此我们可以将所有值存储在排序数组中。 然后每个查询就变成了排序列表上的范围计数问题，可以通过两次二分搜索来回答。 

这将问题从“每个查询重新计算结构”转变为“预计算一次，快速查询”。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 每个查询的强力扫描 |$O(t \cdot n)$|$O(n)$| 太慢了 |
 | 预计算+二分查找|$O(n + t \log n)$|$O(n)$| 已接受 |

 这里$n$是有效的数量$x$， 大致$10^5$。 

## 算法演练

 我们首先生成表单的所有有效值$x(x+1)(x+2)$直到该值超过$10^{16}$。 由于函数呈三次增长，因此我们只需要迭代$x$最多约$10^5$，这是安全的。 

我们将每个计算值存储在一个列表中。 因为该函数对于正数是严格递增的$x$，列表已经排序，因此不需要额外排序。 

对于每个查询$[L, R]$，我们使用二分查找来确定有多少预先计算的值位于该区间内。 具体来说，我们找到价值至少为的第一个位置$L$，以及值大于的第一个位置$R$。 这两个位置之间的差异给出了答案。 

### 步骤

 1. 迭代$x = 1, 2, 3, \dots$并计算$v = x(x+1)(x+2)$。 

我们停下来时$v > 10^{16}$，因为超出此范围，任何查询都不能包含它。 
2. 将所有生成的值存储在数组中`vals`。 

这是有效的，因为每个有效数字都唯一对应于一个$x$。 
3. 对于每个查询$[L, R]$，执行二分查找查找：

 最小索引，其中`vals[i] >= L`和最小索引，其中`vals[i] > R`。 
4. 查询的答案是这两个索引的差值。 

使用二分查找的正确性取决于以下事实：`vals`是严格递增的，因此所有有效数字都是有序的。 

### 为什么它有效

 核心不变量是`vals`按递增顺序包含每个可能的有效产品一次。 由于每个查询仅询问有多少预先计算的值落在某个范围内，因此问题简化为对排序数组中的元素进行计数。 二分搜索无需扫描即可给出精确的边界位置，从而在保持正确性的同时降低复杂性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MAXV = 10**16

vals = []
x = 1
while True:
    v = x * (x + 1) * (x + 2)
    if v > MAXV:
        break
    vals.append(v)
    x += 1

# binary search helpers
def lower_bound(a, target):
    lo, hi = 0, len(a)
    while lo < hi:
        mid = (lo + hi) // 2
        if a[mid] < target:
            lo = mid + 1
        else:
            hi = mid
    return lo

def upper_bound(a, target):
    lo, hi = 0, len(a)
    while lo < hi:
        mid = (lo + hi) // 2
        if a[mid] <= target:
            lo = mid + 1
        else:
            hi = mid
    return lo

t = int(input())
out = []

for _ in range(t):
    L, R = map(int, input().split())
    left = lower_bound(vals, L)
    right = upper_bound(vals, R)
    out.append(str(right - left))

print("\n".join(out))
```预处理循环一次构造所有有效的传奇值，确保每个查询不会重新计算。 停止条件为$10^{16}$避免溢出和不必要的工作。 

二分查找例程是标准的边界查找器。`lower_bound`找到第一个不小于的有效值$L$， 尽管`upper_bound`找到第一个值严格大于$R$。 它们的差异直接计算范围内的有效条目。 

一个微妙的实现细节是安全地使用 Python 整数而无需担心溢出。 然而，明确地停止在$10^{16}$保持生成的列表紧凑并避免不必要的增长。

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 |$O(n + t \log n)$| 预计算一次生成所有有效值，然后每个查询使用两次二分搜索 |
 | 空间|$O(n)$| 存储所有有效产品最多$10^{16}$， 关于$10^5$价值观 |

 约束允许最多$10^6$查询，因此每个查询行为的对数至关重要。 与查询量相比，预计算成本可以忽略不计，并且内存使用量仍然很小。

 ## 边缘情况

 临界边缘情况是间隔完全低于最小有效值。 最小的产品是$1 \cdot 2 \cdot 3 = 6$。 对于像这样的输入$[1, 5]$，该算法会产生一个空计数，因为两个二分搜索都返回索引 0 并且不包含任何值。 

另一种情况是当$L$或者$R$与有效产品完全匹配。 为了$[6, 6]$,`lower_bound`返回索引 0 和`upper_bound`返回索引 1，产生正确的计数 1。单调结构确保匹配边界不会出现歧义。 

大间隔，例如$[1, 10^{16}]$测试全覆盖。 在这种情况下，二分搜索跨越整个预先计算的列表并返回其完整大小，因为所有有效值都在范围内。
