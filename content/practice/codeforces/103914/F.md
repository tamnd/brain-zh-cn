 ## 解决方案
 ## 问题理解

 我们给出了两个整数序列，但是这两个序列都不是直接提供的。 相反，两者都是通过重复应用相同的二次递推模固定值而产生的。 我们首先生成整个序列`s`长度`n`，然后继续同样的递归生成`t`长度`m`无需重置任何内容。 这使得`t`产生相同底层流的延续`s`。 

任务是计算出现在两者中的最长子序列的长度`s`和`t`，其中每个序列内必须保留顺序，但元素不需要是连续的。 

从约束的角度来看，跨测试用例的所有序列的组合长度最多为一百万。 这立即排除了对位置对的任何二次动态规划，因为即使是单个最坏情况测试也会使该方法超出时间限制几个数量级。 解决方案在生成的元素总数中必须接近线性或近线性，可能具有对数因子。 

一个微妙的点来自于生成过程本身。 由于两个序列都来自单个递归链，`s`是`A[1..n]`和`t`是`A[n+1..n+m]`对于单个隐藏数组`A`。 这种结构是避免将两个序列视为独立的关键。 

经常失败的天真的尝试是运行标准 LCS 动态规划或尝试贪婪地匹配相等的值。 在重复值强制做出错误配对决策的简单情况下，这两种方法都会出现问题。 例如，如果`s = [1, 1]`和`t = [1, 1]`，正确答案是 2，但根据实现的不同，第一次出现的贪婪匹配很容易产生 1。 

当值大量重复时，会出现另一种故障模式。 如果`s = [5, 5, 5]`和`t = [5, 5, 5, 5]`，答案是 3，但是不强制执行全局排序的朴素配对策略可能会根据匹配的消耗方式而过高或过低计数。 

核心困难在于每个值可能在两个序列中出现多次，因此问题不仅仅在于相等性检查，而在于选择全局一致的递增配对。 

## 方法

 直接动态规划解决方案定义`dp[i][j]`作为前缀的 LCS`s[1..i]`和`t[1..j]`。 这是有效的，因为每个选择要么跳过一个序列中的一个元素，要么匹配相同的元素。 然而，这需要`O(nm)`时间，当两个序列的长度都达到一百万时这是不可能的。 

关键的观察是这两个序列不是任意的。 它们只是同一生成数组的两个连续段`A`。 这意味着每场比赛`s`和`t`对应于一对索引`(i, j)`这样`i < j`和`A[i] = A[j]`，附加的约束是我们必须选择一组在两个维度上都保持顺序的此类对。 

这将问题转化为在部分有序集中找到最大的点链，其中每个点都是以下索引之间的有效等式：`s`和一个索引`t`。 解决此类问题的标准方法是将它们转换为一个维度上的最长递增子序列，同时按另一个维度排序。 

具体来说，我们处理索引`s`按递增顺序。 对于每个值`v = s[i]`，我们知道所有位置`t`出现相同值的地方。 对于每个这样的位置`j`，我们可以尝试扩展一个以`j`使用之前的最佳子序列结尾`j`。 

为了有效地支持这一点，我们在以下位置维护一个 Fenwick 树：`t`，其中每个索引存储在该位置结束的最佳 LCS 长度。 每个元素来自`s`为多个职位提供更新`t`，并且每次更新都依赖于前缀查询`t`。 

主要的复杂性是值可能会多次出现`t`，但由于所有测试的总输入大小都以一百万为界，因此迭代所有出现的事件总体上仍然可行。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力DP | O(纳米) | O(纳米) | 太慢了 |
 | 芬威克 (Fenwick) 参加比赛 | O((n + m) log m) 摊销 | O(米) | 已接受 |

 ## 算法演练

 我们将整个生成过程视为生成单个数组`A`， 在哪里`s = A[1..n]`和`t = A[n+1..n+m]`。 

1. 一次生成两个序列，同时将每个值的位置存储在`t`。 对于每个索引`j`在`t`，我们追加`j`进入一个由以下键控的列表`A[j]`。 这允许快速查找第二段中出现值的位置。 
2. 初始化 Fenwick 树的索引`t`，其中每个位置代表结束于该位置的最佳 LCS 长度。 最初所有值都为零。 
3. 迭代每个元素`s[i]`为了。 对于当前值`v = s[i]`，检索所有位置的列表`t`在哪里`A[j] = v`。 
4. 对于每个这样的位置`j`, 计算`best = query(j - 1) + 1`， 在哪里`query(j - 1)`返回严格在位置之前结束的最佳子序列`j`在`t`。 这确保我们在第二个序列中保留递增的顺序。 
5. 计算完该值的所有候选更新后，将它们应用到 Fenwick 树。 查询和更新之间的这种分离防止使用来自同一查询和更新的更新`s[i]`以不一致的方式多次。 
6. 跟踪写入 Fenwick 树的最大值； 这是最终的答案。 

每个内部的查询和更新是分离的`s[i]`是必不可少的，因为同一值在`t`不应在一个步骤内相互链接`s`。 

### 为什么它有效

 每个有效的公共子序列对应于选择对`(i, j)`这样两个序列中的索引都会增加。 芬威克树确保当我们将匹配放在位置时`j`在`t`，我们只扩展严格之前结束的子序列`j`。 加工`s`为了保证索引`s`也在增加。 这在由相等值定义的二分匹配图中精确地构造了有效递增链的集合。 

因为每个转换都遵循排序约束，并且每个有效子序列都可以通过逐步选择匹配对来构建，所以存储的最大值正是 LCS 长度。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class Fenwick:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)

    def update(self, i, v):
        while i <= self.n:
            if v > self.bit[i]:
                self.bit[i] = v
            i += i & -i

    def query(self, i):
        res = 0
        while i > 0:
            if self.bit[i] > res:
                res = self.bit[i]
            i -= i & -i
        return res

def solve():
    T = int(input())
    for _ in range(T):
        n, m, p, x, a, b, c = map(int, input().split())

        s = [0] * n
        t = [0] * m

        for i in range(n):
            x = (a * x * x + b * x + c) % p
            s[i] = x

        pos = {}
        for i in range(m):
            x = (a * x * x + b * x + c) % p
            t[i] = x
            if x not in pos:
                pos[x] = []
            pos[x].append(i + 1)

        fw = Fenwick(m)
        ans = 0

        for i in range(n):
            v = s[i]
            if v not in pos:
                continue

            updates = []
            for j in pos[v]:
                best = fw.query(j - 1) + 1
                updates.append((j, best))

            for j, val in updates:
                fw.update(j, val)
                if val > ans:
                    ans = val

        print(ans)

if __name__ == "__main__":
    solve()
```该解决方案仔细维护了一个 Fenwick 树`t`。 每个查询步骤都会计算可以通过匹配当前值来扩展的最佳子序列`s`到出现在`t`。 

一个微妙的实现要求是缓冲每个值的更新`s[i]`。 如果没有这个，在迭代位置时更新芬威克树`t`可能允许稍后出现的相同值错误地依赖于同一迭代的早期更新。 

Fenwick 树内的所有索引都是从 1 开始的，而`t`以从 1 为基础的位置存储以避免前缀查询中的差一错误。 

## 工作示例

 考虑第二个示例情况，其中两个序列都完全由零组成。 

我们有`s = [0, 0, 0]`和`t = [0, 0, 0, 0]`。 

对于中的每个元素`s`, 中的每个位置`t`是一个有效的匹配。 

| 步骤| s[i] | 候选人t职位| 查询结果 | 已应用更新 |
 | --- | --- | --- | --- | --- |
 | 1 | 0 | 1,2,3,4 | 全部 0 + 1 = 1 | 所有位置设置为 1 |
 | 2 | 0 | 1,2,3,4 | 前缀传播到 2 | 值变为 2 |
 | 3 | 0 | 1,2,3,4 | 前缀传播到 3 | 值变为 3 |

 处理完所有元素后，Fenwick 树中的最大值为 3，与预期的 LCS 长度匹配。 

现在考虑一个没有重叠的情况，例如`s = [1, 2, 3]`和`t = [4, 5, 6, 7]`。 两个序列中都没有出现任何值，因此不会执行任何更新。 芬威克树始终保持为零，确认该算法正确处理不相交的字母表。

 ## 边缘情况

 关键的边缘情况是所有值都相同。 在这种情况下，每一个元素`s`匹配中的每个元素`t`，答案应该是准确的`min(n, m)`。 该算法处理这个问题是因为每一步都按顺序扩展所有先前的前缀，并且芬威克树自然地累积最长的链而不会重复计算。 

另一个边缘情况是当之间没有共享值时`s`和`t`。 由于没有位置被更新，芬威克树保持为零并且输出正确地为零。 

第三种情况是值稀疏但不规则地重复。 即使出现次数分布不均匀，该算法也始终通过前缀查询强制执行严格排序，从而防止对匹配项进行无效的重新排序。`t`。
