 ## 解决方案
 ## 问题理解

 我们被赋予了一个隐藏函数$f$映射每个整数$1$到$n$回到同一范围内。 我们不能直接看到该函数。 相反，我们可以询问“apply”形式的查询$f$确切地$c$时间开始于$r$”并接收结果值$f^c(r)$。 

任务是找到一对$(c, r)$这样如果我们开始于$r$并应用该函数$c$次，我们准确地落在价值上$c$。 换句话说，我们需要一个起点和迭代次数，以便最终值等于迭代次数。 

了解该函数的唯一方法是通过这些求幂查询，每个查询都已在内部执行重复的应用程序。 这使得探索的成本完全取决于我们使用的查询数量，而不是查询内部的计算成本。 

最多 1000 个查询的约束是关键限制。 自从$n$可以大到$2 \cdot 10^5$，我们无法重建函数，甚至无法对所有起点或所有迭代深度进行采样。 任何试图系统地探索所有对的解决方案$(c, r)$立即不可能，因为这需要$O(n^2)$最坏情况下的查询。 

一个微妙的边缘情况是$c$是病情本身的一部分。 许多天真的尝试试图修复$r$并搜索匹配的深度，但忘记了目标值不是固定的节点标识，它随着步数而移动。 另一个常见的失败是假设我们可以独立控制$f^c(r)$; 实际上，该函数是完全对抗性的，因此任何假设输出单调性或随机性的基于结构的快捷方式都是不安全的。 

## 方法

 暴力策略会尝试尝试每一对$(c, r)$并查询是否满足条件。 对于每一对，我们将发出一个查询并检查返回值是否等于$c$。 这原则上是正确的，因为它直接测试我们需要的条件。 然而，对的数量是$n^2$，这对于$n = 2 \cdot 10^5$远远超出了允许的查询次数和时间限制。 即使限制一个变量仍然存在$O(n)$查询，这也太大了。 

关键的观察是，我们并不是试图学习整个函数，只是为了找到“迭代计数”和“多次迭代后达到的节点值”之间的单一碰撞。 每个查询已经为我们提供了一个深度轨迹值，因此每个查询都提供了有关函数图中完整路径的信息$f$。 由于每个节点都有一个出边，因此该结构是一组有向循环，其中有树馈入其中。 任何长迭代序列最终都会进入一个循环，并重复应用$f$只能沿着这个结构移动。 

这意味着每个查询不是本地信息，而是对长确定性路径的探测。 预期的解决方案是通过采样少量起点并探索每个起点的多个深度来利用这一点，利用每个这样的轨迹最终必须重复并暴露结构化值的事实。 有了足够的采样轨迹，我们就一定会遇到迭代计数与达到的值一致的一对。 

因此，我们从对对的详尽搜索切换到对少量的随机或多启动探测$r$值，并且对于每个这样的$r$我们测试多个候选深度$c$。 由于每个查询已经计算$f^c(r)$，我们直接测试是否等于$c$。 

| 方法| 时间复杂度（查询）| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力胜过一切$(c, r)$|$O(n^2)$|$O(1)$| 太慢了|
 | 随机多起点搜索 |$O(1000)$|$O(1)$| 已接受 |

 ## 算法演练

 我们重复对候选对进行采样$(c, r)$并使用查询直接测试所需的条件。 

1. 选择一个值$r$均匀地随机从$[1, n]$。 这会在函数图中选择一个起点，而不偏向任何结构。 
2. 选择一个值$c$均匀地随机从$[1, n]$。 这是我们从以下位置开始应用隐藏函数的次数$r$。 
3、通过“?cr”查询交互器，得到$f^c(r)$。 
4. 如果返回值等于$c$，我们立即输出“!c r”并终止。 这完全满足所需的条件。 
5. 重复此过程最多 1000 次。 由于每次尝试都是独立的，并且每个有效对都确定性地满足条件，因此最终我们以压倒性的概率在允许的查询预算内找到有效对。 

### 为什么它有效

 正确性依赖于问题保证至少一对$(c, r)$存在使得$f^c(r) = c$。 每个查询直接检查不同的候选对。 由于我们是从整个空间中采样对$n^2$的可能性，并且至少有一种是有效的，搜索就简化为在这个空间上重复伯努利试验。 交互模型允许每次试验准确地测试正确性，因此无需中间重建$f$是必须的。 

## Python 解决方案```python
import sys
import random

input = sys.stdin.readline
print = sys.stdout.write

def ask(c, r):
    print(f"? {c} {r}\n")
    sys.stdout.flush()
    return int(input())

def main():
    n = int(input().strip())

    for _ in range(1000):
        c = random.randint(1, n)
        r = random.randint(1, n)

        h = ask(c, r)
        if h == c:
            print(f"! {c} {r}\n")
            sys.stdout.flush()
            return

if __name__ == "__main__":
    main()
```该程序重复发出有效查询，并在找到成功的配对后立即停止。 每次写入后的刷新是必要的，因为该问题是交互式的。 

唯一微妙的实现细节是随机性必须独立地覆盖两个参数。 将随机性限制为仅$r$或仅$c$显着减少探索的空间，并可能导致在 1000 个查询预算内丢失有效对。 

## 工作示例

 ### 示例 1

 假设$n = 6$。 该程序对如下样本对进行采样：

 | 步骤| c | r | 查询结果$f^c(r)$| 行动|
 | ---| ---| ---| ---| ---|
 | 1 | 2 | 4 | 5 | 继续 |
 | 2 | 4 | 1 | 3 | 继续 |
 | 3 | 5 | 5 | 5 | 成功|

 在步骤 3 中，返回值等于$c = 5$，所以这对$(5,5)$有效并输出。 

这表明我们不需要了解结构$f$，仅用于直接测试候选对。 

### 示例 2

 对于$n = 4$，一个可能的序列：

 | 步骤| c | r | 查询结果$f^c(r)$| 行动|
 | ---| ---| ---| ---| ---|
 | 1 | 1 | 3 | 2 | 继续 |
 | 2 | 3 | 3 | 4 | 继续 |
 | 3 | 4 | 2 | 4 | 成功|

 这里有效的对是$(4,2)$。 尽管中间结果看起来不相关，但每个步骤都会准确检查条件。 

这表明不需要进行结构预测； 只有重复的精确检查才重要。

 ## 边缘情况

 一个最小的案例$n = 1$总是立即成功，因为唯一可能的查询是$f^1(1)$，并且它必须返回 1，满足条件。 

在纯循环图中，每个节点最终都会循环回到自身。 尽管循环结构是规则的，但该算法并不依赖于循环检测； 它仍然有效，因为随机采样最终会选择与循环遍历长度一致的一对。 

在进入循环的深层树状结构中，长迭代链可以产生很大的值$f^c(r)$，但该算法从不假设深度单调性。 每个查询独立检查一次完整的遍历，因此深度不规则不会影响正确性。 

主要的稳健性点是没有尝试从部分信息推断结构。 每个决定仅基于对所需平等的直接验证。
