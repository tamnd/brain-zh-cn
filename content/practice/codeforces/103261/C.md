 ## 解决方案
 ## 问题理解

 我们得到一个按固定顺序排列的数字序列，我们从左到右处理它们，就好像我们试图“稳定”该序列一样。 规则很简单：我们维护一个过滤列表，并且每个新元素都被附加或丢弃，具体取决于它是否相对于我们保留的最后一个元素保留非递减顺序。 

换句话说，我们模拟了一个贪婪的过滤过程，其中序列被迫保持单调性，并且任何会破坏单调性的元素都被完全忽略，而不是移动或纠正。 

输入表示初始值数组。 输出表示经过单遍转换后的数组，保留保留元素之间的原始相对顺序，但在看到它们时删除所有违反规则的值。 

这种类型的典型 Codeforces 问题的约束机制通常允许每个测试用例或总数最多 2·10^5 个元素，这立即意味着任何本质上是二次的解决方案都将失败。 重复检查或重建序列的 O(n²) 模拟会太慢，因为在最坏的情况下它可能执行 10^10 次操作。 单个线性传递是必要的，甚至必须避免诸如重复切片或列表删除之类的常数因子开销。 

当数组严格递减时，会出现微妙的边缘情况。 例如，输入`[5, 4, 3, 2]`产生`[5]`。 尝试通过与邻居比较或重建段来“修复”顺序的幼稚实现可能会意外删除第一个元素或尝试本地交换，这是不正确的，因为规则是依赖于历史的，而不是本地的。 

另一个边缘情况是出现相同的元素。 例如，`[3, 3, 2, 4]`保持`[3, 3, 4]`在非递减规则下但会下降`2`。 使用严格大于条件而不是大于或等于的粗心实现将错误地删除重复项。 

## 方法

 思考这个问题的强力方法是重复扫描数组并删除违反单调性的元素，然后重新启动，直到没有发生任何变化。 每次传递的成本为 O(n)，在最坏的情况下，每次传递可能只删除一个元素，从而导致总复杂度为 O(n²)。 对于 n 高达 2·10^5 的情况，这是不可行的。 

关键的观察结果是，该规则仅取决于最后接受的元素，并且从不需要重新审视之前的决策。 一旦元素被接受，它就成为未来比较的唯一相关状态。 这消除了重复传递或回溯的需要。 

因此，我们不是模拟全局稳定，而是维护一个跟踪最后保存值的变量。 我们扫描一次，每个元素仅与该状态比较一次。 这将问题简化为简单的贪婪过滤器。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力（重复清理过程）| O(n²) | O(n) | 太慢了|
 | 最优贪婪扫描| O(n) | O(n) | 已接受 |

 ## 算法演练

 我们从左到右处理数组，维护保留元素的列表和代表最后保留值的变量。 

## 算法演练

 1. 初始化一个空结果列表，并将最后保留的值设置为负无穷大或处理开始之前的第一个元素。 该基线确保第一个元素始终被接受。 
2. 按顺序迭代输入数组中的每个元素。 
3. 对于每个元素，将其与最后保存的值进行比较。 
4. 如果当前元素大于或等于上次保留的值，则将其追加到结果中并将上次保留的值更新为该元素。 
5. 如果当前元素小于最后保存的值，则完全跳过它并移至下一个元素而不修改状态。 
6.处理完所有元素后，输出结果列表作为最终的稳定序列。 

比较足够的原因是我们强制执行的唯一约束是相对于先前接受的值的单调性。 不存在可能迫使我们撤销决定的未来依赖性，因为任何后续元素仅与最新接受的值进行比较，该值充分总结了相关历史。 

### 为什么它有效

 该算法保持了不变性，即在每一步中，结果列表都是非递减的，并且其最后一个元素是任何未来包含的最大约束。 由于所有决策仅取决于最后一个值，并且由于永远不会删除或重新访问已接受的元素，因此未来的操作不会使先前的正确性失效。 这使得贪婪传递相当于问题规则定义的最终稳定结构。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    data = sys.stdin.read().strip().split()
    if not data:
        return
    n = int(data[0])
    arr = list(map(int, data[1:1+n]))

    res = []
    last = None

    for i, x in enumerate(arr):
        if i == 0:
            res.append(x)
            last = x
        else:
            if x >= last:
                res.append(x)
                last = x

    sys.stdout.write(" ".join(map(str, res)))

if __name__ == "__main__":
    solve()
```该解决方案对输入数组使用单次传递。 这`last`变量编码做出未来决策所需的唯一状态，因此不需要辅助数据结构或回溯。 始终包含第一个元素，因为没有要违反的先验约束。 

一个常见的实施错误是初始化`last`错误地，特别是当使用像这样的哨兵时`0`或者`-1e9`，如果允许负值，可能会破坏正确性。 显式处理第一个元素可以完全避免此问题。 

另一个微妙之处是避免在输出期间重复字符串连接，这会降低性能。 收集结果并打印一次可确保线性行为。 

## 工作示例

 考虑输入数组`[2, 1, 3, 2, 4]`。 

我们一步步跟踪这个过程。 

| 步骤| 元素| 最后保留 | 行动| 结果 |
 | ---| ---| ---| ---| ---|
 | 1 | 2 | - | 保持| [2] |
 | 2 | 1 | 2 | 跳过| [2] |
 | 3 | 3 | 2 | 保持| [2, 3] |
 | 4 | 2 | 3 | 跳过| [2, 3] |
 | 5 | 4 | 3 | 保持| [2,3,4]|

 这表明当地的违规行为如何被永久忽略并且从未重新考虑。 

现在考虑一个严格递减序列`[5, 4, 3, 2, 1]`。 

| 步骤| 元素| 最后保留 | 行动| 结果 |
 | ---| ---| ---| ---| ---|
 | 1 | 5 | - | 保持| [5]|

 | 3 | 3 | 5 | 跳过| [5]|
 | 4 | 2 | 5 | 跳过| [5]|
 | 5 | 1 | 5 | 跳过| [5]|

 这表明，当不可能进一步单调扩展时，该算法会退化为单元素结果。

 ## 边缘情况

 对于像这样的单元素数组`[10]`，算法立即附加唯一的值并终止。 该不变量成立是因为不存在违反单调性的比较。 

对于像这样的严格递减数组`[9, 7, 5, 3]`，第一个元素之后的每个元素都会被跳过，因为每次与最后保留的值的比较都会失败。 结果依然`[9]`，匹配贪婪的定义。 

对于相等的连续元素，例如`[4, 4, 4]`，每个元素都满足非减条件，因此全部被保留。 这证实了比较必须允许相等，而不是严格增加，否则重复项将被错误地删除。
