 ## 解决方案
 ## 问题理解

 我们有一个由男孩和女孩组成的小系统，每个男孩都有一个固定的愿意与之跳舞的女孩名单。 这个过程是概念性的，而不是模拟的：男孩们按照递增的顺序进行，每个男孩都试图挑选一个他喜欢的仍然有空的女孩。 如果一个男孩喜欢的女孩都被别人占了，他可能会失败。 

问题不在于模拟舞蹈。 相反，对于男孩的每个查询子集，我们必须确定是否存在一种方法，使该子集中的所有男孩在某些有效的选择序列下最终根本没有伴侣。 这里的“没有伴侣”意味着轮到她们时，由于之前有其他男孩选择，所以她们喜欢的女孩都没有空。 

因此，每个查询都在问一个可行性问题：我们是否可以隐式地对选择进行排序，以便查询集中的每个男孩都无法接触到所有他们喜欢的女孩？ 

约束很小：n 和 k 最多为 50，查询最多为 1000。这强烈表明核心结构是子集和二分关系上的组合，并且如果以 50 为界，则针对女孩或男孩的每个查询指数工作是可以接受的。 

一个关键的观察结果是，每个查询都是独立的，但宇宙足够小，我们可以预先计算男孩子集和女孩子集之间的相互作用。 

一个幼稚的错误是按字面解释该过程并按顺序模拟贪婪分配。 这将计算单个结果，但问题是存在的：我们被问的是是否存在子集中的每个人都失败的情况，而不是自然顺序是否会导致失败。 

另一个微妙的陷阱是认为我们必须分配女孩。 我们实际上需要相反的条件：我们希望子集中的所有男孩都被阻止，这意味着对于子集中的每个男孩，他喜欢的每个女孩都“已经被”在他之前的子集中的其他男孩以某种顺序使用。 

## 方法

 蛮力解释将尝试模拟查询中男孩轮流分配女孩的所有可能方式，检查是否存在所有人都失败的时间表。 这立即变得棘手，因为即使对于 50 个男孩来说，排列也是巨大的，而且每个选择都超出了可用的女孩。 

正确的转变是颠倒观点。 我们不考虑分配女孩，而是考虑用同一查询集中其他男孩造成的冲突来覆盖每个男孩的偏好集。 如果一个男孩喜欢的每个女孩都被子集中的另一个男孩“认领”，那么他就失败了。 所以我们实际上是在问，是否存在男孩的排序，使得每个男孩都被较早的男孩所主导，涵盖了所有他喜欢的女孩。 

这变成了二分结构的支配问题。 由于 k 很小，我们可以将每个男孩的偏好列表表示为 k 个女孩的位掩码。 那么条件“男孩我失败了”意味着他的面具中的所有部分都被早期男孩的面具的联合覆盖。 

现在问题变成：是否存在查询子集的排列，使得对于每个位置 i，先前男孩的掩码的并集覆盖当前男孩的掩码。 这相当于询问我们是否可以对集合进行排序，以便每个掩码都包含在其之前的累积并集中。 

这自然可以通过观察来解决：如果存在这样的排序，则子集中所有掩码的并集必须等于某个前缀结构的并集，其中每个步骤添加至少一个后续约束所需的新覆盖女孩。 这导致了对男孩子集的 DP，我们跟踪哪些女孩子集已经可以被“阻止”。

由于 k 只有 50，但掩码是位集，我们可以为男孩的每个子集预先计算他们偏好掩码的并集，但我们需要的不仅仅是这些：我们需要检查是否存在一种顺序，可以逐渐“构建”覆盖范围，从而使每个男孩都在他们的步骤中被阻止。 

我们再次更清晰地重新表述：如果男孩在处理时所有他喜欢的女孩都已经在覆盖集中，则他被成功阻止。 因此，如果我们建立一组 S 全部失败的男孩，则必须存在一个排序，使得 S 的每个前缀都涵盖所有剩余男孩的偏好。 这相当于检查是否不存在“抵抗”结构，其中某些女孩子集仅由男孩覆盖，而男孩本身要求这些女孩更早失败。 

这相当于检查一个封闭条件，该条件可以通过 DP 对具有覆盖女孩位掩码的男孩子集来解决。 

我们定义 dp[男孩的掩码] = 如果我们以某种有效的失败顺序精确处理这个子集，则可以覆盖所有女孩的并集。 我们尝试通过添加一个新男孩来扩展，他的偏好已经完全包含在当前覆盖的集合中。 如果我们可以达到完整的查询集，那么答案是肯定的。 

结束对比：

 | 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力排列 + 模拟 | O(tj!·k) | O(tj!·k) | O(k) | 太慢了 |
 | Bitmask DP 超越男孩 + 工会面具 | O(q · 2^n · k / 64) （由于修剪，有效值要小得多） | O(2^n + n) | O(2^n + n) | 已接受 |

 由于 n ≤ 50 但每个查询都有自己的子集，因此我们实际上在每个查询的子集大小 tj 上运行 DP，使其成为 O(2^tj · k/64)，这在最坏的实际情况下对于 tj 高达 20-25 是可行的，并且设计了约束以允许这样做。 

## 算法演练

 我们对被查询的男孩使用位掩码 DP 独立地解决每个查询。 

1. 对于每个男孩，将他们的偏好列表转换为 k 位整数掩码，其中如果男孩喜欢女孩 g，则设置位 g。 这允许使用位操作进行快速联合和子集检查。 
2. 对于包含 tj boy 的查询，在本地将它们从 0 重新标记为 tj − 1 并提取它们的掩码。 这使每个查询的 DP 保持紧凑，而不是在 n 上全局。 
3. 在这些 tj boy 的子集上初始化 DP 数组。 设 dp[mask] 表示如果我们将男孩精确地排列在其中，则可以已经“覆盖”的女孩集合`mask`以某种有效的顺序，使所有这些都失败。 
4. 从 dp[0] = 0 开始，表示不处理男孩也不处理女孩。 
5. 迭代从 0 到 2^tj − 1 的所有子集 mask。对于每个 mask，尝试添加一个不在 mask 中的新男孩 i。 仅当当前覆盖的集合已包含我喜欢的男孩的所有女孩时，转换才有效。 这确保了当我们接下来放置男孩 i 时，他没有可用的女孩。 
6. 如果有效，计算 next_mask = mask ∪ {i} 并更新 dp[next_mask] = dp[mask] ∪ choice[i]。 工会扩大了覆盖范围，将这个男孩贡献的一切都包括在内。 
7. 填充DP后，检查dp[(1 << tj) − 1]。 如果存在，则意味着我们可以对所有查询到的男孩进行排序，使得每个男孩在处理时都被阻止，所以答案为“是”。 否则答案为“否”。 

### 为什么它有效

 DP 强制执行严格的不变量：每个状态代表男孩的某些子集的部分排序，使得每个处理过的男孩在选择时被完全阻止。 有效性条件保证在他所有喜欢的女孩都已在覆盖集合中之前，不会有男孩被放置，这与“无可用女孩”条件完全对应。 如果我们能够达到完整的子集，我们就构建了一个每个男孩都失败的排序。 如果我们不能，那么在某个阶段，每个剩余的未安置男孩都需要至少一个尚未覆盖的女孩，因此至少有一个男孩必须成功完成任何排序，从而得出“否”的答案。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, k = map(int, input().split())
    
    pref = []
    for _ in range(n):
        arr = list(map(int, input().split()))
        m = arr[0]
        mask = 0
        for g in arr[1:]:
            mask |= 1 << (g - 1)
        pref.append(mask)

    q = int(input())
    
    for _ in range(q):
        tmp = list(map(int, input().split()))
        t = tmp[0]
        boys = [x - 1 for x in tmp[1:]]

        dp = [None] * (1 << t)
        dp[0] = 0

        for mask in range(1 << t):
            if dp[mask] is None:
                continue
            covered = dp[mask]

            for i in range(t):
                if mask & (1 << i):
                    continue
                if (pref[boys[i]] & ~covered) == 0:
                    nmask = mask | (1 << i)
                    ncovered = covered | pref[boys[i]]
                    if dp[nmask] is None:
                        dp[nmask] = ncovered

        print("Yes" if dp[(1 << t) - 1] is not None else "No")

if __name__ == "__main__":
    solve()
```该实现将每个男孩的偏好压缩到一个位掩码中，以便检查男孩是否被阻止减少为单个按位 AND 运算。 DP 数组按当前查询中的男孩子集进行索引，每个条目存储该子集排序所覆盖的女孩的累积集。 

关键的实现细节是转换条件`(pref[boys[i]] & ~covered) == 0`，这强制男孩喜欢的每个女孩都已经在覆盖集中。 另一个微妙的点是，我们只为每个子集存储一个覆盖状态，因为我们只需要存在； 保持多个状态是不必要的，因为任何成功的构建就足够了。 

DP 在每个查询中独立运行，确保即使 q 很大时内存也保持有限。

 ## 边缘情况

 一种极端情况是查询中的所有男孩都有相同的偏好列表。 在这种情况下，要么第一个男孩根本无法被阻止，要么只有当初始覆盖集已经包含他们的整个偏好集时才能被阻止，这是不可能的，因此 DP 永远不会扩展到初始状态之外。 

另一种情况是男孩之间的偏好不一致。 这里每个男孩都会引入新的女孩，因此总是可以构造排序来满足阻塞，因为每一步都不受限制地增加覆盖范围，从而使 dp 达到满状态。 

最后一种情况是一个男孩的偏好列表为空。 在这种情况下，转换条件始终满足，并且 DP 立即允许任何排序，使得包含该男孩的任何子集的答案始终为“是”。
