 ## 解决方案
 ## 问题理解

 给定一个仅由字符 A、B 和 C 组成的字符串。我们希望从该字符串中提取尽可能多的不相交的索引三元组。 每个三元组必须选择三个位置 i、j、k，其中 i < j < k，并且这些位置处的字符必须按顺序形成模式 ABC 或按顺序形成反转模式 CBA。 

一旦索引在三元组中使用，就不能在任何其他三元组中重复使用。 任务是最大化可以形成多少个有效的三元组。 

约束 n ≤ 3 × 10^5 意味着任何比每次传递的线性或接近线性时间更差的解决方案都将陷入困境。 对索引三元组的三次或二次方法是立即不可能的，因为它将涉及 n^3 或 n^2 运算的顺序，这远远超出了一秒限制所允许的范围。 如果天真地实现，即使是在不仔细记录的情况下重复扫描字符串以查找模式的贪婪尝试也可能会退化为二次行为。 

这个问题的一个微妙问题是，如果操作不当，局部贪婪匹配可能会失败。 例如，如果我们总是匹配我们找到的最早的 ABC，而不考虑全局结构，我们可能会在以后阻止更好的配对。 类似地，如果我们只在一个方向上贪婪地形成三元组（比如总是从左到右的 ABC），我们就会错过需要对称推理的有效 CBA 结构。 

另一个极端情况是角色严重不平衡。 例如，像“AAAABBBBBCCCCC”这样的字符串包含许多每个字符，但除非排列正确，否则没有有效的三元组。 任何仅基于频率的幼稚计数方法都会高估或错位结构。 

## 方法

 暴力解决方案将尝试所有可能的索引 i < j < k 的三元组，并检查字符是否形成 ABC 或 CBA，同时确保所选索引不被重复使用。 这在概念上很简单，但需要在不相交约束下跟踪组合。 即使我们尝试贪婪地标记已使用的索引，我们仍然需要重复扫描有效的三元组，并且每次扫描的成本可能为 O(n)。 在最坏的情况下，重复此操作最多 O(n) 次会导致 O(n^2) 行为。 

关键的观察是，我们并不是真正构建任意三元组，而是匹配三个固定字符类型上的有序模式。 每个三元组按索引顺序是 A → B → C 或 C → B → A。 中间的字符始终是 B，这表明了一种自然分解：每个有效的三元组恰好使用一个 B、一个 A 和一个 C。排序约束仅决定 A 是在 B 之前还是之后，对于 C 也是如此。 

这将问题简化为将 B 与相反两侧的 As 和 C 配对。 我们可以确定有多少个 B 作为中心，而不是全局地考虑三元组，并且对于每个这样的选择，我们可以计算可以形成多少个有效的三元组。 

更有用的理解方法是，对于每个 B，我们可以尝试使用左侧的 A 和右侧的 C 形成 ABC 三元组，或者使用左侧的 C 和右侧的 A 形成 CBA 三元组。 一旦使用了B，它的贡献就固定了。 这自然表明了一种贪婪扫描策略，我们使用前缀和后缀信息维护两侧可用的 A 和 C 计数。 

当我们从左到右扫描时，我们预先计算 A 和 C 的前缀计数，后缀计数类似。 然后，对于 S[i] = B 的每个位置 i，我们知道它之前存在多少个 A，之后存在多少个 C，从而给出以 ABC 为中心的三元组的潜在数量。 对称地，我们还知道以 CBA 为中心的三元组之前存在多少个 C，之后存在多少个 A。 每个 B 最多贡献三倍，因此我们贪婪地为每个 B 选择最佳可用选项。 

这种本地选择是安全的，因为一旦我们通过计数考虑了消耗的字符，每个 B 都是独立的，并且我们永远不会由于减少可用计数而重用索引。

 ## 算法演练

 1. 从左到右扫描字符串时计算 A 和 C 的前缀计数。 这告诉我们，对于任何位置，其左侧有多少可用的 A 和 C 字符。 
2. 从右到左计算 A 和 C 的后缀计数。 这给出了任意位置右侧存在多少个可用的 A 和 C 字符。 
3. 迭代字符串中的每个索引 i。 当S[i]不是B时，跳过它，因为只有B可以作为三元组的中心。 
4. 对于位置 i 处的每个 B，计算两个可能的贡献。 1 是用左边 1 个 A 和右边 1 个 C 可以组成的 ABC 三元组的数量。 另一个是使用左侧一个C和右侧一个A的CBA三元组的数量。 
5. 选择生成有效三元组并消耗可用字符的选项。 减少相应的前缀和后缀的可用性，以便不重用索引。 
6. 累积形成的三元组的数量并继续扫描。 

### 为什么它有效

 每个有效的三元组都由 B 索引及其每一侧的一个字符的选择来唯一确定。 前缀和后缀计数代表不相交的索引池，因此一旦一个字符被消耗为三元组，它就不能出现在另一个有效的构造中。 每个 B 处的贪婪选择是安全的，因为后续决策无法回收已使用的索引，并且每个 B 最多参与一个三元组。 这强制执行从局部可行分配构建的全局不相交匹配，而没有重叠。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    s = input().strip()
    n = len(s)

    preA = [0] * (n + 1)
    preC = [0] * (n + 1)

    for i in range(n):
        preA[i + 1] = preA[i] + (s[i] == 'A')
        preC[i + 1] = preC[i] + (s[i] == 'C')

    sufA = [0] * (n + 1)
    sufC = [0] * (n + 1)

    for i in range(n - 1, -1, -1):
        sufA[i] = sufA[i + 1] + (s[i] == 'A')
        sufC[i] = sufC[i + 1] + (s[i] == 'C')

    usedA = usedC = 0
    ans = 0

    for i, ch in enumerate(s):
        if ch != 'B':
            continue

        leftA = preA[i] - usedA
        leftC = preC[i] - usedC
        rightA = sufA[i + 1]
        rightC = sufC[i + 1]

        # Try ABC: A left, C right
        if leftA > 0 and rightC > 0:
            usedA += 1
            usedC += 1
            ans += 1
        # Try CBA: C left, A right
        elif leftC > 0 and rightA > 0:
            usedA += 1
            usedC += 1
            ans += 1

    print(ans)

if __name__ == "__main__":
    solve()
```前缀数组计算每个位置之前存在多少个 A 和 C 字符，而后缀数组计算每个位置之后的字符数。 变量usedA和usedC跟踪早期三元组中已经消耗了多少变量，确保不相交。 

在每个 B 处，我们首先检查是否可以形成 ABC 三元组。 如果没有，我们尝试组建CBA三人组。 优先顺序是任意的，因为两者都消耗一个 A 和一个 C，并且两者在资源使用上是对称的。 

## 工作示例

 考虑字符串 ABCBBAC。 

我们计算前缀和后缀计数，然后扫描 B 位置。 

| 我| 字符 | 左A | 左C | 右A | 右C | 选择|
 | --- | --- | --- | --- | --- | --- | --- |
 | 3 | 乙| 1 | 0 | 1 | 1 | ABC |
 | 4 | 乙| 1 | 0 | 1 | 0 | CBA不可能，跳过|

 位置 3 处的第一个 B 使用位置 0 处的 A 和位置 6 处的 C。之后第二个 B 无法形成有效的三元组。 

这显示了字符的消耗如何防止重用并强制不相交。 

现在考虑一下 BACABA。 

| 我| 字符 | 左A | 左C | 右A | 右C | 选择|
 | --- | --- | --- | --- | --- | --- | --- |
 | 1 | 乙| 1 | 0 | 2 | 1 | ABC |
 | 4 | 乙| 1 | 1 | 0 | 0 | 无 |

 第一个 B 形成有效的 ABC 三元组。 由于双方都缺乏可用端点，第二个 B 无法完成任何有效模式。 

这些痕迹显示了算法如何自然地优先考虑早期可行的匹配并避免过度使用稀缺字符。 

## 复杂度分析

 | 测量| 复杂性 | 说明|

 ## 边缘情况

 像“BBBBBB”这样的字符串仅包含潜在的中心，但不包含有效的 A 或 C 端点。 该算法处理每个 B，但发现 leftA、leftC、rightA、rightC 均为零，因此没有形成三元组。 

像“AAACCCBBB”这样的字符串有丰富的端点，但对于许多 B 位置来说，它们都位于错误的一边。 前缀和后缀分离可确保不会创建无效配对，并且仅计算结构上有效的三元组。 

像“ACACACBBB”这样高度倾斜的情况表明，早期的 B 位置可以消耗有限的端点，从而阻止以后的匹配。 usedA 和usedC 跟踪确保一旦A 或C 被提交，它就不会被重用，从而在贪婪选择下保持正确性。
