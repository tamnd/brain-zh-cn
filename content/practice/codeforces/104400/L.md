 ## 解决方案
 ## 问题理解

 我们得到了一份长度为一天的时间表`n`，其中每分钟恰好有一辆公共汽车到达，并且每个到达都标有公共汽车号码。 所以输入数组是一个标识符序列，第 i 个值告诉我们哪辆公共汽车在第 i 分钟到达。 

之后，我们收到`q`查询。 每个查询都询问特定的巴士号码`x`和一个位置`y`，并且我们必须报告第 y 次出现总线的分钟`x`发生在时间表中。 如果该总线出现的次数少于`y`次，我们回来`-1`。 

核心操作不是关于时间本身，而是关于过滤子序列内的索引出现。 对于每个公交车号码，我们从概念上关心它出现的所有位置的列表。 

约束条件是`n, q ≤ 100000`。 每个查询直接扫描会太慢。 如果每个查询扫描整个数组，我们得到`O(nq)`，在最坏的情况下大约需要 10^10 次操作，远远超出了 Python 中 2 秒允许的范围。 

一个更微妙的限制是总线编号可以大到 10⁹，因此我们不能将它们用作直接数组索引。 任何解决方案都必须依赖于散列或坐标压缩。 

一些边缘情况在这里很重要。 如果总线仅出现一次，则任何要求第二次出现的查询都应返回`-1`。 如果所有值都相同，则查询大值`y`价值观常常会失败。 如果总线从未出现过，则每次查询它都会立即`-1`。 这些情况破坏了在不检查列表长度的情况下假设存在的解决方案。 

## 方法

 蛮力的想法很简单。 对于每个查询`(x, y)`，我们扫描整个时间表并计算`x`直到我们到达第 y 场比赛。 一旦我们命中它，我们就返回索引。 如果我们先完成数组，我们返回`-1`。 

这是正确的，因为它直接模拟了问题的定义。 问题是成本。 每次查询的成本`O(n)`时间，所以跨越`q`我们执行的查询最多`n × q`比较。 如果两者都达到 100000，这就显得太大了。 

关键的观察结果是时间表没有改变。 每个查询都询问相同的固定序列。 这意味着我们可以对所有出现的事件进行一次预处理，为每个总线编号存储它出现的位置列表。 然后，每个查询都变成一个简单的数组查找：该列表的第 y 个元素（如果存在）。 

这将问题从重复扫描转变为索引访问。 繁重的工作转移到阵列上的一次传递中。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(nq) | O(1) | O(1) | 太慢了|
 | 预计算位置 | O(n + q) | O(n) | 已接受 |

 ## 算法演练

 1. 创建一个字典，将每个公交车号码映射到它出现的位置列表。 这种结构是必要的，因为总线数量大且稀疏，因此数组索引不可行。 
2. 从左到右扫描时间表。 对于每一分钟 i，将 i 附加到与该位置的公交车号码相对应的列表中。 这会构建精确的发生顺序，这是必需的，因为查询依赖于时间顺序。 
3. 预处理后，每个总线编号都有一个出现索引的排序列表，自然是按升序排列，因为我们是从左到右插入的。 
4. 对于每个查询`(x, y)`，检查是否`x`字典中存在。 如果没有，立即输出`-1`因为公共汽车永远不会出现。 
5. 如果存在，检查列表是否至少有`y`元素。 如果没有，则输出`-1`因为请求的事件不存在。 
6. 否则输出index处的元素`y-1`在列表中。 这直接对应于时间顺序中第 y 次出现。 

这种索引起作用的原因是我们在预处理过程中保留了准确的时间顺序，因此列表顺序相当于时间顺序。 

### 为什么它有效

 该算法依赖于每个公交车号码的不变量`x`，其存储的列表包含所有索引，其中`x`出现，严格按升序排列。 由于每个查询都要求及时出现第 y 次，因此这相当于该有序列表中的第 y 个元素。 不需要重新排序或重新计算，因此每个查询都减少为对正确的预计算结构的恒定时间查找。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, q = map(int, input().split())
    a = list(map(int, input().split()))

    pos = {}

    for i, v in enumerate(a, start=1):
        if v not in pos:
            pos[v] = []
        pos[v].append(i)

    out = []

    for _ in range(q):
        x, y = map(int, input().split())
        if x not in pos:
            out.append("-1")
        else:
            lst = pos[x]
            if y <= 0 or y > len(lst):
                out.append("-1")
            else:
                out.append(str(lst[y - 1]))

    sys.stdout.write("\n".join(out))

if __name__ == "__main__":
    solve()
```预处理循环构建一个以总线号为键的哈希图，存储所有出现的索引。 使用`enumerate(..., start=1)`确保我们存储从 1 开始的位置，匹配问题的分钟索引。 

通过检查字典成员资格和列表边界，在恒定时间内处理每个查询。 边界条件`y > len(lst)`至关重要，因为缺少它会导致超出范围的访问或对频率的错误假设。

 ## 边缘情况

 从未出现的总线完全通过字典成员资格来处理。 什么时候`x`不存在，查找立即失败并返回`-1`无需触及任何列表。 这可以避免意外的 KeyError 或不正确的默认假设。 

出现次数少于请求次数的总线依赖于存储的列表长度检查。 例如，如果`pos[x] = [2, 5]`并且查询要求`y = 3`，条件`y > len(pos[x])`触发、返回`-1`。 如果没有此检查，则访问`pos[x][2]`会崩溃的。 

大的`y`值是安全的，因为它们不需要迭代。 该算法从不计算查询期间的出现次数，所以极端`y`除了单次比较之外，值不会影响性能或正确性。
