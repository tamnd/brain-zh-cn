 ## 解决方案
 ## 问题理解

 我们有固定的可用时间（以小时为单位），在此期间可以通电。 每一小时恰好对应于机器的一个洗涤周期。 衣服有多种颜色，每种颜色都有一定数量的必须全部清洗的物品。 

洗衣机每小时有两个独立的约束。 首先，一个小时最多可以处理`P`不同的颜色。 其次，对于任何选择的颜色，在该小时内最多可以清洗`K`该颜色的物品。 超过可用的`H`小时，我们必须决定是否存在选择`P`使所有衣服都充分清洗，如果是这样，找到最小的`P`。 

所以结构是：每种颜色`i`有需求`A[i]`。 每个小时都给我们`P`并行的颜色“槽”，每个槽贡献最多`K`洗涤该颜色的物品。 一种颜色可能会在多个小时内出现，并随着时间的推移而积累容量。 

key输出是最小的整数`P`这样就存在一个时间表`H`小时完成所有工作。 如果即使每小时有无限的并行颜色，总容量也不足，那么答案是不可能的。 

限制很大：多达一百万种颜色和非常大的值`A[i]`和`H`。 这立即排除了任何超过几个小时的模拟或每个项目的调度。 每小时或每个项目迭代的任何内容都太慢。 我们需要一个解决方案，将所有内容简化为数组上的聚合算术。 

当考虑总容量但忽略每种颜色的限制时，会出现微妙的失败情况。 例如，如果`A = [100, 100]`,`K = 10`,`H = 1`，那么即使总容量看起来很大，每种颜色也只能贡献`10`一小时内，所以两者都无法完成。 另一个问题是当一个非常大的`A[i]`无论有多大，都会强制多个小时`P`是。 

## 方法

 如果我们固定一个值`P`，我们可以检查可行性。 每个小时内，每种选择的颜色贡献最多`K`洗过的物品，所以穿过`H`小时，每种选择的颜色最多可以贡献`H * K`总计项目。 这立即意味着每种颜色的要求：每个`A[i]`必须满足`A[i] ≤ H * K`，否则再多的并行度也无济于事，因为即使每小时选择一次颜色，它仍然不能超过该上限。 

假设每种颜色都可行，真正的限制是每小时必须同时激活多少种颜色。 对于一种颜色`i`，它需要`ceil(A[i] / K)`“时间段”，意味着它必须出现在许多不同的时间里，因为在一小时内它只能贡献`K`。 

所以每种颜色都会消耗`t[i] = ceil(A[i] / K)`小时。 每小时最多可主持`P`颜色，所以跨越`H`小时，可用颜色槽总数为`P * H`。 我们必须分配所有`t[i]`将单位放入这些槽中，注意每个单位占用该颜色的一小时槽。 这将问题简化为检查总和是否`t[i]`适合`P * H`，但还有一个额外的微妙之处：没有任何一种颜色可以超过`H`，这已经由可行性检查处理。 

因此可行性变为：`ceil(A[i]/K)`必须 ≤`P * H`。 

这将问题转化为单调谓词`P`。 如果某个`P`作品，任何更大的`P`也有效，因为我们只增加每小时的容量。 这种单调性允许二分查找`P`。 

我们还需要一个下限：每个单位槽要求每小时至少一种颜色`P ≥ max(t[i]) / H`，但二分查找可以自然地处理它。 

蛮力会尝试增加`P`从1开始检查可行性，每次检查都扫描`N`价值观。 那将是`O(N * P)`在最坏的情况下，这是不可行的`P`可以很大。 

优化的解决方案使用每次可行性检查一次通过以及二分搜索`P`。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(N * P) | O(1) | O(1) | 太慢了|
 | 最优（二分搜索）| O(N log N) | O(N log N) | O(1) | O(1) | 已接受 |

 ## 算法演练

 ### 1. 预先计算每种颜色的小时要求

 对于每种颜色`i`, 计算`t[i] = (A[i] + K - 1) // K`。 这表示该颜色必须出现多少小时才能完成清洗所有物品。 

推理是每个小时最多贡献`K`单位为颜色，因此除以`K`将物品需求转化为时间需求。 

### 2. 检查全局可行性范围

 如果有的话`t[i] > H`，立即返回`-1`。 这意味着即使我们将每个小时都奉献给这种颜色，它仍然无法完成。 

此步骤尽早隔离不可能的情况，而不考虑`P`。 

### 3.二分查找最小P

 我们搜索`P`在范围内`[1, N]`。 上限是安全的，因为在任何时间我们都不能分配超过`N`颜色。 

对于每位候选人`P`，我们检查总需求是否适合容量`P * H`。 

### 4. 给定 P 的可行性检查

 计算`sum(t[i])`。 如果`sum(t[i]) ≤ P * H`， 然后`P`就足够了。 

解释是每小时提供`P`可用的颜色槽位，因此在所有时间内我们都有总容量`P * H`。 

### 5.返回最小的有效P

 二分搜索确保我们收敛到最小值`P`则满足可行性。 

### 为什么它有效

 每种颜色独立地需要固定的出现小时数，等于`ceil(A[i]/K)`。 颜色之间唯一的耦合是每小时限制`P`。 超过`H`小时，我们正好有`P * H`颜色插槽。 由于除了通过插槽竞争之外，颜色不会产生干扰，因此可行性简化为简单的包装约束。 二分查找之所以有效是因为增加`P`严格增加可用槽位，同时不降低可行性，因此谓词是单调的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    N, K, H = map(int, input().split())
    A = list(map(int, input().split()))

    t = []
    total = 0

    for x in A:
        need = (x + K - 1) // K
        if need > H:
            print(-1)
            return
        t.append(need)
        total += need

    # binary search for minimum P
    lo, hi = 1, N
    ans = N

    while lo <= hi:
        mid = (lo + hi) // 2

        if total <= mid * H:
            ans = mid
            hi = mid - 1
        else:
            lo = mid + 1

    print(ans)

if __name__ == "__main__":
    solve()
```该代码首先将每种颜色压缩为其所需的小时数。 早期检查`need > H`捕获由于单一颜色超过总可用时间窗口而导致的不可能性。 

变量`total`累积所有小时需求。 可行性检查通过使用此预先计算的总和来避免重新扫描阵列。 然后二分查找检查是否`mid * H`足够大以容纳所有所需的颜色外观。 

一个常见的陷阱是忘记约束不是关于物品，而是关于每种颜色的小时时段。 转变为`ceil(A[i]/K)`是关键的结构步骤。

 ## 边缘情况

 一个重要的边缘情况是当单一颜色超过每种颜色的总容量时。 用于输入`N=1, K=3, H=2, A=[10]`，我们计算`t = 4`，大于`H`。 算法立即返回`-1`，即使二分搜索会尝试分配容量，也会正确拒绝。 

另一种情况是所有颜色都很小但数量非常多。 为了`N=10^6, K=1, H=1`，每种颜色正好需要一个小时时段，所以`total = N`。 答案就变成了`P = N`，因为每个小时只能处理`P`颜色，而且只有一小时。 该算法处理这个问题是因为它比较`N`反对`P * H`。 

最后一个微妙的情况是`H`足够大，每种颜色的可行性都成立，但总需求仍然紧张。 二分查找正确找到最小的`P`允许总体打包，并且不会被各个分布误导，因为所有耦合都在聚合中捕获`P * H`约束。
