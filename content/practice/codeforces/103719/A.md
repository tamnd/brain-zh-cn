 ## 解决方案
 ## 问题理解

 我们维护一长排石头，每块石头都有一个数值。 最初，每个位置都从固定基线开始，通常为零。 之后，应用一系列操作。 有些操作修改单个位置，而另一些操作则用新值覆盖整行。 在更新之间，我们需要报告所有宝石值的总和。 

关键的困难不是计算单个总和，而是在混合更新下进行计算：一次影响每个元素的全局覆盖，以及仅影响一个索引但即使在未来的全局覆盖之后仍必须正确反映的本地更新。 输出只是询问当前总和的每个查询的结果。 

尽管界面看起来很简单，但约束意味着石子的数量和操作的数量都可能足够大，以至于不可能对整个数组进行任何每次操作的遍历。 在每次全局覆盖后重新计算总和的简单方法将在每个此类查询中花费线性时间，从而在最坏的情况下导致二次行为，这远远超出了典型的两秒限制可以处理大输入的能力。 

有一些边缘情况暴露了不正确的天真的推理。 假设我们用值 5 覆盖整个数组，然后将单个位置更新为 10。粗心的实现可能会忘记，在添加新值之前必须从先前的全局基线中减去单个位置。 例如，将大小为 4 的数组中的所有值设置为 5 后，总和为 20。如果我们随后将位置 2 设置为 10，则正确的总和将变为 25，而不是 30，因为位置 2 之前贡献的是 5，而不是 0。 

当发生多次全局覆盖时，会出现另一种微妙的情况。 如果我们将所有值设置为 3，然后设置为 7，则必须忽略第二次覆盖之前发生的任何较早的单独更新，因为它们已被有效擦除。 未能跟踪此重置边界会导致陈旧的更新与新的全局状态混合在一起。 

## 方法

 直接的暴力解决方案显式地维护整个数组。 点更新只是分配一个索引，而求和查询则扫描整个数组。 最困难的部分是全局覆盖操作，这需要一一更新所有 n 个元素。 这使得每个全局操作为O(n)，如果此类操作频繁出现，则总成本变为O(n·q)，当n和q都很大时，该成本太大。 

关键的观察结果是操作的结构将时间分为多个阶段。 全局覆盖后，每个元素共享相同的值，直到本地更新更改它为止。 我们只需要记住当前的全局基线并跟踪与它的偏差，而不是物理地重写数组。 

诀窍是将数组视为由两层组成。 第一层是最近一次完全覆盖分配的全局值。 第二层由覆盖后应用的单独更正组成。 每次我们修改单个索引时，我们都会通过删除旧的贡献并添加新的贡献来调整总和。 为了有效地做到这一点，我们必须知道自上次全局覆盖以来该索引是否已被修改。 如果没有，则其旧值是当前的全局基线。 

这将每个操作减少到 O(1)，因为我们从不显式地触及整个数组。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力阵列重建 | O(nq) | O(n) | 太慢了 |
 | 惰性全局值 + 点增量 | O(q) | O(n) | 已接受 |

 ## 算法演练

1.初始化一个全局值，代表最近一次完全覆盖后所有元素的当前值。 最初该值为零，因为尚未发生覆盖。 
2. 维护数组当前的总和。 一开始，这也是零。 
3. 维护一个时间戳或版本计数器，每次发生全局覆盖时该计数器都会增加。 这让我们可以区分位置是在上次覆盖之前还是之后更新的。 
4. 对于每个索引，存储其最后更新的版本及其当前值。 当我们再次修改它时，需要重建它的贡献。 
5. 当我们收到值为 x 的全局覆盖操作时，将全局值更新为 x 并将总和重置为 n 乘 x。 还要增加全局版本计数器，以便所有以前的每个索引更新都变得无关紧要。 
6. 当我们收到索引 i 处值为 x 的点更新时，首先确定其先前的贡献。 如果其存储的版本比当前全局版本旧，则其旧值是全局值。 否则，其旧值就是存储的值。 从总和中减去这个。 
7. 将新值 x 赋给位置 i，将其版本标记为当前版本，并将 x 添加到总和中。 
8. 当要求输入总和时，只需输出存储的总和即可。 

核心思想是，每个元素始终具有明确的贡献，该贡献由当前的全局覆盖或其最新的单独更新确定。 任何操作都不需要扫描整个阵列。 

### 为什么它有效

 在任何时间点，每个元素都属于两种状态之一：自上次全局覆盖以来未受影响，或在其之后单独更新。 全局覆盖为所有未触及的元素定义了统一的基线。 由于每个点更新都会使用该元素的正确先前值显式调整总和，因此每次操作后总和保持一致。 版本控制机制确保过时的更新永远不会干扰新的全局状态。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, q = map(int, input().split())
    arr = [0] * n
    ver = [0] * n

    global_val = 0
    global_ver = 0
    total = 0

    for _ in range(q):
        cmd = input().split()

        if cmd[0] == '1':
            i = int(cmd[1]) - 1
            x = int(cmd[2])

            if ver[i] == global_ver:
                old = arr[i]
            else:
                old = global_val

            total -= old
            total += x

            arr[i] = x
            ver[i] = global_ver

        else:
            x = int(cmd[1])
            global_val = x
            global_ver += 1
            total = n * x

        print(total)

if __name__ == "__main__":
    solve()
```该实现依赖于将逻辑状态与物理存储分离。 数组`arr`和`ver`存储每个索引的覆盖及其相对于上次全局覆盖的新鲜度。 这对`(global_val, global_ver)`代表最后一次完全重置。 在处理点更新时，关键的一步是判断前一个值是否来自`arr[i]`或来自`global_val`，这正是版本检查编码的内容。 

一个常见的错误是在发生全局覆盖时忘记重置每个索引的历史记录。 版本计数器不会清除数组，而是使旧条目自动无效，而无需触及它们。

 ## 边缘情况

 当在几个点更新后发生全局覆盖时，所有先前的每个索引状态都变得无关紧要。 版本计数器通过使旧条目在逻辑上与当前全局版本不兼容来确保这一点。 例如，如果我们在完全重置之前更新索引 1，则该更新不得影响任何后续计算。 

输入：```
2 3
1 1 5
2 3
3
```执行显示，在全局覆盖之后，索引 1 被视为具有值 3，无论其之前的分配如何，并且总数变为 6。过时的值 5 永远不会再次使用，因为其版本与当前全局版本不匹配。 

即使更新以任意顺序交错，该机制也能保证正确性。
