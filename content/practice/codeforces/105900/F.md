 ## 解决方案
 ## 问题理解

 我们给定三个整数$a$,$b$， 和$M$。 任务不是计算通常意义上的“获取索引并返回值”的斐波那契数，而是评估索引本身就是乘积的斐波那契数：我们需要$f_{a \cdot b}$，然后取模$M$。 

关键对象是从零开始索引的斐波那契数列，其中$f_0 = 0$,$f_1 = 1$，后面的每一项都是前两项的总和。 输出只是除以后的余数$(a \cdot b)$- 第一个斐波那契数$M$。 

限制条件非常大：$a$,$b$， 和$M$最多可达约$10^9$。 这立即排除了通过对索引进行迭代或递归来直接计算斐波那契数的可能性。 甚至在索引处计算单个斐波那契数$10^9$使用线性递归是不可能的，因为它需要$10^9$步，这远远超出了任何时间限制。 产品$a \cdot b$甚至更大，因此我们必须将指数视为抽象数字并依赖斐波那契数的结构特性。 

第二个微妙点是模数$M$也很大并且不一定是素数。 这排除了假设可逆性或模除技巧的技术。 一切都必须适用于任意模数。 

一个天真的但常见的失败案例来自于对索引的误解。 例如，如果$a = 3$,$b = 2$，那么我们需要$f_6$，即$8$。 粗心的解释可能会计算出$f_3 \cdot f_2 = 2 \cdot 1 = 2$，这是完全不相关的。 

另一个微妙的边缘情况是中间乘法的溢出$a \cdot b$。 尽管最终的斐波那契计算将以不同的方式完成，但索引乘积本身必须在没有大整数的语言中小心处理，尽管 Python 在这里是安全的。 

## 方法

 一种直接方法是逐一计算斐波那契数，直到达到索引$a \cdot b$，然后取模$M$。 这在原则上是正确的，因为递归唯一地定义了序列。 然而，它需要索引中的线性时间，对于超过几百万的输入，这立即变得不可行，并且这里索引高达$10^{18}$。 

关键的观察结果是斐波那契数在矩阵求幂下表现良好。 复发$$f_n = f_{n-1} + f_{n-2}$$可以编码为线性变换：$$\begin{bmatrix}
f_{n+1} \\
f_n
\end{bmatrix}
=
\begin{bmatrix}
1 & 1 \\
1 & 0
\end{bmatrix}
\begin{bmatrix}
f_n \\
f_{n-1}
\end{bmatrix}$$这意味着计算$f_n$简化为固定 2x2 矩阵的幂$n$。 

矩阵求幂允许我们计算$n$- 第一个斐波那契数$O(\log n)$使用重复平方的时间。 每个乘法都是恒定大小的，并且所有运算都以模数完成$M$，因此值仍然有界。 

剩下的难度就是指数$a \cdot b$。 我们不需要显式生成达到该索引的斐波那契值； 我们只需要计算指数，然后应用快速取幂。 由于乘法$a \cdot b$适合 64 位范围（最多$10^{18}$），我们可以安全地直接在 Python 中计算它，或者在需要时使用模块化归约。 

因此问题归结为计算$f_{a \cdot b} \bmod M$使用快速矩阵求幂。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O(a \cdot b)$|$O(1)$| 太慢了|
 | 矩阵求幂|$O(\log (a \cdot b))$|$O(1)$| 已接受 |

 ## 算法演练

 我们将斐波那契计算重写为矩阵幂问题并对其进行有效评估。 

1. 计算指数$n = a \cdot b$。 这就是我们需要的斐波那契位置。 乘法在 Python 中是安全的，因为整数是无界的，但在其他语言中，这必须使用 64 位算术。 
2.定义变换矩阵$$T =
\begin{bmatrix}
1 & 1 \\
1 & 0
\end{bmatrix}$$该矩阵编码连续斐波那契状态之间的递推关系。 
3. 计算$T^n$使用二进制求幂。 在每一步中，当相应位为$n$已设置。 每个乘法均以模进行$M$，因此所有中间值仍然有界。 
4. 求幂后，结果矩阵的右上角结构包含斐波那契值。 具体来说，$f_n$可以从矩阵形式中提取，通常作为从标准基向量开始时​​第一行的第二个元素。 
5、输出$f_n \bmod M$。 

二进制取幂之所以重要，是因为它利用指数的二进制表示，用平方运算的对数序列代替了线性乘法链。 

### 为什么它有效

 该算法依赖于应用矩阵的不变量$T$一旦转变$(f_n, f_{n-1})$进入$(f_{n+1}, f_n)$。 通过归纳法，应用$T^k$精确推进斐波那契状态$k$步骤。 所以，$T^n$应用于基向量$(f_1, f_0)$产生$(f_{n+1}, f_n)$，并且所需的斐波那契值直接嵌入到该结果中。 由于矩阵乘法是结合律，因此通过重复平方求幂可以保留正确性，同时仅更改计算顺序。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def mat_mul(a, b, mod):
    return [
        [
            (a[0][0]*b[0][0] + a[0][1]*b[1][0]) % mod,
            (a[0][0]*b[0][1] + a[0][1]*b[1][1]) % mod
        ],
        [
            (a[1][0]*b[0][0] + a[1][1]*b[1][0]) % mod,
            (a[1][0]*b[0][1] + a[1][1]*b[1][1]) % mod
        ]
    ]

def mat_pow(mat, exp, mod):
    res = [[1, 0], [0, 1]]
    base = mat

    while exp > 0:
        if exp & 1:
            res = mat_mul(res, base, mod)
        base = mat_mul(base, base, mod)
        exp >>= 1

    return res

def fib(n, mod):
    if n == 0:
        return 0
    T = [[1, 1], [1, 0]]
    res = mat_pow(T, n-1, mod)
    return res[0][0] % mod

a, b, M = map(int, input().split())
n = a * b
print(fib(n, M))
```核心思想是矩阵幂函数`mat_pow`，它实现了 2x2 矩阵的二进制提升。 乘法例程`mat_mul`确保每个条目都按模减少$M$，防止溢出并保持值一致。 

功能`fib(n, mod)`处理矩阵幂定义和斐波那契索引之间的逐一移位。 自从$T^{n-1}$应用于基向量产生$f_n$，我们显式地在指数中减一。 特殊情况$n = 0$单独处理是因为矩阵公式不能自然地表示它。

def solve():
    import sys
    input = sys.stdin.readline

    def mat_mul(a, b, mod):
        return [
            [(a[0][0]*b[0][0] + a[0][1]*b[1][0]) % mod,
             (a[0][0]*b[0][1] + a[0][1]*b[1][1]) % mod],
            [(a[1][0]*b[0][0] + a[1][1]*b[1][0]) % mod,
             (a[1][0]*b[0][1] + a[1][1]*b[1][1]) % mod]
        ]

    def mat_pow(mat, exp, mod):
        res = [[1, 0], [0, 1]]
        base = mat
        while exp > 0:
            if exp & 1:
                res = mat_mul(res, base, mod)
            base = mat_mul(base, base, mod)
            exp >>= 1
        return res

    def fib(n, mod):
        if n == 0:
            return 0
        T = [[1, 1], [1, 0]]
        res = mat_pow(T, n-1, mod)
        return res[0][0] % mod

    a, b, M = map(int, input().split())
    print(fib(a*b, M))

    sys.stdin = io.StringIO(inp)

 ## 边缘情况

 一个脆弱点是处理$n = 0$。 如果$a = 0$或者$b = 0$，那么乘积为零，答案一定是$f_0 = 0$。 在矩阵公式中，以零求幂会产生恒等式，无需特殊处理就不会直接编码斐波那契数。 该代码显式检查这种情况以避免返回错误的矩阵条目。 

另一个边缘情况是大模数$M = 1$。 任何斐波那契数模 1 都为零，但如果每次乘法没有模数约简，中间值可能会增长并使推理变得复杂。 该实现始终减少模数$M$每一步都如此，所以这会正确退化。 

第三个微妙的情况非常大$a \cdot b$。 即使产品到达周围$10^{18}$，二进制求幂仅取决于位数，因此算法保持稳定并且不依赖于对数缩放之外的幅度。
