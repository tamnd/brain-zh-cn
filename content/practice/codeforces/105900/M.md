 ## 解决方案
 ## 问题理解

 我们得到一个有向加权图，其中节点代表加油站，边代表具有行驶距离的单向道路。 我们从节点 1 开始，必须到达节点 N。从 1 到 N 的每条有效路由都有两个相互竞争的标准。 

首先，路线的总距离必须尽可能小。 在达到这个最小总距离的所有路线中，我们接下来关心路线在停靠点之间的燃料消耗方面有多“平滑”。 平滑度通过沿路径使用的最大单边权重来衡量。 最终目标是选择一条最短路径，其中最小化最大边权重。 

所以输出是两个值。 第一个是从 1 到 N 的最短可能距离。第二个是在所有最短路径中，该路径上使用的最大边的最小可能值。 

约束最多可达 200,000 个节点和边，权重最多可达 100,000。 这立即排除了任何三次甚至二次的。 一次 Dijkstra 运行是可以的，但任何涉及重新计算每条边的最短路径或枚举所有路径的操作都不行。 

当存在多个最短路径时，就会出现一个微妙的问题。 一种仅跟踪距离的简单最短路径算法将很乐意返回任何最短路径，但不一定是具有最佳瓶颈的路径。 另一个陷阱是首先尝试优化瓶颈，因为瓶颈最小路径可能不是总距离最短的。 

一个小的说明性冲突：

 输入：```
3 3
1 2 1
2 3 10
1 3 5
```经过 1 → 2 → 3 的最短距离为 6，但直接路径 1 → 3 的距离为 5，对于第一个标准无效。 最短路径中，只有 1 → 2 → 3 存在，所以答案为 (11, 10)。 天真的“首先最小化最大边缘”会错误地选择 1 → 3。 

另一个边缘情况是具有不同瓶颈的相等最短距离：```
3 3
1 2 1
2 3 2
1 3 3
```1→2→3 和 1→3 的距离都是 3。第一个有瓶颈 2，第二个有 3。正确答案是 (3, 2)。 

## 方法

 一个蛮力的想法是枚举从 1 到 N 的所有路径，计算它们的总距离和最大边权重，然后根据字典顺序选择最佳对。 这在概念上是正确的，但立即不可行。 即使在稀疏图中，路径的数量也可以呈指数增长，在类似 DAG 的结构中很容易达到 2^N。 每个路径评估的成本为 O(N)，因此总数变为指数且无法使用。 

一种更结构化的尝试是在最短距离上运行 Dijkstra 一次，然后将注意力仅限制在至少一条最短路径上的边缘。 这导致构建一个子图，其中仅当 dist[u] + w =​​= dist[v] 时才允许使用边 (u, v)。 这是有效的，因为任何最短路径都必须在每一步上满足这个等式。 

一旦我们有了这个最短路径 DAG，问题就简化为在其中选择一条从 1 到 N 的路径，从而最小化最大边权重。 这是一个经典的“最小化路径上的最大边”问题，可以通过另一个最短路径计算来解决，其中路径成本是沿路线的最大边而不是总和。 由于边缘松弛变为`max(current, w)`，我们可以在 DAG 上运行第二个类似 Dijkstra 的进程。 

关键的见解是目标的分离。 最短路径条件定义了一个约束曲面，在该曲面内我们优化了次要目标。 Dijkstra 的结构保证在我们推理二次成本之前所有最短距离都是固定的。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | 指数| O(N) | 太慢了 |
 | 两阶段 Dijkstra (dist + 约束瓶颈 DP) | O((N + M) log N) | O((N + M) log N) | O(N + M) | 已接受 |

 ## 算法演练

 ### 最优策略

 1. 从节点 1 运行 Dijkstra 以计算到每个节点的最短距离。 这确定了到达每个站点的最低可能总成本。 如果没有这个，我们就无法在每个节点本地定义“最短路径”的含义。 
2. 构造一个过滤邻接结构，仅保留满足 dist[u] + w =​​= dist[v] 的边 (u, v, w)。 这确保了每条保留的边都与至少一条最短路径兼容。 任何违反此条件的边都不会出现在有效的最短路径中，因为它会严格增加总距离。 
3. 在这个过滤后的图上，我们现在计算辅助 DP 值：从 1 到每个节点的任何路径上的最小可能最大边权重，仅限于最短路径。 
4. 初始化第二个类似 Dijkstra 的过程，其中节点 v 处的状态成本是迄今为止使用的最大边的最小可能值。 将源值设置为 0。 
5. 使用按该瓶颈值排序的优先级队列。 从节点 u 开始，遍历一条边 (u, v, w) 时，新的成本变为 max(current_cost, w)。 这精确地模拟了沿路径遇到的最大边缘。 
6. 如果这个新的瓶颈值改善了其当前的最佳值，则放松 v。 
7. 处理后，节点 N 的答案是 dist[N] 作为最短距离，best_bottleneck[N] 作为最短路径约束下的最小化最大边。 

### 为什么它有效

 第一个 Dijkstra 锁定精确的最短距离。 这将图变成了分层结构，其中每个有效的移动都必须保持最佳的距离增长。 在这个结构中，从 1 到 N 的任何路径都对应于原始图中的最短路径。 

由于每个保留的边都保留了局部最短路径最优性，因此连接这些边可以保留距离的全局最优性。 这意味着我们不再搜索所有路径，而仅搜索最短路径集。 

在该限制集中，第二个过程探索所有可行的最短路径，但使用单调成本函数（迄今为止看到的最大边缘）对它们进行比较。 此函数与 Dijkstra 式松弛兼容，因为扩展路径不能减少其最大边缘。 这保证了贪婪扩张仍然有效。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

import heapq
from collections import defaultdict

def dijkstra_dist(n, adj):
    INF = 10**30
    dist = [INF] * (n + 1)
    dist[1] = 0
    pq = [(0, 1)]
    
    while pq:
        d, u = heapq.heappop(pq)
        if d != dist[u]:
            continue
        for v, w in adj[u]:
            nd = d + w
            if nd < dist[v]:
                dist[v] = nd
                heapq.heappush(pq, (nd, v))
    
    return dist

def dijkstra_bottleneck(n, adj, dist):
    INF = 10**30
    best = [INF] * (n + 1)
    best[1] = 0
    pq = [(0, 1)]
    
    while pq:
        cur, u = heapq.heappop(pq)
        if cur != best[u]:
            continue
        for v, w in adj[u]:
            if dist[u] + w != dist[v]:
                continue
            nc = max(cur, w)
            if nc < best[v]:
                best[v] = nc
                heapq.heappush(pq, (nc, v))
    
    return best

def solve():
    n, m = map(int, input().split())
    adj = [[] for _ in range(n + 1)]
    
    for _ in range(m):
        a, b, w = map(int, input().split())
        adj[a].append((b, w))
    
    dist = dijkstra_dist(n, adj)
    best = dijkstra_bottleneck(n, adj, dist)
    
    print(dist[n], best[n])

if __name__ == "__main__":
    solve()
```该解决方案首先构建完整的有向邻接表。 第一个 Dijkstra 计算距节点 1 的精确最短距离，稍后将其用作过滤条件。 

第二阶段重用相同的邻接列表，但只允许保留最短路径相等的转换。 这种过滤是在松弛期间内联完成的，而不是预先构建第二个图表，这节省了内存和时间。 

一个关键的实现细节是相等性检查`dist[u] + w != dist[v]`。 这必须准确； 使用`>`或者`<`比较会错误地允许非最短边或丢弃有效边。 

瓶颈DP使用第二优先级队列，因为状态空间仍然受益于贪婪扩展。 尽管成本不是相加的，但最大函数沿扩展保持单调，使得 Dijkstra 式排序有效。

 ## 边缘情况

 一个常见的失败是独立于最短路径计算来处理瓶颈优化。 在直接边的跳数较短但权重较长的示例中，朴素的极小极大方法会错误地选择它。 

该算法可以处理这个问题，因为这些边缘在第一次距离传递期间被删除。 例如：```
3 3
1 2 1
2 3 10
1 3 5
```Dijkstra 给出 dist[3]=5。 边 2→3 不在任何最短路径上，因为 1→2→3 总计 11，超过 5，因此被忽略。 剩余的路径结构仅允许有效的最短路径，并且仅在其中计算瓶颈，从而保持正确性。
