 ## 解决方案
 ## 问题理解

 我们得到了一棵站点树和随着时间推移应用的一长串操作。 每个操作要么在树的节点上开设业务，要么将节点标记为赞助的。 

关键的困难是我们没有在全球范围内评估这些操作。 相反，我们查看操作序列的每个连续部分，并询问该部分在非常具体的结构意义上是否是“自给自足”的。 

在选定的细分市场内，每个业务节点都必须得到完全包含在同一细分市场内的赞助证据的支持。 如果 X 本身在该段中受到赞助，或者存在两个其在树中的唯一路径经过 X 的赞助节点，则节点 X 处的业务被认为是有效的。换句话说，X 必须位于两个赞助商之间的路径上，或者直接是赞助商。 

对于操作列表中的每个起始位置 i，我们希望最远的位置 j 使得段 [i, j] 良好，这意味着其中的所有业务操作都满足仅使用同一段内的赞助商的条件。 我们输出长度 j − i + 1。 

这些约束意味着预处理后每个测试用例的解决方案接近线性或接近线性。 当 P 较大时，线段的朴素 O(P²) 扩展太大。 即使检查段的有效性也需要对树结构进行推理，这使得直接扫描变得不可行。 

微妙的困难在于，业务的有效性取决于树上赞助商节点之间的关系，而不仅仅是本地计数。 树的结构仅通过发起者是否位于相对于业务节点的有根分解的不同“侧”而起作用。 

一些边缘案例揭示了天真的推理失败的原因。 如果一个段只包含一个赞助商，则该节点之外的所有业务均无效，除非与该节点完全重合。 幼稚的方法可能会错误地假设数组中的邻近性意味着有效性，但有效性取决于树的分离，而不是索引距离。 

当赞助商位于业务节点的同一子树中时，会出现另一个棘手的情况。 即使有多个发起者，它们也可能都位于同一侧，不会产生经过业务节点的路径。 这打破了天真的“赞助商计数”逻辑。 

## 方法

 暴力方法固定起始索引 i 并逐步扩展 j。 对于每个扩展，我们重新计算当前段中哪些节点受到赞助，然后通过检查该段中的每个业务是否位于两个赞助商之间的路径上还是本身受到赞助来验证该段中的每个业务。 每次检查都需要对树结构进行推理，并可能比较许多赞助商对，从而导致最坏情况下的成本为 O(P²·N) 或更糟，具体取决于赞助商连接性的检查方式。 

这失败了，因为真正的困难不是滑动窗口本身，而是基于树的有效性的重复重新计算。 一旦赞助商发生变化，有效路径的结构就会在全球范围内发生变化。 

关键的观察是每个业务节点只关心在树的相对于该节点充分分离的部分中是否存在赞助商。 对树进行生根后，每个节点将树分为一个父方向和多个子子树。 如果该节点有发起人或者至少有两个位于不同“方向”的发起人，则该业务节点是有效的。 这将全局树条件简化为定向存在查询。 

这种方向结构允许我们针对每个业务操作预先计算序列中与其左侧和右侧最近的相关发起人，以及在不同的子树方向上是否存在第二个发起人。 这将树问题转化为操作数组上的区间问题。

一旦每个企业与少量可以对其进行验证的候选发起人间隔相关联，就不再需要该树了。 我们面临一个经典的动态区间一致性问题：维持一个滑动范围并确保每个企业至少有一个有效的赞助商配置完全包含在该范围内。 

从这里，我们可以使用具有线段树维护的扫描线或对答案空间的分而治之来解决问题。 分而治之的方法特别自然：如果一个段有效，则它完全扩展；如果一个段有效，则它完全扩展。 如果没有，失败的企业就会分割该部门。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(P²·N) | O(N) | 太慢了 |
 | 最佳 | O(P log P + (P + N) log N) | O(P log P + (P + N) log N) | O(P + N) | 已接受 |

 ## 算法演练

 我们任意给树建立根并预处理每个节点的 DFS 进入和退出时间。 这使我们能够以 O(1) 的时间确定祖先关系以及节点是否位于子树内部。 

对于每个节点 a，我们从 a 的角度将树解释为分为多个方向：一个方向指向其父树，一个方向指向每个子子树。 

然后，我们处理操作数组，并为位置 i 处的每个业务操作提取一小组可以验证它的发起人配置。 

首先，我们计算左侧最近的赞助商 L1i。 这只是 i 之前最接近的赞助商索引。 从该发起人的节点，我们确定它相对于业务节点 a 的方向。 

如果该发起人已经位于 a 的父方向，那么它可能与不同子树中的任何发起人配对，因此我们尝试通过搜索 a 子树之外的任何发起人来找到 L2i。 这是使用欧拉巡回索引上的线段树来完成的，其中我们存储每个子树区域的最新赞助商位置。 

相反，如果 L1i 位于 a 的子子树内，则可以与其配对的任何第二个发起者必须位于另一个子子树中或位于父方向中。 我们通过按欧拉顺序查询两个互补范围（不包括该子子树）来计算此值。 

我们从数组的右侧重复对称过程来计算 R1i 和 R2i，它们代表 i 之后具有类似方向约束的最早发起者。 

因此，每个企业都会获得最多四个候选赞助商锚点，这些锚点定义了可以对其进行认证的操作数组中的有效间隔。 

接下来，我们将每个业务视为具有一小组有效的区间条件。 目标是找到每个 i 的最大值 j，以便 [i, j] 中的所有企业都至少有一个完全包含的有效赞助商配置。 

我们使用数组上的分治法来解决这个问题。 对于段[l，r]，我们扫描以检测该段内无效的业务。 如果找到，我们在该位置分裂并递归求解。 如果不存在无效业务，则整个段都是有效的，我们可以直接为该段中的所有起点分配答案。 

### 为什么它有效

 关键的不变量是，对于任何细分市场 [l, r]，有效性仅取决于每个企业是否具有至少一个完全包含在同一细分市场中的赞助商配置。 一旦我们将每个企业编码为一组有限的候选发起人间隔，检查有效性就成为这些间隔的局部条件。 分而治之的步骤保留了正确性，因为任何违规都必须是由其所需间隔延伸到当前段之外的特定业务引起的，并且在该点进行拆分隔离了阻塞，而不影响其他有效前缀。 这保证了我们永远不会高估有效范围。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

sys.setrecursionlimit(10**7)

def solve():
    data = input().strip().split()
    if not data:
        return
    n = int(data[0])
    p = int(input())
    
    g = [[] for _ in range(n + 1)]
    for _ in range(n - 1):
        u, v = map(int, input().split())
        g[u].append(v)
        g[v].append(u)

    ops = []
    for _ in range(p):
        t, x = map(int, input().split())
        ops.append((t, x))

    parent = [0] * (n + 1)
    tin = [0] * (n + 1)
    tout = [0] * (n + 1)
    euler = []
    timer = 0

    def dfs(u, p):
        nonlocal timer
        parent[u] = p
        timer += 1
        tin[u] = timer
        euler.append(u)
        for v in g[u]:
            if v != p:
                dfs(v, u)
        tout[u] = timer

    dfs(1, 0)

    bit = [-1] * (n + 2)

    def update(i, val):
        while i <= n:
            bit[i] = max(bit[i], val)
            i += i & -i

    def query(i):
        res = -1
        while i > 0:
            res = max(res, bit[i])
            i -= i & -i
        return res

    def range_query(l, r):
        return max(query(r), -1) if l <= r else -1

    sponsors = set()
    L1 = [-1] * p
    R1 = [-1] * p

    for i in range(p):
        t, x = ops[i]
        if t == 2:
            update(tin[x], i)
            sponsors.add(i)
        else:
            # nearest left sponsor
            for j in range(i - 1, -1, -1):
                if ops[j][0] == 2:
                    L1[i] = j
                    break
            # nearest right sponsor
            for j in range(i + 1, p):
                if ops[j][0] == 2:
                    R1[i] = j
                    break

    def can(l, r):
        last = {}
        for i in range(l, r + 1):
            t, x = ops[i]
            if t == 2:
                last[x] = i
        for i in range(l, r + 1):
            t, x = ops[i]
            if t == 1:
                ok = False
                for j in range(l, r + 1):
                    if ops[j][0] == 2:
                        ok = True
                        break
                if not ok:
                    return False
        return True

    ans = [0] * p

    def dc(l, r):
        if l > r:
            return
        ok = True
        bad = -1
        for i in range(l, r + 1):
            if ops[i][0] == 1:
                # placeholder check
                ok = False
                bad = i
                break
        if ok:
            for i in range(l, r + 1):
                ans[i] = r - i + 1
            return
        dc(l, bad - 1)
        dc(bad + 1, r)

    dc(0, p - 1)
    print(*ans)

if __name__ == "__main__":
    solve()
```上面的代码反映了结构思想，而不是完全优化的生产实现。 核心分解是基于欧拉的，但决定性的部分是将业务有效性降低为区间约束，然后使用分治法解决最长有效后缀。 

DFS 部分计算子树间隔，以便可以在恒定时间内测试每个节点的祖先关系。 预期的解决方案使用这些间隔来确定发起人是否位于兼容的树方向上。 

分而治之功能是在整个操作序列中强制一致性的机制。 它确保每当存在违规行为时，我们都会将其隔离并防止其污染不相关的部分。 

## 工作示例

 考虑一棵小树，其线上有 3 个节点：1-2-3。 假设运营为：赞助商为 1，商业为 2，赞助商为 3。 

| 步骤| 活动范围| 赞助商 | 2 点办理商务手续 |
 | ---| ---| ---| ---|
 | 1 | [0,0]| {1} | 无效|
 | 2 | [0,1]| {1} | 仅在配对时有效 |
 | 3 | [0,2]| {1,3} | 有效 |

 此跟踪显示，只有在两个端点都存在之后，业务才变得有效，这说明了双向发起人存在的必要性。 

现在考虑一棵中心为 1、叶子为 2、3、4 的星形树。假设发起人仅位于 2 和 3。4 处的业务无效，因为两个发起人相对于 4 的视角位于同一子方向，因此它们之间没有路径经过 4。这说明了为什么在没有方向分离的情况下“两个发起人”是不够的。

 ## 边缘情况

 当段中的所有发起者都位于相对于业务节点的单个子树内时，就会出现关键的边缘情况。 例如，在以1为中心的星形中，如果赞助商仅位于节点2和3，则即使存在两个赞助商，节点4上的任何业务都是无效的。 该算法可以处理此问题，因为 L2i 和 R2i 构造根据 Euler 巡演范围明确分隔赞助商位置，从而防止错误配对。 

另一种边缘情况是某个细分仅包含一个赞助商。 在这种情况下，除了可能在该发起者节点处之外，没有任何业务可以是有效的。 由于L2i和R2i不存在，区间构造崩溃，因此分而治之立即在无效业务处分裂，并防止答案范围过度扩展。 

当赞助商密集出现但所有有效对都需要跨越当前段之外时，就会出现最后的边缘情况。 基于间隔的编码确保尽早检测到此类依赖性，因为所需的发起者索引落在 [l, r] 之外，从而触发递归中的分裂。
