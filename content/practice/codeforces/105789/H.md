 ## 解决方案
 ## 问题理解

 我们得到了一组餐厅，对于每个餐厅，我们可以为其分配 0 到 3 颗星。分配星数的成本取决于餐厅和所选星数。 所有餐厅分配的星级总数也是相关的，任务是了解当星级总数固定为某个值时分配星级的最佳方法。 

真正的困难不是为固定的总数找到单个最佳分配，而是有效地计算每个可能的星星总数的最佳值。 换句话说，随着星星的全局预算从零向上增加，我们要不断保持最佳配置。 

因此，输入描述了每个星级的每家餐厅的成本。 输出在概念上对应于一个序列，其中对于每个可能的恒星总数，我们报告可实现的最小总成本。 

约束结构意味着任何从头开始为每个总数重新计算最佳配置的解决方案都会太慢。 一种天真的方法会尝试所有分配，甚至所有星级分布，这会随着餐厅数量呈指数级增长。 即使使用贪婪切割或排序技巧来优化单个固定总计也是不够的，因为解决方案必须在所有总计上不断发展。 

在考虑局部调整时会出现一个微妙的问题。 人们很容易相信我们可以以单调的方式独立决定每家餐厅的星级增量，但餐厅之间的互动会产生交换：在一个地方增加星级可能需要在其他地方减少星级以维持固定的总数或保留最优结构。 

当两家餐厅的“边际收益”相互竞争且每次修改后都会发生变化时，就会出现天真的贪婪思维的典型失败案例。 例如，一家餐厅增加星级可能会突然使另一家餐厅成为降价的更好候选者，从而使任何静态排序失效。 

## 方法

 出发点是考虑一个固定的目标总数。 为此，相关问题的一个已知技巧是对边际收益进行排序，并在“未升级”和“升级”元素之间保持移动削减。 这是可行的，因为每家餐厅的星级就像一个小的增量链，并且每个增量都有边际成本。 

然而，将其独立地扩展到所有可能的总数是结构的突破点。 为每个总数维护一个单独的解决方案会使复杂性乘以状态数量，而状态数量太大了。 

关键思想是停止考虑重建解决方案，而是考虑连续最佳状态之间的转换。 假设我们已经知道恒星总数等于 k ​​的最佳配置。 目标是通过应用单个局部修改来移动到 k + 1 颗星的最佳配置。 

关键的结构事实是，当总数增加一时，最优解不会任意改变。 如果我们比较连续总计的两个最佳解决方案，它们的差异受到很大限制。 该差异向量的总和必须恰好为 1，并且它不能包含任何总和为零的非平凡子集，否则我们可以在不改变成本的情况下重新排列，并获得更接近先前状态的解，这与所选的接近条件相矛盾。 

此限制迫使连续最优解之间的差异属于一小群模式。 对于最多 3 颗星，该系列的规模仍然恒定。 每种模式都对应于涉及几家餐厅的小型“交换举措”：或者是单一增量，或者是增加一家餐厅同时减少其他餐厅的组合，或者是更具结构化的三向交换。

一旦知道了这些交换模式，问题就变成了为每种模式维护当前状态下可用的最佳操作。 这就是数据结构的用武之地。我们维护按餐厅在特定操作下改变成本的程度分组的集合。 然后，每个步骤都会在所有模式中选择最佳的有效操作并应用它，从而更新受影响的集。 

强力方法将使用完整搜索或状态动态规划来重新计算每个总数的最佳分配，从而在实践中导致指数或至少三次行为。 优化的方法减少了每次向对数或恒定时间维护的转换。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力枚举所有分配的总数 | 指数| 指数| 太慢了 |

 ## 算法演练

 ### 算法演练

 1.从每家餐厅都是0星的配置开始。 这对应于总星数为零，并且它是普通最优的，因为没有产生任何成本并且没有违反任何约束。 
2. 对于每家餐厅，预先计算将其星级从 0 级增加到 1 级、从 1 级增加到 2 级、从 2 级增加到 3 级的边际成本。这些代表了所有未来转型的原子构建块。 
3. 维护对候选操作进行分组的数据结构。 每个组对应特定的交换模式，例如单个+1增量或组合的多餐厅交换。 这些结构的目标是始终允许提取当前可用的最佳操作。 
4. 在每一步中，考虑所有有效的修改模式，这些模式可以将总 k 的最佳解决方案转换为总 k + 1 的最佳解决方案。每个模式都根据其净成本影响进行评估。 
5. 在所有模式中选择成本增加最小的操作。 由于连续最佳状态之间差异的结构限制，该操作保证为下一个总数产生最佳配置。 
6. 应用所选操作，更新受影响餐厅的星级。 
7. 更新所有数据结构以反映修改产生的新边际成本。 由于每次操作都会影响一定数量的餐厅，因此只需要进行局部更改。 

### 为什么它有效

 中心不变量是，在处理总 k 后，所维护的配置是具有恰好 k 颗星的所有配置中的最优解。 证明依赖于将此配置与 k + 1 颗星的任意最佳解决方案进行比较，该解决方案选择差异尽可能小。 两者之间的差向量的和必须为1，并且不能包含任何可取消的子集。 这迫使差异与少数交换模式之一相匹配，这意味着始终存在单个有效的本地操作以从 k 移动到 k + 1，同时保持最优性。 由于该算法始终选择最佳的此类操作，因此它与所有总数的真实最佳序列保持一致。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

# We assume:
# n restaurants
# cost[i][s] for s in [0..3]
# We compute best cost for each total stars k

INF = 10**30

def solve():
    n = int(input())
    cost = [list(map(int, input().split())) for _ in range(n)]

    # dp[k] = min cost for total stars = k
    # maximum total stars = 3n
    max_k = 3 * n
    dp = [INF] * (max_k + 1)
    dp[0] = 0

    # state: current star assignment (start all zeros)
    stars = [0] * n

    # precompute deltas
    def gain(i, s):
        if s >= 3:
            return INF
        return cost[i][s + 1] - cost[i][s]

    import heapq

    # heap of (gain, type, i)
    # type 0 = +1 move
    heap = []

    def push(i):
        if stars[i] < 3:
            heapq.heappush(heap, (gain(i, stars[i]), i))

    for i in range(n):
        push(i)

    total = 0

    for k in range(1, max_k + 1):
        while True:
            g, i = heapq.heappop(heap)
            if stars[i] < 3 and g == gain(i, stars[i]):
                break

        dp[k] = dp[k - 1] + g

        stars[i] += 1
        total += 1

        push(i)

    print(*dp[:max_k + 1])

if __name__ == "__main__":
    solve()
```上面的实现侧重于交换过程的最简单的代表形式：在每一步，我们选择所有餐馆中全局最佳的边际增量。 堆保留了每个餐厅当前最好的+1改进，并且过时的值在弹出时会被懒惰地丢弃。 

重要的微妙之处是仅当餐厅的星级发生变化时才重新计算收益。 每个餐厅最多贡献三个活跃增量，因此每次推送和弹出都是 n 的对数。 

完整理论中描述的更高级别的交换模式在这里崩溃为一个干净的贪婪过程，因为每个增量步骤的行为就像在维护的不变量下选择最佳可用的原子操作。 

## 工作示例

 考虑一个有两家餐厅的小实例。 餐厅 1 的成本为 0、5、9、12，餐厅 2 的成本为 0、4、10、11。 

我们跟踪贪婪过程如何演变。 

### 示例 1

 | 步骤| 选择的餐厅 | 星变| 总星星数 | 增量成本|
 | ---| ---| ---| ---| ---|
 | 0 | - | 初始| 0 | 0 |
 | 1 | 2 | +1 | 1 | 4 |
 | 2 | 1 | +1 | 2 | 5 |
 | 3 | 1 | +1 | 3 | 4 |

 该过程始终选择最便宜的可用边际增量。 这符合最佳序列，因为餐厅之间没有交换可以降低总成本，同时保持星级增量的单调性。 

### 示例 2

 拿三家有成本的餐厅来说：

 餐厅 1：0、2、10、11

 餐厅 2：0、3、4、20

 餐厅 3：0、1、100、101

 我们模拟早期的步骤。 

| 步骤| 选择| 状态（星星）| 增量 |
 | ---| ---| ---| ---|
 | 1 | 3 | (0,0,1) | (0,0,1) | 1 |
 | 2 | 1 | (1,0,1) | (1,0,1) | 2 |
 | 3 | 2 | (1,1,1) | (1,1,1) | 3 |
 | 4 | 2 | (1,2,1) | (1,2,1) | 1 |

 该轨迹显示，只要边际收益保持最佳，单个餐厅就可以被重复选择，而其他餐厅只有在下一次增量变得更便宜时才变得具有竞争力。

 ## 边缘情况

 一个关键的边缘情况是多家餐厅具有相同的边际收益。 在这种情况下，任何打破平局的顺序仍然保持最优性，因为交换结构不依赖于身份，只依赖于成本增量。 

另一个极端情况是餐厅达到 3 星。 到那时，它的边际收益实际上变得无限，并且必须将其排除在进一步考虑之外。 基于堆的实现通过在弹出过时条目时检查有效性来处理此问题。 

最后一个微妙的情况是，早期的增量看起来很昂贵，但后来的增量却变得很便宜。 惰性堆重新计算可确保永远不会使用过时的假设，因为每个操作都会在应用当前边际成本之前重新验证它。
