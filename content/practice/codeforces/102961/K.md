 ## 解决方案
 ## 问题理解

 我们得到一个整数序列，表示不同数字的类似排列的排列。 任务是确定按升序处理所有数字需要多少“轮次”，其中每轮包括从左到右扫描序列，并在数字出现在下一个所需的顺序位置时拾取数字。 

更具体的思考方式是，我们想要按顺序读取值 1 到 n，但我们被迫重复从左到右遍历数组。 每次遍历时，我们都会选择尽可能多的下一个值，而无需在数组中向后移动。 一旦我们无法再继续当前的pass，我们就开始一个新的pass。 

输入给出 n，后跟一个长度为 n 的数组，其中包含从 1 到 n 的每个整数恰好一次。 输出是一个整数，即按升序使用序列所需的完整从左到右传递的数量。 

约束结构意味着 n 可以很大，通常高达 200,000 或类似 Codeforces 类型的问题。 这立即排除了任何重复模拟完全遍历阵列的方法。 在最坏的情况下，每个数字扫描一次数组的简单模拟的行为类似于 O(n²)，这会太慢。 

当数组已经排序时，会出现微妙的边缘情况。 在这种情况下，所有内容都会在一次传递中消耗掉。 相反的极端是反向排序数组，其中每个元素都强制进行新的传递，因为下一个所需的数字在数组中出现的时间早于前一个。 

例如，如果 n = 5 并且数组为 [5, 4, 3, 2, 1]，则在完整传递完成之前我们无法选择 1，因此每个数字都需要自己的传递，从而导致输出 5。如果不仔细重置传递之间的状态，简单的每传递扫描方法可能会意外地错误地重新计算进度。 

## 方法

 蛮力的想法是从字面上模拟这个过程。 我们维护当前尝试选择的目标值，从 1 开始。我们从左到右重复扫描数组。 每当我们遇到当前目标时，我们都会增加目标。 一旦我们到达数组的末尾，我们就会计算一次并重新开始扫描，直到所有数字都被消耗掉。 

这是有效的，因为它忠实地遵循流程的定义。 问题是成本。 在最坏的情况下，每次传递可能只消耗一个元素，因此我们扫描整个数组以查找 n 个值中的每一个。 这导致大约 n 遍，每次执行 O(n)，产生 O(n²) 总操作。 

关键的观察结果是，重要的不是扫描本身，而是连续数字在数组中相对于彼此出现的位置。 如果数字x+1出现在x的左边，那么当到达x时我们就不能继续当前的遍历，因为我们在这次扫描中已经通过了x+1。 每次发生这种情况，我们都被迫开始一个新的过程。 

因此，我们不是模拟传递，而是跟踪数组中每个值的位置。 我们遍历从 1 到 n 的值并比较位置。 每当当前值的位置小于前一个值的位置时，就意味着单次扫描中顺序中断，需要重新扫描。 

这将问题简化为计算位置序列减少了多少次。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(n²) | O(1) | O(1) | 太慢了|
 | 最佳 | O(n) | O(n) | 已接受 |

 ## 算法演练

 1. 构建数组`pos`在哪里`pos[x]`存储值的索引`x`在给定的排列中。 这将问题从处理数组本身转换为处理位置，这使得比较时间恒定。 
2. 初始化计数器`rounds = 1`因为总是需要至少一次才能开始处理序列。 
3. 迭代`x`从 2 到 n，比较`pos[x]`和`pos[x-1]`。 每次比较都会检查我们是否可以在同一扫描中继续当前的递增序列。 
4. 如果`pos[x] < pos[x-1]`, 增量`rounds`1. 这意味着价值`x`出现在之前`x-1`在数组中，所以在从左到右的扫描中我们会遇到`x-1`首先，因此无法处理`x`在同一个通道中。 
5.循环结束后，输出`rounds`。 

### 为什么它有效

 该算法依赖于单次扫描只能处理索引按递增顺序出现的值的不变量。 每次我们检测到连续值的索引序列减少时，我们都会识别两次扫描之间的边界。 由于值严格按照递增的数字顺序消耗，因此之间的每次反转`pos[x]`和`pos[x-1]`强制进行新的遍历。 此类中断的次数与所需的传球次数完全匹配。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

n = int(input().strip())
a = list(map(int, input().split()))

pos = [0] * (n + 1)
for i, v in enumerate(a):
    pos[v] = i

rounds = 1
for x in range(2, n + 1):
    if pos[x] < pos[x - 1]:
        rounds += 1

print(rounds)
```第一个循环构建位置图，以便每个值在 O(1) 时间内知道其位置。 这一点至关重要，因为它将重复扫描转变为简单的比较。 

第二个循环实现了核心观察：我们只关心连续值是否以递增的索引顺序出现。 初始化为 1 反映了这样一个事实：即使是完美有序的数组也需要至少一次传递。 

一个常见的错误是初始化`rounds`到 0 并尝试仅在中断时增加，这会少计数 1。 另一个微妙的问题是将值与索引混淆，比较必须在值的位置之间进行，而不是值本身。

def run(inp: str) -> str:

 ## 边缘情况

 单元素数组，例如`[1]`是最简单的可能场景。 位置数组仅包含一个值，并且连续对上的循环永远不会运行，从而留下`rounds = 1`。 该算法无需特殊外壳即可正确处理此问题。 

一个反向排序的数组，例如`[4, 3, 2, 1]`产生位置，其中每个`pos[x] < pos[x-1]`确实如此。 循环递增`rounds`每一步都经过 4 次。 这符合每个数字必须在单独的遍历中处理的直觉，因为每个下一个所需的值都出现在数组中的较早位置。 

部分有序的情况，例如`[3, 1, 2, 4]`当位置顺序从 2 移动到 3 时，会产生一个中断。 该算法准确地检测到 1 个减少并输出 2，从而正确地将序列分为两次扫描。
