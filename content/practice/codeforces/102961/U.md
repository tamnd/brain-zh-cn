 ## 解决方案
 ## 问题理解

 我们拥有一组独立的工厂机器，每台机器都可以重复生产相同的物品。 第 i 台机器每隔固定的时间生产一个物品，因此如果它运行的总时间为 T，则它的贡献大致为 T 除以其处理时间，向下舍入到整个物品。 

除了这些机器之外，工厂还必须生产一定数量的产品。 所有机器并行运行，其输出累积。 任务是确定所需的最短时间，以使所有机器生产的物品总数至少达到所需的目标。 

输入包括机器数量、生产要求和机器速度列表。 每个速度表示机器生产单个产品所需的时间。 输出是一个整数，表示满足生产要求的最短时间。 

约束结构通常意味着最多大约 10^5 台机器和较大的目标值，通常高达 10^18 或类似的数量级。 这立即排除了任何逐步推进时间的模拟，因为即使在时间单位上迭代也将远远超出可行的限制。 在最坏的情况下，即使每个单元的模拟也会爆炸到 10^18 次操作，而且即使每个事件的模拟仍然太慢，因为所有机器上的事件密集地交错。 

一些边缘情况往往会打破天真的推理。 如果许多慢速机器中有一台非常快的机器，那么任何贪婪地分配每台机器工作而不随着时间的推移聚合工作的方法都会低估贡献。 例如，如果机器时间为 [1, 100, 100] 并且目标为 5，则正确答案为 5，因为第一台机器单独完成了所有操作。 错误的贪婪分布试图按顺序“分配项目”给机器而不建模并行时间，这将错误地分散负载。 

当所有机器都很慢但目标很小时，就会出现另一种微妙的情况。 例如，目标为 1 的机器 [10, 10, 10] 应返回 10。任何错误地尝试平均处理时间的方法都会错误地建议更小的时间。 

## 方法

 蛮力的想法是从零开始模拟时间并逐步递增。 在每个时间值，我们通过将经过的时间除以处理时间并对机器进行求和来计算每台机器生产了多少个物品。 一旦总数达到目标，我们就返回当前时间。 

这种方法是正确的，因为它直接反映了随着时间的推移生产的定义。 每台机器在时间 T 准确地贡献了 Floor(T / a_i) 项，因此检查每个时间步可以保证正确性。 

失败点是时间轴的增长。 如果机器时间和目标大小很大，则答案最多可达 10^18。 即使单次超过这个范围也是不可能的。 每一步的计算生产成本也为 O(n)，因此总复杂度变为 O(n·答案)，这是无法使用的。 

关键的观察是，时间 T 产生的项目总数在 T 中是单调的。如果我们在时间 T 有足够的项目，那么任何更大的时间也将满足要求。 这种单调结构使我们能够用二分搜索代替随时间推移的线性搜索。 

我们不是模拟每一个时刻，而是提出一个可行性问题：给定固定时间 T，所有机器可以一起生产至少 k 个物品吗？ 此检查速度很快，因为它只需要对所有机器求和下限(T / a_i)。 有了这个谓词，我们二分搜索满足它的最小 T。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力模拟| O(n·T) | O(n·T) | O(1) | O(1) | 太慢了|
 | 准时二分查找 | O(n log T) | O(n log T) | O(1) | O(1) | 已接受 |

 ## 算法演练

我们将问题简化为找到使生产函数达到目标的最小时间 T。 

1. 定义一个函数，通过对所有机器上的 T // a_i 求和来计算所有机器在给定时间 T 内生产的物品数量。 该函数对固定时刻的整个工厂输出进行建模。 
2. 选择时间搜索范围。 下限为零，上限可以安全地是最慢的可能场景，其中一台机器单独生产所有项目，通常设置为 max(a_i) * k。 
3. 在此时间范围内进行二分查找。 在每个中点 T，使用步骤 1 中的函数计算总产量。 
4. 如果 T 处的生产至少是目标，则将 T 记录为候选答案并向左移动搜索，因为我们试图最小化时间。 
5. 否则，由于T太小，无法满足生产要求，因此向右移动搜索。 
6. 继续直到搜索空间耗尽，返回最小的可行 T。 

关键的设计选择是每次都从头开始重新计算检查函数。 这是可以接受的，因为计算与机器数量呈线性关系，而搜索深度在答案范围内呈对数关系。 

### 为什么它有效

 正确性依赖于生产函数的单调性。 对于任何固定的机器速度集，增加 T 只能增加或维持生产的产品数量，而不能减少它。 这保证了可行性条件在数轴上形成一个连续的区间：低于某个阈值的所有时间都是无效的，高于该阈值的所有时间都是有效的。 因此，二分搜索可以保证收敛到最小有效时间，而不会跳过真正的边界。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def can(machines, t, k):
    total = 0
    for a in machines:
        total += t // a
        if total >= k:
            return True
    return False

def solve():
    n, k = map(int, input().split())
    machines = list(map(int, input().split()))

    lo, hi = 0, min(machines) * k
    ans = hi

    while lo <= hi:
        mid = (lo + hi) // 2
        if can(machines, mid, k):
            ans = mid
            hi = mid - 1
        else:
            lo = mid + 1

    print(ans)

if __name__ == "__main__":
    solve()
```解决方案的核心是`can`函数，评估候选时间是否充足。 它使用整数除法对每台机器的贡献求和，直接编码每台机器按时间完成的完整项目数`t`。 

二分搜索保持了可能答案的缩小窗口。 每当中点可行时，它就会被存储，并继续向左搜索以找到更小的有效时间。 上限选择为`min(machines) * k`因为即使是最快的机器也能在这段时间内生产出所有物品，这使其成为安全的最坏情况界限。 

必须小心具有固定整数类型的语言中的溢出，因为`t`和`k`可以很大。 在 Python 中，这是自然处理的。

 二分查找的范围缩小到 8，这是产生至少 7 个项目的最短时间。 这表明可行性如何急剧地从错误转变为正确。

| T | T//10 | T//10 | 总计 | 可行|

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 | O(n log(max(a_i) · k)) | O(n log(max(a_i) · k)) | 每个二分查找步骤都会扫描所有机器，步数在搜索范围内呈对数 |

 ## 边缘情况

 当只有一台机器时，二分查找会分解为寻找达到目标的处理时间的最小倍数。 用于输入`1 5`带机器`[2]`，可行性检查从时间 1 时的 0 个项目增加到时间 10 时的 5 个项目，并且算法正确地收敛到 10，因为只有完整的生产周期才重要。 

当所有机器都相同时，解决方案完全取决于计数乘法而不是分布。 为了`[5, 5, 5]`目标 3、时间 5 已经产生了 3 个项目，任何按顺序为每台机器分配任务的幼稚尝试都会错误地假设需要多个周期，而正确的模型会在一个步骤中聚合贡献。 

当目标非常大时，例如接近 10^18，只有二分搜索仍然可行。 该算法从不随时间显式地迭代，并且每一步对于机器来说都是有界总和，即使在极端规模下也能确保正确性。
