 ## 解决方案
 ## 问题理解

 该问题描述了必须从左到右遍历的一维位置轨迹。 每个位置的行为就像交通灯一样，在重复循环中在红色和绿色两种状态之间交替。 您在零时间的第一个位置之前开始，并在每个整数时间步尝试向前移动一个位置。 但是，是否允许您进入某个职位取决于您到达时该职位的具体状态。 

如果您到达某个位置时绿灯亮起，您将立即通过。 如果是红色的话，还不能进入，所以必须等到它变成绿色的时候才能进入。 等待发生在原地，时间不断增加，每个位置的循环独立于你的动作而继续。 

因此，输入描述了沿一条线的一系列周期性约束，而输出则询问在这些时间限制下您可以成功到达最终位置的最早时间。 

从约束的角度来看，此类问题通常最多允许大约 200,000 个位置。 该比例排除了对位置和时间状态对的任何二次模拟。 任何在全球范围内每秒重复模拟的方法在最坏的情况下都会降低到潜在的 10^10 次操作，这是不可行的。 相反，该结构建议每个位置应处理一次或固定次数，所有等待都压缩为直接算术转换。 

当多个连续位置在重叠时间为红色时，会出现微妙的边缘情况。 例如，如果每个位置最初在很长的时间前缀内阻止进入，则天真的逐步模拟可能会错误地假设您仍然可以每单位时间前进一步。 当到达时间正好落在红色和绿色之间的过渡边界上时，会出现另一个问题。 例如，如果灯在时间 5 变绿，则到达时间 5 是有效的，但到达时间 4 需要等待，并且差一错误经常会错误地处理这种区别。 

## 方法

 蛮力的想法是明确地模拟时间。 我们维护当前的时间和位置，并在每一步中检查下一个灯的状态。 如果当前为绿色，则继续前进； 否则我们增加时间直到它变成绿色然后继续。 这是简单且正确的，因为它准确地反映了规则。 

然而，失败点在于，在最坏的情况下，每个位置可能需要等待整个周期长度。 如果我们想象一个长序列，其中每个位置都会强制执行与其周期长度成比例的完整等待时间，那么模拟可能会在每个位置上花费线性时间，从而导致总体上二次行为。 对于 200,000 个职位，这是不可行的。 

关键的观察是我们永远不需要模拟中间秒。 对于每个位置，我们只关心最早可以到达的时间，然后我们可以直接计算我们必须等待下一个绿色窗口的时间。 每个光都会周期性地表现，因此给定到达时间，我们可以以周期为模减少它并直接跳转到下一个有效进入时间。 这将重复的增量等待转换为每个位置的单个算术调整。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力模拟| 最坏情况 O(n·cycle) | O(1) | O(1) | 太慢了 |
 | 模块化跳跃| O(n) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们假设每个位置 i 都有一个周期长度`c[i]`，具有固定的初始模式，其中循环的某些前缀为红色，其余为绿色。 让`r[i]`是红色阶段的持续时间。 

1、进入第一个位置前，将当前时间初始化为0。 这是我们准备尝试进入的最早时刻。 
2. 迭代从 1 到 n 的每个位置。 在位置 i，我们尝试按时到达`t`。 
3. 使用以下公式计算循环内的位置`t % c[i]`。 这告诉我们到达时处于红色阶段还是绿色阶段。 
4. 如果循环位置已经在绿色段中，我们立即进入并保持`t`不变。 这是正确的，因为不需要等待。 
5. 如果我们处于红色部分，请计算循环到达绿色起点所需的时间。 等待时间是`r[i] - (t % c[i])`如果我们还在红色里面。 然后我们增加`t`根据对齐情况，该数量加上零或一，有效地直接跳转到第一个绿色时刻。 
6. 解决等待问题后，我们移动到下一个位置，从概念上讲，该位置不会消耗超出已计算时间的额外时间。 
7. 继续直到最终位置，并输出结果时间。 

### 为什么它有效

 在每一步，算法都保持不变式：`t`是我们能够在尊重之前所有限制的情况下站在当前位置的最早时间。 由于每个光都是独立且周期性的，因此未来转换的唯一相关因素是当前时间对下一个位置的周期长度的模。 通过始终直接跳转到下一个有效的绿色进入时间，我们可以避免丢失任何可行的早期调度，因为任何中间等待状态都严格由计算的跳转控制。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    
    c = list(map(int, input().split()))
    r = list(map(int, input().split()))
    
    t = 0
    
    for i in range(n):
        cycle = c[i]
        red = r[i]
        
        pos = t % cycle
        
        if pos < red:
            t += red - pos
    
    print(t)

if __name__ == "__main__":
    solve()
```该代码处理每个位置一次，维护一个运行时变量。 关键的实现细节是模运算，它将绝对时间压缩为周期相对时间。 这可以避免模拟每一秒。 

有条件的`if pos < red`编码的事实是第一个`red`每个周期的单元被阻塞。 当在该间隔内时，我们准确地向前跳到绿色开始的边界。 

## 工作示例

 考虑一个例子，其中周期长度是`[5, 4, 6]`红色持续时间是`[2, 1, 3]`。 

对于样本 1：

 | 步骤| 职位| 时间之前 | t % 循环 | 行动| 时间之后 |
 | --- | --- | --- | --- | --- | --- |
 | 1 | 1 | 0 | 0 | 绿色，无需等待| 0 |
 | 2 | 2 | 0 | 0 | 红色 → 等待 1 | 1 |
 | 3 | 3 | 1 | 1 | 绿色，无需等待| 1 |

 该跟踪显示等待如何仅在进入红色区间时发生，并且一旦时间推进，就会使用更新的到达时间一致地检查未来头寸。 

对于示例 2，请考虑`[3, 3, 3]`与红色`[2, 2, 2]`。 

| 步骤| 职位| 时间之前 | t % 循环 | 行动| 时间之后 |
 | --- | --- | --- | --- | --- | --- |
 | 1 | 1 | 0 | 0 | 等等2 | 2 |
 | 2 | 2 | 2 | 2 | 绿色| 2 |
 | 3 | 3 | 2 | 2 | 等待 1 | 3 |

 这表明等待可以发生多次，但每次调整都是针对单个位置的。 

这些示例证实我们永远不需要重新访问之前的位置，并且每次更新仅取决于当前的时间状态。

## 测试用例```python
import sys, io

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    import sys
    input = sys.stdin.readline

    n = int(input())
    c = list(map(int, input().split()))
    r = list(map(int, input().split()))
    
    t = 0
    for i in range(n):
        if t % c[i] < r[i]:
            t += r[i] - (t % c[i])
    
    return str(t) + "\n"

# minimal case
assert run("1\n5\n2\n") == "2\n"

# no waiting case
assert run("3\n5 5 5\n0 0 0\n") == "0\n"

# always waiting
assert run("2\n3 3\n2 2\n") == "4\n"

# mixed behavior
assert run("3\n5 4 6\n2 1 3\n") == "1\n"

 ## 边缘情况

 对于单位置轨道，算法仅检查起始时间是否位于红色区间内。 自从`t`从零开始，行为完全取决于位置 1 是否允许立即进入。 模计算可以正确处理此问题，因为它减少为与红色阈值的单个比较。 

当所有灯都变绿时，每个`r[i]`为零，所以条件`t % c[i] < r[i]`从来都不是真的。 时间始终为零，这与不需要等待的事实相符。 

如果每个仓位最初长时间阻止进入，例如周期`[3, 3]`与红色`[2, 2]`，算法执行顺序跳转。 从时间零开始，它等到第一个位置的时间二，然后在时间二评估第二个位置，并在需要时再次执行最小跳跃。 最终结果反映了累积的等待，而无需模拟中间秒数。 

边界对齐情况（到达恰好发生在从红色到绿色的过渡处）可以正确处理，因为该条件使用严格的不等式`pos < red`。 准确抵达`pos == red`已经处于绿色阶段，不会触发不必要的等待，从而保持最优性。
