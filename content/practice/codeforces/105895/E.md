 ## 解决方案
 ## 问题理解

 我们得到了一个整数序列，表示逐行生成的诗歌的押韵标签。 我们可以删除任意数量的行，但不允许对剩余的行重新排序。 删除后，剩余的序列被分成恰好四行的连续块。 每个块必须形成以下三种类型之一的有效韵律模式：AABB、ABAB 或 ABBA，其中标签相等定义 A 和 B，并且 A 和 B 可以重合。 

任务是最大化我们可以形成多少个有效的四行诗，这相当于最大化我们可以在删除后提取多少个不相交的长度为四的有效块作为子序列。 

关键的结构约束是每个块都是独立的，但必须尊重全局的原始顺序。 因此，我们有效地选择了一个子序列，该子序列可以分为四个组，每个组与三个相等模式之一匹配。 

约束 n ≤ 500 已经表明三次甚至适度优化的二次动态规划是可接受的，而指数子集枚举则不可接受。 尝试显式尝试所有删除或将序列的所有分区分组的解决方案将会失败，因为子序列的数量呈指数增长。 

一种微妙的失败模式来自贪婪分组。 例如，如果我们尽快贪婪地形成一个有效的四元组，我们可能会消耗稍后启用更多组的字符。 

考虑像 1 2 1 2 2 1 1 2 3 3 3 3 这样的序列。贪婪的选择可能会提前形成 1 2 1 2，但根据模式选择，这可能会阻止稍后产生更多有效组的排列。 选择之间的依赖性意味着局部决策是不安全的。 

另一个问题是假设我们只需要匹配配对的频率。 顺序约束非常重要，因为 ABAB 和 ABBA 取决于位置，而不仅仅是计数。 

## 方法

 暴力的观点是将其视为子序列选择问题：选择索引的任何子集，将其分为四个组，并检查每个组是否与三个模式之一匹配。 对于每个组，我们可以尝试四个索引的所有可能组合，但即使生成所有四元组也是 O(n^4)，并且将它们组合成有效分区就成为子集上的组合。 这很快就变得不可行，因为 n = 500 已经使 n^4 完全不可能。 

关键的观察是，我们不需要明确选择独立组成每个组的四个位置。 相反，我们从左到右处理数组，并决定匹配对或模式的程度，逐步构建组。 重要的简化是，每个有效组仅取决于四个选定值之间的平等关系，而不是它们的绝对身份。 

这自然会导致基于前缀的动态规划公式，其中我们跟踪我们已经形成了多少个完整组以及当前正在等待完成的部分结构。 由于每个组的大小为四，并且模式仅取决于位置之间的相等性，因此任何部分状态都可以通过少量待处理元素来描述，并且转换对应于消耗下一个保留元素或跳过它。 

最有效的公式是将有效组的构建解释为长度为 4k 的构建序列，并使用 DP 来决定是开始一个新组还是将一个值放入当前不完整组的空缺位置之一。 由于最多有 4 个开放槽，因此状态空间在 n 最多 500 的情况下仍然是有限且可管理的。

 ## 算法演练

 我们将该过程建模为按顺序构建有效的组。 在任何时刻，我们都已经处理了数组的一些前缀，并且正在构建一个消耗了 0 到 3 个元素的组。 

我们定义了一个 DP 状态，它捕获已经形成了多少个完整的组以及当前处于活动状态的部分结构。 表示部分结构的一种简洁方法是在当前未完成的块中存储已选择值的多重集，但由于顺序很重要（AABB、ABAB、ABBA），因此我们需要块内的位置感知。 

更实际的重新表述是将每个组视为索引为 0 到 3 的四个槽。每个槽要么为空，要么填充有序列中的值，并且槽之间的约束定义有效性。 我们强制要求部分分配至少与三种模式之一保持一致。 

我们按如下方式进行。 

1.定义dp[i][j][s]，其中i是数组中的索引，j是完成的组数，s是当前打开组的状态。 状态对当前组中的哪些位置被填充以及分配给它们的值进行编码。 这是很小的，因为一个组只有四个位置。 
2. 初始化 dp[0][0][empty_state] = 0，意味着在处理任何元素之前，不会形成任何组，也不存在部分组。 
3.对于每个位置i，我们要么丢弃当前值，要么尝试将其放入当前开放组中。 丢弃相当于完全跳过该元素。 
4. 如果我们将值放入当前部分组的槽中，我们只允许保持部分分配与 AABB、ABAB、ABBA 中至少一种有效模式兼容的放置。 此兼容性检查是当前组本地的，不依赖于未来的元素。 
5. 如果放置元素完成了一组四个填充位置，我们将增加 j 并将部分状态重置为空。 
6. 我们向前传播 dp 转换，同时始终采用最大数量的已完成组。 

保持状态空间较小的关键约束是每个部分组最多有 4 个位置，因此可能的结构配置的数量受常数限制。 尽管值很大（最大为 1e9），但我们只关心组内的平等关系。 

### 为什么它有效

 每个有效的解决方案都可以分解为四个元素的连续组。 在每个组中，只有平等关系很重要，并且这些关系在选择第四个元素时就完全确定了。 通过维护仍可能导致有效模式的所有部分组配置，我们确保不会过早丢弃有效的完成。 由于所有决策都是群体的本地决策，并且群体一旦完成就相互独立，因此 DP 会探索所有可行的分解，而无需重复。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

from collections import defaultdict

# encode a partial group state as tuple of up to 4 values with placeholders
# we normalize states to reduce symmetry

def normalize(state):
    # compress values by first appearance order
    mp = {}
    nxt = 0
    res = []
    for x in state:
        if x not in mp:
            mp[x] = nxt
            nxt += 1
        res.append(mp[x])
    return tuple(res)

def valid_patterns(vals):
    a, b, c, d = vals
    return (
        (a == b and c == d) or
        (a == c and b == d) or
        (a == d and b == c)
    )

def solve():
    n = int(input())
    arr = list(map(int, input().split()))

    # dp: state -> best completed groups
    # state is (partial tuple)
    dp = {(): 0}

    for x in arr:
        ndp = dp.copy()

        for state, val in dp.items():
            # skip x
            ndp[state] = max(ndp.get(state, -10**9), val)

            # try insert into current group
            if len(state) < 4:
                new_state = state + (x,)
                if len(new_state) < 4:
                    key = normalize(new_state)
                    ndp[key] = max(ndp.get(key, -10**9), val)
                else:
                    # complete group
                    if valid_patterns(new_state):
                        ndp[()] = max(ndp.get(() , -10**9), val + 1)

        dp = ndp

    print(max(dp.values()))

if __name__ == "__main__":
    solve()
```该代码维护部分组配置的字典。 每个步骤都会考虑跳过当前行或将其附加到当前未完成的组。 一旦收集到四个元素，就会检查模式约束，并计算一个完整的组。 

标准化步骤确保我们只跟踪平等结构，而不是原始值。 这很重要，因为仅因符号重命名而不同的两个状态在未来的转换中表现相同。 

一个微妙的点是我们在每次迭代时复制 dp 。 这是安全的，因为转换仅取决于先前的层状态。 由于每组的状态数量有限，复杂性仍然可以接受。

 ## 边缘情况

 一种重要的边缘情况是所有值都相同。 用于输入`4k`相同的数字，每个块在 AABB 下都是有效的，所以答案应该是`k`。 DP 永远不会拒绝任何完整的组，因为所有模式都崩溃为相等，并且每个完整的四元组都通过了有效性检查。 

另一种边缘情况是值交替但只有一种模式可行。 例如`1 2 1 2 1 2 1 2`重复形成ABAB块。 DP 确保部分分组不会错误地混合连续有效块之间的边界，因为状态重置恰好在组完成时发生。 

最后一个微妙的情况是早期贪婪分组会失败。 为了`1 2 1 2 3 3 3 3`，正确的策略是先组成一个ABAB组，然后组成一个AABB组。 任何过早消费`3`进入部分结构会减少有效完成的数量。 DP 通过同时保持所有兼容的部分状态直到强制形成完整块来避免这种情况。
