 ## 解决方案
 ## 问题理解

 我们得到一个嘈杂的黑白图像，表示为$n \times m$字符网格。 每个单元格要么是 0，要么是 1，其中 1 对应于白色像素，0 对应于黑色像素。 该图像最初是由以固定字体编写的干净二进制字符串生成的，然后被随机像素翻转损坏。 

重要的隐藏结构是网格不是任意的。 它是一个渲染的 OCR 条带：每一列对应于相同底层数字或字符模式的垂直切片，原始干净图像包含编码二进制字符串的水平结构。 噪声以低概率翻转单个像素，因此大多数列仍然类似于其真实图案。 

任务是恢复生成图像的原始二进制字符串。 

约束的高度较小，但宽度较大。 高度约为 64 到 66，宽度可达 1000。这立即表明每一列可以独立处理或聚合成低维表示。 任何试图将图像建模为一般 2D 模式匹配问题或对所有像素运行昂贵的全局优化的解决方案仍然会通过，但前提是它能有效地简化为每列分类。 

天真的读者可能会尝试将图像解释为任意噪声，并尝试使用整个网格的模式匹配来重建字符串。 当考虑完整图像的组合时，这种方法会失败，因为状态空间巨大。 

一个更微妙的失败案例来自于独立处理每一行。 由于每个像素的噪声是独立的，因此按行多数投票似乎是合理的，但行并不独立地对字符进行编码； 字符是垂直一致的图案。 忽略列结构会破坏信号。 

另一种不正确的方法是全局阈值整个图像，例如通过计算每列的总数并与固定阈值进行比较，例如$n/2$。 这在预期中有效，但当字符具有结构不对称时会失败，因为即使在干净的数据中，某些列本质上也更白或更黑。 

关键是每一列都是固定 64 维二进制向量的噪声观测，并且属于同一字符区域的所有列的基础信号是一致的。 

## 方法

 暴力的观点是考虑每个可能的二进制字符串并检查它是否可以在翻转模型下生成观察到的噪声图像。 这相当于搜索所有可能的字符串并计算整个网格的可能性。 即使限制合理的字符串长度，输出字符串的长度也会呈指数级爆炸，并且完全不可行。 

第二个残酷的想法是尝试模板匹配：对于每个可能的字符位置，尝试对齐候选模式并对所有行和列的像素相似性进行评分。 这仍然很昂贵，因为它需要重复扫描大型子矩阵并与许多假设进行比较。 

转折点来自于认识到数据是如何生成的。 每列都是单个字符切片的垂直编码，每个像素的噪声都是独立的。 这意味着每个输出符号的最佳决策仅取决于该列位置的所有行的聚合证据。 

一旦发现这一点，问题就会分解为按列去噪。 对于每一列，我们计算它包含多少白色像素和黑色像素。 由于噪声以很小的概率翻转每个像素，因此列中的多数值极有可能与该位置处的原始干净列结构相匹配。 干净的图像本身在对应于相同字符区域的列之间具有一致的结构，因此重建的输出简化为确定每一列是否属于字体编码的“白色主导”或“黑色主导”区域。 在这个问题中，它直接映射回二进制输出字符。 

因此，解决方案变成了对每列的行进行简单聚合，然后进行阈值处理。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力（全局重建）| 输出长度呈指数增长 | 高| 太慢了|
 | 列多数聚合|$O(nm)$|$O(1)$额外（除了输入）| 已接受 |

 ## 算法演练

 1.读取尺寸网格$n \times m$，将其存储为字符行。 这是必要的，因为我们稍后需要按列访问。 
2.对于每个列索引$j$，计算所有行中 1 的数量。 这将该垂直切片的所有噪声观测值聚合成单个统计数据。 
3. 比较该列中 1 的数量与 0 的数量。 如果更频繁，则将该列视为代表输出中的 1； 否则将其视为 0。基本原理是噪声是对称的且每个像素独立，因此多数投票是真实基础列值的最大似然估计。 
4. 按从左到右的顺序将每列的结果位附加到最终输出字符串中。 
5. 打印重构的二进制字符串。 

### 为什么它有效

 每列都可以看作$n$对隐藏真实值的独立伯努利观测值，该值以概率翻转$p = 0.05$。 观察到的列的期望值偏向于真实位，并且通过测量的集中，噪声压倒大多数的概率在$n$。 自从$n \approx 64$，错误多数决定的概率可以忽略不计，并且在最多 1000 列中，重建保持稳定。 这使得按列多数成为原始信号的一致估计器。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

n, m = map(int, input().split())
grid = [input().strip() for _ in range(n)]

res = []

for j in range(m):
    ones = 0
    for i in range(n):
        if grid[i][j] == '1':
            ones += 1
    zeros = n - ones
    res.append('1' if ones > zeros else '0')

print("".join(res))
```实现直接遵循列聚合思想。 网格按行存储，因为输入以该格式到达，并且我们逐列迭代以避免重复扫描子字符串。 

唯一微妙的决定是严格比较`ones > zeros`。 这很重要，因为关系应该始终映射到 0，尽管$n \in [64, 66]$，从统计上看，平局很少见，但如果噪音完全平衡，仍然有可能。 确定性的平局决胜保证了稳定的输出。

## 测试用例```python
import sys, io

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    n, m = map(int, input().split())
    grid = [input().strip() for _ in range(n)]
    res = []
    for j in range(m):
        ones = 0
        for i in range(n):
            if grid[i][j] == '1':
                ones += 1
        res.append('1' if ones > n - ones else '0')
    return "".join(res)

# minimal case
assert run("1 1\n1\n") == "1"

# clean columns

# majority with noise
assert run("3 3\n110\n101\n000\n") == "100"

 ## 边缘情况

 当列在 0 和 1 之间完美平衡时，就会出现微妙的边缘情况。 例如：```
2 3
10
01
```按列计数在每一列中完全相关。 该算法始终将联系映射为 0。 逐步：

 第 1 列有一个 1 和一个 0，因此 1 不大于零，输出为 0。第 2 列的行为相同，第 3 列也是如此。输出变为`000`。 

尽管在规定的噪声模型下这种情况不太可能发生，但确定性的平局打破可以防止实现之间的未定义行为或不一致的输出。 

另一个边缘情况是最大宽度。 和$m = 1000$，该算法仍然只执行 1000 列扫描，每次最多 66 行，保持执行稳定，没有任何内存压力或缓存问题。
