 ## 解决方案
 ## Problem Understanding

 我们得到了固定的请求序列，以及最多大小的小型“缓存”$m$，最初为空。 当我们从左到右扫描序列时，我们遇到的每个元素要么已经在缓存中，要么不在缓存中。 

如果当前元素已在缓存中，我们将获得一分。 如果它不在缓存中，则在此简化版本中不会发生任何不良情况，但我们可以选择将其插入缓存中，可能会逐出一个现有元素。 在这个简单的版本中，插入和逐出不需要任何成本，因此获得分数的唯一方法是安排缓存，以便将来的元素在出现时就已经存在。 

任务是在扫描期间在线选择所有缓存更新，以便使缓存命中总数最大化。 

限制总体上很小，总共$n$所有测试用例最多 800 个。这会立即排除任何比粗略更差的解决方案$O(n^2)$，但仍然允许检查每个步骤的缓存并模拟未来行为的策略。 

一个微妙的一点是，尽管插入总是免费的，但粗心的策略仍然可能会严重失败。 例如，始终保留最近看到的元素并不是最佳选择。 

考虑一个像这样的序列$1, 2, 3, 1, 2, 3$和$m = 2$。 如果我们总是保留最新的两个元素，我们最终可能会得到$\{2, 3\}$在最后三个元素之前，错过了未来的热门$1$。 相反，最佳策略会跟踪哪些元素很快就会需要。 

关键的困难在于，错过的决策不是关于当前元素，而是关于保留对稍后将再次出现的元素的访问。 

## 方法

 一个蛮力的想法是在每一步之后模拟所有可能的缓存内容。 在每个位置，当发生未命中时，我们会尝试所有可能的选择来逐出哪个元素以及插入哪个元素。 这形成了一个分支过程，其中每个状态最多是大小的子集$m$，并且转换取决于下一个元素。 即使进行了积极的修剪，状态的数量也会以组合方式增长：$\binom{n}{m}$，这远远超出了可行的范围$n = 800$。 

该结构得以简化，因为所有命中都有相同的奖励并且没有惩罚。 唯一有意义的决定是将哪些元素保留在缓存中以最大化未来的命中率。 这将问题变成了具有完整未来知识的经典最优缓存问题。 

众所周知的最优策略是总是驱逐下一次出现距离最远的元素。 如果一个元素不再出现，它始终是被驱逐的最佳候选者。 这种贪婪规则是最优的，因为它推迟了失去对很快需要的元素的访问，从而保留了近期的收益。 

要应用此功能，我们需要在每个步骤中知道每个缓存元素下一次出现的时间。 我们为每个值维护其未来位置的列表，并在向前扫描时前进一个指针。

 ## 算法演练

 我们维护一个缓存集，并为每个值维护一个指向其未来下一次出现的指针。 

1. 为每个值预先计算其出现的所有索引的列表。 这让我们能够快速了解​​未来发生的情况。 
2. 对于每个值，维护一个指向其列表的指针，指示严格在当前位置之后第一次出现的值。 当我们在数组中向前移动时，当我们传递事件时，我们会向前移动这些指针。 
3. 维护当前缓存最多为一组$m$价值观。 
4、加工位置时$i$，检查是否$a_i$已经在缓存中了。 如果是，请将分数加一，因为这是缓存命中。 
5. 如果$a_i$缓存中没有，我们考虑插入它。 如果缓存有空闲空间，我们就直接插入。 否则，我们必须驱逐一个现有元素。 
6. 为了选择要逐出哪个元素，我们检查当前缓存中的每个元素并计算其下​​一个出现的索引。 要删除的最佳候选者是下一次出现的时间最远的候选者，或者根本不存在的候选者。 
7. 逐出该元素后，插入$a_i$进入缓存。 

这种方法有效的关键原因是，下次使用的时间很长的元素对于保留未来的命中来说最没有直接价值。 删除它不会损害近期结果，同时保留早期重用的元素可以保留有保证的收益。 

### 为什么它有效

 在任何一步，缓存都代表一个选择$m$我们承诺为了未来的利益而保留的元素。 假设我们考虑替换一个元素$x$在缓存中包含当前元素$a_i$。 如果$x$将来比其他缓存元素更早使用$y$，交换它们只能通过保留来延迟或更安全地保留未来的点击$x$。 这建立了一个贪婪的交换论点：任何违反“最远的下一次使用被驱逐”规则的配置都可以通过交换来改进，而不会减少未来的命中。 重复这种交换会在每一步产生最佳结构。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    INF = 10**18
    
    for _ in range(t):
        n, m = map(int, input().split())
        a = list(map(int, input().split()))
        _ = list(map(int, input().split()))  # x_i all 1
        _ = list(map(int, input().split()))  # y_i all 0
        _ = list(map(int, input().split()))  # z_i all 0

        pos = [[] for _ in range(n + 1)]
        for i, v in enumerate(a):
            pos[v].append(i)

        ptr = [0] * (n + 1)

        def next_occ(v, i):
            lst = pos[v]
            while ptr[v] < len(lst) and lst[ptr[v]] <= i:
                ptr[v] += 1
            if ptr[v] < len(lst):
                return lst[ptr[v]]
            return INF

        cache = set()
        score = 0

        for i, v in enumerate(a):
            if v in cache:
                score += 1
            else:
                if len(cache) < m:
                    cache.add(v)
                else:
                    worst = None
                    worst_next = -1
                    for u in cache:
                        nxt = next_occ(u, i)
                        if nxt > worst_next:
                            worst_next = nxt
                            worst = u
                    cache.remove(worst)
                    cache.add(v)

        print(score)

if __name__ == "__main__":
    solve()
```该解决方案的组织方式是在每个步骤中维护每个缓存元素的未来的正确视图。 这`pos`列表存储所有出现的情况，并且`ptr`延迟前进，以便计算下一个事件的效率得以摊销。 缓存本身是一个集合，每次未命中都会触发对缓存的扫描，以识别未来有用性最不紧急的元素。 

关键的实现细节是`next_occ(v, i)`始终严格返回索引后的第一个匹配项`i`，并且指针前进确保我们不会重复重新扫描旧位置。

 ## 边缘情况

 一种边缘情况是某个值在插入后不再出现。 在这种情况下，它的下一次出现将被视为无穷大，这使得只要它与任何可重用元素一起出现在缓存中，它就成为最佳逐出候选者。 算法自然会先将其删除，防止浪费缓存空间。 

另一种情况是缓存中的所有值具有相同的下一个出现位置。 在这种情况下，任何驱逐选择都是等效的，并且算法仍然会正确运行，因为替换其中任何一个都不会改变未来的命中潜力。 

最后一种情况是当前元素已经在缓存中，但很快就会再次出现。 指针机制确保我们仍然立即计算命中，并且将来发生的事件将被正确识别，而无需在该步骤进行任何缓存修改。
