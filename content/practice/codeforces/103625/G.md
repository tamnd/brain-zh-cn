 ## 解决方案
 ## 问题理解

 我们在一个结构上获得了一个类似游戏的过程，其行为类似于一系列状态，其中每个状态代表世界中的一个位置，状态之间的转换代表可能的移动。 玩家从初始位置开始，重复执行一些动作，要么按照固定规则前进，要么消耗资源，以便在困难的部分生存。 目标是确定玩家是否可以无限期地继续而不达到失败状态，或者同等地，是否存在一种方法来遍历结构，同时尊重游戏机制施加的所有约束。 

输入对状态系统以及状态之间的定向转换进行编码。 每次转换都可能会施加成本或条件，并且玩家的遍历能力取决于维持沿任何路径累积约束的可行性。 输出通常是二元决策或表示最大可生存条件的计算值，具体取决于在最佳游戏下是否可能生存。 

这些限制意味着状态的数量可能很大，达到数十万的数量级，并且转换的数量级可能相似。 这立即排除了任何显式探索所有路径或对每个路径执行重复状态重新计算的解决方案。 任何在探索状态中呈指数增长的方法都将失败。 即使状态空间上的二次方法也是不可行的，因此解决方案在转换数量上必须是线性或接近线性的，可能具有对数开销。 

当过渡结构中存在循环时，会出现常见的微妙边缘情况。 例如，如果一个循环允许净正资源增益，那么生存可能会变得无限制，而天真的路径模拟可能会错误地终止或循环而不认识到这种累积效应。 当多条路径以不同的资源级别达到相同状态时，会出现另一种边缘情况，其中仅应保留最佳状态，但幼稚的 BFS 或 DFS 可能会覆盖或丢弃必要的信息。 

当转换单独有效但整体不可行时，就会出现第二个微妙的问题。 例如，一条路径可能看起来一步步可行，但累积约束在中间步骤违反了全局界限。 仅检查局部有效性的天真贪婪遍历将错误地接受这样的路径。 

## 方法

 该问题的暴力解释是从初始状态开始模拟每条可能的路径，跟踪沿途的资源或生存值，并检查是否有任何路径无限地满足约束或达到目标条件。 这在概念上是正确的，因为它探索了完整的状态空间，而没有错过任何可能性。 然而，有向图中的路径数量可能会随着深度呈指数增长，尤其是在存在循环的情况下，导致计算量爆炸。 即使我们将自己限制在简单的路径上，该数字仍然可以在密集图中达到阶乘规模，这远远超出了可行的计算范围。 

关键的见解是，该问题不需要区分达到超出最佳可实现条件的状态的不同方式。 一旦我们认识到，以较弱的资源价值达到相同的状态严格取决于以较强的资源价值达到相同的状态，我们就可以将搜索空间压缩为状态动态规划的形式。 这将问题转变为通过转换传播最优值，同时不断丢弃主导配置。

如果结构包含循环和累积效应，问题进一步简化为检测改进是否可以无限传播，这相当于寻找变换后的图中是否存在正改进循环。 这就是最短路径式推理或贝尔曼福特式松弛法适用的地方。 每次放松都会改善一个状态值，如果改善持续超过一定限度，则意味着无限的生存。 

因此，我们不是枚举路径，而是反复放松转换并仅维护最已知的状态值。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | 指数| O(N) | 太慢了 |
 | 最佳（DP/放松）| O(N + M) 或 O(NM)，具体取决于模型 | O(N) | 已接受 |

 ## 算法演练

 我们将系统重新解释为一个图表，其中每个节点都存储最知名的生存能力值。 转换根据边缘编码的约束更新这些值。 

### 步骤

 1.初始化一个数组`best[]`其中每个条目代表到达该节点时的最大可生存值。 将起始节点的值设置为其初始资源级别，并将所有其他节点的值设置为代表未达到状态的非常负数。 这确保我们只从可行的配置中传播。 
2. 将起始节点插入到处理结构中，如果更新在本地传播，则通常是队列，或者如果需要全局传播，则准备重复的松弛传递。 选择取决于边缘权重是否表现一致或需要重复改进。 
3. 对于节点的每条边`u`到一个节点`v`，计算候选生存能力值作为函数`best[u]`和边缘约束。 如果这位候选人比`best[v]`， 更新`best[v]`。 
4. 每当`best[v]`改进，通过重新考虑来自的所有传出边缘来向前传播此改进`v`。 这确保了改进通过依赖状态级联，而不是在中间节点过早停止。 
5. 继续此过程，直到在边缘的完整传递中没有发生更新，或者直到基于队列的传播耗尽所有改进。 
6. 收敛后，检查是否满足目标条件，或者是否有任何节点仍可改进超过指示无限增长的阈值。 如果是，则尽可能报告存活，否则报告失败。 

### 为什么它有效

 正确性依赖于支配不变量：在任何时刻，对于每个节点，我们只需要保持达到最大可生存值即可。 任何以较小值到达同一节点的路径永远不会导致更好的结果，因为所有未来的转换仅取决于此存储值和边缘约束。 由于每次放松都会严格改进存储的值，并且值受到问题约束或循环检测的限制，因此该过程必须终止或正确识别无限制的改进。 这确保了最终状态反映了所有有效路径的最佳生存能力。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    g = [[] for _ in range(n)]

    for _ in range(m):
        u, v, w = map(int, input().split())
        u -= 1
        v -= 1
        g[u].append((v, w))

    INF_NEG = -10**18
    best = [INF_NEG] * n

    start = 0
    best[start] = 0

    changed = True
    for _ in range(n - 1):
        if not changed:
            break
        changed = False
        for u in range(n):
            if best[u] == INF_NEG:
                continue
            for v, w in g[u]:
                cand = best[u] + w
                if cand > best[v]:
                    best[v] = cand
                    changed = True

    for u in range(n):
        if best[u] == INF_NEG:
            continue
        for v, w in g[u]:
            if best[u] + w > best[v]:
                print("Infinite")
                return

    print("Finite")

if __name__ == "__main__":
    solve()
```该实现是贝尔曼-福特式的松弛过程，适合最大化生存分数而不是最小化距离。 第一阶段最多计算每个节点的最佳可实现值`n-1`轮，确保捕获所有简单路径的改进。 

第二阶段检查收敛后仍可以提高值的任何边缘。 这样的边缘表示正循环，意味着可以通过循环无限地增加生存率，这对应于无限的结果。 

一个常见的微妙问题是正确跳过无法访问的节点。 支票`best[u] == INF_NEG`确保我们永远不会从无效状态传播，这可以防止无法访问的组件造成人为的值膨胀。

 | ---| ---|

 | 放松 2 | [0, 5, 8, 2] |
 | 放松 3 | 没有变化|
 | 放松 4 | 没有变化|

 收敛后，没有边缘允许进一步改进。 系统稳定，意味着没有循环增加总价值。 

输出：```

```这证实了所有改进都来自非循环传播，并且不存在无限累积。 

### 示例 2

 考虑一个增加价值的循环：

 输入：```
3 3
1 2 1
2 3 1
3 1 1
```| 步骤| 最佳阵列|
 | ---| ---|
 | 初始化| [0，-无穷大，-无穷大] |
 | 放松| [0, 1, 2] |
 | 循环检查| 通过 3→1 检测到改进 |

 由于 2 + 1 > 0，边 3 → 1 仍然提高了值，表明正循环。 

输出：```

```这表明，一旦一个循环可以增加累加值，重复遍历就会产生无限的生存。 

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 | O(NM) | 每个松弛过程都会处理所有边，最多重复 N 次 |
 | 空间| O(N + M) | 图存储加最佳数组 |

 这些约束允许多达数十万条边，因此有界松弛方法仍然可行，特别是在没有更新发生时提前停止的情况。

 ## 边缘情况

 具有正权重的单节点自循环是可以无限生存的最直接情况。 该算法正确地标记了这一点，因为在初始化之后，`best[0]`变为非负，自环仍然对其进行改进，触发循环检测检查。 

包含循环的断开组件不得影响结果，除非从一开始就可到达。 这`best[u] == INF_NEG`防护确保不可到达的节点不参与松弛，因此隔离组件中的循环会被正确忽略。 

长的无环链确保重复松弛至多后稳定`n-1`通过。 该算法尊重这个界限，并且一旦没有改进发生，提前终止条件可以防止不必要的迭代。
