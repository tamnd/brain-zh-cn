 ## 解决方案
 ## 问题理解

 我们给出了一个简单的线性规则来转换整数输入$x$转化为输出值$ax + b$。 任务是对连续值重复应用此规则$x$，从 1 开始一直到$k$，并按顺序打印所有结果输出。 

更具体地说，想象一台机器接受一个数字，将其乘以$a$，然后添加$b$。 我们向这台机器提供输入 1、2、3，一直到$k$，我们将每个结果记录为一个序列。 最终的输出就是这个转换值的列表。 

约束非常小，所有参数都以 100 为界。这意味着每项的任何合理计算都已经足够快，甚至最简单的模拟运行时间也可以忽略不计。 除了直接评估之外，无需进行优化。 

输入结构本身不存在棘手的隐藏边缘情况，但一个常见的错误是误读序列开始。 该序列必须开始于$x = 1$， 不是$x = 0$。 如果有人错误地从 0 开始，第一项就变成$b$，除非明确意图，否则无效。 

另一个潜在的错误是试图过于聪明并推导出封闭形式的序列，而没有意识到每一项都是独立的。 由于每个术语仅取决于其自身$x$，项之间不存在重复或交互。 

## 方法

 最直接的方法是独立计算每一项。 对于每个$x$从 1 到$k$，我们评估$ax + b$并立即打印结果。 这是可行的，因为表达式是线性的并且不依赖于先前的计算。 

强力解释可能建议构建一个值列表，然后打印它们，但即使这样在这种情况下也已经是最佳的。 操作总数为$k$，每个操作都是一个常数时间的算术计算。 

关键的观察结果是该问题没有状态依赖性。 每个输出完全由其索引决定$x$，因此存储中间结果或尝试增量更新没有任何好处。 任何超出简单循环的优化只会增加复杂性，而不会减少运行时间。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力（直接评估）| O(k) | O(1) | O(1) | 已接受 |
 | 最佳（同样的想法）| O(k) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们使用表达式的定义逐项计算每一项。 

1. 读取整数$a$,$b$， 和$k$。 这些定义了线性变换和我们必须生成的项数。 
2. 迭代$x$从 1 到$k$。 每个值$x$对应于序列中的一项。 
3. 对于每个$x$, 计算$ax + b$。 这是给定代数表达式的直接求值。 
4. 存储或立即打印计算值。 由于输出顺序很重要，因此我们保持增加$x$。 
5. 在一行中用空格分隔值。 

循环内的计算是恒定时间，因此循环本身完全定义了运行时间。 

### 为什么它有效

 输出中的每一项都是通过替换特定值来独立定义的$x$代入同一个公式。 项之间不存在依赖性，因此顺序或独立地评估它们会产生相同的结果。 该循环简单地枚举了域$x$并对每个元素应用相同的确定性函数。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

a, b, k = map(int, input().split())

res = []
for x in range(1, k + 1):
    res.append(str(a * x + b))

print(" ".join(res))
```该解决方案读取三个整数，然后通过迭代所有有效值来构造序列$x$。 每个计算值都会立即转换为字符串，以避免在连接过程中重复转换。 最终输出打印为单个空格分隔行。 

一个微妙的实现细节是使用`range(1, k + 1)`而不是从 0 开始。从 0 开始会错误地产生$b$作为第一项，它与所需的序列定义不匹配。 

使用列表和`" ".join(...)`确保输出格式一致并避免循环内重复打印开销。

 ## 边缘情况

 一种边缘情况是当$k = 1$。 在这种情况下，循环只运行一次并产生一个值$a \cdot 1 + b$。 该算法自然地处理这个问题，因为范围仅包括起点。 

另一个边缘情况是当$b = 0$。 该序列变成了一个纯算术级数，从$a$。 该算法仍然对每个索引应用相同的公式，无需特殊处理即可产生正确的缩放。 

最后的边缘情况是当两者$a$和$b$均处于最大值。 即便如此，Python 中的每次计算都在安全整数范围内，并且不会出现溢出或精度问题。
