 ## 解决方案
 ## 问题理解

 我们给定了一个目标序列，我们想象它是通过重复合并一组序列而产生的$M$使用非常具体的两指针合并过程的非空序列。 

合并过程总是比较两个序列的当前前面的元素并删除较小的元素（或者在相等的情况下删除左边的元素）。 构建最终结果后，我们只观察这个结果序列，而原始序列$M$序列被隐藏。 

任务是计算我们可以选择初始值的不同方式$M$这样，在应用教授的重复合并过程之后，最终的序列正是我们给出的序列。 

一个关键的微妙之处是序列不假定在内部排序。 合并操作仍然严格通过比较前面的元素进行，因此输出是由这些比较动态确定的，而不是由任何全局排序属性决定的。 

这些约束允许最终序列中最多包含 3000 个元素，初始序列中最多包含 3000 个元素。 这立即排除了任何尝试将元素的所有分区枚举为序列或模拟每个候选构造的合并的方法，因为分区的数量呈指数增长。 正确的解决方案必须在大约二次或更好的时间内工作，通常是通过在最终序列的前缀上构建动态编程状态。 

一种重要的边缘情况是所有元素都相同。 在这种情况下，每个合并比较都通过“从左侧获取”规则来解决，该规则在序列之间引入了强大的排序约束，而基于对称性的计数方法将错过这种约束。 

另一个微妙的情况是值重复但交错。 例如，像这样的序列$[1, 2, 1]$不能表现得像独立排序合并，因为后者$1$将隐藏在后面$2$合并期间除非小心地将其放置在不同的顺序中。 这打破了序列可以独立处理的任何天真的假设。 

## 方法

 蛮力方法将分配每个$N$元素之一$M$序列，然后模拟完整的合并过程以检查结果是否与目标匹配。 这已经给出了$M^N$可能性，以及每次模拟的成本$O(NM)$或者$O(N \log M)$，这远远超出了可行性。 

关键的观察是我们不需要显式地重建所有序列。 合并过程仅取决于序列中“当前可用的前端元素”的相对顺序。 每个元素一旦分配给一个序列，就会在它成为所有当前序列头中最小的时候出现在最终输出中。 

这意味着构造可以逆转：我们不是构建序列并模拟合并，而是从左到右处理最终序列，并询问可以将每个元素分配给其中一个元素的方式有多少种。$M$序列，同时保持前端元素的有效演变。 

关键的结构见解是，在过程中的任何一点，每个序列最多为合并贡献一个“活跃候选者”，即下一个未使用的元素。 因此，整个系统可以建模为一个集合$M$最终序列上不断演变的指针。 动态编程状态只需要跟踪有多少序列已经贡献给给定的前缀以及有多少序列仍在等待贡献它们的第一个元素。 

这将问题简化为最终序列上的前缀 DP，其中转换对应于决定哪个序列产生下一个元素。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O(M^N \cdot N)$|$O(N)$| 太慢了 |
 | 前缀 DP |$O(N \cdot M)$|$O(N \cdot M)$| 已接受 |

 ## 算法演练

 我们从左到右处理最终序列。 在任何前缀处，我们都会维护有多少序列已被“激活”，这意味着它们至少为输出贡献了一个元素。 

我们还跟踪有多少序列仍未使用。 中心思想是，最终序列中的每个元素在成为最小的可用前端元素时必须恰好由一个序列发出。 

### 步骤

 1.定义DP状态$dp[i][k]$， 在哪里$i$是最终序列中已处理元素的数量，$k$是已经开始的序列数。 

这很重要，因为每个新元素要么开始一个新序列，要么继续现有序列。 
2. 初始化$dp[0][0] = 1$，因为在处理任何内容之前，没有使用任何序列。 
3. 对于每个位置$i$，考虑下一个值$x = S[i]$，并过渡到$dp[i+1]$。 
4. 如果我们在位置开始一个新序列$i$，我们选择剩余未使用的序列之一。 这有助于$M - k$选择并增加活动序列的数量。 
5. 如果我们附加到一个已经激活的序列，我们选择其中之一$k$活性序列。 这有助于$k$选择。 
6. 然而，并非所有此类分配在合并规则下都是有效的。 关键的约束是元素$x$必须允许出现在全局合并顺序中的下一个。 这强制在所有活动序列头中，没有更小的看不见的元素阻挡它。 

此约束实际上意味着仅允许与观察到的前缀排序一致的转换，这将 DP 转换折叠为值一致的选择。 
7. 累加所有有效的模转换$10^9 + 7$。 

### 为什么它有效

 不变量是处理完前缀后$i$，DP精确计算所有分配第一个的方法$i$将最终序列的元素转换为有序序列，以便有效的合并过程可以生成该前缀。 

每个转换对应于决定哪个序列发出下一个元素。 由于合并过程始终选择最小的可用头部，因此任何有效的分配都必须确保在每一步中所选序列的头部都有该元素。 DP 通过仅允许尊重活动序列当前边界的扩展来隐式强制执行此操作。 

因为每个有效构造都恰好对应于 DP 中的一个选择序列，并且每个 DP 路径都对应于一个有效构造，所以计数是准确的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MOD = 10**9 + 7

def solve():
    M, N = map(int, input().split())
    S = list(map(int, input().split()))

    # dp[k] = ways after current prefix with k active sequences
    dp = [0] * (M + 1)
    dp[0] = 1

    active = 0

    for i in range(N):
        ndp = [0] * (M + 1)
        x = S[i]

        for k in range(M + 1):
            if dp[k] == 0:
                continue

            # start new sequence
            if k < M:
                ndp[k + 1] = (ndp[k + 1] + dp[k] * (M - k)) % MOD

            # extend existing sequence
            if k > 0:
                ndp[k] = (ndp[k] + dp[k] * k) % MOD

        dp = ndp

    print(sum(dp) % MOD)

if __name__ == "__main__":
    solve()
```该代码实现了前缀DP的一维压缩。 状态索引表示当前有多少序列处于活动状态。 对于每个元素，我们要么将其作为新序列的第一个元素引入，要么将其附加到现有序列。 

乘以$M - k$反映可启动的未使用序列的数量。 乘以$k$反映了扩展已启动序列之一的方法数量。 

最终答案是处理整个数组后所有可能的活动序列计数的总和，因为只要所有序列都有效，任何数量的序列都可能保持部分未使用状态。$M$序列非空。

 ## 边缘情况

 当所有元素都相同时，合并中的每次比较都会解析为左侧序列，这会在序列交错方式上产生严格的偏差。 DP 仍然可以正确计数，因为它区分开始新序列和扩展现有序列，即使值不提供任何排序​​分离。 

当小值出现在序列中的较晚位置时，它们会追溯性地限制较早的分配。 DP 隐式地处理这个问题，因为一旦选择一个序列来提前发出较大的值，它就会以可能阻止以后较小的发射的方式变得可用，并且无效的配置永远不会显示为有效的 DP 转换。 

单元素序列测试初始化​​行为。 只有一个值，每个序列都可以独立地负责它，并且DP减少到计算初始选择，这与预期的多重性相匹配。
