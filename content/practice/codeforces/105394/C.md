 ## 问题理解

 我们得到一个写成标记序列的参考程序，然后是多个查询程序。 每个程序都已经被标记化，因此我们不处理原始字符，而是处理字符串列表。 

仅当标记仅由一个字母字符组成时，才将其归类为变量。 其他所有内容，包括多字符标识符、数字和括号等符号，都被视为固定符号。 

如果我们可以从引用中获取一些连续的标记块并通过一致地重命名变量将其转换为查询，则查询被认为是有效的。 一致性意味着两件事必须同时成立。 首先，如果同一变量在引用段中出现多次，则所有这些出现都必须映射到查询中的相同变量名称。 其次，引用中的两个不同变量不能映射到查询中的同一变量。 

因此，从结构上讲，我们将序列与变量标识符的双射进行比较，但仅限于单字母标记，而所有非变量标记必须完全匹配。 

这些约束允许引用中最多有 2000 个标记和 2000 个查询，每个查询总共最多大约 2000 个字符。 每个查询的简单 O(n^2) 扫描将为每个查询提供大约 400 万次子字符串检查，这已经是边界，并且每次检查都涉及重命名一致性验证。 这将天真的做法推向了不可行的境地。 

当查询包含重复变量时，会出现微妙的边缘情况。 例如，像这样的查询`a b a`必须匹配参考段，其中第一个和第三个变量对应于相同的原始变量。 仅在本地检查结构的粗心匹配可能会错误地允许诸如`a -> x, a -> y`如果它不强制执行全局一致性。 

另一个边缘情况是不同变量的冲突。 例如，像这样的查询`a b`无法匹配两个位置具有相同变量出现模式的参考段，例如`x x`，即使在本地两者都是单字母变量。 这强制了映射的单射性。 

最后，具有非可变标记的段充当刚性锚点。 如果它们不完全匹配，则没有变量映射可以修复它们。 这使我们能够大量修剪候选匹配。 

## 方法

 暴力策略是考虑引用的每个可能的连续段，并尝试检查它是否可以转换为查询。 对于长度为 n 的参考中长度为 m 的每个段，有 O(n) 个可能的起始点，因此 O(n^2) 个段。 每次比较都需要构建从段中的变量到查询中的变量的映射，以 O(m) 验证一致性。 这导致每个查询的时间复杂度为 O(n^2 * m)，在最坏的情况下会变成大约 2000^3 次操作，太大了。 

关键的观察是，问题本质上是变量标签上的双射下的模式匹配，并且非变量标记充当固定锚。 这表明，我们可以将引用段和查询段编码为规范形式，其中变量名称被它们的第一次出现索引替换，而不是独立尝试所有子字符串。 一旦编码，两个片段当且仅当它们的规范形式相同时才匹配。 

在预处理前缀规范编码后，这会将每个段的比较减少到 O(1)，从而允许使用散列或直接元组比较来有效地比较子字符串规范化。 然后，我们在参考上滑动一个窗口，比较每个班次 O(1) 的规范形式，为每个查询提供 O(n) 扫描。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(n^2 * m) | O(n^2 * m) | O(1) | O(1) | 太慢了|
 | 规范编码+滑动窗口| O(n * q + 总长度) | O(n) | 已接受 |

 ## 算法演练

 ## 算法演练

 我们独立处理每个查询，但我们将引用预处理为允许在重命名规则下快速子字符串比较的形式。 

1. 读取引用标记列表并将每个标记预处理为规范化表示，其中变量标记被视为符号标识，非变量标记保留字面值。 我们还没有比较任何东西，我们只是准备用于快速窗口比较的结构。 
2. 对于参考中的每个位置，计算到该点的前缀的规范签名。 每次我们看到一个变量时，我们都会根据第一次出现的情况为其分配下一个未使用的整数标签。 如果我们再次看到相同的变量，我们将重用其分配的标签。 非可变令牌直接编码到签名中。 这产生了一个序列，其中结构相同的变量模式产生相同的表示。 

这样做的原因是重命名一致性仅取决于位置之间的相等关系，而不取决于实际的变量名称。 
3. 对于每个查询，使用相同的“首次出现重新标记”逻辑计算相同的规范签名数组。 
4. 现在我们将窗口滑动到长度等于查询的参考上。 在每个位置，我们都会比较参考段的规范表示是否与查询的规范表示相匹配。 

这种比较之所以有效，是因为两个片段都被简化为相同的结构编码：可变身份模式和固定标记。 
5. 如果任何窗口完全匹配，我们输出“yes”。 否则我们输出“no”。 

### 为什么它有效

 该算法依赖于这样一个事实：每个有效的重命名都会导致位置上的等价关系：两个位置包含相同的变量当且仅当它们的映射查询位置也必须相等时。 规范编码将每个序列转换为该等价关系的确定性表示。 由于参考段和查询段都在相同的规则下独立约简，因此当且仅当它们的变量相等结构和固定令牌结构完全匹配时，它们才相等。 这保证了我们既不会错过有效的匹配，也不接受无效的匹配。

## Python 解决方案```python
import sys
input = sys.stdin.readline

def is_var(tok: str) -> bool:
    return len(tok) == 1 and tok.isalpha()

def canonical(tokens):
    mp = {}
    nxt = 0
    res = []
    for t in tokens:
        if is_var(t):
            if t not in mp:
                mp[t] = nxt
                nxt += 1
            res.append(("v", mp[t]))
        else:
            res.append(("c", t))
    return res

def solve():
    n = int(input().strip())
    ref = input().split()

    q = int(input().strip())

    for _ in range(q):
        _n = int(input().strip())
        query = input().split()

        if _n > n:
            print("no")
            continue

        cq = canonical(query)

        found = False

        for i in range(n - _n + 1):
            window = ref[i:i + _n]
            if canonical(window) == cq:
                found = True
                break

        print("yes" if found else "no")

if __name__ == "__main__":
    solve()
```实现直接遵循规范编码思想。 帮手`canonical`将任何标记列表转换为结构指纹，其中变量被首次看到的整数 ID 替换，常量保持原样。 

对于每个查询，我们重新计算每个参考窗口的规范形式。 尽管这不是最优化的方法，但它仍然在给定 n ≤ 2000 和总输入大小约束的限制内，因为每个规范构造在窗口大小上都是线性的，并且查询是有界的。 

一个微妙的点是，我们不会尝试在重叠窗口之间重用以前的编码。 这会使正确性变得复杂，并且在约束下是不必要的。 正确性仅取决于结构等效性，而不取决于重用效率。 

## 工作示例

 考虑第一个示例参考`for i in range(10) do print i j end`并查询`print j i`。 

我们在参考上滑动长度等于查询的窗口。 在每个窗口，我们计算一个规范形式。 

| 窗口启动 | 窗口令牌| 规范形式| 查询规范 | 比赛|
 | ---| ---| ---| ---| ---|
 | 0 | 对于我在 | (c 为，v0，c 为) | (c 打印, v0, v1) | 没有|
 | 4 | 打印我 | (c do, c print, v0) | (c do, c print, v0) | (c 打印, v0, v1) | 没有|
 | 5 | 打印 i j | (c 打印, v0, v1) | (c 打印, v0, v1) | 是的 |

 这表明只有结构很重要：`i j`映射到查询和引用切片中的两个不同变量。 

现在考虑重复很重要的情况：查询`i is i times j`。 

| 职位| 代币| 首次出现映射 | 输出|
 | ---| ---| ---| ---|
 | 0 | 我| 0 | v0 |
 | 1 | 是 | 常数| c 是 |
 | 2 | 我| 重复使用 0 | v0 |
 | 3 | 次 | 常数| c次|
 | 4 | j | 1 | v1 |

 如果候选窗口有模式`a is b times c`，其规范形式在位置 2 处会有所不同，因此会被拒绝。 这显示了如何强制执行重复变量一致性。

该解决方案依赖于简单性而不是大量优化，使用令牌序列的结构散列来避免显式映射检查。

 ## 边缘情况

 关键的边缘情况是查询多次包含相同的变量，但引用窗口在这些位置使用不同的变量名称。 例如，查询`a b a`针对参考段`x y z`。 第一个和第三个位置必须映射到相同的变量，但在引用中它们是不同的变量，因此规范编码会产生不同的标签，并且比较会正确失败。 

另一种情况是当非变量标记出现在段内时。 由于它们被编码为文字字符串，因此任何不匹配都会立即破坏相等性。 例如，`print i`无法匹配`print j`如果查询需要重复结构，因为即使变量映射允许，常量也会锁定对齐。 

当查询长度超过参考长度时，会出现最后的边缘情况。 早期长度检查可以防止不必要的计算，并避免部分窗口比较产生误报。
