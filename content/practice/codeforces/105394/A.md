 ## 解决方案
 ## 问题理解

 我们拥有一个由人们组成的社交网络，其中的友谊形成了一个无向图。 外星人不能独立绑架个人。 如果他们从一组已连接的朋友中选择任何一个人，则必须同时选择整个已连接的组，以免发现有人失踪。 

任务是确定宇宙飞船所需的最低载客量，以便完成所有所需的绑架。 由于每个连接的组件必须被视为一个整体，因此一次旅行可以携带一个完整的组件，并且船舶必须能够安装最大的此类组件。 

输入描述了一个包含多达 200,000 人和 200,000 个好友的图表。 这种规模排除了任何二次方法，例如使用新的遍历重复检查每个节点的可达性。 任何正确的解决方案都必须在线性时间内处理该图，通常为 O(n + m)，因为即使 O(n log n) 也是可以接受的，但没有必要。 

当友谊稀少或完全不存在时，就会出现微妙的失败案例。 如果没有边，则每个节点都是孤立的，并且每个节点都形成大小为 1 的组件，因此答案为 1。一个天真的错误是假设至少存在一条边并仅从邻接列表开始组件跟踪，这将完全错过孤立的节点。 

另一种边缘情况是图完全连接时。 在这种情况下，答案是 n，因为单个组件包含所有人。 任何错误计算边缘或假设存在多个组件的解决方案都会低估所需的容量。 

## 方法

 解决问题的直接方法是，将每个人作为起点，每当遇到未访问的节点时，就进行图遍历，例如深度优先搜索或广度优先搜索。 每次遍历都会发现一个连接的组件，然后我们计算该组件中有多少个节点。 我们跟踪所有组件的最大尺寸。 

这种暴力的想法是正确的，因为它明确地模拟了规则：友谊强制分组，而连通性定义组成员资格。 成本来自于对大图的重复遍历。 在最坏的情况下，如果我们粗心并且没有正确标记访问过的节点，我们可能会多次重新探索图的大部分，从而导致 O(nm) 行为。 即使有了正确的访问跟踪，我们仍然需要遍历每个节点和边一次，这变成了 O(n + m)，这很好。 

关键的见解是该问题不需要我们模拟多次行程或任何动态过程。 它只要求无向图中最大连通分量的大小。 一旦我们认识到每个溯因完全对应于一个连接的组件，整个问题就简化为计算组件大小并取最大值。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 反复天真的探索 | O(纳米) | O(n + m) | 太慢了|
 | DFS/BFS 连接组件 | O(n + m) | O(n + m) | 已接受 |

 ## 算法演练

1. 根据友谊对构建图的邻接列表表示。 这允许有效地遍历任何人的邻居，而无需重复扫描所有边缘。 
2. 维护一个为所有人初始化为 false 的访问数组。 这确保了每个节点在所有遍历中只被处理一次。 
3. 迭代从 1 到 n 的所有人。 对于每个人，如果没有被访问过，则从该节点开始遍历。 
4. 在遍历过程中，使用堆栈或队列来探索同一连通组件中的所有可达节点。 当发现每个访问过的节点时，立即对其进行标记，避免再次访问。 
5. 统计本次遍历过程中访问了多少个节点。 该计数表示当前连接组件的大小。 
6. 使用迄今为止遇到的最大组件尺寸更新答案。 
7.处理完所有节点后，输出最大值。 

此过程有效的原因是每次遍历都与无向图中连通分量的定义完全匹配。 一旦节点被标记为已访问，它就不能属于另一个组件，因此每个节点都只贡献一个组件大小。 因此，这些尺寸中的最大值是可以处理任何所需绑架团体的最小船舶容量。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    adj = [[] for _ in range(n + 1)]

    for _ in range(m):
        a, b = map(int, input().split())
        adj[a].append(b)
        adj[b].append(a)

    visited = [False] * (n + 1)
    best = 1

    for i in range(1, n + 1):
        if visited[i]:
            continue

        stack = [i]
        visited[i] = True
        size = 0

        while stack:
            v = stack.pop()
            size += 1

            for to in adj[v]:
                if not visited[to]:
                    visited[to] = True
                    stack.append(to)

        if size > best:
            best = size

    print(best)

if __name__ == "__main__":
    solve()
```邻接表构造确保我们可以有效地遍历友谊。 访问数组保证每个节点都被处理一次，防止重复计数。 DFS 堆栈避免了 Python 中大型图的递归深度问题。 变量`best`存储遇到的最大连通分量。 

一个微妙的实现细节是将节点标记为在推送时而不是弹出时访问。 这可以防止将同一节点多次插入堆栈，这会增加密集图中的运行时间。

 ## 边缘情况

 没有边的图是不正确实现的最脆弱的情况。 用于输入`3 0`，所有节点的邻接表都是空的。 该算法仍然迭代每个节点，为每个节点启动大小为 1 的 DFS。 最大值仍为 1，这是正确的。 

完全连接的图会暴露错误，其中遍历被错误地限制为直接邻居或访问标记被延迟。 用于输入`4 6`对于所有可能的边，从节点 1 开始进行遍历，最终到达所有节点，产生大小 4。任何过早标记或不正确的邻居迭代都会低估此大小。 

链图测试可达性的传播。 在`1-2-3-4`，从1开始必须通过中间节点最终达到4。 基于堆栈的 DFS 确保传递闭包得到充分探索，因此组件大小变为 4，而不是 1 或 2，如果仅计算直接邻居，则会发生这种情况。
