 ## 解决方案
 ## 问题理解

 我们得到一个无向多重图，其中每条边连接两个城市并带有颜色标签。 任务是构建一条封闭的行走，每条边都只使用一次，因此从结构上讲，这是欧拉游览要求，但对顺序有一个额外的限制：游览中的连续边不允许共享相同的颜色，并且第一条和最后一条边的颜色也必须不同。 

输出不仅仅是“是”或“否”。 如果存在有效的旅行，我们必须输出一个起始城市，然后输出描述遍历顺序的所有边索引的排列。 如果不存在这样的遍历，我们输出-1。 

约束足够小，线性或近线性图遍历都是可行的。 对于最多 1000 条边，任何每次测试大约为 O(M) 或 O(M log M) 的算法都是可以接受的，而任何 M 中的二次方仍然会通过，但没有必要。 多达 1000 种颜色的存在表明可能会出现基于颜色的状态跟踪，但边缘计数使得显式操作邻接结构变得易于管理。 

简单的欧拉电路构造（Hierholzer 算法）通常可以解决问题，但它完全忽略了颜色。 挑战在于，即使存在欧拉电路，颜色限制也可能会破坏有效性，并且我们必须确保以受控方式选择边。 

一些边缘情况非常重要。 

一种是通常意义上的欧拉图，但与节点相关的所有边都具有相同的颜色。 例如，一颗星形以节点 1 为中心，所有边的颜色均为 1。仅当度数为偶数时才存在欧拉游览，但即使如此，任何遍历都会交替穿过中心的边，并且必须连续重复相同的颜色，这违反了约束。 正确的输出是-1。 

另一种情况是底层图中存在多个有效的欧拉之旅，但只有一些满足颜色交替。 邻接列表的朴素 DFS 排序可以轻松选择有效的欧拉结构，但会在本地失败颜色约束。 

第三种微妙的情况是当图形具有多个连接的组件时。 标准欧拉游览需要连接性（忽略孤立的顶点），因此天真的方法可能会尝试构建每个组件的游览并将它们连接起来，但这会失败，因为游览必须是所有边上的单个循环。 

## 方法

 蛮力的想法是将问题视为搜索所有可能的欧拉之旅。 我们可以尝试使用具有回溯功能的 DFS 构建有效的边排列：在每一步中，我们选择任何不违反前一条边颜色约束的未使用边。 这是正确的，因为它探索与约束一致的所有可能的边排序，并最终找到有效的欧拉电路（如果存在）。 

然而，分支因子很大。 在 d 度的节点处，有 d 个可能的下一条边，并且在 M 个边上，可能性的数量大致像阶乘行为一样增长。 即使进行了修剪，在密集情况下搜索也会退化为指数时间。 当M达到1000时，这是不可行的。 

关键的观察是我们仍然从根本上构建欧拉电路，因此 Hierholzer 的算法结构是必要的。 唯一的自由是我们消耗邻接边的顺序。 我们不是全局回溯，而是在边缘选择期间局部强制执行颜色约束。 

我们的见解是，只有当我们卡在一个节点上但在其他地方仍然有不同颜色结构的未使用边缘时，才会出现故障情况。 这表明我们应该贪婪地构建欧拉之旅，但始终确保我们不会陷入最后一个边缘颜色等于第一个边缘颜色的情况。

标准技巧是运行 Hierholzer 算法，但要小心选择下一个边缘，避免连续重复相同的颜色。 由于图尺寸很小，我们可以维护每个节点的邻接列表，并动态选择与最后一个颜色不匹配的边，仅在必要时回退。 如果我们到达不存在有效边的点，那么构造就会失败。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 边排列上的强力 DFS | O(M!) | O(M)| 太慢了 |
 | Hierholzer 具有颜色感知边缘选择 | O(M)| O(M + K) | 已接受 |

 ## 算法演练

 我们使用堆栈构建修改后的欧拉之旅，类似于 Hierholzer 的算法，但我们跟踪最后使用的边的颜色。 

1. 为每个城市构建邻接列表，存储成对的（邻居、边 id、颜色）。 我们还维护一个已使用的边数组。 

原因是欧拉遍历要求我们只消耗一次边，而邻接结构是我们唯一的访问机制。 
2. 选择任何至少有一条边的起始城市。 这里成为游览的起点。 

起点不影响欧拉电路的正确性； 任何有效的周期都可以轮换。 
3. 使用起始城市和不同于所有真实颜色的哨兵先前颜色值初始化堆栈。 
4. 当堆栈不为空时，检查顶部的当前城市。 
5. 从这个城市开始，迭代其邻接列表并查找颜色与先前边缘颜色不同的未使用边缘。 

这确保了我们永远不会在构造中放置具有相同颜色的两个连续边缘。 
6. 如果找到这样的边，则将其标记为已使用，将邻近城市压入堆栈，记录游览中的边 id，并更新最后的颜色。 

这就是欧拉遍历的向前运动。 
7. 如果当前节点不存在有效边，则从堆栈中弹出该节点。 

这对应于 Hierholzer 算法中所有传出边都用尽时的回溯。 
8、遍历完后，检查是否所有的边都被使用了。 如果不是，输出-1。 
9. 最后，确保构造序列中的第一条边和最后一条边具有不同的颜色。 如果不是，输出-1。 

正确性取决于我们从不允许局部颜色冲突，并且我们仍然确保通过欧拉结构每条边仅使用一次。 

### 为什么它有效

 该算法维护两个耦合的不变量。 首先，堆栈结构保证我们始终构建有效的部分欧拉路径，这意味着我们只遍历未使用的边，而不会重新访问它们。 其次，在每个扩展步骤中都会强制执行颜色约束，因此不会引入无效的邻接。 

由于每条边最终要么被使用，要么被证明在颜色约束下无法到达，因此失败意味着不存在有效的排序。 Hierholzer 算法中固有的回溯行为确保了如果存在有效的排序，我们不会过早地陷入阻止所有替代方案的死胡同。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m, k = map(int, input().split())
    adj = [[] for _ in range(n + 1)]

    edges = [None] * (m + 1)
    for i in range(1, m + 1):
        u, v, c = map(int, input().split())
        adj[u].append((v, i, c))
        adj[v].append((u, i, c))
        edges[i] = (u, v, c)

    used = [False] * (m + 1)

    start = 1
    for i in range(1, n + 1):
        if adj[i]:
            start = i
            break

    stack = [(start, -1)]
    tour_nodes = []
    tour_edges = []
    last_color = -1

    while stack:
        u, _ = stack[-1]
        found = False

        while adj[u]:
            v, eid, col = adj[u].pop()
            if used[eid]:
                continue
            if last_color == col:
                continue
            used[eid] = True
            stack.append((v, col))
            tour_edges.append(eid)
            last_color = col
            found = True
            break

        if not found:
            stack.pop()

    if len(tour_edges) != m:
        print(-1)
        return

    # check first/last color condition
    first_color = edges[tour_edges[0]][2]
    last_color = edges[tour_edges[-1]][2]
    if first_color == last_color:
        print(-1)
        return

    print(start)
    print(*tour_edges)

if __name__ == "__main__":
    solve()
```邻接列表存储完整的边元数据，以便我们可以在遍历期间验证颜色约束。 使用的数组确保每个边仅被消耗一次。 

栈模拟欧拉遍历。 每次前进时，我们都会立即强制执行颜色约束，这是与标准 Hierholzer 的唯一偏差。 

最后的验证步骤是必要的，因为即使满足局部约束，循环环绕条件仍然可能失败。

 ## 边缘情况

 即使存在欧拉电路，所有边共享单一颜色的配置也会立即破坏结构。 在所有边颜色均为 1 的三角形中，任何遍历都必须连续或环绕放置相同的颜色，因此当算法检测到相同的第一个和最后一个颜色时，算法会在最终验证步骤中正确失败。 

多边情况（例如由四个交替颜色的边连接的两个城市）仍然会成功，因为贪婪选择总是在每一步找到不同颜色的边，并且堆栈确保我们不会过早终止。 

断开连接的图会在消耗所有边之前强制耗尽堆栈。 条件`len(tour_edges) != m`准确地捕捉到这一点，确保我们不会错误地输出部分游览。
