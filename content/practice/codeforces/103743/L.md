 ## 解决方案
 ## 问题理解

 我们得到一个仅由字符 A、B 和 C 组成的字符串。您应该将其视为从左到右排列的一排菱形。 该过程允许我们重复挑选连续的三块钻石，在当前配置中形成图案 A、B、C。 选择从某个位置开始的这样的块后，字符串会发生变化：根据所选块是从当前动态变化的字符串中的奇数位置还是偶数位置开始，删除该块的两个外部元素或删除中间的元素。 每次操作后，剩余的菱形会关闭并从左到右重新索引，然后重复该过程。 

我们的任务不是最大化移除多少钻石，而是最大化在不再存在有效的 A-B-C 块之前可以执行的此类操作的数量。 

重要的困难在于“奇数索引”的含义在原始字符串中并不固定。 每次删除都会更改所有剩余字符的索引，这意味着即使我们选择相同的原始段，潜在操作的奇偶校验也会随着时间而变化。 

n 最大为 2×10^5 的约束意味着任何在每次操作后重复重建字符串或扫描所有子字符串的解决方案都将太慢。 在每次删除后检查所有三元组的简单模拟将降级为二次或更糟糕的行为，因为每个操作都可以移动 O(n) 个元素，并且可以有 O(n) 个操作。 

一个微妙的边缘情况来自于删除导致的奇偶校验翻转。 某个时刻被“奇数索引”的三元组在经过几次早期删除后可能会变成“偶数索引”，这会更改删除的内容，从而以非本地方式更改字符串的结构。 

例如，在像 ABCABCABC 这样的字符串中，重复选择重叠的三元组可能会改变奇偶校验并导致不同的删除结果，从而改变未来有效操作的数量。 忽略这种动态奇偶校验的贪婪方法会多计数或少计数操作。 

## 方法

 暴力破解的想法很简单：维护当前字符串，从左到右扫描以查找 A、B、C 的任何出现，选择一个，根据其当前位置应用删除规则，重建字符串，然后重复，直到不再存在有效的三元组。 这是正确的，因为它直接模拟该过程，但每个操作都需要扫描并可能重建线性结构。 在最坏的情况下，最多执行 O(n) 次操作，并且每次操作的工作时间为 O(n)，这将变为 O(n^2)，这对于 n 高达 2×10^5 来说太慢了。 

关键的观察结果是，字符串结构仅在所选三元组周围发生局部变化。 其他一切都保持相同的相对顺序，我们需要有效跟踪的唯一全球影响是指数如何变化。 我们维护一个支持两件事的结构，而不是在每次操作后物理地重建字符串：检查当前字符串中的位置当前是否形成 A-B-C 模式，并确定该位置在当前动态索引中是奇数还是偶数。 

这建议用动态删除结构来表示字符串，例如活动字符上的二叉索引树。 每个角色一开始都是活着的，删除则将位置标记为已删除。 BIT 允许我们通过计算在对数时间内有多少个活字符来计算任何原始位置的当前索引。

然后，我们维护一组候选索引或队列，其中原始字符串具有 A、B、C。每次处理候选索引时，我们都会验证它在当前活动结构中是否仍然有效，因为之前的删除可能会破坏或移动它。 如果有效，我们计算其当前索引，应用正确的删除规则，并更新活动结构。 删除后，只有受影响位置周围的恒定大小邻域才能创建或销毁新的有效三元组，因此我们在本地重新计算候选者。 

这将问题转化为小移动窗口上的动态维护任务，而不是重复的全扫描。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力模拟 | O(n^2) | O(n^2) | O(n) | 太慢了 |
 | 使用 BIT + 本地更新进行动态模拟 | O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 我们将字符串视为固定位置，并使用单独的结构来跟踪哪些字符仍然存在。 我们还在原始索引中维护有效 A-B-C 三元组的候选起始位置，并且我们不断更新它们针对当前活动集的有效性。 

## 算法演练

 1. 在位置 1 到 n 上构建二叉索引树，其中每个位置以 1 开头，表示存活。 这允许我们查询在给定的原始索引之前还有多少个字符仍然存在，这给出了它在重新索引的字符串中的当前位置。 这是必需的，因为奇偶校验取决于实时索引，而不是原始索引。 
2. 使用每个索引 i 初始化一个队列，使得 s[i]、s[i+1]、s[i+2] 等于原始字符串中的 A、B、C。 这些只是潜在的起点； 有些可能会因删除而失效。 
3. 重复从队列中取出一个候选索引i。 在使用它之前，请验证位置 i、i+1 和 i+2 是否仍然有效，并且仍然形成 A、B、C。如果不是，则丢弃它。 
4. 使用 BIT 上的前缀和查询计算 i 在实时字符串中的当前位置。 如果这个位置是奇数，我们删除位置 i 和 i+2； 否则我们删除位置 i+1。 这与应用于当前状态的问题规则相匹配。 
5. 对于我们删除的每个位置，更新 BIT 以将其标记为已删除。 这确保了未来的位置查询反映压缩的结构。 
6. 删除后，重新检查 [i−2, i+2] 范围内所有可能的起始索引，因为只有这些索引会受到删除的影响。 对于每个这样的索引j，如果它在当前活动配置中形成A、B、C，则将其推入队列。 
7. 继续，直到队列为空。 成功删除的次数就是答案。 

关键思想是，每个结构变化仅影响可能的新三元组的恒定邻域，因此我们永远不需要重新扫描整个字符串。 

### 为什么它有效

 在每一步中，算法仅处理当前活动配置中有效的三元组。 BIT 保证计算出的索引奇偶校验对于当前状态始终是正确的，因为它精确计算某个位置之前剩余的元素数量。 本地重新插入步骤确保不会遗漏新形成的 A-B-C 三元组，因为任何新的有效三元组必须至少包含一个与删除相邻的字符。 由于删除仅影响本地连接，因此每个未来候选者必须出现在前一个操作的恒定距离内。 这保持了候选生成的完整性，同时避免了全局重新扫描。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class BIT:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)

    def add(self, i, v):
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

    def range_sum(self, l, r):
        return self.sum(r) - self.sum(l - 1)

def solve():
    s = input().strip()
    n = len(s)

    alive = BIT(n)
    for i in range(1, n + 1):
        alive.add(i, 1)

    q = []
    inq = [False] * (n + 1)

    def is_alive(i):
        return alive.range_sum(i, i) == 1

    def valid(i):
        if i < 1 or i + 2 > n:
            return False
        if not (is_alive(i) and is_alive(i + 1) and is_alive(i + 2)):
            return False
        return s[i - 1] == 'A' and s[i] == 'B' and s[i + 1] == 'C'

    for i in range(1, n - 1):
        if s[i - 1:i + 2] == "ABC":
            q.append(i)
            inq[i] = True

    ans = 0
    head = 0

    while head < len(q):
        i = q[head]
        head += 1
        if i < 1 or i + 2 > n:
            continue
        if not valid(i):
            continue

        pos = alive.sum(i)

        if pos % 2 == 1:
            remove = [i, i + 2]
        else:
            remove = [i + 1]

        for x in remove:
            if alive.range_sum(x, x):
                alive.add(x, -1)

        ans += 1

        for j in range(i - 2, i + 3):
            if j < 1 or j + 2 > n:
                continue
            if valid(j):
                q.append(j)

    print(ans)

if __name__ == "__main__":
    solve()
```BIT 跟踪在任何索引之前仍然存在多少字符，这正是确定用于奇偶校验的当前重新索引位置的因素。 有效性检查确保我们只对先前删除后仍然存在的三元组进行操作。 局部重新插入循环至关重要，因为删除字符后，可能会在修改区域形成新的 ABC 模式。 

队列允许包含重复项或过时的条目，并且保留了正确性，因为每个候选项在使用前都会重新验证。

 ## 边缘情况

 像 ABC 这样的最小字符串测试基本有效性条件，因为只有一个操作是可能的，并且在删除之后不再保留任何进一步的结构。 

诸如 ABCABCABC 之类的重复模式测试算法是否正确处理删除后新三元组的级联形成。 修改区域周围的候选重新插入可确保发现新的重叠模式。 

不带任何 ABC 子字符串的字符串可确保队列驱动的进程立即正确终止，而不会尝试无效操作。
