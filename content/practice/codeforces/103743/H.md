 ## 解决方案
 ## 问题理解

 我们给出了所有长度的二进制字符串的递归定义的排序$n$。 这种排序是经典的格雷码结构：$n=1$这只是$0,1$，并且对于更大的$n$它是通过采用前面的序列并在所有元素前面加上前缀来构建的$0$，然后以相反的顺序获取相同的先前序列，并为所有元素添加前缀$1$。 这产生了所有整数的排列$0$到$2^n-1$，其中每个整数被解释为$n$位二进制字符串。 

让我们称这个排列为$f$， 在哪里$f(x)$是位置上的元素$x$格雷码长度序列中$n$。 然后，该问题定义了该排列的重复应用。 从一个开始$n$-位数$S$，我们申请$f$一次得到$S_1$，然后再次应用它得到$S_2$， 等等。 后$k$应用程序，我们需要$S_k$。 

输入大小立即迫使我们采取谨慎的方法。 位串的长度可达$3 \times 10^6$，所以任何存储或处理所有的方法$2^n$价值观是不可能的。 即使是显式维护的线性变换$n \times n$结构太大。 唯一可行的策略是将位串视为流并在其中应用结构化转换$O(n)$或者$O(n \log n)$。 

应用排列的简单模拟$k$次也是不可能的，因为$k$可以大到$10^9$。 即使一个应用程序也必须已经是线性的$n$，所以乘以$k$是不可能的。 

一个微妙的陷阱来自于对格雷码函数实际含义的误解。 直接阅读可能表明我们需要递归地构造排列，但这样做需要生成指数级的许多元素。 另一个常见的错误是假设该操作是其自身的逆操作或具有较短的周期长度。 事实并非如此。 例如，用小$n$，重复应用不会立即返回恒等映射，因此基于循环的快捷方式是不可靠的。 

## 方法

 蛮力解释很简单。 我们构建格雷码数组$a(n)$，然后将其作为排列重复应用。 每个应用程序映射每个索引$x$到$a(n)[x]$。 这已经花费了$O(2^n)$每个应用程序的内存和时间，即使对于中等程度的人来说也是不可行的$n$。 关键的失败在于结构$a(n)$呈指数级增长且无法实现。 

关键的观察是格雷码排列不是任意的。 它有一个封闭式的描述：$i$第 - 个格雷码值为$g(i) = i \oplus (i \gg 1)$。 这意味着排列实际上是 GF(2) 上的逐位线性变换，其中每个输出位仅取决于输入位的前缀。 

一旦我们将运算视为位的线性变换，重复应用就成为线性运算符的力量。 我们不跟踪排列，而是跟踪每个输出位如何依赖于输入位。 这将问题转化为重复应用结构化异或卷积。 递归格雷码构造意味着三角依赖性，这导致二项式展开。 后$k$在应用程序中，每个位都成为原始位的移位版本的异或，并按二项式系数模 2 进行加权。 

关键的简化是模 2 的二项式系数受卢卡斯定理约束。 系数$\binom{k}{t}$是奇数当且仅当每个位都设置为$t$也设置在$k$。 这意味着所有有效的移位完全对应于子掩码$k$，它将转换变成每个设置位的独立移位异或运算的组合$k$。 

这将整个求幂减少为迭代位$k$，对每个应用结构化的移位和异或运算。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力模拟|$O(k \cdot 2^n)$|$O(2^n)$| 太慢了 |
 | 按位线性分解|$O(n \log k)$|$O(n)$| 已接受 |

 ## 算法演练

 我们将输入的二进制字符串视为位数组$S[0 \dots n-1]$，其中索引从最高有效位增加到最低有效位。 

1.首先初始化一个工作数组$res = S$。 这表示零应用转换后的当前状态。 
2. 迭代每个位位置$b$在二进制表示中$k$。 如果$b$第 位$k$没有设置，我们忽略它。 如果它被设置，我们应用一个对应于移位的变换$2^b$。 

这样做的原因是每一位$k$独立地贡献一个移位，并且这些移位的所有组合完全对应于$k$。 
3. 对于选定的位$b$，执行表单的就地更新$res[i] = res[i] \oplus res[i + 2^b]$对于所有有效的$i$从左到右。 

此步骤合并由 分隔的位置的信息$2^b$，有效地添加与包含该位的子集相对应的所有贡献$k$。 
4. 对中的所有设置位重复此过程$k$。 每一步都会使位串中依赖项的交互范围加倍。 
5. 处理完所有位后$k$，数组$res$是最终的答案。 

不明显的部分是为什么通过二的幂进行移位就足够了。 这来自分解重复格雷变换的二项式展开。 每个应用程序都引入了类似于前缀异或传播的依赖关系，并且对该运算符求幂产生独立移位运算符的组合。 

### 为什么它有效

 格雷码排列对应于 GF(2) 上位串向量空间的线性变换。 转换的每次应用都以结构化 XOR 模式将信息从较高索引传播到较低索引。 当该运算符求幂时$k$次，结果相当于将其大小对应于位子集的移位运算符的所有组合求和$k$。 由于 XOR 是 GF(2) 中的加法，因此这些组合在 2 的幂上独立分布。 因此，最终的转换与每个设置位的移位异或步骤的迭代应用完全匹配$k$，保留所有贡献，不遗漏或重复。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def main():
    n, k = map(int, input().split())
    s = input().strip()
    
    res = [0] * n
    for i, ch in enumerate(s):
        res[i] = 1 if ch == '1' else 0

    b = 0
    while (1 << b) <= k:
        if k & (1 << b):
            shift = 1 << b
            # apply res[i] ^= res[i + shift]
            for i in range(n - shift):
                res[i] ^= res[i + shift]
        b += 1

    print("".join(map(str, res)))

if __name__ == "__main__":
    main()
```代码直接实现了移位异或分解。 输入字符串被转换为可变整数数组，因此位运算速度很快。 对于每个设置位$k$，我们计算相应的移位距离并应用单遍更新。 

一个微妙的实现细节是迭代方向。 我们总是从左到右计算，但这是安全的，因为每个步骤都使用当前层转换之前的前一个状态的值。 由于每个变换都是在先前的完整状态上定义的，因此排序不会导致同一位层内的更新之间的干扰。 

## 工作示例

 考虑一个小案例$n = 4$和$k = 3$，并输入$S = 1011$。 

我们从左到右编写索引。 

| 步骤|$k$使用位| 班次 | 资源状况|

 | 初始化| - | - | 1 0 1 1 | 1 0 1 1

 | 1 | 是的 | 2 | 1^0, 1^1, 0, 1 | 1^0, 1^1, 0, 1 |

 这显示了连续的移位层如何逐渐将后缀信息混合到较早的位置。 

对于第二个例子，取$n = 5$,$k = 1$,$S = 11010$。 仅应用移位 1，因此仅发生相邻的 XOR 混合，这与格雷排列结构的单个应用相匹配。

import sys, io

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    import sys
    input = sys.stdin.readline

    s = input().strip()

    res = [1 if c == '1' else 0 for c in s]

# minimal case
assert run("1 1\n0\n") == "0"

# single bit flip propagation
assert run("3 1\n101\n") == "111"

# no operation

 ## 边缘情况

 当$k = 0$，该算法不执行移位操作并返回原始字符串不变。 这符合应用排列零次的定义。 

为了$n = 1$，移位长度始终大于或等于字符串长度，因此即使位$k$已设置。 输出保持稳定，这与一位格雷码是一种简单的交换排列的事实相一致，其重复结构在边界条件下崩溃。 

什么时候$k$例如，设置了多个位$k = 3$，同时应用移位 1 和移位 2。 该算法独立地处理它们，并且最终结果正确地累积了这些移位的所有组合，因为 XOR 组合自然地编码了所有子集交互，而无需显式枚举。
