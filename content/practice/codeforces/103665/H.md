 ## 问题理解

 我们得到一个二进制字符串$t$，并且我们可以将它与特殊的无限二进制字符串族进行比较$s_m$。 每个$s_m$是固定的：它从 0 开始并交替每个位置，所以它看起来像 0101… 直到长度$m$。 

任务是找到最小的长度$m$这样字符串$t$可以作为子序列获得$s_m$。 子序列意味着我们可以从中删除字符$s_m$不改变剩余字符的顺序，并且剩余的必须匹配$t$确切地。 

所以我们不嵌入$t$作为一个连续的块。 我们正在选择交替模式字符串中的位置，并且我们想知道该交替模式必须有多长，以便所有字符$t$可以按顺序匹配。 

约束条件$|t| \le 10^5$迫使我们在线性时间内思考。 任何模拟建筑的方法$s_m$明确地是不可能的，因为$m$它本身可以变大，可能与答案成正比。 二次或偶数$O(m)$每次测试尝试都是遥不可及的。 

一个天真的方向是尝试增加$m$并贪婪地检查是否$t$是一个子序列$s_m$。 这太慢了，因为每次检查都是$O(m)$， 和$m$本身可能很大，我们可能需要反复增加它。 

当贪婪地思考而不仔细跟踪状态时，会出现更微妙的故障模式。 例如，如果$t = 111$，人们可能会错误地假设，因为 1 出现在$s_m$，它总是很容易嵌入，但真正的限制是我们必须将交替模式扩展多远才能按顺序对齐足够的 1，同时尊重子序列匹配。 

关键的难点在于每个角色$t$迫使我们前进$s_m$，有时我们可能需要“扩展”超出当前预期的奇偶校验模式以匹配所需的位。 

## 方法

 蛮力的想法很简单：修复一个$m$、构建或模拟$s_m$，并检查是否$t$是一个子序列。 该检查是线性的$m$因为我们扫描$s_m$并贪婪地匹配字符$t$。 在最坏的情况下，如果答案很大，我们对许多值重复此操作$m$，导致二次行为。 

核心观察是我们实际上不需要构建$s_m$。 的结构$s_m$是完全确定性的：在位置$i$，该值很简单$i \bmod 2$，从0开始。这意味着当我们扫描时$s_m$，唯一重要的状态是当前头寸平价。 

我们可以直接模拟子序列匹配过程：我们沿着无限交替序列“行走”，一次前进一步，并且仅当当前符号与下一个所需字符匹配时$t$我们会消费它吗？ 最小的$m$正是我们消耗完整个之后到达的位置$t$。 

微妙之处在于我们事先并不限制固定的前缀长度。 相反，我们在概念上根据需要延长交替序列，并且在我们完成匹配的那一刻$t$，当前位置给出最小长度。 

这将问题转化为对想象的无限交替字符串的单一贪婪扫描，其线性关系为$|t|$。

 ## 算法演练

 我们模拟沿着无限长的绳子行走$010101...$在尝试匹配的同时$t$作为子序列。 

1. 初始化一个指针，代表我们在交替序列中的当前位置。 从位置 0 开始，对应于值 0。 
2. 初始化索引$i = 0$用于扫描字符串$t$。 我们会尽力匹配$t[i]$与交替序列中的当前位置。 
3.同时$i < |t|$， 比较$t[i]$与当前的交替字符。 如果位置为偶数，则该字符为0； 如果是奇数，则为 1。 
4.如果当前字符匹配$t[i]$， 进步$i$通过1.这意味着我们成功放置了$t[i]$在子序列中的这个位置。 
5. 在所有情况下，将交替序列中的位置前移 1，因为子序列匹配允许跳过字符。 
6.一次$i$达到$|t|$，我们已经匹配了所有字符。 当前位置在交替序列中最小$m$允许嵌入$t$。 

关键的行为是我们决不后退。 每个不匹配仅仅意味着我们跳过一个字符$s_m$，每场比赛消耗一个字符$t$。 

### 为什么它有效

 在任何时候，我们都会维护已经匹配的不变量$t[0..i-1]$在交替字符串中使用一些递增的位置序列。 下一个可用位置是保持顺序的最小可能索引。 因为交替字符串是固定且确定的，所以一旦出现匹配就贪婪地获取匹配可以确保我们最小化最后一个字符使用的最终位置。 匹配字符的任何延迟只会将其位置进一步向右推，而不是更早，因此贪婪策略会产生最小的可能$m$。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = input().strip()
    pos = 0  # current position in s_m (conceptually infinite)
    
    for ch in t:
        # advance until we can match ch
        while pos % 2 != (ch == '1'):
            pos += 1
        # match at this position
        pos += 1
    
    print(pos)

if __name__ == "__main__":
    solve()
```该代码隐式地使用位置奇偶校验来处理交替序列。 对于中的每个字符$t$，它向前移动，直到奇偶校验与所需的位匹配，然后消耗它并继续。 最终值`pos`正是最小长度$m$，因为它代表最后一个匹配字符之后的第一个位置。 

一个微妙的点是`pos`总是代表下一个空闲位置$s_m$，所以在放置最后一个字符后，正确答案正是`pos`， 不是`pos - 1`。

 ## 边缘情况

 一个边缘情况是当$t$已经是 1。在这种情况下，无法使用位置 0，因此算法必须在第一次匹配之前前进到位置 1。 例如，与$t = 1$，我们从 pos = 0 开始，查看不匹配情况，递增到 1，然后在那里匹配。 输出变为 2，这正确反映了包含 1 子序列的最短交替前缀的长度为 2。 

另一种情况是长时间运行相同的字符，例如$t = 000...0$。 交替的结构迫使我们跳过每一个奇怪的位置，因此随后的每场比赛都需要一次向前跳两步。 该算法自然地累积这些跳跃，产生一个最终位置，该位置的增长大约是所需零数量减一的两倍，与实际嵌入要求相匹配。 

最后一个微妙的情况是交替模式，例如$t = 101010$。 即使这符合结构$s_m$，子序列嵌入仍然取决于对齐。 该算法在最早可能的兼容位置匹配每个字符，并且由于奇偶校验交替，它最终会消耗完全连续的位置而不会产生额外的延迟，从而产生$m = |t|$或非常接近，具体取决于起始对齐。
