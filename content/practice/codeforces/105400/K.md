 ## 解决方案
 ## 问题理解

 我们得到一个数组，并询问是否可以使用相邻交换将其转换为排序数组，但有一个约束，随着过程的进行，交换逐渐变得更加困难。 

该操作不是标准交换。 在整个过程中的第 j 次交换时，只有当两个值中较小的值至少为 j 时，我们才可以交换位置 i 和 i+1 处的元素。 这意味着早期的掉期是允许的，但后期的掉期要求双方的参与价值都越来越大。 交换计数在整个序列中是全局的，而不是每个位置，因此每次交换都会消耗一个单位的“权限级别”。 

任务是确定是否存在以非降序对数组进行排序的此类交换序列。 

n 最大为 100000 的约束立即排除了任何交换序列的模拟。 即使是最佳的类似冒泡排序的过程也可能需要 O(n^2) 次交换，并且每次交换的有效性取决于其在全局操作顺序中的位置，这种耦合时间和状态的方式使得直接模拟变得不可行。 

全球互换指数出现了一个微妙的问题。 天真的解释可能会错误地假设每个位置都有自己的阈值，但第 j 个交换适用于整个进程历史记录。 这打破了局部性并使标准的贪婪泡沫推理无效。 

暴露危险的一个小例子是一个数组，其中小元素必须与许多较大元素交叉。 即使每个交换在本地看起来都是有效的，累积的交换指数最终也会变得太大，使得未来所需的交换变得不可能。 

## 方法

 蛮力的想法是在检查交换有效性的同时模拟冒泡排序。 我们重复扫描数组，在允许的情况下交换相邻的反转。 每次交换都会增加一个全局计数器 j，并且我们验证 min(a[i], a[i+1]) ≥ j。 

这在原则上是正确的，因为任何有效的交换序列都对应于正在解析的相邻反转的某些序列。 然而，在最坏的情况下，排序可能需要 O(n^2) 次交换，并且每个步骤都涉及扫描或维护结构，使得总成本为 O(n^2)。 当n达到100000时，这远远超出了可行的极限。 

关键的观察是扭转视角。 我们不考虑互换，而是考虑在互换指数的全球不断增加的约束下元素必须如何移动。 关键的限制是，每次执行交换时，两个交换值中较小的一个必须至少是交换数 j。 这意味着小值必须出现在交换序列的早期； 否则，随着 j 的增长，它们将变得不可用。 

现在从排序的角度来考虑这个过程。 较小的元素不能参与多次互换，因为每次涉及它们的互换都会消耗一个单位的全球预算。 因此，每个元素都有一个与其价值相关的自然“移动预算”：如果一个元素的价值为 x，则它不能参与超过时间 x 的交换。 这将问题转变为检查每个元素是否可以到达其排序位置，而不会被迫进行太多交换。 

我们可以通过模拟最终的排序顺序并跟踪每个元素必须移动多远来对此进行建模。 一个关键的转换是要注意，只有值≥其交换深度约束的元素才能参与必要的反转。 这导致使用芬威克树或反转式计数进行贪婪的可行性检查，但对每个值可以“负担”多少有效交换增加了限制。 

一种更简单的查看方法是从左到右处理数组，同时维护将每个元素带到正确位置所需的交换次数，并确保所需的交换计数永远不会超过该元素的值。 

这将问题转化为约束反演距离可行性检查。

 ## 算法演练

 我们将问题解释为检查每个元素是否能够“承受”它必须参与的交换次数以达到其排序位置。 

1. 构建数组的排序版本，并将每个值按排序顺序映射到其最终位置。 这给了我们每一个事件必须结束的地方。 
2. 遍历数组，同时维护一个结构，该结构允许我们计算当前定位的元素在最终放置方面较大或较小之前的数量。 
3. 对于每个元素，计算它相对于排序顺序贡献了多少个反转。 这代表它必须有效参与多少互换。 
4. 检查每个元素的反转要求是否超过其值。 如果是这样，该元素将在大于其值的时间 j 被强制进行交换，从而违反了最小约束。 
5. 如果所有元素都满足此可行性条件，则输出 YES，否则输出 NO。 

关键思想是，每次反转对应于最终必须发生的一次交换，并且全局交换指数单调增长。 由于值为 x 的元素无法在超过 x 的交换中幸存，因此任何大于 x 的反转负载都意味着不可能。 

### 为什么它有效

 相邻交换的过程通过一次解决一个反转来转换数组。 每个反演分辨率对应于至少一次涉及该反演中较小元素的交换。 因为交换号 j 必须满足 j ≤ min(a[i], a[i+1])，所以任何反转中的最小元素限制了该反转可以在多晚的时间内解决。 

因此，每个元素都对其作为交换的较小一侧可以参与的反转次数施加了硬性限制。 如果所需数量超过其值，则任何交换顺序都无法避免违反全局约束。 因此，倒转会计捕获了所有必要的掉期义务，而无需模拟该过程。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class BIT:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)

    def add(self, i, v):
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

def solve():
    t = int(input())
    for _ in range(t):
        n = int(input())
        a = list(map(int, input().split()))

        vals = sorted(set(a))
        comp = {v: i + 1 for i, v in enumerate(vals)}

        bit = BIT(len(vals))
        inv = [0] * n

        # count how many greater elements appear before each position
        for i in range(n):
            x = comp[a[i]]
            inv[i] = i - bit.sum(x)
            bit.add(x, 1)

        # now check feasibility via reverse pass
        bit = BIT(len(vals))
        for i in range(n - 1, -1, -1):
            x = comp[a[i]]
            greater_on_right = bit.sum(len(vals)) - bit.sum(x)
            if greater_on_right > a[i]:
                print("NO")
                break
            bit.add(x, 1)
        else:
            print("YES")

if __name__ == "__main__":
    solve()
```该解决方案压缩了值，因为在计算反转时只有相对顺序很重要。 芬威克树用于有效计算与反转相关的计数。 

正向传递构建频率信息，但关键的检查是在反向传递中：对于每个元素，我们计算有多少个更大的元素位于其右侧。 该数字对应于必须向左交换的次数。 如果这超过了元素的值，它就无法在交换序列中存活足够长的时间。 

遍历时严格检查条件； 如有违反，我们立即拒绝。

 ## 边缘情况

 关键的边缘情况是所有值都相等。 在这种情况下，每次交换始终有效，因为 min(a[i], a[i+1]) 始终等于最多 n 的交换索引，并且不存在排序约束。 该算法正确返回 YES，因为没有元素必须跨越任何严格大于的元素。 

当最小元素接近数组末尾时，会出现另一种边缘情况。 该元素积累了许多所需的交换，并且由于其值很小，因此它很快就会违反约束。 反向传递检查通过计算有多少个较大的元素位于其右侧来立即捕获这一点。 

第三种边缘情况是严格递减数组。 每个元素都必须跨越所有比自己小的元素，最小的元素会积累最大的负担。 一旦该负担超过元素的值，该算法就会拒绝，这与维持所需交换序列的可能性相匹配。
