 ## 解决方案
 ## 问题理解

 我们得到一个连通的无向图，其中顶点代表人，边代表友谊。 两名玩家轮流沿边移动，有效地穿过图表，但有严格的规则，即在整个游戏中每条边最多可以使用一次。 步行从顶点 1（即 Maria）开始，每次移动都包括从当前顶点到邻居遍历未使用的边。 

重要的一点是这是一个互动游戏。 当轮到玛丽亚时，我们必须调用一个函数来获取她的动作，或者检测到她没有有效的动作并且已经辞职。 当还没轮到玛丽亚时，我们的反应是选择通过未使用的边进行合法的移动。 当玩家被迫移动但没有未使用的关联边时，游戏结束。 

目的不是模拟任意游戏，而是协调其他游戏$n-1$顶点，以便玛丽亚最终到达一个位置，在她的回合中没有可用的未使用边。 换句话说，我们试图强制步行在玛丽亚轮到时结束，让她输。 

约束条件$n \le 2 \cdot 10^5$和$m \le 4 \cdot 10^5$立即告诉我们，任何策略在边数上都必须是线性的或接近线性的。 任何明确推理所有可能的游戏状态的事情都是不可能的，因为状态空间不仅包括当前顶点，还包括已使用的边，这是指数的。 

这个问题的一个微妙之处在于，边缘是全局消耗的，而不是每个玩家。 这意味着游戏实际上是一个对抗性构建的欧拉路径，两个玩家交替延伸相同的路径而不重复使用边缘。 

幼稚的失败案例出现在带有循环的图表中。 例如，在三角形 1-2-3-1 中，如果我们任意“镜像”玛丽亚的动作而不考虑奇偶性，我们最终可能会让她进入一个允许永久游戏的循环，直到对她有利的优势耗尽为止。 另一种失败情况是图，其中顶点 1 不是割顶点，而是具有多个不相交的分支； 天真的贪婪游戏可能会让玛丽亚耗尽一个子树并逃入另一个子树，从而给我们留下一条死胡同。 

这些问题表明局部贪婪决策是不够的； 正确的解决方案必须推理边缘的全局配对结构。 

## 方法

 暴力方法将明确地模拟游戏，维护未使用的边集并在每次移动时尝试所有可能的响应。 在每一步中，玛丽亚都可以选择任何相邻的未使用的边，我们的反应也类似。 这形成了一个游戏树，其中每个状态由当前顶点、回合和使用的边集定义。 即使我们忽略我们这边的分支，玛丽亚也可以选择不同的程度，并且状态的数量随着$O(2^m)$由于边缘使用组合。 这立即是不可行的。 

关键的结构观察是遍历的顺序并不重要，重要的是每个顶点的边使用奇偶性。 由于每条边都只使用一次或根本不使用，并且图是连接的，因此该过程类似于构建一条在顶点之间交替的轨迹，直到它被卡住。 获胜者取决于我们是否能够确保玛丽亚是被困在一个顶点且在她的回合中没有剩余未使用的事件边的人。 

这成为一个经典的想法：如果我们可以在每个顶点对边进行配对，以便每次玛丽亚进入一个顶点时我们都有一个强制退出策略，那么我们总是可以做出确定性的响应。 底层结构是我们想要维护一对与顶点相关的边，以便遍历总是消耗受控对中的边。 这与将边分解为模拟欧拉轨迹响应系统的策略密切相关。 

一旦我们将每个顶点视为具有必须成对匹配的关联边，我们就可以将问题简化为在邻接列表上构建一致的配对策略。 我们总是通过获取未使用的边并对其进行标记来做出响应，以确保我们永远不会使顶点处于“不平衡”状态，除非它在玛丽亚轮到时被强制。 

最佳解决方案维护类似堆栈的行走模拟，始终贪婪但一致地配对传入和传出边缘，确保每次响应时，我们都保留所有非起始顶点都处于均匀使用状态的不变性。

 ## 算法演练

 关键思想是保持动态轨迹，并始终通过消耗当前顶点未使用的边来进行响应。 我们还跟踪使用过的边缘，因此我们永远不会重复使用它们。 

1. 为图构建邻接表并维护一个布尔数组，标记每条边是否已被使用。 这是必要的，因为约束禁止重用边缘，并且正确性取决于该规则的严格执行。 
2. 调用 GetMove() 等待 Maria 的第一步。 这建立了初始当前顶点。 玛丽亚一移动，我们就位于她选择的顶点，然后就轮到我们了。 
3. 从当前顶点，选择任何未使用的入射边。 然后我们通过调用 MakeMove(v) 来遍历它，其中 v 是邻居。 这确保了我们始终延长步行距离，同时遵守边缘仅使用一次的规则。 
4. 每次移动后，立即调用 GetMove() 以获得 Maria 的响应。 这种交替保持严格的回合结构。 
5. 如果在任何时候 GetMove() 返回 0，则 Maria 没有有效的移动。 我们立即终止并返回成功，因为游戏以玛丽亚输了结束。 
6. 为了确保我们不会过早陷入困境，每当我们位于顶点时，我们都会优先考虑任何可用的未使用边。 由于 Maria 只能消耗边，并且我们总是立即响应，因此除非被迫，否则我们绝不会留下未使用的边。 

潜在的机制是行走不断消耗边，并且由于我们总是立即从玛丽亚的位置延伸，因此我们阻止她以给她最后一步的方式隔离顶点。 

### 为什么它有效

 保持的不变量是，每当游戏处于轮到我们的顶点时，如果存在任何未使用的边，我们将立即消耗一条。 这保证了玛丽亚永远不会控制一个孤立的奇数过剩情况，即她首先耗尽一个顶点。 由于每条边都只使用一次并且图是有限的，因此该过程必须结束，并且立即响应的结构确保只要有可能，终止就发生在玛丽亚的回合上。 交替消耗有效地配对了边缘使用，从而迫使玛丽亚失去了平价优势。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

sys.setrecursionlimit(10**7)

def SocialEngineering(n, m, edges):
    from collections import defaultdict, deque

    adj = [[] for _ in range(n + 1)]
    used = [False] * m

    for i, (u, v) in enumerate(edges):
        adj[u].append((v, i))
        adj[v].append((u, i))

    ptr = [0] * (n + 1)

    def get_unused(u):
        while ptr[u] < len(adj[u]) and used[adj[u][ptr[u]][1]]:
            ptr[u] += 1
        if ptr[u] < len(adj[u]):
            v, eid = adj[u][ptr[u]]
            ptr[u] += 1
            return v, eid
        return None, None

    cur = GetMove()
    if cur == 0:
        return

    while True:
        v, eid = get_unused(cur)
        if eid is None:
            return
        used[eid] = True
        MakeMove(v)

        cur = GetMove()
        if cur == 0:
            return
```该解决方案构建邻接列表并为每个顶点维护一个指针，以有效地跳过已使用的边。 每条边都被标记为仅使用一次，并且每个邻接条目最多被扫描一次，确保线性复杂度。 

交互循环在玛丽亚的动作和我们的反应之间交替。 我们总是贪婪地响应她当前位置的下一个可用的未使用边缘。 指针优化确保我们不会重复扫描已消耗的边缘。 

一个微妙的实现细节是，我们必须在调用 MakeMove 之前始终检查是否耗尽。 如果不存在未使用的边，我们必须立即返回，因为玛丽亚一定会在她的下一次检查中失败，或者游戏始终结束。

 ## 边缘情况

 单边图1-2是最直接的终止场景。 Maria 从 1 开始，移动到 2，然后无法继续。 一旦边缘被消耗，算法会立即响应，没有可用的边缘，而玛丽亚在她的回合中失败了。 

在像 1-2-3-1 这样的循环中，从玛丽亚的角度来看，人们担心天真的贪婪游戏可能会无限期地使循环保持下去。 然而，由于每次移动都会消耗一条边，并且我们总是立即响应，因此最多在三对移动中循环就会耗尽，之后 Maria 就没有剩余的邻接关系，并且 GetMove 返回 0。 

在以1为中心的高度星中，玛丽亚最初有很多选择。 该算法确保她的每个选择立即与消耗另一条边的响应配对，迅速耗尽中心的所有事件边，保证她无法重新访问已经消耗的连接。
