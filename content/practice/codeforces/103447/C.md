 ## 解决方案
 ## 问题理解

 我们有一棵有根的树，只有叶子对最终目标很重要。 每片叶子都已经有所需的最终颜色，而内部节点根本没有目标颜色。 最初，什么也没有画。 我们可以重复执行选择节点的操作`x`和一种颜色`c`，然后我们重新绘制子树中的每一个叶子`x`着色`c`，覆盖以前存在的任何内容。 我们的任务是使用尽可能少的操作，以便每片叶子最终都具有所需的颜色。 

重要的细节是操作并不局限于单个叶子，它会淹没整个子树，但只影响叶子。 内部节点只是定义哪些叶子被分组在一起的结构。 

约束条件`n ≤ 10^5`意味着我们需要本质上线性或接近线性的东西。 任何试图显式模拟操作或考虑叶子子集的解决方案都会立即变得太慢。 我们需要树 DP 或 DSU-on-tree 风格的方法，其中每个边或节点仅被处理少量次。 

当相同颜色的多个叶子分布在树的不同部分时，贪婪直觉的一个微妙的失败案例就会出现。 

例如，假设根具有三个子分支，每个子分支都包含一个颜色为叶子的叶子`1`，但这些分支直到根为止都是不相交的。 如果我们独立地处理每个叶子，我们可能会假设需要三个操作。 实际上，根部的单个操作可以将所有它们绘制在一起。 因此，相同颜色出现在树中的结构与有多少叶子一样重要。 

另一个常见的陷阱是假设我们可以通过获取其子树并集来独立地解决每种颜色。 这种情况会被打破，因为一个操作绘制了整个子树，因此颜色通过树结构中的重叠进行交互。 

## 方法

 一个直接的暴力想法是考虑每一个可能的操作序列。 在每一步中，我们选择一个节点和一种颜色并应用它，然后尝试达到目标配置。 这会立即爆炸，因为即使是单个路径也有很多选择，并且每个操作都可能影响树的大部分。 分支因子是`O(n)`每一步和深度也可以`O(n)`，使得这完全不可行。 

一种更结构化的蛮力是逆向思考。 我们可以想象为每个叶子分配最后一个操作，而不是向前构建颜色。 每个叶子都需要通过某个祖先节点上的操作以正确的颜色“覆盖”。 然而，如果不同的叶子位于同一子树中并且具有相同的最终颜色，则它们可以共享相同的操作。 问题在于将相同颜色的叶子分组为可重用的子树操作。 

关键的观察是，重要的不是单个叶子，而是相同颜色的叶子如何通过树结构连接起来。 如果相同颜色的两个叶子位于某个节点的不同子分支中，那么它们仍然可以通过该节点或更高节点的单个操作来统一。 如果它们位于树的完全独立的部分，并且在根之前从未相遇，那么除了更高的祖先之外，它们无法统一。 

这导致了一个干净的重新表述：对于每种颜色，仅查看具有该颜色的叶子，并考虑它们在树中形成了多少个连接的组件。 每个这样的组件至少需要一个操作，因为没有任何单个子树操作可以同时影响不同断开部分中的叶子而不错误地接触其他颜色。 

答案就是由叶子形成的连通分量的所有颜色的总和。 

为了有效地计算这个问题，我们进行自下而上的遍历，并为每个节点维护其子树中出现的颜色。 组合子子树时，如果多个子树中出现相同的颜色，则这些出现对应于在此节点合并的单独组件，从而减少所需的操作。 

我们为每个节点维护一个从颜色到包含该颜色的子子树数量的映射。 每当一种颜色出现在多个子项中时，我们就会计算合并数。 这可以使用从小到大的集合合并来有效地实现。

 | --- | --- | --- | --- |
 | 暴力操作 | 指数| 大| 太慢了|
 | DSU 在树上计算颜色分量 | O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 1. 为树求根并执行后序遍历，以便在其父级之前处理子级。 这确保了当我们处理一个节点时，我们已经知道每个子子树中存在什么颜色。 
2. 对于每个节点，维护一组出现在其叶子之间的子树中的颜色。 该集合表示“哪些颜色在此节点下处于活动状态”，而不存储叶级详细信息。 
3. 还维护每个节点的频率图，其中`freq[c]`计算有多少个不同的子子树包含颜色`c`。 我们不计算有多少叶子，只计算子子树是否至少包含该颜色一次。 
4. 处理节点的每个子节点，并使用从小到大的策略将其颜色集合并到父节点中。 在合并期间，对于从子节点插入的每种颜色，更新当前节点中该颜色的频率计数。 
5. 合并所有子节点后，扫描该节点的频率图。 对于每种颜色`c`，如果它出现在`k`不同的子子树，那么这些`k`出现次数对应于`k`在此节点相遇的独立组件。 我们必须考虑到`k - 1`合并以统一它们，所以我们添加`k - 1`到答案。 
6. 节点的结果颜色集只是所有子集的并集。 这会向上传递给其父级。 
7. 最终答案是所有节点上所有这些“合并成本”的总和。 

这是正确的原因是，当限制在原始树上时，每种颜色的叶子都会形成森林。 每当两个相同颜色的组件在一个节点处相遇时，该节点就是它们可以统一的第一个位置，并且我们必须执行一次操作才能将每个附加组件合并到第一个组件中。 该算法精确地计算这些必要的合并，仅此而已。 

## Python 解决方案```python
import sys
input = sys.stdin.readline
sys.setrecursionlimit(10**7)

from collections import defaultdict

def solve():
    n = int(input())
    p = list(map(int, input().split()))
    c = list(map(int, input().split()))

    g = [[] for _ in range(n)]
    for i, par in enumerate(p):
        g[par - 1].append(i + 1)

    # DSU on tree: store set of colors in subtree
    sets = [set() for _ in range(n)]
    freq = [defaultdict(int) for _ in range(n)]
    ans = 0

    def dfs(u):
        nonlocal ans

        # initialize leaf color
        if c[u] != 0:
            sets[u].add(c[u])

        for v in g[u]:
            dfs(v)

            # small-to-large merge
            if len(sets[u]) < len(sets[v]):
                sets[u], sets[v] = sets[v], sets[u]
                freq[u], freq[v] = freq[v], freq[u]

            # merge child v into u
            for col in sets[v]:
                if col not in sets[u]:
                    sets[u].add(col)
                freq[u][col] += 1

        # count merges needed at this node
        for col, cnt in freq[u].items():
            ans += max(0, cnt - 1)

    dfs(0)
    print(ans)

if __name__ == "__main__":
    solve()
```DFS自底向上构建子树信息。 这`sets[u]`结构跟踪下面存在哪些颜色`u`， 尽管`freq[u]`跟踪有多少子子树贡献每种颜色。 小到大的交换对于保持总复杂性接近线性至关重要，因为每种颜色在组之间仅以对数方式移动多次。 

一个微妙的点是我们只增加`freq[u][col]`每个子树每个颜色一次，而不是每个叶子一次。 这确保我们计算的是结构存在而不是多样性。 

## 工作示例

 考虑一棵树，其根有两个孩子，每个孩子都是一片叶子，有颜色`1`和`2`。 

我们首先处理左叶，然后处理右叶。 

| 节点| 子集合并 | 节点频率| 添加到答案 |
 | --- | --- | --- | --- |
 | 叶 1 | {} → {1} | {1:0 内部} | 0 |
 | 叶 2 | {} → {2} | {2:0 内部} | 0 |
 | 根 | {1} + {2} | {1:1, 2:1} | 0 |

 多个子子树中不会出现颜色，因此不会添加合并成本。 但是，我们总共仍然需要两次操作，每个叶子一次。 这是因为每个叶子都形成自己的单例组件。 

现在考虑一个具有三个子树的根，每个子树都包含一个颜色的叶子`1`。 

| 节点| 子集合并 | 节点频率| 添加到答案 |
 | --- | --- | --- | --- |
 | 叶 A | {1} | {1:0} | 0 |
 | 叶 B | {1} | {1:0} | 0 |
 | 叶C | {1} | {1:0} | 0 |
 | 根 | {1}+{1}+{1} | {1:3} | 2 |

 从根本上讲，颜色`1`出现在 3 个不同的子子树中，所以我们添加`3 - 1 = 2`。 这相当于将三个独立的组件合并为一个，需要两次操作。 

这表明该算法捕获相同颜色仅在较高节点处连接的情况。

 ## 边缘情况

 一个关键的边缘情况是所有叶子共享相同的颜色但分布在不同的分支上。 该算法正确识别内部节点处的多个子子树出现，并累积必要的合并，直到它们在更高级别上统一，从而确保仅保留一个有效操作。 

另一个重要的情况是每片叶子都有不同的颜色。 在这种情况下，任何节点上的频率计数都不会超过 1，因此不会对合并进行计数，并且答案等于叶子的数量，这反映出每个叶子必须独立处理，因为不可能进行分组。 

最后，倾斜的树就像链条一样。 在链中，每个节点只有一个子节点，因此兄弟节点之间不会发生合并。 频率图永远不会产生大于 1 的值，这与线性结构中不可能进行跨分支优化的事实相匹配。
