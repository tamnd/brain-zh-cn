 ## 解决方案
 ## 问题理解

 我们正在模拟一个动态聊天侧边栏，它始终首先显示最近的对话。 每次 Polycarp 向朋友发送消息时，该朋友的聊天就会成为最近活跃的，并移至列表的最顶部。 如果之前从未联系过该朋友，则会在顶部创建一个新的聊天条目。 如果该朋友已存在于列表中，则他们的现有条目将从当前位置删除并重新插入到前面。 

输入是按照消息发送顺序排列的一系列收件人姓名。 处理完所有操作后，输出是唯一好友的最终排序，从最近发送消息到最近最少发送消息。 

该约束允许最多 200,000 条消息，并且每个名称的长度最多为 10。这立即排除了任何重复扫描或移动每条消息的列表的解决方案。 从 Python 列表或数组中间删除元素的简单方法每次操作都会花费线性时间，这在最坏的情况下会导致二次行为。 

当重复向同一个朋友发送消息时，会出现微妙的边缘情况。 例如，如果输入是`a b a`，则处理后：

 - 之后`a`，列表是`a`- 后`b`，列表是`b a`- 后`a`，列表变为`a b`简单地在前面插入而不删除旧的出现的幼稚实现会错误地产生重复项，例如`a b a`，这违反了每个好友最多出现一次的要求。 

另一个边缘情况是所有消息都发送给同一个人。 该列表应始终以该人这一条目结束，无论发送了多少条消息。 

## 方法

 暴力模拟保留当前聊天列表。 对于每个传入的名称，它会扫描列表以检查该名称是否已存在。 如果是，它会删除该条目，然后在前面插入该名称。 如果没有，则只需将其插入到前面即可。 

这是正确的，因为它反映了所描述的确切行为。 然而，每个操作都需要一个线性搜索来查找位置，并需要另一个线性操作来删除或移动元素。 在所有消息都不同的最坏情况下，我们重复扫描不断增长的列表，导致大约 1 + 2 + 3 + ... + n 次操作，即 O(n²)。 当 n 达到 200,000 时，这太慢了。 

关键的观察是，我们只关心名称是否存在及其当前位置，并且我们需要在查找和重新排序时快速更新。 这建议将用于存在跟踪的基于散列的结构与支持前面快速插入的结构结合起来。 单独的集合无法保持顺序，单独的列表对于成员更新来说太慢。 解决方案是维护一个用于成员资格的集合和一个用于排序的类似双端队列的结构，但更简单的是，我们可以使用集合加上向后构建的列表进行模拟，或者更干净地，我们使用集合从末尾进行处理并构造一次最终顺序。 

一个特别清晰的见解是反向遍历消息。 当从最后一条消息扫描到第一条消息时，我们第一次遇到名称时，它一定是该聊天在答案中的最终位置。 任何较早发生的事件都是无关紧要的，因为在转发过程中较晚发生的事件会覆盖较早发生的事件。 因此，我们可以通过向后迭代并收集首次出现的名称来构建结果。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力模拟| O(n²) | O(n) | 太慢了 |
 | 使用 Set | 进行反向扫描 O(n) | O(n) | 已接受 |

 ## 算法演练

 我们处理从最后一个收件人到第一个收件人的消息列表，维护一组已包含在答案中的名称和一个最终顺序的列表。 

1. 从一个空集合和一个空列表开始。 

该集合会跟踪哪些好友已被列入最终排序。 
2. 从列表末尾向开头迭代消息收件人。 

这种逆转之所以有效，是因为在确定最终新近度时，后面的消息会覆盖前面的消息。 
3. 对于每个名称，检查它是否已在集合中。 

如果它已经存在，我们会跳过它，因为它的最终位置已经确定。 
4. 如果它不在集合中，我们将其添加到结果列表中并将其标记为已看到。 

我们第一次遇到逆序名称时对应于它在原始序列中最近出现的情况。 
5. 继续，直到处理完所有消息。 
6. 最后，反转构建的列表，因为我们按照最终出现的相反顺序收集名称。 

### 为什么它有效

 每个朋友的最终位置仅由他们在输入序列中最后出现的位置决定。 从末尾开始处理可以保证我们第一次遇到某个名称时恰好是它在前向时间中最后一次出现。 通过跳过所有后续的遭遇，我们确保每个朋友都只包含一次，并且按照正确的新近度顺序。 该集合强制执行唯一性，而反向遍历则对“最近获胜”规则进行编码。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    arr = [input().strip() for _ in range(n)]
    
    seen = set()
    res = []
    
    for i in range(n - 1, -1, -1):
        name = arr[i]
        if name in seen:
            continue
        seen.add(name)
        res.append(name)
    
    print("\n".join(res))

if __name__ == "__main__":
    solve()
```该解决方案首先读取所有名称，以便我们可以向后遍历它们。 这`seen`set 确保每个名称仅输出一次。 我们以反向最终顺序附加名称并直接按该顺序打印它们，因为如果我们正确解释构造，则不需要反转：我们第一次从后面看到名称是其最终位置，并且我们按该顺序附加。 

关键的实现细节是正确地剥离输入线； 如果没有它，尾随换行符将破坏集合中的相等检查。 此外，我们避免在前面进行任何列表删除或插入，这会降低性能。

 ## 边缘情况

 对于重复的相同消息，例如`a a a a`，算法从后面开始处理，立即取第一个`a`它遇到，然后跳过所有其他。 该集合可以防止重复，因此输出仍然是单个`a`，满足聊天唯一的要求。 

对于交替模式，例如`a b a b a`，反向扫描遇到`a`首先，然后`b`，并忽略较早发生的情况，产生`a b`。 这直接反映了只有每个好友的最后一条消息决定了最终的排序。 

对于严格增加的唯一名称，每个名称都按反向扫描顺序添加一次，产生相反的输入，这与所有不同聊天序列中的“最新位于顶部”行为一致。
