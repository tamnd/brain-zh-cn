 ## 解决方案
 ## 问题理解

 我们会得到一些卡片，并被要求重复构建尽可能高的“卡片金字塔”，移除使用过的卡片，并继续，直到无法构建更多的金字塔。 最终答案是在所有迭代中构建了多少金字塔。 

高度为 1 的金字塔是最简单的结构，消耗固定数量的牌。 对于更大的高度，结构是递归构建的：高度金字塔$h$坐落在一个底座的顶部，底座本身由较小的高度为 1 的金字塔和一排卡片组成，下面是一个高度为 1 的金字塔$h-1$。 重要的结果是每个高度对应于固定的卡片总数。 

这个过程本质上是贪婪的。 在每个阶段，我们总是构建适合剩余卡牌的最高金字塔。 减去其成本后，我们对剩余部分重复相同的决定。 

约束允许最多$t \le 1000$测试用例和总数$n$最多$10^9$。 这立即排除了任何逐卡构建金字塔的模拟，甚至在每个测试用例的所有可能高度上线性迭代的模拟。 尝试每个减法步骤的每个高度的解决方案可能会退化为粗略的$O(n)$在最坏的情况下，这是不可接受的。 

当剩余的牌很小时，就会出现微妙的边缘情况。 例如，如果$n = 1$，根本无法建造金字塔，答案一定是零。 假设每次迭代至少有一个构造的天真的贪婪实现可能会意外地尝试无效的高度计算或输入不正确的循环条件。 

如果我们尝试在不缓存的情况下从头开始重复重新计算金字塔大小，则会出现另一种失败情况。 由于金字塔成本呈二次方增长，因此重新计算它们可达$n$对于每个测试用例来说都太慢了。 

## 方法

 第一个自然尝试是直接模拟该过程。 对于给定的$n$，我们尝试所有可能的高度$h$，计算需要多少张卡，然后选择最大的有效卡。 减去该成本后，我们重复。 这在原则上是正确的，因为该问题明确定义了一个贪婪的构造过程。 

问题是效率。 如果我们从头开始重新计算每个高度的成本，则每次检查都是$O(h)$除非我们预先计算，并且我们可能会对每个测试用例的许多步骤重复此操作。 在最坏的情况下，$n$当重复构建小金字塔时，下降缓慢，导致多次迭代。 这将复杂性推向$O(n)$或者在测试用例中更糟。 

关键的观察是每个金字塔高度都有一个独立于上下文的固定成本。 一旦我们推导出高度所需卡片数量的公式$h$，我们可以预先计算所有可能的值$10^9$。 成本函数呈二次方增长，因此最大高度仅约为$2.5 \times 10^4$。 这使得存储所有可能的金字塔成本变得可行。 

一旦我们有了这个列表，每一步就变成：找到不超过剩余卡牌的最大金字塔成本，减去它，并增加计数。 因为我们总是选择尽可能大的高度，所以我们在每个阶段都有效地以最佳方式减少了问题。

 ## 算法演练

 首先，我们推导并预先计算每个高度的金字塔的成本。 让$f(h)$是需要的卡片数量。 根据构造规则，每次高度增加都会增加可预测数量的卡片，从而导致重复次数呈二次方增长。 我们预先计算所有值$f(h)$直到他们超过$10^9$。 

其次，我们将这些值存储在排序列表中，以便我们可以有效地查询适合给定剩余预算的最大金字塔。 

然后对于每个测试用例：

 1. 从给定数量的卡片开始$n$并将计数器设置为零。 
2. 虽然$n$至少是最小金字塔的成本，找到最大的$f(h)$这样$f(h) \le n$。 这对应于选择最高的有效金字塔。 
3. 减去该成本$n$。 
4. 由于我们已经成功构建了一座金字塔，因此将答案加一。 
5. 重复，直到无法形成金字塔。 

关键的推理步骤是为什么在每个阶段选择最大的可能金字塔是有效的。 任何较小的选择都会留下更多的牌，但由于下一步会独立于先前的选择而重复相同的贪婪规则，因此延迟使用牌不会增加金字塔的总体数量。 

### 为什么它有效

 每个金字塔结构都会消耗固定数量的卡牌，剩下的卡牌数量与我们如何达到该状态无关。 贪心选择总是最大化即时消耗，并且由于所有成本都是固定且不重叠的，因此该过程相当于从固定值的多重集中重复减去最大可能的元素。 任何后来的决定都不可能使先前可行的更大金字塔变得不可能，因为可行性仅取决于剩余总量，而不取决于结构。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

# Precompute pyramid costs
MAX_N = 10**9
costs = []

h = 1
while True:
    # f(h) = h(3h+1)/2
    c = h * (3 * h + 1) // 2
    if c > MAX_N:
        break
    costs.append(c)
    h += 1

def solve(n):
    ans = 0

    while n >= 2:
        # find largest cost <= n
        lo, hi = 0, len(costs) - 1
        best = -1

        while lo <= hi:
            mid = (lo + hi) // 2
            if costs[mid] <= n:
                best = mid
                lo = mid + 1
            else:
                hi = mid - 1

        if best == -1:
            break

        n -= costs[best]
        ans += 1

    return ans

def main():
    t = int(input())
    for _ in range(t):
        n = int(input())
        print(solve(n))

if __name__ == "__main__":
    main()
```核心实现细节是使用封闭式公式预先计算所有有效的金字塔成本$f(h) = \frac{h(3h+1)}{2}$。 这避免了查询期间的任何递归构造逻辑。 

在每个测试用例中，使用二分搜索来定位最大的可行金字塔成本。 这是必要的，因为当存在许多金字塔尺寸时，每次迭代线性扫描所有高度会降低性能。 

循环条件`n >= 2`反映了这样一个事实：高度为 1 的金字塔已经花费了 2 张牌，因此任何低于该高度的金字塔都会立即终止。

 ## 边缘情况

 当$n = 1$，算法立即退出，因为最小的金字塔需要 2 张卡。 循环条件阻止任何选择高度的尝试，并且答案保持为零。 

对于像这样的情况$n = 2$，仅高度 1 可用。 该算法选择成本 2，减去它，然后终止。 这证实了即使不存在更大的结构，二分搜索也能正确识别最小的有效成本。 

对于较大的值，多个相同高度的金字塔连续拟合，例如$n = 14$，算法重复选择高度 2。每次迭代都会根据更新的余数重新计算可行性，确保重复选择不依赖于先前的结构选择，而仅依赖于剩余预算。
