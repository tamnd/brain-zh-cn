 ## 解决方案
 ## 问题理解

 我们得到了一段长度的视频`c`秒。 观看的每一秒都需要消耗`a`数据单位，而互联网连接可以下载`b`每秒连续的数据单位。 

如果我们过早开始观看，播放可能会停止，因为在某些时候视频需要的数据量将超过目前已下载的数据量。 但是，即使在观看时下载也会继续，因此在播放期间可以部分重新填充缓冲区。 

目标是选择一个整数等待时间`t`在开始视频之前，这样一旦开始播放，它就可以连续播放直到结束，任何时候都不会超过可用的下载数据。 我们想要最小的这样`t`。 

关键的数量不仅仅是总数据量，而是已下载数据和观看消耗数据之间的运行平衡。 时`t0`，下载量为`b * t0`。 开始播放后的时间`t`，观看的部分是`t0 - t`，要求`a * (t0 - t)`数据。 条件变为：`b * t0 ≥ a * (t0 - t)`为所有人`t0`在`[t, t + c]`。 

约束很小，有`a, b, c ≤ 1000`，因此即使对可能的等待时间进行模拟也是可行的，但直接的连续时间检查仍然会变得不必要的复杂。 实际结构是线性的，因此我们期望有一个直接的公式。 

当连接相对于消耗速度非常快时，就会出现微妙的边缘情况。 例如，如果`b ≥ a`，那么流式传输始终是安全的，无需等待。 另一个边缘情况是当`b`仅略小于`a`，其中天真的“比较所需总量与下载总量”方法失败了，因为它忽略了中间缓冲区的耗尽。 

## 方法

 一个蛮力的想法是尝试每个可能的等待时间`t`，逐秒模拟播放，并检查缓冲区是否曾经变为负值。 对于每位候选人`t`，我们将跟踪下载的数据和消耗的数据`c`秒。 这是可行的，因为状态的演化是确定性的，但在最坏的情况下它太慢了：最多`O(c^2)`如果我们测试所有操作`t`并模拟`c`每次的步骤。 

关键的观察是约束`b * t0 ≥ a * (t0 - t)`是线性的`t0`，所以如果它在最坏点成立，那么它在所有地方都成立。 最糟糕的时刻发生在播放结束时`t0 = t + c`。 如果我们确保该点的不等式，早期时刻会自动满足它，因为双方都是线性增长的，并且消耗的斜率高于下载的斜率。 

所以我们只需要强制执行：`b * (t + c) ≥ a * c`展开给出：`b * t + b * c ≥ a * c`所以：`b * t ≥ c * (a - b)`自从`a > b`，右边为正，我们可以直接求解：`t ≥ ceil(c * (a - b) / b)`我们取满足此要求的最小整数。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力模拟 | O(c^2) | O(c^2) | O(1) | O(1) | 太慢了 |
 | 数学推导| O(1) | O(1) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们通过强制缓冲区即使在最糟糕的播放时刻也不会变得不足来得出最短等待时间。 

1. 计算播放期间每秒的净赤字率：`a - b`。 这是观看时消耗的数据量多于补充的数据量。 
2. 计算整个视频持续时间内累积的总赤字`c`，即`c * (a - b)`。 这表示在开始播放之前必须预加载多少缓冲区。 
3.观察等待`t`秒产生`b * t`缓冲区的单位。 
4. 要求预加载的缓冲区足以弥补全部不足：`b * t ≥ c * (a - b)`。 
5. 求解`t`使用带上限的整数除法：`t = (c * (a - b) + b - 1) // b`。 
6. 输出`t`。 

我们关注总赤字而不是每秒跟踪的原因是下载和消耗都是线性发展的。 一旦开始时的总缓冲区足以覆盖一段时间内的净损失，中间点就不会违反约束，因为系统以恒定速率发展。 

### 为什么它有效

 在播放过程中，缓冲区以恒定速率变化`b - a`，这是负数，因为`a > b`。 因此，缓冲区在播放的最后时刻被最小化。 确保缓冲区最后为非负数可以保证它永远不会更早变为负数。 这将一段时间内的连续可行性条件简化为一个端点处的单一不等式。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    a, b, c = map(int, input().split())
    
    deficit = c * (a - b)
    
    if deficit <= 0:
        print(0)
        return
    
    t = (deficit + b - 1) // b
    print(t)

if __name__ == "__main__":
    solve()
```该代码计算观看完整视频所需的额外数据总量，然后确定需要下载多少秒才能累积该数据量。 表达式`(deficit + b - 1) // b`是一个标准的上限划分，以确保我们不会低估等待时间。 

一个常见的陷阱是忘记整数除法必须向上舍入，因为等待时间略少于所需时间会导致播放接近结束时失败。

 ## 边缘情况

 当出现一种边缘情况时`b`非常接近`a`。 用于输入`a = 100, b = 99, c = 1`，赤字为`1`,所以等待`1`第二个提供了足够的缓冲区。 该算法计算`ceil(1 / 99) = 1`，符合要求。 

另一种边缘情况是视频很长但不平衡程度很小的情况，例如`a = 5, b = 4, c = 1000`。 总赤字变为`1000`，每一秒的等待只贡献`4`，所以结果是`250`。 天花板部门无需模拟即可精确处理此问题。 

最后一个边缘情况是`c = 1`。 那么条件就简化为确保一秒有足够的缓冲，公式就变成了`ceil((a - b) / b)`，这正确地符合“只有播放的第一秒很重要”的直觉。
