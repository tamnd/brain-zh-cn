 ## 解决方案
 ## 问题理解

 每个查询给出一个整数段，并询问该段中有多少个数字在变换函数 g 下共享特定值。 函数 g 接受一个数字并重复将其替换为其数字的乘积，直到该值变为单个数字。 最后一位数字是 g(x) 的结果。 例如，47 变为 4 × 7 = 28，然后 2 × 8 = 16，然后 1 × 6 = 6，因此 g(47) = 6。 

任务是回答最多 20 万个这样的查询，其中每个查询要求最大一百万个范围内的范围 [l, r] 和 1 到 9 之间的目标数字 k。每个查询的输出是该范围内乘法数字根等于 k 的整数的计数。 

这些约束立即迫使我们远离每个查询的模拟。 对于 [l, r] 中的每个 x 重新计算 g(x) 的简单方法在最坏的情况下每个查询需要多达一百万次操作，总共导致大约 2 × 10^11 次操作，这远远超出了可行的范围。 

一个微妙的点是，虽然 x 上升到 10^6，但 g(x) 总是很快崩溃到一位数。 这意味着只有九种可能的结果，因此可以预先计算跨域的重复结构。 

边缘情况是由数字乘法的行为引起的。 包含零的数字会立即归零，但 k 保证在 1 到 9 之间，因此这些值与计数无关。 另一个极端情况是像 10 或 1000 这样的数字，它们在乘法下会变成零，并且不能意外地计入任何正 k。 

## 方法

 暴力方法针对每个查询中的每个数字独立评估 g(x)。 对于每个 x，我们重复数字相乘，直到达到一位数字。 每个数字最多需要进行几位数字乘法，但对于所有查询来说，这会变得太大，因为每个查询最多可以跨越一百万个数字，并且有多达二十万个查询。 

关键的观察结果是变换 g(x) 仅取决于 x 而不是查询。 一旦我们知道一百万个 x 的 g(x)，每个查询就变成了对预先计算的数组的简单计数问题。 

我们预先计算一个数组，其中每个位置存储 g(x)。 然后，对于从 1 到 9 的每个数字 k，我们在所有 x 上构建一个前缀和，使得 g(x) = k。 之后，使用前缀差异在恒定时间内回答每个查询。 

这将重复的重新计算转换为单个预处理过程。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(Q · N · log N) | O(Q · N · log N) | O(1) | O(1) | 太慢了|
 | 最优（预计算+前缀和）| O(N log N + Q) | O(N log N + Q) | O(N) | 已接受 |

 ## 算法演练

 我们首先计算 1 到 1,000,000 之间每个 x 的 g(x)。 这是通过用数字的乘积重复替换数字直到它变成单个数字来完成的。 此步骤是必要的，因为一旦计算，所有查询都会重用这些值。 

接下来，我们存储有多少个数字产生从 1 到 9 的每个可能数字。我们不是只存储原始频率，而是构建前缀和，以便我们可以有效地回答范围查询。 

然后，我们将原始查询 [l, r, k] 转换为前缀和的差值：答案是 prefix[k][r] 减去 prefix[k][l − 1]。 这是可行的，因为前缀数组会累积从 1 到每个位置的计数。 

最后，我们打印每个查询的结果。 

### 为什么它有效

 正确性来自于 g(x) 是 x 的纯函数并且与查询结构无关的事实。 计算完成后，x 的每次出现都恰好为一位数字桶贡献一个单位。 前缀总和会累积保留这些计数，因此任何间隔查询都会减少为减去两个累积总数，而不会丢失或重复计算任何贡献。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MAXN = 10**6

def prod_digits(x):
    while x >= 10:
        p = 1
        while x > 0:
            x, d = divmod(x, 10)
            p *= d
        x = p
    return x

g = [0] * (MAXN + 1)
for i in range(1, MAXN + 1):
    g[i] = prod_digits(i)

prefix = [[0] * (MAXN + 1) for _ in range(10)]

for i in range(1, MAXN + 1):
    val = g[i]
    for k in range(1, 10):
        prefix[k][i] = prefix[k][i - 1]
    prefix[val][i] += 1

q = int(input())
out = []

for _ in range(q):
    l, r, k = map(int, input().split())
    out.append(str(prefix[k][r] - prefix[k][l - 1]))

print("\n".join(out))
```该解决方案首先定义乘法数字约简函数。 它会重复将数字相乘，直到数字变成一位数，这样即使对于经过多个中间阶段（例如 39 或 88）的输入也能确保正确性。 

然后我们预先计算一百万以内的所有值的 g(x)。 这是昂贵的部分，但只完成一次。 

之后，我们为从 1 到 9 的每个数字构建前缀数组。每个前缀数组跟踪索引 i 之前有多少个数字具有给定的 g 值。 

使用前缀范围的减法来回答查询，这避免了任何重新计算。 

## 工作示例

 考虑一个小的说明性案例，我们在预处理 g(x) 后查询几个区间。 

让我们在小范围内追踪一个简化版本：

 | 我| g(i) | g(i) | 前缀[4][i] | 前缀[6][i] |
 | ---| ---| ---| ---|
 | 1 | 1 | 0 | 0 |
 | 2 | 2 | 0 | 0 |
 | 3 | 3 | 0 | 0 |
 | 4 | 4 | 1 | 0 |
 | 5 | 5 | 1 | 0 |
 | 6 | 6 | 1 | 1 |
 | 7 | 7 | 1 | 1 |
 | 8 | 8 | 1 | 1 |
 | 9 | 9 | 1 | 1 |

 对于查询 [4, 7, 6]，我们计算 prefix[6][7] − prefix[6][3] = 1 − 0 = 1。 

现在考虑一个查询 [1, 9, 4]，我们计算 prefix[4][9] − prefix[4][0] = 1 − 0 = 1，因为在此范围内只有数字 4 映射到 4。 

这些跟踪表明，一旦构建了前缀数组，每个查询就会减少到两次表查找和一次减法。 

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 | O(N log N + Q) | O(N log N + Q) | 1e6 以内的每个数字都通过数字乘积进行归约，则每次查询的时间复杂度为 O(1) |
 | 空间| O(N) | 前缀数组存储每个数字的累积计数 |

 预处理完全符合限制，因为超过一百万个整数的 N log N 运算在 Python 中是可以接受的，并且查询会在恒定时间内得到答复。

 ## 边缘情况

 一个关键的边缘情况是包含零的数字。 例如，x = 105 在第一个数字乘积后立即变为 0。 由于 k 始终介于 1 和 9 之间，因此此类数字永远不会对任何答案产生影响。 预处理确保它们存储在 g(x) = 0 下，并且不会进入任何 k ≥ 1 的 prefix[k] 数组。 

另一种情况是重复的数字塌陷。 像 39 这样的数字会变成 27，然后是 14，然后是 4。如果中间步骤被错误地提前截断，最终的映射将会是错误的。 循环结构确保完全减少，直到保留一个数字，因此无论所需的步骤数如何，最终的分类都是稳定的。 

最后，像 [1, 1e6] 这样的统一范围测试前缀和是否在整个域中正确累积。 由于每个 x 都被分配了一个 g(x)，因此 k = 1 到 9 的总计数与 N 一致，这可以作为预处理正确性的健全性检查。
