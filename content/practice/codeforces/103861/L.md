 ## 解决方案
 ## 问题理解

 我们得到一个长度为 n 的数组，它从零开始。 我们不是直接观察数组，而是仅通过二进制字符串得知最终状态：每个位置告诉我们该索引处的最终值是否为零。 创建数组的隐藏过程是一系列 Fenwick 树点更新，其中位置 p 的每次更新都会向 p 添加一些实际值 v，然后通过索引 p、p + lowbit(p)、p + lowbit(p + lowbit(p)) 等向上传播。 

关键的困难在于一次更新不仅仅影响一个位置。 它影响由索引的二进制表示形式确定的结构化索引集。 任务是仅根据最终的零/非零模式重建可能产生它的此类更新的最小数量。 

输入规模很大，最多有10^5个测试用例，总共n最多10^6。 这迫使每个测试用例都有一个线性或近线性的解决方案。 任何显式模拟更新或尝试搜索操作子集的操作都是立即不可能的，因为即使每个测试用例的 O(n log n) 总体来说也太慢了。 

一个微妙的一点是，值是实数，因此可以取消。 即使位置受到更新的影响，只要值完全取消，位置也可能变为零。 但是，由于我们正在最小化更新次数，因此我们只关心是否可以为更新分配值，以便最终的零/非零模式匹配。 

一个天真的错误是认为每个标记为 1 的位置都需要在该位置进行独立更新。 这会立即失败，因为一次更新可能会影响许多位置。 例如，如果我们更新位置 1，它会影响所有位置。 因此，单个操作可以满足许多 1，但也可能干扰 0，这使得结构变得不平凡。 

另一个误导性的情况是出现交替模式时。 例如，像 101010 这样的字符串可能会诱使人们贪婪地在每 1 处进行更新。但是重叠传播可以使更少的更新就足够了，或者相反，由于零的约束，强制更多的更新。 

## 方法

 暴力视图会尝试将每个所需的更新分配给某个位置和值，然后模拟 Fenwick 传播并检查生成的零/非零模式是否与目标字符串匹配。 这变成了一个组合构造问题：我们有效地选择了一组起始位置和实际值。 即使我们限制自己选择起始位置，也有 2^n 种可能性，并且每个候选都需要 O(n log n) 模拟。 这是完全不可行的。 

关键的见解是扭转观点。 我们考虑的不是更新产生价值，而是需要多少独立的“影响源”来解释最终的模式。 每次更新都会引入遵循 Fenwick 树结构的传播路径，该结构与索引的二进制分解紧密相关。 

从位置 p 开始的 Fenwick 更新对通过重复添加低位段定义的范围内的所有索引做出贡献。 这种结构意味着每个索引 i 都受到在所有位置 p 开始的更新的影响，使得 p 位于隐式 Fenwick 树中 i 的特定祖先链上。 

现在考虑从左到右处理索引。 在位置 i，如果我们看到 1，则它必须从到达 i 的某个更新中接收到至少一个非零贡献。 如果我们确保始终“激活”最少数量的更新来覆盖无法解释的新 1，那么我们就可以有效地维持所需的活跃 Fenwick 传播链数量。

关键的观察结果是芬威克结构的行为就像二元提升区间的森林。 每个索引 i 引入了一个要求，如果先前的索引被清零约束，则该要求始终无法满足。 零充当阻止者：它们禁止任何活动的更新链覆盖它们，除非安排取消，但由于我们正在最大限度地减少更新数量，因此我们更愿意避免引入不必要的重叠链。 

这减少了跟踪有多少独立段必须从我们遇到尚未被先前有效传播结构“覆盖”的 1 的位置开始。 Fenwick 结构确保覆盖依赖性与低位前缀分解一致，因此问题归结为计算在相对于索引的二进制分解扫描字符串时需要多少个新的独立开始。 

简化依赖结构后，最佳解决方案变成贪婪线性扫描，其中我们维护解释所有 1 所需的最小活动更新源集，同时尊重在不引入额外独立更新的情况下不能强制零变为非零。

 ## 算法演练

 我们从左到右处理字符串，同时维护有多少独立的 Fenwick 更新源当前处于“活动”状态并且仍然能够为未来的位置做出贡献。 

1. 我们初始化一个计数器，表示当前有多少个更新链可以解释未来的 1。 
2. 当我们到达值为 1 的位置 i 时，我们首先尝试使用现有的活动链。 如果存在，我们分配它来覆盖这个位置，因为重用结构永远不会增加操作数量。 
3. 如果没有可用的活动链，我们必须在位置 i 开始新的更新。 这对应于对 Fenwick 更新过程的一次调用，因为之前没有任何操作可以解释这一点 1。 
4. 当我们遇到 0 时，我们必须确保不会意外地保留不必要的主动覆盖，从而迫使该位置变为非零。 如果存在活动链，我们会通过“结束”或以与 Fenwick 传播边界一致的方式消耗覆盖范围来解决此问题，从而有效减少可用链的数量。 
5. 我们将继续这个过程直到最后，每次我们被迫引入新的更新源时都会进行总结。 

关键的困难是正确解释“主动链”。 用 Fenwick 的术语来说，每次更新都定义了与二进制分解一致的传播模式。 一条链仅在没有结构冲突（强制归零）阻止的情况下跨段保持可用。 

### 为什么它有效

 每次更新都可以解释为在 Fenwick 晶格中创建一个独立的传播源。 任何 1 都必须由至少一个来源来解释，因此答案是一个下限。 零限制了哪些索引集可以共享源，因为如果单个更新将不可避免地导致零位置，则在不引入取消复杂性的情况下无法重用该结构。 由于取消不会减少最佳构造中所需源的数量，因此我们可以将每个必要源视为结构上独立的。 贪婪扫描构建了这些源的最大重用，确保每个 1 都被覆盖，同时仅当没有现有结构可以合法扩展到当前索引时才引入新源。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        n = int(input())
        s = input().strip()

        ans = 0
        active = 0

        for ch in s:
            if ch == '1':
                if active > 0:
                    active -= 1
                else:
                    ans += 1
                    active += 1
            else:
                active = 0

        print(ans)

if __name__ == "__main__":
    solve()
```该代码维护两个数量。 答案是计算我们必须创建多少个新的 Fenwick 更新源。 这`active`变量表示有多少源仍可用于覆盖即将到来的 1。 

当我们看到 1 时，如果可能的话，我们会重用现有的源，否则我们会创建一个新的源。 当我们看到 0 时，我们会重置活动重用容量，因为覆盖该位置的任何正在进行的传播与在最终模式中保持该索引为零的约束不兼容。 

关键的实现细节是遇到零时立即发生重置。 这确保了我们永远不会在禁止的位置上错误地重用源。

 ## 边缘情况

 最小的边缘情况是单个零。 该算法立即产生零操作，因为不需要更新来保持数组为零，并且不会创建活动链。 

一开始的单个更新源会创建一个更新源，之后不会再进行主动重用，这符合至少需要一次 Fenwick 更新才能产生任何非零值的想法。 

像这样的图案`101010`很重要，因为它交替强制重置和新源。 该算法在每个零处重置活动容量，这可以防止跨间隙的非法重用，并且零后出现的每个孤立的 1 都会启动一个新的源。 跟踪它可以确认，每次从零到一的转换都会增加答案，除非存在一条活跃的链，而在该模型下，该链永远不会跨越零而生存。
