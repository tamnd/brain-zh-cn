 ## 解决方案
 ## 问题理解

 我们得到一个无向图，其中顶点 1 是起始中心，顶点 n 是我们最终必须清除的最终目标。 除 1 之外的每个顶点都包含一个具有初始强度的 Boss。 玩家也有力量值，并随着时间的推移而进化。 

时间在离散的日子里进展。 每天开始时，剩余的每个 Boss 都会将其强度增加固定数量 B。之后，玩家恰好选择从顶点 1 可以到达的一个顶点并与其 Boss 战斗。 如果玩家当前的强度严格大于该 Boss，则该 Boss 会被永久移除，玩家获得 A 强度。 

行动受到活着的老板的限制。 仅当存在一条所有中间顶点都没有活动 boss 的路径时，才可以从顶点 1 访问顶点。 这意味着活着的老板会充当阻碍者，除了我们试图达到的战斗终点之外。 

任务是确定击败顶点 n 的 Boss 所需的最少天数，或者报告不可能。 

这些约束表明需要一个高度优化的解决方案。 测试用例中的顶点和边的总数可以达到 10^6，因此任何以简单方式重复访问边或重新计算可达性的方法都会太慢。 任何 n 的二次方甚至每个状态重复的 BFS 都会被立即排除。 

一个微妙的困难来自于时间和结构之间的相互作用。 一个节点可能在图连通性方面是可达的，但由于强度不足而仍然无法被杀死。 相反，一个节点可能足够弱但无法到达，因为单个活动顶点阻塞了所有路径。 

一些失败案例凸显了其中的陷阱。 

考虑一个线图 1 - 2 - 3 - 4，其中只有顶点 2 阻挡了通往其之外所有内容的路径。 即使顶点 4 足够弱，可以提前被杀死，但在顶点 2 被移除之前，它也无法到达。 仅检查强度而忽略可达性的幼稚方法将错误地允许跳过 2。 

另一种失败模式来自时机。 假设一个顶点在第一天还算勉强太强，但在第二天由于玩家按 A 缩放而变得可杀死。如果 B 很大，延迟也可能比玩家更快地增加 Boss 强度，因此假设“等待总是更好”或“尽早总是更好”的不正确的贪婪策略将会失败。 

正确的解决方案必须同时跟踪动态删除的顶点下的可达性以及节点最早被击败的日期。 

## 方法

 暴力模拟会尝试日复一日地处理游戏。 每天，它都会仅使用已经死亡的顶点作为中间节点来重新计算从 1 可以到达哪些顶点，然后检查所有可到达的顶点以查看哪些可以被杀死。 这需要每天进行 BFS 或 DFS，最多需要 n 天，最坏情况下的复杂度为 O(n(n + m))，这远远超出了限制。 

关键的观察是，唯一有意义的状态不是活的和死的 Boss 的完整配置，而是通过已清除的节点可到达的顶点集以及当天的索引。 一旦顶点变得可到达，它就永远不会再次变得不可到达，因为删除顶点只会打开路径。 

这表明将可达性视为从节点 1 开始不断增长的边界，其中只有已清除的顶点才能用作内部路由点。 每次我们杀死一个顶点时，它都会永久扩展可遍历的区域。

独立地，力量条件仅取决于过去了多少天，因为玩家强度和老板强度都随着击杀数量（每天一个）线性变化。 如果我们将 k 表示为已执行的击杀次数，则第二天玩家的强度为 l1 + kA，而 Boss v 的强度为 l_v + kB。 因此，每个顶点在 k 上都有一个阈值，超过该阈值它就变得可杀死，但只有在它变得可到达时才可以考虑它。 

这自然会导致顶点上的最短路径样式过程，其中“距离”是顶点可以被击败的最早一天（或击杀数），并且当顶点可以从已清除的区域到达时发生转换。 

每次我们删除一个顶点时，我们都会将其邻居解锁为可能可达的，并计算它们可以被杀死的最早可能的日期。 这形成了类似 Dijkstra 的扩展，其中优先级是最早可行的杀戮日。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 使用 BFS 进行日常模拟 | O(n(n + m)) | O(n(n + m)) | O(n + m) | 太慢了 |
 | Dijkstra 超过可达状态 | O((n + m) log n) | O((n + m) log n) | O(n + m) | 已接受 |

 ## 算法演练

 ### 1. 根据终止顺序对流程进行建模

 我们将这个过程解释为选择杀死顶点的顺序。 如果某个顶点作为第 k 个动作被杀死（第一次杀死从 k = 0 开始），则当天为 k + 1，玩家强度为 l1 + kA，每个 Boss 的强度为 l_i + kB。 

这将时间转换为单个整数参数 k，从而简化了比较。 

### 2.推导杀死顶点的可行性条件

 当且仅当严格不等式成立时，顶点 v 才能在步骤 k 处被杀死：

 l1 + kA > l_v + kB。 

重新排列给出：

 l1 - l_v > k(B - A)。 

这告诉我们 v 变得可杀死的最早 k，前提是它是可到达的。 

成长的方向很重要。 如果 A > B，玩家就会随着时间的推移获得优势，因此等待会有所帮助。 如果 A ≤ B，延迟永远不会提高可行性，只会使杀戮变得更加困难或不变。 

### 3. 保持从顶点 1 开始的动态可达性

 只有通过已杀死的顶点连接到 1 的顶点才是可到达的。 活动顶点会阻止遍历。 

因此，我们维护一组已被杀死的“激活”顶点。 由此，我们可以扩展到尚未被杀的邻居。 仅当至少一个邻居已被激活时，顶点才成为候选顶点。 

这确保我们只考虑演化图中当前可到达的顶点。 

### 4. 在最早的终止时间上使用优先级队列

 我们维护一个最著名的值 dist[v]，即 v 在可达状态下可以被杀死的最小 k。 我们从已在 k = 0 处激活的顶点 1 开始。 

对于每个激活的顶点 u，我们扫描其尚未激活的邻居 v。 如果在步骤 k 发现 v，我们根据不等式计算 v 的最早可行的终止步骤 k'。 我们还确保 k' ≥ k 因为我们无法在它变得可达之前杀死它。 

我们将 (k', v) 推入优先级队列，始终扩展具有最小 k' 的顶点。 

### 5. 处理顶点以增加终止时间

 我们反复从优先级队列中提取具有最小 k 的顶点。 如果它已经最终确定，我们就跳过它。 否则，我们将其标记为已杀，将当前 k 设置为该值，并隐式更新玩家强度。 

然后，我们放松所有邻居，由于较早的可达性，可能会减少他们最早的可行杀戮时间。 

### 为什么它有效

在任何时候，该算法都确保优先级队列中的每个顶点都可以通过已被杀死的顶点到达，并且其存储的值是在当前进程下可以被杀死的最早步骤。 因为当我们添加杀死的顶点时，可达性和可行性都只会单调提高，因此使用最小 k 提取的任何顶点都无法在以后得到改进。 这与 Dijkstra 算法的正确性原理完全相同：一旦选择了最小可行状态，未来的路径就无法产生更小的有效杀伤时间。 

## Python 解决方案```python
import sys
import heapq

input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        n, m, A, B = map(int, input().split())
        
        g = [[] for _ in range(n + 1)]
        for _ in range(m):
            u, v = map(int, input().split())
            g[u].append(v)
            g[v].append(u)
        
        l = [0] + list(map(int, input().split()))
        
        INF = 10**30
        
        dist = [INF] * (n + 1)
        used = [False] * (n + 1)
        
        # node 1 is already reachable and "killed" at time 0
        dist[1] = 0
        pq = [(0, 1)]
        
        def earliest_k(v, k):
            # check minimal k' >= k such that:
            # l1 + k'*A > l[v] + k'*B
            # (A - B)*k' > l[v] - l1
            
            if A == B:
                if l[1] > l[v]:
                    return k
                return INF
            
            if A > B:
                # k' > (l[v] - l1) / (A - B)
                need = l[v] - l[1]
                if need < 0:
                    k0 = 0
                else:
                    k0 = need // (A - B) + 1
                return max(k, k0)
            
            # A < B: only possible if already satisfied at current k
            if l[1] + k * A > l[v] + k * B:
                return k
            return INF
        
        while pq:
            k, u = heapq.heappop(pq)
            if used[u]:
                continue
            used[u] = True
            
            for v in g[u]:
                if used[v]:
                    continue
                nk = earliest_k(v, k)
                if nk < dist[v]:
                    dist[v] = nk
                    heapq.heappush(pq, (nk, v))
        
        print(-1 if dist[n] == INF else dist[n])

if __name__ == "__main__":
    solve()
```该代码构建图并在顶点上运行 Dijkstra 风格的过程，其中优先级是顶点可以被击败同时仍可从已处理的节点访问的最早一天。 功能`earliest_k`编码了玩家和 Boss 强度之间依赖于时间的不平等，并尊重这样一个事实：等待可能会有所帮助或有害，具体取决于 A 是否超过 B。 

优先级队列保证顶点按照其最佳杀死日的升序最终确定，并且`used`数组确保每个顶点都被处理一次。

 ## 边缘情况

 当 A ≤ B 时，就会出现关键的边缘情况。在这种情况下，每次延迟要么毫无作用，要么使未来的战斗变得更加困难。 该算法通过仅在顶点可到达且已可杀死的确切时刻接受该顶点来处理此问题。 如果它在发现后不能立即杀死，它就会被有效地丢弃。 

另一个边缘情况是当顶点 n 由于一长串活着的 Boss 而最初无法到达时。 该算法正确地拒绝放松第一个被阻止的顶点之外的节点，直到该阻止者被明确杀死，因为可达性仅通过激活的顶点传播。 

最后一个微妙的情况是，当一个顶点很早就变得可到达，但只有在 k 增加时才变得可杀死。 优先级队列确保它不会过早被选择，而是等待，直到其计算出的最早可行的 k 成为所有候选者中的最小值。
