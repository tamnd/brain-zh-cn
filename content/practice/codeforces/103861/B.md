 ## 解决方案
 ## 问题理解

 我们得到一个数字字符串并要求考虑它的每个子字符串。 对于每个子字符串，我们研究将其拆分为六个连续的非空部分的方法。 当第一、第二和第五部分是相同的字符串，并且第三和第六部分也是相同的字符串，而第四部分不受限制时，分割被认为是有效的。 

对于每个子串，我们计算存在多少个这样的有效六部分分割，这个计数称为该子串的“美度”。 最终任务是将输入字符串的所有子字符串的美丽值相加。 

因此，问题不仅仅是对一个字符串内的模式进行计数，而是对所有子字符串间隔进行聚合计数，其中每个间隔本身允许多个内部分区配置。 

这些约束允许每个测试用例最多 5000 个字符，总长度最多 30000。这立即排除了任何尝试枚举子字符串然后独立重新检查所有分区的解决方案。 检查所有子串然后尝试每个子串内的所有分割点的简单方法会导致三次或更糟糕的行为，这会太慢。 

一个更微妙的问题是分区在子字符串之间严重重叠。 重复片段的固定匹配可以同时产生许多子串。 任何正确的解决方案都必须重用计算，而不是从头开始为每个子字符串重新计算模式匹配。 

值得强调的一个小边缘情况是所有字符都不同。 在这种情况下，不会出现重复的段相等，因此答案为零。 另一种情况是当字符串恒定时，重复的段相等性变得丰富，并且简单的计数很容易通过在不同子字符串之间重复贡献而过度计数。 

## 方法

 蛮力解释很简单。 对于每个子串$t[l..r]$，我们想尽办法选择五个切割位置，形成六个部分。 对于每个这样的分割，我们检查是否$s_1 = s_2 = s_5$和$s_3 = s_6$。 每次检查都涉及比较子字符串，这可能会花费段长度的线性时间。 

即使使用散列优化子串比较，在长度子串内选取五个剪切位置的方法数量$m$是$O(m^5)$，并且有$O(n^2)$子串。 这已经使得暴力破解在$n = 5000$。 

关键的观察是，有效分区的结构几乎完全是通过选择重复块的端点来确定的。 一旦我们修复了第一次出现的位置$s_1$和第二个块$s_3$，其余部分由等式强制。 换句话说，我们可以考虑两个模式块，而不是六个片段：一个重复三次，一个重复两次，它们之间有一个自由分隔符。 

这将问题转化为计算具有固定偏移量的匹配子串对的出现次数，然后聚合它们在所有可能的子串边界上的贡献。 这自然是通过预先计算子串的相等性然后扫描可能的段长度和起始位置来处理的。 

提高效率的标准方法是固定重复块的长度。 假设重复块的长度$a = |s_1| = |s_2| = |s_5|$和第二个重复块$b = |s_3| = |s_6|$。 那么每个有效的结构对应选择一个起始索引$i$，并确保子字符串具有以下模式：$a, a, b, x, a, b$,

 这减少了检查固定偏移处子字符串的相等性$i$。 

然后我们对每一对进行计数$(a, b)$, 有多少个位置$i$满足：$t[i:i+a] = t[i+a:i+2a] = t[i+3a+b:i+3a+b+a]$和$t[i+2a:i+2a+b] = t[i+3a:i+3a+b]$。 

这成为子字符串相等查询的问题，可以使用滚动哈希在恒定时间内回答，然后对所有有效的求和$(a, b, i)$。 

最后的优化是绑定枚举：对于每个起始索引，我们只考虑$a$和$b$使整个结构适合绳子。 这导致$O(n^2)$具有恒定时间检查的枚举。

 ## 算法演练

 我们使用滚动哈希预处理来实现快速子字符串相等性检查。 

1. 预先计算整个字符串的前缀哈希值和基数幂。 这使我们能够在恒定时间内比较任意两个子字符串。 这很重要的原因是每个有效的配置仅取决于多个子字符串的相等性，并且重复的直接比较将主导运行时。 
2.修复起始索引$i$这将作为第一段的开始$s_1$。 每个有效分区都锚定在某个这样的位置，因此迭代所有分区$i$确保完整性。 
3. 对于每个$i$，迭代可能的长度$a$重复块的$s_1, s_2, s_5$。 约束条件是$i + 2a$必须仍在范围内，因为在第二个块开始之前我们至少需要该块结构的三个副本。 
4. 对于每个选择$a$，迭代可能的长度$b$对于第二个重复块$s_3, s_6$。 我们确保长度的完整结构$3a + 2b + 1$（包括免费部分$s_4$) 适合字符串内部。 
5. 对于每个三元组$(i, a, b)$，使用哈希比较验证相等条件：首先检查三个$a$-length 段相等，然后检查两个$b$- 长度段相等。 由于预处理，每次检查都是恒定时间。 
6. 如果两个条件都成立，则将全局答案加 1。每个有效配置恰好对应于一个子串的一个有效分区，因此不需要进一步调整。 

### 为什么它有效

 该算法枚举了起始点和块长度的每种可能选择，并且对于每种选择，它都会精确检查定义有效分区的约束。 因为每个有效分区都是由其起始索引和块大小唯一确定的$a$和$b$，有效分区和计数配置之间存在一对一的映射。 散列步骤确保相等性检查的正确性而不会产生歧义，并且有界循环确保所有可行的配置都被访问一次。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    s = input().strip()
    n = len(s)
    if n < 6:
        print(0)
        return

    base = 91138233
    mod = (1 << 61) - 1

    pref = [0] * (n + 1)
    power = [1] * (n + 1)

    for i in range(n):
        pref[i + 1] = (pref[i] * base + (ord(s[i]) - 48)) & mod
        power[i + 1] = (power[i] * base) & mod

    def get_hash(l, r):
        return (pref[r] - (pref[l] * power[r - l]) & mod) & mod

    ans = 0

    for i in range(n):
        max_a = (n - i) // 3
        for a in range(1, max_a + 1):
            # need space for 3a + at least 2b + 1
            max_b = (n - i - 3 * a) // 2
            if max_b <= 0:
                break
            h1 = get_hash(i, i + a)
            h2 = get_hash(i + a, i + 2 * a)
            h3 = get_hash(i + 3 * a, i + 3 * a + a)
            if not (h1 == h2 == h3):
                continue

            for b in range(1, max_b + 1):
                h4 = get_hash(i + 2 * a, i + 2 * a + b)
                h5 = get_hash(i + 3 * a, i + 3 * a + b)
                if h4 == h5:
                    ans += 1

    print(ans)

if __name__ == "__main__":
    solve()
```该实现依赖于前缀哈希来将子字符串比较减少到恒定时间。 外循环固定起始索引，然后第一个内循环固定重复块长度$a$。 一旦三$a$- 段被验证相等，我们只扫描$b$检查两者是否相等的同时$b$- 段。 

一个微妙的细节是检查的顺序：我们首先验证涉及三个的更严格的条件$a$- 迭代之前的段$b$。 当结构已经无效时，这可以避免不必要的内循环工作。 另一个重要的一点是仔细的边界控制，因为两者$3a$和$3a + 2b$必须保持在字符串范围内。

 ## 边缘情况

 一种关键的边缘情况是字符串长度恰好为 6。在这种情况下，每个有效配置只有一种可能的分区结构。 该算法正确地处理了这个问题，因为所有循环都限制$a \ge 1$,$b \ge 1$，并自动防止超出字符串边界的溢出。 

另一个边缘情况是具有大量相同字符的字符串。 在这里，从相邻索引开始存在许多重叠分区，但每个分区都是独立计数的，因为算法锚定在每个有效的起始位置并且不合并配置。 

最后的边缘情况是当没有有效的三重相等时$s_1, s_2, s_5$存在。 在这种情况下，算法会尽早修剪$a$-级别并避免不必要的$b$-完全迭代，同时保证正确性和效率。
