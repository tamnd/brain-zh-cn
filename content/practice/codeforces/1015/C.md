 ## 解决方案
 ## 问题理解

 我们得到了一组歌曲，每首歌曲都有一个原始大小和一个较小的大小（如果我们选择压缩它）。 所有歌曲必须复制到总容量固定的闪存驱动器上。 唯一可用的决定是压缩哪些歌曲。 压缩一首歌曲会用较小的歌曲替换其大小，我们希望确保最终的总大小不超过容量。 

目标不是决定是否可以容纳歌曲，因为这总是可以通过压缩所有内容并检查总数来回答。 真正的任务是尽量减少我们选择压缩的歌曲数量，同时仍然使总大小符合限制。 

关键的困难来自于压缩歌曲会根据歌曲的不同带来不同的好处。 有些歌曲压缩后可以节省大量空间，而另一些歌曲则节省很少。 该决策是关于选择项目的子集，其中每个选择都会将总大小减少已知量。 

这些约束促使我们寻求线性或近线性的解决方案。 对于多达 100,000 首歌曲，任何尝试所有子集的解决方案都立即不可能，因为它会呈指数级增长。 即使是二次解也会太慢。 这强烈表明我们需要一种基于排序的贪婪策略。 

当压缩所有歌曲还不够时，就会出现微妙的边缘情况。 在这种情况下，没有任何子集可以提供帮助，因为压缩只会减小大小。 另一个不太明显的陷阱是假设我们应该始终压缩具有最大原始大小的歌曲。 这是不正确的，因为重要的不是绝对大小，而是通过压缩实现的减少。 

## 方法

 直接的暴力方法将尝试歌曲的每个子集，在压缩子集中的歌曲后计算结果的总大小，并采用适合容量的最小子集大小。 从概念上讲，这是可行的，因为它探索了所有可能性，但本质上是指数性的，需要检查 2^n 个子集。 当n达到100,000时，这是完全不可行的。 

关键的观察结果是，压缩歌曲总是将总大小减少固定量：原始大小和压缩大小之间的差异。 如果我们将这种减少定义为增益，那么每首压缩歌曲都会独立地减少总和。 问题变成选择最少数量的项目，其总增益至少超过容量。 

这变成了一个经典的贪婪选择问题。 如果我们想使用尽可能少的物品来达到减少目标，我们应该首先选择提供最大减少量的物品。 按增益降序对歌曲进行排序可确保每次选择的压缩在每次操作节省的空间方面尽可能高效。 

我们首先计算没有压缩的总大小。 如果已经在容量范围内，则不需要压缩。 否则我们计算需要减少多少。 然后，我们根据压缩效益对所有歌曲进行排序，并贪婪地选取最大的歌曲，直到满足要求。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(2^n) | O(2^n) | O(n) | 太慢了 |
 | 最优（贪心排序）| O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 令 Total 为所有原始歌曲大小的总和。 对于每首歌曲，将增益定义为 a_i - b_i，这是压缩后节省的空间量。

1. 计算所有未压缩歌曲的总大小。 这给出了做出任何决策之前的基线存储需求。 
2. 如果total已经小于或等于m，则立即输出0，因为不需要压缩。 
3. 计算多余的空间，定义为我们必须移除多少空间才能装入驱动器。 
4. 对于每首歌曲，计算其增益值并将其与歌曲一起存储。 
5. 按增益降序对所有歌曲进行排序。 这确保我们始终首先考虑最有效的压缩。 
6. 迭代排序列表，从所需的多余量中减去每个选定的增益，并计算我们压缩的歌曲数量。 
7. 一旦剩余的多余量变为零或负数，则停止并输出压缩歌曲的数量。 

我们提前停止的原因是，一旦达到所需的减少量，添加更多压缩只会增加计数，而不会提高可行性。 

### 为什么它有效

 正确性依赖于这样一个事实：每次压缩都是独立的，并且对总减少量有附加贡献。 由于每首压缩歌曲都会将总和减少固定量，因此问题相当于选择权重总和至少达到目标的最小数量的元素。 在这种设置中，按重量降序排序是最佳的，因为用较大的未使用增益替换任何选定的较小增益只能减少或维持所需的项目数量。 这个交换参数保证了最大收益的贪婪前缀是最优的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

n, m = map(int, input().split())

songs = []
total = 0

for _ in range(n):
    a, b = map(int, input().split())
    total += a
    songs.append(a - b)

if total <= m:
    print(0)
    sys.exit()

songs.sort(reverse=True)

need = total - m
cnt = 0

for gain in songs:
    need -= gain
    cnt += 1
    if need <= 0:
        print(cnt)
        break
```该解决方案首先读取所有歌曲大小并计算总大小和压缩每首歌曲的好处。 这种分离很重要，因为我们再也不需要直接压缩大小，只需要它们的差异。 

在检查了不需要压缩的简单情况后，我们计算必须释放多少空间。 按降序对增益进行排序可确保我们优先考虑最有效的压缩。 

该循环保持运行减少并计算压缩了多少首歌曲。 一旦达到所需的减少量，我们就会立即停止，以确保最少的数量。 

一个常见的错误是忘记按增益排序，而是按原始大小或压缩大小排序，这与实际的优化目标不符。

import sys, io

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    import sys
    input = sys.stdin.readline

    n, m = map(int, input().split())
    songs = []
    total = 0

    for _ in range(n):
        a, b = map(int, input().split())
        total += a
        songs.append(a - b)

        return "0"

    songs.sort(reverse=True)

    need = total - m
    cnt = 0

    for g in songs:
        need -= g
        cnt += 1
        if need <= 0:
            return str(cnt)

    return "-1"

 ## 边缘情况

 当压缩所有歌曲仍无法满足容量时会出现一种边缘情况。 例如，如果总大小为 50，m 为 10，但即使完全压缩后总大小为 15，算法也会正确返回 -1，因为累积增益永远不会达到所需的阈值。 

另一个边缘情况是当总计已经等于 m 时。 在这种情况下，所需的归约为零，算法立即返回0，而不会进入贪婪循环。 这可以防止错误地压缩不必要的歌曲。 

第三种情况是增益相同时。 排序步骤不依赖于稳定性，因为相等增益之间的任何顺序都会产生相同数量的压缩，并且无论排序如何，前缀条件都确保正确性。
