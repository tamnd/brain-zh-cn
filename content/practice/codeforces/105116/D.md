 ## 解决方案
 ## 问题理解

 我们得到了一系列硬币类型，其中每种类型都比前一种更有价值。 汇率沿着链条成倍增：固定数量的 i 类型硬币可以兑换一枚 i+1 类型硬币。 因为这适用于每个相邻的货币对，所以任何硬币类型都可以通过沿路径相乘或除以任何其他类型来表示。 

玩家从多组硬币开始，每种类型都有一个计数。 随着时间的推移，硬币被添加到收藏中。 每次更新后，我们都会被问到这样一个问题：当前持有的所有代币的总价值是否至少与由 x 个类型 y 的代币组成的某个目标包的价值一样大？ 

因此，每个查询要么是计数向量的一个坐标上的点更新，要么是所有硬币的当前加权总和与单个加权束之间的比较。 

关键的困难在于每种类型的权重并不是独立的。 类型 i 的值取决于其上方所有汇率的乘积，因此直接数字转换很快就会爆炸。 当 n 达到 200,000，q 达到 100,000 时，每次查询后重新计算全局值的速度太慢，甚至维护显式的大整数也变得不可行，因为中间值的大小呈指数级增长。 

一个幼稚的错误是使用前缀乘积将所有内容转换为类型 1 值并维护单个大整数。 这会失败，因为像 10^9 这样的乘法链重复 200,000 次会产生数十万位的数字，使得每个操作都太慢。 

另一个微妙的失败案例来自使用浮点或上限整数比较巨大的值。 如果我们过早截断较大的值，则两个不同的大小可能会变得难以区分，并且当双方都超过上限但不相等时，比较可能会变得不正确。 

## 方法

 一个简单的想法是为每种硬币分配一个固定的重量，等于其在类型 1 硬币中的价值。 那么总价值只是计数和重量的点积。 更新很容易，但查询需要维护一个重复递增的大整数，并且加法涉及非常大的乘法器。 这会导致大量的大整数运算，并且在值沿着整个链重复增长的最坏情况下变得太慢。 

结构洞察是我们实际上并不需要精确的值。 每个查询仅要求比较两个数量。 我们不需要维护精确的大整数，而只需要维护足够的信息来确定排序。 

这表明在饱和数字系统中维护值，其中任何超出特定阈值的值都被视为“无限”，因为一旦值足够大，它将始终主导可以在输入约束中表示的任何可能的查询。 除此之外，我们使用交换结构向上传播贡献，确保更新实际上保持本地和对数，而不是全局。 

我们将当前多重集存储在硬币类型的线段树中，其中每个节点保留以类型 1 单位表示的线段值，但上限为固定的饱和限制。 更新调整单个叶子并向上重新计算。 查询计算同一上限系统下的当前值和目标值并进行比较。 

关键的想法是，只要饱和仅消除相对于任何查询阈值来说已经太大而无关紧要的值之间的区别，比较就保持正确。

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 每个查询的强力转换 | 每个查询 O(n) | O(n) | 太慢了 |
 | 饱和线段树| 每次操作 O(log n) | O(n) | 已接受 |

 ## 算法演练

 1. 预先计算从每种类型到类型 1 的前缀转换因子。不要存储精确的乘积，而是以早期饱和的方式维护它们。 如果产品超出了选定的大范围，我们就会限制它。 
2. 在硬币类型上构建线段树。 每个叶子存储该硬币类型对总价值的贡献，已经缩放为类型 1 单位并饱和。 
3. 对于更新查询，增加单个币种的计数。 我们将这个增量转换为其类型 1 贡献并将其添加到相应的叶子上，然后重新计算父节点。 如果任何节点超过饱和界限，我们就会限制它。 
4. 对于比较查询，我们使用相同的前缀转换逻辑计算 x 类型 y 的硬币的值，同样采用饱和度。 
5.最后，我们将当前多重集的饱和总值与查询束的饱和值进行比较，并输出前者是否至少是后者。 

这样做的原因是，每次真正的比较仅取决于一个值是否最终主导另一个值。 一旦某个值超出了在相同饱和规则下任何查询派生的数字可达到的最大有意义范围，其确切大小就无关紧要了。 系统状态和查询值在相同截断下被对称处理，保留顺序。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

INF = 10**30

n = int(input())
a = list(map(int, input().split()))
b = list(map(int, input().split()))
q = int(input())

# prefix conversion to type 1 with saturation
pref = [1] * (n + 1)
for i in range(1, n + 1):
    if i == 1:
        pref[i] = 1
    else:
        val = pref[i - 1] * a[i - 2]
        if val > INF:
            val = INF
        pref[i] = val

class SegTree:
    def __init__(self, arr):
        self.n = len(arr)
        self.t = [0] * (4 * self.n)
        self.build(1, 0, self.n - 1, arr)

    def cap(self, x):
        return INF if x > INF else x

    def build(self, v, l, r, arr):
        if l == r:
            self.t[v] = self.cap(arr[l] * pref[l + 1])
        else:
            m = (l + r) // 2
            self.build(v * 2, l, m, arr)
            self.build(v * 2 + 1, m + 1, r, arr)
            self.t[v] = self.cap(self.t[v * 2] + self.t[v * 2 + 1])

    def add(self, v, l, r, idx, val):
        if l == r:
            self.t[v] = self.cap(self.t[v] + val * pref[idx + 1])
        else:
            m = (l + r) // 2
            if idx <= m:
                self.add(v * 2, l, m, idx, val)
            else:
                self.add(v * 2 + 1, m + 1, r, idx, val)
            self.t[v] = self.cap(self.t[v * 2] + self.t[v * 2 + 1])

    def query_sum(self):
        return self.t[1]

st = SegTree(b)

for _ in range(q):
    t, x, y = map(int, input().split())
    y -= 1
    if t == 1:
        st.add(1, 0, n - 1, y, x)
    else:
        # compute value of x type-y coins in type 1 units
        need = x * pref[y + 1]
        need = INF if need > INF else need
        print(1 if st.query_sum() >= need else 0)
```线段树存储已归一化为类型 1 单位的每种类型的贡献。 更新仅触及一片叶子并向上传播，保持全局总和一致。 

唯一微妙的设计选择是饱和度。 INF 阈值确保我们永远不会花时间表示超出可能影响任何比较的数字。 

## 工作示例

 ### 示例 1

 考虑一个具有 3 种硬币类型且汇率为 2 和 3 的小型系统。因此，类型 2 等于类型 1 的 2 个，类型 3 等于类型 1 的 6 个。假设最初我们每种类型都有 1 个硬币。 

我们处理一个添加 2 个类型 2 的硬币的查询，然后询问我们是否能买得起 1 个类型 3 的硬币。 

| 步骤| b1 | b2 | b3 | 总计（类型 1）| 查询值 |
 | --- | --- | --- | --- | --- | --- |
 | 初始| 1 | 1 | 1 | 1 + 2 + 6 = 9 | - |
 | 添加 2 类型2 | 1 | 3 | 1 | 1 + 6 + 6 = 13 | 1 + 6 + 6 = 13 | - |
 | 查询 | - | - | - | 13 | 6 |

 由于 13 ≥ 6，答案是 1。这显示了所有类型如何一致地映射到单个尺度。 

### 示例 2

 现在考虑饱和度很重要的情况。 假设汇率很大，并且重复的更新使值超出上限。 一旦超出界限，系统就会将当前总和和查询阈值存储为 INF，但由于两者都是在相同的饱和规则下计算的，因此比较保持稳定。 

这表明该算法从不区分已经与排序无关的值。

 ## 边缘情况

 一种边缘情况是重复更新将单一类型的贡献远远超出任何查询值。 如果没有饱和，这将导致整数溢出或过多的计算。 有了饱和度，一旦数值超过INF，就统一处理，进一步增长不影响比较。 

当查询要求非常高类型的硬币时，会出现另一种边缘情况，其中转换乘数非常大。 前缀乘积立即饱和，这意味着所需的值变为 INF。 任何同样饱和的电流总和都将正确比较，即使未跟踪确切的幅度，也能保留预期的顺序。
