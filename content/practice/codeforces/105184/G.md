 ## 解决方案
 ## 问题理解

 我们提供三种类型的可用手镯牌：一种代表一对零，另一种代表混合对，最后一种代表一对一。 每种瓷砖类型都有有限的库存，分别由 n、m 和 k 给出。 此外，我们还得到了一个由二进制字符串 S 描述的目标圆形手镯图案。 

我们没有被要求建造整个圆形手镯。 相反，我们想要选择 S 的一个连续段，将其视为线性子串，并使用可用的图块构造尽可能长的段。 S 中的每个字符对应于最终构造中的单个位，相邻位确定需要哪个图块。 不同的是，我们可以翻转混合图块的方向，这意味着通常贡献 01 的图块也可以用作 10。其他两种图块类型是对称的，不受翻转的影响。 

任务是确定 S 的连续子串的最大长度，该子串可以在不超过每种图块类型的可用计数的情况下形成。 

对于图块计数和字符串长度，约束都达到 1e6。 这立即排除了子串的任何二次解。 任何解决方案都必须在线性或接近线性的时间内处理字符串，并且任何针对每个子字符串的重新计算都过于昂贵。 

当人们假设可以从字符串的开头贪婪地解决问题而不考虑所有起始位置时，就会出现一种微妙的失败情况。 例如，如果 S 在 0 和 1 之间频繁交替，则对混合图块的需求会波动，并且以不同的偏移量开始会改变我们是否首先用完特定图块类型。 从索引 0 开始的简单的从左到右构造会错过稍后开始的最佳子字符串。 

## 方法

 暴力方法会尝试每个起始索引 i 并将子字符串向右扩展，同时跟踪需要多少个 00、01/10 和 11 转换。 对于每次扩展，我们维护由子串引起的相邻对的计数。 每次扩展子字符串时，我们都会更新所需的图块计数并根据 n、m 和 k 检查可行性。 

这可以正常工作，因为每个子字符串都根据资源限制进行显式测试，但速度太慢。 有 O(N^2) 个子串，并且在最坏的情况下增量更新计数仍然会导致二次行为，因为每个扩展都是 O(1) 但执行了 O(N^2) 次。 

关键的观察是子串的可行性仅取决于相邻字符之间的转换计数。 每个子串都提出一个要求：每个相等的相邻对根据值消耗 00 或 11 瓦片，并且每个不同的对消耗 01 瓦片，无论方向如何，因为允许翻转。 

这减少了跟踪 S 上的滑动窗口的问题，同时维护了三个用于转换的计数器。 一旦我们修复了左端点，我们就可以贪婪地扩展右端点，直到超出其中一个资源。 然后我们向前移动左端点，保持每步 O(1) 的计数。 这是一个经典的两指针或滑动窗口可行性问题，其中约束相对于窗口扩展是单调的。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(N^2) | O(N^2) | O(1) | O(1) | 太慢了 |
 | 滑动窗 | O(N) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们通过计算每个子字符串包含的每种类型的转换数量来处理每个子字符串。 对于子串，每个相邻对都恰好贡献一个资源类别。 

我们维护一个窗口 [l, r) 和反映窗口内相邻转换数量的计数器 cnt00、cnt01、cnt11。

1. 初始化l = 0，r = 0，所有计数器清零。 此时，窗口是空的并且毫无意义。 
2. 一次展开 r 一个字符。 当添加 S[r] 时，如果 r > l，我们考虑在 S[r-1] 和 S[r] 之间形成的新边。 我们根据该边沿是 00、11 还是混合来更新相应的计数器。 此步骤编码了这样一个事实：只有相邻关系才重要，而不是绝对位置。 
3. 更新计数器后，检查 cnt00 <= n、cnt01 <= m 和 cnt11 <= k 是否成立。 如果成立，则当前窗口是可行的，因此我们用其长度更新答案。 
4. 如果窗口变得不可行，我们必须从左侧缩小它。 在向前移动 l 之前，我们删除边缘 (S[l], S[l+1]) 的贡献（如果它存在于窗口内），从而减少相应的计数器。 然后我们增加 l 并继续缩小，直到窗口再次有效。 此步骤有效，因为删除元素只会减少计数，因此最终会恢复可行性。 
5. 继续，直到 r 到达字符串末尾。 

关键的细节是每条边最多添加和删除一次，因此总工作保持线性。 

它为何有效是基于单调可行性属性。 如果窗口不可行，则进一步扩展它无法解决违规问题，因为所有计数器相对于 r 都是非递减的。 同样，从左边收缩严格减少计数器，保证过程始终朝着可行性方向发展，而无需重新访问状态。 这确保了每个有效子串在某个点都被视为窗口边界配置，并记录最大可行长度。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m, k = map(int, input().split())
    S = input().strip()
    if len(S) <= 1:
        print(len(S))
        return

    def classify(a, b):
        if a == b == '0':
            return 0
        if a == b == '1':
            return 2
        return 1

    cnt = [0, 0, 0]  # 00, 01, 11
    l = 0
    ans = 0

    for r in range(len(S)):
        if r > l:
            c = classify(S[r-1], S[r])
            cnt[c] += 1

        while cnt[0] > n or cnt[1] > m or cnt[2] > k:
            if l + 1 <= r:
                c = classify(S[l], S[l+1])
                cnt[c] -= 1
            l += 1

        ans = max(ans, r - l + 1)

    print(ans)

if __name__ == "__main__":
    solve()
```核心实现维护转换计数，而不是针对每个子字符串重新计算它们。 函数分类对三种边缘类型进行编码，确保每个相邻对只贡献一次。 

左指针更新在移位 l 之前小心地消除了传出边缘的贡献。 当窗口大小变为一时，条件 l + 1 <= r 可防止无效访问，因为单个字符没有转换。 

一个常见的陷阱是忘记约束适用于转换，而不是字符的原始计数。 另一个是缩小窗口时错误地更新计数器，这会破坏不变性并导致计数过多。 

## 工作示例

 考虑 S = 0101，其中 n = 0、m = 3、k = 0。仅允许混合图块。 

我们跟踪窗口扩展。 

| r | S[r] | 窗口| cnt00 | cnt01 | cnt11 | 有效 |
 | --- | --- | --- | --- | --- | --- | --- |
 | 0 | 0 | 0 | 0 | 0 | 0 | 是的 |
 | 1 | 1 | 01 | 0 | 1 | 0 | 是的 |
 | 2 | 0 | 010| 0 | 2 | 0 | 是的 |
 | 3 | 1 | 0101| 0 | 3 | 0 | 是的 |

 完整的字符串是可行的，所以答案是 4。这证实了交替结构仅消耗混合图块。 

现在考虑 S = 00110，其中 n = 1，m = 1，k = 0。不允许混合图块，因此禁止过渡。 

| r | S[r] | 窗口| cnt00 | cnt01 | cnt11 | 有效 |
 | --- | --- | --- | --- | --- | --- | --- |
 | 0 | 0 | 0 | 0 | 0 | 0 | 是的 |
 | 1 | 0 | 00 | 00 1 | 0 | 0 | 是的 |
 | 2 | 1 | 001| 1 | 1 | 0 | 是的 |
 | 3 | 1 | 0011| 1 | 1 | 1 | 是的 |
 | 4 | 0 | 00110 | 2 | 1 | 1 | 无收缩|

 当 r = 4 时，窗口变得无效，因为 cnt00 超过 n。 该算法会移动 l 直到恢复可行性，确保找到最佳的有效后缀。 

这些痕迹表明，可行性完全由相邻结构驱动，而不是全局计数。 

## 复杂度分析

 ## 边缘情况

 单字符串是最简单的边界。 由于没有相邻对，因此不消耗任何资源。 当 r = 0 时，该算法正确地将 ans 初始化为至少 1，并且不会发生计数器更新。 

完全交替的弦强调对 m 的依赖。 每一步都会增加cnt01，并且算法永远不会收缩，除非超过m。 这验证了混合边缘的正确分类和持续增长的正确处理。 

像 00000 这样的字符串强调 00 个图块的累积。 每个扩展添加一个 cnt00，一旦超过 n，左指针向前移动，删除旧的 00 贡献。 正确性依赖于对称的添加-删除行为，确保收缩操作期间不会发生重复计数。
