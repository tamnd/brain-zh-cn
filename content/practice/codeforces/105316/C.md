 ## 解决方案
 ## 问题理解

 我们给出了几个独立的场景，其中一匹马从位置 0 开始，仅沿着一条线朝正方向移动。 沿着这条线，食物盘放置在不同的位置。 马到达后，每道菜都需要一定的时间来吃。 向前移动也会花费时间，与行驶的距离成正比：如果马从位置 x 移动到位置 y，它会花费 y − x 分钟的时间。 

对于每个测试用例，我们必须确定马在总时间限制 D 内可以吃多少道菜，假设它可以选择最佳的菜肴子集和顺序，但它永远不能向后移动。 

关键的相互作用是旅行时间和吃饭时间之间的相互作用。 距离较远的菜肴​​价格昂贵，而且也需要时间来享用。 这就产生了单调移动和累积成本的生产线上的调度问题。 

这些约束足够大，以至于每个测试用例具有二次行为的任何解决方案都会失败。 由于所有测试用例的 n 总和最多为 10^5，因此 O(n^2) 方法已经处于临界状态或太慢，而更糟糕的情况显然是不可能的。 这促使我们采用基于排序的贪婪或前缀优化技术，每个测试用例可能为 O(n log n) 或 O(n)。 

当仅基于距离或仅基于进餐时间的贪婪选择失败时，就会出现微妙的边缘情况。 

考虑菜肴：```
D = 5
positions: 1 2 5
times:      1 1 1
```如果我们总是先走尽可能远的，那么去5需要花费5分钟的路程和1分钟的吃饭时间，已经超过了D。但是先选更近的菜可以让菜品总数更多。 仅基于位置的天真贪婪会失败。 

另一种失败模式来自于忽略累积时间：即使每个菜肴单独可行，它们的组合行程累积也会使先前有效的选择无效。 

## 方法

 强力解释是尝试菜肴的每个子集，按位置对每个子集进行排序，从 0 计算总行程时间并求和进食时间，并检查它是否适合 D。这是正确的，因为它尊重单调运动约束，但它会组合爆炸。 每个测试用例有 2^n 个子集，对于每个子集，我们将花费 O(n) 计算可行性，从而导致指数时间。 

一旦我们停止以子集的方式思考，而是按位置对菜肴进行排序，问题的结构就变得易于管理。 由于马永远不会向后移动，任何有效的计划都对应于选择一些递增的位置序列并按该顺序访问它们。 一旦排序，所选类似前缀子集的行程时间仅取决于最大的所选位置，加上跳过间隙的调整。 

关键的简化是以将全局移动与增量决策分开的方式重写所选集合的成本。 如果我们固定集合并按位置排序，则总行程成本只是从 0 开始的连续选定点之间的距离总和。这表明我们可以增量思考：我们从左到右延伸一条路径，并在每个点决定是否包括下一个菜肴。 

我们没有明确枚举子集，而是使用贪婪策略来维护一组选定的菜肴，并且对于给定数量的菜肴，总用餐时间最少。 为了最大限度地提高数量，我们希望保持较低的总成本，同时允许尽可能多的早点菜肴。 这自然会导致最小堆方法：我们以递增的位置顺序迭代菜肴，保持运行时间成本，如果我们超过 D，我们会丢弃迄今为止选择的菜肴中最昂贵的就餐时间菜肴。 这样做的原因是，旅行成本是通过订购固定的，因此只有就餐时间可以调整，并且删除最差的就餐时间项目可以提供保持可行的最佳机会。 

这将问题转化为在累积约束下选择最大数量的项目，其中项目按排序顺序处理，并且我们贪婪地保持可行性。

 | --- | --- | --- | --- |
 | 暴力子集 | O(2^n·n) | O(2^n·n) | O(n) | 太慢了 |
 | 用堆贪婪排序 | O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 我们首先按位置对所有菜肴进行排序。 这是必要的，因为运动是严格向前的，因此任何最佳路线都必须遵循递增的位置顺序。 

接下来，我们从左到右迭代菜肴，同时保持所选菜肴的运行结构。 对于每道菜，我们计算从之前选择的位置到达它所需的额外时间，加上它的用餐时间。 我们维持总成本和结构，使我们能够在需要时取消之前选择的进餐时间。 

我们对所选菜肴的食用时间使用最大堆。 这使得我们能够在总数超过 D 时有效地丢弃浪费时间最多的菜肴，因为删除最长的用餐时间可以最大程度地降低成本。 

在每一步中，我们插入当前菜肴，添加其行程贡献和用餐时间，然后通过删除最差的候选者来重复修复违规行为，直到总时间再次在 D 内。 

最后，答案是任意时刻保留在选择中的最大菜肴数量。 

### 为什么它有效

在任何时候，我们都在考虑按排序顺序排列的菜肴前缀。 在从此前缀中选择 k 道菜的所有方法中，最佳方法是最小化总用餐时间，因为行程成本由最右边选择的位置决定，并且顺序是固定的。 贪心策略准确地保持了这个属性：对于固定的大小，它总是以与可行性一致的方式保持可达候选者中 k 个最小的进食时间。 任何用更长的用餐时间替换所选菜肴的解决方案只会增加总成本，而不会提高可达性，因此基于堆的替换保留了最优性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

import heapq

def solve():
    t = int(input())
    for _ in range(t):
        n, D = map(int, input().split())
        p = list(map(int, input().split()))
        t_eat = list(map(int, input().split()))
        
        dishes = sorted(zip(p, t_eat))
        
        heap = []
        total_time = 0
        prev_pos = 0
        best = 0
        
        for pos, eat in dishes:
            travel = pos - prev_pos
            total_time += travel + eat
            prev_pos = pos
            
            heapq.heappush(heap, -eat)
            
            while heap and total_time > D:
                removed = -heapq.heappop(heap)
                total_time -= removed
            
            best = max(best, len(heap))
        
        print(best)

if __name__ == "__main__":
    solve()
```该解决方案对菜肴进行分类，以便利用连续选定位置之间的差异逐步处理行程。 变量`total_time`累积运动时间和进食时间。 

堆将进食时间存储为负值，以使用 Python 默认的最小堆模拟最大堆。 每当总时间超过限制时，我们就会删除食用时间最长的菜肴，因为它对违规的影响最大。 

变量`best`跟踪在任何前缀步骤中可以维护的最大盘子数量。 

一个微妙的点是我们总是更新`prev_pos`即使我们稍后移除一个盘子。 这是安全的，因为堆清除只会影响进食时间； 移动结构由排序位置上的迭代顺序固定。 这确保了旅行积累和选择之间的一致性。

 ## 边缘情况

 距离原点较远、进食时间较长的一道菜的行为很简单：如果 D 足够，算法将其添加一次，并且永远不会删除它。 由于不存在与其他盘的竞争，堆永远不会触发删除，答案仍然是 1。 

当 D 甚至小于第一次旅行加上进食成本时，堆立即溢出，唯一的元素被删除，留下一个空的选择。 该算法正确报告零。 

密集的小位置盘子和后来的大异常值显示了修剪的预期行为：早期的盘子缓慢积累，但较远的盘子会导致较大的行程跳跃，迫使去除高进食时间。 堆确保只有低成本组合才能生存，从而匹配最优选择策略。
