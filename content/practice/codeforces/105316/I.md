 ## 解决方案
 ## 问题理解

 我们在二维整数网格上给出了几个圆，每个圆都由一个中心点和一个半径定义。 读取所有圆圈后，我们收到一系列查询点。 对于每个查询点，我们必须计算给定的圆中有多少个包含或接触该点。 

当点与圆心之间的欧氏距离小于或等于半径时，圆“覆盖”该点。 由于所有坐标都是整数并且半径都很小（最多 10），因此我们本质上是在重复检查有界几何条件。 

输入大小建议多个测试用例，所有用例总共最多 100,000 个圆圈和 100,000 个查询。 这立即排除了任何为每个查询检查每个圆圈的解决方案。 一种天真的方法会在最坏的情况下执行多达 1010 次距离检查，这远远超出了时间限制。 

一个微妙但重要的约束是半径非常小，最多为 10。这强烈暗示每个圆只影响平面的一个微小的局部区域，并且我们可以利用这个局部性，而不是将圆视为全局几何对象。 

暴露幼稚错误的一种极端情况是许多圆在一个点上严重重叠。 例如，如果所有圆都以 (1, 1) 为中心，半径为 10，则该点附近的每个查询都应计算所有圆。 简单的解决方案在逻辑上仍然有效，但速度太慢。 真正的风险在于优化尝试根据没有精确距离验证的边界框错误地跳过检查，这可能会导致在 x² + y² 恰好等于 r² 的边界处出现差一错误。 

## 方法

 蛮力方法很简单：对于每个查询点，迭代每个圆并使用平方距离计算该点是否位于其中。 这是正确的，因为每个圆的定义都是直接且独立的。 但是，它对每个测试用例执行 n × q 距离检查。 当 n 和 q 的总约束达到 10⁵ 时，这变成了大约 1010 次操作，这是不可行的。 

关键的观察来自半径范围。 每个圆只影响 (2r + 1) × (2r + 1) 正方形内的点，由于 r ≤ 10，因此该区域最多有 21 × 21 = 441 个网格点。 这表明了观点的逆转：我们可以将每个圆的贡献分配给它覆盖的所有整数点，而不是询问哪些圆包含它。 

我们可以预先计算网格点上的频率图。 对于每个圆，我们迭代所有整数偏移量 (dx, dy)，使得 dx² + dy² ≤ r²，并增加对应点 (x + dx, y + dy) 的计数器。 处理完所有循环后，每个查询就变成直接字典查找。 

这是可行的，因为每个圆的格点总数受一个常数 (~300-400) 限制，因此预处理在 n 中是线性的，具有较小的常数因子。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(nq) | O(1) | O(1) | 太慢了 |
 | 预先计算覆盖图 | O(n·r² + q) | O(n·r² + q) | O(n·r²) | O(n·r²) | 已接受 |

 ## 算法演练

 我们将每个圆转换为它覆盖的整数网格点集合并累积贡献。

1. 创建一个哈希映射（或字典），用于存储每个整数点有多少个圆覆盖它。 该地图代表所有可能的查询点的预先计算的答案。 
2. 对于每个具有圆心 (cx, cy) 和半径 r 的圆，迭代从 -r 到 r 的所有整数偏移 dx。 对于每个 dx，从 −r 到 r 迭代 dy，并检查是否 dx² + dy² ≤ r²。 这个条件确保我们只包含圆边界内部或圆边界上的点。 
3. 对于每个有效的 (dx, dy)，递增键 (cx + dx, cy + dy) 处的计数器。 这有效地将圆“绘制”到网格上，增加了所有受影响的整数点的覆盖计数。 
4. 处理完所有圆后，通过读取地图中存储的值来回答每个查询点（qx，qy）。 如果该点不存在，则其值为零。 

关键的想法是我们将计算从查询时间转移到预处理时间。 每个圆圈仅对一个小的固定邻域做出贡献，从而使整体工作易于管理。 

### 为什么它有效

 正确性依赖于每个圆对每个格点独立贡献的事实。 预处理步骤精确地枚举每个圆满足圆不等式的整数点。 由于每个圆中每个这样的点都被访问一次，因此任何坐标处的存储值等于其几何条件包括该点的圆的数量。 查询只是检索此预先计算的聚合，因此不会丢失任何信息或错误地重复计算。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    out = []

    for _ in range(t):
        n, q = map(int, input().split())

        mp = {}

        circles = []
        for _ in range(n):
            x, y, r = map(int, input().split())
            circles.append((x, y, r))

        # precompute offsets for each possible radius (small bound r <= 10)
        for x, y, r in circles:
            rr = r * r
            for dx in range(-r, r + 1):
                dy_limit = int((rr - dx * dx) ** 0.5)
                for dy in range(-dy_limit, dy_limit + 1):
                    px = x + dx
                    py = y + dy
                    key = (px, py)
                    mp[key] = mp.get(key, 0) + 1

        for _ in range(q):
            x, y = map(int, input().split())
            out.append(str(mp.get((x, y), 0)))

    print("\n".join(out))

if __name__ == "__main__":
    solve()
```该解决方案首先构建一个字典，存储受任何圆影响的所有整数点的覆盖计数。 对于每个圆，它使用方程 dx² + dy² ≤ r² 仅枚举圆内的有效整数偏移量。 内部循环受到半径约束的严格限制，因此它仍然有效。 

一个微妙的实现细节是使用平方距离而不是欧几里德距离，避免浮点精度问题。 另一个细节是使用由元组键控的字典，考虑到生成点的总数有限，该字典足够有效。

 ## 边缘情况

 关键的边缘情况是查询恰好位于圆边界上。 对于以 (0, 0) 为圆心、半径为 2 的圆，必须计算点 (2, 0)，因为 2² = 4 等于 r²。 该算法包含这一点，因为条件 dx² + dy² ≤ r² 明确允许相等，并且当 dx = 2, dy = 0 时生成整数枚举保证 (2, 0)。 

另一种情况是多个圆在相同坐标处重叠。 如果十个圆都共享中心 (5, 5)，半径为 1，则 (5, 5) 处的地图条目在预处理过程中会增加十倍。 (5, 5) 处的查询只需读取该累加值，确认线性累加行为，而不会造成圆之间的干扰。
