 ## 解决方案
 ## 问题理解

 给定一个二进制数组，每个元素不是 0 就是 1。我们可以重复选择起始位置并对长度为 3 的连续块进行操作。 该操作将所有三个值替换为 XOR，因此三元组变成重复三次的单个位：如果三元组中 1 的数量为偶数，则全部变为 0，或者如果 1 的数量为奇数，则全部变为 1。 

目标不是计算一个值，而是确定我们是否可以使用最多线性数量的此类操作将整个数组转换为全零，并且如果可能的话，实际构造一个有效的操作序列。 

重要的结构约束是每个操作仅触及三个连续元素并完全覆盖它们。 这意味着信息会被积极破坏，但奇偶校验信息也会在本地传播。 

从复杂度的角度来看，n个测试用例的总和高达2e5。 任何比每个测试用例的线性更糟糕的方法，尤其是任何二次方的方法，例如通过回溯重复模拟天真的贪婪扫描，都不会通过。 这立即表明我们需要一种建设性的贪婪策略，以单次从左到右的方式处理数组，或类似的策略。 

短数组和奇偶校验结构会产生微妙的边缘情况。 

一种边缘情况是当 n 等于 3 时。例如，如果数组是 [1, 0, 0]，应用一次运算会得到 [1, 1, 1]，并且没有办法将其减少到全零。 在这种情况下，假设我们总是可以“修复”本地问题的幼稚方法将错误地报告“是”。 

另一种边缘情况是数组末尾有一个 1，例如 [0, 0, 1, 0]。 影响最后一位的任何操作都必然会拖入其他两个位置，因此不可能在不在其他地方引入新位置的情况下隔离该单个位置。 

这些案例表明，问题的根本在于控制奇偶校验传播，而不是局部消除。 

## 方法

 暴力破解的想法很简单：将每个位置视为应用或不应用操作，模拟长度为 n 的所有可能序列，并检查是否可以到达零数组。 每个操作都会更改三位，因此状态图的大小为 2^n，分支因子最大为 n，这使得超过 n 大约 20 时完全不可行。 

稍微更好的蛮力改进是状态上的 BFS，但状态空间仍然是 2^n，因此它仍然是指数级的。 

关键的见解是停止思考任意序列，而是强制执行贪婪的从左到右标准化。 一旦我们决定在位置 i 发生什么，我们就可以在本地修复约束并确保位置 i 变为零，而不必担心以后再次发生，因为任何涉及 i 的未来操作都必须从 i-2、i-1 或 i 开始，并且我们可以通过构造来防止这种情况发生。 

关键的观察是，我们只需要使用从当前索引开始的操作从左到右“修复”数组，并且我们永远不需要重新访问较早的位置。 这将问题转变为建设性的清理，其中每个决策都消除了局部缺陷并推动剩余的问题向前发展。 

这减少了维护一致的前缀不变式的任务，同时确保后缀仍然可以被解析。

 ## 算法演练

 我们从左到右处理数组，在索引 n - 3 处停止，因为操作需要三个连续元素。

1. 从索引 0 开始，移动到索引 n - 3。 
2. 如果当前元素为1，我们必须在该索引处应用操作。 这是因为以受控方式影响位置 i 而无需重新访问较早位置的唯一方法是锚定从 i 开始的操作。 
3. 通过计算位置 i、i + 1、i + 2 处的三元组的 XOR 来应用该运算，并将所有三个设置为该值。 将索引 i + 1（基于 1 的索引）记录为操作。 
4. 继续向前扫描。 
5. 处理到n - 3 后，检查最后三个元素是否全为零。 如果不是，我们会通过检查是否可以消除剩余的模式来尝试进行最终清理； 如果不是，答案是不可能的。 
6. 如果成功则输出记录的序列。 

关键的设计选择是，每当我们在位置 i 遇到 1 时，我们立即使用唯一可用的本地操作消除它。 这可以防止未解决的问题传播到后面的位置。 

### 为什么它有效

 该算法保持不变，即当前索引之前的所有位置都已固定为零，并且永远不会以改变它们的方式再次触及。 选择每个操作以解析最左边的剩余非零位置，并且因为每个操作仅影响从该位置开始的大小为 3 的窗口，所以后面的操作不能在不违反有效移动结构的情况下将非零值重新引入到已处理的前缀中。 这种贪婪的固定确保了如果存在解决方案，该过程不会阻止未来必要的修正，因为任何有效的解决方案都可以转换为遵循从左到右消除顺序的解决方案。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        n = int(input())
        a = list(map(int, input().split()))
        
        ops = []
        
        for i in range(n - 2):
            if a[i] == 1:
                x = a[i] ^ a[i+1] ^ a[i+2]
                a[i] = a[i+1] = a[i+2] = x
                ops.append(i + 1)
        
        if any(a):
            print("NO")
        else:
            print("YES")
            print(len(ops))
            print(*ops)

if __name__ == "__main__":
    solve()
```实现直接遵循贪心思想。 该循环仅运行到 n - 2，因为每个操作都需要完整的三元组。 每次遇到 1 时，我们都会立即对三元组进行归一化。 这是驱使所有人向左、消灭他们的核心机制。 

最后检查`any(a)`这是至关重要的，因为局部修复仍然会在最后两个位置留下残留图案，而由于缺乏有效的三重窗口而无法解析该残留图案。 

一个微妙的实现点是索引：操作被记录在基于 1 的索引中，因为问题陈述需要这种格式的位置。 混合基于 0 的索引和基于 1 的索引是此处错误答案的常见来源。

 ## 边缘情况

 对于像 [1, 0, 0] 这样的输入，算法会立即应用索引 0 处的操作，将其转换为 [1, 1, 1]。 由于 n 等于 3，因此无法进行进一步的操作，并且最终检查正确地检测到该数组不全为零。 

对于像 [0, 0, 1, 0, 0] 这样的输入，索引 2 处的 1 会强制执行将影响力分散到两侧的操作，并且后续传递无法在不重新引入早期值的情况下隔离它。 决赛`any(a)`check 捕捉到了这种不可能性。 

对于全为 1 的输入，重复从左到右消除会减少受控级联中的数组，如果长度足够，最终会清除它，这表明贪婪传播能够在结构允许的情况下解析密集配置。
