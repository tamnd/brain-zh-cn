 ## 解决方案
 ## 问题理解

 我们得到了一系列奶酪块，每块奶酪块都有一个初始新鲜度值。 时间在离散的日子里前进。 每天，所有剩余的碎片都会同时失去一个单位的新鲜度。 每天，Tryna 最多只能吃一块，而且他很挑剔：当天他只能吃当前新鲜度能被二整除的一块。 

目标是确定根据这些规则可以食用所有奶酪块所需的最短天数。 

关键的微妙之处在于，一块是否可以食用不仅取决于它的价值，还取决于它被食用的日期，因为新鲜度会随着时间的推移而均匀下降。 因此，问题不仅仅是调度删除，而是在依赖于时间的奇偶校验约束下调度它们。 

限制允许最多 100000 件，新鲜度值最多为 1000000。这立即排除了任何对日常饮食的模拟。 如果采用天真的方法，迭代数天并尝试选择有效的奶酪，则可能会达到最大新鲜度值，但该值太大了。 任何有效的解决方案都必须将问题简化为线性时间内的计数和组合问题。 

当我们总是选择任何当前可食用的奶酪而不考虑未来的奇偶约束时，就会出现天真的贪婪模拟的常见失败案例。 

例如，如果早期选择首先消耗“灵活”项目，我们可能会阻止调度仅适合稍后一天的特定奇偶性的项目。 由于新鲜度每天都会发生变化，因此延迟某件物品可能会改变它在以后几天是否可以食用，因此贪婪的本地选择可能会默默地失败。 

## 方法

 第一直觉是每天模拟这个过程。 每天，我们都会更新所有新鲜度值并扫描当前新鲜度为偶数的有效项目。 如果可能的话，我们会删除一个。 这是正确的，但速度非常慢，因为每天最多需要扫描 n 个项目，并且在最坏的情况下最多可能有 O(max(ai)) 天，这导致在最坏的情况下大约需要 10^11 次操作。 

关键的观察结果是，新鲜度平价以一种非常结构化的方式演变。 d 天后，最初新鲜度为 a 的奶酪变成了 a - d。 在 d 天可以食用的条件是 a - d 是偶数，这相当于 a 和 d 具有相反的奇偶性，因为减去 d 会根据 d 翻转奇偶性。 

重写这个约束给出了一个更清晰的解释：每种奶酪根本不依赖于大小，只依赖于它是否必须在奇数或偶数日食用。 这将问题转换为具有两种类型的作业和两种类型的时隙的调度问题。 

我们将问题简化为将每种奶酪分配给不同的一天，其中每一天都有固定的奇偶性，并且每种奶酪都需要特定的奇偶性。 唯一剩下的限制是每天最多只能处理一块奶酪。 

所以问题就变成了：我们有两组项目，一组必须进入奇数天，一组必须进入偶数天，并且我们需要最小的天数前缀，以便可以容纳所有项目。 

我们计算有多少项目需要奇数天，有多少项目需要偶数天，然后找到最小的 T，使得直到 T 的可用奇数和偶数槽都足够。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 日常模拟 | O(n·max(ai)) | O(n·max(ai)) | O(n) | 太慢了 |
 | 奇偶校验计数和调度| O(n) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们首先根据何时允许食用每种奶酪将其转换为简单的平价要求。 然后我们计算每个奇偶类别有多少种奶酪。 最后，我们计算所需的最小天数，以便两个奇偶校验类别都适合可用的天数。

1. 对于每种初始新鲜度为 a 的奶酪，确定必须在哪一天食用。 由于新鲜度每天减少 1，因此新鲜度的奇偶性每天都会交替，因此该条件仅取决于 a 与日期索引之间的奇偶性对齐。 
2. 将每种奶酪分为两组，一组需要奇数天，一组需要偶数天。 令 k_odd 和 k_even 为它们的计数。 
3. 考虑 T 天的前缀。 奇数天数为 (T + 1) // 2，偶数天数为 T // 2。 
4. 我们需要 k_odd <= (T + 1) // 2 和 k_even <= T // 2 同时保持。 
5. 尝试T的两种结构情况。如果T是偶数，T = 2m，那么奇数天和偶数天都是m，所以我们需要m >= max(k_odd, k_even)。 这给出了候选 T = 2 * max(k_odd, k_even)。 
6. 如果T是奇数，T = 2m + 1，则奇数天为m + 1，偶数天为m。 我们要求 m >= k_even 且 m + 1 >= k_odd，这意味着 m >= max(k_even, k_odd - 1)。 这给出了候选 T = 2 * max(k_even, k_odd - 1) + 1。 
7. 答案是这两个有效结构中的最小值。 

它起作用的原因是，一旦我们确定了长度 T，调度的唯一结构就是存在多少个奇数和偶数时隙。 由于每个奶酪都是独立的，并且只有奇偶校验约束，因此只要计数匹配，任何分配都是可行的。 除了槽位可用性之外，没有任何交互，因此将问题简化为二分计数约束是准确的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = list(map(int, input().split()))
    
    k_odd = 0
    k_even = 0
    
    for x in a:
        # determine required parity of day
        # x - (d-1) even => parity(d) = parity(x+1)
        if (x + 1) % 2 == 1:
            k_odd += 1
        else:
            k_even += 1
    
    # even T = 2m
    t_even = 2 * max(k_odd, k_even)
    
    # odd T = 2m + 1
    m = max(k_even, k_odd - 1)
    if m < 0:
        m = 0
    t_odd = 2 * m + 1
    
    print(min(t_even, t_odd))

if __name__ == "__main__":
    solve()
```该代码首先将每种奶酪简化为奇偶校验要求，并计算每个类别中有多少奶酪。 然后，它评估最佳调度长度的两种可能的结构形式（偶数和奇数），并取最小值。 关键细节是处理奇数长度情况下的不对称约束，其中奇数天与偶数天相比有一个额外的槽位。 

一个微妙的实现点是确保当 k_odd 为零或一时 k_odd - 1 不会变为负数，因为在该范围内自动满足约束 m >= k_odd - 1。 

## 工作示例

 ### 示例 1

 输入：```
4
500001 500002 500003 500004
```我们计算奇偶校验要求：

 | 步骤| 价值| (a+1)%2 | 类别 |
 | --- | --- | --- | --- |
 | 1 | 500001 | 甚至| 偶数日 |
 | 2 | 500002 | 奇数| 奇数日 |
 | 3 | 500003 | 甚至| 偶数日 |
 | 4 | 500004 | 奇数| 奇数日 |

 所以 k_odd = 2，k_even = 2。 

现在评价：

 | T型| 公式| 价值|
 | --- | --- | --- |
 | 甚至| 2 * 最大值(2,2) | 4 |
 | 奇数| 2 * 最大值(2,1) + 1 | 5 |

 最小值为 4。 

这表明平衡分布完美地适合交替奇偶校验插槽，而不需要额外的松弛。 

### 示例 2

 输入：```
3
500001 500003 500004
```奇偶校验分类：

 | 价值| 类别 |
 | --- | --- |
 | 500001 | 偶数日 |
 | 500003 | 偶数日 |
 | 500004 | 奇数日 |

 所以 k_even = 2，k_odd = 1。 

现在计算：

 | T型| 价值|
 | --- | --- |
 | 甚至| 2 * 2 = 4 | 2 * 2 = 4
 | 奇数| 2 * 最大值(2,0) + 1 = 5 |

 答案是4。 

这表明，即使当一种奇偶校验类占主导地位时，最优调度也会简单地扩展，直到存在足够的限制奇偶校验时隙。

 ## 边缘情况

 当只有一种奶酪时，算法将其分类到一个奇偶校验桶中，并根据可行性立即返回 1 或 2，这与始终需要至少一天的事实相匹配。 

当所有奶酪都属于同一奇偶性类别时，该公式就会简化为强制该奇偶性有足够的槽位，这变成了简单的加倍模式。 偶数长度结构占主导地位并产生正确的线性缩放。 

当奇偶校验类之间的不平衡很大时，奇数长度的情况变得很重要，因为它为一种奇偶校验类型提供了一个额外的时隙。 该算法正确地测试了这两种结构，确保选择最小的扩展，而不是过度承诺均匀长度的计划。
