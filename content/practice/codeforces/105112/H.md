 ## 解决方案
 ## 问题理解

 我们得到了一组整数，并且允许我们构建一个算术表达式，该算术表达式仅使用每个整数一次。 唯一可用的运算是加法和乘法，我们可以自由插入括号来控制计算顺序。 目标不仅仅是形成任何有效的表达式，而是最大化其数值。 

关键的困难在于乘法和加法在分组下以非平凡的方式相互作用。 由于所有数字都是正数，因此乘法往往会主导加法，但加法可用于在将因子相乘之前使因子膨胀。 

约束很大：最多 100,000 个数字，每个数字最大为 1,000,000。 这立即排除了任何尝试探索表达式树或子集动态编程的方法。 甚至$O(n^2)$推理太慢了。 该解决方案必须有效地是线性或近线性的，也许有一个$O(n \log n)$排序步骤。 

当存在像 1 这样的小数字时，就会出现微妙的边缘情况。 由于乘以 1 没有任何作用，但在乘法之前加 1 会增加一个值，因此它们的位置变得至关重要。 例如，使用输入`1 1 1 1`，天真的贪心乘法给出 1，但将它们分组为`(1+1)*(1+1)`给出 4，这绝对更好。 

另一个边缘情况是当所有数字都很大时。 那么乘法几乎总是有益的，但我们仍然必须决定分组。 例如，`13 37 1`在乘法之前将 1 加到 13 时效果最佳。 

核心挑战是确定括号的最佳结构，而不是暴力破解所有可能性。 

## 方法

 强力解决方案将在数字上尝试所有可能的二元表达式树，为每个内部节点分配加法或乘法。 即使忽略交换律，树形状的数量也是加泰罗尼亚数$C_{n-1}$，并且每个内部节点有2种操作选择，因此总搜索空间呈指数增长。 即使对于$n = 20$，更不用说$10^5$。 

关键的观察是，乘法是结合性的，加法是结合性的，但混合它们会创建一个结构，其中只有一个相互作用很重要：我们是否在将一组数字相乘之前使用加法“升级”一组数字。 由于所有数字都是正数，因此增加任何因子都会增加最终乘积，因此我们希望最大化一组构造项的乘积，其中每个项都是某个子集的总和。 

这将问题简化为将数字分组，对每个组求和，然后将所有组的总和相乘。 剩下的问题是如何以最佳方式组建这些群体。 

最优策略源于经典的不等式直觉：只有当我们仔细确保 1 用于膨胀其他数字而不是浪费在大和中时，将两个和相乘才比将所有内容合并为一个巨和更有利。 事实证明，最好的结构是形成一个乘法链，其中我们将大于 1 的数字分别分组，并在乘法之前使用 1 来增加这些组。 这导致了一种规范结构：取所有大于 1 的数字，将它们的“增广和”相乘，然后通过将它们添加到一个选定的因子中来处理 1。 

达到最大值的表达式树总是有一个“乘法主干”，其中每个节点要么是一个基数，要么是吸收一些 1 的和。 该结构简化为排序和贪婪附加 1 秒。

 ## 算法演练

 1. 将输入数字分为两组：1 组和非 1 组。 原因是 1 在乘法和加法中的表现不同，因为它对于乘法是中性的，但对于增加总和很有用。 
2. 如果没有大于 1 的数字，则返回所有数字的全括号总和。 在这种情况下，乘法没有任何帮助，最佳值只是它们的总和。 
3. 如果有大于 1 的数字，则将每个数字视为乘法结构的基本组成部分。 
4. 选择一个非一数字作为最终乘法链的锚点。 使用加法将所有剩余的 1 添加到该锚点中。 这增加了该因子的值而不改变乘法结构。 
5. 对于所有剩余的非一数字，将每个数字视为乘法链中的独立因子。 
6. 使用乘法组合所有因子，插入括号，以便计算顺序强制执行预期的分组。 
7. 确保所有 1 只使用一次并且仅在加法组内使用，除非不可避免，否则绝不作为独立的乘法因子。 

### 为什么它有效

 该构造依赖于以下不变量：最终产品中的每个因子要么是大于 1 的单个整数，要么是该整数与一定数量的 1 的总和。 由于乘法在正整数上是单调的，因此增加任何因子的值都会严格增加总乘积。 因此，每个 1 都应该分配给一个因子，而不是拆分到多个位置，因为拆分会减少边际收益。 一旦这个问题被解决，剩下的结构就纯粹是独立优化因素的乘法，这确保了全局最优性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = list(map(int, input().split()))

    ones = []
    others = []

    for x in a:
        if x == 1:
            ones.append(x)
        else:
            others.append(x)

    # case: all ones
    if not others:
        expr = "(" + "+".join("1" for _ in ones) + ")"
        print(expr)
        return

    # sort to have deterministic structure
    others.sort()

    # attach all ones to the first non-one number
    # build first factor
    first = str(others[0])
    if ones:
        first = "(" + first + "+" + "+".join("1" for _ in ones) + ")"

    # remaining factors
    factors = [first] + [str(x) for x in others[1:]]

    # multiply them all
    expr = factors[0]
    for f in factors[1:]:
        expr = "(" + expr + "*" + f + ")"

    print(expr)

if __name__ == "__main__":
    solve()
```代码首先将输入分为 1 和非 1，因为它们在构造中扮演着根本不同的角色。 如果一切都是一，那么唯一有意义的操作就是加法，所以我们只需用显式括号将它们相加即可。 

当至少存在一个大于一的数字时，我们对非一值进行排序以使输出具有确定性，尽管正确性不依赖于排序。 然后我们将所有的都附加到第一个因素中。 这反映了一种贪婪的决定，即所有附加的“免费提升”都应该集中在一个地方，从而最大化乘法增益。 

最后，我们使用乘法连接所有因素，仔细地将每个步骤括在括号中以强制执行计算顺序并避免运算符优先级的歧义。 

## 工作示例

 ### 示例 1

 输入：`1 2 3 4`我们将值分为 one = [1]，others = [2,3,4]。 

| 步骤| 当前结构| 行动|
 | ---| ---| ---|
 | 1 | 2, 3, 4 | 识别非人|
 | 2 | 2, 3, 4 | 将 1 附加到第一个因子 |
 | 3 | (2+1), 3, 4 | 建立第一因素|
 | 4 | ((2+1)*3), 4 | 依次相乘 |
 | 5 | (((2+1)*3)*4) | 最终表达 |

 这会产生一个等价于的表达式`3*((1+2)*4)`重新排列分组后的值。 该迹线表明所有加法灵活性都集中到一个乘法分支中。 

### 示例 2

 输入：`13 37 1`我们将一个= [1]，其他= [13, 37]分开。 

| 步骤| 当前结构| 行动|
 | ---| ---| ---|
 | 1 | 13、37 | 识别非人|
 | 2 | (13+1), 37 | 将 1 附加到第一个 |
 | 3 | ((13+1)*37) | 乘|

 这符合最佳想法，即 1 应该增加一个乘法因子，而不是单独使用。 

跟踪表明将 1 分发到其他地方不会改进产品。

 ## 边缘情况

 关键的边缘情况是所有数字都为 1 时。该算法通过生成求和表达式来处理此问题，因为乘法会将所有数字压缩为 1。对于输入`1 1 1`, 施工产量`(1+1+1)`，这是最优的。 

另一种边缘情况是单个大数，例如`1000000`。 该算法立即返回它，因为没有任何操作可以提高它的值。 

混合情况如`1 1 100`说明了为什么集中注意力很重要。 该算法产生`(100+1+1)`，并且由于只有一个乘法分量可用，因此跨多个因素的替代分布无法改善乘积。
