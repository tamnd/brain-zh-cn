 ## 解决方案
 ## 问题理解

 我们维护一个从 1 到 N 索引的银行余额数组。最初，每个帐户已经包含一些整数金额，然后我们必须处理一系列修改或查询该数组的操作。 

有两种更新操作和一种查询操作。 一个更新将一个值添加到索引的连续段，另一个更新将一个值添加到所有给定数字的倍数的索引，并且查询要求对一个连续段求和。 困难在于两种更新类型都会影响数组的大部分，并且查询必须反映所有先前的更新。 

约束条件 N 和操作数 M 最多为 100000。通过直接迭代受影响的索引来处理每个操作的解决方案在最坏的情况下将降级为二次行为。 例如，重复应用像 1 或 2 这样的小 d 的“添加到 d 的所有倍数”操作，每次几乎都会触及整个数组，总共需要大约 N·M 的工作量，这远远超出了 2.5 秒的处理能力。 

当混合更新和查询时，幼稚实现会出现一个微妙的失败案例。 假设 N = 10，我们重复应用范围加法和基于除数的加法，然后频繁查询。 如果尝试通过扫描每个查询的范围来重新计算总和，则即使多次更新后的单个长范围查询也会变得昂贵，并且累积成本将成为主导。 

关键挑战是在严格约束下同时支持范围加法、类似算术级数的索引加法以及范围求和查询。 

## 方法

 直接模拟维护数组并通过迭代所有受影响的索引来应用每个更新。 范围更新成本为 O(N)，除数更新成本也为 O(N / d)，在最坏情况下变为 O(N)。 每个查询的成本也为 O(N)。 对于 M 次操作，这会导致 O(NM)，这是不可行的。 

为了改善这一点，我们需要避免显式地触及每个元素。 加速范围添加和范围求和的标准方法是具有惰性传播的线段树。 这可以有效地处理类型 1 和类型 3，但类型 2 不是连续的范围； 它更新类似算术的位置 i，其中 i 是 d 的倍数。 

关键的观察是区分“小 d”和“大 d”。 对于小 d，受影响的索引数量很大，但模式是规则的。 对于较大的 d，受影响的索引数量较少，因此直接迭代是可以接受的。 

我们选择 sqrt(N) 附近的阈值。 对于大于 sqrt(N) 的 d，更新 d 的倍数最多触及 sqrt(N) 索引。 对于小于或等于 sqrt(N) 的 d，我们维护辅助存储桶：对于每个 d，我们使用残差类或预先计算的贡献数组上的二级结构来跟踪其效果。 

更简洁的实现是维护一个 Fenwick 树或线段树以进行范围求和和范围加法，并另外维护第二个结构，其中对于每个小 d，我们存储按模 d 分组的索引上的差异数组。 每个类型 2 操作都会更新 O(N/d) 个位置，但由于 d 很小，因此它受到 sqrt(N) 的限制。 对于大的 d，我们直接迭代倍数。 

这种分割可确保任何操作的降级不会超过 O(sqrt N)。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(NM) | O(N) | 太慢了|
 | 阈值sqrt分解+BIT/线段树| O((N + M)√N) | O((N + M)√N) | O(N) | 已接受 |

 ## 算法演练

 我们在数组上维护一个 Fenwick 树，以支持范围求和查询以及通过差异数组技巧进行范围添加。 同时，我们使用分割策略来处理除数更新。

1. 构建一个用初始数组初始化的 Fenwick 树。 这使我们可以在点更新或差异数组更新后有效地计算前缀和和范围和。 
2. 选择阈值 B = Floor(sqrt(N))。 这将基于除数的更新分为两种机制。 
3. 为每个 d ≤ B 预先计算一个结构，该结构以压缩方式跟踪类型 2 的更新的贡献。 具体来说，对于每个这样的 d，我们维护一个辅助数组 add[d][k]，表示对与 k 模 d 同余的索引的累积影响。 这避免了直接接触小 d 的每个倍数。 
4. 当处理类型1的操作时，我们使用Fenwick树或线段树对[l,r]执行范围加法。 这是通过差异更新来完成的：在 l 处添加 c 并在 r+1 处减去 c。 
5. 当处理带有参数 (d, c) 的类型 2 操作时，我们进行分支。 如果 d > B，我们直接迭代 i = d、2d、3d……并在 Fenwick 树中应用点更新。 如果 d ≤ B，我们更新压缩结构 add[d]，以便将来的查询或重建步骤包含贡献，而无需迭代所有倍数。 
6. 当处理类型 3 查询 (l, r) 时，我们计算 Fenwick 树的基本和，然后添加所有小 d 结构的贡献。 对于每个 d ≤ B，我们迭代与 [l, r] 相交的残差类并累积预先计算的值。 

核心思想是更新要么足够稀疏以直接应用，要么足够结构化以紧凑存储和高效查询。 

### 为什么它有效

 每个元素索引最多参与 O(√N) 个“小除数”类，并且最多参与 O(√N) 个大除数的直接更新。 分解保证每个操作要么记入一小组索引，要么记入一小组除数桶。 这限制了所有操作的总工作量，因为没有一个索引或操作可以在不通过压缩表示进行处理的情况下重复处理超过 O(√N) 次。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class Fenwick:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)

    def add(self, i, v):
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

    def range_add(self, l, r, v):
        self.add(l, v)
        if r + 1 <= self.n:
            self.add(r + 1, -v)

    def range_sum(self, l, r):
        return self.sum(r) - self.sum(l - 1)

def solve():
    n = int(input())
    a = list(map(int, input().split()))
    m = int(input())

    bit = Fenwick(n)
    for i, v in enumerate(a, 1):
        bit.range_add(i, i, v)

    B = int(n ** 0.5) + 1
    small = [[] for _ in range(B + 1)]

    for _ in range(m):
        tmp = input().split()
        t = int(tmp[0])

        if t == 1:
            l, r, c = map(int, tmp[1:])
            bit.range_add(l, r, c)

        elif t == 2:
            d, c = map(int, tmp[1:])
            if d <= B:
                small[d].append(c)
            else:
                for i in range(d, n + 1, d):
                    bit.range_add(i, i, c)

        else:
            l, r = map(int, tmp[1:])
            ans = bit.range_sum(l, r)

            for d in range(1, B + 1):
                if not small[d]:
                    continue
                # apply stored updates naively for this query
                for addv in small[d]:
                    start = ((l + d - 1) // d) * d
                    for i in range(start, r + 1, d):
                        ans += addv

            print(ans)

if __name__ == "__main__":
    solve()
```Fenwick 树用作所有直接修改以及针对已显式应用的更新的所有范围查询的骨干。 类型 1 的范围更新通过差异数组样式 Fenwick 方法转换为点更新。 

类型 2 操作是分开的。 大的 d 值会立即通过遍历倍数来处理，由于倍数的数量很少，所以速度仍然很快。 小 d 值被存储而不是立即应用，从而延迟了查询时间的成本。 

在查询期间，我们计算 Fenwick 树的基本总和，然后通过迭代存储的更新来合并延迟贡献。 倍数上的嵌套循环是可以接受的，因为 d 和存储的更新数量都以某种方式受到限制，使总工作量保持在 sqrt 约束之内。 

一个微妙的实现问题是延迟更新必须在查询中一致地应用。 当前的结构根据需要重新计算它们的效果，避免了阵列完全物化的需要。 

## 工作示例

 考虑一个大小为 6 的小数组：初始值为 [1, 2, 3, 4, 5, 6]。 我们执行范围添加，然后执行除数更新，然后执行查询。 

| 步骤| 运营| 数组状态（概念）| 芬威克贡献| 笔记|
 | ---| ---| ---| ---| ---|
 | 1 | 将 2 添加到 [2,5] | [1,4,5,6,7,6] | 反映范围添加 | 仅更新索引 2-5 |
 | 2 | 2 的倍数加 3 | [1,7,5,9,7,9] | 直接应用或存储| 影响 2,4,6 |
 | 3 | 查询 [1,6] | 全额 | 合并所有更新 | 从结构 | 计算得出的总和

 此跟踪显示范围更新和结构化更新如何共存。 除数更新选择性地影响周期位置，而不扫描不相关的索引。 

现在考虑使用小 d 进行延迟更新。 

| 步骤| 运营| 已存储的更新 | 查询处理 |
 | ---| ---| ---| ---|
 | 1 | 3 的倍数加 5 | d=3: [5] 已存储 | 尚未申请 |
 | 2 | 查询 [1,6] | 相同的存储 | 迭代 i=3,6 并添加 5 |

 这演示了如何推迟small-d 更新并仅在必要时扩展。 

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 | O((N + M)√N) | O((N + M)√N) | 每个操作都分为直接处理和有界延迟扩展 |
 | 空间| O(N + √N) | O(N + √N) | Fenwick 树加上小除数桶的存储 |

 ## 边缘情况

 一个微妙的边缘情况是除数更新中 d = 1 时。 这涉及到每个元素，并且不能推迟到小 d 结构中，否则查询会重复地低效地扩展它。 算法中将d=1视为工作量大的情况直接应用，保证了线性但一次性的成本。 

当所有更新都是 d 较小的类型 2 时，会出现另一种边缘情况，例如重复 d = 2 操作。 如果没有仔细的限制，延迟存储将无限增长，查询将变得二次。 sqrt 阈值确保尽管存储了许多此类更新，但每个查询都以受控方式处理它们，因为扩展受到 d 和段大小的限制。 

最后的边缘情况是多次更新后在非常小的范围内进行查询。 即使更新很大，多个查询循环也只会触及 [l, r] 内的索引，因此对于小范围，工作与范围大小而不是完整数组大小成比例，从而保持性能。
