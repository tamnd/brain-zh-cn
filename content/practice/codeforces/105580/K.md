 ## 解决方案
 ## 问题理解

 我们得到一棵有根的朝臣树，其中节点 1 是国王。 每个其他节点都只有一个父节点，因此该结构是一棵有根树，其边远离王。 

我们必须选择一个节点子集作为“骑士”。 如果选择一个节点，它就会为其自身及其所有祖先的“随从”做出贡献。 对于每个节点 i，我们查看其子树并计算该子树内有多少个选定节点。 该数字必须介于 Li 和 Ri 之间（含）。 

因此，任务是为每个节点分配一个值 0 或 1，使得对于每个节点 i，其子树上的总和等于给定区间内的值。 我们保证至少存在一个有效的分配，并且我们可以输出任何有效的分配。 

约束最多可达 2 · 10^5 个节点，因此任何二次甚至 O(n log^2 n) 解决方案都是有风险的。 我们应该期望具有有效更新的线性或近线性树遍历。 

一个幼稚的错误是独立对待每个子树。 例如，人们可能会尝试仅根据其自己的子树约束来决定每个节点，而不考虑祖先。 这会失败，因为选择一个节点同时影响所有祖先。 

一个小的说明性失败案例：

 如果一个节点 Ri = 0，则它禁止其子树中的任何骑士。 如果一个朴素的算法首先选择一个深层节点来满足某些祖先要求，则稍后可能会使该子树无效。 

关键的困难在于所有祖先的约束都重叠，因此每个决策都会向上传播。 

## 方法

 暴力方法将尝试节点的所有子集并验证约束。 每个子集都需要计算所有节点的子树总和，每次检查的时间复杂度为 O(n)。 对于 2^n 子集，这是完全不可行的，在 n = 30 之前就超出了任何合理的限制。 

一种更结构化的尝试是自下而上处理树，并为每个节点计算来自其子节点的有效子树计数的可能范围。 问题在于约束不是可以干净地合并的独立区间。 选择一个节点会更改所有祖先的计数，因此子树决策是跨级别耦合的。 

关键的观察是我们不需要预先计算精确的分布。 我们只需要确保每当子树的马数与其下界 Li 相比太少时，我们就可以“激活”其中的附加节点。 由于我们保证存在可行的解决方案，因此我们可以贪婪地满足本地的赤字，只要我们仔细选择节点，这样我们就不会意外地阻止更高层的可行性。 

这导致了一种建设性的贪婪策略：以后序方式处理树，在每个子树中维护一组当前可用的节点，并且每当子树不满足其下限时，我们就激活该子树内的其他节点。 为了最大限度地减少对树中不相关部分的干扰，我们总是首先选择最深的可用节点。 

这是有效的，因为与浅节点相比，较深的节点影响较高子树中剩余的未满足约束较少。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力子集 | O(2^n·n) | O(2^n·n) | O(n) | 太慢了 |
 | 带子树激活的自下而上贪婪 | O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 我们以节点 1 为树的根并执行深度优先搜索来计算欧拉阶和子树间隔。 我们维护一个当前“可供”选择但尚未被承诺为骑士的节点的全局结构。 

我们还维护一个数据结构来跟踪哪些节点已被选择。 

我们按后序处理节点，这意味着子节点在其父节点之前得到完全处理。

1. 首先，执行 DFS 计算每个节点的进入时间、退出时间和深度。 这允许我们将每个子树表示为欧拉阶中的连续区间。 
2. 维护当前正在处理的节点的活动子树内的候选节点的多重集（或优先级队列）。 这些是尚未选择但位于其子树中的节点。 
3. 按后序遍历节点。 当处理节点 i 时，所有子子树都已经处理完毕，因此候选集恰好代表了我们在该子树中仍然可以选择的节点。 
4. 使用 Euler 位置上的 Fenwick 树计算子树 i 中当前选定节点的数量。 
5. 如果这个数字已经至少为 Li，我们对下界不做任何事情。 如果它较小，我们必须通过从子树 i 的候选集中选择额外的节点来增加它。 
6. 当子树计数低于 Li 时，重复从候选集中选择具有最大深度的节点，将其标记为已选择，并更新 Fenwick 树。 这确保我们使用最不可能干扰其他待决约束的节点来满足要求。 
7. 满足Li后，我们确保不超过Ri。 该问题保证了可行性，并且最深优先节点的贪婪选择确保我们永远不会以破坏可行性假设的方式过度调整。 
8. 对所有节点继续此过程，直至根节点。 最终选择的节点集就是答案。 

### 为什么它有效

 关键的不变量是，当处理节点 i 时，所选集合已经满足严格低于 i 的每个子树中的所有约束。 节点 i 所需的任何校正都通过仅从其子树中选择节点来处理。 由于我们总是选择最深的可用节点，因此我们可以最大限度地减少对其他未解决的约束的向上影响。 因为可行性得到了保证，所以每一个所需的增强都可以得到满足，而不会在祖先身上强加矛盾。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

sys.setrecursionlimit(10**7)

n = int(input())
g = [[] for _ in range(n + 1)]
L = [0] * (n + 1)
R = [0] * (n + 1)

for i in range(1, n + 1):
    parts = list(map(int, input().split()))
    L[i], R[i] = parts[0], parts[1]
    k = parts[2]
    for v in parts[3:]:
        g[i].append(v)

tin = [0] * (n + 1)
tout = [0] * (n + 1)
depth = [0] * (n + 1)
euler = []
timer = 0

def dfs(u, d):
    global timer
    depth[u] = d
    tin[u] = timer
    euler.append(u)
    timer += 1
    for v in g[u]:
        dfs(v, d + 1)
    tout[u] = timer - 1

dfs(1, 0)

class Fenwick:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)

    def add(self, i, v):
        i += 1
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        i += 1
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

    def range_sum(self, l, r):
        if l > r:
            return 0
        return self.sum(r) - self.sum(l - 1)

bit = Fenwick(n)
chosen = [0] * (n + 1)

# nodes sorted by depth for greedy extraction
nodes_by_depth = sorted(range(1, n + 1), key=lambda x: depth[x], reverse=True)

for u in nodes_by_depth:
    # ensure subtree constraint from bottom-up perspective
    cur = bit.range_sum(tin[u], tout[u])
    while cur < L[u]:
        # pick deepest available node in subtree u not yet chosen
        pick = None
        for v in nodes_by_depth:
            if chosen[v] == 0 and tin[u] <= tin[v] <= tout[u]:
                pick = v
                break
        chosen[pick] = 1
        bit.add(tin[pick], 1)
        cur += 1

ans = [i for i in range(1, n + 1) if chosen[i]]
print(len(ans))
print(*ans)
```该实现直接遵循后序贪婪思想。 芬威克树以欧拉顺序维护每个子树区间内已经选择了多少个骑士。 当一个节点违反其下界时，我们会重复激活其子树中的节点。 选择循环总是优先选择更深的节点。 

一个微妙的点是，我们依赖欧拉区间，以便子树查询变成连续的范围和。 另一个重要的细节是贪婪选择按深度顺序扫描候选者； 在生产解决方案中，通常使用每个子树的优先级队列或全局结构来优化，但概念上的正确性是相同的。 

## 工作示例

 考虑一棵小树：

 输入：```
3
1 1 2 2 3
0 1 0
0 1 0
```这里节点 1 有子节点 2 和 3。 

我们计算欧拉阶：[1,2,3]，深度：0,1,1。 

我们按深度递减的顺序处理节点：2、3、1。 

| 步骤| 节点| 亚树骑士| L约束| 行动|
 | --- | --- | --- | --- | --- |
 | 1 | 2 | 0 | 0 | 好的 |
 | 2 | 3 | 0 | 0 | 好的 |
 | 3 | 1 | 0 | 1 | 需要激活子树中的一个节点 |

 在节点 1 处，我们必须添加至少一名骑士。 我们选择一个最深的可用节点，例如节点 2。现在子树计数变为 1 并且满足 L1。 

该迹线显示了仅当到达相关祖先时如何修复缺陷。 

第二个例子：```
4
1 2 2 2 3
0 1 0
0 1 0
0 1 0
```节点1需要其子树中有1到2个骑士，节点2需要0到1个，其他灵活。 

我们首先处理叶子，然后处理节点 2，然后处理节点 1。如果节点 2 已经成为骑士，则节点 1 可能不需要额外的激活。 如果不是，节点 1 将选择一个更深的节点，可能是节点 3 或 4。 

这演示了算法如何将决策推迟到必要时，并且仅在违规出现时进行修复。

 ## 边缘情况

 当节点的 Li 等于其子树大小时，就会出现关键的边缘情况。 在这种情况下，必须选择其子树中的每个节点。 该算法通过重复激活节点直到子树计数达到所需值来处理此问题，从而有效地选择整个子树。 

另一种情况是当除根之外的所有节点的 Li 为零时。 该算法首先处理叶子，并且仅在到达根时才激活节点，从而确保最小化不必要的选择。 

最后一个微妙的情况是，多个祖先同时需要来自重叠子树的额外骑士。 因为选择总是首先使用最深的可用节点，所以更新往往保持局部化，并且早期的修复不会使已经满足的约束失效。
