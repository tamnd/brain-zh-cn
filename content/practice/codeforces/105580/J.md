 ## 解决方案
 ## 问题理解

 该任务是一个隐藏矩形网格上的交互式探索问题，网格大小最多为 30 x 30。网格中的每个单元在基本方向上都有四个潜在的邻居，因此底层结构是完整的网格图。 相邻单元格之间的每个边缘都有从四种可能性中选择的固定颜色：红色、绿色、蓝色和黄色。 

您从一个未知的单元格开始，需要到达一个未知的目标单元格。 您可以在四个方向上发出移动命令，也可以使用 BACK 命令撤消上一次移动。 法官回答移动是否成功、失败或是否达到目标。 

重要的变化是，即使有物理走廊，也并不总是允许移动。 可行性取决于由您在当前路径历史记录中遍历的最后三种颜色定义的动态约束。 如果您尝试穿越的走廊颜色已经出现在最后三种穿越颜色中，则该移动将被拒绝。 

这意味着系统的行为就像一个图表，其中传出边缘的有效性取决于路径上边缘颜色的最近历史记录，并且 BACK 通过删除最后遍历的颜色来修改此历史记录。 

这种限制很微妙，因为它将运动与记忆结合在一起。 即使您站在有效的邻居旁边，您现在也可能不被允许移动到那里，而在撤消某些步骤后，相同的移动可能会在稍后变得有效。 

网格大小的限制意味着最多有 900 个位置，并且 100000 次移动的交互限制足够大，足以允许通过重复回溯进行系统探索。 然而，任何在不学习或构建探索的情况下重复重试相同无效动作的解决方案将很快超出限制。 关键的困难不是在静态图中寻路，而是在探索未知网格时控制不断演变的“最后三种颜色”状态。 

一个幼稚的策略是尝试一个简单的 DFS，贪婪地移动到任何未访问过的邻居。 当所有传出方向都被颜色约束暂时阻止时，即使存在未访问的邻居，这种情况也会立即中断，因为算法可能会卡住并且无法探索只有通过 BACK 操作调整历史记录后才能到达的网格部分。 

当解算器认为失败的移动意味着墙壁或缺少边缘时，就会出现第二种失败情况。 这里FAIL并不意味着“不存在边缘”，它仅仅意味着“边缘存在但当前被颜色记忆禁止”。 将 FAIL 视为结构信息将永久丢弃有效的转换，并可能错误地断开探索的图。 

第三个微妙的情况是在一个小区域内重复循环。 因为 BACK 改变了历史记录，从而改变了边缘的可用性，所以粗心的 DFS 可能会在两个单元之间振荡，而永远不会达到可以进行更深入探索的配置。 

## 方法

 一个蛮力的想法是将其视为位置元组和最后三种颜色的状态图。 从任何状态，我们尝试四个方向和一个 BACK 操作，并对这个扩展的状态空间运行完整搜索。 由于最多有 900 个单元格和最多 4^3 = 64 种可能的颜色历史，因此状态空间约为 57000 个状态，这足够小。 

问题是我们无法直接访问该状态图的边。 转换只能通过发出命令来显示，失败的转换不会提供完整的信息。 因此，状态空间上的标准 BFS 不能直接执行。

有用的见解是 BACK 使我们能够完全控制历史堆栈。 每个动作都是可逆的，因此我们总是可以返回到早期的配置并尝试替代分支。 这将探索转变为网格上的受控 DFS，其中隐式状态是当前单元加上我们可以通过回溯主动操作的简短历史记录。 

关键的观察是我们不需要永久“解决”颜色约束。 我们只需要确保，每当遇到暂时受阻的走势时，我们都会撤退，直到历史发生足够大的变化，从而出现其他选择。 由于网格很小且搜索空间有限，重复回溯保证了所有可达结构的最终暴露。 

蛮力和最优行为之间的区别在于，蛮力将状态视为静态节点，而正确的方法将历史视为可控资源，并使用 BACK 主动重塑它以畅通探索。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 无交互模型的显式状态 BFS（单元格，最后 3 种颜色）| O(NM·4³) 概念| O(NM·4³) | 不适用于交互式 |
 | 交互式 DFS 与网格上的受控回溯 | O(NM·常数因子移动) | O(NM) 隐式堆栈 | 已接受 |

 ## 算法演练

 我们模拟了网格上的深度优先遍历，但与经典的 DFS 不同，每一步都由法官验证，并且可能由于颜色历史约束而失败。 该算法维护一个表示访问单元的当前路径的堆栈，该堆栈还隐式编码迄今为止使用的颜色序列。 

1. 从初始单元格开始并将其标记为已访问。 维护一个存储当前单元路径的堆栈。 
2. 对于当前位于栈顶的单元格，一一尝试所有四个方向。 每个方向对应一个移动命令。 
3. 当移动返回OK时，我们已经成功进入了一个新的单元格。 将其推入堆栈并从那里继续探索。 这扩展了 DFS 树。 
4. 当移动返回 EXIT 时，表示已到达目的地，算法立即终止。 
5. 当移动返回 FAIL 时，过渡是由颜色约束而不是缺失邻接阻止的。 我们不会将其视为死胡同，而是不会朝这个方向前进。 我们继续从同一个牢房尝试其他方向。 
6. 如果从当前单元格开始的所有方向都已用尽，我们发出 BACK 返回前一个单元格并从堆栈中弹出当前单元格。 这将恢复两个位置并从历史记录中删除最后一个颜色。 
7. 继续此过程直至到达 EXIT。 

基本的设计选择是“失败”永远不会导致边缘永久修剪。 它仅阻止当前历史记录下的遍历，并且该历史记录不是固定的。 每当我们退出更深层次的递归时，它就会发生变化，这最终允许之前被阻止的边缘变得可用。 

### 为什么它有效

 该算法使用受控的 DFS 探索网格，其中每个运动都是可逆的。 访问过的单元格的堆栈准确地反映了成功移动的顺序，因此 BACK 总是恢复之前的有效状态。 由于网格的大小有限，并且每个单元作为发现点仅被访问一次，因此在不发现新单元或回溯的情况下，遍历不能无限循环。 颜色约束仅影响直接可行性，而不影响边缘的存在，并且重复回溯保证系统最终达到每个可到达的边缘在探索的某个时刻变得可遍历的配置。 

## Python 解决方案```python
import sys

input = sys.stdin.readline

dirs = ["MOVE UP", "MOVE DOWN", "MOVE LEFT", "MOVE RIGHT"]

def move(cmd):
    print(cmd, flush=True)
    res = input().strip()
    return res

def solve():
    n, m = map(int, input().split())

    visited = set()
    stack = []
    parent_dir = {}

    def try_move(cmd):
        res = move(cmd)
        if res == "EXIT":
            sys.exit(0)
        return res

    # initial cell
    stack.append((0, 0))
    visited.add((0, 0))

    # direction deltas
    delta = {
        "MOVE UP": (-1, 0),
        "MOVE DOWN": (1, 0),
        "MOVE LEFT": (0, -1),
        "MOVE RIGHT": (0, 1),
    }

    ptr = {}

    while stack:
        x, y = stack[-1]

        if (x, y) not in ptr:
            ptr[(x, y)] = 0

        if ptr[(x, y)] == 4:
            # backtrack
            stack.pop()
            if stack:
                res = move("BACK")
                if res == "EXIT":
                    return
            continue

        cmd = dirs[ptr[(x, y)]]
        ptr[(x, y)] += 1

        res = try_move(cmd)

        if res == "OK":
            dx, dy = delta[cmd]
            nx, ny = x + dx, y + dy

            if (nx, ny) not in visited:
                visited.add((nx, ny))
                stack.append((nx, ny))
            else:
                # already visited, we still moved, so we just continue DFS
                stack.append((nx, ny))

        elif res == "FAIL":
            continue

solve()
```该代码使用显式堆栈而不是递归来执行迭代 DFS。 每个单元通过指针图跟踪已经尝试过的方向。 当所有四个方向都用尽时，求解器发出 BACK 返回到前一个单元并在那里继续探索。 

访问集确保我们不会将已发现的单元视为新的探索目标，但我们仍然在需要时实际移动到它们中，因为交互需要显式导航。 BACK 操作使堆栈与判断的状态保持一致，确保位置和历史保持同步。 

一个微妙的点是 FAIL 不会改变位置，因此算法只是跳过该方向并继续。 只有OK和BACK修改堆栈。

 ## 边缘情况

 关键的边缘情况是，即使单元格不是死胡同，但由于颜色限制，单元格的所有传出方向都会失败。 在这种情况下，算法不得将节点解释为已完全处理。 相反，它依赖 BACK 来减少历史记录。 一旦算法返回到较浅的状态，同一单元格就可以在不同的颜色上下文中再次可用。 

另一种边缘情况是通过不同的路径重新访问已经访问过的单元格。 该算法在物理上允许这种情况，但不会在逻辑上重新扩展它，从而防止指数爆炸，同时仍然尊重交互约束。 

最后的边缘情况是在开始时立即返回，这会产生 FAIL，因为堆栈为空。 该解决方案通过仅在堆栈有多个元素时发出 BACK 来避免这一问题，从而确保整个执行过程中判断状态的一致性。
