 ## 解决方案
 ## 问题理解

 我们得到一组固定的字典单词，每个单词都很短，然后是大量的查询。 每个查询提供一个单词和一个排名。 

对于查询词，我们使用后缀匹配将其与每个词典单词进行比较。 如果字典单词与查询单词至少共享一个后缀，则该单词被视为相关。 在所有此类匹配中，我们首先确定出现在查询单词和任何字典单词之间的最大可能后缀长度。 仅保留达到此最大后缀长度的字典单词。 从这些单词中，我们按字典顺序对单词进行排序并选择第 k 个单词。 答案不是单词本身，而是它在输入列表中的原始位置。 如果所选组中没有足够的单词，则答案为-1。 

关键的结构细节是单词非常短，长度最多为 5。这立即限制了可以出现的不同后缀的数量，因为每个单词仅贡献恒定数量的后缀。 

简单的解释会尝试将每个查询单词与所有字典单词进行比较，并直接计算后缀匹配。 对于多达 10^5 个字典单词和 10^5 个查询，即使每次比较都很短，这也会太慢。 

第二个微妙的问题是“最佳匹配”是针对每个查询全局定义的，而不是针对每个单词独立定义的。 如果另一个单词与较长的后缀匹配，则与较短后缀匹配的字典单词是无关的。 

当存在多个后缀长度时，会出现简单的失败情况。 

例如，如果字典单词是`["aba", "caba"]`查询是`"xxcaba"`, 两个词都匹配后缀`"aba"`但仅`"caba"`匹配后缀长度 4。正确答案必须完全忽略所有长度为 3 的匹配。 

另一个陷阱是在按后缀长度过滤后忘记了字典顺序。 两个单词可能都匹配最佳后缀长度，但答案取决于排序顺序，而不是输入顺序。 

## 方法

 直接强力解决方案将每个查询单词与每个字典单词进行比较，并通过从末尾扫描最多 5 个字符来计算它们的最长公共后缀。 由于每次比较的成本为 O(5)，因此单个查询的成本为 O(5n)，所有查询的成本为 O(5nq)，大约是 10^10 次操作。 这已经远远超出了时间限制。 

单词长度最多为 5 的约束完全改变了结构。 每个词典单词仅提供一小组后缀：最后一个字符、最后两个字符等等，直到整个单词。 如果我们反过来思考，我们可以按后缀对字典单词进行预先分组，而不是将查询与所有单词进行比较。 

对于每个字典单词，我们生成其所有后缀并将该单词存储在以该后缀字符串为键的存储桶中。 每个存储桶代表共享该后缀的所有字典单词。 如果查询以相同的后缀结尾，我们可以立即检索该后缀长度的所有匹配单词。 

关键的观察是，答案仅取决于此预先计算的结构中存在的最长后缀。 因此，对于每个查询，我们从最长到最短检查其后缀，直到找到一个非空桶。 该桶正是实现最大可能后缀匹配长度的单词集合。 然后，我们在预处理期间按字典顺序对该存储桶进行一次排序，以便每个查询都可以直接对其进行索引。 

这将每个查询减少为最多检查 5 个后缀，然后执行简单的第 k 个选择。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(nq·5) | O(1) | O(1) | 太慢了|
 | 预先计算的后缀桶 | O((n + q) · 5 + Σ 排序) | O(n·5) | 已接受 |

 ## 算法演练

 1. 对于每个字典单词，通过取最后一个字符、最后两个字符等直到完整单词来生成其所有后缀。 将单词的原始索引存储在由每个后缀作为键控的映射中。 这构建了从任何后缀字符串到包含它的所有单词的直接访问。 
2.处理完所有字典单词后，将每个后缀桶中存储的单词列表按照单词本身的字典顺序排序。 我们保留该单词及其原始索引，以便稍后输出所需的位置。 
3. 对于每个查询词，按长度降序考虑其后缀。 从整个单词开始，逐渐删除第一个字符，直到只剩下一个字符。 
4. 对于每个后缀，检查它是否存在于预先计算的映射中。 找到的第一个后缀对应于与任何字典单词共享的最大可能后缀长度。 
5. 如果在映射中未找到后缀，则输出 -1，因为没有字典单词与查询共享任何后缀。 
6.否则，取相应的桶。 如果其大小小于k，则输出-1。 如果足够大，则选择按字典顺序排列的第k个元素，并输出其存储的原始索引。 

正确性来自于后缀是严格按照长度递减顺序检查的。 第一个匹配保证了最大值，因此该存储桶之外的任何字典单词都不再存在后缀匹配。 该存储桶中的所有单词都完全共享该后缀长度，并且根据问题的定义，匹配较短后缀的任何单词都是不相关的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    words = []
    buckets = {}

    for i in range(n):
        w = input().strip()
        words.append(w)
        L = len(w)
        for j in range(L):
            suf = w[j:]
            if suf not in buckets:
                buckets[suf] = []
            buckets[suf].append((w, i + 1))

    for suf in buckets:
        buckets[suf].sort(key=lambda x: x[0])

    q = int(input())
    for _ in range(q):
        s, k = input().split()
        k = int(k)

        found = None

        for i in range(len(s)):
            suf = s[i:]
            if suf in buckets:
                found = buckets[suf]
                break

        if not found or len(found) < k:
            print(-1)
        else:
            print(found[k - 1][1])

if __name__ == "__main__":
    solve()
```预处理阶段从后缀字符串到共享该后缀的所有字典单词构建字典。 每个存储的条目都保留单词及其原始位置，因为排序是按单词完成的，但输出需要原始索引。 

每个桶都排序一次，这样每个查询都可以直接访问第 k 个字典元素，而无需重新计算。 

查询时，按照从最长到最短的顺序测试查询词的后缀。 第一次成功查找保证了最大后缀匹配长度。 然后使用相应的桶来响应排名请求。 

一个微妙的点是我们从不动态重新计算后缀匹配。 所有比较都简化为对预先计算的键进行字典查找，从而保持查询时间恒定。

现在考虑：

 词典：`["abc", "xbc", "c", "bc"]`询问：`"zzbc", k = 2`| 步骤| 后缀已检查 | 存在于地图 | 已选桶 |

该跟踪显示了后缀选择如何将问题简化为单个预先计算的组，以及该组内的排序如何确定最终答案。 

## 复杂度分析

 | 测量| 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 | O(n·L + q·L + 总排序) | 每个单词最多贡献 5 个后缀，每个查询最多检查 5 个后缀，每个桶排序一次 |

 ## 边缘情况

 查询匹配多个后缀长度的情况是通过从最长到最短扫描后缀来处理的。 第一次成功的查找始终对应于最大可能的匹配长度，因为只有在用尽较长的后缀后才会考虑每个较短的后缀。 

对于像这样的查询`"abcd"`与字典单词`"bcd"`和`"cd"`，算法首先检查`"abcd"`， 然后`"bcd"`，并立即停止在`"bcd"`。 较短的后缀`"cd"`永远不会被考虑，这保留了“最大后缀长度”要求的正确性。 

另一个边缘情况是多个单词共享相同的最佳后缀，但以任意输入顺序出现。 由于每个桶在构造后都按字典顺序显式排序，因此第 k 个元素的选择与输入顺序无关，从而防止了因插入顺序而导致的错误答案。
