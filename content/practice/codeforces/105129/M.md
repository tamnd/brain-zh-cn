 ## 解决方案
 ## 问题理解

 我们正在验证的系统非常简单：对于每个测试用例，我们都会被告知程序应该运行多少次测试以及它实际通过了多少次测试。 由此，我们决定该计划是否完全成功。 

每个输入测试用例都描述一次提交。 第一个数字是该提交的测试用例总数。 第二个数字是提交成功通过的测试用例数量。 所需的决策是二元的。 如果每个测试用例都通过了，我们就声明提交完全正确并输出 AC。 如果哪怕有一个测试用例没有通过，我们就会输出 WA。 

约束很小，每个查询的测试用例数量最多为 100。这意味着即使是最简单的每个测试用例处理在计算成本方面也是微不足道的。 不需要预计算、数据结构或优化技巧，因为每次检查的输入大小是恒定有界的，并且每次检查的操作是恒定时间比较。 

边缘情况主要是关于边界平等。 最重要的情况是当通过的测试数等于测试总数时，包括两者都为 1 的最小可能情况。例如，像这样的输入`1 1`必须输出交流电。 稍微不同的输入，例如`1 0`必须输出 WA，因为至少有一项测试失败。 幼稚实现中的一个常见错误是不小心检查是否`m > 0`而不是`m == n`，这会错误地接受类似的情况`3 2`。 

另一个微妙的问题是忘记了通过所有测试意味着完全平等，而不是“接近完全覆盖”。 输入如`100 99`即使成功率很高，仍然应该产生 WA。 

## 方法

 问题的强力解释将模拟单独检查每个测试用例。 人们可以想象迭代所有 n 个测试并验证每个测试是否被标记为通过。 然而，这是不必要的，因为输入已经提供了聚合信息：成功测试的数量 m。 

在直接模拟中，我们将概念性地重建一个大小为 n 的数组，并将其中 m 个标记为通过，然后验证是否存在任何失败。 每个测试用例的成本为 O(n)。 虽然 n 最多为 100，但这种方法在逻辑上是多余的，因为我们永远不需要检查单个测试结果。 

关键的观察结果是整个问题简化为单个相等性检查。 如果成功测试的数量等于测试总数，则没有失败。 否则，至少存在一处故障。 这将问题从结构上的迭代分解为单个比较。

 | ---| ---| ---| ---|
 | 蛮力 | O(T·n) | O(T·n) | O(1) | O(1) | 已接受但没有必要 |
 | 最佳| O(T)| O(1) | O(1) | 已接受 |

 ## 算法演练

 我们独立处理每份提交内容，并通过直接比较来决定其结论。 

1.读取测试用例T的数量。这决定了我们将评估多少提交。 
2. 对于每次提交，读取两个整数n和m。 第一个是测试总数，第二个是通过的测试数量。 这两个值完全决定了正确性，因此不需要额外的数据。 
3. 比较 m 和 n。 如果它们相等，则意味着每次测试都成功。 否则，至少有一项测试失败。 
4. 如果m等于n，则输出“AC”，否则输出“WA”。 

逻辑有意最小化，因为输入已经编码了完整的评估结果。 

### 为什么它有效

 对于每次提交，失败测试集的大小为 n − m。 仅当该数量为零时才接受提交。 由于 n − m = 0 等价于 m = n，因此等式充分表征了正确性。 测试用例之间不存在隐藏结构或依赖关系，因此这个条件既是必要的也是充分的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    out = []
    for _ in range(t):
        n, m = map(int, input().split())
        if n == m:
            out.append("AC")
        else:
            out.append("WA")
    print("\n".join(out))

if __name__ == "__main__":
    solve()
```该解决方案读取所有测试用例一次并将结果存储在列表中以避免重复的 I/O 开销。 核心决策是平等检查`n == m`，它直接编码是否有测试用例失败。 除了确保正确解析输入行之外，没有任何边缘条件。

 ## 边缘情况

 最小的有意义的情况是`n = 1`。 用于输入`1 1`，该算法比较相等性并立即输出 AC，因为 m 等于 n。 为了`1 0`，条件失败并产生 WA。 这证实了边界值的正确处理，无需任何特殊的外壳。 

对于近边界情况，例如`100 99`，比较仍然正确地识别出至少一个测试失败。 计算不依赖于大小，只依赖于相等性，因此大值与小值的行为相同。
