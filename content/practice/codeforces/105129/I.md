 ## 解决方案
 ## 问题理解

 每个测试用例都会给出一组果汁瓶，其中第 i 个瓶子包含一定数量的升。 Abdelaleem 将恰好为 m 个朋友服务，对于每个选定的瓶子，他可以向每个朋友的杯子中倒入相同整数升的水。 如果他不使用瓶子，那就完全被忽略了。 

对所选瓶子的限制非常严格。 平分给m个朋友后，瓶子里剩下的液体一定不能成为小剩菜。 要么以不剩任何东西的方式使用瓶子，要么以剩余至少 k 升的方式使用瓶子。 如果所选的分流方式不能满足这两个条件，则该瓶子不能用于该配置。 

任务是最大化每个朋友的果汁量，假设他为每个朋友选择使用的所有选定瓶子的单一统一量。 

关键的解释是，我们选择一个整数 x，代表每个朋友从每个用过的瓶子中接收到多少升。 对于每个容量为 a 的瓶子，我们要么不使用它，要么将其拆分，以便分配 m·x 升，留下剩余的 a − m·x。 该余数必须满足 a − m·x = 0 或 a − m·x ≥ k。 

输入大小很大，每个测试用例最多 5 × 10^5 个瓶子，值最多 10^9。 这排除了对每个瓶子的 x 的任何按值搜索。 任何尝试针对每个测试用例独立检查每个候选 x 的可行性的解决方案都会太慢。 该结构强烈表明每个瓶子对x贡献一个约束区间，最终答案是同时满足所有约束的最大整数x。 

当天真的方法假设每个瓶子都必须被使用时，就会出现微妙的失败案例。 例如，如果一个瓶子不能干净地分开或留下足够的剩余物，则应该简单地跳过它。 另一种失败情况是将余数条件视为始终要求整除或始终要求余数 ≥ k，而不考虑这两个选项都被允许。 

## 方法

 直接尝试是尝试从 1 到最大 ai 的所有可能的 x 值，并针对每个 x 检查每个瓶子。 对于固定的 x，通过计算 m·x 并检查它是否符合精确除法或留下至少 k 个剩余部分来测试每个瓶子。 这是正确的，但太慢了。 最坏的情况涉及 5 × 10^5 个瓶子和最多 10^9 个候选值，导致不可能进行 10^14 规模计算。 

关键的观察是每个瓶子独立定义允许的 x 值。 对于固定的 a，条件分为两种情况。 

如果瓶子被完全使用，则 a = m·x，当可整除时，它会提供单个候选 x = a / m。 

如果它被部分使用，那么我们需要 a − m·x ≥ k，重新排列为 x ≤ (a − k) / m。 

因此，每个瓶子允许所有 x 达到某个阈值，可能还加上一个额外的精确值。 精确值选项与最大化 x 无关，因为任何单个精确点都不能超过所有约束的最大阈值，除非与其一致。 真正的限制是，对于每个瓶子，我们必须忽略它或确保 x 满足 x ≤ (a − k) / m。 这意味着每个瓶子对 x 施加上限约束。

然而，只有在不强制违反规则的情况下，忽略瓶子才有效。 只有当选择不使用瓶子时，才可以安全地忽略它，因为声明允许完全不喝饮料，所以情况总是如此。 因此，全局可行性降低为确保 x 不超过任何活动瓶子的阈值。 因此，最优 x 是所有有效贡献的最大值，计算为最佳 x，使得跨瓶子至少存在一种有效配置，这只是满足所有每瓶约束的最大可行 x。 

对于每个瓶子，当 a ≥ k 时，最强的约束是 x ≤ (a − k) / m，否则没有任何贡献。 答案就是最大 x，这样至少有一个瓶子可以支持它，并且由于我们正在最大化 x，因此我们在所有有效的每瓶边界上取最大值。 

这将问题简化为扫描所有瓶子并计算从每个瓶子中得出的最佳可能候选值。

 | ---| ---| ---| ---|
 | 暴力破解 x | O(n·最大 a) | O(1) | O(1) | 太慢了 |
 | 每瓶限制提取量| O(n) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们将问题重新解释为选择一个最佳的每个朋友数量 x，该数量与至少一种独立处理每个瓶子的有效方式兼容。 

1. 对于每个值为 a 的瓶子，计算它可以支持的完整 m 份量的最大数量，同时留下零或至少 k 个剩余量。 我们关注“至少保留 k”条件，因为它产生最大的可行 x 范围。 每当 a ≥ k 时，这就会导致候选边界 x ≤ (a − k) // m。 
2. 如果 a < k，则瓶子不能提供有效的剩余配置。 唯一可能的用法是精确除法，仅当 a 可被 m 整除时才有效，得到 x = a // m。 这是一个固定值而不是一个范围。 
3. 对于每个瓶子，提取它可以支持的最佳可能值。 对于 a ≥ k，这是 (a − k) // m。 对于 a < k，如果能被 m 整除，则它是 a // m，否则它没有任何贡献。 
4. 跟踪每瓶所有有效候选者的最大值。 直觉是，任何可实现的 x 必须由至少一种瓶子配置支持，最佳策略是选择允许最大 x 的无瓶颈瓶子。 
5. 返回找到的最大值，如果没有瓶子支持任何有效的份量，则返回 0。 

它之所以起作用是基于这样的结构：每个瓶子除了通过 x 的共享选择之外不会与其他瓶子相互作用。 一旦 x 确定，每瓶的可行性都是独立的。 这种独立性将全局优化分解为从每瓶可行集的并集中选择最佳允许的 x，并且由于我们最大化 x，因此解决方案由所有有效集的最大上限确定。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    T = int(input())
    for _ in range(T):
        n, m, k = map(int, input().split())
        a = list(map(int, input().split()))
        
        ans = 0
        
        for x in a:
            if x >= k:
                ans = max(ans, (x - k) // m)
            else:
                if x % m == 0:
                    ans = max(ans, x // m)
        
        print(ans)

if __name__ == "__main__":
    solve()
```该代码独立处理每个测试用例并维护运行的最大答案。 主要逻辑位于瓶子循环内部，其中每个瓶子都减少为单个候选值。 减法 k 处理剩余约束，而整数除法 m 将剩余可用量转换为每个朋友的分配。 

一个常见的陷阱是忘记了 a < k 的瓶子仍然可以通过精确整除做出贡献。 另一个是混合 (a - k) // m 中的运算顺序，必须在验证 a ≥ k 后计算以避免负面影响。 

## 工作示例

 考虑一个 n = 3、m = 3、k = 2 和瓶子 [10, 7, 4] 的小例子。 

| 瓶子| 一个 | 状况 | 候选人 x |
 | ---| ---| ---| ---|
 | 1 | 10 | 10 (10 − 2) // 3 = 2 | 2 |
 | 2 | 7 | (7 − 2) // 3 = 1 | 1 |
 | 3 | 4 | (4 − 2) // 3 = 0 | 0 |

 最大值为 2。这表明即使一瓶允许更高的值，更紧的瓶子也会限制答案。 

现在考虑 n = 2、m = 4、k = 5 和瓶子 [16, 9]。 

| 瓶子| 一个 | 状况 | 候选人 x |
 | ---| ---| ---| ---|
 | 1 | 16 | 16 (16 − 5) // 4 = 2 | 2 |
 | 2 | 9 | (9 − 5) // 4 = 1 | 1 |

 答案是2，由第一个瓶子驱动。 第二瓶较弱，但不影响最大值。 

这些例子证实了每个瓶子独立地产生 x 上的上限，并且答案由最强的这样的上限决定。

 ## 边缘情况

 一个棘手的情况是当 k 足够大时，大多数瓶子都属于 a < k 类别。 对于输入 n = 2、m = 3、k = 50 以及瓶子 [48, 51]，算法仅当 48 可被 m 整除时才将其视为候选者，否则它不会做出任何贡献。 由于 48 不能被 3 整除，因此被忽略。 对于 51，候选项是 (51 − 50) // 3 = 0，所以答案是 0。 

逐步执行表明不需要中间状态。 每个瓶子都是独立评估的，只有有效的算术路径才能发挥最大作用。
