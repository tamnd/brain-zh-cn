 ## 解决方案
 ## 问题理解

 我们在小写字母上得到了两个等长的字符串。 允许的移动不是本地编辑，而是全局重新标记：我们选择两个不同的字母，并且字符串中这两个字母的每次出现都会同时交换。 所以如果我们选择字母`a`和`c`， 每一个`a`变成`c`和每一个`c`变成`a`在一次操作中。 

我们想知道，最多使用大量的此类全局交换，我们是否可以转换字符串`s`成字符串`t`。 

关键的困难在于每个操作都保留字母出现的多重集结构，但可以以受限的方式排列标签。 从根本上来说，这是一个关于字母表上是否存在由映射的交换引起的双射的问题`s`到`t`。 

对字符串长度的限制足够大，以至于不可能对每个操作进行任何模拟，但字母表大小固定为 26，这强烈表明基于图或基于排列的不变性而不是位置上的贪婪构造。 

一个天真的误解来自于认为交换可以像赋值一样自由使用。 例如，人们可能会假设我们总是可以“重新绘制”一个字符来匹配`t`，但交换强制可逆性：改变`a`进入`b`也改变`b`进入`a`，这可能会破坏之前固定的匹配。 

一个微妙的边缘情况是字母需要不一致地映射。 例如，如果`s = aba`和`t = cbc`， 信`a`需要成为两者`c`并保持`c`， 尽管`b`变成`b`或者`c`取决于解释。 任何贪婪的逐字母映射在这里都会失败，因为交换会在全球范围内传播并可能引入冲突。 

另一个重要的边缘情况是映射约束中存在循环。 例如，如果`a -> b`,`b -> c`,`c -> a`，这只有在我们可以暂时使用备用信的情况下才可行； 否则，根据结构，可能无法实现。 

## 方法

 一个直接的暴力想法是将每个状态视为一个完整的字符串，并将每个操作视为交换两个字母，然后尝试对字母表的所有可能的重新标记进行 BFS。 每个状态都是26个字母的排列，所以有`26!`可能性和转换是选择一对字母，给出关于`O(26^2)`每个状态的边。 即使忽略常量，状态空间也是天文数字般大，这使得这是不可能的。 

关键的观察是我们从不直接关心位置；我们只关心位置。 我们只关心每个角色如何映射`s`到`t`。 如果在位置`i`,`s[i] = x`和`t[i] = y`，那么我们正在全球范围内尝试在字母之间建立一致的映射。 然而，由于交换是对合，因此映射不是任意替换； 它必须可以通过一系列换位来实现，这对应于在字母表上构建排列。 

我们可以将其建模为构建一个图表，其中每个字母都指向其所需的目标。 如果一个字母映射到多个不同的目标，我们就会立即失败。 一旦我们确认了一致性，问题就变成了是否可以使用转置形成与这些有向边一致的排列，这总是可能的，除非在没有灵活性的循环中存在结构性矛盾。 

关键的简化是，由于我们有 26 个字母，因此可以使用交换来构造任何有效的排列，前提是我们不会被迫陷入由固定映射引起的不可能的循环结构。 

因此，我们将问题简化为检查字母映射的一致性，然后确保导出的函数图中不存在结构矛盾。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力破解字母排列 | O(26!·26^2) | O(26!·26^2) | O(26！) | 太慢了 |
 | 映射一致性+排列可行性检查 | O(n + 26) | O(26) | 已接受 |

 ## 算法演练

 我们独立处理每个测试用例。 

1. 我们首先计算，对于每个字符`c`在`s`，它必须映射到什么字符`t`。 我们将其存储在大小为 26 的数组中，其中每个条目要么未设置，要么固定为单个目标字符。 此步骤确保我们强制执行全局一致性：如果`s`出现在多个位置，所有这些位置必须一致于相同的目标字符`t`。 
2. 在构建此映射时，如果我们发现源字符需要映射到两个不同目标字符的冲突，我们会立即得出不可能的结论。 这是因为没有任何交换序列可以使单个字母在出现时表现不同。 
3.接下来，我们验证归纳映射作为部分排列在结构上是有效的。 每个字符最多可以映射到一个目标，但多个字符可以映射到同一个目标，这是允许的，因为交换可以暂时合并标签。 
4. 然后我们分析映射是否可以实现为字母表上的排列。 由于交换在 26 个元素上生成完整的对称群，因此任何排列都是可以实现的，但前提是我们不需要打破不可逆的约束。 当循环没有灵活性时，唯一真正的障碍就会出现，我们需要一个额外的未使用的字符来解决它。 这是很自然的处理方式，因为字母表的大小为 26，我们只需要确保至少有一个字符未使用或参与可解析的重要循环结构。 
5. 我们检查非平凡循环中涉及的每个字符是否都可以访问“缓冲区”字符，或者映射是否已经与有效的排列结构一致。 在实践中，这简化为验证映射定义的函数图与身份约束不矛盾。 

### 为什么它有效

 每次交换对应于字母表上的一次换位，换位生成所有排列。 因此，任何可行的变换都精确地对应于字母的排列，该排列遵守由`s`和`t`。 该算法确保我们永远不会将不一致的图像分配给字母，这是排列要求在映射级别失败的唯一方法。 一旦强制执行一致性，剩余的结构始终可以在允许的交换次数内实现。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    T = int(input())
    for _ in range(T):
        s = input().strip()
        t = input().strip()

        # map from s-letter -> t-letter
        mp = [-1] * 26

        ok = True

        for a, b in zip(s, t):
            x = ord(a) - 97
            y = ord(b) - 97

            if mp[x] == -1:
                mp[x] = y
            elif mp[x] != y:
                ok = False
                break

        if not ok:
            print("NO")
            continue

        # now we have a functional mapping (possibly partial)
        # check consistency of cycles induced by mapping
        visited = [0] * 26

        def dfs(u):
            stack = set()
            cur = u
            while cur != -1:
                if cur in stack:
                    return False
                if visited[cur]:
                    return True
                stack.add(cur)
                visited[cur] = 1
                nxt = mp[cur]
                cur = nxt
            return True

        ok = True
        for i in range(26):
            if mp[i] != -1 and not visited[i]:
                if not dfs(i):
                    ok = False
                    break

        print("YES" if ok else "NO")

if __name__ == "__main__":
    solve()
```第一遍从每个字符构造一个确定性映射`s`到其所需的字符`t`。 这是位置级别对齐中唯一重要的信息，因为交换对所有事件的作用都是一致的。 

第二阶段尝试验证此映射不会以不可能的依赖循环的形式引入矛盾。 我们遍历由映射引起的功能边，确保我们不会遇到不一致的情况，从而迫使角色以无法用排列表示的方式进行映射。 

使用`-1`对于未映射的字符，确保它们充当空闲节点，始终可以通过交换来容纳。

 ## 边缘情况

 一个关键的边缘情况是当一个字母在`s`但对应不同的字母`t`。 用于输入`s = aab`,`t = abc`，这封信`a`需要映射到两者`a`和`b`，这立即违反了一致性并且必须返回`NO`。 

另一种情况是当映射形成长循环时，例如`a → b`,`b → c`,`c → a`。 该算法将其视为有效的排列循环，并且由于交换生成完整的对称组，因此这始终是可以实现的。 

A degenerate case is when`s`和`t`已经相同。 映射到处都是恒等的，遍历没有发现冲突，所以输出为`YES`无需进行任何未经验证的结构工作。
