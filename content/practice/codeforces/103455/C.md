 ## 解决方案
 ## 问题理解

 我们给出了一个简单的红绿灯游戏的时间表，以及一组试图在红灯永久变红之前到达终点线的参赛者。 灯在零时间开始处于红色状态，然后在按升序给出的特定时刻在红色和绿色之间切换。 最后一次记录的开关后，灯将永远保持红色。 

每个参赛者从零位置开始并希望达到目标距离$K$。 当绿灯亮时，他们以恒定的个人速度前进。 当红灯亮时，它们根本不动。 它们可以立即启动和停止，因此无需考虑加速延迟或惯性。 问题是确定每个参赛者是否能够达到这个距离$K$严格在光线变为永久红色之前。 

输入提供灯光变化的数量$M$, 参赛者数量$N$，目标距离$K$、灯光切换状态的时间列表以及每个参赛者的速度。 输出是每个参赛者的二进制指示器：如果及时到达终点则为 1，否则为 0。 

约束足够小，可以对每个参赛者在所有果岭区间进行直接模拟。 和$M, N \le 5000$,天真的$O(MN)$或者$O(M+N)$每个测试策略就足够了，因为操作总数最多约为$2.5 \cdot 10^7$，如果实现得干净的话，它完全符合 Python 的时间限制。 

一个微妙的点是光的交替结构。 灯开始呈红色，因此第一个间隔可达$t_1$是红色，然后是绿色$t_1$到$t_2$，然后再次变为红色，依此类推。 另一个边缘情况是，最终开关后的移动并不重要，因为此后灯永远呈红色。 

一个天真的错误是假设运动是连续的或将所有绿灯时间视为单个块。 例如，如果开关有时$2, 5$，光仅在这些精确间隔之间为绿色，而不是除了这些点之外的所有地方。 对此的误解会导致过多计算移动时间并错误地宣布所有快速玩家为胜利者。 

当参赛者所需的时间与绿色间隔的结束时间完全匹配时，就会出现另一种边缘情况。 由于信号灯在转换时间立即变红，因此只有当他们严格之前或在那一刻完成距离时，在该确切时刻到达终点仍然有效，具体取决于解释。 正确的模型将达到或之前达到截止点视为成功。 

## 方法

 蛮力的想法很简单：独立模拟每个参赛者的光线变化时间线。 保持竞争对手目前已移动的距离，并迭代连续切换时间之间的每个间隔。 对于每个绿色间隔，计算它们行驶的距离（速度乘以持续时间），并累加直到它们达到$K$或者时间线结束。 

这种方法是正确的，因为它直接对真实过程进行建模。 失败点是性能。 每个参赛者都会扫描所有$M$间隔，所以复杂度为$O(NM)$，在最坏的情况下会达到 2500 万次操作。 这是边界，但在优化的 Python 中仍然可以接受，尽管它在概念上变得浪费。 

关键的观察结果是，每个绿色段内的运动都是确定性和线性的，因此我们不需要模拟逐秒的行为。 相反，我们可以独立对待每个竞争对手，但直接根据预先计算的绿色持续时间计算累积可达距离。 这将问题简化为每个参赛者在时间线上一次通过，而无需任何每秒模拟。 

进一步简化是，我们只需要到达之前的总绿灯时间$K$，而不是段内的确切停止点。 如果竞争对手需要时间$K / p_i$，我们可以贪婪地累积绿色间隔，直到超过所需的时间。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 每秒暴力模拟 |$O(N \cdot \sum t)$|$O(1)$| 太慢了|
 | 每个参赛者的间歇模拟|$O(NM)$|$O(1)$| 已接受 |

 ## 算法演练

 我们将切换时间转换为交替的红色和绿色间隔。 由于红灯亮起，第一个间隔$[0, t_1)$是红色的，$[t_1, t_2)$是绿色的，等等。 最后一个间隔之后$t_M$永远是红色的，所以它对运动没有任何贡献。 

对于每个参赛者，我们计算他们在绿色间隔期间可以累积多少距离，直到他们到达$K$或者我们的绿灯时间用完了。 

### 步骤

 1. 通过从第一个红到绿转换开始配对连续的切换时间来构建绿色间隔列表。 由于初始状态为红色，因此第一个可用间隔开始于$t_1$如果$M \ge 1$。 每个绿色区间为$[t_{2i-1}, t_{2i})$。 每个绿色间隔的持续时间为$t_{2i} - t_{2i-1}$。 
2. 每位参赛者的速度$p_i$，计算如果他们连续移动则到达终点所需的时间，即$K / p_i$。 这给出了他们必须积累的目标绿灯时间。 
3. 按顺序迭代绿灯间隔，从剩余所需绿灯时间中减去每个间隔的持续时间。 如果某个时间间隔完全满足剩余要求，我们就会提前停止并将竞争对手标记为成功。 
4. 如果所有果岭间隔都用完，且所需时间仍为正数，则选手无法及时到达终点。 

### 为什么它有效

 关键的不变量是，只有绿色间隔对进度有贡献，并且在每个绿色间隔内，进度以恒定速度呈线性。 因此，到达终点仅取决于累计的果岭时间，而不取决于其如何分配。 该算法通过按时间顺序精确求和可用持续时间来保留此不变性，确保不会重复计算或错误排序绿灯时间。 一旦达到所需的总绿灯时间，相应的物理距离$p_i \cdot t$保证竞争对手已达到或超过$K$。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    m, n, k = map(int, input().split())
    t = list(map(int, input().split()))
    p = list(map(int, input().split()))

    green = []
    # light starts red, so t[0] is red->green
    for i in range(0, m - 1, 2):
        green.append(t[i + 1] - t[i])

    # for each player, check if enough green time exists
    res = []

    for speed in p:
        need = k / speed
        ok = False
        for dur in green:
            if need <= dur:
                ok = True
                break
            need -= dur
        res.append("1" if ok else "0")

    print(" ".join(res))

if __name__ == "__main__":
    solve()
```该代码首先通过配对连续的切换时间来构造可用的绿色段。 循环结束`range(0, m - 1, 2)`反映红到绿和绿到红的结构。 

对于每个参赛者来说，变量`need`表示完成距离需要多少绿灯时间$K$。 每个绿色间隔都会减少这一要求。 那一刻`need`在一段时间内变为负数，则竞争对手成功。 

一个常见的陷阱是尝试直接使用浮点累加来模拟距离。 虽然这在这里有效，但更稳定的解释是将问题减少到所需时间而不是累积距离，这避免了关于部分间隔消耗的推理错误。

 ## 边缘情况

 一种边缘情况是根本没有绿色间隔或者第一次切换发生得很晚。 在这种情况下，所有竞争对手都会失败，除非$K = 0$。 该算法处理这个问题是因为`green`列表变空，因此每个参赛者都会立即耗尽可用时间。 

另一个极端情况是，参赛者所需的时间与所有果岭间隔的总和完全匹配。 在这种情况下，减法循环恰好在零处结束，并且条件`need <= dur`在最终间隔边界触发成功。 这符合允许在最后可能时刻完成的预期行为。 

第三种边缘情况是交替非常小的间隔，其中多个短绿色窗口组合起来勉强达到所需的阈值。 累积减法方法确保碎片无关紧要，因为所有间隔都会按顺序消耗，直到满足要求。
