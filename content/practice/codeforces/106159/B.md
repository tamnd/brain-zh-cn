 ## 解决方案
 ## 问题理解

 我们得到一个数字字符串，它表示两个连接的整数 A 和 B。这两个数字最初来自地址格式，其中 A 和 B 是单独的组成部分，但在消息中它们已粘合在一起，没有任何分隔符。 我们的任务是恢复原来的一对。 

A 和 B 都是 1 到 99 范围内的严格正整数。它们以标准十进制形式书写，因此不允许有前导零。 输入字符串保证是某些 A 和 B 的有效串联，但关键困难在于这种串联可能不是唯一可解码的。 

输出取决于是否存在一种有效方法将字符串拆分为满足约束的两个数字。 如果恰好有一个有效的分割，我们就会打印它。 如果有多个有效的分割，我们打印-1。 

输入长度最多为4位。 这是至关重要的：这意味着我们处理的不是组合搜索空间，而是非常小的、恒定数量的可能性。 就性能而言，尝试所有可能的拆分的任何方法都已经足够了。 

主要的边缘情况是由可变数字长度引起的歧义。 例如，字符串 111 可以解释为 1 和 11，或者 11 和 1，两者都有效。 相反，像 1020 这样的字符串可能只有一种有效的解释，具体取决于 10 和 20 是否有效以及 1 和 020 是否由于前导零规则而无效。 

另一个微妙的情况是，当分割产生大于 99 或等于 0 的数字时，即使它符合数字边界，也必须拒绝该数字。 

## 方法

 暴力破解的想法很简单：尝试所有可能的位置将字符串分成两部分。 由于字符串长度最多为 4，因此最多有 3 个分割点。 对于每个分割，将左子字符串解释为 A，将右子字符串解释为 B，然后验证两者。 

验证意味着检查两个部分都没有前导零，除非它恰好是“0”（这实际上是无效的，因为 A 和 B 从 1 开始），并且两个值都在 1 到 99 之间（包括 1 和 99）。 每次检查的时间复杂度为 O(1)，因此整个过程是常数时间。 

由于输入大小是恒定的，暴力方法已经可以有效地工作。 不需要高级优化，但概念上的简化来自于认识到问题纯粹是约束下的枚举，而不是搜索大空间。 

关键的观察结果是约束完全限制了解决方案空间。 由于 A 和 B 最多各为两位数字，因此任何有效的分割都必须遵守 1 到 2 之间的数字长度。这限制了我们检查第一部分长度为 1 或 2 或 3 的分割，但实际上总字符串长度最多为 4，因此枚举仍然微不足道。 

最后一步是计算存在多少个有效的分割。 如果恰好存在，我们输出它； 否则，歧义会强制输出 -1。

 ## 算法演练

1. 读取输入字符串 S。该字符串表示两个没有分隔符的串联数字，因此每个可能的分割点都是候选解释。 
2. 初始化一个计数器以获取有效解释，并初始化一个变量来存储唯一答案（如果存在）。 
3. 尝试在从 1 到 len(S) - 1 的每个可能位置 i 将 S 拆分为两个非空部分。每个拆分对应于 A = S[:i] 和 B = S[i:]。 
4. 对于每次分割，将两个子字符串转换为整数，并验证两者是否在 1 到 99 的范围内。此步骤确保我们只接受有效的地址组件。 
5. 如果分割有效，则增加计数器并存储该对。 如果找到多个有效的分割，我们就可以得出歧义的结论，但我们会继续逻辑，因为循环很小。 
6. 检查所有分割后，如果恰好存在一对有效的分割，则打印它。 否则，打印-1。 

### 为什么它有效

 每个有效的解决方案都对应于字符串中唯一的剪切位置，因为一旦分割固定，串联就是确定性的。 由于所有约束仅取决于两个零件的数值而不取决于任何全局结构，因此检查所有切割位置会耗尽完整的解决方案空间。 正确性源自以下事实：A 和 B 之间不存在超出其各自有效范围的隐藏转换或依赖关系。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

s = input().strip()

valid = []
n = len(s)

for i in range(1, n):
    a = s[:i]
    b = s[i:]

    # no leading zeros allowed
    if (a != "0" and a[0] == "0") or (b != "0" and b[0] == "0"):
        continue

    A = int(a)
    B = int(b)

    if 1 <= A <= 99 and 1 <= B <= 99:
        valid.append((A, B))

if len(valid) == 1:
    print(valid[0][0], valid[0][1])
else:
    print(-1)
```代码直接反映了算法。 我们迭代所有分割位置并将子字符串解释为整数。 防御性地包含前导零检查，尽管有效输入通常会避免这种情况。 最终决定完全基于有效分割的数量。 

一个微妙的细节是，当我们找到第一个有效的分割时，我们不会提前返回。 虽然我们可以，但显式地计算所有有效的分割可以使歧义条件更清晰，并避免丢失多个有效的解释。

 ## 边缘情况

 一种边缘情况是当字符串具有重复的数字时，例如`"111"`存在多个有效分割的地方。 该算法评估两个分割位置并正确识别两个有效解释，导致输出 -1。 

另一种边缘情况是当分割产生超出允许范围的数字时。 用于输入`"99100"`，分裂像`99 | 100`失败是因为 100 超过 99，并且所有其他拆分也失败或违反约束，没有留下有效的解释或确保取决于结构的唯一性。 

前导零是另一个关键情况。 为了`"1020"`, 分裂`1 | 020`即使在数字上看起来像 1 和 20，也会被拒绝。检查确保保留子字符串结构，而不仅仅是数字相等。 

通过评估每个分割并应用相同的有效性规则来统一处理每个案例，因此在最终实现中不需要特殊的大小写。
