 ## 解决方案
 ## 问题理解

 我们正在维护不断增长的堆栈集合，其中堆栈的数量及其内容随着时间的推移而变化。 最初没有堆栈。 系统按顺序处理一系列操作。 

一个操作创建一个新的空堆栈并将其附加到末尾。 另一个操作将一个值推送到选定的堆栈上。 第三个操作请求曾被推入特定堆栈的第 j 个值，从该堆栈的底部按插入顺序计数。 

因此，每个堆栈的行为类似于插入顺序的标准堆栈，但查询并不要求顶部元素。 相反，他们要求通过插入索引进行历史访问。 

限制非常小，总共最多 100 个操作。 这立即消除了对简单线性处理之外的性能的任何担忧。 任何每次操作执行恒定或线性工作的方法都会顺利通过。 

主要的微妙之处不是性能而是索引的正确性。 每个堆栈独立增长，操作通过基于 1 的标识符来引用堆栈，这些标识符取决于创建顺序。 简单的实现很容易出错，因为混合了堆栈索引或误解第 j 个元素指的是插入顺序而不是反向操作中从顶部开始的深度。 

当人们错误地将堆栈视为需要删除或动态重新索引时，就会发生一种典型的失败情况。 例如，如果我们将第 j 个元素解释为“从顶部开始的第 j 个”，我们可能会错误地反向索引。 实际上，它只是第 j 个推送的元素。 

## 方法

 直接模拟与问题结构完全匹配。 我们维护一个堆栈列表。 每个堆栈都是一个按插入顺序存储值的列表。 当出现类型 1 操作时，我们追加一个新的空列表。 当出现类型2操作时，我们将值追加到相应的堆栈中。 当出现类型3操作时，我们直接索引入栈。 

这是可行的，因为所有操作都是本地的，不需要任何全局结构。 不需要平衡、寻找、坚持。 在 O(1) 更新后，每个查询都会在 O(1) 时间内得到答复。 

蛮力变体在结构上是相同的，因为这里没有有意义的优化问题。 任何使用更复杂的数据结构的尝试只会增加开销而没有好处。 

关键的观察是堆栈是独立的，并且它们的增长历史正是附加值的序列。 因此，第 j 个元素实际上位于基础数组中的索引 j-1 处。

 ## 算法演练

 1. 从一个空列表开始`stacks`。 这表示按创建顺序排列的当前堆栈集合。 每个堆栈本身就是一个按到达顺序存储推送值的列表。 
2. 当类型1操作发生时，追加一个新的空列表到`stacks`。 这个新列表代表一个没有元素的新堆栈。 
3. 当类型2操作时`(i, x)`发生，追加`x`到`stacks[i-1]`。 这模拟推入堆栈 i，保留插入顺序。 
4. 当类型3操作时`(i, j)`发生，输出`stacks[i-1][j-1]`。 这会直接检索该堆栈中第 j 个插入的元素。 

正确性来自于将每个堆栈维护为纯粹的仅附加序列。 由于没有发生删除或修改，插入顺序相当于稳定的位置索引。 

### 为什么它有效

 每个堆栈都是独立的，并且只接收追加操作。 这意味着堆栈内的值顺序正是插入的时间顺序。 第 j 个插入的元素永远不会改变位置，因为没有操作删除或重新排序元素。 因此，按位置索引相当于跟踪插入历史。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def main():
    q = int(input().strip())
    stacks = []

    for _ in range(q):
        parts = input().split()
        t = int(parts[0])

        if t == 1:
            stacks.append([])

        elif t == 2:
            i = int(parts[1]) - 1
            x = int(parts[2])
            stacks[i].append(x)

        else:
            i = int(parts[1]) - 1
            j = int(parts[2]) - 1
            print(stacks[i][j])

if __name__ == "__main__":
    main()
```该实现直接反映了算法。 每个堆栈都存储为 Python 列表，因此追加的时间复杂度为 O(1) 摊销。 堆栈和元素的索引都从从 1 调整为从 0。 

唯一常见的错误是忘记输入中的堆栈索引和元素位置都是从 1 开始的。 每次访问时必须一致地应用转换。

 ## 边缘情况

 一种边缘情况是在堆栈创建后立即查询而不进行推送。 由于问题保证了有效的查询，这种情况在输入中不会发生，但逻辑上访问空堆栈是无效的。 实现依赖于保证，不添加额外的检查。 

另一个微妙的情况是确保基于 1 的索引一致地转换。 例如，堆栈 1 存储在索引 0 处。如果在推送或查询中忘记了此移位，则程序会默默地返回不正确的结果，同时仍保持在范围内。 

最后一种情况是在发生任何推送之前创建多个堆栈。 该结构仍应正确分配空列表而不会出现错误，确保稍后按标识符顺序将土地推入正确的堆栈中。
