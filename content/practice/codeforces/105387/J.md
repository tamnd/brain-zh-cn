 ## 解决方案
 ## 问题理解

 我们正在模拟在代表商店的网格上的受限行走。 网格有$n$行和$m$列，其中每个单元格要么是空的，要么是阻塞的。 一个人从网格的左下角开始，然后遵循一长串的移动命令。 每个命令都会尝试在四个基本方向之一上移动一步，但移动不是自由的：如果该方向的下一个单元格被阻挡或位于网格之外，则该人会立即停止并停留在当前位置。 

任务是在处理很长的此类命令序列后计算最终位置。 

关键的解释是每个命令都尝试精确移动一步，而不是连续滑动。 关于在障碍物处停止的措辞实际上与直接相邻单元无关，因为每个命令的移动都是离散的。 

这些限制以非常直接的方式产生影响。 网格最多可达$1000 \times 1000$，它足够小，可以直接存储并在恒定时间内建立索引。 命令字符串最多可达$2 \cdot 10^6$，这迫使解决方案在恒定时间内处理每个字符。 任何重新计算路径、扫描线或模拟定向扫描可见性的方法都会变得太慢，因为这会使网格大小乘以命令数量。 

一个天真的但常见的错误是将每个命令解释为一直持续到遇到障碍物的“射线”。 例如，如果我们处于某个位置并收到`R`，人们可能会错误地向右扫描，直到撞到墙上。 这将使每个命令$O(m)$在最坏的情况下，导致$O(nm|s|)$的行为，这是完全不可行的。 

另一个微妙的问题是坐标方向。 起始位置是左下角的单元格，移动方向以非标准方式增加或减少行索引和列索引。 混淆第一行是顶部还是底部会导致错误的运动解释。 

最后，边界处理至关重要。 试图离开网格的移动必须被完全忽略，而不是部分应用。 

## 方法

 强力解释将每个命令视为可能需要沿该方向扫描网格，直到遇到障碍物或边界。 对于每条指令，我们都会重复地逐步前进，检查每个中间位置的网格单元。 在最坏的情况下，单次移动可能会遍历整行或整列，并且命令字符串中的每个字符都可能发生这种情况。 和$2 \cdot 10^6$命令和最多$10^6$网格单元，这会退化为大量单元检查，远远超出允许的限制。 

关键的观察结果是运动不是连续的。 每个命令仅尝试移动一个单元格。 一旦我们意识到障碍物只在紧邻的单元格中起作用，整个模拟就变成了一个简单的状态更新：检查一个邻居，然后移动或停留。 

这将问题从路径模拟减少到直接网格步进。 我们只需要每条指令进行恒定时间的数组查找。

 | 暴力破解（按命令扫描）| (O( | s | \cdot \max(n,m))) |
 | 最优（单步模拟）| (O( | s | )) |

 ## 算法演练

 我们将网格视为静态障碍物地图并保持单个当前位置。 

1. 读取网格并将其存储在二维数组中，以便我们可以在恒定时间内测试某个单元格是否被阻塞。 这是必要的，因为每个移动都依赖于立即的邻接检查。 
2.初始化左下单元格的起始位置，对应行$n$， 柱子$1$如果我们使用标准的自上而下索引，或者等效的行$0$， 柱子$0$在从零开始的坐标中。 
3. 对于字符串中的每个命令字符，将预期方向计算为行/列增量。 这将问题转化为重复的状态转换。 
4. 在应用移动之前，计算候选的下一个位置。 这种分离很重要，因为我们必须在提交之前验证该移动。 
5. 检查候选位置是否在网格范围内。 如果在外面，则完全忽略该命令并保持当前位置。 
6. 如果位置在边界内，请检查单元格是否被遮挡。 如果被阻挡，也忽略该移动。 
7. 否则，将当前位置更新为候选单元格。 
8. 处理完所有命令后，以从 1 开始的索引输出最终坐标。 

核心思想是每个命令要么产生有效的相邻转换，要么导致无操作，并且没有命令会影响多个单元格。 

### 为什么它有效

 该算法保持了处理第一个之后的不变量$i$命令，当前位置正是顺序应用这些命令的结果，并且在每一步都强制执行立即阻塞和边界约束。 每个命令都是独立的，因为它仅取决于当前位置，而不取决于该位置之外的任何早期路径历史记录。 由于我们在提交移动之前评估合法性，因此我们永远不会输入无效的单元格或离开网格，并且由于每个命令都按顺序处理一次，因此不会跳过任何有效的转换。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def main():
    n, m = map(int, input().split())
    grid = [input().strip() for _ in range(n)]
    s = input().strip()

    # convert to 0-based indexing, start bottom-left
    x, y = n - 1, 0

    for c in s:
        nx, ny = x, y

        if c == 'U':
            nx -= 1
        elif c == 'D':
            nx += 1
        elif c == 'L':
            ny -= 1
        else:  # 'R'
            ny += 1

        if 0 <= nx < n and 0 <= ny < m and grid[nx][ny] == '.':
            x, y = nx, ny

    print(x + 1, y + 1)

if __name__ == "__main__":
    main()
```该解决方案将网格读入内存，因此每次障碍物检查都是直接数组访问。 该位置存储在从零开始的坐标中，原点实际上位于数组的左上角。 由于问题定义了左下角的开始，因此我们初始化$x = n - 1, y = 0$。 

每个命令都会转换为增量更新。 我们总是首先计算候选位置，然后在提交之前根据边界和障碍物约束对其进行验证。 这可以避免意外的部分更新或不一致的状态更改。 

最终输出根据需要转换回从 1 开始的索引。

 ## 边缘情况

 一种极端情况是每一步都试图离开网格。 例如，从左下角开始，重复发出`D`或者`L`命令。 该算法独立评估每个移动，检测候选位置是否出界，并保持当前状态不变。 No accumulation of invalid movement occurs.

 另一种情况是起始位置周围完全被封锁。 即使命令建议移动，每个相邻单元检查也会失败，因为`#`，所以位置永远不会改变。 该算法仍然在每个命令的恒定时间内执行所有检查。 

第三种情况是非常长的命令字符串。 由于每次迭代仅执行算术和单个数组查找，因此运行时间线性扩展并且不依赖于网格尺寸。
