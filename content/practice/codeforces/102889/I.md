 ## 解决方案
 ## 问题理解

 我们得到了一系列整数，表示随时间推移收集的评分。 这些评级已按收到的顺序排序。 任务是将这个序列精确地划分为 k 个连续组，并保持顺序，其中每个组对应于一天，并且每个元素必须恰好属于一天。 

对于每个组，我们从子数组中计算一个值：对组中的所有数字进行按位与，并对同一组中的所有数字进行按位或。 然后我们对这两个结果进行异或。 分区的总分是所有 k 组的该值的总和。 我们想要选择使总分最大化的分区。 

因此，问题的主要结构是数组上的分区 DP，但段的成本函数并不平凡，并且取决于整个段之间的按位交互。 

约束 n ≤ 2000 和 k ≤ n 表明需要 O(n²k) 或更好的解决方案。 n 中具有重常数的任何三次方仍可能通过，但 O(n³k) 或显式枚举分区太慢。 

当 k = 1 或 k = n 时，会出现微妙的边缘情况。 如果 k = 1，我们必须将整个数组作为一个段，并在整个范围内计算 AND/OR 表达式。 如果 k = n，则每个段的长度为 1，并且由于单个数字的 AND 和 OR 相等，因此每个段的贡献为零，因此答案始终为 0。未显式考虑单元素段的朴素分区 DP 可能会错误地假设到处都是非零贡献。 

另一个重要的边缘情况是值相同时。 如果所有 a[i] 都相等，则对于任何段 AND 等于 OR 等于相同的值，因此 XOR 始终为零。 无论分区如何，这都会迫使答案为零，并且任何优化都不能错误地尝试从拆分中“获得”价值。 

## 方法

 暴力解决方案会尝试一切可能的方法将数组分成 k 段。 路数是组合的，大致为 C(n-1, k-1)，它是 n 的指数。 对于每个分区，我们将通过扫描每个段并计算 AND 和 OR 来计算段值，每个段的成本为 O(n)，导致类似 O(n * number_of_partitions) 的结果，即使对于中等 n 的情况也是完全不可行的。 

更结构化的暴力方法使用动态编程。 让 dp[i][j] 表示使用分成 j 段的前 i 个元素的最佳值。 对于每个 dp[i][j]，我们尝试所有先前的切割位置 p < i 并计算段 [p, i]。 这给出了转换 dp[i][j] = max(dp[p][j-1] + cost(p+1, i))。 计算 cost(p+1, i) 需要 O(n)，因此总复杂度变为 O(n³k)，这对于 n 高达 2000 来说太大了。 

关键的见解是利用细分成本的结构。 对于任何段，该值是（段的AND）XOR（段的OR）。 AND 和 OR 在段的扩展上都是单调的：AND 仅在扩展时减少或保持不变，OR 仅增加或保持相同。 这表明当我们扩展一个段时，按位结构会以受控的方式发生变化。 

我们不是从头开始为每个段重新计算 AND 和 OR，而是增量地维护它们。 对于固定的 i，我们可以向后扩展以 i 结尾的段，每一步以 O(1) 的时间更新 AND 和 OR。 这将每次查询的成本计算从 O(n) 减少到 O(1)，将 DP 转换变成 O(n²k)。 

我们进一步观察到 k ≤ n 允许段上的标准 DP，并且可以通过扫描先前位置同时保持运行 AND/OR 值来优化转换。 

| 方法| 时间复杂度| 空间复杂度| 判决|
 |---|---|---|---|
 | 蛮力 | 指数| O(n) | 太慢了 |
 | 带重新计算的 DP | O(n3k) | O(n3k) | O(nk) | O(nk) | 太慢了 |
 | 通过增量 AND/OR 优化 DP | O(n²k) | O(n²k) | O(nk) | O(nk) | 已接受 |

 ## 算法演练

我们将 dp[j][i] 定义为使用分成 j 段的前 i 个元素可以获得的最大分数。 答案是dp[k][n]。 

1. 为从 1 到 n 的所有 i 初始化 dp[1][i]。 对于单个段，我们通过维护运行 AND 和 OR 值来计算前缀 [1..i] 的成本。 这给出了构建更大分区的基值。 

2. 对于从 2 到 k 的每个段数 j，我们计算从 j 到 n 的每个 i 的 dp[j][i]。 我们无法计算比段更少的元素，因此 i 从 j 开始。 

3. 为了计算 dp[j][i]，我们考虑每个可能的最后切割位置 p，其中 j-1 ≤ p < i。 最后一段是 [p+1..i]，因此 dp[j][i] = dp[j-1][p] 的 p 上的最大值加上段 [p+1..i] 的成本。 

4. 我们不是从头开始重新计算每个段的 AND 和 OR，而是修复 i 并将 p 从 i-1 向后移动到 j-1，维护两个运行变量：cur_and 和 cur_or。 当我们通过包含 a[p] 向左扩展段时，我们更新 cur_and &= a[p] 和 cur_or |= a[p]。 这使得每个段的成本计算为 O(1)。 

5. 对于每个候选 p，我们计算 cur_and XOR cur_or 并相应地更新 dp[j][i]。 

关键思想是，对于固定的右端点 i，向后扫描 p 可以为我们提供增量更新的所有可能的段结尾，因此每个 dp[j][i] 的计算时间为 O(n) 。 

### 为什么它有效

 DP 确保每个分区都由其最后切割位置精确表示一次。 AND 和 OR 的增量维护是正确的，因为这两种操作都是关联的，并且不依赖于段内的排序，只依赖于元素的成员资格。 由于每个可能的段通过向后扫描在每个 dp 状态下被枚举一次，因此不会遗漏任何有效分区，并且不会重复计算无效分区。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, k = map(int, input().split())
    a = list(map(int, input().split()))

    INF = -10**18
    dp = [[INF] * (n + 1) for _ in range(k + 1)]

    # base: 1 segment
    for i in range(1, n + 1):
        cur_and = a[0]
        cur_or = a[0]
        for j in range(1, i):
            cur_and &= a[j]
            cur_or |= a[j]
        dp[1][i] = cur_and ^ cur_or

    for seg in range(2, k + 1):
        for i in range(seg, n + 1):
            cur_and = a[i - 1]
            cur_or = a[i - 1]
            best = INF

            for p in range(i - 1, seg - 2, -1):
                cur_and &= a[p]
                cur_or |= a[p]
                val = cur_and ^ cur_or
                best = max(best, dp[seg - 1][p] + val)

            dp[seg][i] = best

    print(dp[k][n])

if __name__ == "__main__":
    solve()
```DP表是按段数逐行构造的。 第一行被显式地预先计算为完整前缀段成本，这避免了引入额外的虚拟转换。 对于后面的每一行，我们固定端点 i 并向后扫描以有效地计算所有可能的最后一段。 

内部循环是优化发生的地方：我们在向左扩展段时重用之前计算的值，而不是为每个候选段重新计算 AND 和 OR。 

必须小心 cur_and 和 cur_or 的初始化。 必须为每个 i 重置它们，否则值会在不同端点之间泄漏。 此外，后向循环边界确保每个分区都有足够的元素来满足段计数约束。

 ## 边缘情况

 当 k 等于 n 时，每个段只有一个元素。 每个这样的段的 AND 等于 OR，因此 XOR 为零，并且 DP 正确地将所有单元素贡献初始化为零。 

当所有元素相同时，每个可能的段都具有相同的 AND 和 OR，产生零贡献。 DP 永远不会从分裂或合并中找到正增益，因此最佳答案仍然为零。 

当 k 等于 1 时，算法简化为计算单个前缀段。 dp[1][n] 的初始化直接计算整个数组成本，确保正确性而无需转换。 

在大值快速将 AND 减为零的情况下，向后扫描仍然可以正常工作，因为 cur_and 稳定在零，而 OR 继续累加，确保段值仍然可以准确计算而无需重新计算。
