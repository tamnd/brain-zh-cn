 ## 解决方案
 ## 问题理解

 我们得到一个单行字符串，用于表示两个表达式之间的算术相等。 每个表达式可以包含十进制数字和符号`+`和`-`。 减号既可以充当二元减法运算符，也可以充当翻转后面数字的符号的一元符号，并且多个一元符号可以连续出现。 

任务是将输入字符串分为三类。 首先，字符串可能格式错误，这意味着它违反了语言本身的语法规则，例如包含非法字符、缺少结构或不完全包含一个`=`。 其次，一旦求值，它在语法上可能是有效的，但在算术上是错误的。 第三，它在语法上可能是有效的，而且在数字上也是正确的。 

关键的困难在于有效性取决于解析非常宽松的语法：重复的一元`+`和`-`允许使用符号，数字可以有前导零，并且表达式不受空格或格式超出允许字符的限制。 与此同时，格式错误的输入必须尽早被拒绝，即使它们可以用数字来解释。 

该限制是单行最多 300 万个字符。 这立即排除了任何重复切片字符串、构建大型中间子字符串或使用带有回溯的递归解析的方法。 输入大小的任何二次方都会失败，甚至线性解决方案也必须仔细实现，以避免 Python 开销，例如重复串联或正则表达式回溯。 

一些边缘情况很微妙。 

第一个重要案例是围绕运算符的畸形结构。 例如，`2+2=4+`格式错误，因为运算符出现在末尾，后面没有数字。 天真的求值者可能会忽略尾随运算符并错误地将其视为有效语法。 

第二种情况是无效字符。 输入类似`2*2=4`必须立即归类为畸形。 仅关注数字而忽略未知字符的解析器会错误地处理`*`作为噪音并继续。 

第三种情况是附近不明确的一元链`=`。 例如，`-+10=10`语法上有效但错误。 错误地合并一元运算符或忽略符号交替规则的解析器可能会错误地评估左侧表达式。 

最后，空洞的表情`=`均无效。 字符串像`=1+2`或者`1+2=`格式错误，因为每一侧都必须包含至少一个有效的数字标记。 

因此，核心挑战是将严格的语法验证与高效的单遍评估结合起来。 

## 方法

 暴力解释首先将每一侧完全解析为标记，构建显式表达式树或转换为后缀表示法，然后评估两侧。 这需要对字符串进行多次传递，以及与输入大小成比例的令牌列表或堆栈的中间分配。 

在最坏的情况下，每个字符都会贡献多个结构：令牌、AST 节点或堆栈帧。 如果不小心实现字符串切片或重复串联，这会导致严重的常数因子和潜在的二次行为。 

关键的观察是我们实际上不需要构建任何结构。 语法非常简单，我们可以通过维护运行总和和当前符号来评估单遍中的每一侧，同时在读取每个字符时验证语法。 

然后，相等检查简化为动态计算两个整数并进行比较，同时使用小型有限状态逻辑检测格式错误的结构：当前是否在数字内部，是否需要运算符以及表达式是否已正确启动。 

这将解析转换为具有恒定内存的流处理。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力解析 + AST | O(n) 到 O(n²) | O(n) | 太慢了 |
 | 带验证的单次评估 | O(n) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们在一次扫描中处理字符串并分别评估两侧。 

1. 首先，验证是否有一个`=`存在于字符串中。 如果不是，则输入会立即出现格式错误，因为该结构不是等式。 
2. 将处理逻辑上分成两半`=`。 我们不会物理地分割字符串； 相反，当我们遇到时，我们会扫描并切换上下文`=`。 
3. 对于每一边，维护三个变量：运行总计、当前符号（+1 或 -1）以及指示我们当前是否期望数字的标志。 该标志对于检测格式错误的语法至关重要。 
4.当我们遇到`+`或者`-`，我们更新当前的符号。 一个`+`保持不变，a`-`翻转它。 如果我们连续看到多个符号，它们会自然地组合成数字前面的最后一个符号。 
5. 当我们遇到一个数字时，我们解析完整的连续数字。 我们使用当前符号将其乘以总和，然后将符号重置为正值，并标记我们现在等待下一个运算符。 
6.如果遇到数字以外的字符，`+`,`-`， 或者`=`，输入立即出现格式错误。 
7. 如果运算符出现在需要数字的位置（例如在另一个运算符之后或在边的开头），则仅当表达式为一元时，该表达式才仍然有效。 这意味着我们允许`+`或者`-`当需要一个数字时，但不允许将二元运算符放置在无效位置，例如`2+`结束时或`=+`。 
8. 双方处理完毕后，比较评估的总数。 如果匹配，则输出`YES`。 如果语法有效但值不同，则输出`NO`。 

### 为什么它有效

 不变的是，在扫描中的任何点，运行总计表示具有正确累积符号效果的所有完全消耗的数字的值，并且当前符号表示自最后一个数字以来看到的所有一元运算符的组合效果。 

因为每个数字都被消耗一次，并且每个符号只影响下一个数字，所以不存在延迟依赖性。 这保证了流计算等同于完整表达式解析。 格式错误检测是完整的，因为每个非法结构都对应于一种状态，其中要么需要一个数字但未找到，要么出现一个禁止字符，这两种情况都会在遍历过程中立即捕获。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def evaluate(expr):
    n = len(expr)
    i = 0
    total = 0
    sign = 1
    expect_number = True

    if n == 0:
        return None, False

    while i < n:
        c = expr[i]

        if c in '+-':
            if expect_number:
                if c == '+':
                    pass
                else:
                    sign *= -1
                i += 1
            else:
                return None, False

        elif c.isdigit():
            j = i
            while j < n and expr[j].isdigit():
                j += 1
            num = int(expr[i:j])
            total += sign * num
            sign = 1
            expect_number = False
            i = j

        else:
            return None, False

    if expect_number:
        return None, False

    return total, True

def solve():
    s = input().rstrip('\n')

    if s.count('=') != 1:
        print("ERROR")
        return

    left, right = s.split('=')

    lv, ok1 = evaluate(left)
    rv, ok2 = evaluate(right)

    if not ok1 or not ok2:
        print("ERROR")
    elif lv == rv:
        print("YES")
    else:
        print("NO")

if __name__ == "__main__":
    solve()
```该解决方案将解析分离到一个可重用的求值器中，该求值器在线性时间内处理一个表达式。 它保留一个运行总和和一个符号累加器，立即应用每个解析的数字。 这`expect_number`标志强制结构正确性，确保运算符不会出现在无效位置。 

分裂于`=`是安全的，因为该问题保证了格式正确的情况下只有一个相等符号，而其他任何内容都会立即被归类为格式错误。 

主要的微妙之处在于一元运算符的处理。 当一个`+`或者`-`出现在需要数字的位置，它会修改符号状态而不是被视为二元运算符。 当看到一个数字时，会应用并重置累积的符号，确保像这样的链`-+-+-5`都得到正确处理。

 ## 边缘情况

 一个重要的边缘情况是以多个一元符号开头的表达式，例如`-+-+-5=5`。 在解析期间，每个符号在消耗数字之前翻转或保留当前符号状态。 评估器以期望数字的状态开始，因此它会重复更新`sign`如果没有错误，则在读取数字序列后应用它。 结果正确地将链简化为单个有效符号。 

另一个边缘情况是尾随运算符，例如`2+`。 解析时`2`，状态转变为“期待操作员”。 相遇`+`有效，因为它是二元运算符放置，但在`+`字符串结尾之前没有数字。 最后检查`if expect_number`捕捉到这一点并将表达式标记为格式错误。 

第三个边缘情况是等号周围缺少边。 用于输入`=1+2`，左边是空的。 评估器立即返回格式错误，因为它以`expect_number`在没有消耗任何数字的情况下仍然如此。 同样的逻辑对称地适用于`1+2=`。
