 ## 解决方案
 ## 问题理解

 我们在游戏中被赋予了固定的阶段顺序，其中阶段$i$对应于解锁或获得$i$-th 结局。 离开舞台$i$上台$i+1$需要已知的时间$d_{i+1}$，因此除非我们选择暂停，否则进展时间是累积的并且严格增加。 

每个阶段$i$，还有一个事件：在绝对时间开始的流$a_i$。 如果我们恰好在那个时间（或者还没有通过），我们可以选择停止进展，如果需要的话等待，然后观看。 观看时间为零，但看完后我们立即继续下一阶段。 如果我们跳过它，它就会永远丢失，并且以后我们永远无法返回它。 此外，一旦我们越过了阶段$i$，我们不被允许回去观看之前的直播。 

因此，该过程是一个向前传递各个阶段的过程，在每个阶段我们可以：

 1. 立即继续，丢失流，或者
 2. 确保我们准时$a_i$，观看直播，然后继续。 

我们的目标是最大化我们能够观看的流媒体数量。 

阶段结束后的时间$i$完全取决于我们之前选择参加哪些直播，因为暂停只能延迟我们前进，而绝不会后退。 

约束条件达到$n \le 2 \cdot 10^5$，因此任何解必须接近线性或$n \log n$。 决策的二次模拟太慢了，因为每个状态都取决于之前的所有选择。 

一个微妙的问题是，只有当我们足够早时，等待才有用。 如果在阶段$i$我们已经过去了$a_i$，那么该流永远不可能。 如果可能的话，当贪婪地总是获取流时，就会出现另一种失败情况：这样做可能会延迟我们太多，以至于我们会丢失许多实际上更有价值的后来的流。 

例如，假设观看较早的直播迫使我们来得太晚，无法观看后面的两个直播，否则我们本来可以观看这两个直播。 总是获取可用流的天真的贪婪策略在这里失败了，因为它没有考虑未来时间的影响。 

## 方法

 关键的困难在于每个决定都会影响当前时间，因此影响未来的流。 天真的暴力会按时间顺序尝试所有流子集，每次模拟结果时间线。 从概念上讲，这是可行的，因为每个选择都唯一地决定时间演化，但它立即爆炸：每个步骤分为两个选择，导致$2^n$可能性，甚至模拟单个路径$O(n)$，所以总复杂度是指数级的。 

更结构化的蛮力会尝试在阶段和当前时间上进行动态规划，但时间是一个连续值，最多可达$10^9$，所以这个状态空间太大了。 

关键的观察是，在每个阶段，唯一有用的决定是我们是否有能力在时间上准确地调整自己$a_i$不损害未来的机会。 如果我们已经过去了$a_i$，流消失了。 否则，我们可以选择“等待”，但等待是有代价的：它将所有未来的到达时间提前。 

这表明我们不应该跟踪确切的时间，而应该跟踪我们愿意引入多少松弛时间。 一个更有用的观点是考虑我们是否可以按顺序将一组选定的流“适合”到时间轴中。 每次我们获取流时，我们都会强制当前时间至少变为$a_i$，然后继续前进。 

这成为一种经典的贪婪可行性结构：我们尝试以递增的索引顺序选择流，但仅在不破坏相对于时间进展的可行性的情况下才接受流。 连续阶段之间的转换成本是固定的，因此我们可以维持当前的完成时间并在本地测试每个流。 

不明显的简化是，最优策略对应于按顺序做出局部最优接受/拒绝决策，因为采用流引入的任何延迟都是不可逆的，并且只影响未来的可达性，而不会影响过去的选择。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O(2^n \cdot n)$|$O(n)$| 太慢了|
 | 最优贪心模拟 |$O(n)$|$O(1)$| 已接受 |

 ## 算法演练

 我们维持当前时间$t$，代表我们在所选时间表中完成当前阶段的时刻，以及我们成功观看的流媒体数量的计数器。 

我们还利用了这个阶段的事实$1$开始于时间$0$，并离开舞台$i$到$i+1$总是添加$d_{i+1}$。 

### 步骤

 1.初始化当前时间$t = 0$并回答$ans = 0$。 

这代表我们在任何游戏开始之前就开始并且没有观看任何内容。 
2. 阶段迭代$i = 1$到$n$。 

在每个阶段，我们在概念上到达时间$t$完成前一阶段后。 
3.如果$t > a_i$，完全跳过此流。 

这是被迫的：我们已经错过了它的开始时刻，再拖延只会增加时间。 
4. 否则，选择观看直播$i$， 放$t = a_i$，并增加$ans$。 

我们同步到流时间，这可能涉及等待，并且我们承诺这个选择。 
5. 如果$i < n$，将时间提前$d_{i+1}$。 

该模型在完成当前阶段后立即完成下一阶段。 

### 为什么它有效

 关键的不变量是处理阶段之后$i$，变量$t$是我们到达阶段的最早时间$i+1$考虑到迄今为止做出的决定，同时最大化与这些决定一致的所有时间表中观看的流的数量。 

每一步，如果我们能参加直播$i$，这样做绝不会减少未来可行决策的数量，而只会增加增加的影响$t$到$a_i$。 任何跳过流的替代时间表$i$但稍后仍能参加是不可能的，因为直播是固定时间的。 因此，唯一有意义的决定是我们是否还能达到，如果是，我们就接受。 

这确保了每个流要么在可行时被采用，要么永久无法到达，并且在不违反时间限制的情况下，任何选择的重新排列都无法提高计数。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = list(map(int, input().split()))
    d = list(map(int, input().split()))

    t = 0
    ans = 0

    for i in range(n):
        if t <= a[i]:
            t = a[i]
            ans += 1

        if i < n - 1:
            t += d[i]

    print(ans)

if __name__ == "__main__":
    solve()
```该解决方案维护单个时间线变量`t`表示完成最后考虑的阶段后的当前完成时间。 比较`t <= a[i]`编码流是否仍然可达。 如果是，我们“等待”到那个时间，这是通过设置建模的`t = a[i]`，我们将其视为已观看。 

添加的`d[i]`每个阶段之后都是至关重要的，因为它确保时间始终按照移动到下一个结局的固定成本进行。 一个常见的错误是添加`d[i+1]`而不是`d[i]`，这会导致索引不对齐，因为`d`从过渡到第 2 阶段开始。

 ## 边缘情况

 一个关键的边缘情况是当前时间恰好等于$a_i$。 在这种情况下，条件必须允许获取流。 支票`t <= a[i]`确保平等被接受。 如果写成`t < a[i]`，恰好在当前时间发生的流将被错误地跳过。 

另一个边缘情况是等待流将时间提前太多，以致所有剩余的流都变得无法访问。 该算法自然会处理这个问题，因为每个后续比较都会失败一次`t`超越未来`a[i]`值，强制跳过并保持正确性。 

最终的边缘情况非常小或非常大`d[i]`。 由于该算法仅将这些值累积到`t`，溢出在 Python 中不是问题，但在固定宽度整数语言中，这将需要 64 位存储。
