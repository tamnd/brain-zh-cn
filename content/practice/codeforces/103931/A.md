 ## 解决方案
 ## 问题理解

 我们得到一个长度为 8 的固定方程格式，总是写成两个两位数相加并等于另一个两位数。 结构总是`??+??=??`，其中每个`?`是一个数字。 加号和等号固定在相同位置，所有其他位置都是数字。 允许使用前导零，因此每个两个字符块代表 00 到 99 之间的数字。 

一个猜测方程`E`已提交，系统返回反馈字符串`C`使用三种颜色的相同长度。 每个位置描述了相应的字符如何`E`匹配隐藏的正确方程。 绿色表示角色正确且位置正确。 紫色表示该角色存在于隐藏方程式中的某个位置，但不在该位置。 黑色意味着该字符没有出现在隐藏方程中，或者它出现的次数比猜测的少，因此额外出现的次数不会被计入。 

任务是重建所有有效的方程，与给定的猜测进行比较时，这些方程可以准确地产生该反馈`E`。 

关键是有效的候选答案不是任意的。 它必须满足两个独立的条件。 首先，它必须是一个语法上有效的方程，其形式为`??+??=??`。 其次，它必须在算术上正确，这意味着当解释为允许有前导零的整数时，左侧的总和必须等于右侧的总和。 

结构上的约束很小，但组合上的约束很大。 三个数字块的范围都是从00到99，因此在按算术正确性过滤之前最多有一百万个可能的方程。 这个值足够小，可以直接进行暴力破解，但前提是每个候选人的反馈检查有效。 

天真的误解通常来自于将反馈视为每个角色独立的反馈，而没有正确处理重复项。 例如，如果猜测包含重复的数字，则数字的紫色匹配数量取决于它在隐藏答案中出现的次数。 简单地检查成员资格的粗心实现会过度计数匹配并产生不正确的过滤。 

另一个微妙的情况是候选答案中出现重复数字。 考虑像这样的猜测`11+11=22`。 如果正确答案只有一个`1`，其中只有一个位置应该是绿色或紫色，其他位置必须是黑色。 这种多重性上限至关重要，并且必须在模拟中强制执行。 

## 方法

 直接的强力解决方案考虑每个可能的方程`a + b = c`，其中每个`a`,`b`， 和`c`范围从 00 到 99。对于每个三元组，我们将其格式化为字符串表示形式并验证算术正确性。 这最多产生 1000000 个候选者，并且只有一小部分满足要求`a + b == c`。 

对于每个有效的候选者，我们根据给定的猜测模拟 Nerdle 反馈`E`。 该模拟必须复制类似 Wordle 的计数匹配。 我们首先分配所有绿色匹配，其中字符的位置完全一致。 然后，我们通过计算候选字符中未使用字符的频率并将其与剩余的猜测字符进行匹配来处理剩余字符，根据可用性生成紫色或黑色。 

这种强力方法之所以有效，是因为搜索空间很小。 然而，主要成本是反馈模拟。 如果仔细实现，每次比较都是 O(8)，因此总复杂度约为 800 万次操作，这很容易足够快。 

关键的观察结果是不需要更智能的修剪。 该问题的设计使得所有算术有效方程的详尽枚举就足够了。

 ## 算法演练

 我们迭代三个数字块的所有可能值并按算术有效性进行过滤，然后检查与反馈字符串的一致性。 

1. 迭代`a`从0到99，`b`从 0 到 99，计算`c = a + b`。 如果`c > 99`，跳过它，因为它不能容纳两位数。 这确保我们只生成有效的方程结构。 
2. 转换`a`,`b`， 和`c`转换为带有前导零的两个字符的字符串。 构造完整的方程串`S = format(a) + "+" + format(b) + "=" + format(c)`。 
3.比较`S`针对给定的猜测`E`来计算反馈。 我们首先标记所有位置`S[i] == E[i]`为绿色并从进一步考虑中删除这些字符。 此步骤是必要的，因为精确匹配必须始终优先。 
4. 对于剩余的不匹配位置，我们计算隐藏候选的字符频率`S`。 然后我们扫描不匹配的位置`E`。 如果剩余频率池中存在字符，我们分配紫色并减少计数。 否则我们分配黑色。 
5. 如果构造的反馈与给定的字符串完全匹配`C`，我们将方程存储为有效。 
6. 处理完所有候选后，输出计数和所有有效方程。 

核心设计选择是我们从不尝试象征性地“解决”约束。 相反，我们直接测试每个可行的方程并依赖于较小的域大小。 

### 为什么它有效

 该算法是正确的，因为每个有效的隐藏答案都必须是枚举算术解决方案之一。 给定猜测和隐藏答案，反馈函数是确定性的，并且我们的模拟准确地再现了该函数，包括多重性约束。 因此，任何与反馈相匹配的候选者都与观察到的结果一致，并且在枚举过程中将发现任何一致的方程。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def feedback(guess, target):
    n = 8
    res = [""] * n
    used_guess = [False] * n
    used_target = [False] * n

    for i in range(n):
        if guess[i] == target[i]:
            res[i] = 'G'
            used_guess[i] = True
            used_target[i] = True

    freq = {}
    for i in range(n):
        if not used_target[i]:
            freq[target[i]] = freq.get(target[i], 0) + 1

    for i in range(n):
        if res[i]:
            continue
        if freq.get(guess[i], 0) > 0:
            res[i] = 'P'
            freq[guess[i]] -= 1
        else:
            res[i] = 'B'

    return "".join(res)

def solve():
    E = input().strip()
    C = input().strip()

    ans = []

    for a in range(100):
        for b in range(100):
            c = a + b
            if c >= 100:
                continue

            s = f"{a:02d}+{b:02d}={c:02d}"
            if feedback(E, s) == C:
                ans.append(s)

    print(len(ans))
    for x in ans:
        print(x)

if __name__ == "__main__":
    solve()
```该解决方案是围绕所有算术有效方程的直接枚举构建的。 这`feedback`函数实现了严格的两遍匹配策略。 第一次通过锁定绿色位置，确保在频率匹配开始之前不考虑精确匹配。 第二遍使用频率字典来强制执行问题所需的多重性约束。 

一个常见的陷阱是跳过第一遍并直接进行频率匹配，当重复的数字在特定位置对齐时，这会破坏正确性。 

## 工作示例

 考虑一个小的说明性案例，其中猜测是`E = 11+45=56`候选答案是`11+45=56`。 

| 步骤| 猜测与候选| 绿党 | 剩余频率 | 反馈 |
 | ---| ---| ---| ---| ---|
 | 初始| 11+45=56 vs 11+45=56 | 全部 | 无 | GGGGGGGG |

 这确认了完美匹配，产生了所有绿色瓷砖。 

现在考虑一个不匹配的候选者，例如`11+46=57`。 

| 步骤| 猜测与候选| 绿党 | 剩余频率 | 反馈 |
 | ---| ---| ---| ---| ---|
 | 初始| 11+45=56 vs 11+46=57 | 部分（前缀匹配）| 还剩多少位 | 混合 |

 在这里，位置匹配的数字变成绿色，而其他数字则通过频率进行评估。 这演示了位置正确性和多集正确性如何相互作用。 

轨迹显示反馈既不是纯粹的位置反馈，也不是纯粹的基于集合的反馈，而是两者的混合，这就是为什么需要进行两次仿真的原因。

 ## 边缘情况

 一种微妙的情况是，猜测包含重复的数字，但候选答案包含的次数较少。 例如，如果猜测是`11+11=22`候选人是`12+10=22`，只有其中之一`1`s 可以匹配。 基于频率的第二遍确保在分配一个匹配项后，剩余的匹配项将自动强制为黑色。 

另一个边缘情况是算术正确性过滤掉了大多数候选者。 例如，`99+99=198`无效，因为结果超过两位数，因此根本不会考虑。 这可以防止不正确的环绕行为。 

最后一种边缘情况是反馈仅包含黑色图块。 在这种情况下，任何与超过允许重数的猜测不共享字符的候选方程都将通过。 模拟仍然正确地处理了这个问题，因为所有绿色匹配都首先被删除，留下的频率计数自然可以防止过度计入共享数字。
