 ## 解决方案
 ## 问题理解

 我们正在使用一个无限网格，其中每个整数坐标单元都以相同的概率独立地着色为黑色或白色。 唯一允许的操作是查询单元格并接收其颜色。 每个测试用例的目标是输出一组精确的`n`不同的单元具有相同的颜色，并在 8 向邻接下形成单个连接的组件，这意味着在定义连接时也允许对角线移动。 

重要的困难不是建立连通性，而是仅使用无限随机场上的点查询来找到足够大的单色连通分量。 由于网格是无限的，因此没有有助于探索的边界，并且每个单元最初都是未知的。 

这些限制主要体现在查询预算方面。 所有测试用例最多允许 30000 个查询，最多 50 个测试用例`n`最多 300 个。这立即排除了任何试图全面探索大区域或系统地枚举密集区域中所有候选者的策略。 任何解决方案都必须延迟增长组件，并避免重新访问太多断开连接的区域。 

一个天真的但危险的想法是选择一个起始单元，然后在查询邻居的同时尝试扩展 BFS 区域。 问题是随机起始单元的连通分量可能很小，因此这种方法可以用少于`n`可到达的相同颜色的单元格。 另一种故障模式是从附近位置重复重新启动 BFS，这可能会浪费对重叠或已探索的小组件的查询。 

一个微妙的边缘情况是，早期探索发现一个非常小的单色簇几乎完全被相反的颜色包围。 例如，被白色单元包围的单个黑色单元将立即终止 BFS 扩展。 如果策略始终在本地重新启动，则它可能会反复遇到此类陷阱并耗尽查询预算，而不会达到大小`n`。 

## 方法

 暴力破解的想法很简单：选择一个起始坐标，显示它，然后执行 BFS 或 DFS，扩展到相同颜色的所有 8 个邻居，根据需要查询每个看不见的邻居。 这正确地构建了该颜色的最大连通分量。 问题是，在随机网格中，大多数组件都很小，并且包含随机单元的组件的预期大小不能保证达到 300。如果我们在组件太小时重新启动此过程，我们可能会反复陷入小的有限簇。 在确定性策略的最坏情况下，每次尝试在失败之前只能探索一个很小的区域，从而导致大量的查询浪费。 

关键的观察结果是，尽管单个组件可能很小，但无限网格包含许多不相交的区域，并且恒定比例的细胞属于足够大的组件，可以在消亡之前显着生长。 这提出了一种摊销策略：在远离先前探索的地方重复采样新的种子点，并完全扩展它们连接的单色分量。 由于组件是不相交的，并且探索是局部的，因此每个成功的大型组件都直接有助于达到规模`n`，而失败的尝试仅消耗与小组件的大小成比例的有限工作。 

关键的结构技巧是将网格视为黑盒随机图并重复“探测和洪水填充”，直到找到足够大的连接组件。 因为我们只需要尺寸的一个组成部分`n`，我们不关心全局结构，只关心我们遇到的第一个足够大的局部结构。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 随机种子的重复 BFS | O(查询 × 探索的单元格) | O（探索的细胞）| 已接受（摊销）|
 | 系统网格探索| O(R²) 查询区域 R | O(R²) | 太慢了 |

 ## 算法演练

 我们在所有尝试中维护一组访问过的单元格，并且反复尝试增长单色分量，直到收集到`n`细胞。 

1. 从以前未使用过的新种子坐标开始，例如通过迭代固定序列，例如`(i * 1000, i * 1000)`为了增加`i`。 我们这样做是为了避免重复攻击同一局部区域。 
2. 查询种子单元格。 这给了我们颜色`c`，它成为本次尝试的目标颜色。 
3. 使用种子单元初始化 BFS 队列，并将其标记为已访问。 
4.虽然队列不为空，但我们仍然需要超过`n`单元格，弹出一个单元格`(x, y)`并检查所有 8 个相邻坐标。 
5. 对于每个邻居，如果以前从未查询过，则对其进行查询。 如果它的颜色是`c`，将其添加到 BFS 队列并将其标记为当前组件的一部分。 如果它是不同的颜色，请将其标记为已阻止，并且永远不要重新访问它。 
6. 继续，直到任一组件达到尺寸`n`或者 BFS 完全耗尽所有可到达的相同颜色的单元格。 
7. 如果我们达到了尺寸`n`，立即输出那些`n`单元格并停止测试用例。 如果不是，则丢弃该组件并从新的种子坐标重新启动。 

重要的细节是查询是全局缓存的，因此每个单元格最多查询一次。 这确保了当不同的 BFS 尝试在空间中重叠时我们不会浪费查询。 

### 为什么它有效

 每个 BFS 只探索显示图中的一个 8 连接单色分量。 由于网格是无限的且随机着色，因此每个单元都恰好属于一个这样的组件，并且重复采样最终会落在足够大的组件上。 由于我们仅在收集时终止`n`单元由单个组件组成，连接性由结构保证。 该过程无法混合颜色，因为扩展仅沿着边缘到相同颜色的邻居。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

# interactive solution skeleton

sys.setrecursionlimit(10**7)

dx = [-1, -1, -1, 0, 0, 1, 1, 1]
dy = [-1, 0, 1, -1, 1, -1, 0, 1]

seen = {}  # (x,y) -> color

def query(x, y):
    if (x, y) in seen:
        return seen[(x, y)]
    print(f"? {x} {y}", flush=True)
    c = input().strip()
    seen[(x, y)] = c
    return c

def solve_case(n):
    used_seed = 0

    while True:
        sx = used_seed * 1007
        sy = used_seed * 1009
        used_seed += 1

        c = query(sx, sy)

        if c not in ('B', 'W'):
            continue

        q = [(sx, sy)]
        comp = [(sx, sy)]
        comp_set = {(sx, sy)}

        while q and len(comp) < n:
            x, y = q.pop()

            for k in range(8):
                nx, ny = x + dx[k], y + dy[k]
                if (nx, ny) in seen:
                    if seen[(nx, ny)] == c and (nx, ny) not in comp_set:
                        comp_set.add((nx, ny))
                        comp.append((nx, ny))
                        q.append((nx, ny))
                else:
                    col = query(nx, ny)
                    if col == c:
                        comp_set.add((nx, ny))
                        comp.append((nx, ny))
                        q.append((nx, ny))

                if len(comp) >= n:
                    break

        if len(comp) >= n:
            print("! " + c + " " + " ".join(f"{x} {y}" for x, y in comp[:n]), flush=True)
            return

def main():
    t, n = map(int, input().split())
    for _ in range(t):
        solve_case(n)

if __name__ == "__main__":
    main()
```该实现依赖于查询单元的记忆，这很重要，因为来自不同种子的 BFS 扩展经常重叠。 如果没有缓存，同一个单元格可能会被多次查询，很快就会耗尽限制。 

种子选择使用简单的确定性进程来确保我们穿过无限网格的不同区域，而不是重复探测密集的局部集群。 

一个微妙的点是 BFS 仅通过相同颜色的边缘执行，并且我们从不将相反颜色的单元入队。 这保证了每个`comp`列表始终是颜色导出子图中的有效连接组件`c`。 

## 工作示例

 由于网格是交互式的和随机的，我们在一个小的固定区域上模拟一个简化的场景。 

假设原点周围的网格为：

 | 细胞| 颜色 |
 | ---| ---|
 | (0,0) | (0,0) | 乙|
 | (1,0)| 乙|
 | (0,1)| 西 |
 | (1,1) | 乙|

 我们想要`n = 3`。 

### 追踪

 | 步骤| 行动| 队列 | 组件|
 | ---| ---| ---| ---|
 | 1 | 查询 (0,0)=B | [(0,0)] | [(0,0)] |
 | 2 | 展开 (0,0)，参见 (1,0) | [(1,0)] | [(0,0),(1,0)] |
 | 3 | 展开 (1,0)，忽略 (0,1)=W | []| [(0,0),(1,0)] |

 这里组件的大小为 2，因此该种子失败，我们在其他地方重新启动。 

现在假设下一个种子位于更密集的区域：

 | 步骤| 行动| 队列 | 组件|
 | ---| ---| ---| ---|
 | 1 | 查询 (10,10)=W | [(10,10)] | [(10,10)] |
 | 2 | 扩大邻居，找到另外 2 个 W 细胞 | 成长| 成长|
 | 3 | 达到尺寸 3 | 停止| 3 格 |

 这证明了关键行为：小簇会提前终止，但一旦找到有利区域，较大的簇就会迅速积累足够的细胞。 

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 | O(q) | 每个查询处理一次并缓存，BFS 扩展与发现的单元成正比 |
 | 空间| O(q) | 存储所有查询的坐标及其颜色|

 主导因素是查询数量。 由于每个单元最多查询一次，并且每次 BFS 扩展仅添加新的边界单元，因此总工作量保持在 30000 次查询的全局限制内。 网格的随机结构确保了足够的探索路径在实践中足够快地导致大型组件。

 ## 边缘情况

 一个关键的边缘情况是当起始种子位于一个微小的孤立组件中时，例如被相反颜色包围的单个单元。 BFS 立即终止，返回大小为 1 的分量。算法通过丢弃结果并选择远处的新种子来处理此问题。 由于种子确定性地在无限网格上移动，因此重复的失败不会将算法困在局部区域中。 

另一个边缘情况是 BFS 尝试之间重复重叠。 如果没有缓存，相同的边界单元将被重复查询，很快就会浪费查询预算。 这`seen`字典确保每个坐标在全局范围内查询一次，因此重叠只会提高效率而不是损害效率。 

最后的边缘情况是早期成功：BFS 可能已经达到大小`n`在充分探索其组成部分之前。 该算法正确地立即停止扩展并输出第一个`n`收集的节点，保证保持连接，因为它们都是通过 BFS 边缘从种子中发现的。
