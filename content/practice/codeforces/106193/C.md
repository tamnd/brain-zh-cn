 ## 解决方案
 ## 问题理解

 给定一个 32 位的非负整数，我们需要将其转换为可变长度字节编码。 我们并不总是使用四个字节，而是使用打包成字节的 7 位块来表示数字。 每个字节在其低位部分存储数字的 7 位，而该字节的最高位充当连续标志。 如果设置了该标志，则意味着后面有更多字节； 如果为零，则该字节是最后一个字节。 

字节首先从最高有效侧生成，因此第一个字节包含数字的最高位 7 位块，最后一个字节包含最低位块。 这本质上是一个以大端顺序编写的带有延续位的基于 128 的表示形式。 

一个关键的限制是我们不允许引入前导零块。 实际上，这意味着我们不能用低 7 位全为零的字节开始编码，除非数字本身为零。 

输入大小很小，因为数量最多为$2^{31} - 1$。 这意味着二进制表示最多有 31 位，因此分成 7 位块最多产生 5 个字节。 任何在每次测试中运行恒定时间的解决方案都是微不足道的，甚至重复的位操作也可以忽略不计。 

当数字为零时，出现主要的边缘情况。 简单的分块方法可能会产生空序列或单个零字节，并且很容易意外地违反连续位分配方式的“无前导零字节”规则。 

另一个微妙的问题是排序。 如果首先从最低有效侧提取 7 位块并立即打印，则结果将相反，除非明确更正。 例如，编码112025需要从最高有效位开始分组； 反转顺序会产生语法上有效的字节序列，但解码值不正确。 

## 方法

 思考这个问题的一个直接方法是模拟从数字中重复提取 7 位块。 在每一步中，我们可以取出最低 7 位，将数字右移 7，并将这些块存储在列表中。 这会自然分解为 128 位基数。 

这种蛮力想法在表示数字方面是正确的，但它以相反的顺序产生数字，从最不重要到最重要。 然后，我们需要在构造字节之前反转列表。 理论上，这种方法的成本是恒定时间的，因为我们最多执行五次迭代，但概念问题不是性能，而是排序和标志放置的正确性。 

关键的观察结果是编码完全是基于 128 的表示形式，但使用连续位写入。 计算完所有 128 基数后，我们只需按照从最高有效位到最低有效位的顺序输出它们即可。 除了最后一个打印字节之外的每个数字都必须设置其连续位。 

因此，任务简化为将整数转换为基数 128，存储数字，然后通过适当的位操作反向输出它们。 这种结构使问题具有确定性，并且消除了对复杂的按位流的任何需要。

 ## 算法演练

 我们通过重复提取 7 位块来处理该数字，直到该数字变为零。 

1. 使用掩码重复取出数字的最低 7 位，并将它们存储为一个块。 然后将数字右移 7 位。 这将从最低有效位到最高有效位以 128 为基数构建数字。 
2. 将所有块存储在列表中。 在我们设置连续标志之前，每个块代表一个字节的有效负载。 
3. 如果列表为空，则意味着数量为零，因此我们显式设置单个块值 0。 
4. 反转列表，以便我们现在首先处理最重要的块。 
5. 对于每个块，将其转换为一个字节。 对于除最后一个块之外的所有块，通过添加 128 将最高位设置为 1。对于最后一个块，保持不变。 

反转的原因是连续性取决于是否存在更重要的块。 这种依赖关系只有在收集所有块之后才知道，而不是在提取期间。 

### 为什么它有效

 该算法构造数字的以 128 为基数的表示形式。 每个提取的块精确对应于 128 基数中的一位数字。由于任何整数都有唯一的 128 基数表示，因此分解是唯一的。 连续位不会改变块的数值，它仅对结构进行编码。 因此，在完全分解后分配连续位可以保持正确性，并且反转可以确保首先发出最重要的块，从而匹配所需的大端编码。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

n = int(input().strip())

chunks = []

if n == 0:
    chunks = [0]
else:
    while n > 0:
        chunks.append(n & 127)
        n >>= 7

chunks.reverse()

res = []
for i in range(len(chunks)):
    if i != len(chunks) - 1:
        res.append(chunks[i] | 128)
    else:
        res.append(chunks[i])

print(*res)
```代码直接遵循分解策略。 循环`n & 127`提取最低 7 位，对应于 128 基数。 移位 7 移动到下一个数字。 对零的显式检查确保我们输出单个字节而不是空序列。 

反转步骤至关重要，因为连续位取决于最终排序中的位置，而不是提取顺序。 与 128 的按位 OR 为除最后一个字节之外的所有字节设置连续标志。

此跟踪显示分解如何与基数 128 数字对齐，以及在应用连续标志之前如何需要反转。

 ## 边缘情况

 ### 情况：n = 0

 输入为零，因此不运行 7 位分解循环。 该算法显式分配单个块 [0]，生成字节 0。没有连续位集，这符合编码不得引入前导零块的要求。 

### 案例：n = 127

 二进制恰好适合 7 位。 提取产生单个块[127]。 由于只有一个块，因此不添加连续位。 输出为 127，它正确表示没有额外字节的值。 

### 案例：n = 128

 二进制需要两个 7 位块。 提取得到 [0, 1]，反转为 [1, 0]。 第一个字节变为 1 | 128 = 129，第二个是 0。这与预期的跨字节边界的基于 128 的分割相匹配，并演示了对较高块中前导零有效负载的正确处理。 

### 案例：n = 255

 提取产生块 [127, 1]，反转为 [1, 127]。 第一个变成 1 | 128 = 129，第二个仍为 127。该算法正确编码需要多个字节的值，而不会放错连续位，这表明排序的处理与位内容无关。
