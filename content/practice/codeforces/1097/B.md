 ## 解决方案
 ## 问题理解

 我们得到了一个圆形锁，其行为就像一个 360 度的表盘。 从零开始，我们执行一系列旋转，每次旋转都有固定的幅度，但我们可以自由选择其方向：顺时针或逆时针。 顺时针可以被认为是增加角度，而逆时针可以被认为是减去角度。 

按顺序应用所有旋转后，指针以模 360 的某个最终角度结束。任务是确定是否存在所有旋转的方向选择，以使最终位置精确返回到零。 

输入尺寸非常小，最多 15 次旋转，每次旋转以 180 度为界。 这立即表明对所有方向选择的指数搜索是可行的，因为符号分配的总数最多为 2^15，即大约 32,000 个配置。 即使对每个配置进行简单的评估，这也完全在限制范围内。 

一个微妙的点是，这个问题纯粹是关于模 360 的净和。没有中间约束，因此顺序是固定的，只有符号重要。 

边缘情况大多与小n有关。 如果 n = 1，则仅存在一次旋转，而以零结束的唯一方法是该单次旋转为 0 或 360，但由于值至少为 1，因此答案始终为“否”。 天真的误解可能会尝试在每一步立即减少模 360，但这不会影响正确性，因为我们只关心最终的总和。 

另一种边缘情况是当总和超过 360 多次时。 例如，像 180、180、180 这样的旋转仍然可以工作，因为方向可以抵消形成 360 的倍数。 

## 方法

 思考这个问题最直接的方法就是尝试一切可能的方式为每次旋转分配方向。 对于每个角度，我们将其相加或相减，然后检查所得总和是否能被 360 整除。这种强力方法会尝试所有 2^n 符号分配。 它是正确的，因为它显式枚举了每个有效配置。 

这种方法的成本呈指数级增长。 当 n = 15 时，我们评估多达 32768 个配置，并且对于每个配置，我们计算 O(n) 的总和，给出大约 5×10^5 次操作。 这在 Python 中已经很好了，但我们可以进一步简化。 

关键的观察是，我们不需要任何超出强力的优化，因为约束足够小。 该问题本质上是一个子集和变体，其中每个元素可以是正数或负数。 这直接映射到位掩码枚举。 

因此，最佳解决方案就是迭代从 0 到 (1 << n) - 1 的所有掩码，将每个位解释为符号选择，计算结果总和，并检查它是否等于 360 的倍数。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力破解（位掩码）| O(n·2^n) | O(n·2^n) | O(1) | O(1) | 已接受 |
 | 最佳（相同）| O(n·2^n) | O(n·2^n) | O(1) | O(1) | 已接受 |

 在这个问题中，由于限制，蛮力已经是最优的。 

## 算法演练

1. 阅读角度列表。 每个角度代表一个固定的贡献，其方向未定。 
2. 使用从 0 到 (2^n - 1) 的位掩码迭代符号分配的所有可能子集。 每个位指示我们是顺时针还是逆时针旋转该步骤。 这对所有可能的有效决策路径进行编码。 
3. 对于每个位掩码，将运行总和初始化为零。 该总和表示应用所选方向后的净角位移。 
4. 对于旋转列表中的每个索引 i，检查掩码的第 i 位。 如果已设置，则将 a_i 添加到总和中，否则减去 a_i。 这对每次旋转的两个可能方向进行建模。 
5. 处理给定掩码的所有旋转后，计算模 360 的总和。如果它等于 0，则存在有效配置，我们可以立即停止。 
6. 如果没有掩码产生零模结果，则断定没有方向序列将指针返回到起始位置。 

### 为什么它有效

 每个有效的方向序列都唯一对应于一个位掩码。 从方向选择到 +a_i 或 -a_i 的转换保留了准确的最终角度。 因为我们枚举了所有 2^n 种可能性，所以我们涵盖了所有可能的最终结果。 模条件捕获了锁的圆形性质，确保 360 度的任何倍数都等于零。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = [int(input()) for _ in range(n)]
    
    for mask in range(1 << n):
        total = 0
        for i in range(n):
            if mask & (1 << i):
                total += a[i]
            else:
                total -= a[i]
        if total % 360 == 0:
            print("YES")
            return
    
    print("NO")

if __name__ == "__main__":
    solve()
```该实现直接遵循前面描述的位掩码枚举。 每个掩码代表顺时针或逆时针方向的完整分配。 内循环计算产生的位移。 模数检查手柄环绕在圆形表盘上。 

一个常见的错误是忘记减法与加法同样有效。 两者必须对称地考虑。 另一个潜在的陷阱是过早尝试减少以 360 为模的中间值，这在数学上是安全的，但没有必要，并且可能会模糊推理。

 ## 边缘情况

 对于 n = 1，算法同时检查 +a 和 -a。 由于 a 介于 1 和 180 之间，都不等于 360 的倍数，因此没有掩模满足条件，从而正确生成 NO。 

对于所有角度均为 180 的情况，例如 180 180，枚举包括 (+180, -180) 和 (-180, +180)，两者总和为零，正确返回 YES。 

对于较大的混合输入（如 90、90、90、90），存在多个掩码，其总和为 360 或 0，并且穷举搜索保证至少找到一个有效的掩码。
