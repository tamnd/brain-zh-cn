 ## 解决方案
 ## 问题理解

 我们得到了一组括号字符串，每个字符串都是左括号和右括号的某种组合。 从这些字符串中，我们可以形成不相交的对。 每个字符串最多可以使用一次，当我们将两个字符串连接成对时，得到的字符串必须是正确的括号序列。 

正确的括号顺序意味着，如果我们从左到右阅读，我们在任何前缀中看到的右括号永远不会多于左括号，并且最后左括号和右括号的总数相等。 

任务是最大化我们可以形成多少个这样的有效对。 

这些约束将我们推向 O(n log n) 或 O(n) 解决方案。 我们可能有多达 100000 个字符串，总长度高达 500000，因此任何比较所有对或直接模拟串联的方法都太慢。 我们需要将每个字符串压缩成一个小的签名，然后进行贪婪匹配。 

一个微妙的困难来自这样一个事实：一根弦​​的特征不仅仅在于它的总平衡。 具有相同总余额的两个字符串在连接时可能表现得非常不同，因为中间前缀可能会低于零。 

一个天真的错误是只跟踪净余额。 例如，仅仅因为“())(”和“(”的总和取消而将它们视为兼容是不正确的，因为“())(”在内部无效并且不能安全地吸收另一个前缀。

 ## 方法

 一个蛮力的想法是尝试所有字符串配对并检查连接是否产生有效的括号序列。 对于每一对，我们模拟串联并验证前缀平衡。 每次检查都会花费两个字符串总长度的线性时间，因此在最坏的情况下，这将变为 O(n²·L)，这远远超出了可行的限制。 

关键的观察是每个字符串都可以用两个值来概括：它的最终余额和它的最差前缀余额。 如果我们将“(”的balance定义为+1，将“)”定义为-1，那么每个字符串都会产生一个最小前缀和和一个最终和。 这两个值完全决定了它是否可以安全地与另一个相反“类型”的字符串组合。 

如果我们将永远不会变负且以非负平衡结尾的字符串视为“左重候选”，而将永远不会反转正且以非正平衡结尾的字符串视为“右重候选”，那么配对就会减少为匹配兼容的签名而不是原始字符串。 

该问题变成了两个类别之间的匹配任务，其中每个字符串对其可以容忍的不平衡程度做出了限制。 

分类后我们可以将一切简化为贪婪配对策略。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力配对+验证| O(n²·L) | O(1) | O(1) | 太慢了 |
 | 平衡签名+贪心匹配| O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 我们处理每个字符串并计算两个值：其净余额和最小前缀余额。 

1. 对于每个字符串，通过扫描字符计算其平衡。 '(' 递增，')' 递减。 还跟踪扫描期间的最小前缀余额。 这告诉我们弦线曾经低于零多远。 
2. 如果一个字符串的前缀和低于零，它的行为就像一个“重闭合”字符串。 否则，它是“安全打开”的。 我们根据字符串是否以正余额或负余额结尾将字符串分为两组，但关键的分类是基于它们是否可以安全地附加到一对的左侧或右侧。 
3. 对于每个字符串，我们将其压缩为一对 (net_balance, min_prefix)。 这对准确地捕获了附加时它如何转换初始余额。 
4. 我们将字符串分为两类：

净余额非负的字符串和净余额为负的字符串。 直观上看，正数需要开盘支撑，负数提供收盘支撑。 
5. 对于非负组，我们按其最小前缀约束按升序排序。 这确保我们首先使用“最脆弱”的字符串，这意味着它们不能容忍太多的初始不平衡。 
6. 对于负组，我们按其反向约束进行排序，有效地优先考虑最容易匹配的字符串。 
7. 我们贪婪地从每一边匹配一个。 我们维护可行前缀的计数器：如果正组中的字符串与负组中的字符串余额互补且不违反前缀约束，则它们可以与负组中的字符串配对。 
8. 每一次成功的匹配都会形成一对有效的字符串，并且两个字符串都会被排除在考虑范围之外。 

### 为什么它有效

 核心不变量是我们总是首先匹配最受约束的可用字符串。 具有更严格的最小前缀条件的字符串的有效伙伴较少，因此推迟它只能减少最终的匹配数量。 排序确保如果有可能配对，我们就会在两个候选者仍然可用时遇到它。 平衡约束保证任何接受的对都连接成全局有效的序列，因为一旦按正确的顺序放置，这两个字符串都会引入不可逆的前缀冲突。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def analyze(s):
    bal = 0
    mn = 0
    for c in s:
        if c == '(':
            bal += 1
        else:
            bal -= 1
        if bal < mn:
            mn = bal
    return bal, mn

n = int(input())
pos = []
neg = []

for _ in range(n):
    s = input().strip()
    bal, mn = analyze(s)
    if bal >= 0:
        pos.append((mn, bal))
    else:
        neg.append((mn - bal, -bal))

pos.sort()
neg.sort()

i = j = 0
ans = 0

while i < len(pos) and j < len(neg):
    mn1, bal1 = pos[i]
    mn2, need = neg[j]

    if bal1 + need >= 0:
        ans += 1
        i += 1
        j += 1
    else:
        i += 1

print(ans)
```该解决方案首先将每个字符串压缩为平衡配置文件。 辅助函数计算净余额和最小前缀值，这确定字符串如果放置在串联中间是否会变得无效。 

然后根据字符串是否以非负余额或负余额结尾来划分字符串。 这种分离至关重要，因为它使我们能够将一侧视为提供多余的开放能力，而将另一侧视为需要它。 

排序确保我们首先尝试匹配最严格的字符串。 两个指针扫描两个组，并且仅在形成匹配或由于无法配对而跳过候选者时才前进。 

一个常见的实施陷阱是忽略前缀最小值并仅使用净余额。 这会导致配对字符串全局取消但局部破坏，从而使正确性失效。

 ## 边缘情况

 一种边缘情况是字符串已经是完整的正确括号序列，例如“()”。 这些具有零净余额和非负前缀行为。 该算法将它们视为灵活的元素，并且它们自然地与其他中性或互补字符串配对，而不会违反约束。 

另一个边缘情况是早期严重负数但后来恢复的字符串，例如“())(()”。即使净余额可能为零，前缀下降也使它们在许多配对中无法使用。最小前缀跟踪可确保此类字符串被正确分类为限制性字符串，并且仅当存在有效的合作伙伴来补偿其早期赤字时才进行配对。

 最后一种边缘情况是所有弦都严重向一个方向倾斜。 在这种情况下，排序仍然确保我们以正确的顺序尝试所有可能的匹配，但是当不存在进一步的有效补集时，贪婪扫描自然停止，从而避免计数过多。
