 ## 问题理解

 两名玩家轮流写一个数字。 首先，对手宣布一个整数`n`。 看到之后我们选择自己的整数`m`。 谁写的数字较大，谁就立即获胜。 

目标是始终保证胜利，无论发生什么`n`是，同时尊重我们选择的数字必须位于之间的约束`0`和`100000`。 

这将问题转化为在固定上限下做出的简单决策。 我们并不是试图在多轮中动态做出反应或维护数据结构，我们是在观察一个输入后选择一个值。 

约束条件`1 ≤ n ≤ 100`至关重要。 它告诉我们，与最大允许值相比，对手的数量总是非常小`m`。 这已经表明，任何试图仔细比较或适应每个案例的策略都是不必要的，因为可行的范围`m`比对手能产生的任何东西都大得多。 

一个天真的错误是计算类似的东西`m = n + 1`。 这适用于此类类型的大多数竞争性编程问题，但在这里它不是必需的，并且在保证严格的优势裕度方面也不是最佳的。 更重要的是，粗心的实现可能仍然在范围内，但会在以下情况下失败：`n`可能接近`100000`。 

例如，如果规则不同：

 输入：```
100000
```然后`m = n + 1`会超出限制。 在这个问题中我们是安全的，但约束正是允许更简单的常数解的原因。 

## 方法

 暴力解释是尝试每一个可能的值`m`从`0`到`100000`并选择一个严格大于`n`。 这是正确的，因为我们直接模拟每个候选人的规则并检查它是否获胜。 成本顶多是`100001`检查，即使对于许多测试用例来说，这也是微不足道的。 

然而，这种方法是不必要的，因为问题的结构不依赖于除了单一比较之外的任何交互。 一旦我们意识到获胜仅取决于严格大于`n`，我们就不再需要搜索了。 

关键的观察是我们想要尽可能大的允许数量。 自从`100000`是我们允许输出的最大值，我们检查它是否保证胜利。 因为`n ≤ 100`，它总是认为`100000 > n`，因此最大可能的选择已经满足每种情况下的获胜条件。 

这将问题简化为常数时间决策。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(100000) | O(1) | O(1) | 已接受 |
 | 最佳 | O(1) | O(1) | O(1) | O(1) | 已接受 |

 ## 算法演练

 1.读取整数`n`来自输入。 这代表了对手的固定数量。 
2. 认识到我们的号码`m`必须同时满足两个条件：它必须在`[0, 100000]`，并且它必须严格大于`n`。 
3.观察最大可能的候选者`100000`在约束下有效。 
4. 隐式比较：since`n ≤ 100`，不等式`100000 > n`总是成立。 
5、输出`100000`因为它既有效又保证获胜。 

### 为什么它有效

 该算法依赖于问题的单调结构：任何较大的数字总是支配任何较小的数字。 因为我们可以在允许的范围内选择最大可能的值，而对手的值严格限制在远低于它的范围内，所以最大值始终是占优策略。 没有任何配置可以让较小的选择获得更好的性能`100000`，因为获胜仅取决于严格大于`n`。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input().strip())
    print(100000)

if __name__ == "__main__":
    solve()
```该解决方案读取单个输入值并立即输出`100000`。 没有分支，因为约束`n`保证这个值总是超过它。 该实现避免了不必要的比较或算术，保持逻辑最小化和健壮。
