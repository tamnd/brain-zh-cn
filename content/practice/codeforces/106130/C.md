 ## 问题理解

 我们正在与隐藏的尺寸排列进行交互$n$。 排列在任何查询之前都是固定的，每个查询允许我们提交任何长度 -$n$大批。 系统会响应我们提交的数组和隐藏排列之间有多少位置完全匹配。 

任务是最多使用重建整个排列$n^2$每个测试用例都有这样的查询。 

关键的限制是$n \le 100$，以及总和$n^2$所有测试用例的上限为 10000。这立即告诉我们二次查询策略是可以接受的，但是任何三次或需要大量每次查询计算的策略也可以，因为每个查询本身只是$O(n)$。 真正的限制是查询的数量，而不是每次查询的成本。 

天真的误解通常来自于将每个查询视为提供位置信息。 实际上，每个响应只是一个全局分数，因此如果不设计隔离信息的查询，直接重建是不可能的。 

如果我们假设单个随机或贪婪重建策略有效，就会出现微妙的失败情况。 例如，如果我们尝试根据本地查询为每个位置独立分配值，我们很快就会遇到歧义：

 如果$n = 3$隐藏的排列是$[2,3,1]$，像独立测试每个职位的候选人这样的幼稚尝试很容易产生冲突的作业，因为反馈不能定位错误。 

核心挑战是设计查询，让我们以受控方式“比较替代方案”，以便可以确定地解决每个位置。 

## 方法

 蛮力的想法是通过尝试每个位置的所有可能性来完全重建排列。 对于职位$i$，我们可以尝试设置$a_i = x$剩下的任意填充，然后观察匹配数是否增加。 然而，这从根本上来说是不可靠的，因为响应聚合在所有位置上都是匹配的。 一个坐标的任何变化都会以一种不可隔离的方式影响全局得分，除非仔细控制数组的其余部分。 

如果我们尝试暴力破解整个排列，我们将需要测试所有$n!$候选人，这甚至是完全不可行的$n = 100$。 

关键的观察是我们不需要一次隔离一个位置。 相反，我们可以维持排列的_当前猜测_并逐步纠正它。 每个查询都会将我们的猜测与隐藏的排列进行比较，并且响应会准确地告诉我们当前有多少位置是正确的。 

这创造了一个自然的优化方向：如果我们交换当前猜测中的两个位置并且正确匹配的数量增加，则交换是有利的； 如果它减少，我们就撤消它。 这本质上是在全局分数指导下对排列进行局部搜索。 

问题的结构使其可行，因为评分函数是一致且确定的。 每次交换都会以可预测的方式将分数最多改变 2，因此我们可以安全地使用它来收敛到正确的排列$O(n^2)$互换。

 ## 算法演练

 我们维护一个当前数组$p$，最初设置为任何排列，通常$[1, 2, \dots, n]$。 

1. 我们用当前排列查询交互器并存储匹配数$k$。 该值表示在我们当前的猜测下有多少位置已经正确。 
2. 我们迭代所有索引对$(i, j)$和$i < j$，并考虑交换$p[i]$和$p[j]$。 
3. 交换后执行查询并获取新的匹配计数$k'$。 这告诉我们交换是否改善了与隐藏排列的对齐。 
4.如果$k' > k$，我们保留交换，因为它增加了正确位置的数量。 我们还更新了$k \leftarrow k'$。 
5. 如果$k' \le k$，我们恢复交换，因为它不会提高正确性。 这确保我们永远不会失去进展。 
6. 我们继续扫描对，直到没有交换可以提高分数，此时排列必须与隐藏排列相匹配。 

该过程正确终止的原因是分数介于 0 和$n$，并且每次接受的交换都严格将其增加至少 1。由于我们只接受改进，所以我们不能循环。 

## 为什么它有效

 评分函数使用固定排列来计算精确匹配。 任何交换仅影响两个位置，并且分数的变化仅取决于这两个位置之前是正确还是不正确。 如果交换增加了分数，则意味着至少一个先前不正确的位置已变为正确，而不会破坏多个正确位置。 因为我们只接受严格改进的动作，所以我们单调增加固定点的数量，直到达到$n$，它完全对应于隐藏的排列。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def ask(a):
    print("?", *a)
    sys.stdout.flush()
    return int(input())

def solve():
    n = int(input())
    p = list(range(1, n + 1))

    cur = ask(p)

    for i in range(n):
        for j in range(i + 1, n):
            p[i], p[j] = p[j], p[i]
            nxt = ask(p)

            if nxt >= cur:
                cur = nxt
            else:
                p[i], p[j] = p[j], p[i]

    print("!", *p)
    sys.stdout.flush()

t = int(input())
for _ in range(t):
    solve()
```该代码首先读取排列的大小并初始化一个简单的恒等猜测。 这是一个自然的基线，因为它保证了有效的排列结构。 

每个查询都会评估当前猜测与隐藏排列的接近程度。 嵌套循环系统地尝试所有成对交换。 每次交换后，我们立即测试更改是否改善了对齐。 如果没有，我们就将其撤消以保持单调性。 

每次输出后的刷新是交互协议所要求的； 如果没有它，交互器将无法及时接收查询，从而破坏同步。 

## 工作示例

 考虑一个隐藏的排列$p = [2, 1, 3]$, 初步猜测$[1, 2, 3]$。 

### 追踪

 | 步骤| 当前数组 | 查询结果 |
 | --- | --- | --- |
 | 初始化| [1,2,3]| 1 |
 | 交换(0,1) | [2,1,3]| 3 |

 第一个查询显示一个正确的位置。 交换位置 0 和 1 后，分数跳至 3，这意味着交换对齐了整个排列，因此被接受。 

这表明，当身份排列接近目标时，有益的交换会立即暴露出正确的结构。 

现在考虑$p = [3,2,1]$。 

| 步骤| 当前数组 | 查询结果 |
 | --- | --- | --- |
 | 初始化| [1,2,3]| 1 |
 | 交换(0,2) | [3,2,1]| 3 |
 | 接受| [3,2,1]| 3 |

 在这里，单个远程交换同时纠正了两个不匹配的位置，展示了评分函数如何引导我们即使在反馈最少的情况下也能实现全局正确性。

 ## 边缘情况

 对于$n = 1$，该算法执行单个查询并立即返回唯一可能的排列。 无需考虑交换，因此身份初始化已经正确。 

为了$n = 2$，恰好存在一种可能的交换。 如果最初的猜测不正确，则交换必然会将匹配计数增加到 2，因此始终会被接受。 这表明即使在最小的非平凡情况下，单调改进属性也成立。 

对于较大的$n$，该算法依赖于这样一个事实：任何不正确的排列必须包含至少一对交换可改善对齐的位置。 贪婪地接受改进互换保证了收敛，因为分数不能降低并且上限为$n$。
