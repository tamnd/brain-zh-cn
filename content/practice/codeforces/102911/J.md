 ## 解决方案
 ## 问题理解

 我们有一组学生和几个学生组织。 每个组织都由一组官员组成，每个官员都必须亲自出席其组织的舞会才能使活动顺利进行。 

每个组织的舞会必须恰好安排在两天中的一天：周六或周日。 关键的限制是学生不能参加同一天安排的两个不同的舞会。 由于一名学生可能属于多个组织，这会在至少共享一名官员的事件之间造成冲突。 

任务是将每个组织分配到星期六或星期日，这样就不需要学生在同一天参加两个组织。 如果这是不可能的，我们必须报告失败。 如果存在多个有效的时间表，我们还更喜欢一个能够最大限度地增加星期六安排的组织数量的时间表，但任何有效的最大星期六解决方案都是可以接受的。 

输入自然是学生和组织之间的二分关联结构：每个组织都是一个集合，集合之间的重叠定义了冲突。 如果两个组织至少共享一名学生，则它们会发生冲突，这意味着它们不能在同一天被分配。 

需要注意的主要限制是规模。 所有组织的成员条目总数可能达到数十万，因此任何明确比较所有组织对的解决方案都是不可能的。 对组织的二次检查将立即失败。 相反，我们必须将问题简化为由共享学生构建的图结构，并在近线性时间内处理它。 

当一名学生属于三个或更多组织，而这些组织都通过该学生成对重叠时，就会出现微妙的边缘情况。 例如，如果一个学生在三个组织中，则所有三个组织都会通过该共享成员相互约束。 在不考虑传递传播的情况下对每个组织进行的天真的贪婪分配很容易在以后产生矛盾。 

另一个极端情况是当冲突图包含奇数循环时。 例如，三个组织两两共享不同的重叠学生，会强制形成一个不能一致地使用两种颜色的约束循环，从而使调度变得不可能。 检测这一点需要全局推理而不是局部检查。 

## 方法

 蛮力的想法是独立对待每个组织并尝试周六或周日的所有任务。 对于 M 个组织，这会产生 2^M 种可能性。 对于每项作业，我们都会通过检查每个学生并确保他们所属的所有组织中最多一个被分配到周六，最多一个到周日来验证有效性。 构造和验证单个分配已经需要 O(成员资格之和)，因此总数变为 O(2^M ·total_membership)，即使对于数十个 M，这也是远远不可行的，更不用说数十万了。 

关键的观察是，一旦我们意识到他们强制执行什么，我们实际上并不直接关心学生。 每个学生都会在他们所属的所有组织之间引入约束：这些组织中没有两个可以共享同一天。 这相当于说，对于每个学生来说，包含他们的组织在冲突图中形成了一个派系。 整个问题简化为检查该图是否是二分图，因为我们需要分配两种颜色（天）之一，以使相邻组织不同。 

一旦被视为二分检查，问题就变成了标准的图形着色。 不同的是，图表没有明确给出； 它是通过共享学生隐式定义的。 我们必须通过迭代每个学生并连接他们出现的所有组织来构建邻接性。

最大化周六作业的要求不会改变可行性。 任何二分图的每个连接组件都具有两种有效的着色，直至交换颜色。 因此，我们可以选择每个组件的方向，以便更多的节点落入星期六一侧。 每个组件的局部翻转产生全局最大值。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力作业 | O(2^M·total_membership) | O(2^M·total_membership) | O(2^M·total_membership) | O(成员总数) | 太慢了|
 | 图构建+二分着色| O(N + 总成员数) | O(N + M + 总成员数) | 已接受 |

 ## 算法演练

 我们将每个组织建模为图中的一个节点。 如果两个节点至少共享一名学生，我们将连接它们。 由于为每个学生显式构建所有成对边缘的成本可能很高，因此我们使用标准技巧来迭代每个学生并连接其成员列表中的所有组织。 

1. 我们为所有组织创建一个邻接列表。 对于每个学生，我们收集他们所属的组织列表，然后连接该列表中的连续组织以形成边。 这已经足够了，因为如果一个学生属于 k 个组织，将它们连接成一条链已经通过二分约束的传递性强制整个组的一致性。 
2. 我们对所有组织进行图遍历。 对于每个未访问的组织，我们启动 BFS 或 DFS，并尝试使用值 0 和 1（代表星期六和星期日）对连接的组件进行 2 色着色。 我们指定任意的起始颜色。 
3. 在遍历过程中，当我们访问从组织 u 到 v 的边时，如果 v 未分配，则为 v 分配 u 的相反颜色。 如果它已经被分配并且与你的颜色匹配，我们立即知道配置是不可能的并停止。 
4. 完成连接组件后，根据最初的选择，我们可能有两种有效的颜色。 为了最大化星期六的分配，我们计算有多少个节点获得颜色 0 和颜色 1，并根据需要翻转整个组件，以便颜色 0 对应于较大的组。 
5. 最后，我们根据每个组织的最终颜色输出分配的日期。 

正确性来自于独立处理每个连接的组件。 在组件内部，每个约束都由边强制执行，并且二分性保证当且仅当不存在奇数循环时才能满足所有约束。 翻转步骤不会破坏有效性，因为交换颜色保留了邻接约束，同时仅更改标记为星期六的一侧。 

## Python 解决方案```python
import sys
input = sys.stdin.readline
from collections import defaultdict, deque

def solve():
    n, m = map(int, input().split())
    
    orgs = []
    student_map = defaultdict(list)
    
    # read input
    for i in range(m):
        r = int(input())
        names = input().split()
        orgs.append(names)
        for name in names:
            student_map[name].append(i)
    
    adj = [[] for _ in range(m)]
    
    # build graph: connect organizations sharing students
    for members in student_map.values():
        for i in range(len(members) - 1):
            u = members[i]
            v = members[i + 1]
            adj[u].append(v)
            adj[v].append(u)
    
    color = [-1] * m
    answer = [None] * m
    
    for i in range(m):
        if color[i] != -1:
            continue
        
        queue = deque([i])
        color[i] = 0
        comp = [i]
        
        ok = True
        
        while queue:
            u = queue.popleft()
            for v in adj[u]:
                if color[v] == -1:
                    color[v] = color[u] ^ 1
                    comp.append(v)
                    queue.append(v)
                elif color[v] == color[u]:
                    ok = False
        
        if not ok:
            print("NO")
            return
        
        cnt0 = sum(1 for x in comp if color[x] == 0)
        cnt1 = len(comp) - cnt0
        
        flip = cnt1 > cnt0
        
        for x in comp:
            final_color = color[x] ^ flip
            answer[x] = "Saturday" if final_color == 0 else "Sunday"
    
    print("YES")
    for x in answer:
        print(x)

if __name__ == "__main__":
    solve()
```代码直接遵循图构建思想。 邻接列表是使用共享学生成员资格构建的，并且使用 BFS 来分配交替的日期。 唯一微妙的实现细节是，我们必须在遍历期间存储每个连接的组件，以便我们可以在知道两个颜色计数后决定是否翻转它。 如果不存储组件节点，则优化步骤将不可能进行。 

这里的一个常见陷阱是忘记学生可能会出现在许多组织中。 如果我们尝试连接学生列表中的所有对，我们将面临每个学生 O(k^2) 条边的风险。 这里使用的线性链连接避免了爆炸，同时仍然保留了二分约束。 

## 工作示例

 考虑一个包含三个组织 A、B 和 C 的小型配置，其中 A 和 B 共享一个学生，B 和 C 共享另一个学生，但 A 和 C 不直接共享任何人。 

### 示例 1

 输入结构：

 A 包含 Alice 和 Bob

 B 包含鲍勃和查理

 C包含查理

 | 步骤| 处理节点| 颜色分配 | 队列状态|
 | --- | --- | --- | --- |
 | 开始| 一个 | A = 0 | [一] |
 | 访问 A | B 分配 1 | [乙] | |
 | 访问 B | C 分配 0 | [C] | |
 | 访问 C | 完成 | []| |

 遍历后，颜色为A=0，B=1，C=0。 

这显示了传递约束如何通过共享成员正确传播，即使两个组织不直接重叠。 

### 示例 2

 考虑具有两个独立组件的断开连接场景，其中一个组件具有更多颜色为 0 的节点，另一个组件具有更多颜色为 1 的节点。 

| 组件| 节点| 原色 | 翻转决定后|
 | --- | --- | --- | --- |
 | 1 | 3 个节点 | 2 个零，1 个一 | 保持|
 | 2 | 4 个节点 | 1 零，3 1 | 翻转|

 这表明翻转是独立应用于每个组件的，确保最大分配到周六而不影响可行性。 

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 | O(N + 总和 r_i) | 每个学生到组织的边在图构建和 BFS 遍历期间都会处理一次 |
 | 空间| O(N + M + 总和 r_i) | 邻接列表以及成员资格和 BFS 状态的存储 |

 这些约束允许多达数十万个成员资格条目，并且算法对每个条目处理恒定的次数。 这使得运行时间舒适地保持在典型的 2-3 秒约束环境的限制内。

 ## 边缘情况

 一个重要的边缘情况是学生属于大量组织。 在这种情况下，我们不会显式连接所有对； 相反，我们依赖于通过链边缘的 BFS 传播。 该算法仍然正确地强制该学生组中的所有组织在二分区之间交替，因为连接性会传递限制。 

另一个极端情况是一个完全脱节的组织，不与其他人共享学生。 这些节点形成大小为一的独立组件。 该算法最初为它们分配颜色 0，它们自然地为“最大化星期六”目标做出贡献，没有任何限制。 

最后的边缘情况是由奇怪的重叠周期引起的不一致。 例如，三个组织中，A与B共享一个学生，B与C共享一个学生，C与A共享一个学生。在BFS着色时，最终会产生矛盾，已经着色的节点被分配与邻居相同的颜色，立即触发拒绝并输出NO。
