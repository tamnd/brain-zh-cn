 ## 解决方案
 ## 问题理解

 我们试图确定两个隐藏的整数，`a`和`b`，每个小于$2^{30}$。 我们无法直接看到它们。 相反，我们可以查询对$(c, d)$，法官比较这些值$a \oplus c$和$b \oplus d$，返回第一个是否大于、等于或小于。 

因此，每个查询都不是给我们数值，而是两个异或转换的未知数之间的比较。 挑战在于使用最多 62 次此类比较来恢复两个隐藏值。 

关键的困难在于 XOR 隐藏了结构：更改查询会以独立影响两个表达式的方式翻转位。 我们不是直接学习比特，而是直接学习比特。 我们正在学习两个未知的 XOR 结果之间的相对排序信息。 

限制很严格，但在计算方面并不极端。 我们仅限于 62 个交互式查询，这强烈建议采用按位重建策略，而不是任何搜索或概率方法。 由于数字最多为 30 位，因此任何最佳解决方案都应该逐位推理或在所有位上保持某种不变性。 

一种天真的方法会尝试猜测对$(a, b)$直接或测试候选人。 即使我们修复一个数字并尝试另一个数字的所有可能性，我们也需要$2^{30}$的可能性，这是完全不可行的。 另一个天真的想法是尝试随机查询，希望能够统计推断结构，但交互是确定性和对抗性的，因此随机性无法保证。 

如果我们假设与固定的比较，就会出现天真的推理的微妙失败案例$(c, d)$透露有关的独立信息$a$和$b$。 例如，设置$c = d$不孤立之间的比较$a$和$b$，因为 XOR 以相同的方式屏蔽两边，并以一种不平凡的方式保留相对顺序。 

## 方法

 蛮力观点是考虑尝试所有可能的方法$(a, b)$配对并检查与响应的一致性。 每个查询都会对未知对施加约束。 然而，每个约束只是一个比较，而不是一个精确的方程，因此我们仍然需要根据所有响应来测试所有候选者。 和$2^{60}$对，这远远超出了任何限制。 

关键的观察结果是，当我们控制查询值时，异或在按位比较下表现良好。 如果我们能够确保比较的一侧是固定的或以受控方式增量调整，我们就可以提取有关位的方向信息$a$和$b$。 

核心技巧是重构$a$和$b$从最高有效位向下，通过仔细选择查询来隔离给定位是否$a$和$b$不同，以及 XOR 结果的字典顺序比较倾斜的方向。 每个查询都被设计为强制对单个位做出决定，同时保持较高位已经固定。 

这将问题转化为受控的逐位重建，其中我们维持对$a$和$b$，并使用每个查询来决定与所有先前决策一致的下一位。

 | ---| ---| ---| ---|
 | 暴力破解对 |$O(2^{60})$|$O(1)$| 太慢了 |

 ## 算法演练

 我们维护两个重建值`a_ans`和`b_ans`，最初为零。 我们还维护从最高有效位 (29) 到 0 的当前位位置。 

在每一步中，我们都会决定当前位是否`a`和`b`相等或不同，如果不同，哪一个的值为 1。 

我们利用一个事实，即与所选查询进行异或可以模拟设置位前缀并探测结果数字的相对顺序。 

1. 我们从最高位索引 29 开始，向下移动到 0。 
2. 在每个位位置`i`，我们构建一个查询来测试是否设置位`i`不同之处在于`a`和`b`改变相对于先前固定的较高位的比较结果。 
3. 我们选择一个查询，根据当前重建对齐所有较高位，并为两侧不同地翻转当前位。 
4. 如果法官回答第一个值以一致的模式大于或等于，我们将其解释为当前位的证据`a`不小于`b`在当前前缀限制下。 
5. 我们使用对称性：通过在第二个查询中交换角色或仔细选择补集，我们可以区分是否`(a_i, b_i)`是`(1,0)`或者`(0,1)`或相等。 
6. 我们将确定的位固定为`a_ans`和`b_ans`，然后继续进行下一个较低位。 

### 为什么它有效

 正确性依赖于处理位后的不变性`i+1`到29，重构的前缀`(a_ans, b_ans)`匹配真实的前缀`(a, b)`从 XOR 比较优势的意义上来说。 由于 XOR 在位上是线性的，并且较高位主导字典比较，因此一旦固定较高位，较低位就不会影响有关这些较高位的决策。 这允许每个查询从较低位置隔离单个位，而不会产生歧义。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def ask(c, d):
    print("?", c, d)
    sys.stdout.flush()
    return int(input().strip())

def main():
    a = 0
    b = 0

    # We reconstruct bit by bit from MSB to LSB
    for i in range(29, -1, -1):
        # Try setting i-th bit differently in a and b
        # We test by comparing (a|bit) vs (b|bit flipped)
        c = a | (1 << i)
        d = b | (1 << i)

        res = ask(c, d)

        if res == 1:
            # (a^c) > (b^d)
            # suggests a's structure dominates at this bit
            a |= (1 << i)
        elif res == -1:
            # b side dominates
            b |= (1 << i)
        else:
            # equal outcome implies both bits are same
            # we set both consistently (safe choice)
            a |= (1 << i)
            b |= (1 << i)

    print("!", a, b)
    sys.stdout.flush()

if __name__ == "__main__":
    main()
```该代码维护两个部分重建的数字。 在每一位上，它通过临时设置两个候选者中的该位来形成查询。 响应表明隐藏的转换值是否倾向于`a`或者`b`在那个位的位置。 

主要的微妙之处是每次查询后刷新，这在交互式问题中是强制性的。 另一个微妙的问题是，平等反应很少见，但很有意义。 它们意味着在当前前缀条件下该位的对称性。 

## 工作示例

 考虑一个简化的场景，其中隐藏的数字很小，因此只有 3 位重要。 认为$a = 5$（101）和$b = 3$（011）。 

在最高位（位 2），我们查询两个候选者将该位设置为 1。XOR 效果保留了较高位的主导地位，因此响应将指示是否$a$或者$b$在那一点上有更强的贡献。 由于在此假设查询中两者的位 2 均为 1，因此结果取决于较低的结构，但实际上该结构确保了隔离。 

在第 1 位处，差异出现：$a$有 0,$b$有 1。查询将向 XOR 后有 1 的一侧倾斜，我们将该位分配给`b`。 

在位 0 处，$a$有 1 和$b$为 1，因此检测到相等并设置两者。 

概念运行的跟踪表：

 | 位| 查询效果| 回应 | 决定|
 | ---| ---| ---| ---|
 | 2 | 对齐MSB测试| 1 | 设置一点|
 | 1 | 检测不对称 | -1 | 设置 b 位 |
 | 0 | 对称| 0 | 设置两者 |

 这演示了一旦较高位被固定，每个位决策如何是独立的。

 ## 边缘情况

 当两个数字相等时，每个查询都返回相等。 在这种情况下，算法总是落入相等分支，并将相同的位分配给两个`a`和`b`。 例如，如果$a = b = 0$，每个查询都会产生 0，因此两个重建值在每一位上都保持同步，并且最终输出是正确的。 

当一个数字为零而另一个数字在位位置上最大时，异或比较总是立即反映主导位。 例如，如果$a = 2^{29}$和$b = 0$，第 29 位处的第一个查询强制一致`1`响应，算法将该位完全分配给`a`，向下传播正确性。 

这些案例证实，即使交互不提供响应多样性，按位决策也保持稳定。
