 ## 解决方案
 ## 问题理解

 我们在数组上反复执行全局“级别降低”。 在每一步中，我们都会查看当前存在的所有正值，确定其中最小的值，输出该值，然后将每个正值减少相同的量。 零保持不变，不再贡献。 

即使数组之前完全变为零，此过程也会持续进行固定数量的操作。 一旦一切都为零，每个剩余的操作都会输出零。 

重要的结构是，数组从底部开始逐层剥离：每个操作都会从所有剩余的正条目中删除当前的最小正“高度”。 

约束允许最多 100,000 个元素和 100,000 次操作。 任何通过在每一步中扫描整个数组来重新计算最小值的解决方案在最坏的情况下都会导致大约 10^10 次操作，这远远超出了时间限制。 即使每一步重复进行完整排序或完整重新计算也不会通过。 

一些边缘案例清楚地暴露了幼稚的方法。 

如果数组已经全部为零，则答案就是 k 个零。 假设至少存在一个正值的简单实现可能会尝试计算最小值并崩溃或返回无效值。 

如果 k 远大于不同正值的数量，则该过程最终稳定在全零并继续打印零。 例如，`[5]`k = 3 产生`5, 0, 0`。 在数组变空后提前停止而不填充剩余操作的解决方案是不正确的。 

## 方法

 直接模拟很容易想象：重复扫描数组，找到最小的非零元素，从每个正元素中减去它，然后打印它。 这是正确的，因为它完全遵循操作定义。 然而，每一步的成本为 O(n)，在最坏的情况下，我们可能会执行 O(k) 步，从而得到 O(nk)。 两者都达到10^5，这就变成了10^10次操作，这是不可行的。 

关键的观察是，我们实际上并不关心每时每刻不断变化的数组值。 当数组被“展平”时，我们只关心最小正值的序列。 每次我们减去当前的最小正值，至少有一个元素变为零，并且活动值集合会缩小。 该过程相当于按升序重复提取不同的正值。 

如果我们对数组进行排序，操作顺序就会变得更加清晰。 每当我们从一个不同的值移动到下一个更大的值时，该差异对应于一个新的减法层。 相同的值折叠在一起，产生零长度间隙。 特定值贡献的次数取决于有多少个不同的“级别”保持在其之上。 

因此，对数组进行排序并处理相等值的组允许我们计算每个最小值出现的次数，而无需模拟每次减法。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力模拟| O(nk) | O(nk) | O(1) | O(1) | 太慢了 |
 | 排序+分组 | O(n log n) | O(n log n) | O(1) 或 O(n) | 已接受 |

 ## 算法演练

 1. 对数组进行非降序排序。 这确保了相等的值被分组，并且最小的剩余值始终可以在结构的前面访问。 
2. 迭代已排序的数组，跟踪正元素中之前看到的值。 每次遇到新的不同正值时，我们都会计算从此时起有多少元素保持活动状态。 
3. 对于每个不同的值，确定将打印多少次。 这等于严格大于或等于该值的剩余元素数量减去早期级别已“消耗”的元素。 
4. 根据剩余的活动元素数量，重复输出连续不同值之间的差异。 从概念上讲，每组相等的值贡献一个“层高”减法步骤。 
5. 一旦我们产生了 k 个输出就停止。 如果我们在达到 k 之前耗尽所有正值，则为剩余操作附加零。 

正确性来自于以下不变量：在每个减法步骤之后，正值的多重集恰好是原始值减去已通过的不同阈值的数量。 排序直接揭示了这些阈值，因此每个不同的值对应于一个步骤，在该步骤中它成为新的最小值。 

该算法之所以有效，是因为该过程相当于剥离排序值级别。 每个级别的变化对应于排序数组中严格递增的值，并且每个这样的变化恰好生成一个输出，直到所有元素都耗尽。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, k = map(int, input().split())
    a = list(map(int, input().split()))
    
    a.sort()
    
    res = []
    i = 0
    
    while i < n and len(res) < k:
        if a[i] == 0:
            i += 1
            continue
        
        # current minimum positive value
        current = a[i]
        
        # skip all equal values
        j = i
        while j < n and a[j] == current:
            j += 1
        
        # number of remaining active elements
        remaining = n - i
        
        res.append(current)
        
        # after subtracting, all elements in [i, j) become zero,
        # so we conceptually move forward
        i = j
    
    # if we still need outputs, fill with zeros
    while len(res) < k:
        res.append(0)
    
    print("\n".join(map(str, res)))

if __name__ == "__main__":
    solve()
```该实现依赖于排序来公开重复减法过程的结构。 指针`i`始终指向剩余的最小正值。 我们使用以下方法将相等的值分组在一起`j`，因为它们在减法下表现相同。 每组产生一个与当前最小值相对应的输出。 

剩余的零填充处理在 k 次操作之前耗尽所有元素的情况。 

一个微妙的点是我们从不显式地从数组中减去值。 该操作隐含在分组逻辑中：一旦我们传递了一个值，它就被认为完全从未来的最低考虑中删除了。

 我们输出每个不同级别一次，直到数组耗尽。 消耗完所有值后，我们用零填充。 

这表明每个不同的排序值贡献一个减法层。

 输入：```
4 5
10 3 5 3
```排序后的数组是`[3, 3, 5, 10]`。 

| 步骤| 我| 当前| 输出| 保持活跃 |
 | ---| ---| ---| ---| ---|

 每次我们移动到更高的不同值时，我们都会将该值作为先前减少后的下一个最小值发出。 

这证实了排序将重复的减法过程转换为一系列阈值跳跃。

 ## 边缘情况

 如果数组仅包含零，则算法立即进入填充阶段。 由于从未遇到正值，因此排序扫描不会产生输出，并且最终循环输出 k 个零。 这与定义匹配，因为每个操作都会看到一个全零数组。 

对于像这样的输入`[5]`当 k 大于 1 时，算法输出`5`on the first pass, then recognizes no remaining positive values and fills with zeros. This matches the idea that once the only element is reduced, the process stabilizes.

 如果所有元素都相等，则说`[3,3,3]`，排序后的扫描会产生一个不同的值。 该值输出一次，所有剩余操作均为零。 这反映了一个减法步骤同时删除所有元素的事实。
