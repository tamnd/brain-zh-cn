 ## 解决方案
 ## 问题理解

 这座城市被表示为一排塔楼，每座塔楼都有正高度。 你可以重复执行一个非常具体的操作：拿一个塔并将其移动到相邻的塔上，将它们合并成一个塔，其高度为两个塔的总和。 合并后，这两个位置成为一塔，因此塔总数减少一塔。 

经过任意次数的此类合并后，如果剩余的塔序列从左到右的高度不降低，则认为它们是“好的”。 任务是找到达到任何此类良好配置所需的最小合并操作数。 

以更具结构性的方式重新构建它，将原始数组压缩为连续的段。 每个部分对应一个最终的塔，其高度是其内部元素的总和。 操作规则保证段必须是连续的，因为合并仅发生在邻居之间。 

目标是选择将数组划分为多个段，以便段总和从左到右不递减，同时最大化段数。 每次合并都会将段数减少 1，因此最小化操作相当于最大化可以保留的有效段数。 

限制允许最多 5000 个塔。 具有三次行为的解决方案太慢，甚至具有大量过渡的二次方法也可能处于边缘状态。 这强烈表明线性或近线性贪婪或基于堆栈的构造。 

当本地合并看起来有益但实际上阻碍了更好的未来结构时，天真的推理的微妙失败案例就会出现。 例如，如果每当看到相邻塔之间的减少时就贪婪地合并，则可能会过早过度合并并不必要地减少最终段的数量。 正确的解决方案必须强制段总和的全局单调性，而不是对原始值的局部成对决策。 

## 方法

 暴力解释将尝试所有可能的方法来合并相邻的塔，直到序列变得非递减。 每次操作都会将塔的数量减少一个，并且在每一步中您都可以选择任何相邻的一对进行合并。 这会创建一个巨大的分支过程，因为每次合并后结构都会发生变化，并且未来的合并选择也会有所不同。 即使您将其建模为将数组的所有分区探索为连续的段，分区的数量也会以 n 为指数，这使得它在超出非常小的输入时不可行。 

关键的观察结果是，最终状态完全由划分为连续的段决定，并且有效划分的唯一约束是这些段的总和必须从左到右不递减。 我们可以从左到右贪婪地构造最优分区，同时保持有效性，而不是搜索所有分区。 

决定性的结构是，每当我们有三个连续的段，其总和为 A、B、C 且 A ≤ B > C 时，中间段 B 就会阻止可行性。 合并 B 和 C 会增加右段并可能恢复单调性。 这种行为自然可以通过维护一堆段总和并立即修复违规来处理，类似于离散序列的等渗回归。 

这将问题变成了单遍，每当违反单调性时我们都会向后合并，确保我们始终以最大数量的有效段结束。

 ## 算法演练

 我们从左到右处理数组，同时维护一个堆栈，其中每个元素代表最终分区中某个段的总和。

1. 从空堆栈开始。 数组的每个元素最初形成自己的段。 
2. 对于数组中的每个值，使用该单个值创建一个新段并将其推入堆栈。 
3. 压入后，检查最后两个段是否违反非递减顺序，即倒数第二个段的总和大于最后一个段。 
4. 每当存在此类违规时，通过将最后两个段替换为总和为总和的单个段来合并它们。 然后再次重复此检查，因为新的合并段现在可能违反其先前邻居的单调性。 
5. 继续此过程，直到堆栈为空或所有相邻段的和都按非递减顺序排列。 
6. 处理完所有元素后，堆栈中剩余的段数就是最终可能的最大塔数，答案是执行的合并次数，等于 n 减去堆栈大小。 

这种重复合并起作用的原因是相邻段之间的任何违规都无法在以后由未来的元素修复。 如果右侧出现较大的段，则对左侧排序约束没有帮助，因此恢复可行性的唯一方法是立即合并有问题的段。 

核心不变量是，每次迭代之后，堆栈代表已处理前缀的有效分区，并且在所有此类分区中，它保持与非递减约束一致的最大可能的段数。 每当出现违规时，延迟合并只会推迟不可避免的纠正，而不会增加段的数量，因此立即合并可以保持最优性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = list(map(int, input().split()))

    stack = []

    for x in a:
        stack.append(x)

        while len(stack) >= 2 and stack[-2] > stack[-1]:
            stack[-2] = stack[-2] + stack[-1]
            stack.pop()

    # number of merges = n - number of segments
    print(n - len(stack))

if __name__ == "__main__":
    solve()
```该实现直接反映了段的构造。 每个堆栈元素是一个当前段的总和。 while 循环是关键的修复步骤：每当新形成的段与其左邻居相比导致减少时，我们立即将它们合并。 重复条件是必要的，因为合并可能会向左传播违规行为。 

最终答案是根据段计数的总减少量来计算的，它与执行的操作数相匹配。 

## 工作示例

 考虑输入`5 8 2 7 3 1`。 

我们在每次插入后跟踪段堆栈：

 | 步骤| 已插入 | 堆栈（段总和）|
 | ---| ---| ---|
 | 1 | 8 | [8] |
 | 2 | 2 | [8, 2] → 合并 → [10] |
 | 3 | 7 | [10, 7] → 合并 → [17] |
 | 4 | 3 | [17, 3] → 合并 → [20] |
 | 5 | 1 | [20, 1] → 合并 → [21] |

 该过程不断合并，因为每个新元素都太小而无法维持不递减的段和。 最终堆栈的大小为 1，因此答案是 5 − 1 = 4 次操作。 

现在考虑`3 5 2 1`。 

| 步骤| 已插入 | 堆栈|
 | ---| ---| ---|
 | 1 | 5 | [5]|
 | 2 | 2 | [5, 2] → 合并 → [7] |
 | 3 | 1 | [7, 1] → 合并 → [8] |

 再次，所有内容都折叠成一个片段。 该方法正确地识别出不可能有稳定的分裂。 

这些痕迹表明，该算法在本地重复执行单调性，直到整个前缀变得一致，从而保证了全局正确性。

 ## 边缘情况

 单个塔的最小输入表明堆栈逻辑永远不会触发合并，并且答案为零，因为没有操作可以改进或更改顺序。 

严格递减序列如`4 3 2 1`在每一步都会触发重复合并。 每次新插入都会导致对累积段的违反，并且重复合并会将所有内容折叠到一个段中，从而产生 n − 1 次操作。 该算法自然地处理这个问题，因为每次合并都会减少堆栈大小，直到恢复单调性。 

具有小的局部反转的情况，例如`1 5 2 3`说明了为什么贪婪的相邻比较是不够的。 价值`2`强制合并`5`，但这不需要折叠整个前缀，并且堆栈仅通过在必要时进行合并来正确保留最大分段。
