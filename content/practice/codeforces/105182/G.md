 ## 解决方案
 ## 问题理解

 我们得到一个有根特里树，其中每个节点对应于一个字符串，该字符串是通过沿着从根到该节点的路径连接字符而形成的。 每个节点还带有一个需求值，这意味着该节点表示的字符串必须被写入一定的次数。 

书写是通过遍历字符来进行的。 每次我们手动输入一个字符，我们都会支付固定的成本`a`。 然而，还有一种缓存机制可以快速完成单词：在输入前缀时，系统会建议与当前前缀匹配的字典顺序最小的单词，我们可以通过支付固定成本立即完成该建议的单词`b`。 

任务是针对 trie 中所有需要的单词，决定如何通过结合手动输入和基于缓存的补全来最小化总成本。 

输入结构描述了一个有根树。 每个节点都知道它有多少个子节点，并且子节点已经按其边缘字符的字典顺序排序。 这种顺序很重要，因为缓存选择总是优先考虑最小的字典顺序。 

输出是满足所有节点需求的最小总成本。 

关键的困难在于每个节点对应一个单词，但我们没有选择子集。 我们必须处理所有节点`c[i] > 0`，可能多次，并决定如何共享前缀以及何时使用缓存“提前完成”。 

从约束条件来看，`n`最大为 2 × 10^5，因此任何在树大小上比线性或近线性更差的解决方案都会失败。 这立即排除了节点之间的任何成对推理或任何在路径上重复重新计算的动态编程。 该解决方案必须在单次遍历中处理 trie，或者相当于线性时间聚合。 

当每个节点独立考虑缓存使用时，就会出现幼稚的危险情况。 例如，如果两个节点共享一个长前缀，但仅在最后一个字符不同，则单独处理它们会重复计算前缀成本并错过共享结构。 另一个微妙的失败案例是假设缓存总是对每个单词有帮助：实际上，付费`b`仅当足够的剩余后缀成本超过时才有意义`b`，这取决于子树结构，而不仅仅是单个节点。 

## 方法

 蛮力策略将模拟独立编写每个所需的单词。 对于每个节点`c[i] > 0`，我们将计算它的字符串长度`len(i)`并承担成本`len(i) * a`，可选择用缓存成本替换最后一段`b`。 这立即忽略了 trie 的共享前缀结构，但即使我们尝试通过显式遍历共享前缀来修复该问题，我们仍然需要推理重叠路径上的缓存使用组合。 

更深层次的问题是缓存决策在整个子树上交互。 如果我们决定在某个节点使用缓存，我们实际上就不再为该事件中的所有更深的字符付费。 因此，问题不是每个节点的优化，而是子树优化，其中每个节点贡献权重，并且决策向上传播。 

关键的观察是，这本质上是 trie 上的树 DP，其中每个节点代表一个字符串，并且单词的每次出现都是独立贡献的。 我们想要计算为子树中的所有请求提供服务的成本，并决定是使用缓存“停止”在节点上还是继续输入更深的内容。 

如果我们将打字视为从根部遍历边缘，那么每次我们下降边缘时，我们都会付出代价`a`每个活跃遍历的每个字符。 然而，由于单词按字典顺序分支，缓存总是选择最小的延续，这意味着当我们决定在节点上使用缓存时，我们实际上承诺通过一次付款来完成其子树中的所有需求`b`而不是继续遍历。 

这导致了一个清晰的 DP 解释：在每个节点，我们要么通过扩展到子节点来支付，要么停下来支付`b`一次针对通过字典顺序一致的遍历聚合的整个子树需求。 

这将问题简化为计算每个节点满足所有节点的最佳成本`c[i]`在其子树中，按顺序组合子结果，同时提出尊重字典遍历顺序的累积成本的概念。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | 指数 / O(字符串总数 × 长度) | O(n) | 太慢了 |
 | 最佳| O(n) | O(n) | 已接受 |

 ## 算法演练

 我们在 DFS 中处理 trie，自下而上计算 DP 值。 

1. 定义`dp[u]`作为满足节点子树中所有需求的最小成本`u`，假设我们开始于`u`已经输入了它的前缀。 

这个公式允许我们将前缀成本与子树决策分开。 
2.对于一个叶子节点，如果有需求`c[u]`，我们必须通过键入所有出现的内容或重复使用缓存来付费。 由于缓存立即结束该单词，因此每次出现都可以独立处理，因此成本为`c[u] * min(depth_cost, b)`在哪里`depth_cost`由父累积隐式处理。 

实际上，除了 DP 聚合之外，叶子不需要特殊处理。 
3. 对于内部节点，我们首先按字典顺序计算所有子节点的 DP。 

排序很重要，因为缓存总是选择与当前前缀一致的字典顺序最小完成，因此子树合并必须遵守此顺序以保持转换的正确性。 
4. 然后我们考虑两种处理子树的方法`u`。 

第一种方式是正常继续：我们结合所有子 DP 值并添加边贡献的遍历成本，有效支付`a`对于所有所需访问的每个边遍历。 

第二种方法是使用缓存`u`。 如果我们在这里使用缓存，那么以 为根的子树中的所有需求`u`可以立即满足成本`b`，忽略进一步的遍历。 

这是有效的，因为一旦我们提交缓存完成`u`，系统直接输出下字典顺序最小的补全`u`，这与 trie 排序约束一致。 
5. 因此，递推式变为：

 dp[u] = min(b, (dp[v] + 根据所有需求需要将边遍历到 v 并返回的成本) 的子项总和)

 微妙的部分是边缘成本`a`与子树被访问的次数成正比。 这相当于自下而上累积加权贡献，其中每个所需的叶子贡献一次遍历其路径上的每个边。 
6. 为了有效地计算这一点，我们执行后序 DFS 并维护子树需求总和。 每条边都有贡献`a * total_demand_in_child_subtree`。 
7. 最后，在每个节点，我们将完整的遍历成本与`b`并取最小值。 

### 为什么它有效

 关键的不变量是对于每个节点`u`，DP 正确地表示了在两个详尽的选择下满足其子树中所有需求的最低成本：要么我们从不使用缓存`u`或者我们只使用一次缓存`u`终止其下面的所有处理。 任何最佳解决方案都可以转化为缓存使用仅发生在替换整个剩余子树的节点上的解决方案，因为子树内的部分缓存始终可以向上移动到最高使用的节点，而不会增加成本。 这消除了重叠的缓存决策，并将问题减少到独立子树选择。 

## Python 解决方案```python
import sys
input = sys.stdin.readline
sys.setrecursionlimit(10**7)

n, a, b = map(int, input().split())

children = [[] for _ in range(n)]
c = [0] * n

for i in range(n):
    parts = list(map(int, input().split()))
    c[i] = parts[0]
    k = parts[1]
    if k:
        children[i] = parts[2:]

def dfs(u):
    total_cost = 0
    total_cnt = c[u]

    for v in children[u]:
        child_cost, child_cnt = dfs(v)
        total_cost += child_cost
        total_cnt += child_cnt

    # cost contributed by edges is proportional to number of words
    total_cost += total_cnt * a

    # option: use cache at this node
    total_cost = min(total_cost, b)

    return total_cost, total_cnt

ans, _ = dfs(0)
print(ans)
```DFS 返回子树中的成本和终止字的数量。 需要计数，因为每个单词都会影响所有祖先边缘的遍历成本。 乘法`total_cnt * a`捕获这样一个事实：每个所需的单词都需要在此节点级别进行一个字符遍历。 

缓存决策在聚合子节点后应用，因为节点处的缓存会覆盖所有子树结构。 

一个微妙的点是，我们将所有子树需求视为独立单元，对边缘成本的贡献相等。 这是正确的，因为每个所需的单词每次出现都需要通过其路径上的每个边缘一次，除非被缓存替换。

 ## 边缘情况

 一个关键的边缘情况是当`b = 0`。 在这种情况下，缓存应始终在覆盖所有需求的最高可能节点上使用，从而将整个答案压缩为零。 DP 正确地处理了这个问题，因为每个节点都将其累积成本与零进行比较并立即选择缓存。 

另一个边缘情况是当`a = 0`。 这里输入是免费的，因此缓存永远没有用处，除非它也为零。 该算法正确地累积了零边缘成本，使得`b`无关紧要，除非不可能进行负面比较。 

当需求分布在严重重叠的深层分支时，就会出现结构性边缘情况。 例如，如果需要子树下的所有叶子，则子树计数会变大，并且乘法`total_cnt * a`除非选择缓存，否则占主导地位。 DP 确保在子树累积变得比子树累积更昂贵的点处准确地考虑缓存`b`，防止过度计数。
