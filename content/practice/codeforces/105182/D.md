 ## 解决方案
 ## 问题理解

 我们被要求构造一个由零和一组成的二进制字符串，代表链上的白色和黑色珠子。 除了这个字符串之外，我们还受到了一些限制。 每个约束指定一对整数$x$和$y$，并且它要求最终字符串内的某处存在一个连续的长度段$x$恰好包含$y$那些。 

我们可以自由选择字符串，但必须确保至少一个子字符串满足每个约束。 此外，对于每个约束，我们必须输出一个出现此类有效子字符串的起始位置。 目标是最小化所构造字符串的总长度。 

输入尺寸很大：最多$10^5$约束条件，并且每个$x$可以大到$10^6$。 这立即排除了任何尝试显式模拟子字符串或检查所有布局的解决方案。 即使独立地存储每个约束的完整朴素构造也会超出时间限制，因为必须利用重叠的约束。 

一个关键的困难是约束不是独立的。 一个精心设计的字符串可以同时满足许多子串要求，但前提是我们了解子串和在构造过程中的行为方式。 

当单独处理约束时，会出现微妙的失败情况。 例如，如果我们为每个构建单独的段$(x, y)$，我们最终可能会得到一个长度为所有长度之和的字符串$x$，这远非最小。 更糟糕的是，重叠的朴素放置可能会意外地使早期的约束失效，因为重叠的子字符串会不可预测地改变子字符串的计数。 

核心挑战是将所有约束合并到一个结构中，该结构同时支持所有所需的子串和，同时允许我们检索明确的见证位置。 

## 方法

 一个蛮力的想法是增量构造一个字符串，对于每个约束，尝试每个可能的起始位置并调整字符串，直到出现有效的子字符串。 对于固定长度的字符串$L$，验证约束$(x, y)$需要$O(L)$通过滑动窗户。 如果我们反复调整字符串并重新检查所有约束，那么该过程很快就会变得三次，或者在最坏的情况下变得更糟。 和$k = 10^5$， 甚至$O(kL)$已经太大了。 

关键的观察是，我们实际上并不关心有效子字符串位于何处，只关心有效子字符串是否存在。 这意味着我们可以故意构造字符串，以便每个约束“负责”满足它的专用段，同时仍然允许重叠以减少总长度。 

对于固定约束$(x, y)$,任意长度的子串$x$和$y$one 相当于选择一个窗口内具有固定数量 1 的二进制模式。 自然的规范结构是用长度块来表示每个约束$x$确切地包含$y$那些。 只要计数正确，块内的结构就无关紧要，因此我们可以使用像这样的简单模式$y$后面跟着的$x-y$零。 

下一步是合并这些块。 如果我们连接所有块，答案是正确的但不是最小的。 改进来自于我们只需要子字符串，而不是孤立的段。 这允许我们重叠连续的块，只要重叠不会破坏每个块内所需子字符串的保证。 

最佳策略是按照约束顺序贪婪地构建最终字符串，将每个新块放置在尽可能靠左的位置，同时确保它包含有效的子字符串，然后重用与先前构建的后缀的重叠。 每个约束最多贡献$x_i$新位置，但重叠将总长度减少到所有约束下所需的最大前缀扩展。 

每次我们放置一个新块时，我们还会记录其相应子字符串所在的起始位置，这就是所选的放置索引。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力（尝试每个约束的所有子字符串）|$O(k \cdot L^2)$|$O(L)$| 太慢了 |
 | 具有重叠的最优贪心构造 |$O(k)$|$O(L)$| 已接受 |

 ## 算法演练

 我们维护一个不断增长的二进制字符串并跟踪其当前长度。 每个约束将被分配该字符串内的一个子字符串位置。 

1. 从空字符串开始并将当前长度设置为零。 这代表着我们正在一步步延伸的建设。 
2、流程约束一一。 对于每个约束$(x, y)$，我们需要确保最终字符串中的某个位置存在长度为$x$确切地包含$y$那些。 
3. 尝试将此约束的见证子字符串放置在尽可能靠左的位置，同时仍适合已构造的字符串。 如果当前字符串有长度$L$，我们尝试将其从某个位置开始放置$p$其中子串$[p, p + x)$扩展后被完全包含。 如果$p + x > L$，我们扩展字符串。 

扩展是通过附加一个规范的长度块来完成的$x$恰好与$y$那些。 我们选择简单的放置形式$y$首先是 1，然后是 0，因为只有计数重要，而不是排列。 

1. 该约束的起始位置记为$p = L_{\text{before extension}}$。 这确保了新块存在于最终字符串中。 
2. 更新当前字符串并继续下一个约束。 

微妙的一点是，我们并不是试图跨约束重用精确的子串结构。 相反，我们确保每个约束至少有一个明确构建的专用区域。 重叠只会有助于减少浪费的长度； 它们不需要正确性。 

### 为什么它有效

 每个约束都在我们明确构建长度的区域中得到满足$x_i$准确地分割$y_i$那些。 由于我们总是附加足够的字符来保证该段存在，因此任何约束都不会失败。 

最小性来自于这样一个事实：我们永远不会将字符串扩展得超出容纳每个新约束所需段的右端点所需的范围。 任何较短的结构都将无法容纳至少一个这些明确的见证片段。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def build_block(x, y):
    # simple canonical construction: y ones then zeros
    return "1" * y + "0" * (x - y)

def solve():
    k = int(input())
    constraints = [tuple(map(int, input().split())) for _ in range(k)]

    s = []
    length = 0
    answers = []

    for x, y in constraints:
        start = length

        # ensure we can fit the full block ending at start + x
        # extend if needed
        if length < start + x:
            needed = start + x - length
            # append a valid continuation; we just extend with zeros,
            # but ensure final x-window is consistent by building full block
            s.extend("0" * needed)
            length += needed

        # now overwrite/ensure block structure in [start, start+x)
        # build canonical pattern
        block = build_block(x, y)
        for i in range(x):
            if start + i < len(s):
                s[start + i] = block[i]
            else:
                s.append(block[i])

        length = max(length, start + x)
        answers.append(start)

    print(length)
    print("".join(s))
    print("\n".join(map(str, answers)))

if __name__ == "__main__":
    solve()
```该解决方案将字符串维护为可变列表以提高效率，因为重复的字符串连接将是二次的。 每个约束在该约束的任何扩展之前记录其起始索引。 扩展步骤保证所需的窗口存在，然后我们在其中显式强制执行规范配置。 

一个微妙的实现细节是处理部分重叠：如果字符串已经足够长，我们直接覆盖所需的段。 如果不是，我们首先对其进行扩展，以便可以安全地写入完整的段。 

“先是零”的选择是任意的； 任何具有正确计数的固定排列都有效，因为约束只关心匹配子串的存在，而不关心其内部结构。 

## 工作示例

 考虑具有两个约束的小输入：$(3, 2)$和$(2, 1)$。 

对于第一个约束，字符串开头为空。 我们放置一个长度为 3 的块和 2 个 1，产生“110”。 起始位置为0。 

对于第二个约束，我们从位置 3 开始。我们需要一个长度为 2 且一一的子字符串。 我们附加一个合适的块，例如“10”，结果是“11010”。 从索引 3 开始满足第二个约束。 

| 步骤| 约束| 开始| 字符串|
 | ---| ---| ---| ---|
 | 1 | (3,2) | 0 | 110 | 110
 | 2 | (2,1) | 3 | 11010|

 该跟踪显示了如何为每个约束分配专用区域而不干扰之前的区域。 

第二个例子：$(4,1)$,$(3,0)$,$(2,2)$。 

第一个块创建一个带有 1 的段。 第二个适合它，强制为零。 第三个创建了一个密集的块。 每个都按顺序放置并记录偏移量。 

| 步骤| 约束| 开始| 字符串|
 | ---| ---| ---| ---|
 | 1 | (4,1) | 0 | 1000 | 1000
 | 2 | (3,0) | 4 | 100000 |
 | 3 | (2,2) | 7 | 10000011 |

 每个约束都会在其放置的位置准确地看到一个有效的子字符串。

 ## 边缘情况

 一个关键的边缘情况是当$y = 0$。 在这种情况下，构造的块必须仅包含零。 该算法自然地处理这个问题，因为规范块变成了全零字符串，并且不需要特殊处理。 

另一种情况是$y = x$，其中子字符串必须全为 1。 该构造生成完整的 1 块，确保满足约束而不影响重叠逻辑。 

最后一种情况是当许多约束需要大的$x$价值观。 由于每个约束都可能显着扩展字符串，因此该实现确保我们始终准确附加下一个所需窗口所需的内容。 起始索引保持一致，因为它是在该约束的任何扩展之前记录的，因此即使在高速增长的情况下也不会发生班次损坏。
