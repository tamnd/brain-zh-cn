 ## 解决方案
 ## 问题理解

 我们得到了一系列随时间采样的下载速度和固定的窗口大小$k$。 对于每个连续的长度段$k$，我们需要计算该路段的平均速度。 一旦计算出平均值，就会根据其大小以不同的单位显示：如果平均值至少为 1024，则以 MiBps 为单位显示，否则以 KiBps 为单位显示。 

从概念上讲，该任务是数组上的滑动窗口聚合问题，然后是简单的基于阈值的单位转换。 每个窗口恰好产生一个输出值，并且输出必须为每个窗口独立地保留单位规则。 

如果$n$很大，例如高达$10^5$，每个窗口总和的简单重新计算需要$O(k)$每个职位的工作量，导致$O(nk)$总操作。 在最坏的情况下$k$也很大，这变得太慢了，因为它会接近$10^{10}$运营。 

更微妙的一点是，输出取决于平均值，而不是总和。 这意味着每个窗口都需要除法，如果过早应用舍入，粗心的整数除法可能会导致不正确的单元分类。 正确的方法必须确保在除法之前准确计算每个窗口的总和。 

当值徘徊在阈值边界附近时，就会出现典型的边缘情况。 例如，如果$k = 3$一个窗口的总和正好是$3071$，平均值为$1023.666...$，应归类为 KiBps，而不是 MiBps。 任何将原始总和与 1024 进行比较而不是比较平均值的实现都会错误地对此类情况进行分类。 

## 方法

 暴力解决方案很简单：对于每个起始索引$i$，我们扫描下一个$k$元素，计算它们的总和，除以$k$，然后应用单位转换规则。 这是正确的，因为它直接遵循问题的定义，独立处理每个窗口。 

这种方法的问题是重复工作。 每个新窗口几乎与前一个窗口完全重叠，但强力方法从头开始重新计算全部总和。 为了$n$窗户尺寸$k$，这大约导致$n \cdot k$补充。 当两者$n$和$k$很大，这太慢了。 

关键的观察结果是，连续的窗口恰好有一个元素不同：前一个窗口的最左侧元素被删除，并添加右侧的新元素。 这使我们能够保持连续的金额。 我们不是重新计算每个窗口总和，而是通过减去传出值并加上传入值来在恒定时间内更新它。 这减少了总工作量$O(nk)$到$O(n)$。 

有效地维护总和后，在恒定时间内计算每个窗口的平均值，并立即应用单位转换规则。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 |$O(nk)$|$O(1)$| 太慢了 |
 | 滑动窗|$O(n)$|$O(1)$| 已接受 |

 ## 算法演练

 1. 计算第一个的总和$k$元素。 这形成了初始窗口并为所有后续更新建立了基线值。 
2. 将总和除以计算该初始窗口的平均值$k$。 根据该平均值是否至少为 1024，以 MiBps 或 KiBps 为单位输出。 
3. 从索引迭代$k$到$n-1$，将每个位置视为新窗口的右端点。 
4. 对于每个新位置，通过减去离开窗口的元素并添加进入的新元素来更新运行总和。 这使总和与当前窗口保持一致，而无需从头开始重新计算。 
5. 更新总和后，除以计算新平均值$k$。 
6. 根据计算出的平均值再次应用单位转换规则并输出结果。 

核心不变量是，在每次迭代中，运行总和恰好等于最后一次迭代的总和$k$以当前索引结尾的元素。 由于每次更新都会精确删除一个旧元素并添加一个新元素，因此总和始终与当前窗口匹配。 这保证了每个计算的平均值都是正确的，因此基于它的每个单元分类也是正确的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    data = list(map(int, input().split()))
    n, k = data[0], data[1]
    arr = data[2:]

    window_sum = sum(arr[:k])
    out = []

    def format_avg(avg):
        if avg >= 1024:
            return str(avg) + " MiBps"
        else:
            return str(avg) + " KiBps"

    avg = window_sum // k
    # if exact average is required as float, adjust here; problem assumes integer style display
    out.append(format_avg(avg))

    for i in range(k, n):
        window_sum += arr[i]
        window_sum -= arr[i - k]
        avg = window_sum // k
        out.append(format_avg(avg))

    sys.stdout.write("\n".join(out))

if __name__ == "__main__":
    solve()
```该解决方案一次性读取数组，并使用第一个窗口的直接前缀来初始化第一个窗口总和。$k$元素。 滑动更新步骤是通过添加传入元素并删除传出元素来实现的，从而保留恒定时间更新。 

一个微妙的实现问题是整数除法。 如果原始问题需要实值平均值，那么使用楼层划分将是不正确的。 在这种情况下，在与 1024 进行比较之前，应该使用浮点运算将和除。但是，滑动窗口的结构保持不变。 

输出格式化函数封装了阈值逻辑，以便统一处理每个窗口结果。

 ## 边缘情况

 当窗口平均值极其接近 1024 阈值时，就会出现关键的边缘情况。 考虑窗口总和为 2047$k = 2$，平均值为 1023.5。 正确的分类是 KiBps，即使涉及的所有值可能单独超过 1024。该算法可以正确处理此问题，因为分类仅取决于完全求和后计算的窗口平均值。 

另一个边缘情况是当$k = 1$。 在这种情况下，每个窗口都是一个元素，滑动更新退化为直接输出每个值。 不变量仍然成立，因为“删除旧元素，添加新元素”步骤变得微不足道，并且每个总和完全等于元素本身。 

最后的边缘情况是所有值都为零时。 在整个执行过程中，每个窗口总和保持为零，并且输出始终保持在 KiBps 单位。 滑动窗口仍然执行更新，但每次更新都会精确取消，无需特殊的大小写即可保留正确性。
