 ## 解决方案
 ## 问题理解

 有五个固定角色，必须由恰好五个不同的玩家来担任。 每个玩家都有一个编码为长度为 5 的二进制字符串的约束，其中`1`表明玩家有能力扮演相应的角色。 有效的阵容是五个角色和五个不同球员之间的一对一分配，这样每个分配的球员都有资格胜任自己的角色。 

任务是计算存在多少个有效分配，接管池中五名不同球员的所有可能选择，以及将他们分配到五个位置的所有有效方法。 如果至少一个角色由不同的球员担任，则两个阵容被视为不同。 

输入规模高达二十万名玩家。 每个玩家贡献一个 5 位能力掩码。 一个天真的解释立即建议选择任意 5 个玩家并检查所有排列，这已经表明组合爆炸远远超出了枚举所能处理的范围。 当 n 达到 2×10^5 时，任何接近 O(n^5) 甚至 O(n^3 · 5!) 的操作在一秒限制下都是不可行的。 即使 O(n^2) 也是临界值，但只有非常小的常数才可能可接受，因此解决方案必须避免任意玩家子集之间的成对交互。 

当多个玩家共享相同的能力模式时，幼稚推理的微妙失败案例就会出现。 例如，如果十名玩家都可以扮演每个角色，那么有效阵容的数量不仅仅是“选择 5 名玩家”，而是“选择 5 个有序分配”，并且忽略角色分配排列的朴素计数将根据对称性的处理方式而少算或多算。 另一个陷阱是将玩家视为仅通过面具即可互换的玩家，这会错误地合并不同的个体，即使他们贡献了不同的选择。 

## 方法

 直接暴力方法试图枚举五个玩家的每个子集，然后在检查有效性的同时将它们排列在五个角色中。 对于每个选定的集合，最多有 5 个！ 作业，并且检查有效性是每个作业的固定时间。 单独子集的数量是$\binom{n}{5}$，其增长量级为 n^5 / 120，当 n 为 2×10^5 时，已经超过 10^25。 即使我们以某种方式将自己限制在小子集上，组合结构仍然太大而无法明确探索。 

关键的观察结果是，问题的大小实际上是由角色数量控制的，而不是玩家数量。 只有五个职位，因此任何描述部分分配的状态仅取决于哪些角色已被填补。 这提出了一种动态编程公式，我们在其中一个一个地处理玩家，并维护我们可以实现每个填充角色子集的方式。 

每个玩家要么被忽略，要么被用来填补他们能够扮演的仍然空缺的角色之一。 由于只有 5 个角色，因此状态空间只有 2^5 = 32，这使得有效跟踪所有配置成为可能。 处理每个玩家都会更新这 32 个状态，从而产生与 n 成线性关系的总复杂度。 

暴力方法在概念上是有效的，因为它明确地尊重所有约束，但它会失败，因为它重复地重新计算等效的部分赋值。 只有已填充角色的集合才重要的观察结果将对 n 的指数依赖性折叠成角色子集上的恒定大小状态机。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(n^5 · 5!) | O(n^5 · 5!) | O(1) | O(1) | 太慢了 |
 | 玩家的位掩码 DP | O(n·2^5·5) | O(2^5) | O(2^5) | 已接受 |

 ## 算法演练

 我们将这个过程建模为逐渐决定使用哪些球员以及他们如何为填补这五个角色做出贡献。 

1.定义一个DP数组，其中dp[mask]代表bitmask对应的分配角色的方式数`mask`使用玩家的前缀。 掩码中的每一位对应一个已填充的角色。 这紧凑地捕获了所有部分有效的分配。 
2. 初始化 dp[0] = 1，因为只有一种方法可以不向任何角色分配任何内容。 
3. 按输入顺序遍历每个玩家。 对于每个玩家，构造一个新的DP数组ndp，初始等于dp，代表跳过该玩家的选择。 
4. 对于每个当前状态掩码，尝试将当前玩家分配给掩码中仍未填充且玩家能力字符串具有`1`。 对于每个这样的角色，通过添加 dp[mask] 来更新 ndp[mask ∪ {j}]。 这表示使用该玩家来扩展部分任务。 
5.处理完播放器后将dp替换为ndp。 这确保了每个玩家最多被使用一次并且只被考虑一次。 
6. 处理完所有玩家后，答案为 dp[(1<<5) - 1]，代表五个角色全部填满。 

正确性取决于以下事实：每个有效分配在构建它的玩家的处理顺序中都有唯一的“最后出现”。 由于我们只在玩家序列中向前移动，并且从不重复使用玩家，因此每个分配都只计算一次。 

保持的不变性是，在处理前 i 个玩家之后，dp[mask] 等于选择这些 i 玩家的子集并将它们准确分配给 mask 中的角色的方法数量，同时尊重兼容性。 每个转换要么保留子集（跳过），要么通过将当前玩家分配给新角色来扩展它，而不会违反唯一性或兼容性约束。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MOD = 10**9 + 7

def main():
    n = int(input())
    can = [input().strip() for _ in range(n)]

    dp = [0] * 32
    dp[0] = 1

    for i in range(n):
        ndp = dp[:]  # skipping player i
        mask_allowed = 0

        for j in range(5):
            if can[i][j] == '1':
                mask_allowed |= (1 << j)

        for mask in range(32):
            if dp[mask] == 0:
                continue
            base = dp[mask]
            # try assign this player to each available role
            avail = mask_allowed & (~mask)
            sub = avail
            while sub:
                bit = sub & -sub
                ndp[mask | bit] = (ndp[mask | bit] + base) % MOD
                sub -= bit

        dp = ndp

    print(dp[31] % MOD)

if __name__ == "__main__":
    main()
```该实现保留了一个 32 元素的 DP 数组，对应于五个角色的所有子集。 对于每个玩家，我们计算他们可以扮演的角色的位掩码。 这个转换小心地确保我们只将玩家分配给当前掩码中尚未使用的角色。 

可用角色的子掩码上的内部循环是安全的，因为掩码大小固定为 5，因此在实践中迭代子集的时间是恒定的。 在转换期间应用模数以防止溢出。 

一个微妙的点是副本的使用`ndp = dp[:]`。 这对于避免允许单个玩家在同一迭代中多次贡献至关重要。 如果没有这种分离，更新将在同一玩家的处理中错误地链接。

 ## 边缘情况

 一个关键的边缘情况是没有球员可以担任特定角色。 例如，如果每个字符串都有一个`0`在第三个位置，则尝试包含该角色的每个 DP 状态仍然无法到达，并且最终答案正确地保持为零，因为 dp[11111] 从未形成。 

当许多玩家共享相同的能力掩码时，就会出现另一种情况。 该算法不会折叠它们； 每个都按顺序独立处理。 这确保了选择相同玩家的不同子集仍然是单独计算的，因为即使转换相同，dp 也会通过不同的处理步骤演变。 

最后一个微妙的情况是一个玩家可以担任多个角色。 子掩码迭代确保同一玩家在每个 DP 状态中对所有有效角色选择做出贡献一次，而无需在单次迭代中混合分配。
