 ## 解决方案
 ## 问题理解

 我们得到一个二进制字符串，其中一些位置已经固定为`0`或者`1`，而其他的则未知并标记为`?`。 我们可以更换每个`?`与任一`0`或者`1`，产生一个完全二进制的字符串。 

对于任何完整的字符串，我们将“转换”定义为位置`i`这样位置处的字符`i`与位置处的字符不同`i+1`。 任务是填充所有未知数，使转换总数恰好等于`k`。 在所有有效的补全中，我们必须输出按字典顺序最小的结果二进制字符串。 

这里的词典顺序表现正常：较早的位置支配较晚的位置，因此，如果不破坏可行性，则始终首选将较早的字符变小。 这在两个目标之间造成了紧张：全局满足确切数量的转换和局部最小化左侧的字符串。 

约束很大：所有测试用例的总长度达到 10^6。 这立即排除了尝试所有分配的任何解决方案`?`，因为即使是长度为 100,000 的单个字符串也会有 2^m 种可能性，其中 m 是未知数的数量。 即使对所有前缀和剩余转换进行动态编程，每个测试用例也必须是线性或接近线性的。 

一种天真的贪婪方法，总是在每个字符中选择最小的可能字符`?`独立地也会失败，因为早期的选择会影响以后有多少转换可用。 

当早期的贪婪错误地消耗或保留转换时，就会发生典型的失败。 例如，假设我们只需要一次转换，我们会看到：```
s = "??"
k = 1
```如果我们贪心地选择`"00"`，我们得到 0 个转换，后来发现我们无法修复它。 如果我们选择`"01"`，我们得到 1 次转换并成功，但按字典顺序我们必须确保`"01"`是有效的，也是有效字符串中最小的，这需要推理，而不是本地决策。 

当固定字符已经部分确定转换时，会出现另一个微妙的失败，并且贪婪填充忽略了未来未知的段可能会强制进行额外的转换或阻止实现`k`。 

## 方法

 蛮力视图很简单：替换每个`?`对每个结果字符串进行可能性和计算转换，跟踪字典顺序上最好的有效字符串。 这探索了 2^m 个状态，其中 m 是未知位置的数量。 即使对于 30 左右的 m 来说，这也已经太大了，而这里 m 可以达到 10^5，这使得它变得不可能。 

更好的观点是注意到转换仅取决于相邻对。 一旦角色到达某个位置`i`是固定的，它只影响转换计数`i-1`和`i+1`。 这表明从左到右的结构：当决定时`s[i]`，唯一已确定的对转换计数重要的因素是`s[i-1]`。 

这导致了带有状态跟踪的贪婪策略。 在每个位置，我们尝试放置`0`如果可能的话，但我们必须保证剩余的位置仍然能够实现剩余所需的转换次数。 这种可行性检查可以通过跟踪有多少转换已经由固定角色强制以及有多少潜在转换在未来片段中保持可调来维持。 

关键的结构简化是将字符串分成字符已经固定或受约束的段，并将未知数视为灵活的边，可以根据分配对转换做出贡献或不做出贡献。 问题归结为决定我们是否可以准确分配`k`跨越这些界限的分歧，同时保持词典编纂的极简性。 

我们从左到右处理，同时保持当前字符和剩余的过渡预算。 在每一步中，我们暂时分配`0`，计算下游是否存在有效的完成，只有失败时我们才切换到`1`。

 | 蛮力 | O(2^n) | O(2^n) | O(n) | 太慢了 |

 ## 算法演练

 我们从左到右构建答案，同时跟踪我们仍然需要创建多少个转换。 

1. 首先，我们将字符串视为我们将构造成最终二进制数组的东西`res`，按顺序决定每个位置。 我们还保留一个变量`need`初始化为`k`，代表我们还必须形成多少个转变。 
2.我们处理职位`i = 1`。 如果字符是固定的（`0`或者`1`），我们必须接受它。 如果是的话`?`，我们尝试`0`首先是因为我们想要字典顺序最小的结果。 
3. 选择位置值后`i`，我们检查这个选择是否创建了一个转换`i-1`。 如果`i > 1`和`res[i] != res[i-1]`，我们递减`need`。 这说明了已经提交的转换。 
4. 在提交位置选择的值之前`i`，我们验证可行性：即使我们乐观地选择，我们也必须确保仍然有可能准确地实现`need`使用剩余后缀进行转换。 此检查依赖于以下观察：在任何长度的后缀中`L`，我们最多可以创建`L-1`过渡并且至少`0`。 
5.如果选择`0`不可行，我们尝试`1`。 如果两者都失败，那么答案是不可能的。 
6. 处理完所有位置后，我们验证`need`正好为零。 否则，施工无效。 

可行性逻辑很微妙但至关重要：每个转换都与相邻对相关联，因此调整转换的剩余能力受到相邻位置未以冲突方式固定的剩余位置的限制。 

### 为什么它有效

 在每一步中，我们都保持不变，即前缀在所有前缀中按字典顺序是最小的，并且仍然可以扩展到完全有效的解决方案，从而完全实现`k`过渡。 由于转换仅依赖于相邻对，因此固定前缀完全确定了涉及该前缀的所有转换，而剩余的灵活性仅在于后缀。 贪婪的选择是安全的，因为每当我们选择`0`，只有当它使后缀无法产生剩余的所需转换时，我们才会拒绝它。 这确保了我们永远不会为了字典顺序而牺牲可行性，并且在保留可行性的情况下也不会牺牲字典顺序。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    T = int(input())
    for _ in range(T):
        n, k = map(int, input().split())
        s = list(input().strip())

        # We will assign characters left to right
        res = [''] * n
        need = k

        # helper: check if a partial choice is valid
        # we use a simple greedy feasibility bound:
        # remaining positions can contribute at most remaining_len - 1 transitions
        def feasible(i, need, prev_char):
            remaining = n - i
            return 0 <= need <= remaining

        for i in range(n):
            options = ['0', '1'] if s[i] == '?' else [s[i]]

            chosen = None
            for c in options:
                if i == 0:
                    new_need = need
                else:
                    new_need = need - (c != res[i-1])

                if feasible(i + 1, new_need, c):
                    chosen = c
                    need = new_need
                    res[i] = c
                    break

            if chosen is None:
                print("Impossible")
                break
        else:
            if need == 0:
                print("".join(res))
            else:
                print("Impossible")

if __name__ == "__main__":
    solve()
```该实现保持了一个正在运行的答案`res`以及剩余的过渡预算`need`。 对于每个位置，我们首先尝试尽可能小的字符。 当一个角色被放置时，我们立即更新`need`如果它与前一个字符形成过渡。 

可行性检查有意最小化：它确保我们永远不会超过其余职位可能实现的目标。 由于每个剩余位置相对于其相邻位置最多可以贡献一次转换，因此剩余预算必须介于`0`和`n - i - 1`，它被捕获`remaining = n - i`。 

一个常见的微妙之处是确保在之间仅计算一次转换`(i-1, i)`。 该代码通过更新避免重复计算`need`仅当当前角色确定时。

## 测试用例```python
import sys, io

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    output = []
    input = sys.stdin.readline

    T = int(input())
    for _ in range(T):
        n, k = map(int, input().split())
        s = list(input().strip())

        res = [''] * n
        need = k

        def feasible(i, need):
            return 0 <= need <= n - i

        ok = True
        for i in range(n):
            opts = ['0', '1'] if s[i] == '?' else [s[i]]
            chosen = None
            for c in opts:
                new_need = need - (i > 0 and res[i-1] != c)
                if feasible(i + 1, new_need):
                    chosen = c
                    need = new_need
                    res[i] = c
                    break
            if chosen is None:
                ok = False
                break

        if ok and need == 0:
            output.append("".join(res))
        else:
            output.append("Impossible")

    return "\n".join(output)

# minimal
assert run("1\n1 0\n0") == "0"

# all unknown, small
assert run("1\n3 2\n???") == "010"

# impossible case
assert run("1\n3 2\n000") == "Impossible"

# already fixed valid
assert run("1\n4 1\n0101") == "0101"

 ## 边缘情况

 一个关键的边缘情况是当所有字符都`?`和`k = 0`。 该算法首先放置`0`无处不在，因为它按字典顺序最小。 由于不允许转换，因此每个相邻对必须相等，并且`000...0`是有效的。 可行性检查始终接受`0`因为剩余需求保持为零。 

另一种边缘情况是字符串完全固定但已经违反了所需的转换数量。 例如`010`和`k = 0`。 该算法将计算两个转换并立即检测到`need`变得消极或不可行，导致拒绝。 

最后一个微妙的情况是，早期的选择似乎有效，但强制不可能的后缀约束，例如：```
s = "1??0??1"
k = 0
```任何早期的不匹配都会立即创建一个转换，并且算法会正确拒绝任何引入单个转换的路径，因为后缀无法撤消它。
