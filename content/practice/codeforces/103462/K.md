 ## 解决方案
 ## 问题理解

 我们得到一个整数数组和一个特殊值$k$。 该过程是动态的：只要数组仍然包含至少一个等于$k$，我们重复应用一个操作，将数组的每个正元素减一。 零保持不变，并且值永远不会低于零。 

对于任何固定长度的连续子数组$m$，我们想象运行这个全局进程直到没有$k$保留在整个数组中，然后在过程稳定后计算该子数组中出现了多少个零。 最终答案是所有长度上这些零计数的总和 -$m$子数组。 

关键点是递减操作是全局且重复的，因此每个元素每“轮”恰好减少一次，而任何元素$k$仍然存在于数组中的某个位置。 

这些约束促使我们走向线性或接近线性的行为。 和$n$最多$10^6$，任何显式模拟每个递减轮次的解决方案都是立即不可能的。 甚至$O(n^2)$通过滑动窗口会太慢，因为有$O(n)$窗户和每个尺寸$O(n)$。 因此，目标是对数组进行单次传递或少量传递$O(n)$或者$O(n \log n)$复杂。 

一个微妙的问题是理解停止条件。 全局递减的数量不是预先固定的； 这取决于所有发生的情况所需的最大轮数$k$消失。 任何试图模拟直到没有的幼稚方法$k$存在会过度递减或重复扫描数组，这两种方法都太昂贵。 

边缘情况出现在以下情况：

 最小的情况，例如$n = m = 1$可能会混淆对全球进程的解释。 

如果数组不包含$k$无论如何，该操作永远不会触发，因此数组保持不变并且不会创建零。 任何假设至少一轮的解决方案都会错误地减少值。 

如果所有元素都等于$k$，每一轮都会均匀地减少所有正值，因此数组的行为就像一个单一的递减时间线，直到所有值都达到零。 轮数正好变成$k$，所有元素最后都变为零，这强烈影响每个子数组的贡献。 

## 方法

 蛮力的想法是直接为每个子数组模拟所描述的过程。 对于每个长度的窗口$m$，我们将复制子数组，重复减少所有正元素，同时$k$存在，最后计数为零。 这在逻辑上是正确的，但其代价是巨大的。 

每个子数组可能需要$O(\max a_i)$递减轮数，并且每一轮都会触及$O(m)$元素。 高达$10^6$子数组，这会退化为以下顺序$10^{12}$在最坏的情况下操作，这是不可能的。 

关键的观察结果是该操作是全局且统一的：每个元素都减少完全相同的步数，直到过程停止。 唯一重要的是在所有出现之前发生了多少轮$k$被淘汰。 令该数字为$T$。 然后每个元素$a_i$变成$\max(0, a_i - T)$。 

所以整个问题归结为寻找$T$，然后对于每个窗口计算有多少个元素正好$T$，因为这些元素恰好为零。 

这将问题从重复模拟转换为单个导出阈值问题，然后进行滑动窗口频率计算。

 | 最佳 |$O(n)$|$O(n)$| 已接受 |

 ## 算法演练

 ### 关键思想：将动态降低到单一阈值

 我们首先确定在所有之前发生了多少次全局减量操作$k$价值观消失。 每个操作将所有正值减一，因此元素等于$k$准确地生存下来$T = k$变为零之前的轮数。 如果至少有一个$k$，该过程恰好持续$k$步骤； 如果没有$k$，它持续零步。 

因此每个元素都按照规则进行转换$a_i \to \max(0, a_i - T)$， 在哪里$T = k$如果$k$存在，否则$T = 0$。 

变换后窗口中零的个数正好是满足条件的元素个数$a_i \le T$。 然而，由于只有完全为零的元素才重要，因此我们计算有多少原始值满足$a_i \le T$，只有当$a_i \le k$特别是变为零。 

### 步骤

 1. 扫描一次数组，检查值是否存在$k$出现。 如果没有，请设置$T = 0$; 否则设置$T = k$。 这决定了全局减量轮次的总数。 
2. 将问题转化为计数问题，对于每个长度的子数组$m$，有多少个元素满足$a_i \le T$。 这些正是进程结束后将变为零的元素。 
3.构建二进制数组$b$在哪里$b_i = 1$如果$a_i \le T$， 否则$0$。 这对每个位置在最终状态中是否贡献零进行编码。 
4. 计算所有长度的总和 -$m$的总和的窗口$b$价值观。 这可以使用滑动窗口来完成：维护当前窗口总和，添加传入元素，删除传出元素，然后累加结果。 
5. 最终的答案是总的累积窗口总和。 

### 为什么它有效

 该过程对每个元素应用相同数量的递减操作，因此相对顺序并不重要。 每个元素在其初始值至多时独立地变为零$T$。 因此，该变换将问题简化为静态阈值滤波器，然后是标准滑动窗口聚合。 不变的是之后$T$步骤中，每个元素的值仅取决于其原始值，而不取决于位置或交互作用，因此窗口贡献变得可加且独立。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m, k = map(int, input().split())
    a = list(map(int, input().split()))
    
    has_k = any(x == k for x in a)
    T = k if has_k else 0
    
    b = [1 if x <= T else 0 for x in a]
    
    window_sum = sum(b[:m])
    ans = window_sum
    
    for i in range(m, n):
        window_sum += b[i] - b[i - m]
        ans += window_sum
    
    print(ans)

if __name__ == "__main__":
    solve()
```解决方案首先确定特殊值是否$k$存在。 这控制是否发生任何递减轮次。 然后该数组被转换为布尔指示符数组，标记最终将变为零的元素。 

滑动窗口维护每个长度的子数组中此类元素的数量$m$。 Each update adjusts the window in constant time, and the running total accumulates contributions from all windows.

 A common mistake is trying to simulate the decrement process explicitly. Another is recomputing each window sum from scratch, which would be$O(nm)$并大规模失败。

 ## 边缘情况

 当数组中不包含$k$，转换步骤永远不会被触发。 该算法通过设置来处理这个问题$T = 0$，意味着没有元素满足$a_i \le T$除非它已经为零，这在约束下永远不会发生。 因此，所有子数组的滑动窗口总和为零，这与没有任何变化的事实相匹配。 

当所有元素都相等时$k$，每个位置都满足阈值条件。 二进制数组变成全1，每个窗口和等于$m$。 滑动窗口精确累加$n - m + 1$的贡献$m$，匹配过程完成后的统一全零行为。 

对于单元素数组，窗口总和为 1 或 0，具体取决于是否$k$匹配该元素。 滑动窗口无需特殊处理即可正确退化为单个值。
