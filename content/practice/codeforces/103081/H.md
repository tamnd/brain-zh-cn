 ## 解决方案
 ## 问题理解

 我们有一个系统$N$雕像标记为$0$到$N-1$。 超过$N$天，每个小雕像被插入到架子上一次，然后又被移除一次，因此每个小雕像都定义了一个连续的活动间隔$[l_j, r_j)$。 在任何一天$i$，架子上恰好包含了间隔涵盖当天的小雕像，形成了一组$S(i)$。 

与这个不断发展的集合并行，我们模拟了一些$x$开始于$x_0 = 0$。 当天$i$，我们查看当前值$x_i$，我们通过计算当前货架上至少有多少个雕像有标签来增加它$x_i$。 更新后的值变为$x_{i+1}$，但取模$N$。 该过程适用于所有$N$天，我们输出$x_N$。 

关键难点在于门槛$x_i$随着时间的推移而变化，并且取决于之前的计数。 这使得计数变得动态，因为每天都会对同一不断变化的集合提出不同的查询：“至少有多少个活跃元素$x_i$？”。

 限制条件$N \le 100{,}000$立即排除任何每天重新计算活动集或扫描活动集的解决方案。 一个简单的模拟是$O(N^2)$在最坏的情况下，因为每天可能需要扫描所有活动的雕像。 

一个微妙的点是，每天的活跃集并不是任意的，它是通过间隔插入和删除来定义的。 每个雕像都出现在一个连续的范围内，这意味着随着时间的推移，结构完全由间隔事件决定，而不是独立更新。 

一种容易被忽视的边缘情况是$x_i$以模数增长并包裹$N$。 幼稚的实现可能会忘记与计数阈值的模交互。 

例如，假设$N = 3$，并在某一天所有雕像$\{0,1,2\}$是活跃的，并且$x_i = 2$。 然后答案计算元素$\ge 2$，这只是一个元素。 但如果更新推送$x_{i+1}$到$3$，它包裹到$0$，极大地改变下一个查询。 任何治疗方法$x$因为单调是不正确的。 

另一个微妙的问题是，同一个小雕像在连续的时间间隔内做出贡献，因此状态仅在$2N$终点，不是任意的。 忽略这一点会导致每天重新计算，而不是事件驱动的更新。 

## 方法

 直接模拟保持当前设置$S(i)$采用平衡结构，例如标签上的多重集或 Fenwick 树。 每天，我们根据当天的事件插入和删除雕像，然后通过计算元素来回答查询$\ge x_i$。 这已经是$O(N \log N)$，但仍然依赖于有效回答“count ≥ x”。 

如果我们维护标签和 Fenwick 树上的频率数组，我们可以支持插入和删除的点更新，以及计数的前缀和。 然后每个查询减少为一个前缀和查询：总活跃减去前缀和最多$x_i - 1$。 

然而，这仍然留下了我们每天查询完全动态结构的依赖性，同时也被迫精确处理$N$天按顺序。 

关键的观察是，我们实际上不需要处理任意查询顺序：日期的顺序是固定的，并且结构以可预测的方式演变。 而不是将其视为$N$独立的范围计数查询，我们可以预先计算活动间隔，然后按顺序处理事件，在标签上维护芬威克树。 

每个小雕像恰好贡献两个事件：插入和移除。 经过几天的时间，我们相应地更新了结构。 那么每天的查询就只是一个 Fenwick 前缀查询。 

这减少了维护动态集的问题$O(\log N)$更新和查询。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 每日暴力扫描 |$O(N^2)$|$O(N)$| 太慢了 |
 | Fenwick 活跃时间间隔 |$O(N \log N)$|$O(N)$| 已接受 |

 ## 算法演练

 我们将所有雕像作为间隔进行处理并日复一日地进行模拟。 

1. 解析输入并重建每个雕像$j$、插入日期和删除日期。 这给出了一对$(l_j, r_j)$。 这一步将问题从“每天的日志”转换为区间形式，更容易聚合。 
2. 构建两个按天索引的事件列表：一个用于插入，一个用于删除。 对于每个雕像$j$，我们添加$j$到当天的插入列表$l_j$，以及当天的删除列表$r_j$。 这确保我们可以在线性时间内增量更新活动集。 
3. 维护索引上的 Fenwick 树$0 \ldots N-1$，其中每个索引存储小雕像当前是否处于活动状态。 当雕像变得活跃时，我们添加$+1$在其索引处； 当它变得不活跃时，我们减去$1$。 这个结构允许我们查询有多少活跃的小雕像位于标签的任何前缀中。 
4. 初始化$x = 0$。 我们还认为芬威克树总是准确地代表$S(i)$，当天的活动集。 
5.每一天$i$从$0$到$N-1$，首先应用当天计划的所有删除，然后应用所有插入。 只要在计算当天集合的查询之前应用当天的两个操作，顺序并不重要。 
6. 计算$y_i$作为带有标签的活动雕像的数量$\ge x$。 这是获得的$\text{active\_count} - \text{prefix\_sum}(x-1)$。 
7. 更新$x \leftarrow (x + y_i) \bmod N$。 这是该值唯一发生变化的地方，它仅取决于当前的活跃分布和当前阈值。 
8.处理全天后，输出$x$。 

### 为什么它有效

 在任何时刻，芬威克树都准确地代表集合$S(i)$因为每个小雕像都精确地贡献了一次插入和一次删除，并且这些精确地应用在其生命周期的边界处。 因此，每个查询都会在正确的集合上进行评估。 的计算$y_i$正是问题更新规则的定义，因为芬威克树将活动元素划分为以下元素$x_i$以及高于或等于的。 模更新仅影响未来的查询，不会追溯改变正确性，因此模拟仍然忠实于原始过程。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class Fenwick:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)

    def add(self, i, v):
        i += 1
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        if i < 0:
            return 0
        i += 1
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

    def range_sum(self, l, r):
        return self.sum(r) - self.sum(l - 1)

def solve():
    n = int(input())
    insert = [[] for _ in range(n)]
    remove = [[] for _ in range(n)]

    for day in range(n):
        parts = input().strip().split()
        i = 0
        while i < len(parts):
            sign = parts[i][0]
            j = int(parts[i][1:])
            if sign == '+':
                insert[day].append(j)
            else:
                remove[day].append(j)
            i += 1

    for j in range(n):
        d = int(input())
        remove[d].append(j)

    fw = Fenwick(n)
    active = 0
    x = 0

    for day in range(n):
        for j in remove[day]:
            fw.add(j, -1)
            active -= 1
        for j in insert[day]:
            fw.add(j, +1)
            active += 1

        less = fw.sum(x - 1)
        y = active - less
        x = (x + y) % n

    print(x)

if __name__ == "__main__":
    solve()
```该实现首先重建所有间隔端点。 关键的设计选择是将每天的插入和删除列表分开，以便每个事件只处理一次。 Fenwick 树存储每个雕像索引的活动指示器，从而实现快速前缀查询。 

一个微妙的实现细节是处理$x = 0$。 前缀查询使用$x - 1$，这变成$-1$，在这种情况下，芬威克树安全地返回零。 这避免了特殊的套管。 

另一个重要的细节是保持明确的`active`柜台。 虽然可以从 Fenwick 树查询总活动计数，但保留单独的计数器可以避免每天进行额外的对数运算。

 ## 边缘情况

 一种极端情况是所有雕像在最后几天之前都被移除。 活动集变空，所以$y_i = 0$， 和$x$保持不变。 在这种情况下，芬威克树自然会返回零，因为所有位置均处于非活动状态。 

另一种情况是当$x$由于累积而变大并回绕到零。 前缀查询逻辑干净地处理了这个问题，因为$x-1$变为负数并产生零，这意味着所有活动元素都被计算在内。 

最后一个微妙的情况是同一天发生多次插入和删除。 该算法每天批量处理它们，并且由于芬威克树只关心每天的最终状态，因此只要在查询之前应用所有更新，当天的顺序就不会影响正确性。
