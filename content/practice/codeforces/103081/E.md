 ## 解决方案
 ## 问题理解

 我们得到了一份由多种成分组成的食谱，对于每种成分，我们都知道两个数字：烤一个蛋糕需要多少成分，以及厨房目前有多少成分。 任务是确定我们可以烘烤的完整蛋糕的最大数量，以免原料耗尽。 

一个蛋糕需要同时满足所有原料的要求，所以如果任何一个原料不足，我们就无法再增加蛋糕的数量。 每种成分独立地规定了可以生产的蛋糕数量的上限。 因此，最终答案由最严格的成分决定。 

限制条件非常小，最多有 10 种成分。 这立即表明，即使是非常直接的计算也足够了，因为任何对每种成分进行持续工作的解决方案都将立即运行。 除了简单的算术之外，不需要进行优化。 

当原料需求量相对于供应量较大时，就会出现微妙的边缘情况。 例如，如果每个蛋糕需要一种成分 10 个单位，而只有 5 个单位可用，则无论其他丰度如何，仅该成分就迫使答案为零蛋糕。 另一种情况是，除了一个极其严格的约束之外，所有成分都很丰富，这成为瓶颈。 

示例：

 输入：```
2 5
70 1000
```第一种成分最多允许 2 // 5 = 0 个蛋糕，因此即使第二种成分充足，答案也是 0。 简单地求和或平均比率的方法是不正确的。 

## 方法

 蛮力的想法是一张一张地模拟烘焙蛋糕。 对于每个尝试的蛋糕数量 k，我们检查每种成分是否至少有 k 倍其所需数量的可用量。 我们增加 k 直到这个条件不成立。 虽然正确，但这种方法最多执行 O(answer × N) 检查，并且在成分量很大的病态情况下，答案可能达到 10^4 或更多，与所需的相比，会出现不必要的重复。 

关键的观察是每种成分独立地限制蛋糕的数量。 如果一种原料需要每个蛋糕有 r 单位，而我们有一个可用单位，那么仅该原料允许的蛋糕数量就是 a // r。 由于所有成分必须同时满足，因此总体答案是这些每种成分限制的最小值。 这将问题从重复模拟转变为单遍计算。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力模拟| O(N × 答案) | O(1) | O(1) | 太慢了 |
 | 每种成分的最低含量| O(N) | O(1) | O(1) | 已接受 |

 ## 算法演练

 1.读取成分的数量N，然后读取代表每种成分的所需量和可用量的每对值。 这设置了定义可行的蛋糕生产的约束。 
2. 对于每种成分，通过整数除法将可用数量除以所需数量，计算出它可以单独支持多少个蛋糕。 此步骤将原始资源约束转换为统一的“蛋糕容量”指标。 
3. 维护一个初始化为非常大的值的变量，代表当前蛋糕数量的最佳上限。 
4. 对于每种成分，通过取其当前值和计算出的成分容量之间的最小值来更新此变量。 这确保了最终结果遵守最严格的约束。 
5.处理完所有原料后，输出最终的最小值，该值代表在不违反任何约束的情况下可以制作的蛋糕的最大数量。 

为什么有效：每种成分都对可行的解决方案施加了独立的上限。 任何有效的解决方案必须同时满足所有边界，因此解决方案空间是这些约束的交集。 上限的交集由最小界限确定，这保证了没有过度使用任何成分，同时最大化完整蛋糕的数量。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    ans = 10**18

    for _ in range(n):
        need, have = map(int, input().split())
        ans = min(ans, have // need)

    print(ans)

if __name__ == "__main__":
    solve()
```该解决方案增量读取输入并保持运行最小值。 关键的实现细节是整数除法，它正确地计算出每种成分可能的蛋糕数量。 将答案初始化为一个非常大的数字可确保第一个成分正确设置基线。 

一个常见的错误是颠倒划分顺序。 正确的计算是有//需要，而不是需要//有。 后者会颠倒约束并产生无意义的结果。 另一个微妙的点是确保整数除法而不是浮点，因为小数蛋糕是无关紧要的。

 ## 边缘情况

 一个重要的边缘情况是，即使是一个蛋糕，原料的数量也不足。 

输入：```
1
5 3
```该算法计算 3 // 5 = 0。由于运行最小值一开始很大，因此它立即变为 0，并保持为 0 直到结束。 这正确地反映了不可能有蛋糕。 

另一种情况是所有成分的比例相同。 

输入：```
3
2 10
2 10
2 10
```每种成分的产量为 10 // 2 = 5，因此最低限度始终为 5。 该算法不会错误地对容量进行乘法或求和，否则会错误地夸大结果。
