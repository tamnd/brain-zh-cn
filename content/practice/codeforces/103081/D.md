 ## 解决方案
 ## 问题理解

 我们得到一个无向加权图，其中顶点是交叉点，边是有长度的街道。 Phoebe 始终在节点 0 处开始和结束每个慢跑会话。每个慢跑会话都是一次封闭步行，总长度限制在下限之间$L$和一个上限$U$。 

还有一条额外的簿记规则可以推动目标的实现。 当菲比第一次穿过一条街道的任何部分时，它就被认为是“新的”，如果慢跑过程中至少包含一条在之前的任何过程中从未使用过的街道，那么它就被称为有趣的。 一旦街道被使用，它就不会再带来新奇感，即使它在以后的运行中被部分穿越。 

任务是计划尽可能多的有趣的慢跑课程。 每个会话可以自由地重复使用旧街道，但要算作新会话，它必须引入至少一条以前未使用的街道。 

约束条件很大，最多可达$10^5$交叉路口和$10^5$街道。 这立即排除了任何更接近的情况$O(S^2)$甚至重复每条边的最短路径计算。 单个线性或$O(S \log S)$绕过 Dijkstra 是正确的尺度。 

声明中的一个微妙之处是，菲比可能只穿过街道的一部分，但当她接触街道的那一刻，它就被认为是完全“看到”的。 这消除了对部分边缘使用进行推理的需要。 每条边要么未使用，要么已被先前有趣的运行消耗。 

第二个微妙之处是，一次运行只需要包括至少一条新街道，但也可能包括许多条。 一种天真的解释表明我们可能需要将边缘划分为复杂的结构，但事实证明这是不必要的。 

一些重要的边缘情况：

 如果图只有一条边$0 - 1$与长度$3$,$L = 7$,$U = 7$，则不存在有效的运行，因为任何封闭的运行必须至少具有长度$6$，所以答案是$0$。 

如果存在多个边形成循环，则幼稚的方法可能会尝试贪婪地构建长路径，并错误地假设边不能以结构化方式在不同候选运行之间重用，而真正的解决方案仅取决于节点的可达性和最短路径结构$0$。 

## 方法

 直接的暴力解释试图模拟该过程。 人们可以想象重复搜索一条从节点 0 开始和结束、长度为$[L, U]$，其中包含至少一个未使用的边，然后将这些边标记为已使用并继续。 即使我们可以检测单个步行的可行性，枚举此类步行也会带来爆炸性的组合。 由于循环，可能的行走数量随着路径长度呈指数增长，甚至决定每次运行包含哪个边子集也会导致难以处理的搜索空间。 

关键的观察结果是“兴趣”条件几乎完全解耦运行。 每条边仅在第一次使用时才重要。 如果我们可以将每条边分配给最多一个运行，并确保对于每个选定的边，存在一些包含它的有效闭合游走，那么我们可以将每个这样的分配实现为单独的运行。 由于边在这种意义上是独立的，因此游程的最大数量变成了至少一个有效的闭合游走可以单独支持的边的数量。 

因此，问题简化为独立测试每条边：我们能否从 0 构建任何包含该边且总长度为$[L, U]$？ 

对于无向加权图，从 0 开始强制遍历边的最短可能闭合游走$(u, v, w)$是通过从 0 到$u$, 采取优势$v$，并从返回$v$到 0。成本为：$$\text{dist}(0, u) + w + \text{dist}(0, v)$$任何其他包含边缘的有效行走只能更长，因为所有边缘权重都是正的。 这是至关重要的：一旦我们有了包含边缘的最小可能长度，我们总是可以通过绕过从 0 开始的循环来进一步增加长度，而不会失去可行性或违反积极性约束。 

因此，当且仅当其最小强制循环长度至多时，一条边才可用$U$。 如果这个条件成立，我们总是可以延长步行范围以降落在内部的某个地方$[L, U]$。 下界$L$并不限制可行性，因为我们总是可以通过插入弯路来增加长度。 

这将问题简化为单源最短路径计算，然后对边缘进行线性扫描。

 | 暴力枚举运行 | 指数| 高| 太慢了 |
 | Dijkstra + 每边检查 |$O(S \log I)$|$O(I + S)$| 已接受 |

 ## 算法演练

 ### ## 算法演练

 1. 使用 Dijkstra 算法计算从节点 0 到每个其他节点的最短路径距离。 这给出了从主路口到图中任意点的最小旅行成本。 此步骤是必要的，因为每个候选封闭游走必须在节点 0 处开始和结束，因此距 0 的距离定义了“到达”任何边缘的最便宜的方式。 
2. 对于每条街道 (u, v, w)，计算强制包含该街道的封闭步行的最小成本。 计算公式为 dist[u] + w + dist[v]。 这个想法是，我们从一个端点从 0 最优地到达边缘后进入边缘，然后从另一端点最优地返回到 0 。 
3. 检查该最小所需成本是否至多为U。如果超过U，则即使包含该边的最短可能有效闭合游走也太长，因此该边永远不能在任何有效运行中使用。 
4. 统计所有满足条件的边。 每个这样的边都可以分配给不同的运行，因为运行只需要至少一个新边，并且我们始终可以独立地为每个边构建单独的游走。 
5. 输出总计数作为感兴趣的运行的最大数量。 

### 为什么它有效

 关键的不变量是，对于每条边，我们正确地识别包含该边的从 0 开始和结束的任何闭合步行的最小可能长度。 由于所有边权重都严格为正，因此包含相同边的任何替代行走只能增加总长度。 这使得计算值成为真正的下限。 

一旦该下限位于区间内$[L, U]$，可以通过插入弯路来向上延伸步行，而无需从路线中删除边缘，这意味着可行性仅取决于最小值是否至多$U$。 由于每条边都可独立分配给一次运行，因此最大化运行减少为计算可行边。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

import heapq

def solve():
    I, S, L, U = map(int, input().split())
    g = [[] for _ in range(I)]
    edges = []

    for _ in range(S):
        u, v, w = map(int, input().split())
        g[u].append((v, w))
        g[v].append((u, w))
        edges.append((u, v, w))

    INF = 10**18
    dist = [INF] * I
    dist[0] = 0
    pq = [(0, 0)]

    while pq:
        d, u = heapq.heappop(pq)
        if d != dist[u]:
            continue
        for v, w in g[u]:
            nd = d + w
            if nd < dist[v]:
                dist[v] = nd
                heapq.heappush(pq, (nd, v))

    ans = 0
    for u, v, w in edges:
        if dist[u] + dist[v] + w <= U:
            ans += 1

    print(ans)

if __name__ == "__main__":
    solve()
```该解决方案首先构建图表并从节点 0 运行 Dijkstra 来计算最短距离。 这是唯一需要的全局结构，因为对边的每个可行性检查仅取决于到其端点的距离。 

之后，使用最小强制闭合行走公式独立评估每条边。 支票反对$U$是足够的，因为任何可行的最小值自动意味着可扩展到所需的区间。 

一个常见的实现陷阱是忘记图是无向的，因此必须将两个方向添加到邻接列表中。 另一个微妙的点是我们从未明确使用$L$在最终检查中，因为通过循环增加路径长度的能力使得下限与可行性无关。

```最短路径：

 距离[0] = 0，距离[1] = 3

这表明即使最小周期低于$L$，它仍然有效，因为额外的弯路可以将长度增加到正好 7。

 ## 边缘情况

 关键的边缘情况是，图包含可到达的边，但仍然太昂贵，无法包含在从节点 0 开始的任何闭合步行中。例如，如果到两个端点的最短路径很大，则即使是小边也会变得无效，因为回程占主导地位。 该算法正确地拒绝了此类边缘，因为计算出的强制循环超过了$U$。 

另一个边缘情况是当$L$很大但是$U$仅比最小周期稍大。 该解决方案仍然有效，因为任何可行的最小值低于或等于$U$可以走弯路充气。 该算法隐式依赖于从 0 开始的正权重循环的存在，这始终允许受控长度增加而不破坏有效性。 

最后，当图是一棵树时，每条边都形成唯一的路径，解决方案简化为检查每条边的路径通过 0 循环是否适合$U$，公式直接处理，无需特殊外壳。
