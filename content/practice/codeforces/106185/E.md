 ## 解决方案
 ## 问题理解

 我们有一棵树$n$行星。 每个行星都通过一条双向空间路线与其父行星相连，并且每条路线都有一个截止日期。 仅当您开始穿越该路线时，当前时间不大于其截止时间，该路线才可以使用。 穿越任何路线总是只需要一个时间单位。 

您从时间 0 的行星 1 开始。任务是至少访问所有其他行星一次，但有一个变化：您可以立即传送到您已经访问过的任何行星。 传送不消耗时间，但只能传送到之前访问过的星球，所以第一次到达星球仍然必须通过正常的路线旅行。 

因为隐形传送消除了物理步行穿过树中已经探索过的部分的需要，所以真正重要的结构是第一次访问行星的顺序。 每次你第一次到达一个新星球，时间都会增加一倍。 

目标是确定是否存在访问所有行星的顺序，以便每次遍历一条路线来发现新节点时，该路线尚未过期。 如果存在这样的顺序，我们必须输出访问行星 2 到 2 号行星的任何有效顺序$n$。 

输入编码有根树：planet$i$连接到$p_i$，并且该边有截止日期$e_i$。 自从$p_i < i$，该结构已经隐式以 1 为根。 

约束条件$n \le 1000$允许$O(n^2 \log n)$甚至$O(n^2)$方法。 带有堆的线性贪婪解决方案就足够了。 

一个微妙的失败案例来自于忽略这样一个事实：即使节点的最后期限很紧，也无法在其父节点之前访问该节点。 

例如，假设一个孩子的截止日期较早，但其父母的截止日期被推迟：

 输入：```
3
1 0
1 0
```两条边均在时间 0 到期。任何有效的访问订单都必须是`1, 2, 3`或者`1, 3, 2`。 两个子节点必须分别在时间 1 和 2 被访问，但是两条边都需要时间 0，所以遍历是不可能的。 仅检查每个节点的截止日期而不考虑父级约束的简单方法将错误地接受这种结构。 

## 方法

 如果我们忽略最后期限，问题就很简单：我们可以从节点 1 开始并运行任何 DFS 订单，因为远距传送允许我们跳回任何访问过的节点并继续在其他地方探索。 

困难在于，每个新发现的节点都会消耗一个单位时间，并且用于发现节点的每条边在当时必须仍然有效。 这意味着访问顺序是我们控制的真正变量。 

关键的观察是每个节点$i$（为了$i > 1$) 只能通过其唯一的父边首次到达$(p_i, i)$。 如果节点$i$被访问的位置$k$按照顺序（节点1从1开始），然后遍历边$(p_i, i)$发生在某个时间$k-1$。 因此约束变为：$$k - 1 \le e_i \quad \Rightarrow \quad k \le e_i + 1$$因此，每个节点在访问顺序中的位置都有一个截止日期。 

我们还必须尊重节点不能在其父节点之前被访问。 这形成了根树优先级约束：父级必须早于子级出现。 

因此，问题变成在单个时间线上安排单位时间任务，其中每个任务都有截止日期和先决条件（其父任务）。 在每个时间步，我们选择一个可用节点（其父节点已被访问过）并将其永久固定到顺序中。 

蛮力的想法会尝试与父约束一致的所有排列，每次都检查截止日期。 在最坏的情况下这是阶乘，即使对于$n=20$。 

最优结构来自于经典的贪婪调度原则：在所有当前可用的节点中，总是首先选择截止时间最小的节点。 直觉是，延迟紧迫期限的节点只会降低未来步骤的可行性，而尽早采取它可以为较宽松的节点保留灵活性。 

我们维护一个“可用”节点的优先级队列，这意味着其父节点已被访问过。 每次我们提取截止时间最小的节点时，将其分配给访问顺序中的下一个位置，然后激活其子节点。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力排列 |$O(n!)$|$O(n)$| 太慢了|
 | 贪婪堆（最早截止日期优先）|$O(n \log n)$|$O(n)$| 已接受 |

 ## 算法演练

 我们将树的根设为 1。每个节点$i > 1$有一个父母$p_i$和最后期限$e_i$，转化为位置约束$pos[i] \le e_i + 1$。 

1. 从已在位置 1 访问过的节点 1 开始。这是最初唯一可用的节点，因为所有其他节点都依赖于它。 
2. 维护由截止日期决定的可用节点的最小堆$e_i$。 最初，插入节点 1 的所有子节点。 
3. 反复从堆中提取期限最小的节点。 这个节点是当前所有可到达的选择中最紧急的，因此延迟它的风险是最大的。 
4. 将其分配到访问顺序中的下一个位置。 如果违反其截止日期，则意味着当前位置减 1 超过$e_i$，构造立即失败，因为未来的重新安排无法解决此问题。 
5. 将此节点标记为已访问，并将其所有子节点推入堆中，因为它们现在可以通过传送到此节点来访问。 
6. 继续，直到处理完所有节点。 如果每个节点都放置成功，则记录的顺序就是答案。 

### 为什么它有效

 在每一步中，可用节点集表示先决条件已满足的所有节点。 其中，选择最小的截止日期可以确保任何接近不可行的节点都不会被推迟到更灵活的节点之后。 如果存在有效的排序，则总有一种方法对其进行转换，以便在不违反优先级约束的情况下以非递减的截止日期顺序处理节点，因为将较晚的宽松截止日期节点与较早的严格截止日期节点交换永远不会损害可行性，而且可以修复违规问题。 

## Python 解决方案```python
import sys
input = sys.stdin.readline
import heapq

def solve():
    n = int(input())
    parent = [0] * (n + 1)
    deadline = [0] * (n + 1)
    g = [[] for _ in range(n + 1)]

    for i in range(2, n + 1):
        p, e = map(int, input().split())
        parent[i] = p
        deadline[i] = e
        g[p].append(i)

    visited = [False] * (n + 1)
    visited[1] = True

    heap = []
    order = []

    for v in g[1]:
        heapq.heappush(heap, (deadline[v], v))

    t = 1  # position index (node 1 is at time 0 / position 1)

    while heap:
        e, v = heapq.heappop(heap)

        # v will be visited at position t+1, so time is t
        if t > e:
            print("no")
            return

        order.append(v)
        visited[v] = True

        for to in g[v]:
            heapq.heappush(heap, (deadline[to], to))

        t += 1

    if len(order) != n - 1:
        print("no")
        return

    print("yes")
    print(*order)

if __name__ == "__main__":
    solve()
```该实现准确地反映了贪婪过程。 堆存储按截止日期排序的所有当前可用节点。 变量`t`跟踪根节点之后已经放置了多少个节点，因此下一个节点的当前访问时间是`t`。 

一个常见的陷阱是时间与位置的处理不一致。 这里节点1隐式地占据了位置1，所以当我们要放置下一个节点时，它的遍历时间正好是`t`， 不是`t+1`。 支票`if t > e`对条件进行编码$t \le e$。 

另一个微妙的点是，我们从不明确强制超出父激活范围的连接，因为远距传输允许我们始终到达任何已经访问过的节点。 唯一的结构要求是只有在处理其父级之后才推送子级。 

## 工作示例

 考虑一棵小树：```
1
2 1
3 0
```节点 2 的截止时间为 1，节点 3 的截止时间为 0。 

| 步骤| 堆（截止日期、节点）| 所选节点 | 时间 t | 有效的？ |
 | --- | --- | --- | --- | --- |
 | 开始 | (1,2), (0,3) | - | 1 | - |
 | 1 | (0,3), (1,2) | (0,3), (1,2) | 3 | 1 | 1 ≤ 0 失败 |

 这会立即失败，因为节点 3 必须在时间 1 被访问，但只允许时间 0。算法正确地拒绝。 

现在考虑：```
1
2 1
3 1
```| 步骤| 堆| 选择| t | 结果 |
 | --- | --- | --- | --- | --- |
 | 开始 | (1,2), (1,3) | (1,2), (1,3) | - | 1 | - |
 | 1 | (1,2), (1,3) | (1,2), (1,3) | 2 | 1 | 好的 |
 | 2 | (1,3) | 3 | 2 | 2 > 1 失败 |

 这表明，即使是对称的最后期限也可能根据顺序而失败，并且贪婪的选择对于检测不可行性是必要的。 

这些痕迹证实，正确性取决于尊重当前可用节点中的最早截止日期，而不仅仅是全局排序。

 ## 边缘情况

 一种重要的边缘情况是根的多个子节点具有相同的截止日期。 该算法可以选择任一顺序，但可行性取决于尊重两者仅在访问根之后才可用。 堆自然地处理这个问题，因为联系是任意的。 

另一个边缘情况是长链，其中截止日期沿着路径严格减少。 这总是可行的，因为每个节点在需要时变得可用，并且贪婪过程反映了链顺序。 

更微妙的情况是深层节点的截止日期比浅层同级子树更早。 堆确保其父级可用时首先对其进行处理，从而防止其被延迟到不相关的分支之后。
