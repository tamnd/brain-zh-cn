 ## 解决方案
 ## 问题理解

 我们得到一排盘子，每个盘子里放着一个苹果（`a`）或香蕉（`b`）。 我们从初始配置开始，希望使用一系列操作将其转换为目标配置，但这些操作不是任意的单个交换。 相反，每个操作都是由艾米或贝西这两只狗之一执行的确定性“运行”，并且每次运行都会在受约束的从左到右遍历中修改多个位置。 

每只狗首先嘴里含着一个水果，然后从最左边的盘子开始扫描。 在扫描过程中，它会忽略与其当前持有的匹配的车牌。 当它发现第一个盘子里含有相反的水果时，它会执行交换：它将它所持有的东西放在该盘子上，拿起盘子里的水果，然后继续扫描。 最终，它在遇到的下一个相反类型的板上执行第二次交换，然后返回主位置并结束操作。 如果在任何时候不存在所需的相反水果，则操作失败。 

艾米开始拿着一根香蕉，贝西开始拿着一个苹果，他们的行为对于苹果和香蕉来说是对称的。 我们必须决定是否可以应用最多 10000 个这样的操作，选择 Amy 和 Bessie 调用的顺序，将初始字符串转换为目标字符串。 

关键的限制是每个操作实际上都是全局的，可能以结构化的方式影响两个头寸，而不是局部交换。 这立即表明我们处理的不是独立位置，而是扫描过程引起的全局奇偶性或排序约束。 

由于 n 最多为 100，因此对操作序列进行简单的指数搜索是不可能的，因为即使 2^100 也远远超出了限制，并且分支因子实际上每步为 2，最多可达 10000 个步骤。 任何正确的解决方案都必须依赖于变换过程的结构不变量，而不是对所有序列的模拟。 

当初始配置已经等于目标时，即使问题状态 s 和 t 不同，也会出现微妙的边缘情况，因为推理过程中的中间状态可能会错误地假设身份无法达到或微不足道。 另一种边缘情况是，其中一个字符串中某个字符的出现次数很少，使得狗无法在运行过程中执行第二次所需的遭遇，即使存在局部不匹配，也会导致失败。 

主要困难在于每个操作都在字符串中强制执行全局“配对”行为，因此可行性取决于所需的重新排列是否可以在两个交替扫描规则下分解为有效的配对交互。 

## 方法

 强力方法会将每个操作视为整个字符串状态的转换。 从任何配置中，我们都可以模拟 Amy 的操作或 Bessie 的操作，如果有效则生成下一个状态。 这将问题转化为图可达性问题，其中节点都是 2^n 个可能的字符串，边对应于应用两个操作之一。 即使我们忽略不可达状态，每个节点最多也有两个传出转换。 

然而，状态空间以 n 为指数，即使是 BFS 也会探索天文数字般的大量配置。 关键的失败点是，n 达到 100 已经使得可能的状态数量完全不可行。 

结构上的突破来自于观察到每个操作都不是任意的，而是表现得像对不匹配符号的受控两指针配对过程。 每个操作以固定的从左到右的顺序有效地消耗两次出现的相反类型。 这意味着该进程不会自由地排列字符串； 相反，它在不同字符的位置之间强制执行严格的交错约束。 

重新构建问题，每个操作都可以被视为纠正交替不匹配的特定模式，并且整个系统的行为就像维持两个位置序列之间的一致性，其中`a`和`b`必须在交替约束下进行匹配。 这将问题简化为检查 s 和 t 之间的失配结构是否可以使用有限数量的结构化对操作来解决，这相当于验证某种诱导的不平衡事件序列在两个交替的堆栈式变换下是否可行。 

关键的观察是，我们永远不需要模拟完整的转换；我们不需要模拟完整的转换。 相反，我们跟踪每种类型存在多少个不匹配，以及它们是否可以以与每个操作的强制从左到右贪婪性质一致的方式进行配对。 这会导致建设性的贪婪检查，要么构建有效的序列，要么证明不可能。

 ## 算法演练

 核心思想是将每个字符串位置重新解释为必须通过不匹配索引之间的结构化配对过程来满足的要求。 

我们定义 s 和 t 不同的位置，将它们分为两类： s 具有不同的位置`a`并且 t 有`b`，以及 s 所在的位置`b`并且 t 有`a`。 转换过程必须使用每个操作以约束顺序有效处理两个不匹配的操作将一个多重集转换为另一个多重集。 

该算法进行如下。 

1.收集s[i]与t[i]不同的所有索引i，并根据不匹配方向将它们分成两个有序列表。 一个列表存储需要的职位`a → b`, 其他商店`b → a`。 顺序是从左到右的索引顺序。 
2. 观察到每个操作以结构化交替方式恰好翻转两个位置，这意味着每个操作以兼容的顺序消耗每个列表中的一个不匹配项。 这导致两个列表之间存在配对约束。 
3. 尝试贪婪地模拟匹配：在两个不匹配列表上维护两个指针。 在每个步骤中，对每个列表中的下一个可用的不匹配项进行配对。 如果一个列表先于另一个列表耗尽，或者如果排序约束阻止按所需序列进行配对，则转换是不可能的。 
4. 每次成功配对对应序列中的一个操作。 操作类型（Amy 或 Bessie）取决于当前启动所需校正模式的不匹配类型； 这可以从配对序列中领先的列表推断出来。 
5. 通过为每个配对附加正确的字符来构造操作字符串。 确保总操作不超过10000次； 否则返回失败。 

这种贪婪配对起作用的原因是每个狗操作内的扫描顺序强制最早可用的不匹配首先进行交互。 除了从左到右扫描允许的范围之外，没有自由跳过或重新排序交互，因此任何有效序列都必须尊重不匹配位置的自然顺序。 这将问题转变为检查两个有序序列是否可以在不违反单调性的情况下交织成有效的交替对。 

保持的不变性是，在处理 k 个操作之后，前 k 个配对的不匹配已在两个列表中以严格递增的索引顺序进行匹配。 这确保了未来的操作不需要重新访问较早的索引，这将与每个操作中嵌入的从左到右的扫描约束相矛盾。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve_case(n, s, t):
    ab = []
    ba = []
    
    for i in range(n):
        if s[i] == t[i]:
            continue
        if s[i] == 'a' and t[i] == 'b':
            ab.append(i)
        else:
            ba.append(i)
    
    # If counts differ, impossible under symmetric pairing constraint
    if len(ab) != len(ba):
        return None
    
    ops = []
    
    i = j = 0
    turn = 0  # 0 means AB consumes next, 1 means BA consumes next
    
    while i < len(ab) and j < len(ba):
        # each operation pairs one ab and one ba mismatch
        if turn == 0:
            # try consume ab first
            if i >= len(ab):
                return None
            ops.append('A')
            i += 1
            j += 1
        else:
            if j >= len(ba):
                return None
            ops.append('B')
            i += 1
            j += 1
        
        turn ^= 1
    
    if i != len(ab) or j != len(ba):
        return None
    
    if len(ops) > 10000:
        return None
    
    return ''.join(ops)

def solve():
    out = []
    while True:
        line = input().strip()
        if not line:
            break
        n = int(line)
        if n == 0:
            break
        s = input().strip()
        t = input().strip()
        
        res = solve_case(n, s, t)
        if res is None:
            out.append("no")
        else:
            out.append("yes")
            out.append(res)
    
    print("\n".join(out))

if __name__ == "__main__":
    solve()
```该实现首先将不匹配位置提取到两个单调列表中。 这种分离至关重要，因为两只狗的扫描行为都保留从左到右的顺序，因此只有索引顺序很重要。 

配对循环强制执行操作类型的严格交替，同时以锁步方式消耗两个不匹配列表。 变量`turn`编码在每个配对校正步骤中首先应用哪只狗的结构动作。 每次迭代仅消耗每个不匹配列表中的一个元素，反映了每个有效操作恰好修复两个不匹配位置的事实。 

一个关键的微妙之处是我们从不尝试模拟艾米或贝西的内部扫描。 相反，我们依赖这样一个事实：任何有效的全局变换必须对应于索引顺序中一致的不匹配配对，这足以重建有效的操作序列（如果存在）。 

## 工作示例

 考虑示例情况，其中 s =`bba`且 t =`bab`。 

我们首先列出不匹配项：索引 1 不同 (`b → a`是 BA），索引 2 不同（`a → b`根据方向反向分类为AB）。 这会产生两个可以配对的列表。 

| 步骤| ab 指针 | ba指针| 操作选择 | 状态进展|
 | --- | --- | --- | --- | --- |
 | 1 | 0 | 0 | 乙| 第一个配对校正 |
 | 2 | 1 | 1 | 一个 | 第二对校正|

 构造的序列交替操作并消耗两种不匹配类型，产生有效的转换序列。 

现在考虑一种情况，其中 s =`aaab`且 t =`bbaa`。 不匹配结构变得扭曲：有更多`a → b`转变比`b → a`，使得配对不可能。 

| 步骤| ab 计数 | 巴计数| 决定|
 | --- | --- | --- | --- |
 | 初始| 2 | 1 | 检测到不匹配不平衡|

 这表明，无论顺序如何，不平衡都会立即阻止任何有效序列。

 ## 边缘情况

 一种关键的边缘情况是存在但仅在一个方向上的不匹配。 例如，如果每个不同的位置都是`a`在 s 和`b`在t中，则没有对应的`b → a`配对的位置。 该算法立即拒绝这种情况，因为两个不匹配列表的大小不相等，这正确地反映了没有狗操作序列可以修复不平衡的转换。 

另一种边缘情况是存在不匹配但严重交错的情况。 即使如此，该算法也不依赖于邻接性，而仅依赖于排序的索引顺序，因此交错不会影响正确性。 配对仍然在排序索引上同步进行，确保遵守从左到右的约束。 

最后的边缘情况是所需操作的数量超过 10000 的限制。尽管 n ≤ 100 在实践中不太可能出现这种情况，但算法在输出序列之前显式检查此边界，确保即使在简并结构中也符合输出约束。
