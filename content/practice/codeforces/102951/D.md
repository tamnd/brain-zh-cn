 ## 解决方案
 ## 问题理解

 我们从一个概念上非常大的数组开始，索引从 0 到$10^9 - 1$，但最初每个位置都包含零。 我们没有显式存储该数组，而是提供了两种类型的修改和查询范围的操作。 

第一种类型的操作对段应用增量。 给定一个段$[l, r)$和一个值$v$，该范围内的每个位置都有$v$添加到其当前值。 这些更新重叠并累积，因此每个位置的最终值是覆盖该位置的所有更新的总和。 

第二种类型的操作要求范围总和。 给定一个段$[l, r)$，我们需要当前存储在该区间内的所有值的总和。 这不仅仅是计算更新，而是在应用所有范围添加后对数组的实际值求和。 

关键的难点是坐标范围达到$10^9$，因此任何显式构建数组的解决方案都是不可能的。 即使是坐标压缩的密集数组，也只有在我们小心地避免迭代所有位置时才可行。 

约束条件意味着最多$10^5$更新和$10^5$查询。 通过扫描受影响的更新来处理每个查询的解决方案的行为类似于$O(NQ)$，这远远超出了可行的限度。 即使按位置模拟也是不可能的，因为数组长度在内存或时间中不可用。 

当更新和查询严重重叠并触及边界点时，会出现微妙的边缘情况。 例如，考虑在某个点相遇的两个更新：

 输入：```
1 1
0 5 2
0 5
```正确答案是 10，因为从 0 到 4 的每个元素都变成 2。一种幼稚的实现，错误地将范围视为两端包含或差一不匹配$[l, r]$相对$[l, r)$要么会重复计数，要么完全错过最后一个元素。 该问题始终使用半开区间，即使数据结构本身是正确的，混合约定也会导致不正确的聚合。 

## 方法

 暴力解释很简单：应用每次更新后，我们将显式维护数组并通过对范围求和来重新计算答案。 每次更新都触及$O(10^9)$最坏情况下的位置，每个查询也会扫描$O(10^9)$职位。 即使我们假设只有触摸的区域很重要，重叠更新也会使有效成本爆炸到大约$O(N \cdot 10^9 + Q \cdot 10^9)$，这是无法使用的。 

第一个结构观察是，虽然坐标空间很大，但只有更新边界才重要。 在两个连续的事件点之间，数组的值是恒定的。 这意味着数组可以被压缩成由所有定义的段$l$和$r$端点。 最多有$2N + 2Q$这样的位置，因此问题从一个巨大的隐式数组减少到一组可管理的断点。 

一旦我们压缩坐标，每次更新都会成为一个更小的索引空间上的范围添加。 剩下的挑战是有效支持两种操作：范围添加和范围求和。 这正是具有差分技巧的 Fenwick 树或线段树的经典设置。 

关键的见解是通过数组的差异结构来处理数组。 如果我们维护一个支持范围加法的结构，我们可以将其转换为差异数组上的点更新。 然后前缀和重建实际值。 为了回答范围和查询，我们需要前缀和的前缀和，这导致维护两棵芬威克树：一棵用于系数，一棵用于加权系数。 这个标准技巧允许在对数时间内进行这两种操作。 

所以转变是：暴力通过显式模拟起作用，但由于规模而失败。 坐标压缩减小了宇宙的大小。 基于Fenwick的范围更新+范围查询将问题转化为前缀算术。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 |$O(N \cdot 10^9 + Q \cdot 10^9)$|$O(10^9)$| 太慢了 |
 | 坐标压缩+Fenwick树|$O((N+Q)\log (N+Q))$|$O(N+Q)$| 已接受 |

 ## 算法演练

 我们首先提取所有重要的坐标。 这些都是每$l$和$r$来自更新和查询。 我们对它们进行排序并删除重复项，构建从真实坐标到紧凑范围的压缩索引映射$[0, M)$。 

每次更新$[l, r), v$被转换为压缩索引$[l', r')$。 然后，我们使用芬威克树技巧应用范围添加。 我们不是更新间隔中的每个元素，而是维护一个允许我们在前缀上添加线性效果的结构。 

我们使用两棵 Fenwick 树。 一个跟踪某个位置受到影响的次数，另一个跟踪累积的加权贡献。 这种配对允许在对数时间内重建前缀和。 

对于每次更新，我们在端点处对每棵树执行两个点更新，对范围添加的效果进行编码。 

对于每个查询$[l, r)$，我们计算前缀和$r-1$并减去前缀总和$l-1$。 这将得出该时间间隔内的总和。 

这样做的原因是范围加法变成了差异信号，而前缀重建将该信号转换为实际值。 

### 为什么它有效

 该算法保持了 Fenwick 结构代表数组的差分分解的不变量。 每次范围更新相当于添加一个阶跃函数。 任何前缀和查询都会评估这些阶跃函数到某个点的积分。 因为积分是线性的，所以重叠更新只是简单地添加它们的贡献而不会产生干扰，并且前缀的减去可以精确地隔离任何段。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class Fenwick:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)

    def add(self, i, v):
        i += 1
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        if i < 0:
            return 0
        i += 1
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

def add_range(bit1, bit2, l, r, v):
    bit1.add(l, v)
    bit1.add(r, -v)
    bit2.add(l, v * (l - 1))
    bit2.add(r, -v * (r - 1))

def prefix_sum(bit1, bit2, i):
    return bit1.sum(i) * i - bit2.sum(i)

def range_sum(bit1, bit2, l, r):
    return prefix_sum(bit1, bit2, r - 1) - prefix_sum(bit1, bit2, l - 1)

def main():
    n, q = map(int, input().split())
    updates = []
    queries = []

    coords = []

    for _ in range(n):
        l, r, v = map(int, input().split())
        updates.append((l, r, v))
        coords.append(l)
        coords.append(r)

    for _ in range(q):
        l, r = map(int, input().split())
        queries.append((l, r))
        coords.append(l)
        coords.append(r)

    coords = sorted(set(coords))
    idx = {x: i for i, x in enumerate(coords)}

    m = len(coords)
    bit1 = Fenwick(m)
    bit2 = Fenwick(m)

    for l, r, v in updates:
        l = idx[l]
        r = idx[r]
        add_range(bit1, bit2, l, r, v)

    out = []
    for l, r in queries:
        l = idx[l]
        r = idx[r]
        out.append(str(range_sum(bit1, bit2, l, r)))

    print("\n".join(out))

if __name__ == "__main__":
    main()
```该代码首先压缩坐标，以便将所有有意义的边界映射到连续的索引空间中。 这是必要的，因为 Fenwick 树需要密集索引。 

这`add_range`函数将范围添加编码为两个 Fenwick 更新，将效果拆分为前缀可调整的组件。 这`prefix_sum`函数使用标准二树技巧，结合原始累加和校正项来重建某个点的实际值。 

最后，通过减去两个前缀和来回答每个查询，从而准确地隔离所需的段。 

一个微妙的细节是所有范围都被视为半开。 减法使用$r - 1$和$l - 1$小心，确保相邻段之间没有逐一泄漏。 

## 工作示例

 考虑一个小案例：

 输入：```
2 2
1 4 2
3 6 1
1 6
2 5
```我们压缩坐标：$[1, 2, 3, 4, 5, 6]$。 

第一次更新后，值$[1,4)$是+2。 第二次之后，$[3,6)$添加+1。 

我们可以跟踪前缀值：

 | 位置 | 更新1后| 更新2后| 决赛|
 | ---| ---| ---| ---|
 | 1 | 2 | 0 | 2 |
 | 2 | 2 | 0 | 2 |
 | 3 | 2 | 1 | 3 |
 | 4 | 0 | 1 | 1 |
 | 5 | 0 | 1 | 1 |

 询问$[1,6)$总和为 2 + 2 + 3 + 1 + 1 = 9。 

查询$[2,5)$总和为 2 + 3 + 1 = 6。 

该迹线与 Fenwick 重建通过前缀差异计算的结果相匹配。

 ## 边缘情况

 一种边缘情况是仅更新触摸端点。 例如，更新$[0, 1)$接下来是一个查询$[1, 2)$。 正确的结果为零，因为范围不重叠。 Fenwick 结构可以处理此问题，因为差异数组精确地消除了边界处的贡献。 

另一种情况是完全重叠的更新，例如$[0, 10)$重复了很多次。 该值应随更新次数线性缩放，并且前缀重建可确保每次重叠都被累积而不是被覆盖。 

最后的边缘情况是查询与更新边界完全匹配。 由于该实现始终使用半开区间并压缩端点，因此在边界处精确查询不会意外地包括相邻段。
