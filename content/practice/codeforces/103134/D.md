 ## 解决方案
 ## 问题理解

 我们有一组参与者，每个参与者从某一天开始就可以使用，并且在之后的每一天都保持可用状态。 因此，每个参与者贡献一个阈值时间，并且在任何选定的一天，我们都会考虑阈值最多为该天的所有参与者。 

对于任何一天$D$，出席的参与者数量只是值的计数$a_i \le D$。 然而，只有当该计数能被 3 整除时，竞赛才有效。在所有此类有效日期中，我们想要最大可能出席人数意义上的最早一天，或者更准确地说，是产生最多参与者数量同时仍能被 3 整除的日期。如果不存在这样的一天，我们输出 -1。 

关键的微妙之处在于天数不受输入大小的限制。 每个$a_i$可以大到$10^{18}$，因此答案始终是给定值之一，或者在概念意义上可能是它们“之间”，但计数变化的唯一一次恰好是在$a_i$价值观。 两个连续不同的之间$a_i$，活跃参与者的集合不会改变。 

这已经意味着任何候选最佳日期都可以简化为不同的日期之一$a_i$，因为严格在两个值之间选择一天会产生与较小边界相同的参与者计数，但永远不会提高可行性。 

对所有可能的天数进行简单扫描是不可能的，因为范围可达$10^{18}$。 即使按排序顺序扫描所有值也只有在我们压缩重复项并仅在断点处推理时才可行。 

这里重要的边缘情况是重复值和整除性对齐。 例如，如果许多参与者在同一天都有空，则计数可能会跳跃超过 1，并完全跳过 3 的倍数。 例如，如果在某一天计数从 2 变为 4，则这两个计数之间不存在有效的日期，因为要求取决于 3 的精确倍数。 

另一个边缘情况是当最终的参与者总数不能被 3 整除时。即使最后一天包括所有人，它也可能是无效的，我们需要找到计数达到 3 倍数的最接近的较早前缀。 

## 方法

 蛮力的想法是按升序模拟日子，并计算每一天有多少参与者活跃。 由于活动仅在每次发生变化$a_i$，我们将对数组进行排序并扫描它，重新计算每个不同值的计数。 这为我们提供了精确的前缀大小，并允许检查能否被 3 整除。 

然而，由于粗心的实现，日复一日地迭代是不可能的，因为$10^{18}$范围。 即使迭代所有不同的值也可以，但每次从头开始重新计算计数会太慢，导致$O(n^2)$最坏情况下的行为。 

关键的观察是，对数组进行排序为我们提供了一个前缀结构，其中答案仅取决于前缀大小。 在已排序的位置$i$，活跃参与者的数量为$i+1$。 我们只关心索引$i+1$可被 3 整除。在这些指数中，最佳日期就是该指数的值。 如果多个值相等，我们仍然将它们视为不同的参与者，因此重复项仅影响计数，而不影响排序。 

因此，问题简化为排序和扫描一次，选择前缀大小可被 3 整除的最后一个索引。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 数天的暴力模拟 | O(10^18) | O(10^18) | O(n) | 太慢了 |
 | 排序 + 前缀检查 | O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 1. 将所有参与者可用天数读入数组，因为每个值代表一个阈值，超过该阈值后参与者将始终在场。 对这些值进行排序将使我们了解活动集如何随着时间的推移而增长。 
2. 按非降序对数组进行排序，以便我们按照参与者可用的顺序对其进行处理。 这将问题转化为跟踪前缀大小，而不是在任意天进行推理。 
3. 使用索引迭代排序数组$i$，此时活跃参与者的数量是$i+1$。 这是有效的，因为具有较小或相同阈值的所有参与者都恰好是前缀中的参与者。 
4. 对于每个索引$i$，检查是否$i+1$能被 3 整除。如果不是，请跳过它，因为那天将违反团队规模必须是 3 的倍数的约束。 
5. 每当$i+1$能被3整除，记录对应的日值$a[i]$作为候选答案。 我们总是覆盖以前的候选者，因为后来的指数对应于更多的参与者，这在目标下严格来说是更好的。 
6. 处理完所有值后，输出最后记录的有效日。 如果没有找到有效索引，则输出-1。 

### 为什么它有效

 任何一天$D$，参与者的数量正好是排序后的前缀的大小$a_i$小于或等于$D$。 由于活跃参与者的集合仅在以下情况下发生变化：$D$跨越数组中的一个值，答案中每个有意义的变化都对应于已排序的位置之一。 能被3整除的条件仅取决于前缀长度，因此检查索引可以直接捕获所有可能的有效配置，而不会丢失中间日期或需要考虑连续范围。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = list(map(int, input().split()))
    a.sort()

    ans = -1

    for i in range(n):
        if (i + 1) % 3 == 0:
            ans = a[i]

    print(ans)

if __name__ == "__main__":
    solve()
```排序步骤至关重要，因为它将问题从对任意激活时间的推理转换为对前缀大小的推理。 然后，循环直接对约束进行编码，即只有能被 3 整除的计数才有效。 最终作业始终保留最新的有效日期，该日期对应于约束下可能的最大出勤率。 

这里的一个常见错误是尝试评估所有唯一值并使用与排序顺序不同步的指针重新计算计数。 当存在重复项时，这通常会导致差一错误或错过转换。 前缀解释完全避免了这种情况。

 ## 边缘情况

 关键的边缘情况是所有值都相同。 在这种情况下，每个前缀跳转都在同一天，因此算法仍然必须单独对待每个参与者。 排序保留了这一点，并且只有前缀大小控制有效性，因此 3 前缀的最后一个倍数正确给出了答案。 

另一个边缘情况是当$n < 3$。 由于不存在大小可被 3 整除的前缀，因此算法正确地将答案保留为 -1。 

最后一个微妙的情况是，存在有效的前缀大小，但在不同的日子里无法达到。 即使多个有效前缀大小由于重复而对应于同一天的值，选择最后一次出现的情况仍然会给出正确的最大可行出席率，因为重复代表同时加入的不同参与者。
