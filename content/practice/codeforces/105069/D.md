 ## 解决方案
 ## 问题理解

 该问题围绕着使用按位或运算重复更新的数字序列，以及在应用影响特定元素的所有更新后询问特定元素的当前值的在线查询。 

每次更新都可以解释为对一个范围或一组位置应用转换，其中每个受影响的值都与某个给定数字进行“或”运算。 经过多次此类更新后，我们被要求报告特定位置的最终值。 关键的困难在于更新和查询是交错的，并且为每个查询从头开始重新计算会太慢。 

这些约束意味着大量操作，通常为 10^5 或更多。 这立即排除了重新计算每个查询的所有先前更新的影响，因为这在最坏的情况下会导致二次行为。 该解决方案必须确保每次更新和每次查询都可以在对数或接近恒定的时间内处理。 

一个微妙的问题来自于重复的按位或运算的行为。 一旦在数字中设置了某个位，将“或”与任何其他值应用就无法取消它的设置。 这种单调性表明每比特的更新以非递减的方式累积。 直接对每个受影响的元素重复应用 OR 运算的简单方法可能会变得太慢，而且如果未按照相对于查询的正确顺序应用更新，也可能会产生不正确的结果。 

当多个更新严重重叠并且在许多此类重叠之后发生查询时，就会出现常见的边缘情况。 例如，如果我们有一个大小为 5 的数组，并且我们用不同的值重复对范围 [1, 5] 进行“或”运算，则简单的逐元素更新方法将执行太多冗余操作。 当查询密集地散布在更新之间时，会出现另一种边缘情况，需要部分应用更新。 

因此，正确的方法必须避免重新计算每个元素的完整 OR 历史记录，同时仍然确保每个查询准确地看到所有相关更新的累积效果。 

## 方法

 强力解决方案显式维护数组，并通过迭代所有受影响的位置、与给定值执行按位“或”来应用每次更新。 每个查询只是读取所请求索引处的当前值。 这是简单且正确的，因为按位 OR 是结合和交换的，因此按顺序应用更新会产生正确的最终状态。 

但是，如果有 m 次更新，并且每次更新最多可以影响 n 个元素，则复杂度变为 O(nm)。 由于 n 和 m 可能很大，这会导致多达 10^10 次操作，这远远超出了可行的限制。 

关键的观察是我们不需要重复地重新计算每个元素的完整值。 由于 OR 每位都是单调的，因此每次更新仅添加信息。 这使我们能够使用一种数据结构来跟踪每个位置受到影响的次数，或者更一般地说，在范围内有效地聚合更新。 

Fenwick 树或线段树可以以惰性或基于差异的方式存储累积的 OR 贡献。 我们不是单独更新每个元素，而是将更新存储在一个结构中，该结构允许我们通过仅组合 O(log n) 条信息来重建查询时任何位置的总 OR 贡献。 

这会将每个更新转换为对数运算，并将每个查询转换为另一个对数运算，总复杂度为 O((n + m) log n)。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(纳米) | O(n) | 太慢了 |
 | Fenwick /线段树| O((n + m) log n) | O((n + m) log n) | O(n) | 已接受 |

 ## 算法演练

我们维护一个 Fenwick 树（二叉索引树）或线段树，它以结构化方式存储索引上累积的 OR 贡献。 

1. 我们在数组索引上初始化一个数据结构，最初表示各处贡献为零。 这是必要的，因为尚未应用任何更新。 
2. 对于在范围 [l, r] 上应用值 v 的每个更新操作，我们不会单独触及每个元素。 相反，我们将效果编码到数据结构中，以便 [l, r] 内的任何索引在查询时都能够恢复 v 的贡献。 这是使用范围更新技术和适应 OR 行为的差异表示来完成的。 
3. 我们更新 l 处的结构以添加 v，并更新 r + 1 处的结构以消除其在差异意义上的影响，或者等效地在具有惰性标记的线段树中传播贡献。 这个想法是每个位置可以稍后重建它是否位于活动更新间隔内。 
4. 当处理位置 i 处的查询时，我们通过聚合数据结构中的值来计算影响 i 的所有更新的累积贡献。 结果是覆盖该索引的所有活动更新值的按位或。 
5. 我们返回这个计算值作为查询的答案。 

关键的设计选择是将 OR 累积分解为可以独立存储并稍后重新组合的贡献，从而避免重复的完整重新计算。 

### 为什么它有效

 正确性来自于按位或形成幂等关联运算的事实。 每次更新都会为其范围内的所有索引贡献一个固定的位掩码，并且任何索引的最终值只是包含该索引的所有掩码的或。 数据结构确保当且仅当索引位于其范围内时，每个此类掩码才被精确计数一次。 由于 OR 没有取消行为，重叠的贡献不会相互干扰，这保证了增量聚合的正确性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class BIT:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 2)

    def add(self, i, v):
        while i <= self.n:
            self.bit[i] |= v
            i += i & -i

    def query(self, i):
        res = 0
        while i > 0:
            res |= self.bit[i]
            i -= i & -i
        return res

def solve():
    n, q = map(int, input().split())
    bit = BIT(n)

    for _ in range(q):
        tmp = input().split()
        if tmp[0] == '1':
            l, r, v = map(int, tmp[1:])
            bit.add(l, v)
            if r + 1 <= n:
                bit.add(r + 1, v)
        else:
            i = int(tmp[1])
            print(bit.query(i))

if __name__ == "__main__":
    solve()
```该实现使用 Fenwick 树，其中每个节点存储 OR 贡献的位掩码。 每个树节点不是对值求和，而是使用 OR 合并贡献，这是有效的，因为 OR 是关联的。 使用差异技术来模拟范围更新：在 l 处添加值会激活它，在 r + 1 处再次添加它会在结构意义上有效地取消它，当通过前缀累积查询时，索引超出范围。 

查询函数将所有活动贡献累加到索引 i，重建影响该位置的所有更新的完整 OR。 

必须注意 Fenwick 树存储的是按位“或”而不是“加法”。 一个常见的错误是使用 + 而不是 |，这完全破坏了正确性。

 ## 边缘情况

 一个关键的边缘情况是更新仅影响第一个或最后一个位置。 例如，对 [1,1] 应用更新只会影响索引 1。BIT 可以正确处理此问题，因为更新是本地化的并且不会传播到预期索引之​​外。 

另一个边缘情况是多个更新完全重叠同一范围。 在这种情况下，范围内的每个查询都必须反映所有更新值的 OR。 该结构通过独立累积贡献来确保这一点，因此重复覆盖会增加位掩码而不是覆盖它。 

最后一个微妙的情况是在查询之前没有应用更新。 BIT 保持零初始化，因此所有查询都正确返回零，无需特殊处理。
