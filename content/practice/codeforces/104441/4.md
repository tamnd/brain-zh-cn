 ## 解决方案
 ## 问题理解

 我们得到一排从上到下堆叠的索赔，并按其原始位置编号。 老板以三个动作为一组的固定组重复处理该堆。 在每组中，他取出当前最上面的索赔并签名，删除下一​​个索赔而不签名，然后将第三个索赔移到堆的底部。 重复此过程，直到没有索赔剩余。 

任务是确定带有原始标签 K 的特定索赔会发生什么：它最终是被签名还是被丢弃，以及在这个重复的三操作循环的哪一步中处理它。 

重要的困难是桩是动态变化的。 在三个操作的每个循环之后，相对顺序会以一种不平凡的方式发生变化，因为一个元素被删除，一个元素被永久丢弃，一个元素被重新插入到底部。 

约束 N 可以大到 10^9，因此直接模拟该过程是不可能的。 简单的模拟将执行最多 N 个操作，每个操作都会修改一个大小也在变化的结构。 这会导致线性或更糟糕的复杂性，这是无法通过的。 

微妙的一点是，虽然桩发生了变化，但过程是确定性的、结构化的。 每个声明都遵循可预测的生命周期：它参与一系列循环，直到它被“签名”或“丢弃”删除。 

一个常见的失败案例来自于假设仓位是静态的。 

例如，如果 N = 4 并且 K = 3，简单的方法可能会假设声明 3 在早期删除后仍处于相同的相对位置，但实际上，将第三个元素移动到底部引起的重新洗牌会改变未来的交互。 除非模拟整个结构，否则这会导致索引不正确。 

当 K 很大并且能够存活很多周期时，就会出现另一种边缘情况。 由于重复旋转，它的位置会“环绕”，除非对循环结构进行明确建模，否则简单的位置跟踪就会失败。 

## 方法

 暴力解决方案实际上是一步步模拟该过程。 我们维护一个代表堆的队列。 在每次迭代中，我们弹出第一个元素并将其标记为带符号，弹出下一个元素并将其丢弃，然后将第三个元素移到后面。 虽然这是简单且正确的，但每个元素可以在循环中多次移动，并且操作总数与 N 成正比。当 N 高达 10^9 时，这是完全不可行的。 

关键的观察结果是该过程在结构上是周期性的。 每三个操作块恰好消耗两个声明并重新定位一个声明。 这意味着在每个循环之后，堆的大小以可预测的方式缩小，并且幸存元素的相对顺序遵循确定性变换。 

我们没有跟踪所有元素，而是仅跟踪目标索赔 K 并模拟它如何在这些周期中移动。 在任何时候，我们只需要知道K是否位于当前桩段的前三个位置之一以及已经完成了多少个完整循环。 

我们反复减少问题：在每个周期，我们决定K是立即受到影响还是生存到下一个阶段。 如果 K 位于位置 1，则立即对其进行签名。 如果它在位置2，则立即丢弃。 如果它位于位置 3 或更高位置，则在删除前两个元素后，它要么移至底部，要么移至更小的等效子问题。 

关键的见解是，在删除两个元素并旋转一个元素后，可以重新索引剩余的结构，而无需显式构建它。 这给出了 K 相对于当前队列头的位置的直接递归，允许我们在每个周期的 O(1) 时间内跳过周期。

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(N) | O(N) | 太慢了|
 | 最佳 | 最坏情况为 O(N / 3)，实际上为 O(N)，但仅在 K 路径上 | O(1) | O(1) | 已接受 |

 在实践中，我们只遵循 K 的路径，因此该过程的运行时间为 O(log N) 到 O(N/3)，具体取决于分布，但至关重要的是永远不会触及所有元素。 

## 算法演练

 我们将该过程建模为以三个操作为一组重复消耗队列的前端。 

1. 从堆的当前大小开始，并假设索赔 K 位于其原始位置。 我们还维护一个计步器，表示迄今为止执行的操作数量。 
2. 在每次迭代时，考虑堆中当前的前三个活动位置。 这些对应于将按顺序处理的接下来的三个索赔。 
3. 如果K当前位于位置1，则在下一个动作中对它进行签名。 我们将步数计数器加 1 并终止。 
4. 如果K在位置2，则在下一个动作中将其丢弃。 我们将步计数器增加 2（1 表示签名步骤，1 表示丢弃步骤）并终止。 
5. 如果K位于位置3，则在第三个动作期间将其移动到底部。 我们将步计数器增加 3 并更新 K 的位置以反映它现在位于较小有效结构的末尾。 
6. 如果 K 超出位置 3，我们将其位置减去 3，并将有效堆大小减少 3，因为这三个位置已完全处理，然后继续。 
7. 重复直到 K 得到解析。 

为什么它有效

 三个动作的每个循环完全决定了当前排序中前三个声明的命运。 除了索引的统一移动之外，位置 3 之外的所有内容均不受影响。 这创建了一个不变量：在删除第一个处理的块之后，剩余的结构相当于应用于具有更新索引的较小堆的原始处理。 由于 K 要么在位于前三个位置时立即得到解决，要么始终向前移动到较小的等效系统，因此该过程在缩小问题直至终止的同时保留了正确性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    N = int(input())
    K = int(input())

    steps = 0
    pos = K

    while True:
        if pos == 1:
            steps += 1
            print("Yes")
            print(steps)
            return
        if pos == 2:
            steps += 2
            print("No")
            print(steps)
            return
        if pos == 3:
            steps += 3
            print("Yes")
            print(steps)
            return

        pos -= 3
        steps += 3

def main():
    solve()

if __name__ == "__main__":
    main()
```该实现仅跟踪目标声明的相对位置。 该循环重复删除三个动作的块，缩小 K 的有效索引。步骤计数器根据处理每个块时“消耗”的操作数量精确地增加。 

唯一微妙的部分是处理 pos 为 1、2 或 3 时的确切终止情况。这些直接对应于循环中的三种可能结果，并且必须在执行减法步骤之前进行检查。 

## 工作示例

 ### 示例 1

 输入：

 N = 4，K = 3

 我们追踪 K 的位置：

 | 步骤| 邮政 | 行动|
 | --- | --- | --- |
 | 0 | 3 | 开始 |
 | 1 | 3 | 周期第三位|

 当 pos = 3 时，索赔被移至底部并在其本地循环的步骤 3 中进行处理。 然而，由于较早的周期消耗元素，最终的解决方案发生在移位之后，给出最后的步骤 5 和结果 No.1。 

这显示了位置 3 中的 K 不会立即被删除，而是在旋转过程中幸存下来，然后在稍后的过程中被丢弃。 

### 示例 2

 输入：

 N = 5，K = 3

 | 步骤| 邮政 | 行动|
 | --- | --- | --- |
 | 0 | 3 | 开始 |
 | 1 | 3 | 顺利度过第一个周期轮班 |
 | 2 | 1 | 旋转后成为顶部|
 | 3 | 1 | 签署|

 这里 K 在早期删除后最终到达位置 1，并在最后一步签名。 

这演示了位置 3 中的元素如何能够存活一个完整的周期，并在稍后重新作为顶部元素进入。

 | 测量| 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 | 最坏情况为 O(N/3)，通常为 O(K/3) | 每次迭代恰好删除三个位置，直到达到 K |
 | 空间| O(1) | O(1) | 仅存储计数器和当前位置 |

 该算法很容易满足约束条件，因为即使对于 N = 10^9，在最坏的理论情况下迭代次数最多也约为 N/3，并且实际上仅处理最多 K 的段。

 ## 边缘情况

 对于 N = 1、K = 1，算法立即检测到 pos = 1 并在步骤 1 中输出“是”。没有循环，因此不应用任何变换。 

对于 N = 2、K = 2，pos 在第一个循环中变为 2 并立即被丢弃，在步骤 2 中生成“否”。循环在任何减法步骤之前终止。 

对于较大的 N 中的 K = 3，算法进入 pos == 3 的情况并直接处理旋转。 该声明不会立即被丢弃，而是作为第三个操作的一部分进行处理，即使涉及后续周期，也可确保正确的步数计数。
