 ## 解决方案
 ## 问题理解

 我们得到了一系列食品包装，每个包装都标有一种类型。 还有$n$探险的参与者，时间以天为单位。 每天，每个参与者只消耗一包。 关键的限制是参与者必须在整个探险过程中坚持单一食物类型，尽管不同的参与者可能会选择不同的类型。 

因此，对于每个参与者，我们有效地分配一种食物类型，然后我们必须检查仅使用可用的包装可以每天为每个人提供食物多久。 参与者指定类型$x$消耗一包类型$x$每天，所以如果我们分配$k$参加者输入$x$，然后我们度过的每一天$k$包类型$x$。 因此，天数受到每种类型的总包裹数量的限制。 

输入给出参与者的数量$n$, 可用包的数量$m$，以及包类型列表。 输出是我们可以维持此类任务的最大全天数。 

约束很小，$n, m \le 100$，因此即使是尝试许多候选答案或重复重新计算计数的解决方案也能顺利通过。 这表明该解决方案的结构更多的是关于频率的推理，而不是优化繁重的计算。 

当总供应量极度倾斜时，就会出现微妙的边缘情况。 例如，如果所有包裹都是一种类型，但有很多参与者，那么答案就受到可以分配到该类型的人数的限制。 另一种极端情况是每种类型出现的次数都很少。 即使有很多参与者，限制因素也变成了跨类型可用贡献的总和，而不是参与者的数量。 

一个常见的错误是认为每个参与者根据频率独立地贡献天数，而不考虑将更多参与者分配给某种类型会线性增加每日消耗，这可以完全消除天真的贪婪分配的可行性。 

## 方法

 天真的想法是固定几天$d$，然后检查我们是否可以为每个参与者分配一种食物类型，使得每个分配的类型至少有$d$每个参与者的包裹。 如果出现类型$c$次，最多可以支持$\lfloor c / d \rfloor$参与者$d$天。 因此可行性就变成了检查所有类型的总和是否$\lfloor c_i / d \rfloor$至少是$n$。 

如果我们尝试一切可能$d$，答案最多是$m$，所以检查每个$d$最多$m$通过频率扫描给出了一个解决方案$O(m \cdot 100)$，这在约束条件下是微不足道的。 这已经有效了。 

然而，关键的见解是我们不需要迭代所有$d$。 函数“支持的参与者数量”$d$天”是单调递减的$d$。 如果$d$可行，则所有较小的值也可行。 这种单调性允许对答案进行二分搜索。 

所以不要尝试所有$d$，我们二分查找最大值$d$，并为每个候选人使用频率计数计算支持的参与者总数。 每张支票都是$O(100)$，并且搜索是$O(\log 100)$。

 ## 算法演练

 1. 计算每种食品类型有多少个包装。 我们将频率存储在按类型索引的数组中。 
2. 定义函数`can(d)`检查我们是否能够维持$d$天。 对于每种类型和频率$c$，我们计算它可以支持多少参与者$d$天，即$c // d$。 我们将这些值相加。 
3. 如果支持的参与者总数至少为$n$， 然后$d$天是可行的，因为我们可以在不超过供应量的情况下将参与者分配给类型。 
4. 二分查找$d$从 1 到$m$，跟踪最大可行值。 
5.输出最大可行值$d$。 如果即使$d = 1$失败，输出0。 

步骤 3 有效的关键原因是每种类型都像一个资源池，分配一个参与者每天消耗一个单位。 按类型对参与者进行分组可以最大限度地重用相同的消费模式，因此最好的策略始终是将尽可能多的参与者打包到高频类型中。 

### 为什么它有效

 在任何固定天数$d$，一个类型$c$包最多可以支持$\lfloor c / d \rfloor$参与者。 这个界限很紧，因为每个参与者都准确地消费$d$远征期间的单位。 对类型求和得出可以同时分配有效食物类型的参与者总数。 如果这个总和至少是$n$，我们可以贪婪地分配参与者，因为类型之间不存在超出计数能力的交互。 这确保了可行性与计算条件完全匹配，从而使二分搜索决策函数正确。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    a = list(map(int, input().split()))
    
    freq = [0] * 101
    for x in a:
        freq[x] += 1
    
    def can(d):
        total = 0
        for c in freq:
            total += c // d
        return total >= n
    
    lo, hi = 1, m
    ans = 0
    
    while lo <= hi:
        mid = (lo + hi) // 2
        if can(mid):
            ans = mid
            lo = mid + 1
        else:
            hi = mid - 1
    
    print(ans)

if __name__ == "__main__":
    solve()
```频率数组将问题压缩为恒定大小的状态，因为值的界限为 100。可行性检查使用整数除法来直接建模每种食物类型在给定天数内可以维持多少个完整参与者分配。 然后，二分查找反复细化候选答案。 

一个常见的陷阱是在迭代期间迭代原始列表而不是频率数组`can`，这会错误地独立对待每个包并破坏使划分有意义的分组逻辑。

 ## 边缘情况

 当只有一个参与者时，答案会减少到任何单一类型的最大频率，因为该参与者可以选择最丰富的食物。 该算法处理这个问题是因为$d > \max c_i$，所有贡献都变为零，可行性立即失败，留下正确的最大值。 

当总体上只有一个包但参与者很多时，每个$d \ge 1$产生的总分配为零，因此所有的可行性都失败$d$，二分查找正确返回 0。 

当所有包都相同时，可行性就变得简单$\lfloor m / d \rfloor \ge n$，这正是分频的计算结果。 该算法自然地捕获了这一点，无需特殊的外壳，因为只有一个频率是非零的。 

当分布稀疏时，楼层总和可以防止过度计数。 即使包裹总数超过$n \cdot d$，不均匀分布仍然会使分配变得不可能，并且按类型划分正确地强制执行该约束。
