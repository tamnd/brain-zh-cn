 ## 解决方案
 ## 问题理解

 我们给出了一个非常长的整数序列，但没有明确提供。 相反，它以压缩形式作为等值游程给出。 每次运行都会显示一个值`v`被重复`l`次，并且相邻的运行总是具有不同的值。 展开所有运行将给出一个序列`A`长度`k`。 

我们被要求考虑如何将这个完整的序列准确地分割成`r`连续的非空子数组。 对于每个分割，每个子数组都有一个和，我们根据每个子数组的和是偶数还是奇数对每个子数组进行分类。 让`p`是具有奇数和的子数组的数量并且`q`是总和为偶数的数，所以`p + q = r`。 

对于每个查询`r`，我们不构造一个最佳分割。 相反，我们考虑将数组拆分为的所有可能方法`r`部分，我们提出两个单独的极值问题：奇数和段的最大可能数量，以及偶数和段的最大可能数量。 

关键的困难在于数组长度和总扩展大小都可能非常大，最多可达`2 × 10^5`运行至`2 × 10^5`查询，总扩展长度也可能很大。 任何显式构建数组或尝试所有分区点的解决方案都是不可能的。 游程的结构以及只有奇偶性才重要的事实表明，实际值除了奇数或偶数之外无关紧要。 

一种简单的方法是扩展数组，然后对分区计数上的每个查询使用动态编程，但这会太慢，因为即使单个 DP 超过`k`在最坏的情况下，每个查询将是二次的。 

当运行量很大并且值频繁交替奇偶校验时，就会出现微妙的边缘情况。 例如，像这样的序列`1,1,1,2,2,3,3,3`可以有多个仅依赖于奇偶校验转换的最优分割策略，并且在不仔细跟踪前缀和的情况下，运行边界处的贪婪分割并不总是足够的。 

## 方法

 暴力解释是扩展序列并尝试所有可能的放置方式`r-1`之间切入`k-1`职位。 对于每个结果分区，计算段总和并计算有多少个是奇数或偶数。 这本质上是组合的，并且增长得像`O(C(k, r))`仅用于枚举分割，甚至计算每个分割成本`O(k)`。 即使对于`k = 200000`。 

一旦我们注意到只有奇偶校验才重要，我们就可以大大简化。 每个元素贡献要么`0`或者`1`模 2。当其元素的 XOR 为 1 时，段和恰好为奇数。这将每个段评估减少为前缀奇偶校验差异。 

现在问题变成了：我们正在切割一个二进制数组，每个段都通过其端点的异或来标记。 重要的观察结果是，奇数段的数量仅取决于有多少切割位置在前缀奇偶校验中产生翻转。 

这导致了标准压缩：我们只关心运行中的奇偶校验结构。 在一系列相同值中，如果值为奇数，奇偶校验会按预期交替变化；如果值为偶数，则奇偶校验保持稳定。 因此，每次运行都会贡献一个结构化的奇偶校验更改块，无需扩展即可对其进行汇总。 

一旦我们压缩到奇偶校验运行并在运行边界上跟踪前缀奇偶校验，问题就简化为决定可以放置多少次切割来最大化所需奇偶校验的片段。 最优策略在运行边界上变得贪婪：每次我们可以启动一个奇偶校验与目标匹配的段时，我们都会这样做，并跟踪由于奇偶校验约束而剩余多少不可避免的“坏”段。 

最终的解决方案本质上是在选择时计算每个奇偶校验可以强制有多少段`r`段，这取决于存在多少个等前缀奇偶校验状态之间的转换。 

关键的加速是每次运行都会贡献 O(1) 状态转换，并且在预处理前缀奇偶校验结构后，每个查询都可以在 O(1) 内得到答复。

 ## 算法演练

 1. 将每个值替换为其奇偶校验，因为只有奇数/偶数对和而言才重要。 这会将数组转换为二进制序列，其中每个元素都是 0 或 1。 
2. 从概念上讲，在扩展数组上构建前缀奇偶校验，但直接在运行中计算它。 扫描运行时，在每次运行后维护累积奇偶校验，而不扩展它。 连续的偶数值不会改变奇偶性； 一系列奇数值翻转奇偶校验`l mod 2`。 
3. 将运行结构转换为“有效奇偶校验翻转”序列。 每次运行都会贡献一次翻转或不翻转，具体取决于其值是否为奇数以及其长度是否为奇数。 这将问题简化为压缩奇偶校验转换序列。 
4. 在这个压缩结构上预先计算两个数组：一个描述如果我们贪婪地尝试最大化奇数和段，我们可以制作多少个段，另一个描述偶数和段。 这是通过跟踪我们可以在前缀奇偶校验与所需的段奇偶校验对齐的位置进行剪切的次数来完成的。 
5. 对于每个查询`r`，计算最好的可能`p`抓住尽可能多的“好”切入机会`r`，受到结构约束的限制。 然后计算`q = r - p`。 同样，计算最好的可能`q`独立地使用对称推理。 
6. 输出每个查询的两个值。 

它起作用的原因与每个段完全由前缀奇偶校验数组中端点之间的奇偶校验差异决定的不变量有关。 由于前缀奇偶校验是由序列固定的，因此每次切割仅决定我们是否使用奇偶校验与目标匹配的前缀索引。 运行压缩准确地保留了所有前缀奇偶校验转换，因此不会丢失最佳分割。 贪婪选择是有效的，因为每个段决策仅取决于本地前缀奇偶校验可用性，并且早期的选择永远不会将未来的可用性增加到超出预先计算结构中已考虑的范围。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def build_prefix_info(runs):
    # runs: list of (value_parity, length)
    # convert each run into parity flip contribution
    flips = []
    for val, length in runs:
        if val % 2 == 0:
            flips.append(0)
        else:
            flips.append(length % 2)

    # prefix parity after each run boundary
    pref = [0]
    cur = 0
    for f in flips:
        cur ^= f
        pref.append(cur)

    # count how many times each parity occurs at run boundaries
    cnt0 = sum(1 for x in pref if x == 0)
    cnt1 = len(pref) - cnt0
    return cnt0, cnt1

def solve():
    n, m = map(int, input().split())
    runs = []
    for _ in range(n):
        v, l = map(int, input().split())
        runs.append((v, l))

    cnt0, cnt1 = build_prefix_info(runs)

    # In a binary prefix parity array, best number of segments of parity X
    # is bounded by number of prefix positions with that parity minus 1
    max_odd = max(cnt0, cnt1) - 1
    max_even = len(runs) - max_odd

    out = []
    for _ in range(m):
        r = int(input())
        # split r segments between odd and even
        # maximize independently under global cap
        p = min(r, max_odd)
        q = min(r, max_even)
        out.append(f"{p} {q}")

    print("\n".join(out))

if __name__ == "__main__":
    solve()
```该实现首先将每次运行压缩为是否有助于奇偶校验翻转。 只有具有奇数值的运行才能更改奇偶校验，并且偶数长度的奇数运行在内部抵消，因此每次运行都成为单个二进制切换。 运行边界上的前缀奇偶校验数组捕获所有可能的重要段端点。 

前缀奇偶校验的计数为0或1决定了有多少个段端点可以产生奇数或偶数段。 由于每个段对应于选择两个具有奇偶校验差异的前缀索引，因此给定类型的最大段数受到存在多少个正确奇偶校验的端点的限制。 

然后通过夹紧来回答每个查询`r`独立于可实现的最大奇数或偶数段计数。 这反映了这样一个事实：一旦我们耗尽给定奇偶校验类的所有可用端点，无论我们如何分割，我们都无法创建该类型的附加段。 

## 工作示例

 考虑扩展序列为的样本输入`5, 5, 5, 2, 2, 7`。 

我们只处理奇偶校验，所以这变成`1,1,1,0,0,1`。 

元素的前缀奇偶校验：

 | 我| 价值| 前缀奇偶校验|
 | --- | --- | --- |
 | 0 | - | 0 |
 | 1 | 1 | 1 |
 | 2 | 1 | 0 |
 | 3 | 1 | 1 |
 | 4 | 0 | 1 |
 | 5 | 0 | 1 |
 | 6 | 1 | 0 |

 为了`r = 2`，最好的奇数段结构对应于选择对齐前缀奇偶校验的切割，因此两个段都具有 XOR 1，这在结构中可能两次。 对于偶数段，我们选择前缀奇偶校验匹配的剪切，也产生两个偶数段。 

现在考虑一个更简单的情况：`A = [1, 2, 3]`, 扩展奇偶校验`[1,0,1]`。 

前缀奇偶校验：

 | 我| 前缀 |
 | --- | --- |
 | 0 | 0 |
 | 1 | 1 |
 | 2 | 1 |
 | 3 | 0 |

 为了`r = 2`，我们可以在索引 1 或 2 处拆分。选择在 1 处拆分会给出段`[1]`和`[2,3]`奇偶校验为 1 和 1。这显示了选择剪切位置如何改变两者`p`和`q`直接通过前缀奇偶校验差异。 

这些跟踪证实段奇偶校验完全由前缀奇偶校验对齐控制，而不是由元素值本身控制。

 ## 边缘情况

 使用奇值的单次运行测试前缀奇偶校验逻辑是否正确处理强制每次拆分的最小结构。 由于只有一个元素，因此如果值为奇数，则对一个段的任何划分都会产生单个奇数和，并且偶数段为零。 

全偶序列，例如`2,2,2,2`根本不产生奇偶校验翻转。 前缀奇偶校验数组保持不变，因此任何分段仅产生偶数和。 在这种情况下，算法不得错误地计算幻象奇数段。 

严格交替奇偶校验模式，例如`1,2,1,2,1,2`产生最大的奇偶校验变化。 这里每个切割位置都会改变前缀奇偶校验结构，并且算法必须正确计算存在多少个有效端点以构造奇数和偶数段，而无需重复计算转换。 

最后，案例`r = k`强制每个元素成为一个单独的段。 在这种情况下，答案简化为计算单个元素奇偶校验，并且运行压缩中的任何错误都会立即产生不正确的总数。
