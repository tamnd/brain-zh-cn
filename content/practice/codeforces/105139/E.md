 ## 解决方案
 ## 问题理解

 每个测试用例都描述了一个编程竞赛的简单餐饮计划。 有$n$参赛者总数。 确切地$x$他们中的一些人选择了烤鸡肉汉堡套餐，而其余的人则选择了烤鸡肉汉堡套餐。$n - x$选择辣鸡肉汉堡套餐。 辣套餐的价格是$a$，烤套餐的价格是$b$。 任务是计算为所有参赛者准备食物的总成本。 

该结构纯粹是累加性的：每个参赛者根据自己的选择恰好贡献一个固定成本，因此总成本是通过计算每个类别的数量并乘以相应的价格来确定的。 

约束允许最多$10^4$测试用例，每个测试用例使用的值高达$10^4$。 这立即意味着任何解决方案都必须在每个测试用例中以恒定的时间运行。 即使每个测试用例进行线性扫描也会产生不必要的开销，但在技术上仍然是安全的； 然而，任何涉及嵌套循环的事情$n$如果$T$很大。 

这种形式的问题中唯一微妙的问题是具有固定宽度整数的语言中的整数溢出，但在 Python 中这不是问题。 另一个可能的错误是交换计数或误解哪个组对应于哪个价格。 具体的失败案例是$n = 5, x = 5, a = 10, b = 1$。 正确答案是$0 \cdot 10 + 5 \cdot 1 = 5$。 错误的实现可能会错误地应用$a$到烧烤组，生产$50$，这显然是错误的。 

## 方法

 一种直接的方法是单独模拟每个参赛者。 对于每个$n$参赛者，我们决定他们是否属于$x$烤的并添加其中之一$a$或者$b$到运行总计。 这是有效的，因为它准确地反映了问题定义。 然而，它执行$n$每个测试用例的操作，导致$T \cdot n$最坏情况下的操作。 两个值都达到$10^4$，这样可以达到$10^8$对于这样一个简单的算术问题来说，这是不必要的。 

关键的观察是我们永远不需要知道个别参赛者的身份。 只有计数才重要。 一旦我们准确地认识到这一点$x$人们付钱$b$确切地说$n - x$人们付钱$a$，整个问题分解为一个表达式。 这消除了所有迭代并将每个测试用例减少到恒定数量的算术运算。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力模拟|$O(n)$每个测试用例|$O(1)$| 太慢了 |
 | 直接配方|$O(1)$每个测试用例|$O(1)$| 已接受 |

 ## 算法演练

 1.读取测试用例数量$T$。 每个测试用例都是独立的，因此它们之间不传递任何状态。 
2. 对于每个测试用例，读取整数$n, x, a, b$。 这些完全决定了成本的构成。 
3. 计算辣度集的数量为$n - x$。 这是烤组的补充。 
4. 计算总成本为$(n - x) \cdot a + x \cdot b$。 这直接来自每个小组贡献的总和。 
5. 立即输出测试用例的计算值，然后再转到下一个测试用例。 

### 为什么它有效

 该算法依赖于这样一个事实：总成本是独立项目的线性总和。 每个参赛者都为总和贡献了一项，该项的值仅取决于二元分类：辣的或烤的。 由于每个类别的计数是固定且不相交的，因此用乘法替换单独的求和可以保持精确的相等性。 不存在隐藏的依赖关系或排序效应，因此聚合到计数中不会丢失信息。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        n, x, a, b = map(int, input().split())
        spicy = n - x
        total = spicy * a + x * b
        print(total)

if __name__ == "__main__":
    solve()
```该解决方案读取每个测试用例并直接应用导出的公式。 唯一重要的实现细节是正确计算$n - x$对于辣组。 由于 Python 支持大整数，因此乘法顺序在 Python 中无关紧要，但在固定宽度整数语言中，需要使用 64 位整数。

 ## 边缘情况

 当$x = 0$，没有人选择烤套餐。 该公式简化为$n \cdot a$。 该算法计算$n - x = n$，因此辣组正确地吸收了所有参与者，并且烤术语消失了。 

什么时候$x = n$，大家都选择烤套餐。 辣数变为零，总数变为$n \cdot b$。 减法步骤确保辛辣贡献恰好为零，无需特殊外壳。 

什么时候$n = 1$，该公式仍然表现一致，因为$x = 0$或者$x = 1$，并且只有一项有贡献。 该算法不需要分支，这避免了边缘情况条件并保持实现的统一。
