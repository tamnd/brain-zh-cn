 ## 解决方案
 ## 问题理解

 我们举办了一个有固定人数客人的聚会，每位客人都准确地说出了他们满意的一种纸杯蛋糕口味。 每种口味由 1 到 M 之间的整数标识。Alice 只能购买 12 个一包的纸杯蛋糕，这意味着如果她订购 x 几十个某种口味，她就会收到 12 个该口味的纸杯蛋糕。 

要求是，对于每种口味，爱丽丝必须购买足够的纸杯蛋糕，以便至少与喜欢该口味的客人数量一样多的该口味的纸杯蛋糕。 对于所有 M 口味，输出不是纸杯蛋糕的数量，而是每种口味的几十个纸杯蛋糕的数量。 

关键的转变在于，问题简化为计算每种口味的需求，然后将每种口味的需求独立地四舍五入到最接近的 12 的倍数。 

约束足够小，直接频率计数就足够了。 N 最大为 10000，M 最大为 100，我们可以安全地扫描所有访客一次并在线性时间内计算频率。 即使稍微昂贵的方法也可以通过，但任何涉及对每个口味的客人进行嵌套迭代的方法都将是不必要的矫枉过正。 

当某种口味的客人数量正好可以被 12 整除而不是不能被 12 整除时，就会出现一个微妙的失败案例。 例如，如果 12 位客人想要口味 1，那么正好 1 打就足够了。 如果13位客人想要口味1，那么1打不够，我们必须买2打，给24个纸杯蛋糕，尽管只需要13个。 在这些情况下，使用整数除法而不向上舍入进行除法的简单方法会低估。 

另一个极端情况是根本没有客人想要某种口味。 该风味的正确输出是 0 打。 任何错误地初始化计数或忘记处理零频率风格的实现都可能由于盲目应用舍入逻辑而意外输出 1。 

## 方法

 思考这个问题的一种强力方法是模拟逐步购买纸杯蛋糕。 对于每种口味，我们可以反复增加几十个，直到该值的 12 倍至少达到所需的客人数量。 这是正确的，因为它直接强制执行约束，但在最坏的情况下，我们可能会为每种风格迭代最多 N/12 个步骤。 对于 M 种口味，这变成了 O(M * N)，这在这里仍然可以接受，但不必要是间接的。 

更直接的观察是，每种味道都是独立的。 口味之间不存在耦合，因为纸杯蛋糕是特定于口味的，并且每个口味都有限制。 这意味着我们只需要每种口味的频率计数，然后将每个计数转换为除以 12 的上限。关键的简化是认识到面包店约束仅引入舍入，而不引入交互。 

一旦我们接受每种风格都简化为单个算术运算，问题就变成了一个简单的计数任务，然后是整数上限除法。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(M * N / 12) | O(M)| 结构太慢，没必要 |
 | 最佳| O(N + M) | O(M)| 已接受 |

 ## 算法演练

 ### 步骤

 1. 读取 N 和 M，然后读取客人偏好列表。 

这直接表示了每种口味的需求，这是与输出相关的唯一信息。 
2.初始化一个数组`cnt`大小 M+1 到零。 

每个索引对应一种口味，我们累计有多少客人需要它。 
3. 迭代所有客人并递增`cnt[f_i]`对于每个偏好。 

这将整个输入压缩为每种口味的需求计数。 
4. 对于从 1 到 M 的每种口味 i，计算所需的数十个：`(cnt[i] + 11) // 12`。 

这是整数上限除以 12，确保我们永远不会少买纸杯蛋糕。 
5. 在其自己的行上输出每个计算值。 

### 为什么它有效

 每种口味的需求和供应都是完全独立的。 唯一的限制是每单位购买 12 个纸杯蛋糕。 对于任何非负整数需求 d，使得 12x ≥ d 的最小整数 x 恰好是 d/12 的上限。 由于没有一种口味可以替代另一种口味，因此单独解决每种口味会产生全局最优解决方案，并且购买的纸杯蛋糕总量最少。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    arr = list(map(int, input().split()))

    cnt = [0] * (m + 1)

    for f in arr:
        cnt[f] += 1

    out = []
    for i in range(1, m + 1):
        out.append(str((cnt[i] + 11) // 12))

    sys.stdout.write("\n".join(out))

if __name__ == "__main__":
    solve()
```该实现首先构建一个频率表，以便所有后续计算每种风格的复杂度为 O(1)。 使用`(cnt[i] + 11) // 12`是没有浮点运算的上限除法的标准整数技巧。 

一个微妙的点是索引：口味从 1 开始，因此频率数组的大小必须为 M+1，而不是 M。这可以避免客人选择口味 M 时出现差一错误。

 ## 边缘情况

 一个关键的边缘情况是某种口味的需求为零。 例如：

 输入：```
5 3
1 1 1 1 1
```计数数组变为cnt[1]=5，cnt[2]=0，cnt[3]=0。 应用上限除法得到 1, 0, 0。 

该算法正确地为口味 2 和 3 生成零，因为`(0 + 11) // 12 = 0`。 一个有缺陷的实现错误地强制每种口味至少 1 打，即使没有客人请求该口味，也会错误地输出正值。 

另一个边缘情况是 12 的精确倍数。对于 cnt[i]=12，我们得到`(12+11)//12 = 1`，符合一打就完全足够的事实。 这证实了舍入逻辑在干净除法的情况下不会过度购买。
