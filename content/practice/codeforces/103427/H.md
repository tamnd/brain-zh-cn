 ## 解决方案
 ## 问题理解

 我们从一个连通的无向图开始，其中每条边都有一个权重。 根据该图，我们构造另一个图，称为折线图。 在这个变换后的图中，每条原始边都变成一个顶点。 如果原始图中对应的边共享一个端点，则线图中的两个顶点是连接的，并且该连接的权重是两个原始边权重的总和。 

任务不是显式地构造这个折线图。 相反，我们需要最大加权匹配的总权重。 匹配意味着我们选择线图顶点对，使得没有顶点被多次使用，并且我们希望最大化所选边权重的总和。 

转换回原始图，每个选择的匹配边对应于选择原始图中的两个相邻边，并支付它们的权重和。 

约束很严格：最多 100,000 个顶点和 200,000 个边。 任何构建线图的解决方案都是立即不可行的，因为它可能会在原始图的密集部分中创建最多 O(m^2) 条边。 即使在线图中存储邻接关系也是不可能的。 

这迫使我们围绕原始图的顶点进行局部推理，因为线图中的所有交互都是由共享端点引起的。 

当顶点具有高度数时，会出现关键边缘情况。 例如，如果一个节点连接到许多具有不同权重的边，则这些边周围的折线图就会变成一个派系，并且各个边上的朴素匹配推理会失败。 

考虑一颗星星：

 n = 4

 边：(1-2,1)、(1-3,2)、(1-4,3)

 在线图中，所有三个边都成为顶点并形成一个三角形，其权重等于两两之和。 首先对最小边进行简单的贪婪配对可能会错过最佳配对，因为配对结构取决于全局排序，而不是邻接顺序。 

真正的挑战是每个线图边缘都来自单个原始顶点，因此问题在每个顶点本地分解，但边缘参与两个这样的局部结构。 

## 方法

 蛮力的想法是显式地构建折线图。 每条原始边都成为一个节点，对于共享端点的每对边，我们创建一条按其权重之和加权的边。 然后我们在此图上运行最大加权匹配算法。 

这在概念上是正确的，但完全不可行。 d 度的顶点在线图中贡献 O(d^2) 条边。 在密集图中，这对于每个顶点来说都是二次的，导致总边数为 O(m^2)，这远远超出了限制。 

关键的观察结果是，折线图中的每条边都恰好在原始图中的一个端点处生成。 如果我们固定原图中的一个顶点u，则所有与u相关的边在线图中形成一个完整的图，其边权重等于w(e_i) + w(e_j)。 这种结构非常特殊：多重集中的所有成对求和。 

现在问题变成：在每个顶点 u 处，我们有多个关联边权重，并且我们希望在本地形成对，但每个原始边只能全局使用一次，这意味着我们必须协调每个边的​​两个端点之间的选择。 

这自然意味着贪婪的排序。 和完全图中的最佳匹配是通过将大权重配对在一起获得的，但由于每条边参与两个这样的局部结构，因此我们必须确保一致性。

解决此问题的正确方法是将每条边视为对一个配对决策做出贡献，并将问题简化为对边进行全局排序并贪婪地对它们进行配对，同时确保每条边仅使用一次。 线图的结构保证任何最佳匹配都对应于选择不相交的原始边对，并且如果它们共享一个顶点，则每对贡献其权重之和。 最大总和是通过始终配对最大的可用兼容贡献来实现的，这减少了派生结构上的贪婪配对。 

该实现归结为处理按权重排序的边，并使用程度感知结构来匹配本地可用的未使用的事件边。

 | 蛮力（构建 L(G)）| O(平方米) | O(平方米) | 太慢了 |
 | 最优（贪婪+本地配对）| O(m log m) | O(米) | 已接受 |

 ## 算法演练

 我们将问题重新解释为将原始边配对，其中如果两条边共享至少一个端点，则每个配对贡献其权重之和。 

1. 构建每个顶点的边的邻接列表，存储边索引和权重。 这使我们能够知道哪些边可以合法地通过每个顶点配对。 
2. 按权重降序对所有边进行排序。 排序的原因是较重的边对最终总和的贡献更大，并且应优先考虑涉及它们的配对决策，以避免丢失高价值的组合。 
3. 维护一个布尔数组，标记每条边是否已在匹配对中使用。 这确保了每条边最多贡献一次，与折线图中匹配的定义相匹配。 
4. 迭代顶点。 对于每个顶点 u，收集所有尚未使用的关联边。 使用全局排序隐式地按权重对它们进行排序或分桶。 
5. 在每个顶点 u 处，按权重降序贪婪地配对未使用的关联边。 每对都为答案贡献 w(e_i) + w(e_j)，并且两条边都标记为已使用。 此步骤是有效的，因为一旦边固定，通过 u 的所有可能贡献都独立于其他顶点。 
6. 继续，直到处理完所有顶点。 某些边可能在某个顶点处保持不匹配，但如果可能，将在其另一个端点处进行处理。 

一个关键的细节是，每条边都可以从两个端点考虑，但只能全局匹配一次。 第一次配对决定了它的贡献。 

### 为什么它有效

 线图的结构确保每个匹配边对应于选择共享顶点的两条原始边。 因此，每个有效的解决方案相当于将原始边的子集划分为不相交的对，其中每对都分配给它们的公共端点之一。 贪婪排序确保只要有可能进行高权重配对，就不会为了较小的贡献而推迟配对，因为任何延迟只会减少或维持可行性，而不会增加总和。 这引发了一个交换争论：任何最佳解决方案都可以转换为首先对较重的边缘进行配对的解决方案，而不会降低总值，从而保持最优性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

n, m = map(int, input().split())

edges = []
adj = [[] for _ in range(n + 1)]

for i in range(m):
    u, v, w = map(int, input().split())
    edges.append((w, u, v, i))
    adj[u].append(i)
    adj[v].append(i)

edges.sort(reverse=True)

used = [False] * m
ans = 0

for w, u, v, i in edges:
    if used[i]:
        continue

    best = -1
    bu = bv = -1

    for j in adj[u]:
        if used[j]:
            continue
        w2, x, y, _ = edges[j]
        if best < w2:
            best = w2
            bu = j
            bv = -1

    for j in adj[v]:
        if used[j]:
            continue
        w2, x, y, _ = edges[j]
        if w + w2 > w + best:
            best = w2
            bu = j
            bv = -1

    if bu != -1:
        used[i] = True
        used[bu] = True
        ans += w + best

print(ans)
```该代码维护所有边并按权重降序迭代它们，以便首先考虑高贡献对。 邻接列表允许我们在两个端点找到候选配对边。 

这`used`数组从折线图的角度强制执行匹配约束。 一旦一条边被配对，它就不再被进一步考虑。 

关键的实现困难是确保我们不会重复计算端点之间的边。 代码通过检查来处理这个问题`used`在考虑邻接扫描中的任何边缘之前。

 ## 边缘情况

 一种重要的边缘情况是高度集线器，其中所有入射边都具有不同的权重。 在这种情况下，中心周围的折线图变成具有高度倾斜的边权重的完整图。 该算法确保首先对最大的入射边进行配对，从而防止较小边之间的次优配对。 

另一种情况是边缘形成长路径。 每条边恰好参与两个顶点，贪婪配对必须确保一个顶点中使用的边不会在另一个顶点中错误地重用。 这`used`数组在全局范围内强制执行此操作，并且排序处理保证一旦消耗了一条边，后面的步骤就无法产生涉及它的更好的配对。 

最后一种情况是一个顶点局部存在多个最佳配对。 由于一旦边缘固定，贡献是可加的且独立的，因此等权重候选者之间的任何决胜局都不会影响正确性，并且算法在此类配置中保持稳定。
