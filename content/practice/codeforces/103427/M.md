 ## 解决方案
 ## 问题理解

 我们得到一个小写字符串。 对于该字符串的每个前缀，我们需要查看该前缀内所有可能的连续子字符串，并选择字典顺序最大的子字符串。 在该最大子串值的所有出现中，我们必须输出最左边出现的最早开始位置和结束位置。 

重新表述这一点的一种有用方法是想象一次将字符串增长一个字符。 附加每个新字符后，我们查看以该位置结尾的整个前缀，并询问：是否列出每个子字符串并按字典顺序对它们进行排序，哪个是最后一个，以及它第一次出现的开始和结束位置。 

限制最多为一百万个字符。 任何显式枚举子字符串或比较每个前缀的许多子字符串的解决方案都将立即被排除。 即使扫描长度为 i 的前缀的所有子字符串也已经花费了 O(i²)，在最坏的情况下这将导致大约 10^2 次操作。 这远远超出了 2 秒所能处理的范围。 

主要困难在于答案会逐渐变化，但简单的重新计算会从头开始重复解决完整的“前缀中的最大子串”问题。 

一些边缘情况值得隔离。 

如果字符串像“abcdef”一样严格递增，那么在每一步中，最佳子字符串始终是最后一个字符，因为任何较早开始的较长子字符串将由于其第一个字符而按字典顺序较小。 

如果字符串像“fedcba”一样严格递减，则每个前缀处的最佳子字符串将成为整个前缀，因为任何较早开始的子字符串都以较大的字符开头。 

对于像“ababab”这样的字符串，许多子字符串会重复，并且最大子字符串通常从相同模式的不同出现处开始。 关键的微妙之处在于我们必须返回最左边的匹配项，而不仅仅是任何最大子字符串。 

仅跟踪最大后缀或仅比较后缀的简单方法会失败，因为不能保证最佳子字符串从前缀末尾开始。 

## 方法

 蛮力的想法很简单。 对于每个以位置 i 结尾的前缀，我们枚举每个 r ≤ i 的子串 S[l..r]，按字典顺序比较它，并跟踪最大值。 这是正确的，因为它从字面上评估了定义。 问题是每个前缀包含 O(i²) 个子串，并且在最坏的情况下比较子串可能需要 O(i)，因此完整的解决方案退化为 O(n³)。 即使使用散列来减少比较，我们仍然有 O(n²) 个子串，这对于 n 达到 10⁶ 来说太大了。 

关键的观察是我们实际上不需要比较任意子字符串。 如果我们固定一个起始位置l，则从l开始的最佳子串就是当前前缀的后缀S[l..i]。 因此，对于每个前缀，我们真正要问的是：在该前缀的所有后缀中，哪个后缀按字典顺序最大，以及它最早出现在哪里。 

这减少了维护动态增长字符串的字典顺序最大后缀的问题。 这是一个经典的结构问题，可以使用类似于杜瓦尔林登分解算法的贪婪指针方法在线性时间内解决。 我们的想法是，我们不是比较所有后缀，而是维护一个代表迄今为止看到的最佳后缀的候选区间，并且只有在通过向前扫描发现更好的后缀时才更新它。 

关键的结构特性是，一旦已知从位置 l 开始的后缀比从位置 r 开始的另一个后缀更差，那么对于任何未来的前缀，l 永远不会再次成为最佳。 这使我们能够永久消除大范围的候选人。

 | 蛮力 | O(n3) | O(1) | O(1) | 太慢了 |
 | 最佳 | O(n) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们为当前前缀的最佳后缀维护两个候选起始位置。 直观上，它们代表两个竞争的后缀，我们在扩展字符串时按字典顺序比较它们。 

1.初始化两个指针i和j，其中i是当前最佳候选后缀起始，j是我们尝试改进的挑战者后缀起始。 最初我们将 i 设置为 1，j 设置为 2。 
2. 逐个字符地比较从 i 和 j 开始的子字符串，直到两者都在当前前缀之内。 当 S[i+k] 等于 S[j+k] 时，我们提前偏移 k。 此步骤查找两个后缀不同的第一个位置。 
3. 如果我们发现S[i+k] < S[j+k]，那么从j开始的后缀更好，所以我们丢弃i并将i移动到j。 否则，我们丢弃 j。 这个决定是合理的，因为字典顺序仅取决于第一个不同的字符。 
4. 解决此比较后，我们将 j 前进到下一个未见过的起始位置，并与当前最佳 i 重复比较过程。 
5. 每当前缀扩展一个字符时，我们就继续这一过程，确保 i 始终代表当前前缀的最佳后缀起始位置。 对于每个前缀长度 t，我们输出 i 作为字典顺序最大后缀的起始索引，结束索引为 t。 

关键的实现技巧是，不会对每个前缀从头开始进行比较。 指针前进确保每个字符在整个运行过程中最多参与恒定数量的比较。 

为什么它有效是基于优势不变量。 在任何时刻，i 都是最小的索引，使得从 i 开始的后缀在所有尚未消除的后缀中按字典顺序最大。 每当我们比较 i 和 j 时，其中一个对于所有未来的扩展来说都是严格更差的，因为它们的顺序是由第一个不匹配决定的。 因此，一旦起始位置被丢弃，前缀的任何扩展都无法使其再次成为最佳位置。 这保证了我们只永久丢弃位置，并且由于每个位置最多被丢弃一次，因此总工作量保持线性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    s = input().strip()
    n = len(s)
    s = " " + s  # 1-index

    i = 1
    j = 2

    # best start for current prefix is i
    # we maintain j as candidate
    while j <= n:
        a = i
        b = j

        # compare suffixes starting at a and b
        k = 0
        while b + k <= n:
            if s[a + k] == s[b + k]:
                k += 1
                continue
            break

        # decide which is better
        if b + k > n:
            # b suffix exhausted, b is smaller or equal
            j += 1
            continue

        if a + k > n or s[a + k] < s[b + k]:
            i = j
            j += 1
        else:
            j += 1

    # for every prefix, best suffix starts at i, but we must recompute per prefix
    # so we rerun with incremental tracking
    i = 1
    j = 2
    res = []

    for r in range(1, n + 1):
        while j <= r:
            a = i
            b = j

            k = 0
            while b + k <= r:
                if s[a + k] == s[b + k]:
                    k += 1
                    continue
                break

            if b + k > r:
                j += 1
                continue

            if a + k > r or s[a + k] < s[b + k]:
                i = j
                j += 1
            else:
                j += 1

        res.append((i, r))

    print("\n".join(f"{l} {r}" for l, r in res))

if __name__ == "__main__":
    solve()
```该代码使用 1 索引来简化后缀比较。 外循环增加前缀。 对于每个前缀长度 r，我们确保 j 永远不会超过 r，因此仅考虑有效的后缀开头。 

内部比较循环逐字符前进，直到发现不匹配或一个后缀用完。 这种不匹配决定了哪个起始位置更好。 失败的候选人将被永久丢弃。 

微妙的部分是确保 j 只向前移动并且永远不会向后重置。 这就是使总复杂度保持线性的原因。

 ## 边缘情况

 对于像“aaaaa”这样的字符串，每个后缀都是相同的。 该算法自始至终比较相等的字符，并且从不丢弃最早的索引，因此所有前缀的 i 都保持为 1。 输出变为 (1,1), (1,2), ..., (1,5)，这是正确的，因为每个子串都相连，并且我们选择最左边的匹配项。 

对于“abcde”，比较总是会立即失败，而会选择新字符，因此前缀 i 的最佳后缀始终是 i 处的单个字符。 该算法重复地将 i 更新到当前位置，从而产生正确的单例间隔。 

对于“edcba”，每个新字符都较小，因此后面开始的后缀无法击败第一个字符。 该算法从不替换 i=1，并输出递增的右端点。 

对于“ababab”，等前缀比较会在发生不匹配之前导致长联系。 当发生不匹配时，后面的“b”后缀最终会取代前面的“a”开头。 该算法确保丢弃的起始位置永远不会重新进入，因此每个位置只会发生一次跳跃。
