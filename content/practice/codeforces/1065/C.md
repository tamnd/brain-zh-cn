 ## 解决方案
 ## 问题理解

 我们得到了一组垂直塔，每个塔都用正整数高度表示。 在一次操作中，我们选择目标高度$H$，然后每座塔的高度都高于$H$被精确地缩减为$H$，而较短的塔保持不变。 此操作的成本是所有塔中移除的立方体总数，因此它取决于所选的高度和塔的当前配置。 

我们可以执行多次此类操作。 仅当操作的成本不超过给定限制时，操作才被视为有效$k$。 目标是执行最少数量的有效操作，直到所有塔最终具有相同的最终高度。 

这里的重要结构是每个操作仅降低高度，并且每个操作选择应用于所有塔的全局阈值。 这意味着该过程是一系列下降的“水平”，最终状态必须是系统中已经存在的某个高度，或者可以通过重复降低达到的高度。 

约束条件很大：最多$2 \cdot 10^5$塔。 这排除了任何模拟每个可能高度并在二次时间中简单地重新计算成本的解决方案。 重复切片的直接模拟将在每个操作中重复扫描所有塔，从而导致$O(n^2)$最坏情况下的行为，速度太慢。 

当许多塔楼的高度相似时，就会出现一个微妙的问题。 重复切割到下一个不同高度的天真贪婪选择并不总是尊重成本约束，因为将太多的减少分组到一个切片中可能会超过$k$，而过于激进的拆分会增加操作数量。 

一个小例子就说明了其中的困难。 假设高度是$[100, 99, 98]$和$k = 3$。 成本直接削减至 98$(100-98) + (99-98) = 3$，这是有效的。 但先切割到 99，然后切割到 98 也是有效的，但需要更多操作。 总是切入下一个明显高度的粗心策略可能会不必要地过度计算操作。 

关键的困难在于决定切片高度的放置位置，以便每个切片“足够便宜”，同时最大限度地减少切片数量。 

## 方法

 暴力策略会尝试模拟所有可能的有效切片序列。 从任何当前的高度集，我们可以尝试每一个可能的目标高度$H$生成一个有效的切片，计算其成本，然后递归地继续。 这很快就会变成指数，因为每个切片都会创建一个等于可能阈值数量的分支因子。 即使我们限制独特的高度，每一步都需要所有塔的计算成本，从而导致$O(n^2)$每个路径并使整体搜索不可行。 

关键的观察结果是，唯一有意义的候选切片高度是塔的不同高度，按降序排序。 任何最佳策略都只能“降落”在这些值上，因为切割到不存在的中间高度总是以切割到下一个现有水平为主，而在未来的操作中没有额外的好处。 

一旦我们对不同的高度进行排序，我们就可以反向思考：我们从顶层向下逐渐降低高度。 每次我们从一个高度层移动到下一个较低层时，我们都会产生与该层之上的塔楼数量成正比的成本。 

我们可以不向前模拟切片，而是可以贪婪地处理从高到低的高度并进行分组缩减：累积如果我们缩减到下一个级别将删除多少个立方体，并且每当累积成本超过$k$，我们“关闭”一个切片并开始一个新的切片。 

这将问题转化为排序的唯一高度上的分段问题，其中每个分段对应一个切片，分段的成本是该范围内所有塔楼贡献的总垂直落差。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | 指数 /$O(n^2)$|$O(n)$| 太慢了 |
 | 最佳 |$O(n \log n)$|$O(n)$| 已接受 |

 ## 算法演练

 我们首先将高度压缩成频率图，以便我们知道每个不同高度存在多少个塔。 然后我们将这些高度按降序排列。 

1. 构建所有高度的频率图，以便我们知道每个级别当前有多少个塔。 这是必要的，因为成本取决于每一步受影响的塔数量。 
2. 将唯一高度按降序排序。 我们从上到下进行处理，因为切片决策取决于有多少质量高于每个阈值。 
3. 保持当前水平之上当前“活动”的塔数量的运行计数。 最初，所有塔都处于活动状态，因为尚未切断任何东西。 
4. 维护当前切片的运行累计成本。 从高处移动时$h_i$到$h_{i+1}$，每个活动塔贡献一滴$h_i - h_{i+1}$，所以我们添加$(h_i - h_{i+1}) \times \text{active}$到成本。 
5. 如果添加此成本超过$k$，我们最终确定当前切片的高度$h_i$，增加答案，并重置成本累加器。 我们还从该级别开始重新评估当前段，因为已经放置了切片边界。 
6. 完成切片后，我们继续向下处理，跟踪活动塔，每当我们遇到新的高度水平时，活动塔就会增加（因为这些塔在我们下降时加入活动集）。 
7. 处理完所有级别后，如果还有未完成的切片，我们将其算作一次最终操作。 

重新启动步骤背后的原因是，一旦我们超过$k$，当前的高度减少分组成本太高，因此最后一次有效切割必须发生在前一个边界处。 

### 为什么它有效

 在任何时候，所有高于当前水平的塔都会对进一步的减少做出统一的贡献。 这使得在两个连续不同高度之间下降的成本与活动塔的数量成线性关系。 由于成本在整个高度区间内是相加的，并且在每个区间内是独立的，因此问题简化为将排序序列划分为多个段，使得每个段的总和最多为$k$。 每当段超过时贪婪地选择切割$k$是最优的，因为延迟切割只会进一步增加成本，而不会降低未来每单位高度的成本。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, k = map(int, input().split())
    h = list(map(int, input().split()))
    
    freq = {}
    for x in h:
        freq[x] = freq.get(x, 0) + 1

    heights = sorted(freq.keys(), reverse=True)

    active = 0
    cost = 0
    ans = 0

    i = 0
    while i < len(heights):
        if active == 0:
            active = freq[heights[i]]
            i += 1
            continue

        if i < len(heights):
            dh = heights[i-1] - heights[i]
            add_cost = dh * active

            if cost + add_cost <= k:
                cost += add_cost
                active += freq[heights[i]]
                i += 1
            else:
                ans += 1
                cost = 0
                active = 0
        else:
            break

    if active > 0:
        ans += 1

    print(ans)

if __name__ == "__main__":
    solve()
```该代码首先计算频率，以避免重复处理相同的高度。 按降序排序可确保我们始终在高度空间中向下移动。 

变量`active`和`cost`表示当前切片状态。`active`跟踪有多少塔仍高于当前阈值，以及`cost`跟踪当前切片中已删除多少个立方体。 

关键的转变是计算`dh * active`，它编码了这样一个事实：每个活动塔在两个相邻高度水平之间移动时都会均匀减小。 

每当添加下一个间隔将超过$k$，我们最终确定一个切片并重置状态，这对应于在最后一个有效高度边界处放置一个切割。 

## 工作示例

 ### 示例 1

 输入：```
5 5
3 1 2 2 4
```频率排序：

 | 身高| 频率| 活跃| 成本增加 | 运行成本| 行动|
 | ---| ---| ---| ---| ---| ---|
 | 4 | 1 | 1 | - | 0 | 开始 |
 | 3 | 1 | 1 | (4-3)*1 = 1 | 1 | 继续 |
 | 2 | 2 | 2 | (3-2)*2 = 2 | 3 | 继续 |
 | 1 | 1 | 4 | (2-1)*4 = 4 | 7 > 5 | 7 > 5 切|

 第一次切割后，我们从高度 2 重新开始。 

| 身高| 频率| 活跃| 成本增加 | 运行成本| 行动|
 | ---| ---| ---| ---| ---| ---|
 | 2 | 2 | 2 | - | 0 | 开始 |
 | 1 | 1 | 4 | (2-1)*4 = 4 | 4 | 完成 |

 总切片 = 2。 

该痕迹表明，超过$k$当累积的垂直减少变得过于昂贵而无法继续分组时，恰好强制边界。

 输入：```

| 身高| 频率| 活跃| 成本| 行动|
 | ---| ---| ---| ---| ---|

这证实了统一数组总是只需要一次操作。 

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 |$O(n \log n)$| 排序不同高度占主导地位，扫描呈线性 |
 | 空间|$O(n)$| 频率图和高度表|

 约束允许最多$2 \cdot 10^5$元素，所以$O(n \log n)$解决方案在时间限制内非常适合。 内存使用量与不同高度和频率的数量呈线性关系，在 256 MB 以下也是安全的。

 ## 边缘情况

 当所有塔都具有相同高度时，就会出现关键的边缘情况。 在这种情况下，算法会初始化单个活动组并且不会累积任何成本，因此它只生成一个切片，这是正确的，因为不需要减少，但我们仍然计算最终操作。 

另一个微妙的情况是当$k$非常小，迫使每个高度过渡都成为自己的切片。 该算法会处理这个问题，因为每次`add_cost`超过$k$，它会立即重置，确保没有段违反约束。
