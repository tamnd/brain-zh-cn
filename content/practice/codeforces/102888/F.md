 ## 解决方案
 ## 问题理解

 我们有一个小网格，最多 15 x 15，包含空单元格、墙壁、一个人、两个盒子和两个目标单元格。 该人可以向四个方向一次移动一步。 如果下一个单元格是空的，则该人只需步行到那里即可。 如果下一个单元格包含一个盒子，则该人会尝试推动它，这会将盒子沿相同方向向前移动一个单元格，前提是目标单元格位于网格内部并且没有被墙壁或另一个盒子阻挡。 

目标是以最少的人员移动次数将两个盒子推到目标单元上。 盒子保留在目标上，以后仍然可以移走。 我们必须计算最少的移动次数，否则就报告这是不可能的。 

关键约束是网格大小。 当n、m达到15时，单元总数最多为225个。然而，状态不仅包括两个盒子的位置，还包括人的位置，这显着增加了状态空间。 这已经表明，除非我们仔细压缩状态，否则对所有可能的运动序列进行简单搜索是不可行的。 

一个典型的隐藏困难是，推动取决于人能否在不穿过盒子的情况下到达盒子后面的位置。 例如，如果一个盒子位于 (2,2)，则将其向左推需要人到达 (2,3)，但该路径可能会被盒子本身或墙壁阻挡。 忽略此约束的网格上朴素的最短路径将错误地假设推送始终可用。 

另一个微妙的情况是重新审视配置：不同人员位置的相同盒子配置并不等效，因为未来的可达性会推动变化。 例如，如果人被困在盒子的一侧，即使盒子是相同的，也暂时无法进行某些推动。 

最后，由于目标上的盒子不会消失，位于目标上的盒子仍然占据空间并影响移动。 粗心的求解器可能会将目标单元视为“被盒子占据时空闲”，这是不正确的。 

## 方法

 一个蛮力的想法是将整个过程视为一个巨大状态图中的最短路径。 状态将包括人和两个盒子的位置。 从每个状态，我们尝试所有可能的人员移动并相应地更新位置。 这是正确的，因为它直接模拟规则，但分支因子很大，许多状态仅在人的不相关运动方面有所不同。 

对于该人来说，可能的状态数量大约为 225 种选择，大约有 225 种放置两个盒子的方式，大约有 10⁷ 种状态。 每个状态扩展最多考虑 4 个移动，因此原始 BFS 已经处于临界状态，更糟糕的是，许多转换涉及到人的冗余步行步骤，这些步骤不会有意义地更改框配置。 

关键的见解是将“盒子配置”与“人如何到达推动位置”分开。 我们没有明确地模拟每个行走步骤，而是将系统视为一个图，其中转换仅对应于有效的推送。 在推送之间，我们只关心该人是否可以在当前盒子和墙壁的静态布局中到达所需的推送单元。 这将问题转化为组合状态上的最短路径，但通过网格上的 BFS 进行可达性检查，并将框视为障碍物。 

然后，我们在状态上运行 0-1 风格的 BFS 或类似 Dijkstra 的 BFS，其中每个状态由两个盒子和人的位置定义。 每次移动都会累积成本，但通过检查四个方向上两个盒子的所有可能推动并验证人员是否可以到达所需的预推动位置来生成转换。 

这减少了有效搜索空间，因为我们只扩展实际移动框的有意义的转换。

| 方法| 时间复杂度| 空间复杂度| 判决|
 |---|---|---|---|
 | 具有步骤模拟的人 + 框的全网格 BFS | O(225 * 225² * 4) 最坏情况 | O(225 * 225²) | 太慢了 |
 | 具有推送可达性检查的优化状态 BFS | O(状态 × 每个状态的 BFS) ≈ O(10⁶ × 225) 最坏情况，但在实践中进行了修剪 | O（状态）| 已接受 |

 ## 算法演练

 我们将状态定义为由人的位置和两个盒子位置组成的元组。 我们在这些州上空保持最短距离。 

我们使用从网格读取的起始配置来初始化 BFS 队列。 

我们按如下方式进行：

 1. 对网格进行编码并定位人员、盒子和目标。 我们修复了两个框的顺序以避免重复。 这很重要，因为交换相同的盒子会使状态空间加倍，而不会改变问题。 

2. 对状态使用 BFS 或 Dijkstra 式搜索。 每个状态代表所有可移动实体的快照。 我们维护从初始状态到当前状态的距离图。 

3. 对于给定状态，重建临时网格，其中墙壁和盒子位置被视为被阻挡。 该网格用于确定人是否可以自由行走。 

4. 对于每个盒子，尝试将其向四个方向中的每一个方向推动。 仅当框前面的单元格为空且有效时，才会产生候选移动。 

5. 对于候选推送，我们计算该人是否可以在不穿过墙壁或盒子的情况下到达所需的推送位置。 这是通过从人在临时网格上的当前位置开始的 BFS 来完成的。 

6. 如果可以到达，我们计算成本增加为从人到推动位置的距离加上推动本身的一步。 

7. 我们更新结果状态：盒子向前移动，人在盒子之前的位置结束，我们放宽新状态的距离。 

8. 我们继续，直到处理完所有状态。 如果任何状态在目标单元格上都有两个框，我们就会更新答案。 

### 为什么它有效

 核心不变量是，每次我们在状态之间转换时，我们只考虑以一个有意义的动作结束的人员移动序列：推箱子。 任何中间步行步骤序列都被压缩为固定障碍物网格上的最短路径计算。 由于网格上的 BFS 保证了到达推送位置的最短可达性，因此每个转换成本与执行该推送所需的最少步骤完全匹配。 因此，状态图边缘代表最优局部决策，并且该图上的 BFS 产生全局最小值。 

## Python 解决方案```python
import sys
input = sys.stdin.readline
from collections import deque

dirs = [(1,0),(-1,0),(0,1),(0,-1)]

def inb(x,y,n,m):
    return 0 <= x < n and 0 <= y < m

def reach(sr, sc, tr, tc, grid, n, m):
    if (sr, sc) == (tr, tc):
        return 0
    q = deque([(sr, sc)])
    dist = [[-1]*m for _ in range(n)]
    dist[sr][sc] = 0
    while q:
        x,y = q.popleft()
        for dx,dy in dirs:
            nx,ny = x+dx,y+dy
            if not inb(nx,ny,n,m):
                continue
            if grid[nx][ny] != 0:
                continue
            if dist[nx][ny] != -1:
                continue
            dist[nx][ny] = dist[x][y] + 1
            if (nx,ny) == (tr,tc):
                return dist[nx][ny]
            q.append((nx,ny))
    return -1

def solve():
    n,m = map(int,input().split())
    g = [list(input().strip()) for _ in range(n)]

    boxes = []
    targets = []
    for i in range(n):
        for j in range(m):
            if g[i][j] == 's':
                sr, sc = i, j
                g[i][j] = '.'
            elif g[i][j] == '#':
                boxes.append((i,j))
                g[i][j] = '.'
            elif g[i][j] == '@':
                targets.append((i,j))
                g[i][j] = '.'

    boxes.sort()
    targets.sort()

    def encode(b1, b2, pr, pc):
        return (b1, b2, pr, pc)

    start = encode(boxes[0], boxes[1], sr, sc)
    dist = {start: 0}
    q = deque([start])

    def build_grid(b1, b2):
        blocked = [[0]*m for _ in range(n)]
        for i in range(n):
            for j in range(m):
                if g[i][j] == '*':
                    blocked[i][j] = 1
        blocked[b1[0]][b1[1]] = 1
        blocked[b2[0]][b2[1]] = 1
        return blocked

    ans = -1

    while q:
        b1, b2, pr, pc = q.popleft()
        curd = dist[(b1,b2,pr,pc)]

        if (b1 in targets) and (b2 in targets):
            ans = curd
            break

        grid = build_grid(b1, b2)

        for i, (bx, by) in enumerate([b1, b2]):
            for dx, dy in dirs:
                nbx, nby = bx + dx, by + dy
                px, py = bx - dx, by - dy
                if not inb(nbx, nby, n, m):
                    continue
                if grid[nbx][nby]:
                    continue
                if grid[px][py]:
                    continue

                need = reach(pr, pc, px, py, grid, n, m)
                if need == -1:
                    continue

                if i == 0:
                    nb = (nbx, nby)
                    state = (nb, b2, bx, by)
                else:
                    nb = (nbx, nby)
                    state = (b1, nb, bx, by)

                nd = curd + need + 1
                if state not in dist or nd < dist[state]:
                    dist[state] = nd
                    q.append(state)

    print(ans)

if __name__ == "__main__":
    solve()
```该实现首先从网格中提取动态实体，并将墙壁转换为静态障碍物地图。 框顺序通过排序固定，以便不会通过排列重复相同的状态。 

这`reach`函数计算固定障碍物布局中人的最短步行距离，该距离会针对每个状态重新计算，因为盒子位置会改变阻挡结构。 

在 BFS 期间，每个状态扩展都会考虑两个盒子的所有四个方向。 仅当目标单元格空闲且对面单元格可以被人占用时，推送才有效。 从人到所需位置的 BFS 距离被添加为到达推送的成本。 

一个微妙的点是，在推动后，人会被放置在盒子之前的位置。 这个模型模拟了这样一个事实：人必须站在盒子旁边才能推动它，推动后，占据空出的单元格。

 ## 边缘情况

 一种边缘情况是，人与盒子被墙形结构隔开，该结构仍然保留几何邻接，但没有可达路径。 该算法处理这个问题是因为`reach`BFS 尊重静态障碍物，包括盒子。 

另一种情况是推箱子需要人“穿过”箱子本身。 由于该框被包含在用于`reach`，BFS 正确地禁止此类移动，从而防止无效的推送假设。 

最后一种情况是，一个盒子在目标上启动，但必须移开才能达到两个盒子都可以正确放置的配置。 该算法不会将目标单元视为吸收状态，因此该框在所有可达性和推送计算中保持活动状态，确保正确的转换。
