 ## 解决方案
 ## 问题理解

 该任务可以用几何术语来理解。 想象一下在网格上绘制的露营地，其中一些单元格被帐篷或建筑物占据，其余部分是空地。 一个人沿着占用区域的外边界行走，始终恰好停留在占用空间和空白空间之间的边缘。 目标是计算这次行走的长度，以沿网格边缘的单位步数进行测量。 

输入描述了一个矩形网格。 每个单元格指示该位置是属于露营地还是空地。 输出是一个整数，表示围绕所有占用单元的外边界的总长度。 

从复杂性的角度来看，测试用例中的网格大小通常最多可达 10^5 个单元格左右，或者每个维度最多可达数千个单元格。 这立即排除了任何模拟围绕边界逐步行走或对每个单元执行重复洪水填充的解决方案。 与细胞数量 O(nm) 或等效的 O(总细胞) 呈线性关系的解决方案是唯一可行的方向。 

当试图从一个被占用的单元格开始并使用方向规则绕着它走来明确地追踪边界时，就会出现一个幼稚的错误。 考虑露营地有多个断开连接的组件的情况：

 输入：```
1 0 1
0 0 0
1 0 1
```从第一个被占用的单元开始并尝试“沿着边缘”的天真的步行器将完全错过其他组件，除非它为每个区域小心地重新启动。 当形状有孔时，会发生另一种微妙的故障：

 输入：```
1 1 1
1 0 1
1 1 1
```正确的输出同时计算外边界和内腔边界。 仅遵循外部轮廓的简单周界轨迹会低估答案，因为它忽略了内部孔的贡献。 

这些案例表明局部边界穿越是脆弱的。 正确的方法必须是全局性的，并系统地考虑每个暴露的边缘。 

## 方法

 蛮力的想法是将每个占用的单元格视为一个独立的正方形，并尝试确定其边界对最终周长的贡献有多大。 对于每个单元格，我们检查其四个邻居。 如果邻居在网格外或空着，则该侧向周长贡献一个单位。 将所有占用的单元格的总和得出总边界长度。 

这种方法是正确的，因为每个单元格的每个暴露边缘都被精确计数一次。 然而，如果实施效率低下，很容易高估性能。 在尺寸为 n × m 的完整网格的最坏情况下，我们对每个单元执行四次检查，从而导致 4nm 操作。 这仍然是线性的，但如果重复重新行走共享边缘，任何模拟行走或构造显式边界路径的尝试都可能在病态布局中降级为 O((nm)^2)。 

关键的见解是周界不是路径问题而是局部邻接问题。 每个细胞仅根据其直接邻居独立做出贡献。 这完全消除了遍历的需要，并将问题简化为简单的计数。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 |---|---|---|---|
 | 暴力边界追踪 | O((nm)^2) 最坏情况 | O(纳米) | 太慢了|
 | 每个细胞的邻居计数 | O(纳米) | O(1) 或 O(nm) | 已接受 |

 ## 算法演练

 1. 读取网格并识别所有被占用的单元格。 每个占据的单元被视为贡献潜在边界边缘的单位正方形。 

2. 对于每个被占用的单元格，检查其四个相邻方向：上、下、左、右。 每个方向代表一个可能的边界贡献。 

3. 对于给定方向，检查相邻单元格是否在网格之外或为空。 如果是这样，请将周长计数加一。 这是有效的，因为任何缺失的邻居都意味着暴露的边缘。 

4. 在移动到下一个单元之前，累积当前单元所有四个方向的贡献。 这确保每个单元独立地贡献其所有暴露的边缘。 

5. 对整个网格的贡献求和并输出最终的总数。 

检查邻居而不是构建边界的原因是每个边界段唯一对应于由一个占用单元和一个未占用邻接组成的对。 这避免了单元之间共享的边的双重遍历。 

### 为什么它有效

 最终周界中的每条边均由一个已占用的单元格和一个相邻的空位置或出界位置唯一确定。 任何边都不能被计算两次，因为它总是归因于该邻接的被占用的一侧。 相反，必须计算每个暴露的侧面，因为它代表从占用空间到空白空间的过渡。 这种一对一的映射保证了正确性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    grid = [list(input().strip()) for _ in range(n)]
    
    dirs = [(1,0), (-1,0), (0,1), (0,-1)]
    perimeter = 0

    for i in range(n):
        for j in range(m):
            if grid[i][j] != '1':
                continue
            for di, dj in dirs:
                ni, nj = i + di, j + dj
                if ni < 0 or ni >= n or nj < 0 or nj >= m or grid[ni][nj] == '0':
                    perimeter += 1

    print(perimeter)

if __name__ == "__main__":
    solve()
```实现直接遵循邻接原则。 网格被扫描一次，对于每个占用的单元格，我们检查四个邻居。 边界检查确保我们正确地将网格边缘处理为暴露的周边。 一个常见的错误是忘记将出界邻居视为贡献边，这会导致沿外边界的计数不足。 

另一个微妙的问题是输入处理：将网格视为整数与字符串必须保持一致。 这里我们比较字符，因此条件显式检查`'1'`。

 ## 边缘情况

 对于单个占用的单元格，所有四个方向都在网格之外，因此算法正确地计算了四个暴露的边缘。 

输入：```
1 1
1
```该循环访问唯一的单元格，检查四个邻居，所有边界检查均失败，并将周长增加到 4。 

对于全空网格，没有单元格触发任何贡献，因此输出保持为零。 

输入：```
2 2
00
00
```既然条件`grid[i][j] != '1'`到处都失败，不会发生邻居检查，并且算法以零终止。 

对于完全填充的块，内部共享边会取消，因为两个占用单元之间的每个邻接永远不会被视为边界。 只有外边缘起作用，产生矩形块的预期周长 2(n + m)。
