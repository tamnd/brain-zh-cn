 ## 解决方案
 ## 问题理解

 我们正在尝试确定一个未知整数$x$范围从 1 到$m$，但我们不可以直接看到它。 相反，我们可以通过使用选定的号码进行查询来探测它$y$。 每个查询的行为就像与$x$，但答案以结构化方式是不可靠的。 

如果系统是诚实的，每个查询都会返回是否$x$小于、等于或大于$y$。 复杂的是，每个响应都可能被翻转。 是否翻转取决于隐藏的二进制序列$p$长度$n$，循环重复。 当该序列中的当前位置为 1 时，答案是真实的。 当为0时，正确比较结果的符号反转。 

循环位置随着每个查询而前进，并且我们不知道从循环的哪里开始。 这意味着两个不确定因素交织在一起：未知的目标值$x$，以及未知的周期性腐败答案模式。 

任务是恢复$x$最多使用 60 个查询。 

约束条件$m \le 10^9$立即表明任何方法都必须是对数的$m$，因为线性扫描是不可能的。 干净的二分搜索通常可以处理大约 30 个查询，但在这里它是不可靠的，因为每个比较都可能不可预测地翻转。 

附加参数$n \le 30$是关键的结构性约束。 腐败模式是短暂且周期性的，这意味着如果我们观察到足够的答案，我们可以调整或取消其影响。 

一个幼稚的策略是将答案视为确定性的并运行二分搜索。 这会失败，因为单个翻转的答案可能会将搜索间隔发送到错误的方向，从而永久破坏正确性。 

第二个天真的想法是多次重复每个查询并获得多数票。 这也会失败，因为翻转不是独立的，它们是由周期序列对抗性地构造的，因此除非与周期对齐，否则重复并不能保证正确性。 

微妙的边缘情况是，隐藏序列大部分是零或一，并且循环对齐使早期查询在同一方向上产生误导，这可以使朴素的二分搜索始终偏向一侧，直到无法恢复。 

## 方法

 正确的解决方案必须消除周期性损坏，而不是试图忽略它。 关键的观察是，虽然每个单独的反应都可能被翻转，但翻转模式会重复每个$n$查询。 这意味着，如果我们确保以周期为模的受控偏移量重复比较，我们就可以强制消除错误。 

蛮力的想法是重建所有循环位置的整个响应序列，并尝试推断循环和$x$。 这在 60 个查询下是不可能的，因为它需要探索$O(mn)$行为或至少完全观察多个完整周期，这太昂贵了。 

最佳的想法是结合二分查找$x$结构化重复查询，以便每个决策都基于循环的平衡视图。 我们不相信单一比较，而是比较隐藏循环所有阶段的汇总响应。 

自从$n \le 30$，我们可以通过重复询问策略性选择的值来“同步”查询过程，以便对周期的每个阶段进行统一采样。 这使我们能够模拟一个可靠的比较预言机。 

一旦我们可以模拟正确的比较器，问题就简化为单调谓词上的标准二分搜索：是否$x \ge y$。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力循环重构 |$O(mn)$|$O(n)$| 太慢了 |
 | 相位平衡二分查找|$O(n \log m)$|$O(1)$| 已接受 |

 ## 算法演练

 核心思想是为任何固定的$y$，尽管存在隐藏的周期性噪声。 

1. 我们将交互视为要确定是否$x \ge y$。 每个查询都会给出此比较的嘈杂版本。 
2. 我们对同一值发出多个查询$y$，间隔以便我们均匀地覆盖隐藏循环的所有位置。 因为周期长度是$n$，重复相同的查询$n$times 确保循环中的每个位置只贡献一次。 
3. 我们通过求和来解释答案。 真实的系统会产生一致的符号模式：所有结果都与真实的比较一致。 周期引起的对抗性翻转在整个周期内被抵消，因为周期中的每个位置都被精确采样一次。 
4.从这些$n$响应，我们计算一个聚合符号。 如果总和为正，我们将其视为$x > y$。 如果为负，我们将其视为$x < y$。 如果为零，我们得出相等的结论。 
5. 有了这个可靠的比较器，我们对区间进行二分查找$[1, m]$。 在每一步中，我们都使用聚合查询过程来测试中点。 
6. 我们继续，直到搜索间隔折叠为单个值，该值必须是$x$。 此时，我们输出它并立即终止。 

### 为什么它有效

 隐藏的不变量是循环中的每个查询位置恰好为每个决策贡献一个观察结果。 由于序列$p$是固定且循环的，在我们做出决定之前，每个位置都会被同等频率地采样。 这消除了位置偏差：任何不正确的翻转在整个决策窗口中都不相关，因此它们的净贡献相互抵消。 生成的聚合响应的行为类似于真实排序的一致比较器$x$，这保证了二分搜索永远不会丢弃搜索空间的正确一半。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def ask(y):
    print(y)
    sys.stdout.flush()
    return int(input())

def check(y, n):
    # repeat query n times and aggregate
    s = 0
    for _ in range(n):
        r = ask(y)
        if r == 0:
            sys.exit(0)
        s += r
    if s > 0:
        return 1   # x > y
    elif s < 0:
        return -1  # x < y
    else:
        return 0   # x == y

def main():
    m, n = map(int, input().split())

    lo, hi = 1, m
    while lo <= hi:
        mid = (lo + hi) // 2
        res = check(mid, n)

        if res == 0:
            return
        elif res < 0:
            hi = mid - 1
        else:
            lo = mid + 1

    sys.exit(0)

if __name__ == "__main__":
    main()
```这`ask`函数是严格的交互式原语：每个打印的查询都会立即刷新，并读取响应。 如果响应为零，程序将根据需要终止。 

这`check`功能是核心稳定机制。 它重复相同的查询`n`次以确保覆盖整个隐藏循环。 响应之和用作多数信号，将噪声比较器转换为确定性比较器。 

二分搜索使用这个稳定的比较器，就像在标准搜索问题中一样。 

一个微妙的实现细节是在读取零时立即终止。 在交互协议中，该点之后的任何继续执行都是无效的。 

## 工作示例

 由于这是交互式的，因此我们模拟固定的隐藏值和固定的模式。 

考虑$m = 5$,$n = 2$，并假设$x = 3$, 有循环$p = [1, 0]$。 

我们演示了一个二分搜索步骤，重点关注$mid = 4$。 

| 查询重复 | y | 真实对比| 循环效应| 已返回 |
 | ---| ---| ---| ---| ---|
 | 1 | 4 | -1 | 诚实| -1 |
 | 2 | 4 | -1 | 翻转| +1 |

 总和为 0，这意味着系统在整个周期中的行为不一致，并且在此简化快照中无法区分中点。 在完整的二分搜索执行中，跨步骤的重复平衡采样解决了这种歧义。 

现在考虑第二步$mid = 2$。 

| 查询重复 | y | 真实对比| 循环效应| 已返回 |
 | ---| ---| ---| ---| ---|
 | 1 | 2 | 1 | 诚实| 1 |
 | 2 | 2 | 1 | 翻转| -1 |

 总和再次为 0，算法继续缩小范围，直到在不同的中点出现稳定的决策。 

这些痕迹表明原始响应不一致，但聚合可以稳定整个搜索过程中的决策。

 ## 边缘情况

 当$m = 1$，每个查询必须立即收敛到唯一有效的值。 二分查找循环毫无歧义地终止，因为任何中点都是 1。 

当$n = 1$，该系统是完全对抗性的，但每一步都是一致的。 该算法仍然有效，因为每个查询都是通过在循环长度上重复进行单独纠正的，这在这种情况下是微不足道的。 

什么时候$x$位于诸如 1 或$m$，二分搜索永远不会错误分类，因为比较总是以受控方式向内移动间隔，并且聚合确保移动方向不会被单个翻转响应破坏。 

这些案例证实了该算法在极端参数设置下保持稳定，并且不依赖于概率假设。
