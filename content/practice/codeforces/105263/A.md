 ## 解决方案
 ## 问题理解

 我们正在处理网络事件的实时流。 每个事件要么是配置更新，要么是数据包到达。 数据包携带三部分信息：发送者 IP 地址、时间戳和数据包中的字节数。 系统根据最近接受的流量的规则决定是接受还是忽略每个数据包。 

在任何时刻，我们都会维护一个宽度为 δ 的滑动时间窗口，结束于当前数据包时间 t，这意味着我们只关心时间戳在 [t − δ + 1, t] 范围内的已接受数据包。 在此窗口中，我们分别跟踪每个 IP 地址的统计信息：已接受来自该 IP 的数据包数量以及已接受的总字节数。 

如果在假设考虑之后，窗口中从其 IP 接受的数据包数量达到或超过 α，或者窗口中从该 IP 接受的总字节数达到或超过 β，则数据包将被拒绝。 否则，它会被接受并有助于未来的决策。 

挑战在于窗口和阈值都会动态变化，我们必须按顺序处理最多 100,000 个事件，并且时间戳严格增加。 最后一个条件至关重要，因为它允许我们以单调的方式驱逐旧数据包，而不是及时任意搜索。 

为每个查询重新扫描所有先前数据包的简单实现将重复遍历每个数据包最多 O(n) 历史记录，从而导致 O(n²) 行为，这在 10⁵ 操作中是不可行的。 我们需要一个支持增量插入和高效删除过时事件的结构。 

一种微妙的边缘情况是 δ 非常大，可能大于所有时间戳。 在这种情况下，任何内容都不会过期，窗口将成为“迄今为止的所有内容”，仍然必须在不溢出或缓慢扫描的情况下进行处理。 另一个问题是参数更改：α、β 和 δ 可以随时更改，这意味着之前会被接受的数据包现在可能会根据相同的历史记录而被拒绝。 

最重要的隐藏困难是接受的数据包会影响未来的状态，而忽略的数据包则不会。 这意味着我们必须仔细地将“已看到的数据包”与“已计数的数据包”分开，并确保只有接受的数据包才有助于滑动窗口统计。 

## 方法

 暴力解决方案将通过扫描所有先前接受的数据包、过滤当前时间窗口中的数据包并对每个 IP 的计数和字节进行求和来处理每个数据包。 这是正确的，但代价高昂：每个查询可能会触及 O(n) 历史记录，并且最多 10⁵ 查询，在最坏的情况下会导致 101⁰ 次操作。 

关键的观察结果是时间戳严格增加。 这就将滑动窗口问题转化为队列问题。 一旦数据包变得太旧，它就永远不会再变得相关，因此我们可以将接受的数据包存储在 FIFO 结构中，并从前面驱逐过时的数据包。 

为了支持每个 IP 的约束，我们维护每个 IP 的聚合计数器，并在数据包进入和离开窗口时增量更新它们。 当数据包被接受时，我们将其推入队列并更新其 IP 计数器。 在处理每个新数据包之前，我们首先从队列前端逐出过期数据包，并从相应的 IP 状态中减去它们的贡献。 

这将每个数据包减少到 O(1) 摊销工作：每个接受的数据包插入一次并删除一次。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(n²) | O(n) | 太慢了 |
 | 最优（队列+哈希图）| O(n) 摊销 | O(n) | 已接受 |

 ## 算法演练

我们维护一个已接受数据包的队列。 每个条目都存储其时间戳、IP 和大小。 除此之外，我们保留两个哈希映射：一个用于每个 IP 的数据包计数，另一个用于当前窗口内每个 IP 的总字节数。 

我们还维护当前的参数 α、β 和 δ，它们可以随时更改。 

### 步骤

 1. 阅读下一个事件。 如果是配置更新，则替换α、β、δ。 不需要立即进行结构更改，因为在处理未来的数据包时，旧数据包将被延迟验证。 
2. 如果是时间为t的数据包，首先删除过期的数据包。 当队列非空并且前面的数据包的时间戳 < t − δ + 1 时，将其从队列中弹出并从该 IP 的计数器中减去其贡献。 这确保了在评估之前恢复窗口不变量。 
3. 清理后，计算数据包 IP 的当前状态：来自哈希映射的 current_count 和 current_bytes。 
4. 检查添加新数据包是否违反约束。 具体来说，我们测试 current_count + 1 ≥ α 或 current_bytes + size ≥ β 是否。 
5. 如果任一条件成立，则输出“ig”并完全丢弃该数据包。 
6. 否则，输出“ac”，将数据包推入队列，并更新该 IP 的计数器。 
7. 继续下一个活动。 

### 为什么它有效

 核心不变量是，在处理每个数据包之前，队列恰好包含时间戳位于有效窗口 [t − δ + 1, t − 1] 内的已接受数据包。 所有过期的数据包都已被删除，并且所有剩余的数据包都保证与当前决策相关。 

因为我们只添加已接受的数据包，并且仅在数据包超出窗口时将其删除，所以每个 IP 计数器始终反映滑动窗口内已接受流量的真实状态。 由于拒绝可以防止插入，因此被拒绝的数据包永远不会污染未来状态，从而保持正确性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline
from collections import deque, defaultdict

def solve():
    T = int(input())
    for _ in range(T):
        Q = int(input())

        q = deque()
        cnt = defaultdict(int)
        byt = defaultdict(int)

        alpha = beta = delta = 0

        for i in range(Q):
            parts = input().split()
            typ = int(parts[0])

            if typ == 2:
                alpha = int(parts[1])
                beta = int(parts[2])
                delta = int(parts[3])
            else:
                ip = parts[1]
                t = int(parts[2])
                sz = int(parts[3])

                # remove expired
                left_bound = t - delta + 1
                while q and q[0][0] < left_bound:
                    tt, ip0, sz0 = q.popleft()
                    cnt[ip0] -= 1
                    byt[ip0] -= sz0

                c = cnt[ip]
                b = byt[ip]

                if c + 1 >= alpha or b + sz >= beta:
                    print("ig")
                else:
                    print("ac")
                    q.append((t, ip, sz))
                    cnt[ip] += 1
                    byt[ip] += sz

        print("--")

if __name__ == "__main__":
    solve()
```队列仅存储已接受的数据包，因此计数器的每次更新都与有效贡献完全对应。 驱逐步骤在检查新数据包之前执行，以便窗口始终与当前时间戳一致。 

一个微妙的实现细节是将左边界计算为 t − δ + 1。这与包含窗口定义相匹配，并确保当 δ = 1 时正确处理，其中仅当前时间戳有效。 

决策检查使用≥，而不是>，因为达到阈值本身就已经违反了规则。

 ## 边缘情况

 当 δ 足够大以至于没有数据包过期时，就会出现关键的边缘情况。 在这种情况下，队列单调增长，但保留了正确性，因为驱逐条件永远不会触发。 该算法仍然有效，因为计数器始终反映所有可接受的历史记录。 

另一种情况是当 α 或 β 等于 1 时。对于 α = 1，来自同一 IP 的任何第二个接受的数据包立即违反约束，即使它发生在时间较晚的情况下，因为滑动窗口始终至少包括当前数据包。 该算法可以正确处理此问题，因为它在插入之前检查 c + 1 ≥ α。 

第三种情况是参数快速变化。 假设我们在宽松的约束下接受数据包，然后收紧 α 和 β。 我们不会追溯删除数据包； 相反，下一个数据包评估使用更新的约束，而队列仍然仅反映有效的窗口历史记录。 这种分离确保过去接受的数据包保持在状态，但会在新规则下正确影响未来的决策。
