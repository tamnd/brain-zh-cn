 ## 解决方案
 ## 问题理解

 我们得到一个部分指定的二进制字符串，其中一些位置固定为`0`或者`1`还有一些是未知的。 每个未知位置都可以独立地替换为`0`或者`1`。 填充所有未知数后，我们查看最终的字符串并计算有多少相邻字符对不同，这意味着有多少个索引`i`满足`s[i] != s[i+1]`。 

任务是选择替换，以便这个转换次数恰好是`k`，并且在所有有效的补全中，我们必须输出字典顺序最小的结果字符串。 如果没有完成完全可以达到`k`过渡，我们返回`Impossible`。 

这里的字典顺序意味着我们从左到右比较字符串并且更喜欢`0`超过`1`在第一个不同的位置，因此贪婪地提前推零是有益的，但前提是它不会破坏精确到达的可能性`k`稍后进行过渡。 

约束允许字符串的长度达到`10^5`每个测试用例和总长度可达`10^6`，因此每个测试用例的任何解决方案都必须是线性或接近线性的。 任何涉及检查所有完成情况甚至对位置和状态进行二次动态规划的操作都会太慢。 我们被迫采用一种解决方案，该解决方案对每个位置做出从左到右的单一决策，并通过某种形式的预计算来验证可行性。 

当贪婪地选择角色时，会出现一个微妙的困难。 局部最优选择，例如放置`0`在位置`i`可能会导致以后无法达到所需的转换次数，即使有不同的选择，例如`1`会成功的。 这意味着每一个贪婪的决定都必须根据未来的可行性进行验证。 

一些边缘情况值得指出。 

如果字符串没有`?`，我们只检查它的转换计数是否已经等于`k`。 一个天真的解决方案可能仍然尝试“优化”它并意外地更改固定字符，这将是无效的。 

如果`k = 0`，最终的字符串必须是常量，但约束可能会强制两者`0`和`1`在不同的地方，使其不可能。 例如`1?0`永远不可能变得恒定。 

如果`k = n-1`，字符串必须在每个位置交替，但固定约束可能会阻止交替，例如`0?0`，最多只能产生一个转变。 

这些情况都需要全局可行性推理而不是局部决策。 

## 方法

 蛮力方法会尝试所有可能的分配`0`和`1`为所有人`?`位置并计算每个完整字符串的转换数量。 如果有`m`未知数，这给出`2^m`可能性，以及每次检查的成本`O(n)`，导致指数爆炸，即使对于`n = 30`。 

关键的观察是字符串结构具有简单的马尔可夫属性：一旦我们将字符固定在位置`i`，对未来过渡的影响仅取决于该角色，而不取决于早期历史。 这表明对具有代表前一个字符的小状态空间的位置的动态规划公式。 

我们可以针对每个位置进行预先计算`i`以及所有可能的前一个字符`p ∈ {0, 1}`，从后缀可实现的最小和最大转换数`i`如果我们可以自由选择尊重给定固定约束的所有剩余字符，那么直到最后。 这给了我们一个可行性预言：当我们处于位置时`i`，如果我们决定一个角色`c`，我们可以立即判断剩余的后缀是否仍然可以达到所需的剩余转换计数。 

有了这个预言，我们就可以从左到右贪婪地构造答案。 在每一个岗位上，我们都在努力`0`首先检查它是否可以得出有效的完整解决方案； 如果是，我们保留它，否则我们尝试`1`。 这保证了按字典顺序排列的最小输出。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(2^n·n) | O(2^n·n) | O(n) | 太慢了 |
 | DP + 贪婪可行性| 每次测试 O(n) | O(n) | 已接受 |

 ## 算法演练

 我们将问题视为从左到右构建字符串，同时保持已经使用的转换数量。 缺少的部分是能够回答：如果我们修复前缀并在位置选择一个候选字符`i`，剩余的后缀仍然可以准确地产生所需的总转换吗？ 

### 预计算

 1.我们定义一个动态规划表`dp_min[i][p]`和`dp_max[i][p]`， 在哪里`i`是字符串中的一个位置并且`p`是前一个字符（或者`0`或者`1`）。 这些值表示可以从位置形成的最小和最大转换数`i`到`n`，假设位置处的字符`i-1`是`p`。 
2. 我们从右到左计算这些值。 在位置`i`，我们考虑允许放置哪些字符。 如果`s[i]`是固定的，我们只考虑这个值。 如果是的话`?`，我们都尝试一下`0`和`1`。 
3. 对于每个候选字符`c`，我们添加成本`1`如果`c != p`（这会在之间创建一个过渡`i-1`和`i`)，然后我们添加后缀中可能的最佳结果`i+1`前一个字符变为的位置`c`。 
4. 在所有有效选择中`c`，我们取最小值和最大值来填充`dp_min`和`dp_max`。 

经过这一步，我们知道对于任何起始状态，后缀是否可以实现一定范围的转换计数。 

### 贪婪构造

 1. 我们从位置开始构建答案`1`，到目前为止没有使用任何先前的字符和零转换。 
2. 各位置`i`，我们尝试分配`0`首先（以确保字典顺序最少），然后`1`如果需要的话。 
3. 对于候选角色`c`，我们计算它对前一个角色贡献了多少个转换。 如果`i = 1`，这个贡献是`0`。 否则就是`1`如果它与前一个字符不同。 
4. 然后我们检查可行性：选择后`c`，其余后缀必须能够实现等于的转换总数`k`。 这意味着当前的转换加上可能的最佳和最差后缀贡献必须括起来`k`。 
5. 如果可行，我们接受`c`，更新当前的转换计数，然后继续。 

### 为什么它有效

 关键的不变量是在处理位置之前`i`，我们维护一个在所有前缀中按字典顺序最小的前缀，但仍可以扩展为有效的完整解决方案。 DP 确保该位置的决策`i`是安全的：只有当存在至少一个可以满足剩余转换要求的后缀补全时，我们才会提交一个字符。 由于每个未来状态都由 DP 范围完全概括`[dp_min, dp_max]`，以后的决定不依赖于先前字符和当前计数之外的早期任意结构。 

因为每一步都保留了可行性，并且我们总是选择最小的有效字符，所以最终的字符串既有效又按字典顺序最小。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, k = map(int, input().split())
    s = input().strip()
    
    # dp[i][p] -> min/max transitions from i..n given previous char p
    # p = 0 or 1
    INF = 10**9
    
    dp_min = [[0, 0] for _ in range(n + 2)]
    dp_max = [[0, 0] for _ in range(n + 2)]
    
    for i in range(n, 0, -1):
        for p in (0, 1):
            best_min = INF
            best_max = -INF
            
            for c in (0, 1):
                if s[i - 1] != '?' and int(s[i - 1]) != c:
                    continue
                
                cost = 1 if p != c else 0
                
                mn = cost + dp_min[i + 1][c]
                mx = cost + dp_max[i + 1][c]
                
                best_min = min(best_min, mn)
                best_max = max(best_max, mx)
            
            dp_min[i][p] = best_min
            dp_max[i][p] = best_max
    
    res = []
    prev = -1
    used = 0
    
    for i in range(1, n + 1):
        for c in (0, 1):
            if s[i - 1] != '?' and int(s[i - 1]) != c:
                continue
            
            cost = 0 if prev == -1 else (prev != c)
            new_used = used + cost
            
            if new_used > k:
                continue
            
            if i == n:
                if new_used == k:
                    res.append(str(c))
                    prev = c
                    used = new_used
                    break
                continue
            
            mn = new_used + dp_min[i + 1][c]
            mx = new_used + dp_max[i + 1][c]
            
            if mn <= k <= mx:
                res.append(str(c))
                prev = c
                used = new_used
                break
        else:
            print("Impossible")
            return
    
    print("".join(res))

if __name__ == "__main__":
    solve()
```DP 部分构建仅取决于位置和前一个字符的后缀可行性范围。 这就是贪婪构造无需回溯即可发挥作用的原因。 

在构建过程中，我们通过使用小心地分离第一个字符大小写`prev = -1`，确保在字符串开始之前不计算人为转换。 可行性检查比较目标`k`针对可实现的最小和最大总数，保证我们永远不会进入不可恢复的状态。 

一个微妙的实现细节是确保我们在 DP 和贪婪阶段都尊重固定字符。 这两项检查之间的任何不匹配都将导致无效字符串被视为可行。

 ## 边缘情况

 一种边缘情况是完全固定的字符串，不允许做出任何决定。 该算法自然地处理这个问题，因为贪婪步骤首先通过固定约束进行过滤，并且 DP 可行性将接受确切的路径或拒绝某个位置的所有候选路径。 例如，输入`n = 4, k = 2, s = "0101"`通过贪婪，没有任何分支，并且与 DP 可行性完全匹配。 

另一个边缘情况是`n = 1`，无论选择什么字符，都没有过渡。 DP 基本情况正确地为任何后缀产生零转换，因此可行性减少为检查是否`k = 0`。 贪婪阶段只是选择允许的最小字符。 

当出现第三种边缘情况时`k`大于给定固定约束的最大可能转换。 在这种情况下，第一位置的每个候选人都将无法通过可行性检查，因为`k`将躺在一切之外`[dp_min, dp_max]`范围，导致立即拒绝。
