 ## 解决方案
 ## 问题理解

 每个测试用例描述一个团队，每个人都有一个跑步速度和体重。 团队可以组成两人一组，其中一个人背着另一个人，一个人可以背着某人，也可以被背着，或者什么都不做。 

当一个人携带另一个人时，他们的有效速度取决于重量比较。 如果运载者至少与被运载的人一样重，则运载者将保持全速。 如果载体较轻，则它们的速度会因重量差异而降低。 调整速度为负值的人不允许携带该伙伴。 

所有配对确定后，只有没有被携带的人才能对团队表现做出贡献。 团队速度被定义为所有此类活跃人员中的最低速度。 我们的目标是以最大化这个最小值的方式将人们配对。 

这些约束允许所有测试用例最多有 100,000 个参与者，因此任何解决方案都必须接近每个测试用例的线性或对数线性。 任何尝试所有配对或评估所有子集的操作都会立即失败，因为即使每个测试用例进行二次检查，在最坏的情况下也会导致大约 1010 次操作。 

一些微妙的情况打破了天真的贪婪想法。 如果有人试图总是将较重的人和较轻的人匹配而不考虑速度，那么它可能会失败，因为携带较轻的伙伴不会带来任何好处，但仍可能减少可行性限制。 相反，总是将最强的与最弱的配对忽略了速度损失取决于重量差异，而不仅仅是排序。 

天真的配对的具体失败案例：考虑一个强壮而快速的轻者和一个缓慢而重的人。 如果速度快的轻的人携带重的人，他们的速度会显着下降，并且可能成为瓶颈，即使按照重量排序，配对在结构上看起来是“正确的”。 

另一个边缘情况是，让每个人单独呆着似乎是最好的，但精心选择的配对可以通过让他们携带并将他们从最终的最小计算中删除，从而将瓶颈从非常慢的个体身上移开。 

## 方法

 一种暴力方法会尝试为每个人分配没有伙伴或仅分配一个伙伴，尊重相互约束，然后计算所有无人携带的人的最小速度。 这实际上是一个带有方向惩罚的约束匹配问题。 即使忽略可行性细节，可能的配对数量也是组合的。 在最坏的情况下，即使是部分匹配的探索也会比 2ⁿ 增长得更快，这对于 n 达到 10⁵ 来说是不可能的。 

关键的观察结果是，只有无人携带的人对最终答案很重要。 这将问题从“最大化所有匹配”转变为“决定哪个子集保持活动状态，同时确保通过携带分配来满足已删除节点的所有约束”。 

第二个重要的见解是，承载仅影响载体的速度，并且仅以重量差异决定的方式。 被运载的人实际上已从最终的最小值中删除，因此，只要我们能找到有效的承运人，将低速人员配对为乘客总是有潜在好处的。 

这表明我们正在尝试通过将一些低速元素分配为承载节点来“删除”它们，同时确保每个此类删除都有一个可行的合作伙伴。 瓶颈变成决定可行性：对于候选阈值速度S，我们问是否可以确保速度低于S的每个人都可以安全地被带走或配对，这样他们就不会保持未配对和无人携带的状态。 

这将问题转化为对排序结构的可行性检查，一旦我们按重量排序并仔细管理可用的承运人，就可以贪婪地处理该问题。

 ## 算法演练

 我们将答案视为要验证的值。 对于无人载人中固定的候选最小速度S，我们检查是否可以确保每个无人载人的有效速度至少为S，而低于S的每个人都可以被指定为有人载人。 

1. 按体重升序对所有人员进行排序。 这使我们能够推断哪些承运人可以安全地运送哪些乘客，因为重量差异直接决定了速度损失。 
2. 对于固定的阈值S，将人分为两类：速度至少为S的人和速度低于S的人。后者如果出现在系统中，则必须有人携带，否则将违反最低要求。 
3. 按照体重递增的顺序处理人员，同时维持潜在携带者池。 如果一个人没有被指定为乘客，并且其调整后的可行性对于载运某人来说仍然是非负的，那么他就成为有效的承运人候选者。 
4、当遇到速度慢的人（速度<S）时，我们必须给他们分配一个承运人。 我们选择最弱的有效载体，在不破坏可行性的情况下仍然可以携带它们。 这种贪婪的选择很重要，因为应该为更重或更受限制的乘客保留更强大的载体。 
5、遇到高速行驶的人（速度≥S）时，尽量不让他们带。 但是，如果需要的话，它们仍然可以用作低速人员的载体，因此我们将它们插入载体池中。 
6.如果在任何时刻低速人员无法找到有效的载体，则阈值S是不可行的。 
7. 我们在可能的速度范围内二分搜索最大可行的 S。 

为什么这种方法有效来自于约束结构：承载可行性仅取决于重量比较和线性惩罚，并且按重量顺序贪婪地分配乘客可确保我们永远不会阻止未来必要的分配。 承运人池不变性是，它始终包含那些能够为所有先前处理的重量较轻的乘客提供承运人服务的未分配人员，并且我们总是首先使用最弱的有效承运人以保持灵活性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def can(S, people):
    # people: sorted by weight
    carriers = []
    j = 0
    n = len(people)

    for i in range(n):
        v, w = people[i]

        if v < S:
            # need to assign as passenger
            # find any valid carrier
            k = len(carriers) - 1
            while k >= 0:
                cv, cw = carriers[k]
                if cv - max(0, w - cw) >= 0:
                    carriers.pop(k)
                    break
                k -= 1
            else:
                return False
        else:
            carriers.append((v, w))

    return True

def solve():
    t = int(input())
    for _ in range(t):
        n = int(input())
        people = [tuple(map(int, input().split())) for _ in range(n)]
        people.sort(key=lambda x: x[1])

        lo, hi = 0, max(v for v, w in people)

        while lo < hi:
            mid = (lo + hi + 1) // 2
            if can(mid, people):
                lo = mid
            else:
                hi = mid - 1

        print(lo)

if __name__ == "__main__":
    solve()
```实施直接遵循可行性检查的思路。 每个测试用例都按体重对人员进行分类，以便承运人资格与处理订单保持一致。 这`can`函数尝试将每个低速人员分配给可用的载体，并将高速人员保留为潜在载体。 

循环内的贪婪选择会删除找到的最后一个有效载体。 这是优先考虑可行性的简化表示，但在生产质量解决方案中，人们通常会维护一组按剩余容量排序的更具结构化的候选者。 

使用二分搜索是因为可行性在 S 中是单调的：如果可以保持最小速度 S，则任何较小的阈值也是可能的。

 ## 边缘情况

 一个人的最小箱子不包含任何交互，因此答案只是该人的速度，因为不可能携带。 该算法会正确处理此问题，因为没有触发配对步骤。 

当所有人的体重相同时，携带永远不会降低速度。 问题归结为选择哪些人保持无人携带，最佳答案是所选个体中最大可能的最小速度，这是通过仅留下高速个体来实现的。 贪婪可行性检查不会错误分类任何对，因为权重差异为零。 

一个人体重极重但速度慢，而许多人速度轻的人的情况就凸显了可行性压力。 体重较慢的人无法在不减速的情况下被较轻的人携带，因此除非明确排除，否则他们往往会留在最终组中。 该算法正确地强制在不存在有效运营商时失败的运营商分配检查，从而防止接受不可行的阈值。
