 ## 解决方案
 ## 问题理解

 此问题中的每个订单都会到达截止日期和所需的产品数量。 工厂从第一天开始每天生产固定数量的产品，并且没有初始库存。 每当一天结束时，当天的产量就会添加到库存中，然后截止日期为该天的任何订单都必须使用当前库存完全满足。 

任务是确定是否存在任何方法来处理所有订单，以便在生产统一且库存随着时间的推移而积累的情况下，每个订单在其截止日期之前得到履行。 

关键的困难在于订单并没有按照输入顺序自然对齐。 具有不同截止日期的两个订单通过共享库存进行交互，因此如果库存以错误的顺序消耗，本地可行的决策可能会在以后破坏可行性。 

订单数量方面的限制很小，每个测​​试用例最多一百个，但截止日期和要求的值可能高达十亿。 这立即排除了明确模拟每一天的可能性。 任何正确的解决方案都必须针对压缩时间事件进行操作，特别是输入中实际出现的不同截止日期。 

一个幼稚但诱人的错误是按照输入顺序甚至按增加的截止日期顺序处理订单，而不仔细处理跨时间间隙的库存积累。 例如，如果一个人贪婪地满足大量的早期需求，而不考虑后来在相同或较早的有效库存水平上的组合需求，则可能会过早耗尽库存。 

当多个订单共享相同的截止日期时，会出现另一种微妙的故障模式。 如果解决方案一一处理它们，但没有确保库存检查符合截止日期的总需求，则它可能会错误地接受多个步骤中的部分满足，而不是验证汇总的需求。 

## 方法

 暴力解释就是日复一日地模拟。 每天，我们将库存增加 k，然后检查当天到期的所有订单并尝试贪婪地履行它们。 这在原则上是正确的，因为它直接反映了所描述的过程，但当截止日期达到 10^9 时，它是完全不可行的。 即使只有 100 个订单，在最大截止日期之前进行模拟也需要每个测试用例进行最多 10^9 次迭代。 

关键的观察结果是连续两个截止日期之间没有任何变化。 产量呈线性累积，订单仅在截止日期前“激活”。 这使我们能够将时间压缩到最后期限的那几天。 

一旦时间被压缩，我们需要确保在每个截止日，截至该点的可用库存足以满足当天到期的所有订单。 然而，仅仅每天独立检查是不够的，因为库存是跨天共享的。 正确的观点是按升序处理截止日期，并维持到当天所需的总需求。 

在每个不同的截止日期，我们计算到目前为止已经生产了多少库存，即 k 乘以当天指数，然后将其与截止日期为当天或更早的所有需求的总和进行比较。 如果在任何时候需求超过供应，时间表就不可能实现。 

这将问题转化为对排序事件的前缀可行性检查。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 日常模拟 | O(最大 ai) | O(1) | O(1) | 太慢了|
 | 按截止日期排序 + 前缀检查 | O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 让我们逐步描述一下工作方法。

1. 按截止日期对所有订单进行分组。 每天汇总所有所需数量。 这可以防止在同一截止日期内重复处理多个订单，并确保我们仅跟踪每个关键时间点的总需求。 
2. 按升序对不同的截止日期进行排序。 这是必要的，因为可行性取决于每个时间点的累积产量。 
3. 维护所需产品的运行前缀和，称为`need`。 当我们迭代排序的日期时，我们将当天的需求添加到`need`。 
4. 每个截止日期`d`，计算可用库存为`k * d`。 这代表从第 1 天到第 d 天的总产量。 
5. 如果在任何时候`need > k * d`，立即返回“否”，因为即使所有未来的生产都得到完美分配，工厂也已经未能在截止日期前完成任务。 
6. 如果所有截止日期均满足此条件，则返回“是”。 

重要的推理步骤是，延迟履行超过截止日期是不可能的，因此必须在每个截止日期边界准确检查可行性，而不是在最后全局检查。 

### 为什么它有效

 在任何时候，库存完全由时间决定，与我们如何将其分配给订单无关。 唯一的限制是截止日期之前的累积需求不得超过同一点之前的累积产量。 如果这对于每个截止日期前缀都成立，那么我们总是可以按时间顺序贪婪地分配生产单位，因为较早的生产总是可以满足较早的截止日期，并且剩余库存自然会结转。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        n, k = map(int, input().split())
        by_day = {}
        
        for _ in range(n):
            a, b = map(int, input().split())
            by_day[a] = by_day.get(a, 0) + b
        
        days = sorted(by_day.keys())
        
        need = 0
        ok = True
        
        for d in days:
            need += by_day[d]
            if need > k * d:
                ok = False
                break
        
        print("Yes" if ok else "No")

if __name__ == "__main__":
    solve()
```该实现使用字典按截止日期对订单进行分组，这确保了同一天的多个订单合并为单个累积需求。 这很重要，因为拆分它们不会改变可行性，但会使跟踪变得复杂。 

对键进行排序可以建立按时间顺序的处理。 变量`need`充当需求的运行前缀总和，在每一步我们将其与生产能力进行比较`k * d`。 由于任意精度整数，乘法在 Python 中是安全的，但在固定宽度语言中，使用 64 位算术很重要。 

早期中断是一种保留正确性的优化，因为一旦前缀违反容量，则任何进一步的处理都无法恢复可行性。

    return solve()

 ## 边缘情况

 一种极端情况是所有订单都有相同的截止日期。 在这种情况下，算法会将所有需求合并为一个值，并将其与当天的 k 次进行比较。 例如，输入`(2,1),(2,1),(2,1)`k = 2 产生 need = 3 和capacity = 4，这是正确接受的。 

另一个边缘情况是截止日期稀疏时，例如第 10^9 天的单个订单。 该算法不模拟中间天数，而是直接将容量计算为 k * 10^9，避免任何时间爆炸，同时仍然检查所需边界的正确性。 

最后一个微妙的情况是，早期的截止日期很紧，但后来的截止日期很宽松。 前缀比较确保一旦早期累积需求可行，后期的松弛就不会影响正确性，因为剩余产能通过递增生产函数隐式结转。
