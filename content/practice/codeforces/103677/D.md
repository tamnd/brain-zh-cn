 ## 解决方案
 ## 问题理解

 我们有一组葡萄类型，每种类型都有一个所需的最低使用量。 酒庄生产瓶装葡萄酒，每一瓶都是经过精挑细选而成$n-1$不同的葡萄品种$n$可用，每种所选类型使用一个单元。 

因此，除了该瓶中排除的一种选定类型之外，每瓶都会为所有葡萄类型贡献 +1 使用量。 

我们的任务是确定所需的最少瓶子数量，以便每种葡萄类型$i$至少使用$a_i$次。 

换句话说，每个瓶子“错过”了一种葡萄类型，我们必须安排这些错过，这样就不会错过任何类型太多次，因为错过意味着该瓶子中没有使用该葡萄品种。 

关键的隐藏结构是，葡萄的使用总次数仅取决于有多少瓶避免使用它。 如果我们做$k$瓶子，然后是葡萄类型$i$准确地使用了$k - b_i$时间、地点$b_i$是有多少瓶排除它。 约束变为：$$k - b_i \ge a_i \quad \Rightarrow \quad b_i \le k - a_i$$每个瓶子只排除一种类型，因此所有排除的总和恰好是$k$：$$\sum b_i = k$$我们正在选择如何在每种类型的上限下分配这些排除项。 

来自约束$n \le 10^5$和$a_i \le 10^9$，任何试图一一模拟瓶子的解决方案都是不可能的。 简单的模拟可能需要多达$10^9$每种类型的操作，远远超出了时间限制。 甚至$O(nk)$方法立即被排除，因为$k$本身可以很大。 

当所有的情况出现时，就会出现微妙的边缘情况$a_i$是平等的。 例如，如果所有值都是 4，那么对称性表明排除项的平衡分布，如果不仔细分析，贪婪的“总是选择当前最小满足类型”可能会失败。 另一个棘手的情况是当一个$a_i$比其他类型大得多，迫使大多数瓶子尽早排除该类型，否则其他类型无法满足。 

## 方法

 暴力解释是一次模拟一个瓶子。 对于每一瓶，我们选择要排除的葡萄类型并更新使用计数。 这是正确的，因为它直接遵循流程定义。 然而，每个瓶子都需要在其中选择一个排除项$n$类型，我们可能需要最多$k$瓶子，哪里$k$可以是以下顺序$10^9$。 这会导致最坏情况的复杂性$O(nk)$，这是不可行的。 

关键的见解是扭转观点。 我们不是决定每瓶中使用哪些葡萄，而是决定每种葡萄被排除多少次。 每一项排除都会减少该葡萄的使用贡献。 问题变成了分布$k$相同的排除标记$n$桶，每个桶$i$有能力$k - a_i$。 

这将问题转化为给定的可行性检查$k$。 一旦我们检查了可行性，我们就可以二分查找最小值$k$。 可行性条件简化为检查是否：$$\sum \max(0, k - a_i) \ge k$$因为每个$k - a_i$是输入的最大次数$i$可以被排除，排除总数必须精确等于$k$。 

然后我们可以二分查找最小的$k$满足这个不等式。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力模拟|$O(nk)$|$O(n)$| 太慢了 |
 | 二分查找+可行性检查|$O(n \log A)$|$O(n)$| 已接受 |

 这里$A = \max a_i$。 

## 算法演练

 ### 最优算法

 1.观察固定数量的瓶子$k$, 每种葡萄类型$i$除了那些被排除在外的瓶子外，所有瓶子都使用它。 这给出了用法$k - b_i$， 在哪里$b_i$是分配给类型的排除数$i$。 
2. 翻译需求$k - b_i \ge a_i$对排除的约束：$b_i \le k - a_i$。 这告诉我们可以“跳过”每种类型多少次。 
3.对于固定的$k$，计算我们可以在所有类型中分配多少总排除项。 这是$\sum \max(0, k - a_i)$，因为类型不能被否定排除，并且如果$a_i > k$，它没有任何贡献。 
4. 检查可行性：总可用排除容量是否至少为$k$，然后我们可以分配每个$k$在不违反任何每种类型限制的情况下，将其装瓶为独特的排除类型。 否则的话，那是不可能的。 
5. 使用二分查找$k$。 下界是$0$，安全上限是$2 \cdot \max(a_i)$，因为除此之外，所有约束都变得可以满足。 
6. 对于二分查找中的每个中间值，在线性时间内对所有值运行可行性检查$a_i$。 

### 为什么它有效

 该算法依赖于每个瓶子恰好贡献一次排除的不变量，因此系统相当于分配$k$将相同的标记放入有界容器中。 可行性检查正确地表征了容器的总容量是否足以容纳所有代币。 如果容量不足，至少一种葡萄品种必须被排除超过允许的次数，这违反了其要求。 如果容量足够，则始终存在排除的贪婪分配，因为超出总容量和每种类型限制的约束之间不存在耦合。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def can(k, a):
    total = 0
    for x in a:
        if k > x:
            total += k - x
    return total >= k

def main():
    n = int(input())
    a = list(map(int, input().split()))

    lo, hi = 0, 2 * max(a)

    while lo < hi:
        mid = (lo + hi) // 2
        if can(mid, a):
            hi = mid
        else:
            lo = mid + 1

    print(lo)

if __name__ == "__main__":
    main()
```代码的核心结构是可行性函数`can(k, a)`，它计算在候选瓶数下可以分配多少个排除项。 每个条目仅在以下情况下做出贡献：$k$超过它，否则该葡萄即使一次也不能被排除而不违反其要求。 

二分查找通过反复测试给定数量的瓶子是否足够来缩小答案范围。 单调性来自于增加$k$永远不会减少排除容量，并且总是线性增加所需的排除。 

一个常见的实施陷阱是忘记条件是全局的：我们不会贪婪地为每个瓶子分配排除项，而是仅依赖于总容量。 任何模拟每瓶决策的尝试都有可能使逻辑过于复杂并引入排序错误。

 ## 边缘情况

 一种葡萄类型的最小输入，例如$n=1, a_1=1$，迫使算法纯粹依赖于可行性不等式。 为了$k=1$，容量为零，所以不可行，而对于$k=2$，容量变为正数并满足条件。 算法正确返回 1。 

严重倾斜的输入，例如$a = [10^9, 1, 1]$强制解决方案将几乎所有排除项分配给第一种类型。 为了$k = 10^9$，第一种类型贡献零容量，而其他类型贡献大容量，使得可行性检查仅在边界处正确通过。 

统一数组，例如$a_i = 5$测试对称性。 对于小$k$，容量不足，但一旦$k$超过 5，每种类型的贡献均等，可行性呈二次方增长$k$，确保二分查找收敛到正确的平衡值。
