[CF 105062A - 是否评级？](https://codeforces.com/problemset/problem/105062/A)

 **评级：** -
 **标签：** -
 **求解时间：** 1m 12s
 **已验证：** 是的

 ## 解决方案
 ## 问题理解

 我们有两个整数，`m`和`d`。 想想`m`作为日历年中的月份数字以及`d`作为一个月中的某一天。 任务是确定该对是否可以代表问题所暗示的简化世界中的有效日历日期。 

实际上，唯一重要的限制是这一天是否`d`甚至对于给定的月份也有意义`m`。 由于没有明确定义年份或日历系统，因此问题隐式地简化为检查该日期是否存在于该月的允许范围内。 

关键的观察是，月份的不同之处在于它们可以包含多少天，以及它们的有效性`(m, d)`取决于该结构而不是任何计算技巧。 

输入尺寸很小，`m ≤ 12`和`d ≤ 31`，这立即排除了任何优化问题的需要。 任何直接编码日历规则甚至硬编码有效对的解决方案就足够了。 

一个微妙的边缘情况来自于不同月份的日期限制不同这一事实。 例如，天真的方法可能会假设每个月有 31 天，这会错误地接受现实日历解释中的无效日期，例如 2 月 30 日或 4 月 31 日。 另一个潜在的陷阱是忘记最小的月份仍然允许至少 28 或 30 天，具体取决于解释，并且错误地过度限制有效对。 

例如，如果有人错误地假设所有月份都有 31 天，他们会接受`(2, 31)`有效，这显然是不正确的。 另一方面，如果有人假设所有月份都有一个统一的较小界限，例如 30 天，他们会错误地拒绝`(1, 31)`。 

因此，问题归结为正确区分哪些月份允许哪些最大天数。 

## 方法

 蛮力思维方式是将每个可能的日历日期视为有效，然后检查是否`(m, d)`出现在所有有效日期的预先计算列表中。 这意味着根据日历规则枚举所有月份以及每月所有可能的日期，将它们存储在一个集合中，并检查成员资格。 这是可行的，因为域非常小，最多只有几百个条目。 

问题不在于性能，而在于不必要的复杂性。 当结构本身固定且微不足道时，构建和查询结构就显得有些过分了。 

更直接的方法是对月份与最大天数的关系进行编码。 月份的长度在 31 天和 30 天之间交替，二月是一个特殊情况。 由于约束如此之小，最干净的解决方案就是简单地硬编码每月的最大天数并进行比较`d`反对它。 

关键的见解是这根本不是一个动态问题。 测试用例或计算之间不存在依赖关系； 答案完全是通过不断的查找来确定的。

 ## 算法演练

 1. 构建每个月的映射`m`至其最大有效天数。 这反映了固定的日历结构，其中每个月都有一个已知的上限`d`。 
2.读取输入对`(m, d)`。 
3.比较`d`与月份允许的最大值相比`m`。 
4.如果`d`在范围内，输出`"YES"`，否则输出`"NO"`。 

第 1 步背后的原因是，所有未来的检查都依赖于恒定的规则，因此预先计算它可以避免重复逻辑或条件。 

### 为什么它有效

 正确性取决于每个月有固定的最大有效天数的不变量，并且任何有效日期必须满足`1 ≤ d ≤ max_days[m]`。 由于映射对日历的精确约束进行了编码，因此每个有效输入都会通过不等式检查，而每个无效输入都会违反不等式检查。 问题中不存在其他结构或隐藏的依赖关系，因此比较是必要且充分的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    m, d = map(int, input().split())

    # days per month in a standard calendar abstraction
    max_days = {
        1: 31,
        2: 28,
        3: 31,
        4: 30,
        5: 31,
        6: 30,
        7: 31,
        8: 31,
        9: 30,
        10: 31,
        11: 30,
        12: 31
    }

    if 1 <= d <= max_days[m]:
        print("YES")
    else:
        print("NO")

if __name__ == "__main__":
    solve()
```该代码直接将日历结构编码到字典中。 这避免了超出单个查找和比较的任何分支逻辑。 关键细节是确保边界包含在内，因为两个端点都是有效日期。 

一个微妙的实现细节是字典必须正确地将二月反映为 28 天，因为忘记这一点是类似问题中错误答案的最常见原因。 其余月份遵循固定模式。

 ## 边缘情况

 一种极端情况是 31 天的月份中的最大有效天数。 用于输入`1 31`，算法检索`max_days[1] = 31`并检查`31 ≤ 31`，通过，产生`"YES"`。 这证实了上边界是包含在内的。 

另一个极端情况是二月溢出。 用于输入`2 29`，查找返回`28`，以及条件`29 ≤ 28`失败，所以输出是`"NO"`。 这表明特定于月份的约束得到了正确执行。 

最后一个极端情况是一般的 30 天月份，例如 4 月。 对于输入，例如`4 30`，映射给出`30`，并且相等性通过，确保仍然正确处理少于 31 天的月份，而无需超出查找范围的特殊情况。
