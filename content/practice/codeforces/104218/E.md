 ## 解决方案
 ## 问题理解

 我们得到了一个代表直山沿线积雪高度的数组。 值是非负的，并且序列从左到右单调非递减。 每个查询都要求一个连续的段，其元素之和恰好等于给定值$K$。 在所有有效段中，我们必须返回“最小”的一个，这意味着它具有最短的长度，如果多个段共享相同的最小长度，我们选择数组中最早出现的一个。 

关键操作不是对数组进行计数或变换，而是在排序约束下定位具有精确目标和的子数组。 

这些限制以非常具体的方式发挥作用。 数组长度可以达到$10^5$，而查询次数最多为$100$。 这立即排除了每个查询都是二次的任何解决方案，因为$Q \cdot N^2$会太大。 每个查询线性扫描数组的解决方案是可以接受的，但任何比每个查询线性更差的解决方案都会失败。 

数组的单调结构并不是主要技术的正确性所直接需要的，但它保证所有值都是正数，这是我们所依赖的真正的结构属性。 

如果采用一种天真的“尝试所有子数组”方法，就会出现一种微妙的失败情况：

 输入：```
5 1
1 2 3 4 5
5
```暴力方法可能会发现`[0,2]`（总和 6）考虑不正确或可能提前停止，具体取决于实施情况。 正确答案是`[1,2]`或者`[4,4]`取决于解释，但仅`[1,2]`对于最小长度为 2 的和 5 有效，并且`[4,4]`也适用于长度 1，这实际上更好，所以正确的答案是`[4,4]`。 This illustrates why correctness depends on systematically exploring all valid windows, not stopping at the first match.

 Another edge case is when multiple identical values exist, producing many overlapping valid segments. 仅记录每个左边界的第一个匹配的粗心实现可能会错过稍后出现的较短片段。 

## 方法

 The most direct approach is to examine every possible subarray, compute its sum, and check whether it equals$K$。 这是正确的，因为它枚举了所有候选人。 然而，每个查询都需要$O(N^2)$检查所有子数组的时间，最多 100 个查询就变成了$10^7$子数组，每个子数组都需要不断的工作，这已经是临界点，而且一旦包含开销，通常会太慢。 

关键的观察结果是所有因素都是积极的。 这允许滑动窗口技术：一旦窗口总和超过$K$，进一步扩大只会增加总和，因此我们可以放心地向前移动左指针来减少总和。 这种单调行为确保每个指针仅在数组中向前移动，从而为每个查询提供线性复杂度。 

我们维护一个运行窗口并增量调整它，而不是重新计算每个子数组的总和。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O(N^2)$每个查询 |$O(1)$| 太慢了|
 | 滑动窗 |$O(N)$每个查询 |$O(1)$| 已接受 |

 ## 算法演练

 对于每个查询值$K$，我们使用两个指针独立扫描数组。 

### 步骤

 1.初始化两个指针$l = 0$和$r = 0$，以及运行总和$s = 0$。 

窗户$[l, r)$代表当前正在考虑的部分。 
2. 通过移动扩大右边界$r$从左到右。 每次增量后，将新元素添加到$s$。 

此步骤确保我们探索有效段的每个可能的结束位置。 
3. 如果在任何时候$s > K$，通过递增从左侧缩小窗口$l$并从中减去元素$s$直到$s \le K$。 

这是有效的，因为所有值都是非负的，因此缩小是减少总和的唯一方法。 
4. 什么时候$s == K$,记录当前间隔$[l, r]$作为候选答案。 

对于这个固定的窗口来说，这个窗口是最小的$r$，因为任何进一步的缩小都会打破平等。 
5. 记录迄今为止看到的最佳间隔。 “最佳”意味着最小的长度在前，如果并列，则最小的起始索引。 
6. 继续，直到右指针到达数组末尾。 

### 为什么它有效

 在任意固定的右端点$r$，该算法确保左指针位于最小索引处，使得$[l, r]$至多是$K$。 因为所有值都是非负数，所以任何较小的值$l$只会增加总和，这意味着它会超过$K$。 因此，如果有效段结束于$r$如果存在，算法将仅以最小形式找到它一次。 由于处理了每个可能的右端点，因此会考虑所有有效的候选端点，并在其中选择全局最佳端点。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, q = map(int, input().split())
    a = list(map(int, input().split()))

    for _ in range(q):
        k = int(input())

        l = 0
        s = 0

        best_len = float('inf')
        best_l = -1
        best_r = -1

        for r in range(n):
            s += a[r]

            while l <= r and s > k:
                s -= a[l]
                l += 1

            if s == k:
                length = r - l + 1
                if length < best_len or (length == best_len and l < best_l):
                    best_len = length
                    best_l = l
                    best_r = r

        print(best_l, best_r)

if __name__ == "__main__":
    solve()
```该解决方案独立处理每个查询，每次都重置滑动窗口状态。 内循环结束$r$每个元素扩展窗口一次，同时内部收缩循环移动$l$仅在必要时转发。 这保证了每个查询的线性行为。 

平局决胜逻辑显式跟踪间隔长度和起始索引，确保即使多个候选者共享相同的总和，也会选择最早的最小段。 

一个常见的错误是假设第一次$s == K$就足够了。 当稍后出现更短的有效段且具有更好的左边界时，这种情况就会失败。 

## 工作示例

 ### 示例 1

 输入：```
6 3
2 2 3 4 5 6
4
7
16
```供查询$K = 4$:

 | r | 我| 窗口| 总和| 行动| 最好的|
 | --- | --- | --- | --- | --- | --- |
 | 0 | 0 | [2] | 2 | 展开 | - |
 | 1 | 0 | [2,2]| 4 | 记录| [0,1]|
 | 2 | 0 | [2,2,3]| 7 → 收缩 | 3 | - |

 最佳间隔是`[0,1]`，但由于动态缩小和后来的改进，正确的最小单元素匹配取决于扫描进度； 最终选择的是`[3,3]`当重新平衡导致稍后端点完全匹配时。 

为了$K = 7$，窗口最终稳定在`[2,3]`。 

为了$K = 16$, 完整前缀`[0,4]`成为第一个有效的最小段。

 ## 边缘情况

 具有许多重复值的情况，例如`1 1 1 1 1`测试算法是否仅仅因为它们出现较早而错误地选择较长的窗口。 滑动窗口仍然独立地评估每个端点，因此当右指针到达匹配位置时总是找到最短的有效段。 

一个案例，其中$K$等于单个元素可确保算法在记录有效窗口之前不会进行不必要的扩展。 正确的行为是立即识别$s == K$在单个索引上。 

一个案例，其中$K$很大并且只能在数组末尾附近实现，这确认了早期的部分匹配不会被错误地作为最终答案返回，因为每个查询都会在整个扫描中跟踪最佳候选者。
