 ## 解决方案
 ## 问题理解

 我们有两个长度相等的字符串。 一个玩家可以随时自由更改任一字符串的任何字符，而另一个玩家可以通过一次移动将整个字符串首尾翻转。 一旦两个字符串变得相同，这个过程就会立即停止，目标是了解如果两个玩家都表现最佳，一个试图尽快完成，另一个试图尽可能延迟相等，这个过程会持续多长时间。 

关键的困难在于游戏并不是要达到固定的目标状态。 第二个玩家可以不断地改变任一弦的方向，这实际上意味着系统可以随时在弦和反弦之间切换。 因此，“匹配位置”的概念是不稳定的：一次移动中的位置 i 可能对应于下一次移动中的位置 n−i+1 。 

输入大小允许字符串最多 100,000 个字符，因此任何模拟移动或明确考虑游戏每一步的解决方案都是不可能的。 即使是 O(n²) 推理也太慢了。 该结构表明，答案必须来自两个弦的少量全局属性，而不是来自动态模拟。 

一个微妙的边缘情况是字符串已经相等。 然后游戏立即以零步结束。 另一个重要的角落是当琴弦彼此相反时，因为第二个玩家的一次翻转可以根据第一个玩家的动作立即同步它们。 

## 方法

 如果我们尝试直接模拟游戏，我们会立即遇到指数分支。 每次移动后，状态取决于更改了哪个字符以及翻转了哪个字符串。 即使我们忽略最优游戏，状态空间也会随着我们交替进行任意修改和全局逆转而增长。 这使得直接的博弈树探索变得不可行。 

关键的简化来自于关注对平等真正重要的因素。 在任何时刻，如果 S 等于 T 或 S 等于反向(T)，则两个字符串被视为相等，具体取决于最后一次翻转如何影响方向。 由于第二个玩家始终可以选择要翻转的字符串，因此系统可以有效地在两种可能的对齐方式之间交替：直接对齐和反向对齐。 

一旦我们修正了这个观点，Alice的操作就变得极其强大。 她可以通过覆盖字符来纠正单个动作中的任何不匹配。 然而，Bob 的逆转可以通过全局交换位置来立即撤销局部对齐结构。 

这就造成了一种情况，即只有弦的“全局兼容性”才重要，并且游戏减少了强制配置所需的步骤，其中一个动作可以使弦在任一方向下都相同。 

关键的观察是，答案仅取决于字符串在一次修改后是否可以相同或反向相同，以及鲍勃是否可以迫使爱丽丝需要额外的校正步骤。 这将问题分解为检查 S 和 T 之间的对称关系及其反转，并推理对抗性反转下所需的最小编辑量。

 ## 算法演练

 我们首先直接比较两个字符串。 如果它们已经相同，则答案为零，因为游戏在任何移动之前停止。

接下来，我们将 S 与 T 进行比较，并将 S 与反向(T) 进行比较。 这两个对齐代表了鲍勃可以通​​过重复翻转来强制执行的唯一有意义的稳定配置。 其他一切都归结为这两种观点之一。 

然后我们计算 S 和 T 之间以及 S 和反向（T）之间的不匹配。 设它们为 d1 和 d2。 这些值代表如果鲍勃从不提供帮助且从不干涉的情况下，爱丽丝将需要多少次直接编辑。 

然而，鲍勃的角色并不是被动的。 通过每次移动翻转一根弦，他可以迫使爱丽丝陷入纠正一个位置不一定会以稳定的方式减少有效失配的情况。 每次翻转都可能会改变对齐方式，这意味着孤立的修正并不总是立即有效。 

关键的见解是鲍勃一次最多可以维持一个“对齐模糊性”。 这意味着有效的游戏长度取决于最佳方向下所需的最小修正次数，如果两个方向非常不同并且都不能在一次移动中立即修复，则还要加上额外的惩罚。 

具体来说，我们计算 S 和 T 之间以及 S 和反向（T）之间的最小不匹配。 如果两个配置在开始时都不相同，则答案是该最小值加一。 

经过这些计算后，我们输出结果值。 

正确性来自于这样一个事实：Alice 的每一步都可以精确修复一个字符不匹配，而 Bob 的翻转只能全局反转对齐，但不能减少不匹配计数。 因此，游戏简化为达到这样一种状态，即一次最终校正使字符串相同，而鲍勃的最佳策略是始终保留一种配置，使至少一个不匹配问题无法解决，直到被迫为止。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input().strip())
    s = input().strip()
    t = input().strip()

    if s == t:
        print(0)
        return

    def dist(x, y):
        return sum(a != b for a, b in zip(x, y))

    t_rev = t[::-1]

    d1 = dist(s, t)
    d2 = dist(s, t_rev)

    print(min(d1, d2))

if __name__ == "__main__":
    solve()
```该解决方案首先处理简单的相等情况。 然后，它根据第二个字符串的两个可能方向计算不匹配计数，因为随着时间的推移，Bob 始终可以强制执行 T 或反向(T)。 最终答案是 Alice 在最佳游戏下可以达到的最佳失配计数。 

该实现使用直接线性扫描来查找不匹配，这在 n 最多 100,000 下是安全的。 反转字符串也是线性的，但只完成一次。

 ## 边缘情况

 对于相等的字符串，算法会正确返回零，因为提前退出会在任何不匹配计算之前触发。 对于反转相等的字符串，第二个比较占主导地位，在反转下产生零不匹配，这反映了 Bob 始终可以保持 Alice 可以用最少的步骤解决的对齐问题。 

对于完全不相关的字符串，两个方向都会产生大量的不匹配计数，并且无论翻转如何，最小值都能正确反映最佳校正路径。
