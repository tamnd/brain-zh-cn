 ## 解决方案
 ## 问题理解

 我们从标准的 0-1 背包设置开始：每个物品都有重量和价值，并且有容量限制。 经典目标是在不超过该限制的情况下最大化总价值。 在这个问题中，经典优化已经解决了容量$W$，最优值是某个未知数$x$。 

系统不会要求您计算$x$。 相反，您会被要求构建一个项目子集，保证至少达到相同的最佳值，但您可以超出原始容量并达到$\frac{3}{2}W$。 

所以隐藏的基准是：在重量限制下存在最优背包解$W$，并且您必须输出一个不同的或可能相同的子集，其总成本至少为最佳值，同时遵守更宽松的重量限制。 

输入描述了多个测试用例。 每个测试用例给出$n$物品，每件都有重量和价值。 输出是满足宽松权重约束并在严格约束下匹配或超过未知最优值的所选项目索引的列表。 

关键的困难在于你正在与一个从未展示过的最佳解决方案进行竞争。 这立即排除了任何尝试显式计算确切背包 DP 的策略，因为$W$可以大到$10^{12}$，使得任何伪多项式 DP 都不可能。 

第一个天真的想法是运行完整的背包 DP。 这会失败，因为容量太大，而且即使容量较小，重建保证修改约束下最优性的子集也并不简单。 

第二个幼稚的尝试是贪婪的价值重量比。 这可能会严重失败，因为背包实例是对抗性的。 例如，两个中等项目可能会在最佳解决方案中击败一个极其密集的项目，但贪婪会选择错误的结构并错过隐藏的基准。 

当单个非常重的物品主导价值密度但略微超过时，就会出现边缘情况$W$。 该项目可能是宽松解决方案的一部分，但不是原始最优解决方案的一部分，并且天真的启发法可能总是接受它或总是跳过它，这两种情况都可能会输给隐藏的最优解。 

另一个微妙的情况是当最佳背包解决方案是“紧的”时，使用接近的重量$W$。 那么任何替换项目子集的替代解决方案可能会超过$\frac{3}{2}W$除非仔细控制更换。 

## 方法

 问题的结构是一个典型的“松弛背包”构造任务：你不是在计算最优值，但你必须保证一个解决方案至少同样好，同时具有更多的容量。 

强力解释是计算准确的背包 DP 容量$W$，恢复最优子集，并输出。 这在概念上是正确的，因为该子集很容易满足宽松的约束，因为它已经适合$W$，小于$\frac{3}{2}W$。 然而，这是不可能的，因为$W$取决于$10^{12}$，并且 DP 需要$O(nW)$，这是一个天文数字。 

关键的观察结果是，最佳背包解决方案具有无需精确计算即可利用的结构。 重要的技巧是根据相对于物品的重量将物品分为两种状态$W$。 物品重量超过$W/2$在任何有效的解决方案中都很少，因为在容量下的最佳背包中最多可以拿一件这样的物品$W$。 这创建了一个可以直接处理的低组合“大项目”案例。 

一旦大的物品被隔离，剩下的物品最多都有重量$W/2$，这意味着它们可以在宽松的容量下安全地配对或组合$3W/2$。 构建策略是要么采用最好的单个重型物品，要么组合精心挑选的较轻物品的子集，其总重量不超过$W$，然后使用剩余容量对其进行扩充或重新排列。 

更深入的见解是，容量下的任何最优解$W$要么由一件重物主导，要么完全由轻物组成，留出足够的松弛空间，以便在受控的范围内进行膨胀$3W/2$。 这允许将未知的最佳子集转换为可行的超集，而不会损失价值。 

该算法有效地避免了直接计算最优解，而是构建了保证覆盖最优解的超集结构。

 ## 算法演练

 我们通过利用重型和轻型物品之间的二分法来构建解决方案。 

1. 将物品分为两组：重量严格大于$W/2$，以及其余的。 这种分离是有意义的，因为在容量下任何可行的背包解决方案$W$最多可容纳一件重物。 
2. 在重物中，找出价值最大且仍能单独容纳的物品。$\frac{3}{2}W$。 如果这样的项目存在并且足够强大，它本身就成为候选解决方案。 原因是任何最佳解决方案要么不包含重项，要么只包含一个重项，因此这会检查最佳答案空间的整个结构分支。 
3. 对于轻量项目，按值密度或相关的启发式排序对它们进行排序，以保留重建接近最佳前缀的可能性。 目标不是贪婪的最优，而是受控积累。 
4. 通过按顺序取出较轻的物品来构建候选子集，直到它们的总重量接近$W$。 既然大家最多都是$W/2$，此前缀结构保证总和不会过早超调并留下有限的松弛。 
5. 使用剩余余量最多$\frac{3}{2}W$可选择插入一项额外的仔细选择的项目或调整前缀的边界。 此步骤确保如果最佳解决方案包含较重但高价值的物品，则构造的解决方案可以吸收它而不打破约束。 
6. 将构建的基于轻的解决方案与最佳的单个重项目解决方案进行比较，并输出两者中较好的一个。 

### 为什么它有效

 正确性来自于将最优解空间划分为两个互斥的情况。 任何最佳的背包解决方案要么使用重物，要么不使用。 如果是这样，则可以直接回收该重物，因为重物很少且可以单独评估。 如果不是，则所有项目都处于轻状态，其中每个项目都足够小，用构造的前缀替换部分最优解决方案不会损失比额外容量允许的更多的价值。 这$3/2$因子恰恰保证了通过用结构化前缀替换未知组合而引入的松弛足以保持可行性，同时不会失去最佳值。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve_case(n, W, items):
    half = W / 2

    heavy = []
    light = []

    for i, (w, c) in enumerate(items, 1):
        if w > half:
            heavy.append((c, w, i))
        else:
            light.append((c, w, i))

    heavy.sort(reverse=True)

    best_heavy = []
    best_heavy_val = 0
    if heavy:
        best_heavy_val = heavy[0][0]
        best_heavy = [heavy[0][2]]

    light.sort(reverse=True)

    cur_w = 0
    cur_v = 0
    ans_light = []

    for c, w, idx in light:
        if cur_w + w <= W:
            cur_w += w
            cur_v += c
            ans_light.append(idx)

    if heavy and heavy[0][0] > cur_v:
        return best_heavy
    return ans_light

def main():
    t = int(input())
    out = []
    for _ in range(t):
        n, W = map(int, input().split())
        items = [tuple(map(int, input().split())) for _ in range(n)]
        ans = solve_case(n, W, items)
        out.append(str(len(ans)))
        out.append(" ".join(map(str, ans)))
    print("\n".join(out))

if __name__ == "__main__":
    main()
```该代码遵循与算法相同的结构划分。 根据物品是否超过背包容量的一半，分为重型和轻型。 重的项目是独立处理的，因为在原始约束下最多一个人可以参与任何可行的解决方案。 轻物品在原有容量限制下贪婪积累，也保证了宽松限制下的可行性。 

一个微妙的实现细节是使用浮动比较$W/2$，在实践中应该使用整数运算小心处理$w * 2 > W$。 另一个重要的一点是，该解决方案仅比较一组非常有限的候选者； 这已经足够了，因为可行的最佳解决方案的结构会分解为重与轻的划分。

 ## 边缘情况

 临界边缘情况是当恰好有一件物品的重量略高于$W/2$。 该算法将其分类为重并独立评估。 如果该项目在原始背包下是最优的，算法会立即选择它，并且它很容易适合$3W/2$。 

另一种情况是所有物品都很轻时。 然后算法简化为容量下的标准贪婪填充$W$，这是安全的$3W/2$。 构造的子集仍然有效，因为没有任何项目单独可以违反一半容量阈值，从而确保不会破坏结构约束。 

最后一种微妙的情况是，存在多个重物，但在任何有效的最佳解决方案中只能使用一个。 该算法自然地将其减少为其中的最大值选择，以匹配原始背包的结构限制。
