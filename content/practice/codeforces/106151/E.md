 ## 解决方案
 ## 问题理解

 我们给出了一组目标，每个目标都有两个参数：基本能量成本和疲劳因子。 运动员可以选择这些目标的任何子集并决定击中它们的顺序。 如果目标在第 j 次成功命中时被击中，则其成本为其基本成本加上与 j 减 1 成比例的附加惩罚，并按其疲劳系数缩放。 我们的目标是在总能源不超过预算的情况下最大限度地实现多少目标，并且在实现最大计数的所有方法中，我们还报告了可能使用的最低能源。 

困难在于，排序非常重要，因为疲劳项随着位置的增加而增加。 如果稍后放置具有较大疲劳系数的目标，其成本会显着提高，因此排序选择并不独立于子集选择。 问题本质上是在问：如果它们的成本取决于它们在所选序列中的排名，我们可以选择多少个项目，以及最大基数的最小可实现成本是多少。 

这些限制表明单纯的阶乘或子集枚举方法是不可能的。 当 N 达到 10^4 时，每个子集大小的任何 O(N^2) 或更差的构造都太慢。 我们需要更接近 O(N log N) 或 O(N sqrt N) 的东西。 

一些边缘行为值得强调。 

如果所有疲劳系数都为零，那么顺序并不重要，我们只需选择最小的基本成本，直到超出预算。 贪婪排序解决方案的作用微不足道。 

如果所有基本成本都为零，那么问题就纯粹是关于疲劳排序，因为后来的选择会累积更多的惩罚。 尽早选择低疲劳项目是最佳选择，建议按 b 排序。 

当贪婪策略忽略基础和疲劳之间的相互作用时，会发生更微妙的失败情况。 例如，首先选择最小的 ai 不一定是最佳的，因为如果尽早选择，稍微大一点的 ai 和小得多的 bi 可能会变得更便宜。 

示例：

 N = 3，B = 10

 a = [5, 4, 4]

 b = [10, 1, 1]

 如果我们按 ai 递增顺序选择：4 (0)、4 (1)、5 (2)，则成本为 4、5、25，导致总计 34，这是无效的。 但首先选择高基础成本可能会减少以后的处罚。 

这说明单独用a排序是错误的，单独用b排序也是不够的。 前缀大小和斜率之间的相互作用是关键难点。 

## 方法

 暴力的想法是尝试每个可能的子集大小 k 并检查我们是否可以在最优排序下以最小可能成本选择 k 个目标。 对于固定的 k，我们需要选择 k 个项目，然后找到最小化 ai + (position - 1) * bi 之和的最佳排列。 这已经很重要了，因为最佳排序取决于所选子集。 

如果我们修复一个子集，最好的排序是按 bi 降序排列，因为疲劳度较高的项目应该更早放置以减少其乘数。 然后固定子集的成本可以在 O(k log k) 中计算。 然而，枚举大小为 k 的所有子集是组合性的并且是不可能的。 

观察的关键是扭转观点。 我们不是固定 k 并搜索子集，而是尝试逐步构建解决方案：假设我们想要选择 k 个项目。 如果我们按 bi 降序对所有项目进行排序，那么当选择 k 个项目时，我们可以有效地决定哪些项目将出现在序列中较早的位置，其中疲劳损失较小。 

我们可以根据 b 顺序重新构建所选集合的成本。 如果我们按降序处理项目，那么当我们选择一个项目时，它的位置取决于我们已经选择的项目数量。 因此，bi 的贡献与其选择时间相关。

这表明了一个贪婪的选择过程，我们尝试以最小的成本维护大小为 k 的候选集。 该结构变得类似于使用加权增量惩罚来选择 k 个项目，这自然会导致按 bi 排序并维护一个跟踪选择了多少个项目的结构。 

更直接、更标准的转变是从边际成本的角度来思考。 如果我们固定顺序，则在最后添加新项目会增加成本 ai 加上当前大小 k 的 k * bi 。 这意味着每个商品的成本与我们已经挑选的商品数量成线性关系。 为了使贪婪决策有效，我们需要一个结构，该结构始终选择在增加 k 的情况下仍然有利的项目。 

正确的方法简化为按 bi 降序排序，并使用选择策略来确保我们始终将最好的 k 项保留在诱导成本结构下。 这可以通过使用对 k 的贪婪或二分搜索结合优先级结构来迭代和维持增加 k 的可行性来实现。 

我们对答案 k 进行二分查找。 对于固定的 k，我们检查是否可以选择总成本最小 ≤ B 的 k 个项目。为了评估可行性，我们按 bi 降序对项目进行排序，并在分配位置 1 到 k 时使用堆来选择有效增量成本最小的 k 个项目。 这会产生该 k 的最佳排列。 

可行性检查之所以有效，是因为一旦 bi 确定了顺序，剩下的唯一选择就是选择哪 k 个项目，并且将较早的位置分配给较高的 bi 可以最大限度地减少总疲劳贡献。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力子集和排列 | O(2^N · N log N) | O(2^N · N log N) | O(N) | 太慢了 |
 | 二分查找+堆的贪婪可行性| O(N log N log N) | O(N log N log N) | O(N) | 已接受 |

 ## 算法演练

 ## 算法演练

 1.将所有目标按bi降序排序。 这确保了如果在不同位置使用两个项目，则较早考虑疲劳系数较大的项目，这与最小化线性惩罚项相一致。 
2.二分搜索最大k，使得可以在预算B内选择k个项目。单调性成立，因为如果k个项目可行，则任何较小数量的项目也是可行的。 
3. 对于二分搜索期间的固定 k，计算我们是否可以最优地选择 k 个项目。 我们模拟按顺序分配位置 1 到 k，并从排序列表中维护 k 个项目的最佳可能选择。 
4. 使用最大堆根据所选项目的增量成本贡献来跟踪它们。 当按排序顺序考虑每个项目 i 时，我们计算其放置在当前位置 j 的贡献，即 ai + j * bi。 我们将其添加到堆中，如果堆大小超过 k，我们将删除最差的项。 
5. 累计堆中当前项目的总成本。 处理完所有项后，如果堆总和≤B，则k是可行的。 
6. 在二分搜索找到最大可行 k 后，我们重新运行相同的选择过程来计算相应的最小成本。 

为什么它有效：

 关键的不变量是，当按降序处理项目时，首先将它们分配到较早的位置不会损害最优性。 任何最佳解决方案都可以转换为遵守该顺序的解决方案，而不会增加成本，因为较早交换更高的 bi 项会减少或保留总疲劳贡献。 堆确保我们始终保留在强加的位置结构下调整成本最小的 k 个项目。 这保证了如果预算内存在大小为 k 的任何子集，算法将找到一个可行的子集。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def feasible(items, k, B):
    if k == 0:
        return True, 0

    # items are (a, b) already sorted by b desc
    import heapq

    heap = []  # max heap via negative values
    total = 0

    for i, (a, b) in enumerate(items[:], start=1):
        # cost if placed at position i in selection order
        cost = a + (i - 1) * b

        heapq.heappush(heap, -(cost))
        total += cost

        if len(heap) > k:
            removed = -heapq.heappop(heap)
            total -= removed

    if len(heap) < k:
        return False, 0

    return total <= B, total

def solve():
    N, B = map(int, input().split())
    a = list(map(int, input().split()))
    b = list(map(int, input().split()))

    items = list(zip(a, b))
    items.sort(key=lambda x: -x[1])

    lo, hi = 0, N
    best_cost = 0

    while lo <= hi:
        mid = (lo + hi) // 2
        ok, cost = feasible(items, mid, B)

        if ok:
            best_cost = cost
            lo = mid + 1
        else:
            hi = mid - 1

    # recompute final cost for best k
    ok, cost = feasible(items, hi, B)
    print(hi, cost)

if __name__ == "__main__":
    solve()
```该解决方案首先按疲劳系数排序，以便随后关于位置相关成本的推理变得稳定。 可行性函数模拟使用保持最小调整成本的堆在此排序下构建大小 k 的最佳可能集合。 二分搜索利用关于 k 的可行性的单调性质。 一个微妙的点是，我们在固定 k 后重新计算最终成本，因为中间二分搜索状态可能会存储非最终 k 值的成本。

 ## 边缘情况

 对于零预算情况，算法在 k = 0 可行的情况下开始二分搜索，并快速拒绝任何 k ≥ 1，因为堆和立即变为正数。 除非未选择任何项目，否则可行性函数会正确返回 false。 

对于单个项目输入，排序是微不足道的，并且堆只包含一个元素。 当且仅当成本 a1 ≤ B 时，二分搜索将 k = 1 识别为可行，与直接解释相匹配。 

对于全零成本，每次可行性检查都对所有 k 返回 true，因此二分搜索正确收敛到 N，并且堆总和始终保持为零，从而确认当 bi = 0 时疲劳不会产生影响。
