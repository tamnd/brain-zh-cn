 ## 解决方案
 ## 问题理解

 我们得到了一组包，每个包都有两个参数。 第一个是基本成本，如果您将该包裹包含在交货批次中，则需要支付一次。 第二个是拥塞系数，它与您决定在同一批次中发送的包裹数量成比例。 

如果您选择在一次运行中恰好交付 K 个包裹，则每个选定的包裹 i 总共花费$a_i + K \cdot b_i$。 关键点是拥塞项取决于所选包的全局数量，而不是单个包。 

您的任务是选择包的子集，并隐式选择其大小 K，以便所选子集的总成本不超过预算 B，同时最大化 K。在达到此最大 K 的所有子集中，您必须报告可能的最小成本。 

选择和成本之间的相互作用是核心难点：选择更多的包会增加所有选定项目的每个包的拥塞惩罚，这会反馈到选择本身的可行性。 

约束允许最多$2 \cdot 10^5$套餐，费用高达$10^9$总预算最多可达$10^5$对于个别参数。 这立即排除了任何显式重新计算每个子集可行性的解决方案。 任何比粗略地更糟糕的方法$O(n \log n)$或者$O(n \sqrt n)$依赖于常量是不安全的。 

一种天真的但有启发性的失败模式是尝试修复 K 并贪婪地选择修改成本下最便宜的 K 个包。 例如，假设所有$b_i$对于某些物品来说较大，而对于其他物品则较小。 如果 K 固定不正确，贪婪选择可能会选择仅在 K 更改后才变得昂贵的项目，从而使之前的决策无效。 

当 K 更改项目的顺序时，会出现另一个微妙的边缘情况。 两个包可以根据 K 交换顺序，因为成本为$a_i + K b_i$。 例如，如果$a_1 < a_2$但$b_1 > b_2$，那么对于小 K 封装 1 更便宜，但对于大 K 封装 2 变得更便宜。 任何排序一次的方法$a_i$或者$a_i + b_i$不考虑K会破坏。 

## 方法

 蛮力的想法是尝试从 1 到 N 中的每一个可能的 K。对于每个 K，我们根据成本公式计算选择最便宜的 K 个包的成本$a_i + K b_i$。 为了直接做到这一点，我们需要评估所有包，计算调整后的成本，对它们进行排序，并选择最好的 K。 

对于固定的 K，排序需要$O(n \log n)$，对所有 K 执行此操作会导致$O(n^2 \log n)$，这对于$2 \cdot 10^5$。 

关键的结构观察是，我们实际上不需要在完整的重新计算意义上独立尝试所有 K。 相反，我们可以问一个决策问题：对于固定的K，我们可以在预算范围内选择K个套餐吗？最低成本是多少？ 如果我们能有效地回答这个问题，我们就可以二分查找 K。 

对于固定的 K，每个包贡献$a_i + K b_i$。 由于 K 是固定的，这只是每件物品的静态重量。 所以最优选择就是 K 个最小值$a_i + K b_i$。 这意味着我们只需要对每个检查排序一次，但即使这样也太慢了。 

我们通过二分搜索 K 并使用基于堆或基于选择的可行性检查来进一步减少这种情况。 然而，还有一个更简单的技巧：对于固定的 K，我们计算值$a_i + K b_i$，使用排序或堆选择 K 最小的$O(n)$或者$O(n \log n)$，并计算总和。 然后我们二分查找K中$O(\log n)$。 

这给出了$O(n \log n \log n)$，这是可以接受的。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 通过完全重新计算对 K 进行暴力破解 |$O(n^2 \log n)$|$O(n)$| 太慢了|
 | 二分查找 K + 选择 K 个最小调整成本 |$O(n \log n \log n)$|$O(n)$| 已接受 |

 ## 算法演练

 1. 定义一个函数，对于给定的 K，计算恰好选择 K 个包的最小总成本。 对于每个包，计算其有效成本$c_i = a_i + K b_i$，因为 K 在该检查中是固定的。 
2. 对所有计算结果进行排序$c_i$值并取最小 K 值的总和。 这对应于最佳选择，因为对于固定的 K，每个包独立贡献，并且除了选择计数之外没有耦合。 
3. 如果该总和小于或等于B，则K 个套餐在预算内是可行的。 否则，K太大。 
4.二分搜索K从0到N。我们维持可行性从true转变为false的范围。 
5. 对于每个中点 K，运行可行性函数。 如果可行，则将下限上移； 否则将上限向下移动。 
6. 二分搜索完成后，我们知道最大可行 K。我们使用相同的过程重新计算其成本，以获得该 K 的最小总成本。 

为什么有效：对于每个固定的 K，成本函数是完全确定的，并且除了通过选择计数之外，跨包独立。 可行性谓词在 K 中是单调的，因为增加 K 会增加每个$c_i = a_i + K b_i$，这只能增加 K 个最小值的总和或使选择变得更加困难。 这种单调性保证了二分查找的正确性，因为一旦 K 变得不可行，所有更大的 K 仍然不可行。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def cost_for_k(a, b, K, B):
    n = len(a)
    vals = [a[i] + K * b[i] for i in range(n)]
    vals.sort()
    s = sum(vals[:K])
    return s

def feasible(a, b, K, B):
    if K == 0:
        return True, 0
    n = len(a)
    vals = [a[i] + K * b[i] for i in range(n)]
    vals.sort()
    s = sum(vals[:K])
    return s <= B, s

def main():
    n, B = map(int, input().split())
    a = list(map(int, input().split()))
    b = list(map(int, input().split()))

    lo, hi = 0, n
    best_k = 0
    best_cost = 0

    while lo <= hi:
        mid = (lo + hi) // 2
        vals = [a[i] + mid * b[i] for i in range(n)]
        vals.sort()
        s = sum(vals[:mid]) if mid > 0 else 0

        if s <= B:
            best_k = mid
            best_cost = s
            lo = mid + 1
        else:
            hi = mid - 1

    print(best_k, best_cost)

if __name__ == "__main__":
    main()
```该代码对答案 K 实施直接二分搜索。对于每个中点，它计算调整后的成本$a_i + K b_i$并通过排序选择K个最小的。 这些 K 值的总和是该 K 的最小可能成本，因为任何其他子集都会用更大的元素替换所选元素。 

二分查找保留了迄今为止看到的最佳可行 K。 当中点可行时，我们存储 K 及其成本，然后尝试增加 K。否则，我们减少 K。这是可行的，因为 K 中的可行性是单调的。 

一个微妙的实现细节是单独处理 K = 0，以避免切片问题并确保预算允许零包时的正确性。 另一个重要的点是在每次迭代中重新计算成本，而不是尝试重用以前的计算，因为成本函数取决于 K。

 ## 边缘情况

 关键的边缘情况是 K 等于 0 时。 在这种情况下，不选择任何包，并且成本恰好为零。 该算法在排序之前显式处理此问题，避免不必要的工作并确保即使预算为零时也是正确的。 

当所有的情况出现时，就会出现另一种边缘情况$b_i$为零。 那么每个项目的成本就与 K 无关，问题就简化为选择 K 最小的$a_i$价值观。 该算法仍然有效，因为$a_i + K b_i = a_i$。 

第三种情况是当$a_i$值为零但是$b_i$是积极的。 尽管单个项目最初看起来是免费的，但增加 K 很快就会使选择变得昂贵。 二分搜索自然会发现拥塞占主导地位的截止点 K，并且每一步的重新计算都可以正确捕获这种非线性行为。
