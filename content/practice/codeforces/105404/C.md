 ## 解决方案
 ## 问题理解

 我们得到一条线上的线段，由两个固定点标记。 该段内有多个对象，每个对象都放置在不同的坐标处。 每个对象最初要么面向左第一个标记，要么面向右第二个标记。 一旦该过程开始，所有物体都会以相同的速度同时沿其选定的方向移动。 当两个物体正面相遇时，两者都会立即消失。 如果一个对象到达任一边界而没有遇到任何东西，它就会安全地离开该段。 

该过程持续进行，直到段内不再有任何对象为止。 任务是确定有多少对象在整个交互过程中幸存下来，这意味着它们最终到达两个端点之一而不会在碰撞中被破坏。 

这些约束允许每个测试用例最多 10,000 个对象和多个测试用例。 这立即排除了任何连续跟踪运动或按事件顺序解决碰撞的模拟。 简单的成对模拟在每种情况下都是二次的，在最坏的情况下会导致大约 10^8 次操作，这在时间紧迫的情况下在 Python 下是边缘或太慢的。 

当沿同一方向移动的物体相邻时，就会出现一个关键的微妙情况。 例如，如果两个对象都向右移动，它们永远不会相互作用，但仅考虑最近碰撞的简单模拟可能会错误地尝试将它们配对。 另一个微妙的情况是交替方向，其中每个物体都可能与附近的相反移动者取消，但前提是正确的配对逻辑。 

## 方法

 蛮力的想法是模拟运动或重复查找下一个碰撞事件。 在每一步中，我们都会识别所有相互移动的相邻物体对，计算它们之间最早的碰撞，删除两者，然后继续。 这在概念上是正确的，因为碰撞仅发生在相邻的相对移动者之间。 然而，在最坏的情况下，保持邻接并重复扫描冲突会导致 O(n^2) 行为，因为每次删除都会触发更新和重新扫描。 

关键的观察是绝对位置是无关紧要的； 只有相对顺序才重要。 当我们按位置对对象进行排序时，系统变成一个一维序列，其中每个对象要么向左移动，要么向右移动。 碰撞只能发生在向右移动的物体和其右侧的向左移动的物体之间，并且更具体地，碰撞总是以类似堆栈的配对方式发生。 

如果我们从左向右扫描，每个向右移动的物体都可以被认为是“等待”与未来向左移动的物体发生潜在的碰撞。 向左移动的物体要么遇到最近的、不匹配的向右移动的物体，要么在不存在的情况下逃脱。 这正是堆栈取消过程的结构。 

因此，我们不是模拟时间，而是按排序顺序处理对象并维护一堆向右移动的对象。 当我们看到一个向左移动的物体时，它会与一个向右移动的物体（如果有的话）取消； 否则，它会生存下来。 向右移动的对象只是被推入堆栈。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力模拟| O(n²) | O(n) | 太慢了|
 | 堆栈扫描| O(n log n) | O(n log n) | O(n) | 已接受 |

 对数因子仅来自排序位置。 

## 算法演练

 我们首先将每个对象与其位置和方向配对，然后按位置对它们进行排序，以便我们按空间顺序处理它们。

1. 按坐标对所有对象进行排序。 这确保了我们尊重从左到右的物理交互顺序，因为碰撞仅取决于相对位置。 
2. 维护一个空堆栈，用于存储当前向右移动并等待潜在碰撞的对象。 
3. 从左到右迭代对象。 对于每个对象，我们检查它的方向。 
4. 如果对象向右移动，我们将其压入堆栈，因为它稍后可能会与向左移动的对象发生碰撞。 
5. 如果对象向左移动，我们尝试解决冲突：当堆栈中有一个向右移动的对象时，我们将其删除，并删除当前向左移动的对象，因为它们会互相消灭。 从逻辑上讲，碰撞立即发生，因此双方都没有进一步参与。 
6. 如果在当前向左移动的对象被移除之前堆栈变空，则该向左移动的对象将存活下来并有效地移出段并到达左边界。 
7. 处理完所有对象后，堆栈中任何剩余的右移对象都将存活，并且任何从未被取消的左移幸存者也将被计数。 

最终的答案是幸存物体的数量。 

### 为什么它有效

 考虑任何向右移动的物体。 它只能与右侧向左移动的物体发生碰撞。 其中，它在时间上遇到的第一个物体正是在空间顺序上距离其右侧最近的向左移动的物体。 堆栈确保我们始终将最接近的未解析的右移动对象与下一个左移动对象相匹配。 因为两者以相同的速度移动，所以不可能进行中间重新排列：顺序被保留，并且配对是严格贪婪但安全的。 每次取消都对应一个唯一的碰撞事件，并且不会跳过或重复计算任何有效的碰撞。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

t = int(input())
for _ in range(t):
    n, m1, m2 = map(int, input().split())
    pos = list(map(int, input().split()))
    dir_ = list(map(int, input().split()))

    arr = list(zip(pos, dir_))
    arr.sort()

    stack_right = 0
    survivors_left = 0

    for _, d in arr:
        if d == 2:
            stack_right += 1
        else:
            if stack_right > 0:
                stack_right -= 1
            else:
                survivors_left += 1

    print(stack_right + survivors_left)
```排序步骤至关重要，因为原始输入顺序并不反映空间交互。 排序后，我们将动态压缩为一次传递。 变量`stack_right`表示未解决的向右移动对象，尚未遇到向左移动的对应对象。 变量`survivors_left`跟踪在到达边界之前从未遇到过向右移动对象的向左移动对象。 

一个常见的错误是假设对称并尝试独立模拟两个方向，但正确的交互严格取决于从左到右的处理顺序。 另一个微妙的点是，我们永远不需要随着时间的推移模拟实际位置； 方向和顺序完全决定结果。

 ## 边缘情况

 最小的单对象情况证实该算法不会意外地需要配对才能生存。 对于一个对象，堆栈保持为空，直接算作幸存者。 

所有物体朝同一方向移动的情况证明没有引入人为碰撞。 由于我们从不比较同向对象，因此堆栈只是增长或保持未使用状态，从而产生正确的完全生存结果。 

完全交替的配置强调取消逻辑。 每个向右移动的对象都会立即按顺序与下一个向左移动的对象配对。 堆栈正确地强制一对一消灭，并且当对完全平衡时最终计数变为零。 

包含所有向左移动的对象的情况可确保堆栈始终保持为空。 每个对象都被归类为幸存者，确认该算法不会错误地需要匹配的右移动对应对象才能生存。
