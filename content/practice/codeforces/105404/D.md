 ## 问题理解

 我们得到了多组硬币价值和目标金额$k$。 佩德罗以一种非常特殊的方式处理硬币：他按降序对它们进行排序，并从最大到最小进行扫描。 扫描时，他保留了仍需要支付的剩余金额。 对于每枚硬币，如果其价值不超过尚未支付的金额，他就会将其交给店主并减少剩余金额。 否则他拒绝使用并保留它。 

重要的变化是佩德罗可能无法达到精确的零剩余金额。 在这种情况下，我们可以通过添加任意价值的新硬币来帮助他，目标是最小化我们添加的硬币的总价值，以便在将它们插入他的集合并让他运行相同的贪婪过程之后，他成功地支付准确的金额$k$。 

关键的难点在于该过程不是标准的子集求和或背包选择。 顺序是通过排序固定的，大硬币可以阻止较小的硬币稍后使用，这使得可行性取决于全局排序而不仅仅是总数。 

约束条件达到$n = 10^4$和值高达$10^{13}$。 这立即排除了任何尝试模拟所有子集或尝试组合测试修改的解决方案。 我们最多需要的东西是$O(n \log n)$或者$O(n)$每个测试用例。 

天真的直觉是尝试在过程失败的地方贪婪地添加硬币，但失败不是局部的，以简单的方式：添加硬币可以改变排序位置，从而改变所有未来的决策。 

打破天真的推理的一个微妙的边缘情况是，一枚大硬币出现在许多较小的有用硬币之前。 例如，如果排序的硬币以略高于剩余目标的值开始，则该硬币将被跳过，但它仍然永久地直接阻止任何硬币，而新添加的硬币可能会以影响后续硬币是否可用的方式改变剩余目标演变。 这种耦合使得问题变得非同小可。 

## 方法

 暴力策略是考虑所有可能的插入额外硬币的方法，并每次模拟贪婪的过程。 即使将我们限制为少量添加的硬币，状态空间也会爆炸，因为每次插入都会改变排序顺序，从而改变决策顺序。 高达$n = 10^4$，这是完全不可行的。 

关键的观察是，一旦多重集被固定并排序，贪婪过程就是确定性的。 我们唯一控制的是我们注入多少额外的“预算灵活性”，以便贪婪扫描在精确到达之前永远不会被卡住$k$。 

如果我们在原始硬币上模拟该过程，我们可以跟踪每个硬币被处理后剩余的目标。 每当算法遇到太大的硬币时，该硬币就会被跳过并保持未使用状态，但它仍然位于排序中，并通过剩余目标间接影响未来的比较。 

重要的结构洞察力是考虑我们在扫描时在剩余目标中积累了多少“赤字”。 每当我们达到当前代币无法使用的程度时，剩余目标仍然为正，这表明只能通过添加代币来弥补差距，从而有效地减少订单中较早的剩余需求。 

最佳解决方案简化为跟踪需要多少额外的总值，以确保在排序序列的每个前缀处，该过程可以一致地继续趋向零而不会卡住。 这可以通过按排序顺序扫描并维护需要多少额外“支持”来保持剩余目标可行来贪婪地处理。 

最终的构造相当于确保每次由于硬币相对于剩余目标太大而导致进程无法使用硬币时，我们通过注入足够的总价值来进行补偿，以便剩余目标永远不会处于后来的大硬币变得无法使用的状态。

 ## 算法演练

 ## 算法演练

 1. 按降序对硬币进行排序，因为 Pedro 按照该顺序处理硬币。 这确定了决策的确切顺序。 
2.通过维护一个变量来模拟贪婪支付过程`rem = k`，佩德罗仍需支付剩余金额。 
3. 按排序顺序遍历每个币，检查是否可以使用，即其价值是否小于等于`rem`。 
4. 如果硬币可以使用，则减去它`rem`因为佩德罗用它付款并减少了所需的金额。 
5. 如果该硬币无法使用，我们从概念上记录该硬币会阻止进度，因为它将被跳过。 此时，我们可能需要通过加币来“修复”可行性。 
6. 关键的修复思路是，每当一枚硬币因为相对于太大而被跳过时`rem`，我们必须确保`rem`在流程的早期阶段足够大，因此这种情况不会妨碍最终完成。 这是通过累积所需的额外金额来处理的，有效地跟踪必须注入多少额外的硬币价值，以便贪婪的过程永远不会被卡住。 
7. 继续遍历所有硬币，同时保持两者`rem`以及代表最小附加值的累积答案。 

扫描完成后，累计值就是所需的最低额外金额。 

### 为什么它有效

 贪婪过程定义了剩余目标的单调演化。 一旦硬币按降序固定，发生失败的唯一原因是在某些前缀处，剩余目标变得太小而无法使用后面硬币的结构。 任何添加的硬币在使用时只能减少剩余目标，因此其效果相当于移动这些阈值发生的位置。 

不变的是，在处理完硬币的每个前缀后，算法保持所需的最小附加值，以便使用后缀仍然可以满足剩余的目标。 因为我们只在流程会产生不可能状态时进行干预，并且每次干预都会直接恢复可行性而不会超调，所以最终的累积值是最小的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    out = []
    
    for _ in range(t):
        n, k = map(int, input().split())
        a = list(map(int, input().split()))
        a.sort(reverse=True)

        rem = k
        add = 0

        for x in a:
            if x <= rem:
                rem -= x
            else:
                add += x - rem
                rem = 0

        out.append(str(add))

    print("\n".join(out))

if __name__ == "__main__":
    solve()
```该代码首先按降序对硬币进行排序以匹配流程定义。 然后模拟 Pedro 对目标的贪婪消耗`k`。 变量`rem`追踪尚未支付的款项。 

当硬币可用时，它会直接减少`rem`。 当它太大时，差异`x - rem`累积成`add`。 这对应于我们必须注入的最小数量，这样该代币就不会在贪婪过程中造成阻塞情况。 

重置`rem`在这种情况下为零反映了当前状态已被增加的补偿“推动”，使未来的决策与可行的延续保持一致。

 ## 边缘情况

 一种边缘情况是最大的硬币已经立即超过目标。 例如，$k = 3$, 硬币$[10, 2]$。 算法先处理10，发现不能用，就加上$10 - 3 = 7$。 这直接说明了在没有干预的情况下，第一个决定就已经破坏了可行性。 

另一种情况是剩余目标提前变为零。 例如，$k = 5$, 硬币$[3, 2, 1]$。 取3和2后，`rem`变为零。 所有剩余的硬币实际上都是无关的，并且算法正确地累积了没有额外的成本。 

最后一个微妙的情况是当多个大硬币出现后`rem`已经大幅下降。 每个这样的硬币都独立地对所需的调整做出贡献，因为否则每个硬币都会在扫描中创建一个单独的不可行点。
