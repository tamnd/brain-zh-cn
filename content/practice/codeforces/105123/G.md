 ## 解决方案
 ## 问题理解

 我们从由四个可能的字符组成的单个环形 RNA 字符串开始。 随着时间的推移，这种结构被两种以非常特定的方式相互作用的操作所改变。 CUT 操作会破坏每次出现的定向核苷酸对之间的结构，分裂链并可能将圆形片段变成线性片段。 SPLICE 操作执行相反类型的相互作用：每当一条线性链以核苷酸 x 结尾而另一条线性链以 y 开头时，这些链就可以粘合在一起，并且这个过程会贪婪地重复，直到不可能进行此类合并为止。 

每次操作后，我们不会被要求重建股线本身。 相反，我们需要在应用操作的所有强制效果后可以存在的最大可能数量的线性链。 圆形线与最终计数无关，只是它们可以通过切割变成线性。 

关键的困难在于这两项行动都是在全球范围内重复进行的。 CUT 可以同时分割多条链上的许多位置，而 SPLICE 可以重复链式合并直到稳定。 由于操作是累积的，因此每个查询都建立在所有先前的转换之上。 

这些约束允许对长度不超过 200,000 的字符串进行最多 200,000 次操作。 任何显式模拟链或重复扫描每个查询的完整结构的方法都会太慢。 即使每个查询的 O(n) 在最坏的情况下也会导致 4e10 次操作，这是不可行的。 该解决方案必须将每个操作减少到更接近常数或对数时间。 

一个微妙的问题是，天真的直觉往往会追踪实际的细分。 这是失败的，因为 CUT 和 SPLICE 都只依赖于核苷酸类型之间的邻接关系，而不依赖于确切的位置或链身份。 该系统从根本上讲是关于四个节点上的有向交互图，而不是可变的字符串列表。 

当操作相互抵消或重复 SPLICE 将所有内容折叠成单个链时，就会出现边缘情况。 另一个重要的极端情况是，当 CUT 引入新的边界时，会立即启用级联的多个 SPLICE 操作，简单的实现会尝试显式模拟，从而变成二次的。 

## 方法

 暴力方法将明确地将所有链维护为列表中的字符串。 CUT 将迭代所有链，扫描每个字符串，并在出现禁止的 x 到 y 转换时进行分割。 SPLICE 会重复扫描所有的链对，每当结束-开始条件匹配时就合并，并重复直到不再可能合并为止。 这种方法在概念上是正确的，因为它完全遵循规则，但它变得极其昂贵。 每个操作仅扫描一次就需要花费 O(n) 的时间，并且 SPLICE 可能需要多次传递，使得最坏情况的复杂度是二次的，或者比所有查询更糟糕。 

关键的观察结果是，该过程并不取决于单个链的身份，而仅取决于字符之间的哪些定向转换是“活跃的”。 每个 CUT 或 SPLICE 仅影响是否允许或抑制核苷酸对之间的转换。 由于只有 4 个核苷酸，因此只有 16 个可能的定向对。 这减少了动态系统在四个节点上的有向多重图中维持活动边的小状态。 

主动转变的每种配置都会导致链如何合并或分裂的固定结构。 一旦预处理为转换计数，所得线性链的数量仅取决于此配置，而不取决于底层序列本身。 这使我们能够维护每个相邻对在原始循环字符串中出现的次数，然后更新每次操作后有多少“断点”保持有效。

因此，我们不是模拟股线，而是跟踪在当前 CUT 规则下有多少相邻对幸存下来，以及 SPLICE 如何通过以受约束的方式有效地反转这些剪切来合并兼容端点。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(nq) 到 O(n²q) | O(n) | 太慢了 |
 | 最佳 | O(n + q) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们将环状 RNA 解释为连续字符之间的有向边序列，包括最后一个到第一个字符。 每个边缘根据其类型有助于未来潜在的切割或拼接。 

我们维护当前存在的线性链数量的全局计数器，并且维护核苷酸之间转换的计数。 

1. 将初始循环字符串转换为频数表cnt[a][b]，统计字符a后面跟着b绕一圈的次数。 这以压缩形式捕获所有可能的切割点。 
2. 维护一个变量totalCuts，它表示当前有多少个转换是“活动分隔符”，这意味着它们会阻止合并到单个线性结构中。 最初，在完全圆形的股中，结构表现为一个组件，因此这从零有效线性段开始。 
3. 对于 CUT(x, y)，我们切换或激活从 x 到 y 的转换作为分隔符。 如果 cnt[x][y] > 0，则这些出现的位置将成为分割点。 每次这样的激活都会使线性段的数量增加 cnt[x][y]，因为每次出现该邻接都会破坏一个循环或链。 
4. 对于 SPLICE(x, y)，我们尝试删除先前强制执行的 x 和 y 转换之间的分隔。 每次删除都会使线性段的数量减少 cnt[x][y]，因为先前分割的边界再次变得可合并。 
5. 每次操作后，答案是剩余活动切割所隐含的当前线性分量数。 由于合并是在稳定之前强制进行的，因此系统总是会崩溃到由活动转换确定的唯一稳定分区。 
6. 处理每个查询后输出当前线性股的数量。 

关键的计算技巧是每个操作仅影响 16 个有向对之一，因此更新时间复杂度为 O(1)。 

### 为什么它有效

 每个链边界都是由原始循环序列中的定向邻接引起的。 CUT 操作将所有出现的选定邻接激活为强制断点，而 SPLICE 则删除该约束。 由于 SPLICE 始终完全传播合并，直到没有有效端点剩余为止，因此在给定当前活动约束集的情况下，系统始终处于最大合并状态。 这使得线性链的数量成为当前活动的邻接对的确定性函数，与中间链的顺序或结构无关。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def idx(c):
    if c == 'A': return 0
    if c == 'C': return 1
    if c == 'G': return 2
    return 3  # U

def solve():
    n, q = map(int, input().split())
    s = input().strip()

    cnt = [[0] * 4 for _ in range(4)]

    for i in range(n):
        a = idx(s[i])
        b = idx(s[(i + 1) % n])
        cnt[a][b] += 1

    active = [[0] * 4 for _ in range(4)]
    answer = 1

    for _ in range(q):
        parts = input().split()
        t, x, y = parts[0], parts[1], parts[2]
        a, b = idx(x), idx(y)

        if t == "CUT":
            active[a][b] = 1
        else:
            active[a][b] = 0

        # recompute number of linear strands induced by active cuts
        # each active edge contributes cnt[a][b] additional breaks
        answer = 1
        for i in range(4):
            for j in range(4):
                if active[i][j]:
                    answer += cnt[i][j]

        print(answer)

if __name__ == "__main__":
    solve()
```该解决方案将循环字符串压缩为 4 x 4 邻接矩阵，以便可以评估每个操作，而无需再次扫描原始字符串。 活动矩阵存储当前是否对给定定向核苷酸对应用 CUT 约束。 

每次查询后，我们从一个组件开始并添加所有活动切割边缘的贡献来重新计算线性链的数量。 每个活动边精确地贡献了原始圆形串中该邻接的出现次数，因为每个这样的出现都会创建一个新的分隔。 

SPLICE 操作只是停用该边沿，立即反转其效果。 

最微妙的部分是在构建 cnt 时将字符串视为圆形，这确保最后一个字符和第一个字符之间的边界被正确计算为可能的剪切位置。

 ## 边缘情况

 当所有核苷酸都相同时，就会出现微妙的边缘情况。 邻接矩阵只有一个非零条目，并且 CUT 和 SPLICE 重复切换相同的转换。 该算法干净地处理了这个问题，因为每个操作仅翻转有源矩阵中的单个单元，并且贡献要么完全包含或排除。 

另一种边缘情况是 CUT 和 SPLICE 在同一对上交替出现。 由于我们每次仅使用活动标志从头开始重新计算，因此不存在漂移或累积错误的风险。 该状态始终反映准确的当前配置。 

最后的边缘情况是最后一个字符和第一个字符之间的圆形边界。 这是在预处理过程中通过显式地将 s[n-1] 到 s[0] 计数为有效邻接来处理的，确保边界处的切割与内部切割的处理方式一致。
