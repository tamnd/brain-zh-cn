 ## 解决方案
 ## 问题理解

 我们得到了一系列任务，按优先级从第一个到最后排序，每个任务都消耗固定数量的 ATP 来完成。 Jasmine 的每日 ATP 预算为$m$，并且她不能重复任务。 不同的是，她并不总是考虑所有任务。 相反，对于每个前缀长度$k$，她只关注第一个$k$任务按优先顺序排列，并且在该限制集中，她希望在不超出 ATP 预算的情况下完成尽可能多的任务。 

对于每个$k$，我们实际上是在问：如果我们只接受任务$1$通过$k$，我们可以选择的最大任务成本是多少，其总和最多为$m$。 

重要的观察是，任务始终按前缀顺序执行，但在该前缀内，我们可以自由选择任何子集。 因此，这不是调度或排序问题，而是在不断增长的前缀上重复的“总和约束下的最佳子集”问题。 

约束条件很大，有$n$最多$2 \cdot 10^5$。 这立即排除了从头开始为每个重新计算最佳子集的可能性$k$。 一个天真的$O(n^2 \log n)$或者$O(n^2)$重新排序或重新扫描前缀的方法会太慢。 我们需要一个增量结构，其中每个步骤都以对数或恒定摊销时间更新先前的答案。 

当成本高度不平衡时，就会出现一个微妙的问题。 For example, if$m = 10$和任务是$[1, 1, 1, 100, 1]$，那么早期的前缀允许许多小的选择，但是一旦出现大的成本，它永远不会属于最佳子集，但它会改变前缀集。 任何假设我们总是在全球范围内选择最便宜的可用任务的不正确的贪婪方法都会失败，除非它在前缀增长下小心地保持可行性。 

## 方法

 考虑每个前缀的直接方法是对其元素进行排序并采用最小成本，直到总和超过$m$。 这是正确的，因为为了在总和约束下最大化计数，选择​​较小的成本始终是最佳的。 然而，为每个前缀重新计算意味着我们重复排序或重建结构最多$O(n)$元素，导致$O(n^2 \log n)$最坏情况下的行为。 

关键的结构观察是当从前缀移动时$k-1$到$k$，我们正好添加了一个新元素。 前缀的最优集合$k$与前缀不同$k-1$只是因为我们可能想要包含这个新元素，如果它有助于在相同预算下增加计数，则可能会驱逐更大的选定元素。 

这建议维护所选任务的动态多重集，始终代表当前前缀的最佳可能子集。 标准结构是最大堆：我们保留所有当前选择的任务并确保它们的总和保持在$m$。 添加新任务时，我们将其插入，如果总数超过$m$，我们删除最大的元素。 删除最大的元素是最佳选择，因为它可以最大程度地减少每次删除的总和并保留尽可能多的小元素。 

因此每个前缀都可以更新$O(\log n)$，并且我们维护当前总和以及所选任务的数量。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力破解（重新计算每个前缀）|$O(n^2 \log n)$|$O(n)$| 太慢了 |
 | 最佳（最大堆增量）|$O(n \log n)$|$O(n)$| 已接受 |

 ## 算法演练

 我们从左到右处理任务，维护始终符合 ATP 预算的选定任务的结构。 

1.初始化一个空的最大堆和一个变量`total = 0`。 堆存储选择的任务成本，以及`total`跟踪他们的总和。 
2. 对于每个任务$a_k$，将其插入堆中并将其成本添加到`total`。 这对应于暂时接受每个新任务，因为我们还不知道它是否会成为该前缀的最终最佳子集的一部分。 
3.如果`total > m`，从堆中删除最大的元素并减去它`total`。 此步骤恢复了可行性。 删除最大的元素是唯一的选择，永远不会过多地减少所选任务的数量，因为它牺牲了最昂贵的任务，同时保留了尽可能多的项目。 
4. 调整后，堆的大小正好是当前前缀可以完成的最大任务数。 输出这个尺寸。 
5. 对所有对象重复此过程$k$从 1 到$n$，为每个前缀生成一个答案。 

为什么这有效与贪婪的交换论点有关。 在任何前缀处，假设我们有一个大小不是最佳的可行子集。 如果它包含较大的元素而排除较小的元素，则交换它们不会使可行性恶化，也不会减少计数。 因此，最佳解决方案总是更喜欢较小的元素，并且堆不变量会在新元素到达时动态地强制执行该结构。 

## Python 解决方案```python
import sys
import heapq

input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    a = list(map(int, input().split()))
    
    heap = []
    total = 0
    out = []
    
    for x in a:
        heapq.heappush(heap, -x)
        total += x
        
        if total > m:
            largest = -heapq.heappop(heap)
            total -= largest
        
        out.append(str(len(heap)))
    
    print("\n".join(out))

if __name__ == "__main__":
    solve()
```核心实现细节是使用通过负值模拟的最大堆，因为Python的`heapq`只提供最小堆。 我们维持`total`明确地是因为重新计算堆总和会太慢。 

一个微妙的点是，当发生溢出时，我们总是删除一个最大的元素。 即使`total`远远高于`m`，每次插入一次删除就足够了，因为删除最大的元素后，剩余的元素都较小或相等，并且堆不变量确保我们最多仅超出新添加元素的贡献的容量。

 ## 边缘情况

 关键的边缘情况是第一个元素已经超出预算。 用于输入`n = 3, m = 5, a = [10, 1, 1]`，堆首先变成`[10]`，总数超出了预算，我们立即将其删除，留下一个空堆。 答案为$k=1$因此为 0，后续的小元素从头开始重建解决方案。 该算法自然地处理这个问题，因为每次插入之后都会有一个校正步骤。 

另一个边缘情况是当许多小元素累积并且大元素出现较晚时。 为了`m = 6, a = [1, 1, 1, 1, 1, 10]`，堆稳定增长到大小 5，然后 10 强制移除自身，而不是任何小元素，从而保留最佳计数。 这显示了堆始终包含前缀的最小成本可行子集的不变量。
