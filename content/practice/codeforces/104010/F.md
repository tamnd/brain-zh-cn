 ## 解决方案
 ## 问题理解

 我们得到一系列按固定顺序排列的问题分数。 每个位置都有一个积极的价值，解决一个问题会产生很多积分。 所有问题的总和定义了一个门槛：参与者必须至少收集到总金额的一半才能获得文凭。 

Alexey 不会自由选择任意子集。 他选择一个起始索引，然后严格向右进行。 在继续前进时，他可以选择跳过整个所选片段中的至多一个问题。 该部分中所有其他访问过的问题均已解决，并收集其分数。 目标是尽量减少他实际解决的问题数量，同时仍达到总分的至少一半。 

关键约束是$n \le 10^5$，这排除了对所有起始位置和段选择的任何二次探索。 任何试图直接模拟每个开始和每个跳跃位置的解决方案都会表现得像$O(n^2)$，对于一秒的限制来说太慢了。 

一个微妙的一点是，最佳线段不一定是最长的或总密度最大的线段。 由于只允许跳过一次，因此可以牺牲高值元素来减少已解决问题的数量，因此最佳答案取决于单个删除元素对段的压缩程度。 

当最佳解决方案需要在大多数小值区域内跳过非常大的值，或者根本没有跳过时，就会出现边缘情况。 例如，如果所有值都相同，则跳过除了稍微缩短计数之外不会改变最优性，并且问题简化为找到达到一半总和的最短前缀。 

另一种边缘情况是单个非常大的元素在数组中占主导地位。 在这种情况下，从该位置开始并且不跳过任何内容是最佳的，因为任何更长的段只会增加已解决问题的数量，而不会提高效率。 

## 方法

 直接方法尝试每个起始索引$k$，然后在累积点的同时模拟向右延伸，可选择跳过线段内所有可能位置中的一个元素。 对于每次开始，我们都会跟踪达到总和一半所需的最小求解元素数。 这需要，对于每次开始，可能扫描到结束并测试跳过位置，导致粗略$O(n^2)$行为。 和$10^5$元素，这是不可行的。 

主要观察结果是，在段内最多跳过一个元素相当于说：在从$k$，我们可以从段和中删除一个元素，并且我们想要调整后的和达到阈值的最早点。 这表明我们希望最大化跳过一个元素的好处，同时最小化段长度。 

我们不是固定起点并扩展，而是反转视角：对于每个可能的右端点，我们想知道以最多一个删除达到阈值的最佳片段。 这自然会导致双指针或滑动窗口结构，因为当我们移动右边界时，总和和有效性条件都会单调演化。 

我们维护一个窗口$[l, r]$并跟踪其总和。 我们还跟踪窗口内的最大元素，因为如果允许我们跳过一个元素，则最佳选择始终是跳过当前窗口中的最大元素。 这将问题简化为检查是否$$\text{sum}(l, r) - \max(l, r) \ge \text{half}$$或者，如果不需要跳过，是否$$\text{sum}(l, r) \ge \text{half}.$$我们滑行$r$从左到右，维持一个数据结构为窗口内最大，并收缩$l$在满足条件的情况下尽可能多地进行。 最好的答案是任何有效窗口中元素的最小数量。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O(n^2)$|$O(1)$| 太慢了|
 | 最佳 |$O(n)$|$O(n)$或者$O(1)$| 已接受 |

 ## 算法演练

 我们首先计算所有值的总和，并将目标定义为其中的一半，这意味着我们至少需要$\lceil \frac{\text{sum}}{2} \rceil$。 

然后，我们在数组上使用滑动窗口，同时维护两条信息：窗口的当前总和以及允许我们查询窗口中最大元素的结构。 

1.我们初始化两个指针$l = 0$,$r = 0$，并将当前窗口总和设置为零。 我们还初始化一个双端队列来维护最大值的候选者。 该双端队列以值的递减顺序存储索引，因此前面总是给出最大值。 
2.我们展开右指针$r$，添加$a[r]$到窗口总和并通过删除所有小于的元素来更新双端队列$a[r]$从后面，然后推$r$。 这确保了在恒定时间内始终可以访问最大值。 
3. 每次展开后，我们检查当前窗口是否有效。 如果总和已达到目标，或者总和减去最大元素达到目标，则窗口有效。 第二个条件对应于在最佳可能元素上使用单个允许的跳过。 
4. 如果窗口有效，我们尝试从左侧缩小它。 搬家前$l$，我们将其从和中删除，如果匹配，则将其从双端队列前面弹出。 我们在有效性保持的情况下继续缩小，因为我们想要尽可能最小的窗口。 
5. 每次调整后，我们用当前窗口大小更新答案。 
6. 我们重复直到$r$到达数组末尾。 

该算法之所以有效，是因为在每一步中，我们都为每个右边界维护尽可能小的左边界，但仍允许通过最多一次删除来达到阈值。 

### 为什么它有效

 对于任何固定窗口，要跳过的最佳元素始终是其中的最大元素，因为删除任何较小的元素只会减少跳过的好处。 因此，跳过哪里的决定并不取决于窗口之外的未来元素。 

滑动窗口确保对于每个右端点，我们始终保持满足可行性的最小左端点。 如果对于相同的右端点可以有一个更小的窗口，那么它就会被发现，因为在有效性保持的情况下，收缩是贪婪地完成的。 这保证了隐式探索所有候选最佳片段，而无需显式枚举起始点。 

## Python 解决方案```python
import sys
input = sys.stdin.readline
from collections import deque

def solve():
    n = int(input())
    a = list(map(int, input().split()))
    
    total = sum(a)
    target = (total + 1) // 2
    
    dq = deque()
    l = 0
    curr_sum = 0
    ans = n
    
    for r in range(n):
        curr_sum += a[r]
        
        while dq and a[dq[-1]] <= a[r]:
            dq.pop()
        dq.append(r)
        
        while dq and dq[0] < l:
            dq.popleft()
        
        def valid():
            if curr_sum >= target:
                return True
            if dq:
                return curr_sum - a[dq[0]] >= target
            return False
        
        while l <= r and valid():
            ans = min(ans, r - l + 1)
            if dq and dq[0] == l:
                dq.popleft()
            curr_sum -= a[l]
            l += 1
            while dq and dq[0] < l:
                dq.popleft()
    
    print(ans)

if __name__ == "__main__":
    solve()
```该实现保留了一个索引双端队列，表示当前窗口中值的单调递减结构。 这允许恒定时间检索最大元素，这对于评估是否可以有效地应用可选跳过至关重要。 

为了清楚起见，有效性检查被显式地编写为函数，但实际上由于双端队列结构，它是恒定时间。 收缩阶段确保我们始终保持每个右边界的最小窗口。 

删除左指针时必须小心：如果传出元素当前是最大候选元素，则必须将其从双端队列中弹出。 未能将双端队列与窗口边界同步是最常见的错误来源。

 ## 边缘情况

 一种边缘情况是最佳解决方案是单个非常大的元素。 例如，在`[100, 1, 1, 1]`，在第一个位置立即满足阈值，并且算法正确识别大小为 1 的窗口，因为无需跳过逻辑即可触发全和条件。 

另一种情况是必须跳过的情况。 在`[1, 100, 1, 1]`，最佳窗口包括中间的大元素，但将其用作跳过的元素，从而允许其余的小元素形成最小长度解决方案。 双端队列确保该最大值始终被视为跳过候选。 

第三种情况是所有元素都相等。 为了`[2, 2, 2, 2]`，跳过不会以有意义的方式提高阈值的可达性，但收缩仍然会找到跨越总和一半的最短段，从而正确地有利于紧凑窗口而不是较长窗口。
