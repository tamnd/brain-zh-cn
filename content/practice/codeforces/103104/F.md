 ## 解决方案
 ## 问题理解

 我们得到了一组电池和一组拍摄地点。 每块电池都有固定数量的能量，每个位置都需要固定数量的能量来完成一次记录会话。 每个电池最多只能使用一次，一旦您开始在某个位置使用电池，它必须完全满足该位置的要求，而无需中断或更换。 

任务是选择电池和位置之间的配对策略，以使完全完成的位置数量最大化。 每个电池最多可以分配到一个位置，每个位置最多可以分配一个电池。 如果所选电池的容量至少等于其所需的持续时间，则位置满足。 

该结构立即暗示了一个类似二分匹配的问题，但对于任何通用的匹配算法来说，约束都太大了。 相反，这是两个多重集上的贪婪分配问题。 

这些约束意味着我们需要接近 O(N log N + M log M) 或 O(N + M log M) 的东西。 当 N 达到 100000，M 达到 300000 时，任何二次方甚至 N 乘以 M 都是不可能的。 尝试每个电池位置配对的解决方案显然是不可行的，因为在最坏的情况下它会达到 3 × 10^10 的比较。 

关键的结构细节是所有拍摄持续时间都是 2 的幂。 这是一个强烈的暗示，分组或基于桶的贪婪匹配将是有效的，因为我们可以按升序处理需求并有效地处理重复值。 

一些边缘情况很重要：

 一个天真的贪婪者在没有仔细排序的情况下分配可以满足某个位置的最小电池可能会失败。 例如，如果电池是 [5, 6]，需求是 [4, 5]，粗心的分配可能会给出 5 → 5 和 6 → 4，这很好，但如果我们以错误的顺序贪婪地反转决策，我们可能会在 5 上浪费 6，并根据实现顺序错误地留下 4 不匹配。 当存在多个相同的要求时，会出现另一个微妙的失败，因为贪婪的选择必须尊重全局最优性而不是局部拟合。 

## 方法

 暴力视图会尝试以所有可能的方式将电池分配到位置并计算最佳匹配。 这相当于尝试二分图中的所有匹配，如果 ai ≥ bi，则存在边。 即使我们将自己限制为贪婪分配，简单的模拟也会针对每个位置检查每个电池，产生 O(NM) 时间复杂度，这远远超出了限制。 

关键的观察是，我们不需要保留超出其所需能量的位置的身份。 由于需求只是大小，我们只关心满足多少，所以这个问题变成了一个经典的“容量≥需求时最大化匹配数”问题。 这通常是通过对两个数组进行排序并从最小到最大进行贪婪匹配来解决的。 

然而，不同之处在于 b_i 是 2 的幂。 这保证了数量非常有限的不同要求值（最多 31 个）。 我们可以将需求压缩到按指数索引的频率桶中，而不是将 M 视为很大。 然后我们根据从小到大的需求水平有效地匹配电池。 

我们按升序对电池进行排序，并按照从最小功率到最大功率的顺序处理需求。 对于每个需求尺寸，我们尝试分配仍然满足它的尽可能最小的电池。 这确保了除非必要，否则我们不会在小需求上浪费大电池。 

我们在电池上维护一个指针，并在满足需求时贪婪地消耗它们。 因为两边都已排序，并且我们只向前移动，所以每个元素都被处理一次。

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力匹配| O(NM) | O(1) | O(1) | 太慢了|
 | 排序+贪心匹配| O((N + M) log N) | O((N + M) log N) | O(1) 额外 | 已接受 |

 ## 算法演练

 1. 读取所有电池容量并按非递减顺序排序。 这使我们能够始终使用仍能满足要求的最小可用电池，并为以后更大的需求保留更大的电池。 
2. 计算两个值的每个幂存在多少个请求。 由于 bi = 2^j，我们将频率存储在数组 freq[j] 中。 这会将输入压缩为最多 31 个类别。 
3. 在排序后的电池数组的开头初始化一个指针i。 该指针代表最小的未使用电池。 
4. 从最小到最大迭代可能的 2 的幂，按所需容量的递增顺序处理 freq[j]。 这确保我们始终首先尝试完成更简单的任务。 
5. 对于每个需求级别 2^j，尝试贪婪地分配电池。 当 freq[j] > 0 且 i < N 时，前进 i 直到找到容量 ≥ 2^j 的电池。 如果不存在，则中断，因为对于该电池范围，无法在该级别或更高级别进行进一步分配。 
6. 当找到有效电池时，将其分配给一个要求，递增 i，递减 freq[j]，并递增答案计数器。 这会消耗电池和位置。 
7. 继续，直到处理完所有要求级别或电池电量耗尽。 

### 为什么它有效

 该算法保持贪婪不变量：在任何时刻，最小的剩余电池对于当前的需求水平来说要么太小，要么是该水平的最佳选择。 由于电池和需求都是按递增顺序处理的，因此我们将较大电池分配给较小需求的任何偏差只会降低未来较大需求的灵活性，而不会增加满足分配的数量。 单调结构确保一旦某个需求跳过某个电池，它就永远无法满足较小的需求，而一旦需求增加，早期的电池就无法再满足它们。 这可以防止任何可能提高总计数的有益交换。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    a = list(map(int, input().split()))
    b = list(map(int, input().split()))

    a.sort()

    freq = [0] * 31
    for x in b:
        freq[x.bit_length() - 1] += 1

    ans = 0
    i = 0

    for j in range(31):
        need = 1 << j
        while freq[j] > 0 and i < n:
            if a[i] >= need:
                ans += 1
                freq[j] -= 1
                i += 1
            else:
                i += 1

    print(ans)

if __name__ == "__main__":
    solve()
```解决方案首先对电池进行分类，以便我们始终可以按升序消耗它们。 频率数组将所有拍摄持续时间压缩到二次幂的桶中，这一点至关重要，因为它消除了匹配循环中对 M 的依赖。 

指针 i 永远不会向后重置。 这是安全的，因为一旦电池对于要求级别来说太小，它就永远不会用于任何更高的级别。 每个电池最多被考虑一次，确保对排序数组进行线性扫描。 

内循环要么消耗电池，要么永久丢弃电池，以保证进度。

 ## 边缘情况

 一种极端情况是所有电池都小于最小要求。 在这种情况下，指针 i 前进穿过所有电池，没有任何匹配，并且答案仍然为零。 该算法正确地跳过所有不可用的电池，因为条件 a[i] < need 只会触发前进。 

当存在许多相同的需求值时，会出现另一种边缘情况。 因为我们处理频率计数，所以每次匹配仅消耗一个需求单位，并且循环将继续，直到电池或需求耗尽。 不存在对相同要求内的排序的依赖，因此不会出现病态行为。 

最后一种边缘情况是一些非常大的电池与许多小电池并存。 这种排序确保在可能的情况下首先使用小型电池，但一旦需求增加，算法自然会转向较大的电池。 由于指针永远不会向后移动，因此会自动保留大电池以满足更大的需求。
