 ## 解决方案
 ## 问题理解

 我们得到一个整数序列，表示放置在一行中的元素。 允许的操作取决于位置：如果元素当前位于位置$k$（1-索引）完全具有价值$k$，那么该元素就有资格被删除。 

这个过程不是任意的。 每当存在多个可移动元素时，我们就被迫删除其中最右边的一个。 每次移除都会改变剩余元素的索引，因此可移除位置的集合会随着时间的推移而演变。 任务是确定是否可以通过重复应用此规则来完全删除所有元素。 如果可能的话，我们还必须输出元素被删除的顺序，但以与删除过程相反的方式报告。 

关键的微妙之处在于移除不是独立的。 过早删除某个元素可能会暴露或破坏其他元素的可移动性，因为位置会发生变化。 独立检查“a[i] = i 的任何索引 i”的幼稚解释将失败，因为索引是动态的，而不是静态的。 

约束足够小，以至于$O(n^2)$模拟是可以接受的。 这表明每个步骤都可以对阵列进行线性扫描，并且我们可以执行最多$n$移除，最多给予$O(n^2)$总操作。 任何依赖于重复全局重新计算的解决方案仍然是安全的。 

一些失败案例是由于粗心的做法而自然产生的。 一是假设我们总是可以删除最左边的有效元素。 例如，考虑$[1, 2, 3]$。 如果我们先删除 1，我们会移动数组并销毁根据“最右侧强制删除”规则后续删除所需的结构。 正确的过程依赖于顺序，贪婪的方向很重要。 

另一个陷阱是忘记每次删除后索引都会发生变化。 将值视为引用原始索引会导致在以下情况下错误删除：$[2, 1, 3]$，其中唯一有效的删除取决于动态重新索引。 

## 方法

 蛮力的想法是从字面上模拟这个过程。 在每一步中，我们扫描数组，找到其中的所有位置$a[i] = i$，选择最右边的位置，将其删除，然后继续。 每次扫描费用$O(n)$，我们最多可以做$n$搬迁、给予$O(n^2)$复杂。 这在一定范围内有效，但隐藏了重复扫描和移动的低效率。 

关键的观察是，该过程相当于重复删除当前“正确”的最右边位置。 一旦某个位置变得可移除，而我们在它是最右边的候选者时选择不移除它，那么以后就永远无法移除它。 这是因为删除右侧的任何内容不会影响其索引条件，但删除左侧的任何内容会改变它并永久打破相等性。 因此，推迟有效的最右侧移除是不可逆转的损害。 

这导致了一个干净的贪婪策略：重复定位最右边的索引$i$这样$a[i] = i+1$，立即将其删除，然后继续。 正确性取决于索引移位的单调性：只有左侧删除会影响位置的索引，因此最右侧的有效元素可以安全提交。

 ## 算法演练

 我们维护数组并重复删除元素，直到所有元素都被删除或不存在有效的删除。 

1. 从当前数组和一个空列表开始`ans`按删除顺序记录删除的元素。 目标是重建完整的有效删除序列。 
2.从右向左扫描数组，找到最大的索引$i$这样$a[i] = i+1$。 这确保我们遵守始终选择最右边的可移动元素的规则。 
3. 如果不存在这样的索引，则进程卡住。 我们无法继续，因为不允许进行任何操作，因此我们输出 NO 并终止。 
4. 如果找到这样的索引，则从数组中删除该元素并将其值附加到`ans`。 删除会移动索引后的所有元素$i$留下一个位置，更改其索引以供将来检查。 
5. 重复此过程，直到数组变空。 
6.最后反转`ans`并输出它。 需要反转是因为我们按照删除顺序记录了元素，而问题要求相反的顺序。 

为什么它有效

 关键的不变量是，如果我们跳过给定步骤中不是最右边有效位置的任何元素，以后就不能安全地删除它。 严格向左删除元素会移动其索引并永久破坏相等条件。 删除其右侧的元素不会影响它。 因此，如果存在可移动元素，那么最右边的元素是唯一安全且不可逆的选择。 这使得贪婪选择既必要又充分，并且当且仅当存在完整删除顺序时，重复应用最终会耗尽所有有效删除。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = list(map(int, input().split()))

    ans = []

    for _ in range(n):
        idx = -1

        for i in range(len(a) - 1, -1, -1):
            if a[i] == i + 1:
                idx = i
                break

        if idx == -1:
            print("NO")
            return

        ans.append(a[idx])
        a.pop(idx)

    print("YES")
    print(*ans[::-1])

if __name__ == "__main__":
    solve()
```实现直接遵循贪心过程。 内循环从末尾开始搜索，以确保我们选择最右边的有效位置。 这`pop`操作删除该元素并移动数组，自然地为后续迭代维护正确的索引。 最终的逆转`ans`将输出与所需的顺序对齐。 

一个微妙的点是使用`i + 1`在比较中。 这反映了条件中从 1 开始的索引，而 Python 数组是从 0 开始的。 仔细对齐可避免验证过程中出现相差一的错误。 

## 工作示例

 考虑数组$[1, 2, 3]$。 

### 轨迹 1

 | 步骤| 数组| 找到索引 | 已删除 |
 | --- | --- | --- | --- |
 | 1 | [1,2,3]| 2 (3=3) | 2 (3=3) | 3 |
 | 2 | [1, 2] | 1 (2=2) | 1 (2=2) | 2 |
 | 3 | [1] | 0 (1=1) | 0 (1=1) | 1 |

 移除命令是$[3, 2, 1]$，并且反转给出$[1, 2, 3]$。 这证实了完全一致的数组在规则下完全崩溃。 

### 轨迹 2

 考虑$[2, 1, 3]$。 

| 步骤| 数组| 找到索引 | 已删除 |
 | --- | --- | --- | --- |
 | 1 | [2,1,3]| 2 (3=3) | 2 (3=3) | 3 |
 | 2 | [2, 1] | 无 | 失败|

 在这里，该过程提前停止，因为没有位置满足条件。 这表明即使某些元素最初与其索引匹配，也并非所有排列都是可解的。 

该跟踪表明该算法正确地强制执行最右侧有效删除的必要性，并在没有保留有效状态时检测到失败。

 ## 边缘情况

 一个重要的边缘情况是只有最后一个元素最初有效。 例如，在$[2, 3, 1]$，只有最后一个位置满足条件。 算法立即选择它，确保正确性。 移除后，剩余的结构变得更小，并被干净地重新评估，保留了不变量。 

另一种情况是存在多个有效位置但只有最右边的位置是安全的。 在$[1, 3, 2]$，位置 1 和位置 3 最初都是有效的。 该算法首先选择位置 3。 相反，删除位置 1 会移动后面的元素并破坏未来的有效性，从而导致死亡状态。 最右边的规则可以防止这种失败路径。 

最后一个微妙的情况是，有效性由于转移而出现得较晚。 在$[2, 1, 3, 4]$，首先删除 4 是必要的，因为它是最右边有效的。 只有在那之后，下一个正确的结构才会出现。 该算法自然会处理这个问题，因为它会在每次删除后重新计算有效性，而不是依赖于过时的位置。
