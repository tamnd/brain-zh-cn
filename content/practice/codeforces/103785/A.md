 ## 解决方案
 ## 问题理解

 我们得到了相同对象的集合和容器容量。 每个容器最多可容纳$K$对象，我们想要放置所有$N$物体放入容器中。 任务是确定最佳包装时所需的最小容器数量。 

输入由两个整数组成。 第一个代表我们总共有多少个对象。 第二个表示单个容器中可以放置的最大对象数量。 输出是一个整数，表示需要多少个容器来容纳所有对象而不超出容量。 

约束足够小，每个测​​试用例的计算时间必须恒定。 即使$N$很大，可能高达$10^9$在典型的竞争性编程设置中，所涉及的操作是简单的算术，因此线性或迭代打包模拟是不必要的，并且如果在许多测试用例中重复，则可能会太慢。 预期的解决方案必须依赖于直接的数学推理而不是模拟。 

当对象数量恰好能被容器大小整除时，就会出现微妙的边缘情况。 例如，如果$N = 10$和$K = 2$，正确答案是 5。一种总是在除法之后添加一个额外容器的简单方法会错误地返回 6。另一种边缘情况是$K > N$。 例如，$N = 3$,$K = 10$，正确答案是 1，而不是 0。纯整数除法$N / K$如果不调整，将错误地给出 0 个容器。 

## 方法

 思考这个问题的一种直接方法是模拟一个一个地填充容器。 我们从第一个容器开始，最多放置$K$对象，从中减去它们$N$，并重复直到没有物体剩余。 每次迭代都代表填充一个容器。 这是正确的，因为它直接反映了约束。 

然而，这种方法的表现大致如下$N / K$迭代，并且在最坏的情况下$K = 1$，它退化为$N$步骤。 如果$N$如果问题的结构是纯粹的均匀打包，那么这会变得低效且不必要。 

关键的观察结果是，除了最后一个容器之外，每个容器都被完全装满。 这意味着我们正在有效地分区$N$将项目按大小分组$K$。 此类群体的数量正好是上限$N / K$。 这完全消除了模拟，并将问题简化为单个算术表达式。

 ## 算法演练

 1. 读取值$N$和$K$。 这些定义了我们必须放置多少物品以及每个容器可以容纳多少物品。 
2. 计算整数除法$N // K$，它计算我们可以形成多少个完整的容器。 这给出了假设完美包装的集装箱的基线数量。 
3、检查除法时是否有余数$N$经过$K$。 如果$N \bmod K \neq 0$，然后还有剩余物品需要额外的容器。 
4. 如果存在余数，则将容器计数加一。 这说明了部分填充的最终容器。 
5. 输出最终计数。 

余数检查背后的原因是整数除法向零截断，丢弃任何未形成完整组的剩余部分。 被丢弃的部分仍然需要额外容器中的空间。 

### 为什么它有效

 除最后一个容器外，每个容器都必须准确包含$K$物品采用最佳包装。 如果我们有多个部分填充的容器，我们可以将它们合并并减少容器的总数，这与最优性相矛盾。 因此，最多可以有一个不完整的容器，并且它存在当且仅当$N$不能被整除$K$。 这保证了计算的上限$N / K$给出最佳容器数量。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

n, k = map(int, input().split())

containers = n // k
if n % k != 0:
    containers += 1

print(containers)
```该解决方案直接对数学观察结果进行编码。 整数除法计算满容器的数量，模检查确定是否需要额外的容器。 不需要循环或额外的数据结构，这使得实现最小化并避免任何性能问题的风险。 

一个常见的错误是忘记剩余的情况。 只写`n // k`失败时$N$不能完全整除$K$，因为剩余的物体仍然需要一个容器。

 ## 边缘情况

 一个重要的边缘情况是当$N < K$。 用于输入`5 10`，算法计算$5 // 10 = 0$并找到一个非零余数，这会触发增量。 最终结果变成1，正确地代表即使少量的物品仍然需要一个容器。 

另一种情况是精确整除，比如`12 3`。 计算结果$12 // 3 = 4$且余数为零。 由于不需要额外的空间，所以答案仍然是 4。这证实了该算法不会过度计算容器。 

第三种情况是最小输入，例如`1 1`。 这里除法的结果是 1 并且不存在余数，因此结果是 1，符合一件物品正好装满一个容器的直观预期。
