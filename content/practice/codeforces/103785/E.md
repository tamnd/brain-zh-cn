 ## 解决方案
 ## 问题理解

 我们得到一行`N`宿舍房间，每个房间都有一定的清洁成本或与之相关的重量。 管理层希望以结构化的方式分配清扫工：他们必须选择周期性模式，而不是任意选择房间。 

通过在每个大小块中精确选择一个位置来形成有效的分配`K`，然后在整个数组中重复此选择。 具体来说，如果我们决定从位置开始`i`，然后我们选择房间`i, i+K, i+2K, ...`直到我们离开数组。 每次这样的选择都会形成一个候选清扫工团队，其总成本是这些选定位置的值的总和。 

关键的限制是`K`划分`N`，所以数组可以完美划分为`K`余数类模`K`。 任务是计算每个残基类别的总成本并返回其中的最小值。 

从约束的角度来看，这是一个经典的线性聚合问题。 即使`N`大约是`2 × 10^5`，我们只对数组执行一次传递，所以`O(N)`解决方案就足够了。 任何涉及对所有起点进行嵌套迭代并重新计算总和的事情都会降级为`O(N^2 / K)`在最坏的情况下，这太慢了`N`很大。 

当所有值都相等或最小类在局部不明显时，就会出现微妙的边缘情况。 例如，如果数组是`[5, 1, 5, 1]`和`K = 2`，残基类别是`[5 + 5]`和`[1 + 1]`，所以答案是`2`。 一个天真的错误是采用单个最小元素或假设局部最小值占主导地位，这会失败，因为我们正在对结构化子序列求和，而不是选择独立元素。 

另一个失败案例是忘记分组是严格按索引模进行的`K`。 任何贪婪地挑选每一个最小的尝试`K`-th 元素不考虑对齐会产生错误的分组。 

## 方法

 蛮力的想法很简单：尝试从`0`到`K - 1`，然后计算索引处元素的总和`i, i + K, i + 2K, ...`。 每笔金额都需要粗略扫描`N / K`元素，因此总复杂度变为`O(K × N / K) = O(N)`。 有趣的是，如果正确实现，即使简单的公式也已经是线性的，但许多简单的实现会重复重新计算部分和或显式构建子序列，从而导致高级语言中的开销和隐藏的二次行为。 

关键的观察是数组自然分解为`K`基于索引模的独立组`K`。 每个索引都属于一个组，并且每个组独立地对候选答案做出贡献。 因此，与其反复行走，不如跳跃大小`K`，我们可以直接累加为`K`一次通过桶。 

这将问题变成了一个简单的分区求和任务：维护一个数组`bucket[0..K-1]`， 添加`arr[i]`进入`bucket[i % K]`，最后取最小桶和。 

暴力破解之所以有效，是因为每个有效选择都与一个残基类别精确对应，但如果以重复跳转的形式实现，它就会变得低效或尴尬。 观察到索引形成不相交的模类，从而将重复遍历减少为单个累积遍历。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力（每次启动时跳跃）| O(N) | O(1) 到 O(N) | 已接受但多余 |
 | 最佳（mod 分组）| O(N) | O(K) | 已接受 |

 ## 算法演练

 1. 阅读`N`和`K`，然后读取数组`arr`。 我们将每个索引视为属于由下式确定的残基类别：`i % K`。 
2. 创建数组`bucket`大小的`K`，初始化为零。 每个条目将存储一个起始偏移类别的总成本。 
3. 迭代所有索引`i`从`0`到`N - 1`。 
4. 添加`arr[i]`到`bucket[i % K]`。 这会将每个元素分配到正确的周期组，而无需显式构造子序列。 这一步是中心变换，避免重复遍历。 
5.处理完所有元素后，计算所有元素中的最小值`bucket[j]`为了`0 ≤ j < K`。 
6. 输出这个最小值作为答案。 

### 为什么它有效

 每个有效的扫描配置完全对应于选择一个残基类模`K`。 桶的构造确保每个元素恰好对一个类有贡献，并且没有元素被重复计算。 由于每个类代表通过步幅步进选择的所有元素的总和`K`，桶值正是所有可能的有效扫描模式的成本。 对这些详尽且不相交的配置采取最小值可以保证获得最佳解决方案。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, k = map(int, input().split())
    arr = list(map(int, input().split()))
    
    bucket = [0] * k
    
    for i in range(n):
        bucket[i % k] += arr[i]
    
    print(min(bucket))

if __name__ == "__main__":
    solve()
```实现直接遵循桶分解的思想。 数组`bucket`存储每个模类的累积和。 模运算确保正确分组，而无需显式迭代大小跳跃`K`。 

一个常见的错误是尝试构建`k`单独的列表然后对它们求和，这是不必要的并且会增加内存开销。 另一个微妙的问题是忘记正确初始化存储桶，特别是在未初始化的数组可能包含垃圾值的语言中。

 ## 边缘情况

 一种边缘情况是当`K = 1`。 在这种情况下，每个元素都属于同一组，因此答案必须是整个数组的总和。 该算法自然地处理这个问题，因为每个索引都满足`i % 1 = 0`，因此所有值都会累积到一个存储桶中。 

另一个边缘情况是当`K = N`。 每个元素形成自己的组，因为每个索引都映射到不同的残基类别。 该算法正确地分配了每个`arr[i]`到一个单独的桶中，​​最小的桶成为数组中最小的元素。 

第三种情况是值交替频繁时，例如`[1, 100, 1, 100, ...]`和`K = 2`。 这里，每个存储桶根据奇偶校验收集所有小值或所有大值，并且算法正确地将它们分开而不混合，从而保留比较所需的结构。
