 ## 问题理解

 我们正在模拟一个传播过程，其中固定数量的“跳线”充当有限的资源，将更新传播到越来越多的计算机上。 该过程从一台已更新的计算机开始，在每个步骤中，我们都尝试使用可用的电缆来扩展更新的计算机的数量。 每根电缆有效地允许一步中一个额外的连接，但有用连接的数量也受到当前已更新的计算机数量的限制。 

状态由两个整数完整描述。 一是当前更新的计算机数量，最初为 1。另一个是跳线数量，在整个过程中保持不变。 在每次迭代中，我们都会增加更新计算机的数量，具体取决于我们是否有足够的电缆来充分利用更新机器的当前前沿，并且我们重复直到所有 n 台计算机都更新为止。 

输出是到达至少 n 个更新的计算机所需的步骤数。 

从复杂性的角度来看，n 可能足够大，以至于任何递增 1 或单独处理每台计算机的模拟都会太慢。 关键的观察是，当资源充足时，当前状态的值至少呈几何级数增长，因此在最佳状态下，迭代次数是 n 的对数。 这使解决方案牢固地处于 O(log n) 或 O(log n) 摊销时间期望中，排除了任何 O(n) 每步扩展。 

当跳线非常小时，特别是 patch = 0 时，会出现微妙的边缘情况。在这种情况下，该过程无法扩展到初始计算机之外，因此，如果 n > 1，则在简单的假设下不可能得出答案。 当 patch 与 n 相比非常大时，就会出现另一种边缘情况，其中每个步骤的增长立即加倍。 

粗心的实现可能会错误地假设线性增长或无法限制 n 处的增长，从而导致不必要的迭代或溢出。 

## 方法

 该过程的强力解释很简单：维护已更新计算机的数量，并重复应用一步可以到达多少台新计算机的规则，一旦计数达到 n 就停止。 每次迭代都会将贡献计算为当前更新机器数量与电缆数量之间的最小值。 这是正确的，因为每台更新的计算机每个步骤最多可以提供一个传出连接，并且每条电缆每个步骤只能使用一次。 

然而，当 n 很大时，这种模拟变得低效。 在最坏的情况下，如果 patch 与 n 相当，则迭代次数很小，但如果 patch 相对于 n 很小，我们仍然需要很多步骤，并且每个步骤只添加一个小的增量。 蛮力方法每次迭代的工作时间为 O(1)，但可能需要 O(n) 次迭代，当 n 很大时，这太慢了。 

关键的见解是，该过程完全由两个量中较小的一个控制：活动计算机的数量和电缆的数量。 当活动计算机的数量小于或等于电缆的数量时，增长是倍增的，基本上是当前状态的两倍。 当活动计算机的数量超过电缆的数量时，增长将变为线性，增加的固定量等于每一步的电缆数量。 这产生了一个两相系统：指数阶段后跟线性阶段。 该算法简化为模拟这些转换，无需任何每个节点的工作，因为每个步骤仅取决于当前值。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力模拟| 最坏情况 O(n) | O(1) | O(1) | 太慢了 |
 | 基于相位的仿真 | O(log n) | O(1) | O(1) | 已接受 |

 ## 算法演练

1. 将更新计算机的数量初始化为 1，将计步器初始化为 0。这表示系统从一台已感染的计算机开始。 
2. 当更新的计算机数量小于n时，确定当前步骤可以到达多少台新计算机。 
3. 计算增量为当前已更新计算机的数量与电缆数量之间的最小值。 这反映了这样一个事实：每台更新的计算机最多只能提供一个传出连接，但我们也受到可用电缆数量的限制。 
4. 将此增量添加到当前已更新计算机的数量。 这会在一个传播步骤后更新可达集。 
5. 将计步器加一以记录已过去的一个时间单位。 
6. 重复，直到更新的计算机数量达到或超过n。 

为什么它有效：关键的不变量是每次迭代之后，值`curr`准确地表示在给定该步骤数的过程限制的情况下可以到达的最大计算机数量。 更新规则正确地模拟了供应（当前更新的计算机）和容量（跳线）之间的瓶颈。 由于每个步骤都完全耗尽两个限制中较小的一个，因此不会留下未使用的隐藏容量，并且模拟不会低估或高估可到达的节点。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, patch = map(int, input().split())
    
    curr = 1
    hours = 0
    
    while curr < n:
        curr += min(curr, patch)
        hours += 1
    
    print(hours)

if __name__ == "__main__":
    solve()
```该实现直接反映了前面描述的状态转换。 变量`curr`跟踪有多少台计算机已更新，以及`patch`是固定的。 每次循环迭代使用以下方法计算正确的增量`min(curr, patch)`，它编码了可用源和可用电缆之间的瓶颈。 

一个常见的错误是错误地使用`curr += patch`无条件地高估增长，当`curr < patch`。 另一个未能停下来`n`，这可能会导致不必要的增长超出目标，甚至在其他语言中溢出。 循环条件确保在达到所需数量时准确终止。

 ## 边缘情况

 一种边缘情况是当 n 等于 1 时。该算法正确返回 0，因为条件`curr < n`开始时为 false，因此不需要传播。 

另一种边缘情况是补丁非常大时。 用于输入`n = 8, patch = 100`，第一次迭代将 curr 从 1 设置为 2，然后将 2 设置为 4，然后将 4 设置为 8。循环分 3 步终止，并且大补丁值永远不会变得相关。 这证实了`min(curr, patch)`配方可以安全地处理过剩产能。 

第三种边缘情况是 patch 为 1 时。`n = 6, patch = 1`，级数为1→2→3→4→5→6，需要5步。 每一步都贡献一个新节点，与线性状态行为相匹配，并且一旦达到阈值，循环就会干净地终止。
