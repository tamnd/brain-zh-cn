 ## 解决方案
 ## 问题理解

 我们得到了一系列排列成一条线的位置。 每个位置最初都包含一定量的垃圾。 有一个清洁过程，涉及选择一些工人，这些工人按顺序在各个地点移动，清理每个地点的垃圾。 当工人到达某个地点时，他们可以清除那里的垃圾，但任何剩余的垃圾可能会被部分结转，具体取决于有多少工人可用。 

关键问题不是模拟固定数量的工人，而是确定所需的最少工人数量，以便在最后一个位置结束时可以完全清除所有位置的所有垃圾。 

每个工人都会为每个地点的处理能力做出贡献，如果一个地点的垃圾相对于可用工人而言过多，则其中一些垃圾必须“向前转移”到下一个地点，而一部分则可以立即清除。 如果经过优化转移后，最后一个位置仍有剩余垃圾，则说明所选工人数量不足。 

输入描述了每个位置的垃圾量。 输出是一个整数：能够在给定规则下处理所有位置的所有垃圾的最小工作人员数量。 

这些约束允许大约有 200000 个量级单位，并且数组大小足够大，以至于不可能进行 O(n^2) 模拟。 尝试模拟每个候选工人数量的工人行为的简单方法是将 O(n) 扫描乘以 O(A) 可能的工人计数，这太慢了。 这立即建议对答案进行对数搜索并结合线性可行性检查。 

当处理位置之间的转移时，会出现一种微妙的故障模式。 如果一个位置的单位数量略多于 k 但不超过 2k，那么它仍然可行，但前提是仔细转发多余的部分。 完全清除或完全转发的幼稚贪婪策略可以打破这种中间状态。 例如，如果 k = 3 并且某个位置有 5 个单位，则本地删除所有 5 个单位是不可能的，但转发所有 5 个单位也是不可能的，因为即使存在更好的分割，下一个位置也可能变得不可行。 

最后一个位置是另一个重要的边缘情况。 最终位置的任何剩余量都无法进一步转移，因此如果剩余量超过直接容量，配置必须立即失败。 

## 方法

 该解决方案的核心结构来自于观察到可行性在工人数量上是单调的。 如果 k 个工人足以清理所有东西，那么任何更大的数字 k + 1 也足够了，因为只有当容量增加时，每个本地操作才会变得更容易。 这种单调性表明二进制搜索最小 k。 

蛮力思想会尝试从 1 到最大可能值的每个可能的 k，并检查 k 个工人是否可以清理所有垃圾。 每次检查都需要模拟沿线的流程，贪婪地分发或转发垃圾。 由于每次模拟都是O(n)，并且k可以大到大约200000，这导致O(nA)复杂度，太大了。 

改进来自于分离关注点。 我们没有将问题视为对 k 的直接模拟，而是将 k 视为可行性参数，并提出一个是或否的问题：k 个工人能否清洁所有位置？ 这将问题转换为单调的决策问题，从而实现二分搜索。 

剩下的挑战是正确实施可行性检查。 在每个位置，我们必须决定本地清除多少垃圾以及转发多少垃圾。 最优策略的结构受到容量限制的强制：每个工人都可以为本地移除和转移做出贡献，唯一有意义的机制是当前数量是否在 k 以内、k 到 2k 之间或 2k 以上。

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 通过模拟对 k 进行暴力破解 | O(nA) | O(1) | O(1) | 太慢了 |
 | 二分查找 + O(n) 可行性检查 | O(n log A) | O(n log A) | O(1) | O(1) | 已接受 |

 ## 算法演练

 ### 固定 k 的可行性检查

 1. 从第一个位置开始，跟踪该位置当前的垃圾量 x。 最初 x 是该位置的输入值。 

该值代表本地垃圾和来自先前位置的任何遗留垃圾。 
2. 如果 x ≤ k，则清除该位置的所有垃圾，并且不携带任何物品。 

这是有效的，因为所有垃圾都可以由该位置的可用工人直接处理。 
3. 如果 x > 2k，则立即失败。 

原因是，即使所有 k 个工人都全力参与转移和搬运，一个地点最多只能处理 2k 个单位，因此超过这个值就无法完成。 
4. 如果 k < x ≤ 2k 并且这不是最后一个位置，则以最佳方式分割工作：本地删除 k 个单元，并将 x − k 个单元转发到下一个位置。 

其思路是，k个工人在本地充分消耗其产能，剩余的多余部分必须以最紧凑的方式结转。 
5. 如果 k < x ≤ 2k 并且这是最后一个位置，则失败。 

最后无法转发，因此超出 k 的任何超出部分都无法解决。 
6. 移动到下一个位置，将所有转发的垃圾添加到其初始数量，然后重复，直到处理完所有位置。 
7.如果所有位置都处理完毕且没有触发失败，则k是可行的。 

### 为什么它有效

 关键的不变量是，在每一步，携带价值代表未来位置必须处理的最小不可避免的剩余物。 转换规则确保我们不会转发超过必要的内容，因为转发额外的内容只会使未来的位置更难满足。 2k 的界限源于以下事实：每个工作人员最多贡献一个单位的本地移除量和最多一个单位的转发能力，因此每个位置的总有效处理量上限为 2k。 任何超过此的配置都无法分解为有效的工作分配。 

二分搜索是有效的，因为可行性函数是单调的：增加 k 永远不会降低在任何位置处理垃圾的能力。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def can(k, a):
    carry = 0
    n = len(a)
    
    for i in range(n):
        x = a[i] + carry
        
        if x <= k:
            carry = 0
        elif x > 2 * k:
            return False
        else:
            if i == n - 1:
                return False
            carry = x - k
            
    return True

def solve():
    n = int(input())
    a = list(map(int, input().split()))
    
    lo, hi = 1, max(a)
    
    while lo < hi:
        mid = (lo + hi) // 2
        if can(mid, a):
            hi = mid
        else:
            lo = mid + 1
    
    print(lo)

if __name__ == "__main__":
    solve()
```实现将可行性检查分离成一个函数`can(k, a)`它模拟了贪婪的前向过程。 变量`carry`表示从前一个位置推送的剩余垃圾，并添加到当前位置。 

关键的细节是转换规则`k < x ≤ 2k`案件。 我们始终准确地转发`x - k`，永远不会多，也不会少。 转发较少意味着在当前位置留下额外的未处理垃圾，这与可行性相矛盾。 转发更多只会让未来的处境变得更加困难，而不会改善当前的处境。 

二分查找范围选择为`[1, max(a)]`因为如果存在任何垃圾，至少需要一名工人，并且`max(a)`工作人员无需转发即可轻松处理任何单一职位。 

## 工作示例

 ### 示例 1

 考虑`a = [3, 2, 4]`。 

我们测试可行性`k = 3`。 

| 我| 一个[我] | 携带| x| 决定| 新携带|
 | --- | --- | --- | --- | --- | --- |
 | 0 | 3 | 0 | 3 | x ≤ k，明确 | 0 |
 | 1 | 2 | 0 | 2 | x ≤ k，明确 | 0 |
 | 2 | 4 | 0 | 4 | k < x ≤ 2k，最后 → 失败 | - |

 所以k=3是不可行的。 

为了`k = 4`:

 | 我| 一个[我] | 携带| x| 决定| 新携带|
 | --- | --- | --- | --- | --- | --- |
 | 0 | 3 | 0 | 3 | 清除 | 0 |
 | 1 | 2 | 0 | 2 | 清除 | 0 |
 | 2 | 4 | 0 | 4 | x ≤ k，明确 | 0 |

 k = 4 有效，所以答案是 4。 

这证实了二分搜索边界行为。 

### 示例 2

 考虑`a = [5, 1, 6]`，k = 4。 

| 我| 一个[我] | 携带| x| 决定| 新携带|
 | --- | --- | --- | --- | --- | --- |
 | 0 | 5 | 0 | 5 | k < x ≤ 2k → 进位 1 | 1 |

 ## 边缘情况

 单个位置超过2k的情况会在可行性检查中立即处理。 例如，如果 k = 3 并且某个位置有 7 个单位，则条件`x > 2k`触发失败。 这对应于即使在工作人员之间进行最大分配也无法在不违反每个工作人员限制的情况下分配工作负载的情况。 

最终位置约束是单独强制的。 如果进位导致最后位置超过 k，即使在 2k 以内，算法也会正确拒绝它。 例如，当 k = 4 且最后位置 x = 6 时，我们进入中间情况并检测到最后转发是不可能的。 

中间分流箱确保多余部分始终以受控方式向前推进。 如果 x = k + t，则精确转发 t 可确保下一个位置仅接收严格必需的内容，从而保留未来位置的可行性，而不是任意增加负载。
