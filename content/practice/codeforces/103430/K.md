 ## 解决方案
 ## 问题理解

 我们给出了一个从 1 到 n 索引的位置系统。 每个位置都有一个规则决定我们下一步移动的位置：从 i 开始，我们要么向前移动一步到 i + 1，要么进行更大的跳跃到 i + k[i]。 两者中的哪一个发生取决于参数 x 的变化。 随着 x 随着时间的推移而增加，一些位置的行为从“平庸”转变为“美味”，这有效地改变了该位置的传出边缘。 

我们需要反复考虑从位置 1 开始沿着这些有向边的路径，直到退出该范围。 每次由于 x 增加而导致配置发生变化时，我们必须重新计算沿这条路径累积的总成本。 

关键的难点在于，每次更新只修改一个节点的出边，但这种变化可以完全重塑指针的树状结构，并且每次修改后我们仍然需要回答关于从 1 开始的路径上的总和的查询。 

这些约束意味着 n 足够大，以至于在每次更新后从头开始重新计算路径太慢。 在最坏的情况下，每次更改后从 1 开始的简单遍历将在每次更新上花费 O(n) 的成本，从而导致总体 O(n^2)，这对于 10^5 左右的典型限制来说太慢了。 

微妙的结构是每个节点都有一个出边，因此该图是一个函数图，它总是形成一个指向前方的有根树的森林。 当沿着这些指针压缩时，每个连接的组件都是链状结构。 

当人们假设更新是本地的时，就会出现一个幼稚的错误。 例如，如果我们只更改从 i 开始的边，并且不重新考虑 i 之前的节点，我们可能会错误地假设路径中未受影响的部分保持稳定。 但是，即使单个节点的更改也可以重定向整个遍历的后缀。 

一个小的说明性案例：

 输入：

 n = 5, k = [ -, 2, 1, 2, 1, 1 ]

 起始边：1→2、2→4、3→4、4→5、5→出

 在节点2更新后，假设2从2→4变为2→3。 现在，从 1 开始的路径从 1→2→4→5 变为 1→2→3→4→5，完全改变了结构和路径和。 任何假设仅在节点 2 进行本地校正的方法都无法考虑下游重新路由。 

因此，我们需要一种支持动态边缘更新的结构，同时仍然有效地回答“从 1 开始的路径之和”。 

## 方法

 暴力方法很简单：每次更新后，从节点 1 开始重建或遍历图，沿着传出边直到退出。 每一步都严格遵循一个指针，因此正确性并不重要。 问题是性能：每个查询可能需要 O(n) 遍历，并且最多 O(n) 更新，这会导致 O(n^2)，这是不可行的。 

关键的观察结果是，该结构是一个有根森林，其中每个节点只有一个传出边缘，并且边缘始终指向更高的索引。 这保证了没有循环并确保单调的向前运动。 该结构的行为就像一组动态重新连接的链表。 

我们需要支持两种操作：更改一个出边，并计算从 1 开始沿路径的总和。像链接剪切树这样的完整动态树结构可以工作，因为它支持路径查询的链接和剪切操作，但对于这种特定结构来说它很重。 

如果我们通过将节点分割成块来利用局部性，问题就会变得更容易管理。 在块内，我们维护预先计算的信息，使我们能够“跳跃”穿过块，而不是逐个节点地步进。 这是应用于指针跳跃结构的经典平方根分解思想。 

我们将节点划分为大小为 m 的块。 对于每个节点 i，我们维护两个值：toi，当沿着位于其块之外的 i 的边时到达的第一个节点，以及 costi，沿路径直到到达 toi 的权重总和。 这将块内遍历压缩为 O(1) 跳转。

当从 1 开始计算路径时，我们使用这些预先计算的值重复地逐块跳转，从而导致 O(n / m) 步骤。 

当节点 i 的边发生变化时，仅包含 i 的块受到影响。 我们以 O(m) 的时间重新计算该块内的所有 toi 和 costi 值。 选择 m ≈ √n 可以平衡重新计算和查询时间。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | 每次更新 O(n) | O(n) | 太慢了 |
 | 块分解| 每次更新/查询 O(√n) | O(n) | 已接受 |

 ## 算法演练

 我们维护一个函数图，其中每个节点指向 i + 1 或 i + k[i]，并将节点分组为固定大小 m 的块。 

1. 将节点数组分割为大小为 m 的连续块。 这确保了块内的任何路径段都足够小，可以有效地重新计算。 
2. 对于每个节点 i，根据其当前状态定义其出边，并显式存储它。 这允许在更新期间快速重新布线。 
3. 对于每个节点 i，通过模拟从 i 开始的指针跳转来计算 toi，直到我们离开块或到达块外的节点。 同时，将 costi 计算为沿该块内路径的边权重之和。 
4. 要回答从节点 1 开始的查询，请重复从当前位置 i 移动到 toi 并将 costi 添加到答案中，直到 i 离开数组。 每次跳转都会跳过整个块的内部结构。 
5. 当更新修改节点 i 时，重新计算与 i 同一块中的所有节点的所有 toi 和 costi 值。 这是通过重新运行块内的指针逻辑来完成的，因为只有内部结构可能发生了变化。 

这足够的原因是，从 1 开始的任何路径都可以分解为一系列块内片段。 每个段都被预先计算的（toi，costi）完全捕获，因此单个边缘变化后的重新计算只需要修复一个本地段。 

### 为什么它有效

 不变量是，对于每个节点 i，toi 和 costi 准确地表示在当前边配置下，跟随传出边直到退出包含 i 的块的结果。 每次更新后，我们都会为受影响的块恢复这个不变量。 由于从 1 开始的每个完整路径都可以分解为不相交的块段，并且每个段都由其存储的摘要正确表示，因此总计算路径总和始终与当前图一致。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    nxt = [0] * (n + 1)
    val = [0] * (n + 1)

    for i in range(1, n + 1):
        t, k = map(int, input().split())
        if t == 0:
            nxt[i] = i + 1 if i + 1 <= n else n + 1
            val[i] = 1
        else:
            nxt[i] = i + k if i + k <= n else n + 1
            val[i] = 1

    import math
    m = int(math.sqrt(n)) + 1
    block = lambda x: (x - 1) // m

    to = [0] * (n + 2)
    cost = [0] * (n + 2)

    def rebuild(b):
        L = b * m + 1
        R = min(n, (b + 1) * m)
        for i in range(R, L - 1, -1):
            j = nxt[i]
            if j > n or block(j) != b:
                to[i] = j
                cost[i] = val[i]
            else:
                to[i] = to[j]
                cost[i] = val[i] + cost[j]

    for b in range((n + m - 1) // m):
        rebuild(b)

    def query():
        i = 1
        res = 0
        while i <= n:
            res += cost[i]
            i = to[i]
        return res

    q = int(input())
    for _ in range(q):
        idx = int(input())
        nxt[idx] = idx + 1 if nxt[idx] != idx + 1 else idx + 2
        b = block(idx)
        rebuild(b)
        print(query())

if __name__ == "__main__":
    solve()
```该实现首先构建初始功能图，为每个节点存储其传出指针和该移动的成本。 通过选择 √n 左右的块大小，然后为每个节点预先计算其块的第一个出口点以及到达该点的成本来构建平方根分解。 

重建功能是解决方案的核心。 它从右到左处理一个块，以便在计算 to[i] 时，我们可以依赖已经计算出的 to[j] 和 cost[j] 的值来计算下一个节点。 这种依赖顺序至关重要，因为每个节点的值都取决于其后继节点。 

每次更新都会更改一个节点的传出边缘并触发其块的完全重建。 查询函数使用预先计算的块出口重复跳转，在 O(√n) 时间内累积成本。 

一个微妙的实现细节是使用哨兵 n+1 处理边界出口，该哨兵充当终止。 如果没有这个，数组末尾的边缘情况将需要额外的分支并存在错误终止的风险。

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 | O(n√n) | O(n√n) | 每次 n 更新都会触发一次 O(√n) 的块重建，并且每次查询的成本为 O(√n) |
 | 空间| O(n) | 数组存储下一个指针和块摘要 |

 √n 分解确保更新和查询保持平衡。 对于 n 高达典型竞赛限制（例如 10^5）的情况，这可以在时间限制内轻松运行。

 ## 边缘情况

 一个重要的边缘情况是当节点的跳转恰好超出 n 时。 在这种情况下，遍历必须立即停止，任何对 nxt 或 cost 数组进行索引的尝试都将无效。 哨兵 n+1 确保统一处理该终止。 

另一种边缘情况是更新发生在块边界附近。 例如，如果 i 是块的最后一个元素，则其更新只能影响该块，但查询可能仍依赖于 to[i] 跳转到下一个块。 从右到左的重建顺序保证了正确性，因为依赖项始终指向前方。 

最后一个微妙的情况是在同一节点上重复更新。 每次更新都必须完全重新计算块，因为过时的值和成本值将持续存在并默默地破坏未来的查询。
