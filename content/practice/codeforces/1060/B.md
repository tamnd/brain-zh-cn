 ## 解决方案
 ## 问题理解

 我们得到一个大整数$n$，我们想把它分成两个非负整数$a$和$b$这样他们的总和就保持不变$n$。 在所有可能的分割中，我们被要求最大化两部分的数字和的总和，这意味着我们想要最大化$S(a) + S(b)$， 在哪里$S(x)$是小数位的总和$x$。 

所以问题不在于选择任意数字，而在于分配数字$n$分成两个正确相加的数字，同时使数字之和尽可能大。 

约束条件$n \le 10^{12}$方法$n$最多有 12 位数字。 直接枚举所有 split$a \in [0, n]$是不可能的，因为它需要迭代到$10^{12}$值，远远超出了可行的范围。 即使检查每个分割并计算数字总和也是如此$O(n \cdot \log n)$，这是完全不可行的。 

这立即迫使我们寻找数字和在带有进位的加法下如何表现的结构属性。 

天真的直觉可能表明，使两者$a$和$b$“平衡”可能会有所帮助，但数字和在进位传播下表现得很奇怪。 真正的复杂性来自进位如何减少数字之和：每当两个数字之和达到 10 或更多时，进位会将该位置的数字之和减少 9。 

出现微妙的边缘情况时$n$例如，是十的幂$n = 1000$。 天真的分裂就像$500 + 500$感觉很平衡，但与类似的结构相比，数字总和相对较小$999 + 1$，更有效地利用携带。 这暗示最佳结构与十进制进位紧密相连，而不是与对称分区紧密相连。 

## 方法

 蛮力方法会尝试所有可能的分割$a$从 0 到$n$，计算$b = n - a$，并评估$S(a) + S(b)$。 这是正确的，因为它会检查每个有效的配置。 然而，它需要$n$迭代次数，以及每位数字总和计算成本$O(\log n)$，所以总复杂度是$O(n \log n)$。 为了$n \approx 10^{12}$，这远远超出了任何可行的运行时间。 

关键的观察是数字和通过进位与加法相互作用。 我们不再考虑选择任意分割，而是逐位查看结构。 除进位外，每个小数位的行为都是独立的。 重要的想法是，最大化数字和的最佳方法是最小化破坏性进位，而是强制两个数字中的数字变得尽可能大。 

这导致了一个标准的贪婪见解：我们尝试构建$a$和$b$使它们的数字尽可能大，同时仍然求和$n$。 一个众所周知的简化是，当我们以创建最大数字 9 和 0 结构的方式“借用”时，就会出现最佳分割。 具体来说，答案是通过尝试拆分来实现的，其中一个数字由前导数字组成，在下降之前饱和到 9，而另一个数字则吸收余数。 

捕获此问题的一个实用方法是尝试拆分$n$在不同的数字边界。 对于每个前缀分割位置，我们模拟使一个数字包含一个前缀，例如$10^k - 1$-风格贡献，最大化本地数字和。 自从$n$最多有12位，我们只需要测试$O(\log n)$来自前缀调整的候选结构。 

因此，我们通过全面搜索来减少问题$a$到由数字结构构造的一小组贪婪候选者。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O(n \log n)$|$O(1)$| 太慢了 |
 | 最佳|$O(d^2)$在哪里$d \le 12$|$O(1)$| 已接受 |

 ## 算法演练

 1. 转换$n$转换成字符串，这样我们就可以直接推断出它的十进制数字。 按数字方式工作至关重要，因为数字和的行为取决于位置之间的进位。 
2. 尝试通过迭代每个可能的位置来构造候选分割，其中我们减少了前缀$n$并将后缀替换为不超过原始数字的最大可能的数字配置。 这体现了将数字推向 9 会增加数字总和的想法。 
3. 对于每个候选构造，计算隐含对$(a, b)$通过模拟剩余的$n$将在修复一个组件后分发。 
4. 计算$S(a) + S(b)$对于每个候选人。 由于 12 位数字的数字和很便宜，因此在实践中，此步骤对于每个候选者来说都是恒定时间的。 
5. 跟踪所有候选的最大值并输出。 

我们只需要这些结构化候选的原因是，任何最佳解决方案都可以转换为其中至少一个数字具有由前缀后跟所有 9 组成的数字模式的解决方案，而不会降低目标。 该变换参数将无限搜索空间减少为一个小的有限集。 

### 为什么它有效

 关键的不变量是，每当数字位置没有完全饱和到 9 时$a$或者$b$，我们可以局部调整分割以将该数字向上推而不违反总和约束，除非进位约束阻止它。 该阻塞点恰好发生在$n$。 因此，任何最佳配置都对应于发生进位转换的边界，而这些边界正是我们列举的边界。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def digit_sum(x: int) -> int:
    s = 0
    while x:
        s += x % 10
        x //= 10
    return s

def solve():
    n = int(input().strip())

    best = digit_sum(n)  # (a=n, b=0) baseline

    # Try splitting by forcing a prefix boundary in b
    # We construct candidates of form:
    # b = 10^k - 1, a = n - b
    p = 1
    while p <= n:
        b = p - 1
        a = n - b
        best = max(best, digit_sum(a) + digit_sum(b))
        p *= 10

    print(best)

if __name__ == "__main__":
    solve()
```该实现是围绕以下观察而构建的：当一个数字接近表格中的多个数字时，最有成效的分割就会发生$10^k - 1$，这是十进制 9 的序列。 我们测试所有这些边界的大小$n$。 

基线案例$(n, 0)$被包括在内，因为它总是有效的，有时是最佳的$n$本身有一个很大的数字和。 

10 的幂循环生成所有可能的前缀对齐“全 9”候选者。 对于每一位这样的候选人$b$，我们计算$a = n - b$，保证约束$a + b = n$总是成立。

## 测试用例```python
import sys, io

def digit_sum(x):
    return sum(map(int, str(x)))

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)

    n = int(input().strip())

    best = digit_sum(n)
    p = 1
    while p <= n:
        b = p - 1
        a = n - b
        best = max(best, digit_sum(a) + digit_sum(b))
        p *= 10

    return str(best)

# provided sample
assert run("35\n") == "17"

# minimum case
assert run("1\n") == "1"

# power of ten case
assert run("100\n") == "19"

# all digits 9
assert run("99\n") == "18"

# large case
assert run("1000000000000\n") == str(max(digit_sum(1000000000000 - (10**k - 1)) + digit_sum(10**k - 1) for k in range(13)))

# symmetric case
assert run("50\n") == str(run("50\n"))

# edge carry-heavy case
assert run("19\n") == "10"

 ## 边缘情况

 对于$n = 1$，唯一有效的分割是$(1, 0)$。 该算法尝试$b = 0$并返回$S(1) = 1$，这是正确的。 

为了$n = 1000$，算法评估$b = 9, 99, 999$。 最好的通常是$b = 999$，产生$a = 1$，给出数字和$1 + 27 = 28$。 该循环正确地达到了这种情况，因为它迭代了 10 的所有幂。 

为了$n = 19$，最好的分割是$10 + 9$，给出数字和$1 + 0 + 9 = 10$。 候选人$b = 9$经过显式测试，算法正确地捕获了这种基于进位的最佳分解。 

这些案例中的每一个都表明该算法不依赖于均匀分布，而是系统地探索进位对齐边界，这正是对朴素分裂进行改进的地方。
