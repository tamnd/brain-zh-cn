 ## 解决方案
 ## 问题理解

 每个电子元件都属于一种类型，每种类型都有一个处理时间。 每种类型都有多个副本。 该机器可以单独加工一个部件（需要付出加工时间），也可以在一次移动中一起加工两个不同类型的部件。 当两个组件一起处理时，该移动的时间成本是两者中较慢的一个，这意味着它们各自时间的最大值。 

任务是将所有组件安排为单个操作或有效对，以便每个组件只处理一次并且总累积时间尽可能小。 

输入描述了多个项目，其中每个项目的权重等于其处理时间，但项目按类型分组，并且同一组内禁止配对。 输出是在一对贡献其两个值中的最大值并且单个贡献其自己的值的规则下的最小可能总成本。 

这些限制意味着最多 1000 个类型，每种类型最多 10,000 个项目。 因此，在最坏的情况下，项目总数可能达到约 1000 万个，因此任何显式扩展每个组件并尝试单独匹配它们的解决方案都会太慢。 该解决方案必须在每种类型的聚合计数级别上工作。 

一个经常失败的天真的想法是贪婪地配对任意两个可用组件而不考虑类型结构。 例如，如果高值类型的所有组件在概念上相互配对，则这是无效的，因为相同的类型无法配对。 当一种类型在计数中占主导地位时，会出现另一种失败情况，例如 100 个时间 10 的副本和 1 个时间 1 的副本。天真的“最大与最大配对”策略可能会留下太多未配对的大项目或创建低效匹配。 

## 方法

 如果我们忽略配对规则，则每个分量将被单独处理，并且答案将只是所有 fi · ti 的总和。 配对之所以变得有益，是因为用单个 max(ti, tj) 替换两个单独的成本 ti + tj 会消除较小的贡献 min(ti, tj)。 因此，每个有效对都会将总成本减少两个值中的最小值。 

这将问题重新表述为最大化总节省量，其中类型 i 和 j 之间的每一对都会节省 min(ti, tj)，并且同一类型的两个组件不能配对。 基本成本是固定的，因此我们只优化通过配对可以节省多少成本。 

强力方法会将每个单独的组件视为一个节点，并尝试计算最大权重匹配，其中边权重为 min(ti, tj)，不包括同一类型内的边。 这是一个多达 1000 万个节点的密集图匹配问题，远远超出了可行的极限。 

关键的结构观察是类型的所有组件都是相同的。 这使我们能够将问题压缩为每种类型的计数，并在聚合组上贪婪地工作。 如果我们总是采用两种当前最“昂贵”的可用类型并尽可能地将它们配对，我们就可以避免在次优匹配中浪费高价值组件。 由于可以使用 min(fi, fj) 批量配对两种类型，因此每个操作都会完全删除至少一种类型，这将堆操作的数量限制为 O(N)。 

这导致了对类型上的最大结构的贪婪过程。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 对所有组件进行强力匹配 | O(M²) 或更差 | O(M)| 太慢了|
 | 聚合类型上的贪婪配对 | O(N log N) | O(N log N) | O(N) | 已接受 |

 ## 算法演练

 我们将所有类型维护在按处理时间排序的结构中，以及它们的剩余计数。

1. 将每种类型作为包含其处理时间的记录插入，并计数到按处理时间排序的最大优先级队列中。 
2. 当至少两种不同类型仍有剩余成分时，提取处理时间最长的两种类型。 
3. 设这两种类型为 A 和 B。我们在它们之间配对尽可能多的组件，这是它们剩余计数中的最小值。 
4. 每个这样的对贡献的成本等于 max(tA, tB)，因此我们将（对的数量）乘以 max(tA, tB) 添加到答案中。 
5. 将两个计数减去所形成的对的数量。 如果某个类型仍有剩余组件，则将其推回到优先级队列中。 
6. 当剩余类型少于两种时，所有剩余组件必须单独处理，因此我们为剩余类型添加remaining_count·t。 

我们总是穷尽两种最高可用类型之间的配对的原因是这些类型主导未来的决策。 在高时间类型中保留部分未使用的容量，同时将其与低时间类型配对只会减少潜在的节省，因为每对的节省是由较小的时间决定的，并且我们希望尽可能保留将高值配对在一起的机会。 

### 为什么它有效

 考虑任何最佳解决方案。 如果它将高时间组件与低时间组件配对，同时存在另一个不同类型的未使用的高时间组件，则交换配对以使两个高时间组件配对在一起永远不会减少总节省，因为当 y 增加时，min(x, y) 也会增加。 这种交换论证表明，最佳解决方案始终可以转化为配对始终发生在当前两种最昂贵的可用类型之间的解决方案。 贪心过程正是构造了这样的配置，因此它匹配最优成本。 

## Python 解决方案```python
import sys
import heapq

input = sys.stdin.readline

def solve():
    n = int(input())
    heap = []
    
    total = 0
    
    for i in range(n):
        f, t = map(int, input().split())
        total += f * t
        heapq.heappush(heap, (-t, f))
    
    saved = 0
    
    while len(heap) > 1:
        t1, c1 = heapq.heappop(heap)
        t2, c2 = heapq.heappop(heap)
        
        t1 = -t1
        t2 = -t2
        
        k = min(c1, c2)
        saved += k * max(t1, t2)
        
        c1 -= k
        c2 -= k
        
        if c1:
            heapq.heappush(heap, (-t1, c1))
        if c2:
            heapq.heappush(heap, (-t2, c2))
    
    if heap:
        t, c = heap[0]
        t = -t
        saved += c * t
    
    print(total - saved)

if __name__ == "__main__":
    solve()
```该解决方案首先计算每个组件单独处理的基准成本。 然后减去通过配对实现的最大可能节省。 

堆存储按处理时间排序的类型，确保我们始终首先考虑最有价值的剩余类型。 当提取两种类型时，算法会使用它们的最小可用计数将它们批量配对，这对于效率至关重要，因为它避免了每个组件的模拟。 

一个微妙的点是，配对后必须仔细更新计数。 仅类型的剩余部分被推回到堆中。 如果一个类型被完全消耗，它将被完全丢弃。 

## 工作示例

 ### 示例 1

 输入类型：

 (7, 2), (1, 2), (10, 3)

 初始状态：

 | 堆内容 | 行动| 结果对 | 已保存 |
 | ---| ---| ---| ---|
 | (10,3), (7,2), (2,1) | (10,3), (7,2), (2,1) | 10 和 7 对 | 2 对 (10,7) | 2·7 = 14 | 2·7 = 14 |
 | (10,1), (2,1) | (10,1), (2,1) | 10 和 2 对 | 1 对 (10,2) | +2·2 = 4 |
 | 剩余 (10,1) | 单身| 1 单 10 | +10 |

 保存的总数为 28，因此最终答案为总数负 28。 

该轨迹表明，重复耗尽前两种类型可以最大化高值之间的交互。 

### 示例 2

 输入类型：

 (10,2), (11,2), (12,2)

 | 堆内容 | 行动| 结果对 | 已保存 |
 | ---| ---| ---| ---|
 | (12,2), (11,2), (10,2) | (12,2), (11,2), (10,2) | 12 和 11 对 | 2 对 | 2·11 = 22 | 2·11 = 22 |
 | (10,2) 左 | 单身| 2 单打 | 20 |

 这表明，尽管 12 最大，但首先将 11 和 12 配对是最佳的，因为它可以保留结构并最大化高价值配对机会。

 ## 边缘情况

 当单一类型主导所有其他类型时，就会出现极端情况。 由于禁止在同一类型内配对，因此即使计数很大，其许多组件也必须保持不配对。 该算法自然地处理这一问题，因为一旦所有其他类型都用尽，堆中仅包含一个条目，并且所有剩余组件都将作为单项添加。 

当多种类型共享相同的处理时间时，会发生另一种情况。 堆顺序可以按任意顺序选择它们，但这不会影响正确性，因为配对成本仅取决于共享值，并且批量配对确保不依赖于内部排序。 

第三种情况是计数高度不平衡，例如一种类型的项目数量远多于所有其他类型的项目总和。 该算法重复配对，直到较小的类型耗尽，然后正确地将多余的类型保留为单个，而不尝试无效的自配对。
