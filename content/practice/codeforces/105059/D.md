 ## 解决方案
 ## 问题理解

 我们的学期分为几天和一系列作业。 每项作业仅在固定的天数间隔内可用，从开始日到截止日，并且可以在该间隔内的任何一天完成。 然而，你愿意做的工作量受到极大限制：每天最多完成一项任务。 

目标不是直接最大化已完成的作业，而是最小化未能提交的作业数量。 还有一个额外的变化：教授允许你“原谅”最多 k 个错过的作业，这意味着只有 k 之外剩余的错过的作业实际上会损害你的分数。 

因此，如果你完成 X 项作业，那么 n − X 就是错过的作业数，而你的最终惩罚是 max(0, n − X − k)。 由于 k 是固定的，核心任务是在每天一项作业的约束下最大化您可以完成的作业数量，同时尊重可用性窗口。 

该结构是一个经典的调度问题：每个作业都是一个具有时间窗口的单位长度作业，每天最多可以托管一个作业。 

约束很大：每个测试最多 2 × 10^5 分配，并且测试之间的总输入大小也以 2 × 10^5 为界。 这排除了任何二次方的情况，例如对照每天检查每个分配或重复扫描活动间隔。 任何解决方案每次测试都必须大致为 O(n log n) 或 O(n)。 

当许多作业共享相同的狭窄间隔时，就会出现微妙的边缘情况。 例如，如果两个赋值都具有区间[1,1]，则即使k很大，也只能完成一个，而另一个也不可避免地会丢失。 另一个特殊情况是间隔很宽但很分散； 不考虑截止日期而贪婪地选择“最早开始”可能会浪费早期时间并阻碍以后的可行任务。 

## 方法

 蛮力的想法很简单：每天进行模拟，并每天选择尚未完成的任何可用任务。 对于每一天，我们都会扫描所有作业以查找间隔包含该天的作业，然后选择一个。 这是正确的，因为它尊重两个约束：每天最多一个并且仅选择有效的作业。 然而，对于 d 天的每一天，我们可能会检查最多 n 个分配，从而产生 O(nd) 行为，当 n 和 d 都很大时，这太慢了。 

关键的观察结果是，我们实际上不需要以天为主轴进行思考。 相反，我们可以将作业视为随着时间的推移而变得“合格”的事件。 一旦到达 t 天，所有 Si ≤ t 的作业都会成为候选作业，其中，我们希望避免在即将到期的作业上浪费机会。 

这导致了一种贪婪的调度策略：按顺序处理天，维护所有当前可用分配的结构，并始终选择截止日期最早的分配。 这是每个时间步单位容量下的经典“具有发布时间和截止日期的间隔调度”。 直觉是，尽早选择一个长期限的作业可以阻止稍后期限紧迫的作业，而反之则是安全的。 

为了避免明确地迭代所有日期，我们通过扫描时间并维护活动间隔的优先级结构来仅模拟相关日期。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 每天暴力模拟扫描所有作业 | O(n·d) | O(n) | 太慢了|
 | 贪婪的最小堆超过截止日期 | O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 我们将每个作业视为在开始日进入系统并在结束日后离开。 我们按照递增的顺序浏览日子，但只有当事件发生时我们才会明确地前进。

1. 按开始日期对所有作业进行排序。 这使我们能够按时间顺序激活它们，而无需重复扫描完整列表。 
2. 维护一个指向已排序作业的指针和一个按结束日期键入的最小堆。 堆代表当前可用但尚未完成的所有分配。 
3. 迭代从 1 到 d 的天数。 在处理 t 天之前，将开始日期 ≤ t 的所有分配插入到堆中。 这确保堆始终包含 t 天的可行分配。 
4. 从堆中删除结束日期 < t 的所有作业，因为它们无法再完成。 此步骤可防止无效选择持续存在。 
5. 如果堆非空，则将日期 t 分配给结束日期最小的作业。 这种贪婪的选择确保我们首先处理最紧急的任务。 
6. 继续，直到处理完所有天数或完成所有作业。 已完成分配的数量是用于分配的成功堆弹出的总数。 
7. 通过计算 n − 已完成，将完成计数转换为惩罚，然后减去 k 宽恕并钳位为零。 

为什么它有效

 堆总是优先考虑当前所有可行的分配中最先到期的分配。 假设我们曾经选择过一份截止日期较晚的工作，而同时存在更严格的截止日期。 这项紧张的工作仍然只能安排在剩余几天的子集上，因此跳过它可能会带来以后无法修复的永久性损失。 通过始终消耗最早截止日期的作业，我们为所有其他作业保留了灵活性，这正是保证单位容量间隔任务的最佳调度的贪婪交换论点。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

import heapq

def solve():
    t = int(input())
    out = []

    for _ in range(t):
        n, d, k = map(int, input().split())
        jobs = []
        for _ in range(n):
            s, e = map(int, input().split())
            jobs.append((s, e))

        jobs.sort()
        i = 0
        heap = []
        done = 0

        for day in range(1, d + 1):
            while i < n and jobs[i][0] <= day:
                heapq.heappush(heap, jobs[i][1])
                i += 1

            while heap and heap[0] < day:
                heapq.heappop(heap)

            if heap:
                heapq.heappop(heap)
                done += 1

        missed = n - done
        out.append(str(max(0, missed - k)))

    print("\n".join(out))

if __name__ == "__main__":
    solve()
```代码首先按开始日期对作业进行排序，以便我们可以随着扫描的进行逐步激活它们。 堆仅存储结束日，因为插入时已经满足开始约束。 

清理循环会删除过期的分配，即使许多作业随着时间的推移变得无效，也能确保正确性。 贪婪的弹出总是对应于当天安排任务。 

最后，我们使用宽恕参数 k 计算惩罚。

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 | O(n log n) | O(n log n) | 每个作业在堆中插入一次，最多移除一次，每次操作对数 |
 | 空间| O(n) | 堆和作业存储最多保存所有活动分配 |

 这些约束允许最多 2 × 10^5 的总分配，因此 O(n log n) 解决方案可以轻松满足时间和内存限制。

 ## 边缘情况

 一种极端情况是多个作业共享同一个有效日。 堆将包含多个相同的结束日，但每天只能弹出一个。 该算法自然地将剩余的保留在堆中，直到它们过期，此时它们被丢弃。 例如：```
n=2, d=1, k=0
(1,1), (1,1)
```在第一天，两个作业都被插入。 堆包含[1,1]。 一项已选定，一项未完成。 第 2 天不存在，因此剩余的工作被视为错过。 输出为 2，符合不可避免的约束。 

当间隔很宽但开始较晚时会出现另一种边缘情况。 该算法正确地将插入延迟到其开始日期，确保它们不会过早地被考虑。 例如：```
n=1, d=5, k=0
(5,5)
```在第 5 天之前没有可用的工作，因此仅在正确的时间考虑。 然后立即安排。
