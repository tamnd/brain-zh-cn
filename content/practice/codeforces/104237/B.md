 ## 解决方案
 ## 问题理解

 我们得到了在平面上排列的无限直线的集合。 每条线要么是垂直的，代表具有固定 x 坐标的大道，要么是水平的，代表具有固定 y 坐标的街道。 每条垂直线与每条水平线恰好相交一次，并且每次这样的交叉都算作一个交点。 

任务是计算所有给定直线之间存在多少个这样的交点。 

输入大小最多为 1000 行。 这立即表明二次方法是可以接受的，因为检查所有线对的解决方案最多执行大约 10^6 次操作，这完全在 Python 的限制范围内。 

一些极端情况在这里很重要。 如果所有线都是垂直的，则没有水平线相交，因此答案必须为零。 如果所有线都是水平的，则同样成立。 另一种微妙的情况是只有一条垂直线或一条水平线。 在这种情况下，答案等于相反组的大小，并且尝试将相同类型的线配对的错误实现将错误地产生额外计数。 

一个天真的但常见的错误是通过比较每对线并在坐标不同时递增来计算交叉点，这在逻辑上与问题的几何形状无关。 相交仅发生在垂直线之间，而不发生在任意对之间。 

## 方法

 蛮力的想法是考虑每对线并检查它们是否相交。 然而，大多数对是不相关的：两条垂直线永远不会彼此相交，两条水平线永远不会彼此相交。 唯一有意义的相互作用发生在垂直线和水平线之间。 

如果我们将输入分为两组：垂直线和水平线，问题就会大大简化。 x = a 处的每条垂直线与 y = b 处的每条水平线恰好相交一次。 这意味着每一对恰好贡献一个交集。 

因此，我们不需要检查 N 行中的所有对，而只需计算有多少行是垂直的，有多少行是水平的。 最终的答案就是他们的产品。 

暴力方法仍然会迭代所有线对和检查类型，从而导致 O(N²) 检查。 优化版本将问题简化为以 O(N) 计算频率。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(N²) | O(1) | O(1) | 太慢但没必要 |
 | 最佳 | O(N) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们将几何问题转化为计算两类线。 

1.读取行数N并初始化两个计数器，一个用于垂直行，一个用于水平行。 这种分离至关重要，因为交叉只发生在类别之间。 
2. 对于每一行，读取其类型。 如果是垂直的，则增加垂直计数器。 如果是水平的，则增加水平计数器。 实际坐标值无关紧要，因为无论位置如何，任何垂直线都会与每条水平线相交。 
3. 处理完所有线后，计算交叉点的数量：vertical_count 乘以horizo​​ntal_count。 这直接源于以下事实：每条垂直线与每条水平线形成交叉。 
4. 输出结果。 

### 为什么它有效

每条垂直线均由固定的 x 坐标定义，并在两个方向上无限延伸。 每条水平线均由固定的 y 坐标定义。 对于任何选定的由一条垂直线和一条水平线组成的线对，它们的交点是保证且唯一的，因为它们定义了单个点 (x, y)。 由于没有重复的线，因此每一对都只贡献一个交点，并且其他类型的线对没有任何贡献。 这在交点与垂直线和水平线的有序对之间建立了一对一的对应关系。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    v = 0
    h = 0
    
    for _ in range(n):
        t, x = input().split()
        if t == 'v':
            v += 1
        else:
            h += 1
    
    print(v * h)

if __name__ == "__main__":
    solve()
```该解决方案在扫描输入一次时维护两个简单的计数器。 读取坐标值但有意忽略，因为它们不影响是否存在交叉点； 只有方向很重要。 

最后的乘法对两组线之间的笛卡尔积进行编码。 一个常见的实现错误是尝试存储坐标并检查相等性或接近性，这是不必要的，并且会错误地使逻辑复杂化。

 ## 边缘情况

 当所有行都是垂直的时，算法读取每行并仅递增垂直计数器。 水平计数器保持为零，因此乘积为零。 例如：

 输入：```
2
v 1
v 2
```执行设置 v = 2，h = 0，给出 0 个交点，这与平行垂直线永远不会相交的事实相匹配。 

当只有一条垂直线和多条水平线时，请说：```
3
v 5
h 1
h 2
```计数器变为 v = 1 且 h = 2。结果为 2，对应于单条垂直线与两条水平线各恰好相交一次。
