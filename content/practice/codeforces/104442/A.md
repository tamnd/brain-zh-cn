 ## 解决方案
 ## 问题理解

 我们得到了一个权重数组，我们必须将这些值准确地分成$k$非空组，其中每个权重恰好属于一个组。 对于任何组，其成本定义为该组内最大值和最小值之间的差。 分区的总成本是这些组成本的总和。 对于每一个$k$从$1$到$n$，我们必须计算最小可能的总成本。 

因此，真正的决定不是关于元素的排序，而是关于我们如何选择连续或不连续的子集来最小化组内范围。 当一个群体的元素价值接近时，它就会变得廉价。 问题是当我们被迫增加组的数量时，最佳可实现的成本如何变化。 

输入尺寸很大：最多$10^5$每个测试的元素以及测试的总和也高达$10^5$。 这会立即排除每次测试的任何二次或尝试所有分区的任何情况。 即使尝试评估子集上的所有分割或 DP 的解决方案也太慢了。 我们需要一些接近排序的东西加上线性或线性算术处理。 

一个微妙的问题是分组不需要按原始顺序连续。 这意味着除非我们首先进行排序，否则关于“将数组分割成段”的天真的直觉会产生误导，因为只有按值的相对顺序对成本定义很重要。 

一个常见的失败案例来自于没有排序的贪婪分组。 例如，使用值$[1, 100, 2, 99]$，对连续元素进行分组给出了无意义的范围，而最佳分组取决于排序结构$[1,2,99,100]$。 

## 方法

 暴力解决方案会尝试所有方法将数组划分为$k$分组并计算每个分区的成本。 即使对于固定$k$，分区的数量是指数的，本质上是第二类斯特林数。 每次评估费用$O(n)$，所以这种方法立即爆炸超出非常小的范围$n$。 

关键的观察结果是，排序后，最佳组的结构与排序顺序对齐。 在任何组内，只有最小值和最大值很重要，因此组内元素的任何交错都无济于事。 如果我们对数组进行排序，则每个组都可以被认为是选择一些元素，但成本仅取决于极端情况。 关键的见解是，按排序顺序合并两个相邻组会产生与边界元素之间的间隙完全相同的成本增加。 

更具体地说，如果我们对数组进行排序，请考虑从其自己组中的每个元素开始。 成本为零。 现在，如果我们合并两个原本是独立单例的相邻组，则成本会因这两个值之间的差异而增加。 这将问题转变为选择我们“激活”哪些相邻间隙作为组之间的剪切。 

因此，我们将问题简化为选择$k-1$之间的削减$n-1$已排序数组中的间隙。 每次削减都会消除潜在的合并成本； 同样，我们保留最大的$k-1$间隙作为分隔符，或者从双重角度来看，我们首先进行最小的合并。 

这导致了一个经典的结构：我们计算排序数组中的所有相邻差异，对它们进行排序，并使用前缀和来构造所有差异的答案$k$。

 | 最佳 | O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 我们将问题转化为对排序值及其相邻间隙进行操作。 

1. 对数组进行非降序排序。 这确保了任何最佳组都可以根据该排序序列中的连续段进行推理，因为在组内混合远距离值只会不必要地增加其范围。 
2. 计算连续元素之间的差异。 这些表示将相邻值“连接”到同一组中的成本。 当我们减少组的数量时，每个差异都是对最终成本的候选贡献。 
3. 观察到，如果每个元素都作为自己的组开始，则总成本为零。 每次我们按排序顺序合并两个相邻组时，都会产生等于它们之间差距的成本。 这将问题转换为选择合并。 
4. 准确地形成$k$组，我们正好需要$n-k$合并。 每次合并对应于获取一个相邻间隙成本。 
5. 由于我们想要最小化总成本，因此我们首先选择最小的可用合并成本。 因此，对间隙数组进行排序。 
6. 在已排序的间隙上构建前缀和。 答案为$k$组是最小组的总和$n-k$差距。 

它起作用的原因来自于结构不变量：按照排序顺序，任何分区都可以表示为相邻元素之间的一组切割。 每次切割对两个元素是否属于同一组都有独立的贡献，并且连接两个段所贡献的成本仅取决于边界间隙。 由于这些贡献是独立且相加的，因此最优性减少为选择最小的间隙进行合并，从而贪婪地最小化每一步的累积成本。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    out = []
    
    for _ in range(t):
        n = int(input())
        a = list(map(int, input().split()))
        
        if n == 1:
            out.append("0")
            continue
        
        a.sort()
        
        gaps = []
        for i in range(n - 1):
            gaps.append(a[i + 1] - a[i])
        
        gaps.sort()
        
        prefix = [0] * (n)
        for i in range(n - 1):
            prefix[i + 1] = prefix[i] + gaps[i]
        
        res = []
        for k in range(1, n + 1):
            merges = n - k
            res.append(str(prefix[merges]))
        
        out.append(" ".join(res))
    
    print("\n".join(out))

if __name__ == "__main__":
    solve()
```该代码首先对数组进行排序，使所有相关结构变成线性。 然后，它构造间隙数组，该数组对相邻元素之间所有可能的成本增量进行编码。 

对间隙进行排序是关键的贪婪步骤：它确保我们在减少组数时始终首先选择最便宜的合并。 前缀数组允许恒定时间查询最小的总和$x$差距，直接映射到拥有的成本$n-x$合并，因此$k$组。 

之间的映射$k$合并数量是主要索引细节：$k$团体意味着$n-k$元素之间的联系。 

## 工作示例

 考虑输入：```
n = 4
a = [5, 2, 3, 7]
```排序后：$[2, 3, 5, 7]$差距：$$[1, 2, 2]$$排序后的间隙仍然存在：$[1, 2, 2]$前缀和：$$[0, 1, 3, 5]$$我们现在计算答案。 

| k | 合并 = n-k | 成本（前缀[合并]）|
 | --- | --- | --- |
 | 1 | 3 | 5 |
 | 2 | 2 | 3 |
 | 3 | 1 | 1 |
 | 4 | 0 | 0 |

 这表明增加组的数量如何消除首先支付更大间隙的需要，只留下较小的内部合并。 

现在考虑：```
n = 5
a = [1, 100, 2, 3, 200]
```排序：$[1,2,3,100,200]$差距：$[1,1,97,100]$已排序的间隙：$[1,1,97,100]$前缀：$[0,1,2,99,199]$| k | 合并| 成本|
 | --- | --- | --- |
 | 1 | 4 | 199 | 199
 | 2 | 3 | 99 | 99
 | 3 | 2 | 2 |
 | 4 | 1 | 1 |
 | 5 | 0 | 0 |

 此示例显示了当允许更多组时如何首先避免大间隙。

 ## 边缘情况

 对于所有值都相同的输入，例如：```
1
5
10 10 10 10 10
```排序不会产生间隙，因此间隙数组全为零。 每个前缀和都为零，因此所有答案都为零。 该算法自然地处理这个问题，因为每个合并成本都是零，所以任何分区都是最优的。 

对于严格递增的序列，例如：```
1
4
1 2 3 4
```差距是$[1,1,1]$。 随着组的减少，前缀总和会增加成本。 该算法正确地将每个间隙解释为独立的合并成本，并按所需合并的递增顺序累积它们。 

对于具有一个主要差距的情况：```
1
5
1 2 3 100 101
```差距是$[1,1,97,1]$，排序为$[1,1,1,97]$。 仅当剩下的组很少时才使用大间隙，因为它总是在早期合并中被跳过。 这演示了始终首先获取最小可用连接成本的贪婪原则。
