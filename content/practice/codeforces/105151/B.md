 ## 问题理解

 我们得到一组雨滴，每个雨滴落在水平线上的一个点上。 每个液滴都会出现在特定的坐标处，并且仅在其自己的下落时间之后才开始膨胀。 一旦液滴落地，它就会以恒定速率对称膨胀：$k$着陆后几秒，它覆盖了从其中心开始的间隔$x_i$按距离向两个方向$k$。 因此，每个液滴形成一个不断增长的间隔，其半径随时间线性增加，但仅在其激活时间之后。 

任务是确定最早的整数矩$T$使得所有这些扩展区间的并集完全覆盖固定段$[0, r]$。 覆盖意味着里面的每个点$[0, r]$一次至少包含一个液滴的扩展区间$T$。 

约束条件很大，最多可达$2 \cdot 10^6$液滴和坐标高达$10^9$。 这立即排除了任何逐步模拟时间或独立地为每个候选时间重新计算覆盖范围的方法。 任何解决方案都必须避免每次检查时重复遍历所有液滴。 我们也无力承担$O(n \log n)$每次在朴素的二分搜索中进行评估，无需仔细的线性合并，因为每次检查都必须接近线性时间。 

问题的一个微妙方面是，液滴不会立即发挥作用。 他们只有在各自的跌倒时间之后才会变得活跃。 这会创建一组与时间相关的间隔，如果假设所有液滴从零时间开始都处于活动状态，则很容易错误处理。 

一些边缘案例暴露了常见的错误。 如果所有液滴在候选时间后落下，则即使它们的位置很密集，覆盖范围也是空的。 例如，$n=2, r=10$，液滴在$x=0, x=10$，都与$t_i=100$。 在$T=50$，不存在覆盖范围，即使空间上一切都是完美的，答案也不可能是 50。 

当液滴正好在某个时间落下时，会发生另一种失败情况$T$。 其当时的半径$T$为零，因此它仅贡献一个点。 将此误解为提供非零间隔会导致错误的提前终止。 

最后，如果假设固定顺序而不考虑基于时间的激活，则会发生贪婪合并失败。 活动间隔集随着变化$T$，因此必须始终相对于固定候选时间来评估覆盖范围。 

## 方法

 一种直接的方法是模拟时间$0$向上。 在每个整数时间$T$，我们计算所有已经落下的水滴，并且对于每个这样的水滴，我们计算其间隔$[x_i - (T - t_i), x_i + (T - t_i)]$。 然后我们合并这些区间并检查它们是否覆盖$[0, r]$。 这是正确的，因为它直接遵循问题定义。 

问题是性能。 在最坏的情况下，我们可能需要检查$10^9$候选次数，每项检查最多需要处理$2 \cdot 10^6$间隔。 这远远超出了任何可行的计算。 

关键的观察结果是覆盖属性在时间上是单调的。 一旦给定时间$T$足以覆盖$[0, r]$, 任何更大的时间$T'$也会起作用，因为每个液滴只会进一步膨胀。 这允许二分搜索$T$。 

针对给定的每个可行性检查$T$可以在$O(n)$通过仅针对具有以下条件的液滴构建有效间隔$t_i \le T$，对它们进行排序或清理，然后合并。 然而，对每张支票进行排序会太慢。 相反，我们按位置对液滴进行一次分类，或以避免重复分类的方式对其进行处理。 最有效的公式是配对$x_i$和$t_i$，并且对于固定的$T$，动态计算间隔并在按左端点排序后执行线性扫描。 

因此解决方案变成二分搜索$T$，每次检查都执行线性变换和线性合并。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力模拟|$O(T \cdot n)$|$O(n)$| 太慢了 |
 | 二分查找+扫描|$O(n \log R)$|$O(n)$| 已接受 |

 ## 算法演练

 1. 仅在需要稳定处理时才按位置对液滴进行排序，但更重要的是，通过基于固定的计算覆盖间隔来独立地处理每个检查$T$。 全局排序本质上不是必需的，因为合并是根据检查完成的。 
2. 定义函数`can(T)`决定是否所有点$[0, r]$有时被覆盖$T$。 
3. 内部`can(T)`，迭代所有液滴。 对于每个液滴$t_i \le T$，计算其半径$d = T - t_i$，因此它的区间$[x_i - d, x_i + d]$。 忽略水滴$t_i > T$因为他们还没有出现。 
4. 收集所有这样的区间，并按左端点升序合并它们。 合并时，保持迄今为止覆盖的最右边的点。 如果在任何时候当前合并的覆盖范围与下一个间隔之间存在间隙，则覆盖范围会立即失败。 
5、合并所有区间后，检查最终覆盖点是否至少达到$r$。 如果是，$T$就足够了。 
6. 二分查找结束$T$在范围内$[0, \max t_i + r]$。 上限来自最坏的情况，其中最右边的覆盖范围必须从远处的液滴传播。 
7. 对于二分查找中的每个中间值，调用`can(mid)`并相应地调整界限以找到最小有效值$T$。 

它之所以有效是基于覆盖范围的单调性。 一旦间隔配置覆盖$[0, r]$在某个时间$T$，增加$T$仅扩大所有活动间隔或激活更多液滴。 这意味着有效时间的集合形成了整数时间线的后缀，这正是二分查找正确性所需的条件。 合并步骤确保我们正确考虑重叠扩展并精确检测间隙，而不会高估覆盖范围。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def can(T, x, t, r):
    intervals = []
    for i in range(len(x)):
        if t[i] <= T:
            d = T - t[i]
            l = x[i] - d
            rgt = x[i] + d
            intervals.append((l, rgt))

    if not intervals:
        return False

    intervals.sort()
    
    cur = 0
    for l, rgt in intervals:
        if l > cur:
            return False
        if rgt > cur:
            cur = rgt
        if cur >= r:
            return True
    return cur >= r

def solve():
    n, R = map(int, input().split())
    x = list(map(int, input().split()))
    t = list(map(int, input().split()))

    lo, hi = 0, R + max(t)
    ans = hi

    while lo <= hi:
        mid = (lo + hi) // 2
        if can(mid, x, t, R):
            ans = mid
            hi = mid - 1
        else:
            lo = mid + 1

    print(ans)

if __name__ == "__main__":
    solve()
```实现的核心是`can`函数，在固定时间重建有效覆盖范围。 仅当每个液滴处于活动状态时，才会将其转换为间隔。 区间展开直接根据时间差计算，这确保了几何解释的正确性。 

合并过程维护单个变量`cur`，它跟踪迄今为止最右边的覆盖点。 之后开始的任何间隔`cur`发出间隙信号，这会立即使候选时间失效。 

二分查找利用了答案的上界为$R + \max(t_i)$，因为在最坏的情况下，位置 0 处的液滴必须扩展至一定距离$R$当它变得活跃后。

 ## 边缘情况

 关键的边缘情况是在候选时间没有液滴处于活动状态。 例如，$T=0$与所有$t_i > 0$产生一个空的区间集。 该算法立即正确返回 false，因为不可能进行覆盖。 

另一种情况是覆盖几乎完成但在单个间隙处失败。 例如，扩展后的间隔可能会产生$[0, 2]$和$[3, 5]$间隔为 2 到 3。合并步骤检测到这一点，因为下一个间隔在`cur`，并且该函数返回 false 而不继续。 

最后一个微妙的情况是液滴恰好在候选时间激活。 由于它的半径为零，因此它的间隔是一个点。 该算法仍然包含它，但它无法弥合间隙，除非它恰好位于现有覆盖范围的边界，这是由合并条件正确处理的`l > cur`。
