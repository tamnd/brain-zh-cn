 ## 解决方案
 ## 问题理解

 我们有偶数个学生，具体来说是 2n 个，每个学生都具有数字化学技能。 老师必须将他们分成 n 个不相交的对，因此每个学生恰好属于一对。 只有当两个技能的绝对差值最多为A时，一对才有效，否则实验室会爆炸，整个分区都无效。 

在有效对中，我们通过第二个阈值 B 对质量进行分类。如果差异最多为 B，则该对产生完美结果；否则，该对将产生完美结果。 如果介于 B 和 A 之间，则结果还算可以接受。 目标是确定是否存在任何有效的配对，如果存在，则最大限度地增加有多少对属于“完美”类别。 

输入由多个独立的测试用例组成，每个测试用例描述一组技能和两个阈值。 对于每个测试用例，我们要么报告不可能性，要么计算在有效配对约束下可实现的完美配对的最大数量。 

约束足够大，以至于整个集合上的任何二次配对策略都太慢。 由于测试总数为 2n 至 2⋅10^5，即使每次测试的 O(n^2) 构造也是完全不可行的。 这立即暗示了基于排序的贪婪结构或两指针结构。 

当天真的贪婪尝试总是将最近的邻居配对而不尊重全局 A 约束时，就会出现一个关键的微妙失败案例。 例如，考虑一个非常小的集群结构，其中局部配对迫使不可能的剩余。 当一个人试图首先最大化完美对而不确保 A 下的可行性时，就会发生另一种失败，这可能会产生稍后无法完成有效对的配置。 

挑战在于平衡两个相互竞争的目标：A 下的可行性和 B 下的最大化计数。 

## 方法

 暴力破解的想法很简单：尝试所有方法将 2n 个元素分成 n 对，检查每对是否满足 A 约束，并计算有多少个元素满足 B 约束。 这探索了大小为 (2n−1)!! 的组合空间，其增长速度快于指数。 即使对于 n = 10，这也已经是巨大的了，所以它甚至无法远程使用。 

一旦我们对数组进行排序，问题的结构就完全改变了。 排序后，A下的可行性变成了一条线上的配对问题：如果两个元素在排序顺序上相距很远，用更接近的匹配替换它们只会有助于可行性。 这暗示最佳解决方案将始终将排序顺序接近的元素配对，并且任何最佳构造都可以转换为尊重排序约束的构造。 

关键的见解是将问题分为两个阶段。 首先，我们确定阈值 A 下是否存在有效配对。这是一个经典的“排序后相邻配对”可行性检查：如果我们对数组进行排序，当且仅当我们可以按顺序配对而不违反 A 时，才存在有效配对，这减少为使用两个指针的贪婪扫描。 

一旦可行性得到保证，我们希望最大化差值 ≤ B 的对。我们可以将 A 下的有效配对视为约束匹配，其中边仅存在于足够接近的元素之间。 其中，我们希望最大化有多少条边落入更严格的阈值 B。该结构成为具有两个距离范围的直线上的贪婪匹配问题。 我们可以首先贪婪地尝试形成 B-好对，但只能以不破坏剩余元素的 A-可行性的方式。 排序结构允许我们维护两个指针，并始终匹配最早的可能元素，同时遵守约束。 

最终的解决方案最终是对排序数组进行贪婪扫描，尽可能形成 B 对，否则我们保留元素以确保 A 可行配对。

 ## 算法演练

 我们按非降序对所有学生技能进行排序。 从这一点来看，我们只对相邻结构进行推理。 

我们在数组上维护一个指针并按顺序构造对，始终获取下一个可用元素。 

1. 对大小为 2n 的数组进行排序。 
2. 从左到右扫描并尝试以不违反 A 约束的方式贪婪地对元素进行配对。 

如果在任何时候当前元素无法使用有效伙伴在距离 A 内配对，我们就断定不存在有效配对。 
3. 在此扫描期间，只要两个连续的可用元素差值≤B，我们就考虑立即将它们配对为“完美”对。 
4. 如果 A 内的下一个可能的配对只能通过更大的间隙才能实现，我们仍然形成该配对，但不认为它是完美的。 
5. 继续，直到所有元素都配对。 

关键的设计决策是我们从不延迟配对可用的 B 有效相邻对。 延迟这样的配对只会导致它稍后被更差的配对取代，因为所有剩余的候选者将在排序顺序中排得更远。 

### 为什么它有效

 排序将问题简化为一条线，其中邻近性完全决定可行性。 A 下的任何有效配对都必须遵守以下原则：如果一个元素与远处的某个元素配对，则所有中间元素必须在同一间隔内配对，这始终允许转换为本地配对的结构。 

在这个规范结构中，贪婪地选择 B 有效相邻对不会损害最优性，因为让两个接近的元素不成对只会增加它们最终必须跨越的距离，而不会提高以后形成另一个 B 对的机会。 A 约束确保我们永远不会跨越禁止的间隙，并且贪婪配对按间隔保留有效的匹配间隔。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    out = []
    
    for _ in range(t):
        n, A, B = map(int, input().split())
        arr = list(map(int, input().split()))
        arr.sort()
        
        i = 0
        ok = True
        perfect = 0
        
        # greedy pairing from left to right
        while i < 2 * n:
            if i == 2 * n - 1:
                ok = False
                break
            
            # if we can form a perfect pair, take it
            if arr[i + 1] - arr[i] <= B:
                if arr[i + 1] - arr[i] > A:
                    ok = False
                    break
                perfect += 1
                i += 2
            else:
                # otherwise we must still pair i and i+1 to maintain feasibility locally
                if arr[i + 1] - arr[i] > A:
                    ok = False
                    break
                i += 2
        
        if not ok:
            out.append("-1")
        else:
            out.append(str(perfect))
    
    print("\n".join(out))

if __name__ == "__main__":
    solve()
```代码首先对技能进行排序，以便所有候选的好对都成为本地技能。 指针`i`遍历数组，始终消耗成对的连续元素。 如果对于A来说差距太大，那么建造就不可能，我们就停止。 

决策点是当`arr[i+1] - arr[i] <= B`。 在这种情况下，我们将其视为完美配对，因为没有理由推迟它，而推迟只会将其推入稍后差距更大的配对。 

每次迭代恰好消耗两个元素，确保排序后的线性时间。 

一个微妙的点是我们从不尝试“跳过”元素。 在此结构中，跳过将破坏剩余元素必须在 A 内的连续段中仍可配对的不变性。

 ## 边缘情况

 一种重要的边缘情况是所有元素都相同。 排序后，每个相邻对的差值为零，即 ≤ B，因此每对都变得完美，并且算法正确地计算了所有 n 对。 

另一种情况是当存在单个大间隙而破坏可行性时。 排序后，如果任何相邻差异超过 A，则根本不可能进行配对，因为任何匹配结构都无法桥接该间隙。 当算法尝试跨越间隙配对并正确返回 -1 时，它会立即检测到这一点。 

当 B 非常小而 A 仅稍大一点时，就会出现一种微妙的情况。 该算法可能经常落入“有效但不完美”分支，但这并不影响正确性，因为完美对仅在局部最优时才被计数，并且不会强制跨越 A 边界。
