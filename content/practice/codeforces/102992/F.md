 ## 解决方案
 ## 问题理解

 我们正在模拟一种情况，即一个人随着时间的推移反复产生概率“烟花批次”。 每次制作尝试都需要固定的时间，并且每个独立制作的烟花“完美”的概率很小。 在生产出一定数量的烟花后，该人可以选择一次“引爆”所有积累的烟花，这会花费额外的固定时间，然后如果该批次中至少存在一个完美的烟花，他们要么成功，要么从头开始继续这个过程。 

决定的不仅仅是产生多少烟花，而是何时停止产生和引爆，因为引爆会重置尝试周期，但会消耗时间。 目标是最小化第一个成功批次发生之前的预期总时间。 

输入描述每个测试用例的三个参数。 第一个是产生单个烟花的时间，第二个是引爆当前存储的所有烟花的时间成本，第三个是决定每个烟花成功概率的概率标度。 输出是在最佳策略下第一次成功爆炸的最短预期时间。 

主要的结构困难在于，该过程是一个重复决策问题，每批次的成功概率都呈指数关系，而每批次的成功概率取决于爆炸前累积的烟花数量。 

从复杂性的角度来看，约束允许最多 10^4 个测试用例和最多 10^9 个大值。 这立即排除了任何模拟流程或以简单的方式迭代可能的大批量大小的策略。 任何解决方案都必须将每个测试用例简化为恒定时间公式或凸函数或单峰函数上的非常小的有界优化。 

一种微妙的失败模式出现在天真的贪婪推理中。 人们可能认为增加批量大小总是有帮助的，因为它增加了成功概率，但它也增加了生产时间的线性成本。 另一种不正确的方法是固定最大尝试次数并模拟预期值，但由于故障概率的指数衰减和大参数范围而导致预期值中断。 

具体的边缘情况是成功概率极小的情况，例如 p = 1，其中每个烟花几乎永远不会成功。 在这种情况下，尽管频繁重置，但每个周期仅产生一枚烟花可能是最佳选择。 另一方面，当 p = 10000 时，单个烟花就已经保证成功，无需进行任何批处理。 任何正确的解决方案都必须顺利地处理两个极端，而没有数值不稳定。 

## 方法

 蛮力策略是考虑每个可能的批量大小 k。 对于每个 k，我们将一个周期的预期时间计算为 k 乘以生产时间加爆炸时间，再乘以成功之前的预期周期数。 一批大小为 k 的成功概率是 1 减去所有 k 个烟花失败的概率，即归一化概率形式中的 1 减去 (1 − p)^k。 这使得预期周期数等于成功概率的倒数。 因此我们可以计算每个 k 的预期总时间并取最小值。 

这种蛮力的想法是正确的，但很快就太慢了，因为 k 可以任意大。 即使我们将 k 限制在一个很大的范围内，该函数也取决于幂运算，必须仔细评估。 更重要的是，最优 k 在任意意义上并不大，而是位于一个非常狭窄的范围内，其中成功概率的边际收益平衡了成本的边际增加。

关键的见解是认识到 k 上的预期成本函数是单峰的。 随着 k 的增加，成功概率呈指数增加，接近 1，而成本呈线性增加。 这将创建一个最小点。 因此，我们可以使用三元搜索或直接单调斜率推理来搜索最佳 k。 

这减少了在 O(log range) 甚至 O(1) 候选检查中评估函数的问题，因为当增加 k 的边际效益不再超过额外的生产时间时，就会出现最优结果。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力破解 k | O(每次测试的 maxK) | O(1) | O(1) | 太慢了 |
 | k 上的三元搜索 | 每次测试 O(log maxK) | O(1) | O(1) | 已接受 |

 ## 算法演练

 1. 将概率 p 转换为每个烟花的成功概率，将其缩放为分数，以便每个烟花独立成功的概率为 p / 10000。这是必要的，因为批量成功取决于重复的独立试验。 
2. 定义一个函数 cost(k)，用于计算每次爆炸前产生 k 个烟花的预期时间。 该函数首先将批次的失败概率计算为 (1 − p)^k，然后将其转换为批次的成功概率。 
3. 将预期周期长度计算为 k 乘以生产时间加上爆炸时间。 这表示检查批次是否成功之前所花费的确定性时间。 
4. 将批量成功概率转换为直到成功为止的预期循环次数，即成功概率的倒数。 将此乘以周期成本即可获得预期的总时间。 
5. 搜索可能的 k 值以最小化预期时间。 由于该函数是单峰的，因此在 k 的整数域上应用三元搜索。 在每个步骤中，比较成本（mid1）和成本（mid2）以确定哪一半包含最优值。 
6. 充分缩小范围后，直接评估所有剩余候选 k 值并取最小值。 

### 为什么它有效

 正确性依赖于预期时间函数的结构，即线性递增项和凸递减指数项之和。 对于大 k，线性项占主导地位，而对于小 k，指数项占主导地位。 这保证了单一的全局最小值。 三元搜索保留了正确性，因为在任何点，函数都不能具有多个不相交的最小值，因此丢弃一侧并不能消除最优解。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve_case(n, m, p):
    # convert probability
    prob = p / 10000.0

    # if probability is 1, one firework always succeeds
    if prob == 1.0:
        return n + m

    def expected(k):
        # probability all k fail
        fail = (1.0 - prob) ** k
        success = 1.0 - fail
        cycle = k * n + m
        return cycle / success

    lo, hi = 1, 200000  # safe upper bound
    for _ in range(60):
        m1 = lo + (hi - lo) // 3
        m2 = hi - (hi - lo) // 3
        if expected(m1) < expected(m2):
            hi = m2
        else:
            lo = m1

    ans = float('inf')
    for k in range(max(1, lo - 5), hi + 6):
        ans = min(ans, expected(k))

    return ans

def main():
    t = int(input())
    for _ in range(t):
        n, m, p = map(int, input().split())
        print(f"{solve_case(n, m, p):.10f}")

if __name__ == "__main__":
    main()
```该解决方案将决策隔离为单个整数参数 k，即每个周期的烟花数量。 功能`expected(k)`直接编码成本和成功概率，搜索阶段依赖于该函数的单峰性质。 最终的局部扫描是必要的，因为浮点三元搜索可能由于舍入效应而错过精确的整数最小值。 

概率计算中必须小心，因为重复取接近 1 的值可能会下溢； 然而，在给定的限制内，如果仔细实现，双精度就足够了。

 ## 边缘情况

 一个重要的边缘情况是当 p 等于 10000 时，这意味着每个烟花都是完美的。 在这种情况下，最优策略退化为产生单个烟花并立即引爆。 任何批处理更多的尝试只会线性增加成本，而不会增加概率，并且算法通过提前返回 n + m 来正确处理这个问题。 

另一种边缘情况是当 p 非常小时。 任何固定的 k 的成功概率变得很小，因此预期的周期数变得非常大。 三元搜索仍然有效，因为成本函数变得以指数衰减为主，将最优值推向较小的 k 值。 

最后一个微妙的情况是当 k 很大时数值不稳定。 在该区域中，(1 − p)^k 可以下溢到零，使得浮点运算中的成功概率等于 1。 该算法通过将 k 限制在合理的范围并直接验证接近最佳值的候选值来避免这种情况，从而在浮点限制的情况下确保正确性。
