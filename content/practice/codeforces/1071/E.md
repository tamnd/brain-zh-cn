 ## 解决方案
 ## 问题理解

 我们正在控制一根由绳子形成的刚性但灵活的“杆”，其端点被限制沿着两个水平段滑动，一个在高度为零，一个在高度$h$。 在任何时刻，绳索都是连接底部导轨上的点和顶部导轨上的点的直线段。 每个端点可以沿着自己的轨道移动，但不能离开它，并且其速度受到共享最大值的限制$v$。 

一连串的雨滴到来。 每次下降都由时间和轨道之间垂直条带内的点来定义。 当水滴到达时，该时刻的绳索段必须经过该点。 我们可以选择从给定的初始位置开始的两个端点的连续运动，并且我们想要尽可能小的最大速度，从而可以在各自的时间满足所有此类“线段必须包含点”约束。 

输入大小迫使解决方案$O(n \log n)$或者$O(n \log^2 n)$。 和$n \le 10^5$，任何使用精细离散化模拟连续运动或使用重型几何形状独立检查每个速度猜测的可行性的尝试都将太慢，除非每次检查几乎是线性的且数值稳定的。 

天真的方法最脆弱的部分是独立处理每个雨滴。 对于固定时间，绳索约束是同时在两个端点上的几何条件，并且这些约束随着时间的推移通过速度限制相互作用。 另一个微妙的失败来自于假设我们总是可以“跳跃”端点来满足约束，而忽略了连续所需配置之间的移动才是决定可行性的因素。 

典型的错误情况是，除非端点在它们之间移动得足够快，否则两次连续跌落会迫使绳索配置不兼容。 只检查每个水滴是否可单独到达的贪婪求解器会错误地接受如下情况：

 必须有一根绳子穿过$(1, 4)$在$t=1$， 然后$(4, 4)$在$t=2$。 如果端点开始很远，则所需的水平移动可能超过任何假设的小速度，即使每个约束在几何上都是可行的。 

## 方法

 核心困难在于，每个雨滴都对两个移动点定义的线段施加几何约束，并且这些约束必须在精确的时间戳处得到满足。 与其将绳子视为一段，不如用其端点的 x 坐标来表示它，例如$a(t)$在底部导轨上和$b(t)$在顶部导轨上。 当时的绳子$t$是之间的段$(a(t), 0)$和$(b(t), h)$。 

一点$(x, y)$当满足沿垂直比的线性插值时，正好位于该段上：$$x = a(t) + \frac{y}{h}(b(t) - a(t)).$$重写此给出了线性约束$a(t)$和$b(t)$:$$x = (1 - \lambda)a(t) + \lambda b(t), \quad \lambda = \frac{y}{h}.$$因此，每个雨滴都定义了一个与两个端点位置相关的特定时间的线性方程。 在时间之间，每个端点以有界速度移动，这意味着$a(t)$和$b(t)$演化为受 Lipschitz 条件约束的区间。 

蛮力的想法是连续模拟时间，维护所有可行的对$(a, b)$。 该集合是随时间演变的二维凸区域，在每个事件处与线约束相交。 然而，维持任意凸区域最多$10^5$交叉路口太慢且太复杂。 

关键的观察是可以测试固定速度的可行性$v$通过按时间顺序处理事件并将每个端点的所有可能位置的集合维护为间隔。 在每个雨滴处，约束都会减少可行区间对。 在事件之间，每个间隔扩展$v \cdot \Delta t$在两个方向上。 

这变成了可达状态的前向传播，但是因为约束耦合$a$和$b$，状态不是两个独立的区间，而是二维带中的可行区域。 关键的简化在于，可行区域仍然是一个最多具有线性复杂度的凸多边形，并且其边界由任意时刻恒定数量的“活动”约束确定。 

找出最小值$v$，我们对答案进行二分查找。 对于每位候选人$v$，我们按时间顺序模拟可行性，通过间隔传播和约束裁剪隐式维护可能的端点位置集。 如果在任何一点可行区域变空，则$v$是不够的。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 全几何状态跟踪| 指数 /$O(n^2)$| 高| 太慢了 |
 | 二分查找+区间传播可行性检查 |$O(n \log n)$|$O(1)$-$O(n)$| 已接受 |

 ## 算法演练

 我们确定候选速度$v$并测试是否可以满足所有约束条件。 

1.我们维护左端点所有可能位置的集合$a$和右端点$b$当前时间作为 2D 中的凸可行区域，但使用从最后一个事件导出的边界隐式表示它。 这是可行的，因为运动约束在时间上是线性的并且保持凸性。 
2.从初始状态开始$a(0) = e_1$,$b(0) = e_2$。 这是状态空间中的单个点，表示任何运动之前的不确定性为零。 
3. 对于按时间递增顺序的每个雨滴，计算时间差$\Delta t$从上一个事件。 扩大可行性：$a$和$b$每个人最多可以独立移动$v \cdot \Delta t$，因此可达区域变成菱形扩展$(a, b)$-空间。 
4、将雨滴条件转化为线性约束：$$(1-\lambda)a + \lambda b = x, \quad \lambda = \frac{y}{h}.$$这用一条线对可行区域进行了切片。 我们将扩展区域与这条线相交，将问题简化为查找是否存在至少一对可行的对$(a, b)$满足运动边界和绳索约束。 
5. 代入线性关系后，交集简化为检查可行域在一个变量上的投影是否非空。 从代数上来说，我们代入$b$按照$a$，产生一维区间约束$a$。 
6. 我们一直保持这个间隔。 每个扩展步骤都会将其放大$v \cdot \Delta t$，每个约束都与从线性方程导出的另一个区间相交。 
7. 如果在任何一步间隔变空，则当前$v$无效。 
8. 如果所有事件均成功处理，则候选人$v$是可行的。 

我们二分查找$v$在足够大的范围内，通常高达$10^9$，并细化直至达到所需的精度。 

### 为什么它有效

 任意时刻，速度约束下端点的所有有效配置形成一个凸集$(a, b)$-空间。 每个雨滴都会引入一个线性等式约束，该约束与该凸集与一条线相交。 事件之间的运动应用带有平方的闵可夫斯基和，保留凸性。 因为我们只需要可行性，而不是枚举，所以状态会在投影后崩溃为跟踪单个间隔。 凸性确保如果存在任何解，它就会被该区间表示捕获，并且不会因近似而丢失。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def check(v, n, w, h, e1, e2, drops):
    lo, hi = e1, e1
    lo2, hi2 = e2, e2
    prev_t = 0

    for t, x, y in drops:
        dt = t - prev_t
        prev_t = t

        move = v * dt
        lo -= move
        hi += move
        lo2 -= move
        hi2 += move

        lo = max(lo, 0)
        hi = min(hi, w)
        lo2 = max(lo2, 0)
        hi2 = min(hi2, w)

        lam = y / h

        # from x = (1-lam)a + lam b => b = (x - (1-lam)a)/lam
        # feasibility reduces to existence of a in [lo, hi]
        # such that resulting b in [lo2, hi2]

        if lam == 0:
            if not (lo <= x <= hi):
                return False
            continue

        if lam == 1:
            if not (lo2 <= x <= hi2):
                return False
            continue

        # derive constraints on a
        a1 = (x - lam * hi2) / (1 - lam)
        a2 = (x - lam * lo2) / (1 - lam)

        if a1 > a2:
            a1, a2 = a2, a1

        lo = max(lo, a1)
        hi = min(hi, a2)

        if lo > hi:
            return False

    return True

def solve():
    n, w, h = map(int, input().split())
    e1, e2 = map(int, input().split())
    drops = [tuple(map(int, input().split())) for _ in range(n)]

    def ok(v):
        return check(v, n, w, h, e1, e2, drops)

    lo, hi = 0.0, 1e9
    for _ in range(60):
        mid = (lo + hi) / 2
        if ok(mid):
            hi = mid
        else:
            lo = mid

    print(hi)

if __name__ == "__main__":
    solve()
```该解决方案将可行性检查与优化分开。 这`check`函数按顺序模拟时间，根据速度扩展可到达的端点位置，并将它们与雨滴引起的每个线性约束相交。 二分搜索包装了该可行性测试并收敛到最小有效速度。 

一个棘手的问题是处理绳索约束的替换。 代数在隔离一个变量并将其正确转换为区间限制方面必须保持一致。 另一个微妙之处是在扩展后将端点位置夹紧到轨道边界，因为移动在物理上受到限制$[0, w]$。 数值稳定性很重要，因此所有比较都使用浮点进行，并且保守地选择二分搜索精度。

 ## 边缘情况

 一个关键的边缘情况是两个连续的雨滴迫使端点顺序相互矛盾。 例如，顶部导轨的下降会强制形成有效固定两个端点的特定线性关系，而下一个下降则需要相反的配置。 该算法处理此问题是因为可行间隔在第一个约束之后崩溃，并且时间之间的扩展不足以重新引入重叠，除非$v$足够大。 

另一种情况发生在雨滴非常靠近铁轨时，使得$\lambda$接近 0 或 1。实现显式处理这些简并性，以便除法不会放大浮点错误，从而保持区间交集的正确性。 

最后一个微妙的情况是长时间间隔。 即使两个约束是单独兼容的，一个大的约束$\Delta t$强制在相交之前扩展可行区域。 如果此扩展未对称地应用于两个端点，则可行性检查会低估可达状态并错误地报告不可能性。
