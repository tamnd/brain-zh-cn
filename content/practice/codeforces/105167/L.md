 ## 解决方案
 ## 问题理解

 我们在 A 站得到了固定的列车序列。每辆列车都有一个从 1 到 n 的唯一标签，但它们以任意顺序出现。 移动汽车的唯一方法是通过三个定向操作：从 A 到中间站 B、从 B 到最终站 C、以及直接从 A 到 C。 

目标是最终 C 站中的所有汽车均按标签严格按升序排列，即首先是 1 辆车，然后是 2 辆车，依此类推，直至 n。 我们必须输出一系列允许的移动来实现这一点，并且在所有有效序列中，我们被要求生成一个具有最少操作数的序列。 如果不可能，我们必须输出-1。 

关键的限制是 A、B 和 C 在阻塞方面都像堆栈一样。 特别是，B 只有一条轨道，因此到达 B 的汽车受到后进先出行为的限制：较晚到达的汽车可能需要提前离开以畅通未来的移动。 

输入大小高达 n = 100000。任何试图模拟任意重新排列或探索移动排列的解决方案都是立即不可行的。 解决方案必须在 n 中是线性的或接近线性的，因为如果涉及大量常数因子，即使 O(n log n) 也是边界线，并且 O(n^2) 完全不可能。 

当结构在 B 中强制执行特定的中间排序且以后无法纠正时，就会出现微妙的失败情况。 

例如，考虑：```
3
2 1 3
```如果一个人天真地发送2到B，然后1到B，然后尝试直接发送3到C，系统就会陷入困境，因为1在B中阻塞了2，但稍后需要2才能在约束下正确处理3。 这说明在不考虑未来需求的情况下任意推入 B 可能会导致流程陷入僵局。 

另一个有问题的案例：```
2
2 1
```这是不可能的，因为 1 必须在 2 之前结束，但 2 首先出现，并根据动作卡在 B 或 C 中。 任何通过 B 发送所有内容的贪婪尝试仍然会失败。 

这些例子表明，可行性取决于我们是否可以避免逆转 B 的堆栈性质所施加的不可避免的顺序约束。 

## 方法

 强力解释会将其视为巨大状态空间中的最短路径问题。 每个状态由 A、B 和 C 中汽车的排序组成。每次移动都会在状态之间转换，我们搜索一个产生排序 C 的序列。即使表示一个状态在 n 中也是线性的，并且分支因子最多为 3，所以 BFS 是完全不可行的。 

更结构化的暴力方法可能会模拟贪婪策略：如果可能，总是将下一个所需的汽车 k 从 A 或 B 移动到 C 中，否则将阻塞的汽车移动到 B 中。这仍然会失败，因为 B 引入了隐藏的排序约束。 一旦一辆车被推入 B，它的相对顺序决定了未来的操作是否仍然可能。 如果没有全局条件，这种贪婪方法可能会以错误的顺序将车辆困在 B 中。 

关键的观察结果是 C 中的最终顺序是固定的：1 到 n。 这迫使我们按照标签递增的顺序处理汽车。 对于每个值 i，我们最终必须在它变得可访问时将其带到 C。 这将问题转化为按排序顺序检索值的模拟，同时尊重 B 的行为类似于堆栈缓冲区。 

关键的结构见解是，我们必须跟踪的唯一有意义的状态是下一辆所需的汽车是否在 A 可用，或者当前隐藏在必须暂时移至 B 的较大汽车下。这减少了维护堆栈的过程，并确保 B 永远不会违反它只能以符合未来需求的后进先出方式临时存储元素的约束。

 ## 算法演练

 我们维护一个指针`need`代表必须移动到 C 的下一辆车。我们还将站 A 模拟为输入序列上的指针，将站 B 模拟为显式堆栈。 

1. 开始于`need = 1`， 指数`i = 0`在初始数组中，以及一个空堆栈 B。我们还维护一个操作的输出列表。 
2. 虽然`need <= n`，我们尽量保证车`need`达到C。 
3.如果B的顶部等于`need`，我们将其从 B 中弹出并移动到 C。我们记录操作“B C”，然后递增`need`。 这是最直接的解决方案，因为 B 已经持有正确的下一辆车。 
4. 否则，如果A处还有车，我们就乘坐下一辆车`x = a[i]`并处理它。 如果`x == need`，我们立即将其从 A 移至 C 并同时递增`i`和`need`。 如果`x != need`，我们将它从 A 移动到 B，将其压入堆栈，然后递增`i`。 这确保了我们不会阻碍进度，同时仍然尊重排序限制。 
5.如果A已穷尽而B的顶部尚未穷尽`need`，进程被卡住并且不存在有效的序列。 我们输出-1。 
6. 继续，直到所有汽车按顺序放置在 C 中。 

中心思想是 B 作为尚未需要的汽车的临时持有结构。 如果一辆车可以立即前往 C，我们永远不会将其推入 B，因为这样做只会延迟进度并可能造成不必要的阻塞。 

### 为什么它有效

 该算法强制汽车严格按升序进入 C。 任何时候，B中的所有汽车都是之前出现过的汽车`need`已达到但不等于`need`。 由于 A 是按顺序处理的，而 B 是一个堆栈，因此 B 中的任何一辆车只有在成为顶部时才能被释放，这恰好符合较小的待处理元素不会被困在必须先使用的较大的元素下面的要求。 如果出现这样的陷阱，表现为无法暴露`need`从 A 或 B 正确地发出不可能的信号。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input().strip())
    a = list(map(int, input().split()))

    need = 1
    i = 0
    st = []
    ops = []

    while need <= n:
        if st and st[-1] == need:
            st.pop()
            ops.append("B C")
            need += 1
            continue

        if i < n:
            x = a[i]
            i += 1

            if x == need:
                ops.append("A C")
                need += 1
            else:
                st.append(x)
                ops.append("A B")
        else:
            print(-1)
            return

    print(len(ops))
    print("\n".join(ops))

if __name__ == "__main__":
    solve()
```实施直接遵循模拟。 指针`i`确保我们按照固定顺序消耗 A 站一次。 堆栈`st`代表 B 站，我们只与它的顶部交互，符合 LIFO 约束。 

一个微妙的细节是移动的优先级：我们总是首先检查 B 是否有需要的元素，然后再从 A 中消耗更多元素。这是必要的，因为否则我们可能会将额外的元素推入 B 并阻止已经可用的解决方案。 

仅当 A 为空且 B 无法提供所需车辆时，才会发生故障情况。 这是系统唯一被证明卡住的点。

### 示例 2

 输入：```
5
1 2 3 5 4
```将 1、2、3 直接放入 C 后，我们在 4 之前到达 5。 

那时：

 - B 为空
 - A 仍然包含 [5, 4]
 - 需要 = 4

 我们必须在 4 之前处理 5，但是 5 更大并且会妨碍正确的排序。 当 5 被推入 B 时，它稍后位于 4 之上，因此不可能在不违反顺序约束的情况下公开 4。 

算法最终达到A为空且B top不为4的状态，触发失败并输出-1。 

这证实了关键的约束：较大的元素不能永久放置在仍需要提前处理的较小元素之上。
