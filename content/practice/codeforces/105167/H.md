 ## 解决方案
 ## 问题理解

 我们得到了一系列饮料，每种饮料都有一种类型和以小时为单位的有效期。 时间离散地推进：Sascha 每小时正好喝一杯饮料，机器必须选择每小时分配哪种饮料。 饮料在固定的小时数后将变得无法使用，这意味着它只能在序列中最早的位置之一使用。 

关键问题不是模拟消费，而是确定所有饮料类型可以完成多少次完整的“回合”。 一轮意味着对于机器中存在的每种类型，萨沙都会设法喝掉至少一种该类型的物品。 如果某种类型不存在，该类型立即使答案为零。 

每种饮料在最终订购中都有一个截止日期位置：如果饮料有价值`b`，它最多可以放置在位置`b + 1`的消费顺序。 同样，如果我们从 1 开始索引消耗，则每个项目都有一个最新的允许位置。 

假设机器处于最佳控制状态。 这意味着我们可以自由选择任何饮料订购，只要我们尊重过期限制，以最大限度地实现完整类型覆盖的回合数。 

目标是计算最大整数`k`这样我们至少可以安排`k`在各自的截止日期之前喝完各种类型的饮料。 

限制很大：每次测试最多喝 3·10^5 杯，总共 5·10^5。 这立即排除了任何尝试通过以二次甚至三次行为的幼稚方式为每个候选步骤构建显式排列来模拟时间表或检查可行性的解决方案。 任何方法都必须接近线性或每次测试 n log n。 

当某些类型完全缺失时，就会出现微妙的边缘情况。 那样的话，哪怕是一轮完整的比赛也是不可能的。 另一个重要的极端情况是，当存在许多同一类型的项目但具有严格的过期限制时，迫使它们占用非常早的槽位，这限制了可行的完整轮数。 

## 方法

 一个直接的解释就是思考逐步构建消费秩序。 每个小时，我们都会选择一种仍然有效的饮料，试图确保随着时间的推移，我们收集所有类型的平衡计数。 人们可以想象模拟所有可能的有效时间表并检查出现了多少个完整的排版集。 然而，排列的数量是 n 的阶乘，甚至带有回溯的贪婪模拟也太慢了。 

一种更结构化的暴力思想是二分搜索`k`，整轮数。 对于固定的`k`，我们需要检查是否可以至少选择`k`每种类型的饮料，以便所有选定的饮料都可以按照截止日期放置到有效的序列中。 这成为一个受限调度可行性问题。 简单的可行性检查可能会按截止日期对所有项目进行排序并贪婪地分配位置，但如果按类型单独完成或重复扫描，则很容易变成 O(n^2 log n)。 

关键的见解是翻转视角：我们没有考虑构建全局序列，而是将每种饮料视为对早期位置的贡献能力，并且我们询问总体上可以“保留”多少项目，同时确保每种类型至少贡献`k`项目。 由于每种类型在目标上都是对称的，因此问题简化为验证我们是否可以选择`k`每种类型的项目都有最后期限，可以让它们占据第一`k · m`职位。 

这将问题转化为经典的带截止日期的调度可行性检查并结合按类型分组。 对于每种类型，我们只关心`k`最好（最大截止日期）的项目，因为任何可行的选择总是更喜欢较晚的截止日期。 在收集所有类型的这些候选者之后，我们测试它们是否可以按截止日期的递增顺序放置：贪婪扫描就足够了。 

这会产生一个干净的二分搜索`k`，每个检查在排序后都是线性的。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力调度| O(n!) 类/指数 | O(n) | 太慢了 |
 | 二分查找+贪心可行性| O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 我们独立处理每个测试用例。 

1. 按类型对所有饮料进行分组，并将其过期时间存储在单独的列表中。 这是必要的，因为每个类型的要求都是对称的，因此决策是按组而不是全局做出的。 
2. 对于每种类型，按降序排列其到期值。 这使我们能够快速识别哪些`k`如果我们的目标是，则最适合包含的项目`k`回合。 原因是，任何最佳选择都不会在同一类型中选择较小的截止日期而不是较大的截止日期。 
3.我们对答案进行二分查找`k`从 0 到最大可能值，该值受到任何类型的最大项目数和`n // m`。 
4.对于固定候选人`k`，我们尝试通过取顶部来构建所选项目的多重集`k`每种类型的截止日期。 如果某个类型的数量少于`k`项，该候选人立即失败。 
5. 我们收集所有类型的选定项目，形成尺寸列表`k · m`。 
6. 我们按截止日期对列表进行排序。 
7. 我们模拟将它们分配到时隙 1 到`k · m`。 在位置`i`，我们检查该项目的截止日期是否至少`i`。 如果任何项目不满足此条件，则该进度表对此不可行`k`。 
8.如果所有项目都通过，我们就可以实现`k`，所以我们向上移动二分查找； 否则，向下。 

关键思想是按截止日期排序强制执行最佳贪婪调度：始终首先使用最受约束的项目。 

### 为什么它有效

 该算法依赖于单调可行性属性。 如果可以安排的话`k`整轮，然后任意`k' < k`也是可行的，因为删除所需的项目只会放松约束。 对于固定的`k`，选择`k`每种类型的最后期限项目是最佳的，因为用更严格的期限替换任何选定的项目只会降低可行性。 最后，通过增加时间的贪婪分配确保如果存在任何分配，则此排序分配会成功，因为它总是首先为最受约束的元素保留早期槽。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def check(k, groups):
    items = []
    for g in groups:
        if len(g) < k:
            return False
        # take k largest deadlines
        for x in g[:k]:
            items.append(x)
    items.sort()
    for i, d in enumerate(items, start=1):
        if d < i:
            return False
    return True

def solve():
    t = int(input())
    for _ in range(t):
        n, m = map(int, input().split())
        a = list(map(int, input().split()))
        b = list(map(int, input().split()))

        groups = [[] for _ in range(m + 1)]
        for i in range(n):
            groups[a[i]].append(b[i])

        for i in range(1, m + 1):
            groups[i].sort(reverse=True)

        if any(len(groups[i]) == 0 for i in range(1, m + 1)):
            print(0)
            continue

        lo, hi = 0, n // m
        ans = 0

        while lo <= hi:
            mid = (lo + hi) // 2
            if check(mid, groups):
                ans = mid
                lo = mid + 1
            else:
                hi = mid - 1

        print(ans)

if __name__ == "__main__":
    solve()
```该实现首先按类型对过期时间进行分组，然后按降序对每个组进行排序，以便选择第一个`k`elements 对应于选择最灵活的项目。 

这`check`功能增强候选人的可行性`k`。 它通过以下方式构建候选池`k`每种类型的项目，然后按截止日期对它们进行全局排序。 贪婪验证步骤检查第 i 个计划项目是否可以存活到时间`i`。 

二分搜索包装了这个可行性测试以找到最大有效值`k`。 

一个常见的实施陷阱是忘记在截止日期条件下索引是从 1 开始的。 另一个微妙的问题是确保每种类型的选择始终使用最长的期限； 否则，可行性可能会被错误地拒绝。 

## 工作示例

 ### 示例 1

 考虑有两种类型的简化情况：

 输入：```
n = 4, m = 2
a = [1, 1, 2, 2]
b = [2, 1, 2, 1]
```我们测试`k = 1`。 

| 步骤| 行动| 项目 |
 | ---| ---| ---|
 | 组 | 按类型划分 | T1=[2,1]，T2=[2,1] |
 | 选择 | 分别获得前 1 名 | [2] + [2] = [2,2] |
 | 排序| 截止日期前 | [2,2]|
 | 分配| 检查位置 | 1≤2 可以，2≤2 可以 |`k = 1`是可行的。 

现在`k = 2`:

 | 步骤| 行动| 项目 |
 | ---| ---| ---|
 | 组 | 按类型划分 | T1=[2,1]，T2=[2,1] |
 | 选择 | 分别获得前 2 名 | [2,1] + [2,1] |
 | 排序| 截止日期前 | [1,1,2,2]|
 | 分配| 检查位置 | 位置 1 有 1≥1 ok，位置 2 有 1<2 失败 |

 所以`k = 2`失败，答案是1。 

这表明即使总计数足够，紧迫的截止日期也会阻碍更深层次的回合。

 ## 边缘情况

 关键的边缘情况是至少一种类型具有零个项目。 在这种情况下，分组会立即显示一个空列表，以及任何积极的列表`k`失败。 该算法通过在二分搜索之前进行早期检查来处理此问题，返回零。 

另一种情况是当最后期限非常紧迫时，例如所有`b_i = 0`。 即使每种类型都有足够的项目，也只能使用第一个位置，所以`k`永远不能超过 1，如果分组力量发生冲突，通常会变为 0。 贪婪的可行性检查抓住了这一点，因为排序的最后期限将立即违反早期位置约束。 

当一种类型有许多大期限项目而另一种类型主要有小期限项目时，就会出现最后一个微妙的情况。 二分查找可能会建议一个大的`k`，但在可行性构建过程中，小截止日期类型将无法通过全局订购步骤，因为其选择的项目会强制提前放置无法满足的需求。
