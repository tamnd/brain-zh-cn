 ## 解决方案
 ## 问题理解

 我们有一个交通信号灯，可以交替选择两条单向街道中的哪一条允许通过。 光的图案是周期性的并且是预先完全已知的。 每分钟属于街道 1 或街道 2，具体取决于此重复模式。 

随着时间的推移，一批汽车到达。 每辆车在特定的分钟到达，必须分配到两条街道之一。 一旦分配完毕，汽车就会加入该街道上的先进先出队列。 汽车只有满足两个条件才能通过：它已经到达，并且交通信号灯当前允许其通行。 在任何一分钟内，最多一辆车可以从当前绿色的街道上通过。 

在时间 t 到达的汽车只能从 t + 1 分钟开始行驶，因此即使信号灯立即变绿，它仍然至少要等到下一分钟。 总成本是所有车辆等待通过的分钟数的总和。 

任务不仅仅是模拟固定分配，而是为每辆车选择是否应该前往街道 1 或街道 2，以使总等待时间最小化。 

这些约束迫使我们为每个测试用例提供近线性或对数线性解决方案。 所有测试中 n 和 m 的总和最多为 100000，这排除了随时间或随分配的任何二次策略。 尝试将所有汽车分配给两条街道的天真的想法立即变得呈指数级增长。 当事件密集时，即使在没有仔​​细结构的情况下独立模拟两个队列也会面临 O(mn) 行为的风险。 

一个微妙的边缘情况来自于同时到达的汽车。 他们必须保留该分钟内的输入顺序，这会影响队列排序。 另一个棘手的情况是，当光线严重偏向一条街道时，另一条街道实际上被“封锁”，这迫使所有汽车进入唯一有效的队列。 当需要未来的光周期来避免拥塞累积时，仅基于当前光颜色的天真贪婪分配会失败。 

## 方法

 暴力策略会考虑两条街道上所有可能的汽车分配。 对于每项任务，我们都会逐分钟模拟两个队列，推进灯光并每分钟最多处理一辆车。 这是正确的，但立即不可行，因为分配的数量是 2^m。 即使 m = 100000，这也是不可能的。 

更结构化的暴力将一一分配汽车并每次重新计算完整的模拟。 这仍然会导致 O(m^2) 或更糟，因为每次插入都可能影响所有未来的队列延迟。 

关键的观察结果是，每辆车仅通过等待时间与系统进行交互，而等待时间完全取决于其所选队列中有多少辆车在其前面，以及该队列到达后获得绿色信号的频率。 如果我们确定一个分配，计算总等待时间是线性的。 

优化来自于我们注意到我们本质上是将作业分配到两台服务器上，而这两台服务器的服务可用性会定期交替。 每条街道的行为就像一个具有已知服务时段的服务流程。 因此，每辆车的决策是在具有不同未来服务时间表的两个队列之间进行选择。 这种结构允许我们按到达顺序处理汽车并维护两个不断发展的队列，在给定当前队列状态的情况下，始终将汽车分配到插入时产生较小增量成本的一侧。 

这将问题简化为贪婪模拟，在该模拟中，我们为每条街道维护该队列中的汽车下次如果追加则完成的时间。 我们总是计算将当前车辆分配到任一街道所导致的增量等待成本，并选择更好的一个。

 ## 算法演练

 我们首先在周期的每一分钟预先计算哪条街道是绿色的。 然后，我们模拟时间前进，同时维护两个队列，每条街道一个，每个队列跟踪下一个可用服务时段何时出现。 

我们按照到达时间递增的顺序处理汽车。 

1. 将到达时间转换为最早可以处理的分钟，即 t + 1。这是每辆车真正的“准备时间”。 
2. 维护两个指针，代表从时间 0 开始，每条街道下一次可以服务汽车的时间。这些指针按照周期性模式前进。 
3. 对于每辆进站车辆，计算分配到街道 1 和分配到街道 2 时其完成时间。这需要模拟在每个队列末尾插入，这意味着我们将该街道的最后计划出发时间与下一个可用服务时段进行比较。 
4. 将汽车分配到完成时间较短的街道。 如果两者相等，则任一选择都有效。 
5. 分配后，通过将车辆推入队列时间线来更新该街道的下一个可用出发时间。 
6. 等待时间累计为完成时间减去准备时间。 
7. 输出总等待时间和分配数组。 

关键思想是，每条街道的行为就像一个确定性处理器，其服务时段是及时固定的，因此附加作业仅取决于最后计划的完成情况。 

### 为什么它有效

 在任何时刻，每条街道都有由重复的交通模式引起的单调的服务机会序列。 由于车辆按到达顺序进行处理，因此每个队列内的相对顺序无需重新考虑。 一旦分配了车辆，它就不能超越该街道上较早的车辆，并且其完成时间仅取决于该街道上最后预定的完成时间。 这创建了一个贪婪选择属性：将一辆车分配到较早完成的街道永远不会恶化未来的可行性，因为它只影响该街道的尾部，而不影响其他队列的结构。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    k = int(input())
    out = []

    for _ in range(k):
        n, m = map(int, input().split())
        s = input().strip()
        t = list(map(int, input().split()))

        # precompute next green street at each cycle position
        # 0-based time modulo n
        green = [0] * n
        for i, ch in enumerate(s):
            green[i] = int(ch)

        # next available time for each street
        # we simulate service slots as time increases
        nxt = [0, 0]

        ans = [0] * m
        total = 0

        # pointers to next usable minute per street
        ptr = 0

        # we maintain current time progression implicitly
        # but we need to simulate service availability
        # so we track for each street the next time it can process a car
        import heapq

        # for each street, store next time it is free to process a car
        free = [0, 0]

        for i in range(m):
            ready = t[i] + 1

            best = None
            best_lane = 0

            for lane in (0, 1):
                cur = max(free[lane], ready)

                # simulate waiting until a green slot for that lane
                # advance until green matches lane
                while True:
                    if green[cur % n] == lane + 1:
                        finish = cur
                        break
                    cur += 1

                if best is None or finish < best:
                    best = finish
                    best_lane = lane

            ans[i] = best_lane + 1
            total += best - ready

            lane = best_lane
            start = max(free[lane], ready)

            while green[start % n] != lane + 1:
                start += 1

            free[lane] = start + 1

        out.append(str(total))
        out.append(" ".join(map(str, ans)))

    print("\n".join(out))

if __name__ == "__main__":
    solve()
```该实现为每个车道保留一个简单的状态：在之前的分配之后，它可以为新车提供服务的最早时间。 对于每辆车，我们都会尝试两条车道，并计算该车道可以接受汽车并发出绿色信号的最早时间。 该时间是通过从最大到达准备度和车道可用性开始找到的，然后向前扫描，直到周期性信号允许该车道。 

选择更好的车道后，我们将该车道的可用性更新为服务时间后一分钟，因为每个绿灯分钟只能有一辆车可以通过。 等待时间贡献是服务时间和准备时间之间的差值。

 ## 边缘情况

 当交通灯在较长一段时间内不允许一个方向时，就会出现一种关键的边缘情况。 在这种情况下，无论队列长度如何，将汽车分配到该方向都会变得昂贵。 该算法会处理此问题，因为扫描下一个有效的绿色槽位总是会将完成时间提前，从而使该车道失去吸引力。 

另一个边缘情况是许多车辆同时到达。 由于准备情况相同，因此顺序由索引保留，并且 FIFO 队列自然得到维护，因为每次分配都会单调地延长通道的空闲时间。
