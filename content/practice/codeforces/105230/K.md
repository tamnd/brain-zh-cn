 ## 解决方案
 ## 问题理解

 我们有一个矩形网格，其中每个单元格代表一堵墙、一条空通道、一个陷阱、起始位置或一个包含一定数量宝藏的单元格。 从起始单元格开始，允许在四个方向上移动。 目标是确定在陷阱施加的非常具体的限制下穿过迷宫时可以收集多少宝藏。 

重要的变化是陷阱不会直接阻止移动。 相反，任何与陷阱直接相邻的单元格都会成为危险区域。 艾林拒绝接受从这个意义上来说不安全的职位。 因此，有效的可穿越区域不仅由墙壁决定，还由陷阱的“影响区”决定，这将附近的单元排除在考虑范围之外。 

每个测试用例都是一个独立的迷宫，我们必须计算从一开始就可以到达的所有数字的总和，同时尊重这些安全约束。 

网格大小可以大到 1000 x 1000。这会立即排除比每个测试用例的线性时间更糟糕的情况，因为完整扫描已经触及多达一百万个单元。 任何过于频繁地重新访问单元格或模拟天真的探索而不进行修剪的解决方案都会超时。 

有些情况往往会破坏简单的解决方案。 

当单元仅通过几何形状即可到达但位于陷阱旁边时，就会出现一种失败情况。```
S1.
.T.
.1.
```一个忽略陷阱影响的天真的 BFS 会到达底部的单元格并计算宝藏，但正确的答案是 0 或 1，具体取决于该单元格是否与陷阱相邻； 在此布局中，它是相邻的，并且必须完全排除在遍历之外。 

另一个微妙的情况是起点本身与陷阱相邻。 幼稚的方法可能仍会向外扩展，但规则甚至阻止了第一步。 

最后，具有多个断开连接的安全区域的网格通常会误导洪水填充实现，这些实现在遍历之前不会正确标记不安全的单元格。 如果在 BFS 期间延迟检查安全性而不是预先计算，则算法可能会意外地“遍历”本应失效的单元格。 

## 方法

 强力解释将把探索模拟为状态搜索，其中每次移动都会根据局部陷阱接近度考虑是否允许进入相邻单元格。 从每个状态，我们将检查所有四个方向，动态检查周围的陷阱，并递归分支。 

从概念上讲，这是可行的，因为每条有效路径都会被探索，并且每当访问单元格时就会积累宝藏。 但问题就出在重复工作上。 在最坏的情况下，每个单元都可以从多个方向进入，并且对于每个进入，我们可以重新扫描邻居以验证安全性。 这将病态配置的复杂性推向了 O(N²M²)，因为本地检查会在许多路径上重复进行。 

关键的观察结果是，细胞的“危险状况”并不取决于我们如何到达那里。 一个单元格在全局范围内是安全的还是不安全的，仅取决于任何相邻单元格是否包含陷阱。 一旦预先计算完毕，网格就成为过滤图上的标准连接问题。 

然后，问题就简化为找到包含起始节点的连接组件，但仅限于安全单元格，并一路求和数值。 

这会将任务转换为最多 N×M 个节点上的单个 BFS 或 DFS，并具有恒定时间转换。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力模拟| O((NM)²) | O(NM) | 太慢了 |
 | 预计算安全+ BFS | O(NM) | O(NM) | 已接受 |

 ## 算法演练

 1. 扫描整个网格一次，记录所有陷阱的位置。 这是必要的，这样我们以后就可以有效地标记它们的影响，而无需重复搜索。 
2. 创建布尔网格`bad`，最初对于每个单元格都是错误的。 我们将用它来标记无法安全访问的单元格。 
3. 对于每个陷阱单元，如果其四个邻居位于网格内且不是墙壁，则将其标记为坏单元。 这一步构建了陷阱引发的“危险区域”。 我们在遍历之前这样做的原因是必须独立于路径选择来了解安全性。 
4. 还将所有陷阱单元本身标记为坏，因为它们无法进入。 
5. 找到起始单元格。 如果它已经被标记为坏，则答案立即为零，因为无法开始任何合法运动。 
6. 从网格的起始单元运行 BFS。 只能进入不是墙壁、没有标记为坏的、并且之前没有被访问过的牢房。 
7. 每当 BFS 访问一个单元格时，如果它包含数字，则将其转换为整数并将其添加到运行总计中。 
8. 继续，直到队列为空。 累加的总和就是答案。 

### 为什么它有效

 正确性取决于“坏”状态与遍历顺序无关的事实。 如果一个单元格是一个陷阱或直接与一个陷阱相邻，那么它就变得不可用，并且该属性不会根据我们的移动方式而改变。 一旦所有不安全的单元被移除，每个剩余的移动都会保持安全并形成标准的无向网格图。 因此，BFS 从一开始就准确地枚举了可到达的安全组件，并且由于每个单元格都被访问一次，因此该组件中的所有可收藏宝藏都被准确地计数一次。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

from collections import deque

def solve():
    n, m = map(int, input().split())
    g = [list(input().strip()) for _ in range(n)]

    bad = [[False] * m for _ in range(n)]
    sx = sy = -1

    traps = []

    for i in range(n):
        for j in range(m):
            if g[i][j] == 'T':
                traps.append((i, j))
            if g[i][j] == 'S':
                sx, sy = i, j

    dirs = [(1,0), (-1,0), (0,1), (0,-1)]

    for x, y in traps:
        bad[x][y] = True
        for dx, dy in dirs:
            nx, ny = x + dx, y + dy
            if 0 <= nx < n and 0 <= ny < m:
                if g[nx][ny] != '#':
                    bad[nx][ny] = True

    if bad[sx][sy]:
        print(0)
        return

    q = deque()
    q.append((sx, sy))
    vis = [[False] * m for _ in range(n)]
    vis[sx][sy] = True

    ans = 0

    while q:
        x, y = q.popleft()

        if g[x][y].isdigit():
            ans += int(g[x][y])

        for dx, dy in dirs:
            nx, ny = x + dx, y + dy
            if 0 <= nx < n and 0 <= ny < m:
                if not vis[nx][ny] and not bad[nx][ny] and g[nx][ny] != '#':
                    vis[nx][ny] = True
                    q.append((nx, ny))

    print(ans)

if __name__ == "__main__":
    solve()
```首先完全扫描网格以定位陷阱和起始位置。 这样就避免了以后的重复工作。 这`bad`数组对所有因接近陷阱而被禁止的单元进行编码。 这种预计算至关重要，因为它将动态安全条件转变为静态属性。 

然后，BFS 的行为类似于标准洪水填充，只不过它会过滤掉墙壁和不安全的单元格。 访问的数组确保每个单元都被处理一次，从而防止指数分支。 

一个常见的错误是在 BFS 扩展期间检查与陷阱的邻接性，而不是预先计算它。 这会导致根据遍历顺序做出不一致的决策。 另一个微妙的问题是忘记阻止陷阱单元本身，如果它们不与另一个陷阱相邻，则可能会错误地允许通过它们。 

## 工作示例

 考虑以下输入：```
4 9
S......T.
#####.###
111.#.111
..T....T.
```预处理后，所有与陷阱相邻的单元格都会被阻塞。 BFS 开始于`S`并只探索安全的走廊。 

| 步骤| 队列| 当前单元格| 行动| 总计 |
 | --- | --- | --- | --- | --- |
 | 1 | (S)| S | 开始 | 0 |
 | 2 | ... | 仅可到达的安全单元| 在安全区域收集数字| 2 |

 遍历很快就会受到限制，因为陷阱附近的大部分网格在搜索开始之前就失效了。 

现在考虑第二个自定义示例：```
3 5
S1T..
..2..
..3..
```标记不安全单元格后，与它相邻的所有单元格`T`被删除。 假设只有底部区域保持连接。 

| 步骤| 队列| 当前单元格| 行动| 总计 |
 | --- | --- | --- | --- | --- |
 | 1 | S | S | 开始 | 0 |
 | 2 | (1) | 1 | 收集| 1 |
 | 3 | (2) | 2 | 收集| 3 |
 | 4 | (3) | 3 | 收集| 6 |

 该痕迹表明，BFS 仅在预先验证的安全结构上运行，而宝藏积累纯粹是可达性的函数。 

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 | O(NM) | 每个单元在预处理期间处理一次，在 BFS 期间处理一次 |
 | 空间| O(NM) | 用于网格状态、访问跟踪和安全标记的数组 |

 这些限制允许每个测试用例最多一百万个单元，因此基于线性扫描的解决方案可以轻松满足时间和内存限制。 该算法避免了重复探索，即使在密集网格中也能确保可预测的性能。

 ## 边缘情况

 一种边缘情况是起点被陷阱包围。 在这样的配置中，预处理步骤将起始单元标记为坏的，并且算法立即终止。 例如：```
3 3
T.T
.S.
T.T
```这`bad`标记从每个陷阱传播到中心单元，因此永远不会启动 BFS，从而正确地产生零。 

另一种情况是，小区只能通过某个区域到达，但随后由于邻接规则而变得无效。 由于所有失效都是在遍历之前完成的，因此 BFS 根本不会考虑这些单元格，从而防止出现任何不一致的部分探索。 

最后一种情况发生在具有稀疏陷阱的大型开放网格中。 尽管大多数单元格在几何上是可达的，但只有那些不与陷阱相邻的单元格仍保留在 BFS 图中。 该算法可以正确地限制探索，而无需模拟可见性或方向推理。
