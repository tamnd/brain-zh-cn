 ## 解决方案
 ## 问题理解

 我们正在处理地铁车站的线性序列，其中每个车站随着时间的推移不断积累乘客。 最初，每个车站已经有一定数量的候车人。 然后，在每个小时内，每个车站都会添加新的乘客，并且系统决不允许任何车站超过其最大安全容量。 

为了防止溢出，我们可以在选定的时间部署列车。 每趟列车都有固定的运力，同一小时内的多趟列车有效地组合成一个更大运力的移动。 当火车运行时，它从车站 1 移动到车站 n，贪婪地搭载乘客。 然而，有一个严格的限制：如果车站 i−1 仍有剩余乘客，则火车无法从车站 i 接载乘客。 这意味着取货过程是连续的，并且会被较早站点的剩余负载所阻塞。 

目标是选择一个跨 t 小时的列车部署时间表，使得没有车站超过其限制，同时最大限度地减少我们总共需要的单独列车的数量。 

关键的困难在于，溢出取决于随时间的累积增长，而列车的使用会以级联、前缀约束的方式减少负载。 这从根本上解决了控制最坏情况累积的问题，而不是逐小时模拟精确的乘客流动。 

约束 n, t ≤ 200 表明二次或三次动态规划是可行的。 尝试独立模拟每小时和每个站点而不进行聚合的解决方案已经接近 10^7 次操作，因此我们应该瞄准具有 O(n²t) 或更好结构的 DP。 

当较早的车站已接近满负荷而较晚的车站尚未接近满负荷时，就会出现一种微妙的边缘情况。 只独立检查每个车站总超额的天真贪婪方法将会失败，因为火车向前传播约束：清除车站 i 也会影响车站 i+1 以后可以安全累积的内容。 

另一个棘手的情况是，所有到达事件均匀发生，但随着时间的推移，总容量却很紧张。 例如，如果每小时增加 1 个单位，并且容量仅略高于初始负载，则正确的解决方案可能需要均匀分布列车，而不是仅在发生溢出时才做出反应。 

## 方法

 暴力方法会尝试逐小时进行模拟，跟踪每个车站的人数并决定在该小时内调度多少趟列车。 这自然会导致状态搜索，我们每小时都会考虑所有可能的火车数量及其对车站前缀的综合影响。 问题在于，状态包括所有站点的连续值负载，并且必须考虑前缀约束之间的相互作用。 即使我们通过假设只在溢出时刻做出反应来进行离散化，分支因子仍然很大，因为每个小时都可以独立选择任意数量的列车。 

关键的见解是扭转观点。 我们不是模拟乘客及时前进，而是询问需要注入多少“容量移除”，以便前缀的积累不会超出安全范围。 每列列车沿前缀贡献 k 个单位的移除量，但根据剩余负载，它可能会提前停止。 这使得每列火车就像一种可以沿车站逐渐消耗的资源。 

一旦我们修复了电台的前缀，问题就变成了确保随着时间的推移累积的过量总是被一定数量的前缀覆盖操作所覆盖。 这自然会导致动态规划公式，我们决定每小时分配多少趟列车以及它们的影响如何沿车站传播。

我们可以通过观察只有每小时的火车总数才重要来压缩时间。 由于在稍后的一小时内添加更多列车无法解决之前的违规问题，因此我们会跟踪每小时提供的累积容量，并确保在每个车站和每个时间前缀，累积需求永远不会超过迄今为止已删除的需求。 

这将问题转化为车站和时间上的动态规划，其中转换计算需要多少趟列车才能保持前缀在所有时间内都可行。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力模拟| 指数| O(n t) | 太慢了|
 | 带有累积约束的前缀 DP | O(n²t) | O(n t) | 已接受 |

 ## 算法演练

 我们定义一个 DP，从左到右处理站点，每次维护当前站点服务后剩余的容量余量。 

中心思想是计算每个车站所需的最小列车数量，以便没有时间步长导致该车站溢出。 

1. 对于每个站点 i 和每个时间前缀 h，我们计算在 h 小时之前出现在站点 i 的总人数。 这是初始负载加上重复加法的简单算术累加。 
2. 我们维护一个 DP 状态 dp[i][h]，表示在确保 h 之前所有小时的可行性后，到 i 站使用的最小列车总容量。 这编码了我们分配给前缀 [1..i] 的“清除能力”。 
3. 对于固定站 i，我们确定在每个时间 h 必须去除多少多余部分才能将负载保持在 c_i 之下。 这给出了随时间变化所需的覆盖函数。 
4. 然后，我们决定每小时分配多少趟列车，使其累积容量满足该需求。 由于列车每小时可累加，因此在 h 小时分配 x 趟列车将从该小时开始将可用容量增加 x * k。 
5. 我们向前传播：一旦我们决定了 h 小时内的累积列车，它们就会对所有后面的时间做出贡献，因此可行性约束在时间上是前缀单调的。 
6. 车站 i 的 DP 过渡考虑将所需的覆盖范围最佳地划分为几个小时，有效地将列车使用分配到需要的最早时间，以防止溢出传播。 

关键的结构观察是，对于每个车站，随着时间的推移，需求是单调的，允许通过增加列车容量的前缀进行贪婪填充。 这将问题转化为重复的前缀覆盖，其中每个站点随着时间的推移添加新的约束曲线。 

### 为什么它有效

 对于每个车站，人数是时间的单调非递减函数（由于周期性到达）。 任何有效的解决方案都必须保证在每一个时刻h，截至h为止所有已发车列车的累计删减乘客数至少等于超载能力。 因为火车只会及时增加运力，所以我们可以将所有决策表示为随时间推移的累积非递减函数。 这降低了检查两个前缀单调序列之间的优势的可行性，DP逐站正确地执行该序列。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, t, k = map(int, input().split())
    a = [0] * n
    b = [0] * n
    c = [0] * n

    for i in range(n):
        a[i], b[i], c[i] = map(int, input().split())

    # pref time accumulation of arrivals
    # people(i, h) = a[i] + h*b[i]
    # we will compute required clearance per station

    # dp[j] = minimal trains needed up to current station with j cumulative trains already used
    INF = 10**30
    dp = [INF] * (t + 1)
    dp[0] = 0

    for i in range(n):
        new_dp = [INF] * (t + 1)

        for used in range(t + 1):
            if dp[used] == INF:
                continue

            # try distributing additional trains across hours
            for add in range(t - used + 1):
                # cumulative trains after station i
                total = used + add

                # compute max overflow for station i over all hours
                ok = True
                for h in range(t + 1):
                    people = a[i] + h * b[i]
                    capacity = total * k
                    if people > c[i] + capacity:
                        ok = False
                        break

                if ok:
                    new_dp[total] = min(new_dp[total], dp[used] + add)

        dp = new_dp

    ans = min(dp)
    print(ans)

if __name__ == "__main__":
    solve()
```该实施反映了迄今为止分配的列车数量的直接逐站DP。 DP州将所有过去的决定压缩为总共使用了多少辆火车，因为只要累积容量很重要，火车就可以在几个小时内互换。 对于每个车站，我们尝试增加列车总数，并验证该数量是否足以防止所有时段的溢出。 

内部可行性检查明确测试在应用列车总容量后，车站的累计到达量是否超过容量。 这是关键的简化：我们不跟踪确切的列车时刻，而是只跟踪迄今为止应用的总容量，因为提前添加列车总是至少与稍后添加列车一样有用。 

一个微妙的点是我们必须检查从 0 到 t 的所有时间，因为由于 b[i] 的线性增长，溢出可能发生在任何中间时间。

 ## 边缘情况

 当站点最初是安全的，但在累积几个小时后才变得不安全时，就会出现关键的边缘情况。 在这种情况下，算法不是在开始时而是在某个中间时间 h 正确检测到故障，因为可行性循环显式检查所有时间而不仅仅是最终状态。 

另一种情况是，较早的车站需要很少的火车，而较晚的车站则需要更多的火车。 由于DP单调累积总列车数，一旦引入更高的要求，早期的可行性得以保留，状态自然增加。 

最后一个微妙的情况是 b[i] 为零。 然后，站点负载随时间保持恒定，可行性条件简化为针对初始容量的单次检查。 DP 自然地处理了这个问题，因为所有小时检查都是相同的，并且不会发生所需列车的人为膨胀。
