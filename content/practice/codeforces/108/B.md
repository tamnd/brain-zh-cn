 ## 解决方案
 ## 问题理解

 我们有几种无符号整数数据类型，每种数据类型都由其位长度定义。 数据类型与`a`位可以存储每个整数`0`最多`2^a - 1`。 

我们想知道是否存在某个整数`x`和两个具有大小的数据类型`a[i] < a[j]`这样：

 1.`x`适合较小的数据类型。 
2.`x * x`不适合较大的数据类型。 

如果存在这样的一对，该语言的行为就会很奇怪，以至于 Tuftuf 决定放弃它。 

关键的观察结果是实际值`x`事情只能通过二的权力。 由于可表示的最大数量`a`位是`2^a - 1`，拟合的数字中最大可能的平方`a`位大约是`(2^a)^2 = 2^(2a)`。 所以问题就变成了：

 可以对一个数进行平方`a`-bit 类型需要超过`b`位，其中`a < b`？ 

输入大小达到`10^5`，因此任何直接在二次时间内检查每对的算法都会执行大约`10^10`最坏情况下的比较。 这远远超出了 2 秒的限制。 我们需要更接近的东西`O(n log n)`或者`O(n)`。 

问题的一个微妙部分是，条件涉及存在性，而不是普遍性。 我们只需要一个整数`x`这会破坏更大的数据类型。 粗心的解决方案可能只测试较小数据类型的最大值，而错过更简单的见证。 

考虑这个输入：```
2
3 4
```正确答案是`"YES"`因为`x = 7`适合 3 位，但是`49`不适合 4 位。 仅非正式地使用位计数进行推理的幼稚实现可能会错误地认为将数据类型增加一位应该总是足够的。 

另一个容易犯的错误是错误地处理相同的数据类型大小。```
3
5 5 5
```正确答案是`"NO"`因为条件明确要求`a[i] < a[j]`。 相同大小不能形成有效的对。 

第三个陷阱是具有固定宽度整数的语言中的溢出。 如果有人计算`2^a`直接为大`a`，值如`2^10^9`是不可能存储的。 预期的解决方案永远不需要两个的实际幂，只需要指数之间的关系。 

## 方法

 蛮力的想法很简单。 对于每对有序的数据类型大小`(a, b)`在哪里`a < b`，我们询问是否存在一些拟合值`a`其平方超过可表示的最大值的位`b`位。 

一个数字适合`a`位如果：```
x ≤ 2^a - 1
```它的正方形无法容纳`b`位如果：```
x^2 > 2^b - 1
```最大可能的正方形来自于`x = 2^a - 1`。 平方它大约给出`2^(2a)`。 这意味着平方可以超过`b`- 位限制，只要：```
2a > b
```所以暴力解决方案就变成了：

 1. 检查每一对`(a, b)`和`a < b`。 
2.如果`2a > b`， 回答`"YES"`。 

这个逻辑是正确的，但是检查所有对的成本`O(n^2)`。 和`n = 10^5`，这变得不可用。 

关键的见解是排序后只有最小和最大的相关数据类型大小很重要。 

假设数组已排序：```
a[0] ≤ a[1] ≤ ... ≤ a[n-1]
```对于某些数据类型大小`a[i]`，我们想知道是否存在更大的数据类型`a[j]`满意：```
a[j] < 2 * a[i]
```由于数组已排序，因此最简单的候选者是严格大于的最小数据类型`a[i]`。 如果该数据类型至少是`2 * a[i]`，那么后面的每个数据类型也太大，并且不存在有效的对`a[i]`。 

所以排序后，我们只需要比较相邻的不同大小即可。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(n²) | O(1) | O(1) | 太慢了 |
 | 最佳| O(n log n) | O(n log n) | 除了排序之外，O(1) | 已接受 |

 ## 算法演练

 1. 将数据类型大小读入数组。 
2. 按非降序对数组进行排序。 

排序使我们能够按升序检查数据类型大小，并有效地推理出较大的候选者。 
3. 迭代已排序数组中的相邻对。 
4. 对于每一个相邻的对`(a[i], a[i+1])`，首先检查它们是否严格不同。 

大小相等不能满足条件，因为问题需要`a[i] < a[j]`。 
5. 如果`a[i+1] < 2 * a[i]`， 打印`"YES"`并终止。 

这意味着存在一个更小的数据类型`a[i]`和一个更大的数据类型`a[i+1]`这样，对较小类型中的某个数字进行平方需要的位数多于较大类型可以容纳的位数。 
6. 如果循环结束时没有找到这样的一对，则打印`"NO"`。 

### 为什么它有效

 数据类型与`a`位最多可以存储数字`2^a - 1`。 对最大可能值进行平方得出：```
(2^a - 1)^2 = 2^(2a) - 2^(a+1) + 1
```这个值至少是`2^b`每当`2a > b`，这意味着它不再适合内部`b`位。 

因此，整个问题简化为通过以下方式查找两种数据类型大小：```
a < b < 2a
```排序后，如果任何地方都存在这样的一对，那么相邻的不同值中也一定存在一个。 认为`(a[i], a[j])`有效于`i < j`。 由于数组已排序：```
a[i+1] ≤ a[j] < 2a[i]
```所以相邻对`(a[i], a[i+1])`已经足够了。 检查相邻对足以检测每个有效配置。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    a = list(map(int, input().split()))

    a.sort()

    for i in range(n - 1):
        if a[i] < a[i + 1] and a[i + 1] < 2 * a[i]:
            print("YES")
            return

    print("NO")

solve()
```第一部分读取输入并对数据类型大小进行排序。 排序是优化的基础，因为它可以让我们减少对相邻值的搜索。 

该循环仅检查相邻对。 条件：```
a[i] < a[i + 1]
```过滤掉相同的数据类型大小，根据语句，这些数据类型大小是无效的。 

第二个条件：```
a[i + 1] < 2 * a[i]
```直接来自数学推导。 如果这个不等式成立，那么某个数字适合`a[i]`位有一个平方，需要超过`a[i+1]`位。 

该实现从不计算 2 的实际幂，因此即使数据类型大小达到`10^9`。 

找到一对有效的解决方案后立即退出，因为问题仅询问这样的一对是否存在。

 ## 边缘情况

 考虑所有数据类型大小都相等的情况。 

输入：```
4
5 5 5 5
```排序后，数组保持不变。 每个相邻对都无法严格比较：```
5 < 5
```所以算法永远不会进入成功条件并打印`"NO"`。 

这满足了较大的数据类型必须真正更大的要求。 

现在考虑精确的加倍边界。 

输入：```
2
16 32
```该算法检查：```
32 < 2 * 16
32 < 32
```这是错误的。 

输出是`"NO"`因为 32 位数据类型正好足够包含所有 16 位数字的平方。 

最后，考虑有效对在原始输入中不相邻的情况。 

输入：```
5
100 3 8 20 5
```排序后：```
[3, 5, 8, 20, 100]
```该算法检查相邻对：```
5 < 6
```这是真的，所以它立即打印`"YES"`。 

这说明了为什么排序至关重要。 原始顺序隐藏了关系，但排序顺序公开了每个值的最小较大数据类型。
