 ## 解决方案
 ## 问题理解

 我们得到一个数组`a`其中每个位置代表一个单词及其分配的分数。 

对于每对索引`(i, j)`和`i < j`，问题定义$$\text{dist}(i,j)=a_i+a_j$$当距离与位置之间的差异完全匹配时，一对被认为是美丽的：$$j-i=a_i+a_j$$对于每个测试用例，我们必须计算有多少对满足此等式。 

数组长度可以大到`2 · 10^5`，以及所有的总和`n`跨测试用例也至多`2 · 10^5`。 任何检查每一对的解决方案都需要大约$$\frac{n(n-1)}2$$比较。 为了`n = 2 · 10^5`，这大约是`2 · 10^{10}`操作，这是完全不可行的。 

总输入大小强烈表明我们需要每个测试用例接近线性时间，或者最坏的情况`O(n log n)`。 

微妙的观察来自于界限`a_i`。 由于每个分数都是正数，如果`a_i >= n`，那么没有对涉及`i`可以永远美丽，因为$$a_i+a_j \ge a_i+1 > n-1$$而最大可能的值`j-i`只是`n-1`。

3
2025 2025 2025
```每个分数都远大于任何可能的指数差异。 答案是：```
0
```不利用这一事实的简单实现仍然是正确的，但会浪费大量时间来检查不可能的对。

## 方法

 最直接的解决方案是检查每一对`(i, j)`, 计算`a_i + a_j`，将其与`j - i`，并计算匹配项。 

该方法显然是正确的，因为它检查了定义中的确切条件。 问题在于它的复杂性。 为了`n = 2 · 10^5`，对的数量大约是二十亿，远远超出了极限。 

为了更快地找到一些东西，重写条件：$$j-i=a_i+a_j$$将涉及相同索引的项移至同一侧：$$j-a_j=i+a_i$$现在这对很漂亮，正是价值所在$$i+a_i$$左端点等于$$j-a_j$$为正确的端点。 

这将问题从检查对转变为计算相等值。 

假设我们从左到右处理索引。 当我们到达位置时`j`，我们需要知道有多少个早期位置`i`满足$$i+a_i=j-a_j$$如果我们维护所有先前看到的值的频率图`i+a_i`，那么有效合作伙伴的数量`j`只是频率`j-a_j`。 

每个索引只贡献一次，给出线性时间解决方案。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(n²) | O(1) | O(1) | 太慢了 |
 | 最优哈希图 | O(n) | O(n) | 已接受 |

 ## 算法演练

 1. 创建一个空的频率图。 
2. 使用从 1 开始的索引从左到右处理数组。 
3. 对于当前位置`j`，计算：$$key = j - a_j$$每个较早的索引`i`满意的`i + a_i = key`与`j`。 

1.添加频率`key`到答案。 

这会计算所有有效的早期伙伴，并且不会计算无效的伙伴，因为转换后的方程完全等于原始条件。 

1. 计算$$store = j + a_j$$并将其插入到频率图中。 

未来的头寸可能会使用该指数作为其左端点。 

1. 继续，直到处理完所有位置。 

### 为什么它有效

 频率图始终包含值`i + a_i`对于严格小于当前位置的指数。 

处理索引时`j`，每一对美丽的情侣都必须满足$$i+a_i=j-a_j$$并且地图中的每个索引都已经有`i < j`。 因此频率为`j-a_j`正是有效的早期索引的数量。 每个漂亮的对在处理其右端点时都会被计数一次，并且没有无效的对可以满足转换后的相等性。 这确定了正确性。 

## Python 解决方案```python
import sys
from collections import defaultdict

input = sys.stdin.readline

def solve():
    t = int(input())

    for _ in range(t):
        n = int(input())
        a = list(map(int, input().split()))

        freq = defaultdict(int)
        ans = 0

        for i in range(1, n + 1):
            x = a[i - 1]

            ans += freq[i - x]
            freq[i + x] += 1

        print(ans)

solve()
```解决方案直接遵循代数变换。 

变量`freq`存储值的计数`i + a_i`来自先前处理的索引。 对于每个位置，我们首先查询`freq[i - a_i]`因为只有较早的索引才能形成有效的对。 计算贡献后，我们插入`i + a_i`进入地图。 

顺序很重要。 如果首先发生插入，则当前索引可能会在以下情况下错误地匹配自身：`a_i = 0`。 该问题保证正值，但保持正确的顺序可以保持逻辑在数学上的精确性。 

在循环中使用从 1 开始的索引可以与原始方程完全匹配，并避免在推导过程中进行额外的调整。

 ## 边缘情况

 考虑：```
1
3
2025 2025 2025
```加工产生钥匙`-2024`,`-2023`， 和`-2022`。 这些值都不会作为存储值出现`i + a_i`，因此每次查找都会返回零。 该算法输出：```

```这是正确的，因为没有指数差异可以达到数千。 

考虑：```

5
1 1 1 1 1
```存储的值变为`2, 3, 4, 5, 6`。 在职位`3`,`4`， 和`5`，查询的值为`2`,`3`， 和`4`，每个都已经出现一次。 答案就变成了`3`，正确计算所有有效对，而不会重复计算。 

考虑：```
1
4
3 1 1 1
```地图演变为：```
after i=1: {4:1}
after i=2: {4:1,3:1}

```在`i=4`，查询值为`3`，存在一次。 答案加一并结束于：```

```匹配唯一美丽的一对`(2,4)`。
