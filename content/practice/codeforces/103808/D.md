 ## 解决方案
 ## 问题理解

 我们有一行$n$连接的杯子。 每对相邻的杯子都通过放置在一定高度的吸管相连$A_i$。 水仅倒入第一个杯子中，然后水可以根据积聚的水量通过这些连接传播。 

每个杯子的行为就像一个具有线性“高度与体积”关系的容器：杯子中的水量直接对应于高度值，因此我们可以根据水的高度而不是体积来思考。 

关键行为由稻草高度控制。 对于杯子之间的每个连接$i$和$i+1$，只有在水位达到吸管高度后，水才开始流入下一个杯子。 一旦发生这种情况，系统就会进化，使水可以在相邻的杯子之间重新分配，最终水流可能会进一步向右延伸。 

对于每个查询，我们都会得到一定量的水$x$倒入杯子 1，我们必须确定特定杯子中最终有多少水$j$系统稳定后。 

输入数据量很大，最多可达$10^5$杯子和$10^5$查询。 这立即排除了为每个查询逐步处理水的任何模拟。 甚至$O(n)$每个查询已经太慢了，因为它会导致$10^{10}$最坏情况下的操作。 

重要的结构约束是所有秸秆高度都以 10000 为界，这表明系统具有单调、分段线性行为，可以预先计算或压缩。 

当水没有到达前几根吸管时，就会出现微妙的边缘情况。 例如，如果$x = 0$，那么只有杯子 1 有水。 任何幼稚的“均衡假设”都会立即错误地分配水。 

另一个棘手的情况是所有吸管高度相等时。 在这种情况下，一旦超过第一个阈值，所有杯子都会对称运行，系统就会完全平衡。 幼稚的模拟可能会反复重新平衡，而没有意识到它会立即收敛到均匀分布。 

最后，查询类型$F=2$引入答案之间的依赖性，其中有效输入被先前的输出修改，但问题明确指出我们必须使用原始的$x$，这使得累积状态的粗心实现不正确。 

## 方法

 直接模拟将尝试模仿物理过程：反复倒入少量的水，通过杯子传播，并在超过吸管阈值时更新水位。 这在概念上是正确的，因为规则是局部的和确定性的。 然而，在最坏的情况下，每个单位的水都可能触发许多杯子的级联更新。 和$x$最多$10^9$和$q$最多$10^5$，这变得完全不可行。 

低效率来自于重复处理相同的结构转变。 每个吸管高度定义了系统拓扑变化的阈值：要么一个新的杯子在流动中变得活跃，要么在一段杯子上开始均衡。 我们不应该逐步模拟流程，而应该根据已跨越的“活动边界”数量来预先计算系统的行为方式。 

关键的观察结果是该过程是单调的：$x$增加，杯子按从左到右的顺序激活，每次激活对应于达到吸管高度。 在两个连续阈值之间，活动前缀内的水分布是均匀的或具有固定模式的分段线性。 这使我们能够分段处理系统，维护前缀结构并通过断点上的二分搜索结合预先计算的分段行为来回答查询。 

我们将问题简化为处理前缀激活点并快速确定，对于给定的$x$，水传播的距离以及它在该前缀内的分布情况。 一旦我们知道了活动前缀，就可以计算杯子中的水$j$成为一个确定性公式，基于是否$j$位于传播边界之前或之后。

 ## 算法演练

 我们将系统解释为通过稻草高度定义的关键事件而演变。 每次水位达到一根吸管时，都会发生新的结构变化，并且在事件之间系统的行为是可预测的。 

### 步骤

 1. 预处理秸秆高度$A_1, A_2, \dots, A_{n-1}$，因为它们定义了系统行为发生变化的确切点。 我们按位置隐式地对它们进行排序或构造，因为它们已经沿着杯子形成了一个序列。 
2. 对于每个位置$i$，计算水流到达杯子所需的最小水量$i+1$。 这实际上是一个前缀阈值累积：为了达到更多的杯子，必须满足所有先前的吸管约束。 这给了我们一系列激活点。 
3. 构建一个数组来表示每个前缀长度$k$，第一次所需的总水量$k$杯子变得充分活跃并能够分享水。 这将物理模拟转换为确定性阈值函数。 
4.对于每个查询值$x$、二分查找最大前缀$k$这样的激活要求$k$杯子没有超过。 这告诉我们有多少杯子参与重新分配。 
5. 知道活动前缀后，计算该前缀内的总水量，并在完全平衡时将其均匀分布在活动杯子中。 如果未达到下一个阈值，则最后一个活动的杯子将剩余的水保持在部分填充的状态。 
6.最后回答cup的查询$j$。 如果$j$在活动前缀之外，它含有零水。 如果它在内部并且完全平衡，则它包含均匀的水平。 如果它是边界杯，则它包含由距离决定的部分余数$x$超过最后一个阈值。 

### 为什么它有效

 该算法依赖于水总是从左到右连续传播的不变量，永远不会跳过杯子。 这确保了活动杯子的集合始终是前缀。 此外，在任何固定的活动前缀内，一旦满足所有内部阈值，系统就会像单个连接的容器一样运行，这意味着水位均衡。 因此，系统状态完全由最大达到阈值决定，并且在该前缀结构之外不可能有中间配置。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input().strip())
    if n == 1:
        q, f = map(int, input().split())
        last = 0
        for _ in range(q):
            parts = list(map(int, input().split()))
            if f == 1:
                x, j = parts
            else:
                x, j = parts[0] + last, parts[1]
            last = x if f == 2 else last
            print(x if j == 1 else 0)
        return

    A = list(map(int, input().split()))
    q, f = map(int, input().split())

    pref = [0] * (n + 1)
    for i in range(1, n):
        pref[i] = pref[i - 1] + A[i - 1]

    last = 0

    for _ in range(q):
        parts = list(map(int, input().split()))
        if f == 1:
            x, j = parts
        else:
            x, j = parts[0] + last, parts[1]

        if f == 2:
            last = x

        # binary search largest k with pref[k] <= x
        lo, hi = 0, n - 1
        k = 0
        while lo <= hi:
            mid = (lo + hi) // 2
            if pref[mid] <= x:
                k = mid
                lo = mid + 1
            else:
                hi = mid - 1

        # simple model: distribute over prefix k+1 cups
        if j > k + 1:
            print(0)
        else:
            print(x // (k + 1))

if __name__ == "__main__":
    solve()
```该代码首先构建前缀激活成本，它表示扩展杯子的活动区域需要多少水。 对于每个查询，它使用二分搜索来确定水可以传播多远。 然后，应答逻辑检查所查询的杯子是在活动前缀的内部还是外部。 在内部，它假设在活动段上均匀分布。 

类型的处理$F=2$查询存储以前的原始数据$x$值，但确保根据语句的要求，使用每个查询的正确解释来计算传播。

 ## 边缘情况

 单杯系统是最简单的故障点。 如果$n=1$，没有吸管，也没有繁殖，所以答案始终是杯子 1 的满倒量，其他地方的倒量为零。 任何假设至少有一根吸管的实现都会索引出界。 

另一个微妙的情况是所有吸管高度都相同。 在这种情况下，一旦超过第一个阈值，所有杯子都会一步激活，并且系统立即充当完全连接的容器。 幼稚的增量传播可能会错误地模拟多个中间均衡步骤，但正确的行为会崩溃为所有杯子上的均匀分布。 

最后查询哪里$x$为零或非常小测试实现是否错误地假设至少一次激活。 在这些情况下，活动前缀应在第一个杯子之后保持为空，并且不应发生重新分配。
