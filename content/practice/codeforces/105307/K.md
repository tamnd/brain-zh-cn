 ## 解决方案
 ## 问题理解

 这个问题中的每个部分都可以被视为最多 14 个可能统计数据中统计类型的子集。 购买药水会为您提供其所有涵盖的统计数据，并且由于主角是“幸运的”，因此每种药水都保证会成功，因此每种选择的药水都确定性地贡献其整套统计数据。 

每天，我们都会获得所需的统计数据子集。 任务是选择一组药水（允许重复，但无关紧要，因为购买重复品永远不会有帮助），使其统计数据集的并集涵盖所有必需的统计数据，同时最小化总成本。 如果使用可用的药水类型无法覆盖所有必需的统计数据，我们输出 -1。 

关键的结构细节是 K ≤ 14，这立即意味着每个统计集都可以编码为长度最多为 14 的位掩码。这将问题推向“小宇宙的子集之上的子集”状态，其中 2^K 最多为 16384，小到足以允许对所有掩码进行动态编程。 

约束 N, M ≤ 2 × 10^4 排除了对所有部分的任何每次查询重新计算或任何试图通过组合搜索独立解决每一天问题的方法。 每天尝试组合药水子集的天真的方法将会呈指数级增长。 

当任何药水中根本不存在所需的统计数据时，就会出现微妙的边缘情况。 在这种情况下，答案始终是-1。 如果尝试通过每个统计数据的成本效率来贪婪地选择药水，则会出现另一种失败模式：由于药水任意重叠，贪婪的选择可能会阻止更便宜的组合。 

贪婪失败的一个小例子是，药水 A 覆盖成本 5 的 {1,2}，药水 B 覆盖成本 5 的 {2,3}，药水 C 覆盖成本 6 的 {1,3}。贪婪可能会选择 A 和 B 以成本 10 覆盖所有三个统计数据，但最佳选择是 C + A 或 C + B，具体取决于结构，并且在更复杂的情况下，贪婪可能会更严重地失败。 正确的解决方案必须对组合进行全局优化。 

## 方法

 暴力解释是独立对待每一天：枚举药水子集并计算其掩码的并集，跟踪覆盖所需掩码的最小成本。 这是正确的，因为它直接匹配问题的定义，但搜索空间是每天 2^N，在 N = 20000 时，这是一个天文数字。即使将子集限制到合理的大小也会失败，因为可能需要多少药水没有限制。 

关键的观察是我们不需要明确考虑药水的子集。 每个部分贡献一个固定掩码，并且组合部分对应于具有附加成本的掩码的按位或。 这是一个小宇宙上的经典“形成每个掩码的最小成本”，可以使用位掩码上的最短路径样式动态编程一次性解决所有掩码。 

我们将 dp[mask] 定义为精确实现该组统计数据的最低成本。 每个药水都充当过渡：从任何当前掩码，我们可以移动到掩码或 potion_mask，但需要额外的成本。 这形成了一个超过 2^K 个节点的图，每个节点重复 N 个转换。 我们不是在每个查询中运行完整的图搜索，而是使用位掩码空间上的多源最短路径计算 dp 一次。

然而，直接从每个状态进行 N 次转换太慢（2^K × N 次转换）。 关键的改进是用单独的药剂初始化 dp，然后使用经典子集 DP 或位掩码上的类似 Dijkstra 传播对所有掩码运行标准松弛。 由于 K 很小，我们可以将其视为 2^K 个节点上具有最多 K 位转换的最短路径，使用每个掩码的预计算药水列表或使用位操作上的 SPFA 式松弛进行优化。 公认的方法通常使用带有优先级队列的 0-1 式松弛结构或 BFS 式松弛，利用每次转换仅增加掩码。 

一旦计算出 dp，每个查询都会通过检查 dp[required_mask] 来回答。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | ---| --- | --- |
 | 每个查询的暴力破解 | O(M·2^N) | O(M·2^N) | O(1) | O(1) | 太慢了 |
 | 最佳位掩码最短路径| O((N + 2^K) · 2^K) | O((N + 2^K) · 2^K) | O(2^K) | O(2^K) | 已接受 |

 ## 算法演练

 1. 将每个药水的统计数据列表转换为长度为 K 的位掩码。这会将每个药水压缩为单个整数状态，该整数状态准确表示其覆盖的统计数据。 
2. 初始化一个大小为 2^K 且无穷大的 dp 数组，表示实现每个统计数据子集的最小成本。 
3. 直接设置 dp[mask] = 每个药水掩模的成本。 这说明购买单一药水始终是一个有效的策略。 
4. 对所有蒙版运行松弛过程。 对于每个当前可达的掩码，尝试应用每个药水掩码并更新组合掩码：

 new_mask = mask OR potion_mask，并且 dp[new_mask] = min(dp[new_mask], dp[mask] + cost_potion)。 

此步骤传播药水组合，确保最终构建任何多药水选择。 
5. 重复放松，直到无法改善为止。 由于掩码仅增加位数，因此该过程在子集的有限格上收敛。 
6. 对于每个查询日，将所需的统计信息转换为位掩码并输出 dp[required_mask]，如果保持无穷大则输出 -1。 

### 为什么它有效

 dp 状态代表所涵盖统计数据的每个子集的最知名成本。 每个有效的药水集都对应于从各个药水掩码开始的一系列 OR 运算。 由于 OR 在集合包含方面是关联且单调的，因此任何部分的组合都可以分解为重复的成对 OR 转换。 这种放宽保证了只要存在更便宜的掩码组合，它最终就会通过某些更新序列被发现。 由于成本只会降低并且掩模是有限的，因此该过程会收敛到每个子集的全局最优值。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

INF = 10**18

def main():
    N, M, K = map(int, input().split())
    
    potions = []
    for _ in range(N):
        n, c = map(int, input().split())
        arr = list(map(int, input().split()))
        mask = 0
        for x in arr:
            mask |= 1 << (x - 1)
        potions.append((mask, c))
    
    max_mask = 1 << K
    dp = [INF] * max_mask
    
    for mask, cost in potions:
        if cost < dp[mask]:
            dp[mask] = cost
    
    updated = True
    while updated:
        updated = False
        for mask in range(max_mask):
            if dp[mask] == INF:
                continue
            base_cost = dp[mask]
            for pmask, pcost in potions:
                new_mask = mask | pmask
                new_cost = base_cost + pcost
                if new_cost < dp[new_mask]:
                    dp[new_mask] = new_cost
                    updated = True
    
    for _ in range(M):
        m = int(input())
        arr = list(map(int, input().split()))
        mask = 0
        for x in arr:
            mask |= 1 << (x - 1)
        ans = dp[mask]
        print(-1 if ans == INF else ans)

if __name__ == "__main__":
    main()
```该解决方案首先将每个部分编码为位掩码，以便统计数据集的并集成为按位或运算。 dp 数组存储每个统计数据子集可实现的最小成本。 我们用单一药水播种 dp，然后反复放松过渡，将现有的可达掩模与任何药水相结合，产生可能更便宜的更大掩模。 重复循环是必要的，因为最佳解决方案可能需要两个以上的药剂，并且必须逐渐发现中间组合。 

查询阶段就变得很简单：每个所需的集合都被转换为位掩码并在 O(1) 查找中得到答案。 

一个微妙的点是，我们不会尝试跟踪“精确的 k 个药水”或在选择顺序上强制执行任何结构，因为成本累积自然会处理无重复的选择。 

## 工作示例

 ### 示例 1

 我们首先将药水转换成面具。 假设我们有掩码 P1、P2、P3 以及相关成本。 

我们用单一药水成本初始化 dp，然后传播组合。 

| 步骤| 考虑面膜| 行动| dp 更新 |
 | --- | --- | --- | --- |
 | 初始化| P1 | dp[P1]=c1 | dp[P1]=c1 | 集|
 | 初始化| P2 | dp[P2]=c2 | dp[P2]=c2 | 集|
 | 放松| P1 | P1 或 P2 | dp[P1∪P2]=最小值|
 | 放松| P2 | P2 或 P3 | dp[P2∪P3]=最小值|

 收敛后，dp 包含所有可达到的统计组合的最佳成本。 直接查找查询掩码，产生输出 4、4、8。 

这演示了如何通过重复的 OR 传播而不是显式枚举来形成多药组合。 

### 示例 2

 用更少的天数，相同的dp表就建一次。 

| 步骤| 面膜| dp值|
 | --- | --- | --- |
 | 初始化| 药水面膜| 基本成本|
 | 放松| 组合面罩| 更新 |
 | 查询 | 全面罩| 7 |

 这表明即使只存在一个查询，完整的预处理仍然可以正确捕获最佳组合。

 ## 边缘情况

 当所需的统计数据从未出现在任何药水中时，就会发生直接失败的情况。 在这种情况下，包含该位的所有 dp 掩码都保持无穷大。 该算法正确地返回 -1，因为没有松弛可以将丢失的位引入任何可达状态。 

另一种边缘情况是最佳解决方案需要组合两种以上的药剂。 由于 dp 不限于成对组合，而是在已形成的状态上反复松弛，因此中间掩模逐渐累积，直到达到完整集合。 贪婪的单步组合会错过这样的链，但迭代松弛确保了任意长度序列的闭合。 

最后的边缘情况是多个药水共享相同的掩模但成本不同。 初始化步骤仅保留最便宜的直接代表，随后的松弛保留最小的成本结构，确保重复不会扭曲转换。
