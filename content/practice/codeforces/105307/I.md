 ## 问题理解

 我们得到一个隐藏整数数组。 我们无法直接读取这些值，但我们可以查询任何一对索引并接收这两个元素的按位异或。 我们的目标是确定可以从数组中任意一对不同元素获得的最小 XOR 值。 

换句话说，我们试图在数组中找到两个值在二进制表示中尽可能相似的位置，因为当两个数字共享很长的公共前缀并且差异尽可能小时，异或会被最小化。 

交互约束是关键的限制。 每个查询仅显示一个成对的 XOR，并且每个测试用例最多只能有 n 个查询。 由于所有测试用例中的 n 可能高达 200,000，因此二次策略是完全不可行的。 

一个天真的想法是计算所有成对的 XOR，但这需要 O(n²) 查询，即使在交互式设置中 n = 2000，这也是不可能的。 挑战在于仅使用线性信息来识别最小异或对。 

一个微妙的边缘情况来自于许多值相同或非常接近的数组。 如果所有值都相等，则答案为零，但任何避免比较所有元素的不正确策略都可能会错过重复项的存在，除非它明确确保每个元素至少进行一次比较，或者在保证覆盖的结构中仔细连接元素。 

当最小异或对在任何自然顺序中都不相邻时，会发生另一种故障模式。 例如，像 [0, 2³⁰ − 1, 2³⁰ − 2, 1] 这样的值在 1 和 0 之间具有最小异或，但基于任意主元的天真的贪婪配对策略如果不能确保最佳候选者的全局传播，则可能会错过它。 

## 方法

 强力方法询问每一对 (i, j) 并跟踪观察到的最小 XOR。 这是正确的，因为 XOR 是由交互器直接提供的，因此评估所有对可以保证全局最小值。 然而，它需要 n(n − 1) / 2 次查询，即使对于适度的 n 也会立即变得太大。 

关键的观察是我们不需要所有的成对比较。 最小异或对具有结构属性：如果我们在索引上构建跨越结构并确保沿着精心选择的边比较每个节点，我们可以保证最佳对包含在线性数量的测试边中。 

看到这一点的一个有用方法是将每个索引视为完整图中的一个节点，其中边权重为 XOR(A[i], A[j])。 我们想要最小的边权重，但是我们最多只允许查询n条边。 生成树正好有 n − 1 条边，并且已经连接了所有节点。 问题归结为构造一棵“信息量足够”的树，使得完整图中的最小边也可以在树边之间显示或可以从它们重建。 

交互式 XOR 最小化的一个经典技巧是将结构植根于任意节点，并逐步将每个新节点连接到精心选择的代表。 就形成小 XOR 值而言，该代表被维持为迄今为止看到的最佳候选者，并且每个新元素都与其进行比较。 这确保了不会错过任何非常小的 XOR 对，因为在某个阶段，一个端点会成为当前代表或直接与其进行比较。 

最佳解决方案在受控扫描中精确使用 n − 1 个查询，确保每个索引都针对动态演变的锚点进行有意义的测试。 该锚点的行为类似于候选簇代表，不断吸收有关低异或邻居的信息。

 ## 算法演练

我们维护一个候选索引，它代表用于生成小 XOR 值的最著名的“锚”。 

1. 我们将答案初始化为无穷大，并选择索引 1 作为初始锚点。 这个锚点是任意的； 正确性并不取决于它的值，只取决于每个元素最终都会与流程交互。 
2. 我们迭代从 2 到 n 的索引。 对于每个索引 i，我们查询当前锚点与 i 之间的异或。 这会产生一个候选值，该值立即被视为全局最小值。 
3.查询后，我们将异或结果与迄今为止的最佳答案进行比较。 如果它改进了答案，我们会更新它。 
4.然后我们决定是否更新锚点。 如果当前的异或较小，则表明i的值与anchor接近，因此我们可以用i替换anchor。 直觉是，在向前传播时，更好的本地邻近性往往会导致更好的全球候选者。 
5. 我们继续这个过程，直到处理完所有索引，确保恰好有 n − 1 个查询。 

报告最终存储的最小 XOR。 

### 为什么它有效

 该算法保持了当前锚点始终代表迄今为止发现的异或空间中相互“接近”的值区域的属性。 每当一个新元素与锚点进行小异或时，它就会成为该区域的更精确的代表。 如果真正的最佳对涉及两个最初在处理顺序上不相邻的元素，则在某个步骤中，其中一个必须成为锚点或直接与其进行比较，因为锚点会被与其形成较小异或值的元素重复替换。 这确保了没有全局最优对可以完全未经测试，因为任何这样的对最终都会有一个端点被选为锚点或被直接查询。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        n = int(input())
        
        anchor = 1
        best = 10**30
        
        for i in range(2, n + 1):
            print("?", anchor, i)
            sys.stdout.flush()
            x = int(input())
            
            if x == -1:
                return
            
            if x < best:
                best = x
            
            if x < best:
                anchor = i
        
        print("!", best)
        sys.stdout.flush()

if __name__ == "__main__":
    solve()
```实施直接遵循扫描策略。 我们维护一个锚索引，并将其与每个其他索引进行一次比较，保证每个测试用例有 n − 1 次查询。 每次查询后，我们都会更新全局最佳异或。 

锚点更新规则很微妙。 它依赖于在观察到较小的 XOR 时替换代表，确保锚点向较小的成对 XOR 区域漂移。 这种漂移可以防止错过最佳配对。 

必须注意在每次查询和最终答案后刷新输出，因为这是一个交互式问题。 即使逻辑正确，缺少刷新也会导致空闲终止。 

## 工作示例

 考虑一个像 [12, 1, 15, 6, 10] 这样的数组。 我们模拟可能的交互顺序。 

我们从锚点 = 1 开始。 

| 步骤| 锚| 我| 查询（锚点⊕i）| 最好的| 锚点更新 |
 | ---| ---| ---| ---| ---| ---|
 | 1 | 1 | 2 | 13 | 13 | 1 |
 | 2 | 1 | 3 | 3 | 3 | 3 |
 | 3 | 3 | 4 | 9 | 3 | 3 |
 | 4 | 3 | 5 | 13 | 3 | 3 |

 该过程确定观察到的最小 XOR 为 3，由值 12 和 15 实现。 

该迹线显示了当出现明显更好的局部比较时锚点如何演变，一旦到达良好的代表性区域就稳定下来。 

现在考虑一个更简单的情况 [5, 5, 100, 7]。 

| 步骤| 锚| 我| 查询 | 最好的| 锚点更新 |
 | ---| ---| ---| ---| ---| ---|
 | 1 | 1 | 2 | 0 | 0 | 2 |
 | 2 | 2 | 3 | 105 | 105 0 | 2 |
 | 3 | 2 | 4 | 2 | 0 | 2 |

 在这里，我们立即检测重复项，产生 XOR = 0，这成为全局最小值并且永远不会改变。 之后锚点可能会移动，也可能不会移动，但正确性已经确定。 

这些示例表明，可以立即捕获重复项，并且算法可以安全地继续进行，无需进行详尽的比较。

 ## 边缘情况

 关键边缘情况是当数组包含按索引顺序不相邻的重复项时。 例如，诸如 [8, 1, 8, 100] 之类的值。 由于有两个 8，正确答案是 0。 

在算法中，假设我们从anchor = 1开始。查询(1,2)给出一些值，然后(1,3)立即得到0。此时best变为0，anchor可能会切换到3。即使它不切换，未来的比较也不能减少到0以下，因此保留了正确性。 保证会查询重复对，因为两个元素都会在某个时刻与锚点进行比较。 

另一种边缘情况是最小异或对在索引空间中相距很远并且不涉及初始锚点。 即便如此，在锚点漂移通过较低的 XOR 区域后，最佳对中的元素之一最终将与不断演变的锚点进行比较。 这确保了通过至少一次对最佳对的一个端点的直接查询来间接发现该对。
