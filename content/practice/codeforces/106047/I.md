 ## 解决方案
 ## 问题理解

 我们得到了一系列值，这些值被一个一个地插入到最初为空的二进制堆数组中。 每次插入都使用标准的“向上筛选”过程：新元素附加在末尾，然后在违反堆属性的情况下与其父元素重复交换。 

不同之处在于堆属性不是固定的。 对于每次插入，堆被视为最小堆或最大堆，具体取决于隐藏的二进制字符串。 如果该位为零，则该结构在插入期间的行为类似于最小堆，这意味着父值不得大于子值。 如果该位为 1，则其行为类似于最大堆，这意味着父值必须不小于子值。 

在所有插入之后，我们得到结果数组，它保证是插入值的排列，但不一定是全局任一解释下的有效堆，因为不同的插入可能使用不同的堆类型。 任务是重建一个二进制字符串，解释每次插入是如何执行的，以便最终的数组正是给定的数组。 如果多个字符串有效，我们必须输出字典顺序最小的一个，最好先输出零。 如果不存在有效序列，我们必须报告不可能性。 

这些约束允许跨测试用例最多插入一百万次，因此任何解决方案在每个测试中都必须基本上是线性或接近线性的。 任何模拟二进制字符串的所有选择的尝试都是指数级的并且立即不可行。 即使天真地尝试回溯插入类型也会导致每一步都有分支，这远远超出了限制。 

当对单一插入类型的贪婪决策随后阻碍了可行性时，就会出现微妙的失败情况。 例如，如果一个值被强制提前插入最大堆，它可能会爬得太高，并使以后所需的堆配置变得不可能。 另一个边缘情况是出现相等的值时：两种堆类型在相等方面的行为相同，因此多个字符串可能有效，并且字典最小性成为主要约束。 

## 方法

 一个自然的起点是在猜测二进制字符串的同时尝试向前模拟该过程。 对于每个插入位置，我们可以尝试最小堆和最大堆行为，运行 sift-up 操作，并检查是否仍然可以到达最终数组。 这立即表明对由当前堆配置和位置组成的状态进行回溯或动态编程。 然而，堆本身可以在决策中具有多种阶乘配置，并且每次插入都可以将元素向上移动对数步。 即使剪枝也是无效的，因为两个不同的决策序列可能会导致相同的前缀结构，但随后会出现分歧。 这使得暴力方法以 n 为指数。 

观察的关键是扭转观点。 我们可以将最终数组解释为确定性筛选路径的端点，而不是考虑每次插入如何修改堆。 每个插入的值都遵循从其插入位置到某些祖先位置的路径，直到由于比较条件而停止。 停止条件仅取决于我们在该步骤中是否强制执行最小或最大行为。 

因此，每个元素的行为在插入时限制了沿着其到根的路径的一段比较。 如果我们考虑向后重建过程，我们实际上是为每个插入分配一个类型，该类型必须与最终数组结构所引起的祖先关系的最终顺序一致。

关键的减少是我们不需要显式地重建中间堆。 我们只需要确保对于每个插入 i，我们可以选择最小或最大行为，以便 v_i 在最终数组中的最终放置与沿着插入路径的向上比较的有效序列兼容。 这成为由堆索引引起的固定隐式树结构上的可行性分配问题，其中每个节点必须满足两个可能的单调约束之一。 

这将问题转化为决定，对于每个 i，我们是否可以强制执行最小类型或最大类型插入，以便沿其插入路径的所有比较都与最终数组一致。 在不破坏可行性的情况下，通过贪婪地优先选择最小类型（零）来获得按字典顺序排列的最小字符串。

 ## 算法演练

 我们按顺序处理插入，同时维护如果我们遵循先前元素的固定类型选择将获得的堆结构。 关键是我们不是以简单的方式向前重建堆，而是模拟最终的堆配置并在分配类型时强制执行一致性约束。 

1. 我们从一个空堆结构开始，按顺序处理元素 v1 到 vn。 每个元素都放置在堆数组中的位置 i 处，我们从概念上模拟其向上筛选路径。 
2. 对于每个插入 i，我们首先尝试将其分配为零类型（最小堆行为）。 我们仅沿着其路径模拟上移比较：当节点不是根时，我们检查交换是否符合最小堆规则，这意味着父节点必须小于或等于停止点处的当前节点。 我们使用最终的数组值作为这些比较的约束。 
3. 如果最小类型赋值导致与最终数组结构矛盾，意味着沿路径存在违反最小堆条件的所需比较，我们拒绝此选择。 
4. 如果 min-type 无效，我们分配类型一（最大堆行为）。 我们类似地沿着相同的插入路径进行验证，但现在条件相反：在停止条件下，父级必须大于或等于子级。 
5. 如果两个赋值都无效，我们得出结论，没有二进制字符串可以生成给定的最终数组，因为没有一致的堆规则可以解释该元素如何到达其位置。 
6. 我们记录每次插入所选择的位，并在概念上接受这定义了有效的插入过程。 

重要的微妙之处在于，每个插入路径仅取决于堆树中的祖先索引，而不取决于全局重新排列。 因此，验证减少为检查每次插入最多 O(log n) 次比较的链。 

### 为什么它有效

 在任何时刻，堆结构仅由先前的插入及其最终位置决定。 元素 i 的插入仅与其祖先链上的节点交互。 最终的数组对每个这样的链上的总排序进行编码，与必须发生的交换序列一致。 选择类型对应于沿该链强制执行单调条件。 因为一旦比较固定，交换就是确定性的，因此任何有效的类型分配都唯一地确定堆的演变。 尽可能贪婪地选择 min-type 可以保留未来的可行性，因为 min-type 施加的排序约束比 max-type 更弱，因此为以后的插入留下了更大的灵活性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def check(v, a, i, is_max):
    x = i
    val = v[i]
    while x > 1:
        p = x // 2
        if is_max:
            if a[p] < val:
                return False
            if a[p] == val:
                break
        else:
            if a[p] > val:
                return False
            if a[p] == val:
                break
        x = p
    return True

def solve():
    n = int(input())
    v = [0] + list(map(int, input().split()))
    a = [0] + list(map(int, input().split()))

    b = ['0'] * (n + 1)

    for i in range(1, n + 1):
        if check(v, a, i, False):
            b[i] = '0'
        elif check(v, a, i, True):
            b[i] = '1'
        else:
            print("Impossible")
            return

    print("".join(b[1:]))

t = int(input())
for _ in range(t):
    solve()
```该代码独立处理每个插入，首先尝试将其验证为最小堆插入。 这`check`函数从插入位置沿着父链向上走，验证沿路径的比较没有与堆规则相矛盾。 一旦出现矛盾，这种类型就被拒绝。 相等充当停止条件，因为一旦遇到相等的值，任一堆类型都无法再强制交换超过该点，而不违反最终配置所暗示的稳定性。 

一个微妙的实现细节是平等的处理。 一次`a[p] == val`，无论堆类型如何，筛选都可以在那里停止，所以我们提前中断。 这可以防止过度限制路径。

 ## 边缘情况

 第一种边缘情况是所有值都相同。 每次插入都满足任何路径上的最小和最大堆条件，因此按字典顺序最小的输出是全零的字符串。 该算法在每一步都正确接受 min 类型，因为相等永远不会违反 min 条件并立即停止向上移动。 

另一种边缘情况是最终数组与插入顺序完全对应。 在这种情况下，每个元素都已经位于正确的堆位置而无需交换，因此每次插入都必须与立即停止一致。 两种堆类型都允许这样做，因此贪婪选择总是选择零。 

最后一种边缘情况是，只有一种插入类型全局可行，但在局部看来，这两种插入类型对于早期步骤都有效。 除非被迫，否则贪心策略会避免采用 max-type，并且由于 max-type 引入了更严格的向上约束，因此任何过早的选择都会消除灵活性。 该算法通过尽可能推迟到最小类型来避免这种情况，并且仅在祖先链验证失败时才进行切换。
