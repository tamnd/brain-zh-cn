 ## 解决方案
 ## 问题理解

 我们得到一个二进制字符串，其中一些位置已经固定为`0`或者`1`，而其他的则未知并标记为`?`。 我们必须更换每一个`?`与任一`0`或者`1`。 填充字符串后，我们查看相邻对并计算连续字符不同的次数。 该计数必须完全等于目标值`k`。 

在所有有效的完成中，我们不仅仅被要求提供任何解决方案。 我们必须构造按字典顺序最小的结果二进制字符串。 

一个关键的解释转变是，目标不是全局计数 1 或 0，而是控制相邻位置之间的转换。 每个位置对字典顺序都有独立的贡献，而约束通过翻转的次数耦合相邻的位置。 

约束允许最多$n = 10^5$每次测试和总计$10^6$总的来说，因此任何解决方案对于每个测试用例都必须基本上是线性的。 对所有分配的二次或状态扩展方法立即不可行，因为每个`?`搜索空间加倍，使暴力破解呈指数级增长。 

当贪婪地从左到右分配字符而不跟踪未来的可行性时，会出现微妙的边缘情况。 例如，如果我们总是选择`'0'`对于一个`?`为了最大限度地减少字典顺序，我们最终可能会被迫超过或低于所需的转换数量。 当我们尝试在本地修复转换而不考虑更改一个字符会影响两个边界时，会出现另一种失败情况，因此决策不是独立的。 

真正的困难在于，每个位置仅通过邻接对转换计数做出贡献，因此问题变成了具有全局变化预算的受限路径构建。 

## 方法

 暴力方法会尝试每一个分配`?`位置并计算相邻不匹配的数量。 如果有$m$未知数，这探索$2^m$对于每种可能性，我们扫描其中的字符串$O(n)$, 给予$O(n 2^n)$在最坏的情况下。 这远远超出了任何可行的限度。 

关键的观察结果是，转换的数量仅取决于我们是否选择与前一个位置相同或不同的位。 这表明了一种动态编程解释，其中状态由最后选择的位以及我们仍需要实现的转换数量决定。 

我们可以考虑从左到右构造字符串。 在每个位置，我们选择一个与约束一致的值，并跟踪我们已经使用了多少个转换。 按字典顺序排列的最小要求表明我们应该始终尝试`'0'`前`'1'`，但前提是它不会使最终目标变得不可能。 因此可行性检查变得至关重要。 

我们定义一个 DP，在给定当前位置、最后一个字符和剩余转换预算的情况下跟踪是否可以完成后缀。 然而，我们不是在所有状态上进行完整的 DP，而是通过注意到可行性仅取决于剩余长度和剩余转换来压缩它：在位置`i`，如果我们放置一点，我们就确切地知道仍然可以实现多少个转换。 

这导致了带有可行性检查的贪婪构造：在每个位置，我们暂时分配`'0'`或者`'1'`，计算这消耗了多少个转换，并验证剩余的所需转换是否可以在剩余的后缀中实现。 后缀可行性简化为一个简单的界限：`len`剩余位置，可能的转换数量范围为`0`到`len-1`。 

因此，在每一步中，我们都会维护当前的转换和先前的字符，并确保剩余的所需内容`k`保持在可行的范围内。

 | 蛮力 |$O(n 2^n)$|$O(n)$| 太慢了|
 | 贪婪+可行性|$O(n)$|$O(1)$| 已接受 |

 ## 算法演练

 我们从左到右构建字符串，同时维护两条信息：先前选择的字符以及到目前为止已形成的转换数量。 

1. 初始化答案构造。 我们将逐个字符构建结果。 一旦确定了前一个字符，我们也会对其进行跟踪。 对于第一个位置，没有前面的字符，所以我们只选择一个与可行性一致的值。 
2. 对于每个位置`i`，确定候选值。 如果`s[i]`是固定的，我们只考虑这个值。 如果是的话`?`，我们尝试`'0'`首先和`'1'`第二是确保按字典顺序排列的最小输出。 
3. 对于每个候选值，计算如果我们将其放在这里将添加多少个转换。 如果没有先前的角色，则该贡献为零。 否则的话，就是`1`如果候选字符与前一个字符不同，否则`0`。 
4. 更新暂定转换计数并计算剩余所需转换：`k - used_so_far - new_transition`。 
5. 检查后缀的可行性。 在位置`i`， 有`n - i - 1`剩余边数，因此未来转换的最大数量为`n - i - 1`最小值是`0`。 如果剩余的所需转换位于该间隔之外，则该候选无效。 
6. 如果候选通过了可行性，我们会修复它，更新当前转换计数和前一个字符，然后继续。 
7、如果某个岗位无人任职，则无法施工。 

### 为什么它有效

 在每一步中，我们都保留前缀是固定的不变式，并且仍然可以通过后缀的某些完成来实现所需的剩余转换数量。 由于每个放置仅影响通过其与前一个字符的边界的转换计数，因此后缀的可行性仅取决于剩余的边数，而不取决于早期的结构。 这确保了任何保留可行性的局部有效选择不会消除所有全局解决方案，并且选择`'0'`首先保证所有可行完成中按字典顺序排列的最小输出。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        n, k = map(int, input().split())
        s = input().strip()

        res = []
        prev = None
        used = 0

        for i in range(n):
            candidates = []
            if s[i] == '?':
                candidates = ['0', '1']
            else:
                candidates = [s[i]]

            chosen = None

            for c in candidates:
                add = 0
                if prev is not None and c != prev:
                    add = 1

                new_used = used + add
                remaining = k - new_used
                rem_len = n - i - 1

                if 0 <= remaining <= rem_len:
                    chosen = c
                    used = new_used
                    prev = c
                    res.append(c)
                    break

            if chosen is None:
                print("Impossible")
                break
        else:
            if used == k:
                print("".join(res))
            else:
                print("Impossible")

if __name__ == "__main__":
    solve()
```解决方案严格从左到右进行。 对于每个位置，我们仅接受一个字符，如果它不会使剩余的目标无法到达。 可行性检查使用以下事实：剩余的转换可以在之间的任何位置`0`以及剩余边的数量。 

一个微妙的实现细节是第一个字符的处理：由于没有前一个字符，我们避免计算转换。 另一个重要的细节是使用`for-else`结构，确保我们能够清楚地区分成功完成和早期失败。

 ## 边缘情况

 一种极端情况是弦完全固定。 在这种情况下，算法永远不会分支，只检查转换计数是否匹配`k`。 用于输入`s = "0101"`和`k = 3`，遍历直接将转换计算为`3`，因此该字符串无需修改即可被接受。 

当出现另一种边缘情况时`k = 0`。 唯一有效的输出是常量字符串。 用于输入`????`，算法总是选择`'0'`因为它永远不会违反可行性，并且由于不允许进行任何转换，因此每一步都会强制执行相同的字符。 

最后一个边缘情况是`k = n - 1`，这意味着每个相邻的对必须不同。 这迫使出现交替模式，并且可行性检查确保一旦在位置 0 做出选择，每个后续位置都是唯一确定的。
