 ## 解决方案
 ## 问题理解

 给定两个长度相等的数组，我们希望将索引范围从 1 到 n 切成几个连续的段。 每个索引必须恰好属于一个段，并且这些段必须覆盖整个数组，没有间隙或重叠。 

对于每个段，我们将第一个数组中的值之和与第二个数组中的值之和进行比较。 仅当该段上的 a 之和至少等于同一段上的 b 之和时，该段才被视为有效。 任务是将数组分成尽可能多的有效段。 

输出是此类分段的最大数量，如果无法进行单个有效分段，则输出为 -1。 

这些限制以一种直接的方式发挥作用。 所有测试用例的 n 总和最多为 5000，这意味着每个测试用例的 O(n²) 解决方案是可以接受的。 如果直接在多个测试中实现，任何立方或更糟的东西都会太慢。 这也表明动态编程或对前缀的贪婪扫描可能就足够了。 

当整个数组不满足条件时，就会出现微妙的边缘情况。 如果 a 的总和小于 b 的总和，则任何分段都不起作用，因为每个分区都包括将整个数组分成多个部分，而这些部分的总和仍然等于全局差值。 例如，如果a全为0，b全为1，则没有任何线段可以满足不等式，所以答案一定是-1。 

当有效段存在但过早贪婪切割失败时，会发生另一种棘手的情况。 例如，如果早期前缀在 (a − b) 方面稍微为负，但后来的元素进行补偿，我们必须确保不会在妨碍形成未来有效段的位置强制剪切。 

## 方法

 一个自然的起点是考虑检查所有可能的分区。 对于每种将数组切割为 k 段的方式，我们验证每个段是否满足求和条件。 这很快就变得不可行，因为分区的数量随着 n 呈指数增长，甚至重复计算段总和也会导致二次或更严重的开销。 

更好的方法是关注前缀差异。 定义一个数组 c，其中 c[i] = a[i] − b[i]。 线段 [l, r] 的条件为该线段上的 c 之和非负。 所以问题就变成了将数组分割成最大数量的连续段，每个段的总和为非负数。 

这种重新制定是强大的，因为它消除了分别跟踪两个阵列的需要，并减少了管理运行平衡的一切。 

关键的观察是，我们希望一旦运行总和在之前为负值或零之后再次变为非负值，就进行削减。 如果我们尽可能延长一个段，同时保持以非负总数结束它的能力，我们就可以最大化段的数量。 这将问题变成了贪婪扫描：维护一个运行总和，每当它达到正值或零值时，我们就可以安全地关闭一个段并重置。 

正确的直觉是，延迟剪切永远不会有助于增加片段的数量。 任何具有非负和的前缀都可以作为有效的段端点，进一步扩展它只会面临失去潜在的早期切割的风险。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力破解分区 | O(2ⁿ·n) | O(2ⁿ·n) | O(1) | O(1) | 太慢了|
 | 贪婪前缀扫描 | O(n) | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们在处理过程中将数组隐式转换为单差数组。 然后我们从左到右扫描，同时保持我们目前有多少盈余的运行平衡。

1. 初始化运行变量 current_sum = 0 和计数器分段 = 0。运行和跟踪当前段内的总计 (a − b)。 
2. 从索引 1 到 n 迭代数组，通过添加 a[i] − b[i] 来更新 current_sum。 这维持了我们目前正在形成的部门的净平衡。 
3. 每次更新后，检查current_sum是否非负。 如果是，我们决定关闭该索引处的当前段，并将段增加 1。然后，我们将 current_sum 重置为 0，以从下一个索引开始一个新段。 
4.处理完所有元素后，如果我们从未形成任何段，即段保持为零，则输出-1。 否则输出段。 

当 current_sum 变为非负值时立即削减的决定是最大化未来灵活性的贪婪选择。 它确保较早的段在仍然有效的同时尽可能短，从而留下更多元素可用于形成附加段。 

### 为什么它有效

 运行总和表示在给定点结束段的可行性。 任何时候总和变为非负数，我们就有一个有效的段。 如果我们延迟此切割，我们只会将更多元素积累到同一段中，这不能增加未来有效段边界的数量，因为我们正在消耗可能形成较早有效端点的索引。 因此，贪心策略总是保留或增加可能的段数，而不是减少它。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        n = int(input())
        a = list(map(int, input().split()))
        b = list(map(int, input().split()))
        
        current_sum = 0
        segments = 0
        
        for i in range(n):
            current_sum += a[i] - b[i]
            if current_sum >= 0:
                segments += 1
                current_sum = 0
        
        if segments == 0:
            print(-1)
        else:
            print(segments)

if __name__ == "__main__":
    solve()
```代码直接遵循贪心解释。 差异的转换以 a[i] − b[i] 的形式内联完成，避免了额外的内存。 关键的精妙之处在于形成段后current_sum的重置，这保证了每个段是独立的。 

条件 current_sum >= 0 允许我们关闭一个段。 我们并不严格等待它变为正数，因为零和段也是有效的。 

## 工作示例

 考虑一个简单的情况，其中数组逐渐积累正余额。 

输入：

 n = 5

 a = [2, 2, 2, 2, 2]

 b = [1, 1, 1, 1, 1]

 | 我| a[i]-b[i] | a[i]-b[i] | 当前总和| 细分 |
 | --- | --- | --- | --- |
 | 1 | 1 | 1 | 1 |
 | 2 | 1 | 0 | 2 |
 | 3 | 1 | 1 | 3 |
 | 4 | 1 | 0 | 4 |
 | 5 | 1 | 1 | 5 |

 每个前缀都是非负的，因此每个元素都成为自己的元素或最小段的一部分，从而产生最大的切割次数。 

现在考虑一个更平衡的情况。 

输入：

 n = 4

 a = [1, 1, 1, 1]

 b = [2, 2, 0, 0]

 | 我| a[i]-b[i] | a[i]-b[i] | 当前总和| 细分 |
 | --- | --- | --- | --- |
 | 1 | -1 | -1 | 0 |
 | 2 | -1 | -2 | 0 |
 | 3 | 1 | -1 | 0 |
 | 4 | 1 | 0 | 1 |

 只有到最后前缀才有效，所以我们只能组成一个段。 这表明该算法正确地延迟了切割直到实现可行性。

 ## 边缘情况

 完全负的阵列差异清楚地表明了故障情况。 

输入：

 n = 3

 a = [1, 1, 1]

 b = [2, 2, 2]

 运行总和变成 -1, -2, -3 并且永远不会恢复。 没有段可以闭合，因此段保持为零，我们输出−1。 该算法正确地避免了产生错误的分段，因为它仅在达到有效的非负边界时才对分段进行计数。 

零差异情况显示最大分割。 

输入：

 n = 3

 a = [5, 5, 5]

 b = [5, 5, 5]

 每个步骤都将 current_sum 保持为零，因此每个索引都会关闭一个段。 结果是 3 段，确认零被视为有效，并且贪婪重置不会失去最优性。
