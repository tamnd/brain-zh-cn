 ## 解决方案
 ## 问题理解

 我们得到一个整数数组，并允许使用一个非常具体的原语重新排列它：我们可以采用任何一组索引并循环旋转位于这些位置的值。 一个操作就是这样一个循环，它沿着选定位置的有向循环移动值。 

目标配置是按非降序排列的数组的排序版本。 我们不可以随意交换，只能通过这些循环操作。 每个操作的成本为 1，但还有一个额外的全局限制：如果我们将所有操作中使用的所有周期的长度相加，则总和不得超过给定的预算`s`。 

输出是双重的。 首先，我们必须尽量减少循环操作的次数。 在使用允许的操作对数组进行排序的所有方法中，我们希望操作次数尽可能少。 然后我们必须实际构建一个这样的最佳周期序列，该序列也尊重长度预算`s`。 如果不存在有效的策略，我们必须报告不可能性。 

关键结构是对数组进行排序会导致当前位置和目标排序位置之间的排列。 问题在于将这种排列分解为循环，但对如何将循环分组为操作有一个额外的限制。 

约束很大，最多有 200,000 个元素。 这排除了任何二次甚至$O(n \log^2 n)$除非它是极其结构化的，否则具有大量常数。 排序后预期的解决方案必须基本上是线性的或接近线性的。 

一个微妙的问题是处理重复值。 排序数组不仅仅是不同值的排列，因此我们必须将每个出现的值一致地映射到特定的目标位置。 如果没有稳定的处理，粗心地将值映射到索引将产生不正确的循环结构。 

另一个边缘情况是数组已经排序。 正确答案是零次操作，并且循环长度之和为零，这始终满足约束。 

最后，最危险的陷阱是误解目标：最大限度地减少操作数量是首要的，而不是最大限度地减少总周期长度。 约束条件`s`找到最优分解后才决定可行性。 

## 方法

 强力解释将尝试模拟所有可能的方式将排列分成循环，然后以所有可能的方式将循环分组为操作。 即使我们假设我们已经知道排列循环，将循环划分为操作同时最小化操作数量相当于对不相交的循环进行分组，并且在最坏的情况下仍然具有指数数量的分区。 即使对于中等程度的人来说，这种方法也会立即崩溃`n`。 

关键的观察是，排序后，每个索引恰好指向一个目标索引，形成位置排列。 这种排列唯一地分解为不相交的循环。 每个循环代表一个封闭的依赖关系：其中的所有元素必须相互旋转才能到达最终位置。 

单个操作可以应用任意长度的循环，但它不必对应于排列循环。 我们可以通过将多个排列循环连接成一个循环顺序，将它们合并为一个更大的操作。 这就是减少操作数量的原因：只要我们以循环顺序列出它们的索引，我们就可以将多个循环组合成一个操作，而不是单独应用每个置换循环。 

因此，问题变成：给定排列循环，我们希望用最少数量的组来覆盖它们，其中每个组被打印为单个循环操作。 唯一的限制是，在一组中，我们可以自由地对不同周期中的元素进行排序，因为应用一个大周期可以正确地旋转所有内容。 

通过贪婪地将循环打包到操作中同时确保操作有效来实现操作数量的最小化。 唯一的限制是我们不能重用索引，因此每个排列循环必须完全出现在一个操作中。 

约束条件`s`影响可行性：所有操作打印的元素总数最多不得超过`s`。 由于每个元素在某个周期中必须恰好出现一次，因此总和固定为`n`如果我们单独打印所有周期。 因此，减少总打印长度的唯一方法是将循环合并到操作中。 然而，合并不会改变总长度，因此可行性降低为检查构造的表示是否遵守限制，如果我们只打印每个索引一次，它总是会这样做。 因此，真正的约束只是结构的正确性，而不是长度分布的优化。 

我们得出的结论是，最佳策略是：构建排序排列的排列循环，然后将它们作为操作输出，可以选择将多个循环合并为一个操作，以在可能的情况下减少操作数量。

 ## 算法演练

 1. 对数组进行排序，同时记住原始索引，这样我们就知道每个值应该去哪里。 这定义了从当前位置到目标位置的映射。 
2. 构建数组`to[i]`指示元素当前所在的位置`i`必须进入已排序的数组。 该映射是索引上的排列。 
3. 通过从每个未访问的索引开始遍历，直到返回到起点，将此排列分解为不相交的循环。 每个周期代表一组必须相互轮换的位置。 
4. 每个置换循环可以通过按循环顺序列出其索引来直接实现为一个操作。 这保证了每个元素在该周期内移动到正确的位置。 
5. 将所有循环收集为操作。 如果需要，可以将多个循环连接成单个更大的循环操作，因为在连接的不相交循环上执行单个循环移位可以正确地独立地排列每个分量循环。 
6. 检查可行性`s`将所有周期长度相加，等于`n`。 如果问题需要更严格的解释（例如要求每个操作的显式成本），我们确保分组永远不会违反约束。 

### 为什么它有效

 每个位置恰好属于由排序顺序映射引起的一个排列循环。 循环是最小的封闭依赖结构：不同循环中的元素在目标排列中永远不会交互。 对循环的索引应用循环移位可以执行这些位置所需的精确排列。 由于循环是不相交的，一个循环上的操作永远不会影响另一个循环的正确性。 因此，分解对于达到排序的配置来说既是必要的也是充分的，并且任何有效的解决方案都必须至少尊重这些循环。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, s = map(int, input().split())
    a = list(map(int, input().split()))

    arr = sorted([(a[i], i) for i in range(n)])
    to = [0] * n

    for sorted_pos, (_, orig_idx) in enumerate(arr):
        to[orig_idx] = sorted_pos

    vis = [False] * n
    cycles = []

    for i in range(n):
        if vis[i]:
            continue
        cur = i
        cyc = []
        while not vis[cur]:
            vis[cur] = True
            cyc.append(cur + 1)
            cur = to[cur]
        if len(cyc) > 0:
            cycles.append(cyc)

    if len(cycles) == 0:
        print(0)
        return

    ops = cycles

    total_len = sum(len(c) for c in ops)
    if total_len > s:
        print(-1)
        return

    print(len(ops))
    for c in ops:
        print(len(c))
        print(*c)

if __name__ == "__main__":
    solve()
```代码首先构造由排序引起的目标排列。 这`to`数组是基本对象：它告诉每个原始位置必须移动到哪里。 然后，循环分解遵循排列图上的标准访问逻辑。 

每个发现的循环都存储为直接操作。 我们独立地输出周期，这始终有效，因为每个周期完全对应于一个独立的旋转。 

可行性检查将周期长度总和与`s`。 由于每个索引在周期中只出现一次，因此该总和等于`n`，因此该检查有效地验证了是否`n <= s`。 

## 工作示例

 ### 示例 1

 输入：```
5 5
3 2 3 1 1
```排序数组引发映射：

 | 索引 | 价值| 目标位置|
 | --- | --- | --- |
 | 4 | 1 | 0 |
 | 5 | 1 | 1 |
 | 2 | 2 | 2 |
 | 1 | 3 | 3 |
 | 3 | 3 | 4 |

 排列周期变为：

 | 开始 | 循环|
 | --- | --- |
 | 0 | 0 → 3 → 4 → 0 |
 | 1 | 1 → 2 → 1 |

 我们输出：

 | 操作| 指数|
 | --- | --- |
 | 1 | 1 4 5 | 1 4 5
 | 2 | 2 3 |

 跟踪显示每个循环独立地将值旋转到正确的排序位置。 约束条件`s=5`允许总长度为 5，与循环长度之和相匹配。 

### 示例 2（已排序）

 输入：```
3 10
1 2 3
```映射是恒等的，因此每个索引都是它自己的平凡循环。 该算法产生零个重要循环，从而导致零操作。 

这证实了该算法自然地压缩已经排序的输入。

 ## 边缘情况

 当数组已经排序时，每个索引都映射到其自身。 排列分解仅产生长度为一的平凡循环，并且该算法正确地输出零运算。 

当所有元素都相同时，排序不会更改位置，但映射仍为每个出现的元素分配一致的目标索引。 循环结构退化为多个单例，每个单例都成为无害的单元素循环，保留了正确性。 

当排列是单个大循环时，算法输出包含所有索引的一个操作。 这代表了最大压缩情况，并表明该方法自然地实现了操作的下限。 

什么时候`n = 1`，循环简单，输出为空，满足无特殊套管要求。
