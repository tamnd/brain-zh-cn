 ## 解决方案
 ## 问题理解

 我们得到一系列排列成行的石头，每块石头都有颜色和重量。 第一个结构观察是相同颜色的连续宝石可以被压缩：在相同颜色的任何最大块内，只有该块中的最大重量才重要。 其他一切都受到支配，永远无法促成最佳策略。 

压缩后，相邻的石头始终具有不同的颜色，因此该阵列的行为就像颜色交替的序列。 问题中描述的过程简化为在约束下重复移除石头，这实际上意味着只有内部移除才能产生价值，而边界石头不能以相同的方式进行计数。 

关键的组合主张是，在长度为 N 的数组中，在尊重所有约束后，实际对得分做出贡献的棋子数量受到可以通过删除“配对”的内部位置数量的限制，结果最多是 (N − 2) / 2 的上限。最左边和最右边的棋子在结构上被排除在以相同方式做出贡献的范围内，因此只有内部结构很重要。 

我们想要的输出是在最佳游戏下可获得的最大总重量，该问题简化为选择最多该大小的子集，但有一个重要的保证，即可以通过有效的策略实现对那么多内部棋子的任何选择。 

一种简单的方法会尝试模拟删除、重新计算相邻合并并搜索操作序列。 这很快就会呈指数增长，因为每次删除都会改变邻接关系和潜在的未来选择。 

当 N 很小时，就会出现边缘情况。 如果 N 等于 1 或 2，则不存在内部元素，因此答案必须为零。 如果N等于3，则恰好有一颗内部石头，但它是否可用取决于规则，并且声明证实它是微不足道的。 

一个更微妙的边缘情况是所有权重都相等。 贪婪模拟可能会建议进行许多删除，但结构界限仍然限制了仅选择固定数量的元素的答案，而与权重均匀性无关。 

## 方法

 暴力解释将问题视为游戏：在每一步，我们选择一块可移动的石头，将其移除，如果需要的话合并邻居，并跟踪所有可能的结果状态。 这导致了一个状态空间，其中每个状态都取决于当前序列和先前的合并。 即使使用记忆化，不同配置的数量也会呈指数增长，因为每次删除都会以不同的方式改变邻接和折叠段。 对于 N 大约 40 来说，这已经是不可行的。 

关键的简化来自于完全忽略动态结构并专注于结构不变量：压缩相同颜色的线段后，唯一有意义的自由度来自内部位置，并且无论如何执行移除，最多有一半的内部位置可以做出贡献。 这将问题从动态过程转变为静态选择问题。 

关键的见解是，该过程保证我们可以将每一个收益“记入”不同的内部宝石中，并且记入的宝石数量不会超过 ⌈(N − 2) / 2⌉ 。 此外，声明中的建设性论点表明，对这么多棋子的任何选择都是可以实现的，因此最佳游戏相当于从内部位置中选择最大可能的权重。 

因此，解决方案变为：对相关权重进行排序并选择最大的 k，其中 k = ⌈(N − 2) / 2⌉。

 ## 算法演练

我们首先压缩输入，以便合并相同颜色的连续石头，仅保留每个块中的最大权重。 此步骤确保段内的进一步简化无法改善答案，因为单色块内的任何非最大元素都受到严格控制。 

压缩后，我们计算缩减后的数组中石子的数量 N。 如果 N 小于或等于 2，则不存在有效的内部贡献，因此答案立即为零。 

然后我们计算有多少石头可能贡献。 该值为 k = (N − 2 + 1) // 2，对应于 ⌈(N − 2) / 2⌉。 该公式反映了这样一个事实：两个端点都不可用，并且每个有效增益都会消耗配对结构中的至少一个内部位置。 

接下来，我们从压缩数组中收集除端点之外的所有权重，因为只有内部石头才有资格被选择。 从这个集合中，我们取 k 个最大值。 

最后，我们将这些 k 值相加并输出结果。 

这样做的原因是，移除的过程总是可以重新安排，这样每个选定的内部石头都可以被“隔离”，而不会降低其价值，并且没有两个选择会干扰超出配对范围。 这在内部位置上创建了隐式匹配结构，确保最佳分数恰好是大小 k 的最佳可行子集的总和。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input().strip())
    colors = list(map(int, input().split()))
    weights = list(map(int, input().split()))

    if n <= 2:
        print(0)
        return

    # compress by color, keep max weight per segment
    comp = []
    i = 0
    while i < n:
        j = i
        best = weights[i]
        while j < n and colors[j] == colors[i]:
            if weights[j] > best:
                best = weights[j]
            j += 1
        comp.append(best)
        i = j

    m = len(comp)

    if m <= 2:
        print(0)
        return

    k = (m - 2 + 1) // 2  # ceil((m-2)/2)

    interior = comp[1:-1]
    interior.sort(reverse=True)

    ans = sum(interior[:k]) if k > 0 else 0
    print(ans)

if __name__ == "__main__":
    solve()
```该实现首先对相同颜色执行运行长度压缩，同时跟踪每次运行中的最大权重。 这确保了正确性，因为每个单色段只有一颗宝石很重要。 

计算压缩列表后，代码使用切片隔离内部元素。 端点被故意排除，因为结构论证保证它们永远不会做出贡献。 

按降序排序可以直接提取前 k 个值。 这避免了对堆结构的任何需要，因为我们只需要一次选择传递。 

唯一的微妙之处是使用上限除法正确计算 k。 将其写为`(m - 2 + 1) // 2`避免浮点运算和差一错误。 

## 工作示例

 ### 示例 1

 考虑一个压缩的权重数组：

 | 步骤| 数组| 内饰 | k | 已选择 | 总和|
 | --- | --- | --- | --- | --- | --- |
 | 开始| [5,1,4,2,6]| [1,4,2]| 2 | [4, 2] | 6 |

 这里 N = 5，所以 k = ceil(3/2) = 2。我们忽略端点 5 和 6。我们取前 2 个内部值 4 和 2，产生 6。这显示了端点优势如何删除大值，即使它们很大。 

### 示例 2

 | 步骤| 数组 | 内饰 | k | 已选择 | 总和|
 | --- | --- | --- | --- | --- | --- |
 | 开始| [10,3,8,7]| [3, 8] | 1 | [8] | 8 |

 这里 N = 4，所以 k = ceil(2/2) = 1。即使两个内部元素都是有效候选元素，但从结构上只能选择一个，因此我们取最大值。 

这些例子证实了该解决方案仅取决于内部排序，而不取决于任何动态过程。

 ## 边缘情况

 对于 N ≤ 2，该算法立即返回零，因为压缩数组没有内部位置。 例如，输入`N=2`总是产生一个空的内部切片，因此排序和求和自然会产生零。 

当所有石头都具有相同的颜色时，压缩会将数组减少为单个元素，因为只有整个块中的最大重量幸存下来。 这会触发`m <= 2`条件并返回零，与压缩后不存在内部结构的事实相匹配。 

当权重很大但集中在端点时，例如`[100, 1, 1, 100]`，算法完全丢弃端点值。 仅有的`[1, 1]`仍然是内部的，因此 k = 1 选择单个 1，正确反映了端点优势不会转化为规则下的可用分数。
