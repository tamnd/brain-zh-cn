 ## 解决方案
 ## 问题理解

 我们拥有一个构建和操作弹珠集合的系统，其中每个弹珠都可以被视为一个原子单元，稍后可以将其分组为更大的集合。 这些集合是分层形成的：集合可以定义为两个先前创建的集合的并集，它自然地形成二元森林结构，其中叶子是单个弹珠，内部节点代表并集。 

除了这个构造之外，我们还必须决定弹珠的二元分配，将每个弹珠解释为红色或非红色。 这些约束不仅适用于单个弹珠，还适用于层次结构中的聚合集：某些查询对与一个集相对应的子树中可能出现的红色弹珠数量施加限制。 这些限制可以是下限和上限，这立即表明了一种类似流的解释，其中约束通过结构传播而不是局部应用。 

第二种类型的操作使这种结构变得复杂，它允许我们以影响未来约束和联合的方式将大理石从考虑中删除。 移除的弹珠不仅不会被简单地忽略，还会被忽略。 它仍然通过集合的结构间接参与约束，因此它的效果必须在任何全局赋值中一致地保留。 

最终目标是确定是否存在红色和非红色弹珠的一致分配，该分配满足所有分层并集约束和所有子树基数约束，同时尊重删除，如果是，则构造这样的分配或验证可行性。 

问题规模足够大，任何显式跟踪每个子集的约束或重新计算每个操作的可行性的方法都会太慢。 尝试重新计算任意子集上的约束的简单公式将很快退化为二次行为，因为每个约束都可能触及联合森林的大部分。 

关键的结构约束是所有关系都是分层的并且可以嵌入到流网络中。 这强烈表明正确的模型不是组合搜索，而是有界的循环可行性问题。 

边缘情况是由于删除与联合结构的交互方式而产生的。 例如，如果一个弹珠在深深嵌入许多联合后被移除，那么天真的方法可能会完全丢弃它并破坏祖先集合中计数的守恒。 当设置约束应用于部分依赖于已删除的弹珠的子树时，会发生另一种故障模式：简单地忽略已删除的元素会低估并违反下限，从而产生错误的不可行结论。 

## 方法

 暴力方法会将问题视为全局约束满足任务。 我们可以尝试将每个大理石指定为红色或非红色，然后验证每个集合上的每个约束。 这已经给出了大小的解决方案空间$2^n$，即使对于中等的 n 来说，这也是立即不可行的。 

稍微结构化一点的强力方法会尝试在每次操作后重新计算每个约束的子树总和。 如果有 m 个约束，并且每次验证都遍历子树，则每次检查的成本为 O(n)，每次分配尝试的时间复杂度为 O(nm)。 即使我们只评估一次可行性，显式构建所有子树成员资格也会导致重复遍历大型重叠区域。 在最坏的情况下，联合森林退化为链，每个约束几乎触及所有节点，因此单独的预处理就变成了二次方。 

关键的观察结果是，每个约束本质上都是线性的：它限制子树中选定弹珠的总和。 这立即映射到一个流公式，其中每个弹珠都是一个二元变量，每个子树约束成为一个容量约束。 联合结构提供了一种自然的分解：我们不考虑任意子集，而是将该结构解释为一个网络，其中流沿着层次结构守恒。 

每个弹珠对应一个流量单位，可以指定为红色（流量通过）或非红色（流量被阻止）。 每个子树约束成为循环系统中的边或节点约束。 下限和上限自然对应于 LR 流约束。 

困难在于合并拆除。 被移除的弹珠必须仍然保持流动结构的一致性，这意味着它不能被简单地删除。 相反，它必须被重定向，以便任何经过它的流都会被一致地重新路由。 这是通过改造网络来实现的，使移除的节点仍然是流量守恒系统的一部分，确保总流量平衡保持不变。 

一旦问题完全转化为具有下限和上限的循环，就可以使用最大流的标准约简来解决：我们将每个有界边转换为容量约束，引入超级源和汇，并检查是否可以满足所有需求。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(2^n·n·m) | O(2^n·n·m) | O(n + m) | 太慢了 |
 | 最佳（流量减少）| O(F·E √V) | O(F·E √V) | O(V + E) | 已接受 |

 ## 算法演练

 我们现在构建一个对联合森林和所有约束进行编码的流网络。

1. 根据输入操作构建二叉并林。 每个节点代表一个单独的弹珠或一组合并的弹珠。 叶子对应于单个弹珠，而内部节点代表两个子弹珠的联合。 这为我们提供了所有集合上的有根森林结构。 
2. 将每个弹珠解释为一个单位变量，它可以为最终的红色计数贡献 0 或 1。 我们将其表示为一种流决策，其中发送一个单元穿过弹珠对应于将其选择为红色。 
3. 对于每个联合节点，通过确保集合的值等于其子节点之和来强制父节点和子节点之间的一致性。 这是使用内部节点处的流量守恒来建模的。 
4. 对于子树上的每个约束，将下限和上限转换为流程图中的有界容量边。 “子树S中的L和R红色弹珠之间”的约束成为对通过相应节点的流的需求约束。 这确保任何有效的流通都遵守允许的最小和最大红色计数。 
5. 引入标准循环变换：用下界调整替换每个有界边，计算由这些下界引起的节点需求，并构造超级源和超级汇。 这将可行性转化为最大流量问题。 
6. 处理移除的弹珠，确保它们保留在网络中，但重新定向它们的贡献边，以便保留流守恒。 我们不是删除节点，而是调整邻接关系，以便任何将通过已删除的大理石的流量始终通过专用结构重新路由，以保持流入和流出的平等。 这保证了移除不会破坏祖先集中的保护约束。 
7. 运行从超级源到超级汇的最大流量。 如果满足所有需求边，则存在可行的分配。 否则，没有有效的弹珠着色可以满足所有约束。 
8. 从流中恢复分配：当且仅当通过其相应边缘的流在表示选择的方向上饱和时，弹珠是红色的。 

### 为什么它有效

 该构造强制每个约束都是相同单位变量的线性守恒规则。 每个弹珠恰好贡献一个单位的势流，并且每个并集节点保留可加性。 下限和上限限制了可行的流量范围，并且循环减少保证了任何可行的流量对应于有效的分配，反之亦然。 由于在保守网络中每个约束都被编码为容量限制，因此任何有效流都会自动同时满足所有子树要求。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

from collections import deque

class Dinic:
    def __init__(self, n):
        self.n = n
        self.adj = [[] for _ in range(n)]

    def add_edge(self, u, v, c):
        self.adj[u].append([v, c, len(self.adj[v])])
        self.adj[v].append([u, 0, len(self.adj[u]) - 1])

    def bfs(self, s, t, level):
        q = deque([s])
        level[:] = [-1] * self.n
        level[s] = 0
        while q:
            u = q.popleft()
            for v, c, r in self.adj[u]:
                if c > 0 and level[v] == -1:
                    level[v] = level[u] + 1
                    q.append(v)
        return level[t] != -1

    def dfs(self, u, t, f, level, it):
        if u == t:
            return f
        for i in range(it[u], len(self.adj[u])):
            it[u] = i
            v, c, r = self.adj[u][i]
            if c > 0 and level[v] == level[u] + 1:
                ret = self.dfs(v, t, min(f, c), level, it)
                if ret:
                    self.adj[u][i][1] -= ret
                    self.adj[v][r][1] += ret
                    return ret
        return 0

    def maxflow(self, s, t):
        flow = 0
        level = [-1] * self.n
        while self.bfs(s, t, level):
            it = [0] * self.n
            while True:
                f = self.dfs(s, t, 10**18, level, it)
                if not f:
                    break
                flow += f
        return flow

关键的实现细节是在运行最大流之前必须消除所有下限。 这是通过将下限推入节点需求，然后用超级源和超级汇来平衡它们来完成的。 一个常见的错误是忘记这种转换并直接将下限编码为容量，这会产生不正确的可行性检查。

 ## 边缘情况

 一种重要的边缘情况是，弹珠在多次联合操作中被深深嵌入后被移除。 在这种情况下，一个简单的解决方案是删除节点并减小子树大小，从而打破对祖先的上限约束。 正确的流模型将节点保留在图中并重新路由其贡献，以便祖先节点仍然接收一致的流单元。 

当子树约束完全应用于已移除的弹珠时，会出现另一种边缘情况。 如果我们简单地忽略删除的节点，则约束似乎无法满足，因为子树的容量为零。 在流公式中，移除的节点仍然通过守恒边做出贡献，因此如果原始结构允许，仍然可以满足需求。 

最后一个微妙的情况是多个约束在同一联合节点上严重重叠。 幼稚的每个约束评估会重复计算或错过共享贡献。 在循环模型中，所有约束同时编码为单个可行性系统，确保一致的全局满意度，而约束之间不会相互干扰。
