 ## 问题理解

 我们得到了一系列魔法文物，每件都有积极的价值$w_i$。 英雄一开始的魔力为零。 每个神器必须只能使用一次，并且每个神器都可以通过两种方式之一激活。 

如果使用魔法激活一件神器，它的价值会添加到英雄当前的魔法力量中。 如果使用力量激活它，它所产生的金币等于神器的价值乘以英雄当时的魔法力量。 激活顺序是自由的，每个神器是否使用魔法或力量的决定也是自由的。 

目标是最大化从力量激活中获得的总金币，同时记住只有魔法激活才能增加用于未来金币收益的乘数。 

输入尺寸较小，$n \le 100$和价值观$w_i \le 100$。 这立即表明二次甚至三次动态规划方法是可以接受的，但所有子集的指数枚举并不可靠，因为$2^{100}$太大了。 

一个关键的结构性观察是，增加未来收益的唯一方法是将一些文物“花费”为纯粹的电力建设者，其余的则作为利润创造者。 这个顺序很重要，因为权力会随着时间的推移而积累，并直接乘以所有后来选择的利润贡献。 

一个微妙的边缘情况是所有文物都被用作魔法。 在这种情况下，根本不会产生黄金，答案是零。 另一个极端情况是，当所有值都用作强度时，但乘数始终为零，因此答案又为零。 最佳解决方案必须平衡这两种角色。 

## 方法

 暴力方法尝试分配每个神器用作魔法或力量的所有可能性，并且还考虑激活顺序的所有排列。 对于每种安排，我们都会模拟该过程，跟踪当前功率并积累黄金。 这已经变得不可行，因为即使忽略排列，也有$2^n$作业和每次模拟成本$O(n)$, 给予$O(n \cdot 2^n)$，这远远超出了限制。 

关键的观察是顺序可以标准化。 如果我们确定哪些神器用于魔法，那么最好首先按照我们希望它们对物质的力量贡献的顺序来应用所有魔法操作，因为延迟魔法只会减少未来的乘数。 一旦理解了这种分离，问题就变成了：选择一个项目子集来贡献力量，然后决定强度项目的顺序，以在不断增长的前缀和下最大化增益。 

这导致了关于有多少神器被用作魔法以及它们的总威力如何构建的动态编程公式。 我们对文物进行分类，以便对它们进行一一推理，保留总魔法力量的可能状态以及我们已处理的物品数量。 每个神器要么增加力量，要么贡献按当前力量加权的黄金，并且通过 DP 转换做出最佳决策。 

该结构本质上是一个类似背包的 DP，其中状态跟踪累积的魔力总和以及在处理项目前缀后该总和的最佳可实现黄金。

 ## 算法演练

 我们将这个过程解释为构建两个组：用于增加魔力的神器和用于赚取金币的神器，其中金币取决于使用每个力量神器时最终累积的力量。 困难在于，力量激活取决于不断发展的力量，因此我们必须仔细模拟力量积累。 

### 步骤

 1. 按值对工件进行排序，或者更准确地说，以任意顺序处理它们，因为所有选择都是对称的。 相反，我们通过前缀处理隐式地关注子集上的 DP。 
2. 定义一个 DP 状态，我们在其中跟踪已处理的工件数量和到目前为止累积的总魔力，并为每个此类状态存储可实现的最大金币。 
3. 初始化 DP，零加工工件、零功率和零金币。 
4. 对于每件有价值的文物$w$，我们考虑每个现有状态的两次转换。 一个转变将神器视为魔法，将当前功率增加$w$而不生产黄金。 另一个转变将其视为力量，产生的黄金等于当前力量乘以$w$，无需改变电源。 
5. 我们仔细更新 DP，以便每个工件仅使用一次，以与处理相反的顺序迭代状态，以避免在同一迭代中重复使用。 
6. 处理完所有工件后，我们扫描所有 DP 状态并取最大的黄金值。 

关键的设计选择是将电力存储为状态变量。 这是必要的，因为金币取决于每次决策时的乘数，而该乘数正是迄今为止累积的魔力。 

### 为什么它有效

 在这个过程中的任何时刻，影响未来结果的唯一信息是当前的总魔力以及剩余的物品。 DP 状态准确捕获此信息。 以相同处理前缀和相同功率结尾的任何两个序列都可以互换以供将来的决策使用，因此仅保留每个状态的最佳黄金是安全的。 这建立了最佳子结构，因为每个工件的决策仅取决于当前功率，而不取决于如何实现该功率。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    w = list(map(int, input().split()))

    max_sum = sum(w)

    # dp[i][p] = max gold after processing some prefix, with power p
    dp = [[-1] * (max_sum + 1) for _ in range(n + 1)]
    dp[0][0] = 0

    for i in range(n):
        wi = w[i]
        for j in range(i, -1, -1):
            for p in range(max_sum - wi, -1, -1):
                if dp[j][p] < 0:
                    continue

                # use as magic: increase power
                dp[j + 1][p + wi] = max(dp[j + 1][p + wi], dp[j][p])

                # use as strength: gain gold
                dp[j + 1][p] = max(dp[j + 1][p], dp[j][p] + p * wi)

    ans = 0
    for j in range(n + 1):
        for p in range(max_sum + 1):
            ans = max(ans, dp[j][p])

    print(ans)

if __name__ == "__main__":
    solve()
```该代码构建了一个 DP 表，该表以已处理的项目数量和当前累积的魔力为索引。 每个神器要么有助于增加力量，要么根据当前力量贡献金币。 

反向迭代结束`j`和`p`确保每个工件仅使用一次，从而防止覆盖状态，从而错误地多次重复使用同一项目。 力量使用的转变`p * wi`，直接编码乘数规则。

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 |$O(n^2 \cdot \sum w)$| 每个项目都会在所有 DP 状态上进行转换，并且每个状态都会考虑两个选择 |
 | 空间|$O(n \cdot \sum w)$| DP 表存储每个前缀和功率值的最佳金币 |

 限制条件$n \le 100$,$w_i \le 100$意味着$\sum w \le 10000$。 因此 DP 的运行时间约为$10^8$最坏情况下的操作，如果仔细实施，在典型的竞争设置中，在优化的 Python 中，在 2 秒内完成操作是可以接受的。

 ## 边缘情况

 具有一个工件的最小案例，例如输入`1 / 5`，迫使算法认识到，在没有任何事先魔法的情况下，任何强度激活都会产生零金币。 DP 从零功率开始，因此两次转换都会导致最终答案为零。 

像这样的统一案例`3 / 1 1 1`证明分裂是必要的。 最佳计划是使用一件神器用于魔法，两件用于力量，给予力量 1，然后给予金币$1 + 1 = 2$。 DP 正确地探索了这两种可能性，因为它保留了所有中间功率状态。 

具有较大早期价值的案例，例如`3 / 10 1 1`，说明了如果我们尝试始终最大化即时黄金，贪婪排序会失败的原因。 最佳解决方案首先将 10 转换为幂，然后将其花费两次，产生 20，DP 通过正确的状态传播捕获该值。
