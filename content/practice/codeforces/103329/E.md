 ## 解决方案
 ## 问题理解

 我们得到一个数组，其中每个位置都被标记为正或非正。 正值将位置标记为好对象，而非正值将其标记为坏对象。 当坏对象连续出现在数组中时，它们自然会形成连续的块，并且每个这样的最大块被视为一个“组”。 

问题的关键结构是交互取决于这些坏对象组以及如何将它们与好对象进行匹配。 同一连续块内的坏对象在配对行为方面是不可分割的，而来自不同块的坏对象的行为是独立的。 此外，如果 j 不在 i 的右侧，则位置 i 处的坏对象只能与好对象 j 配对。 

任务不是构造显式匹配，而是确定在这些约束下是否可以将所有坏对象完全配对。 不可能性的条件是用坏段 [l, r] 和与两个量的比较来描述的：整个数组中坏对象的总数和出现到位置 l 的好对象的数量。 

直观上，每个坏段必须“适合”由所有坏对象加上出现在其左侧不太远的好对象形成的可用匹配容量。 如果任何段相对于可用容量来说太大，则排列失败。 

从复杂性的角度来看，输入是单个数组，任何解决方案都必须以线性或接近线性的时间运行。 尝试显式模拟所有可能的配对或推理匹配的简单方法将是二次的，这对于大约 2 秒和 n 高达 2e5 或 1e5 的典型 Codeforces 约束来说太慢了。 

当坏段很大并且位于数组的早期时，就会出现主要的微妙边缘情况。 例如，如果数组以一长串坏值开始，后来才包含好值，则贪婪的匹配直觉可能会错误地假设未来的好值会有所帮助，即使它们由于索引限制而无法使用。 

当不良对象被分割成许多小片段时，就会出现另一种边缘情况。 一个简单的解决方案可能会独立检查每个坏对象，而不是对它们进行分组，这会破坏逻辑，因为约束是基于段的，而不是基于元素的。 

## 方法

 蛮力视角是明确模拟配对。 我们会尝试将每个坏对象与来自不同段的另一个坏对象或出现在其左侧的合适的好对象进行匹配。 这自然会导致考虑建立双向匹配或重复扫描有效的合作伙伴。 即使采用贪婪策略，每个坏对象也可能会触发整个数组的扫描以查找匹配项，这会导致在每个对象都是坏对象或交替对象的最坏情况下出现 O(n²) 行为。 

关键的简化来自于认识到唯一有意义的障碍不是单独的配对，而是坏元素的整个连续块。 一旦我们将数组压缩成这些块，每个块的行为就像一个单一的约束：它需要来自坏对象的全局池和出现在其左边界之前的好对象的足够“容量”。 

这将问题从动态匹配减少为前缀计费问题。 我们只需要知道总体上存在多少坏对象，以及每个坏段开始之前有多少好对象。 然后，每个部分独立地检查一个不等式，整个答案取决于任何部分是否违反它。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力匹配| O(n²) | O(n) | 太慢了 |
 | 段+前缀分析| O(n) | O(n) | 已接受 |

 ## 算法演练

我们将数组简化为两种信息：好对象的前缀计数和坏对象的最大连续段列表。 

1. 扫描一次数组并计算一个前缀数组，其中 prefixGood[i] 存储有多少个好对象出现在位置 [1, i] 中。 这使我们能够立即知道任意位置左侧有多少可用的好对象。 
2. 在同一次扫描期间，识别坏对象的每个最大连续段。 每个段由其左端点 l、右端点 r 及其长度 len = r - l + 1 表示。这种压缩至关重要，因为约束适用于每个段，而不是每个元素。 
3. 计算整个数组中坏对象的总数。 该值代表声明中提到的全局池P，并且在所有段之间共享。 
4. 对于每个坏段 [l, r]，将 Q 计算为 prefixGood[l]，即位于段开头或之前的好对象的数量。 这是好对象中唯一可以对该段的匹配约束做出有意义贡献的部分。 
5. 检查每个段的条件：其大小是否超过totalBad + prefixGood[l]。 如果任何部分违反了这种不等式，我们立即得出结论：完全配对是不可能的。 
6. 如果所有段都满足条件，则可以进行排列。 

检查背后的原因是每个段必须由所有坏对象和出现在其之前的可用好对象的组合支持。 如果一个段太大，无论后来的好对象如何排列，它都无法分解为有效的配对。 

它之所以有效，是因为我们观察到，一旦我们聚合了坏对象，所有约束都是局部于段边界的。 一个段不能从其起始点的右侧“借用”可用的好对象，并且不同段之间的交互只会增加可用的灵活性，而不会减少它。 因此，唯一可能的故障模式是单个段超过全局定义的总可用容量并达到其边界。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input().strip())
    a = list(map(int, input().split()))

    prefix_good = [0] * (n + 1)
    total_bad = 0

    for i in range(1, n + 1):
        prefix_good[i] = prefix_good[i - 1]
        if a[i - 1] > 0:
            prefix_good[i] += 1
        else:
            total_bad += 1

    i = 1
    ok = True

    while i <= n:
        if a[i - 1] > 0:
            i += 1
            continue

        l = i
        while i <= n and a[i - 1] <= 0:
            i += 1
        r = i - 1

        seg_len = r - l + 1
        good_left = prefix_good[l]

        if seg_len > total_bad + good_left:
            ok = False
            break

    print("YES" if ok else "NO")

if __name__ == "__main__":
    solve()
```该实现首先构建好值的前缀计数，以便查询“索引 l 之前存在多少好对象”变为 O(1)。 同时，它统计全局所有坏对象。 

然后它再次扫描以提取连续的坏段。 每次发现坏段时，它都会计算其长度并将其与导出的容量表达式进行比较。 

一个微妙的实现细节是 prefix_good[l] 是在段的左边界而不是右边界处获取的。 使用 prefix_good[r] 会错误地计算在配对规则下不允许协助该段的好对象。 

循环结构确保每个索引都被访问一次，因此分段和检查在线性时间内发生。

 ## 边缘情况

 一个关键的边缘情况是数组以长坏段开始。 在这种情况下， prefixGood[l] 为零，因此仅根据坏对象总数检查该段。 该算法正确地处理了这个问题，因为该段仍然受到全局坏容量的限制，并且没有未来的好对象可以帮助它。 

当好的对象只出现在最后时，就会出现另一种边缘情况。 在这种情况下，所有早期段的 prefixGood[l] 保持为零，确保不会计算来自右侧的无效借用。 

第三种边缘情况是坏对象被分割成许多单元素段。 该算法仍然将每个段视为单独的约束，并且由于每个段的长度为 1，因此它通常满足不等式，除非全局结构是退化的。
