 ## 解决方案
 ## 问题理解

 我们给出一个连续整数的区间，从`a`并且有长度`l`。 对于每个数字`x`， 定义`F(x)`作为十进制表示形式中幸运数字的计数。 仅数字`4`和`7`被认为是幸运的。 

任务是找到最小的整数`b > a`这样序列`F(a), F(a+1), ..., F(a+l-1)`正好等于`F(b), F(b+1), ..., F(b+l-1)`。 

换句话说，该时间间隔内的幸运数字计数模式必须从以下位置开始重复：`b`。 

的价值观`a`和`l`一样大`10^9`，这立即排除了任何直接扫描大范围的方法。 在最坏的情况下，即使迭代一次间隔中的所有数字也是不可能的，因为`l`本身可能就是十亿。 我们需要利用函数中的结构`F(x)`而不是对每个数字单独进行评估。 

关键的观察是`F(x)`仅取决于十进制数字。 更改较高的数字通常会保留大的连续块上的幸运数字计数模式。 问题变成了构造另一个其数字行为完全匹配的区间。 

一些边缘情况很容易被错误处理。 

考虑：```
a = 7, l = 1
```我们需要最小的`b > 7`和`F(b) = F(7) = 1`。 答案是`14`， 不是`17`， 因为`14`已经有一位幸运数字了。 仅在幸运数字中搜索的解决方案将会失败。 

另一个棘手的情况是当区间跨越十的幂边界时：```
a = 98, l = 5
```顺序是：```
F(98)=0
F(99)=0
F(100)=0
F(101)=0
F(102)=0
```许多数字都产生全零，但是按 10 的简单幂进行移位可能会意外地在进位中引入幸运数字。 进位传播是这个问题的主要危险。 

一个更微妙的例子是：```
a = 447, l = 3
```顺序是：```
3, 1, 1
```因为：```
447 -> three lucky digits
448 -> one lucky digit
449 -> one lucky digit
```粗心的数字替换策略可能会保留第一个值，但会破坏相邻数字之间的局部转换行为。 

整个挑战是在整个区间内保留进位如何影响幸运数字。 

## 方法

 蛮力的想法很简单。 计算序列```
S(i) = F(a+i)
```为所有人`0 <= i < l`，然后搜索增加的值`b`直到相同的序列再次出现。 

这是有效的，因为定义是直接的并且检查相等性很容易。 不幸的是，它变得极其缓慢。 在最坏的情况下`l = 10^9`，所以即使生成原始序列一次也是不可能的。 搜索空间为`b`也是无界的。 

关键的见解是`F(x)`是逐位确定的，当没有进位到达某些位置时，小数加法会周期性地进行。 

假设我们为区间中的每个数字添加足够大的 10 次方。 如果没有任何加法产生进位到较低的位置，则对于间隔中的每个数字，较低的数字保持不变。 只有一位较高的数字发生变化，我们可以选择该数字，所以它既不是`4`也不`7`。 然后幸运数字计数保持不变。 

这将问题从显式匹配序列转变为构建安全移位。 

危险的部分是进位传播。 如果我们添加`10^k`到一个数字，其较高的块接近`999...9`，进位可能会向下波动并改变许多数字。 为了避免这种情况，我们需要找到一个小数位置，使间隔完全适合一个大小的块`10^k`。 

更具体地说，我们寻找最小的`10^k`这样：```
a / 10^k == (a + l - 1) / 10^k
```意味着整个区间位于单个长度块内`10^k`。 

然后精确添加`10^k`仅更改该块上方的数字并保留下方的数字`k`数字在整个时间间隔内保持不变。 由于更改的数字会增加 1，因此只有当该数字变为`4`或者`7`，或者之前是`4`或者`7`。 

我们可以反复添加`10^k`直到修改后的数字避免幸运的转变。 由于只有一位数字发生变化，我们只需要避免四种情况：```
3 -> 4
4 -> 5
6 -> 7
7 -> 8
```所有其他增量都会保留幸运计数。 

这给出了恒定时间的建设性解决方案。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(l × 搜索范围) | O(l) | 太慢了 |
 | 最佳 | O(位数) | O(1) | O(1) | 已接受 |

 ## 算法演练

 1. 让`r = a + l - 1`，区间中的最后一个数字。 
2. 求最小幂`p = 10^k`这样：```
a / p == r / p
```这意味着整个区间位于一个大小的块内`p`。 
3.设：```
prefix = a / p
```这是区间中每个数字共享的较高部分。 
4. 我们想增加这个前缀，同时保持较低的前缀`k`位数不变。 每个候选班次是：```
b = a + t * p
```对于某个正整数`t`。 
5. 将前缀加一仅更改其最后一位十进制数字。 仅当数字转换是以下之一时，幸运计数才会改变：```
3 -> 4
4 -> 5
6 -> 7
7 -> 8
```6. 从`prefix + 1`，找到第一个增量为`prefix`不会改变前缀中幸运数字的数量。 
7. 令所选的新前缀为`np`。 然后构造：```
b = np * p + (a % p)
```8. 输出`b`。 

### 为什么它有效

 较低的`k`区间中每个数字的位数保持不变，因为区间永远不会跨越`p`。 相加的倍数`p`只修改高位数字。 

对于每个偏移量`i`区间内：```
a+i = prefix * p + low(i)
b+i = np * p + low(i)
```在哪里`low(i)`两种表达方式相同。 

下半部分的幸运数字贡献是完全一样的。 我们选择`np`以便`F(prefix) = F(np)`，意味着较高的部分也贡献相同数量的幸运数字。 因此：```
F(a+i) = F(b+i)
```整个间隔。 

由于我们按升序测试候选前缀，因此出现了极小值。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def lucky_count(x: int) -> int:
    cnt = 0
    while x:
        d = x % 10
        if d == 4 or d == 7:
            cnt += 1
        x //= 10
    return cnt

def solve():
    a, l = map(int, input().split())

    r = a + l - 1

    p = 1
    while a // p != r // p:
        p *= 10

    prefix = a // p
    low = a % p

    target = lucky_count(prefix)

    np = prefix + 1
    while lucky_count(np) != target:
        np += 1

    b = np * p + low

    print(b)

if __name__ == "__main__":
    solve()
```第一部分计算最小块大小`p`使得间隔完全保持在一个块内。 这是问题的核心结构观察。 一旦存在这样的块，所有较低的数字在移位倍数的情况下都是稳定的`p`。 

功能`lucky_count`很小，因为数字最多在`10^10`，所以最多有十位数字。 

变量`prefix`代表区间内共享的较高数字。 我们搜索具有相同幸运数字计数的最小较大前缀。 由于位数很少，因此强制执行本地搜索是完全安全的。 

重建：```
b = np * p + low
```准确保留较低的数字。 这是微妙的部分。 仅使用`a + t*p`直接在数学上是等价的，但是从碎片中重建数字使得不变量变得明确。 

一个常见的错误是选择`p`错误地。 条件必须保证整个区间适合一个块，而不仅仅是这样`a`本身与边界对齐。 否则，对于区间内的某些值，进位会出现在较低的数字中。 

另一个常见的错误是忘记了`prefix`可能为零。 助手正确地处理了这个问题，因为零不包含幸运数字。

 ## 边缘情况

 考虑：```
1 1
```我们有：```
F(1)=0
```幸运数字为零的最小较大数字是`2`。 

该算法计算：```
p=1
prefix=1
target=0
```然后立即发现：```
np=2
```和输出`2`。 

现在检查进位边界情况：```
98 5
```间隔从两位数变为三位数：```
98,99,100,101,102
```天真的转变`10`会失败，因为：```
108
```由于进位，在间隔稍后引入了一个幸运数字。 

该算法反而选择：```
p=100
```因为整个区间都适合一个大小的块`100`。 

这使得区间中每个元素的后缀保持稳定。 

最后，考虑：```
447 3
```过渡：```
447 -> 448
```删除幸运计数`3`到`1`。 

如果我们单独修改较低的数字，这种局部行为将立即中断。 

该算法保留整个后缀`7,8,9`，因此相同的过渡模式会自动出现在新间隔中。
