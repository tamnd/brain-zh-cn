 ## 解决方案
 ## 问题理解

 我们需要构造尽可能小的幸运数字，其数字之和为给定值`n`。 

幸运数字可能只包含数字`4`和`7`。 例如，`447`有效，因为每个数字都是`4`或者`7`， 尽管`45`无效，因为数字`5`出现。 

输入给出一个整数`n`，表示所需的数字和。 输出必须是数字上最小的幸运数字，其数字之和恰好为`n`。 如果不存在这样的数字，我们打印`-1`。 

约束上升到`10^6`，这就立即排除了生成幸运数字的可能性。 即便是只有20位数的幸运数字，数量也已经非常庞大了。 我们需要一些在大致线性时间或更好的时间内运行的东西。 

关键的观察是数字之和仅取决于有多少`4`沙`7`我们使用的是。 如果我们使用`a`数字的副本`4`和`b`数字的副本`7`， 然后：$4a + 7b = n$所以问题就变成了寻找非负整数`a`和`b`满足这个方程，同时也产生尽可能小的数字。 

有一些容易被忽略的边缘情况。 

如果`n = 1`，没有解决方案，因为两者都没有`4`也不`7`可以贡献一笔`1`。 正确的输出是：```
-1
```粗心的实现可能会尝试贪心减法并意外进入无限循环或打印空字符串。 

另一个棘手的情况是存在多个分解时。 例如：```
n = 28
```我们可以写：```
28 = 7 + 7 + 7 + 7
```这给出了`7777`， 或者：```
28 = 4 + 4 + 4 + 4 + 4 + 4 + 4
```这给出了`4444444`。 

虽然`4444444`位数越多，数值越小，因为一旦位数不同，位数较少的数字并不总是较小。 实际的规则更微妙：在相同长度的数字中，前面的数字更重要。 我们必须仔细构造字典顺序上最小的有效数字。 

当位数不同时，会出现一种更重要的边缘情况。 考虑：```
n = 11
```可能的分解是：```
4 + 7
7 + 4
```有效的幸运数字是`47`和`74`。 正确答案是`47`因为按字典顺序它更小。 任何以错误顺序附加数字的实现都会在这里失败。 

## 方法

 最直接的暴力破解想法是生成幸运数字并测试它们的数字和。 我们可以递归地构建每个只包含`4`和`7`，计算其数字和，并跟踪最小的有效数字。 

这在概念上是有效的，因为每个幸运数字最终都会生成。 问题在于数量的爆炸性增长。 有长度的幸运数字`k`有`2^k`的可能性。 即使是长度`20`，已经超过一百万候选人。 自从`n`可能达到`10^6`，所需的数字可能包含数十万位数字，因此暴力破解完全不可能。 

问题的结构给出了更简单的解释。 只有在我们决定了多少个之后，数字的实际顺序才重要`4`沙`7`来使用。 如果我们选择`a`四肢和`b`七，则求和条件变为：$4a + 7b = n$现在我们只需要搜索有效的对`(a, b)`。 

下一个问题是如何使结果数字尽可能小。 

如果位数较少的数字都没有前导零，则位数较少的数字始终小于位数较多的数字。 因为每个数字至少贡献`4`，最小化总位数是有益的。 更换两个`4`与一个`7`减少位数，因为：```
4 + 4 = 8
7 = 7
```使用更多`7`s 通常表示较少的数字。 

但数字顺序也很重要。 一旦总位数确定，将较小的数字放在前面会使数字更小。 所以选择计数后`(a, b)`，最优排列为：```
444...4777...7
```与所有`4`首先。 

这导致搜索空间非常小。 我们可以迭代可能的数量`7`s。 对于每个值`b`，我们检查剩余的总和是否可以被整除`4`。 

在所有有效的对中，我们更喜欢总位数最小的对：```
a + b
```如果多对具有相同的长度，我们更喜欢长度更多的那对`4`s 在前面，如果我们最小化，这自然会发生`b`。

 | 蛮力 | O(2^k) | O(2^k) | O(k) | 太慢了 |

 ## 算法演练

 1.读取整数`n`。 
2. 迭代可能的位数`7`s，称之为`b`， 从`0`最多`n // 7`。 
3. 对于每个`b`，计算剩余总和：```
remaining = n - 7 * b
```1. 检查是否`remaining`可以整除`4`。 

如果它不能被整除`4`，则没有有效计数`4`s 为此而存在`b`。 

1. 如果可整除，计算：```
a = remaining // 4
```现在我们有了一个有效的分解：$4a + 7b = n$1. 由于我们迭代`b`从小到大，第一个有效的对自动给出最小的数字。 

较小`b`意味着更领先`4`s 并且通常更有利的字典顺序。 

1. 打印包含以下内容的字符串`a`的副本`'4'`其次是`b`的副本`'7'`。 
2. 如果循环后不存在有效对，则打印`-1`。 

### 为什么它有效

 每个有效的幸运数字都对应于某对`(a, b)`满意：$4a + 7b = n$在由这些数字组成的所有数字中，将所有`4`优先于所有`7`s 产生最小的排列，因为`4 < 7`。 

该算法检查每个可能的计数`7`s 按升序排列。 第一个有效的解决方案最小化了`7`s。 自从更换`7`与`4`s 将较小的数字推向前面，这会产生最小的有效幸运数字。 

不会跳过任何有效分解，因此算法不会错过最佳答案。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())

    for sevens in range(n // 7 + 1):
        remaining = n - 7 * sevens

        if remaining % 4 == 0:
            fours = remaining // 4
            print('4' * fours + '7' * sevens)
            return

    print(-1)

solve()
```该循环会尝试所有可能的计数`7`s 从最小到最大。 对于每个选择，我们检查剩余的总和是否可以完全使用`4`s。 

一旦我们找到有效的分解，我们立即打印答案并停止。 尽早返回很重要，因为以后的解决方案将包含更多`7`s 并产生更大的数字。 

输出建设也很重要。 我们打印所有`4`优先于所有`7`s 因为一旦数字多重集固定，字典顺序就决定了较小的数字。 

一个微妙的细节是循环绑定：```
range(n // 7 + 1)
```如果没有`+1`，我们会错过整个总和仅由以下组成的情况`7`s。 

另一个容易犯的错误是在减去之前检查整除性`7`贡献。 可分割性测试必须适用于考虑所有选择后的剩余值`7`s。

import sys, io

def solve():
    input = sys.stdin.readline

    n = int(input())

    for sevens in range(n // 7 + 1):
        remaining = n - 7 * sevens

        if remaining % 4 == 0:

def run(inp: str) -> str:
    backup_stdin = sys.stdin
    backup_stdout = sys.stdout

    sys.stdin = io.StringIO(inp)

    output = sys.stdout.getvalue()

    sys.stdout = backup_stdout
