 ## 解决方案
 ## 问题理解

 我们有一行工人，每个工人都有两个属性：生产力和工作时间。 工人的贡献被定义为这两个值的乘积。 威廉必须准确地选择一个连续的片段$K$工人，并希望最大化该部门的总贡献。 

所以任务减少为扫描一个长度的数组$N$，其中每个位置都有一个值$a_i = p_i \cdot h_i$，并找到所有固定长度的连续子数组的最大和$K$。 

输入大小达到$N = 10^5$，这排除了任何从头开始为每个窗口重新计算总和的解决方案。 每个段的简单重新计算将花费$O(NK)$，在最坏的情况下变成$10^{10}$，远远超出一秒钟所能容纳的范围。 

正确的解决方案必须重用相邻窗口之间的计算，这意味着每个元素最多应添加和删除一次。 

出现微妙的边缘情况时$K = 1$。 在这种情况下，答案就是最大的单一产品。 任何滑动窗口逻辑仍必须正确处理初始化。 

另一个极端情况是当所有值都相等或负值不存在时，但我们仍然必须小心具有固定整数大小的语言中的溢出。 在 Python 中这不是问题，但在 C++ 中它需要 64 位整数。 

一个小的说明性案例：

 输入：```
4 2
2 3
1 3
3 2
4 1
```产品成为$[6, 3, 6, 4]$。 正确答案是任意两个连续元素的最大和，所以我们比较$(6+3), (3+6), (6+4)$, 给予$10$。 

一种简单的方法可能会独立地重新计算每一对，但这会在重叠的段中重复共享工作。 

## 方法

 蛮力的想法很简单：考虑一段长度的每个可能的起始位置$K$，计算下一个的总和$K$通过迭代元素并跟踪最大值。 这是正确的，因为它直接评估每个有效的候选片段。 然而，对于每个$N-K+1$起始位置，我们做$K$添加，导致大约$N \cdot K$运营。 和$N = 10^5$，当$K$很大。 

改进来自于认识到连续片段严重重叠。 当我们将窗口从$[i, i+K-1]$到$[i+1, i+K]$，只有两个元素发生变化：我们删除$a_i$并添加$a_{i+K}$。 这将重新计算转换为增量更新，将每次转换减少到恒定时间。 

因此，我们不是重新计算总和，而是维护一个运行总和，并在窗口滑过数组时更新它。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O(NK)$|$O(1)$| 太慢了|
 | 滑动窗 |$O(N)$|$O(1)$| 已接受 |

 ## 算法演练

 1. 将每个worker转换为单个值$a_i = p_i \cdot h_i$，因为只有这个产品对最终总和很重要。 
2. 计算第一个的总和$K$元素。 这形成了初始窗口并建立了基线答案。 
3. 将此初始总和设置为当前最佳答案，因为尚未评估其他段。 
4. 将窗口从左向右滑动一个位置。 在每一步中，减去离开窗口的元素并添加进入窗口的新元素。 此更新之所以有效，是因为连续的窗口正好有两个位置不同。 
5. 每次更新后，将新的窗口总和与最佳答案进行比较，并存储较大的值。 
6.处理完所有窗口后，输出遇到的最佳和。 

### 为什么它有效

 该算法维护当前长度的精确总和 -$K$每一步都有窗口。 由于窗口之间的每次转换都通过精确删除一个过时的元素并精确添加一个新元素来保持正确性，因此不会丢失或重复计算任何信息。 每个可能的连续大小段$K$通过这个滑动过程只访问一次，因此所有维护的总和的最大值是真正的全局最大值。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, k = map(int, input().split())
    a = []
    
    for _ in range(n):
        p, h = map(int, input().split())
        a.append(p * h)
    
    window_sum = sum(a[:k])
    best = window_sum
    
    for i in range(k, n):
        window_sum += a[i]
        window_sum -= a[i - k]
        if window_sum > best:
            best = window_sum
    
    print(best)

if __name__ == "__main__":
    solve()
```该解决方案首先将每个工作线程折叠为单个整数值，这将问题简化为标准固定长度最大子数组总和任务。 使用前缀切片计算初始窗口一次，该前缀切片建立运行总和和基线答案。 

然后循环强制滑动不变量：迭代时$i$,`window_sum`始终表示以下元素的总和$i-k+1$到$i$。 每次迭代通过删除过时的左侧元素并添加新的右侧元素来在恒定时间内更新此状态。 

一个常见的错误是使用切片重新计算循环内的总和，这会默默地将解降级为二次时间。 另一个常见的差一错误是在错误的索引处启动滑动循环； 这里它正好开始于`k`，确保第一个整班对应于更换`a[0]`和`a[k]`。

 ## 边缘情况

 当$K = 1$，该算法将窗口初始化为第一个元素，然后在滑动时简单地比较每个单独的产品。 更新规则仍然有效，因为减去和添加相邻元素会退化为替换当前元素。 

为了$K = N$，循环不会执行，因为只有一个有效窗口。 初始总和已经是答案，并且算法正确地返回它而无需任何滑动更新。 

当所有值都相同时，每个窗口总和相等。 该算法重复更新窗口，但从不更改存储的最佳值，这表明它不依赖于值的变化来正确运行。
