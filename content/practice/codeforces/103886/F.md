 ## 解决方案
 ## 问题理解

 给定一个整数数组，并要求将其拆分为恰好 k 个连续的子数组。 对于任何此类分区，每个子数组都有一个跨其元素计算的 OR 值。 之后，我们对所有这些子数组 OR 值进行 AND 运算，生成一个代表分区质量的数字。 

任务是选择分区以使最终 AND 值最大化。 

考虑目标的另一种方法是确定哪些位可以在所有 k 个子数组中幸存下来。 仅当每个子数组至少有一个元素包含该位时，该位才会对最终答案产生影响。 如果甚至有一个子数组遗漏了该位，则该位将被最后的 AND 删除。 

对值的限制最多为 10^9，这将我们限制为最多 30 个相关位。 元素的数量通常足够大，以至于 O(n^2) 或任何重复尝试分区的方法都是不可行的。 我们需要接近线性或每比特具有较小因子的线性。 

天真的思维的一个微妙的失败案例是假设一旦某个位频繁出现，就可以在不协调的情况下使用它。 例如，如果k很大，一个比特可能会出现很多次，但由于聚类的原因仍然无法保证它在每个段中都存在。 另一个棘手的情况是当一位的贪婪分段破坏了更高位的可行性时。 

## 方法

 一个蛮力的想法是尝试将数组分成 k 个段的所有方法，并计算段 OR 的结果 AND。 这会立即组合爆炸，因为存在 C(n-1, k-1) 个可能的分区，即使对于小 n 也是不可行的。 

我们需要扭转观点。 我们不是首先选择段，而是询问哪些位可以强制出现在每个段中。 假设我们修复了一个候选位掩码 x ，代表我们在最终答案中想要的位。 然后我们检查是否可以将数组分成至少 k 个段，使得每个段在其 OR 中包含 x 的所有位。 

这将问题转化为每个位掩码的可行性检查。 关键的观察是位在可行性方面是独立的：如果我们尝试最大化答案，我们可以贪婪地从最高位到最低位构造 x。 这是可行的，因为较高位在最终整数结果中按字典顺序占主导地位，因此我们永远不想为了较低位的组合而牺牲较高位。 

为了检查固定 x 的可行性，我们从左到右扫描，同时累积当前段的 OR 值。 一旦当前段包含 x 的所有位，我们就剪切并开始一个新段。 如果我们可以形成至少 k 个这样的线段，那么 x 是可行的。 

这种贪婪的切割之所以有效，是因为延迟切割无济于事。 如果一个段已经满足x，扩展它只会添加额外的OR位，但不能删除现有的OR位，因此它永远不会增加我们以后形成更多有效段的能力。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | n 中的指数 | O(n) | 太慢了 |
 | 按位贪婪 + 可行性检查 | O(30n) | O(30n) | O(1) 额外 | 已接受 |

 ## 算法演练

 我们从最高有效位到最低有效位一点一点地构建答案。

1. 将答案掩码 x 初始化为 0，这意味着最初不需要任何位。 我们将在保留可行性的同时尝试一点一点地开启。 
2. 迭代从 29 到 0 的位。对于每个位 i，创建一个候选掩码 x' = x，并设置位 i。 这代表了比特 i 可以成为最终答案的一部分的假设。 
3. 检查是否可以将数组分成至少 k 个有效段，使得每个段包含 x' 中的所有位。 为此，请扫描数组，同时维护当前段的运行 OR。 
4. 扫描时，更新每个元素的运行 OR。 每当运行的 OR 包含 x' 的所有位时，我们就关闭当前段并重置运行的 OR。 我们还增加段计数。 
5. 扫描完成后，如果形成的线段数量至少为k，则x'是可行的，因此我们将x永久更新为x'。 否则，我们丢弃位 i 并保持 x 不变。 
6. 处理完所有位后，x 是可实现的最大掩码。 

关键的微妙之处在于我们可以在可行性检查中形成超过 k 个段。 这很重要，因为如果我们可以形成超过 k 个，我们总是可以合并一些相邻的段而不会失去有效性，因为合并只会增加 OR 并保留 x 中所有位的存在。 

为什么它有效：

 正确性取决于两个属性。 首先，固定掩码的贪婪分割在最大化有效片段数量方面是最佳的，因为我们总是在最早的可能点进行切割。 其次，按位构造是安全的，因为可行性是单调的：如果掩码 x 可行，则任何子掩码也是可行的。 这确保了一旦较高位被拒绝，它就永远不会因为添加较低位而变得有效。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def can(mask, a, k):
    cnt = 0
    cur = 0
    full = mask
    for v in a:
        cur |= v
        if (cur & full) == full:
            cnt += 1
            cur = 0
    return cnt >= k

def solve():
    n, k = map(int, input().split())
    a = list(map(int, input().split()))

    ans = 0
    for b in range(29, -1, -1):
        cand = ans | (1 << b)
        if can(cand, a, k):
            ans = cand

    print(ans)

if __name__ == "__main__":
    solve()
```该实现将可行性检查分离为辅助函数。 条件`(cur & full) == full`是确保当前段中存在所有必需位的干净方法。 

贪婪重置`cur = 0`是安全的，因为一旦一个段有效，任何扩展都无助于以后创建更多段，因为 OR 只累积位。 

外循环试图一点一点地改进答案，确保字典序对位重要性的最优性。 

## 工作示例

 考虑一个数组，我们尝试在其中形成 k 段并观察贪婪切割的行为。 

### 示例 1

 输入：

 n = 5，k = 2

 a = [1, 2, 1, 4, 2]

 我们在施工过程中跟踪候选掩模。 

| 位| 候选人面具 | 段数 | 可行| 选好的面膜|
 | --- | --- | --- | --- | --- |
 | 2 | 4 | 1（不能达到两次）| 没有 | 0 |
 | 1 | 2 | 2 | 是的 | 2 |
 | 0 | 3 | 2 | 是的 | 3 |

 最终答案是3。 

这表明可行性如何取决于能够在多个不相交的段中重复“收集”所需的位。 

### 示例 2

 输入：

 n = 6，k = 3

 a = [3, 1, 2, 3, 1, 2]

 测试掩码 3（位 0 和 1）：

 | 分段构建 | 或 | 行动|
 | --- | --- | --- |
 | [3] | 3 | 切|
 | [1,2]| 3 | 切|
 | [3,1,2]| 3 | 切|

 我们获得了 3 个片段，因此掩码 3 是可行的。 

这表明贪婪切割总是产生最早可能的段边界，从而最大化有效段的数量。

 ## 边缘情况

 关键的边缘情况是 k 等于 1 时。在这种情况下，整个数组是一个段，因此答案只是所有元素的或。 该算法自然地处理这个问题，因为任何可行的位掩码都会通过单段检查。 

另一种边缘情况是当 k 相对于 n 较大时。 如果 k 超过了即使对于掩码 0 也可以形成的最大段数，则所有非零掩码的可行性检查都会正确失败，从而导致答案 0。例如，如果 n = 3 并且 k = 4，则除 0 之外的任何掩码都不能生成 4 个有效段。 

一个更微妙的情况是当位聚集时。 假设高位仅出现在数组的单个区域中。 贪婪分段将最多创建一个包含该位的段，从而导致对该位的检查失败，即使它总共出现了很多次。 这是正确的，因为要求是每个段的，而不是全局频率。
