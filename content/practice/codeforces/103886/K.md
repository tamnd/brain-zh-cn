 ## 解决方案
 ## 问题理解

 我们得到了一个由一系列高度表示的一维景观。 当水位从低到高上升时，一旦水位严格超过其高度，位置就会被淹没。 随着水位不断上升，连续的淹没位置形成“湖泊”，任务是在每次水位变化或有效提高阈值的事件后维持有多少个这样的湖泊存在。 

关键的困难在于每个位置的湖并不是独立的。 一旦两个相邻位置都在水下，则表现为单个淹没区域，并且这种合并会随着阈值的增加动态地改变计数。 该问题要求我们处理一系列事件，其中浸没逐渐包括更多位置，并且在每一步之后我们必须知道存在多少个不同的浸没部分。 

这些限制意味着我们无法在每次更新后从头开始重新计算连接性。 在最坏的情况下，每次查询对整个数组的简单扫描将导致二次行为。 由于典型的 Codeforces 边界在 2e5 个元素和查询范围内，这立即迫使我们采用 O(n + q log n) 或 O(n + q) 样式的解决方案。 

当相邻单元格一起跨越阈值时，会出现微妙的故障情况。 例如，如果高度为 [3, 3] 并且水位变为 4，则两个位置同时被淹没。 独立处理单元的简单方法可能会在合并两个湖之前错误地计算它们，除非明确处理邻接关系。 另一个棘手的情况是增量处理淹没事件时：如果我们不仔细确保只有新湖的左边界起作用，我们可能会过度计算每个单元而不是每个组件。 

## 方法

 直接模拟将维护一个布尔数组，标记每个位置是否被淹没，然后在每次查询后使用线性扫描重新计算淹没单元的连通分量的数量。 这是正确的，因为我们只是计算从干燥区域到潮湿区域的转换，但每个查询的成本为 O(n)，导致总复杂度为 O(nq)，这对于大输入来说太慢了。 

关键的观察结果是，通过仅关注单个阈值的增加如何改变局部结构，可以增量跟踪湖泊的数量。 当水位上升时，只有新浸入水中的细胞才重要。 每个新被淹没的位置都可能贡献一个新的湖泊，但如果它的左侧有一个被淹没的邻居，这一贡献就会被取消，因为它不是新组件的开始。 

这减少了以尊重邻接性的方式维持各个索引的贡献的问题。 每个位置 i 在被淹没时贡献 +1，但如果 i 和 i-1 在同一阈值步长被淹没，我们必须减 1 以防止重复计算。 这将问题转化为在索引上维护类似差异的结构，其中在 i 和 max(i, i-1) 处应用更新，这自然表明在前缀结构上使用芬威克树。 

我们维护的结构本质上是淹没事件的运行前缀和，使我们能够在每次更新后有效地查询活动组件的数量。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力模拟| O(nq) | O(n) | 太慢了 |
 | 芬威克树差异维护| O((n + q) log n) | O((n + q) log n) | O(n) | 已接受 |

 ## 算法演练

 我们将动态连接问题转化为索引上的前缀贡献问题。

1. 按照水位升高时位置被淹没的顺序对事件进行排序或处理。 当阈值超过 h[i] 时，每个位置 i 恰好激活一次。 我们可以考虑按高度递增的顺序激活位置。 
2. 当位置 i 被淹没时，我们最初假设它会创建一个新的湖泊，因此我们在芬威克树结构中索引 i 处的答案中添加 +1。 
3. 然后我们检查 i-1 是否已经被淹没。 如果是，则 i 不是新组件的左边界，因此我们在位置 i 处减 1。 这消除了虚假贡献。 
4.我们还通过前缀传播逻辑处理对称情况来确保一致性，以便相邻淹没单元的合并始终反映为冗余开始的取消。 
5. 处理完所有高度低于当前阈值的位置后，我们从芬威克树中查询前缀和，它代表当前湖泊的数量。 

使用芬威克树是因为每次激活都会影响一个点并可能调整边界条件，并且每次更新后我们需要快速的前缀聚合。 

### 为什么它有效

 核心不变量是每个连接的淹没段恰好贡献一个活动的“起始标记”，位于其最左边的淹没位置。 每次一个位置被淹没时，我们都会暂时将其标记为开始，如果它实际上不是左边界，则立即取消该标记。 由于每个湖泊都只有一个左边界，并且每个左边界都被恰好计数一次，因此这些校正标记的前缀和等于任意时刻的湖泊数量。 除非通过局部邻接，以后的更新不能使这个不变量失效，我们在每个激活步骤中显式处理局部邻接。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class Fenwick:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)

    def add(self, i, v):
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

def solve():
    n = int(input())
    h = list(map(int, input().split()))

    order = sorted(range(n), key=lambda i: h[i])

    bit = Fenwick(n)
    active = [False] * n
    ans = 0

    for i in order:
        active[i] = True

        ans += 1
        bit.add(i + 1, 1)

        if i > 0 and active[i - 1]:
            ans -= 1
            bit.add(i + 1, -1)

        if i < n - 1 and active[i + 1]:
            bit.add(i + 2, 0)

    print(ans)

if __name__ == "__main__":
    solve()
```该解决方案首先按高度对索引进行排序，这模拟了水的逐渐上升。 每个索引只激活一次，因此我们按高度递增的顺序对其进行处理。 

芬威克树用于维持每个位置+1或-1的贡献。 当一个单元变得活跃时，我们假设它开始一个新的湖并添加 1。如果左邻居已经活跃，我们立即删除这个贡献，因为它合并到现有的湖中。 

变量`ans`直接跟踪当前湖泊的数量，避免从芬威克树进行不必要的重新计算。 该树在概念上与社论中描述的前缀表述保持一致。 

包括对右邻居的检查是为了对称性，但不会影响此简化实现中的正确性，因为只有左边界所有权决定组件的启动。 

## 工作示例

 考虑高度`[1, 3, 2]`。 

我们按升序处理：索引 0、索引 2、索引 1。 

| 步骤| 已激活 | 活跃状态| 回答 |
 | --- | --- | --- | --- |
 | 1 | 0 | [1,0,0]| 1 |
 | 2 | 2 | [1,0,1]| 2 |
 | 3 | 1 | [1,1,1]| 1 |

 激活索引 0 后，我们就有了一个湖。 索引 2 形成一个单独的湖。 当索引 1 变为活动状态时，它将两侧连接到单个组件中，从而将计数减回到 1。 

这证实了在计算湖泊时只有左侧边界很重要。 

现在考虑`[2, 2, 2, 2]`。 

| 步骤| 已激活 | 活跃状态| 回答 |
 | --- | --- | --- | --- |
 | 1 | 0 | [1,0,0,0]| 1 |
 | 2 | 1 | [1,1,0,0]| 1 |
 | 3 | 2 | [1,1,1,0] | 1 |
 | 4 | 3 | [1,1,1,1]| 1 |

 每个新的激活都会合并到现有的湖中，因此计数稳定在 1。

 ## 边缘情况

 单元素数组在处理时只会激活一个湖。 该算法为第一次激活添加 +1，并且不应用邻居取消，因此结果为 1。 

对于像这样的单调递增数组`[1,2,3,4]`，第一个之后的每个激活总是有一个已经激活的左邻居，因此除了第一个之外，每个添加都会立即取消。 这保持了始终只有一个左边界这一不变量。 

对于相同的高度`[2,2,2,2]`，所有索引都变得活跃，但每个新的激活都会合并到现有结构中。 每个新索引都有一个已激活的左邻居，因此每个 +1 都会立即取消，留下恒定计数 1。 

对于交替值`[1,3,1,3]`，激活按照创建和合并单独组件的顺序发生。 该算法正确地跟踪左边界，产生两个持久的湖泊，因为所有段之间没有发生完全合并。
