 ## 解决方案
 ## 问题理解

 我们有一个项目集合，对于每个项目都有几种互斥的执行方式。 每种方式都有成本和收入。 对于每个项目，我们必须要么准确选择其中一种可用的方式，要么完全跳过该项目。 所有选择方式的总成本不得超过全球预算，而目标是最大化收入总和。 

重新解释结构的一个有用方法是每个项目贡献一个小的“选择组”项目。 从每一组中，我们要么拿走一件物品，要么什么都不拿走。 每个项目的权重等于成本，价值等于收入。 约束是所有组共享一个背包容量，但还有一个额外的限制，即我们不能从每个组中拿走多于一件物品。 

约束是故意不对称的。 项目数量最多为20个，所有项目的选项总数也最多为20个。但是，预算可能非常大，最多可达10^9，这会立即排除任何依赖预算作为维度的DP。 任何尝试迭代成本状态的解决方案都会在内存和时间上失败。 

这将整个困难从基于容量的背包转移到项目或组的子集枚举上。 

当项目只有昂贵的选项但跳过是最佳选择时，就会出现微妙的边缘情况。 例如，如果一个项目有选项（收入 100，成本 10）和（收入 80，成本 9），并且预算为 5，则这两个选项都不可行，因此正确的行为是完全跳过该项目，从中产生收入 0。 任何假设“每个项目必须选择一个选项”的幼稚实施都会错误地强制选择超出预算的选择。 

另一个边缘情况是所有成本为零。 然后每个项目都可以自由选择，并且最佳答案只是每个项目最佳收入选项的总和，因为预算是无关的。 此案例还暴露了错误地将成本约束视为可选而不是严格的实现。 

最后，由于所有项目的总选项很小，关键的结构提示是我们可以将每个单独的选项视为全局子集问题中的原子项，但我们必须确保永远不会从同一项目中选择两个项。 

## 方法

 一种直接的方法是将其视为一个多选背包，其中预算为 DP 维度。 这导致像 dp[i][b] 这样的状态代表第 i 个项目和预算 b 之后的最佳收入。 这是不可能的，因为 b 达到 10^9，所以即使是单个 DP 层也是不可行的。 

我们可以尝试仅使用观察到的成本来压缩预算，但由于成本在 10^9 以内是任意的并且不受总和限制，因此这仍然没有帮助。 

关键的观察结果是，选项总数只有 20 个。这个值足够小，可以对所有可用选项的子集进行指数枚举。 然而，超过 20 个项目的朴素子集枚举给出了 2^20 种可能性，大约是一百万种，完全可以接受。 

复杂的是每个项目的约束：我们不能选择属于同一项目的两个选项。 这表明我们需要根据项目冲突对子集的可行性进行编码。 

解决此问题的简洁方法是将每个选项视为由其项目索引标记的项目。 然后我们枚举所有选项的子集，对于每个子集，我们通过确保没有两个项目共享一个项目来检查有效性。 如果有效，我们会计算总成本和收入并检查预算可行性。 

由于总项目只有 20 个，因此检查每个子集的时间复杂度为 O(20)，总共为 O(20 * 2^20)，这是高效的。

另一种观点是在项目子集上按项目进行 DP，但由于每个项目都有多个选项，因此展平和验证子集更简单且不易出错。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 预算 DP 超过成本 | O(n·b) | O(n·b) | O(b) | 不可能，因为 b 高达 1e9 |
 | 所有选项的子集枚举 | O(2^m·m) | O(2^m·m) | O(米) | 已接受 |

 ## 算法演练

 ## 算法演练

 1. 将所有可用选项收集到一个列表中，其中每个选项存储其项目 ID、成本和收入。 我们通过允许省略其所有选项的子集将跳过项目视为隐式包含。 
2. 使用从 0 到 2^m − 1 的位掩码迭代这些选项的所有子集，其中 m 是选项总数。 
3. 对于每个子集，使用布尔数组或位掩码跟踪是否已选择项目。 如果我们遇到属于同一项目的两个选项，我们立即丢弃这个子集。 这可以防止非法配置。 
4. 在扫描有效子集的同时，累计总成本和总收入。 这种增量计算确保每个子集评估都是 O(m)，而不是从头开始重新计算。 
5. 如果总成本在预算范围内，请用迄今为止看到的最大收入更新答案。 

我们在迭代过程中明确验证而不是预过滤的原因是冲突取决于所选项目的组合，而不是单独的单个项目。 

### 为什么它有效

 每个可行的解决方案都准确对应于选项的子集，其中每个项目最多包含一个选项，并且其总成本不超过预算。 该算法枚举了选项的所有子集，因此它当然枚举了所有可行的解决方案。 有效性检查准确地过滤掉那些违反每个项目一个选项规则的子集，而预算检查则强制执行可行性。 由于我们评估每个有效配置的收入，因此最大记录值必须等于最优解决方案。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, b = map(int, input().split())
    items = []

    for i in range(n):
        k = int(input())
        for _ in range(k):
            r, c = map(int, input().split())
            items.append((i, c, r))

    m = len(items)
    ans = 0

    for mask in range(1 << m):
        used_projects = 0
        total_cost = 0
        total_rev = 0
        ok = True

        for j in range(m):
            if mask & (1 << j):
                pid, cost, rev = items[j]

                if used_projects & (1 << pid):
                    ok = False
                    break

                used_projects |= (1 << pid)
                total_cost += cost
                total_rev += rev

        if ok and total_cost <= b:
            if total_rev > ans:
                ans = total_rev

    print(ans)

if __name__ == "__main__":
    solve()
```该实现将所有项目选项扁平化为单个数组，同时保留项目标识。 子集枚举对这些选项使用位掩码。 对于每个掩码，我们维护一个项目使用位掩码，以确保没有项目贡献多个选定选项。 

成本和收入累加器是简单的求和。 确保有效性后才进行预算检查，避免了浪费比较。 

一个微妙的实现细节是我们为项目使用使用单独的位掩码而不是一组。 这使得每个项目的检查时间为 O(1)，并避免了 Python 开销。

import sys, io

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    import sys
    input = sys.stdin.readline

    n, b = map(int, input().split())
    items = []
    for i in range(n):
        k = int(input())
        for _ in range(k):
            r, c = map(int, input().split())
            items.append((i, c, r))

    m = len(items)
    ans = 0

    for mask in range(1 << m):
        used = 0
        cost = 0
        rev = 0
        ok = True

        for j in range(m):
            if mask & (1 << j):
                pid, c, r = items[j]
                if used & (1 << pid):
                    ok = False
                    break
                used |= (1 << pid)
                cost += c
                rev += r

        if ok and cost <= b:
            ans = max(ans, rev)

    return str(ans)

 ## 边缘情况

 一种边缘情况是预算为零。 该算法仍然枚举子集，但只有总成本为零的子集才能生存。 这正确地产生了零收入或任何零成本选项（如果存在）。 

另一种情况是，一个项目有多个廉价选项，但只应选择一个。 项目位掩码确保从同一项目中选择两个选项的任何子集都会立即被拒绝，因此即使两者单独负担得起，它们也永远不会一起贡献。 

最后一种情况是所有选项都很昂贵。 子集循环仍然运行，但每个非空子集都无法通过预算检查，并且只有空子集仍然有效，产生输出零。
