 ## 解决方案
 ## 问题理解

 我们得到了一系列按固定顺序排列的节日摊位。 每个摊位都有两个属性：杰克进入它会遇到的人数，以及他进入它所获得的幸福感。 杰克从左向右走，决定每个摊位是进入还是跳过。 

该决策由阈值驱动。 只有当一个摊位的幸福度严格大于这个阈值时，它才有资格被考虑。 在符合条件的摊位中，杰克对满足的总人数仍然有硬性限制，他按顺序处理摊位，每当下一个符合条件的摊位不超过限制时，他就会贪婪地进入。 

因此，对于固定阈值，该过程是确定性的：我们从左到右扫描，忽略幸福度太小的摊位，否则在剩余容量允许的情况下尝试采取它们。 目标是从所选摊位中积累的总幸福感。 任务是选择最大化总幸福感的阈值，如果多个阈值给出相同的最佳值，我们必须输出最小的阈值。 

这些约束清楚地表明，任何独立尝试所有阈值并从头开始模拟过程的解决方案都将太慢。 所有测试用例中的每个测试最多有 200,000 个停顿，每个阈值的简单重新计算将导致最坏情况下的二次行为，这远远超出了时间的适应范围。 

在一些微妙的情况下，直觉可能会失败。 一是降低门槛似乎应该有所帮助，因为它允许更多“好”摊位，但实际上可能会造成伤害，因为低幸福感但高成本的摊位会更早得到处理并消耗容量。 

例如，考虑这样一种情况：一个高幸福感的摊位出现较晚，但较早出现了几个中等幸福感的昂贵摊位。 低门槛涵盖了所有这些，杰克的能力在到达最佳摊位之前就耗尽了，从而降低了总体幸福感。 较高的阈值会排除那些早期阻碍者，并允许采取最佳摊位。 

另一个重要的边缘情况是多个阈值给出相同的结果。 由于答案需要最小阈值，因此我们必须按照资格递减的方向正确跟踪最早的阈值，而不仅仅是第一次看到最大值时。 

## 方法

 蛮力的想法很简单。 对于每个可能的阈值，我们运行完整的从左到右的模拟：检查每个摊位是否符合条件，然后在可能的情况下贪婪地接受它。 这是正确的，因为该过程正是问题所定义的。 然而，阈值最多可以采用 1e9 个不同的值，即使我们将自己限制为重要的值，即不同的幸福值，仍然有多达 200,000 个候选值。 每个模拟都是线性的，因此在最坏的情况下这会变成大约 2e10 次操作，这是不可行的。 

关键的观察结果是阈值不会改变处理顺序或容量规则。 它仅控制哪些停顿对贪婪进程“可见”。 因此，问题变成了按照幸福感递减的顺序动态启用摊位并跟踪贪婪选择如何演变的问题。 

如果我们按照幸福感递减的顺序对档位进行排序，那么随着阈值的降低，档位会按照这个顺序一一激活。 在任何时刻，活动集都由该排序列表的前缀组成。 对于每个前缀，我们需要对原始索引顺序运行相同的贪婪扫描但仅考虑活动停顿的结果。

我们不是从头开始重新计算每个前缀，而是增量地模拟该过程。 当停顿变得活跃时，我们仅根据之前最终确定的决策立即决定贪婪进程是否会接受它。 这是有效的，因为索引顺序中的早期决策一旦做出就永远不会改变，并且贪婪规则仅取决于过去的选择。 

为了有效地支持这一点，我们维护一个数据结构，用于存储哪些已接受的停顿对当前总成本有贡献，因此我们可以查询在到达新停顿之前已经使用了多少容量。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力破解阈值 | O(N²) | O(N) | 太慢了|
 | 按幸福感+增量贪心模拟排序 | O(N log N) | O(N log N) | O(N) | 已接受 |

 ## 算法演练

 ### ## 算法演练

 1. 将所有摊位按照幸福感降序排列。 这决定了随着阈值降低，摊位变得合格的顺序。 幸福感较高的摊位会更早出现。 
2. 维护一个结构，跟踪哪些已接受的摊位对会见的总人数有所贡献，并支持对摊位索引进行前缀和查询。 需要了解在给定位置之前已经使用了多少容量。 
3. 维护当前使用的人数和所选摊位累积的总幸福感。 这些代表了当前贪婪过程的状态。 
4. 按照幸福感从高到低的顺序扫过摊位。 当停顿变得活跃时，尝试完全按照原始贪婪规则按索引顺序进行处理。 
5. 对于位置 i 的摊位，计算在 i 之前的摊位中已经选择了多少人。 如果添加此停顿不超过限制 P，则接受它并更新结构和运行总计。 否则，永久跳过它。 
6. 在每个激活步骤之后，记录当前的总幸福感作为候选答案。 对应于这个时刻的阈值正好低于当前的幸福水平，因为这是这个摊位刚刚符合资格的点。 
7.记录迄今为止看到的最好的幸福。 如果稍后在扫描中再次获得相同的幸福感，请将存储的阈值更新为较新的阈值，因为它对应于较小的阈值。 

### 为什么它有效

 一旦我们确定了哪些摊位符合资格，贪婪过程就完全确定了。 当我们按照幸福感递减顺序处理停滞时，我们正在按照阈值降低时出现的顺序精确模拟所有可能的资格集。 由于每个停顿最多决定一次，并且其决定仅取决于索引顺序中先前固定的决定，因此未来的激活不会使先前的选择无效。 这使得增量模拟相当于为每个阈值单独运行完整的贪婪过程。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class BIT:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)

    def add(self, i, v):
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

    def range_sum(self, l, r):
        return self.sum(r) - self.sum(l - 1)

def solve():
    T = int(input())
    for _ in range(T):
        N, P = map(int, input().split())
        h = [0] * N
        p = [0] * N

        for i in range(N):
            h[i], p[i] = map(int, input().split())

        order = sorted(range(N), key=lambda i: -h[i])

        bit = BIT(N)
        used_people = 0
        total_happiness = 0

        best_happiness = 0
        best_threshold = 10**18

        i = 0
        while i < N:
            j = i
            while j < N and h[order[j]] == h[order[i]]:
                j += 1

            for k in range(i, j):
                idx = order[k] + 1

                people_before = bit.sum(idx - 1)

                if people_before + p[order[k]] <= P:
                    bit.add(idx, p[order[k]])
                    used_people += p[order[k]]
                    total_happiness += h[order[k]]

            current_threshold = h[order[i]] - 1 if i < N else 0

            if total_happiness > best_happiness or (
                total_happiness == best_happiness and current_threshold < best_threshold
            ):
                best_happiness = total_happiness
                best_threshold = current_threshold

            i = j

        print(best_happiness, best_threshold)

if __name__ == "__main__":
    solve()
```该解决方案在索引上构建 Fenwick 树，以维护任何前缀中已采用的总人数。 这使得每次可行性检查都可以在对数时间内完成。 

扫荡摊位是按照幸福感递减的顺序进行的。 按等幸福度分组可确保共享相同阈值边界的所有摊位一起处理，因此阈值更新是一致的。 

一个微妙的一点是，一旦摊位因容量而被拒绝，就永远不会重新考虑。 这与原始的贪婪过程相匹配，一旦按照索引顺序做出决策，决策就是最终的。

 ## 边缘情况

 当高幸福度停顿出现较晚但仅因为较早激活的停顿消耗了容量而被阻止时，就会出现一种边缘情况。 在这种情况下，算法正确地避免了早期的过度提交，因为决策是按索引顺序固定的，并且容量检查可以防止无效包含。 

另一个边缘情况是多个摊位共享相同的幸福值。 由于它们在扫描中同时激活，分组可确保它们在一批中进行处理，因此它们都不会错误地依赖于同一值级别内的阈值分割。 

最后一种边缘情况是，由于所有 p 值都超过 P，因此无法采取任何停顿。在这种情况下，芬威克树在整个扫描过程中保持为空，总幸福感保持为零，并且最小阈值自然成为扫描产生的最小可能值，这与定义一致。
