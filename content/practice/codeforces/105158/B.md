 ## 解决方案
 ## 问题理解

 我们得到一个序列$n$游戏回合。 每轮开始时，T0xel 的钱包中都会添加一枚硬币，并且硬币永远不会丢失，除非用完。 回合中$i$，有一个成本$c_i$：他在该轮中每次购买探测器都会消耗$c_i$立即硬币。 他可以在一轮中购买任意数量的探测器，只要他能用截至该时刻积累的硬币来支付即可。 

目标是最大化所有回合中购买的探测器总数。 

关键的动态是，代币会随着时间的推移而积累，因此早期几轮的财务状况较差，而后期几轮的财务状况会更丰富。 这造成了时间和成本之间的耦合：在便宜的未来回合中购买比在昂贵的或早期回合中购买更有价值，但我们必须尊重每轮只生成一枚代币。 

约束条件$n \le 2 \cdot 10^5$意味着任何解决方案在每次操作时都必须本质上是线性或对数的。 任何尝试模拟每种可能的购买分布或重复扫描过去几轮的操作都将失败，因为$O(n^2)$行为。 

当幼稚的贪婪策略在不展望未来的情况下尽可能购买时，就会出现微妙的失败案例。 例如，如果早期轮次的成本很小，那么天真的方法可能会在早期过度支出并导致未来的廉价轮次匮乏。 

当早期回合成本非常大而后期回合成本很小时，就会出现另一种边缘情况。 如果以最佳方式保存硬币，则应保留所有早期硬币以供以后购买，但天真的“如果可能就立即购买”策略会浪费它们。 

这些例子表明，一轮内的局部贪婪决策是不够的； 我们需要一个全球排序原则。 

## 方法

 暴力方法会模拟每一轮，跟踪硬币的数量，并在承受能力的情况下尝试该轮中所有可能的购买数量。 这可以看作是，对于每一轮$i$，选择一个整数$x_i$这样$x_i \cdot c_i \le \text{coins available before round } i$，然后更新状态并继续。 

正确性很简单，因为它完全遵守约束。 然而，跨轮代币的可能分配数量呈指数增长$n$，甚至对硬币数量进行动态规划也是不可行的，因为硬币总数可以达到$O(n)$，导致$O(n^2)$或者更糟糕的转变。 

关键的结构见解是，除了可用硬币的限制之外，每个探测器的购买都是独立的，并且无论何时获得，每个硬币都是相同的。 这使我们能够重新解释该过程如下：每一轮都会产生一个供应单位，每次购买都会消耗$c_i$该供应量的单位，其中成本仅取决于我们选择购买的回合。 

我们可以根据全球预算来思考，并决定何时花费硬币最有利可图，而不是模拟时间。 随着时间的推移，最优策略会随着单调调整而变得贪婪：我们总是希望确保昂贵的购买不会抢占未来进行更便宜的购买的能力。 

这将问题转化为随着预算的增加而导致的成本流优化问题。

 ## 算法演练

 我们按顺序处理回合，同时保持可用硬币的总数。 在每一步中，我们都会决定本轮购买多少个探测器，但我们不会做出不可逆转的局部决策，而是确保购买的分布保持全局最优。 

1. 初始化变量`coins = 0`和`answer = 0`。 这些硬币代表了之前所有轮次的累计预算总额。 
2. 循环迭代$i = 1$到$n$。 每轮开始时，递增`coins`加 1 来计算新赚取的硬币。 
3. 对于当前成本$c_i$，计算如果我们将所有当前代币投入本轮，理论上可以购买多少个探测器：`possible = coins // c_i`。 如果我们要在这里分配所有内容，这代表了累积预算的最佳本地使用。 
4. 添加`possible`到`answer`，因为这些购买已承诺。 
5. 减少`coins`经过`possible * c_i`。 这模拟了与这些购买所用金额完全相同的支出。 
6. 进入下一轮。 

微妙的部分是为什么在每一步立即承诺所有负担得起的购买是有效的。 原因是，如果我们按顺序解释支出机会，那么现在未花费的任何代币在未来几轮中只会变得更贵或同样昂贵。 由于每轮都有固定成本，并且未来的轮次无法追溯降低成本，因此当在共享预算下独立考虑每轮成本时，延迟购买永远不会产生优势。 

### 为什么它有效

 该算法保持一个不变量：处理轮后$i$，剩余的硬币不足以在任何已处理的回合中购买另一个探测器，其成本小于或等于已经最佳处理的探测器。 任何剩余的硬币严格来说最好保存在未来几轮中，因为未来几轮的成本可能更低或相同，而且较早的支出并不能改善以后获得更便宜的机会的机会。 这确保了每枚硬币要么立即转换为当前成本水平的最大可能探测器，要么保存下来以备将来可能更便宜的成本使用，永远不会浪费。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    c = list(map(int, input().split()))
    
    coins = 0
    ans = 0
    
    for i in range(n):
        coins += 1
        ans += coins // c[i]
        coins %= c[i]
    
    print(ans)

if __name__ == "__main__":
    solve()
```实施完全遵循逐步模拟。 关键细节是使用`coins %= c[i]`，仅保留本轮购买最大化后剩余的硬币。 这相当于减去`possible * c[i]`但避免溢出并使状态保持最小。 

一个常见的错误是忘记在处理每一轮之前积累硬币，这打破了时间顺序。 另一种方法是尝试在每轮中贪婪地做出决定，而不正确地将剩余的硬币向前推进。

 ## 边缘情况

 像这样的最小情况$n = 1, c_1 = 1$显示立即转换：赚取一枚硬币并立即花费，产生一个探测器。 

对于像这样的情况$c = [100, 100, 1]$，该算法在前两轮中积累了两个硬币而不花费任何东西，然后在最后一轮中将所有三个硬币最佳地转换为三个购买。 取模运算确保在最后一轮之后，不会留下可能在之前被错误分配的剩余硬币。 

在统一成本情况下，例如$c = [2, 2, 2, 2]$，系统表现稳定：每两枚硬币转换为一个探测器，其余的则正确向前传播。 这证实了没有任何一轮会导致决策之间的低效率或隐藏的交互。
