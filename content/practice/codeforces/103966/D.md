 ## 解决方案
 ## 问题理解

 我们得到了排列成一行的一系列位置，其中每个位置都有一定的权重。 然后我们得到一组独立的查询。 每个查询描述了一个从给定索引开始并以固定步长重复向前跳转的过程。 该过程收集所有访问过的位置的值，直到它离开数组边界，并且我们必须输出每个查询的收集总和。 

因此，每个查询都定义了数组内索引的算术级数：从位置 a 开始，然后是 a + b，然后是 a + 2b，依此类推。 任务是计算所有这些位置处的数组值的总和。 

这些约束允许最多数十万个元素和查询。 简单的每个查询模拟可能需要为每个查询逐步执行最多 n 个元素，这在最坏的情况下会导致大约 n × q 次操作，约为 10^10 的量级。 这远远超出了典型时间限制内可以完成的事情。 

关键的困难在于不同的查询可以有不同的步长，并且重复使用相同的数组，因此我们需要利用这些算术级数行为的结构。 

当步长为 1 时，会出现微妙的边缘情况。在这种情况下，查询会退化为从 a 到 n 的完整后缀和。 第二种边缘情况是步长很大，接近 n，其中每个查询仅涉及一个或两个元素。 仅有效处理一种机制的简单统一优化将在另一种机制上失败。 

另一个不明显的问题是错误地混合策略：如果我们尝试预先计算直到 n 的所有步长，内存或预处理就会变得太大。 如果我们只优化小步长，如果处理不当，大步长聚合起来仍然是 TLE。 

## 方法

 暴力破解的想法很简单：对于每个查询，从 a 开始，重复添加当前索引处的值，然后跳转 b 直到超过 n。 这是正确的，因为它完全模拟了流程定义。 然而，它的成本与每个查询访问的元素数量成正比。 在最坏的情况下，当 b = 1 时，单个查询扫描 O(n) 个元素，如果有 q 个查询，则变为 O(nq)，这太大了。 

改进来自于注意到跳跃形成算术级数，并且可以通过对 b 求模的残差类来查看数组。 固定 b 将索引划分为独立的链：所有 a mod b 形式的索引形成一个链，其中每一步在该链中向前移动一个位置。 如果我们沿着这些残基链预先计算前缀和，则每个查询将成为该预先计算的结构中的恒定时间差查询。 

问题是 b 本身可能很大，并且对所有可能的 b 到 n 的预计算链将花费 O(n^2)。 标准的权衡是将步长分为小范围和大范围。 对于小的 b，我们预先计算所有残基类别的 DP 表。 对于大b，每次查询访问的元素数量很少，因此直接模拟已经足够高效。 

这将问题分为两个机制，它们一起有效地覆盖所有查询。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | O(nq) | O(1) | O(1) | 太慢了 |
 | 残差 DP + 阈值分割 | O(n√n + q√n) | O(n√n + q√n) | O(n√n) | O(n√n) | 已接受 |

 ## 算法演练

 我们选择 √n 附近的阈值 B。

1. 我们预处理从 1 到 B 的所有步长 b 的答案。对于每个 b，我们构建一个大小为 n 的数组，在其中计算沿大小 b 的跳跃的累积和。 这意味着我们从右到左处理索引，以便每个位置都可以重用其链中下一个位置的值。 这是正确的，因为一旦我们知道从 i + b 开始的总和，我们就可以通过添加 a[i] 将其扩展到 i。 
2. 预处理后，通过直接返回步长为 b 的索引 a 处的预先计算值，可以在 O(1) 内回答 b ≤ B 的每个查询。 这是有效的，因为 DP 已经代表了从该索引开始的精确跳转链上的总和。 
3.对于b>B的查询，我们直接模拟。 由于每一步至少跳转 B 个位置，因此访问的元素数量最多为 n/B，这个数量很小。 
4. 我们独立地累加每个查询的总和并输出。 

关键的决策点是 B 处的分割，这确保没有查询在步骤计数上和预处理表示上都昂贵。 

### 为什么它有效

 对于小步长，算术级数的结构与残差类别完美匹配，因此每个索引仅依赖于同一类别中的一个未来索引。 这会创建一个干净的循环，其中每个值都被计算一次并重复使用。 对于大步长，级数足够短，直接枚举永远不会积累足够的总工作量来超过限制。 每个索引要么由预先计算的链逻辑处理，要么在所有查询中仅访问少量次。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    w = list(map(int, input().split()))
    q = int(input())

    B = int(n ** 0.5) + 1

    # dp[b][i] will store sum starting at i with step b, only for b <= B
    dp = [[0] * n for _ in range(B + 1)]

    for b in range(1, B + 1):
        for i in range(n - 1, -1, -1):
            nxt = i + b
            if nxt < n:
                dp[b][i] = w[i] + dp[b][nxt]
            else:
                dp[b][i] = w[i]

    out = []
    for _ in range(q):
        a, b = map(int, input().split())
        a -= 1

        if b <= B:
            out.append(str(dp[b][a]))
        else:
            s = 0
            i = a
            while i < n:
                s += w[i]
                i += b
            out.append(str(s))

    print("\n".join(out))

if __name__ == "__main__":
    solve()
```DP表`dp[b][i]`编码从 开始的算术级数的总和`i`有步骤`b`，因此小步机制中的查询减少为直接查找。 反向循环至关重要，因为每个状态都取决于`i + b`，它必须已经计算出来。 

对于大步长，while 循环仍然高效，因为索引快速增加，保证每个查询的迭代次数很少。 

## 工作示例

 考虑一个小数组`w = [2, 3, 5, 7]`。 

询问`(1, 3)`从索引 0 开始，访问 0、3，求和 2 + 7 = 9。

 这证实了稀疏算术级数的正确遍历。 

现在考虑`(2, 2)`从索引 1 开始。序列为 1, 3，即 3 + 7 = 10。 

| 步骤| 我| 访问值 | 运行总和 |

 这证明了当 b 很小时的 DP 机制，其中捕获了后缀结构的重复重用。

 ## 边缘情况

 对于步长 1，每个查询都变成后缀和。 DP必须正确地向后传播值； 否则，前向 DP 将错误地重用未初始化的状态。 输入`[1,2,3,4]`带查询`(2,1)`必须产生`2+3+4=9`，并且向后递归通过从末尾构建来确保这一点。 

对于非常大的步长，例如`b ≥ n/2`，每个查询最多访问两个元素。 模拟分支安全地处理这个问题，并且循环很快终止，因为`i`几乎立即跳出界限。
