 ## 解决方案
 ## 问题理解

 我们得到一串小写英文字母。 我们可以执行任意次数的操作，其中每个操作都会选择一个连续的子字符串并将其压缩为单个重复的字母，具体取决于该子字符串内有多少个不同的字符。 具体来说，如果所选子字符串包含$x$不同的字母，那么在操作之后该子字符串中的每个字符都成为$x$字母表中的第 - 个字母。 

字符串中的每个位置在整个过程中最多只能参与一个操作，这实际上意味着一旦某个字符被某个操作覆盖，它就再也不能成为另一个操作的一部分。 

目标是在应用任何此类操作序列后生成字典顺序上可能最大的最终字符串。 

字符串长度可达$10^6$，这排除了任何试图显式模拟操作或探索子字符串的解决方案。 任何长度的二次方，甚至$O(n \log n)$子字符串上有大量常量是不安全的。 我们需要一个解决方案来提取操作可​​能实现的全局结构属性。 

当人们试图相信部分操作可以改进前缀时，就会出现一种微妙的边缘情况。 例如，人们可能认为对精心选择的片段应用操作可以增强早期字符，同时保留后来的大字母。 但是，任何操作的输出仅取决于该段中不同字符的数量，并且该值全局受整个字符串中不同字符数量的限制。 这种限制最终导致整个建筑空间崩溃。 

## 方法

 暴力策略将尝试枚举所有可能的不相交子字符串序列，计算结果转换后的字符串，并按字典顺序保留最佳值。 即使限制于字符串的分区，也有指数级的多种方法来分割它，并且对于每个段，我们需要计算不同的计数，使得总工作量远远超出任何可行的范围。 

关键的结构观察是每个操作都会用一个值替换一个段，该值仅取决于该段中出现的不同字符数。 仅当该段包含原始字符串中存在的每个不同字符时，该值才会最大化。 任何丢失哪怕一个字符的段都会立即产生一个严格较小的字母表索引。 

这意味着一个强大的全局瓶颈：我们可以在字符串中的任何位置创建的最大可能字母完全取决于原始字符串中不同字符的数量。 任何部分段的组合都不能超过它，并且任何实现它的段都必须包含原始字符串的所有不同字符，这迫使它跨越整个出现的字符集。 

一旦认识到这一点，解决方案空间就会缩减为两个有意义的候选方案。 要么我们根本不执行任何操作并保留原始字符串，要么我们对整个字符串应用单个操作，将所有内容转换为由不同字母总数确定的统一字符。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | 指数| O(n) | 太慢了 |
 | 最佳| O(n) | O(1) | O(1) | 已接受 |

 ## 算法演练

 1. 扫描一次字符串以确定它包含多少个不同的字符。 该值直接决定了我们可以生成的最强的字母，因为每个操作的结果仅取决于不同的计数。 
2. 计算$k$，字符串中不同字符的数量。 这对应于任何有效操作可实现的最高字母索引。 
3. 构造一个候选字符串，由对应的字母组成$k$, 重复$n$次。 这表示对整个字符串应用一项操作的效果。 
4. 按字典顺序将构造的字符串与原始字符串进行比较，并输出较大的字符串。 

仅比较这两个字符串背后的原因是，任何操作要么保留原始字符，要么用不能超过全局不同计数的字母覆盖片段。 没有中间分段可以产生字典顺序上优越的结构，因为任何部分操作要么减少可实现的字母值，要么将多个字符折叠成不能超过全字符串转换的统一块。 

### 为什么它有效

 关键的不变量是每个操作输出一个由其所选段中不同字符的数量索引的字母，并且该数字始终最多是整个字符串中不同字符的总数。 因此，全局最佳字母是预先固定的，并且等于总的不同计数。 

任何不包含所有不同字符的操作都会严格降低其输出值。 任何包含所有不同字符的操作都必然跨越这些字符出现的整个范围，这使得它实际上相当于对整个字符串进行操作。 因此，在任何地方实现最大可能的字母表字母的唯一方法是单个全字符串操作，任何其他策略要么保留原始字符串，要么削弱至少一个位置而不进行后续补偿。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    s = input().strip()
    
    k = len(set(s))
    best = chr(ord('a') + k - 1)
    candidate = best * len(s)
    
    if candidate > s:
        print(candidate)
    else:
        print(s)

if __name__ == "__main__":
    solve()
```该实现直接对两个候选减少进行编码。 集合构造计算线性时间内不同字符的数量。 由此我们得出可能的最高字母。 候选字符串的形成时间复杂度为 O(n)，并且单个词典比较确定输出。 

一个微妙的细节是，字典比较是直接在字符串上完成的，这是有效的，因为两个字符串的长度相等。 这避免了手动逐字符扫描的需要。 

## 工作示例

 考虑输入`abbbac`。 

不同的字符是`{a, b, c}`， 所以$k = 3$。 候选字符串是`ccc`，重复以匹配长度，给出`cccccc`。 

我们比较`abbbac`和`cccccc`：

 | 步骤| 不同的 k | 候选字符串| 比较|
 | ---| ---| ---| ---|
 | 初始| 3 | cccccc | 与原版比较|

 自从`c`按字典顺序大于`a`，候选人获胜，输出为`cccccc`。 

现在考虑`zzab`。 

不同的字符是`{z, a, b}`， 所以$k = 3$，给出候选人`ccc`。 

| 步骤| 不同的 k | 候选字符串| 比较|
 | ---| ---| ---| ---|
 | 初始| 3 | cccc| 与原版比较|

 我们比较`zzab`和`cccc`。 第一个字符已经决定：`z > c`，所以原始字符串按字典顺序更大，我们输出`zzab`。 

这些示例表明，该算法并不假设转换后的字符串总是更好，只是认为它是唯一值得考虑的替代方案。

 ## 边缘情况

 一个关键的边缘情况是当字符串按字典顺序已经大于任何统一结构时。 例如，在`zzzz`，不同计数为 1，因此候选者为`aaaa`。 算法正确输出`zzzz`因为直接比较保留了原始字符串。 

另一个重要的情况是所有字符都不同，例如`abcdefghijklmnopqrstuvwxyz`。 这里候选者变成了一串`z`s，它在任何原始安排中占主导地位，因为第一个字符立即得到改善。 

最后，在高度重复的字符串中，例如`aaaaabaaaaa`，即使周围有一个独特的结构`b`，候选仍然仅基于总的不同计数，并且算法正确地决定折叠所有内容是否会改善或恶化词典顺序，而无需尝试任何部分分割。
