 ## 解决方案
 ## 问题理解

 我们得到了一家酒店，楼层从下到上编号。 每层楼都有结构限制，限制该楼层或其上方任何楼层可以容纳的客人数量。 这些限制是从建筑物顶部向下累积的：如果我们确定每层有多少客户端，那么对于每层，放置在该层及其上方所有楼层的客户端总数不能超过该层的给定阈值。 

一批 m 个客户到达，但事先不知道他们的偏好。 每个客户端都被统一分配一个从 1 到 n 的随机“等级”。 等级为 r 的客户只愿意留在 1 至 r 层，这意味着他们拒绝任何高于其等级的分配。 

在所有等级公布后，我们以最佳方式将客户分配到楼层，以最大化分配的楼层数总和，同时尊重酒店的结构限制和每个客户的等级限制。 感兴趣的数量是所有随机排名分配的最佳总和的期望值。 

约束 n, m ≤ 2000 意味着任何三次解或任何重复模拟每个配置的分配都太慢。 我们需要更接近 O(nm) 或 O(nm log n) 的东西。 该结构强烈表明我们应该避免模拟所有排名排列，而应使用聚合期望。 

当顶层容量紧张时，就会出现微妙的边缘情况。 例如，如果 w = [2, 2, 1] 且 m = 3，则顶层总共只能容纳一名客户，并且由于累积约束，该楼层的决策会影响所有较低楼层。 忽略后缀耦合的贪婪分配可能会溢出较低的楼层并破坏可行性，即使每个单独的楼层约束看起来单独有效。 

另一个重要的情况是当所有 w_i 都很大时，例如对于所有 i，w_i ≥ m。 然后，问题就简化为在等级约束下纯粹最大化预期分配的下限值，并且任何正确的解决方案都应该干净地简化为对称期望参数，而不是完全涉及容量。 

## 方法

 思考问题的一个直接方法是模拟整个过程。 我们将为 m 个客户端生成所有可能的排名分配，使用贪婪策略计算每个配置的最佳位置，并对结果进行平均。 这在概念上很简单，因为一旦排名固定，分配就变成具有嵌套约束的确定性最大权重打包问题。 然而，等级配置的数量为n^m，这远远超出了任何计算限制。 即使单个配置也需要排序或贪婪分配，因此这种方法会立即失败。 

关键的观察是我们实际上不需要推理完整的配置。 唯一的随机性在于有多少客户有资格进入每个楼层，并且资格是单调的：客户有资格进入所有楼层直至其等级。 这将问题转化为一个结构化的资源分配系统，其中每个楼层 i 都有一组预期可用的客户和来自酒店限制的容量。 

我们不再与个别客户合作，而是将视角转向预期流程。 每个楼层都会接待预期数量的合格客户，我们希望将他们从较高楼层向下贪婪地分配。 楼层的累积约束可以通过求差转换为每层的容量：可以精确放置在第 i 层的客户端数量由 c_i = w_i − w_{i+1} 限制，其中我们定义 w_{n+1} = 0。这将后缀约束转换为独立的每层上限。

现在唯一剩下的困难是资格是随机的。 客户有资格获得楼层 i 的概率为 (n − i + 1) / n。 由于预期是线性的，我们可以使用预期供应而不是完全分布。 最佳预期分配的行为类似于分数贪婪填充过程：我们将预期客户从顶层分配到底层，同时考虑容量和预期可用性。 

一旦我们预先计算每个楼层前缀的预期可用性，这就会导致确定性的 O(n) 传递。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力模拟排名 | O(n^m · m log m) | O(n^m · m log m) | O(米) | 太慢了 |
 | 期望流量+楼层贪婪DP | O(n) 或 O(nm) 取决于实现 | O(n) | 已接受 |

 ## 算法演练

 ### 算法演练

 1. 将后缀约束转换为每层容量。 对于每个楼层 i，定义 c_i = w_i − w_{i+1}，其中 w_{n+1} = 0。这隔离了每个楼层可以准确放置多少个客户端，而不会违反较高楼层的限制。 
2. 计算每个楼层 i 允许单个客户端放置在楼层 i 的概率，即 P(r ≥ i) = (n − i + 1) / n。 这提供了跨楼层的统一预期可用性结构。 
3. 乘以 m 以获得符合 i 层或任何较低楼层资格的预期客户数量。 这给出了楼层的累积供应曲线。 
4. 自上而下处理楼层，维护尚未分配的客户的剩余预期供应。 在每个楼层 i，将尽可能多的客户分配到该楼层，这是 c_i 和当前剩余供应量中的最小值。 
5. 从剩余供应量中减去分配的数量并继续向下。 每个分配的单元对最终答案贡献 i，因此累加 i ×分配[i]。 

这种贪婪步骤是正确的原因是，较高楼层在价值上总是支配较低楼层，因此任何在不违反约束的情况下使客户向上移动的任务交换都会严格改善目标。 

### 为什么它有效

 如果每层楼都遵守自己的边界 c_i，则转换为每层楼的容量可确保永远不会违反结构约束。 预期供应模型之所以有效，是因为每个客户都独立地为席位资格做出贡献，因此系统在预期中呈线性行为。 由于目标在分配中也是线性的，因此可以通过首先贪婪地填充最高价值的楼层直到容量或预期供应耗尽来实现最大化预期分配。 一旦较高楼层部分未满而较低楼层已被占用，重新分配分配就无法改善解决方案。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MOD = 10**9 + 7

def modinv(x):
    return pow(x, MOD - 2, MOD)

def solve():
    n, m = map(int, input().split())
    w = list(map(int, input().split()))

    w.append(0)

    c = [0] * (n + 1)
    for i in range(1, n + 1):
        c[i] = w[i - 1] - w[i]

    # expected total supply for floor i or below from all m clients
    # P(r >= i) = (n - i + 1) / n
    inv_n = modinv(n)

    expected_supply = [0] * (n + 2)
    for i in range(1, n + 1):
        expected_supply[i] = m * (n - i + 1) * inv_n % MOD

    remaining = m
    ans = 0

    for i in range(n, 0, -1):
        # available expected mass for floors >= i is remaining * P(r >= i)
        avail = remaining * (n - i + 1) * inv_n % MOD

        take = min(c[i], avail)
        ans = (ans + take * i) % MOD
        remaining = (remaining - take) % MOD

    print(ans % MOD)

if __name__ == "__main__":
    solve()
```该实现首先使用差异将累积楼层约束转换为独立的每楼层容量。 然后，它使用均匀的排名分布对符合每个楼层后缀资格的预期客户数量进行建模。 主循环从上到下处理楼层，贪婪地分配容量和剩余可用性允许的尽可能多的预期质量。 

微妙之处在于我们从不模拟个人客户。 相反，我们维护一个代表剩余预期可分配质量的单个标量，这是有效的，因为所有客户端都是可交换的，并且只有资格概率很重要。

 ## 边缘情况

 当所有楼层的容量都非常大时，算法纯粹根据预期的排名资格进行分配。 在这种情况下，每个楼层实际上都是无界的，贪婪扫描减少为从上到下按比例分配预期的客户质量。 输出成为对下限指数的清晰线性期望。 

当只有最低楼层具有非零容量时，所有预期质量都会流向楼层 1，无论等级如何。 该算法自然会强制执行此操作，因为较高楼层的容量为零，因此永远不会消耗任何预期供应。 

当接近顶部的容量急剧减少时，算法会优先考虑首先填充这些楼层。 即使较低楼层有很大的剩余容量，它们也不会收到分配，直到较高楼层饱和为止，这与目标函数得出的最优结构相匹配。
