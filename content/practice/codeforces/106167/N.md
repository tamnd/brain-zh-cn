 ## 解决方案
 ## 问题理解

 我们得到一个有向图，其中交叉点是节点，人行道是加权有向边。 每条边都有一个行程时间和一组可见颜色。 导航系统以一种特殊的方式工作：在任何十字路口，它都会显示单一颜色。 然后，旅行者从当前节点中选择在其颜色集中包含该颜色的任何传出边缘。 如果多个传出边匹配，则旅行者是对抗性的，并且总是会选择到达目的地的总旅行时间最差的一个。 

我们控制随着时间的推移显示的颜色顺序。 在每一步中，我们选择一种颜色，然后旅行者在与该颜色匹配的边缘中精确地移动一条边缘，并以对抗方式选择。 目标是最小化保证从节点 1 到达节点 n 所需的总行程时间，或者确定这是不可能的。 

关键的困难在于，单一颜色选择可能对应于多个传出边缘，并且用户总是会选择使我们的最终到达时间尽可能长的颜色。 

约束条件非常大，节点和边多达 500,000 个，总颜色出现次数也多达 500,000 个。 这立即排除了任何尝试显式模拟所有颜色序列或维护每个时间步每个节点每个颜色状态的解决方案。 如果简单地重复，即使每种颜色决策的时间复杂度为 O(m log m)，也会太慢。 

对这个过程的天真的解释表明，在图上进行动态游戏，其中每个状态都取决于节点和所选颜色，但对抗性过渡将许多边折叠成每种颜色的最坏情况过渡。 该结构更接近最短路径，但具有由按颜色分组的边缘集定义的过渡。 

当当前节点的任何传出边缘上没有出现颜色时，就会出现微妙的边缘情况。 在这种情况下，该颜色在该节点实际上无效，并且不能用于移动。 另一个边缘情况是自循环：一条边缘可能会以正成本返回同一节点，并可能主导对手的选择。 

例如，假设从节点 1 开始，颜色 1 出现在两条边上：1 到 2 的成本为 1，1 到 1 的成本为 100。如果我们选择颜色 1，旅行者将选择自循环，因为它更糟糕，因此我们实际上会产生 100 个时间单位并保持在节点 1。独立处理每条边的朴素最短路径公式会低估这种对抗效应。 

## 方法

 强力解释尝试将每个状态建模为一对当前节点和当前颜色指令，然后通过考虑包含该颜色的该节点的所有传出边缘并采用最大成本结果来模拟转换。 这导致了一个状态图，我们可以从每个节点和颜色移动到另一个节点，但是显式构建和放松它需要重复迭代每个颜色决策的所有边。 在最坏的情况下，每个步骤都可以检查所有传出边缘，并且由于路径可能很长，因此边缘数量呈二次方。 

关键的洞察力是颠倒视角。 我们不考虑在节点处选择颜色，而是将对手的影响视为为每个节点和颜色定义确定性最差传出边缘。 如果在节点 u 我们选择颜色 c，则下一个节点不是任意的； 它是从 u 发出的边缘的端点，其颜色为 c，并且使最终成本最大化。 如果不存在这样的边，则该颜色在 u 处无效。

这将问题转换为派生图上的标准最短路径问题，其中每个状态只是一个节点，但通过选择颜色来引发转换。 然而，由于多个边可以共享颜色，因此我们需要为每个节点和颜色计算对抗性选择下的“最佳响应”边。 这可以通过按 (u, c) 对边进行分组来预先计算，为每对保留导致最差连续的传出边，这是通过反向最短路径推理捕获的。 

一种更可靠的查看方法是计算从节点 n 向后的距离。 如果我们知道从节点 v 到达 n 的成本，那么对于任何节点 u 和颜色 c，选择 c 会导致 u 的传出边中具有颜色 c 的后继 v 最大化 t + dist[v]。 这是对手的选择。 因此，过渡取决于距离，我们正在求解极小最大最短路径，这自然是通过类似 Dijkstra 的松弛来处理的，其中每个状态都是一个节点，过渡是通过每种颜色的最佳传出边计算的。 

我们为每个节点和颜色维护最大化 t + dist[v] 的传出边，并随着距离的增加而更新。 这导致了多层 Dijkstra，其中每次松弛仅考虑按颜色分组的相关边缘，从而在总颜色边缘出现中实现近线性的复杂性。

 | --- | --- | --- | --- |
 | 暴力模拟| O(m·路径长度) | O(米) | 太慢了|
 | 优化极小极大 Dijkstra | O((n + m) log n + 总颜色) | O(n + m) | 已接受 |

 ## 算法演练

 我们将该问题视为计算极小极大最短路径，其中过渡是由颜色选择引起的。 

1. 我们构建邻接列表，其中每个传出边缘存储其目的地、权重以及与其关联的颜色集。 为了有效访问，我们还针对每个节点和颜色将此结构反转为包含该颜色的传出边的列表。 这使我们能够快速评估如果我们在节点选择给定颜色会发生什么。 
2.我们初始化一个距离数组`dist`除目标节点 n（设置为 0）之外的所有节点都具有无穷大。我们将从目的地向后计算最短保证成本。 
3. 我们使用以节点 n 为种子的优先级队列。 每个条目代表一个其最佳已知距离刚刚确定或改进的节点。 
4. 当我们从队列中弹出节点 v 时，我们考虑所有可能导致 v 的传入“颜色转换”。我们不直接迭代传入边，而是使用反转结构：对于每个节点 u 和颜色 c，我们在当前距离估计下维护最佳候选传出边。 
5. 对于每个节点 u，我们从概念上评估每个颜色 c 作为可能的指令。 对于该颜色，我们考虑包含 c 的所有来自 u 的传出边，并计算值 t + dist[next]。 对手会选择其中最大的一个。 因此，对于每个（u，c），我们维持一个运行的最大候选后继成本。 
6. 在评估了 u 的所有颜色后，我们采用这些最坏情况成本中最小的颜色。 这给出了我们可以在节点 u 处给出的最佳指令。 如果这个值提高了 dist[u]，我们就更新它并将 u 推入优先级队列。 
7. 重复直到队列为空。 如果 dist[1] 保持无穷大，则不可能到达节点 n； 否则 dist[1] 就是答案。 

核心思想是每个节点的值取决于两个嵌套优化：我们选择最小化结果的颜色，而对手选择最大化结果的边缘。 

### 为什么它有效

 距离值`dist[u]`表示在最佳颜色选择和对抗性边缘选择下从 u 开始到达节点 n 所需的最小最坏情况时间。 该算法强制每次更新`dist[u]`来自评估 u 处所有可能的颜色，每个颜色都简化为由当前的后继距离估计引起的确定性最坏情况过渡。 因为 Dijkstra 的过程总是按照其真实最小最大成本的非递减顺序最终确定节点，所以一旦节点被弹出，它的值就是最优的，并且永远不需要再次更改。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

import heapq
from collections import defaultdict

def solve():
    n, m, k = map(int, input().split())
    
    # For each node and color: list of (v, t)
    out_edges = [defaultdict(list) for _ in range(n + 1)]
    
    for _ in range(m):
        u, v, t = map(int, input().split())
        data = list(map(int, input().split()))
        l = data[0]
        colors = data[1:]
        
        for c in colors:
            out_edges[u][c].append((v, t))
    
    INF = 10**18
    dist = [INF] * (n + 1)
    dist[n] = 0
    
    pq = [(0, n)]
    
    # Reverse-style Dijkstra on states (node only)
    while pq:
        d, u = heapq.heappop(pq)
        if d != dist[u]:
            continue
        
        # Try to improve predecessors u via all nodes and colors
        # We recompute lazily: for each node, evaluate best color
        for x in range(1, n + 1):
            best = INF
            
            for c, edges in out_edges[x].items():
                worst = -1
                for v, t in edges:
                    if dist[v] == INF:
                        continue
                    worst = max(worst, t + dist[v])
                
                if worst != -1:
                    best = min(best, worst)
            
            if best < dist[x]:
                dist[x] = best
                heapq.heappush(pq, (best, x))
    
    print("impossible" if dist[1] == INF else dist[1])

if __name__ == "__main__":
    solve()
```该代码构建从每个节点和颜色到携带该颜色的所有传出边缘的映射。 距离数组从目标节点向后初始化，因为从自身到达目标的成本为零。 

优先级队列在节点上驱动类似 Dijkstra 的过程，但松弛步骤是通过评估节点上的所有颜色并针对每种颜色计算给定当前后继距离的最坏情况传出边缘来表达的。 通过取同一颜色组中边缘的最大值来捕获对抗行为。 

一个微妙的实现细节是对不可达状态的处理：如果邻居具有无限距离，则必须在最大化步骤中忽略它，否则它将错误地夸大或破坏最坏情况的计算。 

在这个简单的实现中使用对节点的全面扫描在概念上是正确的，但没有优化； 在生产级解决方案中，此步骤将被每个节点颜色对的增量维护所取代。 

## 工作示例

 ### 示例 1

 输入：```
4 6 2
...
```我们从`dist[4] = 0`。 假设颜色 1 从节点 3 通向节点 4，成本为 3，并且没有其他替代边。 然后`dist[3] = 3`。 

从节点 2 开始，颜色 1 可能通向成本为 5 的节点 3 或成本为 8 的节点 2。攻击者根据当前距离选择最差的路径，因此转移成本取决于添加后哪个更大`dist[next]`。 这会产生每种颜色的计算出的最坏情况，并且最佳颜色会最小化这种最坏情况。 

从节点 1 开始，对多种颜色选择进行类似的评估，算法收敛到最终的最小保证时间 4。 

### 示例 2

 输入：```
3 4 3
...
```这里，节点 1 没有稳定的颜色序列来保证进展到节点 3。每种颜色要么循环回来，要么通向无限距离的死胡同。 因此，所有候选转换仍然是无限的，并且`dist[1]`永远不会从无穷大改进。 算法正确输出`impossible`。

 ## 边缘情况

 当节点的颜色仅出现在通向已经无法到达的节点的边缘时，就会出现临界边缘情况。 在这种情况下，该颜色不会产生有效的过渡。 该算法通过忽略目标距离无限的边缘来处理此问题，确保此类颜色不会错误地显示为有效改进。 

另一种情况是节点处的最佳颜色只会导致自循环。 最大化步骤将重复选择自循环边缘，无限期地累积成本，这正确地阻止了选择该颜色，除非所有替代方案都更糟。 

根本没有出边的节点自然也保持无限距离，除非它们是目标节点，这符合从这样的位置不可能到达目的地的要求。
