 ## 解决方案
 ## 问题理解

 我们得到一条始终按照固定顺序从城市 1 到城市 n 穿过城市的路线。 在每对连续的城市 i 和 i+1 之间，都有几条替代道路，每条道路都有两个属性：行驶时间和最大支持的车辆重量。 

从起点到终点的单次旅行包括为 i 到 i+1 的每个路段选择一条道路。 但是，只有当汽车的重量不超过该道路的重量限制时，所选道路才可以使用。 总行程时间是所有路段所选道路时间的总和。 

地图确定后，我们会得到许多不同重量的汽车。 对于每辆车，我们必须计算从城市 1 到城市 n 的最短可能总行驶时间，否则报告不存在有效的道路序列。 

图的结构很重要：它是分层的。 每一层对应一个固定的过渡 i 到 i+1，除了每层必须选择一条可行边的全局约束之外，层之间的选择是独立的。 

这些约束足够严格，任何针对每个查询从头开始重新计算答案的解决方案都会太慢。 对于多达 100,000 个路段、100,000 条道路和 100,000 个查询，每个查询对路段的线性扫描将导致大约 10^10 次操作，这远远超出了可行的限制。 即使每个段的每个查询二进制搜索仍然太慢。 

天真的贪婪思维中出现了一个更微妙的问题：对于固定权重，选择每个段最便宜的边是正确的，但是每个查询独立地重新计算它会重复几乎相同的工作。 关键是查询的不同之处仅在于边权重的阈值条件。 

当路段没有可容纳重型汽车的道路时，就会出现边缘情况。 在这种情况下，即使所有其他部分都很好，答案也一定是“不可能”。 另一个极端情况是，当整体最便宜的道路无法用于较重的汽车时，迫使答案跳到更昂贵但有效的道路。 最后，每个路段的多条道路可能在容量范围上重叠，因此如果处理不当，最优选择会非单调地变化。 

## 方法

 直接方法独立评估每个查询。 对于给定的汽车重量，我们扫描每个路段 i 并在容量至少为该重量的道路中选择最短行驶时间。 总结这些就给出了答案。 这是正确的，因为一旦可行性确定，各个部分都是独立的。 问题是运行时：对于每个查询，我们可能会扫描多达 100,000 个路段，并且每个路段可能需要扫描多条道路。 在最坏的情况下，这会变成 10^10 次操作。 

关键的观察是可行性的权重是单调的。 如果一条道路可以承受重量 w，那么它就可以承受任何更小的重量。 这建议按容量排序或清扫。 我们可以一次性处理所有相关更改，而不是单独回答每个查询。 

对于每个路段，最佳可用道路取决于一个阈值：在容量至少为 w 的所有道路中，我们希望时间最短。 随着 w 的减小，更多的道路变得可用，因此每个路段的最佳值只会随着时间的推移而提高。 

这会导致对所有道路进行全局扫描，并按容量或重量排序查询。 我们按照容量降序激活道路，保持每个路段迄今为止的最佳时间。 当处理查询权重 w 时，容量 ≥ w 的所有道路都已处于活动状态，因此维护值通过维护总和在每段 O(1) 时间内给出正确答案。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(q·n·m_i) | O(q·n·m_i) | O(1) | O(1) | 太慢了|
 | 按容量优化扫描 | O((n + m + q) log(n + q)) 或 O(n + m + q) | O(n) | 已接受 |

 ## 算法演练

我们将每条道路视为一个事件，一旦汽车重量阈值低于其承载能力，该事件就变得可用。 我们还将每个查询视为一个事件，我们希望获得当前最佳总行程时间。 

1. 将每条道路转换为与其容量相关的事件 c． 对于从时间为 d 的路段 i 开始的道路，一旦当前权重阈值至多为 c，我们将可能使用它来改进路段 i 的最佳已知选项。 这确保我们只考虑可行的道路。 
2. 将每个查询权重 w 转换为一个事件，该事件在处理完所有容量至少为 w 的道路后询问当前状态。 此时的答案与该车的最佳路线完全对应。 
3.将所有事件按照key降序排序，统一对待道路容量c和查询权重w。 从大到小的处理确保当我们处于阈值 w 时，所有容量≥ w 的道路都已被考虑。 
4. 维护一个数组 best[i]，表示所有激活道路中当前已知的路段 i 的最短行驶时间。 将所有条目初始化为无穷大，因为最初没有道路可用。 
5. 维持所有最佳[i]的运行总和。 该总和表示当前激活的道路集下最佳可行路线的总行驶时间。 当某个细分市场的最佳价值提高时，我们会相应地更新总和。 
6. 浏览事件。 当遇到道路(i,d,c)时，如果d改善了它，则更新best[i]。 当遇到查询w时，如果所有段都可行，则记录当前总和，否则如果任何best[i]仍然无穷大，则记录不可能。 

它起作用的原因来自于单调激活属性。 在扫描过程中的任何一点，激活的道路组正是那些容量至少为当前阈值的道路。 对于每个路段，我们始终保持该路段所有激活道路中的最短时间。 由于一旦可行性确定，每个分段的选择都是独立的，因此这些最小值的总和正是约束下的最优路径成本。 扫描保证每个查询都能看到正确的激活集而无需重新计算。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    
    edges = []
    for _ in range(m):
        i, d, c = map(int, input().split())
        edges.append((c, i - 1, d))  # 0-index segments
    
    q = int(input())
    queries = []
    for idx in range(q):
        w = int(input())
        queries.append((w, idx))
    
    events = []
    
    for c, i, d in edges:
        events.append((c, 0, i, d))  # road event
    
    for w, idx in queries:
        events.append((w, 1, idx, 0))  # query event
    
    events.sort(reverse=True)
    
    INF = 10**30
    best = [INF] * (n - 1)
    total = 0
    inf_count = n - 1
    
    res = ["" for _ in range(q)]
    
    for val, typ, x, d in events:
        if typ == 0:
            i = x
            if d < best[i]:
                if best[i] == INF:
                    inf_count -= 1
                    total += d
                else:
                    total += d - best[i]
                best[i] = d
        else:
            idx = x
            if inf_count > 0:
                res[idx] = "impossible"
            else:
                res[idx] = str(total)
    
    print("\n".join(res))

if __name__ == "__main__":
    solve()
```该代码将每条道路编码为激活事件，将每个查询编码为快照事件。 按容量递减对事件进行排序可确保每次查询时活动集的正确性。 数组`best[i]`跟踪每段最便宜的有效道路，同时`total`增量地维护总和以避免为每个查询重新计算它。 

一个微妙的实现细节是对不可达段的处理。 变量`inf_count`跟踪有多少路段仍然没有有效的道路。 这避免了为每个查询扫描数组，从而使每个查询保持 O(1)。 另一个重要的细节是只有当找到一条更好的道路时才更新总数； 否则我们会错误地重复计算改进。

 ## 边缘情况

 一个关键的边缘情况是，某个路段最初没有可供重型汽车使用的道路。 在扫描中，这表现为 best[i] 保持无限，直到处理足够高容量的边缘。 例如，如果查询在任何兼容边缘激活之前到达，则 inf_count 保持为正，并且答案正确地是“不可能”。 

另一种情况是，当一个路段有许多道路时，时间会减少，但容量也会减少。 该算法确保仅应用在正确阈值下有效的改进，因此低容量廉价道路永远不会错误地超越高容量要求。 

最后一个微妙的情况是多个查询共享相同的权重。 由于事件被排序在一起，所有此类查询都会观察到激活道路的相同快照，从而保证一致的输出而无需重新计算。
