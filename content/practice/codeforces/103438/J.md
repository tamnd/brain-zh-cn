 ## 解决方案
 ## 问题理解

 我们给定一个长度的字符串$2n$，仅由字母 A、B 和 C 组成。任务是将这个字符串的位置集合拆分为$n$不相交的对。 每对必须包含两个索引$l < r$，并且这些位置的字符必须形成三种允许的组合之一：AB、AC 或 BC。 

换句话说，每个索引必须只使用一次，并且每对必须连接两个不同的字母。 我们并不是在任意顺序重排的意义上构建子序列；而是在构建子序列。 我们选择索引对，同时仅保留每对内部的原始排序。 

约束条件$n \le 10^5$表示字符串的长度可达$2 \cdot 10^5$。 任何尝试搜索配对或维护每个子集的复杂状态的解决方案都会太慢。 一个$O(n \log n)$或者$O(n)$需要这种方法，因为可能会通过多达几亿次操作，但是任何二次操作都可能通过$n$不会。 

当一个角色占主导地位时，一种幼稚的失败模式会立即出现。 例如，如果字符串是 AAAA...A，则不存在有效的配对，因为每对都需要两个不同的字符。 当贪婪方法过于急切地对字符进行配对而不考虑未来的可用性时，就会发生更微妙的失败。 例如，在像 ABACBC 这样的字符串中，将第一个 A 与第一个可用的 B 配对可能会阻止稍后涉及 C 的必要配对，即使存在有效的全局匹配。 

核心困难在于选择不是独立的：配对决策会影响以后是否仍可以匹配剩余的字符。 

## 方法

 一个蛮力的想法是将每个位置视为一个节点，并尝试使用回溯构建完美匹配。 在每一步中，我们选择两个未使用的索引，其字符形成有效对，删除它们，然后递归。 这大致探讨了$(2n-1)!!$最坏情况下的可能性，因为在每个阶段我们都会从所有剩余的索引中选择一个索引的合作伙伴。 即使是为了$n = 20$，这变得不可行，并且在$n = 10^5$，这是完全不可能的。 

一旦我们停止考虑全局搜索，转而关注本地可用性，问题的结构就会显着简化。 每种角色类型只需要搭配不同的角色类型即可，总共只有三种类型。 这表明我们不需要跟踪远程组合结构； 我们只需要确保每当我们处理一个字符时，如果可能的话，我们可以立即将其与一些以前见过的不匹配的兼容字符进行匹配。 

这导致了一种使用三个堆栈的贪婪匹配策略，其中一个用于当前未匹配的 A、B 和 C 的索引。 当我们从左到右扫描字符串时，我们尝试将当前字符与先前看到的不同类型的字符（如果可用）配对。 如果不存在兼容的早期字符，我们将存储当前索引以供将来配对。 

关键的想法是，只要我们在配对后来到达的角色时保持灵活性，推迟角色就是安全的。 因为所有边都在不同的字母之间，并且图形是跨不同类型的完整二分图，所以只要我们避免让类型在没有伙伴的情况下搁浅，任何消耗可用匹配的本地配对选择都不会破坏未来的可行性。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力回溯 | 指数| O(n) | 太慢了|
 | 贪婪堆栈匹配| O(n) | O(n) | 已接受 |

 ## 算法演练

 我们维护三个堆栈来存储尚未配对的字符索引。 每个堆栈对应于 A、B 和 C 之一。 

我们还维护一个结果对的列表。 

1. 我们从左到右迭代字符串。 在每个位置，我们都会查看当前角色。 
2. 如果字符是 A，我们首先尝试将其与之前见过的未匹配的 B 进行匹配。如果存在这样的 B，我们弹出其索引并记录该对（B 索引，当前索引）。 如果不存在，我们尝试将其与之前看到的未匹配的 C 进行匹配。如果存在，我们将进行配对（C 索引，当前索引）。 如果两者都不存在，我们将这个 A 索引压入 A 堆栈。 
3. 如果字符是 B，我们首先尝试将其与可用的 A 匹配。如果不可能，我们尝试将其与 C 匹配。如果都不存在，我们将这个 B 索引压入 B 堆栈。 
4. 如果字符是 C，我们首先尝试将其与可用的 A 进行匹配。如果失败，我们尝试将其与 B 进行匹配。否则，我们将这个 C 索引压入 C 堆栈。 
5. 处理完所有字符后，如果任何堆栈非空，我们就会失败，因为某些索引仍然不匹配。 否则，我们正好有$n$对。 

每个步骤内的排序偏好不是任意的。 当选择首先匹配哪种类型时，我们会优先使用稍后替代选项较少的类型。 例如，在处理 A 时，首先将其与 B 进行匹配有助于避免累积 B，而 B 的有效伙伴数量可能会比 C 的数量少，具体取决于未来的结构。 字母之间的对称性使得该规则在所有情况下都是一致的。 

### 为什么它有效

 在扫描中的任何一点，每个堆栈都表示正在等待剩余字符中的兼容伙伴的索引。 不变量是，如果一个字符被压入堆栈，则意味着当时不存在直接兼容的伙伴。 

当我们稍后将传入字符与这些存储的索引之一进行匹配时，我们可以利用最早的机会有效地解决早期的缺陷。 由于所有允许的边都位于不同的字母之间，并且图形在不同类型之间完全连接，因此任何推迟的字符仍然与任何未来的相反类型兼容。 

贪心规则确保我们永远不会同时保留两个不相容的剩余，而以后又无法解决它们。 如果存在解决方案，则始终存在遵循从左到右解析顺序的匹配序列，因此贪婪过程不会过早陷入困境。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input().strip())
    s = input().strip()
    
    stA, stB, stC = [], [], []
    res = []

    for i, ch in enumerate(s, start=1):
        if ch == 'A':
            if stB:
                j = stB.pop()
                res.append((j, i))
            elif stC:
                j = stC.pop()
                res.append((j, i))
            else:
                stA.append(i)

        elif ch == 'B':
            if stA:
                j = stA.pop()
                res.append((j, i))
            elif stC:
                j = stC.pop()
                res.append((j, i))
            else:
                stB.append(i)

        else:  # 'C'
            if stA:
                j = stA.pop()
                res.append((j, i))
            elif stB:
                j = stB.pop()
                res.append((j, i))
            else:
                stC.append(i)

    if stA or stB or stC:
        print("NO")
        return

    print("YES")
    for a, b in res:
        print(a, b)

if __name__ == "__main__":
    solve()
```该实现直接遵循基于扫描的贪婪过程。 每个堆栈存储不匹配字符的索引。 当匹配成功时，我们立即输出该对。 使用从 1 开始的索引与所需的输出格式相匹配。 最终检查确保没有剩余索引，否则表明配置不可能。 

一个微妙的点是我们从不尝试在构建后重新排序对。 正确性取决于每个配对在创建时就已确定的事实。

 ## 边缘情况

 一种病态的情况是指一个角色比其他角色出现的频率更高，例如`A`重复$2n$次。 在这种情况下，所有三个堆栈直到最后都保持为空，并且每个 A 都被压入而没有匹配。 终止时，非空 A 堆栈会触发拒绝，从而正确识别不可能性。 

另一个微妙的情况是交替结构，例如`ABACBC`。 在这里，早期的贪婪选择可能会显得有风险，因为过早地将 A 与 B 配对会减少后来的 C 的可用 B。 但是，堆栈机制确保以后每当存在更好的匹配时，都会立即使用它，并且任何剩余的兼容性都会保留在堆栈中，直到需要时为止。
