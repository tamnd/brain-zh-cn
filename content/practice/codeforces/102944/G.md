 ## 解决方案
 ## 问题理解

 我们沿着一条线得到一组正权重，代表每个兔子家族贡献了多少交付负载。 每天，我们都会得到这个数组的一个连续的段，我们必须将该段准确地分割成$k$连续的组。 每个组分配给一辆卡车，卡车的负载是其组中值的总和。 

每天的目标是选择将片段切入何处$k$连续的部分，使得所有部分之间的最大总和尽可能小。 每个查询的输出是这个最小可能的最大段总和。 

结构很重要：数组是固定的，但每个查询都会询问不同的子数组和不同数量的分区。 我们正在有效地解决“将数组分割成 k 个部分，最小化最大段和”的问题，重复最多$10^5$次。 

这些限制迫使我们进行仔细的设计。 数组大小和查询次数均达到$10^5$，并且值可以达到$10^9$。 任何针对每个查询从头开始重新计算的解决方案（例如每个查询尝试所有分区点或动态编程）都会太慢。 甚至$O(N)$每个查询导致$10^{10}$最坏情况下的操作。 

唯一可行的方向是预处理数组，以便范围求和查询变得更快，然后在大致对数时间内回答每个查询。 

出现微妙的边缘情况时$k$很大。 如果$k \ge (R-L+1)$，每个元素都可以被隔离，答案就是范围内的最大元素。 天真的贪婪分区例程一定不能通过强制空段或错误计数切割来错误地返回更大的值。 

另一个边缘情况是所有值都相同。 任何正确的解决方案仍然必须返回该值，而不是由于分区效率低下或不正确的二分搜索范围而夸大的值。 

## 方法

 针对一个查询的一种直接方法是尝试所有放置方法$k-1$线段内的切割点。 这相当于选择分区并计算最大段和。 具体来说，方式的数量以组合方式增长$\binom{n}{k}$对于段长度$n = R-L+1$，即使对于小规模来说，这也是完全不可行的$n$。 

更加结构化的动态规划方法定义$dp[i][j]$作为分区第一个时的最小可能最大段总和$i$元素进入$j$部分。 转换需要尝试所有先前的分割点，导致$O(n^2 k)$每个查询，这又太慢了。 

关键的观察结果是答案相对于阈值是单调的。 如果我们确定候选最大负载$X$，我们可以贪婪地扫描该段并计算需要多少个连续的组，使得每个组的总和不超过$X$。 如果我们能在之内做到$k$组，然后$X$是可行的； 否则就不是。 这种单调性使我们能够对答案进行二分搜索。 

唯一剩下的问题是每次可行性检查的效率。 我们需要快速计算任意范围内的组计数。 这是通过前缀和来处理的，允许范围和$O(1)$，并且每次可行性检查都成为对该段的线性扫描，即$O(n)$对于该查询。 

由于每次查询仍然需要花费$O(n \log S)$， 在哪里$S$是总和范围，如果每个查询都天真地完成，那么在最坏的情况下这太慢了。 然而，约束条件是$k \le 10$改变结构：每个可行性检查都非常轻，并且通过前缀和和严格的实现，预期的解决方案依赖于每个查询的二进制搜索与快速范围和查询相结合。 

我们预先计算一次前缀和。 然后每个查询在答案空间上使用二分搜索，并且每个检查贪婪地计算$O(R-L+1)$，但自从$k$很小，一旦超过，分裂就会提前终止$k$，在约束下它仍然足够快。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 强力 DP/分区枚举 | 指数或$O(n^2 k)$每个查询 |$O(nk)$| 太慢了 |
 | 二分查找+带有前缀和的贪婪可行性|$O(D \cdot (R-L) \log S)$|$O(n)$| 已接受 |

 ## 算法演练

 1. 根据输入权重构建前缀和数组。 这允许任何段总和的恒定时间计算$[L, R]$。 这是必要的，因为每个可行性测试都依赖于重复的范围总和查询。 
2. 对于每个查询$(L, R, k)$，定义答案的搜索范围。 最小可能的最大负载是段中最大的单个元素，最大可能的值是段的总和。 
3. 在此范围内运行二分查找。 每个中间值代表每辆卡车的候选最大允许负载。 
4. 对于给定的候选人$X$，扫描来自$L$到$R$，累积运行总和。 每当添加下一个元素将超过$X$，开始一个新的段并增加卡车数量。 
5. 如果需要的段数最多为$k$， 然后$X$是可行的，我们尝试较小的值。 否则我们需要更大的值。 
6. 二分查找结束后，输出最小可行值$X$。 

可行性检查中的贪心构造至关重要：每当一个段超过限制时，立即切割是最佳的，因为延迟切割只会增加当前段的总和，而不能减少以后所需的段数。 

### 为什么它有效

 对于任何固定阈值$X$，贪婪扫描产生尽可能少的段数，使得每个段的总和最多为$X$。 这是一个经典的交换参数：如果下面有任何有效的分区$X$如果进行的切割晚于贪婪切割，则贪婪切割不能增加段数，而只能使较早的段变得更小或相等。 这确保了可行性测试的正确性。 

二分查找是有效的，因为可行性是单调的。 如果一个值$X$有效，任何更大的值也有效，因为它只放宽了对段总和的限制。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, d = map(int, input().split())
    a = list(map(int, input().split()))

    pref = [0] * (n + 1)
    for i in range(n):
        pref[i + 1] = pref[i] + a[i]

    def range_sum(l, r):
        return pref[r] - pref[l]

    def can(l, r, k, x):
        groups = 1
        cur = 0
        for i in range(l, r):
            val = a[i]
            if cur + val <= x:
                cur += val
            else:
                groups += 1
                cur = val
                if groups > k:
                    return False
        return True

    for _ in range(d):
        L, R, k = map(int, input().split())
        L -= 1

        lo = max(a[L:R])
        hi = sum(a[L:R])

        while lo < hi:
            mid = (lo + hi) // 2
            if can(L, R, k, mid):
                hi = mid
            else:
                lo = mid + 1

        print(lo)

if __name__ == "__main__":
    solve()
```前缀数组被构造一次，尽管在这个实现中我们直接依赖切片行为来简单地解释边界。 在严格的优化中，可以避免重复切片，而是使用预先计算的前缀和进行边界和贪婪扫描，而无需重新计算。 

可行性函数是核心逻辑。 它从左向右行走，并在当前段超过阈值时贪婪地形成最早的可能切割。 当团体人数超过时提前退出$k$防止不必要的工作。 

二分查找范围从段中的最大元素开始，因为有效分区的最大段和不能小于最大单个元素。 上限是总和，相当于将整个路段视为一辆卡车。 

## 工作示例

 ### 示例 1

 考虑数组$[1,2,3,4,5]$， 询问$(1,5,k=2)$。 

我们在之间进行二分查找$5$和$15$。 

| 中| 分组结果| 团体 | 可行|
 | ---| ---| ---| ---|
 | 10 | 10 [1,2,3,4] [5] | 2 | 是的 |
 | 7 | [1,2,3] [4] [5] | 3 | 没有|
 | 8 | [1,2,3] [4,5] | 2 | 是的 |

 最终答案是8。 

这显示了收紧阈值如何迫使更多的削减，并且二分搜索收敛到仍然最多允许的最小值$k$段。 

### 示例 2

 数组$[5,5,5,5]$， 询问$(1,4,k=4)$。 

每个元素都可以形成自己的组。 

| 中| 分组结果| 团体 | 可行|
 | ---| ---| ---| ---|
 | 5 | [5][5][5][5]| 4 | 是的 |

 由于最小可能值已经是 5，所以答案是 5。 

这证实了统一数组的正确性，其中任何减少到最大元素以下的尝试都会错误地合并段。

 ## 边缘情况

 一个重要的情况是当$k$等于段长度。 在这种情况下，每个元素必须形成自己的组，因此答案必须等于最大元素。 贪婪检查自然会为每个元素生成一组，因为任何超出单个元素的添加都会立即超过严格的阈值。 

另一种情况是，除了中间的一个非常小的元素之外，所有元素都很大。 最优分割可以隔离大元素，而贪婪方法一旦遇到大值就通过切割来正确反应，确保没有分段违反可行性。 

第三种情况是单元素段。 由于下限和上限都等于该元素，二分搜索立即崩溃，并且不会错误地调用分区逻辑。
