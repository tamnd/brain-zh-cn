 ## 解决方案
 ## 问题理解

 我们得到了一个代表城市道路系统的有向图，其中每条道路只能在一个方向上使用，并且每条道路都有单位成本，从某种意义上说，我们最终会计算我们决定保留多少条道路。 

有一个标记为 1 的特殊路口，作为集合点。 总共有 K+1 个人：你和 K 个朋友。 它们中的每一个都从节点 2 到 K+2 之间的不同交汇处开始。 目标是选择有向道路的子集，使得从每个起始交叉点开始，都有一条仅使用所选道路最终通向节点 1 的有向路径。 

选择的成本是其中包含的道路数量。 在所有有效选择中，我们希望道路数量尽可能少。 

另一种表述要求的方式是，我们正在构造原始有向图的一个子图，其中集合 {2, 3, …, K+2} 中的每个源节点都可以到达节点 1，并且我们希望该子图使用尽可能少的边。 

就节点而言，约束非常小，最多有 20 个顶点和最多 40 条边，而 K 最多为 18。这立即表明子集上的指数算法是可接受的，但任何直接在 N 或 M 中指数的算法都是不可接受的。 

天真的解释可能建议独立运行从每个源到 1 的最短路径，并采用这些路径的并集。 这是不正确的，因为独立最短路径可能会以全局次优的方式重叠或发散。 多条路径使用的共享边只应计数一次，并且如果允许更多共享，则最佳结构可能会故意选择更长的单独路由。 

第二个微妙的问题是路径可以任意合并和分支。 由此产生的结构不是一组简单的最短路径，而是一个共享的有向结构，类似于以节点 1 为根的有向 Steiner 树。 

典型的失败情况是，两个源具有几乎不重叠的不相交的最短路径，但稍长的替代路由允许它们共享较大的前缀。 贪婪的单源策略会过度计算边缘。 

## 方法

 如果我们忽略路径之间的相互作用，最简单的想法是计算从每个源节点到节点 1 的最短路径，重建这些路径，并取所有边的并集。 这产生了一个有效的解决方案，因为每个源仍然有一条到 1 的路径。但是，它不是最优的，因为最短路径是独立计算的并且不协调共享结构。 

失败的原因是目标不是最小化每个源的距离，而是最小化所有源中使用的不同边缘的总数。 一旦两条路径共享一个前缀，该前缀成本只需支付一次。 因此，全球解决方案必须明确鼓励合并。 

我们正在构建的结构是一个以节点 1 为根、跨越所有源的有向子图。 这正是具有单位边权重的有向 Steiner 树问题，以节点 1 为根，终端为 K+1 个源节点。 

由于 K 很小（最多 18），我们可以对终端子集使用经典的子集动态规划方法。 关键思想是不仅通过已经连接的终端来描述部分解决方案，而且还通过它们当前在部分结构中“相遇”的节点来描述部分解决方案。 

我们将状态定义为 dp[mask][v]，表示连接状态中所有终端节点所需的最小边数。`mask`这样它们都可以到达所选子图中的顶点 v。 直观上，v 是该部分解中所有路径汇聚的电流吸收器。 

从这个公式来看，有两个自然的操作。 首先，我们可以合并共享同一端点 v 的两个部分解决方案。如果一个解决方案将 mask1 连接到 v，另一个解决方案将 mask2 连接到 v，那么我们可以将它们组合成 mask1 | mask2 位于同一 v 处，无需添加任何新边，因为这两个结构已经独立存在。 其次，我们可以通过图的边扩展连通性，如果存在边 v → u，则将状态从 v 传播到 u，这会增加一条边的成本。 

蛮力视角会考虑所有划分终端的方法以及所有通过图表路由终端的方法，这是组合爆炸性的。 DP 通过将终端集的组合（子集合并）与沿图边缘的移动分开来减少这种情况。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 独立最短路径并集| O(K (M log N)) | O(K (M log N)) | O(N + M) | 错误|
 | 子集上的 Steiner DP | O(3^K + 2^K·M) | O(3^K + 2^K·M) | O(2^KN) | O(2^KN) | 已接受 |

 ## 算法演练

 我们首先将问题转换为有根定向斯坦纳树公式。 我们将节点 1 视为根，将所有节点 2 到 K+2 视为必须连接到结构中的终端。 

1.我们为单个终端初始化dp。 对于每个终端节点 t，我们设置 dp[1 << i][t] = 0，其中 i 对应于该终端。 这代表了一种简单的结构，其中仅连接该终端，并且“交汇点”是终端本身。 
2. 我们准备合并终端子集。 对于任何状态 dp[mask][v]，如果我们已经知道在同一节点 v 结束的两个状态 dp[mask1][v] 和 dp[mask2][v]，我们可以将它们合并到 dp[mask1 | mask2][v] 通过将它们的成本相加。 这是可行的，因为两个子结构在施工成本核算方面是边不相交的，并且我们只是统一终端集，同时保持相同的根 v。 
3. 我们迭代所有掩码并重复应用此子集合并步骤。 这本质上是位掩码上的 DP 子集，其中每个状态可以通过将其掩码分成两个较小的部分来形成。 此步骤确保对于每个掩码，我们都可以从终端的任何分区构建它。 
4. 合并后，我们沿着出边 v → u 传播每个 dp[mask][v]。 如果我们沿着一条边移动，我们用 dp[mask][v] + 1 更新 dp[mask][u]。此步骤允许“交汇点”在图中移动，逐渐将部分解推向节点 1。 
5. 在处理子集合并和边缘松弛之后，答案是 dp[full_mask][1]，因为我们要求所有终端都连接并在节点 1 结束。 

正确性来自于以下不变量：dp[mask][v] 始终表示构建子图所需的最小边数，其中 mask 中的所有终端都可以到达 v。子集合并保留了有效性，因为它结合了两个独立的结构，这两个独立的结构已经保证了同一端点的可达性。 边缘松弛可以保持有效性，因为将路径延伸一条边缘可以保持可达性，同时将成本增加一倍。 在所有掩码和节点上，DP 探索路由和合并路径的所有可能方式，因此最终状态捕获最佳共享结构。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    N, M, K = map(int, input().split())
    edges = [[] for _ in range(N + 1)]
    
    for _ in range(M):
        u, v = map(int, input().split())
        edges[u].append(v)

    terminals = list(range(2, K + 3))
    T = len(terminals)

    INF = 10**9
    dp = [[INF] * (N + 1) for _ in range(1 << T)]

    for i, t in enumerate(terminals):
        dp[1 << i][t] = 0

    for mask in range(1 << T):
        for sub in range(mask):
            if sub & mask == sub:
                other = mask ^ sub
                if other == 0:
                    continue
                for v in range(1, N + 1):
                    if dp[sub][v] == INF or dp[other][v] == INF:
                        continue
                    val = dp[sub][v] + dp[other][v]
                    if val < dp[mask][v]:
                        dp[mask][v] = val

        for v in range(1, N + 1):
            if dp[mask][v] == INF:
                continue
            for to in edges[v]:
                if dp[mask][v] + 1 < dp[mask][to]:
                    dp[mask][to] = dp[mask][v] + 1

    full = (1 << T) - 1
    print(dp[full][1])

if __name__ == "__main__":
    solve()
```该实现保留由终端子集和端点节点索引的 2D DP 表。 初始化步骤以零成本为每个单例终端播种。 子集合并循环在共享节点处组合不相交的掩码，并且转换循环沿有向边推送状态，同时每次扩展对每个选定的边进行一次计数。 

一个微妙的点是子集迭代必须确保子掩码被正确枚举； 内循环结束`sub < mask`位检查可保证正确性，但未进行优化。 考虑到较小的限制，这已经足够了。 

最终答案取自所有终端都已连接且端点为节点1的状态。

 ## 边缘情况

 关键的边缘情况是所有终端都位于通往节点 1 的单个有向链上。在这种情况下，最佳解决方案应该重用该链一次，而不是重新计算单独的路径。 DP 通过在所有节点传播单个部分状态来处理此问题，而不会重复共享前缀的边缘成本。 

另一种边缘情况是每个终端都有到节点 1 的直接边缘。正确答案只是终端的数量，因为不可能共享。 DP 独立地初始化每个终端，然后立即将每个终端放松到节点 1，从而保留正确的线性成本。 

最后一个微妙的情况是，当最佳路径需要在不位于来自各个终端的任何最短路径上的中间节点处合并时。 子集合并步骤确保即使两个终端通过不同的路由到达非根中间节点，它们仍然可以在那里组合并共享向上到根的后续边，捕获全局最优而不是局部最短行为。
