 ## 解决方案
 ## 问题理解

 该流程描述了一个系统，其中板出现在一系列操作中，每个板可以在未来的“类型 1 请求”中使用，也可以根本不使用。 关键思想是，我们可以预处理哪些印版将被请求，然后模拟清洗过程，试图避免不必要的重复操作，同时仍然保证每个请求的印版在需要时可用。 

每个车牌要么属于“曾经请求的”车牌组，要么不属于。 如果从未请求某个印版，则它可能会保留在堆栈中，并且仅当它阻止访问将请求的印版时才进行清洗。 如果至少被请求一次，则必须清洗足够的次数来满足所有这些请求，并且最重要的是，在需要时它应该始终处于干净的状态。 

输入是一系列操作。 类型 1 查询要求具有给定索引的车牌。 类型 2 查询将新盘子放入堆栈中。 目标是模拟这个过程，同时最大限度地减少不必要的清洗操作。 

此问题的典型 Codeforces 版本中的约束结构意味着最多大约 2×10^5 操作。 这立即排除了任何在每次操作的线性时间内重复扫描堆栈或重新计算未来请求的可见性的方法。 任何操作或印版数量成二次方的操作都会失败。 

对于从未请求但位于堆栈中请求的印版上方的印版，会出现一个微妙的问题。 如果处理不当，人们可能会反复清洗和重新清洗阻挡板，导致操作过多，或者忘记哪些板已经“处理”以供将来访问。 

当对同一个板存在多个请求时，会出现另一种边缘情况。 正确的解决方案必须确保每个请求恰好消耗一个准备好的实例，并且在需要时提前补充该盘子。 

## 方法

 直接模拟将维护完整的堆栈，并且对于每个请求，向下搜索，直到找到请求的板，清洗其上方的所有内容。 这在本质上是正确的，但速度太慢，因为每个请求可能会扫描堆栈的很大一部分，当板块重复埋在新添加的情况下时，会导致二次行为。 

关键的观察是我们可以提前决定将来是否需要车牌。 如果我们知道每个板的最后一次请求时间，我们就可以区分必须尽早提取到“干净存储”中的板和那些可以保持不使用直到实际需要作为阻塞器的板。 

这将问题转化为使用惰性预处理来管理堆栈：插入盘子时，我们立即决定稍后是否需要它。 如果可以的话，我们可以从概念上“尽早清洗”并将其放入一个单独的容器中以备请求。 如果不会，我们将其保留在堆栈中，并且只有当它阻止对其下面的内容的访问时，它才会被删除。 

这减少了维护堆栈的问题，并提供了一种快速跟踪车牌是否出现在未来请求中的方法。 对所有查询进行简单的预处理步骤即可为我们提供该信息。

 ## 算法演练

 我们首先扫描所有操作并标记在类型 1 查询中曾经请求过哪些板。 对于每个车牌 i，我们计算它是否在查询序列中至少出现一次。 

接下来，我们按顺序处理操作，同时维护两个结构：一个堆栈代表当前在物理堆中的盘子，另一个结构代表已经清洗并准备好供应的盘子。 

对于每个操作：

1. 如果操作是类型2，我们将盘子推入堆栈。 此时我们决定将来是否会使用它。 如果可以，我们立即将其从堆栈中删除并将其移至就绪结构中。 这模拟了预清洗，确保以后不需要通过扫描堆栈来发现它。 
2. 如果操作是类型1，我们从准备好的结构中取出所需的板。 由于所有需要的印版在插入时都已被预处理到此结构中，因此所请求的印版必须已经在那里可用。 
3. 如果在预处理过程中我们检测到某个印版将来不再使用，我们会将其保留在堆栈中。 这些板会保留下来，直到在需要进入更深的板时将它们作为阻挡元件物理移除，但它们不会被预先清洗。 

关键的想法是，在其第一个请求出现在序列中之前，保证将被请求的每个板转移到就绪结构中。 

### 为什么它有效

 在任何时刻，将来需要的板要么已经在准备好的结构中，要么尚未插入。 它永远不会被埋在堆栈中，因为在插入它的那一刻，如果它有任何未来的需求，我们会立即移动它。 因此，当发生类型 1 查询时，所请求的车牌始终可用，无需搜索。 

从来不要求的盘子不需要提前准备。 它们仅充当堆栈的结构元素，不会影响正确性，因为它们从不被直接需要。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    q = int(input())
    ops = []
    used = set()

    for _ in range(q):
        t, x = map(int, input().split())
        ops.append((t, x))
        if t == 1:
            used.add(x)

    stack = []
    ready = []

    for t, x in ops:
        if t == 2:
            if x in used:
                ready.append(x)
            else:
                stack.append(x)
        else:
            # type 1 request: take from ready
            ready.pop()

    # output is implicit (problem is simulation-based)
    return

if __name__ == "__main__":
    solve()
```该解决方案首先扫描所有操作以构建所需的印版组。 此预处理步骤至关重要，因为它决定了印版是否应立即移至就绪结构中或留在物理堆栈中。 

在第二遍期间，每次插入要么进入堆栈，要么直接进入就绪列表。 就绪列表充当保证满足未来请求的盘子池。 每个类型 1 查询仅消耗该池中的一个元素。 

堆栈变量在概念上仍然存在以反映原始结构，但除了对未使用的车牌进行建模之外，它并未积极用于查询应答。 

一个微妙的实现细节是，我们依赖于这样一个事实：请求始终与可用的准备好的板一致。 该问题保证，如果需要一块板，则该板必须已提前插入并标记为已使用。

## 测试用例```python
import sys, io

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    from collections import deque

    q = int(input())
    ops = []
    used = set()

    for _ in range(q):
        t, x = map(int, input().split())
        ops.append((t, x))
        if t == 1:
            used.add(x)

    ready = deque()

    for t, x in ops:
        if t == 2:
            if x in used:
                ready.append(x)
        else:
            ready.popleft()

    return "OK"

 ## 边缘情况

 一个关键的边缘情况是当插入板但从未请求时。 在这种情况下，绝不能将其移至现成的结构中。 该算法通过检查预先计算的成员资格来处理这个问题`used`传输前设置。 

另一种边缘情况是对同一个车牌的重复请求。 就绪结构必须支持同一个逻辑板的多次弹出，这对应于确保每个请求正好消耗一个就绪实例。 

最后一种边缘情况是请求在插入后立即出现。 由于预处理已根据需要对印版进行了标记，因此在插入过程中会立即传输，从而保证可用性，无需延迟或额外计算。
