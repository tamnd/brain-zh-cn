 ## 解决方案
 ## 问题理解

 我们得到了一系列成分，其中每种成分都有两种获取方式。 您可以直接以固定价格购买它，也可以通过组合其他成分来生产它，这些成分本身也可以购买或生产。 每个生产步骤都会消耗所需的子成分，因此生产一个项目的成本是其依赖项的总和，并且不同结构之间不存在重用或共享。 

在所有成分中，有些成分被标记为所需目标。 任务是计算获得所有所需目标成分所需的最低总成本，利用与直接购买相比可能降低成本的任何组合规则。 

由输入引起的结构是有向非循环依赖图：每个成分都指向构建它所需的成分。 保证任何成分都不能直接或间接依赖于自身，这确保了我们正在研究 DAG 而不是循环系统。 

约束条件很大，成分多达 100,000 个，总依赖项大小为 300,000。 这排除了任何无需记忆而递归地重新计算成本或以二次方方式重复传播更新的方法。 任何解决方案本质上都必须处理每个依赖边恒定的次数。 

当一种成分的制造成本比购买成本低时，就会出现微妙的失败情况，但只能通过一系列依赖关系。 只比较直系孩子的幼稚方法会失败。 另一个陷阱是重复重新计算共享子树的成本，在最坏的情况下很容易爆炸成指数时间。 

考虑这个简单的场景：

 输入：```
2 3
A B
A 10 0
B 100 1 A
C 1 0
```正确答案是 110。如果我们在计算 B 时忘记使用 A 的最便宜的计算成本，而是独立处理依赖关系或多次重新计算 A，我们要么会过度计数，要么会超时。 

另一个极端情况是，购买总是比手工制作更糟糕，但只有在经过多层传播后才会变得明显。 每个节点上任何没有全局传播的贪婪局部决策在这里都会失败。 

## 方法

 蛮力解释很简单。 对于每种成分，通过尝试两种选择来计算其成本：直接购买或递归地求和其所需成分的成本。 然后，对于每种目标成分，将计算出的成本相加。 

这在原则上是正确的，因为每种成分成本定义都得到严格遵守。 然而，递归多次重新访问相同的节点。 在长度为 N 的链式依赖图中，每次调用都会扩展为另一个完整遍历，在最坏的情况下会导致指数重新计算。 即使使用记忆化，粗心的实现仍然可能由于字符串查找和重复解析而反复重新访问依赖关系。 

关键的观察结果是依赖结构是 DAG，因此每个成分成本仅取决于拓扑顺序中先前可计算的成分。 一旦我们计算出某种成分的最低成本，就不需要重新计算。 

这将问题转化为 DAG 上的类似最短路径的传播，其中每个节点的值都是根据已最终确定的邻居计算的。 我们按拓扑顺序处理节点，或者等效地，使用记忆 DFS 来确保每个节点被评估一次。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | 最坏情况 | O(exp) O(N) 递归 | 太慢了 |
 | 最优（DAG DP /拓扑DP）| O(N + E) | O(N + E) | 已接受 |

 ## 算法演练

 我们将每种成分建模为一个节点。 每个节点都有一个基本成本（购买价格）和一个依赖项列表。 目标是计算每个节点可实现的最低成本。 

1. 解析所有成分并为每个成分分配一个唯一标识符。 这允许快速索引而不是重复的字符串比较。 这很重要，因为否则基于字符串的字典查找将在大规模运行时占据主导地位。 
2. 构建依赖图。 对于每种成分，存储构建它所需的成分列表。 还计算每个节点的入度，即它具有的依赖项数量。 
3. 用节点的直接购买价格初始化每个节点的成本。 这代表了我们完全忽略制作的基线选项。 
4. 准备一个包含所有零依赖节点的处理队列。 这些原料只能购买，因此它们的成本从一开始就是最终的。 
5. 按队列顺序处理节点。 对于每个节点 u，将其计算成本视为固定。 然后，对于依赖于 u 的每个成分 v，我们尝试使用 u 的最终成本贡献来放宽 v 的成本。 具体来说，我们为每个 v 维护其依赖成本的运行总和，一旦处理完所有依赖关系，我们就会将该总和与直接购买价格进行比较。 
6. 当一个节点的所有依赖项都已处理完毕后，计算其完整的制作成本作为其依赖项成本的总和。 将其最终成本设置为制作成本和购买成本中的最小值。 
7. 将依赖关系完全解析的节点推送到队列中，继续执行，直到处理完所有节点。 
8. 最后，将计算出的所有所需目标成分的成本相加。 

关键思想是，只有在已知所有成分的依赖关系后才会评估每种成分，从而确保其计算的精心设计成本的正确性。 

### 为什么它有效

维护的不变量是，当处理节点时，其所有依赖项都已计算出最终的最小成本。 由于该图是非循环的，因此存在拓扑排序，并且基于队列的过程恰好强制执行这样的排序。 因此，当我们计算原料的成本时，所使用的每个子原料成本都已经是最优的，并且未来的更新无法进一步降低它。 

## Python 解决方案```python
import sys
input = sys.stdin.readline
from collections import defaultdict, deque

def solve():
    n, m = map(int, input().split())
    targets = input().split()

    idx = {}
    price = [0] * m
    indeg = [0] * m
    deps = [[] for _ in range(m)]
    rev = [[] for _ in range(m)]

    for i in range(m):
        parts = input().split()
        name = parts[0]
        p = int(parts[1])
        g = int(parts[2])

        idx[name] = i
        price[i] = p
        indeg[i] = g

        for j in range(g):
            child = parts[3 + j]
            deps[i].append(child)

    # convert deps to indices and build reverse graph
    for i in range(m):
        new_list = []
        for name in deps[i]:
            new_list.append(idx[name])
            rev[idx[name]].append(i)
        deps[i] = new_list

    sum_dep = [0] * m
    q = deque()

    for i in range(m):
        if indeg[i] == 0:
            q.append(i)

    # topological processing
    while q:
        u = q.popleft()

        for v in rev[u]:
            sum_dep[v] += price[u]
            indeg[v] -= 1
            if indeg[v] == 0:
                # v is ready: decide its final cost
                price[v] = min(price[v], sum_dep[v])
                q.append(v)

    ans = 0
    for t in targets:
        ans += price[idx[t]]

    print(ans)

if __name__ == "__main__":
    solve()
```该实现首先将成分名称映射到索引，以提高图形操作的效率。 依赖列表被转换为邻接形式，以便我们可以向上传播完成。 这`sum_dep`数组累积已处理的依赖项的成本贡献。 

一个微妙的点是，我们只有在节点的入度变为零时才最终确定节点的成本。 这确保我们在决定制作是否比购买便宜之前已经看到了所有必需的组件。 

最后的求和步骤只是检索所有目标成分的计算成本。

 ## 边缘情况

 一种极端情况是，即使手工制作是有效的，购买某种原料的成本却比手工制作的便宜。 例如，如果某个项目具有昂贵的依赖项，算法仍然会正确选择购买成本，因为最后一步会比较两个选项。 

另一种情况是长依赖链。 该算法按照拓扑顺序对每个节点只处理一次，因此即使长度为 100,000 的链也不会导致递归或重复工作。 

最后一种边缘情况是多个依赖项汇聚到一个节点中。 自从`sum_dep`仅在处理每个依赖项后才聚合贡献，最终成本仅反映每个依赖项一次的全部总和，避免重复计算。
