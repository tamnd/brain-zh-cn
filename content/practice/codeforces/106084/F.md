 ## 解决方案
 ## 问题理解

 我们得到了一个 4 x 4 的拉丁方格，上面有四个代表水果的符号，但有些单元格已经是空的。 有效的完整配置始终是拉丁方：每行每列都包含四个水果中的每一个。 输入保证如果我们填充空单元格，则恰好有一个与约束一致的完成。 

从这个完整的有效排列中，一些水果已经被删除，标记为空单元格。 从这个部分观察到的网格开始，两名玩家轮流移除额外的水果。 仅当移除该水果后，剩余的部分填充网格仍然具有返回到原始完整拉丁方的唯一补全时，才允许移动。 

玩家并不是简单地移除尽可能多的水果。 每个人都假设对手也处于最佳状态，并尝试在该限制下最大化自己的最终移除数量。 当无法进行进一步的有效删除时，该过程将停止。 任务是计算在最佳游戏结束时网格中还剩下多少水果。 

该问题的结构规模非常小，因为网格固定为 4 × 4，因此只有 16 个单元格。 这立即排除了每个测试用例的输入大小中任何渐近的情况。 即使我们对配置进行指数搜索，它仍然是可行的，因为状态空间很小并且受到拉丁方规则的严重约束。 

一个微妙的点是游戏描述隐藏了一个更简单的确定性过程。 唯一合法的举动取决于删除后是否保留完成的唯一性。 这意味着真正的问题是哪些单元在结构上对于保持拉丁方的独特性是必要的。 

一种可能使天真的方法感到困惑的边缘情况是假设如果一个单元当前在完成中没有被“强制”，那么它总是可移除的。 这是错误的，因为删除一个单元格可能会带来局部不可见的歧义。 

例如，考虑这样一种情况：删除一条线索使得两个对称完成成为可能，即使每个单独的单元格看起来仍然与原始完成一致。 正确的检查必须考虑全局唯一性，而不是局部一致性。 

另一个边缘情况是假设以固定顺序进行贪婪删除就足够了。 删除的顺序可以暂时改变哪些单元格可以安全删除，因此我们必须推断出一个不存在进一步安全删除的固定点。 

## 方法

 蛮力视角就是从字面上模拟游戏。 从当前网格中，我们考虑所有可能的有效移除序列，在玩家之间交替，并计算最佳游戏下剩余水果的最终数量。 每个状态都需要检查移动是否合法，而合法性检查本身需要计算当前网格存在多少个拉丁方完成。 

即使只有 16 个单元，博弈树也很大，因为每个单元删除分支，并且在每个节点我们可能会从头开始重新计算唯一性。 在最坏的情况下，这会导致序列数量的阶乘增长，乘以回溯求解器的成本，这对于如此小的结构化系统来说是不必要的开销。 

关键的观察结果是，博弈并不依赖于分支意义上的长期战略规划。 仅当保留唯一性时，移动才有效。 一旦一个单元格是可移动的，两个玩家最终都喜欢拿走所有这些可移动的单元格，因为留下一个可移动的单元格永远不会帮助任何一个玩家达到最佳的游戏状态。 因此，该过程会反复删除任何单元格，这些单元格的删除使解决方案保持唯一，直到不存在这样的单元格为止。

这将问题转化为计算仍然唯一确定拉丁方的线索的最小子集。 该子集在拉丁方理论中被称为临界集。 最终的答案就是这个不可约核心中的细胞数量。 

我们可以通过重复测试移除来计算它。 对于每个填充的单元格，我们暂时将其删除并检查生成的部分网格是否仍然具有一个有效的拉丁方补全。 如果是，则该单元是可拆卸的。 我们会继续申请删除，直到无法再删除为止。 

为了支持这一点，我们需要一个求解器来计算具有固定约束的 4 x 4 拉丁方的完成情况，一旦找到多个解决方案就提前停止。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力博弈树模拟| 16 + 重指数 | 高递归状态 | 太慢了|
 | 具有唯一性检查的迭代剪枝| 常数有界（小网格回溯）| O(1) | O(1) | 已接受 |

 ## 算法演练

 我们将网格视为一个约束系统并对其进行反复修剪。 

1. 从给定的部分填充的 4 x 4 网格开始，其中空单元格被忽略约束。 
2. 定义一个函数，计算有多少个与当前网格一致的完整拉丁方补全。 该函数逐行执行回溯，确保每个符号在每行和每列中恰好出现一次，如果找到多个解决方案，则提前停止。 
3. 对于当前网格，逐个尝试每个填充的单元格。 暂时删除它并在修改后的网格上运行完成计数器。 
4. 如果移除该单元格仍能准确完成一次，则将该单元格标记为可移除。 
5. 检查完所有单元格后，同时或以任意顺序依次删除所有标记的单元格。 
6. 重复该过程，直到完全通过没有发现可移动的单元。 

我们在每一轮之后重新计算的原因是，删除一个单元可以改变约束结构，这可以使以前不可移动的单元可移动。 

### 为什么它有效

 在任何阶段，网格都保证具有精确的一次完成，或者通过删除保留唯一性的约束从这样的网格导出。 删除单元格只能增加或保留完成次数。 因此，该过程单调地削弱约束，直到达到一个固定点，其中每个剩余的填充单元对于唯一性都是至关重要的。 该固定点不依赖于安全移除的顺序，因为任何可移除单元最终都可以被移除，而不会影响最终的不可约约束集。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

FRUITS = ['G', 'L', 'M', 'S']
idx = {c: i for i, c in enumerate(FRUITS)}

def count_solutions(grid):
    row_used = [0] * 4
    col_used = [0] * 4
    empty = []

    for r in range(4):
        for c in range(4):
            ch = grid[r][c]
            if ch == 'X':
                empty.append((r, c))
            else:
                bit = 1 << idx[ch]
                row_used[r] |= bit
                col_used[c] |= bit

    sys.setrecursionlimit(10000)
    cnt = 0

    def dfs(i):
        nonlocal cnt
        if cnt > 1:
            return
        if i == len(empty):
            cnt += 1
            return

        r, c = empty[i]
        for k in range(4):
            bit = 1 << k
            if not (row_used[r] & bit) and not (col_used[c] & bit):
                row_used[r] |= bit
                col_used[c] |= bit
                dfs(i + 1)
                row_used[r] ^= bit
                col_used[c] ^= bit

    dfs(0)
    return cnt

def solve():
    grid = [list(input().strip()) for _ in range(4)]

    filled = True
    while filled:
        filled = False
        to_remove = []

        for r in range(4):
            for c in range(4):
                if grid[r][c] != 'X':
                    ch = grid[r][c]
                    grid[r][c] = 'X'
                    if count_solutions(grid) == 1:
                        to_remove.append((r, c))
                        filled = True
                    grid[r][c] = ch

        for r, c in to_remove:
            grid[r][c] = 'X'

    ans = sum(grid[r][c] != 'X' for r in range(4) for c in range(4))
    print(ans)

def main():
    t = int(input())
    for _ in range(t):
        solve()

if __name__ == "__main__":
    main()
```实现的核心是回溯求解器，最多可计算两次完成。 它使用位掩码来跟踪每行和每列中使用的符号，这使得每次分配的有效性检查时间恒定。 

外循环重复扫描所有填充的单元格并测试是否可以在不破坏唯一性的情况下删除每个单元格。 一个常见的实现错误是在迭代过程中立即删除单元，这可能导致依赖于顺序的行为。 在这里，首先收集移除并在扫描后应用。 

另一个微妙之处是求解器中的早期停止。 我们只关心完成次数是否正好为 1 或者大于 1，因此 DFS 一旦找到两个解就终止。 这对于性能稳定性至关重要。

 ## 边缘情况

 一种重要的边缘情况是，删除一个单元格不会立即创建多个补全，而是在进一步删除后会导致未来的歧义。 迭代重新计算步骤可以处理这个问题，因为在每一轮之后，我们都会在更新的约束系统下重新评估所有单元格，确保没有潜在的歧义未被检查。 

另一种边缘情况是多个单元同时可拆卸。 以不同的顺序删除它们似乎很重要，但由于我们总是根据当前的唯一性条件验证删除，因此任何有效的删除顺序都会导致相同的固定点，其中不存在进一步的安全删除。 

最后的边缘情况是网格已经最小化确定时。 在这种情况下，每次删除尝试都无法通过唯一性检查，并且算法会正确输出填充单元格的初始数量而不进行修改。
