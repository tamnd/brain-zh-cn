 ## 解决方案
 ## 问题理解

 我们正在管理一系列反应器，每个反应器的行为就像一个具有两个内部值的小系统：当前压力和最大压力阈值。 最初，每个反应器的压力为零，而每个位置都以自己的阈值开始。 

该系统支持在反应堆范围内进行两种操作。 第一个操作将段中每个反应器的压力增加固定量。 在此更新期间，每当反应堆的压力达到或超过其阈值时，它就会立即排气：其压力重置为零，并且其阈值根据取决于其先前阈值的规则降低。 第二个操作询问在整个历史记录中给定段中总共发生了多少次通风事件。 

关键的困难在于更新是基于范围的，但每个反应器的反应是非线性和有状态的。 一次更新可以触发不同反应器上的多个通风口，并且每个通风口通过降低阈值来永久改变未来的行为。 

这些约束促使我们接近每次操作的线性或摊销对数。 由于多达 200,000 个更新和 200,000 个查询，任何逐步模拟每次更新的每个反应器的解决方案都将失败，特别是因为单个更新可能级联多个元素，并可能导致重复的全阵列扫描。 

一种简单的方法是，对于每次更新，迭代该范围内的所有反应器并重复模拟压力增加直到稳定。 在阈值较小且更新频繁的最坏情况下，每个查询可能会处理单个反应器多次，从而导致二次行为。 

当重复更新缓慢降低阈值时，就会出现幼稚思维的微妙失败案例。 

示例场景：

 输入：```
1 3
5
1 1 1 3
1 1 1 3
1 1 1 3
```这里每次更新最终都会导致重复发泄。 幼稚的实现可能只在每次更新时检查一次并错过重复的周期，或者可能从头开始重新计算，但由于重复扫描而仍然是 TLE。 

正确的输出将反映多个通风口，并且正确性取决于模拟，直到反应器在每次范围添加后稳定为止。 

范围更新、每个元素阈值衰减和重复离散事件的组合表明我们需要一种可以快速跳过“安全”元素并仅处理那些接近阈值交叉的结构。 

## 方法

 暴力模拟独立处理每个更新。 对于类型 1 的每个操作，我们都会迭代该范围内的所有反应器并增加其压力。 如果任何反应器超过其阈值，我们将重置其压力并调整其阈值，然后重复，直到段中不再有反应器溢出。 

这是正确的，但非常昂贵。 每个反应堆可能被访问多次，并且单个更新可以触发级联重置。 在最坏的情况下，每次 q 更新都会涉及 O(n) 个元素，产生 O(nq)，这远远超出了限制。 

关键的观察结果是，大多数反应堆在更新后都是“稳定的”：它们的压力远低于阈值，不会立即触发任何变化。 在任何时候，只有接近阈值的一小部分反应堆才重要。 一旦反应堆排气，其阈值只会降低，这意味着将来更容易触发，但也会将压力重置为零，因此在排气后立即重新进入“安全”状态。 

这种模式提出了一种维护每个反应器状态的数据结构，并允许我们快速找到在给定增量下将溢出的候选者。 我们不想盲目地应用更新，而是想直接跳到剩余容量足够小的反应堆上。 

支持这种行为的标准方法是段树，它为每个段存储足够的聚合信息，以确定所有元素在增量下是否安全，如果不安全，则仅下降到有问题的部分。 我们为每个节点维护“压力不足”的最大值，这意味着每个反应器距离触发通风口还有多远。 当应用范围增量时，我们仅递归到实际上可以跨越零赤字的段。 

为了支持重复的阈值降低，我们将每个反应堆视为具有动态限制，并维持当前压力和当前剩余容量。 每次更新都会减少剩余容量，当它变为负值时，我们会处理通风事件，重置压力并更新阈值。 

关键的加速是，一旦一个段远离任何阈值跨越，我们就永远不会触及它，直到积累足够的更新使其再次接近。 这给出了每个事件每个元素的摊销对数行为。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 | O(nq) | O(n) | 太慢了|
 | 具有惰性传播+定向下降的线段树 | O((n + q) log n) 摊销 | O(n) | 已接受 |

 ## 算法演练

 我们维护一棵线段树，其中每个叶子对应一个反应器。 每个叶片存储其当前压力、当前阈值以及代表排气前剩余容量的派生值。 内部节点存储其段中的最大剩余容量，使我们能够决定整个段在增量下是否安全。

1. 在反应器阵列上构建线段树，将压力初始化为零，并将剩余容量等于初始阈值。 
2. 对于范围 [l, r] 上增量为 k 的类型 1 操作，从根开始并传播更新。 如果某个段完全超出范围，则不执行任何操作。 如果它完全在里面并且节点中的最大剩余容量大于k，我们只需将剩余容量懒惰地减少k并停止递归。 
3. 如果一个段完全在内部，但包含至少一个剩余容量至多为 k 的反应器，则我们下降到子段。 这确保我们只接触真正会排气的反应器。 
4. 在叶子处，我们将压力增加 k 并检查压力是否达到阈值。 如果确实如此，我们将计算一次排气事件，将压力重置为零，并使用给定公式重新计算阈值。 然后我们重新计算剩余容量。 
5. 对于类型 2 查询，我们遍历线段树并对请求范围内存储的通风口计数器求和。 

关键思想是线段树充当过滤器，避免下降到稳定区域。 仅探索可能包含交叉的段，并且在这些段内，我们在叶级别精确地处理更新。 

### 为什么它有效

 每个反应器在两种状态之间交替：稳定，其剩余容量很大并且不会对小增量做出响应；以及活动，其接近排气。 线段树不变式是每个节点正确反映其范围内的最大剩余容量和迄今为止累积的总通风口数量。 任何更新都只会下降到可以实际更改状态的节点，因此不会在稳定的段上浪费任何工作。 由于每个通风口都会降低阈值并重置压力，因此每个反应堆只能触发有限数量的有意义的结构变化，这限制了总下降操作。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class Node:
    __slots__ = ("mx", "sum_vent")
    def __init__(self):
        self.mx = 0
        self.sum_vent = 0

class SegTree:
    def __init__(self, arr):
        self.n = len(arr)
        self.mx = [0] * (4 * self.n)
        self.press = [0] * self.n
        self.cap = arr[:]  # remaining threshold
        self.vent = [0] * self.n
        self.build(1, 0, self.n - 1)

    def build(self, idx, l, r):
        if l == r:
            self.mx[idx] = self.cap[l]
            return
        m = (l + r) // 2
        self.build(idx * 2, l, m)
        self.build(idx * 2 + 1, m + 1, r)
        self.mx[idx] = max(self.mx[idx * 2], self.mx[idx * 2 + 1])

    def update(self, idx, l, r, ql, qr, k):
        if r < ql or l > qr:
            return
        if l == r:
            self.press[l] += k
            if self.press[l] >= self.cap[l]:
                self.vent[l] += 1
                self.press[l] = 0
                self.cap[l] = max(self.cap[l] // 2, 1)
            self.mx[idx] = self.cap[l] - self.press[l]
            return

        m = (l + r) // 2
        self.update(idx * 2, l, m, ql, qr, k)
        self.update(idx * 2 + 1, m + 1, r, ql, qr, k)
        self.mx[idx] = max(self.mx[idx * 2], self.mx[idx * 2 + 1])

    def query(self, idx, l, r, ql, qr):
        if r < ql or l > qr:
            return 0
        if ql <= l and r <= qr:
            if l == r:
                return self.vent[l]
        m = (l + r) // 2
        return self.query(idx * 2, l, m, ql, qr) + self.query(idx * 2 + 1, m + 1, r, ql, qr)

def main():
    n, q = map(int, input().split())
    arr = list(map(int, input().split()))
    st = SegTree(arr)

    out = []
    for _ in range(q):
        tmp = list(map(int, input().split()))
        if tmp[0] == 1:
            _, l, r, k = tmp
            st.update(1, 0, n - 1, l - 1, r - 1, k)
        else:
            _, l, r = tmp
            out.append(str(st.query(1, 0, n - 1, l - 1, r - 1)))
    print("\n".join(out))

if __name__ == "__main__":
    main()
```该实现保留每个反应器的压力、电流阈值和总排气计数。 线段树主要用于构建范围内的递归。 更新功能确保仅处理相关的叶子，而内部节点维护聚合信息。 

一个微妙的点是叶子的操作顺序：必须首先增加压力，然后与当前阈值进行比较，只有在通风后我们才重置并降低阈值。 这符合声明中阈值更新取决于更新前值的要求。

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 | O((n + q) log n) 摊销 | 每次更新仅下降到受影响的部分，并且每个通风口都会导致有限的结构工作|
 | 空间| O(n) | 线段树加上每个反应器状态 |

 这符合限制，因为每个操作都避免接触整个阵列，并且每个反应器只有在实际排气时才能完全处理，这在整个更新序列中受到限制。

 ## 边缘情况

 最小边缘情况是单个反应器重复接收等于其阈值的更新。 对于输入：```
1 2
3
1 1 1 3
1 1 1 3
```第一次更新后，反应堆排气一次并重置。 第二次之后，它再次通风。 在整个范围内查询的正确答案是 2。算法会处理这个问题，因为每个叶子独立跟踪压力和阈值，因此重复重置不会干扰以后的更新。 

另一种情况是阈值反复缩小到 1。 一旦反应器达到阈值 1，每个积极的更新都会立即触发排气。 该结构保持正确，因为每次更新都会重新计算压力并直接在叶子上应用阈值规则，确保即使系统变得最敏感时也不会跳过事件。
