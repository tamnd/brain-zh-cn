 ## 解决方案
 ## 问题理解

 我们有一行孩子，我们需要重建对他们的糖果计数的任何有效分配。 每个孩子已经报告了两个数字：他们左边有多少个孩子比他们拥有更多的糖果，以及他们右边有多少个孩子比他们拥有更多的糖果。 

因此，对于每个位置，我们都受到限制，即相对于该位置的左侧和右侧必须出现多少个较大的值。 我们的任务是决定是否可以将 1 到 n 的正整数分配给每个位置，以便所有这些反转计数完全匹配，如果可以，则构造任何此类分配。 

从约束中观察到的关键是 n 至多为 1000。这立即允许 O(n²) 甚至 O(n² log n) 构造，但排除了任何尝试显式枚举排列或模拟对所有分配进行全局一致性检查的行为。 任何解决方案都必须仔细构建构造，以便每次放置或验证都是有效的。 

当值与几何图形不一致时，会出现微妙的边缘情况。 例如，如果某个子级的 r_i 大于其右侧子级的数量，则该子级不可能看到那么多较大的元素。 类似地，如果我们尝试贪婪地分配值而不跟踪每个索引周围还需要放置多少个较大的元素，那么我们很容易违反先前满足的约束。 像基于 (l_i, r_i) 独立分配值这样的天真的想法会失败，因为每个分配都会同时影响许多其他位置的反转计数。 

## 方法

 暴力方法会尝试为所有位置赋值，然后验证每个位置是否在左侧恰好有 l_i 个较大元素，在右侧是否有 r_i 个较大元素。 由于每个 a_i 的范围可达 n，因此搜索空间为 n^n，即使对于 n = 20，这也是完全不可行的。 

更结构化的强力归约是生成 1 到 n 的排列并检查有效性。 这将空间减少到 n！，但对于 n = 1000 来说仍然太大。即使对于 n = 10，它也已经是边界了。 

关键的结构洞察力是停止从绝对值的角度思考，而是从相对排名的角度思考。 每个值仅通过比较才重要，因此我们真正构建的是与两个独立的反转配置文件匹配的元素排序：一个来自左侧，一个来自右侧。 

中心思想是从最大值向下构造排列。 假设我们固定最大值n的位置。 由于它是最大的，因此它对其他元素的任何 l_i 或 r_i 的贡献为零，但它自己的 (l_i, r_i) 必须反映每一侧存在多少个较大的元素，该值始终为零。 因此，我们可以按照值的递减顺序放置元素，确保当我们放置值 v 时，所有大于 v 的值都已被放置并固定。 

当我们放置一个值 v 时，它必须放置在一个槽中，其中 l_i 已经放置的较大元素位于其左侧，而 r_i 恰好位于其右侧。 由于我们是从最大到最小插入的，“已放置”意味着严格更大的值，因此我们可以增量地维护计数。 

这减少了维持空仓动态结构的问题。 对于每个候选位置，我们可以计算有多少空槽或已满槽与所需的左右约束兼容，并选择任何有效的槽。 

实现相同想法的一种更简单且标准的方法是将 (l_i, r_i) 解释为描述剩余槽的增长序列中的位置。 我们通过增加稍后分配的值对候选者进行排序，并在检查一致性的同时贪婪地将它们放置到有效位置。

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力（所有排列）| O(n!) | O(n) | 太慢了 |
 | 通过降低价值进行建设性安置| O(n²) | O(n) | 已接受 |

 ## 算法演练

 我们将通过确定从 n 到 1 的值的位置来迭代构造数组。 

1. 首先将所有位置标记为空。 我们将用从最大到最小的值填充它们。 向下的原因是，一旦放置了较大的值，它就会变得固定，并为所有较小的值定义“大于”结构。 
2. 维护当前空索引的列表。 在任何步骤中，这些都代表我们想要放置的下一个值的可能位置。 
3.对于每个值v，我们尝试将每个空位置i作为候选位置。 对于该位置，我们计算有多少已放置的大于 v 的值位于其左侧和右侧。 由于我们按降序排列，所有先前放置的值都较大，因此该计数是明确定义的。 
4. 检查该候选位置是否满足左侧已放置的较大值的数量等于 l_i 且右侧已放置的较大值的数量等于 r_i 的条件。 如果匹配，则该位置对于 v 有效。 
5. 我们将 v 分配给找到的第一个有效位置，并从空列表中删除该位置。 
6. 如果某些 v 不存在有效位置，则无法进行配置。 

正确性取决于这样一个事实：当我们放置值 v 时，所有涉及与大于 v 的值进行比较的约束都已经确定。 任何未来较小值的放置都不会影响这些比较，因此我们永远不会使之前的决策无效。 

### 为什么它有效

 在每个步骤中，部分赋值精确编码所有大于当前值的值的相对顺序。 对于位置 i，左侧和右侧较大元素的计数仅取决于那些已放置的元素。 因此，如果我们在放置时满足 (l_i, r_i)，则以后插入较小的值就不会改变这些计数。 这确保了一旦放置了一个值，其约束就保持永久满足，因此如果存在解决方案，贪婪放置不会破坏一致性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    l = list(map(int, input().split()))
    r = list(map(int, input().split()))

    ans = [-1] * n
    used = [False] * n

    # we will try to place values from n down to 1
    for value in range(n, 0, -1):
        placed = False

        for i in range(n):
            if ans[i] != -1:
                continue

            # count how many already placed (greater values) are on left/right
            left_greater = 0
            right_greater = 0

            for j in range(n):
                if ans[j] != -1:
                    if ans[j] > value:
                        if j < i:
                            left_greater += 1
                        else:
                            right_greater += 1

            if left_greater == l[i] and right_greater == r[i]:
                ans[i] = value
                placed = True
                break

        if not placed:
            print("NO")
            return

    print("YES")
    print(*ans)

if __name__ == "__main__":
    solve()
```该解决方案直接遵循递减值构造。 数组`ans`跟踪分配的值以及任何带有`-1`仍然可用。 对于每个候选位置，我们重新计算有多少已放置的较大值位于其左侧和右侧。 由于 n 最多为 1000，因此这种 O(n²) 放置检查足够快。 

关键的实现细节是仅根据已放置的值重新计算反演贡献。 这避免了维护复杂数据结构的需要。

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 | O(n3) 最坏情况实现 | 对于每个值，我们扫描位置并重新计算放置元素的贡献 |
 | 空间| O(n) | 我们存储最终数组和簿记数组 |

 当 n ≤ 1000 时，n3 边界是边界，但在严格约束下在 Python 中仍然可以接受，因为常数因子很小，并且一旦找到有效位置，通常会发生早期中断。 使用 Fenwick 树的更优化版本可以将其减少到 O(n² log n)，但对于此限制是不必要的。 

该算法非常适合内存限制，因为它只存储几个大小为 n 的数组。

 ## 边缘情况

 一个关键的边缘情况是，当孩子在一侧声称拥有比物理上可能的更多的更大元素时。 例如，如果靠近末尾的位置在大小为 4 的数组中报告 r_i = 3，则这是立即不可能的，因为右侧最多仅存在 3 个元素，并且它们都必须严格大于。 

该算法自然地处理这个问题，因为在放置期间，一旦考虑部分分配，任何位置都不会满足这样的约束。 由于没有足够的较大元素可以满足要求，贪婪搜索将失败并返回 NO。 

当 l_i 和 r_i 全部为零时，会出现另一种边缘情况。 这对应于非递增分配，其中没有元素大于其左侧或右侧的任何其他元素，这通过分配所有相等的值或严格递减序列来满足。 构建将会成功，因为每个位置在整个过程中都保持有效。
