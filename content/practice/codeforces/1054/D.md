 ## 解决方案
 ## 问题理解

 我们得到一个整数数组，其中每个值都精确地表示为$k$位。 除了数组之外，我们还可以进行非常具体的转换：对于任何位置，我们可以翻转数字的所有位，将其转换为其按位补码$k$位空间。 

目标不是直接优化数组值，而是最大化数组上定义的组合量：异或非零的子数组的数量。 如果子数组所有元素的 XOR 不等于 0，则该子数组被视为“良好”。 

因此，该任务变成了一个全局优化问题：通过选择性地补充某些元素，我们希望塑造前缀异或行为，以便尽可能多的子数组异或非零。 

关键的困难在于子数组异或结构取决于前缀异或相等性。 当两个前缀 XOR 值匹配时，子数组的 XOR 值为零。 因此，最大化好的子数组相当于最小化重复的前缀异或冲突。 

约束很大，最多有 200,000 个元素和 30 位整数。 任何试图模拟所有操作子集或直接计算子数组的解决方案都是不可行的。 甚至$O(n^2)$太慢了，因为它意味着周围$4 \times 10^{10}$最坏情况下的操作。 

该操作本身引入了一个微妙的结构：翻转一个值的所有位相当于使用固定掩码进行异或$M = (2^k - 1)$。 这意味着每个元素恰好有两个状态，原始状态或与$M$，我们正在为每个位置选择一个状态。 

当认为本地决策就足够时，就会出现天真的错误。 例如，贪婪地选择翻转以使每个前缀 XOR 唯一可能会失败，因为翻转一个元素会更改所有后续前缀 XOR 值。 

另一个常见的陷阱是尝试直接优化子数组计数，而不将问题转换为前缀异或冲突。 这种方法往往会重复计算或错过分段之间的交互。 

## 方法

 强力解释很简单：对于索引的每个子集，决定是否翻转每个元素，计算结果数组，然后计算有多少个子数组具有非零异或。 这在概念上是有效的，因为它直接评估目标。 然而，它需要$2^n$配置以及每次评估成本$O(n^2)$或者$O(n)$带有前缀XOR，使其完全不可行。 

关键的结构性转变来自于重写目标。 一个子数组$[l, r]$当且仅当前缀 XOR 为 XOR 时，XOR 为零$l-1$等于前缀 XOR at$r$。 因此，最大化好的子数组相当于最小化前缀 XOR 值之间的相等对。 

这将问题转化为使用每个元素的本地二进制选择来控制前缀 XOR 的多重集。 补码操作与 XOR 干净地交互：翻转元素通过固定掩码切换其贡献，因此前缀 XOR 值以可预测的线性方式移动。 

关键的观察是，我们可以在掩码变换下推理每个前缀异或的“值类”，而不是推理单个元素。 每个前缀状态都可以与其补充状态配对，并且转换会影响我们是否留在同一类或切换。 

这将问题简化为前缀 XOR 状态上的贪婪构造，其中每个位置贡献两个可能的下一个状态，并且我们选择能够最小化前缀 XOR 序列中的冲突的状态。 可以通过跟踪每个 XOR 状态或其补体出现的频率并确保我们使序列偏向于前缀 XOR 值的多样性来导出最佳策略。 

实际上，我们在图中构造一条路径，其中每个节点都有两个可能的传出转换，并且我们选择转换来最大化不同的前缀 XOR。

 | --- | --- | --- | --- |
 | 蛮力 |$O(2^n \cdot n)$|$O(n)$| 太慢了 |
 | 最佳 |$O(n)$|$O(2^k)$| 已接受 |

 ## 算法演练

 我们定义$M = 2^k - 1$，因此对一个值求补相当于与$M$。 

我们在决定是否翻转每个元素时维护运行前缀 XOR。 

1. 在扫描数组时计算前缀 XOR，但允许每个元素贡献其中之一$a_i$或者$a_i \oplus M$。 

位置处的前缀异或$i$成为我们想要控制的状态。 
2. 维护一个哈希图或数组，计算每个前缀 XOR 值出现的次数。 
3. 各位置$i$，我们考虑两种可能的下一个前缀异或值：一种无需翻转获得，一种通过翻转获得。 
4. 我们选择迄今为止较少“使用”的产生前缀 XOR 状态的选项，即在前缀 XOR 历史中出现次数较少的选项。 
5. 更新所选前缀 XOR 的频率并继续。 
6. 处理完所有元素后，使用标准恒等式计算好子数组的数量：

 总子数组减去相等前缀 XOR 值对。 

最后一步之所以有效，是因为每一对相等的前缀 XOR 值都定义了一个零 XOR 子数组。 

### 为什么它有效

 整个过程依赖于控制前缀异或值之间的冲突。 每个决策都在两个对称状态之间进行局部选择：当前状态和由固定掩码翻转的状态。 由于这些状态在状态空间中形成不相交的对，因此贪婪选择总是将构造推向在不同值之间传播前缀 XOR 出现的方向。 

关键的不变性是，在每一步中，我们在给定到目前为止处理的前缀的情况下维持前缀异或状态的最佳可能分布，并且每个选择仅通过更新的频率结构影响未来的冲突。 由于成本函数纯粹是在相等前缀对上相加，因此重复状态的局部最小化会产生全局最优配置。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, k = map(int, input().split())
    a = list(map(int, input().split()))
    
    mask = (1 << k) - 1
    
    # prefix xor frequencies
    freq = {}
    
    px = 0
    freq[0] = 1
    
    for x in a:
        # option 1: no flip
        v1 = px ^ x
        
        # option 2: flip
        v2 = px ^ (x ^ mask)
        
        # choose less frequent prefix state
        if freq.get(v1, 0) <= freq.get(v2, 0):
            px = v1
        else:
            px = v2
        
        freq[px] = freq.get(px, 0) + 1
    
    # count pairs of equal prefix XORs
    total = 0
    for c in freq.values():
        total += c * (c - 1) // 2
    
    # total subarrays = n*(n+1)/2
    total_subarrays = n * (n + 1) // 2
    
    print(total_subarrays - total)

if __name__ == "__main__":
    solve()
```该代码维护运行前缀 XOR，同时在原始贡献和补充贡献之间进行贪婪选择。 掩码对全位翻转操作进行编码，因此两个选项始终都是有效状态。 

频率字典跟踪每个前缀 XOR 出现的频率。 由于等前缀 XOR 对完全对应于零 XOR 子数组，因此从总子数组中减去此类对的数量即可得出最终答案。 

一个微妙的点是我们在构造过程中没有显式跟踪子数组。 相反，我们完全依赖于前缀 XOR 多重性，它将所有子数组结构压缩为单个频率计算步骤。

 ## 边缘情况

 关键的边缘情况是位掩码下的所有元素都相同或高度对称。 在这种情况下，幼稚的贪婪选择可能会在相同的前缀 XOR 值之间振荡，从而产生重复的冲突。 正确的处理取决于确保每个决策通过前缀异或频率跟踪一致地考虑当前和翻转的贡献，从而防止状态的病态聚类。
