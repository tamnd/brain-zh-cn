 ## 解决方案
 ## 问题理解

 我们正在模拟物体从位置在一维线段上移动`0`到固定目的地`k`。 每个模块从这条线上的某个位置开始，以每秒一个单位的恒定速度向右移动。 每个病毒也从同一条线上的某个位置开始，但以相同的速度向左移动。 

由于两者以相同的速度向相反的方向移动，任何模块和病毒要么永远不会相遇，要么在完全由其初始位置决定的确定时间恰好相遇一次。 当它们相遇时，它们可能会进行交互，具体取决于我们是否已获得病毒模块兼容性条目。 如果没有这样的条目，它们只是相互穿过。 如果有条目，则双方都具有能力值的情况下会发生战斗，并且可能会移除一个或两个。 幸存的模块也可能失去容量，并且幸存的病毒继续向左移动，并且稍后仍然可以遇到其他模块。 

最终目标不是明确地模拟随时间变化的位置，而是确定哪些模块在所有交互中都幸存下来，然后将它们按到达时间排序输出到服务器。 由于每个模块都以一速度向位置移动`k`，到达时间仅取决于其起始位置。 

这些约束足够严格，以至于不可能对所有可能的碰撞进行简单的成对模拟。 模块数量多达1000个，病毒数量多达5000个，交互规则多达50万条。 每秒检查所有模块病毒对或模拟的解决方案显然会超出限制。 相反，我们必须依赖这样一个事实：相互作用是稀疏的并且完全由初始位置决定。 

当多个交互同时发生时，会出现一个微妙的问题。 由于模块可以通过早期的战斗进行修改，因此处理同时碰撞的顺序可能会影响结果。 任何正确的解决方案都必须尊重对同时性的一致解释。 

一种边缘情况是模块和病毒在同一位置启动。 在这种情况下，它们会在零时间立即发生碰撞，并且必须像任何其他碰撞一样进行处理。 

另一个边缘情况是交互列表中存在病毒模块对，但它们的位置是这样的：`(virus_position - module_position)`很奇怪。 然后它们在半整数时间相遇，并且永远不会在整数秒精确碰撞。 这些对必须被忽略。 

最后，一个模块可能会涉及多次交互，每次生存后其容量都会发生变化。 假设模块容量固定的粗心实现将产生不正确的结果。 

## 方法

 强力解释将尝试每秒模拟系统，更新所有位置并检查所有可能的重叠。 这在概念上很简单，但不可行：位置范围高达 10^9，因此即使是单个模拟也需要数十亿个步骤，并且每个步骤都需要检查所有模块是否有病毒。 

更结构化的蛮力通过计算所有可能的成对碰撞时间来改善这一点。 对于每个可能相遇的模块-病毒对，我们计算相遇时间并按时间顺序模拟事件。 这已经减少了事件处理的问题，但如果我们考虑所有的话，风险仍然很大`n * m`对，最多可达 500 万对。 

关键的观察是我们不需要考虑所有对，只需考虑输入查询中明确给出的对。 该问题提供了一组稀疏的交互作用，并且只有这些交互作用对于确定生存和容量变化才重要。 其他所有的遭遇都是无关紧要的，因为它没有任何影响。 

一旦我们将自己限制在`q`指定的交互作用，系统就成为一个经典的事件模拟问题。 每个事件都有一个由初始位置确定的时间戳。 我们按时间对事件进行排序并按顺序处理它们，更新模块和病毒状态。 

剩下的挑战是正确解决交互问题并处理模块容量丢失或被删除时的状态变化。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力对模拟 | O(n·m·T) | O(n·m·T) | O(1) | O(1) | 太慢了 |
 | 所有对的事件模拟 | O(n·m log (n·m)) | O(n·m log (n·m)) | O(n·m) | 太慢了 |
 | 稀疏事件模拟 | O(q log q) | O(q log q) | O(n + m + q) | O(n + m + q) | 已接受 |

 ## 算法演练

 ### 关键思想

 我们将每个有效的病毒模块交互视为特定时间的预定事件，然后仅模拟这些事件。 

### 步骤

 1. 读取所有模块数据并按名称存储每个模块及其初始容量和位置。 

我们还存储当前的可变容量和活/死标志，因为战斗会修改两者。 
2. 读取所有病毒数据并按其 ID 和位置以及活/死标志存储每个病毒。 
3. 阅读`q`交互规则。 对于病毒和模块之间的每条规则，计算它们是否确实满足。 

模块位于位置`x`和病毒在位置`y`走向彼此只有当相遇`y > x`和`(y - x)`是均匀的。 会议时间是`(y - x) // 2`。 
4. 为每个有效会议创建一个事件`(time, virus_id, module_id, virus_capacity_against_module)`并将其存储在列表中。 
5. 按时间对所有事件进行排序。 如果多个事件共享同一时间，则它们会一起处理。 
6. 按时间递增顺序迭代事件：

 - 如果模块或病毒已经死亡，则跳过该事件。 
- 如果病毒没有定义容量（由于过滤而不会发生），则跳过。 
- 比较模块容量`C`和病毒容量`X`。 

- 如果`C > X`，病毒死亡，模块容量变为`C - X`。 
- 如果`C < X`，模块死亡。 
- 如果相等，双方都会死。 
7. 处理完所有事件后，收集所有仍处于活动状态的模块。 
8. 按到达时间对剩余模块进行排序。 由于所有模块都以速度 1 移动到位置`k`，到达时间为`k - position`，因此通过增加到达时间排序相当于通过减少初始位置排序。 

### 为什么它有效

 每个可以影响系统的交互都被编码为确定性事件。 由于运动是均匀的，因此没有事件依赖于中间位置随机性。 唯一不断变化的状态是模块容量和活动状态，并且两者仅在事件时间更新。 因为每个事件都完全捕获一种可能的交互，所以跳过任何事件将对应于删除真实的物理遭遇，而处理任何额外的事件将对应于发明从未发生的碰撞。 

正确性取决于连续事件之间的不变量，任何一对事件都不会发生任何可能影响另一个事件结果的变化。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m, q = map(int, input().split())

    modules = {}
    module_pos = {}
    module_alive = {}
    module_cap = {}

    for _ in range(n):
        name, cap, pos = input().split()
        cap = int(cap)
        pos = int(pos)
        modules[name] = name
        module_pos[name] = pos
        module_alive[name] = True
        module_cap[name] = cap

    virus_pos = {}
    virus_alive = {}

    for _ in range(m):
        vid, pos = map(int, input().split())
        virus_pos[vid] = pos
        virus_alive[vid] = True

    events = []

    for _ in range(q):
        vid, mod, cap = input().split()
        vid = int(vid)
        cap = int(cap)

        if mod not in module_pos or vid not in virus_pos:
            continue

        x = module_pos[mod]
        y = virus_pos[vid]

        if y <= x:
            continue

        diff = y - x
        if diff % 2 != 0:
            continue

        t = diff // 2
        events.append((t, vid, mod, cap))

    events.sort()

    for t, vid, mod, cap in events:
        if not module_alive[mod] or not virus_alive[vid]:
            continue

        C = module_cap[mod]
        X = cap

        if C > X:
            module_cap[mod] -= X
            virus_alive[vid] = False
        elif C < X:
            module_alive[mod] = False
        else:
            module_alive[mod] = False
            virus_alive[vid] = False

    remaining = [name for name in module_pos if module_alive[name]]

    remaining.sort(key=lambda x: -module_pos[x])

    print(len(remaining))
    print(" ".join(remaining))

if __name__ == "__main__":
    solve()
```模块和病毒字典保留静态输入属性（例如位置），而单独的数组则跟踪动态状态。 这种分离很重要，因为职位永远不会改变，但能力和生存状态会改变。 

事件列表是核心结构：每个条目编码一个可能的冲突。 对它进行排序可以确保我们以正确的时间顺序处理交互。 

最终的排序步骤使用初始位置而不是直接模拟到达时间，因为速度是均匀的并且到达顺序在时间零时完全确定。 

## 工作示例

 ### 示例 1

 输入：```
modules:
A(0, 5), B(3, 2)
viruses:
V1(6), V2(7)
events:
(V1, A, 4), (V2, B, 2)
```活动时间：

 | 时间 | 活动 | 模块帽 | 病毒帽 | 结果|
 | --- | --- | --- | --- | --- |
 | 3 | V1-A | 5 | 4 | A 幸存（cap 1），V1 死亡 |
 | 2 | V2-B | 3 | 2 | B 幸存（第 1 章），V2 死亡 |

 按时间顺序处理：

 | 步骤| 活跃模块 | 行动|
 | --- | --- | --- |
 | t = 2 | 甲、乙| B 幸存，上限减少 |
 | t = 3 | 甲、乙| A幸存，上限减少|

 两个模块都保留下来，最终顺序取决于位置：A 第一。 

这表明按时间排序的事件可确保正确的级联更新。 

### 示例 2

 考虑同时发生的事件：

 | 时间 | 活动 | 模块帽 | 病毒帽 |
 | --- | --- | --- | --- |
 | 2 | V1-A | 3 | 3 |
 | 2 | V2-A | 2 | 1 |

 如果首先处理 V1-A，A 会立即死亡并跳过 V2-A。 如果相反，A 会在第二个事件中死亡。 不管怎样，A 都会被移除，但病毒的存活率会有所不同。 这说明了为什么平局打破只能在本地起作用，但最终的模块集在一致的处理下保持一致。

 ## 边缘情况

 一个关键的边缘情况是多个病毒在同一时间戳与同一模块发生冲突。 该算法按顺序处理它们，但仍然保留正确性，因为每个事件都会立即更新模块状态，同时匹配有序微交互的一致物理解释。 

另一个重要的情况是当病毒遇到未定义容量的模块时。 这些事件会被尽早过滤，确保它们根本不会进入模拟，从而避免不必要的状态检查。 

最后，从未出现在任何事件中的模块保持不变，并且仅包含在基于位置的最终排序列表中，从而确认事件系统仅影响相关对象。
