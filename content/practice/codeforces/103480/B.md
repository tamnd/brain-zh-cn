 ## 问题理解

 我们得到了几个独立的测试用例。 在每个测试用例中，我们收到一个正整数数组，任务是计算有多少个子数组的总和恰好等于 7777。子数组是通过选择两个索引来定义的$l$和$r$和$l \le r$，并对它们之间的所有元素求和。 我们不会被要求找到最大值或最小值，只需要计算有多少个不同的连续段产生目标总和。 

这些限制以非常直接的方式产生影响。 数组长度最大可达$10^5$，并且最多可以有 10 个测试用例。 所有子数组的简单二次枚举将涉及大约$10^{10}$最坏情况下的运行，远远超出一秒所能运行的范围。 这立即排除了任何从头开始重新计算每对端点总和的方法。 

有一个结构约束变得很重要：每个元素都是正数，且最多 5000 个。这保证了前缀和严格递增，并且滑动窗口行为得到明确定义，而无需担心负校正。 

幼稚实现的一个微妙的失败案例是重复重新计算子数组和。 例如，如果我们固定左端点并扩展右指针，同时重复求和，我们最终仍然会这样做$O(n^2)$补充。 另一个陷阱是忽略多个子数组可能严重重叠，并且如果在没有正确前缀对齐的情况下对总和使用散列，则会错误地对它们进行计数。 

## 方法

 蛮力方法很简单：枚举每一对$(l, r)$，计算子数组的和，并检查它是否等于 7777。这是正确的，因为它直接遵循问题的定义。 然而，从头开始计算每个总和的成本$O(n)$，并且有$O(n^2)$对，导致$O(n^3)$时间。 即使我们使用前缀和数组优化每个子数组和，将每个查询减少到$O(1)$，我们还有$O(n^2)$每个测试用例的子数组总数对于$n = 10^5$。 

关键的观察来自于所有元素的积极性。 由于所有数字都是正数，因此当我们向右移动时，前缀和严格增加。 这使我们能够维护一个滑动窗口：对于每个右端点，当总和超过目标时，我们可以向前移动左端点。 不变的是当前窗口总和始终被有效维护，并且没有有效子数组被跳过或重复计算。 

这将问题转化为对数组的两指针扫描，其中每个指针最多仅向前移动$n$次，产生一个$O(n)$每个测试用例的解决方案。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力 |$O(n^2)$或者$O(n^3)$|$O(1)$| 太慢了|
 | 最优两指针 |$O(n)$|$O(1)$| 已接受 |

 ## 算法演练

 我们维护一个滑动窗口$[l, r]$及其当前总和。 

1.初始化两个指针$l = 0$,$r = 0$，并且运行总和等于 0。这表示在处理任何元素之前有一个空窗口。 
2. 从左向右逐步移动右指针。 在每一步中，将当前元素添加到运行总和中。 这扩大了窗口并确保我们考虑以以下结尾的每个可能的子数组$r$。 
3. 当当前总和超过 7777 时，向前移动左指针并减去离开窗口的元素。 这是有效的，因为所有元素都是正数，所以从左边缩小只能减少总和，而不能增加总和。 
4. 调整窗口后，如果总和等于 7777，则增加答案。 这会计算以当前结尾的唯一子数组$r$其总和与目标匹配。 
5. 继续，直到右指针到达数组末尾。 

正确性取决于以下事实：在每个位置$r$，算法找到所有有效的左边界$l$这样子数组的和正好是 7777。因为该数组只包含正数，所以每个子数组最多有一个有效的调整序列$r$，并且不会跳过任何有效的子数组。 

### 为什么它有效

 关键的不变量是在每一步中，窗口$[l, r]$是当前右端点的最小可能左边界，使得总和不超过 7777。因为元素为正，所以移动$l$向前单调减少总和，因此每个指针最多移动$n$次。 每个有效子数组结束于$r$必须恰好在窗口总和从过大的总和缩小后变为 7777 时出现，因此每个解都只计算一次。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    arr = list(map(int, input().split()))
    
    target = 7777
    l = 0
    s = 0
    ans = 0
    
    for r in range(n):
        s += arr[r]
        
        while l <= r and s > target:
            s -= arr[l]
            l += 1
        
        if s == target:
            ans += 1
    
    return ans

t = int(input())
out = []
for _ in range(t):
    out.append(str(solve()))
print("\n".join(out))
```代码直接镜像了滑动窗口的过程。 变量`s`跟踪当前窗口总和以及左指针`l`仅在必要时才移动以将总和保持在范围内。 条件`s == target`恢复可行性后才进行检查，确保正确性。 每个测试用例使用一次传递可以保持实现的高效和简单，避免任何前缀和字典或散列。 

## 工作示例

 考虑数组`[2000, 3000, 2777, 1000]`。 

我们跟踪滑动窗口：

 | r | 我| 窗口| 总和| 行动|
 | --- | --- | --- | --- | --- |
 | 0 | 0 | [2000] | 2000 | 2000 展开 |
 | 1 | 0 | [2000, 3000] | 5000 | 展开 |
 | 2 | 0 | [2000, 3000, 2777] | 7777 | 找到匹配项 |
 | 3 | 0 | [2000, 3000, 2777, 1000] | 8777 | 向左收缩|

 当我们到达$r = 2$，总和恰好匹配一次，因此我们计算一个子数组。 在$r = 3$，总和超出目标，需要缩小，但没有出现精确匹配。 

现在考虑`[7777, 1, 7776]`。 

| r | 我| 窗口| 总和| 行动|
 | --- | --- | --- | --- | --- |
 | 0 | 0 | [7777] | 7777 | 比赛|
 | 1 | 0 | [7777, 1] | 7778 | 收缩|
 | 1 | 1 | [1] | 1 | 没有匹配|
 | 2 | 1 | [1, 7776] | 7777 | 比赛|

 此示例表明，缩小窗口后可以出现有效的子数组，并且算法在每次调整后正确地重新评估条件。

 ## 边缘情况

 一个重要的边缘情况是单个元素等于 7777 时。对于输入`[7777]`，算法开始于`r = 0`，添加该元素，立即找到`s == 7777`，所以它只计算一个子数组。 

另一种情况是当所有元素都很小时并且没有组合达到 7777。例如`[1, 2, 3]`导致窗口总和永远不会超过目标，并且由于它永远不会等于 7777，因此答案仍然为零。 

更微妙的情况是总和短暂超过目标并且必须缩小才能再次找到有效窗口。 在`[7777, 1, 7776]`，窗口首先在索引 0 处匹配，然后扩展到目标之外，然后缩小，然后找到另一个有效的子数组。 滑动窗口正确地处理了这个问题，因为它总是在再次检查相等性之前恢复可行性。
