 ## 解决方案
 ## 问题理解

 我们被要求确定一系列绘画比赛的生产规模。 每场比赛都有已知数量的参与者，并且每个参与者只消耗一个画板。 除此之外，每场比赛还必须额外获得相当于该比赛参赛者 10% 的画板作为安全缓冲区。 

输入给出了几个独立的测试用例。 对于每个测试用例，我们都会得到比赛的数量，然后是每次比赛的参与者人数。 任务是计算所有比赛所需的画板总数，其中每个比赛独立贡献自己的需求。 

一个关键的细节是，“10% 的额外费用”不会在不同的比赛中共享。 它适用于每场比赛，因此舍入行为是根据值而不是全局进行的。 

约束足够小，即使每个测试用例进行线性扫描也很容易就足够了。 每个测试用例最多有 1000 场比赛和 100 个测试用例，每个测试用例的直接 O(N) 解决方案可以在限制内轻松运行。 

主要的微妙之处在于当结果不是整数时如何解释 10% extra。 由于画板是离散的，因此不能存在小数，因此多余的部分必须四舍五入。 这就是许多简单解决方案失败的地方：使用整数除法或浮点截断会产生不正确的总计。 

当参与者数量较少时，就会发生典型的失败场景。 例如，如果比赛有1人参加，10%就是0.1，四舍五入后必须成为额外的1个画板。 像这样的简单计算`a // 10`会错误地给出 0。 

## 方法

 蛮力的想法是独立计算每个竞赛，使用浮点运算计算其所需的画板，然后将所有内容相加。 这原则上是正确的，因为该要求是在不同的比赛中相加的。 但是，它存在精度问题和不正确舍入的风险，特别是在使用浮点乘法和转换为整数时。 

更强大的强力方法会明确计算每个值 10% 的上限，但即便如此，粗心的实现通常会引入相差一的错误。 

关键的观察是每个比赛都是独立的并且贡献准确`a_i + ceil(0.1 * a_i)`画板。 这将问题转化为简单的每个元素转换，然后进行求和。 

我们通过重写上限项来完全避免浮点。 表达式`ceil(a / 10)`可以计算为`(a + 9) // 10`。 这可以避免精度错误并保证所有整数输入的正确性。 

一旦认识到这一点，解决方案就变成了单次线性传递。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力（浮点+舍入）| O(N) | O(1) | O(1) | 有风险/可能会失败 |
 | 最优（整数上限算术）| O(N) | O(1) | O(1) | 已接受 |

 ## 算法演练

 ### 最优算法

 1. 对于每个测试用例，阅读比赛数量和参与者人数列表。 这建立了我们将处理的独立单元。 
2. 初始化累加器`total = 0`。 这将存储所需画板的最终数量。 
3. 每个竞赛值`a`，计算本次比赛贡献的所需画板。 基本要求是`a`，因为每个参与者都需要一个画板。 
4. 将备份要求计算为`ceil(a / 10)`，这确保至少 10% 的额外费用始终向上舍入。 我们将其计算为`(a + 9) // 10`。 +9 确保任何非 10 的倍数在整数除时正确舍入。 
5. 将两部分添加到总计中。 
6.处理完所有比赛后，以要求的格式输出累计总数。 

### 为什么它有效

 每场比赛都是独立的，因此总要求是每场比赛要求的总和。 唯一重要的部分是正确计算线性分数的上限。 身份`(a + 9) // 10`完全匹配`ceil(a / 10)`对于所有整数`a >= 1`。 由于每一项都是独立计算和求和，没有交互，因此比赛之间不存在交叉污染，保证了正确性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    T = int(input())
    for tc in range(1, T + 1):
        n = int(input())
        arr = list(map(int, input().split()))
        
        total = 0
        for a in arr:
            total += a + (a + 9) // 10
        
        print(f"Case #{tc}: {total}")

if __name__ == "__main__":
    solve()
```该解决方案独立读取每个测试用例并一次性处理该列表。 表达式`(a + 9) // 10`是关键的实施细节； 将其替换为`int(a * 0.1)`或者`a // 10`由于截断而不是上限，两者都会不正确。 

格式化步骤完全遵循所需的输出结构。

 ## 边缘情况

 最重要的边缘情况是当`a < 10`。 在那个政权下，`a / 10`小于 1，但实际上仍然非零，因此上限必须返回 1。 

例如：

 输入：```
1
1
7
```该算法计算：`7 + (7 + 9) // 10 = 7 + 1 = 8`。 

一个简单的实现使用`7 // 10`会产生`7`，完全缺少所需的额外画板。 这正是一种无声的低估错误，在大输入上显示正确，但在小约束重测试中失败。 

另一种边缘情况是 10 的精确倍数。对于`a = 20`,`(20 + 9) // 10 = 2`, 匹配`ceil(2.0)`确切地。 任何使用浮点舍入的尝试`int(a * 0.1 + 0.5)`由于浮动精度，尤其是对于较大的值，存在不正确行为的风险，因此整数公式避免了整个类别的错误。
