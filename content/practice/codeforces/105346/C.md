 ## 解决方案
 ## 问题理解

 我们得到一个代表一排灯笼的二进制字符串，其中每个位置要么点亮，要么不点亮。 在一次操作中，我们可以选择任何连续的段并翻转其中的每一位，将零变成一，一变成零。 目标是使用尽可能少的此类段翻转来达到整个字符串变得均匀（全零或全一）的配置。 

输入大小最多可达十万个字符。 这立即排除了任何尝试所有可能的片段或显式模拟翻转序列的方法。 任何比线性或近线性时间更糟糕的事情都会很困难，因为子串的数量本身就是二次的。 

一个天真的想法是尝试选择每个可能的最终目标状态，要么全零，要么全一，然后贪婪地或天真地模拟翻转段，直到字符串匹配。 另一种常见的错误尝试是独立翻转每个最大不匹配字符块，而不考虑合并机会，这可能会导致操作计数过高。 

例如，考虑`010`。 如果我们的目标是全零，则翻转中间部分给出`000`一举一动。 单独翻转每个不匹配字符的贪婪会错误地建议两个操作。 相反，在`01010`，粗心的局部决策很容易多算或少算，因为翻转一个片段会改变多个未来的边界。 

微妙的问题是，翻转不会孤立地在本地起作用，它会改变邻接结构，因此实际成本取决于连续字符之间的转换而不是单个字符本身。 

## 方法

 暴力方法会考虑每个可能的段翻转序列。 即使我们限制自己只翻转有助于实现目标的线段，选择线段的方法数量也是指数级的，因为每次翻转后，字符串都会发生变化并创建新的可能线段。 当 n 非常小时，这很快就变得不可行。 

更结构化的蛮力会尝试修复目标字符串（全零或全一），然后模拟贪婪修正。 如果我们从左向右扫描，每当遇到与目标不匹配的情况时，我们就会从该位置向前翻转，直到解决不匹配问题。 这对于固定目标是正确的，但仍然需要推理强制翻转多少次。 

关键的观察是，一旦我们确定了目标，最佳策略就完全由字符串中的转换决定。 假设我们的目标全为零。 每次我们看到转变`0 → 1`，这意味着我们正在进入一个最终必须翻转以修复这些问题的部分。 同样，每次转变`1 → 0`标记该段的结束。 每个连续的块必须恰好翻转一次，并且每次翻转都可以最佳地覆盖整个块。 

因此，将字符串转换为全零所需的翻转次数等于连续的 1 块的数量。 对称地，转换为全 1 所需的翻转次数等于连续的 0 块的数量。 答案就是这两个值中的最小值。 

问题从动态选择段减少到对二进制字符串中的运行进行计数。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 | 指数| O(n) | 太慢了|
 | 最佳 | O(n) | O(1) | O(1) | 已接受 |

 ## 算法演练

 1. 扫描一次字符串，统计有多少个连续的段`'1'`存在。 每当一个新的段开始`'1'`出现并且前一个字符不出现`'1'`。 如果我们的目标是把一切都归零，那么这就计算出必须处理多少个孤立的 1 组。 
2. 扫描一次字符串并统计有多少个连续的段`'0'`使用相同的逻辑存在。 如果我们的目标是把所有东西都变成一个，那么这代表了所需的翻转次数。 
3. 比较两个计数并输出较小的值。 这对应于选择两种可能的最终均匀状态中更便宜的一个。 

为什么这样有效：可以假设最佳解决方案中的每次翻转都覆盖了与目标不同的相同字符的整个最大连续段。 将一个段拆分为多个翻转永远不会有帮助，因为翻转较大的连续区域不会增加成本，但会减少碎片。 因此，每个最大块恰好贡献一个必要的操作，并且单独块之间的交互不能减少这一计数。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def count_blocks(s, ch):
    cnt = 0
    n = len(s)
    i = 0
    while i < n:
        if s[i] == ch:
            cnt += 1
            while i < n and s[i] == ch:
                i += 1
        else:
            i += 1
    return cnt

def solve():
    n = int(input().strip())
    s = input().strip()

    ones = count_blocks(s, '1')
    zeros = count_blocks(s, '0')

    print(min(ones, zeros))

if __name__ == "__main__":
    solve()
```功能`count_blocks`计算给定字符的连续运行次数。 它在字符串中前进，并且仅在遇到新块的开始时才增加计数器，立即跳过该块的其余部分。 这确保了线性遍历。 

我们计算一块和零块的数量，因为它们对应于两种可能的目标配置。 最终答案是最小值，因为我们可以自由选择我们想要实现的统一状态。 

一个常见的错误是试图明确地模拟翻转。 这是不必要的，因为在最佳解决方案中翻转永远不需要重叠，并且问题的结构保证了块边界完全决定成本。

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    input = sys.stdin.readline

    def count_blocks(s, ch):

        i = 0

                while i < n and s[i] == ch:

 ## 边缘情况

 像这样的字符串`11111`没有内部转换。 该算法将 1 个 1 块和 0 个 0 块计为 0。 最小值为零，与不需要翻转的事实相匹配。 

为了`010101`，每个字符交替，产生三个一块和三个零块。 该算法正确返回三个，反映出每个孤立的字符在任何目标选择下都需要自己的校正。 

对于像这样的图案`000111000`，中间段是唯一与选定的全零目标不同的段。 该算法计算单个单块，正确捕获一次翻转可以在一次操作中覆盖整个中间区域。
