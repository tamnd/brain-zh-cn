 ## 解决方案
 ## 问题理解

 我们得到一个仅由四个字符组成的循环字符串，每个字符映射到一个小整数权重。 该字符串排列成一个环，因此在最后一个字符之后我们回绕到第一个字符。 我们可以在这个圆上选择多个切割位置，将其分成几个连续的圆段。 

每个段都被视为普通字符串（不再是循环），并被分配一个多项式哈希，其中较早的字符贡献较小的 31 次方，而较晚的字符贡献较大的次方。 一个分区的总分是所有段的哈希值之和。 目标是选择削减以使总分最大化。 

弦是圆形的约束是关键的难点。 线段可以环绕末端，而分区本质上是选择不相交的弧，仅覆盖该圆一次。 

当 n 达到 200,000 时，尝试所有切割组合的任何解决方案都是不可能的，因为选择切割的方法数量是指数级的。 即使是区间内的二次 DP 也会太慢。 我们应该期望 O(n) 或 O(n log n) 解决方案，因为我们只能承担对字符串进行少量恒定次数的处理。 

一个微妙的问题来自于散列中的循环结构和指数权重之间的相互作用。 一个幼稚的错误是将字符串视为线性并在本地贪婪地切割，忽略环绕交互。 另一种常见的失败是正确计算每个段的哈希值，但忽略了更改一次切割会影响整个段的指数结构，而不仅仅是其边界。 

一个具体的失败案例是一个圆圈上有一个像“hehan”这样的字符串。 如果我们在看到价值下降时贪婪地削减而不考虑轮换，我们可能会选择一个糟糕的起点，使高价值角色提前进入低权力权重的细分市场，从而显着降低总分。 

## 方法

 暴力方法是将圆中连续字符之间的每个边子集枚举为潜在的分割点。 对于每个选择，我们独立计算所有段哈希并对它们求和。 这是正确的，因为它直接遵循问题的定义：一旦修复了切割，段哈希就是确定性的。 然而，有 n 条边，因此这会导致 2^n 个可能的分区，甚至在 O(n) 中计算每个分区也会导致不可行的 O(n·2^n) 复杂度。 

为了改进，我们需要了解剪切如何影响哈希结构。 在片段内，靠近右侧的字符贡献更大的 31 次方。这意味着同一片段中后面的位置支配前面的位置。 因此，在任何段中，将较大值的字符放置在右端是有利的。 

这一观察结果表明，分区的质量在很大程度上取决于我们是否在序列值“下降”的位置进行切割。 如果我们避免在这样的下降处进行切割，那么大的值可能会被迫进入段中的低权重位置，这是次优的。 相反，每次下降时进行切割可确保每个段局部不递减，从而将小值与小幂对齐，将大值与该段内的大幂对齐。 

关键思想是，通过精确切割下一个字符严格小于当前字符的位置（考虑循环邻接）来获得最佳分区。 这将圆分成最大的非递减分段，并且这种结构与每个分段内 31 次方的单调增长完全一致。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 枚举所有剪切| O(2^n·n) | O(2^n·n) | O(n) | 太慢了 |
 | 贪婪地削减下降| O(n) | O(n) | 已接受 |

 ## 算法演练

我们将每个字符映射到其给定的数值：a 到 1、e 到 2、h 到 3、n 到 4。 

然后我们处理一次圆形字符串并决定在哪里放置切口。 

1. 我们以循环顺序遍历字符串，并将每个字符与下一个字符进行比较。 

如果当前值大于下一个值，我们会在它们之间标记一个剪切点。 

这确保我们打破每一个递减的转变。 
2. 我们使用这些切割点将圆分成几段。 现在，每个段都是映射值的最大非递减序列。 

这种结构保证了在一个细分市场内，价值不会随着我们的前进而下降。 
3. 对于每个段，我们直接使用定义计算其哈希值。 

由于线段是不相交的并且恰好覆盖了圆一次，因此我们可以安全地累积它们的哈希值。 
4. 我们将所有段哈希相加以获得最终答案。 

### 为什么它有效

 在一段内部，当我们向右移动时，31 的指数会增加，因此每个位置都比前一个位置变得更重要。 如果较大的值较早出现，较小的值较晚出现，则贡献不是最佳的，因为较大的值乘以较小的 31 次幂。在下降时精确切割可确保值沿线段不减少，从而防止这种错位，因此较大的值自然会移向较高权重的位置。 两个相邻非递减段的任何合并都会在边界处引入下降，这将严格减少至少一对元素的加权和贡献，因此此类合并无法改进答案。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MOD = 998244353
BASE = 31

val = {'a': 1, 'e': 2, 'h': 3, 'n': 4}

def solve():
    s = input().strip()
    n = len(s)
    a = [val[ch] for ch in s]

    # find cut positions on circular boundary
    cuts = [False] * n
    for i in range(n):
        if a[i] > a[(i + 1) % n]:
            cuts[i] = True

    # build segments
    ans = 0
    i = 0
    while i < n:
        j = i
        seg = []
        while True:
            seg.append(a[j])
            if cuts[j]:
                break
            j = (j + 1) % n
        i = (j + 1) % n

        # compute hash of segment
        h = 0
        for x in seg:
            h = (h * BASE + x) % MOD
        ans = (ans + h) % MOD

    print(ans)

if __name__ == "__main__":
    solve()
```该实现首先将字符转换为数字权重，然后以循环顺序识别发生严格下降的所有位置。 这些位置定义了段边界。 

之后，我们遍历圆圈一次，通过沿着边缘构建每个线段，直到到达切割点。 每个段都是独立处理的，其哈希值以标准多项式形式计算。 

一个微妙的点是我们永远不需要显式旋转字符串。 从索引 0 开始并进行后续切割就足够了，因为切割集完全确定了循环上的分区。 

## 工作示例

 考虑示例字符串`henan`，映射为`h=3, e=2, n=4, a=1, n=4`。 

我们检查循环过渡：

 | 我| 一个[我] | 一个[i+1] | 切？ |
 | --- | --- | --- | --- |
 | 0 | 3 | 2 | 是的 |
 | 1 | 2 | 4 | 没有|
 | 2 | 4 | 1 | 是的 |
 | 3 | 1 | 4 | 没有|
 | 4 | 4 | 3 | 是的 |

 因此，切割被放置在 0、2、4 处，将圆分成遵循这些边界的线段。 

然后，我们独立计算每个段哈希并对它们求和。 

这演示了即使最佳切割在线性视图中视觉上不明显，圆形下降如何强制分割。 

现在考虑`aenhan`，映射为`1 2 4 3 1 3`。 唯一的下降地点是`4->3`和`3->1`，生成在段的正确末端隔离高值元素的段，确保它们在哈希计算中获得更大的功率。

 ## 边缘情况

 对于像这样的单字符字符串`a`，圆有一个顶点和一个自循环过渡。 该算法没有看到严格的下降，并生成包含整个字符串的单个段，这正确地生成哈希值 1。 

对于像这样的完全递减循环`n h e a n`，每个过渡都是一个下降，因此每个角色都成为自己的片段。 每个段哈希都会减少到其单个映射值，总和只是所有权重的总和，这与最佳配置匹配，因为任何合并都会在 31 的递增幂下将较大值放在较小值之前，从而减少较大值的贡献。 

对于像这样的完全递增循环`a e h n a`，在环绕边缘处恰好有一个下降，产生等于整个字符串的单个段。 这确保了大值保留在其权重最大的段的右侧，这在指数加权结构下是最佳的。
