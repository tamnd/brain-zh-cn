 ## 解决方案
 ## 问题理解

 我们得到了几个长度相等的二进制字符串。 将它们视为一个矩阵`n`行和`m`列，其中每个条目为 0 或 1。每行是一个字符串，每列是所有字符串共享的一个位位置。 

然后我们执行一系列操作。 每个操作选取两行`i`和`j`，一段列`[l, r]`，和一个概率`p/100`。 有了这个概率，操作就会“激活”，如果激活了，我们就会覆盖行`j`通过与行进行按位与运算来对该段进行操作`i`。 换句话说，对于段中的每个位置，`sj[x]`变成`sj[x] & si[x]`。 

因此，成功的操作只会将 1 变成 0，而不是相反。 

完成所有操作后，我们从概念上逐列对所有行进行按位与，并计算最终与中有多少个位置为 1。 由于操作是概率性的，因此答案是此类位置的预期数量，以 998244353 为模。 

一个关键的观察结果是，除非通过共享相同的操作，否则列不会相互交互。 特定列最终是否贡献 1 仅取决于该列内发生的情况。 这使得问题在每个比特位置上都是可分离的，最终的答案是所有列上所有行在该列上保持为 1 的概率之和。 

限制非常严格：`n ≤ 1000`,`m ≤ 4000`， 和`q ≤ 2 × 10^5`。 这立即排除了通过每行每操作更新来独立模拟每列的完整过程，因为这将是以下顺序：`q * m`，太大了。 

一个天真的错误是每列独立地处理每个操作并直接乘以生存概率。 但这会失败，因为操作的效果取决于行的演变状态，而不仅仅是初始位。 第二个微妙的失败情况是忽略传播：变为 0 的行稍后可以进一步传播零，因此单个操作可以间接影响许多其他行。 

例如，假设第 1 行是`100`，第 2 行是`111`，第 3 行是`111`。 如果第 1 行在某个位置有 0，并且我们成功申请`1 → 2`， 然后`2`在该位置变为 0，之后`2 → 3`可以进一步传播。 仅考虑操作的直接影响的解决方案将错过这种级联。 

## 方法

 一种蛮力的方法是模拟整个过程。 对于每个位位置，我们模拟整个操作序列，每次随机决定它是否激活。 然后我们重复多次并根据经验估计概率。 这在概念上是正确的，但在约束下在计算上无用，因为即使是一次模拟也是如此`O(nm + qn)`为了准确性，我们需要多次重复。 

另一种直接确定性方法是尝试跟踪所有可能配置的精确概率分布`n`每个位的行。 这是不可能的，因为状态空间是指数的。 

关键的结构见解是停止考虑完整配置，而是仅跟踪每行每比特的边际概率。 即使行是相关的，单个行的“仍然为 1”概率的演变也可以精确更新，因为每个操作都有一个简单的单调效应：它要么不执行任何操作，要么强制执行一个约束，可能根据源行将一些 1 变成 0。 

我们独立地重新解释每一位。 修复一列。 此列中的某些行以 0 开头。 这些行充当永久的“污染源”：一旦一行变为 0，它就永远保持为 0。 一次手术`i → j`（激活时）说：如果`i`此时为 0，则`j`必须变成0。 

因此，零沿着成功激活的操作向前传播，但只能从传播时已经为零的源传播。 

这导致有向图上的概率感染过程，其边以激活概率的时间顺序出现。 我们只需要以 1 开头的行不会被此进程感染的概率。 

我们可以为每一行和每一位维护按时间顺序处理操作后该行仍然为1的概率。 处理操作时`i → j`，如果成功的话，`j`变为 0 如果`i`已经是 0。这给出了一个干净的概率更新规则。 

对于固定位，令`dp[j]`是该行的概率`j`处理完一些操作的前缀后仍然是1。 最初，`dp[j]`如果输入中该位为 0，则为 0，否则为 1。 

处理操作时`(i, j, p)`影响该位的概率`p/100`它激活了。 如果激活的话，`j`变为 0 如果`i`已经是 0。这给出了一个增加机会的转变`j`变为 0 的概率与`i`已经是0了

 这会产生每个操作的线性更新，并且对所有行的贡献求和给出了所有行对于该位保持为 1 的概率。 

蛮力之所以有效，是因为它明确地跟踪每个状态。 最佳方法之所以有效，是因为我们从不需要完整状态，只需要每行每位的生存概率，并且转换保留线性结构。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 全模拟/暴力破解 | 指数或蒙特卡罗| O(纳米) | 太慢了 |
 | 行上的每比特概率 DP | 最坏形式的每位 O(qn) | O(n) | 可接受聚合洞察 |

 ## 算法演练

 我们独立处理每一位，并计算在所有操作之后，每行该位中仍然有 1 的概率。 

1.对于固定位位置，初始化一个数组`dp`尺寸的`n`， 在哪里`dp[j]`如果原始字符串在行中有 1，则为 1`j`，否则为 0。这表示该行的概率`j`一开始还是安全的。 
2. 预先计算每个操作是否影响当前位，即该位是否位于其区间内`[l, r]`。 如果不是，则可以忽略该位。 
3. 按顺序处理操作。 对于一次手术`(i, j, p)`影响当前位，计算其激活概率`p = p / 100`。 
4. 更新`dp[j]`利用这一行的事实`j`仅当操作激活且行时，此操作中才会丢失`i`已经不安全了。 的概率`j`保持安全的程度根据之前安全的可能性以及避免感染的可能性而增加`i`。 
5. 处理完所有操作后，计算所有行的乘积`dp[j]`。 该值是所有行在该位仍然为 1 的概率。 
6. 将该值与所有位位置相加，并输出对 998244353 取模的结果。 

### 为什么它有效

 这个过程是单调的：一旦一个位连续变为 0，它就永远不会返回到 1。每个操作要么保留当前状态，要么添加一种新的零传播方式。 因为每个操作仅取决于源行当前是否为 0，并且因为这种依赖性通过概率线性进入，所以我们可以安全地更新边际概率，而无需跟踪联合分布。 每比特的预期贡献正是从初始零开始的传播链到达任何以 1 开始的行的概率。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MOD = 998244353

def modinv(x):
    return pow(x, MOD - 2, MOD)

def main():
    n, m = map(int, input().split())
    a = [input().strip() for _ in range(n)]

    q = int(input())
    ops = []
    for _ in range(q):
        i, j, l, r, p = map(int, input().split())
        ops.append((i - 1, j - 1, l - 1, r - 1, p))

    # precompute inverse 100
    inv100 = modinv(100)

    ans = 0

    # process per bit
    for bit in range(m):
        dp = [0] * n

        # initial survival probability per row for this bit
        for i in range(n):
            dp[i] = 1 if a[i][bit] == '1' else 0

        # process operations affecting this bit
        for i, j, l, r, p in ops:
            if l <= bit <= r:
                prob = p * inv100 % MOD

                # probability j becomes unsafe increases if i is unsafe
                # we track safety, so we scale down survival
                # dp[j] = dp[j] * (1 - prob * (1 - dp[i]))
                # rewrite carefully in modular form

                fail_from_i = (1 - dp[i]) % MOD
                add_fail = prob * fail_from_i % MOD
                dp[j] = dp[j] * (1 - add_fail) % MOD

        cur = 1
        for i in range(n):
            cur = cur * dp[i] % MOD

        ans = (ans + cur) % MOD

    print(ans)

if __name__ == "__main__":
    main()
```该代码迭代每一位，并维持每行在该位中作为 1 存活的概率。 每个操作仅在覆盖当前位时才起作用。 转换仅取决于源行的当前生存概率，并乘法减少目标行的生存概率。 

所有行的最终乘法计算该位中没有行被损坏的概率，并且位之间的求和给出了预期的计数。 

一个微妙的实现细节是模概率算术。 每个概率更新都必须以 998244353 为模进行，除以 100 是使用模逆来处理的。

 ## 边缘情况

 关键的边缘情况是所有行的某个位中都已经有 1。 在这种情况下，只有从变为 0 的行引入依赖性的操作才能降低最终概率。 该算法处理这个问题是因为`dp`所有行都从 1 开始，只有成功的操作才会减少它。 

另一种边缘情况是所有行的一位中都有 0。 然后`dp`到处都是从零开始，每项贡献都是零。 该算法立即对该位产生零贡献。 

最后一个微妙的情况是一长串操作，其中零传播到多行。 尽管这看起来像一个多步依赖问题，但每次操作的概率更新已经考虑了通过中间状态的间接传播，因此不需要特殊处理。
