 ## 解决方案
 ## 问题理解

 我们得到一个由小写英文字母组成的长字符串。 从这个字符串中，我们可以删除字符并保持剩余字符的顺序，形成一个子序列。 

目标是确定我们是否可以准确地选取 17 个字符来形成非常严格的模式：前 5 个字符全部相同，接下来的 7 个字符也全部相同（可能是不同的字母），最后 5 个字符再次全部相同（可能是第三个字母）。 这三个块是独立的，因为每个块都是恒定的，但不同块的字母可能会或可能不会彼此匹配。 

如果存在这样的子序列，我们必须输出任何一个有效的子序列。 如果不存在，我们输出`none`。 

输入大小最多可达一百万个字符。 这立即排除了任何试图构造所有子序列甚至所有位置组合的方法。 任何在位置对上具有大常数的二次甚至接近线性的东西都会太慢。 我们需要一些东西，要么将字符串压缩成有用的结构，要么将搜索减少到少量恒定数量的候选。 

一个微妙的问题是这三个片段可以重用同一个字符。 例如，像这样的字符串`aaaaaaaaaaaaaaaaa`有效，因为所有三个段都可以是同一个字母。 在这种情况下，假设分段必须不同的粗心方法将失败。 

另一种边缘情况是，某个字符出现了足够多的次数，但它们交错太多，无法按顺序形成所需的块。 例如，一个字符可能会出现 20 次，但如果我们贪婪地获取第一个块的较早出现次数，则可能会破坏后面块的可行性，除非我们显式检查排序。 

## 方法

 一个蛮力的想法是尝试为三个块中的每一个选择一个字母，然后搜索 5 次出现，然后再搜索 7 次，然后再搜索 5 次。 每个块有 26 个选择，因此有 26³ 种可能性。 对于每个三元组，我们扫描字符串或使用指针来查找有效的子序列。 每次尝试的简单扫描都会花费 O(n)，导致大约 26³·n 次操作，这对于高达 10⁶ 的 n 来说太大了。 

我们可以通过预先计算每个字符的出现位置列表来改进。 然后检查我们是否可以在给定索引之后放置一个包含 k 个出现的块，就变成了在该列表中向前行走或使用二分搜索的问题。 这将对固定的字母三元组的检查减少到非常小的、有效的常数或对数。 

关键的观察结果是，除了排序限制之外，结构在字母之间完全独立。 我们永远不需要在块内混合字母，因此每个块只是从单个预计算列表中选择的出现次数。 这将问题转化为测试三个序列中有序拣选的可行性。 

因为字母表大小固定为 26，所以一旦每次检查有效，迭代所有三元组是可以接受的。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 对子序列的暴力破解 | 指数| O(1) | O(1) | 太慢了 |
 | 使用线性扫描尝试所有三元组 | O(26立方·n) | O(1) | O(1) | 太慢了 |
 | 每次检查预计算位置 + 二分查找 | O(26立方·log n) | O(n) | 已接受 |

 ## 算法演练

 对于每个字符，我们存储它出现在字符串中的所有索引的排序列表。 这让我们可以直接跳转到给定位置之后下一个有效出现的字符。 

1. 构建数组`pos[c]`对于每个字符`c`，包含按升序出现的所有索引。 这种预处理是必要的，这样我们就可以快速定位任意位置后第 k 次出现的位置。 
2. 迭代所有有序的字符三元组`(a, b, c)`从`'a'`到`'z'`。 每个三元组代表三个块的候选模式。 
3. 对于固定三元组，尝试从左到右贪婪地构造子序列。 我们维护一个指针`cur`这代表我们接下来可以使用的最小索引。 
4. 尝试选取出现 5 次的字符`a`从`pos[a]`，每次严格选择之后的第一个出现`cur`。 如果在任何时候出现的此类情况少于 5 次，则该三元组会立即失败。 
5.成功拾取第一个块后，更新`cur`到最后选择的位置`a`。 
6. 重复相同的过程以选取 7 个出现的字符`b`后`cur`, 更新`cur`再次回到最后选择的位置。 
7. 最后，尝试选取 5 次出现的字符`c`后`cur`。 如果成功，我们就构造了一个有效的子序列并可以输出相应的字符。 

贪婪选择在固定三元组内起作用的原因是，延迟块内的任何选择只会减少未来的可用性，而永远不会增加它。 由于块内的所有字符都是相同的，因此选择最早的有效位置可以保留最大的灵活性。 

### 为什么它有效

 对于任意固定三元组`(a, b, c)`，该算法始终根据位置构造字典顺序最早的有效子序列。 如果该三元组存在任何有效子序列，则存在一个子序列，其中每个选定的出现都被最早可能的有效出现替换，而不会破坏排序约束。 这个交换参数确保贪婪选择不会消除同一个三元组可能存在的解决方案。 由于测试了所有可能的三元组，因此必须发现任何有效的模式。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    s = input().strip()
    n = len(s)

    pos = [[] for _ in range(26)]
    for i, ch in enumerate(s):
        pos[ord(ch) - 97].append(i)

    def take(lst, start, k):
        res = []
        cur = start
        for _ in range(k):
            # find first index > cur
            # binary search
            lo, hi = 0, len(lst)
            while lo < hi:
                mid = (lo + hi) // 2
                if lst[mid] > cur:
                    hi = mid
                else:
                    lo = mid + 1
            if lo == len(lst):
                return None, start
            cur = lst[lo]
            res.append(cur)
        return res, cur

    for a in range(26):
        for b in range(26):
            for c in range(26):
                cur = -1

                res_a, cur_a = take(pos[a], cur, 5)
                if res_a is None:
                    continue

                res_b, cur_b = take(pos[b], cur_a, 7)
                if res_b is None:
                    continue

                res_c, cur_c = take(pos[c], cur_b, 5)
                if res_c is None:
                    continue

                idxs = res_a + res_b + res_c
                out = [''] * 17
                for i, p in enumerate(idxs):
                    out[i] = s[p]
                print(''.join(out))
                return

    print("none")

if __name__ == "__main__":
    solve()
```该解决方案首先按字符对索引进行分组，这是实现高效跳转的核心结构。 辅助函数`take`负责提取给定位置之后接下来的 k 个事件。 它对预先计算的列表使用二分搜索来定位下一个有效索引，确保每个步骤都是字符频率的对数。 

三重循环枚举了三个块的所有可能的字符分配。 虽然看起来很大，但常数因子固定为 26 立方，在紧张的时间限制下，当每次可行性检查都很高效时，这个常数已经足够小了。 

一旦找到有效的三元组，构造就会立即完成，因此在大多数情况下程序会提前停止。 

## 工作示例

 考虑输入字符串`aaaaacccccccaaaaa`。 

我们可以追踪三重奏的成功尝试`(a, c, a)`。 

| 步骤| 块| 人物 | 所选职位 | 当前指针|
 | --- | --- | --- | --- | --- |
 | 1 | 前 5 | 一个 | 0,1,2,3,4 | 4 |
 | 2 | 下一页 7 | c | 5,6,7,8,9,10,11 | 11 | 11
 | 3 | 最后 5 | 一个 | 12,13,14,15,16 | 16 | 16

 构造完这些索引后，我们直接输出对应的字符，形成一个有效的子序列。 

该跟踪表明该算法正确地遵守了跨块的排序约束并且从不重用较早的位置。 

现在考虑一种早期发生故障的情况，例如`ababa`重复多次，但单个字符的连续可用性不足。 对于三重喜欢`(a, a, a)`，第一个块可能会成功，但第二个块可能会失败，因为在第一次选择后没有留下足够的出现次数，这可以正确防止无效输出。

 ## 边缘情况

 一种边缘情况是所有三个段都使用相同的字符。 在这种情况下，该算法仍然有效，因为同一个列表被重复使用了三次。 前 5 个选择减少了可用后缀，但由于列表是向前扫描的，后面的选择会自动遵循顺序。 

另一种情况是，某个字符在全局范围内出现了足够多次，但在早期选择后不处于可用序列中。 例如，出现的次数可能在开始时很密集，后来稀疏。 里面的二分查找`take`确保我们始终遵守严格的顺序，因此我们永远不会错误地重复使用较早的位置。 

最后的边缘情况是存在多个有效的三元组。 该算法在第一次成功构造时停止，这很好，因为该问题允许任何有效的子序列。
