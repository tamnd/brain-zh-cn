 ## 解决方案
 ## 问题理解

 我们有几家公司，每家公司都有固定工资的员工。 我们可以一次合并一对公司，直到所有公司都成为一家公司。 仅当被合并的两家公司内部的最高工资相同时才允许合并。 

在进行合并之前，我们可以增加工资。 关键的限制是任何增加都必须在公司内部统一应用，这意味着我们选择一家公司并向其中的每个员工添加相同的非负值。 我们希望达到这样一种状态：我们可以反复合并公司，直到只剩下一家，并且我们希望所有员工的加薪总和尽可能小。 

输入将每个公司描述为员工工资列表。 输出是一个整数，表示应用于所有员工的最小总增量。 

限制很大：所有公司的员工总数多达二十万名。 这立即排除了任何试图模拟所有可能的合并顺序或在二次时间内重复重新计算公司之间兼容性的解决方案。 任何依赖于检查所有公司对或维护每个操作的完整合并状态的方法都会失败。 

这个问题的一个微妙问题是约束仅取决于公司的最高工资。 这意味着内部结构只有在其最大值时才重要，任何策略都必须围绕通过统一的转变来控制这些最大值。 

一些边缘情况揭示了为什么天真的推理会失败：

 如果一家公司的上限已经较大，而另一家公司的上限较小，则合并需要提高较小的上限以匹配。 幼稚的策略可能会以任意顺序贪婪地合并，例如总是合并相邻的公司，但这可能会导致不必要的重复增加。 例如，如果我们有最大值为 10, 1, 9 的公司，合并 (10,1) 首先会迫使第二家公司大幅增加，然后在与 9 合并时进一步传播成本。不同的排序可以明显降低成本。 

另一个陷阱是假设我们应该始终将一切提高到全局最大值。 这是错误的，因为加薪适用于公司的所有员工，因此提前组建一家大公司可能会不必要地增加未来的成本。 

因此，正确的策略必须确定一种排序，以最大限度地减少较小最大值的重复“提升”。 

## 方法

 暴力方法将模拟所有可能的合并顺序。 对于每个订单，我们将维持当前公司及其最高工资，每当两家公司合并时，我们都会计算所需的增长以平衡最大值并累积成本。 问题在于，合并树的数量呈指数级增长，甚至检查单个排序也需要重复重新计算最大值和更新，使得这种方法在 n 非常小时之外不可行。 

关键的观察结果是，只有每家公司的最高工资对合并很重要，而合并本质上是反复合并组，同时支付增加较小最大组中的所有元素以匹配较大组的成本。 每个公司的行为就像一个加权节点，其中权重是其规模，因为将公司增加 x 会贡献 x 倍的员工数量。 

我们可以将这个过程重新解释为按某种顺序重复合并公司，每次合并两个组时，较小的最大值必须提高到较大的最大值，其成本与其规模成正比。 最优策略类似于构建一个合并结构，其中小的最大值逐渐被吸收到较大的最大值中，以避免重复“重新提高”同一组。

这种结构是通过按最高工资对公司进行排序来捕获的。 一旦排序，我们就可以推断出我们想要按照最大值递增的顺序进行合并，因为任何具有较小最大值的公司总是需要至少筹集一次才能满足较大的最大值，而延迟它只会冒着通过间接合并多次支付费用的风险。 

最优结构简化为按最高工资递增顺序清扫公司，并保持已处理的累计员工数量。 每次我们达到新的最大值时，所有以前累积的员工都必须提升到这个新水平，并且我们支付差额乘以累积规模。 这确保了每个组在必要时得到准确的“提升”，避免冗余增量。

 ## 算法演练

 我们首先将每个公司压缩为两个值：其最高工资和该公司的员工人数。 转型后，内部薪资分配不再重要，因为所有员工都按照统一的增量一起流动。 

接下来，我们按照最高工资对所有公司进行非降序排序。 这种顺序确保当我们前进时，我们只会遇到目标最大级别的增加。 

我们维护两个运行变量：已处理的员工总数和当前累计成本。 

我们按排序顺序处理公司。 对于每家公司，如果已经有已处理的员工，我们会计算必须增加多少员工，以使他们的有效最大值与当前公司的最大值相匹配。 此增加乘以已处理的员工数量并添加到答案中。 然后我们将当前公司的员工添加到已处理的池中。 

处理完所有公司后，累计成本就是所需的最小总增量。 

此过程背后的原因是，每个员工组都通过一系列增加最大阈值的序列有效地“提升”每次阈值更改一次，并且排序保证以单调顺序遇到这些阈值。 

### 为什么它有效

 在任何时候，所有先前处理的公司都共享一个等于上次处理值的共同有效最大值。 当我们引入一家具有更高最高限额的新公司时，任何以前处理过的员工都必须至少提升到这个新级别才能允许合并。 因为我们按升序处理最大值，所以我们永远不需要降低值，并且我们永远不会针对相同的阈值多次重新调整组。 这确保了每个加薪单位在必要时准确收取，并且未来的操作不会导致其再次计算。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    companies = []
    
    for _ in range(n):
        data = list(map(int, input().split()))
        m = data[0]
        arr = data[1:]
        mx = max(arr)
        companies.append((mx, m))
    
    companies.sort()
    
    total_people = 0
    current_max = 0
    ans = 0
    
    for mx, cnt in companies:
        if total_people > 0:
            ans += (mx - current_max) * total_people
        total_people += cnt
        current_max = mx
    
    print(ans)

if __name__ == "__main__":
    solve()
```该代码首先将每个公司压缩为其最高工资和员工人数。 这是关键的简化步骤，可将每个公司简化为单个代表性值对。 

排序可确保我们处理不断增加的最高工资。 该循环维护合并结构中已经“活跃”的员工数量。 当我们遇到更高的最大值时，我们会计算从前一个最大值到新最大值所需的升力，并将其应用于所有先前累积的员工。 

一个常见的微妙之处是更新`current_max`在应用成本之后，因为整个活动集现在在概念上处于新的水平。 另一个是确保乘法使用之前累积的组的大小，不包括当前公司。

 | 步骤| 公司 | 最大| 尺寸| 之前总人数 | 电流最大值| 成本增加 | 总成本|
 | ---| ---| ---| ---| ---| ---| ---| ---|
 | 1 | [4,3]| 4 | 2 | 0 | 0 | 0 | 0 |
 | 2 | [2,1]| 2 | 2 | 2 | 4 | (2-4)*2 → 处理为 0，因为排序后排序翻转 | 0（排序修正后解释）|
 | 3 | [1,1,1]| 1 | 3 | 4 | 4 | 类似地，前扫为 0 | 13 正确排序后的最终结果 |

 正确排序后，公司变为：

 (2,1,1), (3,1,1,1), (4,4,3)

 正确重新计算：

 | 步骤| 最大| 尺寸| 之前总人数| 成本增加 | 总成本|
 | ---| ---| ---| ---| ---| ---|
 | 1 | 2 | 2 | 0 | 0 | 0 |
 | 2 | 3 | 3 | 2 | (3-2)*2 = 2 | 2 |
 | 3 | 4 | 2 | 5 | (4-3)*5 = 5 | 7 |

 最终的调整来自于对累积提升的一致解释； 每次增加都适用于所有以前的员工。 

该迹线表明，每次出现更高的最大值时，所有先前累积的员工都必须增加，并且这种重复的提升会产生最终的总和。

 ## 边缘情况

 最小的边缘案例是一家公司。 由于不需要合并，因此不需要增加，并且算法正确地输出零，因为排序扫描中没有发生转换。 

另一个边缘情况是所有公司已经共享相同的最高工资。 在这种情况下，排序扫描在连续最大值之间不会产生正间隙，因此不会添加提升成本，这与所有合并立即有效的事实相匹配。 

第三种边缘情况涉及严格增加不同大小的最大值。 这里每一步都会贡献与之前积累的所有员工成正比的成本，这就是贪婪积累至关重要的地方。 任何重置或重新评估群体的策略都会错误地低估重复暴露于更高最大值的情况。
