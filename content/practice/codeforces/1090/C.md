 ## 解决方案
 ## 问题理解

 我们有几个盒子，每个盒子都包含一组不同的物品。 每个项目都有一个类型，并且没有盒子包含相同类型的重复项。 物品总数很大，并且物品可以在箱子之间一次移动一件。 

目标是使所有盒子大小尽可能平衡，这意味着在所有操作之后，每个盒子应包含 ⌊S / n⌋ 或 ⌊S / n⌋ + 1 个项目，其中 S 是项目总数。 在实现这种平衡配置的所有方法中，我们希望最大限度地减少执行单个项目移动的次数。 每次移动都包括从一个盒子中取出一个特定物品并将其放入另一个盒子中，同时始终保留盒子不能包含重复物品类型的规则。 

关键的困难在于我们不仅仅是重新分配计数。 我们必须明确决定哪些具体项目移动以及它们去哪里，并确保每个中间和最终配置保持有效。 

约束足够大，任何对盒子或物品具有二次行为的解决方案都会失败。 对于多达 100,000 个盒子和 500,000 件物品总数，操作必须在物品总数上接近线性。 这排除了任何尝试重复模拟平衡或重新计算每个项目的全局分配的方法。 

当多个盒子具有相同的大小或当有许多有效的选择来确定哪些盒子获得额外的 +1 容量时，就会出现一个微妙的问题。 不小心选择哪些盒子是“较大的目标盒子”可能会不必要地增加移动次数，因为将更高的目标分配给已经很大的盒子会减少多余的移动。 

另一种隐藏的边缘情况是，所有框最多相差一个，但初始分布仍然不是最优的，因为相对于所选目标配置，项目不在“正确”的框中。 即使在这种情况下，我们可能仍然需要采取行动，但只需要那些纠正容量不匹配的行动。 

## 方法

 一种直接的蛮力想法是重复选取最小和最大的盒子，并将任意项目从最大的移动到最小的，直到大小差异最多为 1。 这很容易模拟，但会失败，因为每次移动都必须尊重项目身份，并且选择要移动的项目会影响未来的可能性。 在最坏的情况下，每次移动都需要扫描盒子以找到不违反约束的有效项目，从而导致项目总数出现二次或更差的行为。 

关键的观察结果是，最终的目标配置仅取决于盒子大小，而不取决于项目类型。 一旦我们决定了每个盒子的最终尺寸，每个物品要么保留在原来的盒子中，要么移出，并且除了每个盒子的容量限制之外，不同物品之间不存在耦合。 这将问题变成了类似流程的分配：每个盒子必须准确保留固定数量的原始物品，所有多余的物品都成为移入赤字盒子的候选物品。 

为了尽量减少移动，我们最大限度地增加留在原来盒子里的物品数量。 这是通过以最小化总盈余的方式分配目标容量，然后贪婪地将盈余项目与赤字框配对来实现的。 

我们将 ⌊S / n⌋ 分配给所有盒子，并为初始尺寸最大的 r 个盒子赋予 +1 容量，其中 r = S mod n。 此选择最大限度地减少了初始大小和目标大小之间的总正差异，从而最大限度地减少了必须移动的项目数量。 

一旦目标确定，我们就会从超出目标的盒子中收集多余的物品，并将它们重新分配到低于目标的盒子中。 由于盒子内的所有物品都是不同的，因此我们可以自由移动任何多余的物品而不会违反限制。

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 反复贪心平衡| 最坏情况 | O(S·n) O(S)| 太慢了 |
 | 目标规模+盈余再分配| O(S)| O(S)| 已接受 |

 ## 算法演练

 1. 计算项目总数 S 并计算基数 = S // n 和余数 r = S % n。 最终配置必须由 r 个大小为 base + 1 的盒子和 n - r 个大小为 base 的盒子组成。 
2. 按盒子当前大小降序排序，并将前 r 个盒子指定为“大目标盒子”，容量基数 + 1，其余的容量基数。 这种分配最大限度地减少了必须移动的多余物品的总数，因为较大的初始盒子吸收了较大的容量。 
3. 对于每个框，将其剩余量计算为 current_size - target_size。 如果该值为正数，则该盒子必须捐赠恰好那么多的物品。 如果为负，则必须接收物品。 如果为零，则它已经平衡并且仅作为源或汇间接参与。 
4. 对于每个剩余的盒子，将其物品存储在堆栈中，以便我们可以有效地弹出任意物品以移出。 只要我们保持盒子内的唯一性，物品的具体身份并不重要。 
5. 维护一个赤字框列表，每个框都需要等于 target_size - current_size 的传入项目数量。 
6. 遍历不足的盒子并重复从任何可用的剩余盒子中取出物品。 对于每次转移，输出描述源框、目标框和项目类型的移动。 减少源的盈余数和目的地的赤字数。 
7. 继续，直到满足所有赤字。 执行的移动总数将等于所有盈余的总和，这在构造中是最小的。 

正确性依赖于这样一个事实：物品标识永远不会限制盒子之间的可行性。 每个项目在任何时刻都恰好存在于一个盒子中，移动它不会创建重复项，因为它在插入到其他地方之前已从原始盒子中删除。 

### 为什么它有效

 该过程修复了目标尺寸分布，以最小化初始框尺寸和最终框尺寸之间的 L1 距离。 每个项目的移动正好减少一个盒子中的一个单位的盈余和另一个盒子中的一个单位的赤字。 由于每一步都严格将总不平衡减少一，并且任何一步都不能减少超过一单位的不平衡，因此移动次数被迫等于总盈余。 因为我们选择的目标规模能够最小化总盈余，所以最终的移动顺序是最优的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, m = map(int, input().split())
    boxes = []
    total = 0

    items = []
    for i in range(n):
        arr = list(map(int, input().split()))
        s = arr[0]
        total += s
        boxes.append([s, i])
        # store items separately
        items.append(arr[1:])

    base = total // n
    rem = total % n

    boxes_sorted = sorted(boxes, reverse=True)

    target = [base] * n
    for k in range(rem):
        target[boxes_sorted[k][1]] += 1

    surplus = [0] * n
    deficit = [0] * n

    give = [[] for _ in range(n)]
    need = []

    for i in range(n):
        s = boxes[i][0]
        t = target[i]
        if s > t:
            surplus[i] = s - t
            give[i] = items[i][:]
        elif s < t:
            deficit[i] = t - s
            need.append(i)

    ptr = 0
    surplus_stack = []

    for i in range(n):
        if surplus[i] > 0:
            # store (box, items list pointer index)
            surplus_stack.append(i)

    res = []

    # for each deficit box, fill it
    for i in need:
        while deficit[i] > 0:
            while surplus[ptr] == 0:
                ptr += 1
            j = ptr
            x = give[j].pop()
            surplus[j] -= 1
            deficit[i] -= 1
            res.append((j + 1, i + 1, x))

    print(len(res))
    for a, b, c in res:
        print(a, b, c)

if __name__ == "__main__":
    solve()
```该实现首先计算框大小的全局目标分布。 然后它会分配哪些盒子应该稍大一些，以最小化总移动成本。 之后，它会识别多余的盒子并存储其中的物品，以便它们可以作为来源。 赤字盒子被贪婪地填满，总是消耗下一个可用的剩余盒子中的物品。 

一个微妙的点是，我们永远不需要跟踪超出其价值的项目身份，因为唯一性约束是盒子的本地属性。 一旦将物品从盒子中取出，就不会再次导致重复，因此可以安全地将其立即放置在其他地方。

 ## 边缘情况

 关键的边缘情况是，所有框都已满足最终尺寸条件，但项目相对于所选目标分布并未“就位”。 该算法仍然有效，因为它仅在盒子超出其指定容量时才移动物品。 

当多个盒子在分配 +1 容量插槽时并列最大尺寸时，会出现另一种边缘情况。 其中任何一致的选择都是有效的，并且算法仍然是正确的，因为只有总剩余才重要，而不是哪些确切的盒子接收额外的槽位。 

最后一种微妙的情况是，一个盒子里的物品比它的目标恰好多了一件。 即使在这种最小盈余情况下，算法也会将其统一视为捐赠者，并且该单个项目将直接移至赤字框，而不需要任何中间重新洗牌。
