 ## 解决方案
 ## 问题理解

 我们有一排狐猴，每只狐猴都有不同的高度。 狐猴目前以某种任意顺序排列，而不是按高度排序。 国王希望将这一阵容转变为连续狐猴的每个选定部分都按照高度递增顺序在内部排序的阵容。 

该操作并不直接涉及整个数组的交换。 相反，我们首先将数组分成几个连续的段。 每个细分市场的成本都是固定的，与我们创建的细分市场数量成正比。 分割后，在每个段内我们可以执行相邻的交换，每次交换花费一个单位，因此我们可以自由地对每个段进行排序，但要为我们在其中修复的每个倒置付费。 

因此，任务是决定在哪里将数组切割成段，以便总成本是段数乘以常数加上对每个段进行排序所需的相邻交换数之和，并且使总成本最小化。 

输入由狐猴的数量和开始一段的成本参数组成，然后是高度的排列。 输出是使用相邻交换对每个段进行排序的最小可能总成本。 

对 n 的约束足够大，任何二次甚至接近二次的每段推理都会失败。 由于数组最多可以有 3×10^5 个元素，因此不可能在每个可能的分段内进行 O(n^2) 的排序模拟。 任何解决方案都必须依赖于线性或近线性摊销推理，通常使用贪婪结构或反转的全局属性。 

当数组已经排序时，会出现微妙的边缘情况。 在这种情况下，分成许多段不会进一步降低交换成本，但仍然会增加固定段成本。 每当出现局部无序时就贪婪地开始一个新的分段的幼稚方法可能会过度分段并错过将所有内容保留在一个分段中的最佳选择。 

当数组反向排序时会出现另一种边缘情况。 在这种情况下，排序成本最大，但尽早分割可以减少每个段的反转成本。 然而，由于段开销，过多的分割可能会占主导地位，因此解决方案必须仔细平衡反转减少与固定成本。 

## 方法

 暴力方法会尝试所有可能的方法将数组分割成连续的段。 对于每个分段，我们将计算独立排序每个分段所需的相邻交换数量，这相当于计算该分段内的反转。 这是正确的，因为每个段最终都使用相邻交换完全排序，并且反转计数恰好是此类交换的最小数量。 

然而，对于固定分段，计算反转计数至少需要使用芬威克树或合并排序的线性或对数线性时间。 由于分段的数量是指数级的，这立即变得不可行。 即使限制我们对段端点进行动态编程也会导致 O(n^2) 转换，这对于 n 高达数十万来说太慢了。 

关键的见解是停止考虑任意分段，而是考虑何时合并两个相邻分段是有利的。 假设我们已经有了某个位置的最优解。 延长最后一段要么使成本保持不变（除了反转贡献），要么迫使我们在决定削减时支付额外的段成本。 该决定仅取决于合并引起的反转成本的增加是否超过固定分部成本。

这将问题转变为维护一个增量跟踪反转计数的结构，并允许我们贪婪地决定扩展当前段是否有益。 关键的观察结果是，反演贡献在片段上是相加的，当我们合并片段时，只有跨边界反演发生变化。 这允许线性扫描，其运行成本可以有效更新。

 ## 算法演练

 1. 我们维护一个当前段和一个数据结构，可以在添加元素时动态计算反转次数。 这是必要的，因为段的成本完全取决于其反转计数。 
2. 我们从左到右迭代数组，将每个元素添加到当前段并更新由这次插入引起的反转计数。 反演更新是使用芬威克树或值的平衡结构来完成的。 
3. 在每个职位上，我们都会考虑继续当前业务是否有利。 该决定基于比较将元素保留在当前段中与在此处开始新段的成本，这会重置反转累积但会增加固定段成本。 
4. 当开始一个新段时，我们重置反转跟踪结构并从该点继续处理。 这有效地削减了连续的边际反转成本变得太大的数组。 
5. 我们将总成本累积为所有段成本加上每个段内计算的所有反演贡献的总和。 

经过这些步骤后，我们获得了所有有效分段的最小总成本。 

### 为什么它有效

 正确性来自于这样一个事实：一个段内的反转成本独立于其他段，而跨段的反转一旦进行切割就会消失。 每个元素仅对涉及同一段中其之后的元素的反转做出贡献，因此一旦我们决定剪切，我们将永久消除与较早元素的任何未来交互。 这就形成了一种单调的结构：一旦削减变得比继续更便宜，以后以降低总成本的方式再次合并就永远不会有利。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

class Fenwick:
    def __init__(self, n):
        self.n = n
        self.bit = [0] * (n + 1)

    def add(self, i, v):
        while i <= self.n:
            self.bit[i] += v
            i += i & -i

    def sum(self, i):
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

def solve():
    n, x = map(int, input().split())
    a = list(map(int, input().split()))

    fw = Fenwick(n)
    cur_inv = 0
    segments = 1
    total = 0

    for i, v in enumerate(a):
        greater = fw.sum(n) - fw.sum(v)
        cur_inv += greater

        fw.add(v, 1)

        # decide whether to cut here
        # cut if keeping would be worse than paying new segment cost
        if cur_inv >= x:
            total += cur_inv
            segments += 1
            fw = Fenwick(n)
            cur_inv = 0

    total += cur_inv + segments * x
    print(total)

if __name__ == "__main__":
    solve()
```芬威克树维护当前段中出现过每个值的元素数量。 插入新值时，它创建的反转数量恰好是先前看到的较大元素的数量，这是使用前缀和计算的。 

贪婪切割条件反映了积累反转成本和支付固定段成本之间的权衡。 当当前段内累积的反转变得过于昂贵时，最好进行重置。 

一个常见的陷阱是忘记每次切割后必须重置反转计数。 另一个微妙的问题是确保每个段的段成本精确计算一次，包括在循环后处理的最后一个。

 ## 边缘情况

 对于已经排序的输入，例如`1 2 3 4 5`，反转计数器永远不会增长，因此算法永远不会触发切割并只生成一个段，这是最佳的，因为添加更多段只会增加固定成本。 

对于反向排序的输入，例如`5 4 3 2 1`，反演在每一步都会累积，因此算法往往会提前削减。 每次切割都会重置结构并防止反转成本的二次增长，这符合当无序是全局时每个片段应保持较短的直觉。 

对于像这样的情况`1 3 2 5 4 6`，反转出现在小的孤立对中。 该算法使分段保持更长的时间，因为累积成本增长缓慢，这表明它对局部而不是全局无序结构敏感。
