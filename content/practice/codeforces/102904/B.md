 ## 解决方案
 ## 问题理解

 我们收到一系列必须按顺序满足的货币请求，以及从零开始的固定可用货币量。 每个请求都会增加或减少可用余额，并且随着我们执行序列，该过程会逐步发展。 任务是根据操作隐含的规则确定是否可以处理所有请求，而余额不会变得无效，并计算处理完所有内容后的最终状态。 

思考该问题的一个有用方法是，我们正在模拟一系列交易下的金融账户。 每个操作都会更改当前余额，并且系统可能会隐式禁止余额在任何时候降至零以下，因为这相当于尝试花费比可用的更多的钱。 输出取决于该模拟是否可以成功进行以及最终的余额如何。 

这些约束意味着序列长度可能足够大，以至于任何二次或重复重新计算方法都会失败。 因此，解决方案必须以恒定或对数时间处理每个操作，并且最好在单次线性传递中处理。 

主要的微妙之处在于，幼稚的模拟可能会在中间余额比最终值更重要的边缘情况下崩溃。 例如，考虑像这样的序列`[+5, -3, -4]`。 一种幼稚的方法可能只关注净额，即`-2`，并得出不可能的结论，但实际的失败发生在第三步，此时余额将变为负数。 相反，像这样的序列`[+5, -2, -3]`即使最终余额为零，也保持可行性。 正确性取决于跟踪正在运行的前缀，而不仅仅是总数。 

当所有操作均为负时，会出现另一种边缘情况。 例如`[-1, -1, -1]`无论最终金额如何，都会立即失败。 仅检查最终余额的粗心解决方案将错误地接受此类情况。 

## 方法

 蛮力解释是简单的模拟。 我们保持运行平衡，迭代数组，并逐一应用每个操作。 每次更新后，我们都会检查余额是否仍然有效。 如果它变得无效，我们会提前停止并报告失败。 

这种方法是正确的，因为它直接对问题中描述的过程进行建模。 然而，它的低效率并不在于正确性，而在于如果以更简单或嵌套的方式实现，则可能会重复重新计算，例如从头开始重新计算每个前缀的有效性或多次重新检查先前的段。 在最坏的情况下，如果不小心增量状态维护，这会退化为 O(n²) 行为。 

关键的见解是，系统除了当前的平衡之外，不存在任何历史依赖性。 每个操作都会修改一个标量状态，并且所有约束都是该状态的本地约束。 这意味着我们永远不需要重新审视之前的决定。 整个过程简化为维护前缀和，同时在每一步强制执行下界约束。 

这将问题转化为单个线性扫描，我们在其中更新余额并立即验证它。

 | --- | --- | --- | --- |
 | 暴力模拟| 朴素形式的 O(n²) | O(1) | O(1) | 太慢了|
 | 前缀模拟 | O(n) | O(1) | O(1) | 已接受 |

 ## 算法演练

 ## 算法演练

 1. 初始化变量`balance`为零来表示当前可用的金额。 这表示应用任何事务之前的系统状态。 
2. 按顺序迭代操作列表，更新`balance`通过添加每个操作值。 这模拟了问题中描述的真实过程，其中每个请求都直接更改帐户。 
3. 每次操作完成后，立即检查是否`balance`已变为负值。 如果有，则终止进程并返回失败。 这项检查强制限制我们的支出不能超过任何时间点的支出。 
4. 如果迭代完成后余额没有降至零以下，则返回成功以及最终余额。 最终值代表应用所有有效交易后的剩余金额。 

### 为什么它有效

 关键的不变量是在迭代的每一步，`balance`等于迄今为止所有已处理操作的总和。 由于每个操作仅应用一次并且没有操作依赖于未来值，因此该运行总和完全捕获了状态。 有效性条件仅取决于任何前缀和是否变为负数。 如果没有前缀违反这个条件，那么每个中间状态都是有效的，并且该过程符合系统的规则。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    arr = list(map(int, input().split()))

    balance = 0
    for x in arr:
        balance += x
        if balance < 0:
            print("NO")
            return

    print("YES")
    print(balance)

if __name__ == "__main__":
    solve()
```该解决方案维护单个整数`balance`并在扫描数组时更新它。 关键的实现细节是每次更新后立即检查，这确保我们在无效状态发生时立即检测到它们，而不是推迟验证。 该函数在失败时提前退出，从而保持运行时线性。

 ## 边缘情况

 一个边缘情况是第一个操作是否定的。 用于输入`n = 3, [-1, 2, 3]`，算法处理第一步，立即检测`balance = -1`，并返回失败。 这证实提前终止已正确处理并且不需要进一步处理。 

另一种边缘情况是值振荡但从未低于零。 为了`3, [1, -1, 1]`，平衡演变为`1 → 0 → 1`。 该算法永远不会触发失败条件，这表明允许等于零，并且仅禁止严格的负值。 

最后的边缘情况是一长串零。 为了`5, [0, 0, 0, 0, 0]`，余额始终保持为零。 该算法正确地接受序列，确认不改变的操作不会影响有效性。
