 ## 问题理解

 我们得到了一个歌曲排名列表，其中每首歌曲最初位于从 1 到 n 的唯一位置。 对于每首歌，我们都会收到关于其当前位置是否可以接受或应该改变的统一意见。 最后的任务是构建这些歌曲的新排列，这意味着我们必须将每首歌曲分配到一个新的排名，同时尊重所有意见。 

每种观点都限制了歌曲的新位置与其原始位置的比较。 如果一首歌曲被评为太低，它必须移动到一个更好的（较小的编号）排名。 如果太高，则必须移动到更差（编号更大）的排名。 如果它被标记为正确，它必须准确地保持在原来的位置。 输出的不是最终排名本身，而是逆映射：在新排名的位置 i 处，我们输出原来的位置占据了它。 

因此，真正的决定是将每个原始索引 x 分配给新位置 pos[x]，形成 1 到 n 的排列，同时遵守 pos[x] < x、pos[x] > x 或 pos[x] = x 等约束。 

约束很严格，因为每个测试用例 n 最多可达 100,000，总大小为 200,000。 这排除了任何二次分配或排列上的天真的回溯。 每个测试用例的任何解决方案都必须接近线性或 n log n。 

当在没有全局结构的情况下应用贪婪直觉时，会出现微妙的失败情况。 例如，如果许多歌曲想要向左移动，并且早期只有几个位置存在，则过早分配一首受限制的小歌曲可能会阻止稍后受限制的歌曲。 当固定位置歌曲被延迟处理而不是立即执行时，就会发生另一种失败。 

一个具体的问题场景是，当两首歌曲都想要位置 1，但只有一首可以占据它时，任意选择会导致将一首固定位置的歌曲挡在位置 2，而别无选择。 

## 方法

 蛮力的观点是将其视为在检查约束的同时将每首歌曲分配到一个空闲位置。 人们可以尝试生成排列并验证它们，或者通过约束检查进行回溯。 这在概念上是有效的，因为我们始终确保每个分配都遵循不等式条件，但它探索了指数搜索空间。 在最坏的情况下，即使修剪仍然会导致阶乘增长，因为每个决策都会减少到在许多有效位置中进行选择。 

关键的结构观察是每首歌曲没有任意约束，而是有效位置的单个区间。 标记太低的歌曲只能到达其原始位置之前的前缀中的某个位置。 标记太高的歌曲只能放在其原始位置之后的后缀中。 一首正确的歌曲恰好有一个允许的位置。 因此，每首歌曲都成为一个区间分配问题：将每个项目分配到其允许范围内的唯一整数位置。 

一旦以这种方式重新构建，问题就变成了在点 1 到 n 上调度间隔，其中每个时隙必须恰好由覆盖它的一个间隔填充。 这是一个经典的贪婪分配问题：当我们从左到右扫描位置时，我们维护当前可用的间隔，并始终选择最紧急放置的间隔。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力排列搜索 | O(n!) | O(n) | 太慢了|
 | 区间贪心赋值 | O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 我们将每首歌曲转换为允许位置的区间。 对于原本位于x位置的歌曲，如果标记为0，则其音程正好是[x, x]。 如果标记为-1，则其有效位置为[1, x−1]。 如果标记为1，则其有效位置为[x+1，n]。 

之后，我们按升序处理从 1 到 n 的位置，并决定哪首歌占据每个位置。

1. 首先，我们根据每首歌的意见将其转换为音程[L，R]。 这给了我们一组必须放置在 n 个槽中的 n 个间隔。 
2. 我们按左端点 L 对间隔进行排序或分组，以便在扫描到达起点时激活它们。 指针遍历这个排序列表。 
3. 我们维护所有 L 已经 ≤ 当前位置 i 的“活动”区间的优先级结构。 当我们到达最早的可能时隙时，间隔就会变为活动状态。 
4. 在为位置 i 选择间隔之前，我们删除所有 R < i 的活动间隔，因为它们不能再分配到任何有效的位置。 这些间隔已经是不可能的，因此如果不能更早放置它们，配置就会失败。 
5. 在所有剩余的活动区间中，我们选择 R 最小的一个并将其分配给位置 i。 

我们选择最小 R 的原因是截止日期较早的时间间隔受到的限制最大。 如果我们推迟它们，它们可能会失去所有有效的时段，而稍后仍然可以放置更灵活的间隔。 

1. 我们将所选间隔标记为已使用，然后继续执行位置 i + 1。 

### 为什么它有效

 在每一步 i 中，算法都会确保如果存在有效分配，则在我们未丢弃的区间中始终至少有一个区间覆盖 i。 选择具有最小右端点的间隔可以保留未来的可行性，因为它可以避免在灵活间隔上浪费早期时隙。 这保持了以下不变性：所有未分配的间隔在未来段中仍然至少有一个可能的位置。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    T = int(input())
    for _ in range(T):
        n = int(input())
        intervals = []
        
        for i in range(1, n + 1):
            p, x = map(int, input().split())
            if p == 0:
                L, R = x, x
            elif p == -1:
                L, R = 1, x - 1
            else:
                L, R = x + 1, n
            intervals.append((L, R, x))
        
        intervals.sort()
        
        import heapq
        res = [0] * (n + 1)
        heap = []
        idx = 0
        
        for i in range(1, n + 1):
            while idx < n and intervals[idx][0] <= i:
                L, R, x = intervals[idx]
                heapq.heappush(heap, (R, x))
                idx += 1
            
            while heap and heap[0][0] < i:
                heapq.heappop(heap)
            
            if not heap:
                print(-1)
                break
            
            R, x = heapq.heappop(heap)
            res[i] = x
        
        else:
            print(*res[1:])

solve()
```代码首先将每个意见转换为具体的区间。 然后它按左端点对这些间隔进行排序，以便我们可以按扫描顺序激活它们。 最小堆用于始终检索当前有效区间中具有最小右端点的区间。 

在每个位置 i，所有 L ≤ i 的区间都插入到堆中。 然后删除 R < i 的无效区间。 堆的顶部给出了最紧急的分配间隔。 这确保我们永远不会延迟严格的约束。 

结果数组存储每个位置放置的原始索引，与所需的输出格式匹配。 

一个常见的陷阱是忘记在选择之前必须丢弃 R < i 的区间。 另一个是误解输出要求，它要求逆映射而不是直接排列。 

## 工作示例

 考虑 n = 3 的小情况：

 输入：

 1 1

 -1 2

 0 3

 间隔变为：

 1 → [2,3]

 2 → [1,1]

 3 → [3,3]

 我们处理职位：

 | 我| 活动间隔| 选择| 原因 |
 | --- | --- | --- | --- |
 | 1 | [2,2] 无活动 | - | 只有 [2,2] 尚未激活 |
 | 2 | [2,3], [2,2] | 2 | 固定间隔必须占据位置 2 |
 | 3 | [1,3], [3,3] | 3 | 固定间隔为 3 或保持有效 |

 这证实了固定约束自然会强制放置，而灵活约束则围绕它们进行调整。 

现在考虑一个竞争激烈的情况：

 n = 4

 1 -1 1

 -1 2

 -1 3

 0 4

 间隔：

 1 → [1,0] 立即无效，除非仔细处理，否则不可能，这说明了为什么区间有效性至关重要。 当位置 1 不存在活动间隔时，算法会提前拒绝。 

这表明可行性在每个步骤中都在本地得到执行。

 ## 边缘情况

 对于单个固定位置歌曲，算法会立即将其放置，因为其间隔为 [x, x]。 堆在该位置恰好包含一个有效选择，因此不会出现歧义。 

对于多首歌曲依赖于早期位置的严格约束链，选择最小 R 的贪婪规则可确保首先放置最受限制的歌曲，从而防止稍后出现死胡同。 

对于不可能的配置，其中所需的间隔在某个步骤中未覆盖任何可用位置，堆会变空，并且算法会立即正确输出 -1，而不是继续进行无效分配。
