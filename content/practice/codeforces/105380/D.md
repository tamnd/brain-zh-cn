 ## 解决方案
 ## 问题理解

 我们有一个数组，我们只关心它的前缀。 对于每个前缀，我们必须确定它是否具有称为“good”的特殊属性。 如果我们可以从两端一个接一个地取出前缀的元素，并将它们放入一个新数组中，以便结果序列按非降序排序，则该前缀被认为是好的。 

在每一步中，我们都可以删除当前段的最左边或最右边的剩余元素。 我们继续下去，直到什么都没有留下。 删除的元素形成一个序列，我们想知道是否存在这样的选择序列，使这个最终序列排序。 

任务是找到给定数组中满足该条件的最长前缀。 

这些约束迫使我们采用接近线性的解决方案。 所有测试用例的元素总数最多为 100,000，因此任何每个测试用例都是二次的方法都将立即失败。 甚至一个$O(n \log n)$仅当仔细实施并每个元素使用一次时，每个测试用例的解决方案才是可接受的； 任何涉及独立重新计算每个前缀的事情都太慢了。 

一个微妙的困难是该操作允许从两端自由选择，这意味着策略数量呈指数级增长。 简单的模拟会尝试许多左/右拾取序列，这在非常小的阵列之外是不可能的。 

贪婪直觉的一个常见失败案例是假设总是选择两端中较小的一个就足够了。 例如，在像这样的数组中$[3, 1, 2]$，即使存在有效序列，贪婪的选择也可能会陷入困境，因为早期的选择会以非局部方式影响后来的可行性。 

另一个陷阱是认为该条件相当于可以通过出队对数组进行排序，但这比标准双端队列排序约束弱，因为只要最终顺序不递减，我们就可以选择任何有效序列，不一定保留相对结构。 

## 方法

 暴力解释是模拟每一步选择左或右的所有可能序列，并检查结果序列是否是非递减的。 对于长度前缀$m$，这形成了大小为的二元决策树$2^m$。 即使检查单调性也是$O(m)$，使总复杂度呈指数级增长。 即使对于$m = 30$。 

关键的观察是，我们实际上并不关心选择的确切顺序，只关心是否存在一种方法可以从两端“消耗”数组，同时保持保持结果序列排序的能力。 这改变了视角：我们不再构建序列，而是尝试描述何时可以进行这种构建。 

思考该过程的一个有用方法是，在每一步中，我们都被迫将左端点或右端点放入一个不断增长的序列中，并且该序列绝不能减少。 如果在任何时刻两端都小于最后选择的值，则该过程是不可能的。 这表明可行性仅通过与最后选择的元素进行局部比较来确定，并且我们可以尝试贪婪地构造最佳可能的序列，同时始终选择保持未来灵活性最大的选项。 

最佳的见解是，如果我们固定最后选择的值，我们希望保持剩余部分尽可能“灵活”。 当两者都有效时选择较小的一端可以为后续步骤保留较大的值。 然而，从固定开始的贪婪模拟对于前缀来说仍然不够，因为我们需要有效地测试所有前缀。 

相反，我们颠倒了观点：我们检查前缀是否可以在总是采用较小有效端点的贪婪过程下被完全消耗。 如果这个贪婪过程成功消耗了整个前缀，那么这个前缀就是好的； 否则就不是。 这是有效的，因为贪婪过程的任何失败都对应于不存在有效选择的状态，这意味着没有序列可以成功。 

然后，我们增量扫描前缀，维护两个指针和当前最小可行阈值。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力（所有左/右序列）|$O(2^n \cdot n)$|$O(n)$| 太慢了 |
 | 每个前缀的贪婪两端结构 |$O(n)$总计 |$O(1)$| 已接受 |

 ## 算法演练

 我们使用两个指针和从两端消耗段的贪婪模拟来增量处理每个前缀。 

1. 在当前前缀的末尾初始化两个指针，左为0，右为i。 我们还从非常小的哨兵值开始跟踪最后选择的值。 
2. 虽然指针之间还留有元素，但我们考虑两端的值。 
3. 如果两端都严格小于最后选择的值，则不存在有效的下一步，并且前缀不好。 
4. 否则，我们选择两个有效候选者中较小的一个，当两者都可用时优先选择较小的端点。 我们从概念上附加它并更新最后选择的值。 
5. 我们重复直到该段耗尽或失败。 
6. 如果我们成功使用整个前缀，我们会将答案更新为该前缀长度。 

关键的想法是，在每一步中，我们总是做出局部最安全的选择，即尽可能小的有效元素。 这为未来的步骤保留了最大的灵活性。 

### 为什么它有效

 贪心策略是有效的，因为任何有效的构造都必须遵守下一个选择的元素至少是前一个元素的约束。 在一个步骤的所有有效选择中，选择最小的可用端点永远不会降低未来步骤的可行性，因为它可以为以后的约束保留更大的元素。 如果一个前缀是可解的，贪婪过程永远不会比有效策略更早陷入困境，因为任何选择更大元素的替代选择只会更积极地限制未来的选择。 

因此，贪婪过程的失败恰好对应于任何有效序列的缺失。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def is_good_prefix(a, n):
    l, r = 0, n - 1
    last = -10**18
    
    while l <= r:
        left = a[l]
        right = a[r]
        
        if left < last and right < last:
            return False
        
        # pick the smallest valid option
        if left >= last and right >= last:
            if left <= right:
                last = left
                l += 1
            else:
                last = right
                r -= 1
        elif left >= last:
            last = left
            l += 1
        else:
            last = right
            r -= 1
    
    return True

def solve():
    t = int(input())
    for _ in range(t):
        n = int(input())
        a = list(map(int, input().split()))
        
        ans = 0
        
        for i in range(n):
            if is_good_prefix(a, i + 1):
                ans = i + 1
            else:
                break
        
        print(ans)

if __name__ == "__main__":
    solve()
```该解决方案重复测试前缀并对每个前缀使用两指针贪婪检查。 关键的实现细节是，一旦失败，我们就停止扫描前缀，因为进一步扩展无法修复已经无效的前缀。 

哨兵值`last`必须小于任何数组值； 使用足够的负数可确保正确性。 比较两端时必须小心`last`，因为在决定可行性之前必须检查两者。

 ## 边缘情况

 对于像这样的单元素数组`[7]`，算法开始于`l == r`，立即选择唯一的值，并成功。 不变式是`last`由于只有一次更新，所以始终是非递减的，因此很容易维护。 

对于像这样的严格递减数组`[5, 4, 3, 2]`，第一步选择`5`，之后两端都小于`last`，因此该过程立即停止。 这与长度 1 之外不存在有效的非递减结构这一事实相符。 

对于像这样的常量数组`[2, 2, 2, 2]`，每一步的两端都等于`last`，所以贪心选择总是成功并且完整的前缀有效。 该算法永远不会陷入困境，因为平等永远不会减少未来的选择。
