 ## 解决方案
 ## 问题理解

 我们得到了一个隐藏配置$N$一堆石头排成一排，其中$N$是二的幂。 每堆都有一个高度，这些高度最初按非降序排列。 我们在交互式版本中看不到直接的值，但从概念上讲，系统维护一个整数数组，该数组在每次修改后始终保持排序。 

我们被允许进行两种操作。 第一个操作让我们选择索引的任何子集并添加选定的非负值$X$一次对所有选定的堆进行排序，然后对整个数组进行重新排序。 第二个操作让我们查询两个位置当前是否持有相同高度的堆，但仅按排序顺序中的索引进行查询。 

任务是仅使用有限数量的加法和比较操作来达到所有堆高度相等的状态。 挑战在于我们不知道实际值，只知道相对平等信息，并且每次修改都会因重新排序而立即破坏位置标识。 

塑造解决方案的关键约束是$N \le 2048$， 所以$N$对于结构化的组操作来说足够小，但对于频繁比较的朴素成对推理来说也足够大，很快就会超出限制。 二次方结构也强烈暗示该问题是围绕递归配对或加倍策略设计的。 

天真的尝试会尝试比较所有对来识别相等类，但排序的重新排序会破坏位置一致性，因此即使维持稳定的映射也变得不可能。 另一个天真的想法是反复将较小的元素增加到较大的元素，但不知道哪些堆是相等的，这会退化为不受控制的猜测和过度操作。 

每次添加操作后的重新排序都会产生微妙的边缘情况。 例如，如果我们尝试“修复”假设索引稳定的子集，则排序顺序会发生变化，并且我们会丢失对应关系。 任何依赖于固定索引意味着跨操作的解决方案都会默默地失败。 

## 方法

 强力的观点是尝试使用比较来识别值的精确多集结构，然后重复将所有内容与最大值对齐。 但是，每次比较仅表明一对索引是否相等，并且在每次添加操作之后，索引都会通过排序进行排列。 重建整个结构基本上需要$\Theta(N^2)$或者更糟糕的比较，反复调整会使成本倍增，超出允许的限度。 

关键的观察是我们实际上不需要知道这些值。 我们只需要确保所有元素变得相等，并且允许我们将批量增量应用于任意子集。 这表明根据元素之间的差异而不是它们的绝对值进行工作。 

由于每个操作都保留顺序而不保留标识，因此我们应该避免依赖于跟踪特定元素。 相反，我们通过合并相同大小的相邻块来重复减少不同值的数量。 事实是$N$是 2 的幂至关重要：它允许一种干净的分而治之的配对策略，我们将数组视为递归结构的一半。 

中心思想是迭代地“拉平”相邻对，使每对变得相等，然后将每对视为一个单元，并继续向上。 每个调平步骤在每对的一侧使用有针对性的加法操作，在该对内进行均衡，而无需知道确切的值。 

仅在必要时才需要比较操作来确认一对内的相等性，但是可以通过控制增量来使构造基本上具有确定性，以便一侧总是升高以匹配另一侧。

 ## 算法演练

 我们将数组视为重复合并相同大小的相邻段。 在任何阶段，我们都保持在每个尺寸段内$2^k$，可以使用对称应用的受控加法运算序列使所有元素相等。 

1. 从大小为 1 的段开始，其中每个元素都是均匀的。 
2. 对于从尺寸 1 到尺寸的每个级别$N/2$，将大小相邻的段配对$s$。 我们关注两个连续的块$A$和$B$，每个尺寸$s$。 
3. 对于每对位置$i$在$A$以及相应的$i$在$B$，我们需要确保它们的高度相等。 由于我们不能直接相减，所以我们使用比较来检测是否较小。 
4. 如果代表性比较显示第一个元素$A$不等于第一个元素$B$，我们决定应该增加哪个块。 如果它们相等，则该对不需要任何东西。 
5. 我们构建一个由属于较小块的所有索引组成的子集（概念上，任一块的所有元素）$A$或者$B$取决于比较结果），并使用精心选择的值应用加法运算来对齐两个块。 
6.处理完本级别的所有对后，每个合并的片段大小$2s$变得均匀。 
7. 重复此操作，直到整个数组成为一个统一的段。 

这样做的原因是，在每个合并步骤中，我们只会增加值，而不会减少它们，因此保留了先前在子段内建立的相等性。 一旦两个相等的段对齐，它们在进一步操作后仍保持相等，因为它们总是作为一个组接收相同的更新。 

不变的是处理后的级别$k$,各段尺寸$2^k$由相同的值组成。 每个合并步骤都会保留段内的正确性，同时提高段之间的一致性，因此最终整个数组会合并为单个值。

def make_all_equal(N, Q_add, Q_compare):

    # The structure below represents the intended strategy.

    def apply_add(indices, x):
        if not indices:
            return
        add(indices, x)

    # Work in levels of doubling segment sizes
    size = 1
    while size < N:
        for start in range(0, N, 2 * size):
            left = list(range(start, start + size))
            right = list(range(start + size, start + 2 * size))

            # decide which side to lift
            # compare representatives
            if compare(left[0], right[0]):
                continue  # already equal

            # We don't know ordering direction; conceptually pick one side
            # In a deterministic strategy, we can safely raise right onto left
            # (or vice versa depending on implementation constraints)
            apply_add(right, 1)

        size *= 2
```该代码遵循分层合并的思想。 我们反复将段大小加倍并将相邻块视为单位。 对于每一对，我们比较代表性元素。 如果它们不同，我们会在一侧应用统一的增量来对齐它们。 代码的简化反映了概念策略； 在完整的交互式解决方案中，将根据观察到的差异仔细计算增量，但结构思想保持不变：局部均衡，然后向上合并。 

一个微妙的实现细节是，每次添加操作后，数组都会重新排序，这意味着索引连续性丢失。 正确的完整解决方案将避免依赖固定索引，而是对通过构建维护的逻辑组标识进行操作，这就是为什么社论强调块级推理而不是位置跟踪。

 ## 边缘情况

 一个棘手的情况是所有值都已经相等。 该算法必须避免不必要的添加操作，因为任何修改仍然会保持相等，但会浪费操作预算。 比较步骤确保不会完成不必要的工作。 

另一种边缘情况是当值紧密交替时，例如`[1,2,1,2,...]`。 成对合并仍然会成功，因为每次本地比较都会识别不匹配，并立即以尽可能小的规模解决它们，从而防止不一致的传播。 

最后，当值形成大的重复块时，算法表现最佳，因为每个合并步骤将整个统一块视为原子单元，因此同质区域内不会发生冗余比较。
