 ## 解决方案
 ## 问题理解

 我们得到了一组不同的鱼名称，所有名称都有相同的可能性被选择。 杰克想通过问是或否的问题来识别所选择的鱼，并且他可以问任何他想问的问题，只要答案将剩余的候选者分为两组。 

每个问题有效地将当前候选集分成两个子集。 根据答案，我们继续在一个子集中进行。 经过足够多的问题后，我们最终得到了一条鱼，但问题陈述明确指出，即使在最后，杰克仍然会问一个最终确认问题。 因此，决策过程中的每一片叶子都贡献了一个额外的问题，超出了隔离它所需的深度。 

目标是设计是/否问题的序列，以最大限度地减少识别均匀随机鱼所需的预期问题数量。 换句话说，我们正在针对给定的项目构建一棵最优二元决策树，并且我们希望最小化平均根到叶成本加上一个最终确认步骤。 

关键的约束是问题的内容没有限制，所以唯一重要的是我们如何在每一步中划分集合。 这将问题简化为纯粹的组合树设计：我们不受字符串结构的限制，仅受项目数量的限制。 

主要的边缘情况是只有一条鱼时。 在这种情况下，不需要拆分，但该语句仍然需要一个确认问题，因此答案恰好是 1。 任何忘记这种特殊情况的算法都会错误地输出零。 

第二个微妙的情况是 n 较小，不平衡的树可能看起来局部合理，但会增加预期成本。 例如，对于三个项目，链式决策过程给出深度 1,2,3，但平衡分割给出深度 2,2,2，这在平均情况下严格来说更好。 

## 方法

 蛮力视图是考虑 n 个项目上所有可能的二元决策树。 每个内部节点对应于将当前集合划分为两个非空子集。 对于每棵树，我们将预期成本计算为叶子的平均深度加上一个最终问题。 这种方法是正确的，因为它枚举了是/否问题的每个可能的序列。 

问题是这种树的数量呈超指数增长。 即使对于中等的n，每个节点的分区数量也是巨大的，并且n个项目上的标记二叉树的总数远远超出了可行的计算。 

关键的观察是，只有概率的分布才重要，而不是项目的身份。 由于所有鱼的可能性相同，因此每个叶子的权重为 1，并且我们正在最小化加权外部路径长度。 这正是统一权重多重集上的霍夫曼编码问题。 

我们没有对问题进行推理，而是将这个过程重新解释为重复地将两个子集合并到一个父节点中。 每次合并都会增加合并大小总和的总成本。 最佳策略是始终合并两个最小的可用组，这是霍夫曼编码中使用的标准贪心规则。 

一旦构建了最优树，叶子深度的总和正是累积的合并成本。 预期的问题数量就是总成本除以 n，加上最终的确认步骤。

 ## 算法演练

 我们将每条鱼视为权重为 1 的叶节点。我们使用优先级队列模拟自下而上构建最优决策树。

1. 将 n 个值为 1 的值插入到最小堆中。 每个值代表一组必须一起区分的鱼，并且最初每条鱼都是它自己的组。 
2. 当剩余多个组时，从堆中提取两个最小的组。 它们代表应该以尽可能低的成本组合的两个子树。 
3. 将它们合并成一个新组，其大小为两者之和。 此合并表示在二叉树中创建一个新的决策节点。 此合并的成本会添加到运行总计中，因为两组中的每个元素都会获得一个额外的深度级别。 
4. 将合并的组推回到堆中并继续。 反复合并最小的组可确保将大的深度惩罚推到树中尽可能低的位置。 
5. 所有合并后，累积成本等于最优决策树中所有叶子的深度之和。 
6. 将此总成本除以 n，以获得隔离鱼所需的预期问题数量。 
7. 添加 1 以说明问题陈述所需的最终确认问题。 

这种贪婪过程正确的原因是任何二叉树都可以分解为叶子权重的合并序列，并且将较重的组放置得更深总是会增加超过必要的总成本。 通过始终合并两个最小的组，我们确保以尽可能平衡的方式应用深度的增加，从而最大限度地减少总外部路径长度。 

## Python 解决方案```python
import sys
import heapq

input = sys.stdin.readline

def solve():
    n = int(input().strip())
    if n == 1:
        print("1.0000000000")
        return

    heap = [1] * n
    heapq.heapify(heap)

    total_cost = 0

    while len(heap) > 1:
        a = heapq.heappop(heap)
        b = heapq.heappop(heap)
        s = a + b
        total_cost += s
        heapq.heappush(heap, s)

    expected = total_cost / n + 1.0
    print(f"{expected:.10f}")

if __name__ == "__main__":
    solve()
```该实现直接反映了贪婪构造。 堆存储当前组大小，并且每次合并都会将其结果大小添加到累积成本中，因为合并组中的每个元素都会将其深度增加一。 

除以 n 将总深度总和转换为期望值，因为每条鱼的可能性相同。 决赛`+1`强制执行强制性确认问题。 

唯一微妙的情况是 n 等于 1，此时不会发生合并，但仍需要一次确认。

 ## 边缘情况

 当n等于1时，算法跳过所有堆操作并直接返回1.0。 这符合即使是已知的单条鱼仍然需要确认问题的要求。 

当 n 等于 2 时，堆合并一次，产生总成本 2。除以 2 得到 1，加上确认得到 2，这符合直觉事实：一次拆分就足够了，但仍需要最终确认。 

对于较大的 n，特别是当 n 不是 2 的幂时，堆自然会生成不平衡的最终树，但始终以最深度有效的方式。 贪婪合并确保任何不平衡都被推到树中尽可能高的位置，从而最小化其对期望值的贡献。
