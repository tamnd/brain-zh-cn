 ## 解决方案
 ## 问题理解

 我们正在尝试重建一个长度未知的二进制字符串$N$通过与法官互动。 每次我们提交候选字符串时，法官都会将其与隐藏密码进行比较，并返回两个字符串从头开始匹配的长度。 

在干净的世界中，这只是最长公共前缀的长度。 问题在于该判断并不完全可靠：在某些固定概率下，返回值可能会被损坏。 腐败并不是我们可以利用的结构化方式任意的，因此唯一稳定的信号来自重复的交互和聚合。 

我们的任务是在有限的尝试次数内仅使用这些嘈杂的前缀长度查询来恢复精确的隐藏二进制字符串。 

关键的困难在于单个查询不能被信任。 如果我们得到确切的答案，问题将减少为贪婪地构建字符串$O(N)$查询。 在这里，噪声迫使我们将每个查询视为一种测量而不是事实。 

对查询的限制意味着我们不能暴力破解所有字符串或过度重复地重新采样所有内容。 任何尝试所有可能性或从头开始重建每比特的方法都会很快超出限制，特别是当$N$很大。 

当噪声始终偏向一个方向时，就会出现一种微妙的边缘情况。 例如，如果两个候选前缀产生非常接近的真实 LCP 值，则单个噪声观察可能会错误地翻转决策。 尽管底层结构很简单，但这使得确定性一次性贪婪构造变得不可靠。 

## 方法

 如果没有噪音，问题就很简单了。 我们可以从左到右构建答案。 在位置$i$，我们将尝试设置前缀并观察返回的 LCP。 如果增加，我们保留该位； 否则我们翻转它。 每个查询都会给出准确的反馈，我们将在$O(N)$查询。 

这个想法在噪声下的强力扩展是仍然使用单个查询来决定每一位。 问题是直接的：噪声会减少或夸大报告的前缀长度，这意味着单个决策可能是错误的，并且该错误会传播到所有后续位置。 在最坏的情况下，整个重建工作都会崩溃。 

关键的观察结果是，即使有噪音，每个查询也不是无用的。 它仍然包含有偏差的信号：正确的前缀往往会比不正确的前缀产生更大的 LCP 值。 这允许我们将返回的值视为以真实前缀匹配长度为中心的随机变量。 一旦我们接受了这一点，问题就变成了统计决策问题，而不是精确查询问题。 

我们不相信单个测量结果，而是多次重复每个候选查询并比较汇总结果。 对于每个位置，我们测试两个可能的位并选择具有较高预期前缀分数的位。 由于不正确的前缀会更早中断，因此即使有噪音的测量结果在求平均值时也能以很高的概率保留排序信息。

 ## 算法演练

 我们一次构建一个字符的答案字符串，维护一个我们认为与隐藏密码匹配的不断增长的前缀。 

1.初始化一个空字符串$s$。 这将成为我们重建的密码。 
2. 对于每个位置$i$从$0$到$N-1$，我们决定下一个字符是否是`'0'`或者`'1'`。 
3. 为了测试候选位，我们形成一个由已经固定的前缀组成的完整查询字符串$s$，后跟候选位，然后是任意填充符（我们可以用零填充，因为只有前缀正确性对于位置处的决策很重要$i$）。 
4. 我们多次提交此查询，收集返回的 LCP 值。 这种重复是必要的，因为单个结果可能会被损坏，但重复的样本集中在真实的前缀长度周围。 
5. 我们计算候选人的平均反应`'0'`和候选人`'1'`。 正确的位是产生更高预期 LCP 的位，因为它更有可能与较长前缀的隐藏字符串相匹配。 
6. 将更好的部分添加到$s$，将其锁定为最终答案的一部分。 
7. 一旦确定了所有位置，输出重构的字符串并终止。 

其工作的关键结构原因是 LCP 函数对于正确的前缀是单调的。 扩展正确的前缀总是会将匹配边界向右移动，而错误的位会导致立即丢弃。 当采集足够的样本时，噪声会扰乱值，但不会改变预期的顺序。 

### 为什么它有效

 在任何位置$i$，假设前缀$s[0..i-1]$已经是正确的了。 隐藏字符串在位置有固定位$i$。 如果我们测试正确的位，预期的 LCP 会增加到超过$i$比测试错误位的概率更高，因为不匹配被延迟了。 即使单个查询有噪音，平均也可以保留这种分离。 因此，每个决策步骤都会选择真实字符串的统计上占主导地位的延续，并且不会丢弃任何较早的正确前缀。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

import random

def ask(s: str) -> int:
    print(s, flush=True)
    return int(input().strip())

def decide(prefix: str, n: int, bit: str, samples: int = 5) -> float:
    # build candidate string and average noisy LCP responses
    # we pad to full length; suffix does not matter for LCP beyond mismatch
    best_score = 0
    total = 0
    for _ in range(samples):
        res = ask(prefix + bit + "0" * (n - len(prefix) - 1))
        total += res
    return total / samples

def main():
    n = int(input().strip())
    s = ""

    for i in range(n):
        score0 = decide(s, n, "0")
        score1 = decide(s, n, "1")

        if score1 > score0:
            s += "1"
        else:
            s += "0"

    # final verification (optional)
    print(s, flush=True)
    verdict = input().strip()
    if verdict == "1":
        return

if __name__ == "__main__":
    main()
```核心实施思想是我们从不依赖单一法官的响应。 每个位决策都由多个样本支持，我们比较聚合的 LCP 值而不是原始输出。 填充确保所有查询保持有效长度 -$N$字符串。 

这里的一个常见错误是忘记一旦发生第一次不匹配，后缀对于前缀比较并不重要。 这使我们能够安全地用零填充而不影响决策信号。

 考虑一个小的隐藏字符串`101`。

这证实了即使在噪声下，正确的第一位也始终在期望中占主导地位。 

### 步骤 2：决定第二位

 | 前缀 | 候选人| 查询 | LCP 样品 | 平均 |

 | 1 | 1 | 110 | 110 1, 1 | 1 |

 我们选择`1`。 

噪声样本有时会低估真实的 LCP，但重复采样可以保留正确的排序。

 ## 边缘情况

 最小输入尺寸$N = 1$是最简单的压力情况。 该算法在以下两者之间做出单一决定：`'0'`和`'1'`。 即使有噪声，重复采样也能确保正确的位比错误的位产生更高的预期匹配，因此算法可以正确收敛。 

完全统一的字符串，例如`000...0`是另一个重要案例。 在这里，两个候选决策仅在第一个不匹配点处表现不同。 该算法仍然有效，因为不正确的候选者总是更早地终止前缀，即使噪声偶尔会夸大值，也会产生始终较小的预期 LCP。 

高度交替的弦，例如`1010...`强调跨多个步骤传播正确性。 必须维护每个正确的前缀，以便下一个决策保持有意义。 由于该算法从不修改过去的决策，因此正确性取决于每个步骤的统计稳定性，这是通过重复采样来保持的。
