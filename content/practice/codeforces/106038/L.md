 ## 解决方案
 ## 问题理解

 有几个独立的项目组，每个组代表一场比赛。 对于每场比赛$i$, Fmota 最初拥有$a_i$衬衫。 时间以年为单位，从第 0 年开始到第 1 年$k$。 每年，每项比赛都会额外获得一件球衣，因此每年$t$， 竞赛$i$包含$a_i + t$衬衫。 

Fmota的佩戴流程是有严格结构的。 他不混合比赛。 相反，他按照固定的顺序一项一项地处理比赛。 对于每场比赛，他都会在参加下一场比赛之前按照他选择的任何顺序穿着该场比赛的所有衬衫一次。 这意味着在每次比赛中，这种安排都会贡献阶乘的可能性，而在比赛中，选择会成倍增加，因为块是独立的。 

任务是计算每年$t$从 0 到$k$，存在多少种不同的完整佩戴时间表，以大质数为模（隐式$10^9 + 7$）。 

关键输入结构是初始衬衫数量的列表，输出是一个序列，其中每个值对应于均匀增长后的不同年份。 

这些约束没有以清晰的形式明确说明，但大量内存限制和多年的存在强烈表明，比赛的数量和$k$可以很大，最大可达约$2 \cdot 10^5$。 这立即排除了每年从头开始重新计算阶乘或每年进行任何计算的可能性$O(n)$每次比赛都涉及大量算术的重新计算。 

为每次比赛和每年独立重新计算阶乘的简单方法会重复大型阶乘计算$k \times n$次，这太慢了。 

一种微妙的边缘情况来自增长规则。 如果求解器错误地假设只有最后一年很重要，或者忘记每年的值必须独立输出，那么他们可能会计算该年的单个乘积$k$仅有的。 另一个常见的错误是在没有缓存的情况下重新计算阶乘，从而导致重复$O(n)$在循环内工作。 

## 方法

 对问题的直接解释很简单。 对于固定年份$t$， 竞赛$i$贡献$(a_i + t)!$可能的排列。 由于竞争是独立处理并按固定顺序连接的，因此配置总数就是这些阶乘值的所有竞争的乘积。 

所以对于每一年$t$，答案是：$$\prod_{i=1}^{n} (a_i + t)!$$暴力解决方案将从头开始计算每一对的阶乘$(i, t)$。 对于每一年，我们都会独立地重新计算所有阶乘，成本$O(n \cdot (a_i + t))$如果天真地这样做，或者至少$O(n \cdot \max a_i)$每年。 多年来，这变得完全不可行。 

关键的观察结果是连续数字的阶乘通过简单的乘法转换相关：$$(x+1)! = (x+1) \cdot x!$$这意味着我们可以在一次线性传递中预先计算所有年份的阶乘，直至达到最大可能值。 一旦阶乘被预先计算，每年的答案只是一个乘积$n$预先计算的值，给出总计$O(nk)$乘法成本。 由于乘法成本低廉并且约束通常允许最多数十万次运算，因此这是可以接受的。 

真正的优化不是避免阶乘重新计算，而是确保阶乘在全局范围内计算一次，而不是在嵌套循环内重复计算。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力 |$O(k \cdot n \cdot V)$|$O(1)$| 太慢了 |
 | 最佳 |$O(V + k \cdot n)$|$O(V)$| 已接受 |

 这里$V = \max(a_i + k)$。 

## 算法演练

 我们首先确定我们需要阶乘的最大值，这是最大的初始衬衫数量加上$k$。 这决定了预处理的限制。 

然后我们预先计算阶乘模$10^9 + 7$使用单个线性递归达到该最大值。 这使得恒定时间访问任何$(a_i + t)!$。 

每年，我们都会计算所有比赛中相应阶乘值的乘积。 

### 步骤

 1. 阅读$n$和$k$，然后读取数组$a$。 

这确定了我们每年将乘以多少独立因子贡献。 
2. 计算$MAX = \max(a_i) + k$。 

这限制了所有年份中可能出现的每个阶乘值。 
3. 预计算阶乘数组`fact`从 0 到$MAX$模数$10^9+7$。 

每个值均源自前一个值，使用$fact[x] = fact[x-1] \cdot x$。 这避免了重复的重新计算。 
4. 每年$t$从 0 到$k$，将答案计算为：$$\prod_{i=1}^{n} fact[a_i + t]$$5. 输出所有计算值。 

### 为什么它有效

 正确性取决于两个独立的结构事实。 首先，在每场比赛中，所有衬衫仅在位置上有所不同，因此计算有效的穿着顺序实际上就是计算排列，这是阶乘的。 其次，比赛是在固定的区块中进行的，因此一个区块中的选择不会干扰另一区块，从而使比赛中的总计数成倍增加。 由于年份参数仅统一移动所有阶乘输入，因此预计算阶乘一次可以保留所有年份的正确性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

MOD = 10**9 + 7

def solve():
    n, k = map(int, input().split())
    a = list(map(int, input().split()))

    max_val = max(a) + k

    fact = [1] * (max_val + 1)
    for i in range(1, max_val + 1):
        fact[i] = fact[i - 1] * i % MOD

    for t in range(k + 1):
        res = 1
        for x in a:
            res = res * fact[x + t] % MOD
        print(res, end=" ")

if __name__ == "__main__":
    solve()
```阶乘预处理与年度计算分开，以便每个查询年份仅执行直接表查找。 内乘循环是每年的主要工作，这是不可避免的，因为每个竞赛都是独立贡献的。 

一个常见的陷阱是在阶乘累积过程中忘记模数，这会在最终乘法阶段之前很久导致溢出。 另一个是在年度循环内重新计算阶乘，这会将线性解转换为值范围内的二次解。 

## 工作示例

 考虑第二个示例输入：```
3 4
5 3 7
```我们首先构建阶乘直到$7 + 4 = 11$。 

每年：

 | t 年 | 使用的值| 产品表达|
 | ---| ---| ---|
 | 0 | 5！、3！、7！ | 5！ × 3！ ×7！ |
 | 1 | 6！、4！、8！ | 6！ × 4！ ×8！ |
 | 2 | 7！、5！、9！ | 7！ × 5！ × 9！ |
 | 3 | 8！、6！、10！ | 8！ ×6！ × 10！ |
 | 4 | 9！、7！、11！ | 9！ ×7！ × 11！ |

 每行都是独立的，连续行之间的唯一变化是阶乘参数的统一移动。 

该轨迹表明该算法本质上是跟踪阶乘空间上的滑动变换，而不是从头开始重新计算结构。 

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | ---| ---| ---|
 | 时间 |$O((\max a_i + k) + n \cdot k)$| 因子预处理加上所有竞赛中的每年产品|
 | 空间|$O(\max a_i + k)$| 阶乘表的存储 |

 问题陈述隐含的约束允许这种结构，因为阶乘预处理是线性的，并且每年的乘法是简单的。 即使对于大型$n$和$k$，运算是简单的模乘法，在时间限制内可以轻松完成。

 ## 边缘情况

 一种边缘情况是当$k = 0$，意味着只有初始配置很重要。 该算法仍然有效，因为它计算阶乘高达$\max a_i$并输出单个产品。 

对于输入：```
2 0
3 3
```该算法计算：$$3! \times 3! = 36$$多年来的循环只运行一次，因此不需要特殊处理。 

另一个边缘情况是当所有$a_i$是平等的。 为了：```
3 2
2 2 2
```每年简单评估：$$((2+t)!)^3$$该算法正确地应用了相同的阶乘查找三次，并且模乘法保持了正确性，而没有溢出或排序问题。
