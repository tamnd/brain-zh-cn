 ## 解决方案
 ## 问题理解

 我们得到一个项目集合，其中每个项目都有两个附加值。 一个值代表收益，解释为“你学到了多少”，另一个值代表成本或痛苦，解释为“有多少伤害”。 

从这些项目中，我们可以选择任何子集，可能包括全部或仅一个。 对于任何选定的子集，其得分定义为所选项目的学习值总和减去这些所选项目中的最大痛苦值。 任务是针对输入项的每个前缀计算通过从该前缀中选择任何子集可获得的最大可能分数。 

因此，对于每个前缀，我们正在有效地解决所有子集上的优化问题：最大化总增益减去子集中的最差（最大）成本。 

约束足够大，不可能枚举子集。 强力方法将检查每个前缀的所有 2^n 个子集，即使 n 约为 40，这也是立即不可行的。即使 n 高达 200000，也需要接近 O(n log n) 或 O(n) 的解决方案。 

一个微妙的边缘情况来自于选择空子集的可能性。 如果我们不采取任何项目，则学习总和为零，并且不存在最大痛苦值。 在对该问题的大多数解释中，这贡献为零，并且必须在最终答案中将其视为基线。 另一个棘手的情况是所有学习值都为负时。 当每个项目都是负数时，始终包含具有正向学习的项目的天真贪婪策略将会失败，因为最佳子集可能仍然包含控制最大痛苦惩罚的必要项目。 

## 方法

 直接方法会尝试前缀的所有子集。 对于每个子集，我们计算学习值的总和并减去该子集中的最大疼痛值。 这是正确的，因为它完全符合定义。 然而，对于每个大小为 n 的前缀，有 2^n 个子集，计算每个子集的分数的成本为 O(n)，这会导致指数爆炸，无法在时间限制内运行。 

关键的观察结果是，元素之间唯一的非线性相互作用是最大痛苦项。 学习值的总和是相加且独立的，但最大痛苦仅取决于所选子集中的单个最差元素。 这表明我们应该以哪个元素负责该最大值为条件。 

将项目 j 固定为所选子集中具有最大疼痛值的元素。 一旦这个问题被解决，所有其他选择的项目的疼痛都必须小于或等于 j 的疼痛。 此外，j 必须包含在子集中。 在这种情况下，目标变成选择痛苦最多为 bj 的任意项目子集，同时确保包含 j，最大化学习值的总和，最后减去 bj 一次。 

在 pain ≤ bj 的受限项目集中，除了最终的减法之外，选择之间没有耦合。 因此，对于每一项，我们仅在其提高总和时才将其包括在内。 这意味着我们包括所有具有正学习值的项目，并且我们始终包括 j，无论其符号如何，因为需要它来定义最大痛苦。 

这减少了通过痛苦对项目进行排序并维护前缀上的运行结构的问题。

 | ---| ---| ---| ---|
 | 对子集的暴力破解 | O(2^n·n) | O(2^n·n) | O(n) | 太慢了 |
 | 排序+前缀优化| O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 我们按照疼痛值递增的顺序处理项目，以便在每一步我们准确地知道在给定的最大疼痛阈值下允许将哪些项目包含在一起。

1. 按照疼痛值非降序对所有项目进行排序。 这确保了当我们将特定项目视为最大痛苦候选者时，所有早期项目都是其子集中的有效选择。 
2. 保持运行值`pos_sum`，它存储迄今为止处理的项目中所有正学习值的总和。 这代表了我们可以不受限制地自由采取的最佳贡献，因为除非需要，否则消极的学习项目永远不会有益。 
3. 迭代已排序的项目。 对于每个项目j，将其疼痛值bj视为候选子集的最大疼痛。 
4. 对于此项目，从以下位置开始计算候选分数`pos_sum`，然后纠正项目 j 必须包含在内的事实，即使其学习值为负。 这是通过添加来处理的`min(a_j, 0)`。 
5. 从结果中减去 bj，因为对于其中 j 是定义元素的任何子集，最大的痛苦只支付一次。 
6. 跟踪所有项目的最大值，并与零进行比较以允许空子集作为有效答案。 

### 为什么它有效

 在任何时候，最大疼痛为 bj 的所有子集都必须将 j 作为其定义的最大疼痛元素，这意味着不会出现具有更高疼痛的项目，并且必须存在至少一个具有恰好 bj 的疼痛的项目。 在所有这些子集中，通过包含每个具有正学习价值的项目（因为它们会增加分数而不影响惩罚）并排除除所需元素 j 之外的所有负项目，可以使总和最大化。 目标的结构保证所包含的项目之间不会超出此约束，因此这种对最大痛苦锚的贪婪分解是无损的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n = int(input())
    items = []
    for _ in range(n):
        a, b = map(int, input().split())
        items.append((b, a))  # sort by pain

    items.sort()

    pos_sum = 0
    ans = 0

    i = 0
    while i < n:
        j = i
        # process all with same b together
        while j < n and items[j][0] == items[i][0]:
            j += 1

        # compute answers for this group using current prefix
        for k in range(i, j):
            b, a = items[k]
            cur = pos_sum + (a if a < 0 else 0) - b
            if cur > ans:
                ans = cur

        # update prefix after evaluating this group
        for k in range(i, j):
            b, a = items[k]
            if a > 0:
                pos_sum += a

        i = j

    print(max(ans, 0))

if __name__ == "__main__":
    solve()
```排序步骤确保我们永远不会错误地包含比当前锚点具有更高痛苦的项目。 前缀和仅累积正学习值，因为除非强制，否则负学习值永远不会全局有益，并且在将每个项目评估为最大疼痛锚时，会显式处理强制包含。 

为了保证正确性，对相等的疼痛值进行分组并不是严格要求的，但它通过确保仅在评估给定疼痛级别的所有候选者之后才应用前缀贡献来保持逻辑清晰。 

## 工作示例

 考虑包含三个项目的输入：（学习、痛苦）等于 (5, 4)、(2, 1) 和 (3, 3)。 

按疼痛排序，我们得到 (2,1)、(3,3)、(5,4)。 我们从`pos_sum = 0`。 

在第一项 (2,1) 处，我们将候选计算为 0 + 0 - 1 = -1，然后更新`pos_sum`至 2。由于子集为空，答案仍为 0。 

在第二项 (3,3) 处，候选变为 2 + 0 - 3 = -1，则`pos_sum`变成5。 

在第三个项目 (5,4) 处，候选者变为 5 + 0 - 4 = 1。这将答案提高到 1，对应于选择所有项目并付出最大的痛苦 4。 

该跟踪显示了算法如何延迟对负面项目的承诺，并仅将它们评估为所需的锚点。

 ## 边缘情况

 当所有学习值都是负数时，算法仍然将每个项目评估为潜在的最大痛苦锚，但永远不会从中受益`pos_sum`。 最好的结果要么是不选择任何内容而变成零，要么是强制包含比其他内容更糟糕的内容。 

例如，对于输入 (−2, 1)、(−1, 3)，排序顺序给出候选值 -1 和 -3，均低于零，因此算法正确输出零。 

当多个项目共享相同的疼痛值时，在前缀更新之前，每个项目仍被评估为可能的锚点。 这确保了不同项目充当最大痛苦的子集不会被错误地合并，从而保留绑定约束的正确性。
