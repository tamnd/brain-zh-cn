 ## 解决方案
 ## 问题理解

 我们有三种类型的票：黄色、红色和蓝色。 这些门票可以按照严格的层次结构向上转换。 十张蓝色票可以兑换一张红色票，十张红色票可以兑换一张黄色票。 最后，十张黄票可以兑换一份奖励。 

该过程是完全确定性的：转换始终以 10 为基数单位保留值，但方向很重要，因为我们只能在层次结构中向上移动。 任务是计算如果我们从黄色、红色和蓝色门票的初始计数开始并应用任何转换序列，可以获得多少最终奖励。 

约束允许三个值中的每一个都大到 2 × 10^9，因此任何解决方案都必须在每个测试用例的恒定时间内运行。 在最坏的情况下，任何重复应用转换的模拟都太慢，因为简单的转换循环可能需要多达数十亿的步骤。 

天真的思维的一个微妙的失败案例是假设我们只在可能的情况下贪婪地朝一个方向进行转换而不考虑进位传播。 例如，从大量蓝色门票开始，粗心的实施可能会重复地将蓝色一次转换为红色，然后将红色转换为黄色，然后将黄色转换为奖励，这会在完成之前很长时间就超时。 

另一个概念上的陷阱是在将蓝色转换为红色或将红色转换为黄色后提前停止，而没有完全传播进位。 由于转换级联，部分标准化可能会低估黄票的最终数量，从而低估奖励的数量。 

## 方法

 暴力法是直接模拟交换规则。 我们反复将十张蓝色门票一组转换为红色门票，然后将十张红色门票一组转换为黄色门票，最后将十张黄色门票一组转换为奖励。 每次转换都会减少一个类别并增加另一个类别，我们会继续应用这些操作，直到无法再进行更改为止。 

这种方法是正确的，因为它准确地反映了规则。 然而，它的弱点在于每次转换仅在本地减少十倍的计数，并且在价值严重集中在蓝票中的病态情况下，我们可能需要在系统稳定之前执行非常大量的重复减少。 尽管每个单独的步骤都很简单，但步骤总数可能会与输入的大小成比例地增长，这使得对于高达 2 × 10^9 的值来说是不可行的。 

关键的观察结果是，该系统本质上是一个以 10 为基数的三位数位置数字系统。 蓝色票代表以10立方为基数的层次结构中的个位，红色代表十层，黄色代表百层。 每个完全标准化都对应于将整个状态转换为单个整数值，然后重复除以 10。 我们不需要一步步模拟转账，而是可以直接计算在结转所有蓝色和红色贡献后还存在多少张有效黄票。 

这将问题简化为简单的算术进位传播。

 ## 算法演练

 我们将蓝色票解释为最小单位，红色为10张蓝色票，黄色为10张红色票。 我们首先将所有内容转换为等量的黄票，然后计算这些黄票产生多少奖励。

1. 使用整数除以 10 将蓝票转换为红票。商贡献给红票，余数保留为剩余的蓝票。 其余部分与更高的转化无关，因为它无法形成完整的红票。 
2. 将蓝色转换后新创建的红色门票添加到现有的红色门票中。 
3. 使用整数除以 10 将全部红票转换为黄票。同样，剩余的红票不能形成完整的黄票，因此不再进一步贡献。 
4. 将红色转换后新创建的黄票添加到现有黄票中。 
5. 将黄票总数除以10，换算为奖励，直接给出答案。 

每个阶段都是以 10 为基数的系统中的进位运算，并且顺序很重要，因为较低级别的转换向上传播。 

它的工作原理基于一个不变量：在每一步之后，从较低级别到下一个级别的所有可能的转换都已被完全应用。 在处理蓝票结束时，没有任何 10 张蓝票组未使用； 处理完红票后，没有10张红票组未使用。 由于每次转换都会保留以 10 为基数的加权单位的总值，因此贪婪地折叠每个级别无法更改顶层的最终商。 该系统的行为类似于算术加法中的重复进位标准化。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

y = int(input())
r = int(input())
b = int(input())

# convert blue -> red
r += b // 10
b = b % 10

# convert red -> yellow
y += r // 10
r = r % 10

# convert yellow -> rewards
print(y // 10)
```该实现直接遵循进位传播结构。 每个转换步骤都使用整数除法和模数来将完整的十组与剩余部分分开。 顺序很重要：蓝色必须在红色处理为黄色之前处理为红色，否则我们将失去更深层次的贡献。 

一个常见的错误是尝试在一个表达式中折叠所有内容而不进行中间更新。 虽然可能，但在跟踪携带时很容易出错。 保持三个层次的明确可以确保正确性并使传播结构清晰。

import sys, io

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    import sys
    input = sys.stdin.readline

 ## 边缘情况

 一种边缘情况是所有值都为零。 该算法不执行任何转换，并且整数除法正确会产生零奖励。 国家始终保持稳定。 

另一种边缘情况是仅存在蓝色票据时，例如输入`0 0 100`。 蓝色转换为红色为`100 // 10 = 10`，然后红色转换为黄色`10 // 10 = 1`，最后不会产生奖励，因为黄色只有 1。算法正确地处理了这个问题，因为每个阶段都完全应用整数除法，而不需要迭代。 

第三种情况是当值略低于转换阈值时，例如`9 9 9`。 蓝色不能形成完整的红色票，因此将其作为剩余部分丢弃。 红色变成`9 + 0 = 9`，它也不会转换为黄色。 黄色仍然是9，没有达到奖励阈值，导致输出为零。
