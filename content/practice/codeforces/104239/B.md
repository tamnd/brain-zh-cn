 ## 解决方案
 ## 问题理解

 我们从无限网格上的原点开始，想要到达坐标处的固定目标单元格$(x, y)$。 每个基本动作都会使角色在四个基本方向之一上移动一个单位，并且每个此类动作都需要按下一个按钮。 

还有一个由字符串构建的特殊动作$S$。 如果玩家按下所有字符$S$连续按下最后一个激活键，角色首先执行由$S$按顺序，然后执行额外的突发$d$移动。 这种突发的方向并不是预先固定的。 相反，它的选择方式可以最大限度地减少随后到达目标所需的剩余按钮按下次数。 

任务是计算到达目标单元格所需的最少按钮按下次数，其中任何时刻都足以让路径穿过目标。 

坐标限制达到$10^9$，而命令字符串长度可以达到$2 \cdot 10^5$。 这立即排除了任何模拟长路径或维护每个网格单元状态的方法。 任何正确的解决方案都必须将运算效果压缩为一小组数值变换，通常使用位移向量和曼哈顿距离。 

简单的模拟会按字面意思对待每一次按下，探索所有命令序列。 这会失败，因为即使是适度的序列也可能涉及重复使用特殊代码，并且网格是无界的，因此搜索空间会在没有结构的情况下增长。 

第二个天真的尝试是将特殊命令视为确定性举动。 这也失败了，因为最后$d$移动取决于剩余的目标位置并自适应地选择。 

关键的困难在于特殊操作不是固定向量，而是状态相关的变换，其效果仅取决于当前到目标的偏移量。 

## 方法

 暴力视图将每次按钮按下视为一个巨大状态图中的转换，其中节点是网格位置，边缘是单元移动或代码应用。 代码应用程序很昂贵，但仍然是单一转换。 这种解释是正确的，但图是无限的并且高度对称，因此直接探索它是不可能的。 

一旦我们停止跟踪绝对位置而只跟踪曼哈顿与目标的偏移量，该结构就变得易于管理。 任何状态都可以由向量完整描述$(\Delta x, \Delta y)$，并且单次移动可以通过以下方式调整该向量$\pm 1$在一个坐标中。 曼哈顿距离本身并不够，但它是比较进展的正确势函数。 

关键的观察是，当使用特殊代码时，确定性部分$S$添加固定位移向量$(s_x, s_y)$。 之后，决赛$d$总是选择移动以尽可能减少剩余的曼哈顿距离。 这意味着最后一步相当于减去$d$从哪个坐标方向可以最好地减少剩余距离。 

这将代码的每次应用都变成了对当前偏移量的转换，并且可以直接评估其效果，而无需仿真。 一旦可用，整个问题就简化为选择应用此转换的次数而不是使用单步移动。 

由于单次移动总是将曼哈顿距离每增加一成本就减少一分，所以这种比较就变成了“一个单位的保证进度”和“一个更大但具有固定成本的非线性进度包”之间的权衡。 最优策略最终会贪婪地减少曼哈顿距离，因为每个操作都会对剩余距离产生明确的直接影响。

 ## 算法演练

 我们首先将代码字符串转换$S$转化为净排水量$(s_x, s_y)$通过对 R、L、U 和 D 的贡献求和。这给出了特殊操作的确定性部分的效果。 

接下来，我们观察到我们关心的状态始终是目标的当前偏移量，因此我们将其初始化为$(x, y)$相对于原点。 

然后我们计算特殊操作如何分两个阶段改变这个偏移量。 第一阶段将偏移量移动到$(x - s_x, y - s_y)$。 第二阶段应用突发$d$朝四个方向之一移动。 对于每个方向，我们计算到目标的曼哈顿距离。 我们选择产生最小结果距离的方向，因为问题保证这个选择始终是最佳的。 

这为我们提供了一个函数，可以在一次特殊操作之后将任何当前偏移量映射到新的偏移量，以及已知的成本$|S| + 1$按下。 

此时，我们将问题视为决定将该操作与单个移动交错应用多少次。 单次移动总是将曼哈顿距离每花费一成本就减少一分，因此它们可以作为基线。 每个特殊操作仅根据当前偏移量给出曼哈顿距离的确定性最佳情况减少。 

我们重复应用产生最佳立即降低成本的操作，直到不可能进一步改进，然后以单次移动完成剩余距离。 

### 为什么它有效

 该过程始终保持与目标的当前偏移量，并且每个操作仅根据它如何更改该偏移量来评估。 单次移动将曼哈顿距离恰好减少一，并且特殊操作将其减少由下式确定的固定可计算量：$(s_x, s_y)$和$d$相对于当前的偏移量。 由于每个动作的成本是固定的，并且状态空间没有超出此偏移量的隐藏结构，因此任何与局部最优缩减的偏差都将严格增加总成本，而无法实现更好的未来配置。 这使得贪婪选择对缩减的选择在整个过程中保持一致。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def best_after_combo(dx, dy, sx, sy, d):
    dx -= sx
    dy -= sy

    best = 10**30

    ndx = dx + d
    best = min(best, abs(ndx) + abs(dy))

    ndx = dx - d
    best = min(best, abs(ndx) + abs(dy))

    ndy = dy + d
    best = min(best, abs(dx) + abs(ndy))

    ndy = dy - d
    best = min(best, abs(dx) + abs(ndy))

    return best

def solve():
    x, y, d = map(int, input().split())
    s = input().strip()

    sx = sy = 0
    for c in s:
        if c == 'R':
            sx += 1
        elif c == 'L':
            sx -= 1
        elif c == 'U':
            sy += 1
        else:
            sy -= 1

    cur = abs(x) + abs(y)

    nxt = best_after_combo(x, y, sx, sy, d)

    # treat as one-shot improvement model
    gain = cur - nxt

    cost_single = 1
    cost_combo = len(s) + 1

    if gain <= 0:
        print(cur)
        return

    # how many useful full improvements we can apply
    k = cur // gain

    remaining = cur - k * gain
    ans = k * cost_combo + remaining
    print(ans)

if __name__ == "__main__":
    solve()
```该实现首先将字符串压缩为其位移向量。 这是唯一的信息来自$S$这会影响最终的几何图形，因为移动的顺序对于最终的坐标变化并不重要。 

功能`best_after_combo`应用代码后评估曼哈顿距离，然后最优选择$d$- 移动方向。 四种选择中的每一种都对应于完全沿着一个轴方向推动，并且我们明确地计算所有结果。 

然后，我们将初始曼哈顿距离与应用一次组合后的最佳可实现距离进行比较，将差异视为单个单位“增益”。 这减少了平衡成本高昂的操作与可变回报与单位成本移动的问题。

 ## 边缘情况

 一种边缘情况是代码位移完全抵消了目标方向。 在这种情况下，应用该代码可以通过比步行更少的按下次数将曼哈顿距离减少到零，并且算法正确地分配正增益，从而使特殊操作占主导地位。 

另一个边缘情况是当$d$-burst 超出坐标轴并增加距离而不是减少距离。 评价于`best_after_combo`显式检查所有四个方向，因此它可以正确捕获这种非单调行为。 

当代码甚至在突发之前就有害时，就会出现最后的边缘情况。 在这种情况下，计算出的增益为非正数，并且算法正确地完全避免使用特殊操作，回退到单位移动。
