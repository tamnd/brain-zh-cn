 ## 解决方案
 ## 问题理解

 我们得到了一组容器，每个容器都有固定的重量 2 和相关的成本。 每个容器在结构上并不唯一，但每个输入行描述一组相同的容器：一个计数$k_i$和成本$W_i$，意味着有$k_i$可用容器，每个成本$W_i$。 

在需求方面，我们接到了多个订单。 每个订单指定一个编号$t_j$和一个高度参数$h_j$。 每个订单都需要准确选择$h_j$容器，因为每个容器贡献重量 2，所需总重量为$2h_j$。 每个选择的集装箱都会贡献其成本，我们必须使用相同的全球可用集装箱池独立满足每个订单。 同一物理容器不能在多个订单之间重复使用，因此一旦一个容器类型被一个订单消耗，其容量就会在全局范围内减少。 

目标是最大限度地降低履行所有订单的总成本，或者如果可用容器无法满足所有所需的选择，则报告不可能性。 

如果我们重新解释它，结构就会变得更清晰：我们有一组项目，每个项目的选择单位权重为 1（因为每个容器恰好贡献一个单位的“2 权重”）和成本。 我们必须准确分配$h_j$每个订单的项目以及跨订单的分配必须是不相交的。 

这些限制非常重要。 集装箱组总数最多为10,000个，所有订单的总需求量为5,000个。 这种不对称至关重要：供应量大且结构合理，但需求相对较小。 任何尝试模拟每个容器分配的算法都会太慢，因为在最坏的情况下，简单的扩展可能会达到 10^7 项。 同时，任何将每个组视为有限容量资源的算法都表明采用背包式结构，但具有多种需求。 

当容量不足时，即使总计数看起来很大，也会出现边缘情况。 例如，如果所有集装箱都很便宜，但容量较小，并且单个订单需要大量$h$，即使我们可能会失败$\sum k_i$如果我们错误地处理分组约束就足够了。 另一种失败模式是忘记每个组的容量是有限的； 将每个组视为无限会导致不正确的可行性。 

当必须集体考虑订单时，会出现第二种微妙的边缘情况。 如果一个人贪婪地使用本地最便宜的容器来满足每个订单而不考虑未来的订单，我们可能会耗尽低成本组并迫使稍后使用更高成本的容器，从而增加总成本或导致失败。 

## 方法

 直接的暴力方法将尝试单独处理每个容器。 我们将每个组扩展为$k_i$项目，然后将它们分配给订单。 每个分配决策都涉及选择哪些项目按哪个顺序排列，这自然会导致多维分配或类似流程的公式。 即使对所有商品和所有订单进行简单的动态编程，也需要迭代多达 10,000 个组，并且每个订单可能需要 10,000 个需求单位，从而产生以下顺序的操作计数：$10^8$到$10^9$，在 1 秒限制下太慢了。 

关键的观察结果是所有容器的重量相同，仅成本和多样性不同。 这减少了选择总共的问题$H = \sum h_j$来自多组加权选择的项目，每个组的约束$i$最多贡献$k_i$选择。 

我们不再考虑单个订单，而是将所有需求聚合为一个全局需求：我们必须准确选择$H$集装箱总数。 订单是分开的这一事实仅对可行性分组有影响，但由于所有项目在重量上都无法区分并且在成本上独立，因此任何有效的分配$H$可以重新排列物品以满足任意订单。 

这就把问题转化为一个有容量的有界背包$H$。 各组$i$贡献高达$k_i$物品，每件都有成本$W_i$。 我们想要准确地选择$H$项目最小化总成本。 

有界背包的标准技巧是容量的二进制分解。 各组容量$k_i$被分成两个束的幂，这样我们就可以将问题视为 0/1 背包$O(n \log k_i)$项目。 自总计$H \le 5000$，超过这个容量的背包DP是可行的。 

我们维护一个 DP 数组，其中$dp[x]$是精确挑选的最低成本$x$容器。 我们初始化$dp[0] = 0$以及所有其他的无穷大。 对于每个分解的包，我们执行标准的 0/1 背包过渡$H$到捆绑尺寸。 

这是有效的，因为每个捆绑都强制我们不能超过可用供应的约束，同时允许在对数分解中有效地重用组。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力扩张|$O(\sum k_i \cdot m)$|$O(\sum k_i)$| 太慢了 |
 | 有界背包 DP |$O(n \log k_{\max} \cdot H)$|$O(H)$| 已接受 |

 ## 算法演练

 ## 算法演练

 1. 计算所需容器总数$H = \sum h_j$。 如果$H > \sum k_i$，立即返回-1，因为即使忽略成本我们也无法满足需求。 这是可行性的必要条件。 
2.创建一个大小为DP的数组$H+1$， 在哪里$dp[x]$存储准确挑选的最低成本$x$容器。 初始化$dp[0] = 0$和所有其他值作为无穷大，因为最初我们没有选择任何内容。 
3. 对于每个容器组$i$，分解其容量$k_i$化为二束的幂。 例如，如果$k_i = 13$，我们将其分为 1,2,4,6。每个捆绑包代表选择许多相同的项目，每个捆绑包的成本等于捆绑包大小乘以$W_i$。 

这种分解确保我们将有界选择转换为多个 0/1 选择，而不会丢失任何组合。 
4. 对于每个捆绑包$(s, cost)$，以相反的顺序更新 DP 数组$H$下降到$s$。 对于每个$x$，尝试拿起包裹并放松$dp[x]$使用$dp[x - s] + cost$。 

反向遍历保证每个包最多使用一次，保持 0/1 背包语义的正确性。 
5. 处理完所有组后，检查$dp[H]$。 如果仍然无穷大，则返回-1； 否则返回$dp[H]$。 

### 为什么它有效

 在处理过程中的任何时刻，$dp[x]$表示精确选择的最小成本$x$到目前为止，各组的集装箱均已处理完毕，未超出其容量。 二进制分解确保最多的每个有效子集$k_i$每组中的项目可以表示为束的总和，因此不排除任何可行的解决方案。 反向 DP 转换可确保不会重复使用任何捆绑包，从而保持容量限制的完整性。 由于每个有效的分配$H$items 恰好对应于一种可行的捆绑组合，DP 探索所有有效的配置并返回其中的最小成本。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

INF = 10**18

n = int(input())
groups = []
total_supply = 0

for _ in range(n):
    k, w = map(int, input().split())
    groups.append((k, w))
    total_supply += k

m = int(input())
h_list = []
H = 0
for _ in range(m):
    t, h = map(int, input().split())
    H += h

if H > total_supply:
    print(-1)
    sys.exit()

dp = [INF] * (H + 1)
dp[0] = 0

for k, w in groups:
    if k == 0:
        continue
    cnt = k
    base = 1
    while cnt > 0:
        take = min(base, cnt)
        cost = take * w
        cnt -= take
        base <<= 1

        for x in range(H, take - 1, -1):
            if dp[x - take] + cost < dp[x]:
                dp[x] = dp[x - take] + cost

if dp[H] >= INF:
    print(-1)
else:
    print(dp[H])
```该解决方案首先聚合可行性，然后应用经典的有界背包变换。 唯一的微妙之处是维护每个二进制块的反向 DP 更新，确保每个块被使用一次。 分解循环必须小心地减少剩余容量，否则最后一个块可能会超出原始界限。

 ## 边缘情况

 一种重要的边缘情况是，总供应量充足，但跨群体的分配阻碍了确切的满意度。 例如，如果我们需要 5 个容器，但有组 (k=3) 和 (k=3)，可行性就很好。 然而，如果我们错误地独立对待组，我们可能会过度分配一个组，并导致以后的转换失败。 DP 通过实施全球容量限制来避免这种情况。 

另一种情况是所有成本为零。 DP 仍应正确计算可行性，而不会溢出或不正确的无穷大处理。 由于转换使用加法，因此将无穷大初始化为大常数可确保零成本束正确传播而不会失真。 

最后一个边缘情况是$H = 0$。 由于不需要容器，该算法应立即返回 0。 DP 初始化已经处理了这个问题，因为$dp[0]$设置为零并且不需要任何转换。
