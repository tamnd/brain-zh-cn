 ## 解决方案
 ## 问题理解

 我们正在使用一棵有根树，其中每个顶点都带有颜色标签。 根固定在顶点 1。在这棵树上，随着时间的推移，我们必须支持两种操作：对单个顶点重新着色，以及查询特定子树内存在给定颜色的顶点数。 

这里的子树是指到根的路径经过给定顶点的所有顶点。 因此，每个查询本质上都是在问：如果我们查看节点的连接“后代区域”，当前有多少个节点具有颜色 x？ 

输入是动态的，因为颜色在查询之间会发生变化。 这立即排除了每个查询从头开始重新计算子树统计信息的任何解决方案，因为更新会破坏任何预先计算的静态答案。 

约束总体上很严格：在所有测试用例中，顶点和查询的总数最多为 2×10^5。 这意味着任何解决方案的每次操作都应该表现接近线性或对数线性。 任何类似于每个查询重新计算 DFS 结果或直接扫描子树的操作都会超出限制。 

当许多节点共享相同的颜色并在密集区域中重复更新翻转颜色时，会出现微妙的边缘情况。 例如，考虑一条由 200000 个节点组成的链，最初颜色均为 1。如果我们天真地查询每个节点的子树计数，则每个查询都会退化为扫描 O(n)，这将是灾难性的。 

另一个微妙的失败案例是忘记子树成员资格是结构性的，而不是基于值的。 具有相同颜色的两个节点不可互换，除非它们位于同一子树内； 混合这些概念会导致计数不正确。 

## 方法

 回答查询的直接方法是使用 DFS 或 BFS 遍历给定节点的子树，并计算有多少节点与查询的颜色匹配。 这是正确的，因为它明确地检查了所需的顶点。 然而，每个查询可能会触及 O(n) 个节点，并且最多 2×10^5 个查询，这会导致最坏情况下的 O(nq) 行为，这远远超出了可行的限制。 

关键的观察结果是，如果我们使用欧拉遍历将树展平为数组，子树查询就会变得更加简单。 每个子树对应于该遍历顺序中的一个连续段。 这将问题转化为维护动态数组，我们需要在一定范围内支持点更新和频率查询。 

此时，我们需要一个数据结构来回答：一个值在一个段中出现了多少次，并且更新改变了单个位置的值。 一个自然的结构是从颜色到排序集的映射或每种颜色的 Fenwick 树，但维护每种颜色的 Fenwick 树在内存中成本太高。 

相反，我们扭转了观点：我们不是跟踪每个颜色的位置，而是使用可以有效聚合计数的结构来跟踪每个位置的颜色。 标准解决方案是为每种颜色维护一组有序的欧拉位置。 然后，每个查询减少为计算该颜色有多少个位置位于给定区间内，这可以使用二分搜索来回答。 

更新变成从一组中删除并插入另一组中。 两个操作都是 O(log n)，查询也是 O(log n)。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 每个查询的强力 DFS | O(nq) | O(n) | 太慢了 |
 | 欧拉之旅 + 按颜色排序的套装 | O((n+q) log n) | O((n+q) log n) | O(n) | 已接受 |

 ## 算法演练

1. 为树建立一个邻接表，并以节点 1 为根。我们需要一个确定性的遍历顺序，以便每个子树都成为一个连续的区间。 
2. 运行 DFS 为每个节点分配一个进入时间tin[v] 和一个可选的子树范围[tin[v], tout[v]]。 DFS 顺序确保 v 的所有后代都位于连续段中。 这就是用数组结构代替树结构的核心变换。 
3. 维护一个字典，将每种颜色映射到该颜色当前出现的欧拉位置的排序容器。 首先，将tin[v]插入到c[v]对应的集合中。 
4. 对于类型 2 查询 (v, x)，将 v 的子树解释为区间 [tin[v], tout[v]]。 答案是颜色 x 集中落在该区间内的元素数量。 这是使用二分搜索计算的：upper_bound(tout[v]) 减去 lower_bound(tin[v])。 
5. 对于类型 1 更新 (v, x)，我们必须反映颜色变化。 从旧颜色集中删除tin[v]并将其插入新颜色集中。 然后更新v的存储颜色。 

正确性取决于始终保持集合与当前着色同步。 

### 为什么它有效

 DFS 排序保证 subtree(v) 精确对应于欧拉时间中的连续间隔。 每个节点在此顺序中只出现一次，因此通过索引存储节点相当于将它们存储在树的线性表示中。 因此，每个颜色集代表该颜色在数组中出现的确切位置集，并且每个子树查询都成为该数组上的范围计数问题。 由于更新仅在集合之间移动单个元素，因此结构始终保持一致。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    n, q = map(int, input().split())
    c = [0] + list(map(int, input().split()))

    g = [[] for _ in range(n + 1)]
    for _ in range(n - 1):
        u, v = map(int, input().split())
        g[u].append(v)
        g[v].append(u)

    tin = [0] * (n + 1)
    tout = [0] * (n + 1)
    timer = 0

    stack = [(1, 0, 0)]  # node, parent, state (0 enter, 1 exit)
    while stack:
        v, p, state = stack.pop()
        if state == 0:
            timer += 1
            tin[v] = timer
            stack.append((v, p, 1))
            for to in g[v]:
                if to != p:
                    stack.append((to, v, 0))
        else:
            tout[v] = timer

    from collections import defaultdict
    import bisect

    pos = defaultdict(list)

    for i in range(1, n + 1):
        pos[c[i]].append(tin[i])

    for k in pos:
        pos[k].sort()

    def count(color, l, r):
        arr = pos.get(color, [])
        return bisect.bisect_right(arr, r) - bisect.bisect_left(arr, l)

    for _ in range(q):
        t, v, x = map(int, input().split())
        if t == 1:
            old = c[v]
            if old == x:
                continue
            old_arr = pos[old]
            idx = bisect.bisect_left(old_arr, tin[v])
            if idx < len(old_arr) and old_arr[idx] == tin[v]:
                old_arr.pop(idx)

            bisect.insort(pos[x], tin[v])
            c[v] = x
        else:
            print(count(x, tin[v], tout[v]))

def main():
    t = int(input())
    for _ in range(t):
        solve()

if __name__ == "__main__":
    main()
```DFS 到 Euler 的转换是迭代实现的，以避免递归深度问题。 每个节点接收唯一的进入时间，并且从这些时间导出子树边界。 

颜色到位置的映射是使用保持排序的列表来实现的。 查询使用二分搜索来有效地计算范围计数。 

更新步骤会在插入新位置之前小心地删除旧位置，以确保一致性。 一个常见的错误是忘记更新颜色数组`c[v]`，这将使未来的更新不同步。

 | 测量 | 复杂性 | 说明|

 ## 边缘情况

 当更新分配与节点已有的颜色相同的颜色时，就会出现一种微妙的情况。 在这种情况下，如果不加保护，移除和重新插入会破坏结构。 当新旧颜色匹配时，该实现会显式跳过工作，从而保持设置的完整性。 

另一种情况是深链，其中子树范围是大的连续段。 如果没有欧拉扁平化，这些查询将退化为完整扫描； 在这里，它们减少为简单的间隔计数，因此即使有 200000 个节点的路径仍然高效。 

最后，同一节点上的重复更新强调了删除逻辑的正确性。 由于位置在每个颜色列表中唯一存储，因此未能删除确切的锡索引会累积重复项并导致答案膨胀。 二分搜索删除确保列表始终反映树的真实当前状态。
