 ## 解决方案
 ## 问题理解

 我们有几组朋友，我们想为每个组选择一家咖啡馆，让该组中尽可能多的人感到高兴。 

每个人都有一个由一小组成分描述的偏好，这些成分被编码为一个短字符串。 汉堡包也由一组成分来描述。 如果一个人喜欢的所有成分都出现在汉堡中，那么他就会对汉堡感到满意。 如果一家咖啡馆至少有一个汉堡包能让一个人满意，那么这家咖啡馆就被认为对一个人来说是好的。 

对于每个组，任务是选择一家咖啡馆，使该组中满意的人数最大化。 如果多个咖啡馆达到相同的最大值，我们会选择索引最小的一家。 

这里的关键结构是每个偏好和每个汉堡最多使用 6 个不同的字母，因此即使字母表有 26 个字母，每组也非常小。 这立即表明子集和位掩码推理将是高效的，因为 6 元素集的子集总数仅为 64。 

主要困难在于，我们将每个组与每个咖啡馆进行比较，两者都可以达到 1000，而人和汉堡的总数可以达到 50000。每个咖啡馆内的人和汉堡之间的天真全对匹配太慢了。 

当咖啡馆里的多个汉堡部分符合一个人的偏好，但没有一个完全包含它时，就会出现微妙的边缘情况。 例如，如果一个人喜欢`{a, b}`咖啡馆有汉堡`{a}`和`{b}`，人不满意。 只有完全遏制才重要，而不是部分重叠。 

另一个特殊情况是，咖啡馆没有可以满足团体中任何人需求的汉堡。 如果所有咖啡馆的表现都同样糟糕，那么仍然需要考虑并选择该咖啡馆，因为我们仍然选择平局中最小的指数。 

## 方法

 解决问题的直接方法是从字面上模拟一切。 对于每个组和每个咖啡馆，我们迭代每个人，然后迭代该咖啡馆中的每个汉堡，检查汉堡是否包含该人所需的所有成分。 这是正确的，因为它准确地反映了定义。 然而，复杂性成为团体、咖啡馆、人和汉堡的产物，考虑到限制，复杂性太大了。 

关键的观察结果是，每个汉堡和每个偏好都是最多 6 个不同字母的集合。 我们可以预先计算相反的方向，而不是重复检查子集关系：对于给定的汉堡，我们可以枚举其成分集的所有可能的子集。 每个这样的子集都对应于一个对该汉堡感到满意的人。 由于汉堡最多有 6 种成分，因此此扩展最多创建 64 个子集。 

这将每个咖啡馆转变为它可以满足的所有偏好集的预先计算集。 一旦我们有了这个，检查一个人就变成了一个简单的会员资格测试。 然后，我们通过扫描所有人并计算咖啡馆的预先计算集中包含多少个他们的偏好蒙版来评估每个咖啡馆。 

改进来自于将重复的子集检查替换为每个汉堡的小型恒定扩展以及随后的快速哈希查找。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力人汉堡匹配 | O(G·C·P·B) | O(G·C·P·B) | O(1) | O(1) | 太慢了 |
 | 每个汉堡的子集扩展 + 会员计数 | O(汉堡总数·2^6 + 总人数·咖啡馆) | O(总展开状态) | 已接受 |

 ## 算法演练

 1. 将每个成分字符串转换为超过 26 个字母的位掩码。 每个字符对应一个位位置。 这允许子集检查变成位操作。 
2. 为每个咖啡馆制作一套汉堡的所有成分面具，可以间接满足某人。 对于每个汉堡面具，生成该面具的所有子集并将它们插入咖啡馆的集合中。 这是有效的，因为任何对该汉堡感到满意的人都必须具有等于这些子集之一的偏好。 
3. 经过预处理后，每个咖啡馆现在都包含了它可以通过至少一个汉堡满足的所有偏好蒙版的完整表示。 
4. 对于每个组，计算其人员中相同偏好面具的频率。 这避免了多次重复扫描相同的首选项。 
5. 对于每个咖啡馆，通过迭代组中所有不同的偏好掩码并检查该掩码是否存在于咖啡馆的预先计算集中，计算组中有多少人感到满意。 乘以频率。 
6. 跟踪每组得分最高的咖啡馆，通过选择最小的指数来打破平局。 

### 为什么它有效

 当且仅当存在至少一个其成分集是他们偏好集的超集的汉堡时，每个人都会感到满意。 枚举每个汉堡的所有子集可以准确捕获该汉堡可以满足的所有可能的偏好集。 将咖啡馆中所有汉堡的并集保留了这种等价性，因此最终集合中的成员资格在逻辑上与原始满足条件相同。 由于每个组的分数只是独立人员的总和，因此通过频率进行计数可以保持正确性，而不会改变底层的组合结构。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def mask_of(s):
    m = 0
    for ch in s.strip():
        m |= 1 << (ord(ch) - 97)
    return m

def all_submasks(mask):
    # generate all subsets of up to 26-bit mask, but actual size <= 6 bits
    bits = []
    for i in range(26):
        if mask & (1 << i):
            bits.append(i)
    res = []
    k = len(bits)
    for sub in range(1 << k):
        m = 0
        for j in range(k):
            if sub & (1 << j):
                m |= 1 << bits[j]
        res.append(m)
    return res

n = int(input())
groups = []

for _ in range(n):
    k = int(input())
    freq = {}
    for _ in range(k):
        m = mask_of(input())
        freq[m] = freq.get(m, 0) + 1
    groups.append((k, freq))

m = int(input())
cafes = []

for _ in range(m):
    l = int(input())
    s = set()
    for _ in range(l):
        bm = mask_of(input())
        for sm in all_submasks(bm):
            s.add(sm)
    cafes.append(s)

for _, freq in groups:
    best_cafe = 1
    best_score = -1

    for idx, cafe_set in enumerate(cafes, start=1):
        score = 0
        for pm, cnt in freq.items():
            if pm in cafe_set:
                score += cnt

        if score > best_score or (score == best_score and idx < best_cafe):
            best_score = score
            best_cafe = idx

    print(best_cafe)
```该实现首先将每个成分字符串编码为位掩码。 这使得子集关系变得便宜并且完全避免了字符串操作。 用于生成子掩码的辅助函数枚举汉堡活动位的所有子集，这是安全的，因为每个字符串的不同成分的数量最多为 6。 

每个咖啡馆都会构建一个哈希集，其中包含它可以满足的所有偏好掩码。 该集合是关键的加速结构，用 O(1) 成员资格测试代替重复的子集检查。 

对于每个组，我们使用频率图压缩相同的偏好。 这很重要，因为组可以包含重复的首选项字符串，单独处理它们会不必要地重复相同的工作。 最后的评分步骤只是检查每个咖啡馆集合中每个独特面具的成员资格并乘以其频率。 

在扫描过程中通过跟踪获得最高分数的最小索引来处理平局决胜。 

## 工作示例

 考虑一个由一组和两家咖啡馆组成的简化情况。 

团体偏好是`{a}`,`{ab}`,`{b}`。 1号咖啡厅有汉堡`{ab}`和`{c}`。 2号咖啡厅有汉堡`{b}`和`{ac}`。 

对于 Cafe 1，子集展开为`{ab}`产生`{}`,`{a}`,`{b}`,`{ab}`。 咖啡厅1可以满足`{a}`和`{ab}`但不是`{b}`。 所以分数是2。 

对于 Cafe 2，扩展为`{b}`产生`{}`,`{b}`。 扩展`{ac}`产生`{}`,`{a}`,`{c}`,`{ac}`。 所以Cafe 2可以满足`{a}`和`{b}`得分为 3。 

| 咖啡馆|`{a}`|`{ab}`|`{b}`| 分数 |
 | --- | --- | --- | --- | --- |
 | 1 | 是的 | 是的 | 没有| 2 |
 | 2 | 是的 | 没有| 是的 | 2 |

 该跟踪表明，正确性取决于完整的子集包含而不是部分重叠，并且预先计算的闭包正确地捕获了满意度。 

现在考虑一个所有人都有相同偏好的群体`{a}`咖啡馆有汉堡`{b}`,`{c}`仅有的。 它的子集闭包不包含`{a}`，所以分数保持为零，这与定义一致。 

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 | O(汉堡总数·2^k + m·总人数) | 每个汉堡最多扩展为 64 个子集，每个咖啡馆通过扫描一次群体偏好来评估 |
 | 空间| O(m·64 + 总不同偏好) | 每个咖啡馆都存储一组可达子集 |

 该界限完全符合限制，因为 50,000 个汉堡总共扩展到约 320 万个子集插入，并且每次查找都是恒定时间的。 最终评估最多执行 5000 万次哈希检查，这在 Python 中是可以接受的。

 ## 边缘情况

 任何咖啡馆中没有汉堡包含给定成分集的情况仍然会产生有效答案，因为所有咖啡馆得分为零并且选择了最小索引。 例如，如果所有人都喜欢`{a}`但所有汉堡只包含`{b}`，每个咖啡馆都有一个空的满意度集，答案是咖啡馆 1。 

另一种情况是汉堡恰好有六种成分。 即使这样，子集扩展仍然限制在 64 个元素，因此预处理成本不会爆炸。 该算法将其视为较小的汉堡，以确保在极端情况下的一致行为。 

组内重复的偏好不会影响正确性，因为它们在评分之前会聚合到频率图中。 这可以防止冗余迭代导致的重复计算。
