 ## 解决方案
 ## 问题理解

 我们有几种类型的建筑部件。 每种类型都有固定的质量和有限的供应。 我们想要用这些零件组装几个相同的套件。 “相同”意味着每个套件使用每种类型的完全相同数量的零件。 如果一个套件使用 2 个螺钉和 1 个横梁，则所有其他套件必须使用完全相同的 2 个螺钉和 1 个横梁。 

每个套件都有总质量约束：所选零件的质量总和必须位于给定区间内，从最小允许质量到最大允许质量。 

任务是确定我们可以构建多少个相同的套件，同时遵守这两个约束：我们不能超过任何零件类型的可用供应，并且每个套件必须满足质量间隔。 

输入范围的类型数量较小，但每种类型的项目数量可能很大。 这立即排除了任何模拟一次分配一个套件的零件或直接枚举所有可能的套件组成的方法。 质量限制相对较小，这表明如果我们可以控制多重性，则子集和式推理是可能的。 

一个微妙的困难是套件的数量和套件的组成是耦合的。 选择更丰富的套件会减少我们可以构建的副本数量，而选择最小的套件可能会允许许多副本，但无法满足最低质量要求。 

当人们试图贪婪地挑选零件直到质量进入允许的范围时，就会出现天真的错误。 但这失败了，因为必须在所有套件中重复使用相同的贪婪组合，因此局部决策会影响全局可行性。 

作为一个具体的失败示例，假设我们有两种类型：一种轻但丰富的部分，一种重但稀缺的部分。 贪婪地添加重型零件可能会成为一个有效的套件，但供应太快，导致副本数量减少过多。 正确的答案取决于平衡构图和重复，而不仅仅是形成一套有效的套件。 

另一种极端情况是没有任何单一组合达到最小质量。 在这种情况下，即使供应充足，答案也是零。 

## 方法

 如果我们忽略所有套件必须相同的要求，那么问题就变成了有界背包变体：我们尝试看看是否可以形成总质量位于允许区间内的有效零件子集。 但在这里，问题在于我们必须多次重复同一个子集。 

直接的暴力方法将尝试描述单个套件的所有可能的向量。 对于每个候选组合物，我们将计算其总质量，然后根据最严格的零件类型计算我们可以负担多少个副本。 如果有 n 种类型，并且每种类型最多可以使用 ci 次，那么即使将 xi 限制为 [0, ci] 也会导致巨大的搜索空间，大致以 n 为指数。 这变得完全不可行。 

关键的观察是将两个决策分开。 首先我们确定要构建多少个套件 k。 一旦 k 确定，每个零件类型 i 最多可以在单个套件中使用楼层 (ci / k) 次。 这将问题转变为检查是否存在总重量位于该区间内的物品的有界组合。 可行性检查成为经典的有界背包可达性问题。 

由于 mmax 至多为 10^4，因此我们可以在可实现的总和上提供动态规划解决方案。 唯一剩下的挑战是每个项目计数的界限可能很大，因此我们使用二进制分割来压缩它们。 每个容量为 U 的类型被拆分为 O(log U) 个伪项，从而将有界背包简化为标准的 0/1 背包。 

最后，我们二分搜索 k，因为可行性是单调的：如果我们可以构建 k 个套件，那么我们也可以构建任何更小的数字。

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力破解构图| 指数| O(1) | O(1) | 太慢了 |
 | 二分查找+有界背包DP| O(n * mmax * log ci * log 答案) | O(mmax) | 已接受 |

 ## 算法演练

 我们首先猜测我们想要构建多少相同的套件，并验证该猜测是否可能。 

1.确定套件的候选数量k。 对于每个部件类型 i，计算一个套件中可以使用该部件的多少个副本，即 ci // k。 这强制 k 个相同的套件不会超过可用库存。 此步骤将全局重复约束转换为本地每个套件的限制。 
2. 使用二元分解将每个有界项（具有极限 Ui = ci // k 的 mi）转换为多个 0/1 项。 例如，限制 13 变为 1、2、4、6。其中每一个都表示在套件中获取相同零件类型的多个副本。 
3. 在可达到的质量上运行背包 DP，最高可达 mmax。 我们维护一个布尔数组 dp，其中 dp[s] 意味着我们可以使用分解的物品形成总质量恰好为 s 的套件。 
4. 处理完所有项后，检查 [mmin, mmax] 中的 s 是否有任何 dp[s] 为 true。 如果是，则这个k可行，否则不可行。 
5. 从 0 到安全上限的二分搜索 k，通常是零件总数除以每个套件的最低可能要求。 最大的可行 k 就是答案。 

### 为什么它有效

 对于固定的 k，每个有效的解决方案都完全对应于选择尊重每种类型限制 ci // k 的项目多重集。 二进制分解保留了有限使用的所有有效组合。 DP 枚举了这些约束下所有可达到的总和。 由于每个有效套件都必须具有所需间隔内的总质量，因此检查该间隔内的可达性可以准确捕获可行性。 k 的单调性确保二分搜索找到最大可能数量的相同套件。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def can_make(k, n, mmin, mmax, m):
    # build bounded knapsack using binary splitting
    dp = [False] * (mmax + 1)
    dp[0] = True

    for mi, ci in m:
        if k == 0:
            u = ci
        else:
            u = ci // k
        if u == 0:
            continue

        cnt = 1
        while u > 0:
            take = min(cnt, u)
            weight = mi * take

            # 0/1 knapsack transition
            for s in range(mmax, weight - 1, -1):
                if dp[s - weight]:
                    dp[s] = True

            u -= take
            cnt <<= 1

    for s in range(mmin, mmax + 1):
        if dp[s]:
            return True
    return False

def solve():
    n, mmin, mmax = map(int, input().split())
    m = [tuple(map(int, input().split())) for _ in range(n)]

    # binary search answer
    lo, hi = 0, 10**6 + 5

    while lo < hi:
        mid = (lo + hi + 1) // 2
        if can_make(mid, n, mmin, mmax, m):
            lo = mid
        else:
            hi = mid - 1

    print(lo)

if __name__ == "__main__":
    solve()
```该代码首先定义了对固定数量的套件的可行性检查。 在此检查中，每个项目类型都会转换为多个有界块，以便背包成为这些块的标准子集和。 DP 数组的大小最大为 mmax，因为任何高于该值的总和都是无关紧要的。 

二分查找封装了这个检查。 如果 k 个套件都是可能的，那么所有较小的值也是可能的，这保证了单调搜索的正确性。 

一个微妙的实现细节是在添加每个块时反向迭代背包。 这可以防止多次重复使用同一块，从而在分解后保留 0/1 性质。

3 12 13

 | 1 | (3,8) | 8//k| 更新可达到的金额| 部分 |

 对于 k = 4，每个套件最多可以使用 (2,1,2) 个，大致取决于划分。 DP 最终通过 2×3 + 1×7 达到总和 13。 由于13在[12,13]内部，k是可行的。 

这证实了可行性取决于组合和有限重复。 

### 示例 2

 输入：```
2 5 6
2 3
4 1
```尝试 k = 1 允许所有项目。 DP 可以形成和 2、4、6 等，因此 [5,6] 中存在有效和。 因此 k ≥ 1 有效。 

尝试 k = 2 会减少容量：第二种类型变为 0，第一种类型变为 1。只有和 0 或 2 是可能的，因此区间是不可到达的。 这表明增加 k 如何通过收紧界限来破坏可行性。 

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 | O(log K * n * mmax * log ci) | 对 k 进行二分搜索，每次检查都是有二分分裂的有界背包 |
 | 空间| O(mmax) | DP 阵列可达到最大允许质量 |

 这些约束允许 mmax 最大为 10^4，n 最大为 50，这使得 DP 易于管理。 二进制拆分可确保即使较大的 ci 值也不会爆炸成每种类型的线性复杂度。

 ## 边缘情况

 当 mmin 非常小但 mmax 很紧时，就会出现极端情况。 在这种情况下，即使许多和可以达到，DP 仍然必须将自己限制在上限，否则它可能会错误地接受有效范围之外的和。 

当 k 变得足够大以致所有类型的 ci // k 都为零时，就会出现另一种边缘情况。 在这种情况下，只有 k = 0 或 k = 1（取决于解释）仍然可行。 该算法自然地处理这个问题，因为 DP 不接收任何项目，只检查间隔中是否存在零质量。 

第三种情况是单一类型主导解决方案。 例如，如果一个项目的质量恰好在该区间内，则 DP 必须正确地允许选择多个副本，最多可达每 k 的有界限制，而不混合其他类型。 二进制拆分可确保正确处理这些重复的贡献，而不会过度计数。
