 ## 解决方案
 ## 问题理解

 给定两个长度相等的整数数组，并且允许我们应用一种操作：选择两个位置并同时交换两个数组中这些位置的元素。 换句话说，我们不是在每个数组内独立交换值，而是交换整个对，因此每个索引的行为就像一张持有一对的“卡”$(a_i, b_i)$，我们可以排列这些卡。 

在执行最多有限次数的交换后，目标是对这些对重新排序，以便第一个数组从左到右不递减，第二个数组也从左到右不递减。 

因此，被重新排列的真实对象不是两个单独的数组，而是一系列 2D 点。 每个操作都会对这些点进行排列，并且我们需要两个坐标同时排序的排列。 

这些约束意味着每个测试的实例大小较小，通常允许$O(n^2)$甚至$O(n^3)$建筑。 这立即排除了尝试所有索引排列之类的事情，因为它是阶乘增长的。 它还表明需要一个建设性的排序过程，可能会明确地产生排列，而不仅仅是检查可行性。 

当两个数组在排序上“不一致”时，就会出现微妙的边缘情况。 例如，如果一个索引的值较小$a$但很大$b$，另一个有一个大$a$但很小$b$，那么没有任何排序可以正确放置两者。 

示例：

 输入：```
a = [1, 2]
b = [2, 1]
```在这里，将索引 1 放在索引 2 之前$a$排序但中断$b$，同时逆向修复$b$但打破了$a$。 所以答案是不可能的。 

这种冲突是问题的关键结构性障碍。 

## 方法

 我们可以将每个索引视为一个点$(a_i, b_i)$。 由于我们可以使用交换任意排列索引，因此任务变为：我们能否将这些点排列在单个序列中，使得两个坐标不递减？ 

暴力方法将尝试索引的所有排列并检查两个数组是否已排序。 这是正确的，但即使对于温和的人来说也是不可行的$n$，因为它是$O(n!)$，甚至与$n = 10$，这变得太大了。 

一种更结构化的尝试是按一个坐标排序，例如$a$，并希望$b$也会自动排序。 这有时有效，但当两个点具有交叉顺序时会失败：其中一个点的顺序较小$a$但更大$b$，另一个有更大的$a$但较小$b$。 在这种情况下，排序依据$a$迫使增加$a$但会减少$b$，违反要求。 

关键的观察是我们需要在两个维度上保持一致的单一排序。 这意味着这些点不得包含任何跨坐标的“反转”。 如果对于某对$i, j$，我们有$a_i < a_j$但$b_i > b_j$，那么这两个元素不能在任何有效的最终序列中一致地排序。 

因此可行性的条件变为：当我们对索引进行排序时$a$，序列$b$值必须已经是非递减的，并且关系也必须是一致的，这样它们就不会引入歧义。 同样，排序依据$a$并排序$b$必须导致元素具有相同的相对顺序。 

一旦建立了有效的目标顺序，构建答案就很简单：我们模拟将元素交换到其目标位置，类似于选择排序，记录每个交换操作。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 蛮力排列 | O(n!) | O(n) | 太慢了|
 | 排序+可行性检查+建设性交换| O(n^2) | O(n^2) | O(n) | 已接受 |

 ## 算法演练

 ### 最佳构造

 1.对待各个指标$i$作为一对$(a_i, b_i)$，并将其与原来的位置关联起来。 
2. 按对对索引进行排序$(a_i, b_i)$按字典顺序。 这给出了候选的最终排列，其中如果存在这样的排列，则两个坐标都是非递减的。 
3. 通过检查两个序列是否按照此排序顺序来验证可行性$a$和$b$是非递减的。 如果任一数组中出现任何减少，则得出结论：不存在有效排列。 
4. 如果可行，我们现在知道索引的目标顺序。 构建一个数组`pos`它存储工作排列中每个索引的当前位置。 
5. 从左到右迭代位置。 对于每个位置$i$，根据目标顺序找到应该放在那里的索引。 
6. 如果该索引尚未位于位置$i$，使用一系列交换将其交换到位。 每个交换交换两个头寸并更新其记录的头寸。 记录每次操作。 
7. 继续，直到所有位置都符合目标顺序。 由此产生的交换序列就是答案。 

### 为什么它有效

 正确性来自于将问题简化为构建尊重全局全序的排列。 一旦验证了按对排序顺序以保持两个坐标不减少，任何偏离它都必然会引入反转$a$或在$b$。 构造阶段只是使用允许的交换实现此排列的受控方式，并且由于交换可以实现任何排列，因此唯一真正的约束是目标顺序是否有效。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        n = int(input())
        a = list(map(int, input().split()))
        b = list(map(int, input().split()))

        idx = list(range(n))
        idx.sort(key=lambda i: (a[i], b[i]))

        # check feasibility
        ok = True
        for i in range(n - 1):
            if a[idx[i]] > a[idx[i + 1]] or b[idx[i]] > b[idx[i + 1]]:
                ok = False
                break

        if not ok:
            print(-1)
            continue

        # build current permutation
        pos = list(range(n))
        where = list(range(n))
        target = idx[:]

        ops = []

        for i in range(n):
            if where[i] == target[i]:
                continue
            j = pos[target[i]]

            # swap i and j
            x, y = where[i], where[j]
            where[i], where[j] = where[j], where[i]
            pos[x], pos[y] = pos[y], pos[x]

            ops.append((i + 1, j + 1))

        print(len(ops))
        for i, j in ops:
            print(i, j)

if __name__ == "__main__":
    solve()
```该代码首先使用对的字典排序来构造索引的候选排序。 然后，它验证此排序不会违反任一数组中的单调性。 如果是这样，我们立即返回不可能。 

第二阶段是直接排列构造。 数组`pos`和`where`跟踪每个索引的当前位置以及哪个索引位于每个位置。 每个交换操作都被转换为允许的移动格式并记录。 

这里的一个常见实施陷阱是在每次交换后无法一致地更新两个跟踪阵列。 如果其中之一`pos`或者`where`变得不一致，以后的交换将引用错误的位置并破坏结构。

 ## 边缘情况

 关键的边缘情况是一个坐标中的值相等但另一个坐标中的值不相等。 例如：```
a = [1, 1, 2]
b = [3, 2, 1]
```在这里，联系$a$迫使我们完全依赖$b$， 但$b$在连接块内严格递减。 可行性检查发现了这一点，因为即使$a$是非减的，$b$排序后违反单调性。 

另一种边缘情况是存在多个有效排列时。 该算法不会尝试最小化交换，它只保证正确性，因此任何一致的交换序列都是可以接受的。
