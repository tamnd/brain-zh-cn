 ## 解决方案
 ## 问题理解

 我们给定了三个字符串，代表三个数字，但它们并不是通常意义上的固定数字。 相反，每个字符串的行为可能类似于数字上的模式。 目标是确定我们是否可以将数字分配给字母，以便前两个字符串成为有效数字 A 和 B，第三个字符串成为它们的总和 C，并且分配在同一字母的所有出现中保持一致。 

核心约束是字母和数字之间的双射。 每个不同的字母必须恰好映射到一位数字，并且每个数字只能由一个字母使用。 一旦在三个字符串中的任何一个中为字母分配了数字，该分配就必须在任何地方都成立。 同时，所得数值必须满足算术关系 A + B = C。 

输出本质上是对所有可能的一致分配的可行性检查：是否存在至少一个从字母到数字的映射，使得当字符串被解释为数字时方程有效。 

这些约束在两个方面隐含着重要意义。 首先，在此版本的问题中，每个字符串的长度非常小，大约为 3 个字符，这迫使 A 和 B 的数值最多为 999。这立即限制了实际数值的搜索空间。 其次，因为我们正在处理双射约束，所以任何构建映射的尝试都必须在全局范围内进行验证，而不是在每个数字位置进行局部验证。 

一个天真的但危险的边缘情况是，当相同的字母出现在不同的位置时，会迫使数字角色相互矛盾。 例如，如果一个字母同时出现在 A 和 C 中，则它必须与算术结果得出的数字匹配。 仅以数字方式检查 A + B = C 而不验证字母映射的一致性的粗心方法将错误地接受无效情况。 

另一个微妙的情况是由前导零引起的。 如果映射在多字符数字的最高有效位置将字母分配为零，则通常是无效的。 然而，由于数字构造是直接通过迭代与字符串长度一致的有效整数值得出的，因此如果我们只考虑 A、B 和 C 的正确整数表示，则隐式避免前导零。 

## 方法

 蛮力的观点很简单，因为数字界限非常小。 由于 A 和 B 最多为 999，因此我们可以迭代所有可能的对 (A, B)。 对于每一对，我们计算 C = A + B，然后尝试检查我们是否可以一致地将字母分配给数字。 

这种方法的正确性来自于彻底覆盖可能的数字分配的整个空间。 对于每个候选三元组（A，B，C），我们验证数字模式是否可以在双射约束下与给定的字符串匹配。 此验证是通过并行扫描所有三个字符串并维护两个字典来完成的：一个从字母到数字，另一个从数字到字母。 如果出现任何矛盾，则该作业无效。 

瓶颈是候选对的数量。 即使使用小字符串，朴素循环也会运行高达 1000 × 1000 种可能性，并且每次检查最多扫描几个数字，使其处于临界状态，但在严格的约束下仍然可以接受。 更深层次的问题是，如果数值范围增加，这种方法就无法扩展。 

关键的观察结果是数字空间已经是完全有界和离散的。 我们不是直接搜索任意映射；而是直接搜索任意映射。 相反，我们正在搜索实际的整数三元组（A、B、C）。 这将问题从组合分配问题转换为具有验证步骤的有限枚举问题。

这种转变消除了在外循环中全局推理字母分配的需要。 我们只需要在本地验证每个候选三元组的一致性。

 ## 算法演练

 1. 枚举 A 从 0 到 999 的所有可能值。这个范围足够了，因为输入字符串将 A 的最大长度限制为 3 位数字，因此任何有效的赋值都不能产生更大的数字。 
2. 对于每个 A，枚举 B 从 0 到 999 的所有可能值。我们考虑每个可能的数字配对，因为任何有效的解决方案都必须显示为这些对之一。 
3. 将 C 计算为 A + B。如果 C 超过 999，我们立即跳过它，因为它无法匹配三字符模式。 
4. 将 A、B 和 C 转换为其数字表示形式的字符串，保留位置对齐以供以后匹配。 
5. 检查这些数字串是否能够一致地映射到给定的模式串上。 我们维护从字母到数字的映射以及从数字到字母的反向映射。 
6. 同时逐个位置地遍历三个字符串。 对于每个字母数字对，我们验证与现有映射的一致性。 如果任一方向出现冲突，我们立即拒绝这个三元组。 
7. 如果所有位置一致，我们就断定存在有效分配并提前终止。 

### 为什么它有效

 每个有效的解决方案都唯一对应于某个数字三元组（A、B、C）。 因为 A 和 B 的范围已被完全枚举，所以我们不会错过任何可能的候选者。 验证步骤强制执行字母和数字之间的严格双射，确保只接受结构一致的映射。 由于算术正确性是通过 C = A + B 的构造来检查的，因此任何接受的三元组必须同时满足代数约束和映射约束，以保证正确性。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def ok(s, t, u, a, b, c):
    sa = str(a)
    sb = str(b)
    sc = str(c)
    
    if len(sa) != len(s) or len(sb) != len(t) or len(sc) != len(u):
        return False

    ltod = {}
    dtol = {}

    for ch, d in zip(s, sa):
        if ch in ltod and ltod[ch] != d:
            return False
        if d in dtol and dtol[d] != ch:
            return False
        ltod[ch] = d
        dtol[d] = ch

    for ch, d in zip(t, sb):
        if ch in ltod and ltod[ch] != d:
            return False
        if d in dtol and dtol[d] != ch:
            return False
        ltod[ch] = d
        dtol[d] = ch

    for ch, d in zip(u, sc):
        if ch in ltod and ltod[ch] != d:
            return False
        if d in dtol and dtol[d] != ch:
            return False
        ltod[ch] = d
        dtol[d] = ch

    return True

def solve():
    s = input().strip()
    t = input().strip()
    u = input().strip()

    for a in range(1000):
        for b in range(1000):
            c = a + b
            if c >= 1000:
                continue
            if ok(s, t, u, a, b, c):
                print("YES")
                return

    print("NO")

if __name__ == "__main__":
    solve()
```该解决方案围绕单个辅助函数构建，该函数验证候选数字三元组是否与字母模式兼容。 映射逻辑使用两个字典来保证两个方向的双射，从而防止重复的字母分配和数字重用。 

主循环只是迭代所有可能的 A 和 B 值。 当 c 超过 999 时尽早拒绝，避免了不必要的字符串转换和一致性检查。 正确性取决于每个有效赋值必须对应于枚举范围内的某个整数三元组这一事实。

 ## 边缘情况

 一种边缘情况是相同的字母出现在所有三个字符串中，但通过算术暗示不同的数字角色。 在这种情况下，即使 A + B 在数值上等于 C，映射也会因数字重用冲突而失败。 该算法捕获了这一点，因为反向字典强制数字到字母分配的唯一性。 

另一种边缘情况是需要前导零来匹配长度。 由于验证显式地将数字转换的字符串长度与模式长度进行比较，因此任何不匹配都会自动拒绝无效的解释。 这可以防止隐藏接受值，例如当模式需要两个字符但算术产生一位数字字符串时将“01”解释为有效。
