 ## 解决方案
 ## 问题理解

 我们被赋予了一个在以秒为单位的时间轴上幸存下来的角色。 一开始他有一些最大生命值上限和初始生命值。 每过去一秒，他的生命值就会减少一个单位，如果他的生命值变为零，他就被认为已经死亡。 

有多个敌人，每个敌人必须被击中固定次数才能被击败。 角色每秒精确地执行一次攻击，并且每一秒他都可以选择攻击哪个敌人。 一旦敌人获得了所有所需的命中，就会立即被视为已击败并获得生命恢复加成，但总生命值永远不会超过原始最大上限。 

关键的复杂性是，当我们选择如何在敌人之间分配攻击时，时间会不断消耗生命值。 抵消这种消耗的唯一方法是尽早解决敌人，因为完成敌人会立即恢复生命值。 问题是是否存在某种对所有敌人的命中顺序，使得角色在整个过程中永远不会达到零生命值。 

约束足够大，任何模拟每秒动作的解决方案都是不可能的。 所有测试用例的命中总数可以达到 200000 次，这立即表明任何可行的解决方案都必须将每个敌人视为整个时间块，而不是模拟单个秒。 

当在没有结构的情况下应用贪婪直觉时，就会出现微妙的失败案例。 例如，如果需要的时间太长，尽早优先处理具有大量再生的敌人仍然可能会失败，因为在执行过程中，在获得奖励之前，生命值可能会降至零。 相反，如果只专注于矮小的敌人，如果它延迟了太多的稳定治疗效果，也可能会失败。 正确性取决于对完整任务的排序，而不是单个命中。 

核心困难在于，生存取决于花费的时间和获得的健康之间的持续平衡，并且这种平衡必须在完成工作的每个前缀上保持非负值。 

## 方法

 直接模拟将尝试每秒分配命中，每次更新生命值并尝试击中敌人的所有可能选择。 这显然是指数结构，因为每秒提供最多 n 个选择，并且总秒数等于所有 ti 的总和。 即使进行了优化，这种观点从根本上来说也太细粒度了。 

正确的抽象是将每个敌人视为一项消耗 ti 单位时间的工作，一旦完成，就会提供一次性的生命增益 hi。 在处理工作时，健康状况会随着时间的推移而不断下降，并且仅在完成点时才会增加。 这将问题转化为决定作业的顺序，以便在计划的每个前缀处，累积的运行状况永远不会低于零。 

如果我们确定一个订单，我们可以跟踪一个数量：处理完 k 个敌人后的当前生命值。 安全的条件是，在消灭每个敌人之前，时间损失和之前获得的剩余生命值必须是非负的。 这将问题转换为前缀可行性约束。 

关键的见解是每项工作都有两种相互竞争的影响。 长 ti 是危险的，因为它会在给予任何奖励之前消耗生存时间。 大的嗨是有用的，但只有在完成后才有效。 因此，对于尽早稳定健康而言“高效”的工作应该首先执行，而时间成本昂贵的工作应该推迟，除非它们有足够的补偿。

这自然会导致根据作业在执行过程中使平衡恶化的净趋势进行排序，由数量 ti − hi 捕获。 ti − hi 较小的作业要么消耗更少的时间，要么相对于其成本提供更多的恢复，从而更安全地提前安排。 一旦以这种方式排序，我们只需模拟前缀并验证健康状况永远不会低于零。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 暴力命令敌人 | O(n!) | O(n) | 太慢了|
 | 按 (ti − hi) 排序并模拟 | O(n log n) | O(n log n) | O(n) | 已接受 |

 ## 算法演练

 1. 计算每个敌人的值 di = ti − hi。 该值衡量敌人在应用奖励之前产生的“净伤害压力”。 
2.按照di升序对所有敌人进行排序。 单位时间内危害较小或回报较高的敌人会被放置在较早的位置。 
3. 将当前健康状况初始化为 m，将运行时间计数器初始化为零。 
4. 按排序顺序处理敌人。 对于每个敌人，将时间增加 ti 并减少生命值 ti 以反映其执行过程中的时间流逝。 
5.消灭敌人后，立即给生命加hi代表再生。 
6. 每次更新后，检查健康状况是否仍然严格为正或在完成后立即为零。 如果健康状况低于零，则计划无效。 
7.如果在不破坏条件的情况下处理了所有敌人，则答案是可行的。 

### 为什么它有效

 该过程可以被视为维持单个前缀不变量：在任何时间点，当前生命值等于初始生命值减去所花费的总时间加上从已完成的敌人获得的总再生量。 该值唯一可以恢复的时刻是在完成点，因此时间线上的最小值总是出现在完成敌人之前或之后。 

按 ti − hi 排序可确保更早地处理延迟更危险的敌人，从而防止出现长期高回报工作被推迟并导致过早耗尽的情况。 两个相邻敌人之间的任何顺序颠倒都不会提高可行性，因为交换一对违反顺序的敌人只能稍后转移时间压力，同时延迟奖励，这会削弱早期前缀的生存能力。 这表明排序顺序对于维护非负健康前缀是最佳的。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        n, m = map(int, input().split())
        enemies = []
        for _ in range(n):
            ti, hi = map(int, input().split())
            enemies.append((ti - hi, ti, hi))
        
        enemies.sort()
        
        health = m
        time_spent = 0
        ok = True
        
        for _, ti, hi in enemies:
            time_spent += ti
            health -= ti
            health += hi
            if health < 0:
                ok = False
                break
        
        print("YES" if ok else "NO")

if __name__ == "__main__":
    solve()
```该代码使用派生的优先级密钥 ti - hi 对每个敌人进行编码，并相应地进行排序。 模拟并没有明确地每秒建模； 相反，它将每个敌人压缩到一个单一的过渡中，其中时间和生命值会批量更新。 

一个微妙的实现细节是减法和加法的顺序。 在应用再生之前必须减去时间，因为健康状况是在授予奖励之前的那一刻评估的。 这与问题陈述相匹配，其中由于持续腐烂而发生死亡，但完成后立即应用治疗。

 ## 边缘情况

 当生命值在再生事件之前恰好达到零时，就会出现一个微妙的情况。 该实现将其视为安全的，因为再生是在完成时立即应用的。 例如，如果生命值是 1 并且下一个任务需要 1 秒，则生命值在完成时变为零，但奖励会立即应用并可以恢复它。 

当非常大的 hi 与非常大的 ti 配对时，会出现另一种边缘情况。 一个幼稚的策略可能会因为它的回报而尝试尽早优先考虑它，但如果它导致相对于其收益而言过多的时间消耗，正确的排序就会延迟它。 排序规则会自动处理这个问题，因为这样的任务有很大的 ti - hi 并且会被推到后面，以防止在长时间执行期间过早耗尽。 

最后，m 与各个 ti 值相比较小的情况凸显了为什么排序至关重要。 即使总体治疗是足够的，不正确的早期选择也会造成暂时的赤字，从而在后来的奖励出现之前杀死该过程，而贪婪排序通过使高压任务远离早期前缀来明确避免这种情况。
