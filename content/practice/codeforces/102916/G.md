 ## 解决方案
 ## 问题理解

 我们得到一个字符串和一个数字 k。 我们可以从该字符串中删除字符，同时保留剩余字符的相对顺序。 任务是按顺序准确选取 k 个字符，以便生成的子序列按字典顺序尽可能小。 

输出不是子字符串，因为我们可以在任何地方跳过字符。 它也不仅仅是 k 个最小字符的任意选择，因为必须保留它们在原始字符串中的顺序。 

字符串长度最多可达 1e6 的限制迫使我们远离任何二次方。 任何尝试考虑所有子序列或通过扫描剩余后缀重复模拟贪婪选择的解决方案都会降级为 O(nk) 或更糟，当 n 和 k 都很大时，这太慢了。 

当思考“在每一步中选择尽可能小的字符”时，就会出现一个天真的陷阱。 但这会失败，因为局部的小选择可能会阻止稍后访问更好的全局结构。 

例如，考虑 s = "bcaaa" 和 k = 3。天真的贪婪选择会尽早选择“a”，但如果立即选择第一个“a”，则可能会在以后留下太少的灵活性，最终被迫使用比必要的更糟糕的后缀。 正确答案是“aaa”，这需要故意跳过前面的字符。 

另一种失败模式是尝试重复扫描剩余后缀中的最小字符，然后从那里继续。 这在逻辑上是正确的，但太慢了，因为每一步都会重新扫描缩小的后缀，产生 O(nk) 行为。 

## 方法

 蛮力视角是逐个字符地构建子序列。 在答案的每个位置，我们考虑字符串中每个可能的下一个索引，该索引仍然留下足够的字符来完成长度为 k 的子序列，选择导致字典顺序最小延续的索引，然后递归。 这是正确的，因为它直接强制执行字典顺序的定义，但每个决策都需要扫描后缀并推理可行性，这会导致指数分支，或者如果仔细实施，最多会导致 O(nk)。 

关键的观察是，一旦我们决定将一个字符放置在答案中的某个位置，我们只关心最早的位置，在该位置我们可以安全地选择尽可能小的字符，同时仍然留下足够的字符来完成剩余的长度。 这将问题转化为在滑动可行窗口内重复选择最小字符。 

在任何步骤中，如果我们正在构建答案并且仍然需要 r 个字符，那么我们只能选择一个位置 i，使得 i 之后至少还剩下 r 个字符。 该约束动态地限制搜索窗口。 在该窗口内，选择最小的可用字符始终是最佳的，因为任何较大的字符都会使前缀变得更糟，并且始终有足够的剩余空间来补偿以后。 

该结构提出了一种贪婪算法，具有移动边界和范围上的重复最小查询，可以使用单调堆栈思想或线段树有效地实现。 最简单的竞争性编程解决方案使用贪婪扫描与仔细的指针移动相结合，这已经足够了，因为每个字符都被有效处理了恒定的次数。

 ## 算法演练

我们处理字符串，同时保留答案仍需要多少个字符。 令 r 为我们仍必须选择的字符数。 最初 r = k。 

我们还维护一个指针，指示在选择下一个字符时允许我们扫描多远。 在每一步中，我们都会确定仍然可以使用的最远索引，同时留下足够的字符来完成子序列。 这个边界是n-r。 

在这个有效窗口内，我们需要找到最小的字符。 一旦找到，我们将其附加到结果中，将 r 减一，并严格在所选索引之后继续该过程。 

每个字符选择都会缩小有效区域，并且我们永远不会重新访问不再有用的位置。 

### 为什么它有效

 正确性来自标准交换论证。 假设在某个步骤中我们在位置 i 选择字符 c，但存在一个按字典顺序更小的子序列，该子序列在位置 j > i 选择后面的字符 c' < c。 由于j在可行范围内，用c替换c'会使前缀立即变得更糟，与极小性相矛盾。 相反，任何早于所选最小值的字符要么违反可行性（没有足够的剩余位置），要么按字典顺序更大。 因此，贪心选择始终与最优全局解决方案一致，并且剩余后缀在具有较小 k 的较短字符串上保持相同的子问题。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    s = input().strip()
    k = int(input().strip())
    n = len(s)

    res = []
    i = 0
    remaining = k

    while remaining > 0:
        limit = n - remaining
        best_idx = i

        for j in range(i, limit + 1):
            if s[j] < s[best_idx]:
                best_idx = j

        res.append(s[best_idx])
        i = best_idx + 1
        remaining -= 1

    sys.stdout.write("".join(res))

if __name__ == "__main__":
    solve()
```实现直接遵循贪婪结构。 关键细节是计算`limit = n - remaining`，这增强了可行性：如果我们在位置 i 选择一个字符，则仍然必须有足够的字符来完成子序列。 

内循环只扫描有效窗口，确保正确性。 指针`i`确保我们永远不会重新考虑较早的位置，因此每个索引作为候选索引最多处理一次，然后永久排除。

 ## 边缘情况

 一种微妙的情况是，最佳选择不是全局最小的字符，而是受约束后缀内最小的字符。 

取 s = "acb", k = 2。第一步，选择 'a' 是正确的。 如果 k 是 1，我们仍然会选择“a”。 如果k = 2，选择“a”后，我们必须确保还剩下一个字符，因此我们被迫从后缀“cb”中进行选择。 该算法正确地限制了窗口并选择“b”，给出“ab”。 如果在不考虑可行性的情况下过早选择，总是选择全局最小剩余字符的幼稚方法将会失败。 

另一个边缘情况是最小的字符出现得太早，但会留下足够的剩余字符。 在 s = "baaa", k = 2 中，第一个 'a' 出现在索引 1 处。但是，立即选取它仍然是可行的，因为还剩下足够的字符。 该算法正确计算边界并允许它，产生“aa”。
