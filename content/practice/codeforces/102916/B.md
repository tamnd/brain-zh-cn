 ## 解决方案
 ## 问题理解

 该过程的每个步骤在结构上都是相同的。 Pavel 与一个随机字符重复匹配，该随机字符是从一组固定的字符中统一选择的$n$。 当他遇到角色时$i$，他必须立即选择两个任务中的一个。 每个任务都由一对来描述：时间成本和黄金奖励。 完成所选任务后，该过程会以新的随机角色重复，与之前的一切无关。 

利息数量是游戏的长期效率，意味着随着游戏步数无限增长，总金币除以总时间的限制。 由于字符序列是随机但固定的，因此对于每种字符类型，这成为一个选择始终采取两个动作中的哪一个的问题，以便最大化每单位时间的预期奖励。 

输入大小最多允许二十万个字符，每个字符都有两个可选操作。 任何以简单的嵌套方式重新​​计算每个字符对的全局效果的方法都会立即失败，因为即使$O(n^2)$已经远远超出了极限，甚至$O(n \log n)$如果涉及重复的全局重新计算，则必须仔细证明其合理性。 该结构表明每个字符而不是每个序列位置的决策。 

当一个任务的原始奖励更好但效率更差时，就会出现微妙的失败案例。 例如，一个任务在 100 个时间单位内给出 100 个金币（比率 1），与一个在 1 个时间单位内给出 2 个金币（比率 2）的任务进行竞争。 天真的“独立地为每个字符选择更高的比率”似乎是正确的，但问题是目标是总和的比率，而不是比率的总和，因此如果没有全局一致性论证，局部推理显然不是安全的。 

另一个边缘情况是，当两个任务的效率接近时，全局平均值的微小变化就会翻转最佳选择。 任何假设固定比较阈值而不验证一致性的方法都可能在精心构建的输入上失败，其中最佳全局比率恰好位于局部比率之间。 

## 方法

 暴力的想法是考虑每一个可能的选择分配，对每个角色的意义$i$，选择第一个或第二个任务。 有$2^n$这样的任务。 对于每个任务，我们计算每步的总预期时间和总预期黄金，然后除以得到比率。 

这是正确的，因为它明确枚举了所有策略。 然而，每个角色的策略数量加倍，因此$n = 200{,}000$，这是完全不可行的。 即使在$n = 40$，已经是临界点了。 

关键的观察是我们正在最大化线性函数的比率。 对于任何固定的决策规则，每一步的预期奖励是独立贡献的总和，对于时间也是如此。 困难在于该比率将所有决策结合在一起。 

打破这种耦合的标准方法是猜测比率的最佳值，称之为$\lambda$，并检查我们是否至少可以达到这个效率。 如果我们知道$\lambda$，每个角色决策再次成为局部：对于角色$i$，选择选项 1 有贡献$b_i - \lambda a_i$，选项 2 贡献$d_i - \lambda c_i$。 我们只需选择价值较大的选项即可。 对所有字符求和可以告诉我们这是否猜对了$\lambda$是可以实现的。 

这将问题转化为检查候选比率的可行性。 可行性函数是单调的$\lambda$: 如果某个$\lambda$是可以实现的，任何更小的值也是可以实现的。 这允许对答案进行二分搜索。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 蛮力战胜所有选择|$O(2^n \cdot n)$|$O(n)$| 太慢了|
 | 参数搜索+贪心检查 |$O(n \log W)$|$O(1)$| 已接受 |

 这里$W$是最终答案所需的精度范围。 

## 算法演练

 1. 固定候选值$\lambda$，解释为假设的每单位时间的最大黄金。 该值代表我们正在测试可行性的效率。 
2. 对于每个角色$i$，计算两个分数：$b_i - \lambda a_i$对于第一个任务和$d_i - \lambda c_i$为了第二个任务。 
3. 选择分数较大的任务。 这个本地决定是有效的，因为在固定的情况下$\lambda$，每个角色对全球不平等是否成立都有独立的贡献。 
4. 将所有选定的分数相加。 如果总和非负，则表示总奖励减去$\lambda$次总时间至少为零，因此达到效率$\lambda$是可能的。 
5. 使用二分查找$\lambda$在足够大的间隔内（通常从 0 到高于所有可能比率的安全值）进行细化，直到间隔小于所需的精度。 

### 为什么它有效

 我们测试的数量是$$\sum (b_i \text{ or } d_i) - \lambda \sum (a_i \text{ or } c_i).$$对于固定的$\lambda$，最大化这个表达式相当于独立地最大化每一项。 因此，在这个变换后的目标下，最优的全局政策是通过局部比较获得的。 

如果给定$\lambda$是可行的，任何较小的$\lambda$只增加所有表达式$x - \lambda y$，保持可行性。 这种单调性确保二分搜索收敛到可行性切换的唯一阈值。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def can(v, items):
    total = 0.0
    for a, b, c, d in items:
        if b - v * a >= d - v * c:
            total += b - v * a
        else:
            total += d - v * c
    return total >= 0

n = int(input())
items = [tuple(map(int, input().split())) for _ in range(n)]

lo, hi = 0.0, 1e9

for _ in range(80):
    mid = (lo + hi) / 2
    if can(mid, items):
        lo = mid
    else:
        hi = mid

print(lo)
```核心实现是可行性检查器。 对于每个候选比率，它会评估每个角色的两个选择，并贪婪地选择更好的调整值。 累加的和直接编码是否可以达到猜测的比率。 

二分搜索运行固定次数的迭代，而不是依赖于绝对精度比较。 这避免了浮点不稳定并确保一致的误差范围$10^{-9}$。 

## 工作示例

 考虑一个具有两个字符的小实例：

 字符 1：（a=1，b=2），（c=2，d=3）

 字符 2：（a=3，b=9），（c=1，d=1）

 我们测试一个候选人$\lambda = 2$。 

| 我| 选项 1 得分 | 选项 2 得分 | 选择| 贡献 |
 | --- | --- | --- | --- | --- |
 | 1 | 2 - 2*1 = 0 | 3 - 2*2 = -1 | 3 - 2*2 = -1 1 | 0 |
 | 2 | 9 - 2*3 = 3 | 9 - 2*3 = 3 | 1 - 2*1 = -1 | 1 - 2*1 = -1 | 1 | 3 |

 总和为 3，非负数，因此$\lambda = 2$是可行的。 

该轨迹表明，即使选项 2 对角色 1 具有更高的原始奖励，效率转换也会正确地对其进行惩罚，从而引导决策与全局目标保持一致。 

现在考虑$\lambda = 3$。 

| 我| 选项 1 得分 | 选项 2 得分 | 选择| 贡献 |
 | --- | --- | --- | --- | --- |
 | 1 | 2 - 3*1 = -1 | 2 - 3*1 = -1 | 3 - 3*2 = -3 | 3 - 3*2 = -3 | 1 | -1 |
 | 2 | 9 - 3*3 = 0 | 1 - 3*1 = -2 | 1 - 3*1 = -2 | 1 | 0 |

 总和是-1，所以$\lambda = 3$不可行，表明真正的最优值低于 3。 

## 复杂度分析

 | 测量 | 复杂性 | 说明|
 | --- | --- | --- |
 | 时间 |$O(n \log W)$| 每次可行性检查都会扫描所有字符，二分搜索执行固定次数的迭代 |
 | 空间|$O(n)$| 字符数据的存储|

 的价值$W$对于浮点二分搜索来说，它实际上是常数，因为收敛仅取决于所需的精度，而不取决于输入幅度。 和$n = 200{,}000$，这种方法完全符合限制。 

## 测试用例```python
import sys, io

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    import sys
    input = sys.stdin.readline

    def can(v, items):
        total = 0.0
        for a, b, c, d in items:
            if b - v * a >= d - v * c:
                total += b - v * a
            else:
                total += d - v * c
        return total >= 0

    n = int(input())
    items = [tuple(map(int, input().split())) for _ in range(n)]

    lo, hi = 0.0, 1e9
    for _ in range(80):
        mid = (lo + hi) / 2
        if can(mid, items):
            lo = mid
        else:
            hi = mid

    return str(lo)

# sample-like small checks
assert abs(float(run("1\n1 10 10 70\n")) - 7.0) < 1e-6
assert abs(float(run("1\n2 1 2 1\n")) - 0.5) < 1e-6

# equal options
assert abs(float(run("2\n1 1 1 1\n1 1 1 1\n")) - 1.0) < 1e-6

# skewed large ratio
assert float(run("1\n1 100 100 1\n")) > 1.0

 ## 边缘情况

 两个任务相同的情况，例如$a=b=c=d=1$，结果是每个$\lambda \le 1$是可行的。 该算法精确地收敛于 1，因为两种选择都会在以下位置产生零分：$\lambda = 1$，使得可行性边界尖锐但稳定。 

当一个任务在奖励中占主导地位但效率低下时，就会出现更微妙的情况。 例如，(1, 100) 与 (100, 101)。 对于小$\lambda$，选择第一个； 对于大$\lambda$，第二个变得有竞争力。 二分搜索自然地解决了这个转变点，因为可行性测试完全按照最佳比例翻转，确保不需要不一致的混合策略。
