 ## 解决方案
 ## 问题理解

 我们正在模拟一个具有固定容量的激活物品的充电系统。 每个测试用例给出一个容量`k`和一系列房间：`x`小房间和`y`房间很大。 玩家可以按任意顺序清除这些房间，每个清除的房间都会贡献费用：小房间提供 1 个单位，大房间提供 2 个单位。 每当累计电量达到或超过`k`，该物品使用一次，费用重置为零。 

目标是最大化该项目可以被激活的次数。 

关键难点是房间顺序不固定。 大房间更有价值，但在只需要 1 个单位即可触发使用的情况下，它们也存在“过度收费”的风险，并且额外的单位实际上会被浪费，因为到达后酒吧会立即重置`k`。 

约束条件很大：最多`2 × 10^5`测试用例和值`k, x, y`最多`10^9`。 这立即排除了对房间的任何模拟或每个房间迭代的任何贪婪构造。 每个测试用例必须在恒定的时间内处理。 

一个微妙的边缘情况来自这样一个事实：大房间根据剩余容量的不同而表现不同。 如果当前电量为`k-1`，小房间和大房间都完成了一次使用，但大房间浪费了一个单位。 在保留小房间对于完成剩余容量更有效的情况下，总是首先占用大房间或总是按价值贪婪地打包的幼稚贪婪方法将失败。 

例如，考虑`k = 5, x = 4, y = 2`。 如果我们首先使用大房间，我们可能会反复超出阈值并浪费容量，而混合策略则通过仔细使用小房间来完成精确的间隙来实现更多激活。 

真正的问题不是订购房间，而是计算有多少个完整的尺寸周期`k`我们可以从多个集合中形成`1`沙`2`s。 

## 方法

 暴力方法将明确模拟所有可能的房间排序。 对于每个订单，我们都会维护当前费用，一间一间地使用房间，并计算激活次数。 这是正确的，但完全不可行。 排列数为`(x + y)! / (x! y!)`，即使对于像这样的小输入，这也是一个天文数字`x = y = 50`。 

一种更结构化的暴力改进是在每一步尝试两种房间类型的同时贪婪地进行模拟。 这仍然呈指数级分支，因为决策取决于未来的余数，因此它仍然很棘手。 

关键的观察结果是，顺序仅在我们接近充满电的时刻在本地起作用。 对于大部分过程，我们只关心总累积电荷，而不关心它是如何构建的。 由于所有房间都贡献固定的金额，因此问题减少为最大化我们可以达到的倍数的次数`k`使用多组`1`和`2`。 

我们可以从总和的角度来思考`S = x + 2y`。 如果我们忽略排序，答案的上限是`S // k`。 然而，排序会影响我们能否实现这个界限，因为大房间在使用时可能会造成不可避免的浪费。`k`是奇数或者当我们被迫进入剩余状态时。 

正确的推理方法是将完整周期与剩余行为分开。 每次我们准确地到达`k`，我们消耗一些房间组合，总和至少为`k`。 在所有的形成方式中`k`，使用大房间总是最佳的，除非它稍后会导致更糟糕的剩余模式。 这导致了标准优化：贪婪地处理大房间，但在需要时使用小房间纠正类似奇偶校验的残留。 

最终的解决方案成为每个测试用例的恒定时间算术计算，基于有多少个完整的测试用例`k`块可以由可用的`2`首先是 s，然后是补充`1`s。

 ## 算法演练

 我们想要计算至少可以累积电荷多少次`k`使用`x`那些和`y`两次，每次激活后立即重置。 

1. 计算通过优先考虑大房间可以得到多少充分利用。 由于每个大房间贡献 2，我们首先考虑在形成时我们可以充分利用多少个大房间`k`- 大小的总和。 自然的第一步是计算有多少个完整的`k`我们只能使用大房间来建造街区，即`y * 2 // k`。 这提供了基线，但并未完全捕获混合使用情况。 
2. 不要考虑从头开始构建完整的块，而是将这个过程重新解释为消耗总能量`S = x + 2y`。 最大可能用途受以下限制`S // k`。 如果我们妥善管理剩菜，这给出了我们总能接近的上限。 
3. 唯一的障碍来自于贪婪地使用大房间后剩余的容量。 如果我们使用尽可能多的大房间，我们最终可能会留下无法有效填充的剩余房间`k`仅使用小房间，因为小房间可能不足以反复弥补差距。 
4. 计算大房间的余数贡献模数`k`。 让`large_sum = 2 * y`。 我们从中提取尽可能多的完整用途：`full_from_large = large_sum // k`, 和余数`rem = large_sum % k`。 
5. 现在合并小房间。 大房间后剩余容量为`rem + x`。 每次额外激活都需要`k`单位，所以我们计算`(rem + x) // k`额外用途。 
6.最终答案是`full_from_large + (rem + x) // k`。 

每个步骤都旨在将保证的完整周期与剩余合并分开。 重要的决定是首先处理大房间，因为它们是包装不连续的唯一来源； 一旦模数减少`k`，小房间的行为就像单元填充剩余的间隙。 

### 为什么它有效

 该算法基于以下不变量：在提取尽可能多的完整数据后`k`组从大房间的总贡献，任何剩余的大房间贡献严格小于`k`并且不能再独立形成另一个激活。 此时，剩下的所有灵活性都在于小房间，它只能增加余数，而不能改变其结构。 由于每次激活都需要准确`k`单位，剩下的问题就变成了组合剩余池的纯整数除法。 这保证了任何排序都不能增加完整的数量`k`超出计算表达式的阈值。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def solve():
    t = int(input())
    for _ in range(t):
        k, x, y = map(int, input().split())

        large_sum = 2 * y
        full_from_large = large_sum // k
        rem = large_sum % k

        total_rem = rem + x
        ans = full_from_large + total_rem // k

        print(ans)

if __name__ == "__main__":
    solve()
```代码直接实现了大房间贡献分解以及剩余与小房间合并。 第一个块计算可以从大房间的总价值中提取多少个完全激活。 然后将大房间的剩余部分与所有小房间合并，因为小房间是唯一可以完成下一个激活的部分进度的灵活填充物。 

这里的一个常见错误是尝试模拟排序或贪婪地交错小房间和大房间。 该解决方案通过将过程分解为总贡献和余数的算术来完全避免这种情况。

 ## 边缘情况

 当出现一种边缘情况时`k = 1`。 每个房间都会立即触发激活，所以答案应该是`x + 2y`。 该公式可以正确处理这个问题，因为`large_sum // 1 = 2y`其余部分没有任何贡献。 

另一个边缘情况是只有小房间时。 该算法简化为`x // k`， 自从`large_sum = 0`。 这符合直接直觉。 

当仅存在大房间时，结果变为`(2y) // k`，它匹配将两个分组为大小的块`k`。 

最微妙的情况发生在`k`只是略大于`2y`，其中仅大房间无法形成单一激活。 在这种情况下，大房间的剩余部分将被完全保留并与小房间合并，确保不会因排序而丢失潜在的激活。
