 ## 解决方案
 ## 问题理解

 我们得到了一份人与人之间的汇款清单，其中每条记录都表明一个人代表另一个人付款，从而形成了隐含的债务关系。 从这些交易中，我们可以计算出每个人最终欠下或被欠下多少钱。 正净值意味着该人应该总体收到金钱，而负净值意味着他们必须总体支付金钱。 

任务不是重新计算完整的历史，而是用一个更小的解决方案来取代它。 每个结算操作都是将一定金额从一个人转移到另一个人的单次定向转移。 限制是每个人最多只能发起一次此类转账，但他们可能会多次收到资金。 目标是完全平衡所有债务，使每个人的净余额为零。 

输入大小推动事务列表上的线性或近线性解决方案。 对于多达 20 万人和 1,000,000 笔交易，任何重复模拟配对匹配或重复扫描合作伙伴的方法都会太慢。 该结构强烈建议首先将所有交易分解为净余额，然后在大约 O(N log N) 或 O(N) 时间内执行最终结算计划的单遍构建。 

当考虑任意配对而不跟踪剩余容量时，就会出现微妙的失败案例。 例如，如果一个人欠 100 美元，而与他们匹配的唯一可用债权人只能收到 20 美元，则简单的单对分配将会陷入困境，因为债务人不允许分割其传出转账。 正确的结构必须确保每次传出传输与所选接收器的剩余容量完全匹配，或者我们仅匹配全部金额适合的兼容对。 

当每笔交易在本地而不是每人全局进行平衡时，就会出现另一个边缘情况。 两次中间取消可能会隐藏非零净余额，如果忽略该净余额，则稍后会破坏可行性。 

## 方法

 蛮力的想法是反复挑选任何余额为正的人，并尝试将他们与欠钱的人匹配，直到每个人的余额都为零。 这可以通过维护动态余额列表并重复搜索兼容的货币对来模拟。 每个操作都需要扫描有效的对应项并更新剩余余额。 在最坏的情况下，O(N) 人中的每个人都会触发对剩余池子的扫描，产生 O(N^2) 行为，这远远超出了 N 为 200,000 时的限制。 

关键的观察是，在计算净余额后，交易历史并不重要。 该系统变成了一个简单的流量守恒问题：总流入等于总流出，因此问题简化为将正盈余重新分配为负赤字。 每个人最多可以发起一次转账的限制意味着必须为每个债务人分配恰好一个传出边缘，因此我们必须将每个负余额视为必须一次性发送的不可分割的单位。 

这就将问题转化为“赤字节点”和“过剩容量”之间的贪婪匹配问题。 每个债权人可以接受多笔传入转账，而每个债务人必须指定一个债权人。 唯一的要求是所选债权人在转让时有足够的剩余能力来吸收债务人的全部金额。 执行这一点的自然方法是始终首先匹配最大的剩余盈余，因为较小的盈余更有可能在以后变得无法用于大额债务。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | --- | --- | --- | --- |
 | 暴力配对 | O(N^2) | O(N^2) | O(N) | 太慢了 |
 | 贪婪净余额 + 最大结构 | O(N log N) | O(N log N) | O(N) | 已接受 |

 ## 算法演练

我们首先将所有交易压缩为每个人的单一余额。 

1. 通过迭代所有交易来计算每个人的净余额。 如果 A 为 B 支付了 C，我们将 C 添加到 A 的余额中，并从 B 的余额中减去 C。 这给出了每个人应收到或支付的最终金额。 
2. 将人们分成两组。 一组包含所有余额为正的人，这意味着他们应该收到钱。 另一个包含所有负余额的人，这意味着他们必须付款。 
3. 将所有正余额存储在允许重复提取最大剩余容量的结构中，例如按剩余量键控的最大堆。 每个条目都保留人员索引及其剩余容量。 
4. 迭代所有负余额的人。 对于金额 d 等于减去其余额的每个债务人 i，我们必须准确分配一笔传出转账。 
5. 从堆中提取当前最有能力的债权人。 如果他们的剩余能力小于d，我们暂时将其排除在外，并尝试下一个最大的债权人。 这是有效的，因为不能完全满足该债务人的债权人以后仍然可以对较小的债务人有用。 
6. 一旦我们找到剩余能力至少为 d 的债权人 j，我们就创建金额为 d 的从 i 到 j 的转移。 然后我们将 j 的剩余容量减少 d。 
7. 如果j还有剩余容量，我们将其推回堆中。 否则，我们将丢弃它。 
8. 继续，直到处理完所有债务人。 每个债务人都被分配了一笔准确​​的转出转账。 

正确性依赖于堆始终准确代表所有债权人未使用的接收容量的不变性。 每项转让都会完全消除一个债务人，并恰好减少一个债权人的能力，而不会分割债务人的义务。 

总是选择最大的可用债权人的贪婪选择确保了如果存在可行的分配，我们不会过早地消耗小债权人，从而阻止以后的大额债务。 

## Python 解决方案```python
import sys
import heapq

input = sys.stdin.readline

def main():
    n, m = map(int, input().split())
    bal = [0] * (n + 1)

    for _ in range(m):
        a, b, c = map(int, input().split())
        bal[a] += c
        bal[b] -= c

    pos = []
    neg = []

    for i in range(1, n + 1):
        if bal[i] > 0:
            pos.append([bal[i], i])
        elif bal[i] < 0:
            neg.append([i, -bal[i]])

    heap = []
    for amt, i in pos:
        heapq.heappush(heap, (-amt, i))

    res = []

    for i, need in neg:
        while need > 0:
            amt, j = heapq.heappop(heap)
            amt = -amt

            if amt >= need:
                res.append((i, j, need))
                amt -= need
                need = 0
                if amt > 0:
                    heapq.heappush(heap, (-amt, j))
            else:
                res.append((i, j, amt))
                need -= amt

    print(len(res))
    for a, b, c in res:
        print(a, b, c)

if __name__ == "__main__":
    main()
```该解决方案首先将所有交易合并为每人的单一净余额。 这一步至关重要，因为只有最终的净额才对结算产生影响； 中间转移没有额外的限制。 

正余额会转化为最大堆，以便我们始终可以访问具有最大剩余容量的债权人。 这种选择很重要，因为较小的债权人是更脆弱的资源，如果可能的话，应该为较小的债务保留。 

每个债务人只被处理一次。 内循环将债务贪婪地分配给可用的债权人，并在必要时分割债权人的能力。 尽管债权人可能会收到多笔传入转账，但尊重约束条件，每个债务人始终会被分配单笔传出转账。 

一个微妙的一点是，只有当债权人仍然有剩余容量时，才会将其重新插入堆中。 这可确保堆大小保持在 O(N) 范围内。

 | 时间 | O(N log N + M) | O(N log N + M) | M表示平衡聚合，N log N表示匹配时的堆操作 |

 该算法完全符合约束条件，因为即使是 1e6 事务也只需要线性传递，并且堆操作随着活跃参与者的数量而不是事务的数量而扩展。

import sys, io

def run(inp: str) -> str:
    sys.stdin = io.StringIO(inp)
    import sys as _sys
    from collections import defaultdict
    import heapq

    input = sys.stdin.readline
    n, m = map(int, input().split())
    bal = [0] * (n + 1)

    for _ in range(m):
        a, b, c = map(int, input().split())
        bal[a] += c
        bal[b] -= c

    pos = []
    neg = []

    for i in range(1, n + 1):
        if bal[i] > 0:
            pos.append([bal[i], i])
        elif bal[i] < 0:
            neg.append([i, -bal[i]])

    heap = []
    for amt, i in pos:
        heapq.heappush(heap, (-amt, i))

    res = []

    for i, need in neg:
        while need > 0:
            amt, j = heapq.heappop(heap)
            amt = -amt
            take = min(amt, need)
            res.append((i, j, take))
            amt -= take
            need -= take
            if amt > 0:
                heapq.heappush(heap, (-amt, j))

    out = [str(len(res))]
    for a, b, c in res:
        out.append(f"{a} {b} {c}")
    return "\n".join(out)

 ## 边缘情况

 当所有交易成对抵消时，就会出现关键的边缘情况，因此每个余额都为零。 在这种情况下，算法构建一个空堆并且不处理债务人，从而在不进入匹配阶段的情况下产生零输出操作。 

另一种情况是，一个债权人的能力非常大，而存在许多小债务人。 堆确保债权人被重复使用，逐步减少其剩余容量，同时仍然尊重每个债务人的单一传出约束。 

当债权人的容量被部分使用然后重新插入堆时，会发生最后一个微妙的情况。 正确性取决于在重新插入之前始终更新剩余容量，确保不会重复使用过时的值。
