 ## 解决方案
 ## 问题理解

 我们得到一个人的全名，分成三四个单词。 最后两个词是固定的姓氏，而它们之前的所有词都是名字。 根据这个名称，我们必须重建一个非常具体的候选登录标识符序列，称为 LDAP。 

每个 LDAP 都是通过采用第一个名字的前缀、可选的第二个名字的前缀、然后是完整的第一个姓氏、最后是第二个姓氏的某个前缀来形成的。 系统以“使用第一个名字的多少”的固定递增顺序生成这些 LDAP，并且在每个这样的级别中，它逐渐扩展第二个姓氏，同时尊重前缀如何在各部分之间平衡的约束。 

我们不知道当前未使用哪个 LDAP，但我们可以通过提出 LDAP 字符串来查询系统。 系统会响应该 LDAP 是否已被占用。 我们必须使用最多 20 个查询来找到任何未使用的 LDAP。 

关键的约束是，可能的 LDAP 的空间是结构化的，并且在隐藏的排序中是单调的：当我们增加使用的第一个给定名称的数量时，我们会在候选组中移动，并且在每个组内，后缀扩展遵循可预测的模式。 

时间限制仅在交互开销方面很紧，而不是在计算方面。 查询的数量是真正的约束，因此任何线性扫描所有可能性的解决方案都是不可能的。 即使对所有前缀组合进行适度的枚举也可能会超出限制，因为名称最多可达 20 个字符，可能会提供数百种组合。 

一种简单的方法是按照所描述的顺序生成所有有效的 LDAP，并查询每个 LDAP，直到找到未使用的 LDAP。 这是不安全的，因为在最坏的情况下，候选者的数量大致随着姓氏和名字的前缀长度的乘积而增长，很容易超出查询预算。 

第二种失败模式来自于对顺序的误解。 这一代人并不纯粹是字典式的； 它按照第一个名字的前缀长度进行分组，然后才扩展第二个姓氏。 如果我们假设简单的字典顺序，我们可以跳过有效的候选者或浪费对无效顺序的查询，可能会丢失所需的未使用的 LDAP。 

当名字很短或第二个姓氏很短时，就会出现边缘情况。 特别是，当第一个名字的长度为1时，不存在有意义的“增量前缀策略”，所有变化都来自第二个姓氏扩展。 另一个棘手的情况是当存在多个给定名称时，因为第二个给定名称仅提供单个字母前缀（如果存在），但该前缀在每个组中保持固定。 

## 方法

 暴力策略将完全按照声明中的描述显式模拟 LDAP 生成器。 我们将迭代第一个名字的前缀长度，然后迭代第二个姓氏扩展的可能分割，构建每个 LDAP 候选字符串并查询它。 这是正确的，因为它遵循与系统相同的顺序，因此最终我们会遇到未使用的 LDAP。 但是，生成的字符串数量可能很大。 如果第一个名字的长度为 n，第二个姓氏的长度为 m，则每个级别的组合数量与 m 成正比，并且在所有 n 个级别中，这将变为 O(nm)。 当 n、m 达到 20 时，这已经接近 400 个候选，并且在具有开销限制和最坏情况对抗性排序的交互设置中，这有超过 20 个查询限制的风险。

关键的观察是我们不需要重建完整的序列。 我们只需要任何未使用的 LDAP。 该结构意味着，对于第一个名字的固定前缀长度，LDAP 空间在第二个姓氏扩展方面形成连续的级数。 这使得将构造视为单调决策空间成为可能：如果某个前缀长度允许未使用的 LDAP，我们不需要详尽地探索更小或更大的配置。 

这使我们能够减少搜索第一个给定名称的前缀长度的问题。 对于每个前缀长度，我们构造一个最小有效 LDAP 并检查它是否未使用。 一旦我们找到任何未使用的候选者，我们就可以尝试通过贪婪地扩展第二个姓氏来改进它，因为任何扩展都会保留有效性，直到字符耗尽。 

核心思想是系统保证至少存在一个未使用的LDAP，因此我们在一个具有有限深度和单调扩展的结构化空间中进行搜索。

 | --- | --- | --- | --- |
 | 暴力枚举| O(nm) 查询 | O(1) | O(1) | 太慢了 |
 | 前缀长度引导搜索 | O(n + m) 次查询 | O(1) | O(1) | 已接受 |

 ## 算法演练

 我们模拟 LDAP 构造，同时仔细控制扩展前缀的方式。 

1. 将输入名称拆分为多个组件。 最后两个词是姓氏，它们之前的所有内容形成名字。 我们提取第一个名字、可选的第二个名字、第一个姓氏和第二个姓氏。 
2. 预先计算第二个名字的第一个字母的贡献（如果存在）。 这在使用它的所有结构中保持固定，因此我们从不重复重新计算它。 
3. 将第一个给定名称的可能前缀长度从 1 迭代到其完整长度。 每个前缀长度定义候选 LDAP 的结构层。 
4. 对于每个前缀长度，构造该层中的最小 LDAP。 根据规则，这使用第一个名字的前缀、可选的第二个名字首字母、完整的第一个姓氏以及第二个姓氏允许的最小前缀。 
5. 查询该LDAP。 如果系统响应未使用，我们不会立即停止； 相反，我们尝试一次贪婪地扩展第二个姓氏一个字符，仅在必要时进行查询，因为任何未使用的最小形式都意味着其所有扩展仍然是候选，直到达到约束边界。 
6. 如果采用构造的 LDAP，我们将移至下一个前缀长度，因为该层中的所有候选者都被隐式阻止或已探索。 
7. 一旦我们识别出任何未使用的 LDAP，我们就会立即以所需的格式输出它并终止。 

这样做的原因是，在第一个名字的每个前缀长度内，所有有效的 LDAP 都在第二个姓氏扩展上形成线性进展。 这意味着我们永远不需要探索分支结构。 每个查询要么消除整个前缀层，要么确认我们可以在其中扩展。 由于总共最多有 20 个字符，因此在遍历最多两个线性维度时，我们不能超过 20 个查询。 

## 为什么它有效

 该结构定义了一个分层空间，其中每一层对应于第一个给定名称的固定前缀。 在每一层内，有效的 LDAP 的区别仅在于附加了多少第二个姓氏。 这种交互保证了未使用的 LDAP 存在于至少一层中，一旦我们进入该层，扩展的行为就会单调：如果前缀有效且未使用，则任何较短的前缀也是有效的，我们可以安全地增长它，直到达到边界或使用限制。 这种单调性可以防止回溯，并确保每个查询要么将我们推进到一个新层，要么扩展一个已知的有效前缀。 

## Python 解决方案```python
import sys
input = sys.stdin.readline

def flush():
    sys.stdout.flush()

def query(s):
    print(s)
    flush()
    return input().strip()

def solve():
    parts = input().strip().split()
    if len(parts) == 3:
        g1, sur1, sur2 = parts[0], parts[1], parts[2]
        g2 = ""
    else:
        g1, g2, sur1, sur2 = parts[0], parts[1], parts[2], parts[3]

    g2_initial = g2[0] if g2 else ""

    # try increasing prefix of first given name
    best = None

    for i in range(1, len(g1) + 1):
        base = g1[:i] + g2_initial + sur1

        # start with minimal second surname usage
        cand = base + (sur2[0] if sur2 else "")

        res = query(cand)
        if res == "0":
            # unused, try to extend greedily
            best = cand

            for j in range(1, len(sur2)):
                nxt = cand + sur2[j]
                r = query(nxt)
                if r == "0":
                    best = nxt
                else:
                    break

            print("! " + best)
            flush()
            return

    # fallback (problem guarantees existence)
    print("! " + best)
    flush()

if __name__ == "__main__":
    solve()
```该解决方案仔细维护第一个名字的每个前缀的最小候选，然后使用交互反馈来决定是留在该层还是扩展第二个姓氏。 每次输出后的刷新至关重要，因为没有它，交互器将不会响应。 

一个微妙的实现细节是我们从不尝试推理完整的组合空间。 我们只扩展当前有效的第二个姓氏前缀，这确保我们保持在 20 个查询预算之内。

 ## 边缘情况

 对于第一个给定名称长度为 1 的名称，该算法立即在单层中运行。 例如，“a b c d”仅导致一个有意义的前缀，因此所有决策都是由第二个姓氏扩展驱动的。 该算法通过不依赖更深的前缀迭代来处理这个问题。 

当第二个姓氏为空或非常短时，扩展在最小候选构建后立即停止。 贪婪扩展循环自然终止，无需额外查询。 

当存在多个名字时，仅使用第二个名字的第一个字母。 由于它在所有候选者中都是固定的，因此不会影响分支结构，并且在基础构建期间安全地包含一次。 

每个案例都证实该算法不依赖于关于名称长度分布的隐藏假设，并且在所有有效输入下保持稳定。
