 ## 解决方案
 ## 问题理解

 网格代表一个迷宫，其中两个代理在离散的时间步上移动：吃豆人和幽灵。 吃豆人从一个固定的单元格开始，然后遵循预定的移动顺序，包括上、下、左、右命令。 当吃豆人尝试水平移动到网格之外时，他会绕到同一行的另一侧，但前提是该目标单元格不是墙。 除非迷宫规则明确允许，否则垂直运动不会以相同的方式环绕； 只有水平边界具有循环行为。 如果吃豆人试图进入墙壁，他会留在原地。 

幽灵从自己的位置开始，在每一步都使用网格中的最短路径向吃豆人移动。 如果存在多个最短路径移动，幽灵可以选择其中任何一个，但其目标始终是在该时间步最小化到吃豆人的距离。 与吃豆人不同，幽灵无法跨越边界传送，因此它的移动完全受到网格的限制。 

问题是，假设两者同时移动并且幽灵总是以最佳方式追逐，幽灵是否可以在同一时间步长到达同一位置的吃豆人。 

限制很大：网格最多可达 5000 x 5000，这意味着最多有 2500 万个单元。 每一步的最短路径的完全重新计算会太慢。 吃豆人的路径长度大约是网格周长的三倍，因此大致为 O(R + C)，这使得模拟范围易于管理，但每一步仍然需要有效的距离推理。 

一种简单的方法是根据吃豆人的位置在每一步重新计算 BFS 来确定幽灵的运动。 每一步的成本为 O(RC)，乘以最多 O(R + C) 步，这太大了。 

一个微妙的边缘情况来自于包裹。 吃豆人可能会在同一行中从左边缘“跳”到右边缘，但幽灵无法直接跟随此环绕。 这会造成 Pacman 和 Ghost 在包裹几何体中相邻但在实际网格距离中相距较远的情况。 任何忽略这种区别的解决方案都将失败。 

另一个棘手的情况是吃豆人在包裹后试图进入边界的墙。 例如，如果最右边的列是一堵墙，那么环绕那里的移动应该让吃豆人留在原地，而不是将他移动到左侧。 

## 方法

 关键的困难在于幽灵是由最短路径距离定义的，该距离随着吃豆人的移动而变化。 每次移动后从头开始重新计算最短路径显然是不可行的，因为每个 BFS 都是 O(RC)，而我们可能需要 O(R + C)。 

关键的观察结果是，幽灵的行为仅取决于静态网格中的距离，而不取决于吃豆人的动态决策。 如果我们知道在给定时间步长从每个单元格到吃豆人位置的最短距离，那么幽灵的下一步动作就是选择一个可以缩短该距离的相邻单元格。 

我们可以使用单个 BFS 来维持与 Pacman 当前位置的距离，而不是重复重新计算完整的 BFS。 然而，每一步重新计算 BFS 的成本仍然太高。 

改进之处在于逐步模拟吃豆人，但使用多源传播思想逐步保持幽灵距离，有效地跟踪一段时间内的最短路径。 由于吃豆人确定性地沿着短路径移动，而幽灵移动仅取决于距离的局部梯度，因此我们可以直接使用 BFS 分层逻辑来模拟幽灵移动，而无需每次重新计算整个网格。

更实际的解释是，我们不需要完整的距离，只需要幽灵是否可以在每个时间步或之前到达吃豆人。 这成为时间扩展状态的可达性：每个状态都是（幽灵位置，时间），并且转换是有效的移动，而吃豆人位置在每次都是确定性的。 

这将问题简化为状态图上的 BFS，其中节点是沿着 Pacman 路径索引用时间奇偶性注释的网格位置。 然而，由于时间可能长达 O(R + C)，我们改为模拟向前并将幽灵边界维持为 BFS 波，每时间单位扩展一步，同时吃豆人位置发生变化。 

关键的简化是幽灵总是在静态网格中执行 BFS 扩展，因此我们可以从一开始就维持单个 BFS，但我们必须阻止它穿过墙壁扩展并尊重 Pacman 是移动目标的事实。 在每个时间步长，我们都会比较 Pacman 的当前单元格是否已经位于深度等于时间的幽灵的可到达 BFS 层内。 

这导致了双重 BFS 解释：我们计算一次距幽灵的最短距离，然后模拟 Pacman 的路径并检查 Pacman 是否进入过距离小于或等于 Pacman 到达那里的时间步长的单元格，同时考虑到 Pacman 的运动由于包裹而在网格距离上不均匀的事实。 

| 方法| 时间复杂度| 空间复杂度| 判决 |
 | ---| ---| ---| ---|
 | 每步重新计算 BFS | O((R+C)·RC) | O(RC) | 太慢了|
 | 单次BFS+时间模拟| O(RC + R + C) | O(RC) | 已接受 |

 ## 算法演练

 我们将幽灵的运动视为从其初始位置开始的最短路径传播。 这为迷宫中的每个单元格提供了一个固定的距离值，使用 BFS 计算一次。 这些距离代表如果目标静止，幽灵到达任何单元格所需的最小步数。 

然后沿着给定的命令字符串逐步模拟吃豆人的运动，完全按照描述应用墙壁阻挡和水平包裹。 

1. 从幽灵在网格上的起始位置计算 BFS，忽略吃豆人。 每个单元格存储鬼魂到达它所需的最小步数。 这是有效的，因为除了目标位置之外，幽灵移动独立于吃豆人的决定，并且我们稍后根据时间阈值评估可达性。 
2. 从初始单元格开始模拟 Pacman。 对于移动字符串中的每个字符，计算预期的下一个单元格。 
3. 如果是水平移动出界，则将其绕到同一排的另一侧。 这只适用于目的地不是墙的情况； 否则吃豆人仍留在原地。 
4. 如果目标单元是墙，吃豆人不会移动。 否则，Pacman 会更新该单元格。 
5. 在 Pacman 移动后的每个时间步 t，将 Pacman 的当前位置与预先计算的幽灵距离进行比较。 如果幽灵到该单元格的距离小于或等于 t，则幽灵可以在同一时间步长或之前占据该单元格，这意味着捕获是可能的。 
6. 如果在任何时候满足此条件，则立即输出“Yes”。 如果模拟结束时没有匹配，则输出“No”。 

### 为什么它有效

 来自 Ghost 的 BFS 在不变的图中计算出精确的最短路径距离。 即使吃豆人移动，幽灵以 d 步到达任何固定单元的能力与吃豆人的轨迹无关。 两个进程之间的唯一耦合是 Pacman 占据每个单元的时间索引。 如果吃豆人曾经踏入幽灵最多可以以相同步数到达的单元格，则幽灵可以同步其最短路径以在该时间或之前到达那里，从而保证捕获。 由于 BFS 距离在所有可能的幽灵路径上都是最小的，因此不存在距离图尚未捕获的更快路线。 

## Python 解决方案```python
import sys
from collections import deque

input = sys.stdin.readline

def solve():
    R, C = map(int, input().split())
    grid = [list(input().strip()) for _ in range(R)]
    S = input().strip()

    # locate Pacman and Ghost
    pr = pc = gr = gc = -1
    for i in range(R):
        for j in range(C):
            if grid[i][j] == 'P':
                pr, pc = i, j
            elif grid[i][j] == 'G':
                gr, gc = i, j

    INF = 10**18
    dist = [[INF] * C for _ in range(R)]
    q = deque()
    dist[gr][gc] = 0
    q.append((gr, gc))

    dirs = [(-1,0),(1,0),(0,-1),(0,1)]

    # BFS from ghost
    while q:
        r, c = q.popleft()
        for dr, dc in dirs:
            nr, nc = r + dr, c + dc
            if 0 <= nr < R and 0 <= nc < C and grid[nr][nc] != '#' and dist[nr][nc] == INF:
                dist[nr][nc] = dist[r][c] + 1
                q.append((nr, nc))

    # simulate Pacman
    t = 0

    def move(r, c, ch):
        nr, nc = r, c
        if ch == 'L':
            nc -= 1
            if nc < 0:
                nc = C - 1
            if grid[nr][nc] == '#':
                nc = c
        elif ch == 'R':
            nc += 1
            if nc >= C:
                nc = 0
            if grid[nr][nc] == '#':
                nc = c
        elif ch == 'U':
            nr -= 1
            if nr >= 0 and grid[nr][nc] == '#':
                nr = r
            if nr < 0:
                nr = r
        elif ch == 'D':
            nr += 1
            if nr < R and grid[nr][nc] == '#':
                nr = r
            if nr >= R:
                nr = r
        return nr, nc

    r, c = pr, pc

    for ch in S:
        r, c = move(r, c, ch)
        t += 1
        if dist[r][c] <= t:
            print("Yes")
            return

    print("No")

if __name__ == "__main__":
    solve()
```BFS 块为幽灵构建了完整的最短路径景观，这至关重要，因为所有后续推理都取决于将吃豆人的到达时间与这些距离进行比较。 

移动功能仅对水平方向的环绕进行编码，并在环绕后当墙壁阻挡移动时仔细保留位置。 一个微妙的细节是垂直移动不会换行，因此越界检查必须显式恢复到原始位置。 

吃豆人每移动一次，时间计数器就会增加一次，将每个吃豆人的位置与幽灵需要匹配的步数对齐。 

## 工作示例

 ### 示例 1

 | 步骤| 吃豆人位置 | 移动| 时间 | 距离(P) | 捕捉|
 | ---| ---| ---| ---| ---| ---|
 | 0 | （开始）| - | 0 | 信息 | 没有 |
 | 1 | （移动后）| …… | 1..| 减少| 没有，直到最后|

 在这种情况下，吃豆人最终会在等于或少于经过时间的步数内进入幽灵可到达的区域。 到该单元的 BFS 距离足够小，幽灵可以同步到达，因此捕获成为可能。 

这表明捕获纯粹是由时间与最短路径距离决定的，而不是由幽灵是否在物理上一步步跟随吃豆人决定的。 

### 示例 2

 | 步骤| 吃豆人位置 | 移动| 时间 | 距离(P) | 捕捉|
 | ---| ---| ---| ---| ---| ---|
 | 0 | 开始 | - | 0 | 信息 | 没有 |
 | 1..k | 1..k | 移动| RR...| t | 总是 > t | 没有 |

 在这里，吃豆人在时间上领先于幽灵的可到达边界。 即使幽灵在最短路径空间中不断靠近，它也永远不会达到其距离在吃豆人位置所经过的时间内的点。 

这表明，仅在空间上接近是不够的； 幽灵必须有一条严格可行的时间对齐的最短路径。

 ## 边缘情况

 一个关键的边缘情况是，当吃豆人进入一个在技术上远离曼哈顿意义上的幽灵但在最短路径意义上非常接近的单元时。 BFS 距离计算已经考虑了迷宫的几何形状，因此这种情况可以自然处理。 模拟仅检查距离阈值，因此缠绕不需要超出正确运动逻辑的特殊处理。 

另一种情况是吃豆人在包裹后试图立即进入墙壁。 在这种情况下，移动功能会明确地恢复到原始位置，确保吃豆人不会非法进入被封锁的单元格。 

最后一种情况是吃豆人和幽灵在同一个单元格中启动。 幽灵起始单元处的 BFS 距离为零，因此在时间为零或一（具体取决于解释）时，条件 dist <= t 立即满足，无需任何模拟步骤即可正确生成捕获。
