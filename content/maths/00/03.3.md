## 3.3 定义和命名约定

 定义引入了具有固定含义的新术语。 它将模式变成可重复使用的思想对象。 如果没有定义，数学将是重复描述的集合而不是一个系统。 

一个好的定义可以做三件事。 

| 功能| 意义|
 | ----------------------- | --------------------------------------------------- |
 | 命名一个概念 | 给这个概念一个稳定的把握|
 | 陈述具体条件 | 指定该术语何时适用 |
 | 支持重复使用 | 允许概念出现在定理和证明中 |

 例如：```text
A function f : X → Y is injective if for all x1, x2 ∈ X,
f(x1) = f(x2) implies x1 = x2.
```该定义命名了一个属性。 引入后，我们可以写“让`f`是单射的”，而不是每次都重复完整的条件。

 定义是本地合同。 一旦定义了术语，以后的每次使用都应尊重合同。 如果定义改变，依赖于它的定理也可能改变。 

数学定义通常具有以下形式：```text
An object X is called [name] if it satisfies [conditions].
```或者：```text
We say that X has property P if [conditions].
```条件应该是必要且充分的。 读者应该能够确定一个对象是否满足定义。 

贫穷的：```text
A good function is one that behaves nicely.
```更好的：```text
A function f : X → Y between metric spaces is continuous at x ∈ X
if for every ε > 0 there exists δ > 0 such that
d_X(x,y) < δ implies d_Y(f(x), f(y)) < ε.
```改进的定义给出了检验。 它指定对象、量词以及输入和输出之间的关系。 

定义应在条件之前说明上下文。 短语“连续函数”在度量空间、拓扑空间、序理论和域理论中意味着不同的事物。 明确的定义命名了环境结构。 

| 术语 | 需要上下文 |
 | ---------- | --------------------------------------- |
 | 连续| 拓扑或度量|
 | 简单| 群、图、模、代数 |
 | 正常 | 子群、扩展、运算符、空间 |
 | 常规| 图形、语言、度量、方案 |
 | 完成 | 度量空间、格、理论、范畴 |

 数学重复使用名称。 当上下文清晰时，这是有效的；当上下文不清晰时，这是有效的。 “正常”这个词有很多含义。 正规子群、正规算子和正规拓扑空间是不同的概念。 

命名约定可以减少歧义。 某些类型的名字代表某些角色。 

| 表格 | 典型用途|
 | ---------- | -------------------------------------------------------- |
 |`-ness`| 性质：紧凑性、连通性|
 |`-ity`| 性质：连续性、线性 |
 |`-ism`| 观点：建构主义、形式主义|
 |`-ization`| 流程：完成、本地化|
 |`co-`| 对偶概念：余积、余极限 |
 |`pre-`| 较弱的初步概念：预购、预编|
 |`semi-`| 部分弱化：半群、半环 |
 |`quasi-`| 近似或弱化形式|
 |`pseudo-`| 相似的表面行为，较弱的结构|

 这些模式不是正式的规则，但它们可以帮助读者推断概念之间的关系。 

定义既不能太宽泛，也不能太狭窄。 如果它太宽泛，它会捕获不支持有用定理的对象。 如果它太窄，就会排除自然的例子。 

例如，群的定义足够广泛，可以包括加法下的整数、可逆矩阵、排列和对称性。 它足够窄，足以支持有关恒等、逆、子群和同态的强定理。 

有用的定义确定了后续结果所需的确切结构。 

| 定义风格| 结果 |
 | ---------------- | --------------------------------------- |
 | 太具体了| 定理无法推广 |
 | 太抽象了| 例子变得难以辨认|
 | 选得好| 定理应用广泛且有意义 |

 定义也控制平等。 在定义数学对象时，应该知道什么算作同一个对象。 

例如，图可以定义为：```text
G = (V, E)
```在哪里`V`是一组顶点并且`E`是一组边。 但具有不同顶点名称的两个图可能在结构上相同。 这导致图同构。 

定义给出了原始对象。 等价概念告诉我们如何对它们进行分类。 

好的定义通常会暴露地图。 在许多领域，正确的定义是结构保持图是自然的。 

| 结构| 结构保持图|
 | ----------------- | ------------------------------------------------------ |
 | 集团| 同态 |
 | 戒指| 环同态|
 | 向量空间| 线性图|
 | 拓扑空间| 连续图 |
 | 度量空间| 非膨胀、连续或等轴测图 |
 | 类别 | 函子|

 这些地图揭示了定义旨在保留的内容。 

定义可以是显式的或隐式的。 

明确的定义给出了直接的条件。```text
A group is a set G with a binary operation satisfying associativity,
identity, and inverse laws.
```隐式定义给出了表征一类对象的公理。```text
A field is a structure satisfying the field axioms.
```在实践中，大多数数学定义结合了两者。 他们描述数据并列出定律。```text
A ring consists of a set R with operations + and * such that ...
```这种“数据加法律”的形式很常见。 

| 部分| 示例|
 | ---------------- | ---------------------------------------------------- |
 | 数据| 一套`R`， 运营`+`和`*`|
 | 法律| 结合性、分配性、恒等式 |
 | 派生概念| 单位、理想、模块 |
 | 地图 | 环同态|

 定义应避免隐藏选择。 如果一个对象依赖于一个选择，那么定义应该这样说。 

例如，“选择一个基础`V`”引入了非规范数据。取决于所选基础的结果可能不是固有的`V`。 独立于基础的结果是结构性的。 

仅当结构由给定数据自然确定时才应使用“规范”之类的术语。 如果可以有多种选择，则该结构应称为“选择的”、“固定的”或“选择的”，而不是规范的。 

定义还需要示例和非示例。 当读者能够区分满足定义的内容和不满足定义的内容时，定义就变得可用。 

对于“团体”：

 | 对象| 团体？ | 原因 |
 | ---------------------------------------------------- | --------- | --------------------------------------- |
 | 加法下的整数 | 是的 | 关联性、同一性`0`, 逆 |
 | 加法下的自然数 | 没有 | 无加性逆 |
 | 乘法下的非零实数 | 是的 | 关联性、同一性`1`, 逆 |
 | 乘法矩阵| 有时| 可逆方阵形成群 |

 非例子很重要，因为它们表明哪种条件很重要。 

定义在预期操作下应该是稳定的。 如果一类对象是有用的，那么它通常应该在自然构造下被封闭。 

例如，组的子组是组。 群的乘积就是群。 群的同构副本是群。 这些闭包属性使定义变得稳健。 

实用的定义清单：

 | 问题 | 目的|
 | ---------------------------------- | -------------------------------------- |
 | 需要什么数据？ | 识别对象 |
 | 需要什么法律？ | 修复结构 |
 | 哪些例子可以满足呢？ | 显示相关性 |
 | 哪些非例子失败了？ | 显示边界 |
 | 哪些地图保存了它？ | 呈现出自然的转变 |
 | 适用什么等价关系？ | 支持分类|
 | 后来的哪些定理使用了它？ | 测试有用性 |

 命名应遵循相同的规则。 名称应该简短、稳定，并尽可能与现有约定保持一致。 只有当新名称减少重复解释或确定经常使用的概念时，才应引入新名称。 

坏名字隐藏了结构。 一个好的名字表明了该对象的作用。 

| 弱名| 更好的名字 |
 | ------------- | ----------------------------------- |
 | 不错的功能| 连续功能 |
 | 好套装| 紧凑套装 |
 | 特别地图| 投影图|
 | 事情| 对象、元素、态射、运算符 |
 | 大地产| 完整性、有界性、密度 |

 在数学写作中，定义是基础设施。 它们不是定理之前的装饰。 它们决定定理的含义。 

强大的定义可以最大限度地减少歧义，支持示例，揭示结构，并使后面的推理变得更短。
