## 3.2 正式语言与非正式语言

 数学使用正式和非正式语言。 正式语言提供精确性。 非正式语言具有可读性。 一本好的数学课本通常会将它们结合起来。 

形式语言具有精确的语法。 它指定哪些表达式有效以及语句如何形成。 例如，一阶逻辑具有变量、量词、谓词、连接词和构成规则。```text
∀x (P(x) → Q(x))
```这个表达是正式的。 其结构是明确的。 量词绑定`x`。 蕴涵连接两个谓词。 含义取决于解释，但语法是固定的。 

非正式数学语言是书籍、论文、讲座和笔记中使用的语言。 它将普通散文与符号混合在一起。```text
Every continuous function on a compact interval is bounded.
```这句话虽然不拘形式，但却有精确的数学内容。 在完全正式的设置中，人们将指定域、余域、拓扑、连续性定义和有界性定义。 

| 语言 | 实力| 成本|
 | ------------------- | ---------------------------------- | -------------------- |
 | 正式| 准确可查| 详细 |
 | 非正式| 可读且高效 | 依赖惯例|
 | 结构化非正式| 对人类来说足够精确| 需要纪律 |

 大多数数学都是用结构化的非正式语言编写的。 它并不完全正式，但受到定义、符号和证明规范的约束。 

定理的完整正式版本可能比教科书版本长得多。 例如：```text
If f is continuous on [a,b], then f is bounded.
```形式化版本必须对实数、区间、函数、连续性、顺序、紧致性或完整性以及有界性进行编码。 非正式句子使用共享的数学上下文压缩所有这些。 

这种压缩很有用。 它让读者专注于主要思想。 但压缩可以隐藏假设。 

| 非正式短语 | 隐藏的上下文|
 | ----------------- | ------------------------------------------------- |
 |`f is continuous`| 域、共域、拓扑 |
 |`G is simple`| 团体？ 图形？ 模块？ |
 |`x is small`| 相对于什么界限？ |
 |`almost all`| 有限的例外？ 措施一？ 概率一？ |
 |`canonical`| 由什么普遍属性或惯例定义？ |

 当隐藏的上下文改变了陈述的真实性时，非正式语言就会变得危险。 

例如：```text
Every polynomial has a root.
```对于非常数多项式，这对于实数是错误的，但对于复数是正确的。 该语句需要一个字段。 

更好的：```text
Every nonconstant polynomial over C has a root in C.
```改进后的版本说明了环境结构。 

形式语言通过强制每个对象进入定义的上下文来避免这种歧义。 证明助理严格执行此操作。 术语必须有类型。 定理必须陈述所有假设。 证明必须使用有效的规则。 

非正式数学依赖于读者的推理。 当观众分享惯例时，这是有效的。 当约定不清楚时它就会失败。 

一个好的数学句子使推理负荷变得合理。 它不需要说明每个基本细节，但应该说明影响结果的每个假设。 

贫穷的：```text
The function has an inverse.
```更好的：```text
The function f : X → Y has an inverse because it is bijective.
```更好的是，当结构很重要时：```text
The continuous bijection f : X → Y has a continuous inverse, since X is compact and Y is Hausdorff.
```最终版本包括拓扑结论所需的假设。 

正式和非正式语言的证明风格也有所不同。 

形式证明是一系列有效的推理步骤。 每个步骤都必须由规则、公理、定义或先前证明的定理来证明。 

非正式证明是为受过训练的读者编写的论证。 它可能会省略常规步骤，结合几个逻辑步骤，或使用诸如“清楚地”、“通过构造”或“不失一般性”等短语。 

这些短语应该谨慎使用。 

| 短语 | 可接受时| 风险|
 | -------------------------- | ---------------------------------------------------------- | ------------------------------------------ |
 | 显然| 这一步确实是立即的| 隐藏着一个强硬的论点|
 | 通过建设 | 该对象被定义为满足属性 | 建设可能无法确保这一点|
 | 不失一般性| 对称或简化是明确的 | 可能会错误地丢弃案例 |
 | 如下| 以前的结果直接适用 | 缺少假设|
 | 规范| 选择是自然而不变的 | 可能隐藏任意选择 |

 默认情况下，非正式证明并不弱。 这是一种不同的表示。 如果每个省略的步骤都是可恢复的，那么写得好的非正式证明就可以是严格的。 

关键标准是可重构性。 当有能力的读者可以将其扩展为正式证明而不改变其含义时，证明是可以接受的。 

| 标准| 问题 |
 | ------------- | ------------------------------------------- |
 | 当地意义| 所有符号都在附近定义吗？ |
 | 假设| 是否陈述了必要的假设？ |
 | 推论| 每一步是否合理？ |
 | 范围 | 变量和案例是否受到控制？ |
 | 扩展| 原则上证明可以形式化吗？ |

 形式化对于检查边缘情况很有用。 它强制对域、强制、平等、终止和依赖进行显式处理。 许多非正式错误是类型错误或缺少假设。 

例如，一个证明可以定义一个映射：```text
f : A → B
```但稍后申请`f`到外部的元素`A`。 非正式的符号可能会隐藏这一点。 正式的系统拒绝它。 

类似地，证明可以使用除以`x`无需证明`x ≠ 0`。 正式证明需要缺失的条件。 

非正式写作应该借鉴这一规则。 在使用操作之前，请确保已定义该操作。 在调用定理之前，检查其假设。 在声明唯一性之前，先指定相等关系。 

最好的数学语言在不同级别之间移动。 

| 水平| 示例| 使用|
 | -------- | ---------------------------------- | ---------------- |
 | 散文| “地图保留了距离。” | 解释想法 |
 | 象征性|`d(f(x), f(y)) = d(x,y)`| 国家财产|
 | 正式| 在证明助手中输入定理 | 启用检查 |

 这些级别应该相互支持。 散文解释了原因。 符号说明什么。 形式化检查如何进行。 

一个有用的写作模式是：```text
Introduce the objects.
State the claim.
Give the idea.
Write the proof.
Record the dependencies.
```例如：```text
Let V be a finite-dimensional vector space over a field k, and let T : V → V be linear. We show that T has an invariant subspace under the stated hypotheses. The idea is to use the factorization of the characteristic polynomial. ...
```这种风格在符号操作开始之前提供了足够的结构。 

数学语言应该是正式的，因为歧义会改变意义；而非正式的，因为正式的细节会模糊论证。 精度和可读性不是敌人。 它们是同一设计问题的两个约束。
